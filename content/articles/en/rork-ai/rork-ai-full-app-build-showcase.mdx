## Introduction — What Rork AI Actually Changes About App Development

A lot of people have an app idea they've never acted on. Usually not because the idea isn't good — but because the barrier to execution is genuinely high. Learning Swift or Kotlin, understanding backend architecture, designing UI that doesn't look amateurish, navigating App Store submission — the traditional path is long, and most of it has nothing to do with whether your core idea is valuable.

Rork AI changes this calculus significantly. You describe what you want to build in plain language, and it generates working code, handles UI structure, and iterates with you as the idea evolves. It doesn't eliminate the need for judgment and creativity — but it removes a lot of the friction between having an idea and having a working app.

This guide walks through the full development lifecycle for a **Habit Tracker app** — a genuinely useful product with enough complexity to illustrate all the key patterns. By the end, you'll have a clear picture of how to use Rork AI to build something real.

## Phase 1: Defining What You're Building

### Your First Conversation with Rork AI

Before writing a line of code, you need a clear description of what you're building. The quality of your initial prompt shapes everything that follows. Here's the kind of framing that works well:

```
"I want to build a habit tracking app for iOS and Android.

Core features:
- Add and delete habits
- Daily check-in (mark habit as complete)
- Streak counter (consecutive days completed)
- Weekly and monthly achievement rate charts
- Daily reminder notifications

Target users: people in their 20s–30s focused on self-improvement
Design direction: clean, motivating, minimal clutter"
```

This gives Rork AI enough to generate a meaningful initial scaffold — screen structure, navigation, and basic components — rather than asking it to guess at your intentions.

### Iterating to Clarify Requirements

The first output is a starting point, not a final answer. After reviewing it, you refine:

"The habit cards look too generic. Let me set a custom color for each habit, shown as a left border accent on the card. Also add an emoji picker so each habit has a visual icon."

This kind of specific, concrete feedback drives Rork AI to produce exactly what you have in mind rather than something plausible but off.

## Phase 2: Building the UI

### Screen Architecture

A habit tracker needs four screens: Home (today's habits), Stats (charts and progress), Add/Edit Habit, and Settings. A bottom tab navigator handles the top-level navigation, with the add-habit flow opening as a modal from the home screen.

Tell Rork AI the full navigation structure upfront:

```
"Implement the app with a BottomTabNavigator containing three tabs: Home, Stats, Settings.
Home shows today's habit list with check-off capability.
Stats shows a weekly completion chart using recharts.
Settings handles notification configuration and data reset.
Habit creation opens as a modal from a '+' button in the Home header."
```

### Component Structure

Clean component boundaries make future changes much easier. Being explicit about this in your Rork AI prompt pays off:

```
"Structure the components as follows:
- HabitCard.tsx: renders a single habit with check-off interaction
- HabitList.tsx: renders the list of HabitCards
- ProgressBar.tsx: shows completion percentage
- AddHabitModal.tsx: the habit creation modal
- WeeklyChart.tsx: the weekly stats chart

Keep color definitions in shared/colors.ts."
```

## Phase 3: Data Management

### Local Storage with Custom Hooks

Habit data needs to persist between sessions. Ask Rork AI for a clean hooks-based implementation:

```
"Implement data management using AsyncStorage with three custom hooks:
- useHabits: CRUD operations for habits
- useCompletions: today's check-in state
- useStreaks: consecutive day streak calculation

Include TypeScript types for all data structures."
```

Here's the generated `useHabits` hook:

```typescript
import { useState, useEffect, useCallback } from 'react';
import AsyncStorage from '@react-native-async-storage/async-storage';
import { Habit } from '../types';

const HABITS_KEY = '@habits';

export const useHabits = () => {
  const [habits, setHabits] = useState<Habit[]>([]);
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    loadHabits();
  }, []);

  const loadHabits = async () => {
    try {
      const stored = await AsyncStorage.getItem(HABITS_KEY);
      if (stored) setHabits(JSON.parse(stored));
    } catch (error) {
      console.error('Failed to load habits:', error);
    } finally {
      setLoading(false);
    }
  };

  const addHabit = useCallback(async (habit: Omit<Habit, 'id' | 'createdAt'>) => {
    const newHabit: Habit = {
      ...habit,
      id: Date.now().toString(),
      createdAt: new Date().toISOString(),
    };
    const updated = [...habits, newHabit];
    setHabits(updated);
    await AsyncStorage.setItem(HABITS_KEY, JSON.stringify(updated));
    return newHabit;
  }, [habits]);

  const deleteHabit = useCallback(async (habitId: string) => {
    const updated = habits.filter(h => h.id !== habitId);
    setHabits(updated);
    await AsyncStorage.setItem(HABITS_KEY, JSON.stringify(updated));
  }, [habits]);

  return { habits, loading, addHabit, deleteHabit };
};
```

### Streak Calculation Logic

The streak counter is the emotional core of any habit app — getting the logic right matters:

```typescript
export const calculateStreak = (completions: string[]): number => {
  if (completions.length === 0) return 0;

  const today = new Date();
  today.setHours(0, 0, 0, 0);
  
  let streak = 0;
  let checkDate = new Date(today);
  
  const sortedCompletions = [...completions]
    .map(d => new Date(d))
    .sort((a, b) => b.getTime() - a.getTime());
  
  for (const date of sortedCompletions) {
    const completionDay = new Date(date);
    completionDay.setHours(0, 0, 0, 0);
    
    const diffDays = Math.round(
      (checkDate.getTime() - completionDay.getTime()) / (1000 * 60 * 60 * 24)
    );
    
    if (diffDays === 0 || diffDays === 1) {
      streak++;
      checkDate = completionDay;
    } else {
      break;
    }
  }
  
  return streak;
};
```

## Phase 5: Design Polish

### Iterating on Visual Design

Once core functionality is working, design polish makes the difference between an app people use and one they delete after a week.

Specific design requests work much better than vague ones:

"Add a completion percentage ring to the home screen header — show today's completed habits as a circular progress indicator alongside the date. Use the success green from our color palette."

"Make the Stats screen feel more celebratory. When the weekly completion rate is above 80%, show a small confetti animation. Keep it subtle."

Large design changes are also possible — dark mode, typography updates, spacing system changes — and Rork AI can handle them if you frame the request clearly.

## Practical Tips for Building with Rork AI

**Work in small increments.** Large change requests produce inconsistent results. "Redesign all screens" is harder to get right than "update the habit card layout." Break work into focused, specific requests.

**Read the generated code.** Not necessarily every line, but understand what each piece does. The better your mental model of the codebase, the better your prompts become — and the faster you can spot when something doesn't look right.

**Use version control.** Commit before significant changes. Rork AI might take a direction you don't want, and having a clean rollback point saves frustration.

**Name things explicitly.** If you want Rork AI to use specific variable names, component names, or file paths, say so. It will follow your naming conventions when you state them clearly.
