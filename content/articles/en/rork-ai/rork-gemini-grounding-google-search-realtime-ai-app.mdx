There's a specific type of user complaint that shows up after launching an AI-powered app: "Why doesn't it know about recent news?" It's a fair question, and the answer — that large language models have a training cutoff and simply don't know about things that happened after that point — tends to frustrate users who think of AI assistants as omniscient.

The frustration is understandable. From a user's perspective, if the AI assistant knows about ancient Roman history and can write code in Rust, why would it not know what happened in tech news last week? The distinction between "knowledge encoded in model weights" and "access to live information" is not intuitive to most people. But once a user gets a confidently wrong answer about something recent — a price that changed, a product that got discontinued, a news story that broke yesterday — trust is damaged in a way that's hard to recover.

Grounding with Google Search is Gemini's answer to this problem. When enabled, Gemini autonomously runs Google Search queries when it determines that current information would improve its response, and then uses those search results as the basis for what it tells your users. The model stays the same. What changes is that it now has access to what happened this week, not just what it learned before its training cutoff.

This guide covers everything you need to integrate Grounding into a Rork app: the architecture decisions you need to make upfront, three working code examples you can adapt directly, how to read and display `groundingChunks` in a citation UI, the production pitfalls that will catch you off-guard if you're not prepared, and a practical framework for deciding when Grounding is worth the additional cost.

## What Grounding Actually Does — and What It Doesn't

Let's be precise about the problem Grounding solves. LLMs are trained on a snapshot of text from the internet up to a certain date. After that date, the model is frozen. Gemini 2.5 Pro might have knowledge through early 2025, but it doesn't know what happened in April 2026 unless you tell it — or unless it can go look it up.

For many app use cases, this knowledge cutoff genuinely doesn't matter. A code assistant doesn't need to know today's news. A creative writing tool works fine with decade-old training data. A math tutor has no use for real-time search. But certain categories of app features are fundamentally broken without access to current information:

- **News and current events** — If a user asks about a breaking story, a sports result from last night, or a regulatory change announced this week, the model can only guess or confess ignorance
- **Prices and market information** — Product pricing, stock quotes, exchange rates, and real estate values change constantly; answers from training data will often be wrong
- **Recent software and product releases** — A user asking about the latest version of a library or a newly announced product will get outdated information
- **Upcoming events and schedules** — Concerts, conferences, release dates, and deadlines that were scheduled after the training cutoff simply don't exist in the model's knowledge

Without Grounding, the model handles these queries in one of two ways: it either admits uncertainty ("I don't have information about events after my training cutoff"), or worse, it confidently provides information from its training data that may now be outdated or wrong. The second case is more common and more damaging.

When you enable Grounding, here's the internal sequence that happens on each relevant request:

1. Gemini evaluates whether the user's query requires current information to answer well
2. If yes, it generates one or more Google Search queries automatically, calibrated to retrieve useful results
3. It retrieves and reads the content of those search results
4. It synthesizes a response grounded in the retrieved content, not just its training data
5. It returns the response alongside `groundingChunks` (the sources it consulted), `webSearchQueries` (the exact searches it ran), and `groundingSupports` (which parts of the response are supported by which sources)

Your app code doesn't need to orchestrate any of this search-and-retrieve process. You enable the tool, and Gemini handles the rest. Your responsibility is to handle the metadata that comes back — and to decide when to enable Grounding in the first place.

That last point matters more than it might seem. Grounding adds both cost and latency to every request where it's activated. For queries that don't need current information, enabling Grounding produces no quality improvement while inflating your API costs. Part of building a well-designed Grounding integration is deciding, for each type of query your app handles, whether the fresh information is worth paying for.

## Architecture: Keeping the API Key Server-Side

Before writing any Gemini-specific code, there's an architectural requirement that applies to every AI-powered mobile app: **do not embed your Gemini API key in the app bundle**.

This isn't a matter of following best practices for its own sake. React Native bundles are JavaScript files that run on user devices. They can be extracted, read, and analyzed. If your API key is in the bundle — even if you've tried to obscure it with environment variable bundling — a determined attacker can find it. An exposed key with Grounding enabled can generate substantial API costs very quickly, since Grounded requests consume significantly more tokens than standard requests.

The correct architecture puts a thin server layer between your app and the Gemini API:

```
Rork App → Cloudflare Workers API → Gemini API (with Grounding)
```

This backend layer handles the API key securely, lets you add rate limiting and authentication, and gives you a single place to modify the Gemini configuration without pushing a new app release.

The following implementation uses Cloudflare Workers with Hono. If you haven't set up a Workers backend yet, the [Hono + Cloudflare Workers REST API guide](/en/articles/rork-dev/rork-hono-cloudflare-workers-rest-api-guide) covers the initial setup in detail. Here's the Grounding endpoint:

```javascript
// workers/src/gemini-grounding.ts
import { Hono } from "hono";
import { cors } from "hono/cors";

const app = new Hono();

app.use("/api/*", cors());

app.post("/api/grounded-search", async (c) => {
  const { query, useGrounding } = await c.req.json();

  if (\!query || typeof query \!== "string") {
    return c.json({ error: "query is required" }, 400);
  }

  const apiKey = c.env.GEMINI_API_KEY;

  // Dynamically toggle Grounding based on the client's request
  const tools = useGrounding ? [{ googleSearch: {} }] : [];

  const payload = {
    contents: [{ role: "user", parts: [{ text: query }] }],
    tools,
    generationConfig: {
      // Lower temperature when Grounding is on: we want factual responses, not creative ones
      temperature: useGrounding ? 0.3 : 0.7,
      maxOutputTokens: 1024,
    },
  };

  try {
    const response = await fetch(
      `https://generativelanguage.googleapis.com/v1beta/models/gemini-2.5-pro:generateContent?key=${apiKey}`,
      {
        method: "POST",
        headers: { "Content-Type": "application/json" },
        body: JSON.stringify(payload),
      }
    );

    if (\!response.ok) {
      const errorData = await response.json();
      return c.json({ error: "Gemini API error", details: errorData }, 500);
    }

    const data = await response.json();

    const candidate = data.candidates?.[0];
    const text = candidate?.content?.parts?.[0]?.text ?? "";
    const groundingMetadata = candidate?.groundingMetadata ?? null;

    return c.json({
      text,
      groundingChunks: groundingMetadata?.groundingChunks ?? [],
      webSearchQueries: groundingMetadata?.webSearchQueries ?? [],
      groundingSupports: groundingMetadata?.groundingSupports ?? [],
    });
  } catch (err) {
    return c.json({ error: "Internal error", message: String(err) }, 500);
  }
});

export default app;
```

Set `GEMINI_API_KEY` as a secret using `wrangler secret put GEMINI_API_KEY`. Your mobile app code never touches the key.

One thing worth noting about the temperature setting: when Grounding is enabled, lowering the temperature (here to 0.3) makes the model stick closer to the factual content in the search results rather than embellishing or inferring. For news and price queries, you want the model to report what the sources say, not to synthesize creatively. For standard queries without Grounding, a higher temperature (0.7) produces more natural, varied responses.

## Implementation Pattern 1: News Summarizer

The most intuitive Grounding use case is news summarization. A user types a topic, and your app returns a concise summary of what's currently happening, backed by real sources they can click through to.

```typescript
// components/NewsSearchScreen.tsx
import React, { useState } from "react";
import {
  View,
  Text,
  TextInput,
  TouchableOpacity,
  ScrollView,
  ActivityIndicator,
  StyleSheet,
  Linking,
} from "react-native";

interface GroundingChunk {
  web?: {
    uri: string;
    title: string;
  };
}

interface SearchResult {
  text: string;
  groundingChunks: GroundingChunk[];
  webSearchQueries: string[];
}

const API_BASE = "https://your-worker.your-subdomain.workers.dev";

  const [query, setQuery] = useState("");
  const [result, setResult] = useState<SearchResult | null>(null);
  const [loading, setLoading] = useState(false);
  const [error, setError] = useState<string | null>(null);

  const handleSearch = async () => {
    if (\!query.trim()) return;

    setLoading(true);
    setError(null);
    setResult(null);

    try {
      const response = await fetch(`${API_BASE}/api/grounded-search`, {
        method: "POST",
        headers: { "Content-Type": "application/json" },
        body: JSON.stringify({
          query: `Search for the latest news about the following topic and summarize it in 150 words or fewer. Focus on information from 2025-2026. Topic: ${query}`,
          useGrounding: true,
        }),
      });

      if (\!response.ok) {
        throw new Error(`API Error: ${response.status}`);
      }

      const data: SearchResult = await response.json();
      setResult(data);
    } catch (err) {
      setError(
        err instanceof Error
          ? err.message
          : "A network error occurred. Please try again."
      );
    } finally {
      setLoading(false);
    }
  };

  return (
    <ScrollView style={styles.container}>
      <Text style={styles.title}>Latest News</Text>
      <TextInput
        style={styles.input}
        value={query}
        onChangeText={setQuery}
        placeholder="Enter a topic (e.g. AI regulation 2026)"
        returnKeyType="search"
        onSubmitEditing={handleSearch}
      />
      <TouchableOpacity
        style={[styles.button, loading && styles.buttonDisabled]}
        onPress={handleSearch}
        disabled={loading || \!query.trim()}
      >
        {loading ? (
          <ActivityIndicator color="#fff" />
        ) : (
          <Text style={styles.buttonText}>Search</Text>
        )}
      </TouchableOpacity>
      {error && (
        <View style={styles.errorBox}>
          <Text style={styles.errorText}>{error}</Text>
        </View>
      )}
      {result && (

          <Text style={styles.answerText}>{result.text}</Text>

            <View style={styles.queryBox}>
              <Text style={styles.queryLabel}>Searched for:</Text>
              {result.webSearchQueries.map((q, i) => (
                <Text key={i} style={styles.queryText}>"{q}"</Text>
              ))}
            </View>
          )}
          {result.groundingChunks.length > 0 && (
            <View style={styles.sourcesBox}>
              <Text style={styles.sourcesLabel}>Sources:</Text>
              {result.groundingChunks
                .filter((chunk) => chunk.web)
                .map((chunk, i) => (
                  <TouchableOpacity
                    key={i}
                    onPress={() => chunk.web?.uri && Linking.openURL(chunk.web.uri)}
                    style={styles.sourceItem}
                  >
                    <Text style={styles.sourceTitle} numberOfLines={1}>
                      {chunk.web?.title ?? "(no title)"}
                    </Text>
                    <Text style={styles.sourceUrl} numberOfLines={1}>
                      {chunk.web?.uri}
                    </Text>
                  </TouchableOpacity>
                ))}
            </View>

        </View>
      )}
    </ScrollView>
  );
}
```

There's a design decision embedded in this component that's worth making explicit: it shows `webSearchQueries` alongside the response. This is optional but valuable. When users can see that Gemini ran a search for "AI regulation EU 2026 update" on their behalf, it reinforces that the answer is based on real research, not a guess. In user testing across several apps, this single addition reduced "is this accurate?" follow-up questions significantly — users who see the search queries seem to internalize that the AI actively went and looked things up rather than drawing on static knowledge.

## Implementation Pattern 2: Product Research Assistant

A second strong use case is product and price research. Here's a custom hook that wraps the Grounding search in a reusable form:

```typescript
// hooks/useProductSearch.ts
import { useState, useCallback } from "react";

interface ProductInfo {
  summary: string;
  priceInfo: string | null;
  sources: Array<{ title: string; uri: string }>;
}

const API_BASE = "https://your-worker.your-subdomain.workers.dev";

export function useProductSearch() {
  const [isLoading, setIsLoading] = useState(false);
  const [productInfo, setProductInfo] = useState<ProductInfo | null>(null);
  const [error, setError] = useState<string | null>(null);

  const searchProduct = useCallback(async (productName: string) => {
    setIsLoading(true);
    setError(null);
    setProductInfo(null);

    const prompt = `
Search for current information about the following product and provide:
1. A brief product overview (2-3 sentences)
2. Current retail price range if you can find it
3. One recent news item or update if available (2025-2026)

Product: ${productName}

If you cannot find reliable current information for any of these points, say so
explicitly rather than speculating. Do not fabricate prices or dates.
    `.trim();

    try {
      const response = await fetch(`${API_BASE}/api/grounded-search`, {
        method: "POST",
        headers: { "Content-Type": "application/json" },
        body: JSON.stringify({ query: prompt, useGrounding: true }),
      });

      if (\!response.ok) {
        throw new Error("Search failed. Please try again.");
      }

      const data = await response.json();

      // Basic price extraction — use two-step JSON mode for production
      const priceMatch = data.text.match(
        /\$[0-9,]+(?:\.[0-9]{2})?(?:\s*[-–]\s*\$[0-9,]+(?:\.[0-9]{2})?)?/
      );

      setProductInfo({
        summary: data.text,
        priceInfo: priceMatch ? priceMatch[0] : null,
        sources: data.groundingChunks
          .filter((c: { web?: { uri: string; title: string } }) => c.web)
          .map((c: { web: { uri: string; title: string } }) => ({
            title: c.web.title,
            uri: c.web.uri,
          })),
      });
    } catch (err) {
      setError(err instanceof Error ? err.message : "An error occurred");
    } finally {
      setIsLoading(false);
    }
  }, []);

  return { searchProduct, isLoading, productInfo, error };
}
```

The prompt here includes an explicit instruction: "Do not fabricate prices or dates." This matters more than it might seem. Without this kind of instruction, LLMs sometimes blend search-retrieved information with training data to fill in gaps, which can produce plausible-sounding but incorrect numbers. Explicitly instructing the model to acknowledge uncertainty when its grounded sources don't have an answer produces more honest responses.

## Understanding groundingChunks and Inline Citation UI

The Grounding response includes three pieces of metadata that are worth understanding in detail, because how you use them affects both the quality of your UI and the trust users place in the AI responses.

`groundingChunks` is an array of sources Gemini consulted when forming its response. Each entry can contain a `web` object with `uri` and `title`. These are the pages Google Search returned that Gemini actually read.

`groundingSupports` is more granular — it maps specific passages in the response text to specific chunks. Each entry has a `segment` with `startIndex` and `endIndex` (character positions in the response), `groundingChunkIndices` (which chunks support that segment), and `confidenceScores` (how strongly each chunk supports the claim).

`webSearchQueries` is the list of search queries Gemini generated and ran. This is useful for transparency UI and also for debugging cases where the search seems to have gone off-track.

For a basic citation UI, `groundingChunks` is all you need. For an inline citation experience where specific sentences are linked to specific sources (similar to how academic papers cite sources), you'd use `groundingSupports` to map character positions in the response text to chunk indices.

Here's a filtering function that uses confidence scores to show only the most reliable sources:

```typescript
interface GroundingChunk {
  web?: { uri: string; title: string };
}

interface GroundingSupport {
  groundingChunkIndices: number[];
  confidenceScores: number[];
  segment: { startIndex: number; endIndex: number; text: string };
}

function filterHighConfidenceSources(
  groundingChunks: GroundingChunk[],
  groundingSupports: GroundingSupport[],
  minConfidence = 0.5
): GroundingChunk[] {
  return groundingChunks.filter((_, chunkIndex) => {
    // Find all supports that reference this chunk
    const relevantSupports = groundingSupports.filter((support) =>
      support.groundingChunkIndices.includes(chunkIndex)
    );

    if (relevantSupports.length === 0) {
      // If no supports reference this chunk, include it anyway
      // (some chunks are used for context without direct citation)
      return true;
    }

    // Calculate average confidence across all supports that cite this chunk
    let totalScore = 0;
    let count = 0;
    for (const support of relevantSupports) {
      const indexInSupport = support.groundingChunkIndices.indexOf(chunkIndex);
      if (indexInSupport \!== -1 && support.confidenceScores[indexInSupport] \!== undefined) {
        totalScore += support.confidenceScores[indexInSupport];
        count++;
      }
    }

    return count === 0 || totalScore / count >= minConfidence;
  });
}
```

## Three Production Pitfalls to Avoid

After working with Grounding across multiple apps, there are three failure modes that show up repeatedly. All of them are preventable with the right setup, but none of them are obvious until you've been burned.

**Pitfall 1: Token costs multiply in ways that aren't intuitive**

The cost of a Grounded request isn't just "base request + Google Search fee." When Grounding activates, the model retrieves the full text content of multiple search result pages and includes that content as input context. Depending on the query and the pages retrieved, a request that costs 200 input tokens without Grounding might cost 1,500 to 3,000 tokens with it.

The practical implication is that you cannot estimate Grounding costs from your development testing. Test queries are usually short, clean, and focused — they often retrieve concise search results. Production queries from real users tend to be messier, more ambiguous, and trigger longer, more numerous search results. Budget conservatively and instrument actual token usage from the start. You can log the token counts from the Gemini API response and track them in your backend.

Gemini 2.5 Flash supports Grounding as well and is meaningfully cheaper per token. For most news summarization and product research use cases, Flash quality is sufficient. Consider Flash as the default and reserve Pro for queries that explicitly need deeper reasoning.

**Pitfall 2: Off-topic search results degrade response quality**

Gemini's autonomous search query generation is impressive in general but occasionally misfires. The model generates search queries based on its understanding of the user's request, and sometimes that understanding diverges from what the user actually meant. When this happens, the search results that come back are adjacent but not quite right, and the model incorporates them into a response that sounds plausible but isn't answering the real question.

The most effective mitigation is being explicit in your prompt about what to search for and what to do when the results aren't helpful:

```javascript
// Vague prompt — leaves Gemini to guess what's relevant
const query = `Tell me about ${topic}`;

// Explicit prompt — gives Gemini better search direction and a fallback
const query = `
Search specifically for recent news and updates (2025-2026) about: ${topic}

Focus on factual reporting from reliable sources.
If the search results don't contain relevant or current information about this
specific topic, respond with: "I couldn't find reliable current information
about [topic]. Here's what I know from my training data: [summary]."

Do not present outdated information as current.
`.trim();
```

The explicit fallback instruction ("if you can't find relevant information, say so") is important. Without it, the model sometimes weaves together unrelated search results with training data to produce an answer that has the form of a grounded response but lacks its accuracy.

**Pitfall 3: You can't use JSON mode and Grounding simultaneously**

This trips up a lot of developers because it's not prominently documented. Setting `response_mime_type: "application/json"` in `generationConfig` while also including Grounding tools causes an API error. The two features are mutually exclusive in the current API.

If you need structured output from a Grounded response — for example, you want to extract price, availability, and summary as separate fields — you have two options. The first is to parse the free-text response with regex or string matching (fragile for production). The second, and more reliable, approach is a two-step pattern:

```typescript
async function groundedStructuredSearch(topic: string) {
  // Step 1: Grounded call to get current information as free text
  const groundedResponse = await fetch(`${API_BASE}/api/grounded-search`, {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ query: `Find current information about: ${topic}`, useGrounding: true }),
  });
  const { text: rawInfo, groundingChunks } = await groundedResponse.json();

  // Step 2: Structure the free text as JSON (no Grounding, uses JSON mode)
  const structurePayload = {
    contents: [{
      role: "user",
      parts: [{
        text: `
Extract the following fields from this text and return as JSON only (no explanation):
Text: ${rawInfo}

Schema:
{
  "summary": "one-sentence overview",
  "price": "price range if mentioned, or null",
  "lastUpdated": "date mentioned in text, or null",
  "confidence": "high | medium | low (based on how specific the information is)"
}
        `.trim(),
      }],
    }],
    generationConfig: {
      response_mime_type: "application/json",
      temperature: 0.1,
    },
  };

  // This second call hits Gemini directly or through a separate backend endpoint
  const structureResponse = await fetch(
    `https://generativelanguage.googleapis.com/v1beta/models/gemini-2.5-flash:generateContent?key=${apiKey}`,
    { method: "POST", headers: { "Content-Type": "application/json" }, body: JSON.stringify(structurePayload) }
  );
  const structureData = await structureResponse.json();
  const parsed = JSON.parse(structureData.candidates[0].content.parts[0].text);

  return { ...parsed, sources: groundingChunks };
}
```

The two-step pattern adds latency but produces reliable structured output. In practice, the structuring step (using Gemini Flash on a short prompt) takes under a second, so the total latency is still acceptable for user-initiated searches.

## Dynamic Grounding Toggle and Fallback Strategy

The production pattern that balances quality and cost best is deciding per-request whether Grounding should be enabled, and falling back gracefully when Grounded requests fail:

```typescript
function shouldUseGrounding(query: string): boolean {
  const groundingKeywords = [
    "latest", "recent", "current", "today", "now", "new",
    "price", "cost", "how much", "available", "in stock",
    "news", "release", "just released", "announced", "launched",
    "this year", "this month", "2025", "2026",
    "update", "changed", "different now", "currently",
  ];
  const lower = query.toLowerCase();
  return groundingKeywords.some((kw) => lower.includes(kw));
}

async function searchWithFallback(query: string): Promise<SearchResult> {
  const useGrounding = shouldUseGrounding(query);

  try {
    const controller = new AbortController();
    // Grounded calls can take 5-15 seconds; set a generous timeout
    const timeoutId = setTimeout(() => controller.abort(), 20000);

    const response = await fetch(`${API_BASE}/api/grounded-search`, {
      method: "POST",
      headers: { "Content-Type": "application/json" },
      body: JSON.stringify({ query, useGrounding }),
      signal: controller.signal,
    });

    clearTimeout(timeoutId);

    if (\!response.ok) throw new Error(`HTTP ${response.status}`);
    return await response.json();
  } catch (err) {
    if (useGrounding) {
      console.warn("Grounded request failed, falling back to standard:", err);
      // Retry without Grounding — at least give the user some response
      const fallback = await fetch(`${API_BASE}/api/grounded-search`, {
        method: "POST",
        headers: { "Content-Type": "application/json" },
        body: JSON.stringify({ query, useGrounding: false }),
      });
      if (\!fallback.ok) throw new Error("Both grounded and fallback requests failed");
      const fallbackData = await fallback.json();
      // Signal to the UI that this is a fallback response
      return { ...fallbackData, isFallback: true };
    }
    throw err;
  }
}
```

The `isFallback` flag lets your UI display a subtle note like "Live search unavailable — showing cached knowledge" when the Grounded call fails. This is better than silently showing training-data-only results as if they were current information.

## The Right First Step

The most productive way to learn Grounding is to enable it on one specific feature in an existing app — ideally a feature where you've already received user feedback about stale or wrong answers. Run it in production for a week. Look at the `webSearchQueries` that come back in your logs to understand how Gemini is interpreting your prompts. Compare the source quality in `groundingChunks` against what you'd expect. Watch your token costs.

The combination of real user queries and real search behavior will teach you more about where Grounding adds value — and where it doesn't — than any amount of local testing. From there, the dynamic toggle pattern gives you a clean way to expand Grounding coverage methodically as you understand which query types benefit most.

The goal isn't to enable Grounding everywhere. It's to enable it precisely where your users are currently getting answers that are confidently wrong, and to surface the sources behind those answers in a way that builds trust rather than undermining it.
