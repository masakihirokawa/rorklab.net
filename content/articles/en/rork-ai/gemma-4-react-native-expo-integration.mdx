Two common reasons developers want on-device AI in mobile apps: avoiding per-inference API costs at scale, and building features that work without network access. Gemma 4's E2B and E4B models address both. Apache 2.0 licensing means no distribution friction for App Store or Google Play. The trade-off is managing model file size and device performance — which this guide covers concretely.

## Three Integration Approaches for React Native

**Approach 1: llama.rn (Recommended — iOS + Android)**

[llama.rn](https://github.com/mybigday/llama.rn) is a React Native binding for llama.cpp that runs GGUF format models through native inference engines (Metal on iOS, Vulkan/CPU on Android). Gemma 4 GGUF variants are available on Hugging Face.

```bash
npm install llama.rn
cd ios && pod install
```

```typescript
import { LlamaContext, initLlama } from "llama.rn";
import RNFS from "react-native-fs";

let context: LlamaContext | null = null;

export async function loadGemma4(): Promise<void> {
  const modelPath = `${RNFS.DocumentDirectoryPath}/gemma4-4b.gguf`;

  context = await initLlama({
    model: modelPath,
    use_mlock: true,
    n_ctx: 4096,
    n_threads: 4,
    n_gpu_layers: 32, // Uses Metal (iOS) or Vulkan (Android) if available
  });
}

export async function generate(prompt: string): Promise<string> {
  if (\!context) throw new Error("Model not loaded");

  const result = await context.completion({
    prompt: `<start_of_turn>user
${prompt}<end_of_turn>
<start_of_turn>model
`,
    n_predict: 512,
    temperature: 0.7,
    top_p: 0.9,
    stop: ["<end_of_turn>"],
  });

  return result.text;
}
```

**Approach 2: ONNX Runtime Mobile**

For projects already in the ONNX ecosystem or needing minimal model size, ONNX-format Gemma 4 runs through `onnxruntime-react-native`:

```typescript
import { InferenceSession, Tensor } from "onnxruntime-react-native";

async function runInference(inputText: string): Promise<string> {
  const session = await InferenceSession.create(
    `${RNFS.DocumentDirectoryPath}/gemma4-e2b.onnx`
  );

  const inputIds = tokenize(inputText); // Implement tokenizer separately
  const inputTensor = new Tensor("int64", inputIds, [1, inputIds.length]);
  const outputs = await session.run({ input_ids: inputTensor });

  return decode(outputs.logits);
}
```

**Approach 3: Ollama on a Local Server (Prototyping)**

For development speed, run Ollama on a machine on the same network and call it from the app. Not suitable for production (requires network), but useful for rapid iteration before implementing on-device inference:

```typescript
const OLLAMA_URL = "http://192.168.1.100:11434"; // Local server IP

async function queryGemma4(prompt: string): Promise<string> {
  const resp = await fetch(`${OLLAMA_URL}/api/generate`, {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ model: "gemma4:4b", prompt, stream: false }),
  });
  const data = await resp.json();
  return data.response;
}
```

## Model File Size and Device Requirements

Quantization choice determines the trade-off between quality and storage/memory:

| Quantization | E2B Size | E4B Size | Min RAM | Quality |
|---|---|---|---|---|
| Q4_K_M | ~1.4 GB | ~2.5 GB | 3 GB | Good |
| Q5_K_M | ~1.8 GB | ~3.1 GB | 4 GB | Better |
| Q8_0 | ~2.8 GB | ~5.0 GB | 5 GB | Best |

Q4_K_M is the practical default for most mobile deployments — acceptable quality degradation relative to full precision, manageable on mid-range hardware.

Downloading the model on first launch:

```typescript
import * as FileSystem from "expo-file-system";

export async function ensureModelReady(): Promise<string> {
  const dir = `${FileSystem.documentDirectory}models/`;
  await FileSystem.makeDirectoryAsync(dir, { intermediates: true });

  const modelPath = `${dir}gemma4-e4b-q4.gguf`;
  const info = await FileSystem.getInfoAsync(modelPath);

  if (\!info.exists) {
    const { uri } = await FileSystem.downloadAsync(
      "https://your-cdn.example.com/gemma4-e4b-q4.gguf",
      modelPath
    );
    return uri;
  }
  return modelPath;
}
```

## App Store and Google Play Considerations

**App Store**: If the total app size including model exceeds 4 GB, you need App Thinning and On-Demand Resources (ODR) to separate the model from the main binary. Smaller models (E2B Q4_K_M at ~1.4 GB) stay well under Apple's 4 GB OTA download limit.

**Google Play**: The 100 MB APK/AAB limit means model files always need to be downloaded post-install. Play Asset Delivery (PAD) handles this automatically — the model downloads on first launch, with progress visible to the user.

## Realistic Performance on Current Hardware

Running E4B (Q4_K_M) through llama.rn on an iPhone 15 Pro (A17 Pro chip, Metal acceleration): expect roughly 10–20 tokens/second. That's too slow for real-time chat but entirely adequate for background tasks — voice input → text generation → summary → notification, all completing in under 10 seconds for typical use cases.

Gemma 4's audio input is native to E2B and E4B, but accessing it via llama.rn currently requires separate audio preprocessing. A pragmatic approach: handle audio capture with Expo's AV library or Whisper.rn, then pass the transcript to Gemma 4 for natural language processing. The voice-to-insight pipeline works well in this configuration, even if the two steps aren't a single model call.

Start with text generation tasks in your Rork-designed UI. Once that works, layer in audio and OCR features using the preprocessing wrappers. The incremental approach is faster than trying to wire up full multimodal inference in one step.
