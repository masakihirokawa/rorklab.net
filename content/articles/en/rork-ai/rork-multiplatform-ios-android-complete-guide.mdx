"My Rork app only works on iOS." This is one of the most common issues developers hit after building their first Rork project. The root cause is almost always the same: the initial prompt didn't establish cross-platform intent, so Rork defaulted to iOS-specific patterns.

This guide covers how to build iOS/Android compatible from the start, and how to retrofit compatibility when you haven't.

## How Rork's iOS/Android Support Works

Rork generates React Native (Expo) code. React Native is a framework for building cross-platform apps from a shared codebase — but only if you design for it. Certain prompting patterns cause iOS-specific code to appear in the output.

### Common sources of iOS-only code

**Font specifications**: "Use San Francisco font" or "use the system font" — San Francisco is iOS-only. Android defaults to Roboto.

**iOS-specific visual design**: "Make it look like an iPhone app" — this produces UI that feels unnatural on Android.

**SafeArea handling**: Code written specifically for iPhone notch and Dynamic Island can produce excessive padding on Android devices.

## Cross-Platform Prompting Patterns

The easiest way to avoid compatibility problems is to establish intent from the first prompt.

### Effective prompt structure

```
Create a [app name] that works correctly on both iOS and Android.
Prioritize designs that feel natural to users on both platforms 
over platform-specific aesthetics.
Use system default fonts, and handle SafeArea appropriately 
for both iOS and Android.
```

### What to avoid

```
Create an iPhone app (→ establishes iOS-first bias)
Use iOS-style design (→ skips Material Design considerations)
Use San Francisco font (→ Android incompatible)
```

## Handling Platform Differences in Code

When platform-specific implementations are genuinely required, use the Platform API to branch cleanly.

```javascript
// components/PlatformAwareButton.tsx
import { Platform, StyleSheet, TouchableOpacity, TouchableNativeFeedback, View } from 'react-native';

interface ButtonProps {
  onPress: () => void;
  children: React.ReactNode;
  style?: object;
}

export function PlatformButton({ onPress, children, style }: ButtonProps) {
  // Android: native ripple effect
  if (Platform.OS === 'android') {
    return (
      <TouchableNativeFeedback
        onPress={onPress}
        background={TouchableNativeFeedback.Ripple('#00000020', false)}
      >
        <View style={[styles.button, style]}>{children}</View>
      </TouchableNativeFeedback>
    );
  }
  
  // iOS: opacity feedback
  return (
    <TouchableOpacity onPress={onPress} style={[styles.button, style]}>
      {children}
    </TouchableOpacity>
  );
}

const styles = StyleSheet.create({
  button: {
    padding: 16,
    borderRadius: 8,
    backgroundColor: '#007AFF',
    alignItems: 'center',
  },
});
```

### Platform-appropriate fonts

```javascript
// styles/typography.ts
import { Platform } from 'react-native';

export const typography = {
  // System fonts: iOS uses San Francisco, Android uses Roboto
  body: {
    fontFamily: Platform.select({
      ios: 'System',
      android: 'Roboto',
    }),
    fontSize: 16,
    lineHeight: 24,
  },
  heading: {
    fontFamily: Platform.select({
      ios: 'System',
      android: 'Roboto',
    }),
    fontSize: 24,
    fontWeight: '700' as const,
  },
  // Custom fonts: embed in assets/fonts/ for both platforms
  custom: {
    fontFamily: 'NotoSansJP',
    fontSize: 16,
  },
};
```

## Cross-Platform Issues You Will Miss Without Real Devices

### Keyboard behavior

iOS and Android handle keyboard appearance differently. KeyboardAvoidingView requires platform-specific behavior:

```javascript
import { KeyboardAvoidingView, Platform } from 'react-native';

function FormScreen() {
  return (
    <KeyboardAvoidingView
      behavior={Platform.OS === 'ios' ? 'padding' : 'height'}
      style={{ flex: 1 }}
    >
      {/* Form content */}
    </KeyboardAvoidingView>
  );
}
```

### Android hardware back button

Android devices have a hardware back button. Without handling it, your app exits when users press it mid-flow.

```javascript
import { BackHandler } from 'react-native';
import { useEffect } from 'react';

function ScreenWithBackHandler({ navigation }: any) {
  useEffect(() => {
    const backAction = () => {
      navigation.goBack();
      return true; // Prevent default behavior (app exit)
    };
    
    const subscription = BackHandler.addEventListener('hardwareBackPress', backAction);
    return () => subscription.remove();
  }, [navigation]);
  
  return <View>{/* Content */}</View>;
}
```

### Scroll inertia

iOS has aggressive momentum scrolling; Android is more conservative. If scroll behavior feels wrong on one platform, `decelerationRate` is the control.

## Publishing to Both Stores

### App Store (iOS) checklist

- Apple Developer Program enrollment ($99/year)
- App Store Connect app registration
- Screenshots: iPhone 6.7-inch required; iPad if supporting iPad
- Privacy policy URL (required if collecting any data)

Rork Max's 2-click publishing handles the build and submission mechanics. The preparation above must be complete first.

### Google Play (Android) checklist

- Google Play Console registration ($25 one-time)
- AAB format build via `expo build:android`
- Content rating configuration
- Privacy policy URL

### Realistic publication timeline

```
Day 1–3:  Testing complete on both platforms
Day 4:    Submit to App Store (review: 1–2 days)
Day 4:    Submit to Google Play (review: 3–7 days)
Day 5–6:  App Store approval
Day 7–11: Google Play approval
Day 12:   Coordinated launch on both stores
```

Submit on the same day to minimize the gap between approvals. Apple typically reviews in 1–2 days; Google often takes longer.

## The One Practice That Prevents Most Cross-Platform Problems

Test on real devices early — not simulators. Simulators and emulators miss device-specific rendering differences, touch feedback nuances, and performance characteristics that only appear on actual hardware.

Using Rork Companion on both a physical iPhone and a physical Android device from week one catches problems when they are small fixes. Discovering them during App Store review submission costs days.

A cross-platform app reaches twice the market of a single-platform app with roughly the same development effort when you design for both from day one. That math favors building for both — and starting with the prompting patterns in this guide makes it achievable without doubling your workload.
