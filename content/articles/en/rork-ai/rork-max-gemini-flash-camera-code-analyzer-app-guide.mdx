Here's a scenario most developers have encountered: you're reading a technical book, watching a coding tutorial, or watching someone pair-program in front of you — and there's a piece of code you want to understand right now. You can't copy-paste it. You want an AI to look at it.

That's the app we're building in this guide.

Combining Rork Max with Gemini 2.5 Flash, you can have a working prototype of a camera-based code analyzer in a single day. But if you try to build this naively, you'll hit three specific production issues that almost no tutorial covers. This guide deals with all of them directly — and shows you the complete implementation.

## What You're Building

The app works like this: point your phone's camera at any code — from a textbook, a whiteboard, someone's screen, or a printed handout — and Gemini 2.5 Flash analyzes it and generates an explanation. Users can toggle between three explanation depths: Beginner, Intermediate, and Advanced. Results are saved to a local history. Free users get three analyses per day; premium subscribers get unlimited access.

The target audience is people actively learning to program: students, career switchers, developers picking up a new language. They encounter code in contexts where copy-paste isn't possible, and they want instant comprehension. The market is real and the value proposition is clear.

### Why Gemini 2.5 Flash Specifically

There are several ways to approach the camera-to-AI pipeline. A direct client call to Gemini Flash is the fastest to build. Running through a backend (Supabase Edge Functions or Cloudflare Workers) is more secure. Using an on-device model like Gemma 3 enables offline use.

For the architecture in this guide, we're going with a direct client call. The reasons: no infrastructure to set up, fast iteration, and you can migrate to a backend proxy once you've validated the product. We'll cover the API key security tradeoffs in the pitfalls section.

Gemini 2.5 Flash wins for this use case on two dimensions. Speed matters enormously in a camera app — users are holding their phone pointed at something, waiting. Flash's average response time runs about 40% faster than GPT-4o in practice, and costs significantly less per request. On code recognition specifically, Flash handles low-contrast images (dark IDE themes, printed text with shadows) better than you'd expect from a smaller model.

## Step 2: Camera Permissions — The Full Implementation

Rork Max generates a simplified version of camera permission handling. The simplified version will cause App Store rejections. Here's the complete implementation:

```tsx
// app/camera.tsx
import { CameraView, useCameraPermissions } from 'expo-camera';
import { useState, useRef } from 'react';
import { StyleSheet, Text, TouchableOpacity, View, Alert } from 'react-native';
import { router } from 'expo-router';

export default function CameraScreen() {
  const [permission, requestPermission] = useCameraPermissions();
  const [isCapturing, setIsCapturing] = useState(false);
  const cameraRef = useRef<CameraView>(null);

  // Still loading permission state
  if (\!permission) {
    return <View style={styles.container} />;
  }

  // Permission not yet granted — show explanation screen
  if (\!permission.granted) {
    return (
      <View style={styles.permissionContainer}>
        <Text style={styles.title}>Camera Access Required</Text>
        <Text style={styles.message}>
          This app uses your camera to photograph code for AI analysis.{'\n\n'}
          Photos are sent to Google Gemini for analysis and are not stored on our servers.
        </Text>
        <TouchableOpacity style={styles.grantButton} onPress={requestPermission}>
          <Text style={styles.grantButtonText}>Allow Camera Access</Text>
        </TouchableOpacity>
      </View>
    );
  }

  const handleCapture = async () => {
    if (\!cameraRef.current || isCapturing) return;

    setIsCapturing(true);
    try {
      const photo = await cameraRef.current.takePictureAsync({
        quality: 0.7,    // Critical: see pitfall #1 below
        base64: true,
        exif: false,     // Strip location data from EXIF for privacy
      });

      if (\!photo?.base64) {
        throw new Error('Failed to retrieve image data');
      }

      router.push({
        pathname: '/result',
        params: { imageBase64: photo.base64 },
      });
    } catch (error) {
      Alert.alert(
        'Capture Failed',
        'Something went wrong. Please try again.',
        [{ text: 'OK' }]
      );
      console.error('Camera capture error:', error);
    } finally {
      setIsCapturing(false);
    }
  };

  return (
    <View style={styles.container}>
      <CameraView ref={cameraRef} style={styles.camera} facing="back">
        <View style={styles.overlay}>
          <View style={styles.scanFrame} />
          <Text style={styles.hint}>
            Frame the code within the box. Good lighting improves accuracy.
          </Text>
        </View>
      </CameraView>
      <TouchableOpacity
        style={[styles.captureButton, isCapturing && styles.capturing]}
        onPress={handleCapture}
        disabled={isCapturing}
      >
        <Text style={styles.captureText}>
          {isCapturing ? 'Analyzing...' : 'Capture & Analyze'}
        </Text>
      </TouchableOpacity>
    </View>
  );
}
```

The `quality: 0.7` parameter is load-bearing. On a current iPhone or Pixel, a full-quality photo can be 8–12MB. Gemini's API has request size limits, and oversized images either time out or return errors. At 0.7 quality, a photo typically comes in at 2–3MB — and code recognition accuracy doesn't meaningfully degrade at this compression level.

## Step 3: The Gemini Multimodal API — Handling JSON Reliably

This is where most implementations fail. Without the right configuration, Gemini wraps JSON responses in Markdown code fences (```json ... ```), which breaks `JSON.parse()`. Here's the implementation that prevents this:

```typescript
// utils/gemini.ts
const GEMINI_API_URL =
  'https://generativelanguage.googleapis.com/v1beta/models/gemini-2.5-flash:generateContent';

export type ExplanationLevel = 'beginner' | 'intermediate' | 'advanced';

export interface CodeAnalysisResult {
  language: string;
  explanation: string;
  keyPoints: string[];
  potentialIssues?: string[];
}

const LEVEL_PROMPTS: Record<ExplanationLevel, string> = {
  beginner: `
You are a coding tutor for absolute beginners.
Analyze the code in the image and respond with this exact JSON (no code fences):
{
  "language": "programming language name",
  "explanation": "What this code does, explained like the reader has never coded before. Max 200 words.",
  "keyPoints": [
    "The most important thing this code does (1 sentence)",
    "A key concept a beginner should notice (1 sentence)",
    "What a beginner should learn from this (1 sentence)"
  ]
}
Define any technical terms in plain language.
  `.trim(),

  intermediate: `
You are a code reviewer for mid-level developers.
Analyze the code in the image and respond with this exact JSON (no code fences):
{
  "language": "programming language name",
  "explanation": "What this code does, its design intent, and any patterns it uses. Max 250 words.",
  "keyPoints": [
    "The design pattern or architectural decision being made",
    "How data flows through this code",
    "A practical insight for someone writing similar code"
  ],
  "potentialIssues": [
    "One improvement or watch-out if applicable"
  ]
}
  `.trim(),

  advanced: `
You are a senior engineer doing a technical code review.
Analyze the code in the image and respond with this exact JSON (no code fences):
{
  "language": "programming language name",
  "explanation": "Architecture, performance characteristics, scalability implications, and security considerations. Max 350 words.",
  "keyPoints": [
    "The architectural pattern and why it was likely chosen",
    "Performance or scalability characteristics",
    "Long-term maintainability implications"
  ],
  "potentialIssues": [
    "Security vulnerability or risk if present",
    "Performance bottleneck if present",
    "Refactoring opportunity with highest ROI"
  ]
}
  `.trim(),
};

export async function analyzeCodeImage(
  imageBase64: string,
  level: ExplanationLevel,
  apiKey: string
): Promise<CodeAnalysisResult> {
  const requestBody = {
    contents: [
      {
        parts: [
          {
            inline_data: {
              mime_type: 'image/jpeg',
              data: imageBase64,
            },
          },
          {
            text: LEVEL_PROMPTS[level],
          },
        ],
      },
    ],
    generationConfig: {
      temperature: 0.1,
      topP: 0.8,
      maxOutputTokens: 1024,
      responseMimeType: 'application/json', // This line prevents the JSON-in-markdown problem
    },
  };

  const response = await fetch(`${GEMINI_API_URL}?key=${apiKey}`, {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify(requestBody),
  });

  if (\!response.ok) {
    const errorText = await response.text();
    if (response.status === 400) {
      throw new Error('Image format issue — please retake the photo.');
    }
    if (response.status === 403) {
      throw new Error('API key authentication failed. Check your configuration.');
    }
    if (response.status === 429) {
      throw new Error('Rate limit reached. Please wait a moment and try again.');
    }
    throw new Error(`Analysis failed (status ${response.status})`);
  }

  const data = await response.json();
  const content = data.candidates?.[0]?.content?.parts?.[0]?.text;

  if (\!content) {
    throw new Error('No response from AI. Try a different image.');
  }

  try {
    return JSON.parse(content) as CodeAnalysisResult;
  } catch {
    // Fallback: even with responseMimeType set, Gemini occasionally drifts
    return {
      language: 'Detected',
      explanation: content,
      keyPoints: [],
    };
  }
}
```

The `responseMimeType: 'application/json'` field is the key. Without it, Gemini is free to format its response however it wants — and it often chooses Markdown code fences. With it, Gemini constrains its output to valid JSON. The fallback `JSON.parse` catch block handles the rare cases where the model still drifts.

## Step 4: The Result Screen with Level-Switching Cache

```tsx
// app/result.tsx
import { useLocalSearchParams, router } from 'expo-router';
import { useState, useEffect } from 'react';
import {
  ScrollView, Text, View, TouchableOpacity,
  ActivityIndicator, Alert, StyleSheet
} from 'react-native';
import { analyzeCodeImage, type ExplanationLevel, type CodeAnalysisResult } from '../utils/gemini';
import { saveToHistory } from '../utils/storage';

const LEVEL_LABELS: Record<ExplanationLevel, string> = {
  beginner: 'Beginner',
  intermediate: 'Intermediate',
  advanced: 'Advanced',
};

export default function ResultScreen() {
  const { imageBase64 } = useLocalSearchParams<{ imageBase64: string }>();
  const [level, setLevel] = useState<ExplanationLevel>('beginner');
  const [result, setResult] = useState<CodeAnalysisResult | null>(null);
  const [isLoading, setIsLoading] = useState(true);
  const [error, setError] = useState<string | null>(null);
  // Cache prevents redundant API calls when switching levels
  const [cache, setCache] = useState<Partial<Record<ExplanationLevel, CodeAnalysisResult>>>({});

  useEffect(() => {
    loadAnalysis(level);
  }, [level]);

  const loadAnalysis = async (targetLevel: ExplanationLevel) => {
    if (cache[targetLevel]) {
      setResult(cache[targetLevel]\!);
      return;
    }

    setIsLoading(true);
    setError(null);

    try {
      const analysisResult = await analyzeCodeImage(
        imageBase64,
        targetLevel,
        process.env.EXPO_PUBLIC_GEMINI_API_KEY\!
      );
      setResult(analysisResult);
      setCache((prev) => ({ ...prev, [targetLevel]: analysisResult }));
    } catch (err) {
      setError(err instanceof Error ? err.message : 'Analysis failed');
    } finally {
      setIsLoading(false);
    }
  };

  const handleSave = async () => {
    if (\!result) return;
    await saveToHistory({ result, level, timestamp: Date.now() });
    Alert.alert('Saved', 'Analysis added to your history.');
  };

  return (
    <ScrollView style={styles.container} contentContainerStyle={styles.content}>
      {/* Level switcher */}
      <View style={styles.levelTabs}>
        {(Object.keys(LEVEL_LABELS) as ExplanationLevel[]).map((l) => (
          <TouchableOpacity
            key={l}
            style={[styles.tab, level === l && styles.activeTab]}
            onPress={() => setLevel(l)}
            disabled={isLoading}
          >
            <Text style={[styles.tabText, level === l && styles.activeTabText]}>
              {LEVEL_LABELS[l]}
            </Text>
          </TouchableOpacity>
        ))}
      </View>

      {isLoading ? (
        <View style={styles.loadingContainer}>
          <ActivityIndicator size="large" color="#4CAF50" />
          <Text style={styles.loadingText}>Analyzing with Gemini...</Text>
          <Text style={styles.loadingSubtext}>Usually 3–5 seconds</Text>
        </View>
      ) : error ? (
        <View style={styles.errorContainer}>
          <Text style={styles.errorText}>{error}</Text>
          <TouchableOpacity style={styles.retryButton} onPress={() => loadAnalysis(level)}>
            <Text style={styles.retryText}>Try Again</Text>
          </TouchableOpacity>
          <TouchableOpacity style={styles.backButton} onPress={() => router.back()}>
            <Text style={styles.backText}>Retake Photo</Text>
          </TouchableOpacity>
        </View>
      ) : result ? (
        <View>
          <View style={styles.languageBadge}>
            <Text style={styles.languageText}>{result.language}</Text>
          </View>
          <Text style={styles.explanation}>{result.explanation}</Text>
          {result.keyPoints.length > 0 && (
            <View style={styles.section}>
              <Text style={styles.sectionTitle}>Key Points</Text>
              {result.keyPoints.map((point, i) => (
                <Text key={i} style={styles.keyPoint}>• {point}</Text>
              ))}
            </View>
          )}
          {result.potentialIssues && result.potentialIssues.length > 0 && (
            <View style={styles.issuesSection}>
              <Text style={styles.sectionTitle}>Watch-outs</Text>
              {result.potentialIssues.map((issue, i) => (
                <Text key={i} style={styles.issue}>⚠ {issue}</Text>
              ))}
            </View>
          )}
          <TouchableOpacity style={styles.saveButton} onPress={handleSave}>
            <Text style={styles.saveButtonText}>Save to History</Text>
          </TouchableOpacity>
        </View>
      ) : null}
    </ScrollView>
  );
}
```

The cache object is the design choice worth explaining. When a user switches from Beginner to Advanced, without the cache you'd make a second API call. With 50,000 users, that's a significant cost difference. More importantly, it makes the UI feel instant after the first load per level — which is the right tradeoff.

## Three Production Pitfalls and How to Avoid Them

These are the issues you won't find in tutorials, but that will surface in production.

### Pitfall 1: Dark-Theme Code Has Lower Recognition Accuracy

Light text on a dark background (a common IDE theme) results in noticeably lower recognition accuracy compared to dark text on white. This isn't a Gemini problem specifically — it's a characteristic of multimodal vision models generally.

The practical fix: add guidance text to the camera screen. "For best results, increase screen brightness or photograph printed code." Users self-optimize when given clear instructions. You don't need programmatic brightness detection — a simple text hint in the UI captures 80% of the benefit.

### Pitfall 2: Long Code Gets Truncated

With `maxOutputTokens: 1024`, very long code snippets result in truncated explanations. Increasing the token limit raises both cost and latency.

A better UX solution: implement a crop/selection UI. Let users mark which section of the photo they want analyzed. This is more useful than unlimited tokens, because it lets users ask "explain just this function" rather than "explain this entire 300-line file." React Native Gesture Handler's `LongPressGestureHandler` makes this buildable.

### Pitfall 3: API Key Exposure in Client Bundles

`process.env.EXPO_PUBLIC_GEMINI_API_KEY` is baked into your compiled app bundle at build time. Someone who knows how to inspect Expo bundles can extract it. This isn't an App Store rejection risk, but it's a real security concern at scale.

For a prototype or early launch, the risk is manageable — your API key is only useful to someone who is specifically looking for it. Once you hit 1,000+ users, migrate to a backend proxy: Cloudflare Workers or Supabase Edge Functions receive the image from your app, make the Gemini API call server-side, and return the result. Your actual API key never leaves your server.

## RevenueCat Subscription Integration

Here's the usage-tracking utility that enforces the free tier limit:

```typescript
// utils/usageManager.ts
import AsyncStorage from '@react-native-async-storage/async-storage';

const USAGE_KEY = 'daily_usage';
const FREE_DAILY_LIMIT = 3;

interface DailyUsage {
  count: number;
  date: string;
}

export async function checkAndTrackUsage(): Promise<{
  allowed: boolean;
  remaining: number;
}> {
  const today = new Date().toISOString().split('T')[0];
  const stored = await AsyncStorage.getItem(USAGE_KEY);
  let usage: DailyUsage = stored
    ? JSON.parse(stored)
    : { count: 0, date: today };

  // Reset on new day
  if (usage.date \!== today) {
    usage = { count: 0, date: today };
  }

  if (usage.count >= FREE_DAILY_LIMIT) {
    return { allowed: false, remaining: 0 };
  }

  usage.count += 1;
  await AsyncStorage.setItem(USAGE_KEY, JSON.stringify(usage));
  return { allowed: true, remaining: FREE_DAILY_LIMIT - usage.count };
}
```

And the RevenueCat integration that gates premium access:

```typescript
// utils/subscription.ts
import Purchases, { PURCHASES_ERROR_CODE } from 'react-native-purchases';

export async function isPremiumUser(): Promise<boolean> {
  try {
    const customerInfo = await Purchases.getCustomerInfo();
    return customerInfo.entitlements.active['premium'] \!== undefined;
  } catch {
    return false; // Default to free on any error
  }
}

export async function purchasePremium(): Promise<boolean> {
  try {
    const offerings = await Purchases.getOfferings();
    const monthlyPackage = offerings.current?.monthly;

    if (\!monthlyPackage) throw new Error('No offering available');

    const { customerInfo } = await Purchases.purchasePackage(monthlyPackage);
    return customerInfo.entitlements.active['premium'] \!== undefined;
  } catch (error: any) {
    // Don't surface an error when user cancels — just return false
    if (error.code === PURCHASES_ERROR_CODE.PURCHASE_CANCELLED_ERROR) {
      return false;
    }
    throw error;
  }
}
```

The `PURCHASE_CANCELLED_ERROR` handling is easy to overlook and causes a bad UX when missed. When users tap "Cancel" on the App Store purchase sheet, they should return silently to your app — not see an error dialog.

## App Store Approval: Three Requirements to Get Right

Camera apps with subscriptions get extra scrutiny in review.

**Requirement 1: Specific camera usage description**

In `app.json`:

```json
{
  "expo": {
    "ios": {
      "infoPlist": {
        "NSCameraUsageDescription": "Used to photograph code for AI analysis. Photos are sent to Google Gemini and are not stored on our servers."
      }
    }
  }
}
```

Vague descriptions like "Camera is used for app functionality" get flagged. Be specific about what data goes where.

**Requirement 2: Third-party data disclosure in privacy policy**

Any app that sends user data to a third-party API must disclose this in the privacy policy. Your App Store Connect privacy questions need to reflect that images are sent to Google for analysis.

**Requirement 3: Restore purchase button**

This is the most common rejection reason for subscription apps. Somewhere on your paywall screen, there must be a button that calls `Purchases.restorePurchases()`. Label it "Restore Purchase" or "Already subscribed?" — either works.
