## The AI-Generated UI Problem — Why Every Screen Looks Different

When you generate an app with Rork, the UI is built automatically — but you may have noticed that each screen has a subtly different design flavor. The home screen uses muted colors, but the settings screen suddenly goes vivid. Button corner radius varies between screens. Shadow intensity isn't consistent.

This happens because **AI optimizes each screen independently when no design system is present**. Human designers unconsciously maintain consistency. AI makes isolated decisions per screen unless given explicit rules.

Tomo Tsubota's "Melta UI" — a design system built specifically for Claude Code — pioneered a practical solution to this challenge. This article adapts those insights for Rork app development, showing you how to build a **design system that stabilizes AI-generated UI quality**.

## Three Problems Design Systems Solve

### 1. Color Drift — AI Gravitates Toward "Safe" Choices

Left to its own devices, AI converges on purple gradients, blue primary colors, and white backgrounds — the telltale "AI look." Defining your own color palette prevents this.

### 2. Typography Inconsistency — Font Sizes Everywhere

When heading sizes, body text, and line heights vary between screens, users unconsciously perceive "sloppy app." A typography scale ensures AI uses the same size system every time.

### 3. Component Inconsistency — Three Styles of "Confirm" Button

The same confirmation button shouldn't have different corner radius, shadow, and padding depending on which screen it's on. Component specifications ensure uniform rendering.

## Designing Design Tokens for Rork Apps

### What Are Design Tokens?

Design tokens are visual values — colors, typography, spacing, shadows — defined as named variables. Including them in Rork prompts controls AI output.

## Three Ways to Integrate Design Systems into Rork

### Method 1: Include Rules Directly in Prompts

The simplest approach — include design tokens in your app generation prompt.

```
Build a reading tracker app. Follow these design rules:

[Colors]
- Primary: #2563EB (blue)
- Background: #FAFAFA
- Text: #1F2937
- Border: #E5E7EB
- Subtle shadows only (rgba(0,0,0,0.05) max)

[Typography]
- Headings: 22px / semibold
- Body: 16px / regular / line-height 1.6
- Captions: 13px / regular

[Components]
- Buttons: 12px border-radius, 12px 24px padding
- Cards: 16px border-radius, sm shadow, no border
- Inputs: 6px border-radius, 1px border #E5E7EB
```

### Method 2: Rork Max Custom Theme Configuration

With Rork Max, define design tokens in a `theme.ts` file applied across the entire app.

```typescript
// theme.ts — Custom theme for Rork Max
export const appTheme = {
  colors: {
    primary: "#2563EB",
    primaryLight: "#DBEAFE",
    secondary: "#059669",
    background: "#FAFAFA",
    surface: "#FFFFFF",
    textPrimary: "#1F2937",
    textSecondary: "#6B7280",
    border: "#E5E7EB",
    error: "#DC2626",
    success: "#059669",
  },
  typography: {
    h1: { fontSize: 28, fontWeight: "700" as const, lineHeight: 36 },
    h2: { fontSize: 22, fontWeight: "600" as const, lineHeight: 30 },
    h3: { fontSize: 18, fontWeight: "600" as const, lineHeight: 25 },
    body: { fontSize: 16, fontWeight: "400" as const, lineHeight: 26 },
    caption: { fontSize: 13, fontWeight: "400" as const, lineHeight: 20 },
  },
  spacing: (factor: number) => factor * 8,
  borderRadius: { sm: 6, md: 12, lg: 16, full: 9999 },
} as const;

// Usage:
// <View style={{ padding: appTheme.spacing(2), borderRadius: appTheme.borderRadius.md }}>
//   <Text style={appTheme.typography.h2}>Book List</Text>
// </View>
```

### Method 3: Post-Generation Design Review Prompts

After generating an app, run a compliance check:

```
Review the generated app UI against these criteria:
1. Color palette consistency — only #2563EB, #FAFAFA, #1F2937 and defined colors used
2. Font sizes within the defined scale (13, 16, 18, 22, 28px)
3. All button corner radii are exactly 12px
4. Shadow opacity stays between 0.05 and 0.1

Fix any violations.
```

## The "Why" Behind Design Token Choices

### Choosing Colors Intentionally

If you pick tokens "randomly," you'll end up with the same generic output AI would generate anyway. Be intentional:

- **Primary color**: Match your app's domain — finance uses blue, health uses green, entertainment uses orange
- **Background**: Use off-white (#FAFAFA, #F9FAFB) instead of pure white (#FFFFFF) for eye comfort
- **Shadow intensity**: AI tends to overdo shadows. Cap at `opacity: 0.05–0.07`
- **Accent color**: Pick one complementary color, reserved exclusively for CTAs

### Building a Typography Scale

For mobile apps, a **1.25x scale** (Major Third) works well:

```
13px (caption) × 1.25 = 16px (body)
16px (body)    × 1.125 = 18px (h3)
18px (h3)      × 1.22  = 22px (h2)
22px (h2)      × 1.27  = 28px (h1)
```

## Frequently Asked Questions

### Can I apply a design system with Rork's free plan?

Yes. The prompt-based approach (Method 1) works on any plan. The `theme.ts` approach (Method 2) requires Rork Max, but prompt-based rules deliver significant quality improvements on their own.

### Can too many design tokens hurt AI output quality?

Yes. Overloading tokens can confuse the AI and actually decrease quality. Insights from Tsubota's Melta UI suggest keeping core design elements (colors, typography, spacing, border-radius, shadows) within roughly 400 tokens for optimal results.

### Can I apply a design system to an already-built app?

Yes. Use the design review prompt (Method 3) to audit existing UI and fix violations. For extensive changes, it may be faster to regenerate the app from scratch with the design system defined upfront.

### How do I maintain consistency between iOS and Android?

Rork apps are built on React Native / Expo, so the same code runs on both platforms. For platform-specific adjustments (iOS Safe Areas, Android Material Design), add "follow each platform's design guidelines" to your prompt.

## Wrapping Up — Design Systems Are Your Blueprint for AI

In the age of AI-powered development, the role of design systems has shifted from "shared language for human designers" to **specification documents for AI**. Define design tokens, include them in your Rork prompts, and the quality of AI-generated UI stabilizes dramatically.

Color drift, typography inconsistency, component mismatches — these hallmarks of "obviously AI-made" UI are solvable problems. Start with just two things: a color palette and a typography scale. You'll be surprised how much that alone transforms the output.
