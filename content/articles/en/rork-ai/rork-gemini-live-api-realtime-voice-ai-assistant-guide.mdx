When it comes to voice AI assistants, OpenAI's Realtime API has been the go-to choice. But when you start calculating the cost of daily conversations in a production app, the numbers add up quickly — especially for indie developers running tight margins.

That's where **Gemini Live API** becomes worth a serious look. Google's realtime multimodal API supports bidirectional audio and text streaming, handles interruptions gracefully, and offers competitive pricing for conversation-heavy use cases. It's not a perfect fit for every app, but for many use cases it deserves to be in your evaluation.

This guide walks through everything you need to integrate Gemini Live API into a Rork app — from establishing the WebSocket connection, to streaming microphone audio, to playing back AI responses and handling interruptions. Every section includes working code and notes on the specific gotchas that trip most people up.

## How Gemini Live API Differs from OpenAI Realtime API

Before diving into implementation, it's worth understanding how Gemini Live API works so the design decisions make sense.

Gemini Live API is a WebSocket-based API available through Google AI Studio or Vertex AI. The client streams audio (or text) to Google's servers, and the server responds with audio or text in real time. The overall structure is similar to OpenAI's Realtime API, but there are meaningful differences.

**Mixed input modalities**: Gemini Live API lets you send both audio and text within the same session. This makes it easy to build experiences where the user can voice a question and then type a follow-up — or vice versa — without switching sessions.

**Interruption handling**: When the user starts speaking while the AI is responding, Gemini automatically handles the interruption when new audio input arrives. With OpenAI's Realtime API, you need to explicitly send a `response.cancel` event. Gemini's approach tends to feel more natural in conversation flows.

**Cost structure**: At the time of writing, `gemini-2.0-flash-live-001` offers competitive pricing for high-volume conversation apps compared to OpenAI's Realtime API. That said, pricing changes — always benchmark against your specific usage pattern before committing.

## Architecture: Direct vs. Proxied Connection

You have two options for how your Rork app connects to Gemini Live API.

**Direct connection** means your app opens a WebSocket directly to Gemini's endpoint. This is the fastest to implement and great for development, but it requires storing your API key in the client — a security risk you shouldn't take in production.

**Backend proxy** means a server (Cloudflare Workers, Supabase Edge Functions, etc.) holds your API key and relays messages between your app and Gemini. Your app connects to your own WebSocket endpoint, not Google's directly.

This guide shows the direct approach first to get you up and running, then covers how to build the Cloudflare Workers proxy for production.

## Setting Up the WebSocket Connection

The core of the implementation is establishing and managing the WebSocket connection. The connection URL format and the setup handshake are where most initial confusion comes from.

```typescript
// hooks/useGeminiLive.ts
import { useState, useEffect, useRef, useCallback } from 'react';

const GEMINI_API_KEY = 'YOUR_GEMINI_API_KEY'; // Move to backend in production
const MODEL = 'gemini-2.0-flash-live-001';
const GEMINI_WS_URL = `wss://generativelanguage.googleapis.com/ws/google.ai.generativelanguage.v1beta.GenerativeService.BidiGenerateContent?key=${GEMINI_API_KEY}`;

type SessionState = 'idle' | 'connecting' | 'connected' | 'error';

interface Message {
  role: 'user' | 'model';
  text: string;
}

export const useGeminiLive = () => {
  const ws = useRef<WebSocket | null>(null);
  const [sessionState, setSessionState] = useState<SessionState>('idle');
  const [messages, setMessages] = useState<Message[]>([]);
  const [isModelSpeaking, setIsModelSpeaking] = useState(false);
  const setupComplete = useRef(false); // Guard: don't send audio before setup completes

  const connect = useCallback(async () => {
    if (ws.current?.readyState === WebSocket.OPEN) return;

    setSessionState('connecting');
    setupComplete.current = false;
    ws.current = new WebSocket(GEMINI_WS_URL);

    ws.current.onopen = () => {
      // Send the setup message immediately after connection opens
      const setupMessage = {
        setup: {
          model: `models/${MODEL}`,
          generation_config: {
            response_modalities: ['AUDIO'],
            speech_config: {
              voice_config: {
                prebuilt_voice_config: {
                  // Available voices: Puck, Charon, Kore, Fenrir, Aoede
                  voice_name: 'Aoede',
                },
              },
            },
          },
          system_instruction: {
            parts: [
              {
                text: 'You are a helpful, friendly assistant. Keep your responses concise and conversational.',
              },
            ],
          },
        },
      };

      ws.current?.send(JSON.stringify(setupMessage));
    };

    ws.current.onmessage = (event) => {
      handleServerMessage(event.data);
    };

    ws.current.onerror = (error) => {
      console.error('WebSocket error:', error);
      setSessionState('error');
    };

    ws.current.onclose = (event) => {
      console.log('WebSocket closed:', event.code, event.reason);
      setSessionState('idle');
      setupComplete.current = false;
    };
  }, []);

  const disconnect = useCallback(() => {
    ws.current?.close();
    ws.current = null;
    setSessionState('idle');
  }, []);

  const handleServerMessage = (data: string) => {
    try {
      const message = JSON.parse(data);

      // Wait for setupComplete before allowing audio transmission
      if (message.setupComplete) {
        setupComplete.current = true;
        setSessionState('connected');
        return;
      }

      if (message.serverContent) {
        const { modelTurn, turnComplete } = message.serverContent;

        if (modelTurn?.parts) {
          for (const part of modelTurn.parts) {
            if (part.text) {
              setMessages((prev) => [
                ...prev,
                { role: 'model', text: part.text },
              ]);
            }

            if (part.inlineData?.mimeType?.startsWith('audio/')) {
              playAudioResponse(part.inlineData.data);
              setIsModelSpeaking(true);
            }
          }
        }

        if (turnComplete) {
          setIsModelSpeaking(false);
        }
      }
    } catch (error) {
      console.error('Error parsing server message:', error);
    }
  };

  const sendAudioChunk = useCallback((base64Audio: string) => {
    // Critical: don't send audio until setup is complete
    if (\!setupComplete.current || ws.current?.readyState \!== WebSocket.OPEN) return;

    const audioMessage = {
      realtimeInput: {
        mediaChunks: [
          {
            mimeType: 'audio/pcm;rate=16000',
            data: base64Audio,
          },
        ],
      },
    };

    ws.current.send(JSON.stringify(audioMessage));
  }, []);

  const sendText = useCallback((text: string) => {
    if (\!setupComplete.current || ws.current?.readyState \!== WebSocket.OPEN) return;

    const textMessage = {
      clientContent: {
        turns: [{ role: 'user', parts: [{ text }] }],
        turnComplete: true,
      },
    };

    ws.current.send(JSON.stringify(textMessage));
    setMessages((prev) => [...prev, { role: 'user', text }]);
  }, []);

  return {
    connect,
    disconnect,
    sessionState,
    messages,
    isModelSpeaking,
    sendAudioChunk,
    sendText,
  };
};
```

## Audio Capture: The Format Requirements

Capturing microphone audio in the exact format Gemini expects is the most common source of silent failures. Gemini Live API requires **PCM 16-bit, 16kHz, mono** audio. Anything else gets silently ignored — no error, no response, just nothing happening.

```typescript
// hooks/useAudioCapture.ts
import { Audio } from 'expo-av';
import { useRef, useCallback } from 'react';

// These exact settings match what Gemini Live API expects
const RECORDING_OPTIONS: Audio.RecordingOptions = {
  android: {
    extension: '.wav',
    outputFormat: Audio.AndroidOutputFormat.PCM_16BIT,
    audioEncoder: Audio.AndroidAudioEncoder.DEFAULT,
    sampleRate: 16000,    // Must be 16kHz
    numberOfChannels: 1,  // Must be mono
    bitRate: 256000,
  },
  ios: {
    extension: '.wav',
    outputFormat: Audio.IOSOutputFormat.LINEARPCM,
    audioQuality: Audio.IOSAudioQuality.HIGH,
    sampleRate: 16000,        // Must be 16kHz
    numberOfChannels: 1,      // Must be mono
    bitRate: 128000,
    linearPCMBitDepth: 16,    // Must be 16-bit
    linearPCMIsBigEndian: false,
    linearPCMIsFloat: false,
  },
  web: {
    mimeType: 'audio/webm',
    blobEvent: { timeslice: 250 },
  },
};

export const useAudioCapture = (
  onAudioChunk: (base64Audio: string) => void
) => {
  const recording = useRef<Audio.Recording | null>(null);
  const isCapturing = useRef(false);
  const intervalRef = useRef<NodeJS.Timer | null>(null);

  const startCapture = useCallback(async () => {
    try {
      const permission = await Audio.requestPermissionsAsync();
      if (\!permission.granted) {
        throw new Error('Microphone permission is required');
      }

      // Set audio mode for recording
      await Audio.setAudioModeAsync({
        allowsRecordingIOS: true,
        playsInSilentModeIOS: true,
      });

      const { recording: newRecording } = await Audio.Recording.createAsync(
        RECORDING_OPTIONS
      );

      recording.current = newRecording;
      isCapturing.current = true;

      // Send audio chunks every 500ms
      intervalRef.current = setInterval(async () => {
        if (\!isCapturing.current || \!recording.current) {
          if (intervalRef.current) clearInterval(intervalRef.current);
          return;
        }

        try {
          const uri = recording.current.getURI();
          if (uri) {
            const base64 = await fileUriToBase64(uri);
            onAudioChunk(base64);
          }
        } catch (e) {
          console.error('Error sending audio chunk:', e);
        }
      }, 500);

    } catch (error) {
      console.error('Failed to start audio capture:', error);
      throw error;
    }
  }, [onAudioChunk]);

  const stopCapture = useCallback(async () => {
    isCapturing.current = false;

    if (intervalRef.current) {
      clearInterval(intervalRef.current);
      intervalRef.current = null;
    }

    if (recording.current) {
      try {
        await recording.current.stopAndUnloadAsync();
      } catch (e) {
        console.error('Error stopping recording:', e);
      }
      recording.current = null;
    }

    // Switch audio mode back for playback
    await Audio.setAudioModeAsync({
      allowsRecordingIOS: false,
      playsInSilentModeIOS: true,
    });
  }, []);

  return { startCapture, stopCapture };
};

const fileUriToBase64 = async (uri: string): Promise<string> => {
  const response = await fetch(uri);
  const blob = await response.blob();
  return new Promise((resolve, reject) => {
    const reader = new FileReader();
    reader.onloadend = () => {
      const base64 = (reader.result as string).split(',')[1];
      resolve(base64);
    };
    reader.onerror = reject;
    reader.readAsDataURL(blob);
  });
};
```

## Playing Back AI Responses

When Gemini sends an audio response, you'll receive Base64-encoded PCM audio data. Here's a reliable approach to playing it back:

```typescript
// utils/audioPlayer.ts
import { Audio } from 'expo-av';

let currentSound: Audio.Sound | null = null;

export const playAudioResponse = async (base64Audio: string): Promise<void> => {
  try {
    // Stop and unload any currently playing audio
    if (currentSound) {
      await currentSound.stopAsync();
      await currentSound.unloadAsync();
      currentSound = null;
    }

    await Audio.setAudioModeAsync({
      allowsRecordingIOS: false,
      playsInSilentModeIOS: true,
      staysActiveInBackground: false,
    });

    const { sound } = await Audio.Sound.createAsync(
      { uri: `data:audio/pcm;base64,${base64Audio}` },
      { shouldPlay: true }
    );

    currentSound = sound;

    sound.setOnPlaybackStatusUpdate((status) => {
      if (status.isLoaded && status.didJustFinish) {
        sound.unloadAsync().catch(console.error);
        currentSound = null;
      }
    });
  } catch (error) {
    // Don't rethrow — a failed playback shouldn't crash the conversation
    console.error('Audio playback error:', error);
  }
};

export const stopAudioPlayback = async (): Promise<void> => {
  if (currentSound) {
    try {
      await currentSound.stopAsync();
      await currentSound.unloadAsync();
    } catch (e) {
      console.error('Error stopping playback:', e);
    }
    currentSound = null;
  }
};
```

## Interruption Handling

The hallmark of a natural voice conversation is the ability to interrupt. When your user starts speaking while the AI is mid-response, the AI should stop. Here's how to wire that up:

import { stopAudioPlayback } from '../utils/audioPlayer';

## Common Pitfalls and How to Avoid Them

After helping several developers get this working, here are the issues that come up most often.

**Silent failures from missing `setupComplete` guard**: The most common mistake is sending audio before the setup handshake completes. After the WebSocket opens, you must send the `setup` message and wait for `setupComplete` in the server response. Any audio sent before this is silently dropped. Always use a `setupComplete` flag (as shown above) to guard `sendAudioChunk`.

**Audio format mismatch**: If your audio isn't in the format Gemini expects (PCM 16-bit, 16kHz, mono), you'll get no error — just silence. Gemini doesn't send back an error code for invalid audio format. If you're seeing the WebSocket connection work but no responses coming back, format mismatch is the first thing to check.

**iOS audio session conflict**: iOS requires switching audio session modes between recording and playback. If you try to play audio while the session is in recording mode, you'll get either silence or a crash. The pattern to follow: switch to recording mode (`allowsRecordingIOS: true`) before `startCapture`, and back to playback mode (`allowsRecordingIOS: false`) before playing responses.

**Session timeouts during long pauses**: Gemini Live API will close inactive sessions. For apps where users might set their phone down for a few minutes, implement reconnection logic. A simple exponential backoff reconnect on close event handles this gracefully.

```typescript
// Reconnection with exponential backoff
const reconnect = useCallback(async () => {
  const maxRetries = 3;
  for (let attempt = 1; attempt <= maxRetries; attempt++) {
    try {
      await new Promise(resolve => setTimeout(resolve, 1000 * attempt));
      await connect();
      console.log(`Reconnected on attempt ${attempt}`);
      return;
    } catch (e) {
      if (attempt === maxRetries) {
        console.error('All reconnection attempts failed');
        // Show user-facing error here
      }
    }
  }
}, [connect]);
```

## Moving to Production: The Cloudflare Workers Proxy

When you're ready to deploy, move your API key to a Cloudflare Worker that proxies the WebSocket connection. Here's a minimal implementation:

```javascript
// worker.js (Cloudflare Worker)
// Set GEMINI_API_KEY as a secret via: wrangler secret put GEMINI_API_KEY

export default {
  async fetch(request, env) {
    if (request.headers.get('Upgrade') \!== 'websocket') {
      return new Response('Expected WebSocket', { status: 426 });
    }

    const [client, serverSide] = Object.values(new WebSocketPair());
    serverSide.accept();

    const geminiUrl = `wss://generativelanguage.googleapis.com/ws/google.ai.generativelanguage.v1beta.GenerativeService.BidiGenerateContent?key=${env.GEMINI_API_KEY}`;

    const gemini = new WebSocket(geminiUrl);

    // Relay: client → Gemini
    serverSide.addEventListener('message', (e) => {
      if (gemini.readyState === WebSocket.OPEN) {
        gemini.send(e.data);
      }
    });

    // Relay: Gemini → client
    gemini.addEventListener('message', (e) => {
      serverSide.send(e.data);
    });

    serverSide.addEventListener('close', () => gemini.close());
    gemini.addEventListener('close', (e) => serverSide.close(e.code, e.reason));
    gemini.addEventListener('error', () => serverSide.close(1011, 'Upstream error'));

    return new Response(null, { status: 101, webSocket: client });
  },
};
```

In your app, just replace `GEMINI_WS_URL` with your Worker's URL. Everything else stays the same.

## Advanced: Implementing Voice Activity Detection

The basic implementation above requires the user to manually tap a button to start and stop recording. For a more natural experience, you can implement voice activity detection (VAD) so the app automatically detects when the user starts and stops speaking.

The simplest approach is energy-based detection: measure the audio signal level and consider speech active when it exceeds a threshold. Gemini also has server-side VAD built in — when you enable `automatic_activity_detection` in the setup config, the server handles turn detection for you.

```typescript
// Enabling server-side VAD in the setup message
const setupMessage = {
  setup: {
    model: `models/${MODEL}`,
    generation_config: {
      response_modalities: ['AUDIO'],
      // ... voice config ...
    },
    // Enable automatic activity detection
    realtime_input_config: {
      automatic_activity_detection: {
        disabled: false, // Enable server-side VAD
      },
    },
    system_instruction: {
      parts: [{ text: 'You are a helpful, friendly assistant.' }],
    },
  },
};
```

When server-side VAD is enabled, you can stream audio continuously and let Gemini decide when the user's turn ends. This produces a noticeably more natural conversation rhythm than the push-to-talk approach, but requires more bandwidth since you're streaming even during silence.

For most production apps, a hybrid approach works well: use client-side energy detection to gate when audio is actually sent to the server, reducing unnecessary data transmission, while letting Gemini's VAD determine turn boundaries.

## Handling Network Interruptions Gracefully

Real-world networks are unreliable. Your voice assistant needs to handle connection drops without losing the conversation context or confusing the user.

```typescript
// Enhanced connection management with state persistence
export const useGeminiLiveWithResilience = () => {
  const [conversationHistory, setConversationHistory] = useState<Message[]>([]);
  const lastActiveSessionRef = useRef<string | null>(null);

  const connectWithHistory = useCallback(async () => {
    await connect();

    // After reconnecting, send conversation context in the system instruction
    // Note: Gemini Live doesn't persist session state across connections,
    // so you need to re-inject context if continuity matters
    if (conversationHistory.length > 0) {
      const contextSummary = conversationHistory
        .slice(-5) // Last 5 exchanges
        .map(m => `${m.role === 'user' ? 'User' : 'Assistant'}: ${m.text}`)
        .join('\n');

      // Send context via text message at the start of new session
      setTimeout(() => {
        sendText(`[Context from our previous conversation: ${contextSummary}]`);
      }, 500); // Small delay to ensure setup is complete
    }
  }, [conversationHistory, connect, sendText]);

  return {
    connectWithHistory,
    conversationHistory,
    // ... rest of the hook
  };
};
```

The key thing to understand about Gemini Live API is that each WebSocket connection represents a fresh session. Unlike some stateful APIs, there's no session ID you can pass to resume where you left off. If conversation continuity matters to your use case, you need to manage that state client-side and re-inject it when reconnecting.

## Choosing Between Audio and Text Response Modes

Gemini Live API can respond in either audio, text, or both. The choice affects latency, bandwidth, and user experience.

**Audio-only responses** feel most natural for voice conversations. The response latency is lower because the model generates audio directly without intermediate text. The downside is you can't easily display a transcript of what the AI said.

**Text-only responses** let you display the conversation as text (like a chat interface) and optionally synthesize speech client-side using a different TTS engine. This gives you more control over the voice quality and language, but adds complexity.

**Both audio and text** is useful for accessibility: the user hears the response and also sees a caption. It increases the data sent back from the server, which is worth considering on mobile connections.

```typescript
// Requesting both audio and text responses
generation_config: {
  response_modalities: ['AUDIO', 'TEXT'], // Request both
  speech_config: {
    voice_config: {
      prebuilt_voice_config: { voice_name: 'Aoede' },
    },
  },
},
```

For most voice assistant apps, I'd recommend starting with audio-only and using the `serverContent.modelTurn.parts` array to separately capture any text that comes through. Then decide whether to add transcription based on actual user feedback.

## Testing on Real Devices

The Rork simulator can test the UI, but voice capture requires a real device. There are a few things that don't behave the same in the simulator:

The WebSocket connection works fine in the simulator, so you can test the message-handling logic without hardware. What you can't test is the audio format, the microphone permission flow, and the actual audio playback. These need physical devices.

For iOS testing, build to a device via TestFlight (or direct install with your developer certificate). The `allowsRecordingIOS` mode switch is particularly important to test on real hardware — audio session conflicts that aren't visible in the simulator can crash real devices.

For Android, the PCM recording format is more reliably supported on modern devices, but older Android versions (below API 29) can have inconsistent behavior with `PCM_16BIT` encoding. Test on at least two Android versions if your app needs to support a wide range.

One practical tip: during development, add a debug panel to your screen that shows the current `sessionState`, the last few WebSocket messages (truncated), and audio level indicators. Being able to see exactly what's happening at each step saves hours of debugging.

## A Note on System Prompt Design

The system instruction you include in the setup message has an outsized impact on conversation quality. A few principles that work well for voice-first assistants:

Keep responses short by default. Voice responses that run longer than 30-40 seconds feel like being lectured. Instruct the model to aim for 1-3 sentence responses unless the user explicitly asks for more detail.

Specify a conversational register. "Be friendly and casual" produces very different results from "Be professional and concise." Experiment with the phrasing and listen to whether it sounds natural when played back as audio.

Handle out-of-scope questions explicitly. If your assistant has a defined purpose, tell it what to do when users go off-topic. "If asked about topics unrelated to X, politely explain that you're focused on X and redirect the conversation" is more useful than vague instructions.

These small prompt adjustments will do more for perceived quality than any technical optimization.
