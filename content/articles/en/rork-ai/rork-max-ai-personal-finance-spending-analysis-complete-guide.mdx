Most budgeting apps on the App Store share the same fundamental problem: they record your spending, but they don't tell you what to do about it. You end up with a graph of your bad habits and no clear path forward.

Building a finance app with Rork Max and Gemini changes that. Instead of passive data entry, you get an app that reads a receipt photo, automatically assigns a category, and at the end of the month tells you something like: "Your dining spend is up 40% from last month. Want to set a lower target for next month?" That's the kind of app this guide will help you build.

I'll cover everything from SQLite schema design to secure API key storage to the specific App Store review landmines you'll encounter with financial apps. All code examples are production-ready and include error handling — because an app that crashes when the AI returns an unexpected format is worse than no AI at all.

## Architecture Overview: What We're Building

Before writing any code, here's the complete picture — and the reasoning behind each choice.

**Core features:**

- Camera-based receipt scanning → Gemini Vision extracts amount, store name, and category
- Natural language entry ("just spent ¥430 at the convenience store")
- Monthly and weekly spending charts by category
- Budget targets with progress tracking
- End-of-month AI advice generated from actual spending history

**Tech stack:**

- **Local database**: expo-sqlite (offline-first, privacy-preserving)
- **AI**: Gemini 1.5 Flash (receipt OCR and NLP parsing), Gemini 1.5 Pro (advice generation)
- **Secure storage**: expo-secure-store (API key management)
- **Charts**: Victory Native (see the [interactive chart implementation guide](/en/articles/rork-dev/rork-victory-native-charts-dashboard) for details)

A word on the database choice: I deliberately avoided Firebase and Supabase here. Financial data is among the most sensitive information users store on their phones, and a meaningful segment of users specifically prefers apps that don't upload their data anywhere. An offline-first design with optional cloud sync as a paid feature actually performs better in App Store reviews and tends to have lower early churn. Your setup may differ, but this is worth thinking through before you reach for a cloud backend by default.

## Data Models and Local Database Design

Database schemas are expensive to change later, so it's worth getting this right upfront. Here's the core data model:

```typescript
// types/finance.ts
export type ExpenseCategory =
  | 'food'          // Groceries, convenience stores
  | 'dining'        // Restaurants, cafes, takeout
  | 'transport'     // Train, bus, taxi, gas
  | 'shopping'      // Clothing, electronics, non-food household items
  | 'utilities'     // Electricity, gas, water, phone bills
  | 'healthcare'    // Pharmacy, hospital, gym
  | 'entertainment' // Movies, games, books, hobbies
  | 'education'     // Learning services, educational books
  | 'other';        // Catch-all

export interface Expense {
  id: string;
  amount: number;            // Amount in local currency (integer)
  category: ExpenseCategory;
  description: string;       // Store name or memo
  date: string;              // ISO 8601
  receiptImageUri?: string;  // Local path to receipt image
  aiCategorized: boolean;    // Was this categorized by AI?
  createdAt: string;
}

export interface Budget {
  id: string;
  category: ExpenseCategory;
  monthlyLimit: number;  // Monthly spending cap
  month: string;         // "2026-04" format
}
```

Now the SQLite initialization and CRUD layer, with full error handling:

```typescript
// lib/database.ts
import * as SQLite from 'expo-sqlite';

const db = SQLite.openDatabaseSync('finance.db');

export async function initDatabase(): Promise<void> {
  try {
    await db.execAsync(`
      PRAGMA journal_mode = WAL;

      CREATE TABLE IF NOT EXISTS expenses (
        id TEXT PRIMARY KEY,
        amount REAL NOT NULL,
        category TEXT NOT NULL,
        description TEXT NOT NULL,
        date TEXT NOT NULL,
        receipt_image_uri TEXT,
        ai_categorized INTEGER DEFAULT 0,
        created_at TEXT NOT NULL
      );

      CREATE TABLE IF NOT EXISTS budgets (
        id TEXT PRIMARY KEY,
        category TEXT NOT NULL,
        monthly_limit REAL NOT NULL,
        month TEXT NOT NULL,
        UNIQUE(category, month)
      );

      CREATE INDEX IF NOT EXISTS idx_expenses_date ON expenses(date);
      CREATE INDEX IF NOT EXISTS idx_expenses_category ON expenses(category);
    `);
  } catch (error) {
    console.error('Database initialization failed:', error);
    throw new Error('Failed to initialize database');
  }
}

export async function addExpense(expense: Expense): Promise<void> {
  try {
    await db.runAsync(
      `INSERT INTO expenses VALUES (?, ?, ?, ?, ?, ?, ?, ?)`,
      [
        expense.id,
        expense.amount,
        expense.category,
        expense.description,
        expense.date,
        expense.receiptImageUri ?? null,
        expense.aiCategorized ? 1 : 0,
        expense.createdAt,
      ]
    );
  } catch (error) {
    if (error instanceof Error && error.message.includes('UNIQUE constraint')) {
      throw new Error('This expense already exists');
    }
    throw new Error('Failed to save expense');
  }
}

export async function getExpensesByMonth(month: string): Promise<Expense[]> {
  // month is in "2026-04" format
  try {
    const rows = await db.getAllAsync<any>(
      `SELECT * FROM expenses WHERE date LIKE ? ORDER BY date DESC`,
      [`${month}%`]
    );
    return rows.map(row => ({
      id: row.id,
      amount: row.amount,
      category: row.category as ExpenseCategory,
      description: row.description,
      date: row.date,
      receiptImageUri: row.receipt_image_uri ?? undefined,
      aiCategorized: row.ai_categorized === 1,
      createdAt: row.created_at,
    }));
  } catch (error) {
    console.error('Failed to fetch expenses:', error);
    return []; // Return empty array on error — don't crash the UI
  }
}

export async function getCategoryTotals(
  month: string
): Promise<Partial<Record<ExpenseCategory, number>>> {
  try {
    const rows = await db.getAllAsync<{ category: string; total: number }>(
      `SELECT category, SUM(amount) as total
       FROM expenses
       WHERE date LIKE ?
       GROUP BY category`,
      [`${month}%`]
    );
    const result: Partial<Record<ExpenseCategory, number>> = {};
    rows.forEach(row => {
      result[row.category as ExpenseCategory] = row.total;
    });
    return result;
  } catch (error) {
    console.error('Failed to get category totals:', error);
    return {};
  }
}
```

The `PRAGMA journal_mode = WAL` line at the top is deliberate. WAL (Write-Ahead Logging) allows concurrent reads and writes without blocking — important for a finance app where you're likely displaying aggregated charts while simultaneously recording new expenses. The indexes are also worth creating from day one: adding them after six months of data accumulation incurs a migration penalty.

## Automatic Expense Classification: Gemini Vision API for Receipt Scanning

Here's where the "AI finance app" becomes real. The key insight from building this: if you ask Gemini to "analyze this receipt and tell me the amount and category," you get wildly inconsistent output that's hard to parse reliably. The fix is to demand structured JSON output and treat every response with defensive parsing.

```typescript
// lib/gemini.ts
import { GoogleGenerativeAI } from '@google/generative-ai';
import * as SecureStore from 'expo-secure-store';
import * as FileSystem from 'expo-file-system';

const VALID_CATEGORIES: ExpenseCategory[] = [
  'food', 'dining', 'transport', 'shopping',
  'utilities', 'healthcare', 'entertainment', 'education', 'other',
];

export interface ReceiptAnalysisResult {
  amount: number;
  storeName: string;
  category: ExpenseCategory;
  confidence: number; // 0–1
  rawText?: string;
}

export async function analyzeReceiptImage(
  imageUri: string
): Promise<ReceiptAnalysisResult> {
  // Never hardcode API keys — fetch from SecureStore
  const apiKey = await SecureStore.getItemAsync('gemini_api_key');
  if (\!apiKey) {
    throw new Error(
      'Gemini API key not configured. Please add your API key in Settings.'
    );
  }

  const genAI = new GoogleGenerativeAI(apiKey);
  const model = genAI.getGenerativeModel({ model: 'gemini-1.5-flash' });

  // Convert image to Base64
  let base64Image: string;
  try {
    base64Image = await FileSystem.readAsStringAsync(imageUri, {
      encoding: FileSystem.EncodingType.Base64,
    });
  } catch {
    throw new Error('Failed to read image file');
  }

  const prompt = `
Analyze this receipt image and respond ONLY with a JSON object in this exact format.
Do not include any text outside the JSON.

{
  "amount": <numeric amount>,
  "storeName": "<store name>",
  "category": "<one of: food/dining/transport/shopping/utilities/healthcare/entertainment/education/other>",
  "confidence": <float 0.0–1.0>,
  "rawText": "<key text extracted from the receipt>"
}

Category rules:
- food: Grocery stores, convenience stores (food and household items)
- dining: Restaurants, cafes, fast food, takeout
- transport: Train, bus, taxi, ride-sharing, gas station
- shopping: Clothing, electronics, non-food household items
- utilities: Electricity, gas, water, phone/internet bills
- healthcare: Pharmacy, hospital, clinic, gym
- entertainment: Cinema, games, books, hobbies
- education: Online courses, educational books
- other: Anything that doesn't fit above
`;

  let responseText: string;
  try {
    const result = await model.generateContent([
      prompt,
      {
        inlineData: {
          mimeType: 'image/jpeg',
          data: base64Image,
        },
      },
    ]);
    responseText = result.response.text();
  } catch (error) {
    throw new Error(
      'Gemini API call failed. Please check your network connection.'
    );
  }

  // Defensive JSON parsing
  try {
    // Handle cases where Gemini wraps JSON in a code block
    const jsonMatch = responseText.match(/\{[\s\S]*\}/);
    if (\!jsonMatch) throw new Error('No JSON found in response');

    const parsed = JSON.parse(jsonMatch[0]);

    // Validate required fields
    if (typeof parsed.amount \!== 'number' || parsed.amount <= 0) {
      throw new Error('Invalid amount');
    }
    if (\!VALID_CATEGORIES.includes(parsed.category)) {
      parsed.category = 'other'; // Graceful fallback for unknown categories
    }
    const confidence = Math.min(1, Math.max(0, parsed.confidence ?? 0.5));

    return {
      amount: Math.round(parsed.amount),
      storeName: parsed.storeName || 'Unknown',
      category: parsed.category,
      confidence,
      rawText: parsed.rawText,
    };
  } catch {
    // Parsing failure: return a safe fallback that prompts manual entry
    return {
      amount: 0,
      storeName: 'Unknown',
      category: 'other',
      confidence: 0,
    };
  }
}
```

When `confidence` is below 0.5, surface a UI prompt asking the user to verify the amount and category before saving. This does two things: it prevents silent errors, and it generates correction data that you can use to improve your category mapping logic over time.

## Natural Language Expense Entry

For expenses without receipts — transit cards, contactless payments, cash — natural language input is a better UX than a form. This function parses free-form text like "grabbed coffee for $5 this morning" into structured data:

```typescript
// lib/nlpExpense.ts
import { GoogleGenerativeAI } from '@google/generative-ai';
import * as SecureStore from 'expo-secure-store';

export async function parseNaturalLanguageExpense(
  input: string,
  currentDate: string
): Promise<Partial<Expense> | null> {
  const apiKey = await SecureStore.getItemAsync('gemini_api_key');
  if (\!apiKey) return null; // Silently skip if key not configured

  const genAI = new GoogleGenerativeAI(apiKey);
  const model = genAI.getGenerativeModel({ model: 'gemini-1.5-flash' });

  const prompt = `
Extract expense information from this text and return a JSON object.
Text: "${input}"
Current date: ${currentDate}

{
  "amount": <numeric amount, 0 if unclear>,
  "category": "<category>",
  "description": "<brief description>",
  "date": "<ISO 8601 date, use ${currentDate} if today>"
}

Return null (not a JSON object) if the text doesn't describe an expense.
`;

  try {
    const result = await model.generateContent(prompt);
    const text = result.response.text();
    const jsonMatch = text.match(/\{[\s\S]*\}|null/);
    if (\!jsonMatch || jsonMatch[0] === 'null') return null;

    const parsed = JSON.parse(jsonMatch[0]);
    if (\!parsed.amount || parsed.amount <= 0) return null;

    return {
      amount: parsed.amount,
      category: VALID_CATEGORIES.includes(parsed.category)
        ? parsed.category
        : 'other',
      description: parsed.description || input.slice(0, 50),
      date: parsed.date || currentDate,
    };
  } catch {
    return null; // Fall back to manual entry form silently
  }
}
```

In the UI, run this async while keeping the input field interactive. If parsing fails or returns null, transition smoothly to a manual entry form — the user shouldn't notice that the AI failed.

## Personalized AI Advisor: Injecting Spending History as Context

This is the feature that makes the app worth paying for. The difference between a generic AI response and a useful one comes down to context quality. Rather than sending raw transaction history (expensive, noisy), send a structured summary:

```typescript
// lib/financialAdvisor.ts
import { GoogleGenerativeAI } from '@google/generative-ai';
import * as SecureStore from 'expo-secure-store';

interface MonthlySpendingSummary {
  month: string;
  totalAmount: number;
  categoryBreakdown: Partial<Record<ExpenseCategory, number>>;
  budgets: Partial<Record<ExpenseCategory, number>>;
  topExpenses: Array<{ description: string; amount: number; category: string }>;
}

export async function generateFinancialAdvice(
  summary: MonthlySpendingSummary,
  previousMonth?: MonthlySpendingSummary
): Promise<string> {
  const apiKey = await SecureStore.getItemAsync('gemini_api_key');
  if (\!apiKey) {
    return 'Add your Gemini API key in Settings to receive personalized spending insights.';
  }

  const genAI = new GoogleGenerativeAI(apiKey);
  const model = genAI.getGenerativeModel({
    model: 'gemini-1.5-pro',
    generationConfig: {
      maxOutputTokens: 500, // Keep advice concise — long walls of text don't get read
      temperature: 0.7,
    },
  });

  // Build month-over-month comparison
  const categoryLines = Object.entries(summary.categoryBreakdown).map(
    ([cat, amount]) => {
      const prev =
        previousMonth?.categoryBreakdown[cat as ExpenseCategory] ?? 0;
      const changeStr =
        prev > 0
          ? ` (${(((amount\! - prev) / prev) * 100).toFixed(0)}% vs last month)`
          : '';
      return `- ${cat}: $${amount\!.toFixed(2)}${changeStr}`;
    }
  );

  // Identify over-budget categories
  const overBudgetLines = Object.entries(summary.categoryBreakdown)
    .filter(([cat, amount]) => {
      const budget = summary.budgets[cat as ExpenseCategory];
      return budget && amount\! > budget;
    })
    .map(([cat, amount]) => {
      const budget = summary.budgets[cat as ExpenseCategory]\!;
      return `- ${cat}: $${amount\!.toFixed(2)} (over $${budget.toFixed(2)} budget by $${(amount\! - budget).toFixed(2)})`;
    });

  const prompt = `
You are a friendly, supportive personal finance coach.
Analyze the following spending data and provide 2–3 specific, actionable suggestions.

[${summary.month} spending summary]
Total: $${summary.totalAmount.toFixed(2)}

By category:
${categoryLines.join('\n')}

${
    overBudgetLines.length > 0
      ? `Over budget:\n${overBudgetLines.join('\n')}`
      : 'All categories within budget.'
  }

Top expenses this month:
${summary.topExpenses
    .slice(0, 3)
    .map(e => `- ${e.description}: $${e.amount.toFixed(2)}`)
    .join('\n')}

[Instructions]
- Reference specific numbers from the data above
- Give actionable suggestions, not vague advice like "spend less"
- Be encouraging, not judgmental
- Do NOT recommend any specific investment products, stocks, or financial instruments
- Keep the response to 150–200 words
`;

  try {
    const result = await model.generateContent(prompt);
    return result.response.text();
  } catch (error) {
    console.error('Advice generation failed:', error);
    // Graceful degradation: return a local summary instead of crashing
    return `${summary.month} summary: Total spending was $${summary.totalAmount.toFixed(2)}. Detailed AI analysis requires an internet connection.`;
  }
}
```

The `maxOutputTokens: 500` cap is intentional. I've found that concise, focused advice (150–200 words) gets read and acted on. Long AI essays don't. The token cap also keeps API costs predictable.

The instruction "Do NOT recommend any specific investment products" is not optional — it's an App Store requirement. Omitting it risks Gemini generating responses that look like investment advice, which triggers review rejection.

## Security and Privacy: Handling Financial Data Responsibly

This section exists because financial apps face stricter scrutiny from App Store reviewers — and from users. Three areas require deliberate design.

### Secure API Key Storage

```typescript
// lib/secureStorage.ts
import * as SecureStore from 'expo-secure-store';
import * as FileSystem from 'expo-file-system';

export async function saveApiKey(key: string): Promise<void> {
  // iOS: stored in Keychain
  // Android: stored in EncryptedSharedPreferences
  await SecureStore.setItemAsync('gemini_api_key', key, {
    keychainAccessible: SecureStore.WHEN_UNLOCKED_THIS_DEVICE_ONLY,
    // Prevents the key from appearing in iCloud backups
    // Also prevents migration to a new device via backup restore
  });
}

export async function getApiKey(): Promise<string | null> {
  try {
    return await SecureStore.getItemAsync('gemini_api_key');
  } catch {
    // Keychain access can fail on devices with unusual encryption states
    return null;
  }
}

// Complete data deletion — required if your privacy policy says "stored locally"
export async function deleteAllUserData(
  db: SQLite.SQLiteDatabase
): Promise<void> {
  const errors: string[] = [];

  try {
    await db.execAsync('DELETE FROM expenses; DELETE FROM budgets;');
  } catch {
    errors.push('expense data deletion');
  }

  try {
    await SecureStore.deleteItemAsync('gemini_api_key');
  } catch {
    errors.push('API key deletion');
  }

  try {
    const receiptDir = FileSystem.documentDirectory + 'receipts/';
    const info = await FileSystem.getInfoAsync(receiptDir);
    if (info.exists) {
      await FileSystem.deleteAsync(receiptDir, { idempotent: true });
    }
  } catch {
    errors.push('receipt image deletion');
  }

  if (errors.length > 0) {
    throw new Error(`Partial deletion failure: ${errors.join(', ')}`);
  }
}
```

`WHEN_UNLOCKED_THIS_DEVICE_ONLY` does two things: it restricts access to when the device is unlocked (reasonable for an API key), and it excludes the key from iCloud backup. Without this flag, a user's Gemini API key could end up in their iCloud backup and theoretically accessible from another device.

### Receipt Image Storage Strategy

`FileSystem.documentDirectory` is backed up to iCloud by default. `FileSystem.cacheDirectory` is not. For receipt images, you have two reasonable choices: store them in the cache directory (they may be cleared by the OS), or store them in documents but delete them after successful analysis. Either approach needs to be documented in your privacy policy.

### Transparency About Gemini Processing

When a user scans a receipt, that image leaves their device and is processed by Google's servers. This fact must appear in three places to pass App Store review: the onboarding flow, the privacy policy, and the App Store data disclosure form (under "Data Linked to You" → "Financial Info").

## Five App Store Review Pitfalls for Finance Apps

These are drawn from actual review feedback, not speculation.

**① Language that sounds like investment advice**

Even indirect language like "putting your savings to work" or "growing your money" can trigger a review note. The safest language focuses on tracking and awareness. If you want Gemini to discuss savings goals, add an explicit constraint in the system prompt: "Do not recommend any investment products, financial instruments, or specific financial strategies."

**② Vague camera permission descriptions**

`NSCameraUsageDescription` (and the Android equivalent) must explain the specific use case. "The app uses your camera" fails review. "The app uses your camera to photograph receipts for automatic expense categorization" passes.

**③ Missing financial data disclosure in App Store Connect**

If you're using any analytics SDK (Mixpanel, Amplitude, PostHog), the spending data that flows through those SDKs must be declared under "Financial Information" in App Store Connect's privacy nutrition label. This is easy to overlook when you first add an analytics library.

**④ Paywall design for AI features**

If AI analysis is a premium feature, the free tier must still offer meaningful value — the App Store rejects apps where the free experience is a non-functional preview. The most review-friendly pattern is a usage-based gate: a set number of receipt scans per month for free, with unlimited access on a paid plan. Users understand this model and reviewers accept it.

**⑤ Data deletion functionality must exist and be documented**

If your privacy policy states that data is stored locally, Apple expects a "Delete all data" option to exist in the app. It doesn't need to be prominently featured, but it must be in settings and functional. Reviewers test this.

## Common Implementation Pitfalls

**Pitfall 1: Gemini returns inconsistent JSON structure**

Even with examples in the prompt, Gemini 1.5 Flash occasionally returns a different structure. The `jsonMatch` regex extraction in the code above handles cases where Gemini wraps the JSON in a code block. The `parsed.category` fallback to `'other'` handles cases where an unexpected category string is returned. Both are necessary in production.

**Pitfall 2: Confusing `execAsync` and `runAsync` in expo-sqlite**

`execAsync` runs multiple SQL statements and returns void. `runAsync` runs a single statement and returns a `SQLiteRunResult` with a `changes` field. For INSERT operations where you want to verify success, use `runAsync` and check `result.changes > 0`. Using `execAsync` for inserts silently succeeds even when you might expect a failure signal.

**Pitfall 3: Receipt images filling local storage**

A user who scans three receipts a day will accumulate several hundred MB of images in six months. Design for deletion from the start: either delete the original after a successful analysis, keep only a thumbnail, or implement automatic cleanup for images older than 30 days. Include this behavior in your privacy policy.

**Pitfall 4: Aggregation queries slowing down after several months of data**

Without indexes, `SUM(amount) GROUP BY category WHERE date LIKE '2026-%'` will table-scan every row as data grows. The indexes in the schema above (on `date` and `category`) keep this fast. For apps with very high transaction volumes, consider materializing monthly aggregates into a separate `monthly_summary` table and updating it on every insert.

**Pitfall 5: Unclear failure mode when Gemini returns low-confidence results**

A confidence of 0.3 means the AI is guessing. If you save that data silently, the user's spending history becomes unreliable. The right UX is to surface low-confidence results with the amount field pre-filled but editable, and the category shown as a suggestion rather than a fait accompli. This turns an AI limitation into an intuitive correction flow.

**Pitfall 6: API key not configured on first launch**

The first time a user opens the app and tries to scan a receipt, `SecureStore.getItemAsync` returns null. Without an early check, the app will crash at the API call. The code above throws a descriptive error message before the API call, which you can catch and route to your API key setup screen. First-launch experience matters more than developers tend to think.


## Tying It All Together: The Monthly Summary Screen

Here's how the components above combine into the main screen a user sees every day. This example shows a `MonthlyDashboard` component that queries the database, computes the summary, and offers the AI advice flow:

```typescript
// screens/MonthlyDashboard.tsx
import React, { useState, useEffect, useCallback } from 'react';
import { View, Text, TouchableOpacity, ActivityIndicator, Alert } from 'react-native';
import { getExpensesByMonth, getCategoryTotals } from '../lib/database';
import { generateFinancialAdvice } from '../lib/financialAdvisor';

interface Props {
  currentMonth: string; // "2026-04"
  previousMonth: string; // "2026-03"
}

export function MonthlyDashboard({ currentMonth, previousMonth }: Props) {
  const [totals, setTotals] = useState<Partial<Record<ExpenseCategory, number>>>({});
  const [advice, setAdvice] = useState<string | null>(null);
  const [loadingAdvice, setLoadingAdvice] = useState(false);
  const [totalSpend, setTotalSpend] = useState(0);

  useEffect(() => {
    async function loadData() {
      const monthTotals = await getCategoryTotals(currentMonth);
      setTotals(monthTotals);
      const total = Object.values(monthTotals).reduce((sum, v) => sum + (v ?? 0), 0);
      setTotalSpend(total);
    }
    loadData();
  }, [currentMonth]);

  const requestAdvice = useCallback(async () => {
    setLoadingAdvice(true);
    setAdvice(null);

    try {
      // Build summaries for current and previous month
      const [prevTotals, topExpenses] = await Promise.all([
        getCategoryTotals(previousMonth),
        getExpensesByMonth(currentMonth).then(expenses =>
          expenses
            .sort((a, b) => b.amount - a.amount)
            .slice(0, 5)
            .map(e => ({ description: e.description, amount: e.amount, category: e.category }))
        ),
      ]);

      const prevTotal = Object.values(prevTotals).reduce(
        (sum, v) => sum + (v ?? 0), 0
      );

      const adviceText = await generateFinancialAdvice(
        {
          month: currentMonth,
          totalAmount: totalSpend,
          categoryBreakdown: totals,
          budgets: {},  // Pass real budget data from your budgets table
          topExpenses,
        },
        prevTotal > 0
          ? {
              month: previousMonth,
              totalAmount: prevTotal,
              categoryBreakdown: prevTotals,
              budgets: {},
              topExpenses: [],
            }
          : undefined
      );

      setAdvice(adviceText);
    } catch (error) {
      Alert.alert(
        'Could not generate advice',
        error instanceof Error ? error.message : 'Please try again later.'
      );
    } finally {
      setLoadingAdvice(false);
    }
  }, [currentMonth, previousMonth, totals, totalSpend]);

  return (
    <View style={{ flex: 1, padding: 16 }}>
      <Text style={{ fontSize: 24, fontWeight: 'bold' }}>
        ${totalSpend.toFixed(2)} this month
      </Text>

      {/* Category breakdown — connect to your chart library here */}
      {Object.entries(totals).map(([cat, amount]) => (
        <View key={cat} style={{ flexDirection: 'row', justifyContent: 'space-between', paddingVertical: 4 }}>
          <Text>{cat}</Text>
          <Text>${(amount ?? 0).toFixed(2)}</Text>
        </View>
      ))}

      {/* AI Advice Section */}
      {advice ? (
        <View style={{ marginTop: 24, padding: 16, backgroundColor: '#f0f4ff', borderRadius: 12 }}>
          <Text style={{ fontWeight: '600', marginBottom: 8 }}>AI Insights</Text>
          <Text style={{ lineHeight: 22 }}>{advice}</Text>
        </View>
      ) : (
        <TouchableOpacity
          onPress={requestAdvice}
          disabled={loadingAdvice || totalSpend === 0}
          style={{
            marginTop: 24,
            padding: 14,
            backgroundColor: loadingAdvice ? '#ccc' : '#4f46e5',
            borderRadius: 10,
            alignItems: 'center',
          }}
        >
          {loadingAdvice ? (
            <ActivityIndicator color="#fff" />
          ) : (
            <Text style={{ color: '#fff', fontWeight: '600' }}>
              Get AI spending insights
            </Text>
          )}
        </TouchableOpacity>
      )}
    </View>
  );
}
```

A few design decisions worth calling out here:

The advice request is triggered by the user, not automatic. Auto-generating advice every time the screen loads would burn API quota and slow the app. User-initiated requests also feel less surveillance-y — which matters for a finance app where trust is the product.

The `Promise.all` for loading previous month data and top expenses runs them in parallel, which matters when both database queries are on the critical path of the advice generation call.

The `totalSpend === 0` guard prevents the button from being active at the start of a month when there's nothing to analyze yet. Small UX details like this reduce confused support requests.

## Next Steps After Getting It Running

Once you have the core features working, the highest-leverage next step is getting the app into TestFlight and watching a few real users try to use it. You'll quickly discover UI problems that self-testing can't surface: users who can't figure out how to photograph a receipt in the right orientation, users who want to edit an AI-categorized expense but can't find how, users confused about where the monthly advice appears.

Fix those issues before App Store submission. A smooth UX reduces review rejection risk and dramatically improves early ratings — and early ratings heavily influence App Store search ranking.

The AI advice feature has a compounding quality dynamic: as users correct AI-assigned categories, you accumulate data that can improve your category mapping logic. Even simple changes — like tracking which categories get corrected most often and adding those patterns to your Gemini prompt — will meaningfully improve accuracy over time without any model training.

Start with the MVP. Get it into real hands. Then iterate on the intelligence.
