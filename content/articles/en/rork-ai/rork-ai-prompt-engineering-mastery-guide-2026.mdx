Rork promises that anyone can build apps through natural language. But most developers who use it seriously hit the same wall: the output is "close enough" but never quite right. UIs don't match the vision. Error fixes introduce new bugs. State management ends up tangled. The more complex the feature, the further the AI seems to drift from what you actually wanted.

The root cause is almost never Rork's capabilities. It's the prompt.

Prompt engineering — the practice of designing inputs to get the best outputs from AI — is the single most leveraged skill in AI-assisted development. This guide covers everything you need to master it for Rork specifically: how Rork processes context, seven core principles for writing precise prompts, and a library of 20 production-ready templates for every stage of mobile app development.

If you're new to the topic, [Rork Prompt Engineering Tips for App Design](/articles/rork-basics/rork-prompt-engineering-app-design-tips) and [Rork AI Prompt Troubleshooting FAQ](/articles/rork-ai/rork-ai-prompt-not-working-faq) are great starting points before diving in here.

## Why Your Prompt Is the Bottleneck — Not Rork

Rork AI is powered by a large language model (LLM) under the hood. LLMs are fundamentally probabilistic: they generate the most statistically likely next token given the input. This means the quality of the input directly determines the quality of the output. There is no magic, no intent-reading, no inference beyond what you explicitly provide.

The same task — "add a chat screen" — produces wildly different results depending on how it's framed:

**❌ Low-quality prompt:**
```
Add a chat screen
```

**✅ High-quality prompt:**
```
[Feature] Real-time chat screen

[Stack]
- Supabase Realtime for message subscription
- Message model: { id, userId, content, createdAt }
- FlatList to display history (newest at bottom)

[UI Specs]
- Fixed text input + send button at bottom
- Own messages: right-aligned, blue background
- Others' messages: left-aligned, gray background
- Use KeyboardAvoidingView so input stays visible when keyboard opens

[Constraints]
- No mock data — fetch from Supabase only
- No TypeScript `any` types
```

Both prompts take about the same time once you build the habit. The output quality difference is night and day.

## Understanding How Rork AI Processes Context

Before writing better prompts, it helps to understand how Rork interprets them.

### The Context Window

Every LLM has a "context window" — the total amount of text it can hold in working memory at once. Rork's underlying models support tens of thousands to hundreds of thousands of tokens (roughly 1–2 characters per token). Critically, this window accumulates across your entire conversation history. When it fills up, the oldest messages are dropped.

This is why code generation quality often degrades as a project grows. The AI is working with incomplete — and sometimes contradictory — context.

### Preventing Context Degradation

**Strategy 1: Use a reset prompt when things drift**
```
Let me summarize the current project state before continuing.

Completed features:
- [List of finished screens/functionality]

Current file structure:
- [Key files and their roles]

Active dependencies:
- [Libraries in use]

With this context in mind, please implement the following:
[New instruction]
```

**Strategy 2: Start fresh sessions for major features**
When adding a large new module, opening a new chat and pasting the "project context prompt" (covered later) often yields better results than continuing an old session.

### Code as Context

Rork doesn't just read your prompts — it reads your existing codebase too. This has practical implications:

- **Comments in your code are documentation for the AI.** Well-commented functions give the AI better signals when generating related code.
- **Consistent naming conventions** help the AI infer intent and follow your patterns.
- **TypeScript type definitions** dramatically improve accuracy. A typed interface is a precise specification the AI can reason against.

## Seven Core Principles for Precision Prompts

These principles emerged from extensive trial and error building production apps with Rork. Apply them consistently and you'll rarely get off-target outputs.

### Principle 1: Separate What, How, and Why

Structure prompts in three distinct sections:

```
[WHAT: What to build]
User profile edit screen

[HOW: How to implement it]
- Form validation: react-hook-form + zod
- Data source: Supabase profiles table
- Avatar: expo-image-picker for photo updates

[WHY: Context and constraints]
- Must reference the existing AuthContext for the user object
- After save, navigate back to the home screen
```

### Principle 2: Explicit Prohibitions

AI will choose "sensible defaults" unless told otherwise. Prohibitions prevent unwanted choices:

```
[Constraints — Do NOT]
- Do not use local useState for shared state (use Zustand)
- Do not leave console.log statements in the output
- Do not hardcode API keys in the source
- Do not modify or overwrite [filename].tsx
```

### Principle 3: Specify the Output Format

```
Please structure your response as follows:
1. A 3-sentence explanation of your implementation approach
2. The complete code with inline comments
3. A bulleted "Testing steps" section at the end

All code should be TypeScript with no omitted type definitions.
```

### Principle 4: Scope the Change

AI tends to "improve" things beyond what you asked. Scope your requests explicitly:

```
Limit your changes to these two files only:
- src/screens/ProfileScreen.tsx
- src/hooks/useProfile.ts

Do not modify any other files. Do not refactor unrelated code.
```

### Principle 5: Reference Existing Patterns

```
Follow the same patterns as the existing codebase:
- Use the same styling approach as Button.tsx (src/components/Button.tsx)
- Fetch auth info from useAuth (src/hooks/useAuth.ts)
- New components should follow the same file structure pattern
```

### Principle 6: Always Request Error Handling

```
Handle all of the following error cases explicitly:
- Network error: show user-facing feedback with retry option
- Auth error: redirect to login screen
- Validation error: per-field error messages below each input
- Loading state: show ActivityIndicator while async operations run
```

### Principle 7: Break Complex Features into Phases

Single-prompt attempts at complex features produce lower-quality output. Phase your work:

```
We'll implement this in three phases. Only implement Phase 1 today:

Phase 1: UI components with mock data
Phase 2: Connect to the real API
Phase 3: Error handling, edge cases, and loading states
```

## Screen and Component Generation Prompt Patterns

### Pattern 1: Building a New Screen

```
[New Screen: Task List]

Component: TaskListScreen
Path: src/screens/TaskListScreen.tsx

[Data Type]
type Task = {
  id: string;
  title: string;
  completed: boolean;
  dueDate: Date;
  priority: 'low' | 'medium' | 'high';
};

[UI Requirements]
- FlatList to render tasks
- Completed tasks: checkmark icon + strikethrough title
- Priority color via left border (low=gray, medium=amber, high=red)
- Swipe-to-delete (react-native-gesture-handler)
- Empty state: illustration + "No tasks yet" text

[Navigation]
- "+" button in top-right navigates to TaskCreateScreen
- Tapping a task navigates to TaskDetailScreen with the task id as param

[Styling]
- NativeWind (Tailwind CSS syntax)
- Primary: #4F46E5, Background: #F9FAFB
```

### Pattern 2: Modifying an Existing Component

```
[Modification: Add button variants]

File: src/components/Button.tsx

[What to add]
New `variant` prop: 'primary' | 'secondary' | 'danger' | 'ghost'

Current state: only 'primary' is implemented
Add: secondary (white bg, border), danger (red bg), ghost (no bg, text only)

[Constraints]
- Do not change the existing primary variant
- Loading spinner must work across all variants
- Update TypeScript types accordingly
- Changes must be limited to Button.tsx only

[Usage reference]
<Button variant="danger" onPress={handleDelete} loading={isDeleting}>
  Delete
</Button>
```

### Pattern 3: Form with Validation

```
[Form: User Registration]

Library: react-hook-form + zod

[Validation rules]
- email: valid email format
- password: min 8 chars, must contain letters and numbers
- confirmPassword: must match password
- username: 3–20 chars, alphanumeric + underscore only

[UX requirements]
- Real-time validation on change (immediate feedback)
- Submit button disabled until all fields are valid
- Show loading spinner on submit button while submitting
- Error messages in red text directly below each field

[On submit]
- Success: POST to API → navigate to HomeScreen
- Failure: display error as toast notification (react-native-toast-message)
```

### Pattern 4: Infinite Scroll List

```
[Infinite scroll list implementation]

Data source: Supabase (posts table)
Page size: 20 items

[Requirements]
- Initial load: fetch 20 most recent posts
- At scroll end: fetch next 20
- During fetch: show ActivityIndicator in list footer
- After last page: show "All caught up" message in footer
- Pull-to-refresh: re-fetch from top

[Performance]
- Optimize FlatList initialNumToRender and maxToRenderPerBatch
- Implement getItemLayout if item heights are fixed
- keyExtractor must use the id field

[Error handling]
- Fetch failure: show "Retry" button in footer
- Network offline: show offline banner at top of screen
```

## API Integration and State Management Prompt Patterns

### API Integration Template

```
[API Integration: Product List]

Endpoint: GET /api/products
Response type:
{
  data: Array<{
    id: string;
    name: string;
    price: number;
    imageUrl: string;
    category: string;
  }>;
  total: number;
  page: number;
}

[Implementation approach]
- Use TanStack Query (react-query)
- queryKey: ['products', { page, category }]
- staleTime: 5 minutes (product data doesn't change frequently)
- Auto-retry on error: 3 attempts

[Environment]
- Base URL from process.env.EXPO_PUBLIC_API_URL
- API key in Authorization header as Bearer token
  (token stored in AsyncStorage from auth flow)

[Custom hook design]
- Implement as useProducts(page, category)
- Return: { products, isLoading, error, refetch, fetchNextPage }
```

### Zustand Store Template

```
[Zustand store: Shopping Cart]

File: src/stores/cartStore.ts

[State definition]
type CartItem = {
  productId: string;
  name: string;
  price: number;
  quantity: number;
};

type CartState = {
  items: CartItem[];
  // Derived state
  totalItems: number;
  totalPrice: number;
  // Actions
  addItem: (product: Product) => void;
  removeItem: (productId: string) => void;
  updateQuantity: (productId: string, quantity: number) => void;
  clearCart: () => void;
};

[Implementation requirements]
- Persist to AsyncStorage using zustand/middleware persist
- Memoize totalItems and totalPrice as computed values
- addItem: if item exists, increment quantity by 1
- updateQuantity: if quantity reaches 0, auto-call removeItem
- Use immer middleware for immutable updates

[Testability]
- Add a resetCart function to enable test teardown
```

## Error Debugging Prompt Techniques

Debugging is where prompt quality matters most. Pasting only the error message rarely produces a useful fix. Provide full context.

### The Debugging Template

```
[Bug report]

Error message:
```
TypeError: Cannot read properties of undefined (reading 'map')
  at ProductList.tsx:45
```

Conditions:
- Occurs on first render of ProductList screen (before data loads)
- Does not occur after data is loaded

Problematic code:
```typescript
const ProductList = () => {
  const { products } = useProducts();
  return (
    <FlatList
      data={products.data}  // ← error here
      ...
    />
  );
};
```

Desired fix:
- Either prevent useProducts from returning undefined in initial state
- Or add a fallback for when products.data is undefined
- While loading, show a skeleton loader instead of the list
```

### The Refactoring Template

```
[Refactoring request]

Target: src/screens/HomeScreen.tsx
Current problems:
- File is 400+ lines and hard to read
- Business logic and UI are mixed
- Same data-fetching logic is duplicated in 3 places

Refactoring goals:
1. Extract data fetching into useHomeData custom hook
2. Split into sub-components:
   - HeroBanner.tsx
   - FeaturedProductsSection.tsx
   - RecentOrdersSection.tsx
3. Move shared fetchWithErrorHandling to utils/api.ts

[Hard constraints]
- External behavior and visual output must not change
- Strengthen TypeScript types (eliminate any)
- Add comments explaining intent
```

**Template 1: Social login**
```
Implement Apple Sign In + Google Sign In using @clerk/clerk-expo.

- Must coexist with existing email/password login without breaking it

```
Persist auth state across app restarts:
- Store tokens in expo-secure-store (not AsyncStorage)
- Validate token in AppLoading component on startup
- Valid token: navigate to HomeScreen

- Animation: points appear sequentially on mount

**Template 4: Search with Algolia**
```
Implement real-time search with Algolia:
- 300ms debounce on text input

### Performance

**Template 5: Image list optimization**
```
Optimize heavy image list performance:
- Use expo-image (replacement for react-native-fast-image)
- Blurred placeholder while loading

- Fixed item heights with getItemLayout for smooth scrolling
- Transform image URLs to prefer WebP format

**Template 6: Card expand animation**
```
Implement a card expand animation with react-native-reanimated:
- Initial: card height 60px (title only)
- On tap: smoothly expand to 240px (duration: 300ms)

- Arrow icon rotates 180° during expand
- Implement with useSharedValue + useAnimatedStyle
- Must work correctly on both iOS and Android

### Security

**Template 7: Biometric auth gate**
```
Add a biometric authentication gate with expo-local-authentication:
- Require auth on app launch and on return from background
- Check Face ID/Touch ID availability
- Fallback to PIN input on unsupported devices
- Lock out for N seconds after 3 failures
- Overlay that hides screen content until authentication succeeds
```

**Template 8: API key security**
```
Implement secure API key management:
- All secrets managed via .env + process.env.EXPO_PUBLIC_*
- No keys hardcoded in source (excluded from git history)
- Non-public keys (Secret Keys, etc.) accessed via backend proxy only
- Runtime token management via expo-secure-store
```

## Common Failure Patterns and How to Avoid Them

### Failure 1: Requesting too many changes at once

**Problematic:**
```
Add user auth, a profile screen, a settings screen, and push notifications.
```

**Fix:** Work on one feature at a time. Verify it works before moving to the next.

### Failure 2: Vague style descriptions

**Problematic:**
```
Make it look nice and modern
```

**Better:**
```
Follow Apple Human Interface Guidelines:
- Font: SF Pro (system font)
- Colors: light mode #1C1C1E / dark mode #FFFFFF
- Border radius: 12px
- No drop shadows
- Button height: 50px
- Base spacing unit: 16px grid
```

### Failure 3: "Just fix it" without context

**Problematic:**
```
Something's broken, please fix it
```

**Better:**
```
Problem: [specific symptom]
When it occurs: [trigger/conditions]
Expected behavior: [what should happen]
Error message: [paste it]
What I've tried: [your debugging attempts so far]
```

### Failure 4: Not providing architecture context

When adding features without telling the AI what's already in place, it often generates code that contradicts your existing patterns.

**Fix:** Start every significant prompt with your stack context:
```
Match the existing project architecture:
- State management: Zustand (not Redux)
- Styling: NativeWind
- Navigation: expo-router (not React Navigation)
- Data fetching: TanStack Query
- Backend: Supabase
```

## Building a Prompt Asset Library

The best prompts are the ones you've already refined. Treat them as code — version them, share them, reuse them.

### Folder Structure

Create a `prompts/` directory at your project root:

```
prompts/
├── components/
│   ├── form-with-validation.txt
│   ├── infinite-scroll-list.txt
│   └── animated-card.txt
├── api/
│   ├── rest-api-integration.txt
│   └── realtime-subscription.txt
└── debugging/
    ├── typescript-error.txt
    └── render-performance.txt
```

### The Project Context Prompt

Create a single file that describes your project's full technical context. Paste it at the start of every new Rork session:

```
# Project Context

## App summary
[App name]: [One-line description]

## Tech stack
- React Native + Expo SDK 52
- Navigation: expo-router v4
- State: Zustand + TanStack Query
- UI: NativeWind v4 + shadcn/ui
- Backend: Supabase
- Payments: RevenueCat + Stripe

## Code conventions
- All TypeScript, no `any`
- Functional components only
- Business logic in custom hooks, not screens
- Errors caught by Error Boundaries

## File naming
- Components: PascalCase (e.g., UserCard.tsx)
- Hooks: camelCase (e.g., useUser.ts)
- Utils: camelCase (e.g., formatDate.ts)
```

Pasting this at the start of a session means every subsequent prompt inherits this full context — without you having to repeat yourself.

## Frequently Asked Questions

**Q1. Sometimes longer prompts seem to produce worse results. Am I over-specifying?**

A. It's possible. The issue isn't length — it's relevance. Every token in your prompt competes for the model's attention. If your prompt contains information unrelated to the current task, it dilutes the signal. Keep prompts focused on the task at hand. Use headings and bullet points to structure longer prompts clearly, and check for contradictory instructions. The rule of thumb: one prompt, one task.

**Q2. Should I write prompts in English or Japanese?**

A. Rork's underlying models are primarily trained on English data, so English prompts — especially for technical specifications — often produce more accurate results. That said, modern models handle Japanese well enough for most use cases. A practical approach: start in Japanese, and switch specific technical sections (data types, constraints, API specs) to English if you notice quality degrading. Even writing code comments in English can help.

**Q3. Does prompt strategy differ between Rork and Rork Max?**

A. Yes, meaningfully. Rork Max generates native Swift/SwiftUI code, so prompts can include platform-specific details — HealthKit permissions, SwiftData model definitions, Core ML model input shapes, etc. Rork Max also supports a larger context window, making it more resilient to longer spec documents. Standard Rork is optimized for cross-platform React Native/Expo, so your prompts should stay within that ecosystem. For Rork Max-specific tips, see [Rork Max Prompt Engineering Tips](/articles/rork-dev/rork-max-prompt-tips).

## Wrapping Up: From Issuing Instructions to True Collaboration

When you internalize the techniques in this guide, your relationship with Rork AI shifts. Instead of issuing vague commands and hoping for the best, you're communicating clear specifications to a capable collaborator. You're not asking the AI to guess what you want — you're telling it precisely what you need, in a form it can act on reliably.

Start by copying the templates directly and adapting them to your project. As you build your own prompt library, you'll develop an instinct for what works. The investment compounds: prompts you refine once continue delivering value across every feature you build.
