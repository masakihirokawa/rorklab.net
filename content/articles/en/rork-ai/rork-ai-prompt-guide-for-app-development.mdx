You typed "make a to-do app" into Rork and got something back — but it's not quite what you pictured.

This is a common experience, and the fix is usually in how the prompt is written rather than what's possible in Rork. The AI is trying to fill in gaps with reasonable defaults. The more of those gaps you fill with specific information, the less Rork has to guess.

## Tip 1: Start with who's using the app

User context shapes the UI decisions Rork makes about layout, tone, and complexity.

```
Vague:
"Make a budgeting app"

Better:
"Make a budgeting app for a couple in their 30s who want to 
 take a photo of receipts and log daily expenses. 
 The priority is simplicity — it should feel effortless to use."
```

"Who," "for what purpose," and "in what situation" changes the visual direction and interaction flow Rork produces.

## Tip 2: List the screens upfront

Naming the screens you want tells Rork the intended structure before it has to infer it.

```
"Build a task management app with these screens:
 1. Task list (filterable by complete/incomplete)
 2. Add task screen (title, due date, priority)
 3. Task detail screen (edit, delete, mark complete)
 4. Settings screen (notification toggle)"
```

If you're not sure about all the screens yet, add: "Start with a minimal working version and we can add screens later." This prevents Rork from trying to build everything at once.

## Tip 3: Describe the visual direction in keywords

A few words about aesthetics go a long way.

```
Design direction examples:
- "Minimal and monochromatic"
- "Bright and friendly. Green as the accent color"
- "Professional, business-focused. Dark mode support"
- "Colorful with large text, designed for children"
- "Photo-centric layout like Instagram"
```

Naming a reference app ("similar feel to [App Name]") works well when Rork can recognize it.

## Tip 4: Be specific about data structure

Describing the data model upfront prevents mismatches between what you expect and what Rork builds.

```
"Each entry a user can log:
 - Meal name (text)
 - Calories (number)
 - Time of meal (date/time)
 - Photo (optional)
 - Notes (optional, text)

List view: grouped by date
Summary: show total calories for the day"
```

Writing this detail upfront eliminates follow-up requests like "add calorie tracking" or "group entries by date."

## Tip 5: State what you don't want

Rork makes assumptions. Explicitly excluding things you don't want prevents those assumptions from showing up.

```
"Important: please exclude the following:
 - Social features (sharing, following, etc.)
 - Login or account creation (not needed for v1)
 - Ad placement areas
 - Keep navigation to a minimum — no more than 3 main tabs"
```

Saying what you don't want is just as useful as saying what you do.

## Tip 6: Make revision instructions specific

When asking for changes, pointing to exactly what needs to change prevents unintended edits to other parts of the app.

```
Vague revision:
"Make the design better"

Specific revision:
"In the task list screen, increase the card height slightly,
 set the task name font size to 16px, and change the text color to #333333"
```

Specific scope prevents Rork from rewriting parts of the app that were already working.

## Tip 7: Build incrementally

Complex apps built in one shot tend to miss the mark in multiple ways. Building in stages lets you confirm each piece before adding the next.

```
Phase 1:
"Build a simple notes app. Just add notes, delete notes, 
 and list them. That's all for now."

After reviewing →

Phase 2:
"Add a label feature so I can tag notes with a color.
 Four colors: red, yellow, blue, green."

After reviewing →

Phase 3:
"Add a filter so I can view notes by label color."
```

Each phase is reviewable on its own, and corrections at each step are smaller and more targeted.

## A Complete Example Prompt

Here's a prompt that combines all of the above:

```
[Purpose]
A sleep tracking app to help people log and visualize their 
nightly sleep. Target user: working adults in their 20s-40s 
trying to improve their sleep quality.

[Screens]
1. Home (today's sleep log + 7-day chart)
2. Log entry (bedtime, wake time, sleep quality rating, notes)
3. History (monthly view of past entries)

[Design direction]
Dark theme. Navy and deep blue tones. White and light gray text. 
A calm, nighttime aesthetic.

[Data per entry]
- Bedtime (time)
- Wake time (time)
- Sleep duration (auto-calculated)
- Sleep quality (1-5 score)
- Notes (optional text)

[Exclude]
Social features, login/account creation, ad placements
```

You don't need to write this much every time. If you want to see something first before committing to details, keep the prompt short and revise from there. Find the level of detail that works for how you think through ideas.
