# Rork App Rejected by App Store? Submission Troubleshooting FAQ

You've built an app with Rork, submitted it to the App Store, and—rejection. It's frustrating, but you're not alone. App Store review can feel unpredictable, but most rejections follow predictable patterns. This FAQ covers the seven most common rejection reasons for Rork apps and how to fix them.

## Q1: "Guideline 4.2 — Minimum Functionality" Rejection

**A:** This rejection means Apple believes your app doesn't provide sufficient practical value or core functionality.

### Why Does This Happen?

Apple's Guideline 4.2 requires apps to have "clear and useful functionality." Apps that get rejected typically have:

- **Placeholder-only content** — buttons and text that don't do anything
- **Incomplete features** — a login screen with no working database behind it
- **Demo-only apps** — features that look nice but don't actually work

### The Fix

**1. Ensure core features work end-to-end**

Use your own app and verify:
- All buttons perform their intended action
- Form data is actually saved to a database
- Navigation flows properly
- No dead links or broken features

```javascript
// Bad: button doesn't do anything
<Button title="Save" onPress={() => console.log('Save pressed')} />

// Good: button actually saves data
<Button title="Save" onPress={() => saveToDatabase(userData)} />
```

**2. Write a clear, detailed app description**

Your App Store description should explain exactly what users can do with your app.

```
✗ Bad: "A fun and useful app"

✓ Good: "Task management app where you can create tasks,
organize them by category, mark them complete, and sync across devices."
```

**3. Use real app screenshots**

App Store screenshots must show your actual app in use, with real data and working features—not mockups or concept art.

:::warning
Don't use placeholder images, diagrams, or concept art. Screenshots must accurately represent your functioning app.
:::

## Q2: Missing or Incomplete Privacy Policy

**A:** A privacy policy is mandatory for App Store approval. Apple reviews these very strictly.

### Why It's Required

Apple needs to understand exactly how your app collects, uses, and protects user data. Apps without clear privacy policies get rejected.

### The Solution

**1. Create a comprehensive privacy policy**

Your policy should include:

```markdown
# Privacy Policy

## Data We Collect
- User name, email (if user-provided)
- Device ID (for analytics)
- Location data (if permitted)

## How We Use Your Data
- User authentication and account management
- App analytics and improvement
- Error reporting

## Data Protection
- All data encrypted with SSL/TLS
- Regular secure backups
- Never sold or shared with third parties
```

**2. Add a Privacy Policy URL to App Store Connect**

Under "App Information," provide a link to your privacy policy hosted on a website.

**3. Keep it updated**

Every time you add a new feature that touches user data, update your privacy policy accordingly.

:::tip
Free privacy policy generators (iubenda, Termly) can help, but always customize them to your specific app and data practices.
:::

**4. Accurately complete the App Privacy section**

Since 2020, Apple requires an "App Privacy" label. Answer these questions honestly:

- What data does your app collect?
- Is that data used for tracking?
- Can users delete their data?

## Q3: Screenshots Don't Match the App

**A:** If your App Store screenshots don't accurately represent what users see when they open your app, Apple will reject it.

### Common Mistakes

- **Outdated screenshots** — the app was updated but screenshots weren't
- **Mockup images** — using design comps instead of actual app screenshots
- **Unresolved bugs** — bugs visible in screenshots also appear in the live app

### The Fix

**1. Capture fresh screenshots**

```bash
# Build a test version
eas build --platform ios

# Install and run on a test device
# Take screenshots of the actual app in use
```

**2. Verify screenshots match your live app**

Check:
- Are buttons in the correct positions?
- Is text displaying correctly?
- Are images and icons visible?
- Does the layout match your app?

**3. Include the full user flow**

Include screenshots of:
- Launch/login screen
- Main dashboard
- Key feature screens
- Settings or profile screen

:::warning
If your Rork app displays missing images or has visual bugs, fix those first. Mismatches between screenshots and the actual app are a rejection reason.
:::

## Q4: Crash Logs Detected

**A:** When Apple tests your app and it crashes (fails to launch or crashes during use), it gets rejected.

### Common Crash Causes

- **Memory leaks** — app doesn't properly release memory
- **Initialization errors** — database or API connections fail on startup
- **Type errors** — variables have incorrect types
- **Unhandled network errors** — app crashes when internet is unavailable

### The Fix

**1. Test thoroughly on real devices**

```bash
# Create a test build
eas build --platform ios --profile preview

# Install and run on test devices
# Exercise all features and flows
```

**2. Add proper error handling**

```javascript
// Always handle API failures gracefully
try {
  const data = await fetchDataFromAPI();
  setData(data);
} catch (error) {
  console.error('API Error:', error);
  // Show user a helpful message instead of crashing
  alert('Unable to load data. Please try again.');
}
```

**3. Test with no internet connection**

Simulate offline scenarios to ensure your app handles network failures gracefully.

```javascript
// Handle network errors
const fetchWithFallback = async (url) => {
  try {
    const response = await fetch(url);
    if (!response.ok) throw new Error('Network error');
    return await response.json();
  } catch (error) {
    console.error('Fetch failed:', error);
    return cachedData || null; // return cached data if available
  }
};
```

**4. Check crash logs in TestFlight**

Use Apple's TestFlight beta testing. If crashes occur, the crash logs will tell you exactly where your app is failing.

:::tip
Test on multiple device models and older iOS versions before submitting. Many crashes happen only on older hardware or OS versions.
:::

## Q5: In-App Purchase Configuration Error

**A:** In-App Purchase (IAP) setup is complex. Small mistakes cause rejections.

### Common IAP Mistakes

- **Missing App Store Connect setup** — code references an IAP product that doesn't exist in App Store Connect
- **No test IAP** — sandbox products aren't configured for testing
- **Price mismatches** — prices differ between your code and App Store Connect

### The Fix

**1. Set up IAP in App Store Connect**

```
App Store Connect > My Apps > [Your App] > In-App Purchases

Create new In-App Purchase:
- Product ID: "com.example.app.premium"
- Tier: $0.99 (example)
- Add localized description
```

**2. Verify your Rork code references the correct IAP product ID**

```javascript
// Make sure this ID exactly matches App Store Connect
import { requestPurchase } from 'react-native-iap';

const purchasePremium = async () => {
  try {
    await requestPurchase({
      skus: ['com.example.app.premium'], // Must match App Store Connect
    });
  } catch (error) {
    console.error('Purchase failed:', error);
  }
};
```

**3. Create test accounts for IAP testing**

In App Store Connect, add test accounts that can purchase without real charges. Use these to test IAP before submission.

:::warning
Always test IAP in the sandbox environment. Never test with real payments before submission. Use TestFlight with a test account to verify everything works.
:::

## Q6: Age Rating Not Set

**A:** Every app requires a content rating. Missing this causes rejection.

### What's a Content Rating?

Based on your app's content, Apple assigns a recommended age group:
- Violence, combat → 12+
- Romantic content → 12+
- Explicit content → 17+

Most simple apps qualify for 4+.

### The Fix

**1. Complete the rating questionnaire in App Store Connect**

```
App Store Connect > My Apps > [Your App] > Information >
Content Rating

Answer all questions honestly:
- Does your app contain violence?
- Romantic content?
- Alcohol, tobacco, or drug references?
- Profanity?
```

**2. Be honest in your answers**

Many beginners try to answer "No" to everything to get a 4+ rating, but Apple checks. Dishonest ratings get caught and can get your app removed entirely.

:::tip
Most simple productivity apps, games, and utilities correctly rate as 4+. Don't stress about this—Apple's rating system is straightforward if you're honest.
:::

## Q7: API Keys Hardcoded in Source Code

**A:** Embedding API keys directly in your code is a security risk that triggers rejection.

### Why This Is Dangerous

Hardcoded API keys are trivial to extract by reverse-engineering your app. Once exposed:
- Your API gets abused by attackers
- Your server gets overwhelmed
- Your credentials are compromised

### The Solution

**1. Use environment variables**

```javascript
// Bad: API key visible in source code
const API_KEY = "sk-1234567890abcdef";

// Good: loaded from environment
const API_KEY = process.env.EXPO_PUBLIC_API_KEY;
```

**2. Use a .env file (local development only)**

```
// .env
EXPO_PUBLIC_API_KEY=sk-1234567890abcdef
EXPO_PUBLIC_API_URL=https://api.example.com
```

```javascript
// app.json or app.config.js
import Constants from 'expo-constants';

const API_KEY = Constants.expoConfig.extra.apiKey;
```

**3. Use a backend server (most secure)**

The safest approach: keep API keys on your backend server only. Your Rork app calls your server, which handles API calls.

```
[Rork App] ──→ [Your Backend] ──→ [Third-party API]
           (no API keys)    (API keys secure here)
```

:::warning
Before production submission, scan your code for hardcoded secrets. Use tools like `git grep` to search for patterns like "sk_", "api_key=", or "secret".
:::

## Summary

Before submitting a Rork app to the App Store, verify:

1. **All features work** — no placeholder buttons or unfinished functionality
2. **Privacy policy exists** — it's complete, accurate, and linked
3. **Screenshots are current** — they show your real app, not mockups
4. **No crashes** — tested on multiple devices and iOS versions
5. **Secrets are safe** — no API keys or credentials in code
6. **Rating is set** — content rating questionnaire completed
7. **IAP is configured** — if you use In-App Purchase, it's properly set up

Most Rork app rejections are preventable with careful testing and honest policy documentation. When rejection happens, read Apple's detailed message carefully—they usually tell you exactly what to fix.
