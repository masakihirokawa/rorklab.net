## The Real Limitations of Rork Max's SwiftUI Output

Rork Max's code generation quality is genuinely impressive. I mean that. The speed at which it produces functional UI code is better than I expected. But after using it on real projects, I've noticed consistent patterns in what it misses.

**Accessibility is almost never included.** Rork Max rarely generates `.accessibilityLabel` or `.accessibilityHint`. App Store reviewers care about this, and missing accessibility support is now a common rejection reason. It's not a dealbreaker—it's a 10-minute fix—but you need to know it's coming.

**Error handling is surface-level.** Network requests often lack proper fallbacks. API responses aren't validated thoroughly. This matters, especially with subscriptions. When a billing request fails, you can't just crash or show a generic error message. You need to handle the specific case where the user's payment was declined, or where their connection dropped mid-transaction.

**State management is flat.** For a prototype, Rork Max's approach is fine. But once you have multiple screens sharing data, or you need to persist subscription status across app restarts, the generated structure becomes limiting. You'll find yourself refactoring more than you expected.

None of this is a serious problem. It's just different from the mindset of "ship this to production." Rork Max assumes you'll refine the output. That's healthy. We just need to know where to focus our refinement efforts.

## The App Store Review Gauntlet: Top 5 Rejection Reasons and How to Avoid Them

App Store rejection rates for new apps are around 30%. Most rejections aren't about bugs—they're about guideline compliance, data privacy, or user experience details that the AI doesn't know to prioritize.

### Rejection #1: Missing Privacy Policy and Data Collection Disclosure

This is the most common reason I see. Apps are rejected because they don't clearly disclose what data they collect and why.

Rork Max won't generate a privacy prompt. You need to add one.

### Rejection #3: Inadequate Error Handling

Network requests fail. Sometimes payment fails. Sometimes the user closes the app mid-transaction. Rork Max's networking code doesn't always handle these cases gracefully.

Add proper error handling:

```swift
@MainActor
class NetworkManager: NSObject, ObservableObject {
  @Published var isLoading = false
  @Published var errorMessage: String?
  
  func fetchWithRetry<T: Decodable>(
    url: URL,
    maxAttempts: Int = 3
  ) async throws -> T {
    var lastError: Error?
    
    for attempt in 1...maxAttempts {
      do {
        let (data, response) = try await URLSession.shared.data(from: url)
        
        guard let httpResponse = response as? HTTPURLResponse else {
          throw NetworkError.invalidResponse
        }
        
        guard (200...299).contains(httpResponse.statusCode) else {
          throw NetworkError.httpError(status: httpResponse.statusCode)
        }
        
        return try JSONDecoder().decode(T.self, from: data)
        
      } catch {
        lastError = error
        
        // Exponential backoff before retry
        if attempt < maxAttempts {
          try await Task.sleep(nanoseconds: UInt64(pow(2, Double(attempt))) * 1_000_000_000)
        }
      }
    }
    
    throw lastError ?? NetworkError.unknown
  }
}

enum NetworkError: LocalizedError {
  case invalidResponse
  case httpError(status: Int)
  case unknown
  
  var errorDescription: String? {
    switch self {
    case .invalidResponse:
      return "Server returned an invalid response. Please try again."
    case .httpError(let status):
      return "Server error (\(status)). Please try again later."
    case .unknown:
      return "Something went wrong. Please check your connection and try again."
    }
  }
}
```

This approach retries failed requests with exponential backoff, and gives the user a clear message if things really do fail.

## StoreKit 2: The Complete Subscription Implementation

Rook Max might generate a simple purchase button, but a real subscription system needs a lot more. You need to:

- Fetch available products from App Store
- Handle purchase transactions
- Verify receipts
- Track subscription status
- Handle restore purchases
- Monitor for cancellations

Here's the full implementation:

```swift
import StoreKit

@MainActor
class SubscriptionManager: NSObject, ObservableObject {
  @Published var availableProducts: [Product] = []
  @Published var purchasedProductIDs: Set<String> = []
  @Published var subscriptionStatus: SubscriptionStatus = .none
  @Published var isLoading = false
  @Published var errorMessage: String?
  
  private let productIDs = [
    "com.example.rork.monthly",    // $2.99/month
    "com.example.rork.annual",     // $24.99/year
  ]
  
  private var updateListenerTask: Task<Void, Error>?
  
  override init() {
    super.init()
    setupTransactionListener()
  }
  
  private func setupTransactionListener() {
    updateListenerTask = Task.detached { @MainActor [weak self] in
      for await result in Transaction.updates {
        await self?.handleTransaction(result)
      }
    }
  }
  
  func loadProducts() async {
    isLoading = true
    defer { isLoading = false }
    
    do {
      let products = try await Product.products(for: productIDs)
      self.availableProducts = products.sorted { $0.price < $1.price }
    } catch {
      self.errorMessage = "Failed to load products: \(error.localizedDescription)"
    }
  }
  
  func purchase(_ product: Product) async -> Bool {
    do {
      let result = try await product.purchase()
      
      switch result {
      case .success(let verification):
        let transaction = try checkVerified(verification)
        await transaction.finish()
        await refreshSubscriptionStatus()
        return true
        
      case .userCancelled:
        return false
        
      case .pending:
        self.errorMessage = "Purchase pending. Apple is verifying your payment."
        return false
        
      @unknown default:
        return false
      }
    } catch {
      self.errorMessage = "Purchase failed: \(error.localizedDescription)"
      return false
    }
  }
  
  func restorePurchases() async {
    do {
      try await AppStore.sync()
      await refreshSubscriptionStatus()
    } catch {
      self.errorMessage = "Restore failed: \(error.localizedDescription)"
    }
  }
  
  func refreshSubscriptionStatus() async {
    self.purchasedProductIDs.removeAll()
    
    for await result in Transaction.currentEntitlements {
      if case .verified(let transaction) = result {
        self.purchasedProductIDs.insert(transaction.productID)
      }
    }
    
    let hasSubscription = purchasedProductIDs.contains { id in
      id.contains("monthly") || id.contains("annual")
    }
    
    self.subscriptionStatus = hasSubscription ? .active : .none
  }
  
  private func handleTransaction(_ result: VerificationResult<Transaction>) async {
    do {
      let transaction = try checkVerified(result)
      await transaction.finish()
    } catch {
      print("Transaction verification failed: \(error)")
    }
  }
  
  private func checkVerified<T>(_ result: VerificationResult<T>) throws -> T {
    switch result {
    case .unverified:
      throw SubscriptionError.unverifiedTransaction
    case .verified(let safe):
      return safe
    }
  }
  
  deinit {
    updateListenerTask?.cancel()
  }
}

enum SubscriptionStatus {
  case none
  case active
}

enum SubscriptionError: Error {
  case unverifiedTransaction
}
```

Use this manager throughout your app:

## Pricing Strategy: Regional Pricing and Psychological Anchors

Here's where most developers go wrong. They pick a price, ship it, and hope. But pricing is an art.

Start with your home market. If you're in Japan, price in yen. If you're in the US, price in dollars. For Japan, I'd suggest starting at ¥400/month ($2.99/month). That's low enough that users don't overthink it, high enough that serious users don't feel cheap.

Then adjust for other markets using Purchasing Power Parity (PPP):

- **Japan**: ¥400/month
- **US**: $2.99/month
- **UK**: £1.99/month
- **India**: ₹99/month
- **Germany**: €2.99/month

The goal is psychological consistency, not literal exchange rates.

### Free Trial Psychology

A 7-day free trial increases conversion dramatically. The math is simple: users who experience your value are more likely to pay.

In StoreKit 2, you set the trial period in App Store Connect. But you should also remind users when the trial is ending:

```swift
func scheduleTrialExpirationNotification() {
  let content = UNMutableNotificationContent()
  content.title = "Your free trial ends tomorrow"
  content.body = "Start your subscription to keep all features."
  content.sound = .default
  
  // Send 1 day before expiration
  let trigger = UNTimeIntervalNotificationTrigger(
    timeInterval: 6 * 24 * 3600,
    repeats: false
  )
  
  let request = UNNotificationRequest(
    identifier: "trial-expiration",
    content: content,
    trigger: trigger
  )
  
  UNUserNotificationCenter.current().add(request)
}
```

### Annual Subscription Discount

People are willing to pay more per year if the monthly cost is lower. This is anchoring. A $24.99/year plan feels cheaper than $2.99/month (it's not—it's the same price—but psychologically it reads cheaper).

I'd suggest:

- Monthly: $2.99
- Annual: $24.99 (saves users $11.89/year)

The 29% discount is big enough to feel meaningful without killing your revenue per paying user.

The most important screen in your app is the onboarding. Rork Max will generate a UI walkthrough, but it won't convert.

Good onboarding follows this structure:

1. **Show the problem** — "Do you spend hours writing boilerplate SwiftUI?"
2. **Show the solution** — "Rork Max generates it in seconds"
3. **Give them a win** — Let them use a key feature for free, right now
4. **Ask for money** — Only after they've felt the value

## App Store Optimization (ASO): Getting Your First Visibility

ASO is the App Store equivalent of SEO. The algorithm factors in:

- **Keywords**: What people search for
- **Download velocity**: How fast your app gains downloads
- **Retention**: Do users come back?
- **Reviews**: Star rating and review sentiment

Focus on keywords first. Use [App Annie](https://www.data.ai/) (now acquired by Sensor Tower) to find keywords with decent search volume but low competition.

Good keywords for a Rork Max companion app:
- "Swift UI code generator" (medium volume, medium competition)
- "iOS app builder" (high volume, high competition)
- "Swift UI templates" (low-medium volume, low competition)

Then optimize your screenshots. Rork Max's UI is beautiful, but it's generic. Add text overlays:

**Screenshot 1**: "Code Generation in Seconds" with a big, bold sans-serif font
**Screenshot 2**: "AI-Powered Swift UI" with a code snippet visible
**Screenshot 3**: "From Design to Production" showing the workflow
**Screenshot 4**: "Free trial, then $2.99/month" (be transparent)

And write a compelling description:

> Rork Max generates production-ready SwiftUI code from your designs. No more boilerplate. No more busywork. Just real, working code—every time.
>
> Features:
> • AI-generated SwiftUI components
> • Unlimited design imports (with subscription)

>
> Used by solo developers and small teams shipping 50+ apps to the App Store.

## Retention: Keeping Your Paying Users

Acquisition is expensive. Retention is where you actually build a business.

Track churn weekly. In App Store Connect, check "Subscription Cancellation Rate." Anything above 35% monthly churn is a red flag. If you're seeing that, your product doesn't solve a problem deeply enough, or your onboarding isn't working.

To reduce churn:

1. **Send in-app messages** when users haven't opened the app in 3 days
2. **Add new features** every month—give users a reason to stay
3. **Offer win-back discounts** to users who cancel ("Come back for $1/month for 3 months")
4. **Survey canceling users**—ask why they left
