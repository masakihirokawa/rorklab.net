## The Problem Most App Developers Face

Many indie developers think AdMob revenue is fixed — a variable they can't control. But that's wrong.

Small UI tweaks can **2x to 3x AdMob revenue** without changing the app's core functionality.

This guide shows you how to systematically design, test, and optimize ad placements using Rork and Firebase.

## Why UI Design Controls AdMob Revenue — Real Data

### The AdMob Revenue Formula

```
Revenue = Impressions × CTR × CPM
```

Developers often obsess over CPM (cost per mille), which Google controls. **Focus instead on Impressions and CTR — these you can control.**

### Real Example: Before and After

**Before optimization:**
- DAU: 5,000
- Session length: 3 minutes
- Impressions per session: 1.2
- CTR: 3%
- CPM: 50 yen
- **Daily revenue: 900 yen**

**After UI optimization:**
- DAU: 5,000 (unchanged)
- Session length: 3.5 minutes (users stay longer)
- Impressions per session: 2.0 (better placement)
- CTR: 5% (eye-catching ad positions)
- CPM: 50 yen (unchanged)

**Impact: 2.8x revenue increase, zero code changes, only UI redesign.**

## Ad Placement Science — The Golden Positions

Data from thousands of apps shows which placements work best.

### Highest-performing position: Game Over Screen

When users see "Game Over," they're:
- Relaxed (game ended).
- Mentally waiting for something (menu to load, next action).
- Primed to click.

Showing a fullscreen interstitial here yields:
- **CTR: 8-12%** (vs. 3-5% on average).
- **User frustration: Low** (feels natural).

### Second-best: Pause / Menu Screens

Users have voluntarily paused. They're comfortable waiting. A banner ad above "Resume" button performs well.

### Avoid: In-game interruptions

Ads during active gameplay = destroyed UX = 1-star reviews = uninstalls.

## Ad Format Selection — Banner vs. Interstitial vs. Rewarded

| Format | CTR | CPM | Best Placement | User Churn |
|---|---|---|---|---|
| Banner | 1-2% | 30-70 yen | Top/bottom fixed | Low |
| Interstitial | 8-12% | 100-200 yen | Game over | Medium |
| Rewarded | 20-40% | 30-50 yen | User choice | Low |

### Recommended layout: All three

## A/B Testing Ad Placements with Firebase

Never assume a layout is optimal. Test it.

### Setup: Firebase A/B Testing

1. Go to Firebase Console → A/B Testing
2. Create test "ad_placement_variant_b"
3. Duration: 14 days
4. Target: 50% of users
5. Define success metrics (ARPU target)

### Using Remote Config for Layout Variants

Firebase Remote Config delivers different layouts to different users:

```json
// Variant A (Control)
{
  "banner_position": "bottom",
  "interstitial_frequency": 1.0,
  "show_reward_button": true
}

// Variant B (Test)
{
  "banner_position": "top",
  "interstitial_frequency": 0.5,
  "show_reward_button": true
}
```

In your app code:

```swift
let config = RemoteConfig.remoteConfig()
config.fetch { status, error in
  config.activate()
  
  let position = config.configValue(forKey: "banner_position").stringValue
  if position == "top" {
    placeBannerAtTop()
  } else {
    placeBannerAtBottom()
  }
}
```

### Key Metrics to Track

| Metric | Definition | Target |
|---|---|---|
| **ARPU** | Revenue per user per day | Variant B +15% vs. Variant A |
| **CTR** | Ad click rate | Variant B +20% vs. Variant A |
| **Session Length** | Time per session | Variant B ≥ Variant A (no decrease) |
| **Churn** | Uninstall rate | Variant B ≤ Variant A (no increase) |

### Reading Results

After 14 days:
- **ARPU**: A = 1,500 yen, B = 1,800 yen → **B wins (+20%)**
- **CTR**: A = 4%, B = 5% → **B wins**
- **Session**: A = 4 min, B = 3.9 min → **tie** (minimal impact)
- **Churn**: A = 2.5%, B = 2.4% → **B wins (lower is better)**

**Decision**: Roll out Variant B to 100%.

**Revenue impact**: Monthly ARPU up 20% = 20% more monthly revenue.

## Continuous Optimization Cycle

Run one test per month. Compound improvements across the year.

### Monthly Schedule

**Week 1: Analyze & Hypothesize**
- Review Firebase and AdMob data.
- Identify bottleneck (low CTR? Short session? High churn?).
- Hypothesis: "Moving banner to top will increase CTR from 4% to 5%."

**Week 2: Design & Setup**
- Use Rork to generate new UI variant.
- Define Remote Config variables.
- Build and upload to TestFlight or internal testing.

**Week 3: Run Test**
- Release Variant B to 50% of users.
- Monitor for crashes or bugs.

**Week 4: Analyze & Rollout**
- Compare ARPU, CTR, session length, churn.
- If Variant B wins, roll out to 100%.
- Document results.

## Reading Firebase Analytics + AdMob Data

### Critical Metrics

**ARPU (Average Revenue Per User)**

Formula: Total revenue ÷ Number of users.

Most important single metric.

Example:
```
AdMob (last 30 days):
- Estimated revenue: 150,000 yen
- Ad requests: 500,000
- Valid impressions: 450,000
- Clicks: 22,500
- CTR: 5%
```

If DAU = 5,000:
```
ARPU = 150,000 yen ÷ (5,000 users × 30 days)
     = 1 yen per user per day
```

**CTR (Click-Through Rate)**

Percentage of ads shown that are clicked.

- CTR < 2%: Ad placement is poor. Change positions.
- CTR 2-5%: Average. Room to improve.
- CTR > 5%: Excellent. Don't change.

**CPM (Cost Per Mille)**

Revenue per 1,000 impressions.

Rises when: Ad format matches user intent better.
Falls when: Impressions shift to low-value regions or user demographics change.

**Session Length**

Time users spend per session.

If session length drops 50% after a UI change = ads are intrusive. Reduce ad frequency or change placement.

### Decision Tree

```
1. ARPU down 5%+ this month?
   → Check CPM. If CPM stable, check CTR.
   → If CTR down: ad placement changed? Fix it.
   → If CTR stable: fewer impressions? Check session length.

2. CTR down?
   → Ad placement or format changed?
   → Users ad-fatigued (too many ads)?
   → A/B test: reduce frequency or move placement.

3. Session length down 20%+?
   → Ads are interrupting gameplay.
   → A/B test: fewer interstitials, or move to better moments.

4. CPM down 50%?
   → Geographic shift (traffic from low-CPM countries)?
   → Check AdMob → Performance → Region.
   → If yes, consider region-specific ad formats.
```
