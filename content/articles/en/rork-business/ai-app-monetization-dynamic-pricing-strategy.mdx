Monetization is the eternal challenge for indie developers. Ad revenue alone doesn't provide stable monthly income, and subscriptions bring their own headaches with high churn rates.

Having run apps with AdMob as the primary revenue source for years, I always felt uneasy about earnings fluctuating with ad unit prices. That's what led me to revenue optimization based on user behavior data — analyzing usage patterns to present the right offer at the right time, building a revenue structure that doesn't depend entirely on advertising.

This article walks through monetization techniques implementable at the indie developer scale, with concrete code examples.

## Implementing User Segmentation

The first step in monetization is classifying users by behavior patterns. Showing the same offer to everyone is like casting a net over the entire ocean. Targeting gets you much better results.

```typescript
interface UserBehavior {
  userId: string;
  sessionsLast30Days: number;
  avgSessionDurationMin: number;
  featuresUsed: string[];
  daysSinceInstall: number;
  totalSpent: number;
  lastActiveDate: Date;
}

type UserSegment = "power_user" | "engaged" | "casual" | "at_risk" | "dormant";

function classifyUser(behavior: UserBehavior): UserSegment {
  const daysSinceActive = Math.floor(
    (Date.now() - behavior.lastActiveDate.getTime()) / (1000 * 60 * 60 * 24)
  );

  // No activity for 14+ days → dormant
  if (daysSinceActive > 14) return "dormant";

  // No activity for 7+ days with declining sessions → at risk
  if (daysSinceActive > 7 && behavior.sessionsLast30Days < 5) return "at_risk";

  // High frequency + long sessions + many features → power user
  if (
    behavior.sessionsLast30Days >= 20 &&
    behavior.avgSessionDurationMin >= 10 &&
    behavior.featuresUsed.length >= 5
  ) {
    return "power_user";
  }

  // 3+ times per week → engaged
  if (behavior.sessionsLast30Days >= 12) return "engaged";

  return "casual";
}
```

This segmentation is simple, but the insights it yields are significant. When I analyzed my own app data, power users were just 8% of the user base but accounted for 62% of paid revenue. Meanwhile, the vast majority of casual users contributed only ad revenue. Understanding this imbalance is where strategy begins.

## Segment-Based Offer Optimization

Once you know the user segment, you can design the optimal offer for each.

```typescript
interface Offer {
  type: "subscription" | "one_time" | "discount" | "feature_unlock";
  productId: string;
  price: number;
  message: string;
  displayTrigger: string;
}

function getOptimalOffer(segment: UserSegment, behavior: UserBehavior): Offer | null {
  switch (segment) {
    case "power_user":
      // Offer annual plan (looks cheaper per month)
      if (behavior.totalSpent === 0) {
        return {
          type: "subscription",
          productId: "annual_pro",
          price: 48, // ~$4/month vs $5/month regular
          message: "Thanks for being a loyal user. Get the annual plan for just $4/month",
          displayTrigger: "session_start",
        };
      }
      return null; // Already paying — don't show

    case "engaged":
      // Offer trial-included monthly subscription
      return {
        type: "subscription",
        productId: "monthly_pro_trial",
        price: 5,
        message: "Try all Pro features free for 7 days",
        displayTrigger: "feature_gate", // When they hit a paid feature
      };

    case "at_risk":
      // Win-back with limited discount
      return {
        type: "discount",
        productId: "monthly_pro_discount",
        price: 3, // 40% off regular $5
        message: "We've missed you. Get Pro at a special price (this week only)",
        displayTrigger: "app_open",
      };

    case "casual":
      // One-time purchase for casual users
      return {
        type: "one_time",
        productId: "premium_feature_pack",
        price: 2,
        message: "Unlock your favorite features permanently",
        displayTrigger: "after_3_sessions",
      };

    case "dormant":
      // Handle via push notifications, not in-app
      return null;
  }
}
```

Pay attention to `displayTrigger`. The timing of when you show an offer makes or breaks monetization. Power users see it at session start (when satisfaction is highest), engaged users see it when they access a paid feature (when motivation to pay peaks).

A common mistake is showing every user a popup right after app launch. This damages user experience and directly leads to lower review ratings.

## Building a Churn Prediction Model

If you can intervene before a user churns, you can significantly reduce cancellation rates. You don't need a full machine learning model — rule-based scoring delivers solid accuracy.

```typescript
interface ChurnSignals {
  sessionFrequencyDecline: number;  // Month-over-month session decline rate (0-1)
  featureUsageBreadth: number;      // Breadth of features used (0-1)
  supportTickets: number;           // Support tickets in last 30 days
  failedPayments: number;           // Failed payment attempts
  daysSinceLastPurchase: number;
}

function calculateChurnRisk(signals: ChurnSignals): {
  score: number;      // 0-100 (higher = more at risk)
  topFactor: string;
  recommendation: string;
} {
  let score = 0;
  const factors: { name: string; contribution: number }[] = [];

  // Session frequency decline is the strongest churn signal
  const freqScore = signals.sessionFrequencyDecline * 40;
  score += freqScore;
  factors.push({ name: "session_decline", contribution: freqScore });

  // Low feature usage → may not be finding value
  const breadthScore = (1 - signals.featureUsageBreadth) * 25;
  score += breadthScore;
  factors.push({ name: "low_feature_usage", contribution: breadthScore });

  // High support tickets → frustrated user
  const supportScore = Math.min(signals.supportTickets * 8, 20);
  score += supportScore;
  factors.push({ name: "support_issues", contribution: supportScore });

  // Failed payments → possible intentional cancellation
  const paymentScore = signals.failedPayments * 10;
  score += paymentScore;
  factors.push({ name: "payment_failures", contribution: paymentScore });

  score = Math.min(score, 100);

  const topFactor = factors.sort((a, b) => b.contribution - a.contribution)[0];

  const recommendations: Record<string, string> = {
    session_decline: "Re-engage with personalized content recommendations",
    low_feature_usage: "Show onboarding guide for unused features",
    support_issues: "Proactive support outreach",
    payment_failures: "Send payment method update email",
  };

  return {
    score,
    topFactor: topFactor.name,
    recommendation: recommendations[topFactor.name] ?? "Monitor situation",
  };
}
```

Combining this model with automatic discount offers or personalized notifications for users scoring above 70 has reduced churn rates by 20-30% in practice.

The key is returning not just the score, but also `topFactor` (the primary driver) and `recommendation` (the suggested action). Users churn for different reasons, and sending everyone the same "discount coupon" is ineffective. Users with declining frequency get content recommendations; users with support frustrations get proactive outreach — automating cause-specific responses is what moves the needle.

## Validating Price Changes with A/B Testing

Changing prices is scary. Raise them and you might increase churn; lower them and margins shrink. A/B testing is what takes the fear out of the equation.

```typescript
interface PriceExperiment {
  id: string;
  variants: {
    name: string;
    price: number;
    weight: number; // Traffic allocation (0-1)
  }[];
  metrics: {
    conversions: Map<string, number>;
    revenue: Map<string, number>;
    sampleSize: Map<string, number>;
  };
}

function assignVariant(experiment: PriceExperiment, userId: string): string {
  // Stable assignment via user ID hash
  const hash = simpleHash(userId + experiment.id);
  const normalized = (hash % 1000) / 1000;

  let cumulative = 0;
  for (const variant of experiment.variants) {
    cumulative += variant.weight;
    if (normalized < cumulative) return variant.name;
  }
  return experiment.variants[0].name;
}

function isStatisticallySignificant(experiment: PriceExperiment): {
  significant: boolean;
  winner: string | null;
  confidence: number;
} {
  const variants = experiment.variants.map((v) => {
    const n = experiment.metrics.sampleSize.get(v.name) ?? 0;
    const conversions = experiment.metrics.conversions.get(v.name) ?? 0;
    const rate = n > 0 ? conversions / n : 0;
    return { name: v.name, n, conversions, rate };
  });

  if (variants.some((v) => v.n < 100)) {
    return { significant: false, winner: null, confidence: 0 };
  }

  // Simple Z-test for two variants
  const [a, b] = variants;
  const pooledRate = (a.conversions + b.conversions) / (a.n + b.n);
  const se = Math.sqrt(pooledRate * (1 - pooledRate) * (1 / a.n + 1 / b.n));

  if (se === 0) return { significant: false, winner: null, confidence: 0 };

  const zScore = Math.abs(a.rate - b.rate) / se;
  const confidence = zScore > 2.576 ? 0.99 : zScore > 1.96 ? 0.95 : zScore > 1.645 ? 0.9 : 0;

  return {
    significant: confidence >= 0.95,
    winner: confidence >= 0.95 ? (a.rate > b.rate ? a.name : b.name) : null,
    confidence,
  };
}
```

Hash-based variant assignment ensures the same user always sees the same price. Price changes mid-experiment erode user trust, making this consistency crucial.

Automating statistical significance testing means decisions are based on data, not gut feeling. This level of rigor is entirely achievable for indie developers.

Monetization isn't about extracting money from users. It's about proposing fair value at the right time. Segmentation determines "who," offer optimization determines "what," and churn prediction determines "when." When all three axes are driven by data, you can build a sustainable revenue structure that doesn't depend on ad revenue alone.
