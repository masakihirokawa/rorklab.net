## Understanding K-Factor and Viral Growth Theory

### What Is K-Factor?

K-Factor is borrowed from epidemiology, where it describes how many people one infected person will spread a disease to. In app growth, K-Factor measures how many new users one existing user brings in.

The formula is straightforward:

```
K = i × c

i = average number of invitations sent per user
c = conversion rate of invited users who install the app
```

For example, if each user sends 5 invitations on average and 30% of recipients install the app:

```
K = 5 × 0.30 = 1.5
```

When K > 1, your app is in a viral state. Every new user cohort brings in more users than the previous one. With K = 1.5, 100 users bring in 150 new users in the next cycle, those 150 bring in 225, and so on.

### Realistic K-Factor Benchmarks

Most apps operate with K < 1, which is still perfectly viable when combined with paid acquisition or SEO. But even small improvements in K dramatically reduce your long-term CAC.

Industry benchmarks to calibrate expectations:

- **Social and community apps**: K = 0.3–0.8
- **Fintech and savings apps**: K = 0.1–0.4
- **Games with social features**: K = 0.5–1.2
- **Productivity and work apps**: K = 0.05–0.2

To measure your own K-Factor accurately, you first need a referral tracking system in place—which is exactly what we'll build.

### Viral Cycle Time: The Overlooked Variable

Alongside K-Factor, viral cycle time—the average time between a user sending an invitation and the invited person installing the app—dramatically affects growth speed.

```
Shorter cycle time = faster compounding

Example:
- K = 1.5, cycle = 7 days → ~3x growth in a month
- K = 1.5, cycle = 1 day → thousands of times growth theoretically
```

Reducing cycle time means reducing friction between invitation and installation. That's where Branch.io's deferred deep links shine.

- **Deferred deep linking**: Users who tap a referral link, go through the App Store, install, and open the app are automatically routed to the right screen—with full referral data intact.

```typescript
// lib/branch.ts
import branch, { BranchParams } from 'react-native-branch';

// Call this in your root App component
export async function initBranch(): Promise<void> {
  branch.subscribe(({ error, params, uri }) => {
    if (error) {
      console.error('Branch error:', error);
      return;
    }

    if (params['+is_first_session']) {
      // New install triggered by a referral link
      handleReferralInstall(params);
    } else {
      // Existing user opened a deep link
      handleDeepLink(params);
    }
  });
}

async function handleReferralInstall(params: BranchParams): Promise<void> {
  const referrerId = params['referrer_id'] as string;
  const campaignId = params['~campaign'] as string;

  if (referrerId) {
    // Record the attribution server-side
    await recordReferralConversion({
      referrerId,
      campaignId,
      newUserId: getCurrentUserId(),
    });

    // Trigger reward for the referrer
    await triggerReferrerReward(referrerId);
  }
}
```

### Step 3: Generate Unique Referral Links Per User

Each user should get a unique link that carries their ID and campaign metadata:

```typescript
// hooks/useReferralLink.ts
import { useState, useEffect } from 'react';
import branch from 'react-native-branch';
import { useAuth } from './useAuth';

export function useReferralLink() {
  const { user } = useAuth();
  const [referralLink, setReferralLink] = useState<string>('');
  const [isLoading, setIsLoading] = useState(true);

  useEffect(() => {
    if (!user) return;
    generateReferralLink();
  }, [user]);

  async function generateReferralLink() {
    try {
      const buo = new branch.BranchUniversalObject('referral/' + user.id, {
        title: `${user.displayName} invited you`,
        contentDescription: 'Join and get a special welcome bonus!',
        contentImageUrl: user.avatarUrl,
      });

      const linkProperties = new branch.LinkProperties();
      linkProperties.feature = 'referral';
      linkProperties.campaign = 'user_referral_2026';
      linkProperties.addControlParam('referrer_id', user.id);
      linkProperties.addControlParam('referrer_name', user.displayName);
      linkProperties.addControlParam(
        '$desktop_url',
        'https://yourapp.com/download'
      );

      const { url } = await buo.generateShortUrl(linkProperties);
      setReferralLink(url);
    } catch (error) {
      console.error('Failed to generate referral link:', error);
    } finally {
      setIsLoading(false);
    }
  }

  return { referralLink, isLoading };
}
```

## Incentive Design: The Psychology of Rewards

### The Double-Sided Reward Principle

The most effective referral programs reward both the referrer and the referee. This is the model used by Uber, Airbnb, and Dropbox. When both parties benefit, the referrer has a genuine reason to share, and the new user has a compelling reason to act.

Reward options by app type:

**For referrers (existing users):**
- In-app currency or credits
- Extended access to premium features
- Cash back (especially effective for fintech apps)
- Exclusive badges or status (powerful in community apps)

**For referees (new users):**
- Welcome bonus credits
- Free trial of premium features
- Discount on first purchase

### Designing the Reward Trigger

When you grant the reward matters as much as what the reward is. Granting on install maximizes volume but invites fraud. Granting on first purchase ensures quality but reduces conversion.

```typescript
// types/referral.ts
export type RewardTrigger =
  | 'install'           // Immediate—high volume, higher fraud risk
  | 'registration'      // After account creation—balanced
  | 'first_purchase'    // After paying—highest quality users
  | 'milestone';        // After specific in-app action

// services/rewardService.ts
export async function grantReward(
  userId: string,
  trigger: RewardTrigger,
  referrerId?: string
): Promise<void> {
  const rewardConfig = await getRewardConfig(trigger);

  if (!rewardConfig.enabled) return;

  // Reward the new user
  await addUserCredits(userId, rewardConfig.refereeCredits);
  await sendPushNotification(userId, {
    title: 'Your welcome bonus is here! 🎁',
    body: `${rewardConfig.refereeCredits} credits have been added to your account`,
  });

  // Reward the referrer when conditions are met
  if (referrerId && rewardConfig.referrerCredits > 0) {
    await addUserCredits(referrerId, rewardConfig.referrerCredits);
    await sendPushNotification(referrerId, {
      title: 'Your referral earned a reward! 🎉',
      body: `Someone joined through your link. ${rewardConfig.referrerCredits} credits added!`,
    });
  }
}
```

### Fraud Prevention

Any referral program with valuable rewards will attract fraud attempts. The most common abuse pattern is creating multiple accounts to self-refer. Here are the essential safeguards:

```typescript
// services/fraudDetection.ts
export async function validateReferral(
  referrerId: string,
  newUserId: string,
  deviceInfo: DeviceInfo
): Promise<{ valid: boolean; reason?: string }> {

  // 1. Block the same device from being used for multiple accounts
  const existingAccounts = await getUsersByDevice(deviceInfo.deviceId);
  if (existingAccounts.length > 0) {
    return { valid: false, reason: 'device_already_used' };
  }

  // 2. Rate-limit registrations from the same IP
  const recentRegistrations = await getRecentRegistrationsByIp(
    deviceInfo.ipAddress,
    timeframe: '1h'
  );
  if (recentRegistrations > 3) {
    return { valid: false, reason: 'ip_rate_limit' };
  }

  // 3. Prevent self-referral
  if (referrerId === newUserId) {
    return { valid: false, reason: 'self_referral' };
  }

  // 4. Cap the number of successful referrals per user
  const referralCount = await getReferralCount(referrerId);
  if (referralCount >= MAX_REFERRALS_PER_USER) {
    return { valid: false, reason: 'referral_limit_reached' };
  }

  return { valid: true };
}
```

### What to Test

The variables most worth testing in a referral program:

- **Reward amount and type**: 100 credits vs. 200 credits vs. 7-day premium trial
- **Reward trigger**: on install vs. on registration vs. on first purchase
- **Message copy**: feature-focused vs. emotional vs. scarcity framing

## KPIs and Dashboard for Referral Programs

### The Metrics That Matter

Track these core indicators to understand referral program health:

- **Invitation rate**: % of active users who send at least one invitation
- **Invitation-to-install conversion rate**: % of invited users who install
- **K-Factor**: Calculated weekly from invitation rate and conversion rate
- **Referred user LTV**: Do referred users retain and spend more than organically acquired users?
- **Cost per referred user vs. CAC**: Is the referral program cheaper than paid acquisition?

### Managing Referral Data in Supabase

```sql
-- Referral tracking table
CREATE TABLE referrals (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  referrer_id UUID NOT NULL REFERENCES users(id),
  referee_id UUID REFERENCES users(id),
  referral_link_id TEXT NOT NULL,
  status TEXT NOT NULL DEFAULT 'pending',
  -- status: pending / converted / rewarded / fraudulent
  created_at TIMESTAMPTZ DEFAULT NOW(),
  converted_at TIMESTAMPTZ,
  rewarded_at TIMESTAMPTZ,
  reward_amount INTEGER DEFAULT 0,
  experiment_variant TEXT,
  device_fingerprint TEXT
);

-- K-Factor view (rolling 30 days)
CREATE VIEW referral_kfactor AS
SELECT
  DATE_TRUNC('week', created_at) AS week,
  COUNT(DISTINCT referrer_id) AS unique_referrers,
  COUNT(*) AS total_invitations,
  COUNT(CASE WHEN status = 'converted' THEN 1 END) AS conversions,
  ROUND(
    COUNT(CASE WHEN status = 'converted' THEN 1 END)::numeric /
    NULLIF(COUNT(DISTINCT referrer_id), 0),
    3
  ) AS k_factor
FROM referrals
WHERE created_at > NOW() - INTERVAL '30 days'
GROUP BY week
ORDER BY week DESC;
```

## Troubleshooting Common Issues

### Branch SDK Initialization Timing Problems

**Problem**: Deep link data isn't available when the app launches from an invitation

**Cause**: Branch SDK subscribes asynchronously, and if the component tree renders first, the initial params may be missed

**Solution**:

```typescript
// App.tsx
import { useEffect } from 'react';
import { initBranch } from './lib/branch';

export default function App() {
  useEffect(() => {
    // Initialize Branch immediately on mount
    initBranch();
  }, []);

  // ...
}
```

### Deferred Deep Links Not Carrying Referral Data

**Problem**: After installing from the App Store via a referral link, the app opens but referrer data is missing

**Check these first**:

- **iOS**: Verify Branch keys are configured in `Info.plist`
- **Android**: Confirm `INTERNET` permission and Branch `<intent-filter>` are in `AndroidManifest.xml`
- **Session timeout**: The default 3-second timeout for Branch session initialization may be too short on slow connections—consider increasing to 5 seconds

### Duplicate Reward Grants

**Problem**: Unstable network conditions cause the reward API to be called multiple times for the same referral

**Solution**: Use idempotency keys to prevent duplicate processing

```typescript
async function safeGrantReward(
  userId: string,
  referralId: string
): Promise<void> {
  const idempotencyKey = `reward_${userId}_${referralId}`;

  // Skip if already processed
  const existing = await redis.get(idempotencyKey);
  if (existing) {
    console.log('Reward already granted, skipping');
    return;
  }

  // Set the key with 24-hour TTL before processing
  await redis.set(idempotencyKey, '1', { ex: 86400 });
  await grantReward(userId, 'registration');
}
```

## FAQ

**Q: When is the right time to launch a referral program?**

A: Wait until you have product-market fit. If users don't yet understand or love your app's core value, referral invitations will lead to installs that immediately churn—wasting reward budget and potentially hurting your store rating. A practical threshold: when 30-day retention exceeds 20% and you're seeing organic word-of-mouth begin naturally, you're ready. That usually corresponds to around 1,000+ monthly active users.

**Q: How should I budget for referral rewards?**

A: Your reward cost per acquired user should stay below your organic LTV for that cohort. If your normal CPI is $5 and your referral program acquires users at $2 in reward cost (total rewards ÷ installs from referrals), you're saving 60% on acquisition. Referred users also tend to have higher LTV than ad-acquired users—they came in with a personal recommendation, which correlates with stronger trust and engagement.

**Q: Do I need a paid Branch.io plan as an indie developer?**

A: Branch.io's free plan supports up to 10,000 monthly active users, which is more than enough for early-stage apps. If you want to keep costs at zero, you can combine Expo Router's [deep linking implementation](/articles/rork-dev/rork-max-deep-linking-universal-links) with a lightweight custom attribution backend. The Branch paid tier becomes valuable when you need advanced analytics, fraud scoring, and enterprise support.

**Q: Can fraud prevention be completely foolproof?**

A: No system is completely fraud-proof, but combining device fingerprinting, IP rate limiting, and self-referral blocking catches the vast majority of simple fraud attempts. For higher-value rewards, consider integrating a dedicated fraud detection service like Kount or Sift Science. Start permissive, measure your fraud rate, and tighten controls incrementally—overly aggressive rules will block legitimate users and damage your referral conversion rate.

**Q: How long should I run before evaluating referral program performance?**

A: At minimum 2–3 months of consistent data. The first few weeks are noisy as the program gains awareness and your most enthusiastic users participate. Track weekly K-Factor, invitation rate, and 7-day and 30-day retention of referred users. For deeper context on user acquisition and retention strategy, see [the complete growth and user acquisition guide](/articles/rork-business/rork-app-growth-user-acquisition-retention-guide).
