I released my first wallpaper app back when smartphones were just becoming mainstream. In those days, the formula was simple: build something basic, slap AdMob on it, and let the downloads roll in. That world is gone.

Today the App Store is saturated with wallpaper apps. Low-quality submissions get bounced in review, and even well-built ones struggle to surface unless the business model and content operations are designed carefully. Rork dramatically lowers the implementation cost — but that cuts both ways. Your competitors can ship faster too.

The differentiator now is business design: how you structure monetization, how you deliver content, and how you keep users coming back week after week. This guide covers all three, with working code you can adapt directly in your own Rork project.

## Choosing the Right Business Model

There are three main monetization approaches for wallpaper apps. The right choice depends on how frequently you plan to update content and how tolerant your target audience is to paywalls.

### Freemium + Premium Unlock

You offer 50–100 free wallpapers and require a subscription or one-time purchase to access the rest. This is the most common approach for indie developers, and for good reason — it balances accessibility with revenue potential.

The counterintuitive lesson here: don't be stingy with your free tier. If the free content feels low-effort, users assume the paid content is more of the same. Put your best work in the free tier. The implicit message — "if this is free, imagine what's behind the paywall" — converts far better than teasing users with mediocre samples.

### Fully Free + Rewarded Ads

All wallpapers are free, but downloading requires watching an AdMob rewarded ad. There's no paywall friction, which keeps download rates high. This model works well when you're targeting high-volume, casual users who won't pay but will watch a short video.

The tradeoff is that revenue per user is low, so you need significant scale to hit meaningful numbers.

### One-Time Purchase (Unlock All)

A single $0.99–$4.99 purchase unlocks everything. Simple, transparent, and users appreciate not being charged repeatedly. The downside is obvious: no recurring revenue. This model makes sense for smaller, curated collections — think photo books rather than ongoing services.

**My recommendation is freemium + AdMob rewarded ads in combination.** Free users get the "watch an ad to download" option, and premium subscribers get ad-free downloads. This design maximizes lifetime value because it monetizes both paying users and non-paying ones, while giving non-payers a path to engage without hitting a hard wall.

## Project Architecture in Rork

### Technology Stack

Here's what a production-grade wallpaper app needs:

- **Content storage**: Supabase Storage (CDN-backed image delivery)
- **Database**: Supabase (categories, tags, favorites, scheduled publishing)
- **Payments**: RevenueCat (subscription and/or one-time purchase)
- **Ads**: Google AdMob (rewarded ads for downloads, banner ads for free users)
- **State management**: Zustand or Jotai (download state, favorites)

When working with Rork, resist the urge to prompt for everything at once. Break your build into stages:

```bash
# Stage 1 — UI structure with mock data
"""
Create a wallpaper gallery app with:
- Home screen: category tabs (Nature, City, Abstract, Minimal)
- Grid view: 2-column layout with thumbnail images
- Detail screen: fullscreen image, download button, favorite button
Use mock data for now. Tech: React Native + Expo
"""

# Stage 2 — Wire up Supabase (after Stage 1 is working)
"""
Update the wallpaper grid to fetch from a Supabase `wallpapers` table.
Use SWR-style caching with useQuery.
"""

# Stage 3 — Add premium gating (after Stage 2 is working)
"""
Using RevenueCat, restrict downloads of wallpapers where
isPremium = true to users with an active "premium" entitlement.
Show a paywall screen for non-subscribers.
"""
```

Keeping the app in a working state between stages prevents Rork from generating half-finished implementations. Each prompt builds on verified working code.

### Supabase Database Design

```sql
-- Core wallpaper table

create table wallpapers (
  id uuid default gen_random_uuid() primary key,
  title text not null,
  category text not null check (
    category in ('nature', 'city', 'abstract', 'minimal')
  ),
  tags text[] default '{}',
  -- Separate thumbnail and full-res URLs to optimize bandwidth
  thumbnail_url text not null,   -- Low-res preview (400×700px)
  full_url text not null,        -- Full-res download (1080×1920px+)
  is_premium boolean default false,
  is_active boolean default true,
  published_at timestamptz default now(),  -- Enables scheduled publishing
  download_count integer default 0,
  created_at timestamptz default now()
);

-- RLS: public reads, no auth required
alter table wallpapers enable row level security;

create policy "Public wallpapers readable by anyone"
  on wallpapers for select
  using (is_active = true and published_at <= now());

-- Safe download count increment
create or replace function increment_download_count(wallpaper_id uuid)
returns void as $$
  update wallpapers
  set download_count = download_count + 1
  where id = wallpaper_id;
$$ language sql security definer;
```

The `published_at` column is worth calling out. By setting future dates when you upload content, you can batch-upload a week's worth of wallpapers on a single afternoon and have them drip-release automatically. No cron jobs, no manual publishing — the RLS policy handles the schedule.

## Content Management with Supabase Storage

Bundling images directly in your app binary is a common mistake that compounds over time. Every content update requires a new App Store submission, and review times are unpredictable. Hosting images in Supabase Storage means you can add or remove wallpapers without touching your app code at all.

### Storage Bucket Setup

```sql
-- Thumbnail bucket (enable CDN caching in Supabase dashboard)
insert into storage.buckets (id, name, public, file_size_limit, allowed_mime_types)
values (
  'wallpaper-thumbnails',
  'wallpaper-thumbnails',
  true,
  2097152,   -- 2 MB limit
  array['image/jpeg', 'image/webp']
);

-- Full-resolution bucket
insert into storage.buckets (id, name, public, file_size_limit, allowed_mime_types)
values (
  'wallpaper-full',
  'wallpaper-full',
  true,
  20971520,  -- 20 MB limit
  array['image/jpeg', 'image/webp', 'image/png']
);

-- RLS: anyone can read public images
create policy "Public read for thumbnails"
  on storage.objects for select
  using (bucket_id = 'wallpaper-thumbnails');

create policy "Public read for full images"
  on storage.objects for select
  using (bucket_id = 'wallpaper-full');
```

**Image format recommendation**: prefer WebP over JPEG. WebP delivers 20–30% smaller file sizes at equivalent visual quality, which translates directly to faster load times and lower Supabase bandwidth costs. Most modern iOS and Android versions support WebP natively, and Expo's `<Image>` component handles it without any extra configuration.

### Download Hook with Progress Tracking

The download implementation below handles permissions, album organization, and cleanup. Don't skip the album step — it makes a real difference to users who want to find their saved wallpapers later.

```tsx
// hooks/useWallpaperDownload.ts

import * as MediaLibrary from 'expo-media-library';
import * as FileSystem from 'expo-file-system';
import { useCallback, useState } from 'react';

interface DownloadResult {
  success: boolean;
  error?: string;
}

export function useWallpaperDownload() {
  const [isDownloading, setIsDownloading] = useState(false);
  const [progress, setProgress] = useState(0);  // 0–100

  const downloadWallpaper = useCallback(async (
    wallpaperId: string,
    fullUrl: string,
  ): Promise<DownloadResult> => {
    setIsDownloading(true);
    setProgress(0);

    const tempUri = `${FileSystem.cacheDirectory}wallpaper_${wallpaperId}.jpg`;

    try {
      // 1. Request media library permission
      const { status } = await MediaLibrary.requestPermissionsAsync();
      if (status \!== 'granted') {
        return {
          success: false,
          error: 'Photo library access is required. Please allow access in Settings › Privacy › Photos.',
        };
      }

      // 2. Download full-resolution image to cache
      const downloadResumable = FileSystem.createDownloadResumable(
        fullUrl,
        tempUri,
        {},
        ({ totalBytesWritten, totalBytesExpectedToWrite }) => {
          setProgress(Math.round((totalBytesWritten / totalBytesExpectedToWrite) * 100));
        },
      );

      const result = await downloadResumable.downloadAsync();
      if (\!result?.uri) throw new Error('Download failed: no file URI returned');

      // 3. Save to camera roll
      const asset = await MediaLibrary.saveToLibraryAsync(result.uri);

      // 4. Organize into "Wallpapers" album (create if it doesn't exist)
      //    ⚠️ Skipping this step leaves images scattered in the camera roll
      const album = await MediaLibrary.getAlbumAsync('Wallpapers');
      if (\!album) {
        await MediaLibrary.createAlbumAsync('Wallpapers', asset, false);
      } else {
        await MediaLibrary.addAssetsToAlbumAsync([asset], album, false);
      }

      // 5. Increment download count in background (non-blocking)
      supabase
        .rpc('increment_download_count', { wallpaper_id: wallpaperId })
        .catch((e: unknown) => console.warn('increment_download_count failed:', e));

      return { success: true };

    } catch (error) {
      return {
        success: false,
        error: error instanceof Error
          ? error.message
          : 'Download failed. Please try again in a moment.',
      };
    } finally {
      setIsDownloading(false);
      setProgress(0);
      // Clean up temp file
      FileSystem.deleteAsync(tempUri, { idempotent: true }).catch(() => {});
    }
  }, []);

  return { downloadWallpaper, isDownloading, progress };
}
```

## In-App Purchases with RevenueCat

For foundational RevenueCat setup in Rork, see [the Rork RevenueCat in-app purchase guide](/articles/rork-dev/rork-revenuecat-inapp-purchase). Here's the wallpaper-specific premium status hook:

```tsx
// hooks/usePremiumStatus.ts

import Purchases, { CustomerInfo } from 'react-native-purchases';
import { useEffect, useState } from 'react';

interface PremiumStatus {
  isPremium: boolean;
  isLoading: boolean;
  expirationDate: Date | null;
}

export function usePremiumStatus(): PremiumStatus {
  const [isPremium, setIsPremium] = useState(false);
  const [isLoading, setIsLoading] = useState(true);
  const [expirationDate, setExpirationDate] = useState<Date | null>(null);

  const syncFromCustomerInfo = (info: CustomerInfo) => {
    // "premium" must match the Entitlement ID you configured in RevenueCat
    const entitlement = info.entitlements.active['premium'];
    setIsPremium(\!\!entitlement);
    setExpirationDate(
      entitlement?.expirationDate ? new Date(entitlement.expirationDate) : null,
    );
  };

  useEffect(() => {
    Purchases.getCustomerInfo()
      .then(syncFromCustomerInfo)
      .catch((e) => console.warn('RevenueCat fetch failed:', e))
      .finally(() => setIsLoading(false));

    // Keep in sync after purchases or restores
    const listener = Purchases.addCustomerInfoUpdateListener(syncFromCustomerInfo);
    return () => { listener.remove(); };
  }, []);

  return { isPremium, isLoading, expirationDate };
}
```

In your detail screen, the gate becomes a one-liner:

```tsx
const { isPremium } = usePremiumStatus();

if (wallpaper.isPremium && \!isPremium) {
  return <PremiumPaywall onPurchase={handlePurchase} onRestore={handleRestore} />;
}
```

**Pricing advice for wallpaper apps**: start with two products — a monthly plan ($0.99–$1.99/month) and a one-time unlock ($3.99–$7.99). Add an annual plan priced at roughly 8× the monthly rate (so users perceive ~33% savings) once you have enough data to validate conversion rates.

## AdMob Rewarded Ads for Free Users

For the full AdMob setup, see [the Rork AdMob monetization guide](/articles/rork-business/rork-admob-ad-monetization-complete-guide). Here's a hook that wires rewarded ads into the download flow:

```tsx
// hooks/useDownloadWithRewardAd.ts

import {
  RewardedAd,
  RewardedAdEventType,
  TestIds,
} from 'react-native-google-mobile-ads';
import { useCallback, useEffect, useRef, useState } from 'react';
import { useWallpaperDownload } from './useWallpaperDownload';
import { usePremiumStatus } from './usePremiumStatus';

const AD_UNIT_ID = __DEV__
  ? TestIds.REWARDED
  : (process.env.EXPO_PUBLIC_ADMOB_REWARDED_ID ?? TestIds.REWARDED);

export function useDownloadWithRewardAd() {
  const { isPremium } = usePremiumStatus();
  const { downloadWallpaper, isDownloading, progress } = useWallpaperDownload();
  const pendingDownload = useRef<{ id: string; url: string } | null>(null);

  const [ad] = useState(() =>
    RewardedAd.createForAdRequest(AD_UNIT_ID, {
      requestNonPersonalizedAdsOnly: true,  // GDPR compliance
    }),
  );
  const [adLoaded, setAdLoaded] = useState(false);

  useEffect(() => {
    const unsubLoaded = ad.addAdEventListener(
      RewardedAdEventType.LOADED,
      () => setAdLoaded(true),
    );

    const unsubEarned = ad.addAdEventListener(
      RewardedAdEventType.EARNED_REWARD,
      () => {
        if (pendingDownload.current) {
          const { id, url } = pendingDownload.current;
          pendingDownload.current = null;
          downloadWallpaper(id, url);
        }
        ad.load();  // Preload next ad
      },
    );

    ad.load();
    return () => { unsubLoaded(); unsubEarned(); };
  }, [ad, downloadWallpaper]);

  const handleDownload = useCallback(async (
    wallpaperId: string,
    fullUrl: string,
  ) => {
    if (isPremium) {
      // Premium users: download immediately, no ad
      await downloadWallpaper(wallpaperId, fullUrl);
      return;
    }

    if (adLoaded) {
      pendingDownload.current = { id: wallpaperId, url: fullUrl };
      setAdLoaded(false);
      ad.show().catch(() => {
        // If the ad fails to display, still complete the download
        pendingDownload.current = null;
        downloadWallpaper(wallpaperId, fullUrl);
        ad.load();
      });
    } else {
      // Ad not ready — download anyway to avoid blocking the user
      await downloadWallpaper(wallpaperId, fullUrl);
      ad.load();
    }
  }, [ad, adLoaded, isPremium, downloadWallpaper]);

  return { handleDownload, isDownloading, progress };
}
```

**Critical pitfall**: `RewardedAd.show()` is not awaitable. You must listen for `EARNED_REWARD` to know when the user completed viewing the ad. If a user closes the ad early, `EARNED_REWARD` does not fire — so you need a deliberate policy on whether to allow the download anyway. I recommend allowing it. Forcing users to watch the full ad before downloading generates negative reviews that hurt your App Store ranking more than the lost ad revenue.

## App Store Review: Wallpaper-Specific Pitfalls

Wallpaper apps face a few review scenarios that general apps don't. Here are the ones I've run into personally.

### Copyright and License Documentation

Reviewers sometimes ask for proof that you're authorized to distribute the images. If you're using Unsplash or Pexels assets, keep the license documentation on hand. For AI-generated images (Midjourney, Stable Diffusion, etc.), check the commercial use terms of whichever tool you used — they vary significantly.

### Privacy Manifest Configuration

`expo-media-library` requires two distinct privacy strings in `app.json`. Vague descriptions get flagged.

```json
{
  "expo": {
    "ios": {
      "infoPlist": {
        "NSPhotoLibraryAddUsageDescription": "Used to save wallpapers to your photo library.",
        "NSPhotoLibraryUsageDescription": "Required to organize saved wallpapers into a dedicated 'Wallpapers' album."
      }
    }
  }
}
```

The difference between the two keys: `NSPhotoLibraryAddUsageDescription` covers write access (saving), while `NSPhotoLibraryUsageDescription` covers read access (listing albums to organize into). You need both.

### Content Policy

Wallpaper apps are reviewed for content as well as code. Clearly prohibited content includes:

- Adult or sexually suggestive imagery (even if rated 17+ in your metadata)
- Photos of real celebrities or athletes (right of publicity issues)
- Brand logos or trademarks (copyright)
- Photorealistic weapons or gore

Abstract art, landscapes, illustrations, and AI-generated imagery generally pass without issue — but keep your rating metadata accurate. Reviewers notice mismatches.

## Content Operations for the Long Term

The apps that hold their revenue over years are the ones with a reliable content cadence. "Set and forget" is a strategy that works for about three months, after which user retention drops and so do rankings.

### Automated Publish + Push Notification Flow

```typescript
// supabase/functions/notify-new-wallpapers/index.ts
// Runs on a weekly schedule via Supabase Edge Function cron

import { createClient } from '@supabase/supabase-js';

const supabase = createClient(
  Deno.env.get('SUPABASE_URL')\!,
  Deno.env.get('SUPABASE_SERVICE_ROLE_KEY')\!,
);

Deno.serve(async () => {
  const yesterday = new Date();
  yesterday.setDate(yesterday.getDate() - 1);

  const { data: newWallpapers } = await supabase
    .from('wallpapers')
    .select('id, title, thumbnail_url')
    .gte('published_at', yesterday.toISOString())
    .eq('is_active', true)
    .order('published_at', { ascending: false })
    .limit(5);

  if (\!newWallpapers?.length) {
    return new Response('No new wallpapers today', { status: 200 });
  }

  // Send push via OneSignal
  const res = await fetch('https://onesignal.com/api/v1/notifications', {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      'Authorization': `Basic ${Deno.env.get('ONESIGNAL_REST_API_KEY')}`,
    },
    body: JSON.stringify({
      app_id: Deno.env.get('ONESIGNAL_APP_ID'),
      included_segments: ['All'],
      contents: { en: 'New wallpapers are available\!', ja: '新しい壁紙が追加されました' },
      headings: { en: 'Fresh Wallpapers', ja: 'ウォールペーパー更新' },
      big_picture: newWallpapers[0].thumbnail_url,
    }),
  });

  return new Response(JSON.stringify(await res.json()), { status: 200 });
});
```

"New wallpapers available" notifications consistently outperform generic engagement pushes in open rate. Even weekly sends add up: in my experience, push notifications tied to content updates generate 15–25% of weekly active users returning specifically because of the notification.

### Sustainable Content Cadence

Here's the weekly workflow I run for my wallpaper apps:

- **Monday–Wednesday**: Select and process next week's wallpapers (WebP conversion, resize to 1080×1920px)
- **Thursday**: Upload to Supabase Storage, insert rows into `wallpapers` with `published_at` set to spread across the following week (3–5 per day)
- **Monday–Sunday next week**: Images publish automatically via RLS, no action needed
- **Monday**: Edge Function fires, OneSignal push goes out

Total time investment: 2–3 hours per week after the system is set up.



## Pricing Strategy and App Store Optimization

### Setting Up Your Paywall Screen

The paywall screen deserves as much attention as any other screen in your app. It's where conversions happen, and a generic "Subscribe now" screen leaves a lot of money on the table.

Three elements that make a meaningful difference in wallpaper app paywalls:

**Show the content, not just the price.** Display a grid of your best premium wallpapers behind a subtle blur or overlay. Users need to see what they're unlocking. Text descriptions of "exclusive content" don't convert; visible, beautiful wallpapers behind the paywall do.

**Lead with the annual plan.** Display the annual option first (and prominently), with the monthly option below it as a secondary choice. Frame the annual price as a daily cost ("less than $0.01 per day") rather than a yearly total. This framing consistently improves annual plan take rates, and annual subscribers have dramatically higher LTV than monthly.

**Include a restore button.** Users who reinstall your app or switch devices need to restore purchases. If they can't find this button, you'll get one-star reviews and support requests. Put it somewhere visible — a small "Restore Purchases" link at the bottom of the paywall is sufficient.

### App Store Optimization for Wallpaper Apps

The wallpaper category is competitive, and ranking in search requires deliberate keyword strategy. A few observations from running apps in this space:

The most discoverable wallpaper apps don't just use generic keywords like "wallpaper" or "backgrounds." They target style-specific terms ("minimalist wallpaper," "aesthetic wallpaper," "dark mode wallpaper") and device-specific terms ("iPhone 15 wallpaper," "iPad wallpaper"). These longer-tail terms have less competition and often convert better because users searching with specifics are more committed.

Your screenshots should show wallpapers applied to actual device screens rather than floating in isolation. Users are making a judgment about how the wallpaper will look on their phone — give them the most direct answer possible.

App preview videos (if you make one) work best when they show the download flow in real time: user browses grid, taps a wallpaper, watches a short ad, wallpaper saves to camera roll, appears as the device's home screen. This 15-second narrative demonstrates value more effectively than any static screenshot.

## Analyzing Performance: Metrics That Actually Matter

Most wallpaper app developers track download count and revenue. That's necessary but not sufficient. The metrics that predict whether your app will still be generating revenue a year from now are different.

**Day-7 retention** tells you whether users find ongoing value in your app. For a wallpaper app, a Day-7 retention rate above 20% suggests your content cadence and notification strategy are working. Below 10% is a warning sign that users are treating your app as a one-time download.

**Downloads per session** measures how engaged active users are. If users open your app but rarely download anything, the issue might be content selection (the wallpapers aren't compelling enough), UX friction (the download flow is too many taps), or paywall placement (users hit the paywall too quickly and bounce).

**Rewarded ad completion rate** tells you whether your free users find value in the watch-to-download mechanic. Completion rates above 70% indicate users feel the exchange is fair. Below 50% suggests the ad experience is too disruptive relative to the perceived value of the wallpaper.

You can pull all of these from App Store Connect analytics (for iOS) or Firebase Analytics (for cross-platform). The important thing is to check them weekly, not monthly — wallpaper apps can lose momentum fast if a content drop underperforms and you don't notice until a month later.

## Common Implementation Mistakes and How to Fix Them

After running multiple wallpaper apps and looking at what went wrong early on, here are the mistakes most likely to cost you time or money.

**Mistake 1: Loading full-resolution images in the grid view**

This is the most common performance issue in wallpaper apps. The grid view should always use thumbnail URLs (small, fast-loading previews). The full-resolution image should only load when the user taps to view the detail screen or initiates a download. Loading full-res images in the grid causes scroll jank and burns through user data plans.

The fix is structural, not a patch: maintain separate `thumbnail_url` and `full_url` fields in your database from day one.

**Mistake 2: Not handling the `EARNED_REWARD` event correctly**

A surprising number of implementations call `RewardedAd.show()` and then proceed with the download immediately in the same function, regardless of whether the user actually watched the ad. This happens because developers assume `show()` is synchronous or awaitable.

The correct pattern: store the pending download in a ref, listen for `EARNED_REWARD`, and trigger the download only inside that listener. The hook earlier in this article implements this correctly — use it as a reference.

**Mistake 3: Not preloading the rewarded ad**

If you only call `ad.load()` when the user taps the download button, there will always be a delay before the ad can show. Users experience this as the app "hanging." Load the ad as soon as the detail screen mounts (or even earlier), and reload it after each `EARNED_REWARD` event. The goal is to always have an ad ready before the user asks for it.

**Mistake 4: Overloading the free tier with ads**

There's a temptation to add banner ads, interstitial ads, and rewarded ads all at once to maximize revenue. In practice, showing ads on multiple surfaces in a wallpaper app damages the experience enough to hurt your App Store rating and word-of-mouth. I've had better results keeping ads to a single placement (the rewarded download) and letting the quality of the content do the conversion work.

**Mistake 5: Setting `requestNonPersonalizedAdsOnly` to false without proper consent flow**

If you're distributing your app in the EU or UK, you need a compliant consent flow before serving personalized ads. Google's UMP (User Messaging Platform) SDK handles this. Skipping it exposes you to GDPR risk and can result in your AdMob account being flagged. The setup is about half a day of work and worth doing before your first international release.

```tsx
// Initialize UMP SDK at app startup (App.tsx or _layout.tsx)
import {
  AdsConsent,
  AdsConsentStatus,
} from 'react-native-google-mobile-ads';

async function initializeAdConsent() {
  const consentInfo = await AdsConsent.requestInfoUpdate();
  
  if (
    consentInfo.isConsentFormAvailable &&
    consentInfo.status === AdsConsentStatus.REQUIRED
  ) {
    // Show the consent form — this is mandatory for EU/UK users
    await AdsConsent.showForm();
  }
}
```

Call `initializeAdConsent()` once when the app starts, before initializing any ad units. The UMP SDK caches the result, so subsequent launches skip the form for users who have already consented.

## What to Build First

If you're starting fresh, build the Supabase Storage integration before anything else. Bundling images in the app binary feels convenient on day one, but every content refresh will require a new review submission. That friction compounds.

Once your images live in Supabase Storage and your `published_at` schedule is working, you have the foundation for everything else in this guide — premium gating, rewarded ads, push notifications. Each layer builds cleanly on top.

Start there, get it working, and the rest follows naturally.
