The biggest fork in the road after shipping an app isn't technical — it's deciding how to make money from it.

I've been building apps since 2014, and if there's one lesson that stands out, it's this: there's no single right answer. But there are *very wrong* answers, and I've made most of them.

In this guide, I'll walk you through three monetization models for Rork-built apps — ads, subscriptions, and one-time purchases — with realistic revenue simulations at 5,000 MAU and genre-specific recommendations. By the end, you'll know which model makes sense for your app.

## Three Models, Three Different Futures

Let me lay out the fundamentals first.

**Ad-based** monetization is straightforward: your users enjoy the app free, you show them ads, both of you move on. It scales easily. Even at 5,000 MAU, you can reasonably expect a few thousand dollars a month. The catch? Users tolerate ads, they don't love them. Poor ad placement kills retention.

**Subscription** monetization asks users to pay regularly for ongoing access or features. The profit margin is higher, revenue is predictable, and LTV (lifetime value) balloons. The price you pay for this is complexity: you must constantly deliver value, or your churn rate (cancellation rate) becomes your enemy.

**One-time purchase** monetization is the simplest to implement. Set a price tier in the App Store, let users buy once, and move on. The downside: you're forever chasing new users. Today's revenue doesn't compound.

Each model suits different app types. Puzzle games and news apps lean toward ads. Productivity tools almost always thrive on subscriptions. Niche utilities can survive on one-time purchases alone.

## Model 1: Ad Revenue — The Reality of eCPM

The two numbers that matter in ad monetization are **eCPM** (earnings per thousand impressions) and **fill rate** (what percentage of ad slots actually get filled).

In Japan, eCPM typically sits between $0.50 and $2.00, depending on category. Games trend higher; utilities trend lower. North America and Western Europe see $2.00–$8.00. Southeast Asia runs $0.20–$0.80. These are market averages from 2025–2026.

To maximize ad revenue, your job is to make ads feel like a fair trade, not an interruption.

### Reward Ads: The Carrot Approach

Reward ads work because users *choose* to watch them. In exchange for watching a 30-second ad, they get coins, gems, premium currency, or a free pass to premium features. CTR is high. eCPM is often 20–30% better than standard banners.

If you're building a casual game, drop a reward ad button on the score screen: "Watch an ad for +100 coins?" Most users will take it. Implement it like this:

```typescript
// React Native (Expo) reward ad implementation
import { RewardedAd, RewardedAdEventType } from 'google-mobile-ads-react-native';

const RewardAdButton = ({ onReward }) => {
  const rewardedInterstitial = RewardedAd.createForAdRequest(
    'ca-app-pub-3940256099942544/5224354917', // Test Ad Unit ID
    {
      requestNonPersonalizedAdsOnly: false,
    }
  );

  const handleShowReward = async () => {
    try {
      await rewardedInterstitial.load();
      
      rewardedInterstitial.addAdEventListener(
        RewardedAdEventType.EARNED_REWARD,
        () => {
          console.log('User earned reward');
          onReward(100); // Award +100 coins
        }
      );

      rewardedInterstitial.show();
    } catch (error) {
      console.error('Failed to load reward ad:', error);
    }
  };

  return (
    <TouchableOpacity
      style={styles.button}
      onPress={handleShowReward}
    >
      <Text>Watch ad for +100 coins</Text>
    </TouchableOpacity>
  );
};
```

For productivity apps, the same pattern works: "Watch an ad to unlock this feature for 24 hours" or "Use your free feature pack one more time."

### Interstitial Ads: Timing Is Everything

Interstitial ads (full-screen takeovers) have high eCPM but demand careful placement. Show them too often, and users abandon your app. Show them too rarely, and you leave money on the table.

A practical heuristic: don't show an interstitial to the same user more than once per 30–60 seconds, and only after they've completed a meaningful action (game round, file export, task completion). Space them out so they feel like natural pauses, not punishment.

```typescript
// Frequency capping for interstitials
class AdFrequencyManager {
  private lastAdTime: number = 0;
  private actionCount: number = 0;
  private minIntervalMs: number = 30000; // 30 seconds
  private minActionsBetweenAds: number = 2;

  canShowInterstitial(): boolean {
    const timeSinceLastAd = Date.now() - this.lastAdTime;
    return (
      timeSinceLastAd > this.minIntervalMs &&
      this.actionCount >= this.minActionsBetweenAds
    );
  }

  recordAction(): void {
    this.actionCount++;
  }

  recordAdShown(): void {
    this.lastAdTime = Date.now();
    this.actionCount = 0;
  }
}

// Usage
const adManager = new AdFrequencyManager();

const handleLevelComplete = async () => {
  if (adManager.canShowInterstitial()) {
    await showInterstitialAd();
    adManager.recordAdShown();
  }
  adManager.recordAction();
  navigateToResultsScreen();
};
```

### Waterfall Strategy: Doubling Your eCPM

Here's a power move: don't rely on AdMob alone. Run a **mediation waterfall** — a cascade of ad networks, each given a chance to fill the ad slot before passing to the next.

AdMob gets first shot. If it doesn't fill (or the bid is low), IronSource tries. If that fails, AppLovin gets a turn. By the end, you're pulling ads from whoever bid highest for that impression.

Done right, waterfalls lift eCPM by 50–100%.

```typescript
// Mediation network waterfall (pseudocode)
const mediationConfig = {
  networks: [
    { id: 'admob', priority: 1, timeout: 3000 },
    { id: 'ironsource', priority: 2, timeout: 3000 },
    { id: 'applovin', priority: 3, timeout: 3000 },
  ],
};

const selectBestAdNetwork = async (): Promise<AdNetwork> => {
  const results = await Promise.all(
    mediationConfig.networks.map(async (network) => ({
      network,
      bid: await fetchBid(network.id),
    }))
  );
  
  return results.reduce((best, current) =>
    current.bid > best.bid ? current : best
  ).network;
};
```

## Model 2: Subscriptions — Playing the Long Game

Subscriptions are where indie apps find stability. Instead of chasing new users endlessly, you build a recurring revenue stream from people who value your work enough to pay monthly or yearly.

The LTV (lifetime value) of a subscriber who stays for a year is 12x a one-time purchase at the same price point.

But here's the hard truth: people cancel subscriptions constantly. You need three things for a subscription to stick:

1. **Clear value** — The premium features aren't nice-to-have, they're need-to-have
2. **Regular updates** — At least monthly, users see something new
3. **Free trial** — Give them 3–7 days to experience the full feature set before charging

### Pricing: What the Market Actually Pays

In Japan, the tiers that work:

- **¥480/month** — Light productivity tool or simple utility
- **¥980/month** — Full-featured editor or note-taking app
- **¥1,980/month+** — Professional tools, educational content, or entertainment with fresh material weekly

In the US App Store:

- **$2.99/month** — Lightweight options
- **$4.99/month** — Standard tier
- **$9.99/month+** — Premium, specialized tools

One critical move: **always offer annual billing**. A user who pays ¥5,760 per year instead of ¥480/month is giving you 10x the margin on the same commitment. If just 20% of your subscribers choose annual, your overall revenue jumps 20%.

### StoreKit 2 Implementation

In React Native, [react-native-purchases](https://www.revenuecat.com/) (RevenueCat) has become the standard for subscription management. However, in Expo, RevenueCat's native module can be finicky. For maximum stability, consider EAS Build or direct Expo Modules integration:

```typescript
// Expo Modules StoreKit 2 (pseudocode)
import * as StoreKit from 'expo-store-kit';

const SubscriptionManager = {
  async startFreeTrial() {
    try {
      const product = await StoreKit.fetchProduct('premium_monthly');
      const result = await StoreKit.requestPurchase(product, {
        offerToken: product.introductoryPrice.offerToken,
      });
      return result;
    } catch (error) {
      console.error('Purchase failed:', error);
    }
  },

  async checkSubscriptionStatus() {
    const transactions = await StoreKit.getTransactions();
    const activeSubscription = transactions.find(
      (tx) => tx.isActive && tx.productId === 'premium_monthly'
    );
    return !!activeSubscription;
  },

  async restorePurchases() {
    await StoreKit.syncPurchases();
  },
};
```

### Converting Free Users to Paid

The conversion funnel is where subscriptions win or lose. Here's a flow that works:

1. User installs app → full access to free features
2. Day 3: They hit a premium feature → banner: "Start 7-day free trial"
3. Day 5: Push notification → "Your trial ends in 2 days. You're going to love this."
4. Day 7 morning: "Premium starts today. Cancel anytime."
5. If they cancel: "Try again in 3 days? First week is free."

The key: never surprise them with a charge. Apple and Google penalize deceptive dark patterns.

```typescript
// Trial-to-paid flow
const SubscriptionFlow = {
  async initiateFreeTrial(userId: string) {
    const trialEndDate = new Date();
    trialEndDate.setDate(trialEndDate.getDate() + 7);

    await saveUserData(userId, {
      trialStartedAt: new Date(),
      trialEndsAt: trialEndDate,
      isTrialActive: true,
    });

    // Schedule reminder at day 5
    scheduleNotification({
      date: new Date(trialEndDate.getTime() - 2 * 24 * 60 * 60 * 1000),
      title: 'How are you enjoying Premium?',
      body: 'Your trial ends in 2 days. Cancel anytime.',
    });

    return trialEndDate;
  },

  async handleTrialExpired(userId: string) {
    await StoreKit.requestPurchase(premiumProduct);
  },

  async handleTrialCancelled(userId: string) {
    // Re-offer in 3 days
    scheduleNotification({
      date: new Date(Date.now() + 3 * 24 * 60 * 60 * 1000),
      title: 'Give Premium another try?',
      body: 'First week is free.',
    });
  },
};
```

## Model 3: One-Time Purchase — The Simple Sell

Buy-once is the simplest model to implement. Set a price in the App Store, unlock features on first purchase, done.

The problem: scalability. Your revenue depends entirely on new user acquisition. Once you've sold to the active market, growth stops. Unlike ads (which scale with DAU) or subscriptions (which compound), buy-once is a flat line.

### When One-Time Purchase Works

Some categories thrive on this:

- **Utilities** — Barcode scanner, unit converter, PDF compressor. Single-use value, minimal updates.
- **Education** — Textbooks, courses, reference material. One price for complete access.
- **Games (completed)** — Single-player campaigns or puzzle games with defined endpoints. Not MMOs.
- **Design assets** — Brushes, fonts, templates (Procreate model).

### Price Elasticity: Finding Your Sweet Spot

Here's what real data shows:

| Price | Relative Downloads | Est. Monthly Revenue (5K MAU) |
|---|---|---|
| $1.99 | 100% | $300–$600 |
| $4.99 | 50% | $400–$800 |
| $9.99 | 20% | $300–$600 |
| $19.99 | 8% | $250–$400 |

Price elasticity is brutal. As price rises, downloads fall exponentially. Most utilities find their revenue sweet spot at **$2.99–$4.99**.

## Revenue Simulation: All Three Models at 5,000 MAU

Let's compare apples to apples. Same app, three different monetization strategies. Here are the assumptions:

- MAU: 5,000
- DAU (daily active): 3,000
- Ad impressions per DAU: 8
- eCPM: $1.50 (Japan-focused, games category)
- Subscription price: ¥480/month
- Subscription conversion: 5% (market average)
- Subscription churn: 8% monthly
- One-time purchase price: $3.99
- Purchase conversion: 3%

### Ad Model: Monthly Revenue

```
DAU: 3,000
Impressions per DAU: 8
Total impressions: 3,000 × 8 = 24,000

eCPM: $1.50
Monthly revenue: (24,000 / 1,000) × $1.50 = $36
USD to JPY (1:150): ¥5,400

After 30% platform fees: ¥3,780
```

Ads generate small, reliable income at scale. But per-user value is minimal.

### Subscription Model: Monthly Revenue

```
MAU: 5,000
Conversion rate: 5% = 250 new subscribers/month
Churn rate: 8% = ongoing cancellations

Steady state (month 2+):
Inflow: 250 new per month
Outflow: base × 8%
Equilibrium: 250 / 0.08 = 3,125 subscribers

Monthly revenue: 3,125 × ¥480 = ¥1,500,000
After 30% fees: ¥1,050,000
```

Subscriptions dominate. The difference is staggering: 1 million yen more per month than ads.

### One-Time Purchase Model: Monthly Revenue

```
MAU: 5,000
Conversion: 3% = 150 purchases/month
Price: $3.99

Revenue: 150 × $3.99 = $598.50 ≈ ¥90,000
After 30% fees: ¥63,000
```

One-time purchases outpace ads but pale next to subscriptions. And unlike subscriptions, this won't grow as MAU grows — only the new users buy.

## Hybrid Model: Ads + Subscriptions

In practice, you rarely pick one. Most successful indie apps combine them.

The pattern: free tier with ads + premium tier (ad-free + extra features).

This lets price-insensitive users stay free (you monetize via ads) while power users opt in to ad-free premium. It's the best of both worlds.

```typescript
// Hybrid monetization logic
export const FeatureAccess = {
  async canUseFeature(userId: string, featureId: string) {
    const user = await getUserData(userId);
    
    // Premium = everything
    if (user.isPremium) return true;
    
    // Free = basic features only
    const basicFeatures = ['core-editing', 'export-jpg'];
    return basicFeatures.includes(featureId);
  },

  async handlePremiumFeatureAccess(userId: string, featureId: string) {
    const user = await getUserData(userId);
    
    if (!user.isPremium) {
      showPremiumUpgradeSheet({
        feature: featureId,
        onProceed: () => SubscriptionManager.startFreeTrial(),
      });
      return false;
    }
    return true;
  },

  async shouldShowAds(userId: string): Promise<boolean> {
    const user = await getUserData(userId);
    return !user.isPremium && !user.adsRemoved;
  },
};
```

To boost conversion, use these tactics:

1. **Gradual lockout** — First few uses are free, then features gate
2. **Friction points** — "One more use requires premium"
3. **Value proof** — Show how premium saves time/money numerically

### State Management for Monetization

Use Zustand or Redux to track subscription and purchase state globally:

```typescript
// store.ts
import { create } from 'zustand';

interface MonetizationState {
  isPremium: boolean;
  adsRemoved: boolean;
  
  setPremiumStatus: (status: boolean) => void;
  setAdsRemoved: (removed: boolean) => void;
}

export const useMonetization = create<MonetizationState>((set) => ({
  isPremium: false,
  adsRemoved: false,
  
  setPremiumStatus: (status) => set({ isPremium: status }),
  setAdsRemoved: (removed) => set({ adsRemoved: removed }),
}));
```

## Genre-Specific Recommendations

| Genre | Best Model | Why |
|---|---|---|
| Casual Games | Ads + Remove Ads | Light play demands uninterrupted experience |
| Core Games | One-Time | Depth warrants single price |
| Productivity | Subscription | Daily use = high LTV |
| Photo Editing | Subscription | Regular filters/updates justify recurring fee |
| Utilities | One-Time | Limited, stable feature set |
| News/Content | Subscription | Fresh content is the product |
| Social | Ads | Scale first, monetize later |

## The Phased Approach: How I'd Do It

After 10+ years building apps, here's the path I'd recommend:

**Phase 1 (MAU < 1,000):** Ship with ads. Validate product-market fit.

**Phase 2 (MAU 1,000–5,000):** Add premium features behind a paywall. Measure conversion.

**Phase 3 (MAU > 5,000):** Shift to subscription-first, keep ads as fallback for non-premium users.

This progression keeps complexity low early and compounds revenue as you grow. Rork makes moving between these phases smooth — you're just toggling features and adding purchasing hooks.

Test, learn, adapt. Your "perfect" monetization model is waiting on the other side of real user data.
