# Rork App Rejected by App Store? Complete Guideline Violation Fixes

You've built your app with Rork. You've tested it thoroughly. You hit submit on App Store Connect with confidence. Then—rejection.

The good news: almost every rejection follows predictable patterns, and once you understand the pattern, the fix is straightforward.

The better news: this guide covers the six most common rejection reasons for Rork apps, exact how-to-fix instructions, and real appeal templates you can copy directly into App Store Connect.

## Guideline 4.3 — Spam & Duplicate Apps

### The Problem: Template Similarity

Because Rork uses templates, other Rork apps often share similar layouts, navigation patterns, and feature structures. Apple detects this and flags your app as a "duplicate" or "spam" app—essentially saying, "This looks like every other template-based app we've rejected this week."

Apple rejects under 4.3 when:
- UI/UX appears to be unmodified template design
- Feature set is indistinguishable from dozens of similar apps
- No clear brand identity or custom design elements exist
- The app feels like a copy-paste effort, not original work

### The Fix

**1. Completely customize your UI and branding**

Don't submit the default Rork template. Replace it with:

```javascript
// Change default colors completely
const appTheme = {
  primary: '#1F4788',      // Your brand color (not Rork default)
  secondary: '#FFA500',    // Your accent color
  background: '#F5F5F5'    // Your background
};

// Replace every instance of the Rork default color scheme
StyleSheet.create({
  container: {
    backgroundColor: appTheme.background,
  },
  header: {
    backgroundColor: appTheme.primary,
  },
  button: {
    backgroundColor: appTheme.secondary,
  }
});
```

- **Logo**: Custom-designed (not a generic icon)
- **Color palette**: 3-4 brand colors chosen specifically for your app
- **Typography**: Use custom Google Fonts instead of system defaults
- **Icons**: Consistent icon set (Material Design or custom)
- **Layout**: Modify the default Rork layout structure—if you started with bottom tabs, change to side navigation or top drawer

**2. Build features that don't exist in the Rork template**

Apple wants to see that you've added genuine value beyond the boilerplate. Add at least one of these:

- **AI-powered features**: Use Claude API to auto-summarize, categorize, or recommend content
- **Real-time multiplayer**: Team collaboration, live data sync across devices
- **Rich media**: Video, AR, health tracking integration (if relevant to your app)
- **Custom analytics**: Track meaningful user behavior and personalize experience
- **Advanced search**: Full-text search, filters, sorting not in the basic template

```javascript
// Example: Adding Claude API intelligence
async function analyzeUserInput(userText) {
  const response = await fetch('/api/claude-analyze', {
    method: 'POST',
    body: JSON.stringify({ text: userText })
  });
  // Claude assigns category, priority, or generates a summary
  // This is unique to your app—not part of the template
  return response.json();
}
```

**3. Write compelling App Store marketing**

Your description and keywords need to emphasize uniqueness:

```
❌ Generic: "A task management app built with Rork"

✓ Specific: "AI-powered task intelligence. This app uses machine
learning to automatically prioritize your tasks, estimate time, and
group related items. Real-time collaboration with teammates."
```

### Apple Appeal Template

When Apple rejects for "duplicate/spam," use this exact template in App Store Connect's "Resolution Center":

```
Dear Apple Review Team,

Thank you for reviewing [App Name] (Bundle ID: [bundle-id]).

We respectfully disagree with the rejection reason "Duplicate or
Spam App." We've addressed this feedback with the following
improvements:

**1. Unique Brand Identity**
- Custom logo designed by [Designer/Studio Name]
- Brand color palette: [Describe your specific colors and design]
- Complete UI redesign from default template

**2. Differentiated Features**
- [Feature 1]: AI-powered [functionality]—uses Claude API for intelligence
- [Feature 2]: Real-time [collaborative feature]—unique to this app
- [Feature 3]: [Custom functionality]—not found in template apps

**3. Quality & Completeness**
- 8+ fully-functional screens
- No placeholder content
- Comprehensive testing across iOS versions

**4. App Store Presence**
- Professional marketing materials
- Clear, detailed app description
- Screenshots demonstrating unique features in action

We've invested significant effort in making this a genuinely valuable app that goes far beyond template functionality. We respectfully request reconsideration and re-review.

Sincerely,
[Your Name]
```

## Guideline 2.1 — App Completeness

### The Problem: Unfinished Features & Placeholder Content

Apple's testers spend 5-15 minutes with your app. If they find buttons that don't work, unfinished screens, or features that go nowhere, they'll reject you for "incomplete app."

Common issues:
- Login screen exists but doesn't connect to a backend
- "Settings" button exists but does nothing
- Data doesn't save after user entry
- Missing required test account info for reviewers

### The Fix

**1. Test every single interaction, end-to-end**

Treat your app like Applebot will. Go through it systematically:

```javascript
// Create this checklist and verify every item
const appTestingChecklist = {
  screens: {
    splash: "✓ Loads in 2 seconds, auto-transitions",
    home: "✓ All buttons navigate or trigger actions",
    detail: "✓ Each item shows full details correctly",
    create: "✓ Form input saves to database",
    profile: "✓ User data displays and can be edited",
    settings: "✓ Every toggle/switch actually works"
  },
  navigation: {
    backButton: "✓ Returns to previous screen",
    tabs: "✓ All tabs switch content correctly",
    deepLinks: "✓ URLs open the correct screen"
  },
  features: {
    authentication: "✓ Login/signup works, logout too",
    dataStorage: "✓ Created items persist after app restart",
    networking: "✓ Images/content loads from URLs"
  }
};

// Don't submit until every item is ✓
```

**2. Remove all placeholder text and images**

The Rork template likely includes "Lorem ipsum" or generic images. Find and replace every instance:

```javascript
// ❌ BAD - Rork default
<Text>"Lorem ipsum dolor sit amet, consectetur adipiscing elit"</Text>
<Image source={require('./placeholder.png')} />

// ✓ GOOD - Real content
<Text>"Organize your tasks and boost productivity"</Text>
<Image source={require('./myLogo.png')} />
```

**3. Provide test account credentials**

Apple's testers need to access your app fully. In App Store Connect, under "App Review Information," include:

```
Test Account Information:
---
Email: testuser@example.com
Password: SecureTestPassword123!

Additional Notes:
- Account has sample data pre-populated
- Test user can create new items, edit, and delete
- All payment flows (if applicable) use sandbox Stripe keys
- App is fully functional without test credentials, but
  logged-in features show more content

Reviewer Instructions:
1. Tap "Sign Up" (or you can login with above credentials)
2. Tap the "Create New" button to add an item
3. Swipe left on any item to delete it
4. Tap "Settings" to see all configuration options
```

## Guideline 5.1.1 — Data Privacy & Collection

### The Problem: Missing or Vague Privacy Policy

This is the most strictly enforced guideline. If your Rork app collects any user data at all—emails, names, device info—Apple wants a crystal-clear privacy policy showing what you collect, why, and how you protect it.

### The Fix

**1. Create a comprehensive, publicly-hosted privacy policy**

Don't write a paragraph in App Store Connect. Create a full webpage and host it:

```markdown
# Privacy Policy for [App Name]
Last Updated: March 26, 2026

## What Data We Collect
- User email and name (when you sign up)
- Profile information (optional photo, bio)
- App usage data (which features you use, how often)
- Device information (device type, OS version, unique device ID)
- Location data (only if you grant permission)

## Why We Collect It
- Create and manage your account
- Improve app features and performance
- Send you updates and support messages
- Analyze usage patterns to make the app better

## Who We Share It With
- **Google Analytics**: To understand how users interact with the app
- **Firebase Crashlytics**: To fix bugs and stability issues
- **Amazon AWS**: Our cloud database provider
- **Stripe** (if applicable): For payment processing

See their privacy policies for details:
[Google Privacy](https://policies.google.com/privacy)
[AWS Privacy](https://aws.amazon.com/privacy/)
[Stripe Privacy](https://stripe.com/privacy)

## Your Rights
- Request to see all data we have about you
- Delete your account and all associated data
- Opt-out of analytics

Contact us at: privacy@[yourapp].com

## Data Security
- All data transmitted via encrypted HTTPS
- Database encrypted at rest
- Regular security audits
- Compliance with GDPR and CCPA
```

**2. Add the privacy policy URL to App Store Connect**

```
App Store Connect → Your App → App Information → Privacy
- Privacy Policy URL: https://yourapp.com/privacy
- Contact Email: privacy@yourapp.com
```

**3. Request permissions with clear, honest descriptions**

If your app needs camera, location, contacts, or health data, add these to Info.plist:

```xml
<!-- Info.plist -->

<!-- If requesting location -->
<key>NSLocationWhenInUseUsageDescription</key>
<string>We need your location to show nearby
recommendations on the map. Your location is only
used in-app and never shared with third parties.</string>

<!-- If requesting camera -->
<key>NSCameraUsageDescription</key>
<string>Take photos for your profile picture.
You can always change it later in Settings.</string>

<!-- If requesting contacts -->
<key>NSContactsUsageDescription</key>
<string>Find friends in your contacts and invite them
to collaborate. We don't store or share your contacts.</string>
```

**4. Disable any unnecessary data collection**

```javascript
// ❌ DON'T collect data you don't need
analytics.trackAllUserActions();  // Overkill

// ✓ DO collect only what you use
analytics.track('user_created_task');  // Specific, purposeful
```

## Guideline 4.0 — Design Quality

### The Problem: Bare-Bones Features, Poor UX

Apple evaluates not just whether your app works, but whether it feels polished and complete. A minimal template with 2-3 screens and basic navigation can feel unfinished.

### The Fix

**1. Build a complete feature set**

Minimum viable completeness:

```
Expected Screens (minimum 6-8):
✓ Splash/Launch Screen
✓ Home/Main Feed
✓ Detail/Single Item View
✓ Create/Edit Screen
✓ User Profile
✓ Settings
✓ Search/Browse
✓ Onboarding/Tutorial
```

**2. Polish performance**

Slow apps feel cheap. Optimize:

```javascript
// ❌ SLOW: Load 10,000 items at once
function loadAllPosts() {
  return fetch('/api/posts')  // Fetches every post ever
    .then(response => response.json())
    .then(data => renderAllPosts(data));  // App freezes for 3 seconds
}

// ✓ FAST: Pagination and lazy loading
function loadPosts(page = 1) {
  return fetch(`/api/posts?page=${page}&limit=20`)
    .then(response => response.json())
    .then(data => {
      renderPosts(data);
      // Next 20 load when user scrolls to bottom
    });
}
```

**3. Match modern iOS design standards**

- Use system fonts (SF Pro) unless you have a strong brand reason not to
- Implement safe area insets for notch/Dynamic Island
- Use SF Symbols for consistent icons
- Support dark mode
- Responsive layout (works on all iPhone sizes)

```javascript
// ✓ Support dark mode
import { useColorScheme } from 'react-native';

export default function App() {
  const colorScheme = useColorScheme();

  return (
    <View style={{
      backgroundColor: colorScheme === 'dark' ? '#000' : '#FFF',
      color: colorScheme === 'dark' ? '#FFF' : '#000'
    }}>
      {/* Content */}
    </View>
  );
}
```

## Guideline 2.3.1 — Hidden Features

### The Problem: Important Features Are Hard to Find

Apple's reviewers have limited time. If a key feature is buried three menus deep or hidden behind an unclear UI, reviewers won't find it—and won't rate your app fairly.

### The Fix

**1. Surface all major features in main navigation**

```
Good Navigation Structure:
- Bottom tab bar OR side drawer
- 4-5 main tabs/sections visible at all times
- Everything accessible within 2-3 taps
- No hidden menus or undiscoverable features

Bad Structure:
- Settings hidden in a tiny gear icon
- Create button only visible on home screen
- Export feature buried in a 5-level menu
```

**2. Create an App Preview video**

Submit a 15-30 second video demonstrating every major feature:

```markdown
# App Preview Script (15 seconds)

[0-2 sec] App opens, splash screen → home feed loads
[2-5 sec] User taps "Create New" button
[5-8 sec] User fills out form and saves
[8-10 sec] Item appears in feed, user taps to view details
[10-12 sec] User navigates to Profile screen
[12-14 sec] User navigates to Settings
[14-15 sec] Outro showing app name
```

Tools: Use Quicktime (Mac) to record simulator, then upload to App Store Connect.

## Final Resubmission Checklist

Before you hit "Submit for Review" again:

```markdown
# Pre-Submission Quality Checklist

Functionality:
□ Every button triggers an action
□ Every form input saves and persists after app restart
□ Login/authentication works completely (login + logout)
□ No console errors, warnings, or crashes
□ Tested on both iPhone and iPad

Content:
□ No "Lorem ipsum," "Test," or placeholder text
□ All images load correctly and aren't distorted
□ Privacy policy URL is live and accessible
□ No offensive, misleading, or copyrighted content

Apple Requirements:
□ Test account credentials provided in App Review Info
□ App description matches actual functionality
□ Screenshots are real app screens (not mocks or concept art)
□ Keywords accurately describe the app
□ At least 3 high-quality app screenshots

Design:
□ App works in both light and dark modes
□ Safe area respected (notch/Dynamic Island)
□ No stretched or distorted UI elements
□ Navigation is clear and intuitive
□ Consistent design language throughout

Performance:
□ App launches in under 3 seconds
□ No freezing or stuttering during navigation
□ Images load smoothly (no infinite spinners)
□ Battery drain is reasonable (no background tasks running constantly)
```

## Summary

Rork apps get rejected most often not because of technical flaws, but because they feel incomplete, unpolished, or generic. The path forward is clear:

1. **Customize thoroughly**—replace every default with your brand
2. **Test completely**—find bugs before Apple does
3. **Document honestly**—clear privacy policies, helpful descriptions, test accounts
4. **Appeal respectfully**—acknowledge feedback and explain your fixes

One more tip: when you appeal, be specific. "We fixed the issue" gets rejected. "We replaced all placeholder text with real onboarding content, added AI-powered task categorization, and included detailed privacy documentation" gets approved.
