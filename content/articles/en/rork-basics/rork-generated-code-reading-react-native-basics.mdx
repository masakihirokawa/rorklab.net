You built an app with Rork, shipped it to the App Store, then opened the source to change one tiny thing — and promptly closed the window in defeat. If that sounds familiar, you are in very good company. I spent my first two or three Rork projects completely avoiding the generated code, assuming it was somehow reserved for "real" engineers.

Here is the secret: about eighty percent of what Rork emits is the same handful of patterns, repeated. You do not need to learn React Native as an engineer — you only need to learn how to read it enough to make small surgical edits. This guide is written for that narrower goal, not to turn you into a developer.

## Why reading the code matters, even when you mostly let the AI drive

Handing everything off to the AI is a perfectly valid strategy. But in practice, a few situations tend to show up again and again where reading skills pay off:

- The AI keeps trying to fix the same bug without success, and a two-line manual edit would be faster
- You want to tweak a single piece of copy in a paywall or review prompt
- App Review rejects your submission and asks you to change the color of one button

These are the cases where the fix is small and targeted, but asking the AI to make it usually means **other things mutate that you did not want changed**. Ten minutes of human edits turns into an hour of regeneration lottery, and morale takes a hit.

You do not need the ability to write this code from scratch. You need a **mental map** of what lives where. That map takes far less time to build than most people expect — an afternoon, realistically.

## The only three files worth opening first

A fresh Rork project ships with a lot of files, which is intimidating. For 90% of what you will ever need to touch, three files are enough:

- `app/_layout.tsx` — the overall layout and which screen belongs to which tab
- `app/(tabs)/index.tsx` — the first screen shown when the home tab is selected
- `app.json` — app name, icons, splash screen, version number

A rough heuristic: if you want to change something you *see* on a screen, the relevant file is somewhere under `app/(tabs)/`. If you want to change the **app itself** — its name, icon, or version — `app.json` is almost always the answer. Directories like `node_modules/`, `ios/`, and `android/` exist for advanced use cases; leave them closed at first. In many weeks of maintenance work on a typical small Rork app, I have never had to open any of them manually.

For a broader orientation, [Getting started with Rork](/articles/rork-basics/getting-started) also walks through the folder layout at a high level, and pairs nicely with this article.

## JSX is just "boxes inside boxes" — that is all you really need to know

Open a screen file and you will see a strange HTML-like syntax. This is JSX. A typical Rork-generated home screen looks roughly like this:

```tsx
// app/(tabs)/index.tsx
// Purpose: Home screen — a title and a button, centered
import { View, Text, Pressable, StyleSheet } from 'react-native';

export default function HomeScreen() {
  const handlePress = () => {
    console.log('Button pressed');
  };

  return (
    <View style={styles.container}>
      <Text style={styles.title}>Welcome</Text>
      <Pressable style={styles.button} onPress={handlePress}>
        <Text style={styles.buttonText}>Get started</Text>
      </Pressable>
    </View>
  );
}

const styles = StyleSheet.create({
  container: { flex: 1, alignItems: 'center', justifyContent: 'center' },
  title: { fontSize: 24, fontWeight: 'bold', marginBottom: 20 },
  button: { backgroundColor: '#0A84FF', paddingHorizontal: 24, paddingVertical: 12, borderRadius: 8 },
  buttonText: { color: 'white', fontSize: 16 },
});
```

The trick is to picture each element physically: `<View>` is a box, `<Text>` is a label, `<Pressable>` is anything you can tap. Read out loud, the snippet above is saying "inside a centered box, there is a heading and a blue button stacked vertically." To change "Welcome" to "Welcome back," you edit the text between `<Text>` tags. That is the whole operation.

Note the `handlePress` function. It is nothing more than a small piece of code that runs when the button is tapped. In real apps you will see it open another screen, call an API, or toggle a value — but the shape stays the same: a function defined above `return`, and attached to something with `onPress={...}`. Recognizing this pattern alone explains a surprising amount of what Rork generates.

With practice, you will start to glance at a file and instantly recognize things like *"oh, this is a list of cards arranged vertically"* — which is already most of what reading code is.

## `useState` is the second concept — and it unlocks half the app

Once static screens make sense, the next idea worth your time is **state**: values in a screen that can change as the user interacts. A counter that increments on tap, a search field that reflects what was typed — these are all driven by state.

```tsx
// Purpose: Tap the button to increment a counter
import { useState } from 'react';
import { View, Text, Pressable } from 'react-native';

export default function Counter() {
  // count: the current number (starts at 0)
  // setCount: the only way to update count
  const [count, setCount] = useState(0);

  return (
    <View>
      <Text>Current: {count}</Text>
      <Pressable onPress={() => setCount(count + 1)}>
        <Text>Increment</Text>
      </Pressable>
    </View>
  );
}
```

The pivotal line is `useState(0)`. You are telling the app "remember a value that starts at 0, and allow it to change." The pair `[count, setCount]` gives you a way to read the value and a function to update it — always through `setCount(...)`, never by reassigning `count` directly.

When you scan a Rork-generated file, search for `useState`. You will find it near login info, input text, the currently selected tab, loading flags, and anything else that shifts based on user action. Put differently, **the lines near `useState` are the business logic of the screen**. If the app "does something surprising," this is where to look first.

One related habit that pays off: when something appears to break only after a user does X, open the file, find the `useState` lines, and trace which of them might have changed due to X. You usually find the culprit within a minute.

## Five patterns that show up in almost every Rork output

A shortcut to fluency: these five patterns cover a huge portion of what you will encounter.

1. **Navigation uses `router.push('/path')`** — Rork uses Expo Router, so filenames become routes. If `app/settings.tsx` exists, `router.push('/settings')` opens it
2. **Network calls wrap `fetch` in `async/await`** — if you see `await fetch(...)`, that is where the app talks to an external server
3. **Colors and fonts live in `StyleSheet.create`** — design tweaks rarely require touching anything else
4. **Assets are loaded via `require('./assets/...')`** — replacing the file in the `assets/` folder and updating this path is usually enough
5. **Environment variables show up as `process.env.EXPO_PUBLIC_XXX`** — API keys go in `.env`, and this is how the code reads them

Internalizing just these five takes you most of the way. I usually tell people to keep this short list pinned in a note app for the first month — every time you spot one in the wild, the pattern locks in a little more.

## Start with the smallest possible edit — then go from there

Once you can read, the best way to build confidence is to make the world's smallest change: edit a string inside a `<Text>`, or change a `backgroundColor` in `StyleSheet`. That's it. The moment you see the app rebuild with your edit on a real device, the intimidation factor drops sharply.

After that first win, a sensible sequence is: change a color, change a margin, add a new menu item, duplicate an existing screen and rename it. Each step is tiny, and each one compounds into a real ability to steer the AI rather than depend on it.

If you also want to refine your **prompting** habits so the AI hands you cleaner code to begin with, [the Rork AI prompt engineering mastery guide](/articles/rork-ai/rork-ai-prompt-engineering-mastery-guide-2026) complements this article well. The two skills — reading and prompting — reinforce each other in ways that are hard to appreciate until you have both.

If you find yourself wanting a structured introduction to React Native itself, [Learning React Native by Bonnie Eisenman](https://www.amazon.co.jp/dp/1491989149/?tag=pinocchio-22) is a solid, readable reference. You do not need to read cover to cover — treat it as a buffet and pick the chapters that match whatever you are stuck on this week.

One closing thought: reading code is mostly a confidence problem, not a technical one. The syntax looks unfamiliar, so the brain treats it as a walled garden and refuses entry. But the actual mechanics are no deeper than following a recipe — a View holds other views, a Text shows a string, and a useState lets a number or flag change when the user does something. If you accept that the AI will keep doing 90% of the work, and you only need to step in for the occasional surgical edit, the mountain shrinks to something you can climb on a Saturday afternoon.

For today, open your `index.tsx`, change one word inside a `<Text>`, and watch it appear on your phone. Those ten minutes will pay for themselves many times over.
