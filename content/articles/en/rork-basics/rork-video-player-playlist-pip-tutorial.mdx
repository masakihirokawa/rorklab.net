## The Audio Player Worked Fine — Then Video Broke Everything

If you've built an audio player with Rork before, you know that `expo-av`'s `Audio.Sound` API makes playback straightforward. Hit play, pause, seek — it just works. So you think, "Video should be similar, right?"

Not quite. The moment you swap in the `Video` component, a cascade of new problems appears. The native controls don't match your app's design. Playlist auto-advance doesn't exist out of the box. Fullscreen mode fights with your navigation. Background playback stops dead when the user switches apps.

These issues stem from a fundamental difference: audio playback runs independently of the UI, but video playback is tightly coupled to a visible component on screen. That changes everything about how you architect the player.

This tutorial walks through building a custom video player from scratch in Rork — with draggable seek bar, playlist management, auto-advance, and iOS Picture-in-Picture support. Every code block is complete and ready to run.

## Understanding expo-av's Video Component

`expo-av` provides two playback APIs: `Audio.Sound` for audio and `Video` for video. The `Video` component is a React Native view that renders video frames, which means its lifecycle is tied to the screen.

Here's the base video player component with custom controls:

```tsx
// components/VideoPlayer.tsx
import React, { useRef, useState, useCallback } from 'react';
import { View, StyleSheet, Dimensions, TouchableOpacity, Text } from 'react-native';
import { Video, ResizeMode, AVPlaybackStatus } from 'expo-av';

const { width: SCREEN_WIDTH } = Dimensions.get('window');
const VIDEO_HEIGHT = SCREEN_WIDTH * (9 / 16); // 16:9 aspect ratio

interface VideoPlayerProps {
  uri: string;
  title: string;
  onPlaybackFinish?: () => void;
}

export default function VideoPlayer({ uri, title, onPlaybackFinish }: VideoPlayerProps) {
  const videoRef = useRef<Video>(null);
  const [status, setStatus] = useState<AVPlaybackStatus | null>(null);

  const isPlaying = status?.isLoaded ? status.isPlaying : false;
  const position = status?.isLoaded ? status.positionMillis : 0;
  const duration = status?.isLoaded ? status.durationMillis ?? 0 : 0;

  const handlePlaybackStatusUpdate = useCallback((newStatus: AVPlaybackStatus) => {
    setStatus(newStatus);
    // Detect playback completion for auto-advance
    if (newStatus.isLoaded && newStatus.didJustFinish) {
      onPlaybackFinish?.();
    }
  }, [onPlaybackFinish]);

  const togglePlayPause = async () => {
    if (\!videoRef.current) return;
    if (isPlaying) {
      await videoRef.current.pauseAsync();
    } else {
      await videoRef.current.playAsync();
    }
  };

  const formatTime = (millis: number) => {
    const totalSeconds = Math.floor(millis / 1000);
    const minutes = Math.floor(totalSeconds / 60);
    const seconds = totalSeconds % 60;
    return `${minutes}:${seconds.toString().padStart(2, '0')}`;
  };

  return (
    <View style={styles.container}>
      <Video
        ref={videoRef}
        source={{ uri }}
        style={styles.video}
        resizeMode={ResizeMode.CONTAIN}
        onPlaybackStatusUpdate={handlePlaybackStatusUpdate}
        useNativeControls={false} // We'll build our own controls
      />
      <TouchableOpacity style={styles.overlay} onPress={togglePlayPause}>
        <Text style={styles.playButton}>{isPlaying ? '⏸' : '▶'}</Text>
      </TouchableOpacity>
      <View style={styles.timeBar}>
        <Text style={styles.timeText}>{formatTime(position)}</Text>
        <View style={styles.progressTrack}>
          <View
            style={[
              styles.progressFill,
              { width: duration > 0 ? `${(position / duration) * 100}%` : '0%' },
            ]}
          />
        </View>
        <Text style={styles.timeText}>{formatTime(duration)}</Text>
      </View>
      <Text style={styles.title}>{title}</Text>
    </View>
  );
}

const styles = StyleSheet.create({
  container: { width: SCREEN_WIDTH, backgroundColor: '#000' },
  video: { width: SCREEN_WIDTH, height: VIDEO_HEIGHT },
  overlay: {
    ...StyleSheet.absoluteFillObject,
    height: VIDEO_HEIGHT,
    justifyContent: 'center',
    alignItems: 'center',
  },
  playButton: { fontSize: 48, color: 'rgba(255,255,255,0.8)' },
  timeBar: {
    flexDirection: 'row',
    alignItems: 'center',
    paddingHorizontal: 12,
    paddingVertical: 8,
  },
  timeText: { color: '#aaa', fontSize: 12, width: 45 },
  progressTrack: {
    flex: 1,
    height: 3,
    backgroundColor: '#333',
    borderRadius: 1.5,
    marginHorizontal: 8,
  },
  progressFill: { height: 3, backgroundColor: '#e63946', borderRadius: 1.5 },
  title: { color: '#fff', fontSize: 14, paddingHorizontal: 12, paddingBottom: 8 },
});
```

The key decision here is `useNativeControls={false}`. The OS-provided controls work for quick prototypes, but they become a dead end when you need playlist integration, custom PiP buttons, or branded styling. Building custom controls from the start pays off every time.

## Making the Seek Bar Draggable

The progress bar above is display-only — users can't drag it to jump to a specific position. For a video player, that's a dealbreaker. Here's a draggable seek bar using `PanResponder`:

```tsx
// components/SeekBar.tsx
import React, { useRef } from 'react';
import { View, StyleSheet, PanResponder, GestureResponderEvent } from 'react-native';

interface SeekBarProps {
  progress: number; // 0 to 1
  onSeek: (progress: number) => void;
  onSeekStart?: () => void;
  onSeekEnd?: () => void;
}

export default function SeekBar({ progress, onSeek, onSeekStart, onSeekEnd }: SeekBarProps) {
  const barRef = useRef<View>(null);

  const getProgressFromEvent = (evt: GestureResponderEvent) => {
    return new Promise<number>((resolve) => {
      barRef.current?.measure((_x, _y, width, _height, pageX) => {
        const touchX = evt.nativeEvent.pageX - pageX;
        resolve(Math.max(0, Math.min(1, touchX / width)));
      });
    });
  };

  const panResponder = useRef(
    PanResponder.create({
      onStartShouldSetPanResponder: () => true,
      onPanResponderGrant: async (evt) => {
        onSeekStart?.();
        const p = await getProgressFromEvent(evt);
        onSeek(p);
      },
      onPanResponderMove: async (evt) => {
        const p = await getProgressFromEvent(evt);
        onSeek(p);
      },
      onPanResponderRelease: () => {
        onSeekEnd?.();
      },
    })
  ).current;

  return (
    <View ref={barRef} style={styles.track} {...panResponder.panHandlers}>
      <View style={[styles.fill, { width: `${progress * 100}%` }]} />
      <View style={[styles.thumb, { left: `${progress * 100}%` }]} />
    </View>
  );
}

const styles = StyleSheet.create({
  track: {
    flex: 1,
    height: 20, // Large touch target (visual bar is only 3px)
    justifyContent: 'center',
    marginHorizontal: 8,
  },
  fill: {
    position: 'absolute',
    height: 3,
    backgroundColor: '#e63946',
    borderRadius: 1.5,
  },
  thumb: {
    position: 'absolute',
    width: 14,
    height: 14,
    borderRadius: 7,
    backgroundColor: '#e63946',
    marginLeft: -7,
    top: 3,
  },
});
```

Notice the touch target trick: the `track` view has `height: 20` while the visible bar is only 3px tall. This is the same approach YouTube uses — a thin visual bar with a much larger invisible touch area. Without it, users on small screens will tap and miss, then blame your app for being "unresponsive."

## Playlist Management with Auto-Advance

A video player's real value isn't playing one video — it's the continuous viewing experience. Here's a custom hook that manages playlists with shuffle and repeat modes:

```tsx
// hooks/usePlaylist.ts
import { useState, useCallback } from 'react';

export interface VideoItem {
  id: string;
  uri: string;
  title: string;
  duration?: number; // seconds
  thumbnail?: string;
}

export function usePlaylist(initialItems: VideoItem[]) {
  const [items, setItems] = useState<VideoItem[]>(initialItems);
  const [currentIndex, setCurrentIndex] = useState(0);
  const [isShuffled, setIsShuffled] = useState(false);
  const [repeatMode, setRepeatMode] = useState<'none' | 'all' | 'one'>('none');

  const currentItem = items[currentIndex] ?? null;

  const playNext = useCallback(() => {
    setCurrentIndex((prev) => {
      if (repeatMode === 'one') return prev; // Repeat same video
      const next = prev + 1;
      if (next >= items.length) {
        return repeatMode === 'all' ? 0 : prev; // Loop or stop
      }
      return next;
    });
  }, [items.length, repeatMode]);

  const playPrevious = useCallback(() => {
    setCurrentIndex((prev) => {
      const prevIdx = prev - 1;
      if (prevIdx < 0) {
        return repeatMode === 'all' ? items.length - 1 : 0;
      }
      return prevIdx;
    });
  }, [items.length, repeatMode]);

  const jumpTo = useCallback((index: number) => {
    if (index >= 0 && index < items.length) {
      setCurrentIndex(index);
    }
  }, [items.length]);

  const toggleShuffle = useCallback(() => {
    setIsShuffled((prev) => {
      if (\!prev) {
        // Pin current video at index 0, shuffle the rest
        const current = items[currentIndex];
        const rest = items.filter((_, i) => i \!== currentIndex);
        const shuffled = rest.sort(() => Math.random() - 0.5);
        setItems([current, ...shuffled]);
        setCurrentIndex(0);
      }
      return \!prev;
    });
  }, [items, currentIndex]);

  const cycleRepeatMode = useCallback(() => {
    setRepeatMode((prev) => {
      const modes: Array<'none' | 'all' | 'one'> = ['none', 'all', 'one'];
      const nextIdx = (modes.indexOf(prev) + 1) % modes.length;
      return modes[nextIdx];
    });
  }, []);

  return {
    items,
    currentItem,
    currentIndex,
    isShuffled,
    repeatMode,
    playNext,
    playPrevious,
    jumpTo,
    toggleShuffle,
    cycleRepeatMode,
  };
}
```

The shuffle implementation pins the currently playing video at index 0 before randomizing the rest. Without this, pressing shuffle would immediately jump to a random video — the same UX mistake that Spotify famously fixed in their early days.

The three repeat modes (`none`, `all`, `one`) map directly to what users expect from any media player. Building these into the hook from the start is much cleaner than bolting them on later.

## Building the Playlist UI

The playlist view uses `FlatList` for efficient rendering, showing thumbnails, titles, and duration badges:

```tsx
// components/PlaylistView.tsx
import React from 'react';
import { View, Text, FlatList, TouchableOpacity, Image, StyleSheet } from 'react-native';
import { VideoItem } from '../hooks/usePlaylist';

interface PlaylistViewProps {
  items: VideoItem[];
  currentIndex: number;
  onSelect: (index: number) => void;
}

export default function PlaylistView({ items, currentIndex, onSelect }: PlaylistViewProps) {
  const formatDuration = (seconds?: number) => {
    if (\!seconds) return '--:--';
    const m = Math.floor(seconds / 60);
    const s = seconds % 60;
    return `${m}:${s.toString().padStart(2, '0')}`;
  };

  const renderItem = ({ item, index }: { item: VideoItem; index: number }) => {
    const isActive = index === currentIndex;
    return (
      <TouchableOpacity
        style={[styles.item, isActive && styles.activeItem]}
        onPress={() => onSelect(index)}
      >
        <View style={styles.thumbnailContainer}>
          {item.thumbnail ? (
            <Image source={{ uri: item.thumbnail }} style={styles.thumbnail} />
          ) : (
            <View style={[styles.thumbnail, styles.placeholderThumb]} />
          )}
          <Text style={styles.durationBadge}>{formatDuration(item.duration)}</Text>
        </View>
        <View style={styles.info}>
          <Text style={[styles.title, isActive && styles.activeTitle]} numberOfLines={2}>
            {item.title}
          </Text>
          {isActive && <Text style={styles.nowPlaying}>Now Playing</Text>}
        </View>
      </TouchableOpacity>
    );
  };

  return (
    <FlatList
      data={items}
      keyExtractor={(item) => item.id}
      renderItem={renderItem}
      showsVerticalScrollIndicator={false}
    />
  );
}

const styles = StyleSheet.create({
  item: {
    flexDirection: 'row',
    padding: 12,
    borderBottomWidth: StyleSheet.hairlineWidth,
    borderBottomColor: '#222',
  },
  activeItem: { backgroundColor: '#1a1a2e' },
  thumbnailContainer: { position: 'relative' },
  thumbnail: { width: 120, height: 68, borderRadius: 6, backgroundColor: '#333' },
  placeholderThumb: { justifyContent: 'center', alignItems: 'center' },
  durationBadge: {
    position: 'absolute',
    bottom: 4,
    right: 4,
    backgroundColor: 'rgba(0,0,0,0.75)',
    color: '#fff',
    fontSize: 10,
    paddingHorizontal: 4,
    paddingVertical: 1,
    borderRadius: 2,
    overflow: 'hidden',
  },
  info: { flex: 1, marginLeft: 12, justifyContent: 'center' },
  title: { color: '#eee', fontSize: 14, lineHeight: 20 },
  activeTitle: { color: '#e63946', fontWeight: '600' },
  nowPlaying: { color: '#e63946', fontSize: 11, marginTop: 4 },
});
```

`FlatList` is essential here, not optional. If you render the playlist with `ScrollView` + `.map()`, you'll see frame drops once the list exceeds about 50 items. `FlatList` only renders visible rows, keeping scroll performance smooth regardless of playlist size.

## Picture-in-Picture on iOS

iOS Picture-in-Picture lets users keep watching video in a floating window after leaving your app. The `expo-av` `Video` component supports PiP, but you need to configure the audio session correctly first:

```tsx
// Configure audio session for background playback (call once at app startup)
import { Audio } from 'expo-av';

async function configureAudioSession() {
  await Audio.setAudioModeAsync({
    allowsRecordingIOS: false,
    staysActiveInBackground: true, // Required for PiP audio
    playsInSilentModeIOS: true,    // Play audio even in silent mode
    shouldDuckAndroid: true,
    playThroughEarpieceAndroid: false,
  });
}
```

Without `staysActiveInBackground: true`, the audio track stops the moment the user leaves the app, and the PiP window disappears within seconds.

In Expo's managed workflow, the simplest path to PiP is using native controls (`useNativeControls={true}`), which automatically shows the PiP button on supported iOS devices. If you're using custom controls but still want PiP, you can toggle native controls when the user taps a PiP button:

```tsx
const [useNative, setUseNative] = useState(false);

const enterPiPMode = () => {
  // Temporarily switch to native controls to expose the PiP icon
  setUseNative(true);
};
```

This approach isn't seamless, but it's the most practical solution within the managed workflow. For production apps that need polished PiP behavior, Rork Max with SwiftUI's `AVPlayerViewController` gives you full control. Combine it with the background playback patterns covered in the [audio player tutorial](/articles/rork-dev/rork-audio-player-app-tutorial) for the best results.

## Fullscreen Mode

Fullscreen video requires hiding the status bar, unlocking screen rotation, and restoring everything when the user exits. Here's a hook that handles it all:

```tsx
// hooks/useFullscreen.ts
import { useState, useCallback } from 'react';
import { StatusBar, Dimensions } from 'react-native';
import * as ScreenOrientation from 'expo-screen-orientation';

export function useFullscreen() {
  const [isFullscreen, setIsFullscreen] = useState(false);

  const enterFullscreen = useCallback(async () => {
    StatusBar.setHidden(true, 'fade');
    await ScreenOrientation.unlockAsync();
    setIsFullscreen(true);
  }, []);

  const exitFullscreen = useCallback(async () => {
    StatusBar.setHidden(false, 'fade');
    await ScreenOrientation.lockAsync(
      ScreenOrientation.OrientationLock.PORTRAIT_UP
    );
    setIsFullscreen(false);
  }, []);

  const toggleFullscreen = useCallback(async () => {
    if (isFullscreen) {
      await exitFullscreen();
    } else {
      await enterFullscreen();
    }
  }, [isFullscreen, enterFullscreen, exitFullscreen]);

  const dimensions = isFullscreen
    ? {
        width: Dimensions.get('screen').width,
        height: Dimensions.get('screen').height,
      }
    : {
        width: Dimensions.get('window').width,
        height: Dimensions.get('window').width * (9 / 16),
      };

  return { isFullscreen, toggleFullscreen, enterFullscreen, exitFullscreen, dimensions };
}
```

One subtle but important detail: `Dimensions.get('screen')` versus `Dimensions.get('window')`. The `screen` variant includes the status bar area, while `window` returns only the usable area. In fullscreen mode, you want `screen` — otherwise you'll have a gap at the top where the status bar used to be.

Add `expo-screen-orientation` with `npx expo install expo-screen-orientation`. In Rork, you can simply prompt "add expo-screen-orientation to the project" and it handles the dependency.

## Three Pitfalls That Will Waste Your Afternoon

After building the player, these are the issues that catch most developers off guard. Knowing them ahead of time saves real debugging hours.

**1. Black screen on Android (audio plays, no video)**

Some Android devices render a black frame with `ResizeMode.CONTAIN` while the audio track plays normally. This is a hardware acceleration quirk in `expo-av`. The workaround is deceptively simple:

```tsx
<Video
  style={[styles.video, { opacity: 0.99 }]} // Forces a different render path
/>
```

Setting `opacity: 0.99` looks like a no-op, but it pushes React Native's rendering engine through a different code path that avoids the hardware acceleration bug.

**2. Audio continues after navigating away**

When using React Navigation, leaving the player screen doesn't automatically pause playback. The video component unmounts, but the audio track keeps going. Fix it with the navigation focus hook:

```tsx
import { useIsFocused } from '@react-navigation/native';

const isFocused = useIsFocused();

useEffect(() => {
  if (\!isFocused && videoRef.current) {
    videoRef.current.pauseAsync();
  }
}, [isFocused]);
```

**3. Memory pressure on long videos**

Videos over 30 minutes can push memory usage high enough to trigger OS warnings. Reduce the playback status update frequency from the default 500ms to 1000ms:

```tsx
<Video
  progressUpdateIntervalMillis={1000}
/>
```

This alone won't solve extreme cases, but it cuts the status update overhead in half and buys you headroom for longer content.

## What's Next — Streaming and Offline Playback

You now have a working video player with custom controls, draggable seek bar, playlist management with shuffle and repeat, PiP support, and fullscreen mode. For local files or direct URLs, this is production-ready.

The natural next step for a streaming-focused app is HLS (HTTP Live Streaming) support. `expo-av` handles HLS natively — just pass an `.m3u8` URL as the source. However, controlling adaptive bitrate switching requires native code, which is where Rork Max becomes valuable.

For offline playback, combine `expo-file-system` downloads with [local storage management](/articles/rork-dev/rork-local-storage-asyncstorage-mmkv-sqlite-guide) to track downloaded files, check available storage, and auto-delete expired content. That's a substantial enough topic for its own dedicated guide.

Start with the code in this article, get a working player running, and then decide which features your specific app needs. That's the fastest path from tutorial to production — and the approach I personally recommend for any media app project.
