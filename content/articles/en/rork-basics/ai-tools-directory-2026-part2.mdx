Firebase's database services (Realtime Database and Firestore) are essential for mobile apps built with Rork. They provide real-time data synchronization, offline support, and serverless scaling. Your Rork app typically connects to Firebase for data persistence.

**Use Cases**

- Complex data structures and relationships

- Structured app data
- User profiles and settings
- Analytics and tracking
- Content management
- Social features

**Firebase SDK Integration**

When building Rork apps, Firebase SDK integrates directly:

1. Add Firebase SDK to Rork project
2. Configure Firebase project and credentials
3. Initialize database connection
4. Implement authentication integration
5. Build queries and listeners
6. Implement offline caching
7. Set security rules for data protection

**Data Structure Best Practices**

- Design shallow hierarchies
- Avoid deep nesting

**Pricing**

- Realtime Database: $5-25/GB stored, bandwidth charges
- Firestore: $0.06 per 100K reads, $0.18 per 100K writes
- Storage: $0.18/GB stored, $1/GB downloaded
- Free tier: Generous limits for development

**Security Rules**

Firebase requires explicit security rules:

```javascript
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    match /users/{userId} {
      allow read: if request.auth.uid == userId;
      allow write: if request.auth.uid == userId;
    }
    match /posts/{postId} {
      allow read: if true;
      allow write: if request.auth != null;
    }
  }
}
```

**Offline Support**

Firebase's offline capabilities are crucial for mobile:
- App reads from local cache while offline
- Changes queued locally
- Automatic sync when connection restored
- Conflict resolution handles simultaneous updates
- Essential for reliable mobile experience

**Use Cases for Mobile Apps**

- Ultra-low latency API endpoints
- Global request routing and load balancing
- Real-time features with minimal latency
- Request filtering and manipulation
- Response caching and serving
- Rate limiting per user/IP
- Authentication layer
- Request transformation

**Integration with Rork Apps**

```
Rork App
↓
Cloudflare Worker (Edge)
↓
Origin Server / API / Database
```

Workers intercept requests and can:
- Serve cached responses instantly
- Apply authentication
- Transform requests/responses

**Pricing**

- Free: 100,000 requests/day
- Workers Paid: $5/month, then $0.50 per 10M requests
- KV: Additional charges for storage
- D1: Additional charges for database operations
- Durable Objects: $0.15 per object per day

**Performance Example**

- Traditional cloud function: 200-500ms latency
- Cloudflare Worker: 10-50ms latency
- Cached response: <1ms

**Example: API Endpoint with Caching**

```javascript
export default {
  async fetch(request, env) {
    const cache = caches.default;
    const cached = await cache.match(request);

    if (cached) return cached;

    const response = await fetch(
      'https://api.example.com/endpoint',
      { headers: request.headers }
    );

    const clonedResponse = response.clone();
    await cache.put(request, clonedResponse);

    return response;
  }
};
```

**When to Use Workers**

- APIs that need global distribution

- Rate limiting and protection
- Request transformation and filtering
- Caching and CDN needs

## App Monetization & Billing

### StoreKit 2 (Apple In-App Purchases)

**Overview**

StoreKit 2 is Apple's modern framework for in-app purchases and subscriptions. It's the required approach for monetizing iOS apps, handling payment processing, subscription management, and revenue sharing with Apple.

**Core Capabilities**

- In-app purchases (consumable, non-consumable)
- Auto-renewing subscriptions
- Subscription upgrade/downgrade
- Family sharing support
- Refund handling
- Promotional offers
- Sandbox testing environment
- Server-side validation

**Purchase Types**

- **Consumable**: One-time use (coins, gems, premium features)
- **Non-Consumable**: Permanent unlock (ad removal, themes)
- **Auto-Renewing Subscription**: Monthly/yearly recurring
- **Non-Renewing Subscription**: Fixed-duration access

**Integration with Rork**

Rork handles StoreKit integration:

1. Create products in App Store Connect
2. Configure in Rork app
3. Implement purchase flow UI
4. Handle purchase results
5. Validate purchases server-side
6. Unlock features based on purchase
7. Handle refunds and cancellations

**Best Practices**

- Store receipt validation on backend (never trust client)
- Implement renewal/expiry checking
- Handle network failures gracefully
- Provide sandbox testing before release
- Test edge cases: refunds, cancellations, renewals
- Monitor for fraud
- Set up renewal reminders

**Pricing Strategy Considerations**

- **Freemium**: Free tier with paid features
- **Premium**: Upfront purchase, no ads
- **Subscriptions**: Recurring revenue, more predictable
- **Consumables**: For utility/game apps
- **Trial periods**: 3-14 days standard

**Common Subscription Structures**

- $0.99/month - Budget tier
- $2.99/month - Standard tier
- $9.99/month - Premium tier
- $4.99/month or $39.99/year - Annual discount incentive

**Revenue Sharing**

- Apple takes 30% of all revenue
- Developer keeps 70%
- Small business program: 15% commission (if under $1M annual revenue)

**Sandbox Testing**

Crucial for testing before App Store release:
- Create sandbox test accounts
- Simulate purchases, renewals, cancellations
- Test refund handling
- Verify subscription state
- No real payment in sandbox

**Key Advantages**

- Single API for iOS, Android, web
- Handles StoreKit 2 and Google Play Billing differences
- Subscription state management
- Cross-platform analytics
- Paywall optimization
- Customer analytics
- Integration with multiple payment processors
- Webhook support for backend systems

**Architecture Benefit**

```
Traditional Approach:
iOS → StoreKit 2 API
Android → Google Play Billing API
Web → Stripe API

RevenueCat Approach:
iOS, Android, Web → RevenueCat API → Unified handling
```

**Core Features**

- Subscription lifecycle management
- Feature entitlements (check if user has feature)
- Paywall template system
- Analytics and reporting
- Customer management
- Webhook events
- Paywall testing and analytics

**Implementation Pattern**

1. User initiates purchase in app
2. RevenueCat handles platform-specific billing
3. RevenueCat validates with stores
4. Webhook sends purchase event to backend
5. Backend grants access/features
6. Next time user opens app, RevenueCat confirms subscription

**Pricing**

- 1% of MRR above $10K
- One-time purchases: No fee
- Non-renewing subscriptions: No fee

**When RevenueCat Makes Sense**

- Cross-platform apps (iOS + Android + Web)
- Complex subscription logic
- Paywall optimization needed
- Multiple payment methods
- Multi-currency support
- Significant subscription revenue

**Paywall System**

RevenueCat provides paywall templates:

### EAS (Expo Application Services)

**Overview**

EAS provides a complete ecosystem for building, testing, and distributing React Native and Expo apps. It's particularly valuable for Rork apps built on React Native, handling the complexity of building for iOS and Android.

**EAS Services**

- **EAS Build**: Managed cloud builds for iOS and Android

5. Test and iterate
6. Submit to app stores via EAS Submit

**Time Savings**

- Local builds: 30-60 min (iOS), 10-20 min (Android)

**Pricing**

- Enterprise: Custom pricing

**Over-The-Air Updates (EAS Update)**

- EAS generates update
- Users download on next app open
- Instant rollback if issues
- Perfect for hotfixes

**Usage Example**

**When to Use EAS vs Local Builds**

## Production Strategies & Advanced Topics

### Cross-Platform Monetization Architecture

**Complete Monetization System**

```
iOS App
  → StoreKit 2 → Purchase
  → RevenueCat API → Validates purchase
  → Backend → Grants access
  → Entitlements check on app open

Android App
  → Google Play Billing → Purchase
  → RevenueCat API → Validates purchase
  → Backend → Grants access
  → Entitlements check on app open

Web App
  → Stripe → Purchase
  → Custom backend → Grants access
  → Same entitlements system
```

**Unified Entitlements System**

Key principle: Single source of truth for subscription status
- Backend validates all purchases
- Client shows UI based on entitlements
- Fallback if client state disagrees with server
- Regular refresh of entitlements
- Handle all expiry/renewal scenarios

**Cost-Benefit Analysis**

- StoreKit 2 / Google Play: 30% commission, high discovery
- Direct subscriptions: 0% commission, harder to acquire users
- Hybrid approach: Offer both, let users choose

### Scaling Infrastructure

**Typical Growth Timeline**

- 0-1,000 users: Single app, simple backend
- 1K-10K users: Firebase sufficient
- 10K-100K users: Consider optimization
- 100K-1M users: Dedicated database, caching
- 1M+ users: Full infrastructure optimization

**Scaling Database**

```
Phase 1: Firebase (0-100K users)
Phase 2: Firebase + Caching (100K-1M)
Phase 3: Supabase/PostgreSQL (1M+)
Phase 4: Distributed database, read replicas (10M+)
```

**Scaling APIs**

```
Phase 1: Cloud Functions / Workers
Phase 2: Cloud Run / Cloudflare Workers (with caching)
Phase 3: Load-balanced endpoints, regional distribution
Phase 4: Global edge distribution, CDN
```

**Cost Optimization at Scale**

- Database indexes: Critical for query performance
- Caching: Reduce database hits
- CDN: Reduce bandwidth costs
- Batch operations: Reduce API calls
- Usage monitoring: Alert on cost anomalies
- Reserved capacity: Lock in rates for predictable usage
