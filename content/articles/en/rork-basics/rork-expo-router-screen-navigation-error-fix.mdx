# Fix Navigation Errors in Rork Apps: A Complete Troubleshooting Guide

When you run a Rork-generated app for the first time, navigation issues are among the most frustrating surprises. You tap a button and nothing happens. A red error screen appears saying "Screen not found." The tab bar is gone. The back button doesn't work.

Rork builds on Expo Router — a powerful file-based routing system — but even well-generated code can encounter navigation errors when the file structure shifts slightly or configuration is missing. This guide breaks down the six most common navigation errors Rork users run into, explains why each one happens, and walks you through how to fix them.

## Common Navigation Error Patterns

Here's a quick overview of the errors we'll cover:

- **`Couldn't find a navigation object`** — the navigation hook is called outside a navigator
- **`Screen not found`** — no file exists for the target route
- **Tab bar missing or unresponsive** — tab layout is misconfigured
- **Back button not working** — stack history isn't being built
- **Deep links not working** — missing scheme configuration
- **Scroll position or state not resetting** — focus lifecycle not handled

Let's walk through each one.

## Error 1: `Couldn't find a navigation object`

### What you see

Tapping a button triggers this error in your console:

```
Error: Couldn't find a navigation object. Is your component inside a screen in a Navigator?
```

### Why it happens

This error fires when `useNavigation()` or `useRouter()` is called inside a component that sits outside a Navigator's scope. In Rork-generated code, this commonly happens when a component is used as a modal or popup that isn't connected to the navigation tree.

```tsx
// ❌ Problematic pattern
// A component defined outside a Navigator calling useRouter
export function StandaloneButton() {
  const router = useRouter(); // ← Error thrown here
  return <Button onPress={() => router.push('/detail')} title="Go to Detail" />;
}

// ✅ Fix: Call hooks only inside screens rendered by a Navigator
// Make sure this component is rendered within a Stack.Screen or Tabs.Screen
```

### How to fix it

1. Find the component throwing the error and check where it's being rendered
2. Review `app/_layout.tsx` to confirm the component is inside a navigator tree
3. If not, ask Rork to "move this component inside the navigator scope" and regenerate

## Error 2: `Screen not found` — Route Doesn't Exist

### What you see

Calling `router.push('/detail/123')` or `router.navigate('profile')` results in:

```
Error: No route matching path "/detail/123" was found.
```

### Why it happens

Expo Router maps file paths directly to routes. If the file `app/detail/[id].tsx` doesn't exist, there is no route at `/detail/123`. This most often occurs in Rork apps when the AI generates navigation calls that reference screens it hasn't created yet, or when file names have a typo.

```
# Correct file structure for common routes
app/
├── _layout.tsx
├── index.tsx          → /
├── profile.tsx        → /profile
├── detail/
│   └── [id].tsx       → /detail/123, /detail/456, etc.
└── (tabs)/
    ├── _layout.tsx
    ├── home.tsx       → /(tabs)/home
    └── settings.tsx   → /(tabs)/settings
```

### How to fix it

1. Look at the path you're trying to navigate to (e.g., `/detail/123` needs `app/detail/[id].tsx`)
2. Check the `app/` directory and confirm the file exists
3. For dynamic routes (`[id].tsx`), verify the parameter name in the brackets matches what you're passing
4. Ask Rork to "create the missing screen at `app/detail/[id].tsx`" so it generates the file

## Error 3: Tab Bar Not Showing or Unresponsive

### What you see

The app launches but the bottom tab bar is missing entirely, or tapping a tab does nothing.

### Why it happens

Rork frequently auto-generates tab navigation, but the `(tabs)` folder structure or the `_layout.tsx` configuration can have gaps. The most common causes are a missing `app/(tabs)/_layout.tsx`, a `Tabs.Screen` name that doesn't match the filename, or an undefined tab icon reference.

```tsx
// Correct tab layout (app/(tabs)/_layout.tsx)
import { Tabs } from 'expo-router';
import { Ionicons } from '@expo/vector-icons';

export default function TabLayout() {
  return (
    <Tabs
      screenOptions={{
        tabBarActiveTintColor: '#007AFF',
      }}
    >
      {/* name must match the filename exactly (no extension) */}
      <Tabs.Screen
        name="index"         // maps to app/(tabs)/index.tsx
        options={{
          title: 'Home',
          tabBarIcon: ({ color }) => (
            <Ionicons name="home" size={24} color={color} />
          ),
        }}
      />
      <Tabs.Screen
        name="settings"      // maps to app/(tabs)/settings.tsx
        options={{
          title: 'Settings',
          tabBarIcon: ({ color }) => (
            <Ionicons name="settings" size={24} color={color} />
          ),
        }}
      />
    </Tabs>
  );
}
```

### How to fix it

1. Confirm both the `app/(tabs)/` folder and `_layout.tsx` exist
2. Check that every `Tabs.Screen` `name` matches its file name exactly (`index.tsx` → `name="index"`)
3. Verify the icon library (`@expo/vector-icons` or similar) is installed
4. Ask Rork to "fix the tab navigation layout in `(tabs)/_layout.tsx`" and regenerate it

## Error 4: Back Button Not Working

### What you see

After navigating forward in a stack, the back button in the header disappears or pressing it returns to the wrong screen.

### Why it happens

Stack navigation in Expo Router maintains a history of screens. Using `router.replace()` swaps the current screen instead of stacking a new one — so there's nothing to go back to. This is the right behavior for post-login redirects, but a common mistake in other contexts.

```tsx
// Back button disappears because replace() clears history
router.replace('/home');

// Use push() when you want the back button to work
router.push('/home');

// Use replace() intentionally for screens the user should not return to
// (e.g., after successful login, redirect to main tabs)
router.replace('/(tabs)/home');
```

### How to fix it

1. Check which navigation method you're using (`push`, `replace`, or `navigate`)
2. Use `router.push()` for screens where the user should be able to go back
3. Reserve `router.replace()` for transitions where backward navigation doesn't make sense
4. On Android, also verify the system back button behaves correctly on a real device

## Error 5: Deep Links Not Working

### What you see

Tapping a link from a push notification or social media post either opens the app on the home screen or doesn't open the app at all.

### Why it happens

Deep linking requires a URL scheme to be configured in `app.json` (or `app.config.js`). Rork-generated projects often leave this unconfigured, especially on Android where intent filters are needed.

```json
{
  "expo": {
    "name": "MyApp",
    "slug": "my-app",
    "scheme": "myapp",
    "ios": {
      "bundleIdentifier": "com.example.myapp"
    },
    "android": {
      "package": "com.example.myapp",
      "intentFilters": [
        {
          "action": "VIEW",
          "data": [
            {
              "scheme": "myapp",
              "host": "open"
            }
          ],
          "category": ["BROWSABLE", "DEFAULT"]
        }
      ]
    }
  }
}
```

### How to fix it

1. Open `app.json` and verify the `scheme` field is set
2. For Expo Router, check that `expo-linking` is properly configured
3. Test deep links on a simulator using `npx uri-scheme open myapp://profile --ios`
4. Ask Rork to "add deep link configuration with scheme `myapp` to `app.json`"

For a deeper look at React Native deep linking, [React Native in Action (Nader Dabit)](https://www.amazon.co.jp/dp/4798169889/?tag=pinocchio-22) provides a solid foundation.

## Error 6: Scroll Position or State Not Resetting on Navigation Return

### What you see

After navigating away from a list and coming back, the scroll position stays where it was — or a form isn't cleared when you expect it to be.

### Why it happens

Expo Router keeps stacked screens in memory, so component state persists when you navigate back. Whether this is a bug or a feature depends on your app's needs. When you do want to reset state on return, you need to handle the focus lifecycle explicitly.

```tsx
// Reset scroll position each time this screen comes into focus
import { useFocusEffect } from 'expo-router';
import { useCallback, useRef } from 'react';
import { ScrollView } from 'react-native';

export default function ListScreen() {
  const scrollRef = useRef<ScrollView>(null);

  useFocusEffect(
    useCallback(() => {
      // Scroll to top every time this screen gains focus
      scrollRef.current?.scrollTo({ x: 0, y: 0, animated: false });
    }, [])
  );

  return (
    <ScrollView ref={scrollRef}>
      {/* list content */}
    </ScrollView>
  );
}
```

### How to fix it

1. Use `useFocusEffect` to run code whenever the screen gains focus
2. Reset scroll position with `scrollRef.current?.scrollTo()`
3. Reset form state by calling your state setter inside `useFocusEffect`
4. Tell Rork to "reset the scroll position when this screen comes into focus" for an AI-generated fix

## FAQ

**Q. Will regenerating in Rork automatically fix navigation errors?**

A. Regenerating with the same prompt usually produces the same structure. Be specific: instead of "fix the navigation," try "add `app/detail/[id].tsx` with a back button that returns to the previous screen." The more precise your prompt, the more targeted the fix.

**Q. What is the difference between Expo Router and React Navigation?**

A. Expo Router uses file-based routing (similar to Next.js) — the file path in the `app/` folder becomes the URL route automatically. React Navigation requires you to define your routes in code with Stack, Tab, or Drawer navigators. Rork uses Expo Router by default, so file structure directly controls routing. See the [Expo Router Navigation Guide](/articles/rork-dev/expo-router-navigation-guide) for a full walkthrough.

**Q. Multiple navigation errors are happening at once — where do I start?**

A. Start at the root: review `app/_layout.tsx` and (if you have tabs) `app/(tabs)/_layout.tsx`. A misconfigured root layout breaks everything downstream. Once those look correct, copy the full error message into Rork's chat — the AI is often able to pinpoint the issue directly. For runtime errors beyond navigation, the [Expo Runtime Error Troubleshooting Guide](/articles/rork-dev/rork-react-native-expo-runtime-errors-fix) covers a wider range of issues.

## Summary

Most navigation errors in Rork-generated apps trace back to three root causes: a mismatch between the file structure and the route being navigated to, using the wrong navigation method (`push` vs `replace`), or a missing or misconfigured layout file.

When something breaks, start by checking the `app/` directory for the file that corresponds to your target route. Then review your navigation method choice. If the issue persists, paste the exact error message into Rork's chat — it's often the fastest path to a working fix.
