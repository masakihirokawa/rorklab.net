## Why Connect Rork to GitHub?

Rork lets you generate fully functional mobile apps from plain-English prompts — but as your project grows toward production, you need robust version control and team collaboration workflows. GitHub is the standard answer, and Rork makes it easy to connect.

Rork-generated code is built on React Native with Expo, meaning it's just standard open-source code you own entirely. Putting that code under GitHub version control unlocks:

- **Change history** — see exactly what the AI generated vs. what you changed manually
- **Rollback safety** — revert to any previous working version in seconds
- **Team reviews** — designers and engineers can code-review AI-generated output
- **CI/CD automation** — trigger builds and tests automatically on every push
- **Third-party integrations** — connect Vercel, Cloudflare, EAS, and other tools seamlessly

This guide walks you through the complete workflow: exporting from Rork, setting up a GitHub repository, managing branches, and automating builds with GitHub Actions.

## Prerequisites

Before you start, make sure you have the following:

- A **Rork account** (code export is available on all plans)
- A **GitHub account** (free)
- **Node.js 18 or later** installed locally
- **Expo CLI** (`npm install -g @expo/cli`)
- The **git** command-line tool

> Note: If you're using Rork Max (native Swift), the export flow differs — this guide focuses on the standard **Rork (React Native/Expo)** workflow.

## Step 1: Export Your Code and Connect to GitHub

### 1-1. Export from the Rork Dashboard

Open your Rork project dashboard and click the **"Export Code"** or **"GitHub"** button in the top-right corner.

Rork offers two export methods:

**Method A: ZIP Download**
Download the entire project as a ZIP file. Extract it locally and push to your own GitHub repository.

**Method B: GitHub OAuth Integration (Recommended)**
Authorize Rork via GitHub OAuth, then select a target repository. From that point on, every time you run a prompt, Rork automatically commits the resulting diff to your repository.

### 1-2. Create a New GitHub Repository

```bash
# Using the GitHub CLI
gh repo create my-rork-app --private --description "Mobile app built with Rork AI"

# Or create a repo on github.com and clone it locally
git clone https://github.com/your-username/my-rork-app.git
cd my-rork-app
```

### 1-3. Push Your Initial Commit

If you used the ZIP download method, follow these steps to make your first commit:

```bash
# Extract the ZIP into your repo directory
unzip rork-export.zip -d my-rork-app/
cd my-rork-app

# Set up .gitignore (skip if already included in the export)
cat << 'EOF' > .gitignore
node_modules/
.expo/
dist/
.env
.env.local
*.eas.json
ios/
android/
EOF

# Initialize git and push
git init
git add .
git commit -m "Initial: Rork generated app scaffold"
git remote add origin https://github.com/your-username/my-rork-app.git
git push -u origin main
```

After pushing, verify the React Native/Expo project structure is visible in your GitHub repository.

## Step 2: Branching Strategy for Rork Projects

Because Rork can regenerate large portions of code with each prompt, a solid branching strategy is essential to keep your `main` branch stable.

### 2-1. Recommended Branch Structure

```
main        ← Production releases (stable)
├── develop ← Integration branch for ongoing work
│   ├── rork/feature-login     ← AI-generated login screen
│   ├── rork/feature-profile   ← AI-generated profile page
│   └── fix/bugfix-crash       ← Manual bug fix
```

```bash
# Before running a new Rork prompt, create a feature branch
git checkout develop
git pull origin develop
git checkout -b rork/feature-onboarding

# After generating code in Rork, export and commit the diff
git add .
git commit -m "rork: Add onboarding screen flow (AI generated)"

# Open a Pull Request to develop for review before merging
git push origin rork/feature-onboarding
gh pr create --base develop --title "Add onboarding screen" \
  --body "Onboarding flow generated via Rork prompt"
```

### 2-2. Commit Message Conventions

Use prefixes to clearly distinguish AI-generated code from manual changes:

| Prefix | Meaning |
|--------|---------|
| `rork:` | Code generated by a Rork prompt |
| `fix:` | Bug fix |
| `feat:` | Manually added feature |
| `style:` | UI/design adjustments |
| `chore:` | Config changes, dependency updates |

## Step 3: Automate Builds with GitHub Actions + EAS

Combine GitHub Actions with Expo Application Services (EAS) to trigger automatic builds on every push to `main`.

### 3-1. Configure Expo EAS

```bash
# Install EAS CLI globally
npm install -g eas-cli

# Log in to your Expo account
eas login

# Initialize EAS for your project
eas build:configure
```

This creates an `eas.json` configuration file. Review the build profiles:

```json
// eas.json
{
  "cli": {
    "version": ">= 5.0.0"
  },
  "build": {
    "development": {
      "developmentClient": true,
      "distribution": "internal"
    },
    "preview": {
      "distribution": "internal",
      "ios": {
        "simulator": true
      }
    },
    "production": {
      "autoIncrement": true
    }
  },
  "submit": {
    "production": {}
  }
}
```

### 3-2. Create a GitHub Actions Workflow

```yaml
# .github/workflows/eas-build.yml
name: EAS Build

on:
  push:
    branches:
      - main
  pull_request:
    branches:
      - main

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      # Check out the repository
      - name: Checkout repository
        uses: actions/checkout@v4

      # Set up Node.js
      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: "20"
          cache: "npm"

      # Install dependencies
      - name: Install dependencies
        run: npm ci

      # Authenticate with Expo using a secret token
      - name: Setup EAS
        uses: expo/expo-github-action@v8
        with:
          eas-version: latest
          token: ${{ secrets.EXPO_TOKEN }}

      # Trigger a preview build for iOS simulator
      - name: Build preview (iOS)
        run: eas build --platform ios --profile preview --non-interactive
        # Expected output: Build #XXXX submitted. Build URL: https://expo.dev/...
```

**Setting up the GitHub Secret:**

1. Generate an API token in Expo Dashboard under **Settings > Access Tokens**
2. Add it to your GitHub repo at **Settings > Secrets and variables > Actions** as `EXPO_TOKEN`

Now every push to `main` triggers an EAS build — you can monitor status in the Expo dashboard.

## Common Errors and Fixes

Here are the most frequent issues when connecting Rork code to GitHub, and how to resolve them.

**Error: `failed to push some refs to 'origin'`**

Cause: Local and remote histories have diverged.

```bash
# Rebase and retry
git pull --rebase origin main
git push origin main
```

**Error: `npm ci` fails during CI build**

Cause: The `package-lock.json` exported by Rork may be outdated.

```bash
# Regenerate package-lock.json locally and commit it
rm package-lock.json
npm install
git add package-lock.json
git commit -m "chore: regenerate package-lock.json"
```

**Error: `EXPO_TOKEN` authentication failure**

Cause: Token has expired or has insufficient permissions. Generate a new token in the Expo dashboard and update the GitHub secret.

**Large unexpected diffs after a Rork prompt**

Rork sometimes regenerates the full codebase with each prompt. Use `git diff --stat` to review the scope before committing, and always work on a feature branch rather than directly on `main`.

```bash
# Review changes before staging
git diff --stat HEAD

# Inspect a specific file
git diff HEAD -- src/screens/HomeScreen.tsx
```

## Advanced: Team Collaboration Tips

When working with a team on a Rork project, the following practices make a big difference.

### Pull Request Template

Create `.github/pull_request_template.md` to standardize reviews of AI-generated code:

```markdown
## Summary
<!-- What was built/changed with Rork -->

## Rork Prompt Used
<!-- Paste the prompt(s) you submitted to Rork -->

## Screenshots
<!-- Add simulator or device screenshots -->

## Checklist
- [ ] Manually verified the UI
- [ ] Confirmed no regressions in existing features
- [ ] Checked that no secrets or API keys are included
```

### Protect Main with Branch Rules

In your repository settings, enable **Branch protection rules** for `main`:

- Require pull request reviews before merging
- Require status checks (EAS build) to pass
- Restrict direct pushes to `main`

This ensures every Rork-generated change is reviewed before it reaches production.

## FAQ

**Q: Can I use GitHub integration on the Rork free plan?**

A: ZIP code download is available on all Rork plans including free. The GitHub OAuth integration (direct push from Rork on every prompt) requires a paid plan. Free-plan users can manually push ZIPs to GitHub following the steps in Step 1-3.

**Q: If I manually edit the exported code, will a new Rork prompt overwrite my changes?**

A: Yes — Rork regenerates the full project on each prompt, which can overwrite manual edits. Keep custom modifications in separate branches (prefix them with `custom/`) and merge carefully after each Rork-generated update.

**Q: Can I use private GitHub repositories?**

A: Absolutely. When granting Rork access via GitHub OAuth, you can select private repositories. Always use private repos for projects containing proprietary business logic or API integrations.

**Q: How long do EAS builds take in GitHub Actions?**

A: iOS preview builds (simulator) typically take 5–15 minutes; production builds take 15–30 minutes depending on EAS queue load. For teams needing faster parallel builds, consider an EAS premium plan.
