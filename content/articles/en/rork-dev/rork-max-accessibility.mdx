Accessibility is not a feature—it's a responsibility. When you build with Rork Max, you're creating experiences that should be available to everyone, regardless of physical ability. This guide covers everything needed to make your Rork-generated apps fully accessible: screen readers, dynamic text, color blindness support, and App Store compliance.

## Why Accessibility Matters

**App Store Requirements**: Apple now mandates accessibility compliance for app approval. Apps lacking proper accessibility support are rejected during review.

**Legal Compliance**: The US ADA (Americans with Disabilities Act), EU WCAG 2.1, and similar laws in other countries require digital accessibility. Non-compliance carries legal risk.

**User Reach**: About 15% of the global population (1.1+ billion people) have disabilities. Full accessibility opens your app to this significant market.

**Business Impact**: Better SEO, higher user satisfaction, stronger brand reputation, and reduced support burden from accessibility-related complaints.

## accessibilityLabel: The Foundation

Every interactive element must have a semantic label. Screen reader users cannot see visual cues, so text labels are essential.

```typescript
import { TouchableOpacity, Text, View } from "react-native";

export function AddButton() {
  return (
    <TouchableOpacity
      accessibilityLabel="Create new task"
      accessibilityRole="button"
      onPress={() => console.log("Add task")}
      style={{
        width: 56,
        height: 56,
        borderRadius: 28,
        backgroundColor: "#007AFF",
        justifyContent: "center",
        alignItems: "center",
      }}
    >
      <Text style={{ fontSize: 24, color: "#fff" }}>+</Text>
    </TouchableOpacity>
  );
}
```

**Best Practices for Labels:**

- Keep them concise (2–5 words ideal)
- Don't include role names ("button", "link") — that's what `accessibilityRole` does
- Use action verbs ("Add", "Delete", "Share")
- Localize labels using translation files, not hardcoded strings

## accessibilityHint: Provide Context

Where `accessibilityLabel` describes *what*, `accessibilityHint` explains *why* or *what happens next*. It's read after the user focuses the element.

```typescript
<TouchableOpacity
  accessibilityLabel="Save"
  accessibilityHint="Saves your form changes and closes the editor"
  accessibilityRole="button"
  onPress={handleSave}
>
  <Text>Save</Text>
</TouchableOpacity>
```

Use hints sparingly—only when action consequences are unclear from the label alone.

## accessibilityRole: Semantic Elements

Assign correct roles so screen readers interpret elements appropriately:

```typescript
import { Text, View, Switch, TextInput, ScrollView, Button } from "react-native";

// Button
<TouchableOpacity accessibilityRole="button">
  <Text>Click me</Text>
</TouchableOpacity>

// Search input
<TextInput
  accessibilityRole="search"
  placeholder="Search tasks"
/>

// Radio button
<View accessibilityRole="radio" accessibilityState={{ selected: true }}>
  <Text>Option A</Text>
</View>

// Checkbox
<View accessibilityRole="checkbox" accessibilityState={{ checked: false }}>
  <Text>I agree</Text>
</View>

// Header
<View accessibilityRole="header">
  <Text style={{ fontSize: 24, fontWeight: "bold" }}>Section Title</Text>
</View>

// List with items
<ScrollView accessibilityRole="list">
  {items.map((item, idx) => (
    <View key={idx} accessibilityRole="listitem">
      <Text>{item.title}</Text>
    </View>
  ))}
</ScrollView>
```

Available roles: button, link, header, image, imagebutton, keyboardkey, text, adjustable, checkbox, radio, switch, togglebutton, summary, search, spinbutton, tablist, tab, tabpanel, progressbar, menu, menuitem, none.

## Screen Reader Navigation Order

Control the focus order so VoiceOver and TalkBack navigate logically through your app:

```typescript
import { View, Text, Button, TextInput, AccessibilityInfo } from "react-native";

export function RegistrationForm() {
  const emailInputRef = React.useRef<TextInput>(null);

  return (
    <View
      accessibilityViewIsModal={true} // Confine focus to this view
      accessible={true}
    >
      <Text accessibilityRole="header">Create Account</Text>

      {/* Navigation order: Email → Password → Submit */}
      <TextInput
        ref={emailInputRef}
        placeholder="Email address"
        accessibilityLabel="Email address"
      />

      <TextInput
        placeholder="Password"
        secureTextEntry={true}
        accessibilityLabel="Password"
      />

      <Button
        title="Create Account"
        accessibilityLabel="Create account"
        onPress={handleSignUp}
      />

      {/* Auto-focus first input for screen reader users */}
      React.useEffect(() => {
        if (emailInputRef.current) {
          AccessibilityInfo.setAccessibilityFocus(
            findNodeHandle(emailInputRef.current)
          );
        }
      }, []);
    </View>
  );
}
```

**accessibilityViewIsModal**: When `true`, focus cannot leave this view. Essential for modals, dialogs, and overlays.

## Dynamic Type: Responsive Font Sizing

Users with vision impairment rely on large text. React Native supports dynamic text scaling via the `fontScale` property:

```typescript
import { useWindowDimensions, Text, StyleSheet } from "react-native";

export function AccessibleText({ children, variant = "body" }) {
  const fontScale = useWindowDimensions().fontScale;

  const styles = StyleSheet.create({
    heading: {
      fontSize: 24 * fontScale,
      fontWeight: "bold",
      marginBottom: 16,
    },
    body: {
      fontSize: 16 * fontScale,
      lineHeight: 24 * fontScale,
    },
    caption: {
      fontSize: 14 * fontScale,
    },
  });

  const styleKey = variant as keyof typeof styles;
  return <Text style={styles[styleKey]}>{children}</Text>;
}
```

React Native's `Text` component defaults to `allowFontScaling={true}`, meaning text automatically scales with system settings. Never set `allowFontScaling={false}` unless absolutely necessary.

```typescript
// ✅ Good: Text scales with user preferences
<Text allowFontScaling={true}>
  This text adjusts to the user's accessibility settings
</Text>

// ❌ Avoid: Text ignores user preferences
<Text allowFontScaling={false}>
  This text is fixed size (bad for accessibility)
</Text>
```

Test with maximum font scaling (Settings > Accessibility > Larger Sizes) to ensure layouts don't break.

## Color Contrast: WCAG Standards

Text and background colors must have sufficient contrast. About 8% of people have color vision deficiency; they rely on contrast to distinguish content.

**WCAG Contrast Ratios:**

- **AA (Minimum)**: 4.5:1 for normal text, 3:1 for large text (18pt+)
- **AAA (Enhanced)**: 7:1 for normal text, 4.5:1 for large text

Check contrast with online tools like [WebAIM Color Contrast Checker](https://webaim.org/resources/contrastchecker/).

```typescript
import { View, Text, StyleSheet } from "react-native";

const styles = StyleSheet.create({
  // ❌ Poor contrast: #777 on #f5f5f5 is only 4.48:1
  badExample: {
    color: "#777777",
    backgroundColor: "#f5f5f5",
  },

  // ✅ Good contrast: #333 on #f5f5f5 is 10:1
  goodExample: {
    color: "#333333",
    backgroundColor: "#f5f5f5",
  },

  // ✅ High contrast: #000 on #fff is 21:1
  excellentExample: {
    color: "#000000",
    backgroundColor: "#ffffff",
  },
});

export function AccessibleText() {
  return (
    <View>
      <Text style={styles.goodExample}>Better contrast = better readability</Text>
      <Text style={styles.excellentExample}>Excellent contrast for all users</Text>
    </View>
  );
}
```

Test contrast in both light and dark modes.

## Color Blindness: Don't Rely on Color Alone

About 1 in 12 men and 1 in 200 women have color blindness. Never communicate information using color alone.

```typescript
import { View, Text } from "react-native";

// ❌ Bad: Only red color indicates error
<View style={{ backgroundColor: "#f44336", padding: 12 }}>
  <Text>Error</Text> {/* Text is crucial; color alone isn't enough */}
</View>

// ✅ Good: Color + icon + text
<View
  style={{
    backgroundColor: "#f44336",
    padding: 12,
    flexDirection: "row",
    alignItems: "center",
  }}
>
  <Text style={{ fontSize: 20, marginRight: 8 }}>✕</Text>
  <Text style={{ color: "#fff", fontWeight: "600" }}>Error: Please try again</Text>
</View>

// ✅ Better: Color + icon + text + border pattern
<View
  style={{
    backgroundColor: "#f44336",
    padding: 12,
    flexDirection: "row",
    alignItems: "center",
    borderLeftWidth: 4,
    borderLeftColor: "#c62828",
  }}
>
  <Text style={{ fontSize: 20, marginRight: 8 }}>✕</Text>
  <Text style={{ color: "#fff", fontWeight: "600" }}>Error</Text>
</View>
```

Use the [Color Blindness Simulator](https://www.color-blindness.com/) to preview how your app looks to people with color vision deficiency.

## Touch Target Size: 44×44 Points

Mobile accessibility requires adequate touch targets. Apple and Google recommend a minimum of 44×44 points for all interactive elements:

```typescript
import { TouchableOpacity, View, Text } from "react-native";

// ❌ Too small: 24×24 is difficult to tap accurately
<TouchableOpacity
  style={{ width: 24, height: 24 }}
  onPress={() => {}}
>
  <Text>Delete</Text>
</TouchableOpacity>

// ✅ Correct: 44×44 minimum
<TouchableOpacity
  style={{
    width: 44,
    height: 44,
    justifyContent: "center",
    alignItems: "center",
  }}
  onPress={() => {}}
>
  <Text>Delete</Text>
</TouchableOpacity>

// ✅ Smart: Small visual icon, large touch area using hitSlop
<TouchableOpacity
  style={{ width: 24, height: 24, justifyContent: "center", alignItems: "center" }}
  hitSlop={{ top: 10, bottom: 10, left: 10, right: 10 }} // Extends touch by 10pt on each side
  onPress={() => {}}
>
  <Text style={{ fontSize: 12 }}>✕</Text>
</TouchableOpacity>
```

`hitSlop` expands the touch-sensitive area without changing the visual size. Perfect for icon buttons where visual size must stay small.

<Callout type="info">
**Spacing**: In addition to individual touch targets, maintain at least 8 points of spacing between interactive elements to prevent accidental taps.
</Callout>

## Testing with VoiceOver (iOS)

Enable VoiceOver to test screen reader compatibility:

1. **Xcode Simulator**:
   - Device → More Developer Tools → Accessibility Inspector
   - Or: Settings → Accessibility → VoiceOver → ON

2. **Physical Device**:
   - Settings → Accessibility → VoiceOver → ON

**VoiceOver Gestures**:

- **Single tap**: Select and read element
- **Double tap**: Activate element (button press)
- **Two-finger swipe right**: Next page
- **Two-finger swipe left**: Previous page
- **Rotor (two-finger rotate)**: Jump between headings, links, etc.

Navigate your entire app with VoiceOver enabled. Verify:
- All interactive elements are selectable
- Labels are clear and descriptive
- Navigation order is logical
- Buttons and links activate correctly

## Testing with TalkBack (Android)

Enable TalkBack for Android screen reader testing:

1. **Emulator or Device**:
   - Settings → Accessibility → TalkBack → ON

2. **TalkBack Gestures**:
   - **Single tap**: Select element
   - **Double tap**: Activate element
   - **Swipe right**: Next element
   - **Swipe left**: Previous element
   - **Two-finger swipe right**: Next page

Test the same scenarios as VoiceOver to ensure consistency across platforms.

## Automated Accessibility Testing

Use Jest and React Native Testing Library to catch accessibility issues early:

```typescript
import { render, screen } from "@testing-library/react-native";

test("Add button has correct label", () => {
  render(<AddButton />);
  const button = screen.getByLabelText("Create new task");
  expect(button).toBeVisible();
});

test("Modal confines focus with viewIsModal", () => {
  const { getByTestId } = render(<Modal testID="modal" />);
  const modal = getByTestId("modal");
  expect(modal).toHaveAccessibilityViewIsModal(true);
});

test("Text scales with fontScale", () => {
  render(<AccessibleText>Scalable text</AccessibleText>);
  const text = screen.getByText("Scalable text");
  expect(text).toHaveStyle({ fontSize: expect.any(Number) });
});
```

Automated tests catch missing labels and incorrect roles, but manual VoiceOver/TalkBack testing is essential for real-world usability.

<Callout type="warning">
**Manual testing is mandatory**: Automated tests cannot replicate the actual experience of navigating with a screen reader. Always test with VoiceOver and TalkBack enabled before submitting to the App Store.
</Callout>

## App Store Review Checklist

Apple's accessibility review focuses on these areas:

1. **Screen Reader Support**
   - [ ] All interactive elements have `accessibilityLabel`
   - [ ] VoiceOver navigation order is logical
   - [ ] Decorative images have `accessible={false}`
   - [ ] No elements are permanently hidden from screen readers

2. **Dynamic Type**
   - [ ] Text scales at maximum font size
   - [ ] Layouts don't break at large text sizes
   - [ ] Minimum line height is at least 1.2× font size

3. **Color Contrast**
   - [ ] All text meets WCAG AA (4.5:1) minimum
   - [ ] Dark mode contrast is also adequate
   - [ ] Charts and diagrams use patterns in addition to color

4. **Touch Targets**
   - [ ] All buttons/links are 44×44pt or larger
   - [ ] Spacing between targets prevents accidental taps

5. **Keyboard Navigation**
   - [ ] All functionality is accessible via keyboard
   - [ ] Tab order is logical
   - [ ] Focus indicators are visible

6. **App Store Connect Metadata**
   - Complete the "Accessibility" section:
     ```
     VoiceOver: Yes
     Dynamic Type: Yes
     Larger Sizes: Yes
     Color Blindness Friendly: Yes
     ```

## Common Rejection Reasons

1. **Screen Reader Incompatibility**: Interactive elements don't respond to VoiceOver
2. **Insufficient Contrast**: Text fails to meet WCAG AA ratio
3. **Small Touch Targets**: Buttons are smaller than 44×44pt
4. **Missing Labels**: Elements lack `accessibilityLabel`
5. **Focus Trap**: VoiceOver gets stuck on specific elements
6. **Incomplete Metadata**: Accessibility section in App Store Connect is empty

## Implementation Checklist

- [ ] All buttons have `accessibilityLabel` and `accessibilityRole="button"`
- [ ] Text inputs have `accessibilityRole="search"` or `"textbox"`
- [ ] Sections have headers with `accessibilityRole="header"`
- [ ] Decorative images have `accessible={false}`
- [ ] All text uses `allowFontScaling={true}` (default)
- [ ] Line height is at least 1.2× font size for large text
- [ ] Color contrast ratio is 4.5:1 or higher
- [ ] Touch targets are 44×44pt minimum (including `hitSlop`)
- [ ] Modals use `accessibilityViewIsModal={true}`
- [ ] VoiceOver navigation order is logical
- [ ] Dark mode has adequate contrast
- [ ] App Store metadata includes accessibility features
