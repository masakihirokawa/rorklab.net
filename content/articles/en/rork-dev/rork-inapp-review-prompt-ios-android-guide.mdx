## Why In-App Review Prompts Matter

Your app's rating on the App Store or Google Play has a direct impact on how many people download it. Apps with a rating of 4.0 or higher rank better in search results and instill confidence in potential users. Yet most satisfied users never take the time to leave a review — only a small fraction do so spontaneously.

That's where **in-app review prompts** come in. By surfacing the OS-native review dialog at the right moment — without forcing users to leave the app — you can dramatically increase the volume of reviews you receive.

This guide walks you through implementing in-app review prompts for both iOS and Android in a Rork app, with practical code examples and best practices for maximizing your rating scores.

## iOS Implementation — SKStoreReviewController

### How It Works

On iOS, in-app reviews are powered by **SKStoreReviewController** from the StoreKit framework. A few important things to understand upfront:

- iOS allows the prompt to appear **at most 3 times per year** for the same app version
- The OS makes the final call on whether to show the dialog — calling the API doesn't guarantee it will appear
- The dialog always appears in simulators and development builds (great for testing)

For Rork apps built with Expo, the `expo-store-review` package provides a clean abstraction over the native APIs.

```typescript
// src/hooks/useReviewPrompt.ts
import * as StoreReview from 'expo-store-review';
import AsyncStorage from '@react-native-async-storage/async-storage';

const LAST_PROMPT_DATE_KEY = 'last_review_prompt_date';

/**
 * Hook for triggering an in-app review request
 */
export function useReviewPrompt() {
  const requestReview = async () => {
    try {
      const isAvailable = await StoreReview.isAvailableAsync();
      if (!isAvailable) return;

      // Request the review dialog (OS decides whether to show it)
      await StoreReview.requestReview();

      // Record when we last prompted
      await AsyncStorage.setItem(
        LAST_PROMPT_DATE_KEY,
        new Date().toISOString()
      );
    } catch (error) {
      console.error('Review prompt error:', error);
    }
  };

  return { requestReview };
}
```

### Smart Trigger Conditions

```typescript
// src/utils/reviewTrigger.ts
import AsyncStorage from '@react-native-async-storage/async-storage';

const MIN_LAUNCH_COUNT = 5;          // At least 5 app launches
const MIN_DAYS_SINCE_INSTALL = 3;    // 3 days after install
const MIN_DAYS_BETWEEN_PROMPTS = 60; // 60 days between prompts

/**
 * Determines whether we should show the review prompt
 * @returns true if conditions are met, false to skip
 */
export async function shouldShowReviewPrompt(): Promise<boolean> {
  // 1. Launch count check
  const launchCountStr = await AsyncStorage.getItem('app_launch_count');
  const launchCount = parseInt(launchCountStr || '0', 10);
  if (launchCount < MIN_LAUNCH_COUNT) return false;

  // 2. Days since install check
  const installDateStr = await AsyncStorage.getItem('app_install_date');
  if (installDateStr) {
    const installDate = new Date(installDateStr);
    const daysSinceInstall =
      (Date.now() - installDate.getTime()) / (1000 * 60 * 60 * 24);
    if (daysSinceInstall < MIN_DAYS_SINCE_INSTALL) return false;
  }

  // 3. Time since last prompt check
  const lastPromptStr = await AsyncStorage.getItem('last_review_prompt_date');
  if (lastPromptStr) {
    const lastPromptDate = new Date(lastPromptStr);
    const daysSinceLastPrompt =
      (Date.now() - lastPromptDate.getTime()) / (1000 * 60 * 60 * 24);
    if (daysSinceLastPrompt < MIN_DAYS_BETWEEN_PROMPTS) return false;
  }

  return true; // All conditions satisfied
}
```

### Wiring It Into a Screen

```tsx
// src/screens/HomeScreen.tsx (relevant excerpt)
import { useEffect } from 'react';
import { useReviewPrompt } from '../hooks/useReviewPrompt';
import { shouldShowReviewPrompt } from '../utils/reviewTrigger';
import AsyncStorage from '@react-native-async-storage/async-storage';

export function HomeScreen() {
  const { requestReview } = useReviewPrompt();

  useEffect(() => {
    trackLaunch();
    checkAndRequestReview();
  }, []);

  const trackLaunch = async () => {
    const current = parseInt(
      (await AsyncStorage.getItem('app_launch_count')) || '0', 10
    );
    await AsyncStorage.setItem('app_launch_count', String(current + 1));
  };

  const checkAndRequestReview = async () => {
    const should = await shouldShowReviewPrompt();
    if (should) {
      // Add a short delay so it doesn't feel abrupt
      setTimeout(() => requestReview(), 2000);
    }
  };

  return (
    // ... your regular UI
  );
}
```

## Android Implementation — Google Play In-App Review API

### How It Works on Android

The Google Play In-App Review API works similarly to iOS — it displays a native review sheet inside the app. Key constraints to be aware of:

- Google imposes a **rate limit per user** (the exact number is undisclosed by Google)
- The dialog won't appear on emulators without a real Google Play account configured

The good news: `expo-store-review` abstracts over both platforms, so you can use the **exact same API** for iOS and Android.

```typescript
// src/utils/reviewHelper.ts
import * as StoreReview from 'expo-store-review';
import { Platform } from 'react-native';

/**
 * Triggers in-app review on both iOS and Android:
 * - iOS: SKStoreReviewController
 * - Android: Google Play In-App Review API
 */
export async function triggerInAppReview(): Promise<void> {
  try {
    const isAvailable = await StoreReview.isAvailableAsync();

    if (isAvailable) {
      await StoreReview.requestReview();
      console.log(`[${Platform.OS}] Review prompt requested`);
    } else {
      // Fallback: open the store page directly
      await StoreReview.openStoreReviewPage();
      // Expected: the App Store or Google Play opens to the app's review page
    }
  } catch (e) {
    console.warn('Review prompt skipped:', e);
  }
}
```

## Best Practices for Maximizing Review Quality

### ✅ Choose the Right Moment

The most important rule: prompt users **right after a positive experience**. Asking at the wrong time can backfire.

- **Good moments**: After completing a task, reaching a milestone, achieving a goal, or finishing a session
- **Bad moments**: On first launch, after an error, during checkout, or mid-tutorial

```typescript
// Example: trigger after a user completes their 3rd, 10th, or 50th task
async function onTaskCompleted(taskCount: number) {
  if ([3, 10, 50].includes(taskCount)) {
    const should = await shouldShowReviewPrompt();
    if (should) {
      await triggerInAppReview();
    }
  }
}
```

### ✅ Target Engaged Users

Showing the prompt to everyone equally isn't as effective as targeting users who are genuinely invested in your app. Engaged users tend to leave higher ratings.

```typescript
async function isEngagedUser(): Promise<boolean> {
  const sessionCount = await getSessionCount();
  const totalUsageMinutes = await getTotalUsageMinutes();
  const hasCompletedOnboarding = await AsyncStorage.getItem('onboarding_done');

  return (
    sessionCount >= 5 &&
    totalUsageMinutes >= 30 &&
    hasCompletedOnboarding === 'true'
  );
}
```

### ✅ Use a Custom Satisfaction Gate

A common and effective technique is to show a simple **"Are you enjoying the app?"** screen before surfacing the OS dialog. Happy users go to the official review flow; unhappy users are routed to an internal feedback form — keeping negative sentiment out of the store.

```tsx
function SatisfactionGate() {
  const [step, setStep] = useState<'ask' | 'review' | 'feedback'>('ask');

  if (step === 'ask') {
    return (
      <View>
        <Text>Are you enjoying the app?</Text>
        <Button
          title="Yes, I love it! ❤️"
          onPress={() => {
            setStep('review');
            triggerInAppReview(); // Route to official review dialog
          }}
        />
        <Button
          title="I have some suggestions"
          onPress={() => setStep('feedback')} // Route to internal feedback form
        />
      </View>
    );
  }
  // ... handle review and feedback steps
}
```

## Common Issues and Fixes

### Issue 1: `isAvailableAsync()` Returns `false`

**Cause**: You're running on an Android emulator without a real Google Play account, or you haven't built a production/TestFlight build yet.

**Fix**: Test Android in-app reviews using a build distributed via Google Play Internal Testing on a real device.

### Issue 2: `requestReview()` Is Called But Nothing Appears

**Cause**: The OS rate limit has been reached (normal behavior on iOS after 3 prompts per app version per year).

**Fix**: This is expected. Always have `openStoreReviewPage()` as a fallback for users who want to leave a review but didn't see the in-app dialog.

### Issue 3: App Crashes on Android When Calling the API

**Cause**: Outdated Google Play Services on the device, or a mismatch with the Expo SDK version.

**Fix**: Check your `minSdkVersion` and ensure you're on the latest compatible version of `expo-store-review`.

## FAQ

**Q: Can I use the same code for both iOS and Android?**

A: Yes. `expo-store-review` provides a unified API. Under the hood, it calls SKStoreReviewController on iOS and the Google Play In-App Review API on Android.

**Q: The dialog doesn't appear on Android emulators — is something broken?**

A: No, this is expected. The Google Play In-App Review API requires a real device with Google Play Services. Use Internal Testing on a real Android device to test.

**Q: Can I detect whether the user actually submitted a review?**

A: No. Both iOS and Android intentionally hide this information to protect user privacy. You can't tell if the user tapped "Submit" or dismissed the dialog. The best approach is to monitor your store review counts in the developer console a few days after the prompt window.
