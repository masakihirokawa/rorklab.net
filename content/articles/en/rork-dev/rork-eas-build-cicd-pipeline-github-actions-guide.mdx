When you're building one app with Rork, manually archiving in Xcode and uploading to App Store Connect feels manageable. You do it once a week, maybe twice. It's annoying, but not a crisis.

When you're managing three or four apps simultaneously, that same workflow becomes a genuine productivity problem. I hit that wall about six months ago — sat down one Monday morning, looked at my calendar, and realized I'd spent over six hours the previous week on build logistics alone. Not on features. Not on user research. On clicking "Archive" and waiting.

That's when I committed to properly automating the whole pipeline. And I want to share what I learned, including the parts the docs don't cover well.

The end state we're building toward: merge to `main` and your app is in TestFlight within 20–30 minutes, automatically, while you're already working on the next thing. No Xcode required. No manual uploads. No certificate anxiety.

## Understanding What "CI/CD" Actually Means Here

The term gets used loosely, so let me be specific about what we're building:

**Continuous Integration (CI)** means every code push automatically triggers a build. If something is broken, you find out immediately — not when you try to release next week.

**Continuous Delivery (CD)** means every successful build on certain branches automatically gets delivered to the appropriate testing or distribution channel. The human decision is "merge this PR" — everything after that is automated.

For a Rork-based React Native app, this specifically means:
- Feature branches → development builds (internal distribution, quick feedback loops)
- `develop` branch → staging builds (TestFlight internal group, QA testing)
- `main` branch → production builds (TestFlight + Google Play internal testing, ready for review submission)

## Tools in This Stack and Why

**EAS Build** (Expo Application Services) handles the actual compilation and signing. It runs builds in Expo's cloud infrastructure, which means you don't need a Mac to build iOS apps, and you don't need to maintain local build environments. EAS also manages Apple certificates and provisioning profiles automatically — one of the most painful parts of iOS development, just gone.

**GitHub Actions** handles the orchestration. It watches your repository, detects pushes to specific branches, and triggers the appropriate EAS Build workflow. It also manages secrets securely and sends notifications when builds complete or fail.

**EAS Submit** (included in EAS) handles app store submission. Once a build is complete, `eas submit` authenticates with App Store Connect or Google Play and uploads the artifact automatically.

**EAS Update** handles OTA (over-the-air) updates for JavaScript code changes — no store submission required, delivers to users in minutes.

## Full Pipeline Architecture

Here's what the complete system looks like:

```
feature/my-feature → develop (PR merge)
  └── GitHub Actions: development-build.yml
      └── EAS Build: development profile
          └── QR code / download link posted to PR comment

develop → main (PR merge)
  └── GitHub Actions: staging-build.yml
      └── EAS Build: staging profile
          └── TestFlight (internal testers group)
          └── Google Play (internal testing track)

main (direct or PR merge)
  └── GitHub Actions: production-build.yml
      └── EAS Build: production profile
          └── TestFlight (external testers / App Store submission)
          └── Google Play (internal → promote to alpha/production)
          └── Slack notification: success or failure
```

The key principle: **branch determines environment**. No manual environment selection, no forgetting to switch profiles, no "wait, did I build the staging or production version?"

## Step 1: Account Setup and EAS CLI Installation

### What you need before starting

- Expo account (free at [expo.dev](https://expo.dev))
- Apple Developer Program membership ($99/year — required for TestFlight distribution)
- Google Play Console access ($25 one-time fee)
- GitHub repository containing your Rork-exported React Native code
- Slack workspace with Incoming Webhooks enabled (optional but recommended)

### Installing EAS CLI

```bash
# Install globally so it's available from any directory
npm install -g eas-cli

# Authenticate with your Expo account
eas login

# Confirm the login worked
eas whoami
# Output: your-email@example.com
```

### Initializing EAS in your project

From the root of your Rork project:

```bash
# This creates eas.json and links the project to your Expo account
eas build:configure
```

You'll be asked whether you want to configure the project for iOS, Android, or both. Select both. You can also be asked about build profiles — leave the defaults for now; we'll replace them entirely in the next step.

## Step 2: Designing eas.json for Three Environments

This is the single most important configuration decision in the entire pipeline. I'd estimate half the CI/CD problems I see in the Rork community trace back to a poorly structured `eas.json`. Spend time getting this right.

```json
{
  "cli": {
    "version": ">= 10.0.0"
  },
  "build": {
    "development": {
      "developmentClient": true,
      "distribution": "internal",
      "ios": {
        "simulator": false,
        "resourceClass": "m-medium"
      },
      "android": {
        "buildType": "apk",
        "resourceClass": "medium"
      },
      "env": {
        "APP_ENV": "development",
        "API_URL": "https://dev-api.your-app.com"
      }
    },
    "staging": {
      "distribution": "internal",
      "ios": {
        "resourceClass": "m-medium"
      },
      "android": {
        "buildType": "apk",
        "resourceClass": "medium"
      },
      "env": {
        "APP_ENV": "staging",
        "API_URL": "https://staging-api.your-app.com"
      }
    },
    "production": {
      "autoIncrement": true,
      "ios": {
        "resourceClass": "m-large"
      },
      "android": {
        "buildType": "app-bundle",
        "resourceClass": "large"
      },
      "env": {
        "APP_ENV": "production",
        "API_URL": "https://api.your-app.com"
      }
    }
  },
  "submit": {
    "production": {
      "ios": {
        "appleId": "your-apple-id@example.com",
        "ascAppId": "1234567890",
        "appleTeamId": "XXXXXXXXXX"
      },
      "android": {
        "serviceAccountKeyPath": "./google-services/play-store-key.json",
        "track": "internal"
      }
    }
  }
}
```

### Why `m-large` for production?

The `resourceClass` setting determines the VM size used for your build. Development and staging builds use `m-medium` — fast enough and cheaper. Production iOS builds use `m-large` because Apple Silicon (`m-` prefix) significantly reduces build times, and for production you want speed and reliability over cost savings.

### The non-negotiable: secrets don't belong in eas.json

`API_URL` is fine in `eas.json` because it's not sensitive. But API keys, service role keys, webhook secrets, and payment keys must go somewhere else. `eas.json` lives in your repository — treat it as public.

For sensitive values, use EAS Secrets:

```bash
# Store a secret in EAS (encrypted, accessible during builds)
eas secret:create \
  --scope project \
  --name STRIPE_SECRET_KEY \
  --value "sk_live_your_key_here"

# Separate secrets for staging
eas secret:create \
  --scope project \
  --name STRIPE_SECRET_KEY_STAGING \
  --value "sk_test_your_staging_key_here"

# Verify what's stored (values are not shown)
eas secret:list
```

## Step 3: Converting app.json to Dynamic app.config.js

Rork exports a static `app.json`. For multi-environment support with different bundle IDs, app names, and icons, you need to convert this to a JavaScript configuration that reads environment variables at build time.

Rename `app.json` to `app.json.backup`, then create `app.config.js`:

```javascript
// app.config.js
const IS_DEV = process.env.APP_ENV === "development";
const IS_STAGING = process.env.APP_ENV === "staging";

export default {
  name: IS_DEV ? "MyApp (Dev)" : IS_STAGING ? "MyApp (Staging)" : "MyApp",
  slug: "my-app",
  version: "1.0.0",
  orientation: "portrait",
  icon: IS_DEV ? "./assets/icon-dev.png" : "./assets/icon.png",
  splash: {
    image: "./assets/splash.png",
    resizeMode: "contain",
    backgroundColor: "#ffffff",
  },
  ios: {
    bundleIdentifier: IS_DEV
      ? "com.yourcompany.myapp.dev"
      : IS_STAGING
      ? "com.yourcompany.myapp.staging"
      : "com.yourcompany.myapp",
    buildNumber: "1",
    supportsTablet: false,
  },
  android: {
    package: IS_DEV
      ? "com.yourcompany.myapp.dev"
      : IS_STAGING
      ? "com.yourcompany.myapp.staging"
      : "com.yourcompany.myapp",
    versionCode: 1,
    adaptiveIcon: {
      foregroundImage: "./assets/adaptive-icon.png",
      backgroundColor: "#ffffff",
    },
  },
  extra: {
    apiUrl: process.env.API_URL,
    appEnv: process.env.APP_ENV,
    eas: {
      projectId: "your-eas-project-id-from-eas-build-configure",
    },
  },
  plugins: ["expo-router"],
};
```

Separate bundle identifiers mean you can have dev, staging, and production versions installed simultaneously on the same device. This is more useful than it sounds — when testing, you can switch between environments without uninstalling and reinstalling.

## Step 4: GitHub Actions Workflows in Detail

Create the `.github/workflows/` directory in your project root. Each workflow file is a separate YAML file.

### Production build and submission

```yaml
# .github/workflows/production-build.yml
name: Production Build & Submit

on:
  push:
    branches:
      - main

jobs:
  build-and-submit-ios:
    name: iOS Production Build
    runs-on: ubuntu-latest
    timeout-minutes: 60
    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Setup Node.js 20
        uses: actions/setup-node@v4
        with:
          node-version: "20"
          cache: "npm"

      - name: Install dependencies
        run: npm ci

      - name: Setup EAS
        uses: expo/expo-github-action@v8
        with:
          eas-version: latest
          token: ${{ secrets.EXPO_TOKEN }}

      - name: Build iOS
        run: |
          eas build \
            --platform ios \
            --profile production \
            --non-interactive
        env:
          STRIPE_SECRET_KEY: ${{ secrets.STRIPE_SECRET_KEY }}
          SUPABASE_SERVICE_KEY: ${{ secrets.SUPABASE_SERVICE_KEY }}

      - name: Submit to TestFlight
        run: |
          eas submit \
            --platform ios \
            --profile production \
            --latest \
            --non-interactive
        env:
          EXPO_APPLE_APP_SPECIFIC_PASSWORD: ${{ secrets.APPLE_APP_SPECIFIC_PASSWORD }}

  build-and-submit-android:
    name: Android Production Build
    runs-on: ubuntu-latest
    timeout-minutes: 60
    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Setup Node.js 20
        uses: actions/setup-node@v4
        with:
          node-version: "20"
          cache: "npm"

      - name: Install dependencies
        run: npm ci

      - name: Write Google Play service account key
        run: |
          mkdir -p ./google-services
          echo '${{ secrets.GOOGLE_PLAY_SERVICE_ACCOUNT_JSON }}' \
            > ./google-services/play-store-key.json

      - name: Setup EAS
        uses: expo/expo-github-action@v8
        with:
          eas-version: latest
          token: ${{ secrets.EXPO_TOKEN }}

      - name: Build Android
        run: |
          eas build \
            --platform android \
            --profile production \
            --non-interactive
        env:
          STRIPE_SECRET_KEY: ${{ secrets.STRIPE_SECRET_KEY }}
          SUPABASE_SERVICE_KEY: ${{ secrets.SUPABASE_SERVICE_KEY }}

      - name: Submit to Google Play
        run: |
          eas submit \
            --platform android \
            --profile production \
            --latest \
            --non-interactive

  notify-slack:
    name: Slack Notification
    runs-on: ubuntu-latest
    needs: [build-and-submit-ios, build-and-submit-android]
    if: always()
    steps:
      - name: Notify result
        uses: slackapi/slack-github-action@v1.26.0
        with:
          payload: |
            {
              "text": "Production build complete\niOS: ${{ needs.build-and-submit-ios.result }}\nAndroid: ${{ needs.build-and-submit-android.result }}\nBy: ${{ github.actor }}\nCommit: ${{ github.sha }}"
            }
        env:
          SLACK_WEBHOOK_URL: ${{ secrets.SLACK_WEBHOOK_URL }}
```

### Staging build workflow

```yaml
# .github/workflows/staging-build.yml
name: Staging Build

on:
  push:
    branches:
      - develop

jobs:
  build-staging:
    name: Staging Build (iOS + Android)
    runs-on: ubuntu-latest
    timeout-minutes: 60
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: "20"
          cache: "npm"
      - run: npm ci
      - uses: expo/expo-github-action@v8
        with:
          eas-version: latest
          token: ${{ secrets.EXPO_TOKEN }}
      - name: Build staging
        run: |
          eas build \
            --platform all \
            --profile staging \
            --non-interactive
        env:
          STRIPE_SECRET_KEY: ${{ secrets.STRIPE_SECRET_KEY_STAGING }}
          SUPABASE_SERVICE_KEY: ${{ secrets.SUPABASE_SERVICE_KEY_STAGING }}
```

### OTA update workflow (for JS-only changes)

```yaml
# .github/workflows/ota-update.yml
name: OTA Update

on:
  push:
    branches:
      - main
    paths:
      - "app/**"
      - "components/**"
      - "hooks/**"
      - "lib/**"
      - "constants/**"
      - "utils/**"

jobs:
  update:
    name: Publish OTA Update
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: "20"
          cache: "npm"
      - run: npm ci
      - uses: expo/expo-github-action@v8
        with:
          eas-version: latest
          token: ${{ secrets.EXPO_TOKEN }}
      - name: Publish to production channel
        run: |
          eas update \
            --channel production \
            --message "Update: ${{ github.event.head_commit.message }}" \
            --auto
```

The `paths:` filter is important — it prevents OTA updates from triggering when you change configuration files that don't affect the JS bundle. OTA only works for JavaScript code changes; native module additions require a full build.

## Step 5: Registering GitHub Secrets

Go to your repository on GitHub → **Settings → Secrets and variables → Actions → New repository secret**.

Add each of the following:

**Expo authentication:**
- `EXPO_TOKEN`: In Expo dashboard → Account Settings → Access tokens. Critical: choose **No expiry**. Expiring tokens are a common cause of mysterious CI failures a month after setup.

**Apple submission:**
- `APPLE_APP_SPECIFIC_PASSWORD`: Visit [appleid.apple.com](https://appleid.apple.com) → Sign-In and Security → App-Specific Passwords → Generate. This is separate from your Apple ID password and required for automated App Store Connect uploads.

**Google Play:**
- `GOOGLE_PLAY_SERVICE_ACCOUNT_JSON`: Paste the entire contents of the JSON key file you downloaded from Google Cloud Console. Paste it exactly — including the curly braces.

**App secrets:**
- `STRIPE_SECRET_KEY` and `STRIPE_SECRET_KEY_STAGING`
- `SUPABASE_SERVICE_KEY` and `SUPABASE_SERVICE_KEY_STAGING`
- Any other sensitive values your app needs at build time

**Notifications:**
- `SLACK_WEBHOOK_URL`: Create an Incoming Webhook in your Slack app settings

**Naming requirement:** GitHub Secrets only accept `A-Z`, `0-9`, and `_`. A hyphen in a secret name like `STRIPE-SECRET-KEY` causes it to silently evaluate to empty string. You'll see authentication errors with no mention of the naming issue.

## Step 6: iOS Certificate Management via EAS

Add `credentialsSource: "remote"` to each iOS build profile:

```json
{
  "build": {
    "development": {
      "ios": { "credentialsSource": "remote" }
    },
    "staging": {
      "ios": { "credentialsSource": "remote" }
    },
    "production": {
      "ios": { "credentialsSource": "remote" }
    }
  }
}
```

EAS handles all Apple Developer Portal interactions: creating certificates, creating provisioning profiles, registering devices, renewing certificates before they expire. This replaces an entire Fastlane Match setup.

The first time, Apple's 2FA requires a human in the loop. Run this locally once:

```bash
eas credentials

# Follow the prompts:
# Select platform: iOS
# Select build profile: production
# Choose: Set up credentials automatically
```

EAS will request the necessary credentials from Apple, store them encrypted in EAS infrastructure, and use them for every subsequent build including all CI builds.

## Step 7: Google Play API Configuration

Android store submission is more complex than iOS because it requires a Google service account with specific permissions. Here's the full sequence.

### Creating the service account

1. Sign in to [Google Cloud Console](https://console.cloud.google.com/) with your Play Console account
2. Create or select a project
3. Navigate to **APIs & Services → Library** → find "Google Play Android Developer API" → click Enable
4. Go to **Credentials → + Create Credentials → Service Account**
5. Enter a name like `play-store-ci` → click Create and Continue
6. Select **Owner** as the role → click Done
7. Click on the service account you just created
8. Go to the **Keys** tab → **Add Key → Create new key → JSON** → Download

### Granting Play Console access

1. Open Google Play Console → your app → **Users and permissions → Invite new users**
2. Enter the email from the JSON file (the `client_email` field)
3. Set access level: **Admin**
4. Select the specific app → Save

### The permission propagation delay

After inviting the service account, it can take anywhere from a few minutes to 24 hours for permissions to activate. If `eas submit` returns a 403 immediately after setup, wait and try again before troubleshooting the configuration.

To diagnose which type of error you have:

```bash
# Run locally for clearer error messages
eas submit --platform android --profile production --latest

# "403 Permission denied" → wait for permissions to propagate
# "401 invalid_grant" → regenerate the JSON key (it may be corrupted)
# "404 Application not found" → first APK must be uploaded manually
```

The first release must always be uploaded manually through the Play Console web interface. Automation only works for subsequent updates.

## Step 8: Environment-Aware Code in Your App

With three environments running, you need clean in-app separation of environment-specific configuration and behavior.

```typescript
// utils/config.ts
import Constants from "expo-constants";

type AppEnvironment = "development" | "staging" | "production";

const APP_ENV =
  (Constants.expoConfig?.extra?.appEnv as AppEnvironment) ?? "production";

export const Config = {
  env: APP_ENV,
  apiUrl: (Constants.expoConfig?.extra?.apiUrl as string) ?? "",
  isProduction: APP_ENV === "production",
  isDevelopment: APP_ENV === "development",
  isStaging: APP_ENV === "staging",
} as const;

// Type-safe debug logging — no-op in production
export const debugLog = (...args: unknown[]): void => {
  if (\!Config.isProduction) {
    // eslint-disable-next-line no-console
    console.log(`[${Config.env.toUpperCase()}]`, ...args);
  }
};
```

Connecting this to Sentry for environment-aware error tracking:

```typescript
// app/_layout.tsx
import * as Sentry from "@sentry/react-native";
import { Config } from "@/utils/config";

Sentry.init({
  dsn: process.env.EXPO_PUBLIC_SENTRY_DSN,
  environment: Config.env,
  // Capture all events in dev/staging; sample in production to control costs
  tracesSampleRate: Config.isProduction ? 0.1 : 1.0,
  // Disable Sentry entirely in development to reduce noise
  enabled: \!Config.isDevelopment,
  beforeSend(event) {
    // Strip any accidentally logged sensitive fields
    if (event.request?.data) {
      delete event.request.data.password;
      delete event.request.data.token;
    }
    return event;
  },
});
```

## Step 9: Branch Protection Rules

Without branch protection, the automation doesn't enforce anything — someone could push directly to `main` and bypass the pipeline. Configure branch protection in your GitHub repository:

**Settings → Branches → Add branch protection rule**

For `main`:
- Require pull request reviews before merging: ✓ (1 reviewer minimum)
- Require status checks to pass: ✓
  - Add your GitHub Actions workflow names as required checks
- Require branches to be up to date: ✓
- Restrict who can push to matching branches: ✓

For `develop`:
- Require pull request reviews (optional for smaller teams)
- Require status checks: ✓

This ensures every code change goes through the pipeline and that the pipeline must succeed before merging. Combined with the automated CI/CD, you have a complete quality gate from feature branch through production.

## Step 10: Managing Build Costs

EAS Build uses a credit system. Here's how to stay within reasonable costs:

- Development builds on simulators are free
- Internal distribution builds use fewer credits than production builds
- `m-medium` resource class costs fewer credits than `m-large`

For a typical indie app with daily development builds:
- Keep development profile as `distribution: "internal"` (faster, cheaper)
- Only use `m-large` for production (where time and reliability matter most)
- Use OTA updates for JavaScript changes (no credits consumed)

To monitor usage: Expo dashboard → your account → Billing → Usage

If you're running multiple Rork apps and hitting limits, prioritize production automation (highest value) and keep development builds manual.

## Common Failures and Fixes

**① EXPO_TOKEN silently expires**

Tokens created with a 30-day expiry fail quietly — the error looks like authentication failure, not "token expired." Always create tokens with **No expiry** and update GitHub Secrets whenever you regenerate.

**② Build and submit race condition**

Without `--no-wait`, `eas build` blocks until complete before `eas submit` runs. If you add `--no-wait` for faster queue response, split build and submit into separate jobs with `needs:` dependency.

**③ Google Play key file not found**

If `eas.json` expects `./google-services/play-store-key.json` but the GitHub Actions `echo` step writes it elsewhere, you'll get a permission error pointing at the wrong thing. Add `ls -la ./google-services/` before the submit step to confirm file placement.

**④ Slack notification skipped on upstream failure**

The `if: always()` condition is required on any job that should run even when upstream jobs fail. Without it, a failed build means no notification — exactly when you most need one.

**⑤ Podfile.lock conflict breaks iOS builds**

After adding or updating npm packages with native dependencies, regenerate locally and commit:

```bash
cd ios && pod install && cd ..
git add ios/Podfile.lock
git commit -m "chore: update Podfile.lock"
```

**⑥ Android Java version mismatch**

```
Unsupported class file major version 65
```

Add Java 17 configuration to `android/app/build.gradle`:

```groovy
android {
  compileOptions {
    sourceCompatibility JavaVersion.VERSION_17
    targetCompatibility JavaVersion.VERSION_17
  }
  kotlinOptions {
    jvmTarget = "17"
  }
}
```

## Recommended Rollout Sequence

Setting up the full pipeline in one session is doable but stressful if something breaks. This sequence minimizes risk:

**Week 1:** Set up staging pipeline only. Get `develop → staging build` working reliably. Fix any issues in a low-stakes environment.

**Week 2:** Add the production build workflow. Keep `eas submit` commented out — trigger builds automatically, but review and submit manually.

**Week 3:** Enable automated `eas submit` for production once you've verified the builds are clean and the submission format is correct.

**Week 4:** Add OTA updates. By now you understand which changes are JS-only (safe for OTA) versus native (require full build).

This phased approach means you're never debugging multiple new systems simultaneously.

## Advanced: Multi-App Pipeline Configuration

If you're building several apps with Rork — which is increasingly common as AI-assisted development lowers the cost of app creation — you can share pipeline infrastructure across projects rather than setting it up from scratch each time.

### Shared GitHub Actions using reusable workflows

GitHub Actions supports calling reusable workflows from other repositories. Create a central repository (e.g., `your-org/ci-workflows`) with standardized build workflows:

```yaml
# In your-org/ci-workflows/.github/workflows/eas-build.yml
name: Reusable EAS Build

on:
  workflow_call:
    inputs:
      platform:
        required: true
        type: string
      profile:
        required: true
        type: string
    secrets:
      EXPO_TOKEN:
        required: true
      STRIPE_SECRET_KEY:
        required: true

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: "20"
          cache: "npm"
      - run: npm ci
      - uses: expo/expo-github-action@v8
        with:
          eas-version: latest
          token: ${{ secrets.EXPO_TOKEN }}
      - run: |
          eas build \
            --platform ${{ inputs.platform }} \
            --profile ${{ inputs.profile }} \
            --non-interactive
        env:
          STRIPE_SECRET_KEY: ${{ secrets.STRIPE_SECRET_KEY }}
```

Then in each app's repository, call the shared workflow:

```yaml
# app-1/.github/workflows/production-build.yml
name: Production Build

on:
  push:
    branches: [main]

jobs:
  build:
    uses: your-org/ci-workflows/.github/workflows/eas-build.yml@main
    with:
      platform: all
      profile: production
    secrets:
      EXPO_TOKEN: ${{ secrets.EXPO_TOKEN }}
      STRIPE_SECRET_KEY: ${{ secrets.STRIPE_SECRET_KEY }}
```

This approach means that when you improve the shared workflow — better error handling, updated node version, new notification format — every app benefits automatically.

### Per-app secrets vs. organization secrets

For multi-app setups, consider moving shared secrets (EXPO_TOKEN, Slack webhook) to GitHub organization secrets rather than repository secrets. Organization secrets can be granted to specific repositories, so you manage them in one place but they're accessible across your apps.

Repository secrets (STRIPE_SECRET_KEY, SUPABASE_SERVICE_KEY, APPLE_APP_SPECIFIC_PASSWORD) remain per-repository because they differ per app.

## Monitoring Build Health Over Time

Once CI/CD is running, you need visibility into build success rates and timing. EAS provides a build history dashboard, but you can also extract this data programmatically.

```bash
# Check build history from CLI
eas build:list --limit 10 --platform ios

# Filter by status
eas build:list --status finished --limit 5
eas build:list --status errored --limit 5
```

A useful practice is tracking your **build success rate** weekly. If production builds start failing more than 5% of the time, it's usually a signal that something in your dependency chain or native configuration changed and needs attention.

For Slack notifications, consider enriching the payload with build timing information:

```yaml
- name: Post detailed Slack notification
  uses: slackapi/slack-github-action@v1.26.0
  with:
    payload: |
      {
        "blocks": [
          {
            "type": "section",
            "text": {
              "type": "mrkdwn",
              "text": "*Production Build Complete*\n*iOS:* ${{ needs.build-ios.result }}\n*Android:* ${{ needs.build-android.result }}\n*Branch:* `${{ github.ref_name }}`\n*Author:* ${{ github.actor }}\n*Commit:* <${{ github.server_url }}/${{ github.repository }}/commit/${{ github.sha }}|${{ github.sha }}>"
            }
          }
        ]
      }
  env:
    SLACK_WEBHOOK_URL: ${{ secrets.SLACK_WEBHOOK_URL }}
    SLACK_WEBHOOK_TYPE: INCOMING_WEBHOOK
```

The block kit format renders cleanly in Slack and makes the commit link clickable — much more useful than plain text when you need to trace what changed.

## Handling Hotfixes Without Breaking the Pipeline

When you need to ship an urgent fix, the normal pipeline flow (feature → develop → main) can feel slow. Here's a hotfix strategy that maintains the pipeline's integrity while allowing fast turnaround:

1. Branch from `main` directly: `git checkout -b hotfix/crash-fix main`
2. Make the minimal fix, test locally
3. Open a PR directly to `main` (bypassing `develop`)
4. If branch protection requires a review, have a teammate approve quickly
5. Merge → production build triggers automatically

For truly critical production crashes, you can also use OTA updates if the fix is JavaScript-only:

```bash
# Emergency OTA from local machine (bypasses the full pipeline)
eas update \
  --channel production \
  --message "HOTFIX: crash on launch for users with empty profile" \
  --auto
```

OTA updates reach users faster than store submissions and faster than waiting for GitHub Actions. Keep this option in mind for urgent situations.

## Keeping the Pipeline Secure

A CI/CD pipeline that builds and submits to app stores is a high-value target if any of your secrets are compromised. A few practices that meaningfully reduce risk:

**Rotate secrets regularly.** Set a calendar reminder every 90 days to rotate EXPO_TOKEN, Apple app-specific passwords, and Google Play service account keys. Each service makes rotation straightforward.

**Limit service account permissions.** The Google Play service account initially gets "Owner" access. Once you've confirmed submission works, reduce it to the minimum required role for your use case.

**Use environments in GitHub Actions for production.** GitHub's Environments feature lets you require a manual approval step before production jobs run. This adds a human gate while keeping everything else automated:

```yaml
jobs:
  build:
    runs-on: ubuntu-latest
    environment: production  # Requires approval if configured
    steps:
      # ... your build steps
```

**Audit repository access.** Your CI/CD pipeline is only as secure as your repository access controls. Enable 2FA for all collaborators, and use branch protection rules to prevent direct pushes to `main`.

## What to Do When Builds Break

Builds will eventually fail. The question is how quickly you can diagnose and fix them. Here's a systematic approach:

**Step 1: Check the EAS dashboard first.** EAS often provides more detailed error messages than GitHub Actions. Navigate to `expo.dev/accounts/[your-account]/projects/[project-slug]/builds`.

**Step 2: Categorize the error.** Is it a JavaScript bundle error, a native compilation error, or a signing/certificate error? Each category has different debugging approaches.

**Step 3: Reproduce locally if possible.**

```bash
# Run the equivalent build locally
eas build --platform ios --profile production --local

# Local builds show full output and fail faster
```

**Step 4: Check recent changes.** Build failures after a merge often trace to a specific dependency update or configuration change. `git log --oneline -20` and reviewing what changed in `package.json` or native directories usually reveals the cause.

**Step 5: Search EAS community resources.** The Expo forums and Discord have solutions to most common build errors already documented. The error message is usually specific enough to search directly.

If a build fails in CI but succeeds locally, the most common culprits are: environment variables not set correctly in GitHub Secrets, a dependency that requires platform-specific setup (native modules), or a timing issue with credential access.

The goal is to get from "build failed" to "root cause identified" in under 15 minutes. With practice and the right tooling, this becomes routine rather than stressful.
