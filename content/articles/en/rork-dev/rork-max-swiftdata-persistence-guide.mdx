## What Is SwiftData?

SwiftData is Apple's next-generation data persistence framework, introduced at WWDC23. It dramatically reduces the boilerplate and configuration overhead that came with Core Data, replacing it with a declarative, macro-driven approach built on top of Swift's modern language features.

Since Rork Max generates native Swift apps, you can target iOS 17 and above to take full advantage of SwiftData. Compared to Core Data, the learning curve is significantly gentler, and integration with SwiftUI is seamless — making it an especially compelling choice for indie developers looking to ship apps quickly.

---

## How SwiftData Differs from Core Data

Both SwiftData and Core Data are first-party persistence solutions from Apple, but the developer experience is worlds apart.

Core Data required you to define entities in a `.xcdatamodeld` file and interact with data through specialized APIs like `NSManagedObjectContext` and `NSFetchRequest`. Managing configuration files, threading contexts, and understanding the underlying stack added significant complexity.

SwiftData takes a radically different approach. You define models by adding the `@Model` macro to a regular Swift class. Fetching data is handled through the `@Query` property wrapper, which binds directly to SwiftUI views. There are no configuration files to manage — your schema lives entirely in code.

If you have an existing Core Data project, migration is also possible since SwiftData uses Core Data's storage layer under the hood.

## Defining Models with the @Model Macro

The `@Model` macro is the cornerstone of SwiftData. It turns an ordinary Swift class into a persistent data model.

```swift
import SwiftData

@Model
class Book {
    var title: String
    var author: String
    var genre: String
    var finishedDate: Date?
    var memo: String
    var rating: Int
    var createdAt: Date

    init(
        title: String,
        author: String,
        genre: String = "Uncategorized",
        finishedDate: Date? = nil,
        memo: String = "",
        rating: Int = 0
    ) {
        self.title = title
        self.author = author
        self.genre = genre
        self.finishedDate = finishedDate
        self.memo = memo
        self.rating = rating
        self.createdAt = Date()
    }
}
```

No `.xcdatamodeld` file needed. The property types directly define the schema, which eliminates a whole class of mismatches between your code and your data structure.

### Customizing Attributes

You can add indexes, unique constraints, and Spotlight integration through attribute macros.

```swift
@Model
class Book {
    @Attribute(.unique) var isbn: String
    @Attribute(.spotlight) var title: String
    var author: String
    // ...
}
```

The `@Attribute(.unique)` constraint prevents duplicate records with the same value, while `@Attribute(.spotlight)` registers the property for Spotlight search indexing.

## Defining Relationships

Relationships between models are expressed as regular Swift properties.

```swift
@Model
class Author {
    var name: String
    var bio: String

    @Relationship(deleteRule: .cascade)
    var books: [Book] = []

    init(name: String, bio: String = "") {
        self.name = name
        self.bio = bio
    }
}

@Model
class Book {
    var title: String
    var author: Author?
    // ...
}
```

The `@Relationship` macro lets you specify delete rules. With `.cascade`, deleting an author also removes all associated books. The default `.nullify` simply sets the relationship property to `nil` while keeping the child records intact.

## ModelContainer and ModelContext

To work with SwiftData, you need a `ModelContainer` (which configures storage) and a `ModelContext` (which provides the operational context for CRUD operations).

### Configuring at the App Entry Point

```swift
import SwiftUI
import SwiftData

@main
struct ReadingApp: App {
    var body: some Scene {
        WindowGroup {
            ContentView()
        }
        .modelContainer(for: [Book.self, Author.self])
    }
}
```

The `.modelContainer(for:)` modifier automatically configures storage for the specified model types. This single line replaces all the boilerplate that `NSPersistentContainer` setup required in Core Data.

### Accessing ModelContext in Views

Within your views, use the `@Environment` property wrapper to access the `ModelContext` and perform data operations.

```swift
struct AddBookView: View {
    @Environment(\.modelContext) private var context

    func addBook() {
        let newBook = Book(
            title: "Practical Swift",
            author: "Jane Smith"
        )
        context.insert(newBook)
    }

    func deleteBook(_ book: Book) {
        context.delete(book)
    }
}
```

Call `context.insert()` to add data and `context.delete()` to remove it. Changes are saved automatically, so you typically don't need to call `save()` explicitly.

## Fetching Data with @Query

The `@Query` property wrapper keeps your views in sync with your data automatically.

```swift
struct BookListView: View {
    @Query(sort: \Book.finishedDate, order: .reverse)
    private var books: [Book]

    var body: some View {
        List(books) { book in
            BookRowView(book: book)
        }
    }
}
```

`@Query` supports a rich set of parameters for filtering, sorting, and limiting results.

```swift
// Filtered query
@Query(
    filter: #Predicate<Book> { $0.rating >= 4 },
    sort: \Book.finishedDate,
    order: .reverse
)
private var highRatedBooks: [Book]

// Limited result set
@Query(
    sort: \Book.createdAt,
    order: .reverse,
    limit: 10
)
private var recentBooks: [Book]
```

The `#Predicate` macro provides type-safe filtering with compile-time error checking. Compared to Core Data's `NSPredicate`, you get native Swift expressions that are far more intuitive to write and maintain.

---

## Implementing Dynamic Filters

When you need query conditions that change based on user input, configure `@Query` dynamically in the view's initializer.

```swift
struct FilteredBookListView: View {
    @Query private var books: [Book]

    init(genre: String) {
        let predicate = #Predicate<Book> {
            $0.genre == genre
        }
        _books = Query(
            filter: predicate,
            sort: \Book.finishedDate,
            order: .reverse
        )
    }

    var body: some View {
        List(books) { book in
            BookRowView(book: book)
        }
    }
}
```

The parent view can call `FilteredBookListView(genre: "Fiction")` to display genre-specific results.

## Enabling iCloud Sync

SwiftData makes CloudKit integration straightforward. Just specify the iCloud database option in your `ModelContainer` configuration.

```swift
@main
struct ReadingApp: App {
    var body: some Scene {
        WindowGroup {
            ContentView()
        }
        .modelContainer(for: [Book.self, Author.self], cloudKitDatabase: .automatic)
    }
}
```

In Rork Max, simply adding "enable iCloud sync" to your prompt is enough to have this configuration included in the generated code.

One caveat: `@Attribute(.unique)` constraints may not be fully compatible with CloudKit. If you need uniqueness guarantees with iCloud sync enabled, it's safer to implement validation logic on the application side.

## Schema Migration

When your app evolves and model structures change, SwiftData provides a versioned schema migration system.

```swift
enum BookSchemaV1: VersionedSchema {
    static var versionIdentifier: Schema.Version = Schema.Version(1, 0, 0)
    static var models: [any PersistentModel.Type] {
        [BookV1.self]
    }

    @Model
    class BookV1 {
        var title: String
        var author: String
        init(title: String, author: String) {
            self.title = title
            self.author = author
        }
    }
}

enum BookSchemaV2: VersionedSchema {
    static var versionIdentifier: Schema.Version = Schema.Version(2, 0, 0)
    static var models: [any PersistentModel.Type] {
        [BookV2.self]
    }

    @Model
    class BookV2 {
        var title: String
        var author: String
        var genre: String
        init(title: String, author: String, genre: String = "Uncategorized") {
            self.title = title
            self.author = author
            self.genre = genre
        }
    }
}
```

You can define a `SchemaMigrationPlan` to specify the steps between versions. For lightweight migrations — like adding a property with a default value — SwiftData handles the transition automatically.

## Performance Optimization Tips

Here are key strategies for keeping your SwiftData-powered app performant.

**Leverage lazy loading.** SwiftData lazily loads relationship data. Even if a model has relationships pointing to large datasets, memory is only consumed when the data is actually accessed.

**Limit fetch counts.** Set the `limit` parameter on `@Query` to control how much data is loaded at once. For infinite-scroll interfaces, designing with pagination in mind is essential.

**Use background operations for bulk work.** When importing large volumes of data, use `ModelActor` to avoid blocking the main thread.

```swift
@ModelActor
actor BookImporter {
    func importBooks(from data: [BookData]) {
        for item in data {
            let book = Book(
                title: item.title,
                author: item.author
            )
            modelContext.insert(book)
        }
        try? modelContext.save()
    }
}
```

The `@ModelActor` macro ensures data operations run safely on a background thread, keeping the UI responsive even when processing thousands of records.

## Common Pitfalls to Watch For

A few things worth keeping in mind when using SwiftData with Rork Max.

**iOS 17 minimum requirement.** SwiftData is only available on iOS 17 and later. If you need to support iOS 16, stick with Core Data. Specifying your target OS version in the Rork Max prompt ensures the right framework is selected.

**Testing in previews.** When using SwiftData in SwiftUI Previews, set up an in-memory container for clean, isolated test data.

```swift
#Preview {
    BookListView()
        .modelContainer(for: Book.self, inMemory: true)
}
```

**Handling Optionals.** Using `Optional` properties in SwiftData models means you'll need nil checks in your predicates and views. Where possible, set default values and avoid Optionals to keep your code simple.
