You wake up the morning after a release to find several support messages: "The payment button isn't working." You check the code and spot it immediately — a single wrong condition in a branch you pushed the day before.

The fix takes five minutes. But the App Store review takes at least 24 hours, sometimes days. In the meantime, your users are stuck with a broken app.

EAS Update changes this entirely. For JavaScript and TypeScript changes, you can push fixes directly to users' devices — no review required.

## What Is EAS Update?

EAS Update is Expo's over-the-air (OTA) update delivery service. It lets you push JavaScript bundle and asset changes (images, fonts) to your app users without going through App Store or Google Play review.

Since Rork apps are built on React Native and Expo, EAS Update works out of the box.

### What Can and Can't Be Updated OTA

EAS Update only applies to JavaScript/TypeScript code and static assets. Native code changes, new permissions, or new native modules still require a full app review.

- ✅ Logic bug fixes
- ✅ UI text, colors, and layout changes
- ✅ API endpoint updates
- ✅ Image and font replacements
- ✅ JavaScript library updates
- ❌ New permissions (camera, microphone, etc.)
- ❌ Native module changes
- ❌ App icon or splash screen changes (native side)

The vast majority of everyday bug fixes and text corrections fall within EAS Update's scope.

## How EAS Update Works

EAS Update manages updates through "channels" and "branches."

```
Build (app binary)
  └─ Channel: production  ← What App Store / Google Play users receive
  └─ Channel: staging     ← What testers receive (TestFlight / internal tracks)

Channels point to branches:
Branch: main              ← Where your latest updates land
```

When a user launches your app, it connects to EAS servers and checks whether a new update is available for its channel. If one exists, the bundle downloads in the background and applies on the next launch (or immediately, if you configure it that way).

Users don't need to do anything. Restarting the app is enough to get the latest version.

## Setup Instructions

### 1. Install the Expo EAS CLI

```bash
npm install -g eas-cli

# Log in with your Expo account
eas login
```

### 2. Configure app.json

Add the following to your Rork app's `app.json`:

```json
{
  "expo": {
    "name": "Your App Name",
    "slug": "your-app-slug",
    "runtimeVersion": {
      "policy": "sdkVersion"
    },
    "updates": {
      "url": "https://u.expo.dev/YOUR_PROJECT_ID",
      "enabled": true,
      "fallbackToCacheTimeout": 0,
      "checkAutomatically": "ON_LOAD"
    }
  }
}
```

`YOUR_PROJECT_ID` is set automatically when you run `eas init`:

```bash
eas init
```

### 3. Set Channels in eas.json

Add a `channel` to each build profile in `eas.json`:

```json
{
  "build": {
    "development": {
      "developmentClient": true,
      "distribution": "internal",
      "channel": "development"
    },
    "staging": {
      "distribution": "internal",
      "channel": "staging"
    },
    "production": {
      "distribution": "store",
      "channel": "production"
    }
  }
}
```

With separate channels, testers see staging updates while regular users only receive production updates.

### 4. Deploy an Update

After fixing your code, a single command deploys the update:

```bash
# Deploy to staging channel
eas update --channel staging --message "Fix payment button condition"

# After verification, deploy to production
eas update --channel production --message "Fix payment button condition (prod)"
```

Expected output:
```
✔ Published\!
  Channel: production
  Branch: main
  Update ID: abc123...
  Message: Fix payment button condition (prod)
```

Within a few minutes of this command completing, users' devices will start receiving the update.

## Channel Management and Rollback

### Viewing Update History

```bash
# List updates in the production channel
eas update:list --channel production
```

### Rolling Back

If a bad update slips through, you can instantly revert to a previous version:

```bash
# Roll back to a specific update ID
eas update:republish --channel production --update-id PREVIOUS_UPDATE_ID
```

This is one of EAS Update's most powerful features. No binary rebuild required — you can undo a broken update in under five minutes.

### Staging Verification Before Production

```bash
# Test on staging first
eas update --channel staging --message "New feature: profile image upload"

# After testing, republish to production
eas update:republish \
  --source-channel staging \
  --channel production \
  --update-id STAGING_UPDATE_ID
```

## Common Pitfalls and Fixes

Here are the issues that actually come up after integrating EAS Update.

### ① Updates Not Reaching Users Due to runtimeVersion Mismatch

This is the most common problem. When you rebuild the app after changing native code, the `runtimeVersion` changes. Users on older builds won't receive new updates because the version no longer matches.

```json
// ❌ Manually pinned — easy to drift out of sync
"runtimeVersion": "1.0.0"

// ✅ Tied to SDK version (recommended)
"runtimeVersion": {
  "policy": "sdkVersion"
}
```

With the `sdkVersion` policy, `runtimeVersion` only changes when the Expo SDK version changes — not on every native rebuild. This prevents accidental version mismatches.

### ② Balancing Immediate Updates with User Experience

By default, the app checks for updates at launch, downloads in the background, and applies the update on the **next** launch. This is the smoothest UX.

For critical bug fixes that need immediate application, use `reloadAsync()` from `expo-updates`:

```typescript
import * as Updates from 'expo-updates';

async function checkAndApplyUpdate() {
  try {
    const update = await Updates.checkForUpdateAsync();
    if (update.isAvailable) {
      await Updates.fetchUpdateAsync();
      // Show a dialog to inform the user before reloading
      await Updates.reloadAsync();
    }
  } catch (error) {
    // Silently continue on update check failure
    // Never crash the app over a failed update check
    console.warn('Update check failed:', error);
  }
}
```

Be careful with `reloadAsync()` — it restarts the app abruptly. Reserve it for genuinely critical fixes; the default next-launch behavior is almost always better for user experience.

### ③ Updates Not Working in the Development Environment

EAS Update does not run in Expo Go or Development Builds — this is intentional. To test real update delivery, distribute a `staging` build through TestFlight or Google Play's internal testing track and test from there.

## Production Best Practices

After several months of running EAS Update in production, here's what actually makes a difference.

**Write specific update messages, not generic ones.** "Bug fix" tells you nothing when you're scanning update history six months later. Write something like "Payment flow: fix variable initialization timing (#123)" — specific enough to make rollback decisions in seconds.

**Always test on staging for at least 15 minutes before pushing to production.** Install the staging update on your own device, run through the critical paths, then deploy to production. This simple rule catches nearly every major regression before it reaches users.

**Automate with GitHub Actions.** Automatically deploying to the staging channel on every merge to `main` removes the friction of manual deployments. See the [GitHub Actions + EAS Build CI/CD automation guide](/articles/rork-dev/rork-github-actions-eas-build-cicd) for a ready-to-use workflow.

**Log every update to a shared channel.** Even as a solo developer, posting EAS update results to Slack or Discord gives you a timestamped record of what changed and when — invaluable when debugging a regression that appeared "suddenly."

For testing your app on real devices before pushing updates, check out the [Rork Companion iOS Testing Guide](/articles/rork-basics/rork-companion-ios-real-device-test-guide) as well.

EAS Update has transformed how I handle post-release issues. Perfect code doesn't exist. What matters is having the infrastructure to fix problems in five minutes rather than five days.
