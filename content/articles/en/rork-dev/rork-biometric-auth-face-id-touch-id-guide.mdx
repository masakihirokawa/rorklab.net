## Why Biometric Authentication Matters for Mobile Apps

As mobile apps handle increasingly sensitive data, relying solely on passwords is no longer sufficient. Passwords can be phished, leaked, or forgotten — and typing them on a small screen is cumbersome. **Biometric authentication** solves both problems at once: Face ID, Touch ID, and Android fingerprint sensors let users verify their identity in under a second while maintaining strong security.

With Rork, you can integrate biometric authentication into your app using `expo-local-authentication`, a cross-platform package that provides a unified API for iOS and Android. In this guide, we'll walk through the complete implementation — from setup to security best practices and fallback strategies.

## Understanding expo-local-authentication

The `expo-local-authentication` package is the foundation for biometric auth in Rork apps. It wraps platform-specific APIs — `LAContext` on iOS and `BiometricPrompt` on Android — behind a simple, consistent interface. Here's what it offers:

- **Hardware detection**: Check whether the device supports biometric authentication
- **Enrollment status**: Verify that the user has actually registered their face or fingerprint
- **Authentication dialog**: Display the native OS authentication prompt and handle the result

Since it uses React Native / Expo bridges, you don't need to write any platform-specific code.

```typescript
// Importing and checking supported authentication types
import * as LocalAuthentication from 'expo-local-authentication';

const types = await LocalAuthentication.supportedAuthenticationTypesAsync();
// [1] = FINGERPRINT, [2] = FACIAL_RECOGNITION, [3] = IRIS
console.log('Available auth types:', types);
```

## Step 1 — Project Setup

Start by installing `expo-local-authentication` in your Rork project.

```bash
# Run in your Rork project terminal
npx expo install expo-local-authentication

# For iOS, you need to add a usage description to Info.plist
# Add the following to app.json or app.config.js
```

Add the Face ID usage description to `app.json` — this is required for App Store review.

```json
{
  "expo": {
    "ios": {
      "infoPlist": {
        "NSFaceIDUsageDescription": "We use Face ID to securely authenticate your identity"
      }
    }
  }
}
```

On Android, no additional configuration is needed. The `expo-local-authentication` package automatically adds the `USE_BIOMETRIC` and `USE_FINGERPRINT` permissions to your manifest.

## Step 2 — Checking Biometric Support

Before attempting authentication, you should always check whether the device supports it and whether the user has enrolled their biometric data.

```typescript
import * as LocalAuthentication from 'expo-local-authentication';
import { Alert } from 'react-native';

// Check biometric availability
async function checkBiometricSupport(): Promise<{
  isAvailable: boolean;
  authTypes: LocalAuthentication.AuthenticationType[];
  isEnrolled: boolean;
}> {
  // Does the hardware support biometrics?
  const isAvailable = await LocalAuthentication.hasHardwareAsync();
  
  // Which authentication types are available?
  const authTypes = await LocalAuthentication.supportedAuthenticationTypesAsync();
  
  // Has the user enrolled their biometric data?
  const isEnrolled = await LocalAuthentication.isEnrolledAsync();

  return { isAvailable, authTypes, isEnrolled };
}

// Usage
const support = await checkBiometricSupport();

if (!support.isAvailable) {
  Alert.alert('Notice', 'This device does not support biometric authentication');
} else if (!support.isEnrolled) {
  Alert.alert(
    'Setup Required',
    'Please set up Face ID or Touch ID in your device settings'
  );
}
// Expected output:
// { isAvailable: true, authTypes: [2], isEnrolled: true }
// authTypes: [2] indicates FACIAL_RECOGNITION (Face ID)
```

## Step 3 — Implementing the Authentication Flow

Now let's build the core authentication logic that displays the native biometric prompt and handles the result.

```typescript
import * as LocalAuthentication from 'expo-local-authentication';

interface AuthResult {
  success: boolean;
  error?: string;
}

async function authenticateUser(): Promise<AuthResult> {
  try {
    const result = await LocalAuthentication.authenticateAsync({
      // Message displayed in the authentication dialog
      promptMessage: 'Unlock the app',
      // Label for the fallback button
      fallbackLabel: 'Use passcode',
      // Label for the cancel button
      cancelLabel: 'Cancel',
      // Allow device passcode as a fallback
      disableDeviceFallback: false,
    });

    if (result.success) {
      console.log('Authentication successful');
      return { success: true };
    } else {
      console.log('Authentication failed:', result.error);
      return {
        success: false,
        error: result.error ?? 'Authentication failed',
      };
    }
  } catch (error) {
    console.error('Authentication error:', error);
    return {
      success: false,
      error: 'An error occurred during authentication',
    };
  }
}
// Expected output (success): { success: true }
// Expected output (failure): { success: false, error: "user_cancel" }
```

Setting `disableDeviceFallback` to `false` allows users to fall back to their device passcode (PIN or pattern) if biometric authentication fails repeatedly. This strikes a good balance between security and preventing user lockouts.

## Step 4 — Building the Lock Screen Component

Let's put it all together in a React component that serves as the app's lock screen.

```tsx
import React, { useState, useEffect, useCallback } from 'react';
import { View, Text, TouchableOpacity, StyleSheet, ActivityIndicator } from 'react-native';
import * as LocalAuthentication from 'expo-local-authentication';

export default function BiometricLockScreen({ onAuthenticated }: {
  onAuthenticated: () => void;
}) {
  const [isLoading, setIsLoading] = useState(true);
  const [biometricType, setBiometricType] = useState<string>('');
  const [errorMessage, setErrorMessage] = useState('');

  useEffect(() => {
    (async () => {
      const types = await LocalAuthentication.supportedAuthenticationTypesAsync();
      if (types.includes(LocalAuthentication.AuthenticationType.FACIAL_RECOGNITION)) {
        setBiometricType('Face ID');
      } else if (types.includes(LocalAuthentication.AuthenticationType.FINGERPRINT)) {
        setBiometricType('Touch ID');
      } else {
        setBiometricType('Passcode');
      }
      setIsLoading(false);
    })();
  }, []);

  const handleAuthenticate = useCallback(async () => {
    setErrorMessage('');
    const result = await LocalAuthentication.authenticateAsync({
      promptMessage: `Authenticate with ${biometricType}`,
      fallbackLabel: 'Use passcode',
      disableDeviceFallback: false,
    });

    if (result.success) {
      onAuthenticated();
    } else if (result.error !== 'user_cancel') {
      setErrorMessage('Authentication failed. Please try again.');
    }
  }, [biometricType, onAuthenticated]);

  // Automatically trigger authentication on mount
  useEffect(() => {
    if (!isLoading && biometricType) {
      handleAuthenticate();
    }
  }, [isLoading, biometricType, handleAuthenticate]);

  if (isLoading) {
    return (
      <View style={styles.container}>
        <ActivityIndicator size="large" color="#007AFF" />
      </View>
    );
  }

  return (
    <View style={styles.container}>
      <Text style={styles.lockIcon}>🔒</Text>
      <Text style={styles.title}>App Locked</Text>
      <Text style={styles.subtitle}>
        Authenticate with {biometricType} to continue
      </Text>
      {errorMessage ? (
        <Text style={styles.error}>{errorMessage}</Text>
      ) : null}
      <TouchableOpacity style={styles.button} onPress={handleAuthenticate}>
        <Text style={styles.buttonText}>Unlock with {biometricType}</Text>
      </TouchableOpacity>
    </View>
  );
}

const styles = StyleSheet.create({
  container: {
    flex: 1, justifyContent: 'center',
    alignItems: 'center', backgroundColor: '#F5F5F7',
  },
  lockIcon: { fontSize: 64, marginBottom: 24 },
  title: { fontSize: 22, fontWeight: '700', marginBottom: 8, color: '#1D1D1F' },
  subtitle: { fontSize: 16, color: '#6E6E73', marginBottom: 24 },
  error: { fontSize: 14, color: '#FF3B30', marginBottom: 16 },
  button: {
    backgroundColor: '#007AFF', paddingHorizontal: 32,
    paddingVertical: 14, borderRadius: 12,
  },
  buttonText: { color: '#FFFFFF', fontSize: 16, fontWeight: '600' },
});
```

## Security Design Best Practices

Implementing biometric authentication is only part of the equation. You also need to think about how the broader security architecture works.

### Storing Auth Tokens Securely

After a successful biometric check, you'll likely need to access stored credentials or tokens. Use `expo-secure-store` to encrypt and store sensitive data in the iOS Keychain or Android Encrypted SharedPreferences.

```typescript
import * as SecureStore from 'expo-secure-store';

// Save token securely after authentication
async function saveAuthToken(token: string): Promise<void> {
  await SecureStore.setItemAsync('auth_token', token, {
    keychainAccessible: SecureStore.WHEN_UNLOCKED_THIS_DEVICE_ONLY,
  });
}

// Retrieve token on app launch
async function getAuthToken(): Promise<string | null> {
  return await SecureStore.getItemAsync('auth_token');
}
// Expected output: A JWT token string like "eyJhbGciOiJIUzI1NiIs..."
```

### Tiered Security Levels

Different features warrant different security levels. Consider structuring your app like this:

- **Level 1 (Low)**: App launch — device passcode is sufficient
- **Level 2 (Medium)**: Viewing personal data — require biometric authentication
- **Level 3 (High)**: Payments and transfers — biometric auth plus server-side verification

### Re-authentication on Background Resume

When users switch away from your app and come back, you need a policy for when to require re-authentication. Financial apps should lock immediately, while a note-taking app might allow a 5-minute grace period.

```typescript
// Re-authentication logic for background resume
import { AppState, AppStateStatus } from 'react-native';

const AUTH_TIMEOUT_MS = 5 * 60 * 1000; // 5 minutes
let lastAuthTime = Date.now();

AppState.addEventListener('change', (state: AppStateStatus) => {
  if (state === 'active') {
    const elapsed = Date.now() - lastAuthTime;
    if (elapsed > AUTH_TIMEOUT_MS) {
      requireReAuthentication();
    }
  }
});

function requireReAuthentication() {
  console.log('Session timed out: re-authentication required');
}
```

## Fallback Strategies When Biometrics Aren't Available

Not every user can use biometric authentication. Your app needs to handle these scenarios gracefully:

- Older devices without biometric hardware
- Users who haven't set up Face ID or Touch ID
- Temporary inability due to injury, gloves, or environmental conditions
- Post-biometric data changes that require re-enrollment

A sensible fallback hierarchy looks like this:

1. **Biometric authentication** (Face ID / Touch ID / Fingerprint)
2. **Device passcode** (PIN / Pattern)
3. **In-app PIN** (Custom 4–6 digit code)
4. **Password authentication** (Last resort)

## Testing and Debugging Tips

You can test biometric authentication in simulators and emulators during development.

- **iOS Simulator**: Go to `Features > Face ID > Enrolled` to enable it, then use `Features > Face ID > Matching Face / Non-matching Face` to simulate success or failure
- **Android Emulator**: Use `Extended Controls > Fingerprint` to simulate fingerprint scans

On real devices, make sure to test the following scenarios:

- First-time Face ID / Touch ID authentication success
- Fallback to passcode after authentication failure
- User cancellation behavior
- Re-authentication after returning from background
- App behavior when no biometric data is enrolled

Rork's preview feature makes on-device testing smooth. If you're using Rork Max, also refer to [Rork Max Security Best Practices](/en/articles/rork-dev/rork-max-security-best-practices) for production-ready security configuration.

## Frequently Asked Questions

### Q1. Do I need to handle both Face ID and Touch ID separately?

No. The `expo-local-authentication` package automatically selects the available authentication method. When you call `authenticateAsync()`, it uses whichever biometric is available on the device. However, it's good practice to check `supportedAuthenticationTypesAsync()` so you can display the correct label (e.g., "Unlock with Face ID" vs "Unlock with Touch ID") in your UI.

### Q2. Is biometric authentication alone sufficient for app security?

Biometric authentication is excellent for local identity verification, but it should be paired with server-side authentication (JWT, session management) for a robust security posture. Biometrics confirm that the person holding the device is the enrolled user — but they don't protect against network-based attacks or token theft.

### Q3. What should I be careful about when sending auth results to the server?

Never send a simple "biometric auth succeeded" flag to your server. This can be spoofed. Instead, use biometric authentication to unlock locally stored credentials (via `expo-secure-store`), then send those credentials (such as a JWT) to your server for validation. The server should verify the token's integrity independently.

### Q4. Are there App Store or Google Play review requirements for biometric auth?

On iOS, you must include `NSFaceIDUsageDescription` in your `Info.plist` with a clear explanation of why your app uses Face ID. Missing this will result in App Store rejection. Additionally, if your app requires biometric authentication, you must provide an alternative authentication method for devices that don't support it — otherwise you'll unnecessarily exclude users.

## Summary

Integrating biometric authentication into your Rork app with `expo-local-authentication` is straightforward and dramatically improves both security and user experience. Here are the key takeaways:

- Always check hardware support and enrollment status with `hasHardwareAsync()` and `isEnrolledAsync()` before attempting authentication
- Customize the authentication experience with `authenticateAsync()` options for prompt messages and fallback behavior
- Store authentication tokens securely using `expo-secure-store` and pair biometric auth with server-side validation
- Design a re-authentication policy for background resume that fits your app's security requirements

Security is fundamental to user trust. Use this guide as a starting point to build an authentication experience that's both secure and delightful for your users.
