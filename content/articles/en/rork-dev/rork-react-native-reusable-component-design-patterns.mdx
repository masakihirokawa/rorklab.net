## Introduction — Why Component Design Matters

As your Rork projects grow in complexity, you'll inevitably notice yourself writing similar UI elements over and over. Buttons, cards, input fields, list items — these common patterns start appearing across multiple screens, each with slight variations. When a design change is needed, you're suddenly hunting through dozens of files to update each instance.

Reusable component design solves this problem at its root. A well-designed component can be used across different screens simply by changing its Props, dramatically speeding up development while reducing bugs. In this guide, we'll walk through practical component design patterns tailored specifically for Rork-powered React Native development.

This article is intended for developers who have built one or two apps with Rork and want to take their code quality to the next level.

## Applying Atomic Design Principles in Rork

**Atomic Design**, a widely adopted framework for building design systems, organizes UI elements into five hierarchical levels. For Rork-generated React Native projects, a practical three-tier approach works best.

**Atoms** are the smallest, indivisible UI elements — buttons, text inputs, icons, and labels. They can't be broken down any further.

**Molecules** combine multiple Atoms into small functional units. A search bar (text input + icon + button) or a card component (image + text + button) are typical examples.

**Organisms** assemble Molecules into larger sections of a screen — headers, footers, product lists, or complete forms.

When writing prompts for Rork, being specific about these layers produces better results. Instead of asking Rork to "create a search bar," try: "Create a SearchBar molecule that combines a TextInput atom and an Icon atom with a clear button." This kind of structured instruction helps Rork generate code that's inherently more reusable.

## Props Design Fundamentals — Balancing Flexibility and Type Safety

The key to truly reusable components lies in **thoughtful Props design**. When Props are well-structured, the same component adapts seamlessly to different contexts throughout your app.

Here's an example of a versatile button component with carefully designed Props:

```typescript
// components/atoms/Button.tsx
import React from 'react';
import { TouchableOpacity, Text, StyleSheet, ViewStyle, TextStyle } from 'react-native';

// Clearly separate required and optional Props
interface ButtonProps {
  label: string;               // Required: button text
  onPress: () => void;         // Required: tap handler
  variant?: 'primary' | 'secondary' | 'outline';  // Optional: style variant
  size?: 'small' | 'medium' | 'large';            // Optional: size
  disabled?: boolean;          // Optional: disabled state
  style?: ViewStyle;           // Optional: custom style override
}

export const Button: React.FC<ButtonProps> = ({
  label,
  onPress,
  variant = 'primary',    // Set sensible defaults
  size = 'medium',
  disabled = false,
  style,
}) => {
  return (
    <TouchableOpacity
      onPress={onPress}
      disabled={disabled}
      style={[styles.base, styles[variant], styles[size], disabled && styles.disabled, style]}
    >
      <Text style={[styles.label, styles[`${variant}Label`]]}>
        {label}
      </Text>
    </TouchableOpacity>
  );
};

// Expected usage:
// <Button label="Submit" onPress={handleSubmit} />
// <Button label="Cancel" onPress={handleCancel} variant="outline" size="small" />
```

Three design principles make this component effective. First, `label` and `onPress` are required while styling props are optional with sensible defaults. Second, literal types for `variant` and `size` restrict input to valid values, catching errors at compile time. Third, the `style` prop allows consumers to override styles when needed without modifying the component itself.

## Composition Patterns — children and Compound Components

For more complex UI elements, **component composition** patterns unlock powerful flexibility.

```typescript
// components/molecules/Card.tsx
import React, { ReactNode } from 'react';
import { View, StyleSheet, ViewStyle } from 'react-native';

interface CardProps {
  children: ReactNode;       // Content is completely flexible
  padding?: number;
  shadow?: boolean;
  style?: ViewStyle;
}

export const Card: React.FC<CardProps> = ({
  children,
  padding = 16,
  shadow = true,
  style,
}) => {
  return (
    <View style={[styles.container, shadow && styles.shadow, { padding }, style]}>
      {children}
    </View>
  );
};

// Define sub-components for structured layout
Card.Header = ({ children }: { children: ReactNode }) => (
  <View style={styles.header}>{children}</View>
);

Card.Body = ({ children }: { children: ReactNode }) => (
  <View style={styles.body}>{children}</View>
);

Card.Footer = ({ children }: { children: ReactNode }) => (
  <View style={styles.footer}>{children}</View>
);

// Expected usage:
// <Card>
//   <Card.Header><Text>Title</Text></Card.Header>
//   <Card.Body><Text>Description</Text></Card.Body>
//   <Card.Footer><Button label="Details" onPress={handlePress} /></Card.Footer>
// </Card>
```

By using `children`, the Card's internal structure is entirely controlled by the consumer. Product cards, profile cards, notification cards — they all look different but share the same Card foundation. This compound component pattern is one of the most powerful tools in React development.

## Separating Logic with Custom Hooks

Extracting business logic from UI components is another essential technique for improving reusability. React custom hooks let you move state management and API calls outside of your components entirely.

```typescript
// hooks/useToggle.ts
import { useState, useCallback } from 'react';

// A simple custom hook for managing toggle state
export const useToggle = (initialValue: boolean = false) => {
  const [value, setValue] = useState(initialValue);

  const toggle = useCallback(() => setValue(prev => !prev), []);
  const setTrue = useCallback(() => setValue(true), []);
  const setFalse = useCallback(() => setValue(false), []);

  return { value, toggle, setTrue, setFalse };
};

// Usage:
// const { value: isVisible, toggle: toggleModal } = useToggle(false);
// <Button label={isVisible ? 'Close' : 'Open'} onPress={toggleModal} />
```

Here's a more practical example — a custom hook that encapsulates form field validation:

```typescript
// hooks/useFormField.ts
import { useState, useCallback } from 'react';

interface UseFormFieldOptions {
  initialValue?: string;
  validate?: (value: string) => string | null;  // null = validation passed
}

export const useFormField = ({ initialValue = '', validate }: UseFormFieldOptions = {}) => {
  const [value, setValue] = useState(initialValue);
  const [error, setError] = useState<string | null>(null);
  const [touched, setTouched] = useState(false);

  const onChange = useCallback((text: string) => {
    setValue(text);
    if (touched && validate) {
      setError(validate(text));
    }
  }, [touched, validate]);

  const onBlur = useCallback(() => {
    setTouched(true);
    if (validate) {
      setError(validate(value));
    }
  }, [value, validate]);

  return { value, error, touched, onChange, onBlur };
};

// Expected output:
// { value: 'test@example.com', error: null, touched: true, onChange: [Function], onBlur: [Function] }
```

This single hook handles email inputs, password fields, username fields — any text input that needs validation. The UI component stays clean and focused purely on rendering.

## Prompting Rork for Better Component Architecture

The way you write prompts in Rork directly impacts how reusable your generated components will be. Here are some practical tips that make a real difference.

**Specify Props explicitly**: Instead of "create a button," write "create a reusable Button component with variant (primary/secondary/outline), size (small/medium/large), and disabled Props with TypeScript interfaces."

**Define folder structure**: Tell Rork: "Organize components into components/atoms/, components/molecules/, and components/organisms/ directories."

**Request a shared theme**: Add: "Create a theme.ts file with color, typography, and spacing constants, and reference them in all components." This ensures visual consistency across your entire app.

**Encourage composition**: When adding new screens, specify: "Build this screen using the existing Button, Card, and Input components." This prevents Rork from creating duplicate implementations.

For a deeper dive into architectural patterns that complement component design, check out the [Rork App Architecture Patterns Complete Guide](/en/articles/rork-dev/rork-app-architecture-patterns-mvvm-clean), which covers MVVM, Clean Architecture, and dependency injection strategies.

## FAQ

**Q: Can I use Rork-generated components as-is in production?**

A: Rork generates functional code that works immediately, making it great for prototyping and MVPs. As your project matures, refactoring toward the patterns described in this guide — particularly TypeScript Props definitions and custom hook extraction — will significantly improve maintainability. The transition is gradual and doesn't require a complete rewrite.

**Q: Is Atomic Design overkill for small apps?**

A: For apps with just two or three screens, strict Atomic Design is unnecessary overhead. However, even small projects benefit from two core practices: centralizing common elements like Buttons and Inputs into shared components, and separating logic into hooks. These habits pay off immediately when you add new features.

**Q: TypeScript type definitions feel complex. Where should I start?**

A: Start with Props interfaces — they're the most practical entry point. You can express most component Props with just four types: `string`, `number`, `boolean`, and `() => void`. When prompting Rork, add "include TypeScript type definitions for all Props" to your instructions, and Rork will generate properly typed code automatically.

**Q: How should I test reusable components?**

A: React Native Testing Library is the standard approach. Reusable components are actually easier to test than tightly-coupled ones because their inputs and outputs are clearly defined through Props. Start by writing tests for your Atom components — "given these Props, verify this renders" — and work upward through Molecules and Organisms.

Designing reusable components is a skill that elevates your Rork development from quick prototyping to building production-quality apps. You don't need to achieve perfect architecture from day one. Start by creating a shared Button and Card component, then gradually expand the pattern library as your project grows. By incorporating component design instructions into your Rork prompts, you'll find that the AI generates higher-quality, more reusable code right from the start.
