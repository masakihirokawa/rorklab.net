You ship your Rork-generated app to TestFlight, open it on a real device, and the bundled asset images look great — but the remote images from Supabase Storage or your CDN come up as empty boxes. I've hit this exact symptom three times in the past few months. The root cause was different each time, but the order I check things has stayed the same. This post is about that order.

Here's the thing worth saying upfront: roughly nine times out of ten, the problem is *not* the image component. It's either how the URL gets to the device, or how iOS decides whether to load it. Rork's generated React Native + Expo code is usually fine, so I've learned to debug from the outside in rather than starting with `<Image>`.

## Triage in 30 seconds

Before doing anything else, paste the image URL into your browser's address bar and see if it loads. This single check decides half the cases for you.

- **Loads fine** → URL and server are healthy. Suspect the client side next: ATS, URL encoding, or the missing-size pitfall.
- **Won't load / asks for auth** → It's a storage permission issue. Check your bucket's public-read policy or your signed URL's expiry.

If you see the symptom "works in simulator but not on device" or "works on Android but not iOS," you can almost guarantee the next section — iOS App Transport Security — is responsible.

## Cause 1: iOS ATS is blocking your HTTP image

iOS blocks all `http://` (non-TLS) requests by default. This is Apple's policy, not anything Rork is doing. If your image URL starts with `http://`, this is the first thing to check.

The classic giveaway: the image shows up in Rork's web preview but not in the iOS Companion app or TestFlight build. That mismatch is ATS almost every time.

**The right fix is to serve the image over HTTPS.** Cloudinary, Supabase Storage, Firebase Storage, imgix, and pretty much every modern CDN supports HTTPS out of the box. The HTTP-only case usually only comes up with a self-hosted server that hasn't had Let's Encrypt set up.

If for some reason you can't move to HTTPS, you can add an exception in `app.json` — but I'd avoid doing this in production. Apple's reviewers will ask why.

```json
{
  "expo": {
    "ios": {
      "infoPlist": {
        "NSAppTransportSecurity": {
          "NSAllowsArbitraryLoads": true
        }
      }
    }
  }
}
```

This setting allows *all* HTTP traffic, which is heavy-handed. The proper approach is `NSExceptionDomains` scoped to specific hosts. Treat the snippet above as a debugging aid, not a long-term fix.

A small but important nuance: ATS doesn't care about the URL shown in the React Native source — it cares about the URL the *device* eventually requests. If your CDN or load balancer issues a 301/302 redirect that lands on plain HTTP somewhere in the middle, the request still fails. I've seen teams swear their image URLs were HTTPS only to discover an old subdomain was redirecting to the HTTP origin. Curl with `-IL` against the original URL is enough to surface this:

```bash
curl -IL https://your-cdn.example.com/uploads/photo.jpg
# Watch for any "Location: http://..." in the redirect chain
```

## Cause 2: Spaces or non-ASCII characters in the URL

A surprisingly common case: the file name or path contains spaces, parentheses, or non-ASCII characters that weren't URL-encoded. When Rork composes an image URL from a user-uploaded file, the original filename can leak straight through.

The buggy version:

```tsx
// ❌ Breaks when fileName has spaces, parentheses, or non-ASCII chars
<Image source={{ uri: `${BASE_URL}/uploads/${fileName}` }} />
```

The fix is to wrap the *filename portion only* in `encodeURIComponent`:

```tsx
// ✅ Encode only the filename, preserving the path slashes
<Image
  source={{ uri: `${BASE_URL}/uploads/${encodeURIComponent(fileName)}` }}
  style={{ width: 200, height: 200 }}
/>
```

The key detail: use `encodeURIComponent`, not `encodeURI`. The latter doesn't escape slashes, which is correct for full URLs but useless for protecting a filename. I once spent a full day chasing this exact bug. The most robust answer is to rename uploads server-side to a random ASCII string before storage — that way, no user-supplied character ever ends up in a URL.

## Cause 3: No size means a 0×0 view

This is the other classic "URL is correct but the image is invisible" symptom. React Native's `<Image>` does **not** auto-detect dimensions for remote URIs — only for local `require(...)` assets. If you don't give it width and height, the rendered box is zero by zero, regardless of how successfully the bytes were loaded.

```tsx
// ❌ No size — the bytes load but the box is 0×0
<Image source={{ uri: 'https://example.com/photo.jpg' }} />

// ✅ Explicit dimensions
<Image
  source={{ uri: 'https://example.com/photo.jpg' }}
  style={{ width: '100%', height: 240, borderRadius: 12 }}
  resizeMode="cover"
/>
```

If you swap the URL and nothing visually changes — even though the element is in the tree per the inspector — this is your problem. For variable-aspect images, pass `aspectRatio` in `style` and the height will be derived from the available width.

One more thing here: if you're showing a list of remote images in a `FlatList` or `ScrollView`, missing dimensions can also cause layout jank — items shift as each image loads — even if you eventually do see them. Pinning a deterministic height per row (or pre-storing image dimensions in your database) is what separates a polished feed from one that visibly stutters.

## Cause 4: CORS and caching — the web preview gotcha

If images are missing only in Rork's web preview, it's almost certainly CORS. Native iOS/Android builds don't enforce cross-origin checks, but browsers do, and they enforce them strictly.

For S3 or Supabase Storage, the storage layer must return `Access-Control-Allow-Origin: *` (or your specific domain). Supabase exposes this in the Storage settings UI; for AWS, it's the bucket's "Permissions" → "CORS configuration."

The other adjacent symptom is "I uploaded a new image but the old one keeps showing." That's CDN/browser caching. The simplest fix is a **cache buster** appended to the URL:

```tsx
// Append a version param so the URL changes when the image changes
const cacheBuster = `?v=${user.avatarUpdatedAt}`;
<Image
  source={{ uri: `${user.avatarUrl}${cacheBuster}` }}
  style={{ width: 64, height: 64, borderRadius: 32 }}
/>
```

Users updating their avatar but still seeing the old one is a bug report you'll get over and over if you skip this. Tying the cache buster to a real `updatedAt` timestamp solves it permanently and saves you a lot of support email.

## Make load state visible

The fastest way to disambiguate "didn't load" from "loaded but invisible" is to wire up `onLoad` and `onError` and let the device tell you what happened.

```tsx
import { Image } from 'expo-image'; // strongly recommended

export function RemoteAvatar({ uri }: { uri: string }) {
  return (
    <Image
      source={{ uri }}
      style={{ width: 80, height: 80, borderRadius: 40 }}
      contentFit="cover"
      transition={200}
      onLoad={() => console.log('[Image] loaded:', uri)}
      onError={(e) => console.log('[Image] error:', uri, e?.error)}
      placeholder={require('../assets/avatar-fallback.png')}
    />
  );
}
```

Expected output: `[Image] loaded: https://...` on success, `[Image] error: ... (details)` on failure. I'd push you toward `expo-image` over the built-in `<Image>` whenever practical — it ships with a fallback `placeholder`, blur transitions, and a real memory cache, which makes a meaningful difference on flaky networks. If your Rork-generated code uses the built-in component, swapping it in is usually a clean upgrade.

The pattern I now use across every Rork app I ship is to put a `RemoteImage` wrapper around `expo-image` that always renders a placeholder, always logs errors with the URL attached, and accepts `width`/`height` as required props (TypeScript will catch the omission at the call site). It costs ten lines of code and saves hours over the lifetime of the app — every future "the image isn't showing" bug report comes with the URL already in the logs.

If you take one thing away: paste the image URL in a browser first. That five-second check splits the problem space in half. Once you've localized it to the client, walk the four causes above in order — ATS, encoding, sizing, CORS — and you'll land on the real one within three steps almost every time.
