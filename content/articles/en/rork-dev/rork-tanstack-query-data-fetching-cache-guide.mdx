As your Rork app grows, managing data fetching becomes increasingly complex. Loading states, error handling, caching, background refresh — implementing all of this from scratch is tedious and error-prone. That's where **TanStack Query (formerly React Query) v5** shines.

In this guide, we'll walk through integrating TanStack Query into a Rork app, from initial setup all the way through advanced caching strategies and Supabase integration.

## What Is TanStack Query and Why Use It in Rork?

TanStack Query v5 is a library built specifically for managing **server state** in React and React Native applications. Server state refers to data that lives on your backend — API responses, database records — and behaves differently from local UI state.

Rork is powered by React Native (Expo), so TanStack Query works seamlessly out of the box. Compared to the classic `useState` + `useEffect` pattern, TanStack Query gives you:

- **Automatic caching**: Fetched data is stored in memory and reused, reducing redundant API calls
- **Background refetching**: Stale data is silently refreshed when the app comes back to the foreground
- **Unified loading and error states**: `isLoading`, `isError`, and `data` are all available from a single hook
- **Optimistic updates**: Instantly reflect changes in the UI before the server confirms
- **Infinite scroll and pagination**: Built-in support via `useInfiniteQuery`

You might wonder — "Is `useState` + `useEffect` really not enough?" It works for simple cases, but as your app scales, you'll start fetching the same data in multiple screens, struggling with cache invalidation, and writing repetitive loading logic. TanStack Query handles all of this cleanly.

## Setup: Adding TanStack Query to Your Rork Project

Start by installing the required packages:

```bash
# Install TanStack Query v5
npx expo install @tanstack/react-query

# Optional: DevTools for debugging during development
npx expo install @tanstack/react-query-devtools
```

Next, wrap your app with `QueryClientProvider` in the root layout file (`app/_layout.tsx`):

```tsx
// app/_layout.tsx
import { QueryClient, QueryClientProvider } from "@tanstack/react-query";
import { Stack } from "expo-router";

// Create QueryClient at module level (outside the component)
const queryClient = new QueryClient({
  defaultOptions: {
    queries: {
      // How long data is considered "fresh" (in milliseconds)
      staleTime: 1000 * 60 * 5, // 5 minutes
      // How long unused cache entries stay in memory
      gcTime: 1000 * 60 * 10, // 10 minutes
      // Number of retries on error
      retry: 2,
    },
  },
});

export default function RootLayout() {
  return (
    <QueryClientProvider client={queryClient}>
      <Stack />
    </QueryClientProvider>
  );
}
```

> **Note**: The default `staleTime` is `0`, meaning data is always considered stale. Tune this based on how frequently your data changes.

## useQuery: Clean Data Fetching in One Hook

The core building block is `useQuery`. Here's how to fetch a list of tasks:

```tsx
// hooks/useTasks.ts
import { useQuery } from "@tanstack/react-query";

// Any async function works as a query function
async function fetchTasks() {
  const response = await fetch("https://api.example.com/tasks");
  if (!response.ok) {
    // Throwing an error triggers TanStack Query's error handling
    throw new Error("Failed to fetch tasks");
  }
  return response.json();
}

export function useTasks() {
  return useQuery({
    queryKey: ["tasks"], // Unique cache key
    queryFn: fetchTasks,
  });
}
```

```tsx
// components/TaskList.tsx
import { View, Text, ActivityIndicator } from "react-native";
import { useTasks } from "@/hooks/useTasks";

export function TaskList() {
  const { data, isLoading, isError, error } = useTasks();

  if (isLoading) {
    return <ActivityIndicator />;
  }

  if (isError) {
    return <Text>Error: {error.message}</Text>;
  }

  return (
    <View>
      {data.map((task: { id: string; title: string }) => (
        <Text key={task.id}>{task.title}</Text>
      ))}
    </View>
  );
}
```

Queries with the same `queryKey` share a cache entry, so even if multiple components call `useTasks()`, only one API request fires.

## Designing queryKeys: Handling Dynamic Parameters

When fetching data based on dynamic values like IDs or search terms, include those values in the `queryKey`:

```tsx
// hooks/useTaskDetail.ts
import { useQuery } from "@tanstack/react-query";

async function fetchTaskById(taskId: string) {
  const response = await fetch(`https://api.example.com/tasks/${taskId}`);
  if (!response.ok) throw new Error("Task not found");
  return response.json();
}

export function useTaskDetail(taskId: string) {
  return useQuery({
    queryKey: ["tasks", taskId], // Different taskId = separate cache entry
    queryFn: () => fetchTaskById(taskId),
    enabled: !!taskId, // Don't fetch if taskId is empty
  });
}
```

The `enabled` option is great for conditional fetching — for example, waiting until a user is authenticated before loading their data.

## useMutation: Creating, Updating, and Deleting Data

For write operations, use `useMutation`. Here's an example for adding a new task:

```tsx
// hooks/useCreateTask.ts
import { useMutation, useQueryClient } from "@tanstack/react-query";

async function createTask(newTask: { title: string; completed: boolean }) {
  const response = await fetch("https://api.example.com/tasks", {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify(newTask),
  });
  if (!response.ok) throw new Error("Failed to create task");
  return response.json();
}

export function useCreateTask() {
  const queryClient = useQueryClient();

  return useMutation({
    mutationFn: createTask,
    onSuccess: () => {
      // Invalidate the "tasks" cache to trigger a refetch
      queryClient.invalidateQueries({ queryKey: ["tasks"] });
    },
    onError: (error) => {
      console.error("Mutation error:", error.message);
    },
  });
}
```

```tsx
// components/AddTaskForm.tsx
import { useState } from "react";
import { View, TextInput, Button, Text } from "react-native";
import { useCreateTask } from "@/hooks/useCreateTask";

export function AddTaskForm() {
  const [title, setTitle] = useState("");
  const { mutate, isPending, isError } = useCreateTask();

  const handleAdd = () => {
    if (!title.trim()) return;
    mutate(
      { title, completed: false },
      {
        onSuccess: () => {
          setTitle(""); // Reset form after successful submission
        },
      }
    );
  };

  return (
    <View>
      <TextInput
        value={title}
        onChangeText={setTitle}
        placeholder="Enter task title"
      />
      <Button title={isPending ? "Adding..." : "Add Task"} onPress={handleAdd} />
      {isError && <Text>Something went wrong. Please try again.</Text>}
    </View>
  );
}
```

Calling `invalidateQueries` marks the cached data as stale and triggers a background refetch the next time the query is observed.

## Supabase Integration

Many Rork apps use Supabase as a backend. Since Supabase client methods return Promises, they integrate naturally with TanStack Query:

```tsx
// lib/supabase.ts (your existing setup)
import { createClient } from "@supabase/supabase-js";

export const supabase = createClient(
  process.env.EXPO_PUBLIC_SUPABASE_URL!,
  process.env.EXPO_PUBLIC_SUPABASE_ANON_KEY!
);
```

```tsx
// hooks/useSupabaseTasks.ts
import { useQuery, useMutation, useQueryClient } from "@tanstack/react-query";
import { supabase } from "@/lib/supabase";

// Fetch tasks from Supabase
async function fetchSupabaseTasks() {
  const { data, error } = await supabase
    .from("tasks")
    .select("*")
    .order("created_at", { ascending: false });

  if (error) throw new Error(error.message);
  return data;
}

export function useSupabaseTasks() {
  return useQuery({
    queryKey: ["supabase", "tasks"],
    queryFn: fetchSupabaseTasks,
    staleTime: 1000 * 30, // Consider data fresh for 30 seconds
  });
}

// Toggle task completion
export function useToggleTask() {
  const queryClient = useQueryClient();

  return useMutation({
    mutationFn: async ({
      id,
      completed,
    }: {
      id: string;
      completed: boolean;
    }) => {
      const { error } = await supabase
        .from("tasks")
        .update({ completed })
        .eq("id", id);

      if (error) throw new Error(error.message);
    },
    onSuccess: () => {
      queryClient.invalidateQueries({ queryKey: ["supabase", "tasks"] });
    },
  });
}
```

## Cache Strategy: Understanding staleTime and gcTime

Tuning cache behavior is key to balancing freshness and performance.

**staleTime**: How long data is considered "fresh." During this window, TanStack Query won't refetch even if the component remounts or the window regains focus.

**gcTime** (formerly `cacheTime`): How long unused cache entries stay in memory before being garbage collected.

```tsx
// Cache strategy examples by data type
const cacheStrategies = {
  // User profile — changes rarely
  userProfile: {
    staleTime: 1000 * 60 * 60,      // 1 hour
    gcTime: 1000 * 60 * 60 * 24,    // 24 hours
  },
  // Product inventory — changes frequently
  inventory: {
    staleTime: 0,                    // Always refetch
    gcTime: 1000 * 60 * 5,          // 5 minutes
  },
  // News feed — moderate update frequency
  newsFeed: {
    staleTime: 1000 * 60 * 5,       // 5 minutes
    gcTime: 1000 * 60 * 30,         // 30 minutes
  },
};
```

Matching `staleTime` to the nature of your data reduces unnecessary API calls while keeping the UI up to date.

## Common Issues and How to Fix Them

### Query Never Runs

Check whether `enabled` is unintentionally `false`. Also make sure you're calling hooks inside a component that's wrapped by `QueryClientProvider`.

```tsx
// Bug: passing undefined instead of false
useQuery({ queryKey: ["user", userId], queryFn: ..., enabled: userId });

// Fix: explicitly convert to boolean
useQuery({ queryKey: ["user", userId], queryFn: ..., enabled: !!userId });
```

### Cache Not Updating After Mutation

Verify that the `queryKey` in `invalidateQueries` matches the one used in `useQuery`. Note that `invalidateQueries` performs partial matching by default.

```tsx
// Invalidates all queries starting with ["tasks"]
queryClient.invalidateQueries({ queryKey: ["tasks"] });

// Exact match only
queryClient.invalidateQueries({
  queryKey: ["tasks", specificId],
  exact: true,
});
```

### TypeScript Type Errors

In v5, types are inferred from the `queryFn` return value. You can also specify types explicitly:

```tsx
// Explicit type annotation
useQuery<Task[], Error>({
  queryKey: ["tasks"],
  queryFn: fetchTasks,
});
```

## Frequently Asked Questions

**Q. Does TanStack Query conflict with Redux or Zustand?**

Not at all — they solve different problems. TanStack Query manages **server state** (data from APIs and databases), while Redux and Zustand manage **client state** (UI state, user input). The recommended pattern is to use both: for example, Zustand for cart state and TanStack Query for product listings.

**Q. Can I trigger refetches when the app returns to the foreground?**

Yes, but React Native requires a small setup. TanStack Query listens to `window.focus` by default, which doesn't apply to React Native. Use `focusManager.setEventListener` to respond to `AppState` changes:

```tsx
// lib/queryFocusManager.ts
import { focusManager } from "@tanstack/react-query";
import { AppState, AppStateStatus } from "react-native";

export function setupFocusManager() {
  return AppState.addEventListener("change", (state: AppStateStatus) => {
    focusManager.setFocused(state === "active");
  });
}
```

Call this inside a `useEffect` in your root layout to enable automatic refetching when users return to your app.

**Q. What happens to cached data when the app is offline?**

In v5, you can control offline behavior with the `networkMode` option. Setting `networkMode: "offlineFirst"` returns cached data even when there's no network connection. For a comprehensive look at local storage options, see the [Rork × AsyncStorage / MMKV / SQLite Local Data Management Guide](/articles/rork-dev/rork-local-storage-asyncstorage-mmkv-sqlite-guide).
