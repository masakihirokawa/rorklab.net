## Why User Behavior Analytics Matters

You've shipped your app, downloads are climbing — but retention is flat, and you're not sure where users are dropping off. Sound familiar? This is exactly the kind of problem that user behavior analytics solves.

Firebase Analytics is Google's free analytics platform that tracks user behavior in real time. Paired with Firebase Remote Config, it lets you run A/B tests and make data-driven decisions about your UI, onboarding flow, and conversion funnels — without shipping a new build.

In this guide, we'll walk through integrating Firebase Analytics into a Rork Max app, designing meaningful custom events, and setting up your first A/B test from scratch.

## Prerequisites

Before getting started, make sure you have:

- A **Rork Max** account with an existing app project
- A **Firebase project** (create one at [Firebase Console](https://console.firebase.google.com/))
- **Node.js** 18 or later
- Basic familiarity with React Native / Expo

Since Rork Max provides full native module access, you can use the Firebase SDK without any workarounds. If you haven't already, register your iOS and Android apps in the Firebase Console before proceeding.

## Setting Up Firebase Analytics

### Installing the Packages

In your Rork Max project, install the Firebase packages:

```bash
# Install Firebase Analytics and Remote Config
npm install @react-native-firebase/app @react-native-firebase/analytics @react-native-firebase/remote-config
```

### Initializing Firebase

Set up Firebase in your app's root layout:

```typescript
// app/_layout.tsx
import analytics from '@react-native-firebase/analytics';
import remoteConfig from '@react-native-firebase/remote-config';
import { useEffect } from 'react';

export default function RootLayout() {
  useEffect(() => {
    const initFirebase = async () => {
      // Set default values for Remote Config
      await remoteConfig().setDefaults({
        onboarding_variant: 'control',
        cta_button_color: '#007AFF',
        show_premium_banner: false,
      });

      // Fetch remote values (5-minute cache)
      await remoteConfig().setConfigSettings({
        minimumFetchIntervalMillis: 300000,
      });
      await remoteConfig().fetchAndActivate();

      // Log session start
      await analytics().logEvent('app_session_start', {
        timestamp: new Date().toISOString(),
      });
    };

    initFirebase();
  }, []);

  // ... return layout
}
```

This initializes Remote Config with sensible defaults, fetches the latest values from the server, and logs a session start event to Analytics — all on app launch.

## Designing and Implementing Custom Events

The real power of Firebase Analytics comes from custom events tailored to your app. Here's a reusable helper module that covers the most common tracking patterns.

### Analytics Helper Functions

```typescript
// utils/analytics.ts — Analytics helper functions
import analytics from '@react-native-firebase/analytics';

// Track screen views
export const trackScreenView = async (screenName: string) => {
  await analytics().logScreenView({
    screen_name: screenName,
    screen_class: screenName,
  });
};

// Track button taps
export const trackButtonTap = async (
  buttonName: string,
  screenName: string,
  additionalParams?: Record<string, string>
) => {
  await analytics().logEvent('button_tap', {
    button_name: buttonName,
    screen_name: screenName,
    ...additionalParams,
  });
};

// Track conversion events (purchases, sign-ups, etc.)
export const trackConversion = async (
  conversionType: string,
  value?: number,
  currency?: string
) => {
  await analytics().logEvent('conversion', {
    conversion_type: conversionType,
    value: value ?? 0,
    currency: currency ?? 'USD',
  });
};

// Track funnel steps (onboarding, checkout, etc.)
export const trackFunnelStep = async (
  funnelName: string,
  stepNumber: number,
  stepName: string
) => {
  await analytics().logEvent('funnel_step', {
    funnel_name: funnelName,
    step_number: stepNumber,
    step_name: stepName,
  });
};
```

### Example — Tracking an Onboarding Funnel

```typescript
// screens/Onboarding.tsx
import { trackFunnelStep, trackConversion } from '@/utils/analytics';

export default function OnboardingScreen() {
  const handleStepComplete = async (step: number) => {
    // Record each step completion
    await trackFunnelStep('onboarding', step, `step_${step}_complete`);
  };

  const handleOnboardingFinish = async () => {
    // Record onboarding completion as a conversion
    await trackConversion('onboarding_complete');
  };

  // ... UI components
}
```

Once deployed, you'll see `funnel_step` events flowing into the Firebase Console's Events tab, and you can build a funnel report to visualize drop-off rates at each step.

## Building Your First A/B Test

### Managing Variants with Remote Config

Firebase Remote Config lets you change your app's behavior from the server — no new build required. Here's a custom hook that makes A/B testing straightforward.

```typescript
// hooks/useABTest.ts — Custom hook for A/B testing
import { useState, useEffect } from 'react';
import remoteConfig from '@react-native-firebase/remote-config';
import analytics from '@react-native-firebase/analytics';

interface ABTestConfig {
  parameterKey: string;   // Remote Config key
  experimentName: string; // Name for Analytics logging
}

export function useABTest({ parameterKey, experimentName }: ABTestConfig) {
  const [variant, setVariant] = useState<string>('control');

  useEffect(() => {
    const fetchVariant = async () => {
      // Get the current variant from Remote Config
      const value = remoteConfig().getValue(parameterKey);
      const currentVariant = value.asString() || 'control';
      setVariant(currentVariant);

      // Log which variant was shown
      await analytics().logEvent('ab_test_exposure', {
        experiment_name: experimentName,
        variant: currentVariant,
      });
    };

    fetchVariant();
  }, [parameterKey, experimentName]);

  return variant;
}
```

### Example — A/B Testing CTA Button Copy

```typescript
// components/CTAButton.tsx
import { TouchableOpacity, Text, StyleSheet } from 'react-native';
import { useABTest } from '@/hooks/useABTest';
import { trackButtonTap } from '@/utils/analytics';

export default function CTAButton() {
  // A/B test: switch button text based on variant
  const ctaVariant = useABTest({
    parameterKey: 'cta_button_text',
    experimentName: 'cta_text_experiment_v1',
  });

  const buttonText =
    ctaVariant === 'variant_a'
      ? 'Start Free Today'
      : ctaVariant === 'variant_b'
        ? 'Build Your First App'
        : 'Get Started'; // control

  const handlePress = async () => {
    await trackButtonTap('cta_main', 'home', {
      variant: ctaVariant,
    });
    // ... navigation logic
  };

  return (
    <TouchableOpacity style={styles.button} onPress={handlePress}>
      <Text style={styles.text}>{buttonText}</Text>
    </TouchableOpacity>
  );
}

const styles = StyleSheet.create({
  button: {
    backgroundColor: '#007AFF',
    paddingVertical: 16,
    paddingHorizontal: 32,
    borderRadius: 12,
    alignItems: 'center',
  },
  text: {
    color: '#FFFFFF',
    fontSize: 18,
    fontWeight: '700',
  },
});
```

To activate the experiment, head to the A/B Testing section in Firebase Console. Create a new experiment targeting the `cta_button_text` parameter with three variants: `control`, `variant_a`, and `variant_b`. Firebase will automatically distribute users across the variants and track the performance of each.

## Analyzing Your Data

Firebase Analytics data can be viewed in the Console or exported to BigQuery for advanced analysis.

### Key Metrics to Watch

Start by monitoring these core metrics in the Firebase Console:

- **DAU / MAU ratio**: Daily active users divided by monthly active users. A ratio above 0.2 is considered healthy for most apps.
- **Average session duration**: How long users spend in each session — a direct measure of engagement.
- **Event completion rates**: Track your custom funnel events to see where users drop off.
- **Retention cohorts**: What percentage of users return after 1 day, 7 days, and 30 days.

### Leveraging BigQuery for Deeper Insights

For more granular analysis, enable BigQuery export in the Firebase Console. This gives you raw SQL access to every event, letting you answer questions like "What's the 7-day retention rate for users who completed onboarding?"

```sql
-- 7-day retention for users who completed onboarding
SELECT
  COUNT(DISTINCT CASE WHEN retention_day = 7 THEN user_pseudo_id END)
    / COUNT(DISTINCT user_pseudo_id) AS retention_7d
FROM (
  SELECT
    user_pseudo_id,
    DATE_DIFF(event_date, first_open_date, DAY) AS retention_day
  FROM `your_project.analytics_events`
  WHERE event_name = 'session_start'
    AND user_pseudo_id IN (
      SELECT user_pseudo_id
      FROM `your_project.analytics_events`
      WHERE event_name = 'conversion'
        AND event_params.value.string_value = 'onboarding_complete'
    )
)
```

## Common Issues and Troubleshooting

### Events Not Showing Up in Firebase Console

Firebase Analytics events can take up to 24 hours to appear in the standard reports. For real-time debugging, use **DebugView** in the Firebase Console:

```bash
# Enable DebugView on an iOS simulator
# Add the following launch argument in your Xcode scheme
-FIRAnalyticsDebugEnabled
```

### Remote Config Values Not Updating

Remote Config caches values by default for 12 hours. During development, set `minimumFetchIntervalMillis` to a shorter interval (we used 5 minutes in our setup code above). For production, Google recommends at least 3,600,000 milliseconds (1 hour) to avoid performance issues.

### Insufficient Sample Size for A/B Tests

To reach statistical significance, you generally need at least 1,000–5,000 events per variant. If your app has a small user base, extend the test duration to 2–4 weeks, or reduce the number of variants to concentrate your traffic.

## FAQ

### Q1. Is Firebase Analytics free?

Yes, Firebase Analytics is completely free with no limits on event volume or user count. However, if you enable BigQuery export, standard BigQuery pricing applies (there's a generous free tier).

### Q2. How long should I run an A/B test?

Two to four weeks is the typical recommendation. Shorter tests may not reach statistical significance, while longer tests risk being affected by seasonal changes or external factors. Firebase's A/B Testing dashboard will notify you when enough data has been collected to declare a winner.

### Q3. Can I use Firebase Analytics with standard Rork (not Max)?

Standard Rork uses React Native / Expo, but Firebase's native SDK doesn't work in Expo Go. You'll need to either use the `expo-firebase-analytics` package or create a custom build with EAS Build. Rork Max gives you direct native module access, so the full Firebase SDK works out of the box.

### Q4. Which events should I track first?

Start with three essentials: (1) Screen views (`screen_view`) to map the user journey, (2) key action taps (`button_tap`) to measure engagement, and (3) conversion events (sign-ups, purchases) to track revenue-critical metrics. Once you have this baseline data, you'll quickly spot where to focus your optimization efforts.
