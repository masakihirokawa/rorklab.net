## Introduction — Why New Architecture Matters Right Now

In October 2024, React Native 0.76 shipped with **New Architecture enabled by default** — the culmination of a multi-year overhaul of the entire rendering and communication model. Expo SDK 52 (released November 2024) followed suit, making New Architecture the standard for all new projects.

Since Rork generates Expo-based apps, any project created in 2025 or later is almost certainly running on New Architecture. Yet many developers haven't fully grasped **why it matters, what changed under the hood, and how to make the most of it** in real applications.

This guide breaks down the four core technologies behind New Architecture, walks you through the migration process for Rork apps, shows you how to diagnose and fix compatibility issues, and teaches you optimization patterns that simply weren't possible before.

### Who This Guide Is For

This guide is written for:

- Rork developers experiencing unexpected build errors or runtime crashes they suspect are related to New Architecture
- Developers who want to future-proof their existing apps and maintain long-term compatibility
- Engineers who want to deeply understand React Native's internals and apply that knowledge to performance tuning

## The Four Technologies Behind New Architecture

New Architecture isn't a single change — it's four interconnected technologies working together.

### 1. JSI (JavaScript Interface)

The old architecture relied on an **asynchronous bridge** for all communication between JavaScript and Native code. Every message had to be serialized to JSON, added to a queue, and processed asynchronously — meaning any native call introduced inherent latency.

JSI replaces this bridge entirely. Instead of passing messages, the JavaScript engine (Hermes) can hold **direct references to C++ objects**. This means:

- Native calls can be made **synchronously** from JavaScript
- Zero serialization overhead
- JavaScript objects can directly reference native memory (zero-copy)

```cpp
// Example: registering a JSI native binding in C++
// This function is directly callable from JavaScript — no bridge involved
runtime.global().setProperty(
  runtime,
  "myNativeFunction",
  jsi::Function::createFromHostFunction(
    runtime,
    jsi::PropNameID::forAscii(runtime, "myNativeFunction"),
    1,
    [](jsi::Runtime& rt, const jsi::Value& thisVal,
       const jsi::Value* args, size_t count) -> jsi::Value {
      // Executes directly, no bridge round-trip
      double result = args[0].asNumber() * 2;
      return jsi::Value(result);
    }
  )
);
// Expected behavior: calling myNativeFunction(21) from JS returns 42 instantly
```

### 2. TurboModules

In the old architecture, all Native Modules were initialized eagerly at app startup — even ones the user might never touch. Camera, location, Bluetooth — all loaded into memory before a single screen rendered, unnecessarily extending launch time.

TurboModules introduce **lazy initialization**:

- Modules are initialized only when first accessed
- Type-safe sync and async interfaces via JSI
- Native code is auto-generated from TypeScript spec files (via Codegen)

```typescript
// TurboModule TypeScript Spec — this file generates C++/Swift/Java code
import type { TurboModule } from 'react-native';
import { TurboModuleRegistry } from 'react-native';

export interface Spec extends TurboModule {
  // Synchronous method — true synchrony via JSI
  getDeviceId(): string;

  // Async method — returns a Promise
  requestPermission(permissionType: string): Promise<boolean>;

  // Event listener pattern
  addListener(eventType: string): void;
  removeListeners(count: number): void;
}

export default TurboModuleRegistry.getEnforcing<Spec>('MyTurboModule');
```

### 3. Fabric (New Renderer)

Fabric is a ground-up rewrite of the UI rendering pipeline. In the old architecture, React's virtual DOM reconciliation, Shadow Tree layout calculation, and Native View updates ran on separate threads and communicated asynchronously — often causing layout jank and inconsistent measurements.

With Fabric:

- **A unified C++ Shadow Tree** is safely accessible from the JS thread, UI thread, and background threads simultaneously
- **Synchronous layout measurement**: `onLayout` handlers and `measureInWindow` return immediate results
- **Concurrent rendering support**: React 18's Concurrent Features (`useTransition`, `useDeferredValue`) now work as intended

```typescript
// Synchronous layout measurement with Fabric
// In the old architecture, this required an async callback chain
import { useRef, useCallback } from 'react';
import { View, Text } from 'react-native';

export function MeasurableComponent() {
  const viewRef = useRef<View>(null);

  const handlePress = useCallback(() => {
    // measureInWindow now returns results immediately
    viewRef.current?.measureInWindow((x, y, width, height) => {
      console.log(`Position: x=${x}, y=${y}, Size: ${width}x${height}`);
      // Expected output: Position: x=16, y=84, Size: 343x48
    });
  }, []);

  return (
    <View ref={viewRef}>
      <Text onPress={handlePress}>Tap to measure position</Text>
    </View>
  );
}
```

### 4. Codegen

Codegen generates native interface code (C++ headers, Java/Kotlin stubs, Swift/ObjC stubs) from TypeScript or Flow type definitions. This eliminates the need to write bridge glue code by hand and catches type mismatches at build time rather than runtime.

- No more handwritten bridge code
- Type safety enforced at compile time, not runtime
- Native module implementation becomes dramatically simpler

## Checking and Configuring New Architecture in Rork Apps

### Verifying Your Current Setup

Look at your `app.json` (or `app.config.js`) to confirm New Architecture status:

```json
// app.json — Expo SDK 52+ new project
{
  "expo": {
    "name": "MyApp",
    "slug": "my-app",
    "sdkVersion": "52.0.0",
    "newArchEnabled": true
  }
}
```

With Expo SDK 52+, `newArchEnabled` defaults to `true` even when not explicitly set. Verify your SDK version:

```bash
# Check your Expo SDK version
cat package.json | grep '"expo"'
# Expected output: "expo": "^52.0.0"

# Check React Native version
cat package.json | grep '"react-native"'
# Expected output: "react-native": "0.76.5"
```

### Temporarily Disabling New Architecture

If you need to disable New Architecture while resolving compatibility issues:

```json
{
  "expo": {
    "newArchEnabled": false
  }
}
```

This is a **temporary measure** — support for the old architecture will continue to shrink over time, and this flag will eventually be removed.

## Compatibility Check — Diagnosing Your Libraries

The most common New Architecture migration problem is **incompatible third-party libraries**.

### Using the Official Compatibility Tools

```bash
# Run compatibility checks
npx react-native-community-cli check-android
npx react-native-community-cli check-ios

# Or scan all dependencies at once
npx @react-native-community/upgrade-helper
```

### Common Compatibility Issues and Fixes

**Issue 1: Libraries using legacy NativeModules / NativeEventEmitter**

These libraries work via the Interop Layer but won't benefit from New Architecture's performance improvements.

```typescript
// ❌ Legacy NativeModules access (deprecated)
import { NativeModules } from 'react-native';
const { MyLegacyModule } = NativeModules;

// ✅ New Architecture compatible access
import { TurboModuleRegistry } from 'react-native';
const MyTurboModule = TurboModuleRegistry.get('MyTurboModule');
```

**Issue 2: react-native-camera (legacy camera library)**

The old `react-native-camera` is not New Architecture compatible. Migrate to `expo-camera` or `react-native-vision-camera` v4.

```bash
# Remove the legacy library
npx expo uninstall react-native-camera

# Install a New Architecture compatible alternative
npx expo install expo-camera
```

**Issue 3: react-native-reanimated below v3**

Reanimated has been fully New Architecture compatible since v3. Upgrade if you're on an older version.

```bash
# Check your current version
cat package.json | grep reanimated
# Expected output: "react-native-reanimated": "^3.16.0"

# Upgrade if needed
npx expo install react-native-reanimated@latest
```

### New Architecture Compatibility Status (Key Libraries)

Here's how major libraries stand today:

- **Fully compatible**: react-navigation v7, react-native-reanimated v3+, expo-camera, react-native-gesture-handler v2+, react-native-mmkv v3+, @shopify/flash-list
- **Works via Interop Layer**: react-native-svg (below v15), react-native-maps (below v1.10)
- **Needs verification**: Community libraries with infrequent maintenance updates

## Build Error Troubleshooting

Here are the most common build errors after enabling New Architecture and how to fix them.

### iOS: `Undefined symbol: _OBJC_CLASS_$_RCTBridge`

A library is still referencing legacy bridge classes.

```bash
# Clean CocoaPods and reinstall
cd ios
pod deintegrate
pod install --repo-update
cd ..

# Regenerate native code
npx expo prebuild --clean
```

### Android: `Cannot find symbol: class ReactPackage`

```kotlin
// ❌ Legacy ReactPackage implementation (deprecated in New Architecture)
class MyPackage : ReactPackage {
    override fun createNativeModules(reactContext: ReactApplicationContext)
        = listOf(MyModule(reactContext))
}

// ✅ Migrate to TurboReactPackage
class MyPackage : TurboReactPackage() {
    override fun getModule(name: String, reactContext: ReactApplicationContext)
        = if (name == MyTurboModule.NAME) MyTurboModule(reactContext) else null
}
```

### Metro Bundler: `Cannot use 'import' outside a module`

Occurs when `"type": "module"` is set in `package.json`.

```javascript
// Update metro.config.js
const { getDefaultConfig } = require('expo/metro-config');
const config = getDefaultConfig(__dirname);

// Add mjs to sourceExts
config.resolver.sourceExts.push('mjs');

module.exports = config;
```

## Performance Measurement and Optimization

Once you're running on New Architecture, here's how to unlock its full potential.

### Profiling with React DevTools

```typescript
// Performance measurement utility
import { performance } from 'react-native';

export function measureRender(label: string, fn: () => void): void {
  const start = performance.now();
  fn();
  const end = performance.now();
  console.log(`[${label}] Render time: ${(end - start).toFixed(2)}ms`);
  // Expected output: [MyComponent] Render time: 2.34ms
}
```

### Leveraging React 18 Concurrent Features

Concurrent Features only deliver their full impact in New Architecture environments.

```typescript
import { useState, useTransition, useDeferredValue } from 'react';
import { FlatList, TextInput, Text, View } from 'react-native';

interface Item {
  id: string;
  name: string;
}

export function SearchableList({ data }: { data: Item[] }) {
  const [query, setQuery] = useState('');
  const [isPending, startTransition] = useTransition();

  // Heavy filtering runs at lower priority
  const deferredQuery = useDeferredValue(query);

  const filteredData = data.filter(item =>
    item.name.toLowerCase().includes(deferredQuery.toLowerCase())
  );

  return (
    <View>
      <TextInput
        value={query}
        onChangeText={text => {
          // Input update is high priority — reflects immediately
          setQuery(text);
        }}
        placeholder="Search..."
      />
      {/* While isPending, show previous results at reduced opacity */}
      <FlatList
        data={filteredData}
        style={{ opacity: isPending ? 0.5 : 1 }}
        keyExtractor={item => item.id}
        renderItem={({ item }) => <Text>{item.name}</Text>}
      />
    </View>
  );
}
```

### JSI-Powered Animation Optimization with Reanimated v3

```typescript
import Animated, {
  useSharedValue,
  useAnimatedStyle,
  withSpring,
  runOnJS,
} from 'react-native-reanimated';

// Reanimated v3 + Fabric = animations running directly on the UI thread
// Zero JS thread involvement, silky-smooth 60fps regardless of JS workload
export function OptimizedCard() {
  const scale = useSharedValue(1);
  const opacity = useSharedValue(1);

  const animatedStyle = useAnimatedStyle(() => ({
    transform: [{ scale: scale.value }],
    opacity: opacity.value,
  }));

  const handlePress = () => {
    // worklet: runs entirely on the UI thread — JS thread never touched
    scale.value = withSpring(0.95, { damping: 15 }, () => {
      scale.value = withSpring(1);
    });
  };

  return (
    <Animated.View style={[{ padding: 20, backgroundColor: '#fff' }, animatedStyle]}>
      <Animated.Text onPress={handlePress}>Tap me</Animated.Text>
    </Animated.View>
  );
}
```

## Runtime Error Patterns and Fixes

Here are runtime error patterns that appear specifically after migrating to New Architecture.

**Error 1: `TypeError: null is not an object (evaluating 'module.default')`**

A CommonJS module is being imported in an ESM context incorrectly.

```javascript
// Add to metro.config.js
config.resolver.unstable_enablePackageExports = true;
config.resolver.unstable_conditionNames = ['require', 'default'];
```

**Error 2: `Invariant Violation: No suitable renderer found for...`**

A custom component is trying to interact with UIManager directly, which conflicts with Fabric.

```typescript
// ❌ Direct UIManager access (not compatible with Fabric)
import { UIManager, findNodeHandle } from 'react-native';
UIManager.measure(findNodeHandle(ref.current), callback);

// ✅ Use ref.measureInWindow instead
ref.current?.measureInWindow((x, y, width, height) => {
  // Fabric-compatible measurement
});
```

**Error 3: Custom native module crashes with multiple callback invocations**

The Interop Layer doesn't support modules that call their callbacks more than once.

```typescript
// ❌ Calling a callback multiple times (crashes in New Architecture)
RCT_EXPORT_METHOD(streamData:(RCTResponseSenderBlock)callback) {
  callback(@[@"first"]);
  callback(@[@"second"]); // Crash!
}

// ✅ Use EventEmitter for streaming multiple values
- (void)startStream {
  [self sendEventWithName:@"dataReceived" body:@{@"value": @"first"}];
  [self sendEventWithName:@"dataReceived" body:@{@"value": @"second"}];
}
```

## Migration Checklist

Use this checklist to migrate safely and systematically.

**Pre-Migration**

- Upgrade to Expo SDK 52+ or React Native 0.76+
- Audit all native dependencies for New Architecture compatibility
- Create a feature branch (`feature/new-architecture`) separate from `main`

**Migration Steps**

- Add (or confirm) `"newArchEnabled": true` in `app.json`
- Run `pod install` to update iOS CocoaPods
- Verify `newArchEnabled=true` in Android `gradle.properties`
- Regenerate native code with `npx expo prebuild --clean`

**Validation**

- Run full functional tests on iOS Simulator
- Run full functional tests on Android Emulator
- Verify animations, gestures, camera, and location services
- If crashes occur, use Hermes stack traces to identify root causes

## FAQ

**Q. Are Rork-generated apps automatically compatible with New Architecture?**

A. Rork generates Expo-based projects, so with Expo SDK 52+, New Architecture is enabled by default. However, third-party libraries installed by Rork may not all be compatible. Check your `app.json` for `"newArchEnabled": true`, and if you encounter issues, use this guide's troubleshooting section. Most major libraries are already compatible, and the Interop Layer handles many of the rest.

**Q. How long will the old architecture remain supported?**

A. The React Native team is gradually winding down old architecture support. As of 2026, the Interop Layer allows most legacy libraries to keep working, but new features — Concurrent Features, direct JSI bindings, and more — are New Architecture exclusive. Whether you're starting a new project or maintaining an existing one, migrating sooner is strongly recommended.

**Q. Will New Architecture always improve performance noticeably?**

A. Not always in a way that's immediately perceptible, especially for lightweight apps. However, apps that do heavy list rendering, complex animations, or high-frequency JS-to-native communication will see meaningful improvements. We recommend profiling with React DevTools Profiler or Xcode Instruments to quantify the gains rather than relying on intuition.

**Q. How should I approach migrating a custom native module?**

A. First, check whether the Interop Layer handles it transparently. If not, you'll need to migrate to TurboModule format: write a TypeScript Spec file, run Codegen to generate native stubs, then implement the native side. Refer to the official guide "Migrating a Legacy Native Module to TurboModule" for detailed steps. If the migration is complex, consider temporarily setting `newArchEnabled: false` and addressing one module at a time.

## Summary

React Native New Architecture is the result of years of foundational rethinking, delivering **bridgeless communication via JSI**, **lazy-loaded TurboModules**, **synchronous rendering through Fabric**, and **compile-time type safety with Codegen**.

For Rork developers, the key takeaways are:

- New Architecture is on by default in Expo SDK 52+ and React Native 0.76+
- Most major libraries are already compatible
- Many incompatible libraries work via the Interop Layer, but long-term migration is still needed
- Combining Concurrent Features with Reanimated v3 unlocks animation quality that simply wasn't achievable before

Understanding New Architecture deeply means you can deliver consistently great user experiences and keep your apps maintainable as the React Native ecosystem continues to evolve. Take the migration one step at a time, verify at each stage, and don't hesitate to temporarily fall back if something breaks unexpectedly.

Once you've migrated to New Architecture, combine it with the profiling techniques in [Rork App Performance Optimization — Rendering, Memory & Network Techniques](/articles/rork-dev/advanced-rork-app-performance-optimization) to quantify your improvements concretely. You can also integrate New Architecture compatibility checks into the build pipeline described in [Rork Max × EAS CI/CD Guide](/articles/rork-dev/rork-max-eas-cicd), so every deployment automatically verifies that your architecture upgrade remains intact. Finally, run your test suite — using the patterns from [Rork Max App Testing Guide — Jest, Detox E2E, and Quality Assurance](/articles/rork-dev/rork-max-testing-guide) — against New Architecture to ensure ongoing quality across every release.

For developers who want a more structured understanding of React Native internals, [Fullstack React Native](https://www.amazon.co.jp/dp/4839973453/?tag=pinocchio-22) by the Expo core team is an excellent companion resource to this guide.
