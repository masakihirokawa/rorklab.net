## Introduction: How Haptics Transform the App Experience

Have you ever tapped a button and felt a subtle vibration in response — a tiny pulse that somehow made the interaction feel more satisfying? That's **haptic feedback** at work.

Apple introduced its precision Taptic Engine with the iPhone 7 in 2016, and modern iPhones can deliver incredibly nuanced vibration patterns. Android has matured similarly, and today both platforms offer robust APIs for delivering thoughtful tactile experiences. When you build with Rork, you can tap into these capabilities to give your app that polished, "feels premium" quality that keeps users coming back.

In this guide, you'll learn how to implement haptic feedback in a Rork app — from the fundamentals to real-world implementation patterns.

What you'll learn:
- The different types of haptic feedback and when to use each
- Implementing haptics in Rork using `expo-haptics`
- Cross-platform patterns for iOS and Android
- Design principles to avoid haptic overload
- Common pitfalls and how to fix them

**Who this is for:** Developers with some Rork experience who want to elevate their app's UX. A basic understanding of React Native components will help.

## Types of Haptic Feedback

### iOS Haptic Categories

iOS organizes haptic feedback into three main categories:

**① Impact Feedback**
Simulates a physical collision — the sensation that something has snapped into place. Use it when something meaningful happens in the UI, like placing a dragged item or selecting a cell.

| Intensity | Example Use Case |
|-----------|-----------------|
| `light` | Checkbox toggles, small UI elements |
| `medium` | Standard button presses, list row selections |
| `heavy` | Significant actions, delete confirmations |
| `rigid` | Firm contact sensation (iOS 13+) |
| `soft` | Gentle landing sensation (iOS 13+) |

**② Notification Feedback**
Communicates the result of an operation. Think of it as the tactile equivalent of a toast notification.

| Type | When to Use |
|------|-------------|
| `success` | Operation completed, data saved |
| `warning` | Caution required before proceeding |
| `error` | Validation failed, something went wrong |

**③ Selection Feedback**
A subtle tick used when a value changes incrementally — perfect for pickers and sliders.

### Android Haptic Feedback

Android uses `HapticFeedbackConstants` under the hood, but when building with Rork (React Native / Expo), the `expo-haptics` library abstracts these differences so you can write cross-platform code without worrying about platform internals.

## Implementing Haptics in Rork

### Step 1: Setting Up expo-haptics

In most Rork projects, `expo-haptics` is already included as part of the Expo SDK. If you need to add it manually:

```bash
npx expo install expo-haptics
```

No additional native configuration is required for managed Expo/Rork projects.

### Step 2: Basic Impact Feedback on Button Press

```tsx
import * as Haptics from 'expo-haptics';
import { TouchableOpacity, Text, StyleSheet } from 'react-native';

// A button component with built-in haptic feedback
const HapticButton = ({ onPress, children, intensity = 'medium' }) => {
  const handlePress = async () => {
    // Fire haptic before (or alongside) the action
    await Haptics.impactAsync(
      intensity === 'light'
        ? Haptics.ImpactFeedbackStyle.Light
        : intensity === 'heavy'
        ? Haptics.ImpactFeedbackStyle.Heavy
        : Haptics.ImpactFeedbackStyle.Medium
    );
    onPress?.();
  };

  return (
    <TouchableOpacity style={styles.button} onPress={handlePress}>
      <Text style={styles.label}>{children}</Text>
    </TouchableOpacity>
  );
};

const styles = StyleSheet.create({
  button: {
    backgroundColor: '#007AFF',
    paddingHorizontal: 24,
    paddingVertical: 12,
    borderRadius: 12,
    alignItems: 'center',
  },
  label: {
    color: '#fff',
    fontSize: 16,
    fontWeight: '600',
  },
});

// Usage:
// <HapticButton onPress={() => console.log('tapped')} intensity="medium">
//   Save Changes
// </HapticButton>
```

### Step 3: Notification Feedback for Operation Results

```tsx
import * as Haptics from 'expo-haptics';
import { useState } from 'react';

// Form submission with result-based haptic feedback
const FormSubmitExample = () => {
  const [status, setStatus] = useState<'idle' | 'success' | 'error'>('idle');

  const handleSubmit = async () => {
    try {
      // Async operation (API call, etc.)
      await submitForm();

      // Success: a satisfying, rewarding pulse
      await Haptics.notificationAsync(
        Haptics.NotificationFeedbackType.Success
      );
      setStatus('success');
    } catch (err) {
      // Error: a sharper, attention-grabbing vibration
      await Haptics.notificationAsync(
        Haptics.NotificationFeedbackType.Error
      );
      setStatus('error');
    }
  };

  return (/* ... */);
};
```

### Step 4: Selection Feedback for Pickers and Sliders

```tsx
import * as Haptics from 'expo-haptics';
import { useState } from 'react';
import { View, Text } from 'react-native';

// A scroll picker that gives tactile feedback on each value change
const HapticPicker = ({ items, onSelect }) => {
  const [selectedIndex, setSelectedIndex] = useState(0);

  const handleScroll = (index: number) => {
    if (index !== selectedIndex) {
      // Fire a subtle tick every time the selection changes
      Haptics.selectionAsync();
      setSelectedIndex(index);
      onSelect?.(items[index]);
    }
  };

  return (
    <View>
      {items.map((item, index) => (
        <Text
          key={index}
          onPress={() => handleScroll(index)}
          style={index === selectedIndex ? styles.selected : styles.item}
        >
          {item}
        </Text>
      ))}
    </View>
  );
};

// Expected behavior:
// → Each time the selected value changes, the user feels a light "tick"
// → This mirrors the native iOS picker wheel experience
// → Users can navigate without looking at the screen
```

## Haptic Design Principles

### Principle 1: Less Is More

The fastest way to annoy your users is to over-vibrate. If every tap on every element triggers a heavy impact, users will either disable haptics system-wide or just stop using your app.

**Recommended haptic usage by action type:**

| Action | Recommended Feedback |
|--------|---------------------|
| Save, submit, complete | `Notification.Success` |
| Error or validation fail | `Notification.Error` |
| Primary button tap | `Impact.Medium` or `Impact.Light` |
| List selection, picker change | `Selection` |
| Navigation / screen transition | None |
| Icon tap in a tab bar | None (system handles it) |

### Principle 2: Account for Platform Differences

iOS's Taptic Engine is significantly more expressive than most Android vibration motors. The same API call may feel crisp and precise on an iPhone, but slightly coarser on Android. Here's how to adapt:

```tsx
import { Platform } from 'react-native';
import * as Haptics from 'expo-haptics';

// Platform-adaptive haptic helper
const adaptiveHaptic = async (type: 'success' | 'error' | 'tap') => {
  if (Platform.OS === 'ios') {
    // iOS supports nuanced differentiation
    switch (type) {
      case 'success':
        await Haptics.notificationAsync(Haptics.NotificationFeedbackType.Success);
        break;
      case 'error':
        await Haptics.notificationAsync(Haptics.NotificationFeedbackType.Error);
        break;
      case 'tap':
        await Haptics.impactAsync(Haptics.ImpactFeedbackStyle.Light);
        break;
    }
  } else {
    // On Android, keep it simple to avoid jarring vibrations
    if (type !== 'tap') {
      await Haptics.impactAsync(Haptics.ImpactFeedbackStyle.Medium);
    }
  }
};
```

### Principle 3: Respect User System Settings

iOS allows users to disable haptic feedback system-wide under **Settings > Sounds & Haptics**. The good news: `expo-haptics` automatically respects this preference — your app doesn't need to do anything extra.

That said, consider adding an in-app toggle for haptics in your settings screen. It's a small touch that shows users you care about their preferences.

## Common Errors and Fixes

### Error 1: Haptics Not Firing on Some Devices

Older Android devices or certain device configurations may not support the haptic APIs. Always wrap haptic calls in a try-catch:

```tsx
const safeHaptic = async () => {
  try {
    await Haptics.impactAsync(Haptics.ImpactFeedbackStyle.Medium);
  } catch {
    // Silently ignore — don't surface haptic errors to users
  }
};
```

### Error 2: No Vibration on Android

Check that your `AndroidManifest.xml` includes the VIBRATE permission:

```xml
<!-- android/app/src/main/AndroidManifest.xml -->
<uses-permission android:name="android.permission.VIBRATE" />
```

Rork's standard templates typically include this, but it's worth verifying if you're getting zero feedback on Android.

### Error 3: Haptics Not Working in the Simulator

iOS simulators don't support haptic feedback. Always test haptics on a physical device. Some Android emulators do support vibration, but real device testing is strongly recommended.

## Advanced: Building Custom Haptic Sequences

For special moments in your app — a game achievement, a payment success screen — you can chain multiple haptic calls into a custom sequence:

```tsx
import * as Haptics from 'expo-haptics';

// A "celebration" haptic sequence for milestone moments
const celebrationHaptic = async () => {
  await Haptics.impactAsync(Haptics.ImpactFeedbackStyle.Light);
  await new Promise(r => setTimeout(r, 100));
  await Haptics.impactAsync(Haptics.ImpactFeedbackStyle.Medium);
  await new Promise(r => setTimeout(r, 100));
  await Haptics.notificationAsync(Haptics.NotificationFeedbackType.Success);
};

// Usage: call when user hits a milestone
// await celebrationHaptic();

// Expected output:
// → Light pulse → (100ms pause) → Medium pulse → (100ms pause) → Success notification
// → Creates a "tada" feeling that rewards the user
```

Use custom sequences sparingly — save them for moments that genuinely deserve a celebration. Overusing this pattern diminishes its impact.

## FAQ

**Q. Does using haptic feedback affect App Store review?**

A. Haptic feedback itself doesn't cause App Store rejections. Apple's [Human Interface Guidelines](https://developer.apple.com/design/human-interface-guidelines/playing-haptics) recommend using haptics purposefully — to confirm actions, signal errors, and reflect value changes. Sticking to these three use cases keeps you well within Apple's guidelines.

**Q. What's the difference between expo-haptics and react-native-haptic-feedback?**

A. `expo-haptics` works seamlessly in Expo managed workflow (which Rork uses) with no native configuration needed. `react-native-haptic-feedback` offers more granular Android control but requires native setup. For Rork projects, `expo-haptics` is the recommended choice.

**Q. I'm building a game where collisions happen frequently. Will constant haptics drain the battery?**

A. Frequent haptic calls do affect battery life and device temperature. For high-frequency scenarios like game collisions, use `Impact.Light`, throttle feedback to no more than 10–15 times per second, or make haptics opt-in for players. Aim for feedback that feels responsive without feeling relentless.

- **Reinforces interactions** — users know their tap registered without checking the screen

- **Elevates perceived quality** — small details signal that real care went into the product

Start simple: add `Impact.Medium` to your primary action buttons and `Notification.Success` to your save confirmations. Once you feel the difference on a real device, you'll wonder how you shipped without it.
