## Introduction

With Rork, you can rapidly prototype mobile apps using AI. But as your app grows — more screens, more features, more complexity — you'll inevitably hit the wall of "I can't find anything in this codebase" and "changing one thing breaks something else."

This guide addresses these challenges with three architectural pillars: MVVM (Model-View-ViewModel) for separating UI from logic, Clean Architecture for layered design, and dependency injection for testability. Integrating these patterns into your Rork workflow will help you build apps that remain maintainable in the long run.

## Why Architecture Patterns Matter

### Technical Debt You Don't See in Small Apps

Code generated by Rork often mixes business logic, API calls, and state management within a single component. For a small app with five screens, this works fine. But once you pass twenty screens, problems start surfacing:

- Components exceeding 500 lines, making it hard to locate what needs fixing
- API specification changes requiring simultaneous edits across multiple screens
- Difficulty writing tests because components depend directly on APIs

### Three Problems Architecture Patterns Solve

- **Separation of concerns**: Clearly dividing UI display, business logic, and data fetching responsibilities minimizes the blast radius of changes
- **Reusability**: Extracting logic from components enables sharing across different screens
- **Testability**: Making dependencies injectable enables unit testing with mocks

## The MVVM Pattern — Separation of Concerns with Custom Hooks

### MVVM Structure Basics

MVVM (Model-View-ViewModel) clearly separates the UI layer from business logic. In the React Native / Expo context, it maps like this:

- **Model**: Data type definitions and persistence logic
- **View**: React components (responsible only for display)
- **ViewModel**: Custom hooks (responsible for business logic and state management)

### Implementation Example: Task Management Screen

Start with the Model (type definitions):

```typescript
// models/Task.ts
export interface Task {
  id: string;
  title: string;
  completed: boolean;
  createdAt: Date;
  priority: "low" | "medium" | "high";
}

export interface TaskFilter {
  status: "all" | "active" | "completed";
  priority?: Task["priority"];
}
```

Next, implement the ViewModel (custom hook):

```typescript
// viewmodels/useTaskListViewModel.ts
import { useState, useMemo, useCallback } from "react";
import { Task, TaskFilter } from "../models/Task";

export function useTaskListViewModel(initialTasks: Task[]) {
  const [tasks, setTasks] = useState<Task[]>(initialTasks);
  const [filter, setFilter] = useState<TaskFilter>({ status: "all" });

  // Memoized filtered task list
  const filteredTasks = useMemo(() => {
    return tasks.filter((task) => {
      if (filter.status === "active" && task.completed) return false;
      if (filter.status === "completed" && !task.completed) return false;
      if (filter.priority && task.priority !== filter.priority) return false;
      return true;
    });
  }, [tasks, filter]);

  // Toggle task completion
  const toggleTask = useCallback((id: string) => {
    setTasks((prev) =>
      prev.map((t) =>
        t.id === id ? { ...t, completed: !t.completed } : t
      )
    );
  }, []);

  // Add a new task
  const addTask = useCallback(
    (title: string, priority: Task["priority"] = "medium") => {
      const newTask: Task = {
        id: Date.now().toString(),
        title,
        completed: false,
        createdAt: new Date(),
        priority,
      };
      setTasks((prev) => [newTask, ...prev]);
    },
    []
  );

  // Statistics
  const stats = useMemo(
    () => ({
      total: tasks.length,
      active: tasks.filter((t) => !t.completed).length,
      completed: tasks.filter((t) => t.completed).length,
    }),
    [tasks]
  );

  return {
    filteredTasks,
    filter,
    setFilter,
    toggleTask,
    addTask,
    stats,
  };
}
```

Finally, create the View (component). It uses only the values and functions returned by the ViewModel — no business logic whatsoever:

```tsx
// screens/TaskListScreen.tsx
import React from "react";
import { View, Text, FlatList, TouchableOpacity } from "react-native";
import { useTaskListViewModel } from "../viewmodels/useTaskListViewModel";

export function TaskListScreen({ initialTasks }) {
  const { filteredTasks, toggleTask, stats } =
    useTaskListViewModel(initialTasks);

  return (
    <View style={{ flex: 1, padding: 16 }}>
      <Text style={{ fontSize: 18, fontWeight: "bold" }}>
        Tasks: {stats.active} / {stats.total}
      </Text>
      <FlatList
        data={filteredTasks}
        keyExtractor={(item) => item.id}
        renderItem={({ item }) => (
          <TouchableOpacity onPress={() => toggleTask(item.id)}>
            <Text
              style={{
                textDecorationLine: item.completed
                  ? "line-through"
                  : "none",
                padding: 12,
              }}
            >
              {item.title}
            </Text>
          </TouchableOpacity>
        )}
      />
    </View>
  );
}
```

With this structure, UI changes stay in the View and logic changes stay in the ViewModel, making each independently testable.

## Clean Architecture — Structuring Your App in Three Layers

### Layer Composition

Clean Architecture divides your application into three layers:

- **Domain layer**: Business rules, entities, use cases (no external dependencies)
- **Data layer**: Repository implementations, API clients, local storage
- **Presentation layer**: UI components, ViewModels, navigation

The critical rule is that dependencies flow from the outside in. The Domain layer must never depend on the Data or Presentation layers.

### Directory Structure Design

```
src/
├── domain/
│   ├── entities/          # Entities (type definitions)
│   │   └── Task.ts
│   ├── repositories/      # Repository interfaces
│   │   └── ITaskRepository.ts
│   └── usecases/          # Use cases
│       ├── GetTasksUseCase.ts
│       └── CreateTaskUseCase.ts
├── data/
│   ├── repositories/      # Repository implementations
│   │   └── TaskRepository.ts
│   ├── api/               # API clients
│   │   └── taskApi.ts
│   └── local/             # Local storage
│       └── taskStorage.ts
└── presentation/
    ├── screens/            # Screen components
    ├── components/         # Shared UI components
    └── viewmodels/         # ViewModels (custom hooks)
```

### Implementing Use Cases

A use case describes a single unit of application behavior, with each use case corresponding to one business operation.

```typescript
// domain/usecases/GetTasksUseCase.ts
import { Task, TaskFilter } from "../entities/Task";
import { ITaskRepository } from "../repositories/ITaskRepository";

export class GetTasksUseCase {
  constructor(private taskRepository: ITaskRepository) {}

  async execute(filter: TaskFilter): Promise<Task[]> {
    const tasks = await this.taskRepository.getAll();

    return tasks.filter((task) => {
      if (filter.status === "active" && task.completed) return false;
      if (filter.status === "completed" && !task.completed) return false;
      return true;
    });
  }
}
```

```typescript
// domain/repositories/ITaskRepository.ts
import { Task } from "../entities/Task";

export interface ITaskRepository {
  getAll(): Promise<Task[]>;
  getById(id: string): Promise<Task | null>;
  create(task: Omit<Task, "id" | "createdAt">): Promise<Task>;
  update(id: string, data: Partial<Task>): Promise<Task>;
  delete(id: string): Promise<void>;
}
```

### Repository Implementation (Data Layer)

```typescript
// data/repositories/TaskRepository.ts
import { Task } from "../../domain/entities/Task";
import { ITaskRepository } from "../../domain/repositories/ITaskRepository";
import { taskApi } from "../api/taskApi";

export class TaskRepository implements ITaskRepository {
  async getAll(): Promise<Task[]> {
    const response = await taskApi.fetchTasks();
    return response.map(this.mapToEntity);
  }

  async getById(id: string): Promise<Task | null> {
    const response = await taskApi.fetchTask(id);
    return response ? this.mapToEntity(response) : null;
  }

  async create(
    data: Omit<Task, "id" | "createdAt">
  ): Promise<Task> {
    const response = await taskApi.createTask(data);
    return this.mapToEntity(response);
  }

  async update(id: string, data: Partial<Task>): Promise<Task> {
    const response = await taskApi.updateTask(id, data);
    return this.mapToEntity(response);
  }

  async delete(id: string): Promise<void> {
    await taskApi.deleteTask(id);
  }

  // Convert API response to domain entity
  private mapToEntity(raw: any): Task {
    return {
      id: raw.id,
      title: raw.title,
      completed: raw.completed,
      createdAt: new Date(raw.created_at),
      priority: raw.priority,
    };
  }
}
```

## Dependency Injection — Enabling Testable Code

### Dependency Injection with React Context

Use React's Context API to make repository and use case implementations injectable:

```typescript
// di/ServiceContainer.ts
import React, { createContext, useContext } from "react";
import { ITaskRepository } from "../domain/repositories/ITaskRepository";
import { GetTasksUseCase } from "../domain/usecases/GetTasksUseCase";
import { CreateTaskUseCase } from "../domain/usecases/CreateTaskUseCase";

export interface ServiceContainer {
  taskRepository: ITaskRepository;
  getTasksUseCase: GetTasksUseCase;
  createTaskUseCase: CreateTaskUseCase;
}

const ServiceContext = createContext<ServiceContainer | null>(null);

export function ServiceProvider({
  container,
  children,
}: {
  container: ServiceContainer;
  children: React.ReactNode;
}) {
  return (
    <ServiceContext.Provider value={container}>
      {children}
    </ServiceContext.Provider>
  );
}

export function useService(): ServiceContainer {
  const context = useContext(ServiceContext);
  if (!context) {
    throw new Error(
      "useService must be used within a ServiceProvider"
    );
  }
  return context;
}
```

### Injecting Mocks for Testing

Thanks to dependency injection, you can inject mock repositories during testing:

```typescript
// __tests__/GetTasksUseCase.test.ts
import { GetTasksUseCase } from "../domain/usecases/GetTasksUseCase";
import { ITaskRepository } from "../domain/repositories/ITaskRepository";

// Mock repository
const mockRepository: ITaskRepository = {
  getAll: async () => [
    {
      id: "1",
      title: "Test Task",
      completed: false,
      createdAt: new Date(),
      priority: "medium",
    },
    {
      id: "2",
      title: "Completed Task",
      completed: true,
      createdAt: new Date(),
      priority: "high",
    },
  ],
  getById: async () => null,
  create: async (data) => ({
    ...data,
    id: "new",
    createdAt: new Date(),
  }),
  update: async (id, data) => ({} as any),
  delete: async () => {},
};

describe("GetTasksUseCase", () => {
  it("should filter active tasks", async () => {
    const useCase = new GetTasksUseCase(mockRepository);
    const result = await useCase.execute({ status: "active" });
    // Verify that the completed task is excluded
    expect(result).toHaveLength(1);
    expect(result[0].title).toBe("Test Task");
  });
});
```

## Practical Workflow with Rork

### Communicating Architecture Patterns to Rork

When generating apps with Rork, include your design principles in the prompt to get pattern-aligned code from the start:

```
Create a task management app following these architecture patterns:

1. MVVM Pattern:
   - Business logic goes in custom hooks (viewmodels/)
   - Components are responsible only for display

2. Directory structure:
   - src/domain/entities/ — type definitions
   - src/domain/usecases/ — business logic
   - src/data/repositories/ — API communication
   - src/presentation/screens/ — screens
   - src/presentation/viewmodels/ — custom hooks

3. Dependency injection:
   - Use repositories through interfaces
   - Inject implementations via React Context
```

### Refactoring Existing Code

Here's a step-by-step approach to gradually refactoring Rork-generated code:

1. **Extract Models first**: Move type definitions from components to `domain/entities/`
2. **Create ViewModels**: Extract `useState` and related logic into custom hooks
3. **Introduce Repositories**: Centralize API calls in `data/repositories/`
4. **Add Tests**: Write unit tests for use cases and ViewModels

You don't need to change everything at once. Start with one screen, get comfortable with the patterns, and then gradually expand.

## Frequently Asked Questions

### Do small apps really need Clean Architecture?

For apps with just three to five screens, MVVM (custom hook separation) alone is often sufficient. Introducing all Clean Architecture layers makes more sense once your app has ten-plus screens and multiple data sources (API + local DB). Over-engineering can actually reduce productivity, so adopt patterns incrementally based on your app's scale.

### How do I reconcile Rork-generated code with architecture patterns?

A practical approach is to quickly generate a prototype with Rork, verify it works, then refactor along architectural lines. Including design principles in your Rork prompts can also help generate pattern-aligned code from the start. Rather than chasing perfection, "build something that works, then refine" is the most efficient path.

### Can I use state management libraries (Zustand, Redux) alongside these patterns?

Yes. A natural fit is to use Zustand stores in the ViewModel layer, with components accessing state only through ViewModels. The choice of state management library and architecture patterns are independent decisions — combining them actually makes each layer's responsibilities clearer.

### Should I use TypeScript interfaces or types?

A common convention is to use `interface` for repository definitions (which benefit from `extends`) and `type` for entity data structures (which work better with union and mapping types). The key is establishing a consistent team rule to avoid confusion.

## Summary

Architecture patterns are often seen as something only large-scale apps need, but adopting them early brings clearer code organization and significantly reduces the time spent fixing bugs and adding features.

Separate UI from logic with MVVM, clarify layer responsibilities with Clean Architecture, and ensure testability with dependency injection — these three pillars are the foundation for elevating Rork-generated apps to production quality.

Start by separating the ViewModel (custom hook) in just one screen. Combine it with [advanced state management patterns](/articles/rork-dev/rork-advanced-state-management-patterns) and [custom hook reuse techniques](/articles/rork-dev/rork-custom-hooks-reusable-logic), and add [test automation](/articles/rork-dev/rork-max-testing-guide) to complete a robust app architecture.

For deeper exploration of these concepts, [Clean Architecture](https://www.amazon.co.jp/dp/4048930656/?tag=pinocchio-22) by Robert C. Martin is an essential read that provides the foundational thinking behind all the patterns covered in this guide.
