## Why Image Optimization Can Make or Break Your App

In mobile apps, images are one of the biggest factors affecting user experience. Beautiful visuals make your app appealing, but unoptimized images can slow everything to a crawl and drive users away.

Studies show that roughly 40% of users abandon an app if images take more than 3 seconds to load. Whether you're building with Rork or any React Native/Expo stack, how you handle images directly impacts user satisfaction and store ratings.

This guide walks you through image optimization fundamentals for Rork apps — from caching strategies and memory management to responsive image delivery. Every technique includes working code examples you can drop into your project right away.

## Fast Image Rendering with Expo Image

Since Rork apps are built on React Native + Expo, the `expo-image` library is your best friend for image display. Compared to React Native's built-in `Image` component, `expo-image` comes with built-in disk and memory caching, delivering a significant performance advantage out of the box.

```tsx
// Basic image display with expo-image
import { Image } from 'expo-image';

// Optimized image component with BlurHash placeholder
const OptimizedImage = ({ uri, blurhash, width, height }) => {
  return (
    <Image
      source={{ uri }}
      placeholder={{ blurhash }}
      contentFit="cover"
      transition={300}
      style={{ width, height, borderRadius: 8 }}
      cachePolicy="memory-disk"
      recyclingKey={uri}
    />
  );
};

// Usage:
// <OptimizedImage
//   uri="https://example.com/photo.webp"
//   blurhash="LKO2:N%2Tw=w]~RBVZRi};RPxuwH"
//   width={300}
//   height={200}
// />
// → BlurHash displays instantly, then fades into the actual image
```

Setting `cachePolicy` to `"memory-disk"` ensures that once an image is loaded, it's cached in both memory and on disk. Subsequent displays of the same image skip the network request entirely, resulting in near-instant rendering.

## Designing a Caching Strategy — The 3-Layer Cache Architecture

An effective image caching system uses three layers, each serving a different purpose.

**Memory Cache (L1)** is the fastest layer. It holds images currently on screen or likely to appear soon. Since memory is finite, images are automatically evicted using an LRU (Least Recently Used) policy.

**Disk Cache (L2)** persists images in local storage. Even after an app restart, previously loaded images are available instantly. `expo-image` manages this layer automatically.

**CDN Cache (L3)** operates on the server side. By serving images through a CDN (Content Delivery Network), users receive images from the nearest edge server, speeding up even first-time loads.

```tsx
// Image cache manager leveraging all three layers
import { Image } from 'expo-image';
import * as FileSystem from 'expo-file-system';

const IMAGE_CACHE_DIR = FileSystem.cacheDirectory + 'images/';

// Initialize cache directory
const ensureCacheDir = async () => {
  const dirInfo = await FileSystem.getInfoAsync(IMAGE_CACHE_DIR);
  if (!dirInfo.exists) {
    await FileSystem.makeDirectoryAsync(IMAGE_CACHE_DIR, {
      intermediates: true,
    });
  }
};

// Prefetch images for the next screen
const prefetchImages = async (urls: string[]) => {
  await Promise.all(
    urls.map((url) => Image.prefetch(url))
  );
};

// Check cache size for management
const getCacheSize = async () => {
  const dirInfo = await FileSystem.getInfoAsync(IMAGE_CACHE_DIR, {
    size: true,
  });
  const sizeMB = (dirInfo.size || 0) / (1024 * 1024);
  return sizeMB; // Returns size in MB
};

// Clear cache (call from settings screen)
const clearImageCache = async () => {
  await Image.clearDiskCache();
  await Image.clearMemoryCache();
  console.log('Image cache cleared');
};

// Expected output:
// After prefetchImages completes, those images display instantly
// getCacheSize() → e.g., 45.2 (MB)
```

## Placeholders and Progressive Display

Showing nothing while an image loads is the worst possible experience. Placeholders keep your app feeling responsive even during load times.

**BlurHash** encodes an image into a compact 20–30 character string. By pre-computing this on the server and including it in API responses, you can display a blurred preview instantly before the full image arrives.

```tsx
// Image grid with BlurHash placeholders
import { Image } from 'expo-image';
import { FlatList, View, StyleSheet } from 'react-native';

const ImageGrid = ({ images }) => {
  const renderItem = ({ item }) => (
    <View style={styles.imageContainer}>
      <Image
        source={{ uri: item.url }}
        placeholder={{ blurhash: item.blurhash }}
        contentFit="cover"
        transition={250}
        style={styles.image}
        cachePolicy="memory-disk"
        // Off-screen images load at lower priority
        priority={item.isVisible ? 'high' : 'low'}
      />
    </View>
  );

  return (
    <FlatList
      data={images}
      renderItem={renderItem}
      keyExtractor={(item) => item.id}
      numColumns={2}
      // Limit off-screen render window to save memory
      windowSize={5}
      maxToRenderPerBatch={6}
      removeClippedSubviews={true}
    />
  );
};

const styles = StyleSheet.create({
  imageContainer: {
    flex: 1,
    margin: 4,
    borderRadius: 8,
    overflow: 'hidden',
  },
  image: {
    width: '100%',
    aspectRatio: 1,
  },
});

// Expected behavior:
// 1. BlurHash placeholders appear instantly in the grid
// 2. Images fade in over 250ms once loaded
// 3. Off-screen images are lazy-loaded to conserve memory
```

## Responsive Images — Serving the Right Size for Every Device

The same image doesn't need to be full resolution on every device. An iPhone SE's small screen needs far fewer pixels than an iPad Pro. Sending full-resolution images to all devices wastes bandwidth unnecessarily.

Server-side image resizing services like Cloudflare Images or Imgix let you request images at the exact dimensions you need via URL parameters.

```tsx
// Utility to generate device-optimized image URLs
import { Dimensions, PixelRatio } from 'react-native';

const getOptimizedImageUrl = (
  baseUrl: string,
  displayWidth: number
) => {
  // Account for device pixel density
  const pixelWidth = Math.round(
    displayWidth * PixelRatio.get()
  );

  // Cap at reasonable maximum
  const targetWidth = Math.min(pixelWidth, 1200);

  // Cloudflare Images example
  return `${baseUrl}?w=${targetWidth}&format=webp&q=80`;
};

// Usage:
const screenWidth = Dimensions.get('window').width;
const imageUrl = getOptimizedImageUrl(
  'https://example.com/images/hero.jpg',
  screenWidth
);
// iPhone 15 Pro (PixelRatio: 3, width: 393):
// → "https://example.com/images/hero.jpg?w=1179&format=webp&q=80"
// iPhone SE (PixelRatio: 2, width: 375):
// → "https://example.com/images/hero.jpg?w=750&format=webp&q=80"
```

Using WebP format typically reduces file sizes by 25–35% compared to JPEG at equivalent visual quality. A quality setting of `q=80` strikes a good balance between file size and perceived quality — the compression artifacts are virtually unnoticeable in most cases.

## Memory Management — Preventing App Crashes

Apps that handle lots of images need careful memory management. Memory leaks and excessive usage lead directly to crashes, especially on older devices with limited RAM.

```tsx
// Memory-efficient image list implementation
import { useCallback, useEffect, useRef } from 'react';
import { Image } from 'expo-image';
import { FlatList, AppState } from 'react-native';

const MemoryEfficientImageList = ({ images }) => {
  const flatListRef = useRef(null);

  // Release memory when app goes to background
  useEffect(() => {
    const subscription = AppState.addEventListener(
      'change',
      (nextAppState) => {
        if (nextAppState === 'background') {
          Image.clearMemoryCache();
        }
      }
    );
    return () => subscription.remove();
  }, []);

  const renderItem = useCallback(
    ({ item }) => (
      <Image
        source={{ uri: item.url }}
        style={{ width: '100%', height: 200 }}
        contentFit="cover"
        cachePolicy="disk"
        recyclingKey={item.id}
      />
    ),
    []
  );

  // Improve FlatList scroll performance with fixed layouts
  const getItemLayout = useCallback(
    (data, index) => ({
      length: 200,
      offset: 200 * index,
      index,
    }),
    []
  );

  return (
    <FlatList
      ref={flatListRef}
      data={images}
      renderItem={renderItem}
      keyExtractor={(item) => item.id}
      getItemLayout={getItemLayout}
      windowSize={3}
      maxToRenderPerBatch={4}
      removeClippedSubviews={true}
      initialNumToRender={5}
    />
  );
};

// Expected behavior:
// - Memory usage stays stable during scrolling (no sudden spikes)
// - Memory cache auto-clears when app enters background
// - removeClippedSubviews unmounts off-screen images
```

The `recyclingKey` property enables efficient view recycling inside `expo-image`, keeping memory usage low during scrolling. Setting `cachePolicy` to `"disk"` skips the memory cache entirely and relies only on disk cache — a useful trade-off for image-heavy list screens where memory conservation matters most.

## Measuring Performance and Iterating

To properly evaluate your optimizations, you need concrete numbers. Here's a basic approach to measuring image performance in your Rork app.

```tsx
// Custom hook for tracking image load metrics
import { useState, useCallback } from 'react';

const useImageLoadMetrics = () => {
  const [metrics, setMetrics] = useState({
    loadCount: 0,
    totalLoadTime: 0,
    averageLoadTime: 0,
    cacheHitRate: 0,
    cacheHits: 0,
  });

  const onLoadStart = useCallback(() => {
    return Date.now();
  }, []);

  const onLoadEnd = useCallback(
    (startTime: number, fromCache: boolean) => {
      const loadTime = Date.now() - startTime;
      setMetrics((prev) => {
        const newCount = prev.loadCount + 1;
        const newTotal = prev.totalLoadTime + loadTime;
        const newCacheHits = prev.cacheHits + (fromCache ? 1 : 0);
        return {
          loadCount: newCount,
          totalLoadTime: newTotal,
          averageLoadTime: Math.round(newTotal / newCount),
          cacheHits: newCacheHits,
          cacheHitRate: Math.round(
            (newCacheHits / newCount) * 100
          ),
        };
      });
    },
    []
  );

  return { metrics, onLoadStart, onLoadEnd };
};

// Usage:
// const { metrics } = useImageLoadMetrics();
// console.log(metrics);
// → { loadCount: 50, totalLoadTime: 8500,
//     averageLoadTime: 170, cacheHits: 35,
//     cacheHitRate: 70 }
```

Aim for an average load time under 200ms and a cache hit rate above 60% — these benchmarks correlate with apps that users perceive as fast and responsive. ## Frequently Asked Questions

### What's the difference between expo-image and React Native's built-in Image component?

`expo-image` includes built-in two-layer caching (disk + memory), BlurHash/ThumbHash placeholder support, WebP/AVIF format handling, and view recycling. The standard `Image` component offers only basic caching. For apps with significant image content, `expo-image` delivers noticeably better performance. It's the recommended choice for all new Rork projects.

### How large should I set my cache size limit?

A disk cache limit of 100–200MB works well for most apps. This range keeps frequently displayed images cached without consuming too much of the user's device storage. For photo-heavy apps, you can extend to around 300MB without issues. The default settings in `expo-image` are well-tuned for typical use cases.

### Should I use WebP or JPEG?

As of 2026, WebP is strongly recommended for mobile apps. It delivers 25–35% smaller file sizes than JPEG at equivalent visual quality, and it supports transparency (alpha channel). Both iOS and Android have full WebP support, so compatibility isn't a concern. The easiest way to adopt WebP is to use a server-side image transformation service and specify `format=webp` in your requests.

### Any tips for asking Rork to implement image optimization?

When generating apps with Rork, be specific in your prompts: "Use expo-image for all image display with BlurHash placeholders and disk caching enabled." Adding concrete parameters like "Set FlatList windowSize to 5 and enable removeClippedSubviews" helps Rork generate higher-quality, performance-optimized code.
