## Recognizing Image and Media Loading Failures in Rork Apps

When you're testing a Rork-generated app on a physical device or simulator, you might run into frustrating issues: images that refuse to load, spinners that never stop, or outright crashes when displaying a gallery of photos.

These image and media loading errors are notoriously tricky to diagnose because the root cause can range from simple URL typos to complex memory management issues. Is it a network problem? An image format incompatibility? A corrupted cache? Or is your app running out of memory?

In this guide, we'll walk through each failure pattern systematically, analyze the underlying causes, and provide step-by-step solutions you can apply immediately. Since Rork generates React Native (Expo) code that uses the `Image` component or the `expo-image` library, we'll cover the specific gotchas for each approach.

## Common Causes of Image Display Failures

Image loading failures in Rork apps typically fall into a few well-defined categories. Start by identifying which pattern matches your situation.

**Pattern 1: Incorrect URL or File Path**

This is the most basic cause. The remote image URL has a typo, the local asset path is wrong, or the `require()` statement isn't structured correctly. With Rork-generated code, it's common for placeholder image URLs to remain in the output without being replaced with real assets.

**Pattern 2: Mixed HTTPS/HTTP Content (iOS App Transport Security)**

iOS enforces App Transport Security (ATS) by default, which blocks non-encrypted HTTP connections. If your image server doesn't support HTTPS, images will silently fail to load on iOS devices. This issue is easy to miss because the same images work perfectly on Android.

**Pattern 3: Image Format Incompatibility**

WebP images work well on Android but may cause issues on older iOS versions (iOS 14 and earlier). Similarly, HEIC/HEIF images that are common on iOS won't render on most Android devices without additional libraries.

**Pattern 4: CORS Errors in Web Preview**

When images fail to display in Expo's web preview but work fine on native devices, the culprit is usually Cross-Origin Resource Sharing (CORS) policy. The image server needs to return the appropriate `Access-Control-Allow-Origin` header.

**Pattern 5: Memory Exhaustion from High-Resolution Images**

Loading many high-resolution images simultaneously — especially in a `FlatList` or `ScrollView` — can exhaust available memory and crash your app. This is one of the most common issues in image-heavy applications like galleries or e-commerce apps.

## Step 1: Validate URLs and File Paths

Start by confirming that your image URLs and paths are correct. Use this debug wrapper component to visualize the loading state of every image in your app.

```tsx
// components/DebugImage.tsx
// A wrapper component that surfaces image loading errors visually
import React, { useState } from 'react';
import { Image, View, Text, StyleSheet } from 'react-native';

interface DebugImageProps {
  uri: string;
  width: number;
  height: number;
}

export function DebugImage({ uri, width, height }: DebugImageProps) {
  const [status, setStatus] = useState<'loading' | 'success' | 'error'>('loading');
  const [errorMsg, setErrorMsg] = useState('');

  return (
    <View style={styles.container}>
      <Image
        source={{ uri }}
        style={{ width, height }}
        onLoad={() => setStatus('success')}
        onError={(e) => {
          setStatus('error');
          setErrorMsg(e.nativeEvent.error || 'Unknown error');
        }}
      />
      {status === 'loading' && <Text style={styles.info}>Loading: {uri}</Text>}
      {status === 'error' && (
        <Text style={styles.error}>Error: {errorMsg}{'\n'}URL: {uri}</Text>
      )}
    </View>
  );
}

const styles = StyleSheet.create({
  container: { marginVertical: 8 },
  info: { fontSize: 12, color: '#666' },
  error: { fontSize: 12, color: '#ff4444', backgroundColor: '#fff0f0', padding: 8 },
});

// Expected output:
// On success → Image renders normally, status text is hidden
// On failure → Red error text with the specific error message and URL
```

This component makes it immediately clear which images are failing and why. Once you've identified the problematic URLs, you can fix them directly.

For local assets, make sure your `require()` paths are correct.

```tsx
// ✅ Correct local image loading
<Image source={require('../assets/images/logo.png')} />

// ❌ Common mistake — path doesn't exist
<Image source={require('./images/logo.png')} />

// ❌ Common mistake — dynamic require doesn't work in React Native
const imageName = 'logo';
<Image source={require(`../assets/images/${imageName}.png`)} /> // Won't work
```

## Step 2: Fix Network and Security Configuration

If images fail to load specifically on iOS, App Transport Security is likely blocking the HTTP request. You can add exceptions in your `app.json` (or `app.config.js`).

```json
{
  "expo": {
    "ios": {
      "infoPlist": {
        "NSAppTransportSecurity": {
          "NSAllowsArbitraryLoads": false,
          "NSExceptionDomains": {
            "your-image-server.com": {
              "NSExceptionAllowsInsecureHTTPLoads": true,
              "NSIncludesSubdomains": true
            }
          }
        }
      }
    }
  }
}
```

Avoid setting `NSAllowsArbitraryLoads: true` as a blanket override — it weakens your app's security posture. Instead, migrate your image server to HTTPS and only whitelist specific legacy domains as exceptions.

For CORS issues, verify the response headers from your image server. If you're using Supabase Storage or Cloudflare R2, check the bucket's CORS configuration.

```bash
# Check the response headers for your image URL
curl -I https://your-bucket.supabase.co/storage/v1/object/public/images/photo.jpg

# Expected output (with CORS properly configured):
# Access-Control-Allow-Origin: *
# Content-Type: image/jpeg
# Content-Length: 245890
```

## Step 3: Migrate to expo-image for Better Reliability

The standard React Native `Image` component has known limitations around cache management and memory efficiency. If you're experiencing persistent image issues in your Rork project, migrating to `expo-image` is one of the most impactful fixes you can make.

```bash
# Install expo-image
npx expo install expo-image
```

```tsx
// Using expo-image with automatic cache management
import { Image } from 'expo-image';

function OptimizedImage({ uri }: { uri: string }) {
  return (
    <Image
      source={{ uri }}
      style={{ width: 300, height: 200 }}
      contentFit="cover"
      placeholder={require('../assets/placeholder.png')}
      transition={200}
      cachePolicy="memory-disk"
      onError={(e) => console.warn('Image loading error:', e.error)}
    />
  );
}

// Expected behavior:
// 1. Placeholder image appears immediately
// 2. Remote image fades in over 200ms once loaded
// 3. Subsequent views load instantly from cache
```

Under the hood, `expo-image` uses `SDWebImage` on iOS and `Glide` on Android — both battle-tested native libraries that handle caching, format conversion, and memory management far better than the standard component. Key advantages include automatic memory and disk caching, built-in placeholder and transition support, cross-platform WebP and AVIF support, and efficient memory reclamation for off-screen images.

## Step 4: Optimize Memory for Large Image Lists

When displaying images in a `FlatList` or `SectionList`, you need to configure the list component to manage memory efficiently. Without these settings, rendering hundreds of images will crash your app.

```tsx
// Memory-efficient image list rendering
import { FlatList } from 'react-native';
import { Image } from 'expo-image';

function ImageGallery({ images }: { images: string[] }) {
  return (
    <FlatList
      data={images}
      keyExtractor={(item, index) => `img-${index}`}
      // ✅ Release off-screen elements to save memory
      removeClippedSubviews={true}
      // ✅ Limit the render window to reduce memory pressure
      windowSize={5}
      // ✅ Limit initial render count
      initialNumToRender={6}
      // ✅ Limit batch rendering
      maxToRenderPerBatch={4}
      renderItem={({ item }) => (
        <Image
          source={{ uri: item }}
          style={{ width: '100%', height: 200 }}
          contentFit="cover"
          // ✅ Blurhash placeholder for perceived performance
          placeholder={{ blurhash: 'LKN]Rv%2Tw=w]~RBVZRi};RPxuwH' }}
          recyclingKey={item}
        />
      )}
    />
  );
}
```

The `removeClippedSubviews` and `windowSize` settings are critical here. By default, `FlatList` keeps off-screen elements in memory, which adds up quickly with image-heavy content.

Don't overlook image resolution, either. If your server delivers 4000×3000 pixel originals for what's displayed as a 300×200 thumbnail, each image consumes roughly 48MB of memory. Serve appropriately sized images through your CDN's resize feature, or implement server-side resizing at upload time.

## Verifying the Fix

After applying your fixes, run through this verification checklist.

1. **Test on both platforms**: Launch the app with `npx expo start` and verify image display on both iOS and Android devices
2. **Test network recovery**: Toggle airplane mode on and off, then confirm images reload correctly. If caching is working, previously viewed images should appear instantly
3. **Monitor memory usage**: Use React Native Debugger or Xcode Instruments to watch the memory graph while scrolling through image lists
4. **Check error logs**: Confirm no new errors are appearing in `onError` callbacks or console output

```bash
# Stream React Native logs in real time
npx react-native log-ios
# or
npx react-native log-android

# Filter for image-related errors only
npx react-native log-ios 2>&1 | grep -i "image\|photo\|media\|load"
```

## Prevention — Best Practices to Avoid Future Issues

Here are the design principles that will help you avoid image-related problems from the start.

**1. Build an image optimization pipeline early**

Set up image resizing, compression, and format conversion before your app grows. If you're using Supabase Storage or Cloudflare R2, pair it with an Edge Function that automatically resizes uploads. For a detailed implementation guide, check out [Building File Storage with Rork and Cloudflare R2](/en/articles/rork-dev/rork-cloudflare-r2-file-storage-guide).

**2. Isolate image errors with fallback UI**

Prevent a single broken image from crashing your entire app. Use `expo-image`'s `placeholder` prop or wrap image-heavy sections in React Error Boundaries to gracefully handle failures.

**3. Establish format standards for your project**

Decide on a consistent image format and document it. WebP is generally the best choice for cross-platform apps, but if you need to support iOS 14 or earlier, maintain a JPEG/PNG fallback strategy. For more on image optimization and caching, see [Image Optimization and Cache Strategy for Rork Apps](/en/articles/rork-dev/rork-image-optimization-cache-strategy).

**4. Implement a cache clearing mechanism**

Corrupted caches can cause images to stop displaying even when the source is fine. Include a "Clear Cache" option in your app's settings screen so users can resolve the issue themselves.

```tsx
// Clear expo-image caches programmatically
import { Image } from 'expo-image';

async function clearImageCache() {
  await Image.clearDiskCache();
  await Image.clearMemoryCache();
  Alert.alert('Done', 'Image cache has been cleared');
}
```

## Frequently Asked Questions

### Q1: Only some images fail to load in my Rork-generated app. What should I check first?

Start by using the `DebugImage` component from Step 1 to identify which specific images are failing and what error messages they produce. In most cases, the issue is either a URL typo or a missing CORS configuration on the image server. Also check whether Rork left any placeholder URLs (like `https://via.placeholder.com/...`) in the generated code that were never replaced with real image sources.

### Q2: Images display fine on Android but won't load on iOS. What's causing this?

This is almost always caused by iOS App Transport Security blocking HTTP (non-HTTPS) requests. Follow the ATS configuration steps in Step 2 of this guide. Additionally, if your images are in HEIC format, they won't render on most Android devices — standardize on JPEG or WebP for cross-platform compatibility.

### Q3: My app crashes when scrolling through an image list. How do I fix it?

This is a memory exhaustion issue. Apply the `FlatList` optimization settings from Step 4 — specifically `removeClippedSubviews`, `windowSize`, and `maxToRenderPerBatch`. Just as importantly, check the resolution of images your server is delivering. Serving 4000×3000 pixel images for 300×200 thumbnails wastes approximately 48MB of memory per image, which adds up rapidly in a scrollable list.

### Q4: Should I use expo-image or the standard React Native Image component?

For new projects, `expo-image` is the clear winner. It outperforms the standard `Image` component in caching, placeholder support, format compatibility, and memory efficiency. If you're working on an existing project that uses the standard `Image` component, the migration is relatively straightforward since the APIs are largely compatible. We recommend switching if you're experiencing recurring image display issues.

## Summary

Image and media loading errors in Rork apps stem from a variety of causes, so a systematic approach to diagnosis is essential. Work through the issues methodically: validate URLs first, then check network and security settings, upgrade to `expo-image` for reliability, and finally optimize memory management for image-heavy screens.

Migrating to `expo-image` is arguably the single most effective improvement you can make. For apps that rely heavily on images — galleries, e-commerce, social feeds — it's worth adopting early in your development process.
