## Introduction

February 2026 marked a pivotal moment in React Native's evolution with the release of **React Native 0.84**, which makes **Hermes V1 the default JavaScript engine**, completely replacing JavaScriptCore (JSC). Simultaneously, Expo SDK 55 announced the complete discontinuation of Legacy Architecture, making **New Architecture the only viable path forward**.

This comprehensive guide covers React Native 0.84's transformative changes, the performance revolution brought by Hermes V1, migration strategies, and practical implications for Rork Max application development.

## React Native 0.84: Major Architectural Shifts

### Hermes V1 Becomes the Default Engine

The headline feature of React Native 0.84 is that **Hermes V1 is now the mandatory default JavaScript engine**. This represents the culmination of years of development and marks the end of an era where developers had a choice between Hermes and JSC.

Hermes is Meta's JavaScript engine specifically optimized for mobile environments. Unlike JSC (which originated from Safari), Hermes was architected from the ground up with mobile constraints in mind: low RAM devices, slower processors, and battery efficiency.

Key characteristics of Hermes:
- **Ahead-of-time bytecode compilation**: JavaScript code is compiled during the build process
- **Mobile-first design**: Every feature decision prioritizes mobile constraints
- **Minimal memory footprint**: Aggressive optimization for devices with limited RAM
- **Deterministic performance**: Predictable performance characteristics for better debugging

### New Architecture Becomes Mandatory

Alongside Hermes V1, **New Architecture is now the only supported architecture** in React Native 0.84. Legacy Architecture—the older bridge-based system—is no longer supported in any form.

This consolidation simplifies the React Native ecosystem:
- **TurboModules**: A faster native module bridge replacing the old bridge
- **Fabric**: A modernized renderer that replaces legacy JSX rendering
- **JSI (JavaScript Interface)**: Direct JavaScript-to-native communication without serialization

The transition from dual-architecture support to a single unified architecture represents a major simplification for library authors and app developers alike.

| Garbage Collection Pause | 12ms | 45ms | 73% reduction |

### Bytecode Precompilation: The Game Changer

Hermes V1's revolutionary feature is **precompiling JavaScript to bytecode during the build process**, rather than parsing JavaScript at runtime.

**Traditional JSC Workflow:**
```
1. App launches
2. JavaScript bundle loaded into memory
3. JavaScript parser tokenizes source code
4. AST (Abstract Syntax Tree) generated
5. JIT compilation to machine code
6. Execution begins
```

**Hermes V1 Workflow:**
```
1. Build time: JavaScript → bytecode
2. App launches
3. Bytecode loaded directly
4. Execution begins (skip all parsing overhead)
```

This elimination of startup-time parsing provides:
- **Faster cold starts**: No parser overhead during app launch
- **Reduced memory**: No intermediate representations stored in memory
- **Predictable performance**: Consistent startup times regardless of bundle size

### Generational Garbage Collection

Hermes V1 implements a sophisticated multi-generational garbage collection system:

```
Generation 0 (Young): Most frequently garbage collected
  └─ Objects dying immediately
  └─ Negligible GC pause time

Generation 1 (Intermediate): Less frequently collected
  └─ Objects surviving young generation
  └─ Moderate GC pause time

Generation 2 (Old): Rarely collected
  └─ Long-lived objects
  └─ Minimal collection frequency
```

**Benefits:**
- **Frame rate stability**: Average GC pause reduced from 45ms to 12ms
- **Reduced jank**: Better UI responsiveness during garbage collection
- **Predictable memory**: Aging algorithm ensures old objects aren't repeatedly scanned

## Expo SDK 55: The End of Legacy Architecture

### Breaking Change: Legacy Architecture Removed

Expo SDK 55 makes a definitive architectural choice: **Legacy Architecture is completely removed**. This is not a deprecation—it is an end-of-life announcement.

```
OLD: Developers could choose between Legacy and New Architecture
NEW: New Architecture is the only option
```

For developers still using Legacy Architecture:
- Immediate migration required for Expo SDK 55 compatibility
- No backward compatibility layer exists
- Updates to `app.json` are mandatory

### The newArchEnabled Flag Elimination

In previous Expo versions, New Architecture was opt-in via the `newArchEnabled` flag:

```json
{
  "expo": {
    "newArchEnabled": true,
    "plugins": ["react-native-worklets-core"]
  }
}
```

**In Expo SDK 55, this flag is completely removed and ignored.** New Architecture is the implicit default.

**Updated app.json:**

```json
{
  "expo": {
    "name": "MyApp",
    "version": "1.0.0",
    "plugins": ["react-native-worklets-core"],
    "sdkVersion": "55"
  }
}
```

### EAS Build Acceleration: 30% Faster

Expo SDK 55 introduces advanced caching mechanisms for EAS Build:

**Build Performance Improvements:**

| Scenario | Previous Time | New Time | Savings |
|----------|---------------|----------|---------|
| Cached build (no code changes) | 8 minutes | 2.5 minutes | 69% |
| Incremental build (small changes) | 8 minutes | 5.6 minutes | 30% |
| Clean build | 12 minutes | 10.2 minutes | 15% |

**Cache Optimization Strategy:**
- Layer 1 cache: Build artifacts (Maven, CocoaPods)

The caching system achieves approximately **70% cache hit rates** for typical development workflows.

## Migration Guide: From Older Versions to 0.84

### Pre-Migration Checklist

Before migrating, ensure:
- [ ] Node.js v22.11 or higher installed
- [ ] All npm dependencies up-to-date
- [ ] Git repository is clean (all changes committed)
- [ ] CI/CD pipelines backed up
- [ ] Team informed of breaking changes

### Step 1: Update Dependencies

The safest method is using `expo@latest`:

```bash
npx expo@latest upgrade
```

This command:
- Updates React Native to 0.84
- Updates Expo to SDK 55
- Handles related dependencies
- Creates a migration report

**Alternative: Manual update via package.json**

```bash
npm install react-native@0.84 expo@55
npm install --save-dev @react-native/community-cli@14
```

### Step 2: Remove newArchEnabled Flag

Search `app.json` for any `newArchEnabled` references and remove them:

```diff
{
  "expo": {
    "name": "MyApp",
-   "newArchEnabled": true,
    "plugins": [
      ["react-native-worklets-core"]
    ]
  }
}
```

Also check `eas.json`:

```diff
{
  "build": {
    "production": {
-     "env": { "EXPO_PUBLIC_NEW_ARCH": "true" }
    }
  }
}
```

### Step 3: iOS Configuration Updates

Update `ios/Podfile`:

```ruby
# Ensure this block exists in ios/Podfile post_install
post_install do |installer|
  react_native_post_install(
    installer,
    config[:reactNativePath],
    :mac_catalyst_enabled => false,
    :architecture => :new  # Explicitly set to new
  )
end
```

Install pods:

```bash
cd ios && pod install && cd ..
```

### Step 4: Android Configuration Updates

Update `android/build.gradle`:

```gradle
project.ext.react = [
  enableHermes: true,  // Hermes is now default
  enableNewArchitecture: true  // New Architecture is default
]
```

Update `android/gradle.properties`:

```properties
# Hermes engine
hermesEnabled=true

# New Architecture
newArchEnabled=true
```

### Step 5: Validate Library Compatibility

Some libraries may require updates for New Architecture compatibility:

```bash
# Run Expo's diagnostic tool
npx expo doctor --fix

# View compatibility report
npx expo doctor
```

Look for warnings about:
- Libraries with bridge-only native code
- Older CocoaPods specifications
- Gradle dependency conflicts

### Step 6: Test Locally

**iOS testing:**

```bash
# Clean build
rm -rf ios/Pods ios/Podfile.lock node_modules package-lock.json

# Reinstall
npm install
cd ios && pod install && cd ..

# Run on device/simulator
npx react-native run-ios --mode=Release
```

**Android testing:**

```bash
# Clean build
cd android && ./gradlew clean
cd ..

# Run on device/emulator
npx react-native run-android --mode=release
```

**Expo development:**

```bash
npx expo start

# Press 'i' for iOS simulator
# Press 'a' for Android emulator
```

### Step 7: Verify Hermes Bytecode

Confirm that Hermes bytecode compilation is working:

```bash
# iOS Release build
npx react-native run-ios --mode=Release

# Check device logs
xcrun simctl spawn booted log stream --predicate 'eventMessage contains[cd] "hermes"'
```

Output should show:
```
Loaded Hermes bytecode bundle
Hermes engine initialized
```

### Step 8: Build Optimization Passes

```bash
# Metro bundler optimization report
npx react-native bundle \
  --platform android \
  --dev false \
  --entry-file index.js \
  --bundle-output app.bundle \
  --max-workers 4

# View bundle size analysis
ls -lh app.bundle
```

## Troubleshooting Common Issues

### Issue 1: "Hermes is not enabled"

**Symptoms:**
- App starts slowly
- `HermesInternal` is undefined
- Build log shows JSC instead of Hermes

**Solutions:**

```bash
# Clear all caches
rm -rf node_modules package-lock.json
rm -rf ios/Pods ios/Podfile.lock
rm -rf android/.gradle

# Reinstall
npm install
cd ios && pod install && cd ..

# Verify configuration
grep -r "enableHermes" android/

# Should output: enableHermes=true
```

### Issue 2: "newArchEnabled: true" causes build failure

**Symptoms:**
- "Unknown property: newArchEnabled"
- Build fails at gradle compilation stage

**Solutions:**

```bash
# Remove from all config files
grep -r "newArchEnabled" . --include="*.json" --include="*.gradle"

# Remove matches from:
# - app.json
# - eas.json
# - android/build.gradle
# - android/gradle.properties
```

### Issue 3: "Legacy Architecture library incompatibility"

**Symptoms:**
- Build succeeds, but runtime crash
- "Bridge error" or "Module not found"

**Solutions:**

```bash
# Check library compatibility
npm outdated

# Update specific libraries
npm install @react-native-async-storage/async-storage@latest

# Or run Expo doctor
npx expo doctor --fix
```

### Issue 4: EAS Build cache issues

**Symptoms:**
- Local build works, EAS build fails
- "Native module not found"

**Solutions:**

```bash
# Clear EAS cache
eas build --clear-cache --platform ios

# Force clean build
eas build --platform ios --no-cache

# Check build logs
eas build:view --id <build-id>
```
