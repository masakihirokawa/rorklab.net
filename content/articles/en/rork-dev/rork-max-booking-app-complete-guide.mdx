## Architecture and Design Principles

### Database Schema Design

A well-designed database is the foundation of any booking system. Getting the schema right from the start will make everything else easier.

The core tables for a booking system are:

1. **services** — The types of services offered, including pricing and duration
2. **time_slots** — Available time windows for booking
3. **bookings** — User reservation records
4. **payments** — Stripe payment transaction logs

### Booking Flow Design

The user journey from browsing to confirmed booking follows this path:

1. Select service → 2. Pick a date → 3. Choose a time slot → 4. Review details → 5. Pay via Stripe → 6. Booking confirmed → 7. Confirmation notification

Each step is implemented incrementally using Rork Max.

## Step-by-Step Implementation

### Step 1: Create Supabase Tables

Open the SQL Editor in your Supabase dashboard and run the following schema:

```sql
-- Services table
CREATE TABLE services (
  id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  name TEXT NOT NULL,
  description TEXT,
  duration_minutes INTEGER NOT NULL DEFAULT 60,
  price INTEGER NOT NULL,
  category TEXT,
  image_url TEXT,
  is_active BOOLEAN DEFAULT true,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- Time slots table
CREATE TABLE time_slots (
  id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  service_id UUID REFERENCES services(id),
  date DATE NOT NULL,
  start_time TIME NOT NULL,
  end_time TIME NOT NULL,
  max_bookings INTEGER DEFAULT 1,
  current_bookings INTEGER DEFAULT 0,
  is_available BOOLEAN DEFAULT true,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE(service_id, date, start_time)
);

-- Bookings table
CREATE TABLE bookings (
  id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  user_id UUID REFERENCES auth.users(id),
  service_id UUID REFERENCES services(id),
  time_slot_id UUID REFERENCES time_slots(id),
  status TEXT DEFAULT 'pending'
    CHECK (status IN ('pending','confirmed','cancelled','completed')),
  notes TEXT,
  stripe_payment_intent_id TEXT,
  total_price INTEGER NOT NULL,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW()
);

-- Payments table
CREATE TABLE payments (
  id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  booking_id UUID REFERENCES bookings(id),
  stripe_payment_intent_id TEXT NOT NULL,
  amount INTEGER NOT NULL,
  currency TEXT DEFAULT 'jpy',
  status TEXT DEFAULT 'pending'
    CHECK (status IN ('pending','succeeded','failed','refunded')),
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- Row Level Security policies
ALTER TABLE bookings ENABLE ROW LEVEL SECURITY;

CREATE POLICY "Users can view own bookings"
  ON bookings FOR SELECT
  USING (auth.uid() = user_id);

CREATE POLICY "Users can create bookings"
  ON bookings FOR INSERT
  WITH CHECK (auth.uid() = user_id);

ALTER TABLE time_slots ENABLE ROW LEVEL SECURITY;

CREATE POLICY "Anyone can view available slots"
  ON time_slots FOR SELECT
  USING (is_available = true);
```

Tell Rork Max to "add the Supabase connection configuration" and the AI will generate environment variables and connection code automatically.

The generated calendar component fetches data from Supabase in real time and visually displays availability. Here is the key logic:

```typescript
// Generate marked dates for the calendar
const generateMarkedDates = (slots: TimeSlot[]) => {
  const marked: Record<string, MarkedDateProps> = {};

  slots.forEach(slot => {
    const dateStr = slot.date;
    if (slot.current_bookings >= slot.max_bookings) {
      marked[dateStr] = {
        marked: true,
        dotColor: '#9CA3AF',
        customStyles: {
          container: { backgroundColor: '#F3F4F6' }
        }
      };
    } else {
      marked[dateStr] = {
        marked: true,
        dotColor: '#3B82F6',
        customStyles: {
          container: { backgroundColor: '#EFF6FF' }
        }
      };
    }
  });
  return marked;
};
```

### Step 4: Integrate Stripe Payments

Pre-booking payments help prevent no-shows. Set up a Supabase Edge Function as your payment server:

```typescript
// supabase/functions/create-payment-intent/index.ts
import Stripe from 'https://esm.sh/stripe@14.14.0';

const stripe = new Stripe(Deno.env.get('STRIPE_SECRET_KEY')!);

Deno.serve(async (req) => {
  const { booking_id, amount } = await req.json();

  const paymentIntent = await stripe.paymentIntents.create({
    amount: amount,
    currency: 'jpy',
    metadata: { booking_id },
    automatic_payment_methods: { enabled: true },
  });

  const supabase = createClient(
    Deno.env.get('SUPABASE_URL')!,
    Deno.env.get('SUPABASE_SERVICE_ROLE_KEY')!
  );

  await supabase.from('payments').insert({
    booking_id,
    stripe_payment_intent_id: paymentIntent.id,
    amount,
    status: 'pending',
  });

  return new Response(JSON.stringify({
    clientSecret: paymentIntent.client_secret,
  }));
});
```

Tell Rork Max to "integrate Stripe Payment Sheet into the booking confirmation screen" and the client-side code will be generated automatically. Apple Pay and Google Pay are enabled by default through `automatic_payment_methods`.

### Step 5: Set Up Push Notifications

Configure notifications for booking confirmations and reminders:

```sql
-- Check for 24-hour reminders every hour (Supabase pg_cron)
SELECT cron.schedule(
  'booking-reminders',
  '0 * * * *',
  $$
  SELECT net.http_post(
    url := 'https://your-project.supabase.co/functions/v1/send-reminder',
    body := json_build_object(
      'bookings',
      (SELECT json_agg(row_to_json(b))
       FROM bookings b
       JOIN time_slots ts ON b.time_slot_id = ts.id
       WHERE b.status = 'confirmed'
       AND ts.date = CURRENT_DATE + INTERVAL '1 day'
       AND NOT b.reminder_sent)
    )::jsonb
  );
  $$
);
```

### Automated Cancellation Policies

Implement business-specific cancellation rules that automatically calculate refund amounts:

```typescript
const calculateRefundAmount = (
  booking: Booking,
  slot: TimeSlot
): number => {
  const now = new Date();
  const bookingDate = new Date(
    `${slot.date}T${slot.start_time}`
  );
  const hoursUntil =
    (bookingDate.getTime() - now.getTime()) / (1000 * 60 * 60);

  if (hoursUntil >= 48) return booking.total_price;
  if (hoursUntil >= 24) return booking.total_price * 0.5;
  return 0;
};
```

This applies a full refund for cancellations 48+ hours before the appointment, 50% refund for 24-48 hours, and no refund within 24 hours. Combined with the Stripe Refund API, the entire cancellation and refund process can be fully automated.

## Troubleshooting

### Supabase Realtime Not Working

Verify that your RLS policies are correctly configured. Realtime filters data according to RLS, so the `time_slots` table needs a `SELECT` policy. Also check that the table is enabled for replication in the Supabase dashboard under Database > Replication.

### Stripe Payment Sheet Not Appearing

When using Stripe with Expo, you need `@stripe/stripe-react-native` installed and an EAS development build. The Stripe SDK uses native modules that do not work in Expo Go. Always test with `eas build --profile development`.

### Double Bookings Occurring

Use PostgreSQL transaction isolation when updating `current_bookings` in the `time_slots` table. The query `UPDATE ... WHERE current_bookings < max_bookings` will return zero affected rows if the slot is already full, allowing you to handle the "fully booked" case gracefully. This optimistic locking approach prevents race conditions.

## FAQ

**Q: How do I prevent double bookings from concurrent access?**

Use Supabase transactions combined with unique constraints. Update `current_bookings` in the `time_slots` table with `UPDATE ... WHERE current_bookings < max_bookings`. If the update affects zero rows, the slot is full. This optimistic locking approach safely handles concurrent access.

**Q: Can users view their bookings offline?**

Yes. By enabling offline support in Rork Max, you can cache recent booking data locally and automatically sync when connectivity is restored. See the [Offline-First Architecture Guide](/en/articles/rork-dev/rork-max-offline-first-architecture) for implementation details.

**Q: How do I separate admin and regular user permissions?**

Use Supabase RLS policies combined with custom claims in `app_metadata`. Admins get full read/write access to all bookings, while regular users can only access their own reservations.

**Q: How can I support multiple business locations?**

Add a `shop_id` column to the `services` table and manage services and time slots per location. With tenant-based RLS policies, you can scale the app into a multi-tenant SaaS booking platform.

**Q: What is the best way to generate test data?**

Use the Supabase SQL Editor to bulk insert seed data. Ask Rork Max to "generate SQL to insert 30 sample services and time slots for testing" and the AI will create the seed statements for you.

**Q: Can I send email notifications after a booking is confirmed?**

Absolutely. Combine Supabase Edge Functions with Resend (an email delivery API) to automatically send booking confirmation and reminder emails. Simply tell Rork Max to "add email notifications via Resend integration" and the scaffolding will be generated.
