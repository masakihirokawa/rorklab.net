## What Is Cloudflare R2—and Why Is It a Great Fit for Rork Apps?

When building apps with Rork, one of the first infrastructure questions you'll face is: where should user-generated files go? Profile photos, in-app images, and video content all need somewhere reliable and affordable to live. While AWS S3 and Firebase Storage are popular choices, **Cloudflare R2** has become the go-to solution for many indie developers in 2026.

Cloudflare R2 is an S3-compatible object storage service with one standout advantage: **zero egress fees**. Unlike AWS S3, which charges for every byte transferred out of storage, R2 lets you serve files to your users at no extra cost. This alone can cut infrastructure bills dramatically for apps with active media sharing.

There are three key reasons R2 pairs beautifully with Rork apps.

First, **cost efficiency**. R2's free tier includes 10GB of storage, 1 million Class B operations (reads), and 1 million Class A operations (writes) per month. For early-stage apps and solo developers, this often means running file storage completely free. Second, **Cloudflare Workers integration**. If you're already using Cloudflare Workers as your API layer, you get native R2 bucket bindings with no additional setup. Third, **S3 API compatibility** means you can use existing AWS SDK tooling without rewriting anything.

This guide walks through building a complete file upload pipeline—from creating an R2 bucket to uploading images and videos directly from a Rork (React Native / Expo) app.

## Prerequisites and Project Setup

Before diving in, make sure you have the following ready:

- A Cloudflare account (free tier works)
- A Rork or Expo project with React Native
- Familiarity with Node.js basics
- Basic knowledge of Cloudflare Workers (used as the API backend)

Our architecture routes uploads through Cloudflare Workers rather than letting the mobile app write to R2 directly. This keeps your API credentials server-side and lets you enforce file type validation, size limits, and authentication before anything reaches storage.

## Step 1: Creating Your R2 Bucket

Log in to the Cloudflare dashboard and navigate to **R2 Object Storage** in the sidebar. Click **Create bucket**, give it a meaningful name (e.g., `my-app-uploads`), and select **Automatic** for the region. Cloudflare will distribute your data globally based on where requests originate.

Next, generate an API token so your Workers can interact with R2. Go to **My Profile → API Tokens → Create Token**, and select the **Cloudflare R2:Edit** permission template.

## Step 2: Building the Cloudflare Workers API

Rather than exposing R2 credentials to the mobile client, we create a lightweight Worker that receives upload requests and handles R2 directly.

Start by scaffolding a new Workers project:

```bash
# Install Wrangler CLI if you haven't already
npm install -g wrangler

# Create a new Workers project
npm create cloudflare@latest my-app-api
cd my-app-api
```

Add the R2 binding to your `wrangler.toml`:

```toml
name = "my-app-api"
main = "src/index.ts"
compatibility_date = "2026-04-01"

[[r2_buckets]]
binding = "FILE_BUCKET"
bucket_name = "my-app-uploads"
```

Now implement the upload endpoints in your Worker:

```typescript
// src/index.ts
import { Hono } from 'hono'
import { cors } from 'hono/cors'

type Bindings = {
  FILE_BUCKET: R2Bucket
}

const app = new Hono<{ Bindings: Bindings }>()

// CORS configuration (lock down origins in production)
app.use('*', cors({
  origin: '*',
  allowHeaders: ['Content-Type', 'Authorization'],
  allowMethods: ['GET', 'POST', 'PUT', 'DELETE', 'OPTIONS'],
}))

// Generate upload URL endpoint
app.post('/api/upload/presign', async (c) => {
  const { fileName, contentType } = await c.req.json()

  // Validate file type
  const allowedTypes = ['image/jpeg', 'image/png', 'image/webp', 'video/mp4']
  if (!allowedTypes.includes(contentType)) {
    return c.json({ error: 'Unsupported file type' }, 400)
  }

  // Generate a unique storage key
  const timestamp = Date.now()
  const random = Math.random().toString(36).substring(2, 8)
  const ext = fileName.split('.').pop() || 'bin'
  const key = `uploads/${timestamp}-${random}.${ext}`

  return c.json({
    uploadUrl: `https://my-app-api.workers.dev/api/upload/${key}`,
    fileKey: key,
  })
})

// Accept file upload
app.put('/api/upload/:key{.+}', async (c) => {
  const key = c.req.param('key')
  const contentType = c.req.header('Content-Type') || 'application/octet-stream'

  // Enforce 10MB limit
  const body = await c.req.arrayBuffer()
  if (body.byteLength > 10 * 1024 * 1024) {
    return c.json({ error: 'File too large. Max 10MB.' }, 413)
  }

  // Write to R2
  await c.env.FILE_BUCKET.put(key, body, {
    httpMetadata: { contentType },
  })

  // Return the public CDN URL
  const publicUrl = `https://pub-xxxxx.r2.dev/${key}`

  return c.json({ success: true, url: publicUrl, key })
})

// Delete a file
app.delete('/api/files/:key{.+}', async (c) => {
  const key = c.req.param('key')
  await c.env.FILE_BUCKET.delete(key)
  return c.json({ success: true })
})

export default app
```

Deploy your Worker:

```bash
wrangler deploy

# Expected output:
# Uploaded my-app-api (3.42 sec)
# Published my-app-api (0.21 sec)
# https://my-app-api.your-account.workers.dev
```

## Step 3: Implementing Uploads in Your Rork App

With the backend in place, let's wire up the upload flow in the Rork app (React Native / Expo).

Install the necessary Expo packages:

```bash
npx expo install expo-image-picker expo-document-picker
```

Create a reusable upload utility:

```typescript
// utils/fileUpload.ts
const API_BASE = 'https://my-app-api.your-account.workers.dev'

export interface UploadResult {
  url: string
  key: string
}

/**
 * Uploads a local file to Cloudflare R2 via the Workers API.
 * @param localUri  URI of the local file (from expo-image-picker, etc.)
 * @param fileName  Desired file name
 * @param contentType  MIME type (e.g., 'image/jpeg')
 */
export async function uploadFileToR2(
  localUri: string,
  fileName: string,
  contentType: string
): Promise<UploadResult> {
  // Step 1: Request an upload URL from the Worker
  const presignRes = await fetch(`${API_BASE}/api/upload/presign`, {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ fileName, contentType }),
  })

  if (!presignRes.ok) throw new Error('Failed to get upload URL')

  const { uploadUrl, fileKey } = await presignRes.json()

  // Step 2: Read the local file as binary
  const fileRes = await fetch(localUri)
  const blob = await fileRes.blob()

  // Step 3: PUT the file to R2 via the Worker
  const uploadRes = await fetch(uploadUrl, {
    method: 'PUT',
    headers: { 'Content-Type': contentType },
    body: blob,
  })

  if (!uploadRes.ok) throw new Error('Upload failed')

  const result = await uploadRes.json()
  return { url: result.url, key: fileKey }
}
```

Build an image picker component with upload support:

```typescript
// components/ImageUploader.tsx
import React, { useState } from 'react'
import { View, Image, TouchableOpacity, Text, ActivityIndicator, Alert } from 'react-native'
import * as ImagePicker from 'expo-image-picker'
import { uploadFileToR2 } from '../utils/fileUpload'

interface ImageUploaderProps {
  onUploadComplete: (url: string) => void
}

export default function ImageUploader({ onUploadComplete }: ImageUploaderProps) {
  const [uploading, setUploading] = useState(false)
  const [previewUri, setPreviewUri] = useState<string | null>(null)

  const pickAndUpload = async () => {
    const { status } = await ImagePicker.requestMediaLibraryPermissionsAsync()
    if (status !== 'granted') {
      Alert.alert('Permission Required', 'Please allow access to your photo library.')
      return
    }

    const result = await ImagePicker.launchImageLibraryAsync({
      mediaTypes: ImagePicker.MediaTypeOptions.Images,
      allowsEditing: true,
      aspect: [4, 3],
      quality: 0.8, // Compress to 80% to reduce file size
    })

    if (result.canceled || !result.assets[0]) return

    const asset = result.assets[0]
    setPreviewUri(asset.uri)
    setUploading(true)

    try {
      const fileName = asset.fileName || `photo_${Date.now()}.jpg`
      const contentType = asset.mimeType || 'image/jpeg'
      const { url } = await uploadFileToR2(asset.uri, fileName, contentType)
      onUploadComplete(url)
      Alert.alert('Success', 'Your image has been uploaded!')
    } catch (error) {
      Alert.alert('Error', 'Upload failed. Please try again.')
      console.error('Upload error:', error)
    } finally {
      setUploading(false)
    }
  }

  return (
    <View style={{ alignItems: 'center', gap: 12 }}>
      {previewUri && (
        <Image
          source={{ uri: previewUri }}
          style={{ width: 200, height: 150, borderRadius: 8 }}
          resizeMode="cover"
        />
      )}
      <TouchableOpacity
        onPress={pickAndUpload}
        disabled={uploading}
        style={{
          backgroundColor: uploading ? '#ccc' : '#F6821F',
          paddingHorizontal: 24,
          paddingVertical: 12,
          borderRadius: 8,
        }}
      >
        {uploading
          ? <ActivityIndicator color="#fff" />
          : <Text style={{ color: '#fff', fontWeight: 'bold' }}>Pick & Upload Image</Text>
        }
      </TouchableOpacity>
    </View>
  )
}
```

## Step 4: Video Upload Support

For video files, the same approach works. Use `expo-document-picker` to let users select a video, then pass it through the same upload utility:

```typescript
// utils/videoUpload.ts
import * as DocumentPicker from 'expo-document-picker'
import { uploadFileToR2 } from './fileUpload'

export async function pickAndUploadVideo(
  onProgress?: (percent: number) => void
): Promise<string | null> {
  const result = await DocumentPicker.getDocumentAsync({
    type: 'video/*',
    copyToCacheDirectory: true,
  })

  if (result.canceled || !result.assets[0]) return null

  const asset = result.assets[0]

  // Check against your size limit
  if (asset.size && asset.size > 10 * 1024 * 1024) {
    throw new Error('Video files must be under 10MB.')
  }

  onProgress?.(20)
  const { url } = await uploadFileToR2(asset.uri, asset.name, asset.mimeType || 'video/mp4')
  onProgress?.(100)

  return url
}
```

For larger video files (100MB+), consider using **Cloudflare Stream** instead—it's purpose-built for video hosting, transcoding, and adaptive bitrate delivery.

## Step 5: Production Security Hardening

Before going live, tighten up your Worker's security posture.

Lock down CORS to specific origins:

```typescript
// Production CORS — restrict to your app's domains
app.use('*', cors({
  origin: (origin) => {
    const allowed = ['https://myapp.com', 'exp://localhost:8081']
    return allowed.includes(origin) ? origin : null
  },
}))
```

If your app uses authentication (Supabase Auth, Clerk, etc.), validate JWTs in the Worker before allowing uploads:

```typescript
// Middleware example for JWT validation
app.use('/api/upload/*', async (c, next) => {
  const token = c.req.header('Authorization')?.replace('Bearer ', '')
  if (!token) return c.json({ error: 'Unauthorized' }, 401)
  // Verify token with your auth provider here
  await next()
})
```

## Step 6: Enabling Public CDN Access

To make uploaded files publicly accessible, go to your R2 bucket settings in the Cloudflare dashboard and enable **Public Access**. This gives you a public URL in the format `https://pub-xxxxx.r2.dev/your-file-key`.

You can also attach a **custom domain** (e.g., `https://files.myapp.com/`) through Cloudflare's DNS settings, giving your users a branded URL for all media.

Since R2 files are served through Cloudflare's global CDN, your users get fast load times regardless of their location—without paying a cent for bandwidth.

## Troubleshooting Common Errors

**CORS errors in the app**

Double-check that the `cors()` middleware is applied before your route handlers in the Worker. During development, use `origin: '*'` and tighten it before deploying to production.

**Uploads silently fail on mobile**

Mobile networks can be unreliable. Add retry logic with exponential backoff, and use `expo-network` to check connectivity before initiating uploads. Consider queuing failed uploads locally and retrying when the connection is restored.

**R2 binding not found error**

The `binding` name in `wrangler.toml` and the TypeScript type definition (`Bindings`) must match exactly. If you rename one, update both. Run `wrangler deploy` again to apply binding changes—local dev with `wrangler dev` also picks them up automatically.

**Request body too large**

Workers have a 100MB request body limit. For large video files, switch to R2's native multipart upload API or route uploads directly through a Cloudflare Stream endpoint, bypassing the Worker body limit entirely.

## FAQ

**Q: Should I use Supabase Storage or Cloudflare R2?**

A: If you're already using Supabase for your database and auth, Supabase Storage is the more integrated choice—especially for Row Level Security (RLS) policies on file access. If your backend runs on Cloudflare Workers or you expect heavy traffic with lots of file downloads, R2's zero-egress pricing makes it significantly cheaper at scale.

**Q: Can I do image resizing or format conversion with R2?**

A: R2 itself is pure object storage with no transformation capabilities. For on-the-fly image optimization, pair R2 with **Cloudflare Images**—it handles WebP conversion, resizing, and smart cropping via URL parameters. Alternatively, compress and resize on the device side using `expo-image-picker`'s `quality` and `aspect` options before upload.

**Q: What does R2 actually cost in practice?**

A: The free tier covers 10GB storage, 1M Class A (write) operations, and 10M Class B (read) operations per month. Beyond that, storage costs $0.015/GB/month and Class A operations cost $4.50 per million. Crucially, there are **no egress fees**—serving files to your users is completely free. For most early-stage apps, R2 costs nothing.
