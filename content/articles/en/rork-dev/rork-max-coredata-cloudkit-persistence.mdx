## Why Core Data and CloudKit?

When you're building a mobile app, deciding where and how to store user data is one of the most consequential architectural choices you'll make. Since Rork Max generates native Swift apps, you have direct access to Apple's own persistence stack: **Core Data** for local storage and **CloudKit** for iCloud synchronization.

Compared to third-party backends like Supabase or Firebase, this combination has a compelling advantage — there's no additional server cost, and every iPhone user already has an iCloud account. Your app can sync across devices without requiring a separate sign-up flow, and it works offline by default.

## Core Data Fundamentals

Core Data is Apple's object graph management framework. Under the hood it uses SQLite, but you interact with it through Swift objects rather than raw SQL queries.

### Setting Up NSPersistentContainer

The entry point for Core Data is `NSPersistentContainer`, which manages the persistent store and provides access to managed object contexts.

```swift
import CoreData

class PersistenceController {
    static let shared = PersistenceController()

    let container: NSPersistentContainer

    init() {
        container = NSPersistentContainer(name: "AppModel")
        container.loadPersistentStores { description, error in
            if let error = error {
                fatalError("Failed to load Core Data store: \(error)")
            }
        }
        container.viewContext.automaticallyMergesChangesFromParent = true
        container.viewContext.mergePolicy = NSMergeByPropertyObjectTrumpMergePolicy
    }
}
```

Setting `automaticallyMergesChangesFromParent` to `true` ensures that changes made on background contexts are automatically reflected in the view context. The merge policy determines how conflicts are resolved — more on that later.

### Defining Your Data Model

In a standard Xcode project, you'd define entities in the `.xcdatamodeld` visual editor. With Rork Max, you can describe the model directly in your prompt:

```
Create a task management app.
Persist data using Core Data with the following entities:
- Task: title (String), isCompleted (Bool), createdAt (Date), priority (Int16)
- Category: name (String), color (String)
- Task has a many-to-one relationship with Category
```

Rork Max will generate the data model file along with the corresponding Swift managed object subclasses.

## Integrating CloudKit

### Switching to NSPersistentCloudKitContainer

The beauty of Apple's stack is that adding cloud sync is remarkably straightforward. Replace `NSPersistentContainer` with `NSPersistentCloudKitContainer` and enable a couple of options:

```swift
import CoreData

class PersistenceController {
    static let shared = PersistenceController()

    let container: NSPersistentCloudKitContainer

    init() {
        container = NSPersistentCloudKitContainer(name: "AppModel")

        guard let description = container.persistentStoreDescriptions.first else {
            fatalError("No store description found")
        }
        description.setOption(true as NSNumber,
                              forKey: NSPersistentHistoryTrackingKey)
        description.setOption(true as NSNumber,
                              forKey: NSPersistentStoreRemoteChangeNotificationPostOptionKey)

        container.loadPersistentStores { description, error in
            if let error = error {
                fatalError("Failed to load CloudKit store: \(error)")
            }
        }
        container.viewContext.automaticallyMergesChangesFromParent = true
        container.viewContext.mergePolicy = NSMergeByPropertyObjectTrumpMergePolicy
    }
}
```

The two option keys — `NSPersistentHistoryTrackingKey` and `NSPersistentStoreRemoteChangeNotificationPostOptionKey` — are essential for CloudKit sync. History tracking lets the framework compute deltas, and remote change notifications trigger sync when data arrives from another device.

### Configuring CloudKit in Rork Max

```

```

Rork Max will add the iCloud capability to the project, configure the container identifier, and set up the persistence stack accordingly.

## CRUD Operations with SwiftUI

### Displaying Data with @FetchRequest

SwiftUI's `@FetchRequest` property wrapper gives you a reactive view of your Core Data store. When records change — whether from user input or a CloudKit sync — the UI updates automatically.

```swift
struct TaskListView: View {
    @Environment(\.managedObjectContext) private var viewContext

    @FetchRequest(
        sortDescriptors: [NSSortDescriptor(keyPath: \Task.createdAt, ascending: false)],
        animation: .default
    )
    private var tasks: FetchedResults<Task>

    var body: some View {
        List {
            ForEach(tasks) { task in
                TaskRowView(task: task)
            }
            .onDelete(perform: deleteTasks)
        }
    }

    private func deleteTasks(offsets: IndexSet) {
        withAnimation {
            offsets.map { tasks[$0] }.forEach(viewContext.delete)
            try? viewContext.save()
        }
    }
}
```

### Creating and Saving Records

```swift
private func addTask(title: String, priority: Int16) {
    let newTask = Task(context: viewContext)
    newTask.title = title
    newTask.isCompleted = false
    newTask.createdAt = Date()
    newTask.priority = priority

    do {
        try viewContext.save()
    } catch {
        print("Save error: \(error)")
    }
}
```

When you call `viewContext.save()`, the data is written to the local SQLite store and CloudKit sync kicks off in the background. The user doesn't need to do anything — it just works.

## Conflict Resolution and Best Practices

### Choosing a Merge Policy

When the same record is edited on two devices simultaneously, a conflict occurs. Core Data provides several merge policies to handle this:

| Policy | Behavior | Best For |
|--------|----------|----------|
| `NSMergeByPropertyStoreTrumpMergePolicy` | Remote (store) wins | Server-authoritative data |
| `NSMergeByPropertyObjectTrumpMergePolicy` | Local (in-memory) wins | User-facing edits |
| `NSRollbackMergePolicy` | Discards in-memory changes | Read-only views |

For most user-facing apps, `NSMergeByPropertyObjectTrumpMergePolicy` is the right choice. It preserves whatever the user last typed, which feels natural and predictable.

### Using Background Contexts

Heavy read/write operations on the main thread will freeze your UI. Offload bulk work to a background context:

```swift
func importData(_ items: [ImportItem]) {
    container.performBackgroundTask { context in
        for item in items {
            let task = Task(context: context)
            task.title = item.title
            task.createdAt = Date()
        }
        try? context.save()
    }
}
```

`performBackgroundTask` creates a dedicated context on a background queue and automatically merges changes back to the main context when the block completes.

## Troubleshooting Common Issues

### Sync Not Starting

If CloudKit sync doesn't seem to be working, check the following:

1. **iCloud sign-in**: Verify that the device is signed into an iCloud account in Settings.
2. **Capabilities**: Ensure both iCloud (CloudKit) and Background Modes (Remote notifications) are enabled in the project.
3. **Container ID match**: The container ID in your code must exactly match the one registered in the CloudKit Dashboard.
4. **Test on a real device**: The Simulator's CloudKit support can be unreliable. Always verify sync behavior on physical hardware.

### Disappearing Data

If data vanishes after a sync cycle, the most common cause is a missing `NSPersistentHistoryTrackingKey` option. Without history tracking, the framework can't compute proper deltas and may overwrite local data. Double-check that both required options are set on the store description.
