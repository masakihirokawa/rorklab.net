## The Day Push Notifications Changed My Numbers

From years of building apps independently, push notifications are one of the most commonly "postponed" features. But the moment you implement them, retention climbs and paid conversions follow. That's the reality.

When I added push notifications to my first app, DAU (daily active users) grew 1.4x in two weeks. This guide covers everything I learned — from implementation details to backend architecture — so you can add this to your Rork app and see the same results.

## How Rork's Codebase Relates to Expo Notifications

Apps generated by Rork are React Native (Expo) based. That means you can use Expo's push notification ecosystem directly.

Expo's push notification system has two layers:

**1. Expo Push API** (recommended)
Expo acts as a server-side bridge, abstracting APNs (Apple) and FCM (Google). You send notifications to Expo's API, and it handles delivery to both iOS and Android.

**2. Direct Integration** (advanced)
Connecting directly to APNs / FCM yourself. More control, but significantly more implementation work. The Expo Push API is more than sufficient to start.

## Implementation Step 1: Acquiring Device Tokens

First, add a permission request to your Rork app. Here's how to instruct Rork:

```
Add code to request Expo push notification permissions on app launch (or after onboarding completion).

Requirements:
1. Use the expo-notifications package
2. On permission granted, retrieve the Expo Push Token
3. Store the token in AsyncStorage
4. POST the token to the backend endpoint /api/register-token
5. Fail silently if permission is denied (no error UI)
```

The generated code will look something like this:

```typescript
// hooks/usePushNotifications.ts
import { useEffect } from 'react';
import * as Notifications from 'expo-notifications';
import AsyncStorage from '@react-native-async-storage/async-storage';

export const usePushNotifications = () => {
  useEffect(() => {
    registerForPushNotifications();
  }, []);
  
  const registerForPushNotifications = async () => {
    try {
      const { status } = await Notifications.requestPermissionsAsync();
      if (status \!== 'granted') return;
      
      const token = (await Notifications.getExpoPushTokenAsync()).data;
      await AsyncStorage.setItem('pushToken', token);
      
      // Register with backend
      await fetch('https://your-api.com/api/register-token', {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({ 
          token, 
          userId: await AsyncStorage.getItem('userId'),
          platform: Platform.OS
        })
      });
    } catch (error) {
      // Silent failure
    }
  };
};
```

## Implementation Step 2: Backend Token Management

The server-side implementation for receiving and storing tokens. I use Node.js + PostgreSQL:

```javascript
// routes/notifications.js (Node.js / Express)
const { Pool } = require('pg');
const pool = new Pool({ connectionString: process.env.DATABASE_URL });

// Table creation (one-time setup)
// CREATE TABLE push_tokens (
//   id SERIAL PRIMARY KEY,
//   user_id VARCHAR(255),
//   token TEXT UNIQUE NOT NULL,
//   platform VARCHAR(10),
//   is_active BOOLEAN DEFAULT true,
//   created_at TIMESTAMP DEFAULT NOW(),
//   last_used_at TIMESTAMP DEFAULT NOW()
// );

app.post('/api/register-token', async (req, res) => {
  const { token, userId, platform } = req.body;
  
  try {
    await pool.query(`
      INSERT INTO push_tokens (user_id, token, platform)
      VALUES ($1, $2, $3)
      ON CONFLICT (token) 
      DO UPDATE SET 
        user_id = $1,
        is_active = true,
        last_used_at = NOW()
    `, [userId, token, platform]);
    
    res.json({ success: true });
  } catch (error) {
    res.status(500).json({ error: error.message });
  }
});
```

## Implementation Step 3: Sending Notifications

The delivery logic using Expo's Push API:

```javascript
// services/pushNotification.js
const { Expo } = require('expo-server-sdk');
const expo = new Expo();

async function sendPushNotifications(tokens, message) {
  // Split into batches (Expo limit: 100/request)
  const messages = tokens
    .filter(token => Expo.isExpoPushToken(token))
    .map(token => ({
      to: token,
      title: message.title,
      body: message.body,
      data: message.data || {},
      sound: 'default',
      badge: 1
    }));
  
  const chunks = expo.chunkPushNotifications(messages);
  const tickets = [];
  
  for (const chunk of chunks) {
    try {
      const ticketChunk = await expo.sendPushNotificationsAsync(chunk);
      tickets.push(...ticketChunk);
    } catch (error) {
      console.error('Send error:', error);
    }
  }
  
  // Deactivate invalid tokens (critical\!)
  await handleTicketErrors(tickets, tokens);
  
  return tickets;
}

async function handleTicketErrors(tickets, tokens) {
  const errorTokens = tickets
    .filter(t => t.status === 'error' && t.details?.error === 'DeviceNotRegistered')
    .map((t, i) => tokens[i]);
  
  if (errorTokens.length > 0) {
    await pool.query(
      'UPDATE push_tokens SET is_active = false WHERE token = ANY($1)',
      [errorTokens]
    );
  }
}
```

Handling `DeviceNotRegistered` errors is critical. Ignore it and invalid token sends pile up until Expo throttles your account.

## Designing User Segment Notifications

Sending the same notification to every user isn't the best approach. Here's the segment design I use:

```javascript
// User segment notification strategy

// 1. Active users (opened app within last 7 days)
// Goal: Deepen engagement, introduce new features
// Frequency: Up to 2-3 per week
// Example: "New feature added", "This week's picks"

// 2. Dormant users (7-30 days since last open)
// Goal: Re-activation
// Frequency: Every 2-3 days, max 2-week window
// Example: "Welcome back\! Check out the new [X] we added"

// 3. Free users (no purchase)
// Goal: Drive conversion to paid
// Frequency: Weekly max (don't be pushy)
// Example: "Try Premium features free for 7 days"

// 4. Paying users
// Goal: Loyalty, upsell
// Frequency: Minimal (UX first)
// Example: "Thank you for your support — early access to our new feature"

async function getSegmentTokens(segment) {
  const queries = {
    active: `
      SELECT t.token FROM push_tokens t
      JOIN users u ON t.user_id = u.id
      WHERE u.last_active_at > NOW() - INTERVAL '7 days'
      AND t.is_active = true
    `,
    dormant: `
      SELECT t.token FROM push_tokens t
      JOIN users u ON t.user_id = u.id
      WHERE u.last_active_at BETWEEN NOW() - INTERVAL '30 days' AND NOW() - INTERVAL '7 days'
      AND t.is_active = true
    `,
    free: `
      SELECT t.token FROM push_tokens t
      JOIN users u ON t.user_id = u.id
      WHERE u.subscription_status IS NULL
      AND t.is_active = true
    `
  };
  
  const result = await pool.query(queries[segment]);
  return result.rows.map(r => r.token);
}
```

## Writing Copy That Gets Opened

The most technically sound notification system won't help if nobody opens your notifications.

**Patterns that worked for me:**
- Include numbers: "3 new features", "2 days left", "50% off"
- Genuine urgency (use sparingly): "Today only", "10 spots remaining"
- Personalization: "Hey Sarah, it's been a while"
- Questions that spark curiosity: "Struggling with X? Here's what works."

**Patterns to avoid:**
- Too vague: "You have a notification"
- Too pushy: "Upgrade NOW\!"
- Identical copy every time (users stop opening when they recognize the pattern)

## Building an Analytics Layer

Send, measure, improve — don't stop after sending.

```typescript
// App side: record notification taps
Notifications.addNotificationResponseReceivedListener(async (response) => {
  const { notificationId } = response.notification.request.content.data;
  
  // Log the open event to backend
  await fetch('/api/notification-opened', {
    method: 'POST',
    body: JSON.stringify({ notificationId, openedAt: new Date().toISOString() })
  });
  
  // Navigate to the target screen
  navigation.navigate(response.notification.request.content.data.screen);
});
```

Key metrics to track:
- **Permission grant rate**: % of users who granted permission when asked
- **Open rate**: % of sent notifications that were tapped (healthy range: 5–15%)
- **Conversion rate**: % of notification opens that completed the target action (purchase, share, etc.)

## Where to Go Next

Once your push notification system is solid, consider combining it with **local notifications** (sent from inside the app with no server). For reminder and habit-forming apps, local notifications often outperform push in engagement.

And always watch for **notification fatigue**. The worst outcome isn't low open rates — it's users disabling notifications entirely. The courage to send *less* is as important as the ability to send *more*.
