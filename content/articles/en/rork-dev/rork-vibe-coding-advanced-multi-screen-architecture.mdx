## Introduction

Vibe coding's true power emerges when designing large, complex applications. Enterprise-scale apps with multiple screens, sophisticated state management, and API integration can be specified naturally in Rork.

This guide covers production-level patterns. It targets developers with Rork experience who are planning medium-to-large projects.

## Architecture Design Principles for Scale

### 1. Logical Screen Separation

Managing multiple screens requires following a structured separation of concerns:

**Three-Layer Model:**

- **Presentation Layer**: UI only
- **Business Logic Layer**: API calls, data transformation
- **Data Layer**: Firestore / Supabase communication

When defining multiple screens in vibe coding, craft prompts that preserve this separation:

E-commerce app example:

[Screen 1: Product List]
- Fetch products from Firestore (with category filtering)
- Display each as a card (image, name, price)

[Screen 2: Product Detail]
- Full product info (description, reviews, inventory)
- 'Add to Cart' button

[Screen 3: Shopping Cart]
- List of cart items (quantity, subtotal per item)
- Increment/decrement quantity

[Screen 4: Checkout]
- Shipping address form
- Payment method selector (Stripe integration)

Rork generates an architecture like:

```
src/
├── screens/
│   ├── ProductList.tsx
│   ├── ProductDetail.tsx
│   ├── ShoppingCart.tsx
│   └── Checkout.tsx
├── context/
│   ├── CartContext.tsx
│   └── ProductContext.tsx
├── services/
│   ├── firestoreService.ts
│   ├── stripeService.ts
│   └── authService.ts
└── navigation/
    └── RootNavigator.tsx
```

### 2. Global State Management with Context + Reducer

For complex shared state, specify the Context + useReducer pattern. Manage:

- Login state (user info)
- Shopping cart (product list)
- Filter state (selected category)

Selected category in Screen 1 should persist after navigation.

## Advanced Data Flow Patterns

### Pattern 1: API Response Caching

Cache product data in memory during a user session. Cache expires after 5 minutes. Fetch from cache if available, otherwise query Firestore.

### Pattern 2: Optimistic Updates

When user taps 'Add to Cart':
1. Immediately update UI (optimistic update)
2. Simultaneously save to Firestore asynchronously
3. Rollback on failure

## Implementation Best Practices

### 1. Type Safety with TypeScript

Define TypeScript interfaces for all API responses. Include types for: Product, User, CartItem, Order.

### 2. Centralized Error Handling

Route all API errors through a unified error handler. Display user-friendly messages in the UI. Log detailed errors to Firebase Analytics.

### 3. Performance Optimization

If product list has 1000+ items, use VirtualList (virtual scrolling). Render only visible items on screen.

## Frequently Asked Questions

### Q1: Can I manually edit vibe-coded output?

A: Absolutely. Rork generates standard React Native code that's fully editable in any IDE. Post-generation manual adjustments are expected and supported.

### Q2: What if multiple developers vibe-code the same project simultaneously?

A: Use standard Git workflows for conflict management. For parallel work, split prompts by screen to avoid conflicts.

### Q3: Do vibe-coded screens work with existing component libraries?

A: Yes. Rork supports hybrid approaches. Import design system components while using vibe coding to specify business logic.
