You spend days building your app in Rork Max, finally submit it to App Store Connect — and the next morning you wake up to this in your inbox:

> ITMS-91053: Missing API declarations — Your app's code in the "YourApp.ipa" file uses required reason API. Add a value for the required reason to the PrivacyInfo.xcprivacy file.

If you've never seen this before, it's genuinely confusing. There's no obvious place to start, and the error message doesn't point to a specific file you wrote. I spent a few hours staring at it before I figured out what was going on. Here's what you need to know.

## Why This Rejection Happens

Apple introduced the "Required Reason API" policy in 2024. If your app (or any library it uses) accesses certain device APIs — like reading/writing UserDefaults, checking disk space, or querying file timestamps — you're now required to declare *why* in a Privacy Manifest file called `PrivacyInfo.xcprivacy`.

The catch is that **you don't have to be calling these APIs directly**. Third-party packages bundled inside your Rork Max project often use them under the hood. You're still responsible for declaring them.

This check happens at the automated validation stage in App Store Connect, before any human reviewer sees your app. That's why it can feel random — your app might have passed review before, and now suddenly it doesn't.

## Common APIs That Trigger This Error

The rejection email lists the specific APIs that need declarations. In Rork Max apps, these four come up most often:

- **UserDefaults (NSUserDefaults)** — Used to persist settings and state. Reason code: `CA92.1`
- **File timestamp APIs (NSFileManager date attributes)** — Used for cache management. Reason code: `C617.1`
- **Disk space APIs** — Used to check available storage. Reason code: `E174.1`
- **System boot time APIs** — Used for performance measurement. Reason code: `35F9.1`

Libraries like `@react-native-async-storage/async-storage`, `react-native-mmkv`, and `expo-notifications` — which Rork Max includes by default — are common culprits.

## Two Ways to Fix It

### Option 1: Let Rork AI Handle It (Recommended)

Paste the following into the Rork chat interface, swapping in the API names from your rejection email:

```text
My app was rejected from the App Store with this error:

ITMS-91053: Missing API declarations

APIs flagged:
- NSPrivacyAccessedAPICategoryUserDefaults
- NSPrivacyAccessedAPICategoryFileTimestamp

Please create or update the ios/PrivacyInfo.xcprivacy file with the
following reason codes:
- UserDefaults: CA92.1
- File timestamp: C617.1
```

Rork AI will generate the correct file structure for your project. Just make sure to review the output — the reason codes need to actually match your use case, or you risk another rejection for a different reason.

### Option 2: Edit PrivacyInfo.xcprivacy Manually

In your Rork Max project's `ios/` directory, create or edit `PrivacyInfo.xcprivacy` with the following structure:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<\!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN"
    "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
    <key>NSPrivacyAccessedAPITypes</key>
    <array>
        <\!-- UserDefaults: reading/writing app settings -->
        <dict>
            <key>NSPrivacyAccessedAPIType</key>
            <string>NSPrivacyAccessedAPICategoryUserDefaults</string>
            <key>NSPrivacyAccessedAPITypeReasons</key>
            <array>
                <string>CA92.1</string>
            </array>
        </dict>
        <\!-- File timestamps: cache and sync management -->
        <dict>
            <key>NSPrivacyAccessedAPIType</key>
            <string>NSPrivacyAccessedAPICategoryFileTimestamp</string>
            <key>NSPrivacyAccessedAPITypeReasons</key>
            <array>
                <string>C617.1</string>
            </array>
        </dict>
    </array>
</dict>
</plist>
```

After adding this file, you need to **create a new build** via EAS Build. Your existing binary won't include the changes.

## Don't Forget to Update Your Dependencies

Since late 2024, most major npm packages have started shipping their own Privacy Manifests. If you're running older versions of libraries like `expo-notifications` or `react-native-mmkv`, upgrading them might solve the problem entirely without any manual file editing.

```bash
# Update Expo dependencies to compatible versions
npx expo install --fix

# Diagnose Privacy Manifest issues in your dependency tree
npx expo-doctor
```

`npx expo-doctor` is worth running before every App Store submission. It catches Privacy Manifest gaps in your dependency tree before they turn into rejection emails.

## The EAS Build Cache Problem

One thing that trips people up: after making the fix, they rebuild — and still get rejected. If this happens to you, try clearing the EAS Build cache:

```bash
# Force a clean build with no cached layers
eas build --platform ios --clear-cache
```

Also consider using the "Validate App" option in App Store Connect or Xcode's Organizer before submitting. It runs the same automated checks and will flag ITMS-91053 before the file even reaches Apple's servers.

## Pre-Submission Checklist

Before hitting submit again, run through this list:

- All APIs from the rejection email are declared in `PrivacyInfo.xcprivacy`
- Each API has at least one valid reason code assigned
- `npx expo-doctor` shows no Privacy Manifest warnings
- A fresh EAS Build was created with `--clear-cache`
- App Store Connect validation shows no ITMS-91053 warnings

## Avoiding This in the Future

Add `PrivacyInfo.xcprivacy` to your git repository so it doesn't get lost between projects. Every time you add a new library to your Rork Max app, it's worth a quick check to see whether that library accesses any Required Reason APIs.

Apple's official documentation on [Required Reason APIs](https://developer.apple.com/documentation/bundleresources/privacy_manifest_files/describing_use_of_required_reason_api) is actually quite readable and lists all the valid reason codes. Worth bookmarking.

The fix itself is usually straightforward once you know what's happening. Pull up your rejection email, identify the APIs listed, match them to reason codes, and update the file. Most rejections of this type resolve with a single resubmission.
