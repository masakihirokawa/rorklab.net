## Building Bluetooth LE Apps with Rork

Smartwatches, blood pressure monitors, fitness bands, smart locks — the number of everyday devices communicating over Bluetooth LE (BLE) is growing fast. If you want to build an app that talks to these IoT devices using Rork, the most direct path is **react-native-ble-plx**.

This guide walks you through the essentials of Bluetooth LE, how to add the library to your Rork project, and how to scan for devices, connect to them, and read or write data. Even if you've never built a BLE app before, you'll have working code by the end of this article.

## Bluetooth LE Fundamentals

Before diving into code, it's worth understanding a few key concepts.

**GATT (Generic Attribute Profile)** is the data communication standard for BLE. It defines how two devices exchange information using a two-layer structure called **Services** and **Characteristics**.

- **Service**: A logical grouping of related functionality. For example, the Heart Rate Service (UUID: 0x180D) groups everything related to measuring heart rate.
- **Characteristic**: The actual data within a service. Each characteristic supports operations like Read, Write, and Notify (push updates when the value changes).

Most consumer BLE devices use standardized UUIDs defined by the Bluetooth SIG. Custom hardware typically uses proprietary UUIDs provided in the device's developer documentation.

## Setting Up Your Environment

To add BLE support to your Rork project, you'll need:

- An active Rork or Rork Max project
- An iOS physical device (BLE does not work in the simulator)
- An Android physical device (for Android testing)
- The `react-native-ble-plx` library

Start by giving Rork this prompt:

```
Install react-native-ble-plx and configure it so I can scan for and connect
to Bluetooth LE devices. Support both iOS and Android.
```

Rork will automatically add the dependency and configure the required permissions for both platforms.

## Configuring Permissions

BLE requires explicit permission declarations on both platforms. Rork handles this automatically, but knowing what's happening helps when debugging.

**iOS (Info.plist)**

```xml
<!-- Explain to the user why Bluetooth is needed -->
<key>NSBluetoothAlwaysUsageDescription</key>
<string>Used to communicate with your smart devices via Bluetooth.</string>
<key>NSBluetoothPeripheralUsageDescription</key>
<string>Required to exchange data with Bluetooth devices.</string>
```

**Android (AndroidManifest.xml)**

```xml
<!-- Android 12 and above -->
<uses-permission android:name="android.permission.BLUETOOTH_SCAN" />
<uses-permission android:name="android.permission.BLUETOOTH_CONNECT" />
<!-- Android 11 and below -->
<uses-permission android:name="android.permission.BLUETOOTH" />
<uses-permission android:name="android.permission.BLUETOOTH_ADMIN" />
<uses-permission android:name="android.permission.ACCESS_FINE_LOCATION" />
```

## Initializing the BLE Manager

Initialize a single `BleManager` instance at the app level and reuse it throughout your app. This avoids resource conflicts and makes state management cleaner.

```typescript
import { BleManager, Device } from 'react-native-ble-plx';

// Create once at startup — treat this as a singleton
const bleManager = new BleManager();

// Listen for Bluetooth state changes
const subscription = bleManager.onStateChange((state) => {
  if (state === 'PoweredOn') {
    console.log('Bluetooth is ready');
    subscription.remove(); // Clean up after the first detection
  }
}, true);
```

## Scanning for Nearby Devices

Use `startDeviceScan` to discover nearby BLE peripherals. Because scanning drains battery quickly, always stop the scan once you've found your target device — or after a reasonable timeout.

```typescript
import { useState } from 'react';
import { BleManager, Device } from 'react-native-ble-plx';

const bleManager = new BleManager();

export function useBLEScanner() {
  const [devices, setDevices] = useState<Device[]>([]);
  const [isScanning, setIsScanning] = useState(false);

  const startScan = () => {
    setDevices([]);
    setIsScanning(true);

    // First arg: service UUIDs to filter by (null = all devices)
    // Second arg: options (allowDuplicates: false prevents duplicate entries)
    bleManager.startDeviceScan(null, { allowDuplicates: false }, (error, device) => {
      if (error) {
        console.error('Scan error:', error);
        setIsScanning(false);
        return;
      }
      if (device && device.name) {
        setDevices((prev) => {
          const exists = prev.find((d) => d.id === device.id);
          return exists ? prev : [...prev, device];
        });
      }
    });

    // Auto-stop after 10 seconds
    setTimeout(() => {
      bleManager.stopDeviceScan();
      setIsScanning(false);
    }, 10000);
  };

  const stopScan = () => {
    bleManager.stopDeviceScan();
    setIsScanning(false);
  };

  return { devices, isScanning, startScan, stopScan };
}
```

## Connecting to a Device and Reading Data

Once you've found a device, connect to it and read a characteristic. Here's a complete example using a standard heart rate monitor.

```typescript
// Standard UUIDs defined by the Bluetooth SIG
const HEART_RATE_SERVICE = '0000180d-0000-1000-8000-00805f9b34fb';
const HEART_RATE_CHARACTERISTIC = '00002a37-0000-1000-8000-00805f9b34fb';

async function connectAndReadHeartRate(device: Device): Promise<number> {
  try {
    // Connect with a 5-second timeout
    const connectedDevice = await device.connect({ timeout: 5000 });
    console.log('Connected to:', connectedDevice.name);

    // Discover all services and characteristics on the device
    await connectedDevice.discoverAllServicesAndCharacteristics();

    // Read the heart rate characteristic
    const characteristic = await connectedDevice.readCharacteristicForService(
      HEART_RATE_SERVICE,
      HEART_RATE_CHARACTERISTIC
    );

    // Decode the Base64-encoded byte array
    const rawData = Buffer.from(characteristic.value ?? '', 'base64');
    // Heart rate format: byte 0 = flags, byte 1 = heart rate value
    const heartRate = rawData[1];
    console.log('Heart rate:', heartRate, 'bpm');

    return heartRate;
  } catch (error) {
    console.error('Connection or read error:', error);
    throw error;
  }
}
```

## Subscribing to Real-Time Notifications

Many BLE devices push updates whenever their sensor value changes — rather than waiting for a read request. Use `monitorCharacteristicForService` to subscribe to these live updates.

```typescript
async function subscribeToHeartRate(device: Device, onUpdate: (bpm: number) => void) {
  await device.discoverAllServicesAndCharacteristics();

  // Monitor the characteristic for value changes
  const subscription = device.monitorCharacteristicForService(
    HEART_RATE_SERVICE,
    HEART_RATE_CHARACTERISTIC,
    (error, characteristic) => {
      if (error) {
        console.error('Notification error:', error);
        return;
      }
      if (characteristic?.value) {
        const rawData = Buffer.from(characteristic.value, 'base64');
        const heartRate = rawData[1];
        onUpdate(heartRate);
      }
    }
  );

  // Return an unsubscribe function for cleanup
  return () => subscription.remove();
}
```

## Common Errors and Fixes

BLE development comes with its own set of quirks. Here are the most common issues you'll encounter.

**Error: "Cannot use BLE in current state: PoweredOff"**

The user's Bluetooth is turned off. Show a prompt and guide them to Settings.

```typescript
bleManager.onStateChange((state) => {
  if (state === 'PoweredOff') {
    Alert.alert(
      'Bluetooth is Off',
      'Please enable Bluetooth to connect to your device.',
      [{ text: 'Open Settings', onPress: () => Linking.openSettings() }]
    );
  }
}, true);
```

**Frequent "Device disconnected" errors**

Implement an automatic retry loop to reconnect when the signal drops.

```typescript
async function connectWithRetry(device: Device, maxRetries = 3) {
  for (let i = 0; i < maxRetries; i++) {
    try {
      return await device.connect({ timeout: 5000 });
    } catch (error) {
      if (i < maxRetries - 1) {
        console.log(`Retrying connection (${i + 1}/${maxRetries})...`);
        await new Promise((resolve) => setTimeout(resolve, 1000));
      } else {
        throw error;
      }
    }
  }
}
```

**Android: "Location permission required" when scanning**

On Android, BLE scanning requires location permission due to OS-level restrictions (scanning can reveal physical location). Before calling `startDeviceScan`, request `ACCESS_FINE_LOCATION` using `PermissionsAndroid.request`. Prompt Rork with "Add Android BLE permission handling before scanning" and it will wire this up for you.

## FAQ

**Q: Can I test BLE in the iOS Simulator?**

A: No. Bluetooth LE requires real hardware, so you must test on a physical device. The iOS Simulator and Android Emulator both lack BLE support. If you're using Rork Max, use TestFlight to distribute a build to your test device.

**Q: What's the difference between react-native-ble-plx and @react-native-community/ble-manager?**

A: Both libraries provide similar functionality. `react-native-ble-plx` offers a more modern Promise-based API with first-class TypeScript support and is actively maintained — making it the better choice for new Rork projects. `ble-manager` is older and uses a callback-based API.

**Q: Can I transfer large files (like images) over BLE?**

A: BLE is designed for small, frequent data packets — typically up to 512 bytes per packet (limited by MTU). For large payloads like images, you'd need to chunk the data and implement a custom reassembly protocol, or consider using Wi-Fi / LTE instead. BLE excels at small sensor readings (temperature, heart rate, accelerometer data).

**Q: Can my app receive BLE notifications in the background?**

A: Yes, with proper configuration. On iOS, add `bluetooth-central` to the `UIBackgroundModes` key in Info.plist and enable Background Modes in Xcode Capabilities. Android allows background BLE by default, though you'll need to account for Doze mode restrictions on some devices.

## Summary

With Rork and react-native-ble-plx, you can connect your app to virtually any BLE peripheral — from consumer fitness trackers to custom IoT hardware — without writing complex native code.

Key takeaways from this guide:

- Understand GATT (Services + Characteristics) to navigate any BLE device's data structure
- Use `react-native-ble-plx` for scanning, connecting, and reading data from peripherals
- Subscribe to Notify characteristics for live, push-based sensor updates
- Implement retry logic and clear error messages to handle the inherent instability of wireless connections

BLE-connected apps have a natural moat: they solve problems that purely software apps cannot. Healthcare, smart home, and industrial verticals all have real, paying demand. Start with a simple sensor, ship it, and build from there.
