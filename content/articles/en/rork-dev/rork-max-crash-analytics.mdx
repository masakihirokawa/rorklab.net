# Rork Max Crash Reporting & Monitoring

In production, your Rork Max app will encounter errors you'll never see during development. A user on a slow network experiences a timeout. Another encounters an edge case you didn't anticipate. Without crash reporting, these errors vanish silently—users delete your app, and you never know why.

This guide walks you through integrating **Sentry** and **Firebase Crashlytics** with your Rork Max app, capturing real-world errors, tracking user impact, and setting up alerts to catch problems before your users do.

## Why Crash Monitoring Matters

Every crash is an opportunity cost: app deletion, negative reviews, lost engagement. Without monitoring, you're flying blind.

**What you can't see without monitoring:**
- **Crash frequency**: How many users are affected? Is it 0.1% or 10%?
- **User impact**: Which users? Which devices or OS versions?
- **Error trends**: Are crashes increasing after your latest release?
- **Root causes**: Stack traces, network state, device memory at the moment of crash
- **Breadcrumbs**: What actions led to the crash?
- **Performance degradation**: Slow screens that don't quite crash but frustrate users

Crash reporting tools automatically capture, aggregate, and alert you to these issues, turning silent failures into actionable data.

## Sentry vs Firebase Crashlytics

| Feature | Sentry | Firebase Crashlytics |
|---------|--------|----------------------|
| **Crash Reporting** | ✅ Excellent | ✅ Excellent |
| **Performance Monitoring** | ✅ Built-in (Transactions) | ⚠️ Separate product (Performance) |
| **Error Breadcrumbs** | ✅ Comprehensive | ⚠️ Basic |
| **User Context** | ✅ Rich metadata support | ✅ Good |
| **Source Maps** | ✅ Automatic upload | ✅ Automatic upload |
| **Free Tier** | ✅ 5K events/month | ✅ Unlimited events |
| **Pricing** | Pay-per-event ($29/month min) | Free + analytics features paid |
| **Setup Complexity** | Low | Low |
| **React Native Support** | ✅ First-class | ✅ First-class |

**Choose Sentry if:** You want advanced error grouping, performance monitoring, and detailed breadcrumbs—especially valuable for teams prioritizing observability.

**Choose Firebase Crashlytics if:** You're already using Firebase, prefer Google's ecosystem, and want a free solution with room to grow.

## Sentry Integration with Rork Max

### Step 1: Installation

Add Sentry to your Rork Max project using Expo:

```bash
npx expo install @sentry/react-native
```

<Callout type="info">
Sentry's React Native SDK works seamlessly with Expo. If you're using EAS Build, Sentry will automatically handle source map uploads after your build completes.
</Callout>

### Step 2: Initialize Sentry at App Entry

Create or update your app's entry point to initialize Sentry before any code runs. In Rork Max projects, this typically goes in `app.json` or your root component file:

```javascript
import * as Sentry from "@sentry/react-native";
import { NavigationContainer } from "@react-navigation/native";
import { createNativeStackNavigator } from "@react-navigation/native-stack";

// Initialize Sentry BEFORE anything else
Sentry.init({
  dsn: "https://your-key@sentry.io/your-project-id",
  environment: process.env.NODE_ENV || "development",
  // Track the release version from app.json
  release: "1.0.0",
  // Set a higher tracesSampleRate in development
  tracesSampleRate: process.env.NODE_ENV === "production" ? 0.1 : 1.0,
  // Attaches HTTP client interactions (network requests)
  integrations: [
    new Sentry.ReactNativeTracing(),
  ],
});

// Then define your navigation and screens
const Stack = createNativeStackNavigator();

function App() {
  return (
    <NavigationContainer>
      <Stack.Navigator>
        {/* Your screens here */}
      </Stack.Navigator>
    </NavigationContainer>
  );
}

// Wrap your app with Sentry
export default Sentry.wrap(App);
```

### Step 3: Capture Exceptions Manually

While Sentry automatically captures uncaught exceptions, you'll often want to capture errors explicitly:

```javascript
import * as Sentry from "@sentry/react-native";

async function loadUserProfile(userId) {
  try {
    const response = await fetch(`/api/users/${userId}`);
    const data = await response.json();
    return data;
  } catch (error) {
    // Capture the error with context
    Sentry.captureException(error, {
      tags: {
        user_action: "load_profile",
      },
      extra: {
        userId,
        timestamp: new Date().toISOString(),
      },
    });
    // Re-throw or handle gracefully
    throw error;
  }
}
```

### Step 4: Capture Messages

For non-exception errors or important events:

```javascript
import * as Sentry from "@sentry/react-native";

function processPayment(orderId, amount) {
  if (amount < 0) {
    Sentry.captureMessage("Negative payment amount attempted", {
      level: "warning",
      extra: { orderId, amount },
    });
    return false;
  }
  // Process payment...
}
```

### Step 5: React Error Boundary

Wrap components that might crash to catch errors gracefully:

```javascript
import * as Sentry from "@sentry/react-native";
import { Component } from "react";
import { View, Text } from "react-native";

class ErrorBoundary extends Component {
  state = { hasError: false };

  static getDerivedStateFromError(error) {
    return { hasError: true };
  }

  componentDidCatch(error, errorInfo) {
    // Capture the error to Sentry
    Sentry.captureException(error, {
      contexts: {
        react: {
          componentStack: errorInfo.componentStack,
        },
      },
    });
  }

  render() {
    if (this.state.hasError) {
      return (
        <View style={{ flex: 1, justifyContent: "center", alignItems: "center" }}>
          <Text>Something went wrong. We're looking into it.</Text>
        </View>
      );
    }

    return this.props.children;
  }
}

export default Sentry.withErrorBoundary(ErrorBoundary, {
  fallback: <Text>Error loading component</Text>,
});
```

## Sentry: User Context & Privacy

Tracking which user experienced an error is crucial for prioritization:

```javascript
import * as Sentry from "@sentry/react-native";

function setUserContext(user) {
  Sentry.setUser({
    id: user.id,
    email: user.email,
    username: user.username,
    // Custom fields—be thoughtful about PII
  });
}

function clearUserContext() {
  Sentry.setUser(null);
}
```

<Callout type="warning">
**Privacy & GDPR**: Never send sensitive PII (passwords, credit cards, SSNs) to Sentry. Use anonymized user IDs where possible. Check your privacy policy and ensure users consent to error reporting.
</Callout>

Add additional context for debugging:

```javascript
Sentry.setContext("device", {
  model: "iPhone 13",
  os_version: "15.2",
});

Sentry.setTag("subscription_level", "premium");
```

## Sentry: Performance Monitoring

Track slow screens and operations:

```javascript
import * as Sentry from "@sentry/react-native";

async function fetchListingDetails(listingId) {
  // Start a transaction
  const transaction = Sentry.startTransaction({
    name: "fetch_listing_details",
    op: "http",
  });

  const span = transaction.startChild({
    op: "fetch",
    description: "GET /listings/:id",
  });

  try {
    const response = await fetch(`/api/listings/${listingId}`);
    const data = await response.json();

    span.finish();
    transaction.finish();
    return data;
  } catch (error) {
    span.status = "error";
    span.finish();
    transaction.finish();
    Sentry.captureException(error);
    throw error;
  }
}
```

The `tracesSampleRate` in your config controls what percentage of transactions are recorded. Set it to `1.0` in development to see all data, `0.1` in production to reduce noise.

## Firebase Crashlytics Integration

### Step 1: Install Dependencies

```bash
npx expo install @react-native-firebase/app @react-native-firebase/crashlytics
```

Configure Firebase in your `app.json` (or use `google-services.json` for Android):

```json
{
  "plugins": [
    [
      "@react-native-firebase/app",
      {
        "analytics": false,
        "database": false,
        "messaging": false,
        "storage": false,
        "crashlytics": true
      }
    ]
  ]
}
```

### Step 2: Initialize Crashlytics

```javascript
import crashlytics from "@react-native-firebase/crashlytics";

async function initializeCrashlytics() {
  // Link crashes to a user ID
  await crashlytics().setUserId("user_12345");

  // Enable collection (off by default in debug builds)
  if (!__DEV__) {
    await crashlytics().setCrashlyticsCollectionEnabled(true);
  }
}

// Call at app startup
initializeCrashlytics();
```

### Step 3: Record Errors

```javascript
import crashlytics from "@react-native-firebase/crashlytics";

try {
  riskyOperation();
} catch (error) {
  // Record a non-fatal error
  crashlytics().recordError(error);
}
```

For non-fatal errors without exceptions:

```javascript
crashlytics().log("Payment gateway timeout");
```

### Step 4: Custom Attributes & Breadcrumbs

Add context to crash reports:

```javascript
import crashlytics from "@react-native-firebase/crashlytics";

function startCheckout(orderId, total) {
  crashlytics().setAttribute("current_order_id", orderId);
  crashlytics().setAttribute("checkout_total", String(total));

  // Breadcrumb
  crashlytics().log(`User started checkout for order ${orderId}`);
}
```

## Non-Fatal vs Fatal Errors: Strategy

Decide how to categorize errors:

| Error Type | Sentry | Crashlytics | Example |
|-----------|--------|-------------|---------|
| **Fatal Crash** | `captureException()` | Auto-captured | App freezes, force close |
| **Non-Fatal Exception** | `captureException()` | `recordError()` | Network timeout caught and handled |
| **User Action Warning** | `captureMessage()` | `log()` | Invalid form submission |
| **Performance Alert** | Transactions (if slow) | n/a | Screen takes 5+ seconds to load |

```javascript
// Non-fatal network error—app continues
async function fetchData() {
  try {
    return await fetch("/api/data");
  } catch (error) {
    Sentry.captureException(error, { level: "warning" });
    return null; // App handles gracefully
  }
}

// Fatal error—app can't continue
function validateAuth(token) {
  if (!token) {
    const error = new Error("No auth token");
    Sentry.captureException(error, { level: "fatal" });
    throw error; // App stops here
  }
}
```

## Alerting & Notification Setup

### Sentry Alerts

1. **Log in to sentry.io** and go to your project
2. **Alerts → Create Alert Rule**
3. **Choose conditions:**
   - Error rate > 5% in last hour
   - New issue appears
   - Error count > 100

4. **Set actions:**
   - Send to Slack: `/slack https://hooks.slack.com/...`
   - Email to team
   - PagerDuty integration for critical issues

```
Condition: "A new issue is created"
Actions: Slack channel #app-errors, Email team@company.com
```

### Firebase Crashlytics Alerts

1. **Firebase Console** → Crashlytics
2. **Setup → Alerts**
3. **New Alert:**
   - Crash-free user rate drops below 99%
   - Spike in crashes for a specific version

<Callout type="info">
Both tools let you mute noisy errors (spam, intentional test crashes). Mark them as "resolved" to group future similar errors separately.
</Callout>

## Choosing: Sentry vs Crashlytics for Your Team

**Use Sentry if:**
- You want **advanced error grouping** and need to see patterns (e.g., "all crashes on older Android versions")
- Your team requires **performance monitoring** (slow screens, HTTP request timing)
- You want **detailed breadcrumbs** (last 10 user actions before crash)
- You don't use Firebase and prefer a standalone solution
- You need **custom issue workflows** and integrations

**Use Firebase Crashlytics if:**
- You're already in the **Google/Firebase ecosystem**
- You want a **free solution** with no per-event costs
- Your crashes are primarily from **native code** (NDK errors)
- You need simple, straightforward crash reporting without complexity
- You already use Firebase Analytics and want unified reporting

**Use Both if:**
- Your app is mission-critical and you can justify the complexity
- You want **diversity** (if one service goes down, you still have data)
- Sentry catches app crashes; Crashlytics logs native framework issues

## Monitoring Checklist

Before shipping to production:

- [ ] **Sentry DSN** configured for production environment
- [ ] **Source maps** uploaded to both services (auto-upload on EAS Build enabled)
- [ ] **Release version** set in Sentry config matching `app.json`
- [ ] **User context** implemented—crashes show which user was affected
- [ ] **Error boundaries** wrap risky components (deep navigation, image loading)
- [ ] **Try/catch blocks** around async operations (network calls, file operations)
- [ ] **Alert rules** created for error spikes (> 5% increase)
- [ ] **Slack/email** notifications tested
- [ ] **Privacy policy** updated to disclose error reporting
- [ ] **Sample errors** sent from staging to verify both services receive data
- [ ] **Team members** have dashboard access and know how to respond

## Summary

Crash reporting transforms invisible errors into actionable data. Sentry excels at observability and performance; Crashlytics offers a free, Google-integrated alternative.

For Rork Max apps:

1. **Start with one**: Pick Sentry for advanced features or Crashlytics for simplicity
2. **Instrument gradually**: Begin with automatic crash capture, add user context, then performance monitoring
3. **Set alerts early**: Catch spikes before they impact users at scale
4. **Review regularly**: Weekly check-ins on the crash dashboard catch trends early
5. **Privacy first**: Get user consent and avoid logging sensitive data

Your users won't notice crash reporting—until you fix a bug they reported three weeks ago. Then they'll appreciate the responsiveness.
