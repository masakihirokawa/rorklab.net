## The Login Screen Is Where You Lose Users

The single biggest reason users abandon your login screen is forgotten passwords. In 2026, Passkeys — backed by Apple, Google, and Microsoft — solve this at the root. Users authenticate with Face ID or a fingerprint, and the underlying cryptographic protocol makes phishing attacks nearly impossible.

If you've built an app with Rork and want to add Passkeys, there's an immediate hurdle: Expo Go can't run them. Passkeys rely on native OS authentication APIs, which means you need a Development Build. This guide walks through the entire process, from that migration to a working implementation.

## Why Development Builds Are Non-Negotiable

Passkeys call native APIs directly — `ASAuthorizationController` on iOS and the Credential Manager API on Android. Expo Go's sandboxed environment doesn't allow adding native modules, so any Passkeys library simply won't load.

After generating your project with Rork, switch to Development Builds:

```bash
# Set up Expo Development Build
npx expo install expo-dev-client
npx expo prebuild

# Build for iOS simulator
npx expo run:ios

# Build for Android emulator
npx expo run:android
```

This generates the `ios/` and `android/` directories, unlocking the ability to add any native module. The investment pays off far beyond Passkeys — it opens the door to all native integrations.

## Setting Up react-native-passkeys

The `react-native-passkeys` library provides a unified API across iOS, Android, and web, making it a natural fit for Rork's cross-platform approach.

```bash
# Install the library
npx expo install react-native-passkeys
```

### Domain Verification Files

Passkeys require your app to prove its association with a specific domain. Skip this step and both registration and authentication will fail silently — no error message, no crash, just nothing happening when the user taps the button.

**iOS — Apple App Site Association (AASA) file:**

```json
// https://your-domain.com/.well-known/apple-app-site-association
{
  "webcredentials": {
    "apps": [
      "TEAM_ID.com.yourcompany.yourapp"
    ]
  }
}
```

**Android — Digital Asset Links:**

```json
// https://your-domain.com/.well-known/assetlinks.json
[
  {
    "relation": ["delegate_permission/common.handle_all_urls"],
    "target": {
      "namespace": "android_app",
      "package_name": "com.yourcompany.yourapp",
      "sha256_cert_fingerprints": ["YOUR_SHA256_FINGERPRINT"]
    }
  }
]
```

A common stumbling block: **domain verification doesn't work during local development**. On iOS, you'll need to add the `webcredentials:` Associated Domains entitlement in `app.json`, pointing to the same domain used for `applinks:`.

## Implementing the Registration Flow

Here's the complete code for registering a new Passkey. The flow retrieves a challenge from your server, creates a key pair using the device's biometric authentication, then sends the public key back:

```typescript
// hooks/usePasskeyAuth.ts
import { useState, useCallback } from 'react';
import { Passkeys } from 'react-native-passkeys';
import { Alert } from 'react-native';

const RP_ID = 'your-domain.com';
const API_BASE = 'https://your-domain.com/api/auth';

type PasskeyState = {
  isLoading: boolean;
  error: string | null;
};

export function usePasskeyAuth() {
  const [state, setState] = useState<PasskeyState>({
    isLoading: false,
    error: null,
  });

  // Register a new Passkey (first-time signup)
  const register = useCallback(async (userId: string, userName: string) => {
    setState({ isLoading: true, error: null });

    try {
      // 1. Fetch registration challenge from server
      const optionsRes = await fetch(`${API_BASE}/passkey/register-options`, {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({ userId, userName }),
      });

      if (\!optionsRes.ok) {
        throw new Error('Failed to fetch registration challenge');
      }

      const options = await optionsRes.json();

      // 2. Create key pair via device biometric prompt
      const credential = await Passkeys.create({
        publicKey: {
          challenge: options.challenge,
          rp: { id: RP_ID, name: 'Your App' },
          user: {
            id: options.userId,
            name: userName,
            displayName: userName,
          },
          pubKeyCredParams: [
            { alg: -7, type: 'public-key' },   // ES256
            { alg: -257, type: 'public-key' },  // RS256
          ],
          authenticatorSelection: {
            userVerification: 'required',
            residentKey: 'required',
          },
          timeout: 60000,
        },
      });

      // 3. Send public key to server for storage
      const verifyRes = await fetch(`${API_BASE}/passkey/register-verify`, {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({
          userId,
          credential: {
            id: credential.id,
            rawId: credential.rawId,
            response: credential.response,
            type: credential.type,
          },
        }),
      });

      if (\!verifyRes.ok) {
        throw new Error('Registration verification failed');
      }

      setState({ isLoading: false, error: null });
      return true;
    } catch (err) {
      const message =
        err instanceof Error ? err.message : 'Registration failed';
      setState({ isLoading: false, error: message });
      Alert.alert('Registration Error', message);
      return false;
    }
  }, []);

  // Authenticate with an existing Passkey
  const authenticate = useCallback(async () => {
    setState({ isLoading: true, error: null });

    try {
      const optionsRes = await fetch(`${API_BASE}/passkey/auth-options`, {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
      });

      if (\!optionsRes.ok) {
        throw new Error('Failed to fetch authentication challenge');
      }

      const options = await optionsRes.json();

      const assertion = await Passkeys.get({
        publicKey: {
          challenge: options.challenge,
          rpId: RP_ID,
          userVerification: 'required',
          timeout: 60000,
        },
      });

      const verifyRes = await fetch(`${API_BASE}/passkey/auth-verify`, {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({
          credential: {
            id: assertion.id,
            rawId: assertion.rawId,
            response: assertion.response,
            type: assertion.type,
          },
        }),
      });

      if (\!verifyRes.ok) {
        throw new Error('Authentication failed');
      }

      const { token } = await verifyRes.json();
      setState({ isLoading: false, error: null });
      return token;
    } catch (err) {
      const message =
        err instanceof Error ? err.message : 'Authentication failed';
      setState({ isLoading: false, error: message });
      return null;
    }
  }, []);

  return { ...state, register, authenticate };
}
```

With this hook, your login screen becomes remarkably simple:

```tsx
// screens/LoginScreen.tsx
import { View, Text, TouchableOpacity, ActivityIndicator } from 'react-native';
import { usePasskeyAuth } from '../hooks/usePasskeyAuth';

export default function LoginScreen() {
  const { isLoading, authenticate } = usePasskeyAuth();

  const handleLogin = async () => {
    const token = await authenticate();
    if (token) {
      // Login successful — navigate to main screen
    }
  };

  return (
    <View style={{ flex: 1, justifyContent: 'center', padding: 24 }}>
      <Text style={{ fontSize: 28, fontWeight: 'bold', marginBottom: 32 }}>
        Welcome back
      </Text>

      <TouchableOpacity
        onPress={handleLogin}
        disabled={isLoading}
        style={{
          backgroundColor: '#007AFF',
          padding: 16,
          borderRadius: 12,
          alignItems: 'center',
        }}
      >
        {isLoading ? (
          <ActivityIndicator color="#fff" />
        ) : (
          <Text style={{ color: '#fff', fontSize: 18, fontWeight: '600' }}>
            Sign in with Face ID / Fingerprint
          </Text>
        )}
      </TouchableOpacity>
    </View>
  );
}
```

## Three Pitfalls That Will Cost You Hours

**1. Wrong Content-Type on the AASA file**

The AASA file must be served with `application/json` as its Content-Type. If you're hosting on Cloudflare Pages or Vercel, the `.well-known` directory may not have the correct headers by default. On Cloudflare, add a `_headers` file to set it explicitly. Without this, iOS silently fails domain verification.

**2. Android signing key mismatch in assetlinks.json**

Development builds and release builds use different signing keys. If you only register the debug key's `sha256_cert_fingerprints`, authentication breaks in production. When using EAS Build, run `eas credentials` to retrieve the exact fingerprint and make sure it matches what's in your `assetlinks.json`.

**3. The `userVerification: 'required'` trap**

Setting `userVerification` to `required` means devices without biometric hardware will throw an error. If you need to support older Android devices, consider switching to `preferred` and providing a fallback authentication method like PIN entry. This is a deliberate trade-off between security strictness and device coverage.

## Gradual Adoption Alongside Traditional Auth

Not every user has a Passkey-capable device yet. The pragmatic approach is to offer Passkeys alongside existing authentication methods and nudge compatible users toward the more secure option.

The [biometric authentication guide](/articles/rork-dev/rork-biometric-auth-face-id-touch-id-guide) covers Face ID and Touch ID setups that work well as a fallback when Passkeys aren't available. And the [authentication architecture overview](/articles/rork-dev/rork-auth-firebase-supabase) shows how Firebase Auth and Supabase Auth can coexist with Passkeys, giving users choice while encouraging migration to the strongest option.

## Server-Side Considerations

The client-side implementation is only half the story. Your backend needs to handle WebAuthn challenge generation and credential verification. Libraries like `@simplewebauthn/server` (Node.js) or `py_webauthn` (Python) handle the cryptographic verification, so you don't need to implement it from scratch.

A minimal server endpoint for generating registration options looks like this:

```typescript
// Server-side: POST /api/auth/passkey/register-options
import {
  generateRegistrationOptions,
} from '@simplewebauthn/server';

export async function POST(req: Request) {
  const { userId, userName } = await req.json();

  // Fetch any existing credentials for this user
  const existingCredentials = await db.getCredentialsByUserId(userId);

  const options = await generateRegistrationOptions({
    rpName: 'Your App',
    rpID: 'your-domain.com',
    userID: userId,
    userName: userName,
    // Prevent re-registering the same authenticator
    excludeCredentials: existingCredentials.map((cred) => ({
      id: cred.credentialId,
      type: 'public-key',
    })),
    authenticatorSelection: {
      userVerification: 'required',
      residentKey: 'required',
    },
  });

  // Store the challenge temporarily for verification
  await db.storeChallenge(userId, options.challenge);

  return Response.json(options);
}
```

The key detail here is the `excludeCredentials` array. Without it, a user could accidentally register the same device multiple times, leading to confusion when multiple Passkey prompts appear during login.

## What Passkeys Change for Your App

The most measurable impact of adding Passkeys is login conversion rate. Removing password entry dramatically improves success rates for returning users, especially on mobile where typing is friction.

The implementation overhead is concentrated in two areas: migrating to Development Builds and setting up domain verification. Once those are done, ongoing maintenance is lower than managing passwords — no reset flows, no hashed password storage, no credential stuffing attacks to defend against. As your next step, review your [overall app security architecture](/articles/rork-dev/rork-app-security-complete-implementation-guide) and consider making Passkeys the centerpiece of your authentication strategy.
