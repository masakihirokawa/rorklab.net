Here's a QA workflow that's all too common in solo app development: you finish a build, manually upload the APK to a shared Slack channel, write instructions for how to install it, wait for testers to try it, and then when something crashes, spend the next hour asking "What screen were you on? What did you tap last?" The answer is almost always some variation of "it just crashed."

If you're building apps with Rork and want to take them to production quality, this QA loop is usually the first serious wall you hit. It's not a code quality problem—it's a process problem. And process problems are solvable.

Combine Firebase App Distribution, Crashlytics, and GitHub Actions, and the entire manual loop disappears. Push to `main`, and testers receive an email notification with a direct install link within minutes. Something crashes in the wild, and you get a detailed report with stack traces, breadcrumb logs, and the exact screen the user was on—before they even tell you about it. Run this for a few weeks and your crash-free rate climbs without any heroic debugging sessions.

I've used this setup across several personal apps, and it's genuinely one of those infrastructure investments that pays itself back on the first bad release. Let me walk through the full setup, from Firebase project creation through production operations.

## Why These Three Tools Together

Before getting into the specifics, it's worth understanding why this particular combination works well—because each tool has a distinct and non-overlapping role.

**GitHub Actions** is the orchestrator. It watches for repository events (pushes, pull request merges, scheduled runs) and kicks off your build pipeline automatically. You define the logic once in a YAML file, and it runs reliably without any manual intervention. For Rork apps built on Expo, Actions integrates cleanly with EAS Build.

**Firebase App Distribution** is the delivery layer. Once EAS Build produces an artifact (APK or IPA), App Distribution takes it and sends it directly to your registered testers. They get an email with a one-tap install link, no TestFlight approval queue, no manual sharing. It also keeps a versioned history of all builds you've distributed, which is useful when testers report "the previous version worked but this one doesn't."

**Crashlytics** is the watchdog. The moment your app crashes on any tester's device—or in production—Crashlytics captures the full event: stack trace, device model, OS version, app version, and any custom context you've added. By the time a tester messages you, the report is already waiting in your Firebase console.

You could swap some of these tools. Sentry is a strong Crashlytics alternative—see the [Sentry + Rork guide](/en/articles/rork-dev/rork-sentry-crash-monitoring-setup-guide) for a comparison. Bitrise or CircleCI can replace GitHub Actions. But this specific combination has a meaningful advantage for solo developers: **everything lives in one place**. Your distribution history, crash reports, and performance metrics are all in the Firebase console. You're not stitching together dashboards from three different services.

The cost structure also matters. Crashlytics is permanently free with no event limits—a real differentiator from Sentry's free tier, which caps the number of error events per month. When you're watching costs on a side project, eliminating a recurring line item for crash monitoring adds up over a year.

## Setting Up Firebase and Integrating with Your Rork Project

Start at console.firebase.google.com. Create a new project, then register both your iOS and Android apps. The critical detail here is that the identifiers must exactly match what's in your Rork `app.json`:

```json
// app.json — the identifiers used everywhere in your build pipeline
{
  "expo": {
    "name": "MyApp",
    "slug": "my-app",
    "version": "1.0.0",
    "ios": {
      "bundleIdentifier": "com.yourname.myapp"
    },
    "android": {
      "package": "com.yourname.myapp",
      "versionCode": 1
    }
  }
}
```

After registering each app in Firebase, download `GoogleService-Info.plist` (iOS) and `google-services.json` (Android). Where you place these files trips up a lot of developers new to this setup.

For **Expo managed workflow** (which is what Rork generates): place both files in the project root, then reference them explicitly from `app.json`. The Expo build system handles the rest.

```json
// app.json — Firebase config references + plugin registration
{
  "expo": {
    "name": "MyApp",
    "slug": "my-app",
    "plugins": [
      "@react-native-firebase/app",
      "@react-native-firebase/crashlytics",
      "@react-native-firebase/perf"
    ],
    "ios": {
      "bundleIdentifier": "com.yourname.myapp",
      "googleServicesFile": "./GoogleService-Info.plist"
    },
    "android": {
      "package": "com.yourname.myapp",
      "googleServicesFile": "./google-services.json"
    }
  }
}
```

Install the packages:

```bash
# Core Firebase + Crashlytics (required)
npx expo install @react-native-firebase/app @react-native-firebase/crashlytics

# Performance Monitoring (strongly recommended—it's free and takes 10 minutes to set up)
npx expo install @react-native-firebase/perf
```

One important constraint to know upfront: `@react-native-firebase` contains native code and therefore cannot run in Expo Go or Rork Companion preview mode. You need a custom development build via EAS Build. This is the most frequent source of "why isn't Crashlytics working in preview?" confusion—it's not a configuration problem, it's just a fundamental limitation of how native modules work in Expo's ecosystem.

If you haven't set up EAS Build for your Rork project yet, the [EAS Build + GitHub Actions guide](/en/articles/rork-dev/rork-github-actions-eas-build-cicd) walks through that prerequisite step.

## Crashlytics Setup: From Installation to Real Reports

With packages installed and `app.json` configured, initialize Crashlytics in your app's entry point. For Expo Router, that's `app/_layout.tsx`.

```typescript
// app/_layout.tsx
import { useEffect } from 'react';
import crashlytics from '@react-native-firebase/crashlytics';
import { Stack } from 'expo-router';

export default function RootLayout() {
  useEffect(() => {
    const initCrashlytics = async () => {
      try {
        // Explicitly enable collection
        // (it's on by default, but being explicit makes intent clear)
        await crashlytics().setCrashlyticsCollectionEnabled(true);

        // Set user ID if authenticated—this lets you filter crashes by user
        const userId = await getStoredUserId(); // your auth implementation
        if (userId) {
          await crashlytics().setUserId(userId);
        }

        // Record app version as a custom key for easy filtering
        await crashlytics().setAttribute(
          'app_version',
          process.env.EXPO_PUBLIC_APP_VERSION ?? 'unknown'
        );

        // Record build type to distinguish beta vs production crashes
        await crashlytics().setAttribute(
          'build_type',
          __DEV__ ? 'debug' : 'release'
        );
      } catch (e) {
        // If Crashlytics setup fails, don't let it crash the app
        console.warn('Crashlytics initialization failed:', e);
      }
    };

    initCrashlytics();
  }, []);

  return <Stack />;
}
```

After this initialization, any unhandled native crash will automatically generate a report in your Firebase console with a full stack trace, device information, and OS version.

### Verifying the Integration Works

Crashlytics doesn't send reports in `__DEV__` mode, which makes it harder to confirm your setup is working. Use this test pattern:

```typescript
// TestScreen.tsx — temporary screen for verifying Crashlytics
// Remove this before submitting to the app stores
import crashlytics from '@react-native-firebase/crashlytics';
import { Button, View } from 'react-native';

export function CrashlyticsTestScreen() {
  const testNonFatal = async () => {
    // Sends a non-fatal error—app continues running
    await crashlytics().recordError(
      new Error('Test: non-fatal error from CrashlyticsTestScreen')
    );
    console.log('Non-fatal error sent. Check Firebase console in ~2 minutes.');
  };

  const testCrash = () => {
    // Sends a fatal crash report and terminates the app
    crashlytics().crash();
  };

  return (
    <View>
      <Button title="Test Non-Fatal Error" onPress={testNonFatal} />
      <Button title="Test Fatal Crash" onPress={testCrash} />
    </View>
  );
}
```

Build and install a release build via EAS (the `preview` profile works), trigger both test events, and check the Firebase console after a few minutes. If reports appear, your integration is confirmed.

## Adding Context to Crash Reports: Custom Keys and Breadcrumbs

Getting crash reports is step one. Making those reports actionable is step two. Crashlytics gives you two mechanisms for adding context: **custom keys** (key-value pairs stored with every report) and **logs** (a rolling buffer of recent events, similar to breadcrumbs in Sentry).

### Screen Tracking

Recording the current screen when a crash happens is the single most useful piece of context you can add—it transforms "something crashed" into "the crash always happens on the product detail screen."

```typescript
// hooks/useCrashlyticsTracking.ts
import { useEffect } from 'react';
import { usePathname } from 'expo-router';
import crashlytics from '@react-native-firebase/crashlytics';

export function useCrashlyticsTracking() {
  const pathname = usePathname();

  useEffect(() => {
    const trackScreen = async () => {
      try {
        // Update the custom key with every navigation
        await crashlytics().setAttribute('current_screen', pathname);

        // Log as a breadcrumb entry
        // Crashlytics keeps the last ~64 log entries in each report
        crashlytics().log(
          `[NAV] ${pathname} — ${new Date().toISOString()}`
        );
      } catch {
        // Never let tracking logic affect app behavior
      }
    };

    trackScreen();
  }, [pathname]);
}
```

Call `useCrashlyticsTracking()` from your root layout, and every crash report will show exactly which screen the user was on, plus a timeline of recent navigation events.

### Error Boundaries for JavaScript Errors

React Native has two layers where errors occur: the native layer (crashes Crashlytics catches automatically) and the JavaScript layer (errors React can catch before they propagate to native). An error boundary covers the JavaScript layer.

```typescript
// components/CrashBoundary.tsx
import React, { Component, ErrorInfo, ReactNode } from 'react';
import crashlytics from '@react-native-firebase/crashlytics';

interface Props {
  children: ReactNode;
  fallback?: ReactNode;
  screenName?: string;
}

interface State {
  hasError: boolean;
}

export class CrashBoundary extends Component<Props, State> {
  constructor(props: Props) {
    super(props);
    this.state = { hasError: false };
  }

  static getDerivedStateFromError(): State {
    return { hasError: true };
  }

  componentDidCatch(error: Error, errorInfo: ErrorInfo) {
    // Send to Crashlytics as non-fatal
    crashlytics().recordError(error);

    // Add component stack + screen context as breadcrumbs
    crashlytics().log(
      `[ERROR BOUNDARY] Screen: ${this.props.screenName ?? 'unknown'}\n` +
      `Stack: ${errorInfo.componentStack?.slice(0, 400) ?? 'n/a'}`
    );

    // Attach structured context as custom keys
    crashlytics().setAttributes({
      error_boundary: 'triggered',
      boundary_screen: this.props.screenName ?? 'unknown',
      // Truncate message to fit Crashlytics key value limit
      error_summary: error.message.slice(0, 256),
    });
  }

  render() {
    if (this.state.hasError) {
      // Show fallback UI or return null to silently recover
      return this.props.fallback ?? null;
    }
    return this.props.children;
  }
}
```

Wrap individual screens or feature sections with this boundary, passing a descriptive `screenName`:

```typescript
// app/(tabs)/product/[id].tsx
export default function ProductDetailScreen() {
  return (
    <CrashBoundary
      screenName="ProductDetail"
      fallback={<ErrorFallbackUI message="Unable to load product" />}
    >
      <ProductDetailContent />
    </CrashBoundary>
  );
}
```

### Recording User Actions as Breadcrumbs

For complex user flows—checkout, form submissions, multi-step onboarding—recording key actions as logs creates a timeline you can read backward from a crash.

```typescript
// utils/breadcrumb.ts
import crashlytics from '@react-native-firebase/crashlytics';

export function breadcrumb(action: string, context?: Record<string, string>) {
  const contextStr = context
    ? ' ' + Object.entries(context).map(([k, v]) => `${k}=${v}`).join(' ')
    : '';

  crashlytics().log(
    `[ACTION] ${action}${contextStr} — ${Date.now()}`
  );
}

// Usage throughout your app:
// breadcrumb('checkout_started', { cart_size: '3', total: '2480' });
// breadcrumb('payment_form_submitted');
// breadcrumb('api_call', { endpoint: 'create_order', method: 'POST' });
```

When a crash happens during checkout, you'll see the full sequence of actions the user took, not just the final stack trace.

## GitHub Actions: The Full Automation Workflow

With Crashlytics capturing data from builds, the next piece is automating how those builds reach your testers. Here's a complete, production-ready GitHub Actions workflow.

```yaml
# .github/workflows/distribute.yml
name: Build and Distribute

on:
  push:
    branches: [main]
  workflow_dispatch:
    inputs:
      platform:
        description: 'Platform to build'
        required: true
        default: 'android'
        type: choice
        options: [android, ios, all]

env:
  NODE_VERSION: '20'

jobs:
  # Android build and distribution (runs on every main push)
  distribute-android:
    name: Android Distribution
    runs-on: ubuntu-latest
    if: |
      github.ref == 'refs/heads/main' ||
      github.event.inputs.platform == 'android' ||
      github.event.inputs.platform == 'all'

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: ${{ env.NODE_VERSION }}
          cache: 'npm'

      - name: Install dependencies
        run: npm ci

      - name: Setup EAS
        uses: expo/expo-github-action@v8
        with:
          eas-version: latest
          token: ${{ secrets.EXPO_TOKEN }}

      - name: Restore google-services.json
        run: |
          echo "${{ secrets.GOOGLE_SERVICES_JSON_BASE64 }}" \
            | base64 --decode > ./google-services.json

      - name: Build Android APK
        id: eas_build
        run: |
          eas build \
            --platform android \
            --profile preview \
            --non-interactive \
            --output ./dist/app.apk
        env:
          EAS_NO_VCS: 1

      - name: Upload to Firebase App Distribution
        uses: wzieba/Firebase-Distribution-Github-Action@v1
        with:
          appId: ${{ secrets.FIREBASE_APP_ID_ANDROID }}
          token: ${{ secrets.FIREBASE_TOKEN }}
          groups: qa-testers
          file: ./dist/app.apk
          releaseNotes: |
            Branch: ${{ github.ref_name }}
            Commit: ${{ github.sha }}
            
            ${{ github.event.head_commit.message }}

      - name: Notify Slack on success
        if: success()
        uses: slackapi/slack-github-action@v1.26.0
        with:
          channel-id: ${{ secrets.SLACK_CHANNEL_ID }}
          slack-message: |
            ✅ Android test build ready
            *${{ github.event.head_commit.message }}*
            Check Firebase App Distribution for the install link.
        env:
          SLACK_BOT_TOKEN: ${{ secrets.SLACK_BOT_TOKEN }}

      - name: Notify Slack on failure
        if: failure()
        uses: slackapi/slack-github-action@v1.26.0
        with:
          channel-id: ${{ secrets.SLACK_CHANNEL_ID }}
          slack-message: |
            ❌ Android build failed
            *${{ github.event.head_commit.message }}*
            <${{ github.server_url }}/${{ github.repository }}/actions/runs/${{ github.run_id }}|View build logs>
        env:
          SLACK_BOT_TOKEN: ${{ secrets.SLACK_BOT_TOKEN }}

  # iOS build and distribution (manual trigger or release branch only)
  distribute-ios:
    name: iOS Distribution
    runs-on: macos-latest
    if: |
      github.ref == 'refs/heads/release' ||
      github.event.inputs.platform == 'ios' ||
      github.event.inputs.platform == 'all'

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: ${{ env.NODE_VERSION }}
          cache: 'npm'

      - name: Install dependencies
        run: npm ci

      - name: Setup EAS
        uses: expo/expo-github-action@v8
        with:
          eas-version: latest
          token: ${{ secrets.EXPO_TOKEN }}

      - name: Restore GoogleService-Info.plist
        run: |
          echo "${{ secrets.GOOGLE_SERVICE_INFO_PLIST_BASE64 }}" \
            | base64 --decode > ./GoogleService-Info.plist

      - name: Build iOS IPA
        run: |
          eas build \
            --platform ios \
            --profile preview \
            --non-interactive \
            --output ./dist/app.ipa
        env:
          EAS_NO_VCS: 1

      - name: Upload iOS to Firebase
        uses: wzieba/Firebase-Distribution-Github-Action@v1
        with:
          appId: ${{ secrets.FIREBASE_APP_ID_IOS }}
          token: ${{ secrets.FIREBASE_TOKEN }}
          groups: ios-testers
          file: ./dist/app.ipa
          releaseNotes: |
            Branch: ${{ github.ref_name }}
            Commit: ${{ github.sha }}
            ${{ github.event.head_commit.message }}
```

All required GitHub Secrets:

- `EXPO_TOKEN`: expo.dev → Account Settings → Access Tokens
- `FIREBASE_TOKEN`: run `firebase login:ci` locally, copy the output token
- `FIREBASE_APP_ID_ANDROID`: Firebase console → Project Settings → Your apps → App ID
- `FIREBASE_APP_ID_IOS`: same location, iOS app
- `GOOGLE_SERVICES_JSON_BASE64`: run `base64 google-services.json` and paste output
- `GOOGLE_SERVICE_INFO_PLIST_BASE64`: run `base64 GoogleService-Info.plist` and paste output

And the matching `eas.json`:

```json
{
  "build": {
    "preview": {
      "distribution": "internal",
      "android": {
        "buildType": "apk",
        "gradleCommand": ":app:assembleRelease"
      },
      "ios": {
        "simulator": false
      }
    },
    "production": {
      "android": {
        "buildType": "app-bundle",
        "autoIncrement": true
      },

        "autoIncrement": true
      }
    }

```

With this workflow, every push to `main` automatically triggers a build and sends it to your tester group. Typical turnaround from queue to delivery is 6–8 minutes for Android.

## Performance Monitoring: Making Slowness Visible

Crashes are obvious. Performance degradation is invisible until users start leaving negative reviews about the app feeling "sluggish." Firebase Performance Monitoring makes slowness measurable and trackable over time.

```typescript
// utils/trackedAsync.ts
import perf from '@react-native-firebase/perf';
import crashlytics from '@react-native-firebase/crashlytics';

/**
 * Wraps any async operation with Firebase Performance tracing.
 *
 * Naming conventions that work well in the Firebase console:
 *   - fetch_product_list (data fetching)
 *   - submit_checkout_form (user actions)
 *   - upload_profile_photo (media operations)
 *   - load_home_screen (screen performance)
 */
export async function trackedAsync<T>(
  traceName: string,
  asyncFn: () => Promise<T>,
  attributes?: Record<string, string>
): Promise<T> {
  const trace = await perf().startTrace(traceName);

  // Add optional attributes before the operation starts
  if (attributes) {
    for (const [key, value] of Object.entries(attributes)) {
      await trace.putAttribute(key, value);
    }
  }

  try {
    const result = await asyncFn();
    await trace.putAttribute('status', 'success');
    return result;
  } catch (error) {
    await trace.putAttribute('status', 'error');

    // Route errors to Crashlytics for unified error tracking
    const err = error instanceof Error ? error : new Error(String(error));
    crashlytics().recordError(err);
    crashlytics().log(`Trace "${traceName}" failed: ${err.message}`);

    throw error;
  } finally {
    // Always stop the trace—this records the duration
    await trace.stop();
  }
}

// Usage examples:
//
// const products = await trackedAsync(
//   'fetch_product_list',
//   () => supabase.from('products').select('*').limit(20),
//   { category: 'electronics' }
// );
//
// const order = await trackedAsync(
//   'submit_checkout',
//   () => api.post('/orders', checkoutData)
// );
```

After running Performance Monitoring for a few weeks, you'll have concrete data about which screens are slow, how API response times change between app versions, and whether a specific user action is consistently slower than others. This is the kind of signal that's hard to get from user feedback alone—users rarely say "the product list takes 800ms to load," they just stop opening that screen.

Keep these constraints in mind:
- Trace names: max 100 characters, no leading underscores
- Custom attributes per trace: max 5
- Total custom trace types per app: 300 (group by feature, not by individual endpoint)

## Common Pitfalls: What Actually Goes Wrong

Here are the issues I've hit personally, plus the recurring questions that come up in developer communities.

### 1. Unreadable Stack Traces from Hermes

React Native 0.70+ uses Hermes by default, and Hermes produces minified stack traces. If your Crashlytics reports show function names like `$b`, `_e`, `__apply`, you're missing source maps. EAS Build handles source map upload automatically via the Crashlytics Gradle plugin—but only if `jsEngine` is explicitly set.

```json
// app.json — make Hermes explicit
{
  "expo": {
    "jsEngine": "hermes"
  }
}
```

If stack traces are still unreadable after setting this, try rebuilding with `--clear-cache`. Stale EAS Build cache is the most common culprit.

### 2. GoogleService-Info.plist Not Picked Up by EAS Build

Even with `googleServicesFile` correctly set in `app.json`, EAS Build can sometimes use a cached layer that doesn't include your config file. Force a fresh build with:

```bash
eas build --platform ios --profile preview --clear-cache
```

If the problem persists, verify the file path is relative to the project root (not absolute), and that the file name matches exactly including capitalization.

### 3. Firebase Token Expiration

Tokens from `firebase login:ci` are persistent but can be invalidated by Google's security systems, particularly if your account has unusual login activity or security policy changes. If your workflow starts failing with authentication errors after previously working, regenerate the token and update your GitHub Secret.

```bash
# Generate a new token
firebase login:ci

# The output is the new token to store in GitHub Secrets as FIREBASE_TOKEN
```

### 4. Testers Can't Find the Install Link

Firebase App Distribution sends distribution emails from `noreply@firebase.google.com`. These frequently land in spam, especially for Gmail accounts using aggressive filtering. When you first set up the pipeline, explicitly tell testers what to look for—"check for an email from noreply@firebase.google.com and follow the link to register your device." This eliminates 80% of "how do I install it?" questions.

### 5. Crashlytics Reports Not Appearing for Debug Builds

This is expected behavior. Crashlytics collection is disabled by default in debug mode (`__DEV__ === true`). If you test with a local debug build and don't see reports, that's why—not a misconfiguration. Preview and production builds from EAS send reports normally.

```typescript
// Make the behavior explicit in your initialization
crashlytics().setCrashlyticsCollectionEnabled(\!__DEV__);
```

### 6. "Non-fatal errors" Flooding the Dashboard

If you're using `recordError()` broadly (wrapping all API calls, for example), you may find the Crashlytics non-fatal dashboard flooded with routine errors that don't actually need attention. Be intentional about what you record: unexpected errors are worth tracking, but "user is not authenticated" or "network request timed out after retry" probably aren't.

```typescript
// Selective error recording example
async function fetchUserProfile(userId: string) {
  try {
    return await api.get(`/users/${userId}`);
  } catch (error) {
    // Record auth errors—these might indicate an account issue
    if (error.code === 'auth/invalid-token') {
      crashlytics().recordError(error);
    }
    // Don't record routine network timeouts
    if (error.code === 'network/timeout') {
      console.warn('Network timeout on profile fetch');
    }
    throw error;
  }
}
```

## Crash-Free Rate: Targets, Operations, and the Weekly Review Habit

Crashlytics tracks your "crash-free sessions" percentage—the share of all sessions that completed without a crash. This metric matters because App Store and Google Play both factor app stability into featuring decisions and search rankings.

Practical benchmarks:
- **99.5%+**: Strong. At this level you're in contention for App Store featuring.
- **99.0–99.5%**: Acceptable. Negative reviews about stability may start appearing.
- **Below 99.0%**: Needs intervention. Direct impact on ratings and user retention.

The most sustainable improvement approach I've found is a brief weekly review rather than crash-fixing sprints. Every Monday, open Crashlytics and identify the top three crashes by session impact from the previous week. Pick one and fix it completely—root cause, not symptom. Over several weeks this consistently brings crash-free rates to 99.5%+ without burning out.

**Velocity Alerts** are essential for catching regressions quickly. Enable them from the Crashlytics dashboard (bell icon, top right), and you'll get an email when crash rates spike—usually within an hour of a bad release. Combined with your automated distribution pipeline, you can catch, roll back, and re-release within the same day when something goes wrong.

For a comprehensive view of the pre-release process, the [Rork App Store review checklist guide](/en/articles/rork-business/rork-app-store-review-complete-checklist) covers the broader quality verification steps that complement this crash monitoring setup.

## Managing Tester Groups and Release Notes Effectively

Firebase App Distribution lets you organize testers into named groups, which becomes important once you have more than a handful of people testing different builds. A few patterns that work well in practice:

**Separate groups by role**: Create distinct groups like `internal-team`, `qa-testers`, and `beta-users`. Your GitHub Actions workflow can target different groups based on which branch triggered the build—`main` pushes go to `qa-testers`, `release` pushes go to `beta-users`.

```yaml
# Target different groups based on branch
- name: Upload to Firebase App Distribution
  uses: wzieba/Firebase-Distribution-Github-Action@v1
  with:
    appId: ${{ secrets.FIREBASE_APP_ID_ANDROID }}
    token: ${{ secrets.FIREBASE_TOKEN }}
    groups: |
      ${{ github.ref == 'refs/heads/release' 00260026 'beta-users' || 'qa-testers' }}
    file: ./dist/app.apk
```

**Write meaningful release notes**: The default commit message is often enough for internal builds, but for beta users who are not reading your code, a brief plain-language description helps them know what to test. A good release note pattern:

```
What changed:
- Fixed crash on product detail screen when image fails to load
- Improved checkout form validation error messages
- Added dark mode support for notification screens

What to test:
- Add any item to cart, navigate to product detail, force airplane mode, then try loading
- Submit the checkout form with intentionally wrong card details
- Toggle system dark mode and check all screens
```

**Keep tester lists pruned**: Firebase App Distribution2019s free tier supports up to 50 active testers. Remove testers who have not opened a build in the last 60 days—this keeps your group relevant and your limit from filling up with inactive accounts.

You can view which testers have actually opened each build from the App Distribution console, which is useful for confirming coverage before submitting a release to the App Store.

## Getting Started: The 20-Minute First Step

The full Firebase App Distribution + Crashlytics + GitHub Actions pipeline is a 2–3 hour setup that becomes largely self-maintaining. But you don't need to do it all at once.

If you're starting today, skip App Distribution for now and spend 20 minutes getting Crashlytics integrated and confirmed working. Here's the minimum viable sequence:

1. Create Firebase project, register your app, download config files
2. Install `@react-native-firebase/app` and `@react-native-firebase/crashlytics`
3. Add plugin config to `app.json`
4. Add `initCrashlytics()` call to your root layout
5. Build a preview APK with EAS and trigger a test crash
6. Confirm the report appears in Firebase console

That baseline alone—Crashlytics running on your EAS preview builds—gives you visibility into whether your releases are introducing regressions. The GitHub Actions automation and App Distribution can follow in a second session once you've confirmed the fundamentals work.

The difference between apps that improve incrementally and ones that plateau is rarely about feature development. It's about having a feedback loop tight enough to act on. A crash report that arrives the moment testers hit a bug is a completely different experience from discovering it through a negative App Store review two weeks later.

Build the feedback loop first. Everything else follows from there.
