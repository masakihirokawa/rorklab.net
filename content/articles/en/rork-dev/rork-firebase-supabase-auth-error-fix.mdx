If you've integrated Firebase or Supabase authentication into your Rork app, you've probably run into the frustrating experience of auth working fine in development — then mysteriously breaking in production or on a real device. These kinds of errors hit users at the worst possible moment and can seriously hurt retention.

This guide covers the most common Firebase and Supabase auth errors in Rork apps, their root causes, and how to fix them step by step.

## Common Auth Error Symptoms

When auth breaks in a Rork app, you'll typically see one of these messages in your console or network logs.

**Firebase Auth errors:**
- `auth/network-request-failed` — network connectivity issue
- `auth/user-token-expired` — token has expired and wasn't refreshed
- `auth/invalid-api-key` — wrong or malformed API key
- `auth/unauthorized-domain` — request coming from a domain not in the allowlist
- `auth/too-many-requests` — rate limiting triggered by rapid login attempts

**Supabase Auth errors:**
- `AuthApiError: Invalid login credentials` — wrong email/password
- `JWT expired` — access token past its TTL
- `AuthSessionMissingError` — session data missing from storage
- `new row violates row-level security policy` — RLS blocking the operation
- HTTP `401 Unauthorized` or `403 Forbidden`

Match your symptoms to the relevant fix section below.

## Root Cause Analysis: Why Auth Errors Happen

Auth failures in Rork apps generally fall into five categories.

**Pattern 1: Misconfigured API keys or connection strings**

It's surprisingly easy to paste the wrong API key, especially when juggling development and production environments. A single typo or stale value in your config file is enough to break auth entirely.

**Pattern 2: Token expiry with no auto-refresh**

Firebase ID tokens expire after one hour. Supabase access tokens have a configurable TTL (default: 1 hour). If your app doesn't automatically refresh tokens, users who leave the app open in the background will get auth errors when they return.

**Pattern 3: CORS policy violations (Web Preview)**

When testing your app in Rork's Web Preview, browser CORS policies can block auth requests. This is especially common with Supabase, which requires explicit origin configuration.

**Pattern 4: Overly restrictive RLS policies**

If you've enabled Row Level Security in Supabase, misconfigured policies can prevent authenticated users from reading or writing their own data, resulting in 403 errors after a successful login.

**Pattern 5: Network configuration issues (native builds)**

Auth errors that only appear in native iOS or Android builds often trace back to network-related settings — iOS's App Transport Security (ATS) policy being a common culprit for HTTPS-related failures.

## Step-by-Step Fixes

### Step 1: Verify your API keys and connection config

Start by ruling out the simplest cause.

**For Firebase:**

```javascript
// firebaseConfig.js — double-check every value against Firebase Console
const firebaseConfig = {
  apiKey: "YOUR_API_KEY",
  authDomain: "yourapp.firebaseapp.com",
  projectId: "your-project-id",
  storageBucket: "your-project.appspot.com",
  messagingSenderId: "123456789",
  appId: "1:123456789:web:abcdef"
};

// Confirm Firebase initializes correctly
import { initializeApp, getApps } from 'firebase/app';

const app = getApps().length === 0
  ? initializeApp(firebaseConfig)
  : getApps()[0];

console.log("Firebase initialized:", app.name); // Should log "[DEFAULT]"
```

**For Supabase:**

```javascript
// supabaseClient.js
import { createClient } from '@supabase/supabase-js';

const supabaseUrl = process.env.EXPO_PUBLIC_SUPABASE_URL;
const supabaseAnonKey = process.env.EXPO_PUBLIC_SUPABASE_ANON_KEY;

if (!supabaseUrl || !supabaseAnonKey) {
  throw new Error('Missing Supabase URL or Anon Key');
}

export const supabase = createClient(supabaseUrl, supabaseAnonKey);
```

In Expo, environment variables **must** use the `EXPO_PUBLIC_` prefix to be accessible in client code. After any `.env` changes, run `npx expo start --clear` to flush the cache.

### Step 2: Implement token auto-refresh

**Firebase — listen to auth state changes:**

```javascript
import { getAuth, onAuthStateChanged } from 'firebase/auth';

const auth = getAuth();

export function useAuthState() {
  const [user, setUser] = useState(null);
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    // Firebase auto-refreshes tokens internally; this listener
    // keeps your app state in sync
    const unsubscribe = onAuthStateChanged(auth, (currentUser) => {
      setUser(currentUser);
      setLoading(false);
    });

    return () => unsubscribe();
  }, []);

  return { user, loading };
}
```

**Supabase — persist sessions with AsyncStorage and AppState:**

```javascript
import { createClient } from '@supabase/supabase-js';
import AsyncStorage from '@react-native-async-storage/async-storage';
import { AppState } from 'react-native';

export const supabase = createClient(supabaseUrl, supabaseAnonKey, {
  auth: {
    storage: AsyncStorage,        // Persist session across app restarts
    autoRefreshToken: true,       // Automatically refresh expiring tokens
    persistSession: true,         // Keep session in storage
    detectSessionInUrl: false,    // Must be false for React Native
  },
});

// Resume auto-refresh when app comes back to foreground
AppState.addEventListener('change', (state) => {
  if (state === 'active') {
    supabase.auth.startAutoRefresh();
  } else {
    supabase.auth.stopAutoRefresh();
  }
});
```

### Step 3: Add allowed domains in Firebase

If auth works in native builds but fails in Web Preview, the domain probably isn't in Firebase's allowlist.

1. Open [Firebase Console](https://console.firebase.google.com/)
2. Go to **Authentication → Settings → Authorized domains**
3. Add the following:
   - `localhost`
   - Rork's preview URL (e.g., `*.rork.app`)
   - Your production domain (e.g., `yourapp.com`)

### Step 4: Audit your Supabase RLS policies

If you're seeing 403 errors after a successful login, RLS is likely the culprit. Test by temporarily disabling RLS on the affected table:

```sql
-- Temporary diagnostic step — restore RLS after testing
ALTER TABLE your_table DISABLE ROW LEVEL SECURITY;
```

If that fixes the error, your policy needs updating. Here's a policy that lets authenticated users access only their own data:

```sql
-- Allow authenticated users to manage their own rows
CREATE POLICY "Users can access their own data"
ON your_table
FOR ALL
USING (auth.uid() = user_id)
WITH CHECK (auth.uid() = user_id);
```

Make sure to re-enable RLS once you've confirmed the correct policy is in place.

### Step 5: Check iOS ATS settings

If auth errors only appear on iOS devices, verify your ATS config in `app.json`:

```json
{
  "expo": {
    "ios": {
      "infoPlist": {
        "NSAppTransportSecurity": {
          "NSAllowsArbitraryLoads": false,
          "NSExceptionDomains": {
            "your-supabase-project.supabase.co": {
              "NSExceptionAllowsInsecureHTTPLoads": false,
              "NSIncludesSubdomains": true
            }
          }
        }
      }
    }
  }
}
```

Both Firebase and Supabase use HTTPS, so `NSAllowsArbitraryLoads` should remain `false`. Only add explicit exceptions for custom API domains you control.

## Verification: How to Confirm the Fix

After applying a fix, use these checks to verify everything is working correctly.

**Inspect network requests:**

```javascript
// Add debug logging to Supabase requests
const supabase = createClient(supabaseUrl, supabaseAnonKey, {
  global: {
    fetch: (...args) => {
      console.log('[Supabase] →', args[0]);
      return fetch(...args).then(res => {
        console.log('[Supabase] ←', res.status);
        return res;
      });
    }
  }
});
```

**Check session validity:**

```javascript
// Confirm the current session is active and not expired
const { data: { session } } = await supabase.auth.getSession();
console.log('Session status:', session ? 'Active' : 'None');
console.log('Expires at:', session?.expires_at
  ? new Date(session.expires_at * 1000).toISOString()
  : 'N/A');
```

**Test across all build targets:**

Verify the complete sign-in → use app → sign-out → session restore flow in your development build, TestFlight/internal test build, and production build. Environment-specific bugs only surface when you test in the right context.

## Prevention: Best Practices to Avoid Future Auth Errors

**Centralize error handling:**

```javascript
async function signIn(email, password) {
  try {
    const { data, error } = await supabase.auth.signInWithPassword({
      email,
      password,
    });

    if (error) {
      if (error.message.includes('Invalid login credentials')) {
        throw new Error('Incorrect email or password. Please try again.');
      } else if (error.message.includes('Email not confirmed')) {
        throw new Error('Please verify your email address before signing in.');
      } else {
        throw new Error(`Sign-in failed: ${error.message}`);
      }
    }

    return data.user;
  } catch (err) {
    // Forward to error tracking (Sentry, Firebase Crashlytics, etc.)
    console.error('[Auth] Sign-in error:', err.message);
    throw err;
  }
}
```

**Add auth guards to protected screens:**

Check session validity at critical navigation points and redirect to the login screen if the session has expired. This prevents users from hitting confusing errors mid-session.

## FAQ

**Q1. Auth works in development but breaks only in the production build. Why?**

Production builds disable debug mode, which means some errors are silently swallowed. Enable an error tracking service (Sentry, Firebase Crashlytics) in your production configuration so you can see what's actually failing. Also double-check that your production API keys and project IDs are correctly set in your build environment — it's easy to accidentally ship a build pointing at your dev project.

**Q2. I keep getting "new row violates row-level security policy". How do I fix it?**

This means your RLS policy is blocking the operation. In the Supabase dashboard, go to **Authentication → Policies** and check the policies on the affected table. If no policy exists for `INSERT`, your writes will be blocked by default when RLS is enabled. Also make sure `auth.uid()` isn't returning `null` — if the user isn't authenticated when the operation is attempted, no RLS policy will match.

**Q3. Google/Apple Sign-In fails only on iOS real devices. What should I check?**

For native OAuth on iOS, you need `GoogleService-Info.plist` (Firebase) or the correct `clientId` (Supabase) embedded in your app bundle. Confirm the Bundle ID in your Xcode project or `app.json` exactly matches what's registered in Firebase Console or your OAuth provider. If you're using EAS Build, verify that `eas.json` correctly references the path to `GoogleService-Info.plist`.

**Q4. Users get logged out frequently because JWTs expire. Is there a way to extend this?**

Yes — in the Supabase dashboard, go to **Authentication → Settings** and increase the "JWT expiry limit" (up to 604800 seconds / 7 days). More importantly, implement the `AppState`-based session management from Step 2 so tokens refresh automatically when users return to the app after time away.

**Q5. Auth behaves differently in Rork's Web Preview vs. a native build.**

Rork's Web Preview runs in a browser environment where `AsyncStorage` falls back to `localStorage`, which isn't fully equivalent. Auth flows — especially OAuth redirects — behave differently across environments. Treat Web Preview as a UI preview only, and always verify auth behavior on a simulator or real device.

## Summary

Firebase and Supabase auth errors in Rork apps typically come down to a handful of root causes:

- **Misconfigured API keys** → re-check values against your project console
- **Token expiry** → implement `autoRefreshToken` with `AppState` integration
- **CORS / domain issues** → add your domains to the Firebase or Supabase allowlist
- **RLS policy blocks** → review and update policies step by step
- **iOS ATS settings** → verify `NSAppTransportSecurity` in `app.json`

Auth is the kind of thing that can break silently without good monitoring in place. Getting error tracking set up early pays dividends every time something unexpected happens in production.
