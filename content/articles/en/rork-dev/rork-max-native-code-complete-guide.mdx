Rork Max's ability to generate a working app skeleton in minutes is genuinely impressive. The wall most developers hit comes a few days later, when the app that looked perfect in preview reveals a collection of issues on a real device.

These issues aren't really about Rork being imperfect — they're about the gap between Rork's preview environment and the requirements of a real Expo/React Native app. The patterns repeat, and knowing them in advance saves hours of debugging.

This guide covers the ten most common pitfalls in Rork Max native development, with working code fixes for each.

## Pitfall 1: SwiftUI Code Doesn't Behave as Expected

Rork Max can output SwiftUI code for iOS, but the quality varies significantly based on how specific your prompt is.

**Prompt engineering for better SwiftUI output:**

```
❌ Vague: "Build a task manager in SwiftUI"

✅ Specific: "Build a task manager app with these specifications:
- Target: iOS 17+
- UI framework: SwiftUI with NavigationStack
- State management: @Observable macro (NOT ObservableObject + @Published)
- Persistence: SwiftData
- Features: CRUD tasks, toggle completion status
- Screens: Task list → Task detail/edit (push navigation)"
```

When you don't specify the target iOS version, Rork often generates code using `ObservableObject + @Published`, which is the older pattern. For iOS 17+, `@Observable` is the recommended approach:

```swift
// ❌ Older pattern Rork sometimes generates
class TaskViewModel: ObservableObject {
    @Published var tasks: [Task] = []
}

// ✅ iOS 17+ pattern — request this explicitly in your prompt
@Observable
class TaskViewModel {
    var tasks: [Task] = []
    
    func addTask(_ title: String) {
        tasks.append(Task(id: UUID(), title: title, isCompleted: false))
    }
    
    func toggleTask(_ task: Task) {
        if let index = tasks.firstIndex(where: { $0.id == task.id }) {
            tasks[index].isCompleted.toggle()
        }
    }
}
```

After generation, check the Swift version and iOS deployment target in `Package.swift` or project settings. Rork sometimes targets older SDK versions by default.

## Pitfall 2: Expo Router File Structure Breaks Navigation

Rork Max generates Expo Router (App Router pattern) apps, but the file structure sometimes has gaps that cause routing errors or blank screens.

The expected structure for a tab-based app:

```
app/
├── _layout.tsx         ← Root layout (required)
├── index.tsx           ← Home/redirect screen  
├── (tabs)/
│   ├── _layout.tsx     ← Tab bar configuration
│   ├── home.tsx
│   └── profile.tsx
├── (auth)/
│   ├── login.tsx
│   └── signup.tsx
└── modal.tsx           ← Presented as modal
```

The `_layout.tsx` files are where Rork commonly cuts corners:

```typescript
// app/_layout.tsx — complete root layout
import { Stack } from 'expo-router';
import { useColorScheme } from 'react-native';

export default function RootLayout() {
  const colorScheme = useColorScheme();
  
  return (
    <Stack
      screenOptions={{
        headerStyle: {
          backgroundColor: colorScheme === 'dark' ? '#1a1a1a' : '#ffffff',
        },
        headerTintColor: colorScheme === 'dark' ? '#ffffff' : '#000000',
        headerShadowVisible: false,
      }}
    >
      <Stack.Screen name="(tabs)" options={{ headerShown: false }} />
      <Stack.Screen name="(auth)" options={{ headerShown: false }} />
      <Stack.Screen 
        name="modal" 
        options={{ presentation: 'modal', title: '' }} 
      />
    </Stack>
  );
}
```

```typescript
// app/(tabs)/_layout.tsx — tab bar configuration
import { Tabs } from 'expo-router';
import { Ionicons } from '@expo/vector-icons';

export default function TabLayout() {
  return (
    <Tabs screenOptions={{ tabBarActiveTintColor: '#007AFF' }}>
      <Tabs.Screen
        name="home"
        options={{
          title: 'Home',
          tabBarIcon: ({ color, size }) => (
            <Ionicons name="home" size={size} color={color} />
          ),
        }}
      />
      <Tabs.Screen
        name="profile"
        options={{
          title: 'Profile',
          tabBarIcon: ({ color, size }) => (
            <Ionicons name="person" size={size} color={color} />
          ),
        }}
      />
    </Tabs>
  );
}
```

When you get a blank screen or an error about an unmatched route, check whether the screen name in `<Stack.Screen name="...">` matches the actual filename.

## Pitfall 3: Supabase Auth Not Persisting Sessions

Rork-generated Supabase integration commonly misses two things: session persistence with `AsyncStorage` and auth state routing.

```typescript
// lib/supabase.ts — correct mobile configuration
import { createClient } from '@supabase/supabase-js';
import AsyncStorage from '@react-native-async-storage/async-storage';
import { AppState } from 'react-native';

export const supabase = createClient(
  process.env.EXPO_PUBLIC_SUPABASE_URL\!,
  process.env.EXPO_PUBLIC_SUPABASE_ANON_KEY\!,
  {
    auth: {
      storage: AsyncStorage,       // Mobile requires AsyncStorage
      autoRefreshToken: true,
      persistSession: true,        // Without this, users re-login on every app open
      detectSessionInUrl: false,   // Must be false for native apps
    },
  }
);

// Auto-refresh tokens when app comes to foreground
AppState.addEventListener('change', (state) => {
  if (state === 'active') {
    supabase.auth.startAutoRefresh();
  } else {
    supabase.auth.stopAutoRefresh();
  }
});
```

```typescript
// app/_layout.tsx — auth state routing
import { useEffect, useState } from 'react';
import { Slot, useRouter, useSegments } from 'expo-router';
import { supabase } from '../lib/supabase';
import type { Session } from '@supabase/supabase-js';

export default function RootLayout() {
  const router = useRouter();
  const segments = useSegments();
  const [session, setSession] = useState<Session | null>(null);
  const [isLoading, setIsLoading] = useState(true);

  useEffect(() => {
    supabase.auth.getSession().then(({ data: { session } }) => {
      setSession(session);
      setIsLoading(false);
    });

    const { data: { subscription } } = supabase.auth.onAuthStateChange(
      (_event, session) => setSession(session)
    );

    return () => subscription.unsubscribe();
  }, []);

  useEffect(() => {
    if (isLoading) return;

    const inAuthGroup = segments[0] === '(auth)';

    if (\!session && \!inAuthGroup) {
      router.replace('/(auth)/login');
    } else if (session && inAuthGroup) {
      router.replace('/(tabs)/home');
    }
  }, [session, segments, isLoading]);

  if (isLoading) return null;

  return <Slot />;
}
```

## Pitfall 4: Push Notifications Setup Is Incomplete

Expo Notifications requires configuration in three places: the code, `app.json`, and (for EAS builds) the EAS project. Rork often gets one or two of these right but not all three.

```typescript
// hooks/useNotifications.ts — complete implementation
import { useEffect, useRef, useState } from 'react';
import * as Notifications from 'expo-notifications';
import * as Device from 'expo-device';
import Constants from 'expo-constants';
import { Platform } from 'react-native';

Notifications.setNotificationHandler({
  handleNotification: async () => ({
    shouldShowAlert: true,
    shouldPlaySound: false,
    shouldSetBadge: false,
  }),
});

async function registerForPushNotifications(): Promise<string | undefined> {
  if (\!Device.isDevice) {
    console.warn('Push tokens require a physical device.');
    return;
  }

  const { status: existing } = await Notifications.getPermissionsAsync();
  const { status } = existing === 'granted'
    ? { status: existing }
    : await Notifications.requestPermissionsAsync();

  if (status \!== 'granted') {
    console.warn('Push notification permission denied.');
    return;
  }

  // EAS project ID is required for Expo Push Tokens
  const projectId = Constants.expoConfig?.extra?.eas?.projectId;
  if (\!projectId) throw new Error('EAS Project ID not found in app.json');

  const token = await Notifications.getExpoPushTokenAsync({ projectId });

  if (Platform.OS === 'android') {
    await Notifications.setNotificationChannelAsync('default', {
      name: 'default',
      importance: Notifications.AndroidImportance.MAX,
    });
  }

  return token.data;
}

export function useNotifications() {
  const [pushToken, setPushToken] = useState<string>();
  const notificationRef = useRef<Notifications.Subscription>();
  const responseRef = useRef<Notifications.Subscription>();

  useEffect(() => {
    registerForPushNotifications().then(setPushToken);

    notificationRef.current = Notifications.addNotificationReceivedListener(
      (notification) => console.log('Notification received:', notification.request.identifier)
    );

    responseRef.current = Notifications.addNotificationResponseReceivedListener(
      (response) => {
        const screen = response.notification.request.content.data?.screen as string;
        if (screen) {
          // Navigate to screen from notification tap
        }
      }
    );

    return () => {
      notificationRef.current?.remove();
      responseRef.current?.remove();
    };
  }, []);

  return { pushToken };
}
```

Required `app.json` configuration:

```json
{
  "expo": {
    "extra": {
      "eas": { "projectId": "your-eas-project-id" }
    },
    "plugins": [
      [
        "expo-notifications",
        {
          "icon": "./assets/notification-icon.png",
          "color": "#ffffff"
        }
      ]
    ]
  }
}
```

## Pitfall 5: Camera and Image Picker Not Working

```typescript
// Correct camera/image picker implementation
import * as ImagePicker from 'expo-image-picker';

export async function pickImage(): Promise<string | null> {
  // Request permission first
  const { status } = await ImagePicker.requestMediaLibraryPermissionsAsync();
  
  if (status \!== 'granted') {
    alert('Photo library access is required to pick images.');
    return null;
  }

  const result = await ImagePicker.launchImageLibraryAsync({
    mediaTypes: ImagePicker.MediaTypeOptions.Images,
    allowsEditing: true,
    aspect: [4, 3],
    quality: 0.8, // Compress to 80% — reduce upload size
  });

  if (\!result.canceled && result.assets[0]) {
    return result.assets[0].uri;
  }
  return null;
}
```

Required `app.json` additions:

```json
{
  "expo": {
    "ios": {
      "infoPlist": {
        "NSPhotoLibraryUsageDescription": "Used to select profile pictures and attachments",
        "NSCameraUsageDescription": "Used to take photos"
      }
    },
    "plugins": [
      ["expo-image-picker", { "photosPermission": "Used to select images" }]
    ]
  }
}
```

## Pitfalls 6–10: Quick Reference

**6. React Query cache behaving differently on device**
Set `staleTime` and `gcTime` explicitly. Default values behave differently across environments.

```typescript
const queryClient = new QueryClient({
  defaultOptions: {
    queries: {
      staleTime: 1000 * 60 * 5, // 5 minutes
      gcTime: 1000 * 60 * 10,   // 10 minutes
      retry: 2,
    },
  },
});
```

**7. Android keyboard covers input fields**
Add to `app.json`:
```json
{ "expo": { "android": { "softwareKeyboardLayoutMode": "pan" } } }
```

**8. EAS Build fails on npm install**
Run `npx expo install --fix` first, then `npm install --legacy-peer-deps` if peer dependency conflicts persist.

**9. App Store rejection: missing permission descriptions**
Audit every native module your app uses and add corresponding `NSXxxUsageDescription` strings to `app.json ios.infoPlist`. Apple reviewers check all of them.

**10. Rork generates the same code repeatedly**
Add to your prompt: "Use a different approach than previous generations" or "Refactor the existing implementation, don't rewrite from scratch." Rork tends to converge on familiar patterns in long sessions.

## Prompting for Production-Quality Code

The most consistent way to get better output from Rork Max is to be explicit about your production requirements upfront:

```
Build a [app type] with these specifications:

Tech stack:
- Expo SDK 52 + TypeScript + Expo Router v3 (App Router)
- Backend: Supabase (auth + database)
- State: Zustand for global state
- Styling: NativeWind

Quality requirements:
- Error handling on every async operation (try/catch + user-facing error messages)
- Loading states for all API calls
- Empty state components for empty lists
- TypeScript strict mode (no 'any' types)

Current state: [describe what's already built]
New feature: [exactly what you want to add]
```

Specific prompts produce specific, production-ready code. Vague prompts produce working prototypes that need significant cleanup before shipping. The investment in a well-structured prompt pays back quickly in less debugging time.
