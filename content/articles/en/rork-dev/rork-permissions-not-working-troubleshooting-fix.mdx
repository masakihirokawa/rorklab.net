Everything works fine in Expo Go — the camera launches, location is detected, notifications fire on schedule. Then you hand off a TestFlight build and the first message back is: "The camera dialog never appeared." Sound familiar?

Permission bugs are among the most frustrating to diagnose in Rork-based apps because they often produce no error messages at all. The OS simply silences the request, leaving you with a feature that appears to run but silently fails. This guide covers the three main failure patterns, platform-specific configuration, and how to handle the tricky case of permissions that have already been denied.

## Why Rork AI-Generated Code Tends to Miss Permission Declarations

When Rork generates a camera screen, it correctly adds the `useCameraPermissions` hook to the component — but it doesn't always update `app.json` with the corresponding native declaration (`NSCameraUsageDescription` on iOS, `CAMERA` in the Android permissions array). The component code looks complete, so the issue is invisible until you build.

There are three distinct patterns worth knowing:

**Pattern 1: The dialog never appears at all.** The native-side declaration (`Info.plist` / `AndroidManifest.xml`) is missing, so the OS doesn't know the app is allowed to request that permission. No error, no dialog — just silence. This is the hardest to notice during development.

**Pattern 2: It worked once, then stopped.** The user tapped "Don't Allow" on first launch. On iOS, once a permission is denied, the app cannot prompt again — the only path forward is directing the user to Settings. If your code doesn't handle the `DENIED` state, the feature will appear broken indefinitely.

**Pattern 3: Works in Expo Go, fails in a standalone build.** Expo Go ships with a broad set of pre-declared permissions. If you only tested in Expo Go, you may have been riding those declarations without realizing it. A clean EAS build uses only your app's declarations.

## Setting Up the Native Declarations

The declarations go in `app.json` and need to match exactly what your app requests at runtime.

**iOS (`expo.ios.infoPlist` in `app.json`)**

```json
{
  "expo": {
    "ios": {
      "infoPlist": {
        "NSCameraUsageDescription": "Used to take photos and record videos.",
        "NSMicrophoneUsageDescription": "Used for audio recording in videos.",
        "NSLocationWhenInUseUsageDescription": "Used to find spots near your current location.",
        "NSPhotoLibraryUsageDescription": "Used to save and load photos.",
        "NSPhotoLibraryAddUsageDescription": "Used to save photos to your library."
      }
    }
  }
}
```

**Android (`expo.android.permissions` in `app.json`)**

```json
{
  "expo": {
    "android": {
      "permissions": [
        "CAMERA",
        "RECORD_AUDIO",
        "ACCESS_FINE_LOCATION",
        "ACCESS_COARSE_LOCATION",
        "READ_MEDIA_IMAGES",
        "READ_MEDIA_VIDEO"
      ]
    }
  }
}
```

One important note on `NSLocationAlwaysAndWhenInUseUsageDescription`: declaring this key alone triggers Apple's review process for background location usage. If your app doesn't genuinely need background location, remove it. Apple's Guideline 2.5.4 flags apps that request broader access than necessary, and background location is one of the most scrutinized categories.

## Handling the Full Permission Lifecycle

Rork's generated components often call `requestPermission()` directly without branching on the current status. That works exactly once — after a denial, it's a no-op. Here's a more complete pattern:

```typescript
import { Alert, Linking } from 'react-native';
import { useCameraPermissions, PermissionStatus } from 'expo-camera';

export function useCameraWithFallback() {
  const [permission, requestPermission] = useCameraPermissions();

  const requestWithFallback = async () => {
    // First-time request: show the system dialog
    if (permission?.status === PermissionStatus.UNDETERMINED) {
      const result = await requestPermission();
      return result.granted;
    }

    // Already denied: the OS won't show the dialog again
    // Direct the user to Settings instead
    if (permission?.status === PermissionStatus.DENIED) {
      Alert.alert(
        'Camera Access Required',
        'To use this feature, please enable Camera access in the Settings app.',
        [
          { text: 'Cancel', style: 'cancel' },
          {
            text: 'Open Settings',
            onPress: () => Linking.openSettings(),
          },
        ]
      );
      return false;
    }

    // Already granted
    return permission?.granted ?? false;
  };

  return { permission, requestPermission: requestWithFallback };
}
```

The key distinction is `UNDETERMINED` vs `DENIED`. Once a user denies a permission on iOS, calling `requestPermission()` again does nothing — no dialog, no error. The only viable path is `Linking.openSettings()`, which opens your app's page in the Settings app where the user can manually re-enable it.

## Location Permissions: The Two-Step iOS Upgrade

Location access has more granularity than most other permissions, and iOS enforces a specific ordering that catches developers off guard.

You cannot request "Always Allow" (background location) directly. iOS will internally convert the request to "While Using the App" on the first prompt. The upgrade to "Always Allow" must happen as a second, separate step after the user has already granted foreground access:

```typescript
import * as Location from 'expo-location';

async function requestLocationPermission() {
  // Step 1: Request foreground access first
  const { status: foregroundStatus } =
    await Location.requestForegroundPermissionsAsync();

  if (foregroundStatus \!== 'granted') {
    return false;
  }

  // Step 2: Only request background if your app genuinely needs it
  // Remove this block entirely if you only need "while using" access
  const { status: backgroundStatus } =
    await Location.requestBackgroundPermissionsAsync();

  if (backgroundStatus \!== 'granted') {
    // Fall back to foreground-only functionality
    return 'foreground-only';
  }

  return 'full';
}
```

Most apps that use location for showing nearby content don't need background access at all. Requesting it anyway increases your App Store review friction without adding user value.

## Testing Permissions Correctly

Expo Go is not a reliable environment for testing permission behavior. Two things you must verify in an actual standalone build:

**Test the denial flow.** Build with EAS and install via TestFlight or the Google Play Internal Track. Tap "Don't Allow" when the permission dialog appears, then try to use the feature again. You should see your Settings redirect alert. If nothing happens, your fallback handling is missing.

**Review the usage description strings.** Rork sometimes generates these in English even for Japanese-targeted apps. Apple expects the usage description to be in the same language as the app's primary locale. Go through each key in `app.json` before submitting and write clear, specific descriptions — "Used for X" is more likely to pass than vague phrases like "For app functionality."

## App Store and Google Play Rejection Patterns

Permission-related rejections are common and almost always avoidable.

**App Store rejections to watch for:**

- Declaring permissions your app doesn't use (Guideline 5.1.1) — audit your `Info.plist` and remove unused keys
- Vague or missing usage descriptions (Guideline 2.5.4)
- Requesting background location without a clear justification in the review notes

**Google Play rejections to watch for:**

- Using `READ_EXTERNAL_STORAGE` on apps targeting API level 33+ — Android 13 replaced this with `READ_MEDIA_IMAGES` and `READ_MEDIA_VIDEO`
- Declaring "dangerous" permissions (camera, microphone, contacts) without requesting them at runtime

Android's permission API has changed significantly across OS versions. If you're using an older Expo SDK, check whether your target API level matches your declared permissions — inconsistencies here produce unexpected behavior that's hard to reproduce during development.

## Quick Reference Checklist

Before pushing any build that involves permissions, run through these:

- Every runtime permission request has a matching declaration in `app.json`
- Code handles `UNDETERMINED`, `DENIED`, and `GRANTED` as separate cases
- The `DENIED` path shows a clear alert with an "Open Settings" button
- Tested on a real standalone build (not just Expo Go)
- Usage description strings are in the app's primary language
- No permission is declared that the app doesn't actually use

Catching permission issues at this stage takes five minutes. Catching them after a user complaint — or an App Store rejection — takes much longer.
