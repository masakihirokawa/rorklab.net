## Why Dynamic Island and Live Activities Matter

Dynamic Island, introduced with the iPhone 14 Pro, transforms the camera cutout area into an interactive display for real-time information. From ride-sharing ETAs to sports scores and food delivery tracking, it lets users stay informed without opening your app.

Live Activities work hand-in-hand with Dynamic Island by displaying real-time updates on the Lock Screen through Apple's ActivityKit framework. Since their introduction in iOS 16.1, adoption has surged, and in 2026 they're a standard feature in most popular apps.

With Rork Max, you can generate native Swift code for Dynamic Island and Live Activities without directly working in Xcode. This guide walks you through the core concepts of ActivityKit and provides step-by-step implementation instructions.

## Understanding the ActivityKit Architecture

Implementing Live Activities requires Apple's ActivityKit framework. There are three key components to understand.

**ActivityAttributes**: A protocol that defines the data model for your Live Activity. It separates static data (information that doesn't change) from dynamic data (information updated in real time).

**ActivityContent**: An object that holds the current state. Each update passes a new ContentState instance.

**Widget**: The UI displayed on Dynamic Island and the Lock Screen, built with SwiftUI.

Here's an ActivityAttributes definition using a food delivery tracking example:

```swift
// DeliveryActivity.swift
import ActivityKit
import SwiftUI

struct DeliveryAttributes: ActivityAttributes {
    // Static data (doesn't change after activity starts)
    var restaurantName: String
    var orderNumber: String

    // Dynamic data (updated in real time)
    struct ContentState: Codable, Hashable {
        var status: DeliveryStatus
        var estimatedMinutes: Int
        var driverName: String
    }
}

enum DeliveryStatus: String, Codable {
    case preparing   // Being prepared
    case pickedUp    // Picked up
    case onTheWay    // In transit
    case arriving    // Almost there
    case delivered   // Delivered
}
// Expected behavior: The struct conforming to ActivityAttributes
// serves as the data model for the Live Activity
```

The `restaurantName` and `orderNumber` are fixed when the activity starts, while `status` and `estimatedMinutes` inside `ContentState` change in real time.

## Configuring the Widget Extension in Rork Max

In Rork Max, you can prompt "Add a Widget Extension with Live Activities support" to auto-generate the necessary files. The generated structure includes:

Here's how you build the Lock Screen Live Activity UI with SwiftUI:

```swift
// DeliveryLiveActivityView.swift
import SwiftUI
import WidgetKit
import ActivityKit

struct DeliveryLiveActivityWidget: Widget {
    var body: some WidgetConfiguration {
        ActivityConfiguration(for: DeliveryAttributes.self) { context in
            // Lock Screen UI
            VStack(alignment: .leading, spacing: 8) {
                HStack {
                    Text(context.attributes.restaurantName)
                        .font(.headline)
                        .foregroundColor(.primary)
                    Spacer()
                    Text("Order #\(context.attributes.orderNumber)")
                        .font(.caption)
                        .foregroundColor(.secondary)
                }

                // Progress bar
                ProgressView(value: progressValue(for: context.state.status))
                    .tint(.green)

                HStack {
                    Label(statusText(for: context.state.status),
                          systemImage: statusIcon(for: context.state.status))
                        .font(.subheadline)
                    Spacer()
                    Text("~\(context.state.estimatedMinutes) min")
                        .font(.subheadline)
                        .bold()
                }
            }
            .padding()
            .activityBackgroundTint(.white.opacity(0.9))
        } dynamicIsland: { context in
            // Dynamic Island definition (detailed in next section)
            DynamicIsland {
                DynamicIslandExpandedRegion(.leading) {
                    Label(context.state.driverName, systemImage: "car.fill")
                        .font(.caption)
                }
                DynamicIslandExpandedRegion(.trailing) {
                    Text("~\(context.state.estimatedMinutes) min")
                        .font(.caption)
                        .bold()
                }
                DynamicIslandExpandedRegion(.bottom) {
                    ProgressView(value: progressValue(for: context.state.status))
                        .tint(.green)
                }
            } compactLeading: {
                Image(systemName: statusIcon(for: context.state.status))
            } compactTrailing: {
                Text("\(context.state.estimatedMinutes)m")
                    .font(.caption)
            } minimal: {
                Image(systemName: "bag.fill")
            }
        }
    }

    // Returns progress value based on status
    private func progressValue(for status: DeliveryStatus) -> Double {
        switch status {
        case .preparing: return 0.2
        case .pickedUp: return 0.4
        case .onTheWay: return 0.6
        case .arriving: return 0.8
        case .delivered: return 1.0
        }
    }

    private func statusText(for status: DeliveryStatus) -> String {
        switch status {
        case .preparing: return "Preparing"
        case .pickedUp: return "Picked Up"
        case .onTheWay: return "On the Way"
        case .arriving: return "Arriving Soon"
        case .delivered: return "Delivered"
        }
    }

    private func statusIcon(for status: DeliveryStatus) -> String {
        switch status {
        case .preparing: return "takeoutbag.and.cup.and.straw"
        case .pickedUp: return "bag.fill.badge.checkmark"
        case .onTheWay: return "car.fill"
        case .arriving: return "mappin.and.ellipse"
        case .delivered: return "checkmark.circle.fill"
        }
    }
}
// Expected behavior: The Lock Screen displays the restaurant name,
// status, and estimated time with a progress bar
```

## Designing the Three Dynamic Island Display Modes

Dynamic Island has three display modes that adapt based on user interaction.

**Compact**: The default view when your app is in the background. It occupies the narrow pill-shaped area split into left and right sections. In the code above, this is defined by `compactLeading` and `compactTrailing`.

**Expanded**: Shown when the user long-presses the Dynamic Island. This larger area supports four regions: `leading`, `trailing`, `center`, and `bottom`.

**Minimal**: Displayed when multiple Live Activities are active simultaneously. You only have space for a single icon, so show only the most critical piece of information.

Apple's Human Interface Guidelines recommend the following design principles:

- In Compact mode, place a brand icon on the left and the most important value on the right
- In Expanded mode, create a layout where users can grasp the situation at a glance
- Keep animations subtle and prioritize information legibility
- Ensure good visibility in both dark and light modes

## Starting, Updating, and Ending Live Activities from Your App

Once the Widget Extension UI is defined, you control the Live Activity lifecycle from your main app code.

```swift
// DeliveryManager.swift
import ActivityKit

class DeliveryManager: ObservableObject {
    @Published var currentActivity: Activity<DeliveryAttributes>?

    // Start a Live Activity
    func startTracking(restaurant: String, orderNumber: String, driver: String) {
        // Check if the device supports Live Activities
        guard ActivityAuthorizationInfo().areActivitiesEnabled else {
            print("Live Activities are disabled")
            return
        }

        let attributes = DeliveryAttributes(
            restaurantName: restaurant,
            orderNumber: orderNumber
        )
        let initialState = DeliveryAttributes.ContentState(
            status: .preparing,
            estimatedMinutes: 30,
            driverName: driver
        )

        do {
            let activity = try Activity.request(
                attributes: attributes,
                content: .init(state: initialState, staleDate: nil),
                pushType: .token  // Enable push notification updates
            )
            currentActivity = activity
            print("Live Activity started: \(activity.id)")
        } catch {
            print("Failed to start Live Activity: \(error)")
        }
    }

    // Update the Live Activity
    func updateStatus(newStatus: DeliveryStatus, minutes: Int, driver: String) async {
        guard let activity = currentActivity else { return }

        let updatedState = DeliveryAttributes.ContentState(
            status: newStatus,
            estimatedMinutes: minutes,
            driverName: driver
        )

        await activity.update(
            ActivityContent(state: updatedState, staleDate: nil)
        )
    }

    // End the Live Activity
    func completeDelivery() async {
        guard let activity = currentActivity else { return }

        let finalState = DeliveryAttributes.ContentState(
            status: .delivered,
            estimatedMinutes: 0,
            driverName: ""
        )

        await activity.end(
            ActivityContent(state: finalState, staleDate: nil),
            dismissalPolicy: .after(.now + 300)  // Auto-dismiss after 5 minutes
        )
        currentActivity = nil
    }
}
// Expected behavior:
// startTracking() → Delivery info appears on Lock Screen and Dynamic Island
// updateStatus() → Display updates in real time
// completeDelivery() → Shows "Delivered", auto-dismisses after 5 minutes
```

Setting `pushType: .token` enables remote updates via APNs (Apple Push Notification service), allowing your server to update the Live Activity even when the app is in the background.

## Setting Up Remote Updates via Push Notifications

To update Live Activities when the user isn't actively using your app, you need APNs-based push updates. Here's how to handle push tokens on the client side:

```swift
// Observe push token updates (app side)
func observePushToken() {
    guard let activity = currentActivity else { return }

    Task {
        for await tokenData in activity.pushTokenUpdates {
            let token = tokenData.map { String(format: "%02x", $0) }.joined()
            print("Push Token: \(token)")
            // Send this token to your backend server
            await sendTokenToServer(token: token, activityId: activity.id)
        }
    }
}

private func sendTokenToServer(token: String, activityId: String) async {
    // Register the token with your server
    let payload: [String: Any] = [
        "activity_id": activityId,
        "push_token": token
    ]
    // Replace with your actual server endpoint
    // POST https://your-api.example.com/register-token
    print("Registering token with server: \(payload)")
}
// Expected behavior: After starting a Live Activity, the APNs token
// is generated and automatically sent to the server
```

On the server side, use the received token to send update requests to APNs. With Rork Max and Supabase Edge Functions, you can generate the backend code as well.

The APNs payload format for updates looks like this:

```json
{
  "aps": {
    "timestamp": 1711756800,
    "event": "update",
    "content-state": {
      "status": "onTheWay",
      "estimatedMinutes": 12,
      "driverName": "John Smith"
    }
  }
}
```

Make sure to set the headers `apns-push-type: liveactivity` and `apns-topic: {BundleID}.push-type.liveactivity`.

## Common Errors and How to Fix Them

There are several common pitfalls when implementing Live Activities.

**Missing `NSSupportsLiveActivities` flag**: Without adding `NSSupportsLiveActivities = YES` to your Widget Extension's `Info.plist`, `Activity.request()` will report that activities are disabled. When prompting Rork Max, explicitly state "Add the Live Activities support flag to Info.plist."

**Mismatched ActivityAttributes between Widget and App**: The ActivityAttributes definition must be shared between both the app target and the Widget Extension target. Either create a shared framework target or add the same file to both targets.

**Dynamic Island not showing in Simulator**: Dynamic Island only appears in simulators for iPhone 14 Pro and later models. Select the appropriate device in Xcode's simulator settings, or use the Rork Companion app to test on a physical device.

**Update frequency limits**: Apple throttles Live Activity updates to a certain number per hour. Overly frequent updates will be rate-limited by the system, so design your implementation to only push significant state changes.

## FAQ

**Q: Can Live Activities work on iPhones without Dynamic Island?**

Yes. Live Activities primarily display on the Lock Screen, which works on any iPhone running iOS 16.1 or later, regardless of whether it has Dynamic Island hardware. On devices without Dynamic Island, the Lock Screen view is the only display — the system gracefully falls back.

**Q: How should I prompt Rork Max to generate Live Activities code?**

Be specific about your use case and required display modes. For example: "Implement food delivery tracking Live Activities using ActivityKit. Support all three Dynamic Island modes: Compact, Expanded, and Minimal." Including your data model fields (the types of information you want to display) helps Rork Max generate more complete code.

**Q: How long can a Live Activity remain visible?**

According to Apple's specifications, a Live Activity can stay active for up to 8 hours. After that, it's automatically terminated. You can also set a `staleDate` to indicate when the information might be outdated. When ending an activity programmatically with `activity.end()`, you can control the dismissal timing using `dismissalPolicy`.
