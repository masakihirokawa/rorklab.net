One day, the API error rate in my production app suddenly crossed 5%.

The culprit was an upstream API that had become unstable. Timed-out requests were piling up, the app was throwing errors continuously, and users were staring at an endless loading spinner. If I had implemented a Circuit Breaker at the time, the app could have auto-recovered within ten minutes.

When you are building apps solo, error handling in API communication tends to get postponed. You ship with a simple try/catch around fetch, and only after a real production incident do you realize: this needed more thought from the start.

This article shows you how to implement production-grade API communication in Rork Max, with working code. By combining Circuit Breaker, Retry, and Exponential Backoff, you can build a system that auto-recovers when upstream APIs become unstable — one that users may never even notice is handling failures on their behalf.

## Why Simple fetch Is Not Enough in Production

The API calls that Rork generates are straightforward fetch calls by default.

```typescript
// Typical API call generated by Rork
const fetchUser = async (userId: string) => {
  const response = await fetch(`https://api.example.com/users/${userId}`);
  if (\!response.ok) throw new Error("Failed to fetch user");
  return response.json();
};
```

This works fine in development. But production is a different world. The problems you will encounter fall into three categories.

**Transient failures**: Mobile apps switch between WiFi and 4G, go through tunnels, and get used in elevators. Momentary connection drops are routine — and most of them would resolve with a single retry.

**Cascading failures**: When an upstream API slows down, requests waiting for a timeout start piling up, gradually choking your app. Without a Circuit Breaker, your app keeps hammering an already-broken API, making things worse.

**Infinite waiting**: The default fetch has no built-in timeout. When an API does not respond, users can watch a spinner for ten, twenty, even thirty seconds.

These three problems are what Circuit Breaker, Retry, and Exponential Backoff are designed to solve.

## The Circuit Breaker Pattern: Stopping Failure Propagation

The Circuit Breaker — named after the electrical device — monitors for API failures. When failures exceed a threshold, it "trips" to cut off requests, then automatically tests for recovery after a waiting period.

There are three states: **Closed (normal)** — requests pass through. **Open (tripped)** — requests are immediately rejected with an error. **Half-Open (testing)** — one request is allowed through to check whether the API has recovered.

```typescript
interface CircuitBreakerOptions {
  failureThreshold: number;
  recoveryTimeout: number;
  successThreshold: number;
}

type CircuitState = "CLOSED" | "OPEN" | "HALF_OPEN";

class CircuitBreaker {
  private state: CircuitState = "CLOSED";
  private failureCount = 0;
  private successCount = 0;
  private lastFailureTime?: Date;
  private options: CircuitBreakerOptions;

  constructor(options: CircuitBreakerOptions) {
    this.options = options;
  }

  async execute<T>(fn: () => Promise<T>): Promise<T> {
    if (this.state === "OPEN") {
      const now = new Date();
      const elapsed = this.lastFailureTime
        ? (now.getTime() - this.lastFailureTime.getTime()) / 1000
        : Infinity;

      if (elapsed >= this.options.recoveryTimeout) {
        this.state = "HALF_OPEN";
        console.log("Circuit Breaker: OPEN -> HALF_OPEN");
      } else {
        throw new Error(
          `Circuit is OPEN. Retry after ${
            Math.ceil(this.options.recoveryTimeout - elapsed)
          }s`
        );
      }
    }

    try {
      const result = await fn();

      if (this.state === "HALF_OPEN") {
        this.successCount++;
        if (this.successCount >= this.options.successThreshold) {
          this.reset();
          console.log("Circuit Breaker: HALF_OPEN -> CLOSED (recovered)");
        }
      } else {
        this.failureCount = 0;
      }

      return result;
    } catch (error) {
      this.recordFailure();
      throw error;
    }
  }

  private recordFailure() {
    this.failureCount++;
    this.lastFailureTime = new Date();
    if (this.failureCount >= this.options.failureThreshold) {
      this.state = "OPEN";
      console.log(
        `Circuit Breaker: CLOSED -> OPEN (${this.failureCount} failures)`
      );
    }
  }

  private reset() {
    this.state = "CLOSED";
    this.failureCount = 0;
    this.successCount = 0;
  }

  getState(): CircuitState {
    return this.state;
  }
}

// Usage
const userApiBreaker = new CircuitBreaker({
  failureThreshold: 5,    // Trip after 5 failures
  recoveryTimeout: 30,    // Attempt recovery after 30 seconds
  successThreshold: 2,    // Reset to CLOSED after 2 successes
});
```

When setting thresholds, my starting point is **failureThreshold: 5, recoveryTimeout: 30 seconds**, then I adjust based on real error logs. Setting the threshold too aggressively means the Circuit trips on routine network blips, which is the wrong trade-off.

## Retry with Exponential Backoff: A Smarter Retry Strategy

A naive immediate retry can actually make things worse. If an API is down under heavy load and every client retries simultaneously, you pile even more load onto an already-struggling server. This is the "thundering herd problem."

Exponential Backoff increases the wait time between retries exponentially. Adding Jitter — a random offset — staggers retries across clients so they don't all hit the server at the same moment.

```typescript
interface RetryOptions {
  maxRetries: number;
  baseDelay: number;
  maxDelay: number;
  jitterFactor: number;
  retryableStatusCodes: number[];
}

const DEFAULT_RETRY_OPTIONS: RetryOptions = {
  maxRetries: 3,
  baseDelay: 1000,
  maxDelay: 30000,
  jitterFactor: 0.3,
  retryableStatusCodes: [408, 429, 500, 502, 503, 504],
};

class ApiError extends Error {
  constructor(
    message: string,
    public statusCode: number,
    public response?: unknown
  ) {
    super(message);
    this.name = "ApiError";
  }
}

const sleep = (ms: number) => new Promise((resolve) => setTimeout(resolve, ms));

async function withRetry<T>(
  fn: () => Promise<T>,
  options: Partial<RetryOptions> = {}
): Promise<T> {
  const opts = { ...DEFAULT_RETRY_OPTIONS, ...options };
  let lastError: Error | undefined;

  for (let attempt = 0; attempt <= opts.maxRetries; attempt++) {
    try {
      return await fn();
    } catch (error) {
      lastError = error as Error;

      if (error instanceof ApiError) {
        if (\!opts.retryableStatusCodes.includes(error.statusCode)) {
          throw error; // Non-retryable: fail immediately
        }
      }

      if (attempt === opts.maxRetries) break;

      // Exponential Backoff + Jitter
      const exponentialDelay = Math.min(
        opts.baseDelay * Math.pow(2, attempt),
        opts.maxDelay
      );
      const jitter =
        exponentialDelay * opts.jitterFactor * (Math.random() * 2 - 1);
      const delay = Math.max(0, exponentialDelay + jitter);

      console.log(
        `Retry attempt ${attempt + 1}/${opts.maxRetries} after ${Math.round(delay)}ms`
      );
      await sleep(delay);
    }
  }

  throw lastError ?? new Error("Retry failed");
}
```

Distinguishing which errors are worth retrying is critical. Retrying a `400 Bad Request` (malformed request) or a `401 Unauthorized` (bad credentials) is pointless. Retrying only makes sense for transient conditions like `503 Service Unavailable` or `408 Request Timeout`.

## Timeout Management: Solving the Infinite Wait Problem

fetch has no built-in timeout. You need to implement it with `AbortController`.

```typescript
interface FetchWithTimeoutOptions extends RequestInit {
  timeoutMs: number;
}

async function fetchWithTimeout(
  url: string,
  options: FetchWithTimeoutOptions
): Promise<Response> {
  const { timeoutMs, ...fetchOptions } = options;
  const controller = new AbortController();

  const timeoutId = setTimeout(() => {
    controller.abort(`Request timed out after ${timeoutMs}ms`);
  }, timeoutMs);

  try {
    const response = await fetch(url, {
      ...fetchOptions,
      signal: controller.signal,
    });

    clearTimeout(timeoutId);

    if (\!response.ok) {
      throw new ApiError(
        `HTTP ${response.status}: ${response.statusText}`,
        response.status,
        await response.json().catch(() => null)
      );
    }

    return response;
  } catch (error) {
    clearTimeout(timeoutId);

    if (error instanceof Error && error.name === "AbortError") {
      throw new ApiError(`Request timed out after ${timeoutMs}ms`, 408);
    }

    throw error;
  }
}
```

Timeout values should vary by use case. In practice: 5–8 seconds for list endpoints, 8–12 seconds for detail views and user-initiated actions, 30–60 seconds for file uploads and heavy processing. Set timeouts too aggressively and you will generate errors for users on slow connections — not a good look.

## Combining All Three: The Production API Client

Here is a production-ready API client that integrates Circuit Breaker, Retry, and Timeout.

```typescript
interface ApiClientOptions {
  baseUrl: string;
  defaultTimeout: number;
  circuitBreaker: CircuitBreakerOptions;
  retry: Partial<RetryOptions>;
}

class ResilientApiClient {
  private breakers: Map<string, CircuitBreaker> = new Map();
  private options: ApiClientOptions;

  constructor(options: ApiClientOptions) {
    this.options = options;
  }

  private getBreaker(endpoint: string): CircuitBreaker {
    // Each endpoint gets its own independent Circuit Breaker
    if (\!this.breakers.has(endpoint)) {
      this.breakers.set(
        endpoint,
        new CircuitBreaker(this.options.circuitBreaker)
      );
    }
    return this.breakers.get(endpoint)\!;
  }

  async request<T>(
    path: string,
    options: Omit<FetchWithTimeoutOptions, "timeoutMs"> & {
      timeoutMs?: number;
    } = {}
  ): Promise<T> {
    const url = `${this.options.baseUrl}${path}`;
    const timeoutMs = options.timeoutMs ?? this.options.defaultTimeout;
    const breaker = this.getBreaker(path);

    // Apply: Circuit Breaker -> Retry -> Timeout
    return breaker.execute(() =>
      withRetry(
        () =>
          fetchWithTimeout(url, {
            ...options,
            timeoutMs,
          }).then((res) => res.json() as Promise<T>),
        this.options.retry
      )
    );
  }

  getBreakerStates(): Record<string, CircuitState> {
    const states: Record<string, CircuitState> = {};
    this.breakers.forEach((breaker, endpoint) => {
      states[endpoint] = breaker.getState();
    });
    return states;
  }
}

// App-wide singleton — create once, use everywhere
export const apiClient = new ResilientApiClient({
  baseUrl: "https://api.yourapp.com",
  defaultTimeout: 8000,
  circuitBreaker: {
    failureThreshold: 5,
    recoveryTimeout: 30,
    successThreshold: 2,
  },
  retry: {
    maxRetries: 3,
    baseDelay: 1000,
    maxDelay: 15000,
  },
});

// Using it — Circuit Breaker + Retry + Timeout all handled
const user = await apiClient.request<User>("/users/me");
const posts = await apiClient.request<Post[]>("/posts", {
  timeoutMs: 5000, // Override timeout for this endpoint
});
```

The key design decision here is giving each endpoint its own independent Circuit Breaker. When the user profile API becomes unstable, the posts feed API should continue to work. Sharing a single breaker across all endpoints would cause one failing service to take down everything.

## Integrating with TanStack Query

Many Rork Max apps use TanStack Query for data fetching. Combining it with the resilient API client gives you declarative, clean error handling.

```typescript
import { useQuery } from "@tanstack/react-query";

export function useUser(userId: string) {
  return useQuery({
    queryKey: ["user", userId],
    queryFn: () => apiClient.request<User>(`/users/${userId}`),
    retry: (failureCount, error) => {
      // Do not retry when the Circuit is OPEN — wait for auto-recovery
      if (
        error instanceof Error &&
        error.message.startsWith("Circuit is OPEN")
      ) {
        return false;
      }
      return failureCount < 1;
    },
    staleTime: 5 * 60 * 1000,  // Serve from cache for 5 minutes
    gcTime: 30 * 60 * 1000,
  });
}

function UserProfileScreen({ userId }: { userId: string }) {
  const { data: user, error, isLoading, isError, refetch } = useUser(userId);

  if (isLoading) return <LoadingSpinner />;

  if (isError) {
    const isCircuitOpen = error?.message?.startsWith("Circuit is OPEN");
    return (
      <ErrorView
        message={
          isCircuitOpen
            ? "The server is currently busy. Please try again shortly."
            : "Failed to load data."
        }
        // Hide retry button while Circuit is OPEN — auto-recovery is already in progress
        onRetry={isCircuitOpen ? undefined : refetch}
      />
    );
  }

  return <UserProfile user={user\!} />;
}
```

Watch out for retry stacking: if TanStack Query retries 3 times and the API client also retries 3 times, a single failure can trigger up to 9 requests. My preference is to keep retry logic in the API client and set TanStack Query's retry to at most 1 — or disable it entirely. You can find more on data fetching patterns in our [TanStack Query data fetching guide](/en/articles/rork-dev/rork-tanstack-query-data-fetching-cache-guide).

## Common Mistakes and Pitfalls

Here are three mistakes I see repeatedly in production codebases.

**Pitfall 1: Retrying all errors indiscriminately**

```typescript
// Wrong: retries everything, including errors that will never resolve
async function badRetry<T>(fn: () => Promise<T>) {
  for (let i = 0; i < 3; i++) {
    try {
      return await fn();
    } catch {
      await sleep(1000);
    }
  }
}

// Right: only retry transient server-side errors
async function goodRetry<T>(fn: () => Promise<T>) {
  return withRetry(fn, {
    retryableStatusCodes: [408, 429, 500, 502, 503, 504],
    // 400, 401, 403, 404 are not retried
  });
}
```

Retrying a `401 Unauthorized` three times delays the redirect to the login page by three full seconds — a silent UX failure that is easy to miss in development.

**Pitfall 2: Creating Circuit Breakers inside components**

Circuit Breakers must be shared across the app. A new instance per component means the failure counter resets every time the component unmounts.

```typescript
// Wrong: state is lost on unmount
function MyComponent() {
  const breaker = new CircuitBreaker({ failureThreshold: 5 });
  ...
}

// Right: use the app-wide singleton
import { apiClient } from "./api-client";
function MyComponent() {
  apiClient.request("/endpoint");
  ...
}
```

**Pitfall 3: Forgetting to cancel in-flight requests on unmount**

Even after a component unmounts, a pending fetch will keep waiting for its timeout. Clean up with `useEffect`'s return function.

```typescript
useEffect(() => {
  const controller = new AbortController();

  apiClient
    .request("/data", { signal: controller.signal })
    .then(setData)
    .catch((err) => {
      if (err.name \!== "AbortError") setError(err);
    });

  return () => controller.abort(); // Cancel on unmount
}, []);
```

## Observability: Making Resilience Visible

Once the resilience infrastructure is in place, you need a way to confirm it is actually working. Logging Circuit Breaker state transitions gives you a window into your app's real-world behavior.

```typescript
class MonitoredCircuitBreaker extends CircuitBreaker {
  private readonly name: string;
  private onStateChange?: (from: CircuitState, to: CircuitState) => void;

  constructor(
    name: string,
    options: CircuitBreakerOptions,
    onStateChange?: (from: CircuitState, to: CircuitState) => void
  ) {
    super(options);
    this.name = name;
    this.onStateChange = onStateChange;
  }

  async execute<T>(fn: () => Promise<T>): Promise<T> {
    const previousState = this.getState();
    try {
      const result = await super.execute(fn);
      const currentState = this.getState();

      if (previousState \!== currentState) {
        console.log(`[CB:${this.name}] ${previousState} -> ${currentState}`);
        this.onStateChange?.(previousState, currentState);
      }

      return result;
    } catch (error) {
      const currentState = this.getState();
      if (previousState \!== currentState) {
        console.warn(
          `[CB:${this.name}] ${previousState} -> ${currentState} (failure)`
        );
        this.onStateChange?.(previousState, currentState);
      }
      throw error;
    }
  }
}

// Wiring up Sentry alerts on Circuit state changes
const userApiBreaker = new MonitoredCircuitBreaker(
  "user-api",
  { failureThreshold: 5, recoveryTimeout: 30, successThreshold: 2 },
  (from, to) => {
    if (to === "OPEN") {
      Sentry.captureMessage(`Circuit Breaker opened: user-api`, {
        level: "warning",
        extra: { from, to },
      });
    }
  }
);
```

## A Phased Rollout Roadmap

You do not need to implement everything at once. Introduce these patterns in priority order.

**Step 1 — Today: Add Timeouts**

Apply an 8-second timeout to every API call. This alone eliminates most "infinite loading" reports. Swap one important fetch call for `fetchWithTimeout` and deploy.

**Step 2 — This Week: Add Retry + Exponential Backoff**

Add retry logic around the timeout. At this point you will likely see an immediate reduction in user-reported errors. One `withRetry` wrapper around `fetchWithTimeout` is all it takes.

**Step 3 — This Month: Add Circuit Breakers**

Wire up `ResilientApiClient` as a singleton and gradually replace existing fetch calls. Now your app can survive upstream API instability without user intervention.

Once all three are in place, your API communication is production-grade. The fear of shipping because "what if the API goes down?" starts to fade, replaced by the confidence that your app will handle it gracefully.

Start today with one timeout. That is all.
