## Introduction

One of the most common requests when building business-focused mobile apps is the ability to generate PDF documents — invoices, quotes, work reports, health summaries, and more. If you're building with Rork, you'll be happy to know this is entirely achievable using `expo-print` combined with `expo-sharing`.

In this guide, we'll walk through the entire process of building a PDF generation feature from scratch: creating an HTML template, generating the PDF, and sharing it via the native share sheet on both iOS and Android. If you're new to working with external APIs in Rork, you may also find [Rork's Practical Guide to External API Integration](/articles/rork-dev/rork-external-api-integration-guide) helpful as a starting point.

This article is aimed at developers with some experience in Rork app development who are comfortable with TypeScript and basic React Native patterns.

---

## How PDF Generation Works in Rork

### The Role of expo-print

Expo (the runtime that powers Rork apps) provides `expo-print`, a library that converts HTML strings into PDF files. Here's the basic flow:

1. Build an HTML document as a JavaScript string (with dynamic data injected via template literals)
2. Pass it to `Print.printToFileAsync()` to generate a `.pdf` file on the device
3. Use `expo-sharing`'s `shareAsync()` to display the native share sheet — letting users send the PDF via email, AirDrop, Dropbox, or any other app

The biggest advantage of this approach is that **you design PDFs using HTML and CSS** — the same skills you likely already have. No special PDF formatting library or proprietary template engine required.

### Required Packages

You'll need the following packages in your Rork project. Simply tell Rork: "Add expo-print, expo-sharing, and expo-file-system to the project" and it will handle the installation.

```
expo-print
expo-sharing
expo-file-system (optional, for inspecting or deleting generated files)
```

## Step-by-Step: Building an Invoice PDF App

### Step 1: Create the HTML Invoice Template

Start by defining an HTML template as a TypeScript function. This gives you full control over styling and layout, and makes it easy to inject dynamic data.

```typescript
// lib/invoiceTemplate.ts

interface InvoiceData {
  invoiceNumber: string;
  date: string;
  dueDate: string;
  companyName: string;
  companyAddress: string;
  clientName: string;
  clientAddress: string;
  items: { description: string; quantity: number; unitPrice: number }[];
  taxRate: number;
  note?: string;
}

export function generateInvoiceHTML(data: InvoiceData): string {
  const subtotal = data.items.reduce(
    (sum, item) => sum + item.quantity * item.unitPrice,
    0
  );
  const tax = Math.round(subtotal * data.taxRate);
  const total = subtotal + tax;

  const itemRows = data.items
    .map(
      (item) => `
      <tr>
        <td style="padding: 10px; border-bottom: 1px solid #eee;">${item.description}</td>
        <td style="padding: 10px; border-bottom: 1px solid #eee; text-align: center;">${item.quantity}</td>
        <td style="padding: 10px; border-bottom: 1px solid #eee; text-align: right;">$${item.unitPrice.toLocaleString()}</td>
        <td style="padding: 10px; border-bottom: 1px solid #eee; text-align: right;">$${(item.quantity * item.unitPrice).toLocaleString()}</td>
      </tr>
    `
    )
    .join("");

  return `
    <!DOCTYPE html>
    <html lang="en">
    <head>
      <meta charset="UTF-8" />
      <style>
        body { font-family: 'Helvetica Neue', Arial, sans-serif; margin: 40px; color: #333; }
        h1 { color: #2563eb; font-size: 28px; margin-bottom: 4px; }
        .meta { display: flex; justify-content: space-between; margin-bottom: 30px; }
        .meta-block p { margin: 2px 0; font-size: 13px; }
        table { width: 100%; border-collapse: collapse; margin-top: 20px; }
        th { background: #2563eb; color: white; padding: 10px; text-align: left; font-size: 13px; }
        td { font-size: 13px; }
        .total-area { margin-top: 20px; text-align: right; }
        .total-area p { margin: 4px 0; font-size: 14px; }
        .total-area .grand-total { font-size: 18px; font-weight: bold; color: #2563eb; }
        .note { margin-top: 30px; font-size: 12px; color: #666; border-top: 1px solid #ddd; padding-top: 16px; }
      </style>
    </head>
    <body>
      <h1>Invoice</h1>
      <p style="color:#888; font-size:13px;">Invoice No. ${data.invoiceNumber}</p>
      <div class="meta">
        <div class="meta-block">
          <p><strong>${data.companyName}</strong></p>
          <p>${data.companyAddress}</p>
        </div>
        <div class="meta-block" style="text-align:right;">
          <p>Bill To: <strong>${data.clientName}</strong></p>
          <p>${data.clientAddress}</p>
          <p>Issue Date: ${data.date}</p>
          <p>Due Date: ${data.dueDate}</p>
        </div>
      </div>

      <table>
        <thead>
          <tr>
            <th>Item / Service</th>
            <th style="text-align:center; width:80px;">Qty</th>
            <th style="text-align:right; width:120px;">Unit Price</th>
            <th style="text-align:right; width:120px;">Amount</th>
          </tr>
        </thead>
        <tbody>${itemRows}</tbody>
      </table>

      <div class="total-area">
        <p>Subtotal: $${subtotal.toLocaleString()}</p>
        <p>Tax (${data.taxRate * 100}%): $${tax.toLocaleString()}</p>
        <p class="grand-total">Total: $${total.toLocaleString()}</p>
      </div>

      ${data.note ? `<div class="note"><strong>Note:</strong> ${data.note}</div>` : ""}
    </body>
    </html>
  `;
}
```

All CSS should be inline or placed inside a `<style>` tag — external stylesheets won't be loaded in the PDF rendering context.

### Step 2: PDF Generation and Sharing Logic

Next, wire up the actual PDF generation and sharing using `expo-print` and `expo-sharing`.

```typescript
// lib/pdfService.ts
import * as Print from "expo-print";
import * as Sharing from "expo-sharing";

export async function generateAndSharePDF(html: string, filename: string) {
  try {
    // Convert HTML to a PDF file
    const { uri } = await Print.printToFileAsync({
      html,
      base64: false,
    });

    // Expected output: uri = "file:///var/mobile/.../tmp/xxx.pdf"
    console.log("PDF generated at:", uri);

    // Check if sharing is supported on this device
    const isAvailable = await Sharing.isAvailableAsync();
    if (!isAvailable) {
      console.warn("Sharing is not available on this device");
      return;
    }

    // Open the native share sheet
    await Sharing.shareAsync(uri, {
      mimeType: "application/pdf",
      dialogTitle: `Share ${filename}`,
      UTI: "com.adobe.pdf", // iOS-specific type identifier
    });
  } catch (error) {
    console.error("PDF generation error:", error);
    throw error;
  }
}
```

The file generated by `printToFileAsync()` is saved in the device's temporary directory. Once `shareAsync()` is called, the OS-native share sheet appears — on iOS this includes AirDrop, Mail, and third-party apps; on Android you'll see the standard share dialog.

## Common Errors and How to Fix Them

### Issue 1: Text Appears as Boxes (Encoding Problem)

**Cause**: The PDF rendering engine uses the device's system fonts, and certain characters (especially non-Latin scripts) may not render correctly.

**Fix**: Use a web font loaded via a `@import` URL, or embed the font as Base64 in the HTML. For example:

```html
<style>
  @import url("https://fonts.googleapis.com/css2?family=Noto+Sans+JP&display=swap");
  body {
    font-family: "Noto Sans JP", sans-serif;
  }
</style>
```

Note that web fonts require an active internet connection. For offline apps, Base64-embed the font file directly.

---

### Issue 2: `printToFileAsync` Crashes in Expo Go

**Cause**: `expo-print` has limited functionality in the Expo Go sandbox environment.

**Fix**: Test PDF generation on a real device using EAS Build or Expo Dev Client. Do not rely on Rork's simulator preview for testing print features.

---

### Issue 3: Page Breaks Ruin the Layout

**Cause**: Long tables or content blocks may split awkwardly across pages.

**Fix**: Use CSS page-break properties to prevent splits within important elements:

```css
.no-break {
  page-break-inside: avoid;
}
tr {
  page-break-inside: avoid;
}
```

## FAQ

**Q1: Does expo-print work on both iOS and Android?**

A: Yes, `expo-print` supports both platforms. Note that `Print.printAsync()` (which opens the system print dialog) is iOS-only, but `printToFileAsync()` (used to generate a PDF file) works on both iOS and Android without any issues.

**Q2: Can I control the PDF page size (e.g., A4)?**

A: Yes. Pass `width` and `height` to `printToFileAsync()` in points. For A4, use `{ width: 595, height: 842 }`. For US Letter, use `{ width: 612, height: 792 }`.

**Q3: How do I embed a logo image in the PDF?**

A: The most reliable approach is to encode the image as Base64 and use it as a data URL in the `<img>` tag. You can use `expo-file-system` to read the image file and convert it:

```typescript
import * as FileSystem from "expo-file-system";
const base64 = await FileSystem.readAsStringAsync(logoUri, {
  encoding: FileSystem.EncodingType.Base64,
});
const src = `data:image/png;base64,${base64}`;
// Then use: <img src="${src}" width="120" />
```

**Q4: Can I add page numbers to the PDF?**

A: CSS `@page` rules have limited support in expo-print's rendering engine. A reliable workaround is to calculate the total number of pages programmatically and inject page number HTML into a footer element in the template before generating the PDF.

**Q5: Can I show a preview of the PDF before generating it?**

A: Yes! Use `expo-web-view` to render the HTML as a preview screen, and then trigger `printToFileAsync()` only when the user confirms. This is a great UX pattern that avoids unnecessary file generation.

## Summary

In this guide, we covered how to build a fully functional PDF generation and sharing feature in your Rork app:

- Use `expo-print`'s `printToFileAsync()` to convert HTML into a PDF file
- Use `expo-sharing`'s `shareAsync()` to open the native share sheet on both iOS and Android
- Design your PDF layout with plain HTML and CSS — no proprietary template engine needed
- Handle common issues like font encoding, Expo Go limitations, and page break control
