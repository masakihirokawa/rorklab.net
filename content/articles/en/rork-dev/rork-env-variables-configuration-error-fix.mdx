## When Environment Variables Return Undefined in Rork Apps

You've carefully added your API keys and backend URLs to a `.env` file, but when your Rork-generated app tries to read them, they come back as `undefined`. This is one of the most common and frustrating issues developers encounter when working with Expo and React Native — and it almost always comes down to how the platform handles environment variables differently from what you might expect.

This guide covers every major cause of environment variable failures in Rork apps and walks you through fixing each one.

## Common Causes of Environment Variable Errors

There are five main reasons environment variables fail to load in Rork apps built on Expo and React Native.

**1. Missing `EXPO_PUBLIC_` Prefix**

Since Expo SDK 49, any environment variable you want to access on the client side must start with `EXPO_PUBLIC_`. Variables without this prefix are intentionally excluded from the app bundle for security reasons.

**2. Incorrect `.env` File Location**

The `.env` file needs to be in the project root directory. Placing it in a subdirectory like `src/` or `config/` means it won't be picked up by the bundler.

**3. Stale Metro Bundler Cache**

Metro caches environment variable values at build time. Changing your `.env` file without clearing the cache means the app continues using the old values.

**4. Confusing Build-time and Runtime Variables**

EAS Build environment variables (set in `eas.json`) and local `.env` files operate through different mechanisms. Understanding when each is used prevents a whole class of configuration bugs.

**5. EAS Secrets Misconfiguration**

Secrets registered with EAS may not be properly injected into the build process if they're scoped incorrectly or named wrong.

## Step-by-Step Fix

### Step 1: Verify Your `.env` File

Start by confirming your `.env` file is in the right place with the correct format.

```bash
# Make sure you're in the project root
ls -la .env*

# Check the contents
cat .env
```

Here's what a properly formatted `.env` file looks like:

```bash
# ✅ Correct: client-side variables with EXPO_PUBLIC_ prefix
EXPO_PUBLIC_API_URL=https://api.example.com
EXPO_PUBLIC_SUPABASE_URL=https://your-project.supabase.co
EXPO_PUBLIC_SUPABASE_ANON_KEY=your_anon_key_here

# ✅ Correct: server-side / build-only variables (no prefix needed)
SENTRY_AUTH_TOKEN=your_sentry_token
EAS_PROJECT_ID=your_project_id
```

Watch out for these common mistakes:

```bash
# ❌ Wrong: missing EXPO_PUBLIC_ prefix
API_URL=https://api.example.com
SUPABASE_URL=https://your-project.supabase.co

# ❌ Wrong: values wrapped in quotes
EXPO_PUBLIC_API_URL="https://api.example.com"

# ❌ Wrong: spaces around the equals sign
EXPO_PUBLIC_API_URL = https://api.example.com
```

### Step 2: Check How You're Accessing Variables

In Expo, environment variables are accessed through `process.env`:

```typescript
// ✅ Correct approach
const apiUrl = process.env.EXPO_PUBLIC_API_URL;
const supabaseUrl = process.env.EXPO_PUBLIC_SUPABASE_URL;

// Debug: check values are loaded
console.log('API URL:', process.env.EXPO_PUBLIC_API_URL);
console.log('Supabase URL:', process.env.EXPO_PUBLIC_SUPABASE_URL);
```

If you're migrating from an older setup, you might be using the legacy approach through `expo-constants`. The `process.env` method is now recommended:

```typescript
// ⚠️ Legacy approach (still works but not recommended)
import Constants from 'expo-constants';
const apiUrl = Constants.expoConfig?.extra?.apiUrl;

// ✅ Modern approach (recommended)
const apiUrl = process.env.EXPO_PUBLIC_API_URL;
```

### Step 3: Clear the Metro Cache

After modifying your `.env` file, you must clear the Metro bundler cache:

```bash
# Clear cache and restart
npx expo start --clear

# For a more thorough cleanup
rm -rf node_modules/.cache
npx expo start --clear
```

Running `npx expo start` without the `--clear` flag after changing environment variables is one of the most common reasons values appear stale. Environment variables are statically embedded at build time, so the bundler needs to reprocess them.

### Step 4: Configure EAS Build Environment Variables

For EAS Build (cloud builds), set environment variables in your `eas.json`:

```json
{
  "build": {
    "development": {
      "env": {
        "EXPO_PUBLIC_API_URL": "https://dev-api.example.com",
        "EXPO_PUBLIC_SUPABASE_URL": "https://your-project.supabase.co"
      }
    },
    "preview": {
      "env": {
        "EXPO_PUBLIC_API_URL": "https://staging-api.example.com",
        "EXPO_PUBLIC_SUPABASE_URL": "https://your-project.supabase.co"
      }
    },
    "production": {
      "env": {
        "EXPO_PUBLIC_API_URL": "https://api.example.com",
        "EXPO_PUBLIC_SUPABASE_URL": "https://your-project.supabase.co"
      }
    }
  }
}
```

### Step 5: Manage Secrets with EAS Secrets

For sensitive values like API keys, use EAS Secrets instead of hardcoding them in `eas.json`:

```bash
# Create a secret
eas secret:create --name EXPO_PUBLIC_SUPABASE_ANON_KEY \
  --value "your_anon_key_here" \
  --scope project

# List all secrets
eas secret:list

# Update a secret (delete and recreate)
eas secret:delete --name EXPO_PUBLIC_SUPABASE_ANON_KEY
eas secret:create --name EXPO_PUBLIC_SUPABASE_ANON_KEY \
  --value "new_anon_key_here" \
  --scope project
```

EAS Secrets are automatically injected as environment variables during the build process. The standard workflow is: use `.env` for local development, and EAS Secrets for cloud builds.

## How to Verify Your Fix

Once you've made corrections, verify that everything works with a simple debug screen:

```typescript
// app/debug-env.tsx (temporary debug file)
import { View, Text, ScrollView } from 'react-native';

export default function DebugEnvScreen() {
  const envVars = {
    API_URL: process.env.EXPO_PUBLIC_API_URL,
    SUPABASE_URL: process.env.EXPO_PUBLIC_SUPABASE_URL,
    SUPABASE_ANON_KEY: process.env.EXPO_PUBLIC_SUPABASE_ANON_KEY
      ? '✅ Set (hidden)'
      : '❌ Not set',
  };

  return (
    <ScrollView style={{ padding: 20 }}>
      <Text style={{ fontSize: 20, fontWeight: 'bold', marginBottom: 16 }}>
        Environment Variables Debug
      </Text>
      {Object.entries(envVars).map(([key, value]) => (
        <View key={key} style={{ marginBottom: 12 }}>
          <Text style={{ fontWeight: 'bold' }}>{key}</Text>
          <Text style={{ color: value ? '#22c55e' : '#ef4444' }}>
            {value || 'undefined ❌'}
          </Text>
        </View>
      ))}
    </ScrollView>
  );
}
```

Remember to remove this debug screen before publishing. You don't want to risk exposing configuration details in a production build.

## Preventing Future Issues

Build these habits into your workflow to avoid environment variable problems going forward.

**Include a `.env.example` in Your Repository**

Keep a template with all required variable names (but no real values) so you or your teammates always know what's needed:

```bash
# .env.example (commit this to your repo)
EXPO_PUBLIC_API_URL=
EXPO_PUBLIC_SUPABASE_URL=
EXPO_PUBLIC_SUPABASE_ANON_KEY=
```

**Add `.env` to `.gitignore`**

Never commit actual `.env` files with real credentials:

```bash
# .gitignore
.env
.env.local
.env.production
```

**Create a Type-safe Environment Helper**

In TypeScript projects, wrap your environment access in a utility that warns you about missing values early:

```typescript
// utils/env.ts
function getEnvVar(key: string): string {
  const value = process.env[key];
  if (\!value) {
    console.warn(`⚠️ Environment variable ${key} is not set`);
  }
  return value ?? '';
}

export const ENV = {
  API_URL: getEnvVar('EXPO_PUBLIC_API_URL'),
  SUPABASE_URL: getEnvVar('EXPO_PUBLIC_SUPABASE_URL'),
  SUPABASE_ANON_KEY: getEnvVar('EXPO_PUBLIC_SUPABASE_ANON_KEY'),
} as const;
```

Using this `ENV` object throughout your app prevents typos in variable names and catches missing configuration before it causes runtime errors.

## Frequently Asked Questions

### Q: Can I access environment variables without the `EXPO_PUBLIC_` prefix on the client side?

No — and this is by design. The prefix requirement is a security boundary that prevents server-side secrets (like database passwords) from being bundled into the app, where they could be extracted through reverse engineering. If you need to avoid the prefix, you can pass values through `app.config.js`'s `extra` field, but the `process.env` approach is simpler and recommended.

### Q: Variables work locally but return undefined in EAS Build. What's wrong?

Your local `.env` file doesn't exist in the EAS Build cloud environment. You need to either set the values in `eas.json` under the appropriate build profile's `env` field, or register them as EAS Secrets with `eas secret:create`. For `EXPO_PUBLIC_` variables specifically, `eas.json`'s `env` field is the most reliable approach.

### Q: What's the priority order between `.env.local` and `.env`?

Expo gives `.env.local` the highest priority, followed by `.env`. However, environment-specific files like `.env.production` aren't automatically selected based on the build type — that's not a standard Expo feature. For environment-specific values, use `eas.json` build profiles, where you can set different values for `development`, `preview`, and `production`.

### Q: Is it safe to put Supabase's `service_role_key` in an environment variable?

Only if it stays on the server side. Never use the `EXPO_PUBLIC_` prefix with `service_role_key` — doing so embeds it in the app bundle, making it extractable through reverse engineering. The `service_role_key` grants admin-level access to your database and should only ever be used in server-side code like Supabase Edge Functions or your own API server.

## Summary

Environment variable issues in Rork apps almost always trace back to a missing `EXPO_PUBLIC_` prefix, a stale Metro cache, or confusion between local `.env` files and EAS Build configuration. By following the diagnostic steps in this guide, you can quickly identify and fix the root cause.

While environment variable management might seem like a small detail, getting it right is essential for both security and developer productivity. Adopt the practices outlined here — `.env.example` templates, `.gitignore` rules, and type-safe access helpers — to keep your configuration clean and your builds reliable.
