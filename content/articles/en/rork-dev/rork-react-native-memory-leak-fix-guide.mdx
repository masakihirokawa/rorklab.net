After shipping an app built with Rork, you might start getting user reports that sound something like this: "The app gets slower the longer I use it" or "It crashes when I come back from the background." These symptoms are almost always caused by memory leaks.

AI-generated code tends to get the core logic right, but it doesn't always handle the cleanup side of things — unregistering event listeners, cancelling async operations, unsubscribing from real-time services. "Code that works" and "code that properly releases memory" are two different things. This guide walks through the most common memory leak patterns in Rork-generated code, how to diagnose them with Flipper, and exactly what to change to fix them.

## Why Rork-Generated Code Is Prone to Memory Leaks

Rork generates code in a React Native + Expo setup. React Native handles memory perfectly well when written carefully, but AI-generated code tends to have a few recurring blind spots:

- `useEffect` registers event listeners without returning a cleanup function
- `setInterval` or `setTimeout` is used without clearing the timer on unmount
- Async operations try to update state after the component has been unmounted
- Supabase or Firebase realtime subscriptions are never unsubscribed

None of these are "broken" in the sense that the app won't run — but leave them in place long enough, and memory will quietly pile up.

## Visualizing Memory Usage with Flipper

The first step is confirming that a leak actually exists. Guessing and patching code at random is not a good use of your time.

**Setting Up Flipper (requires macOS)**

```bash
# Download and install Flipper from https://fbflipper.com/
# Connect via a development build, not EAS

npx expo start --dev-client
```

Once Flipper is running and connected to your simulator or device, open the "Memory" tab in the left sidebar. Perform some typical in-app navigation — open a screen, go back, repeat ten times — then stop the recording.

**What to Look For**

In the Memory tab:

- **Heap Size**: If this climbs steadily as you navigate around without ever coming back down, you have a leak
- **Retained Size**: If a particular component has a dramatically larger retained size than others, that's your suspect

A clean, leak-free app should return to roughly its starting heap size after you open and close the same screen ten times. If it doesn't, something is holding on to memory it should have released.

## The Four Most Common Patterns (and How to Fix Them)

### Pattern 1: Missing useEffect Cleanup

This is by far the most frequent issue in Rork-generated code.

```typescript
// ❌ Leaks memory — no cleanup function
useEffect(() => {
  const subscription = someEventEmitter.addListener('update', handleUpdate);
  // This subscription outlives the component
}, []);

// ✅ Fixed — cleanup function returned
useEffect(() => {
  const subscription = someEventEmitter.addListener('update', handleUpdate);
  return () => {
    subscription.remove(); // Runs when the component unmounts
  };
}, []);
```

If you want Rork to fix this for you, the prompt "Add cleanup functions to all useEffect event listeners" works reliably in most cases.

### Pattern 2: Timer Not Cleared on Unmount

This one comes up whenever you're implementing polling or delayed actions.

```typescript
// ❌ Timer keeps firing after the screen is gone
useEffect(() => {
  const timer = setInterval(() => {
    fetchLatestData();
  }, 5000);
  // Nothing stops this timer when the component unmounts
}, []);

// ✅ Fixed
useEffect(() => {
  const timer = setInterval(() => {
    fetchLatestData();
  }, 5000);
  return () => clearInterval(timer); // One line is all it takes
}, []);
```

`setTimeout` follows the same rule: return `() => clearTimeout(timer)` in the cleanup.

### Pattern 3: State Updates After Unmount

If an API call or async operation finishes after the user has already left the screen, the callback will try to write to state in an unmounted component. React 18 stopped showing the console warning for this, but the wasted computation is still real.

```typescript
// ✅ isMounted flag pattern — simple and readable
useEffect(() => {
  let isMounted = true;

  const fetchData = async () => {
    const result = await api.getItems();
    if (isMounted) { // Only update if the component is still alive
      setItems(result);
    }
  };

  fetchData();
  return () => {
    isMounted = false;
  };
}, []);
```

You can also use `AbortController` for this, but the `isMounted` flag is easier to add to existing Rork-generated code without restructuring anything.

### Pattern 4: Supabase Realtime Subscription Not Removed

This one is particularly easy to miss in real-time chat or notification features.

```typescript
// ❌ Subscription stays active after the component unmounts
useEffect(() => {
  const channel = supabase
    .channel('messages')
    .on('postgres_changes', { event: '*', schema: 'public', table: 'messages' }, handleChange)
    .subscribe();
  // No cleanup
}, []);

// ✅ Fixed
useEffect(() => {
  const channel = supabase
    .channel('messages')
    .on('postgres_changes', { event: '*', schema: 'public', table: 'messages' }, handleChange)
    .subscribe();
  return () => {
    supabase.removeChannel(channel); // Fully removes the subscription
  };
}, []);
```

Firebase's `onSnapshot` works the same way: it returns an unsubscribe function that you should call in your cleanup.

## Going Deeper with Xcode Instruments (iOS)

Once you've confirmed a leak with Flipper, Xcode Instruments gives you a more detailed picture on the iOS side. The "Leaks" template shows you exactly which objects are being retained and the full stack trace of where they were allocated.

```bash
# If you're using Rork Max, you have a generated Xcode project
# Open it, then go to Product → Profile → choose the Leaks template
# Operate the app as usual — a red "\!" in the timeline means a leak was detected
```

The learning curve is steeper than Flipper, but when Flipper tells you there's a leak and you still can't find it, Instruments will usually point you straight at the culprit.

## Verifying Your Fixes

After making changes, always verify them with the same method:

1. Connect Flipper
2. Open the affected screen, interact with it, navigate away — repeat 10 times
3. Check whether the Heap Size has returned to its initial level

If it has, you're done. If it's still climbing, there's likely another instance of the same pattern elsewhere. You can ask Rork to scan the entire file: "Check all useEffect hooks in this file and add cleanup functions wherever they're missing."

## Writing Effective Prompts for Rork

If you'd rather have Rork handle the fixes directly, here's a prompt template that works well:

```
Review this file for memory leaks and fix all of the following:
- useEffect hooks that register event listeners without returning a cleanup function
- setInterval or setTimeout calls that aren't cleared on unmount
- Supabase or Firebase subscriptions that aren't unsubscribed
- Async operations that might update state after the component unmounts

After fixing, explain what you changed and why.
```

Combine this prompt with the specific file path (e.g., `src/screens/ChatScreen.tsx`) and Rork will work through each pattern methodically.

## Building the Habit Early

Memory leaks are much easier to prevent than to fix after the fact. A few habits to build into your Rork workflow:

- Whenever you see a `useEffect`, immediately ask yourself whether it needs a cleanup function
- Treat any timer (`setInterval`, `setTimeout`) as automatically requiring a clear
- Assume any external subscription (Supabase, Firebase, WebSocket) needs an explicit unsubscribe
- After changing a screen, run the 10-open-10-close test in Flipper as a baseline check

Apps that handle memory well feel noticeably snappier over long sessions — and users notice. Knowing how to diagnose and fix leaks means that when "the app gets slow" reports start coming in, you have a repeatable process instead of a guessing game.
