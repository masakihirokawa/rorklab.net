## iPad Design Fundamentals

### How iPad Differs from iPhone

The most critical difference when designing for iPad is the shift from single-column to multi-column layouts. While iPhone apps typically use a vertical, single-column navigation flow, iPad apps should leverage the wider display with side-by-side content.

| Aspect | iPhone | iPad |
|--------|--------|------|
| Layout | Single column | 2–3 columns |
| Navigation | NavigationStack (push) | NavigationSplitView (sidebar + detail) |
| Multitasking | Not supported | Split View / Slide Over |
| Input | Touch | Touch + Apple Pencil + Keyboard + Trackpad |
| Drag & Drop | Within app only | Between apps |

### Apple's Key Recommendations

Apple's Human Interface Guidelines emphasize four areas for iPad apps:

1. **Support all screen sizes** — Your app should look great in both portrait and landscape orientations.
2. **Enable multitasking** — Ensure your layout adapts gracefully when used in Split View or Slide Over.
3. **Pointer support** — Add hover effects for trackpad and mouse users.
4. **Keyboard shortcuts** — Provide common shortcuts for external keyboard users.

### Step 2: Implementing NavigationSplitView

NavigationSplitView is the cornerstone of iPad app navigation. It manages a sidebar, content list, and detail view in a single unified interface.

```swift
// NavigationSplitView for iPad-optimized layout
// iPad: Sidebar + list + detail in a three-column layout
// iPhone: Automatically falls back to single-column push navigation

struct ContentView: View {
    @State private var selectedCategory: Category?
    @State private var selectedNote: Note?
    @State private var columnVisibility: NavigationSplitViewVisibility = .all

    var body: some View {
        NavigationSplitView(columnVisibility: $columnVisibility) {
            // Sidebar: Category list
            List(categories, selection: $selectedCategory) { category in
                Label(category.name, systemImage: category.icon)
            }
            .navigationTitle("Categories")
        } content: {
            // Content: Note list
            if let category = selectedCategory {
                NoteListView(
                    category: category,
                    selectedNote: $selectedNote
                )
            } else {
                ContentUnavailableView(
                    "Select a Category",
                    systemImage: "folder",
                    description: Text("Choose a category from the sidebar")
                )
            }
        } detail: {
            // Detail: Note editor
            if let note = selectedNote {
                NoteDetailView(note: note)
            } else {
                ContentUnavailableView(
                    "Select a Note",
                    systemImage: "doc.text",
                    description: Text("Pick a note to start editing")
                )
            }
        }
    }
}

// Expected output:
// iPad landscape → Sidebar | Note list | Note detail (3 columns)
// iPad portrait → Collapsible sidebar with 2 columns
// iPhone → Standard push-based navigation
```

The `columnVisibility` parameter is key here — it lets users toggle sidebar visibility, and the system automatically adjusts the layout when the device rotates.

## Multitasking Support — Split View and Slide Over

### Handling Split View

Most iPad power users rely on Split View to run two apps side by side. While Rork Max apps support Split View by default, you need to ensure your layout adapts smoothly when the window narrows.

**Prompt for Rork Max:**

```
Make sure the layout doesn't break when the app is used in Split View.
Use horizontalSizeClass to switch between Compact and Regular layouts.
```

```swift
// Split View support: Adaptive layout based on screen width
struct AdaptiveNoteView: View {
    @Environment(\.horizontalSizeClass) private var sizeClass
    @State private var selectedNote: Note?

    var body: some View {
        if sizeClass == .regular {
            // Wide screen (iPad full screen or large Split View)
            HStack(spacing: 0) {
                NoteListView(selectedNote: $selectedNote)
                    .frame(width: 320)

                Divider()

                if let note = selectedNote {
                    NoteEditorView(note: note)
                } else {
                    placeholderView
                }
            }
        } else {
            // Narrow screen (iPhone or small Split View)
            NavigationStack {
                NoteListView(selectedNote: $selectedNote)
                    .navigationDestination(item: $selectedNote) { note in
                        NoteEditorView(note: note)
                    }
            }
        }
    }

    private var placeholderView: some View {
        ContentUnavailableView(
            "Select a note to get started",
            systemImage: "square.and.pencil"
        )
        .frame(maxWidth: .infinity, maxHeight: .infinity)
    }
}

// Expected output:
// Regular (iPad full screen / wide Split View) → List on left, editor on right
// Compact (iPhone / narrow Split View) → List view, tap to navigate to editor
```

### Slide Over Considerations

In Slide Over mode, your app's window shrinks to roughly iPhone width. Since `horizontalSizeClass` switches to `.compact`, the adaptive code above handles this automatically. No additional implementation is needed, but testing toolbar and button layouts in this narrow view is recommended.

## Implementing Drag and Drop

One of the iPad's standout features is inter-app drag and drop. Users can drag photos from the Photos app directly into your app — an interaction that feels native and powerful.

**Rork Max prompt:**

```
Add the ability to insert images into notes via drag and drop.
On iPad, support inter-app drag and drop using
dropDestination and the Transferable protocol.
```

```swift
// iPad drag and drop: Accepting dropped images
struct NoteEditorView: View {
    @Bindable var note: Note
    @State private var isDropTargetActive = false

    var body: some View {
        ScrollView {
            VStack(alignment: .leading, spacing: 16) {
                TextField("Title", text: $note.title)
                    .font(.title)

                TextEditor(text: $note.body)
                    .frame(minHeight: 200)

                // Image drop zone
                imageDropArea
            }
            .padding()
        }
        .navigationTitle("Edit Note")
    }

    private var imageDropArea: some View {
        ZStack {
            RoundedRectangle(cornerRadius: 12)
                .strokeBorder(
                    style: StrokeStyle(lineWidth: 2, dash: [8])
                )
                .foregroundStyle(isDropTargetActive ? .blue : .secondary)
                .frame(height: 150)

            VStack(spacing: 8) {
                Image(systemName: "photo.on.rectangle.angled")
                    .font(.largeTitle)
                Text("Drop images here")
                    .font(.subheadline)
            }
            .foregroundStyle(.secondary)
        }
        .dropDestination(for: Data.self) { items, _ in
            for item in items {
                if let image = UIImage(data: item) {
                    note.images.append(image)
                }
            }
            return !items.isEmpty
        } isTargeted: { targeted in
            isDropTargetActive = targeted
        }
    }
}

// Expected output:
// Drag from Photos → dashed area highlights blue → drop to add image
```

## Apple Pencil Integration

Rork Max can generate drawing and handwriting features using Apple's PencilKit framework. This opens up creative and productivity use cases that set your app apart.

**Prompt example:**

```
Add a drawing canvas where users can take handwritten notes
with Apple Pencil. Use PencilKit with PKCanvasView and
show a tool picker for selecting pen types and colors.
```

```swift
// Apple Pencil support: PencilKit drawing canvas
import PencilKit

struct DrawingCanvasView: UIViewRepresentable {
    @Binding var canvasView: PKCanvasView
    @Binding var toolPicker: PKToolPicker

    func makeUIView(context: Context) -> PKCanvasView {
        canvasView.drawingPolicy = .anyInput  // Allow finger drawing too
        canvasView.backgroundColor = .systemBackground

        toolPicker.setVisible(true, forFirstResponder: canvasView)
        toolPicker.addObserver(canvasView)
        canvasView.becomeFirstResponder()

        return canvasView
    }

    func updateUIView(_ uiView: PKCanvasView, context: Context) {}
}

struct HandwritingNoteView: View {
    @State private var canvasView = PKCanvasView()
    @State private var toolPicker = PKToolPicker()

    var body: some View {
        VStack {
            DrawingCanvasView(
                canvasView: $canvasView,
                toolPicker: $toolPicker
            )
        }
        .navigationTitle("Handwritten Note")
        .toolbar {
            ToolbarItem(placement: .topBarTrailing) {
                Button("Clear") {
                    canvasView.drawing = PKDrawing()
                }
            }
        }
    }
}

// Expected output:
// Full-screen canvas for drawing with Apple Pencil
// Tool picker shows pen, marker, and eraser options
```

## Adding Keyboard Shortcuts

External keyboard users on iPad expect keyboard shortcuts for common actions. Adding them makes your app feel professional and efficient.

```swift
// Keyboard shortcut implementation
struct NoteAppView: View {
    @State private var showNewNote = false
    @State private var searchText = ""

    var body: some View {
        NavigationSplitView {
            NoteListView()
                .searchable(text: $searchText)
        } detail: {
            NoteDetailView()
        }
        .sheet(isPresented: $showNewNote) {
            NewNoteView()
        }
        // Command + N to create a new note
        .keyboardShortcut("n", modifiers: .command)
    }
}

// When users hold the Command key on iPad,
// a shortcut overlay displays all available shortcuts.

// Expected output:
// ⌘N → New note sheet appears
// ⌘ long press → Shortcut overlay displays
```

## Common Issues and Fixes

### Layout breaks on iPad

Tell Rork Max: "Avoid fixed widths. Use `frame(maxWidth: .infinity)` and GeometryReader for flexible layouts." Hard-coded sizes cause problems across iPad's various screen dimensions and multitasking configurations.

### Sidebar disappears in certain orientations

Set `columnVisibility` to `.all` on your NavigationSplitView and add `.navigationSplitViewStyle(.balanced)` for consistent column behavior across orientations.

### Apple Pencil input not responding

Verify that `PKCanvasView`'s `drawingPolicy` is set to `.anyInput`. If set to `.pencilOnly`, finger input will be disabled, which can be confusing during testing without a physical Pencil.

## FAQ

### Can I run a Rork Max iPad app on iPhone too?

Yes. NavigationSplitView automatically degrades to NavigationStack-like behavior on iPhone. By using `horizontalSizeClass` to switch between layouts, a single codebase serves both devices. When you tell Rork Max to "support both iPhone and iPad," the AI generates the appropriate adaptive layout code.

### Does this work on iPad mini?

Yes. Although the iPad mini has a smaller screen, its `horizontalSizeClass` is still `.regular`. Setting `columnVisibility` to `.automatic` on NavigationSplitView lets the system decide the optimal column count based on available width. No special handling is required — Rork Max generated code works on iPad mini out of the box.

### Is Stage Manager supported?

SwiftUI apps built with Rork Max automatically support Stage Manager. Since Stage Manager allows freeform window resizing similar to macOS, your adaptive layout responds to size changes just as it does with Split View. No additional code is needed, though testing at small window sizes is recommended to catch any edge cases.
