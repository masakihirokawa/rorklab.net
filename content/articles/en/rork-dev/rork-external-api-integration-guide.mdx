## Why API Integration Matters for Your Rork App

Building a modern app almost always means connecting to external services. Whether you're pulling in weather data, processing payments, or adding AI-powered features, knowing how to work with APIs is essential.

Rork makes it incredibly fast to build React Native / Expo apps with AI assistance, but the API integration layer is where you as a developer add real value. This guide walks you through everything you need to know — from basic fetch calls to production-ready error handling — so you can confidently connect any external service to your Rork app.

Here's what you'll learn:

- Making REST API requests with `fetch`
- Managing API keys securely
- Implementing Bearer token and header-based authentication
- Building robust error handling
- Creating loading UI patterns that keep users informed

## REST API Fundamentals

REST (Representational State Transfer) APIs communicate over HTTP and form the backbone of modern web services. In your Rork app, you'll work with four main HTTP methods:

- **GET**: Retrieve data (e.g., fetching a news feed)
- **POST**: Send or create data (e.g., user registration)
- **PUT / PATCH**: Update existing data (e.g., editing a profile)
- **DELETE**: Remove data (e.g., deleting a post)

API responses typically come back as JSON, and React Native provides the standard JavaScript `fetch` function to handle these requests natively.

## Basic Data Fetching with fetch

Let's start with the simplest possible API call — a GET request to a public endpoint:

```javascript
// Basic GET request
const fetchUsers = async () => {
  try {
    const response = await fetch('https://jsonplaceholder.typicode.com/users');
    
    // Check the response status code
    if (!response.ok) {
      throw new Error(`HTTP error! status: ${response.status}`);
    }
    
    // Parse the JSON response
    const data = await response.json();
    console.log('Users fetched:', data.length);
    // Expected output: Users fetched: 10
    
    return data;
  } catch (error) {
    console.error('Failed to fetch data:', error.message);
    return [];
  }
};
```

Three key takeaways here: `async/await` keeps asynchronous code readable, `response.ok` catches HTTP errors, and `try/catch` handles network failures gracefully.

## Using APIs in Rork Components

In a real Rork app screen, you'll combine `useState` and `useEffect` to fetch and display data. Here's a weather component that demonstrates this pattern:

```javascript
import React, { useState, useEffect } from 'react';
import { View, Text, ActivityIndicator, StyleSheet } from 'react-native';

// Weather display component
export default function WeatherScreen() {
  const [weather, setWeather] = useState(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    const fetchWeather = async () => {
      try {
        setLoading(true);
        const response = await fetch(
          'https://api.open-meteo.com/v1/forecast?latitude=40.7128&longitude=-74.0060&current_weather=true'
        );

        if (!response.ok) {
          throw new Error('Failed to fetch weather data');
        }

        const data = await response.json();
        setWeather(data.current_weather);
      } catch (err) {
        setError(err.message);
      } finally {
        setLoading(false);
      }
    };

    fetchWeather();
  }, []);

  if (loading) {
    return (
      <View style={styles.center}>
        <ActivityIndicator size="large" color="#4A90D9" />
        <Text>Loading weather data...</Text>
      </View>
    );
  }

  if (error) {
    return (
      <View style={styles.center}>
        <Text style={styles.errorText}>{error}</Text>
      </View>
    );
  }

  return (
    <View style={styles.container}>
      <Text style={styles.title}>New York Weather</Text>
      <Text style={styles.temp}>{weather.temperature}°C</Text>
      <Text>Wind Speed: {weather.windspeed} km/h</Text>
    </View>
  );
}
// Expected output: Current temperature and wind speed for New York displayed on screen

const styles = StyleSheet.create({
  container: { flex: 1, padding: 20 },
  center: { flex: 1, justifyContent: 'center', alignItems: 'center' },
  title: { fontSize: 24, fontWeight: 'bold', marginBottom: 10 },
  temp: { fontSize: 48, fontWeight: 'bold', color: '#4A90D9' },
  errorText: { color: 'red', fontSize: 16 },
});
```

The pattern of managing three states — `loading`, `error`, and `data` — ensures users always see appropriate feedback regardless of what happens with the network.

## Securing Your API Keys

Most external APIs require an API key for authentication. Here's how to handle them safely in a Rork app.

### Using Environment Variables

Expo's environment variable support lets you keep API keys out of your source code:

```javascript
// .env file (never commit this to version control)
// API_KEY=YOUR_API_KEY

// Using the environment variable in your app
const API_KEY = process.env.EXPO_PUBLIC_API_KEY;

const fetchData = async () => {
  const response = await fetch(
    `https://api.example.com/data?key=${API_KEY}`
  );
  const data = await response.json();
  return data;
};
// Expected output: API key securely loaded from environment variable for the request
```

### Bearer Token Authentication

For OAuth 2.0 and token-based auth, you'll include the token in the `Authorization` header:

```javascript
// Authenticated request with Bearer token
const fetchProtectedData = async (accessToken) => {
  try {
    const response = await fetch('https://api.example.com/protected/resource', {
      method: 'GET',
      headers: {
        'Authorization': `Bearer ${accessToken}`,
        'Content-Type': 'application/json',
      },
    });

    if (response.status === 401) {
      // Token expired — trigger refresh flow
      console.log('Auth error: please refresh your token');
      return null;
    }

    if (!response.ok) {
      throw new Error(`API error: ${response.status}`);
    }

    return await response.json();
  } catch (error) {
    console.error('Authenticated request failed:', error.message);
    return null;
  }
};
// Expected output: Data securely retrieved from a protected endpoint
```

The golden rule: never hardcode API keys in your source files. Add `.env` to your `.gitignore` and use environment variables or a backend proxy instead.

## Sending Data with POST Requests

When your app needs to submit forms, register users, or create resources, you'll use POST requests:

```javascript
// Sending data with a POST request
const createPost = async (title, body) => {
  try {
    const response = await fetch('https://jsonplaceholder.typicode.com/posts', {
      method: 'POST',
      headers: {
        'Content-Type': 'application/json',
      },
      body: JSON.stringify({
        title: title,
        body: body,
        userId: 1,
      }),
    });

    if (!response.ok) {
      throw new Error(`Submission failed: ${response.status}`);
    }

    const result = await response.json();
    console.log('Created post ID:', result.id);
    // Expected output: Created post ID: 101
    
    return result;
  } catch (error) {
    console.error('Failed to create post:', error.message);
    throw error;
  }
};
```

Always remember to set the `Content-Type: application/json` header and serialize your data with `JSON.stringify()`.

## Error Handling Best Practices

Production apps need to handle network failures, rate limits, server outages, and more. Here's a reusable API client class that covers the most common scenarios:

```javascript
// Robust API client with comprehensive error handling
class ApiClient {
  constructor(baseUrl) {
    this.baseUrl = baseUrl;
  }

  async request(endpoint, options = {}) {
    const url = `${this.baseUrl}${endpoint}`;
    const config = {
      headers: {
        'Content-Type': 'application/json',
        ...options.headers,
      },
      ...options,
    };

    try {
      const response = await fetch(url, config);

      // Handle specific status codes
      if (response.status === 429) {
        const retryAfter = response.headers.get('Retry-After') || 60;
        throw new Error(`Rate limited — please retry after ${retryAfter} seconds`);
      }

      if (response.status >= 500) {
        throw new Error('Server error — please try again later');
      }

      if (response.status === 404) {
        throw new Error('The requested resource was not found');
      }

      if (!response.ok) {
        const errorBody = await response.text();
        throw new Error(`API error (${response.status}): ${errorBody}`);
      }

      return await response.json();
    } catch (error) {
      if (error.name === 'TypeError' && error.message === 'Network request failed') {
        throw new Error('No network connection — please check your internet');
      }
      throw error;
    }
  }

  get(endpoint) {
    return this.request(endpoint, { method: 'GET' });
  }

  post(endpoint, data) {
    return this.request(endpoint, {
      method: 'POST',
      body: JSON.stringify(data),
    });
  }
}

// Usage
const api = new ApiClient('https://api.example.com');
// Expected output: Appropriate error messages thrown based on HTTP status codes
```

This class-based approach gives you consistent error handling across your entire app. You can even tell Rork to "create an API client class with error handling" in your prompt, and it will generate something similar.

## Frequently Asked Questions

### Are there any restrictions on which APIs I can use with Rork?

Since Rork builds React Native / Expo apps, you're making native HTTP requests that aren't subject to browser CORS restrictions. This means you can connect to virtually any REST API directly. That said, always prefer APIs that use HTTPS for security.

### Is it safe to embed API keys directly in the app?

API keys embedded in client-side code can be extracted through reverse engineering. For sensitive keys, route requests through a backend service (like Supabase Edge Functions) so the key never leaves your server. For low-risk, read-only public APIs, client-side keys with usage restrictions are acceptable.

### What should I do when API responses are slow?

Start by implementing a timeout with `AbortController` to cancel requests that take too long. You can also cache responses locally so subsequent loads are instant, and use skeleton loading screens to keep the UI feeling responsive while data loads.

### Any tips for prompting Rork to generate API integration code?

Be specific in your prompts. Instead of "add an API call," try something like "Create a component that fetches data from [API endpoint], displays it in a FlatList, and includes loading spinners and error messages." The more context you give, the better the generated code.
