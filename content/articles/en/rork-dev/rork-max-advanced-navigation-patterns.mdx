## Introduction — Why Navigation Design Matters

The quality of a mobile app is fundamentally shaped by its navigation architecture. No matter how polished your UI components are, a poorly structured navigation system will undermine the entire user experience.

When building apps with Rork Max, the foundation is Expo Router and React Navigation. For simple apps, the default configuration works fine, but once your app grows beyond 20 screens, maintainability deteriorates rapidly without a solid understanding of navigation design patterns.

This article provides a comprehensive guide to implementing advanced navigation techniques in Rork Max that scale to production-grade applications. We'll cover nested structure design principles, custom screen transition animations, authentication flow branching, shared element transitions, and performance optimization — everything you need for a professional-quality app.

This guide assumes familiarity with Rork Max basics and Expo Router fundamentals. For an introduction to Expo Router, see [Expo Router and Rork Navigation Design Guide](/en/articles/rork-dev/expo-router-navigation-guide).

## Mastering Expo Router's File-Based Routing

Expo Router uses file-system-based routing where the file structure inside the `app/` directory directly maps to the route structure. A deep understanding of this system is the first step toward advanced navigation design.

### Route Groups and Layout Hierarchy

In large-scale apps, organizing screens into logical groups is essential. Expo Router's "route groups" (directories wrapped in parentheses) let you structure screens without affecting the URL path.

```typescript
// app/ directory structure design example
// app/
// ├── (auth)/           ← Pre-authentication screen group
// │   ├── _layout.tsx   ← Stack Navigator (no header)
// │   ├── login.tsx
// │   ├── register.tsx
// │   └── forgot-password.tsx
// ├── (tabs)/           ← Main tab group
// │   ├── _layout.tsx   ← Tab Navigator
// │   ├── home/
// │   │   ├── _layout.tsx  ← Stack Navigator (nested within tab)
// │   │   ├── index.tsx
// │   │   └── [id].tsx     ← Dynamic route
// │   ├── search.tsx
// │   ├── notifications.tsx
// │   └── profile/
// │       ├── _layout.tsx
// │       ├── index.tsx
// │       └── settings.tsx
// ├── (modals)/         ← Modal screen group
// │   ├── _layout.tsx   ← Stack Navigator (presentation: modal)
// │   ├── create-post.tsx
// │   └── image-viewer.tsx
// └── _layout.tsx       ← Root layout (auth check)

// app/_layout.tsx — Root layout implementation
import { Stack } from 'expo-router';
import { useAuth } from '@/hooks/useAuth';

export default function RootLayout() {
  return (
    <Stack screenOptions={{ headerShown: false }}>
      <Stack.Screen name="(auth)" />
      <Stack.Screen name="(tabs)" />
      <Stack.Screen
        name="(modals)"
        options={{
          presentation: 'modal',
          animation: 'slide_from_bottom',
        }}
      />
    </Stack>
  );
}
```

Three key benefits define this pattern. First, authentication screens, main screens, and modals are managed as independent groups. Second, each group has its own layout (Navigator), cleanly separating navigation responsibilities. Third, the parenthesized portions don't appear in URL paths, so deep links remain clean and natural.

## Authentication Flow Architecture

The most complex navigation challenge in production apps is branching based on authentication state. In Rork Max, you can implement declarative auth guards by combining Expo Router's `redirect` with `useSegments`.

### Auth Context and Route Guard

```typescript
// contexts/AuthContext.tsx — Authentication state management
import React, { createContext, useContext, useEffect, useState } from 'react';
import { useRouter, useSegments, useRootNavigationState } from 'expo-router';
import * as SecureStore from 'expo-secure-store';

type AuthState = {
  user: User | null;
  isLoading: boolean;
  signIn: (email: string, password: string) => Promise<void>;
  signOut: () => Promise<void>;
};

const AuthContext = createContext<AuthState | null>(null);

export function AuthProvider({ children }: { children: React.ReactNode }) {
  const [user, setUser] = useState<User | null>(null);
  const [isLoading, setIsLoading] = useState(true);
  const router = useRouter();
  const segments = useSegments();
  const navigationState = useRootNavigationState();

  // Initialize auth state
  useEffect(() => {
    const restoreSession = async () => {
      try {
        const token = await SecureStore.getItemAsync('auth_token');
        if (token) {
          const userData = await verifyToken(token);
          setUser(userData);
        }
      } catch (error) {
        console.warn('Session restoration failed:', error);
        await SecureStore.deleteItemAsync('auth_token');
      } finally {
        setIsLoading(false);
      }
    };
    restoreSession();
  }, []);

  // Navigation guard — redirect on auth state changes
  useEffect(() => {
    if (isLoading || !navigationState?.key) return;

    const inAuthGroup = segments[0] === '(auth)';

    if (!user && !inAuthGroup) {
      // Not authenticated → redirect to login
      router.replace('/(auth)/login');
    } else if (user && inAuthGroup) {
      // Authenticated → redirect to main screen
      router.replace('/(tabs)/home');
    }
  }, [user, segments, isLoading, navigationState?.key]);

  const signIn = async (email: string, password: string) => {
    const { token, user: userData } = await authenticateUser(email, password);
    await SecureStore.setItemAsync('auth_token', token);
    setUser(userData);
    // The guard in useEffect automatically handles the redirect
  };

  const signOut = async () => {
    await SecureStore.deleteItemAsync('auth_token');
    setUser(null);
    // The guard in useEffect automatically handles the redirect
  };

  return (
    <AuthContext.Provider value={{ user, isLoading, signIn, signOut }}>
      {children}
    </AuthContext.Provider>
  );
}

export const useAuth = () => {
  const context = useContext(AuthContext);
  if (!context) throw new Error('useAuth must be used within AuthProvider');
  return context;
};
```

### Integrating Onboarding Flows

When displaying onboarding screens for new users, an additional branching layer is needed in the auth flow.

```typescript
// Extended navigation guard
useEffect(() => {
  if (isLoading || !navigationState?.key) return;

  const inAuthGroup = segments[0] === '(auth)';
  const inOnboarding = segments[0] === '(onboarding)';

  if (!user && !inAuthGroup) {
    router.replace('/(auth)/login');
  } else if (user && !user.onboardingCompleted && !inOnboarding) {
    // Onboarding not completed
    router.replace('/(onboarding)/welcome');
  } else if (user && user.onboardingCompleted && (inAuthGroup || inOnboarding)) {
    router.replace('/(tabs)/home');
  }
}, [user, segments, isLoading, navigationState?.key]);
```

The beauty of this design is that auth logic is centralized in a single location. Individual screen components don't need to worry about authentication state, and new screens are automatically protected simply by placing them in the appropriate route group.

## Custom Screen Transition Animations

While default screen transitions are smooth, implementing custom animations that match your app's brand creates a professional impression.

### Custom Stack Navigator Transitions

```typescript
// app/(tabs)/home/_layout.tsx — Custom transition animations
import { Stack } from 'expo-router';
import {
  type StackCardInterpolationProps,
  type StackCardStyleInterpolator,
} from '@react-navigation/stack';
import { Easing } from 'react-native';

// Fade + scale transition (iOS-inspired)
const fadeScaleTransition: StackCardStyleInterpolator = ({
  current,
  next,
  inverted,
  layouts: { screen },
}: StackCardInterpolationProps) => {
  const progress = current.progress;
  const nextProgress = next?.progress;

  const opacity = progress.interpolate({
    inputRange: [0, 1],
    outputRange: [0, 1],
  });

  const scale = progress.interpolate({
    inputRange: [0, 1],
    outputRange: [0.92, 1],
  });

  // Shrink current screen when next screen arrives
  const containerOpacity = nextProgress
    ? nextProgress.interpolate({
        inputRange: [0, 1],
        outputRange: [1, 0.85],
      })
    : 1;

  return {
    cardStyle: {
      opacity: containerOpacity,
      transform: [{ scale }],
    },
    overlayStyle: { opacity },
  };
};

export default function HomeLayout() {
  return (
    <Stack
      screenOptions={{
        cardStyleInterpolator: fadeScaleTransition,
        transitionSpec: {
          open: {
            animation: 'timing',
            config: {
              duration: 350,
              easing: Easing.bezier(0.2, 0.9, 0.3, 1.0),
            },
          },
          close: {
            animation: 'timing',
            config: {
              duration: 280,
              easing: Easing.bezier(0.4, 0.0, 0.2, 1.0),
            },
          },
        },
        gestureEnabled: true,
        gestureDirection: 'horizontal',
      }}
    >
      <Stack.Screen name="index" options={{ title: 'Home' }} />
      <Stack.Screen name="[id]" />
    </Stack>
  );
}
```

### Bottom Sheet Modal Transitions

```typescript
// app/(modals)/_layout.tsx — Bottom sheet modal

export default function ModalLayout() {

          },

            opacity: progress.interpolate({
              inputRange: [0, 1],
              outputRange: [0, 0.5],
            }),
          },

```

The critical factor in animation design is the `Easing.bezier()` curve selection. Material Design 3 motion guidelines recommend `(0.2, 0.9, 0.3, 1.0)` (deceleration) for entering and `(0.4, 0.0, 0.2, 1.0)` (acceleration) for exiting. This combination produces natural, satisfying transitions.

## Shared Element Transitions

The smooth movement of elements from a list screen to a detail screen (shared element transitions) dramatically enhances user experience. Here's how to implement them using React Native Reanimated.

### Basic Shared Element Transition

```typescript
// components/SharedImage.tsx — Shared element component
import Animated, {
  SharedTransition,
  withSpring,
  withTiming,
} from 'react-native-reanimated';

// Custom spring transition definition
const customTransition = SharedTransition.custom((values) => {
  'worklet';
  return {
    width: withSpring(values.targetWidth, {
      damping: 18,
      stiffness: 120,
    }),
    height: withSpring(values.targetHeight, {
      damping: 18,
      stiffness: 120,
    }),
    originX: withSpring(values.targetOriginX, {
      damping: 18,
      stiffness: 120,
    }),
    originY: withSpring(values.targetOriginY, {
      damping: 18,
      stiffness: 120,
    }),
    borderRadius: withTiming(values.targetBorderRadius, {
      duration: 300,
    }),
  };
});

type SharedImageProps = {
  id: string;
  source: { uri: string };
  style?: any;
};

export function SharedImage({ id, source, style }: SharedImageProps) {
  return (
    <Animated.Image
      sharedTransitionTag={`image-${id}`}
      sharedTransitionStyle={customTransition}
      source={source}
      style={style}
    />
  );
}

// screens/ListScreen.tsx — Usage in a list screen
import { SharedImage } from '@/components/SharedImage';
import { Link } from 'expo-router';

function PostCard({ post }: { post: Post }) {
  return (
    <Link href={`/(tabs)/home/${post.id}`} asChild>
      <Pressable style={styles.card}>
        <SharedImage
          id={post.id}
          source={{ uri: post.thumbnail }}
          style={styles.cardImage}
        />
        <Animated.Text
          sharedTransitionTag={`title-${post.id}`}
          style={styles.cardTitle}
        >
          {post.title}
        </Animated.Text>
      </Pressable>
    </Link>
  );
}

// screens/DetailScreen.tsx — Usage in detail screen
function PostDetail() {
  const { id } = useLocalSearchParams<{ id: string }>();
  const post = usePostData(id);

  return (
    <ScrollView>
      <SharedImage
        id={id}
        source={{ uri: post.image }}
        style={styles.heroImage}
      />
      <Animated.Text
        sharedTransitionTag={`title-${id}`}
        style={styles.detailTitle}
      >
        {post.title}
      </Animated.Text>
      {/* Article content */}
    </ScrollView>
  );
}
```

### Key Considerations for Shared Element Transitions

There are several important points to keep in mind when implementing shared element transitions.

First, `sharedTransitionTag` must be unique across the entire app. Always include the item ID in list elements. If the same tag exists on multiple screens, you'll get unexpected behavior.

Second, from a performance perspective, when rendering large numbers of items in a FlatList, configure `initialNumToRender` and `windowSize` appropriately to prevent off-screen items from unnecessarily registering transition tags.

Third, on Android, shared element transition support can be partial in some cases, so it's wise to prepare fallback animations with platform detection.

## Advanced Tab and Drawer Combination Patterns

### Conditional Tab Bar Visibility

Hiding the tab bar on certain screens while showing it on others is a common pattern.

```typescript
// app/(tabs)/_layout.tsx — Conditional tab bar
import { Tabs, usePathname } from 'expo-router';
import { Platform } from 'react-native';
import Animated, {
  useAnimatedStyle,
  withTiming,
  useSharedValue,
} from 'react-native-reanimated';
import { useEffect } from 'react';

// Routes where the tab bar should be hidden
const HIDDEN_TAB_ROUTES = [
  '/home/create',
  '/home/camera',
  '/profile/settings/account',
];

export default function TabLayout() {
  const pathname = usePathname();
  const tabBarVisible = useSharedValue(1);

  useEffect(() => {
    const shouldHide = HIDDEN_TAB_ROUTES.some((route) =>
      pathname.startsWith(route)
    );
    tabBarVisible.value = withTiming(shouldHide ? 0 : 1, { duration: 200 });
  }, [pathname]);

  return (
    <Tabs
      screenOptions={{
        headerShown: false,
        tabBarStyle: {
          position: 'absolute',
          borderTopWidth: 0,
          elevation: 0,
          ...Platform.select({
            ios: {
              shadowColor: '#000',
              shadowOffset: { width: 0, height: -2 },
              shadowOpacity: 0.08,
              shadowRadius: 8,
            },
          }),
        },
      }}
      tabBar={(props) => <AnimatedTabBar {...props} visible={tabBarVisible} />}
    >
      <Tabs.Screen
        name="home"
        options={{
          title: 'Home',
          tabBarIcon: ({ color }) => <HomeIcon color={color} />,
        }}
      />
      <Tabs.Screen
        name="search"
        options={{
          title: 'Search',
          tabBarIcon: ({ color }) => <SearchIcon color={color} />,
        }}
      />
      <Tabs.Screen
        name="notifications"
        options={{
          title: 'Notifications',
          tabBarIcon: ({ color }) => <BellIcon color={color} />,
          tabBarBadge: 3, // Unread notification count
        }}
      />
      <Tabs.Screen
        name="profile"
        options={{
          title: 'Profile',
          tabBarIcon: ({ color }) => <UserIcon color={color} />,
        }}
      />
    </Tabs>
  );
}

// AnimatedTabBar.tsx — Animated tab bar
function AnimatedTabBar({ visible, ...props }) {
  const animatedStyle = useAnimatedStyle(() => ({
    transform: [
      {
        translateY: (1 - visible.value) * 100, // Slide down to hide
      },
    ],
    opacity: visible.value,
  }));

  return (
    <Animated.View style={[styles.tabBarContainer, animatedStyle]}>
      {/* Tab bar rendering logic */}
    </Animated.View>
  );
}
```

## Navigation State Persistence and Restoration

Restoring the previous screen state when a user returns to your app from the background significantly improves UX.

```typescript
// hooks/useNavigationPersistence.ts
import { useEffect, useRef, useCallback } from 'react';
import AsyncStorage from '@react-native-async-storage/async-storage';
import {
  useNavigationContainerRef,
  useRootNavigation,
} from 'expo-router';

const NAVIGATION_STATE_KEY = 'NAVIGATION_STATE_V1';

export function useNavigationPersistence() {
  const navigationRef = useNavigationContainerRef();
  const isRestored = useRef(false);

  // Save state
  const persistState = useCallback(async () => {
    try {
      const state = navigationRef.current?.getRootState();
      if (state) {
        await AsyncStorage.setItem(
          NAVIGATION_STATE_KEY,
          JSON.stringify(state)
        );
      }
    } catch (error) {
      console.warn('Navigation state persistence failed:', error);
    }
  }, [navigationRef]);

  // State change listener
  useEffect(() => {
    const unsubscribe = navigationRef.current?.addListener(
      'state',
      persistState
    );
    return () => unsubscribe?.();
  }, [navigationRef, persistState]);

  // Restore state
  const restoreState = useCallback(async () => {
    if (isRestored.current) return undefined;
    isRestored.current = true;

    try {
      const savedState = await AsyncStorage.getItem(NAVIGATION_STATE_KEY);
      return savedState ? JSON.parse(savedState) : undefined;
    } catch {
      return undefined;
    }
  }, []);

  return { restoreState, persistState };
}
```

When implementing state persistence, keep security in mind. If you're saving route information for screens that require authentication, consider using `expo-secure-store`. Also, include a version key in the storage key to prevent crashes when route structures change between app versions.

### Route Data Prefetching

```typescript
// hooks/usePrefetchRoute.ts — Screen data prefetching
import { useQueryClient } from '@tanstack/react-query';
import { useCallback } from 'react';

export function usePrefetchRoute() {
  const queryClient = useQueryClient();

  const prefetchPost = useCallback(
    (postId: string) => {
      // Prefetch data before the user navigates
      queryClient.prefetchQuery({
        queryKey: ['post', postId],
        queryFn: () => fetchPost(postId),
        staleTime: 5 * 60 * 1000, // Valid for 5 minutes
      });
    },
    [queryClient]
  );

  return { prefetchPost };
}

// Usage: prefetch on list item onPressIn
function PostListItem({ post }: { post: Post }) {
  const { prefetchPost } = usePrefetchRoute();
  const router = useRouter();

  return (
    <Pressable
      onPressIn={() => prefetchPost(post.id)} // Prefetch on touch start
      onPress={() => router.push(`/(tabs)/home/${post.id}`)}
      style={styles.listItem}
    >
      <Text>{post.title}</Text>
    </Pressable>
  );
}
```

By starting the prefetch on `onPressIn` (the moment the finger touches) and navigating on `onPress` (when the finger lifts), those few dozen milliseconds make a significant difference in perceived speed.

## Deep Link Integration Design

In advanced navigation architectures, maintaining consistency with deep links is crucial. Expo Router's file-based routing makes deep link configuration straightforward, but combining it with auth guards requires careful design.

```typescript
// Deep link + auth guard integration pattern
// Add to contexts/AuthContext.tsx

const [pendingDeepLink, setPendingDeepLink] = useState<string | null>(null);

useEffect(() => {
  if (isLoading || !navigationState?.key) return;

  const inAuthGroup = segments[0] === '(auth)';

  if (!user && !inAuthGroup) {
    // Save the deep link before redirecting to auth
    const currentPath = '/' + segments.join('/');
    if (currentPath !== '/(auth)/login') {
      setPendingDeepLink(currentPath);
    }
    router.replace('/(auth)/login');
  } else if (user && inAuthGroup) {
    // After authentication, navigate to the saved deep link
    if (pendingDeepLink) {
      router.replace(pendingDeepLink);
      setPendingDeepLink(null);
    } else {
      router.replace('/(tabs)/home');
    }
  }
}, [user, segments, isLoading, navigationState?.key]);
```

For a deep dive into deep link implementation patterns, see [Deep Linking and Universal Links in Rork Max](/en/articles/rork-dev/rork-max-deep-linking-universal-links).

## Frequently Asked Questions (FAQ)

### Q: Should I use Expo Router or React Navigation directly?

For Rork Max projects, Expo Router is the recommended first choice. It's built on top of React Navigation and provides significant productivity gains through file-based routing, type-safe routing, automatic deep link configuration, and static route analysis. Direct React Navigation usage is only needed for extremely specialized cases that Expo Router doesn't support — such as dynamically switching Navigator types at runtime.

### Q: How do I maintain performance in large-scale apps with 50+ screens?

Three strategies are effective. First, use `lazy: true` and `freezeOnBlur: true` settings to prevent unnecessary screen rendering. Second, use React.lazy and Suspense for lazy loading heavy components. Third, properly partition route groups to keep the Navigator tree shallow. Avoid deep nesting — ideally stay within 2-3 levels.

### Q: When nesting a Stack Navigator inside a tab, how does the back button behave?

Within a tab's Stack Navigator, the back button navigates backward through that specific stack. Once you reach the root screen of the stack, the Android back button exits the app by default. If you want to manage cross-tab navigation history, set `backBehavior: 'history'` in the Tabs options to enable returning to the previous tab.

### Q: The login screen briefly flashes when implementing auth guards. How do I prevent this?

Keep the splash screen visible during the `isLoading` state. Use `SplashScreen.preventAutoHideAsync()` from `expo-splash-screen` and call `SplashScreen.hideAsync()` only after auth state restoration completes. This completely eliminates the flash during the auth check.

File-based routing mastery, authentication flow architecture, custom screen transition animations, shared element transitions, tab and drawer combination patterns, state persistence, performance optimization, and deep link integration design — by properly combining these elements, you can build a robust navigation structure that maintains its integrity as your screen count grows.
