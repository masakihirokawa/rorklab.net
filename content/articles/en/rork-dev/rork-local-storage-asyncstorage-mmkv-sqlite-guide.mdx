## Why Local Data Persistence Matters

Every mobile app needs to store data on the device — whether it's user preferences, cached API responses, or offline-ready content. In Rork apps built on React Native and Expo, choosing the right local storage solution directly impacts performance, user experience, and code maintainability.

Think about it: your app needs to remember login state, restore partially filled forms, cache images for instant loading, and keep working when the network drops. All of these depend on solid local data persistence.

This guide walks you through the three main storage libraries available in Rork apps — **AsyncStorage**, **MMKV**, and **SQLite** — comparing their strengths, trade-offs, and ideal use cases with practical code examples.

**Who this guide is for:**

- Developers building Rork apps who need to decide on a storage strategy
- Those already using AsyncStorage but hitting performance bottlenecks
- Anyone looking to add offline support or search functionality to their app

## AsyncStorage — The Simple Key-Value Store

### How It Works

AsyncStorage is the community-maintained key-value store for React Native. It's the most straightforward option and works out of the box in most Rork projects with minimal setup.

- **Data format**: String key-value pairs (objects stored via JSON.stringify)
- **API style**: Fully asynchronous with Promise-based methods
- **Storage backend**: SQLite on Android (6MB default), plist files on iOS
- **Thread safety**: Internally serialized for safe concurrent access

### Implementation

```typescript
import AsyncStorage from '@react-native-async-storage/async-storage';

// Save user preferences
const saveUserPreferences = async (preferences: {
  theme: 'light' | 'dark';
  language: string;
  notifications: boolean;
}) => {
  try {
    const jsonValue = JSON.stringify(preferences);
    await AsyncStorage.setItem('@user_preferences', jsonValue);
    console.log('Preferences saved successfully');
  } catch (error) {
    console.error('Failed to save preferences:', error);
  }
};

// Load user preferences
const loadUserPreferences = async () => {
  try {
    const jsonValue = await AsyncStorage.getItem('@user_preferences');
    if (jsonValue !== null) {
      return JSON.parse(jsonValue);
    }
    // Return defaults
    return { theme: 'light', language: 'en', notifications: true };
  } catch (error) {
    console.error('Failed to load preferences:', error);
    return null;
  }
};

// Expected output:
// saveUserPreferences({ theme: 'dark', language: 'en', notifications: true })
// => "Preferences saved successfully"
// loadUserPreferences()
// => { theme: 'dark', language: 'en', notifications: true }
```

### When AsyncStorage Is the Right Choice

- User settings (theme, language, notification toggles)
- Temporary auth token storage (use Keychain for sensitive tokens)
- Small caches with a few dozen entries
- Prototypes and MVPs where simplicity matters most

### Limitations

AsyncStorage works well for simple cases, but performance degrades noticeably with hundreds of entries or frequent read/write operations. Its async-only API also means you can't access stored data synchronously at app startup — a common pain point when initializing themes or navigation state.

## MMKV — Blazing-Fast Synchronous Storage

### How It Works

react-native-mmkv is a React Native binding for the MMKV library, originally developed by the WeChat team. It uses memory-mapped files (mmap) to achieve read/write speeds roughly **30x faster** than AsyncStorage.

- **Data format**: Key-value pairs with native type support (strings, numbers, booleans)
- **API style**: Both synchronous and asynchronous access
- **Encryption**: Built-in AES-128 encryption
- **Multiple instances**: Create separate storage instances for different purposes

### Implementation

```typescript
import { MMKV } from 'react-native-mmkv';

// Create a storage instance
const storage = new MMKV({
  id: 'app-storage',
  // Enable encryption (optional)
  // encryptionKey: 'your-encryption-key',
});

// Synchronously save and retrieve data
const saveTheme = (theme: 'light' | 'dark') => {
  storage.set('user.theme', theme);
  console.log(`Theme set to ${theme}`);
};

const loadTheme = (): 'light' | 'dark' => {
  const theme = storage.getString('user.theme');
  return (theme as 'light' | 'dark') ?? 'light';
};

// Storing objects
const saveCartItems = (items: Array<{ id: string; quantity: number }>) => {
  storage.set('cart.items', JSON.stringify(items));
};

const loadCartItems = (): Array<{ id: string; quantity: number }> => {
  const json = storage.getString('cart.items');
  return json ? JSON.parse(json) : [];
};

// Expected output:
// saveTheme('dark')
// => "Theme set to dark"
// loadTheme()
// => "dark"
// saveCartItems([{ id: 'item-1', quantity: 2 }])
// loadCartItems()
// => [{ id: 'item-1', quantity: 2 }]
```

### Integrating MMKV with React State

MMKV provides hooks that tie directly into React's rendering cycle, keeping your stored values and UI in sync automatically.

```typescript
import { useMMKVString, useMMKVBoolean } from 'react-native-mmkv';

function SettingsScreen() {
  // Use MMKV values as React state
  const [username, setUsername] = useMMKVString('user.name');
  const [darkMode, setDarkMode] = useMMKVBoolean('user.darkMode');

  return (
    <View>
      <Text>Username: {username ?? 'Not set'}</Text>
      <Switch
        value={darkMode ?? false}
        onValueChange={(value) => setDarkMode(value)}
      />
    </View>
  );
  // => Changing a value auto-persists to MMKV and triggers a re-render
}
```

### When MMKV Is the Right Choice

- Performance-critical apps that need data immediately at startup
- Frequently accessed settings, flags, and tokens
- Encrypted storage for sensitive local data
- When combined with the [state management patterns for Rork apps](/articles/rork-dev/rork-advanced-state-management-patterns), MMKV becomes especially powerful for global state persistence

## SQLite — Full Relational Database on Device

### How It Works

expo-sqlite provides a SQLite wrapper that brings relational database capabilities to your device. You get table schemas, SQL queries, indexes, and transactions — similar to what you'd use on a server, but running entirely on the phone.

- **Data format**: Tables with rows and columns (relational structure)
- **Queries**: Full SQL support for filtering, sorting, joining, and aggregating
- **Indexes**: Fast lookups even with tens of thousands of records
- **Transactions**: ACID compliance for data integrity

### Implementation

```typescript
import * as SQLite from 'expo-sqlite';

// Open the database
const db = await SQLite.openDatabaseAsync('myapp.db');

// Create tables
await db.execAsync(`
  CREATE TABLE IF NOT EXISTS tasks (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    title TEXT NOT NULL,
    completed INTEGER DEFAULT 0,
    category TEXT,
    created_at TEXT DEFAULT (datetime('now'))
  );
  CREATE INDEX IF NOT EXISTS idx_tasks_category ON tasks(category);
`);

// Add a task
const addTask = async (title: string, category: string) => {
  const result = await db.runAsync(
    'INSERT INTO tasks (title, category) VALUES (?, ?)',
    [title, category]
  );
  console.log(`Task added (ID: ${result.lastInsertRowId})`);
  return result.lastInsertRowId;
};

// Get tasks by category
const getTasksByCategory = async (category: string) => {
  const tasks = await db.getAllAsync(
    'SELECT * FROM tasks WHERE category = ? ORDER BY created_at DESC',
    [category]
  );
  return tasks;
};

// Get completion statistics
const getCompletionStats = async () => {
  const stats = await db.getFirstAsync(`
    SELECT
      COUNT(*) as total,
      SUM(CASE WHEN completed = 1 THEN 1 ELSE 0 END) as done
    FROM tasks
  `);
  console.log(`Completion rate: ${stats.done}/${stats.total}`);
  return stats;
};

// Expected output:
// await addTask('Create shopping list', 'daily')
// => "Task added (ID: 1)"
// await getTasksByCategory('daily')
// => [{ id: 1, title: 'Create shopping list', completed: 0, category: 'daily', created_at: '2026-03-29 ...' }]
// await getCompletionStats()
// => "Completion rate: 0/1"
```

### When SQLite Is the Right Choice

- Managing hundreds to tens of thousands of records
- Frequent filtering, sorting, and aggregation (task managers, expense trackers, inventory apps)
- Structured data with relationships (table joins)
- Offline-first apps that need local completeness before server sync

For a deep dive into building robust offline-capable apps, check out the [offline-first architecture guide for Rork Max](/articles/rork-dev/rork-max-offline-first-architecture). It covers sync engines, conflict resolution, and data reconciliation patterns in detail.

## Comparing the Three Options

The best storage choice depends on your data characteristics and access patterns. Here's how they stack up.

**Read/write speed**: MMKV is the clear winner at roughly 30x faster than AsyncStorage. SQLite performance depends on query complexity, but with proper indexing it handles tens of thousands of records efficiently. AsyncStorage is fine for small datasets but slows down noticeably as data grows.

**Data structure**: AsyncStorage and MMKV are key-value stores, ideal for simple values and JSON objects. SQLite uses a table-based relational model that shines when you need complex queries, filtering, or data relationships.

**Learning curve**: AsyncStorage is the easiest to pick up. MMKV has a very similar API, making migration straightforward. SQLite requires SQL knowledge, though expo-sqlite keeps the JavaScript API clean and approachable.

**Encryption**: MMKV has built-in AES-128 encryption. SQLite can be encrypted with SQLCipher. AsyncStorage offers no encryption, making it unsuitable for sensitive data.

**Practical recommendation**: Start with AsyncStorage for simplicity, upgrade to MMKV when performance matters, and bring in SQLite when you need structured data or search. This incremental approach keeps your architecture clean without over-engineering early on.

## Practical Pattern — Combining Multiple Storage Solutions

In real-world apps, using multiple storage libraries together gives you the best of each. Here's a common pattern for a task management app built with Rork.

```typescript
import { MMKV } from 'react-native-mmkv';
import * as SQLite from 'expo-sqlite';

// MMKV: App settings and UI state (needs fast synchronous access)
const settingsStorage = new MMKV({ id: 'settings' });

// SQLite: Task data (needs search and aggregation)
let db: SQLite.SQLiteDatabase;

const initializeStorage = async () => {
  // MMKV is available synchronously from the start
  const theme = settingsStorage.getString('theme') ?? 'light';
  const lastOpenedTab = settingsStorage.getString('lastTab') ?? 'today';

  // SQLite needs async initialization
  db = await SQLite.openDatabaseAsync('tasks.db');
  await db.execAsync(`
    CREATE TABLE IF NOT EXISTS tasks (
      id INTEGER PRIMARY KEY AUTOINCREMENT,
      title TEXT NOT NULL,
      due_date TEXT,
      priority INTEGER DEFAULT 0,
      completed INTEGER DEFAULT 0
    );
  `);

  console.log(`Initialized: theme=${theme}, tab=${lastOpenedTab}`);
  return { theme, lastOpenedTab };
};

// Save UI state (synchronous, instant)
const saveLastTab = (tab: string) => {
  settingsStorage.set('lastTab', tab);
};

// Search tasks (flexible SQL queries)
const searchTasks = async (keyword: string) => {
  return await db.getAllAsync(
    'SELECT * FROM tasks WHERE title LIKE ? ORDER BY priority DESC',
    [`%${keyword}%`]
  );
};

// Expected output:
// await initializeStorage()
// => "Initialized: theme=light, tab=today"
// saveLastTab('week')  ← saved synchronously, instantly
// await searchTasks('shopping')
// => [{ id: 1, title: 'Shopping list', due_date: '2026-03-30', ... }]
```

The key insight is to **use MMKV for settings and UI state** that need instant synchronous access, and **SQLite for business data** that benefits from structured queries and aggregation.

## Frequently Asked Questions

### Q1: How hard is it to migrate from AsyncStorage to MMKV?

The migration is relatively painless because the APIs are similar in structure. The main changes involve replacing `getItem`/`setItem` with `getString`/`set` and removing `await` keywords since MMKV operations are synchronous. For existing data, implement a one-time migration function that reads from AsyncStorage and writes to MMKV on first launch.

### Q2: What happens to the SQLite database when the app is uninstalled?

The SQLite database file is deleted along with the app. If your app stores important data locally, design a backup strategy that syncs to a server. Combining SQLite with Supabase or Firebase for cloud backup is a common and effective approach.

### Q3: Can I use MMKV in Expo's Managed Workflow?

Yes, starting with Expo SDK 50, you can use react-native-mmkv in Managed Workflow through Config Plugins. Since Rork projects typically use Expo, you'll use it alongside `expo-dev-client`. Adding the Config Plugin to your `app.json` is all it takes to get started.

### Q4: Which storage should I use for sensitive data?

For authentication tokens, personal information, and other sensitive data, use MMKV with its built-in AES-128 encryption or expo-secure-store (a Keychain/Keystore wrapper). The [Rork Max security best practices guide](/articles/rork-dev/rork-max-security-best-practices) covers detailed implementation patterns for secure local storage.

Choosing the right local storage for your Rork app comes down to matching data characteristics with the right tool.
