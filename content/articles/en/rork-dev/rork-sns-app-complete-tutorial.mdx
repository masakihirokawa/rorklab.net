## What You'll Build

This tutorial walks you through building a fully functional social networking app — think Twitter meets Instagram — using **Rork and Supabase**. By the end, your app will include:

- **User authentication** (email/password and Google login)
- **Post creation** with text and image uploads
- **Home timeline** showing only posts from followed users
- **Follow / unfollow** system
- **Likes** with instant optimistic UI feedback
- **Real-time updates** via Supabase Realtime
- **User profiles** with avatar, bio, and post history

This guide is aimed at developers who have already built something with Rork (see [Getting Started](/en/articles/rork-basics/getting-started)) and are ready to tackle a multi-feature, data-driven application. #### Database Schema (SQL)

Open the Supabase SQL Editor and run the following to create all required tables:

```sql
-- User profiles (1:1 with auth.users)
CREATE TABLE profiles (
  id UUID PRIMARY KEY REFERENCES auth.users(id) ON DELETE CASCADE,
  username TEXT UNIQUE NOT NULL,
  display_name TEXT,
  bio TEXT DEFAULT '',
  avatar_url TEXT,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- Posts table
CREATE TABLE posts (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES profiles(id) ON DELETE CASCADE,
  content TEXT NOT NULL,
  image_url TEXT,
  likes_count INTEGER DEFAULT 0,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- Follow relationships
CREATE TABLE follows (
  follower_id UUID NOT NULL REFERENCES profiles(id) ON DELETE CASCADE,
  following_id UUID NOT NULL REFERENCES profiles(id) ON DELETE CASCADE,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  PRIMARY KEY (follower_id, following_id)
);

-- Likes table
CREATE TABLE likes (
  user_id UUID NOT NULL REFERENCES profiles(id) ON DELETE CASCADE,
  post_id UUID NOT NULL REFERENCES posts(id) ON DELETE CASCADE,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  PRIMARY KEY (user_id, post_id)
);

-- Trigger to keep likes_count in sync automatically
CREATE OR REPLACE FUNCTION update_likes_count()
RETURNS TRIGGER AS $$
BEGIN
  IF TG_OP = 'INSERT' THEN
    UPDATE posts SET likes_count = likes_count + 1 WHERE id = NEW.post_id;
  ELSIF TG_OP = 'DELETE' THEN
    UPDATE posts SET likes_count = likes_count - 1 WHERE id = OLD.post_id;
  END IF;
  RETURN NULL;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER likes_count_trigger
AFTER INSERT OR DELETE ON likes
FOR EACH ROW EXECUTE FUNCTION update_likes_count();

-- Enable Row Level Security on all tables
ALTER TABLE profiles ENABLE ROW LEVEL SECURITY;
ALTER TABLE posts ENABLE ROW LEVEL SECURITY;
ALTER TABLE follows ENABLE ROW LEVEL SECURITY;
ALTER TABLE likes ENABLE ROW LEVEL SECURITY;

-- RLS Policies: everyone can read; users can only modify their own data
CREATE POLICY "profiles_read_all" ON profiles FOR SELECT USING (true);
CREATE POLICY "profiles_update_own" ON profiles FOR UPDATE USING (auth.uid() = id);
CREATE POLICY "posts_read_all" ON posts FOR SELECT USING (true);
CREATE POLICY "posts_insert_own" ON posts FOR INSERT WITH CHECK (auth.uid() = user_id);
CREATE POLICY "posts_delete_own" ON posts FOR DELETE USING (auth.uid() = user_id);
CREATE POLICY "follows_read_all" ON follows FOR SELECT USING (true);
CREATE POLICY "follows_manage_own" ON follows FOR ALL USING (auth.uid() = follower_id);
CREATE POLICY "likes_read_all" ON likes FOR SELECT USING (true);
CREATE POLICY "likes_manage_own" ON likes FOR ALL USING (auth.uid() = user_id);
```

> **Pro tip**: Using `likes_count_trigger` keeps the count denormalized in the `posts` table, eliminating the need for expensive aggregate queries every time you load the feed.

import AsyncStorage from '@react-native-async-storage/async-storage';
import { AppState } from 'react-native';

const SUPABASE_URL = process.env.EXPO_PUBLIC_SUPABASE_URL!;
const SUPABASE_ANON_KEY = process.env.EXPO_PUBLIC_SUPABASE_ANON_KEY!;

export const supabase = createClient(SUPABASE_URL, SUPABASE_ANON_KEY, {
  auth: {
    storage: AsyncStorage,
    autoRefreshToken: true,
    persistSession: true,
    detectSessionInUrl: false,

// Refresh token when app comes back to the foreground
AppState.addEventListener('change', (state) => {
  if (state === 'active') {
    supabase.auth.startAutoRefresh();
  } else {

  }
});

### Step 3: Timeline (Posts from Followed Users Only)

This is the heart of your SNS app — a feed that only shows posts from people you follow.

**Rork Prompt (timeline):**
```
Build the home timeline screen.

- Fetch posts only from users the current user follows (plus their own)
- Each post shows: avatar, username, text content, optional image, like count, timestamp
- Infinite scroll (load 20 posts per page)
- Tap a post to open the detail screen
- Like button with instant UI feedback (optimistic update)
- "New post" button at the top of the screen
```

Here's the custom hook that powers the timeline:

```typescript
// hooks/useTimeline.ts — timeline data hook
import { useState, useEffect, useCallback } from 'react';
import { supabase } from '@/lib/supabase';

const PAGE_SIZE = 20;

export function useTimeline(userId: string) {
  const [posts, setPosts] = useState<Post[]>([]);
  const [loading, setLoading] = useState(true);
  const [hasMore, setHasMore] = useState(true);
  const [page, setPage] = useState(0);

  const fetchPosts = useCallback(async (pageNum: number) => {
    // Get IDs of everyone the current user follows
    const { data: followData } = await supabase
      .from('follows')
      .select('following_id')
      .eq('follower_id', userId);

    const followingIds = followData?.map(f => f.following_id) ?? [];
    followingIds.push(userId); // Include the user's own posts

    const { data, error } = await supabase
      .from('posts')
      .select(`
        id, content, image_url, likes_count, created_at,
        profiles!user_id (id, username, display_name, avatar_url),
        likes!left (user_id)
      `)
      .in('user_id', followingIds)
      .order('created_at', { ascending: false })
      .range(pageNum * PAGE_SIZE, (pageNum + 1) * PAGE_SIZE - 1);

    if (error) throw error;
    if (data.length < PAGE_SIZE) setHasMore(false);

    // Attach isLiked flag to each post
    return data.map(post => ({
      ...post,
      isLiked: post.likes?.some(l => l.user_id === userId) ?? false,
    }));
  }, [userId]);

  useEffect(() => {
    fetchPosts(0).then(data => {
      setPosts(data);
      setLoading(false);
    });
  }, [fetchPosts]);

  const loadMore = async () => {
    if (!hasMore) return;
    const nextPage = page + 1;
    const newPosts = await fetchPosts(nextPage);
    setPosts(prev => [...prev, ...newPosts]);
    setPage(nextPage);
  };

  return { posts, loading, hasMore, loadMore, setPosts };
}
```

> **What this does**: The single Supabase `select` call uses a JOIN (`likes!left`) to determine in one round trip whether the current user has liked each post. `!left` means LEFT JOIN — posts with no likes at all still appear.
>
> **Expected output**: An array of post objects, each with an `isLiked: boolean` field added.

### Step 4: Likes with Optimistic UI

Likes should feel instant. We update the UI immediately and roll back only if the API call fails.

**Rork Prompt (likes):**
```
Add a like feature to each post.

- Tapping the heart instantly toggles the like (optimistic UI)
- Write to / delete from the Supabase likes table in the background
- If the API call fails, roll back to the previous state
- Liked heart = red; not liked = gray
```

```typescript
// hooks/useLike.ts — like toggle hook with optimistic update
import { useState } from 'react';
import { supabase } from '@/lib/supabase';

export function useLike(
  postId: string,
  initialIsLiked: boolean,
  initialCount: number,
  userId: string,
  onUpdate: (postId: string, isLiked: boolean, count: number) => void
) {
  const [isLiked, setIsLiked] = useState(initialIsLiked);
  const [count, setCount] = useState(initialCount);

  const toggleLike = async () => {
    // Optimistic update — change UI immediately
    const newIsLiked = !isLiked;
    const newCount = newIsLiked ? count + 1 : count - 1;
    setIsLiked(newIsLiked);
    setCount(newCount);
    onUpdate(postId, newIsLiked, newCount);

    try {
      if (newIsLiked) {
        const { error } = await supabase
          .from('likes')
          .insert({ user_id: userId, post_id: postId });
        if (error) throw error;
      } else {
        const { error } = await supabase
          .from('likes')
          .delete()
          .eq('user_id', userId)
          .eq('post_id', postId);
        if (error) throw error;
      }
    } catch (error) {
      // Roll back on failure
      setIsLiked(initialIsLiked);
      setCount(initialCount);
      onUpdate(postId, initialIsLiked, initialCount);
      console.error('Like update failed:', error);
    }
  };

  return { isLiked, count, toggleLike };
}
```

## Common Errors & Fixes

### Error 1: `RLS policy violation` — Can't read or write data

**Cause**: Row Level Security is enabled but the necessary policies are missing.

**Fix**:
```sql
-- Check existing policies
SELECT * FROM pg_policies WHERE tablename = 'posts';

-- Add a permissive read policy for testing
CREATE POLICY "allow_read" ON posts FOR SELECT USING (true);
```

### Error 2: Real-time updates aren't arriving

**Cause**: Realtime replication is not enabled for the table, or the `filter` syntax is wrong.

**Fix**:
1. Supabase Dashboard → Database → Replication → enable Realtime for the `posts` table
2. Double-check filter format: `user_id=in.(id1,id2)` — commas, no spaces

### Error 3: Images aren't displaying (Storage)

**Cause**: The Storage bucket is set to Private.

**Fix**:
```sql
UPDATE storage.buckets SET public = true WHERE id = 'post-images';
```
Or add a SELECT policy in Dashboard → Storage → Policies:
```sql
CREATE POLICY "public_images" ON storage.objects
  FOR SELECT USING (bucket_id = 'post-images');
```

### Error 4: `Cannot read property 'data' of undefined`

**Cause**: Missing error handling on Supabase queries. Every query returns `{ data, error }`.

**Fix**: Always handle `error` before using `data`:
```typescript
const { data, error } = await supabase.from('posts').select('*');
if (error) {
  console.error('Query error:', error.message);
  return;
}
// Safe to use data here
```

## FAQ

**Q1. Can I run a real SNS app on Supabase's free plan?**

A. Absolutely — for early-stage products. The free tier supports up to 50,000 MAUs, 500 MB of database storage, 1 GB of file storage, and 200 concurrent Realtime connections. Once you start growing, upgrade to the Pro plan ($25/month).

**Q2. My timeline queries are slow. How do I speed things up?**

A. Add composite indexes on the most-queried columns:
```sql
CREATE INDEX idx_posts_user_created ON posts(user_id, created_at DESC);
CREATE INDEX idx_follows_follower ON follows(follower_id);
```
This dramatically speeds up both the follow lookup and the post pagination.

**Q3. Can I compress images before uploading?**

A. Yes — use `expo-image-manipulator`. Just tell Rork: *"Compress images to max 1080px wide and 80% JPEG quality before uploading."* Rork will wire it in automatically.

**Q4. What App Store guidelines apply to SNS apps?**

A. Key requirements include: ① User-generated content moderation (report and block features), ② A clear privacy policy describing what data you collect, ③ Age restrictions if your app could contain adult content. See the App Store publishing guide for a full checklist.

**Q5. Will keeping a Realtime connection open drain my users' batteries?**

A. WebSockets are lightweight, but it's still good practice to disconnect in the background. Use the `AppState` pattern from Step 2 (`startAutoRefresh` / `stopAutoRefresh`) to pause the connection when the app is backgrounded.

**Q6. How do I build a "suggested users" feature?**

A. A simple approach: fetch active users you're not yet following.
```sql
SELECT p.* FROM profiles p
LEFT JOIN follows f ON f.following_id = p.id AND f.follower_id = :current_user_id
WHERE f.following_id IS NULL
  AND p.id != :current_user_id
ORDER BY (
  SELECT COUNT(*) FROM posts WHERE user_id = p.id
    AND created_at > NOW() - INTERVAL '30 days'
) DESC
LIMIT 10;
```
