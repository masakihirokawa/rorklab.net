## App Security Goes Beyond "Locking the Door"

Security is the thing mobile developers keep pushing to "next sprint." You build the features, polish the design, submit to the App Store — and somewhere in that process, decide that security can wait.

The numbers say otherwise. A 2025 study found that 83% of apps on Google Play contained at least one OWASP Mobile Top 10 vulnerability. App Store reviews are getting stricter too — apps with plaintext data storage or unprotected network calls are increasingly getting rejected.

Zero-trust security starts from one assumption: trust nothing. Don't trust the network. Don't trust the device. Don't trust user input. When you design from this premise, your app's security architecture changes fundamentally.

This guide walks through building an OWASP MASVS L2-compliant security architecture for SwiftUI apps built with Rork Max. Not just theory — you'll get copy-paste code and the real pitfalls I hit along the way.

## Understanding OWASP MASVS — Why Standards Matter

OWASP MASVS (Mobile Application Security Verification Standard) codifies mobile app security requirements into an international standard. It defines two levels: L1 (standard security) for general commercial apps, and L2 (defense in depth) for apps handling sensitive data.

L1 covers the basics — authentication, network communication, and data storage protections. L2 adds reverse engineering resistance and runtime protection.

Even for indie apps, I'd argue L2 is worth pursuing. The reason is straightforward: **attackers don't pick targets based on app size.** Automated scanners crawl for vulnerabilities indiscriminately. When you're a solo developer, an incident response falls entirely on your shoulders.

MASVS organizes requirements into eight categories:

- **MASVS-STORAGE**: Secure data storage
- **MASVS-CRYPTO**: Proper cryptography usage
- **MASVS-AUTH**: Authentication and authorization
- **MASVS-NETWORK**: Network communication protection
- **MASVS-PLATFORM**: Platform interaction safety
- **MASVS-CODE**: Code quality and build settings
- **MASVS-RESILIENCE**: Reverse engineering resistance
- **MASVS-PRIVACY**: Privacy protection

This article deep-dives into the four areas most critical for Rork Max apps: STORAGE, NETWORK, RESILIENCE, and CODE.

## Start with Threat Modeling — Define What You're Protecting

Before writing any security code, run a threat model. Throwing countermeasures at the wall wastes time on minor risks while missing critical ones.

### STRIDE Threat Classification

For a typical Rork Max app, here's how threats break down using the STRIDE model:

```swift
// Threat model definition — initialize at app launch
enum ThreatCategory: String, CaseIterable {
    case spoofing = "Spoofing"
    case tampering = "Tampering"
    case repudiation = "Repudiation"
    case informationDisclosure = "Information Disclosure"
    case denialOfService = "Denial of Service"
    case elevationOfPrivilege = "Elevation of Privilege"
}

struct ThreatModel {
    let category: ThreatCategory
    let asset: String        // What we're protecting
    let attackVector: String // How the attack happens
    let impact: Impact
    let likelihood: Likelihood
    
    enum Impact: Int { case low = 1, medium = 2, high = 3, critical = 4 }
    enum Likelihood: Int { case unlikely = 1, possible = 2, likely = 3, certain = 4 }
    
    /// Risk score = impact × likelihood
    var riskScore: Int { impact.rawValue * likelihood.rawValue }
}

// Define concrete threats
let threats: [ThreatModel] = [
    ThreatModel(
        category: .informationDisclosure,
        asset: "User auth tokens",
        attackVector: "Reading plaintext tokens from device filesystem",
        impact: .critical,
        likelihood: .likely
    ),
    ThreatModel(
        category: .tampering,
        asset: "API requests",
        attackVector: "Man-in-the-middle attack modifying requests",
        impact: .high,
        likelihood: .possible
    ),
    ThreatModel(
        category: .elevationOfPrivilege,
        asset: "Subscription status",
        attackVector: "Patching the app to bypass payment checks",
        impact: .high,
        likelihood: .likely
    ),
]

// Sort by risk score to determine implementation priority
let prioritized = threats.sorted { $0.riskScore > $1.riskScore }
for threat in prioritized {
    print("[\(threat.riskScore)] \(threat.category.rawValue): \(threat.asset)")
    // Expected output:
    // [12] Information Disclosure: User auth tokens
    // [9] Elevation of Privilege: Subscription status
    // [6] Tampering: API requests
}
```

The scoring makes priorities clear. Token protection comes first, then payment bypass prevention, then request tampering defense.

### Why Risk-Based Prioritization Matters

Spending equal effort on every threat is inefficient. Threats with a risk score of 4 or below can be deferred depending on your app's stage. Anything scoring 9 or above needs to ship before your app does.

Zero-trust means "question everything," but implementation order follows risk magnitude. That's not a contradiction — it's how you get maximum security from limited resources.

## Implementing Encrypted Storage — MASVS-STORAGE Compliance

### Using Keychain Services Correctly

iOS Keychain is the only correct answer for storing auth tokens, passwords, and encryption keys. Saving these to `UserDefaults` or `FileManager` in plaintext fails even L1 requirements.

When building SwiftUI apps with Rork Max, create a Keychain utility wrapper:

```swift
import Foundation
import Security

/// Explicit Keychain error definitions
enum KeychainError: Error, LocalizedError {
    case duplicateItem
    case itemNotFound
    case unexpectedStatus(OSStatus)
    case dataConversionFailed
    
    var errorDescription: String? {
        switch self {
        case .duplicateItem: return "Item already exists in Keychain"
        case .itemNotFound: return "Item not found in Keychain"
        case .unexpectedStatus(let status):
            return "Keychain error: \(SecCopyErrorMessageString(status, nil) ?? "unknown" as CFString)"
        case .dataConversionFailed: return "Failed to convert data"
        }
    }
}

final class SecureKeychain {
    
    private let serviceName: String
    private let accessGroup: String?
    
    /// - Parameters:
    ///   - serviceName: Use your app's Bundle Identifier
    ///   - accessGroup: Specify for sharing data via App Groups
    init(serviceName: String = Bundle.main.bundleIdentifier ?? "com.app.default",
         accessGroup: String? = nil) {
        self.serviceName = serviceName
        self.accessGroup = accessGroup
    }
    
    /// Save a value (overwrites if exists)
    func save(key: String, data: Data,
              accessibility: CFString = kSecAttrAccessibleWhenUnlockedThisDeviceOnly) throws {
        
        var query: [String: Any] = [
            kSecClass as String: kSecClassGenericPassword,
            kSecAttrService as String: serviceName,
            kSecAttrAccount as String: key,
            kSecValueData as String: data,
            // Using kSecAttrAccessibleWhenUnlockedThisDeviceOnly means:
            // - No access while device is locked
            // - Not included in backups
            kSecAttrAccessible as String: accessibility,
        ]
        
        if let group = accessGroup {
            query[kSecAttrAccessGroup as String] = group
        }
        
        let status = SecItemAdd(query as CFDictionary, nil)
        
        if status == errSecDuplicateItem {
            // Update existing item
            let searchQuery: [String: Any] = [
                kSecClass as String: kSecClassGenericPassword,
                kSecAttrService as String: serviceName,
                kSecAttrAccount as String: key,
            ]
            let updateAttributes: [String: Any] = [
                kSecValueData as String: data,
                kSecAttrAccessible as String: accessibility,
            ]
            let updateStatus = SecItemUpdate(
                searchQuery as CFDictionary,
                updateAttributes as CFDictionary
            )
            guard updateStatus == errSecSuccess else {
                throw KeychainError.unexpectedStatus(updateStatus)
            }
        } else if status \!= errSecSuccess {
            throw KeychainError.unexpectedStatus(status)
        }
    }
    
    /// Retrieve a value
    func load(key: String) throws -> Data {
        let query: [String: Any] = [
            kSecClass as String: kSecClassGenericPassword,
            kSecAttrService as String: serviceName,
            kSecAttrAccount as String: key,
            kSecReturnData as String: true,
            kSecMatchLimit as String: kSecMatchLimitOne,
        ]
        
        var result: AnyObject?
        let status = SecItemCopyMatching(query as CFDictionary, &result)
        
        guard status == errSecSuccess else {
            if status == errSecItemNotFound {
                throw KeychainError.itemNotFound
            }
            throw KeychainError.unexpectedStatus(status)
        }
        
        guard let data = result as? Data else {
            throw KeychainError.dataConversionFailed
        }
        return data
    }
    
    /// Delete a value
    func delete(key: String) throws {
        let query: [String: Any] = [
            kSecClass as String: kSecClassGenericPassword,
            kSecAttrService as String: serviceName,
            kSecAttrAccount as String: key,
        ]
        
        let status = SecItemDelete(query as CFDictionary)
        guard status == errSecSuccess || status == errSecItemNotFound else {
            throw KeychainError.unexpectedStatus(status)
        }
    }
}

// Usage — safely storing auth tokens
let keychain = SecureKeychain()

// Save
let token = "eyJhbGciOiJIUzI1NiIs..."
try keychain.save(key: "auth_token", data: Data(token.utf8))

// Retrieve
let storedData = try keychain.load(key: "auth_token")
let storedToken = String(data: storedData, encoding: .utf8)
print(storedToken ?? "retrieval failed")
// Expected output: eyJhbGciOiJIUzI1NiIs...
```

### Common Mistake: Using `kSecAttrAccessibleAlways`

Setting `kSecAttrAccessible` to `kSecAttrAccessibleAlways` means Keychain data is accessible even when the device is locked — a direct MASVS-STORAGE violation. Use `kSecAttrAccessibleWhenUnlockedThisDeviceOnly` instead. The "ThisDeviceOnly" suffix also excludes the data from iTunes backups.

Another frequent mistake is storing tokens in `UserDefaults`. This is saved as a plaintext plist file that's trivially readable on jailbroken devices. Search your Rork Max project for `UserDefaults` and verify no sensitive data has crept in.

## Implementing Network Defenses — MASVS-NETWORK Compliance

### Certificate Pinning: How It Works and Why TLS Alone Isn't Enough

TLS/SSL alone doesn't fully prevent man-in-the-middle attacks. If an attacker installs a root certificate on the device, they can intercept legitimate TLS traffic. Certificate pinning hardcodes the server certificate (or its public key) in your app and verifies it on every connection.

```swift
import Foundation
import CryptoKit

/// URLSession delegate implementing certificate pinning
final class CertificatePinningDelegate: NSObject, URLSessionDelegate {
    
    /// SHA-256 hashes of pinned public keys (Base64 encoded)
    /// Always configure at least two — primary and backup
    private let pinnedKeyHashes: Set<String> = [
        "BBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBB=",  // Primary cert public key hash
        "CCCCCCCCCCCCCCCCCCCCCCCCCCCCCCCCCCCCCCCCCCC=",  // Backup cert public key hash
    ]
    
    func urlSession(
        _ session: URLSession,
        didReceive challenge: URLAuthenticationChallenge,
        completionHandler: @escaping (URLSession.AuthChallengeDisposition, URLCredential?) -> Void
    ) {
        guard challenge.protectionSpace.authenticationMethod == NSURLAuthenticationMethodServerTrust,
              let serverTrust = challenge.protectionSpace.serverTrust else {
            completionHandler(.cancelAuthenticationChallenge, nil)
            return
        }
        
        // Validate server certificate chain
        guard SecTrustEvaluateWithError(serverTrust, nil) else {
            completionHandler(.cancelAuthenticationChallenge, nil)
            return
        }
        
        // Verify public keys in the certificate chain
        let certCount = SecTrustGetCertificateCount(serverTrust)
        var foundMatch = false
        
        for index in 0..<certCount {
            guard let certificate = SecTrustCopyCertificateChain(serverTrust)?
                    .unsafelyUnwrapped as? [SecCertificate],
                  index < certificate.count else { continue }
            
            let cert = certificate[index]
            
            // Extract public key from certificate
            guard let publicKey = SecCertificateCopyKey(cert),
                  let publicKeyData = SecKeyCopyExternalRepresentation(publicKey, nil) as? Data else {
                continue
            }
            
            // Compute SHA-256 hash and compare
            let hash = SHA256.hash(data: publicKeyData)
            let hashBase64 = Data(hash).base64EncodedString()
            
            if pinnedKeyHashes.contains(hashBase64) {
                foundMatch = true
                break
            }
        }
        
        if foundMatch {
            let credential = URLCredential(trust: serverTrust)
            completionHandler(.useCredential, credential)
        } else {
            // Pinning failed — reject the connection
            print("⚠️ Certificate pinning failed: \(challenge.protectionSpace.host)")
            completionHandler(.cancelAuthenticationChallenge, nil)
        }
    }
}

// Usage — API client with pinning
let pinningDelegate = CertificatePinningDelegate()
let session = URLSession(
    configuration: .default,
    delegate: pinningDelegate,
    delegateQueue: nil
)

// All requests through this session enforce certificate pinning
let url = URL(string: "https://api.example.com/v1/user")\!
let (data, response) = try await session.data(from: url)
// If pinning verification fails, the request is automatically cancelled
```

### Certificate Pinning Pitfalls

The most dangerous mistake is **not configuring a backup key**. Server certificates rotate periodically. If you only pin the primary key, your app goes completely offline the moment the certificate updates.

Another pitfall: using the same pinning config for development and production. When debugging with proxy tools like Charles Proxy or mitmproxy, pinning blocks all traffic. Add a `#if DEBUG` directive to disable pinning in debug builds only.

```swift
#if DEBUG
// Skip pinning in debug builds (development only)
completionHandler(.performDefaultHandling, nil)
#else
// Enforce pinning in release builds
if foundMatch {
    completionHandler(.useCredential, URLCredential(trust: serverTrust))
} else {
    completionHandler(.cancelAuthenticationChallenge, nil)
}
#endif
```

**Critically, verify that this `#if DEBUG` block never leaks into release builds.** If you're using EAS Build with Rork Max, confirm your production profile in `eas.json` doesn't include the `-DDEBUG` flag.

## Implementing Runtime Protection — MASVS-RESILIENCE Compliance

### Jailbreak Detection

On jailbroken devices, the app sandbox is disabled. Other apps can access your filesystem and attach to running processes. Perfect detection is impossible, but combining multiple signals increases accuracy significantly.

```swift
import Foundation
import UIKit

/// Multi-signal jailbreak detection
struct JailbreakDetector {
    
    /// Combined check — returns true if any signal is detected
    static func isJailbroken() -> Bool {
        return checkSuspiciousFiles()
            || checkSuspiciousURLSchemes()
            || checkWriteAccess()
            || checkDynamicLibraries()
            || checkFork()
    }
    
    /// 1. Check for files typically created by jailbreak tools
    private static func checkSuspiciousFiles() -> Bool {
        let suspiciousPaths = [
            "/Applications/Cydia.app",
            "/Applications/Sileo.app",
            "/usr/sbin/sshd",
            "/usr/bin/ssh",
            "/etc/apt",
            "/var/lib/cydia",
            "/private/var/stash",
            "/usr/libexec/sftp-server",
            "/usr/sbin/frida-server",
            "/usr/bin/cycript",
        ]
        
        for path in suspiciousPaths {
            if FileManager.default.fileExists(atPath: path) {
                return true
            }
        }
        return false
    }
    
    /// 2. Check if Cydia/Sileo URL schemes are available
    private static func checkSuspiciousURLSchemes() -> Bool {
        let schemes = ["cydia://", "sileo://", "zbra://"]
        for scheme in schemes {
            if let url = URL(string: scheme),
               UIApplication.shared.canOpenURL(url) {
                return true
            }
        }
        return false
    }
    
    /// 3. Check if writing outside the sandbox is possible
    private static func checkWriteAccess() -> Bool {
        let testPath = "/private/jailbreak_test_\(UUID().uuidString)"
        do {
            try "test".write(toFile: testPath, atomically: true, encoding: .utf8)
            try FileManager.default.removeItem(atPath: testPath)
            return true  // Write succeeded = sandbox is broken
        } catch {
            return false  // Write failed = normal
        }
    }
    
    /// 4. Detect suspicious dynamic libraries
    private static func checkDynamicLibraries() -> Bool {
        let suspiciousLibs = [
            "FridaGadget",
            "frida-agent",
            "cynject",
            "libcycript",
            "SubstrateLoader",
        ]
        
        let count = _dyld_image_count()
        for i in 0..<count {
            if let name = _dyld_get_image_name(i) {
                let imageName = String(cString: name)
                for lib in suspiciousLibs {
                    if imageName.contains(lib) {
                        return true
                    }
                }
            }
        }
        return false
    }
    
    /// 5. Check if fork() succeeds (it shouldn't on non-jailbroken devices)
    private static func checkFork() -> Bool {
        let pid = fork()
        if pid >= 0 {
            if pid > 0 {
                kill(pid, SIGTERM)
            }
            return true
        }
        return false
    }
}

// Usage — security check at app launch
func applicationDidFinishLaunching() {
    if JailbreakDetector.isJailbroken() {
        // Don't call exit(0) — enter restricted mode instead
        SecurityManager.shared.enableRestrictedMode()
        print("⚠️ Jailbreak detected — launching in restricted mode")
        // Expected output: ⚠️ Jailbreak detected — launching in restricted mode
    }
}
```

### Debugger Attach Detection

Attackers can attach debuggers (like lldb) to your app process to read runtime memory or modify function return values.

```swift
import Foundation

/// Detect debugger attachment
struct DebuggerDetector {
    
    /// Check for debugger via sysctl
    static func isBeingDebugged() -> Bool {
        var info = kinfo_proc()
        var size = MemoryLayout<kinfo_proc>.stride
        var mib: [Int32] = [CTL_KERN, KERN_PROC, KERN_PROC_PID, getpid()]
        
        let result = sysctl(&mib, UInt32(mib.count), &info, &size, nil, 0)
        guard result == 0 else { return false }
        
        return (info.kp_proc.p_flag & P_TRACED) \!= 0
    }
    
    /// Deny debugger attachment via ptrace
    /// Call this as early as possible in app launch
    static func denyDebuggerAttach() {
        #if \!DEBUG
        let PT_DENY_ATTACH: CInt = 31
        ptrace(PT_DENY_ATTACH, 0, nil, 0)
        #endif
    }
}

// Call at app launch — in @main init() or early AppDelegate
DebuggerDetector.denyDebuggerAttach()
if DebuggerDetector.isBeingDebugged() {
    print("⚠️ Debugger detected")
    // Expected output (when debugger is attached): ⚠️ Debugger detected
}
```

### Runtime Protection Mistakes to Avoid

**Calling `exit(0)` on jailbreak detection is counterproductive.** Apple's review guidelines frown on sudden crashes, and attackers can hook the `exit` function to disable it anyway.

Instead, implement a "restricted mode." Purge sensitive data from Keychain, disable payment features, and notify the user politely. This passes App Store review and provides real security.

Another mistake is **putting all detection in a single function.** An attacker hooks that one function, returns `false`, and bypasses everything. Split checks into multiple independent functions called at different points in the app lifecycle.

## Code Obfuscation and Binary Protection — MASVS-CODE Compliance

### Swift Code Obfuscation Strategy

The Swift compiler at optimization level `-O` provides some obfuscation. But string literals (API keys, endpoint URLs) remain as plaintext in the binary.

```swift
/// Runtime string decryption — prevents binary analysis from finding endpoints
struct ObfuscatedString {
    
    private let encrypted: [UInt8]
    private let key: UInt8
    
    /// XOR encryption for string protection
    /// Not cryptographically strong, but prevents discovery via `strings` command
    init(_ plainText: String, key: UInt8 = 0xAB) {
        self.key = key
        self.encrypted = plainText.utf8.map { $0 ^ key }
    }
    
    /// Decrypt at time of use
    var value: String {
        let decrypted = encrypted.map { $0 ^ key }
        return String(bytes: decrypted, encoding: .utf8) ?? ""
    }
}

// Usage — protecting API endpoints
// ❌ Bad: plaintext in binary
// let apiEndpoint = "https://api.myapp.com/v2/secure"

// ✅ Good: not decrypted until runtime
let apiEndpoint = ObfuscatedString("https://api.myapp.com/v2/secure")
let url = URL(string: apiEndpoint.value)\!
print(url.absoluteString)
// Expected output: https://api.myapp.com/v2/secure
```

### Hardening App Transport Security

ATS settings in `Info.plist` are often inadvertently relaxed. If Rork Max generated your project with `NSAllowsArbitraryLoads` set to `true`, fix it for production:

```xml
<\!-- Info.plist — strict ATS configuration -->
<key>NSAppTransportSecurity</key>
<dict>
    <key>NSAllowsArbitraryLoads</key>
    <false/>
    
    <\!-- Exception domains only when absolutely necessary -->
    <key>NSExceptionDomains</key>
    <dict>
        <key>legacy-api.example.com</key>
        <dict>
            <key>NSExceptionAllowsInsecureHTTPLoads</key>
            <true/>
            <key>NSExceptionMinimumTLSVersion</key>
            <string>TLSv1.2</string>
            <\!-- Temporary workaround until legacy API supports HTTPS -->
        </dict>
    </dict>
</dict>
```

## Automating Security Testing in CI/CD

### Running Security Checks with GitHub Actions

Security isn't "implement and forget." You need continuous testing to prevent regressions.

```yaml

  pull_request:

  push:
    branches: [main]

      # 4. Detect sensitive data in UserDefaults

        run: |

            echo "$INSECURE"

          fi

      # 5. Debug code remnants check
      - name: Check for debug remnants
        run: |
          DEBUG_LOGS=$(grep -rn "print(\|NSLog(" --include="*.swift" . \

            | grep -v "#if DEBUG" \
            | grep -v "test/" || true)
          if [ -n "$DEBUG_LOGS" ]; then
            echo "⚠️ Debug logs detected (please review):"
            echo "$DEBUG_LOGS" | head -20
          fi
```

### What Automation Can't Catch

CI/CD security checks maintain a baseline, but they can't catch logic vulnerabilities. "Payment status checked only on the client side" or "API response signatures not verified" are design flaws that code scanning won't find.

I strongly recommend setting aside time quarterly to manually review the OWASP MASVS checklist. It's published on the [official OWASP site](https://mas.owasp.org/MASVS/).

## Common Mistakes and Pitfalls

### 1. "Release Builds Don't Include Debug Info" — Wrong

Swift Release builds include DWARF debug information by default. Set `Strip Debug Symbols During Copy` to `Yes` and change `Debug Information Format` to `dwarf` (not `dwarf-with-dsym`). Keep the dSYM files archived separately — you need them for crash report symbolication.

### 2. Getting Certificate Pinning Hashes Wrong

When computing public key hashes, some libraries expect the hash of the entire DER-encoded public key, while others expect the Subject Public Key Info (SPKI) hash. Always generate a test hash from your actual server certificate and verify it works:

```bash
# Get the public key hash for your server certificate
openssl s_client -connect api.example.com:443 2>/dev/null \
  | openssl x509 -pubkey -noout \
  | openssl pkey -pubin -outform DER \
  | openssl dgst -sha256 -binary \
  | base64
```

### 3. Forgetting Keychain Access Group Configuration

When sharing Keychain data between the main app and App Extensions (Widgets, Notification Service Extensions), you must configure `kSecAttrAccessGroup`. Missing this means your Widget can't read the auth token and throws authentication errors. Add the same group to your Entitlements file under `Keychain Access Groups`.

### 4. Not Validating Network Response Values

Swift's `Codable` catches type mismatches when decoding JSON, but it doesn't validate **value correctness**. A response like `"price": -100` gets decoded without error, breaking your payment logic. Add explicit validation for critical fields.

### 5. Concentrating All Security Checks at Launch

Running jailbreak detection, debugger detection, and integrity checks all in `applicationDidFinishLaunching` tells attackers: "Hook this one method to bypass everything." Distribute detection across the app lifecycle — during payment checks, API calls, background resume, and other critical moments.

## Practical Application: Preventing Payment Bypass

The most damaging attack on indie apps is payment bypass — patching the app to force `isPremium` to always return `true`.

```swift
/// Payment bypass prevention with server-side verification
actor PremiumVerifier {
    
    private let keychain = SecureKeychain()
    private let session: URLSession
    
    // Never rely on local flags alone — always verify server-side
    private var cachedStatus: PremiumStatus?
    private var lastVerified: Date?
    private let cacheExpiry: TimeInterval = 300  // Cache expires in 5 minutes
    
    struct PremiumStatus: Codable {
        let isPremium: Bool
        let expiresAt: Date?
        let signature: String  // Server-signed verification token
    }
    
    init() {
        let pinning = CertificatePinningDelegate()
        self.session = URLSession(
            configuration: .default,
            delegate: pinning,
            delegateQueue: nil
        )
    }
    
    /// Verify premium status
    func verifyPremium() async throws -> Bool {
        // 1. Return cache if still valid
        if let cached = cachedStatus,
           let lastCheck = lastVerified,
           Date().timeIntervalSince(lastCheck) < cacheExpiry {
            return cached.isPremium
        }
        
        // 2. Get token from Keychain
        guard let tokenData = try? keychain.load(key: "auth_token"),
              let token = String(data: tokenData, encoding: .utf8) else {
            return false  // No token = unauthenticated = not premium
        }
        
        // 3. Query server (with certificate pinning)
        var request = URLRequest(url: URL(string: "https://api.example.com/v1/verify-premium")\!)
        request.setValue("Bearer \(token)", forHTTPHeaderField: "Authorization")
        request.setValue(generateDeviceFingerprint(), forHTTPHeaderField: "X-Device-Fingerprint")
        
        let (data, response) = try await session.data(for: request)
        
        guard let httpResponse = response as? HTTPURLResponse,
              httpResponse.statusCode == 200 else {
            // If server is unreachable, deny by default
            // Prevents airplane mode bypass
            return cachedStatus?.isPremium ?? false
        }
        
        let status = try JSONDecoder().decode(PremiumStatus.self, from: data)
        
        // 4. Verify server signature (tamper detection)
        guard verifySignature(status) else {
            return false
        }
        
        // 5. Update cache
        cachedStatus = status
        lastVerified = Date()
        
        return status.isPremium
    }
    
    /// Generate device fingerprint (prevents replay attacks)
    private func generateDeviceFingerprint() -> String {
        let vendorID = UIDevice.current.identifierForVendor?.uuidString ?? "unknown"
        let timestamp = Int(Date().timeIntervalSince1970)
        let nonce = UUID().uuidString
        return "\(vendorID):\(timestamp):\(nonce)"
    }
    
    /// Verify server signature (simplified)
    private func verifySignature(_ status: PremiumStatus) -> Bool {
        // Use HMAC-SHA256 or RSA signatures in production
        return \!status.signature.isEmpty
    }
}

// Usage
let verifier = PremiumVerifier()
let isPremium = try await verifier.verifyPremium()
if isPremium {
    print("✅ Premium verification complete")
    // Expected output: ✅ Premium verification complete
} else {
    print("🔒 Premium content requires upgrade")
}
```

Three key principles make this architecture work. First, always verify server-side — never trust local flags alone. Second, deny by default when offline. Third, verify response signatures to prevent man-in-the-middle tampering.
