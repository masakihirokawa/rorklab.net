## Introduction

Live Activities and Dynamic Island are two of the most impactful UI features Apple has shipped in recent years. Since iOS 16.1, apps can surface real-time data — delivery countdowns, sports scores, ride-sharing ETAs — right on the Lock Screen and in the Dynamic Island pill at the top of the display. Users never need to open your app to stay informed.

The traditional challenge? Implementing ActivityKit in Swift has been a multi-day project: you need to define `ActivityAttributes`, build separate SwiftUI views for each Dynamic Island presentation mode, wire up push token handling for server-side updates, and keep everything in sync with your app's state. That complexity has kept many indie developers from shipping the feature.

Rork Max changes the equation. Because it generates native Swift code from plain-language prompts, you can have a working Live Activities integration — including all three Dynamic Island modes — in a few focused prompt iterations. This guide walks you through the complete process using a food delivery app as a concrete, testable example.

## How Live Activities and Dynamic Island Work

### Live Activities

Live Activities use Apple's **ActivityKit** framework to display a persistent, real-time widget on the Lock Screen and Notification Center. They're designed for tasks with a clear start and end — a 30-minute food delivery, a 90-minute match, a Pomodoro session. Apple caps them at 8 hours maximum.

Each Live Activity is defined by an `ActivityAttributes` struct with two distinct parts:

- **Static attributes** — data that never changes during the activity (restaurant name, order ID)
- **ContentState** — mutable data pushed as updates (delivery status, ETA, driver name)

### Dynamic Island

The Dynamic Island (iPhone 14 Pro and later) takes the same `ActivityAttributes` but renders three distinct UI modes:

| Mode | When shown | Approximate size |
|------|-----------|-----------------|
| **Compact** | Another app is in the foreground | ~120×36pt |
| **Minimal** | Two activities running simultaneously | Very small pill |
| **Expanded** | User long-presses the island | Up to 160pt tall |

All three views share the same live `ContentState`, so every update to the activity automatically propagates across every surface. You manage one data model — ActivityKit handles the rest.

## Step-by-Step Implementation

### Step 1: Review the Generated ActivityAttributes

Rork Max produces an `ActivityAttributes` conformance similar to this:

```swift
import ActivityKit
import Foundation

struct DeliveryActivityAttributes: ActivityAttributes {
    // Mutable real-time state (pushed on each update)
    public struct ContentState: Codable, Hashable {
        var status: DeliveryStatus    // Current delivery stage
        var driverName: String        // Assigned driver
        var estimatedArrival: Date    // Dynamic ETA
        var currentStep: Int          // 0–3, drives progress bar UI
    }

    // Static order metadata (set once at activity start)
    var restaurantName: String
    var orderId: String
}

enum DeliveryStatus: String, Codable {
    case preparing  = "Preparing"
    case pickedUp   = "Picked Up"
    case delivering = "On the Way"
    case delivered  = "Delivered"

    var sfSymbol: String {
        switch self {
        case .preparing:  return "fork.knife"
        case .pickedUp:   return "bag.fill"
        case .delivering: return "bicycle"
        case .delivered:  return "checkmark.circle.fill"
        }
    }
}
```

**Expected output:** The file compiles without errors. `ContentState` conforms to both `Codable` and `Hashable` — both are required by ActivityKit.

### Step 3: Implement the Full Activity Lifecycle

The generated `OrderViewModel.swift` handles starting, updating, and ending the activity. Here are the key methods:

```swift
// Call this when the user confirms their order
func startTracking(order: Order) async {
    guard ActivityAuthorizationInfo().areActivitiesEnabled else {
        print("⚠️ Live Activities not authorized")
        return
    }

    let attrs = DeliveryActivityAttributes(
        restaurantName: order.restaurant,
        orderId: order.id
    )
    let state = DeliveryActivityAttributes.ContentState(
        status: .preparing,
        driverName: "Finding a driver…",
        estimatedArrival: Date().addingTimeInterval(1800), // 30 min
        currentStep: 0
    )

    do {
        currentActivity = try Activity.request(
            attributes: attrs,
            content: .init(state: state, staleDate: nil)
        )
        print("✅ Activity started: \(currentActivity!.id)")
    } catch {
        print("❌ Failed to start activity: \(error)")
    }
}

// Call this on each status update from your backend
func updateStatus(_ status: DeliveryStatus, eta: Date, driver: String) async {
    let newState = DeliveryActivityAttributes.ContentState(
        status: status,
        driverName: driver,
        estimatedArrival: eta,
        currentStep: status == .preparing ? 0 :
                     status == .pickedUp ? 1 :
                     status == .delivering ? 2 : 3
    )
    await currentActivity?.update(.init(state: newState, staleDate: nil))
}

// Call this when the order is delivered
func endTracking() async {
    // Dismiss the activity 5 seconds after completion
    await currentActivity?.end(nil, dismissalPolicy: .after(Date().addingTimeInterval(5)))
}
```

## Common Errors and How to Fix Them

**"NSSupportsLiveActivities missing from Info.plist"**

Prompt Rork Max: *"Add NSSupportsLiveActivities set to true in Info.plist."* This is a required entitlement that Rork Max sometimes omits on first generation.

**Dynamic Island not visible in simulator**

Select an iPhone 14 Pro (or later) simulator. Dynamic Island does not exist on earlier hardware models or their simulators.

**Activity.request() throws immediately**

Usually a `Codable` conformance issue in `ContentState`. If any property in the struct doesn't fully conform to `Codable` — including nested enums without raw values — ActivityKit throws at runtime. Prompt: *"Fix any Codable conformance issues in DeliveryActivityAttributes.ContentState."*

**ETA timer shows 0:00 instead of counting down**

`Text(date, style: .timer)` requires the date to be in the future at render time. If your ETA calculation produces a past date, the timer shows `0:00`. Double-check that `Date().addingTimeInterval(...)` produces a future timestamp.

## Advanced: Server-Driven Updates via APNs

Once your backend is set up, you can update Live Activities server-side via Apple Push Notification service (APNs) — no app wake-up required. Ask Rork Max:


The generated code retrieves the activity's `pushToken` and shows the expected JSON payload shape, which your server sends to the APNs Live Activities endpoint. This is the production pattern used by apps like Uber and DoorDash to push real-time location updates directly to the Lock Screen.

## FAQ

**Do Live Activities work on Android?**

No — Live Activities and Dynamic Island are iOS-only features. For Android users, Rork (the standard React Native version) supports foreground service notifications with custom layouts, which serve a similar purpose.

**What's the maximum duration for a Live Activity?**

Apple enforces an 8-hour maximum. After that, the system automatically dismisses the activity. Always call `activity.end()` when the underlying task completes — leaving a stale delivery tracker on someone's Lock Screen is poor UX.

**Can I test Dynamic Island without a physical device?**

Yes. Xcode simulators running iOS 16.1+ correctly preview the Lock Screen Live Activity UI. For Dynamic Island specifically, you need an iPhone 14 Pro (or later) simulator. Rork Max's browser-based streaming simulator also supports Lock Screen preview.
