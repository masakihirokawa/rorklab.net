## Why Your Rork App Needs Unit Tests

Rork and Rork Max are incredible for generating code fast — but verifying that the logic is actually correct is still on you as the developer. As your app grows and you add features, refactor code, or fix bugs, manually checking that nothing broke becomes impossible.

That's where **unit tests** come in. Adding tests to your Rork-generated app gives you:

- Early bug detection (catch issues while you're still in the same file)
- Confidence when refactoring (if tests pass, behavior hasn't changed)
- Living documentation (your tests show exactly how each function is supposed to work)
- CI/CD integration (automated quality gates on every push)

In this guide, you'll learn how to set up **Jest** and **React Native Testing Library (RNTL)** in a Rork Expo project, then write real tests for utility functions, components, and custom hooks.

If you're looking for end-to-end testing (running the full app on a real device), check out the [Rork Max E2E Testing Guide with Detox & Maestro](/articles/rork-dev/rork-max-e2e-testing-detox-maestro-guide).

## What Are Jest and React Native Testing Library?

### Jest

**Jest** is a JavaScript testing framework created by Meta (formerly Facebook). It comes pre-configured in Expo projects and handles four jobs at once:

- **Test runner** — executes your tests and reports results
- **Assertions** — `expect(value).toBe(...)` syntax for verifying behavior
- **Mocking** — replace external dependencies with controlled fakes
- **Snapshot testing** — detect unintended UI changes

### React Native Testing Library (RNTL)

**React Native Testing Library** helps you test components from a user's perspective — not internal implementation details. Instead of checking which functions were called internally, you verify what users actually see and interact with.

```bash
# Verify or install in your Expo project
npx expo install @testing-library/react-native @testing-library/jest-native
```

## Setting Up Your Test Environment

### Step 1 — Check package.json

Open your Rork-generated project and confirm these dependencies are present:

```json
{
  "devDependencies": {
    "jest": "^29.0.0",
    "jest-expo": "^51.0.0",
    "@testing-library/react-native": "^12.0.0",
    "@testing-library/jest-native": "^5.0.0"
  },
  "jest": {
    "preset": "jest-expo"
  }
}
```

If they're missing, install them:

```bash
npx expo install jest-expo jest @testing-library/react-native @testing-library/jest-native
```

### Step 2 — Configure jest.config.js

Create or verify `jest.config.js` at the project root:

```js
// jest.config.js
module.exports = {
  preset: 'jest-expo',
  setupFilesAfterFramework: ['@testing-library/jest-native/extend-expect'],
  transformIgnorePatterns: [
    'node_modules/(?!((jest-)?react-native|@react-native(-community)?)|expo(nent)?|@expo(nent)?/.*|@expo-google-fonts/.*|react-navigation|@react-navigation/.*|@unimodules/.*|unimodules|sentry-expo|native-base|react-native-svg)',
  ],
  moduleFileExtensions: ['ts', 'tsx', 'js', 'jsx'],
  testPathPattern: '(/__tests__/.*|(\\.|/)(test|spec))\\.[jt]sx?$',
};
```

> **Note**: The `transformIgnorePatterns` setting tells Jest which native Expo modules to transform. Without it, you'll get `SyntaxError` on most Expo imports.

### Step 3 — Decide Where to Put Test Files

Two common patterns:

- Co-locate test files with source: `Button.tsx` + `Button.test.tsx` in the same folder
- Separate `__tests__/` directory at the project root

For Rork projects, co-location is easier to navigate and keeps related files together.

```
src/
  components/
    Button.tsx
    Button.test.tsx    ← co-located test
  utils/
    formatPrice.ts
    formatPrice.test.ts
```

## Writing Unit Tests for Utility Functions

Start here — pure JavaScript functions with no UI are the easiest to test and the most valuable to cover.

### Example: Price Formatting Function

```ts
// src/utils/formatPrice.ts
export function formatPrice(price: number, currency: string = 'USD'): string {
  if (price < 0) {
    throw new Error('Price must be a non-negative number');
  }
  return new Intl.NumberFormat('en-US', {
    style: 'currency',
    currency,
  }).format(price);
}
```

```ts
// src/utils/formatPrice.test.ts
import { formatPrice } from './formatPrice';

describe('formatPrice', () => {
  test('formats a USD price correctly', () => {
    expect(formatPrice(9.99)).toBe('$9.99');
    // Expected output: $9.99
  });

  test('formats a JPY price correctly', () => {
    expect(formatPrice(1000, 'JPY')).toBe('¥1,000');
    // Expected output: ¥1,000
  });

  test('handles zero correctly', () => {
    expect(formatPrice(0)).toBe('$0.00');
  });

  test('throws an error for negative values', () => {
    expect(() => formatPrice(-100)).toThrow('Price must be a non-negative number');
  });
});
```

Run the tests:

```bash
npx jest src/utils/formatPrice.test.ts

# Expected output:
# PASS  src/utils/formatPrice.test.ts
#   formatPrice
#     ✓ formats a USD price correctly (3ms)
#     ✓ formats a JPY price correctly (1ms)
#     ✓ handles zero correctly
#     ✓ throws an error for negative values (1ms)
#
# Tests: 4 passed, 4 total
```

## Component Testing with React Native Testing Library

RNTL tests render a component in memory and let you query it like a real user would — by visible text, role, or label.

### Testing a Button Component

```tsx
// src/components/PrimaryButton.tsx
import React from 'react';
import { TouchableOpacity, Text, StyleSheet } from 'react-native';

type Props = {
  label: string;
  onPress: () => void;
  disabled?: boolean;
};

export function PrimaryButton({ label, onPress, disabled = false }: Props) {
  return (
    <TouchableOpacity
      style={[styles.button, disabled && styles.disabled]}
      onPress={onPress}
      disabled={disabled}
      accessibilityRole="button"
      accessibilityLabel={label}
    >
      <Text style={styles.label}>{label}</Text>
    </TouchableOpacity>
  );
}

const styles = StyleSheet.create({
  button: { backgroundColor: '#007AFF', padding: 16, borderRadius: 8 },
  disabled: { opacity: 0.5 },
  label: { color: '#fff', textAlign: 'center', fontWeight: '600' },
});
```

```tsx
// src/components/PrimaryButton.test.tsx
import React from 'react';
import { render, fireEvent } from '@testing-library/react-native';
import { PrimaryButton } from './PrimaryButton';

describe('PrimaryButton', () => {
  test('renders the label correctly', () => {
    const { getByText } = render(
      <PrimaryButton label="Submit" onPress={() => {}} />
    );
    // "Submit" text should be visible on screen
    expect(getByText('Submit')).toBeTruthy();
  });

  test('calls onPress when tapped', () => {
    const mockOnPress = jest.fn(); // mock function
    const { getByRole } = render(
      <PrimaryButton label="Submit" onPress={mockOnPress} />
    );

    fireEvent.press(getByRole('button'));

    // onPress should have been called exactly once
    expect(mockOnPress).toHaveBeenCalledTimes(1);
  });

  test('does not call onPress when disabled', () => {
    const mockOnPress = jest.fn();
    const { getByRole } = render(
      <PrimaryButton label="Submit" onPress={mockOnPress} disabled={true} />
    );

    fireEvent.press(getByRole('button'));

    // Should not fire because button is disabled
    expect(mockOnPress).not.toHaveBeenCalled();
  });
});
```

### Testing Async Components

For components that fetch data, use `waitFor` to let the async operation complete before asserting.

```tsx
// src/components/UserProfile.test.tsx
import React from 'react';
import { render, waitFor } from '@testing-library/react-native';
import { UserProfile } from './UserProfile';

// Mock the fetch API
global.fetch = jest.fn(() =>
  Promise.resolve({
    json: () => Promise.resolve({ name: 'Jane Smith', email: 'jane@example.com' }),
  })
) as jest.Mock;

describe('UserProfile', () => {
  test('displays user data after loading', async () => {
    const { getByText } = render(<UserProfile userId="123" />);

    // Wait up to 1000ms for the API response to render
    await waitFor(() => {
      expect(getByText('Jane Smith')).toBeTruthy();
    }, { timeout: 1000 });
  });
});
```

## Testing Custom Hooks

Custom hooks in your Rork app can be tested independently using `renderHook`.

```ts
// src/hooks/useCounter.ts
import { useState, useCallback } from 'react';

export function useCounter(initialValue: number = 0) {
  const [count, setCount] = useState(initialValue);
  const increment = useCallback(() => setCount(c => c + 1), []);
  const decrement = useCallback(() => setCount(c => c - 1), []);
  const reset = useCallback(() => setCount(initialValue), [initialValue]);
  return { count, increment, decrement, reset };
}
```

```ts
// src/hooks/useCounter.test.ts
import { renderHook, act } from '@testing-library/react-native';
import { useCounter } from './useCounter';

describe('useCounter', () => {
  test('starts at the default value of 0', () => {
    const { result } = renderHook(() => useCounter());
    expect(result.current.count).toBe(0);
  });

  test('increments the count', () => {
    const { result } = renderHook(() => useCounter());
    act(() => {
      result.current.increment();
    });
    expect(result.current.count).toBe(1);
  });

  test('decrements the count', () => {
    const { result } = renderHook(() => useCounter(5));
    act(() => {
      result.current.decrement();
    });
    expect(result.current.count).toBe(4);
  });

  test('resets to initial value', () => {
    const { result } = renderHook(() => useCounter(10));
    act(() => {
      result.current.increment();
      result.current.increment();
      result.current.reset();
    });
    expect(result.current.count).toBe(10);
  });
});
```

## Common Errors and Fixes

### Error 1: `SyntaxError: Cannot use import statement`

**Cause**: Jest can't transform ES Modules from a native dependency.

**Fix**: Add the problematic package to `transformIgnorePatterns` in your config:

```js
// Add packages that need transformation
transformIgnorePatterns: [
  'node_modules/(?!((jest-)?react-native|@react-native(-community)?)|expo(nent)?|@expo(nent)?/.*|react-navigation|@react-navigation/.*|nativewind|react-native-reanimated)',
],
```

### Error 2: `Unable to find an element with the text`

**Cause**: Text case mismatch, or querying before async data loads.

**Fix**:

```tsx
// Case-insensitive match
getByText(/submit/i)

// Wait for async render
await waitFor(() => getByText('Submit'));
```

### Error 3: `act(...)` warnings in the console

**Cause**: State updates are happening outside of `act()`.

**Fix**: Wrap interactions and async operations:

```tsx
// Wrap in act
await act(async () => {
  fireEvent.press(button);
});
expect(mockFn).toHaveBeenCalled();
```

## Checking Test Coverage

See what percentage of your code is covered by tests:

```bash
# Generate a coverage report
npx jest --coverage

# Example output:
# ----------|---------|----------|---------|---------|
# File      | % Stmts | % Branch | % Funcs | % Lines |
# ----------|---------|----------|---------|---------|
# All files |   78.5  |   72.3   |   80.1  |   79.2  |
```

Add convenience scripts to `package.json`:

```json
{
  "scripts": {
    "test": "jest",
    "test:watch": "jest --watch",
    "test:coverage": "jest --coverage"
  }
}
```

A practical coverage target: aim for **80%+ on business logic and utility functions**, and cover the key interactions on your most important components.

## FAQ

**Q. Can I test Rork-generated code without modifications?**

A. Yes, in most cases. If the generated component uses native modules like `expo-camera` or `expo-location`, mock those modules with `jest.mock('expo-camera')`. The component logic itself will be testable without changes.

**Q. Should I use snapshot testing?**

A. Snapshot tests can catch unexpected UI changes, but in a Rork environment where AI frequently regenerates code, snapshots break often and become a maintenance burden. Prefer behavioral tests — "does this text appear?", "does this button call the right function?" — over snapshot comparisons.

**Q. I'm getting TypeScript errors in my test files**

A. Add `"types": ["jest", "@testing-library/jest-native"]` to `compilerOptions` in `tsconfig.json`. Also confirm that `@types/jest` is in your devDependencies.

**Q. How do I run tests automatically in CI/CD?**

A. Add this step to your GitHub Actions workflow:

```yaml
- name: Run tests
  run: npx jest --ci --coverage
```
