## Introduction: What is Expo Application Services (EAS)?

Expo Application Services (EAS) is a managed cloud platform that automates the build and deployment of Rork Max applications—React Native / Expo-based mobile apps—without requiring local macOS, Xcode, or Android Studio environments. Traditionally, iOS builds demanded macOS with Xcode, while Android builds required Android Studio. With EAS, developers can build both iOS and Android apps from any machine: Linux servers, Windows PCs, or GitHub Actions runners.

### Why EAS Matters for Rork Max Development

1. **Eliminates Local Environment Complexity**: No need to install and maintain Xcode, Android Studio, or Xcode command-line tools
2. **Accelerates Release Cycles**: Automated CI/CD pipelines reduce time from commit to production deployment
3. **Over-The-Air (OTA) Updates**: Push JavaScript bundle fixes without waiting for App Store or Google Play review
4. **Streamlines Team Testing**: Preview builds distributed via QR codes enable parallel QA testing
5. **Production Reliability**: Automatic rollback capabilities, environment isolation, and multi-profile support ensure stable releases

This guide provides a complete, production-ready implementation of Rork Max with EAS CI/CD, including real-world code examples and best practices.

## Chapter 1: EAS Setup and Initial Configuration

### Step 1: Install EAS CLI Globally

```bash
npm install -g eas-cli
```

Verify installation:

```bash
eas --version
```

### Step 2: Authenticate with Expo Account

```bash
eas login
```

You'll be prompted to open a browser and log in to your Expo account. If you don't have one, sign up for free at [expo.dev](https://expo.dev).

<Callout type="info">
**Pro Tip**: For CI/CD environments (GitHub Actions, GitLab CI), generate an API token at [expo.dev/settings/access-tokens](https://expo.dev/settings/access-tokens) and store it as the `EXPO_TOKEN` environment variable. This enables passwordless authentication across all pipelines.
</Callout>

### Step 3: Initialize EAS Build Configuration

```bash
eas build:configure
```

This command creates an `eas.json` configuration file in your project root. Below is a comprehensive example with development, preview, and production profiles.

### Complete eas.json Configuration

```json
{
  "cli": {

  },
  "build": {
    "development": {

        "enterpriseProvisioning": "universal-adhoc"

        "gradleCommand": ":app:assembleDebug"
      }
    },
    "preview": {
      "distribution": "internal",
      "channel": "preview",
      "ios": {
        "buildType": "ad-hoc",
        "provisioning": {
          "teamId": "YOUR_APPLE_TEAM_ID"
        }
      },
      "android": {
        "buildType": "apk",
        "gradleCommand": ":app:assembleRelease"
      }
    },
    "production": {
      "distribution": "store",
      "channel": "production",
      "ios": {
        "buildType": "app-store",
        "provisioning": {
          "teamId": "YOUR_APPLE_TEAM_ID"
        }
      },
      "android": {
        "buildType": "aab",
        "gradleCommand": ":app:bundleRelease"
      }
    }
  },
  "submit": {

      "ios": {
        "appleId": "your-email@example.com",

```

### Profile Breakdown

- **development**: For local testing. Internal distribution via Expo Go
- **preview**: For QA teams. Builds can be shared via QR code or internal testing tracks
- **production**: For App Store / Google Play releases. Generates AAB (Android App Bundle) and App Store-ready iOS builds

## Chapter 2: EAS Build for iOS

### iOS Prerequisites

You'll need an Apple Developer Account to build for iOS. Sign up at [developer.apple.com](https://developer.apple.com) and enroll in the Apple Developer Program ($99/year).

### Step 1: Configure iOS Credentials

```bash
eas credentials
```

This interactive command securely stores:
- Apple ID email
- App-specific password (recommended over account password)
- Development and distribution certificates
- Provisioning profiles

<Callout type="info">
**Security First**: Never commit credentials to version control. EAS encrypts and manages all credentials server-side. For CI/CD, use `EXPO_TOKEN` instead of your Apple credentials.
</Callout>

### Step 2: Create Your First iOS Build

```bash
eas build --platform ios --profile production
```

EAS queues the build and displays a build ID. Monitor progress:

```bash
eas build --status
```

Upon completion, you'll receive a .ipa file ready for TestFlight submission.

### Step 3: Submit to TestFlight

```bash
eas submit --platform ios --profile production
```

The app automatically uploads to App Store Connect and becomes available to TestFlight internal testers within minutes.

### iOS Configuration Best Practices

```json
{
  "build": {
    "production": {
      "ios": {
        "buildType": "app-store",
        "provisioning": {
          "teamId": "YOUR_APPLE_TEAM_ID"
        },
        "entitlements": {
          "com.apple.developer.networking.vpn": [
            "allow"
          ],
          "com.apple.developer.pass-type-identifiers": [
            "pass.com.example.rorkmax"
          ]
        },
        "buildNumberPattern": "{{YEAR}}.{{MONTH}}.{{DAY}}.{{BUILD_NUMBER}}"
      }
    }
  }
}
```

### Troubleshooting iOS Builds

If builds fail, check:

```bash
# List all builds with details
eas build --list --limit=10

# View logs for a specific build
eas build --id=BUILD_ID --logs
```

Common issues:
- **Certificate expired**: Re-run `eas credentials` to regenerate
- **Provisioning profile mismatch**: Ensure bundle ID matches provisioning profile
- **Xcode version conflict**: EAS automatically uses the latest stable Xcode version

## Chapter 3: EAS Build for Android

### Android Prerequisites

You'll need a Google Play Developer Account to publish apps. Sign up at [play.google.com/console](https://play.google.com/console) ($25 one-time fee).

### Step 1: Configure Android Keystore

```bash
eas credentials
```

EAS can auto-generate a keystore or import an existing one.

#### Manual Keystore Creation (if needed)

```bash
keytool -genkey -v -keystore my-release-key.keystore \
  -keyalg RSA -keysize 2048 -validity 10000 \
  -alias my-key-alias \
  -storepass mypassword \
  -keypass mypassword
```

Register the keystore with EAS when prompted.

### Step 2: Build APK or AAB

```bash
# APK build (for testing/sideloading)
eas build --platform android --profile preview

# App Bundle (for Google Play)
eas build --platform android --profile production
```

### Android Configuration

```json
{
  "build": {
    "production": {
      "android": {
        "buildType": "aab",
        "gradleCommand": ":app:bundleRelease",
        "env": {
          "TARGET_SDK_VERSION": "34",
          "MIN_SDK_VERSION": "24"
        }
      }
    },
    "preview": {
      "android": {
        "buildType": "apk",
        "gradleCommand": ":app:assembleRelease"
      }
    }
  }
}
```

### Step 3: Submit to Google Play

```bash
eas submit --platform android --profile production
```

First, set up a Google Play service account:

1. Open [Google Cloud Console](https://console.cloud.google.com)
2. Create a new project
3. Enable **Google Play Android Developer API**
4. Create a service account with Editor role
5. Download the JSON key file
6. Configure in `eas.json`:

```json
{
  "submit": {
    "production": {
      "android": {
        "serviceAccount": "./path/to/service-account.json",
        "track": "production"
      }
    }
  }
}
```

### Android Multi-Language Support

For apps supporting multiple regions:

```json
{
  "build": {
    "production": {
      "android": {
        "buildType": "aab",
        "env": {
          "LANGUAGES": "en,ja,es,fr,de"
        }
      }
    }
  }
}
```

## Chapter 4: EAS Update — OTA (Over-The-Air) Deployments

### What OTA Updates Can Change

EAS Update distributes changes to JavaScript code and static assets without requiring new app store submissions.

**Supported Updates**:
- React Native / Rork Max business logic
- UI components and screens
- API calling logic
- StyleSheet and CSS-in-JS styles
- Image and font assets

**Not Supported**:
- New native modules
- Permission changes
- SDK version upgrades
- Compiled native code

### Deploy an OTA Update

```bash
eas update --branch production --message "Critical fix: payment processing crash"
```

This command:
1. Builds the JavaScript bundle
2. Uploads to EAS servers
3. Immediately distributes to all users on the `production` branch

### Detailed Update with Metadata

```bash
eas update --branch production \
  --message "v1.2.1: Improved dashboard performance" \
  --metadata "{\"environment\": \"production\", \"severity\": \"high\"}"
```

### Multi-Branch Release Strategy

```bash
# Canary release to 10% of users
eas update --branch production \
  --message "Experimental: new checkout flow" \
  --percent 10

# Promote to 50% after monitoring
eas update --branch production \
  --message "Expanding rollout to 50%" \
  --percent 50

# Full rollout
eas update --branch production \
  --message "Complete rollout of new checkout" \
  --percent 100
```

### Rollback to Previous Release

```bash
# View update history
eas update --list

# Rollback to a specific update ID
eas update --branch production --rollback-to-release-id=abc123def456
```

<Callout type="info">
**Production Practice**: Always validate OTA updates on a staging branch before production rollout. If issues are detected, rolling back takes seconds using `--rollback-to-release-id`.
</Callout>

### Scheduled Updates

For non-breaking updates, schedule them to deploy during off-peak hours:

```bash
eas update --branch production \
  --message "Scheduled update: 2:00 AM PT" \
  --scheduled
```

## Chapter 5: GitHub Actions CI/CD Pipeline

### Complete GitHub Actions Workflow

Create `.github/workflows/eas-ci.yml`:

```yaml
name: Rork Max EAS CI/CD Pipeline

on:
  push:
    branches:
      - main
      - develop
  pull_request:
    branches:
      - main
  workflow_dispatch:

concurrency:
  group: ${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: true

jobs:
  test:
    name: Lint & Unit Tests
    runs-on: ubuntu-latest
    timeout-minutes: 15
    steps:
      - name: Checkout repository
        uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: "18.x"
          cache: "npm"

      - name: Install dependencies
        run: npm ci

      - name: Run ESLint
        run: npm run lint -- --format json --output-file eslint-report.json || true
        continue-on-error: true

      - name: Run tests with coverage
        run: npm run test -- --coverage --silent

      - name: Upload coverage to Codecov
        uses: codecov/codecov-action@v3
        with:
          files: ./coverage/coverage-final.json
          fail_ci_if_error: false

  build-preview:
    name: Build Preview (iOS & Android)
    runs-on: ubuntu-latest
    needs: test
    if: github.event_name == 'push' && github.ref == 'refs/heads/develop'
    timeout-minutes: 45
    steps:
      - name: Checkout repository
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: "18.x"
          cache: "npm"

      - name: Install EAS CLI
        run: npm install -g eas-cli

      - name: Install dependencies
        run: npm ci

      - name: Build iOS Preview
        id: ios-build
        run: |
          eas build \
            --platform ios \
            --profile preview \
            --non-interactive \
            --wait
        env:
          EXPO_TOKEN: ${{ secrets.EXPO_TOKEN }}

      - name: Build Android Preview
        id: android-build
        run: |
          eas build \
            --platform android \
            --profile preview \
            --non-interactive \
            --wait
        env:
          EXPO_TOKEN: ${{ secrets.EXPO_TOKEN }}

      - name: Comment on PR with preview links
        if: github.event_name == 'pull_request'
        uses: actions/github-script@v7
        with:
          github-token: ${{ secrets.GITHUB_TOKEN }}
          script: |
            github.rest.issues.createComment({
              issue_number: context.issue.number,
              owner: context.repo.owner,
              repo: context.repo.repo,
              body: `## ✅ Preview Builds Ready\n\n📱 **Download Preview App:**\n- [iOS Preview](https://expo.dev/accounts/rorklab/projects/rorkmax)\n- [Android Preview](https://expo.dev/accounts/rorklab/projects/rorkmax)\n\n📖 Scan QR code in Expo Go to test changes.`
            })

  build-and-deploy-production:
    name: Build Production & Deploy
    runs-on: ubuntu-latest
    needs: test
    if: github.event_name == 'push' && github.ref == 'refs/heads/main'
    timeout-minutes: 60
    steps:
      - name: Checkout repository
        uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: "18.x"
          cache: "npm"

      - name: Install EAS CLI
        run: npm install -g eas-cli

      - name: Install dependencies
        run: npm ci

      - name: Get version
        id: version
        run: |
          VERSION=$(node -p "require('./package.json').version")
          COMMIT_SHA=${{ github.sha }}
          echo "VERSION=${VERSION}" >> $GITHUB_OUTPUT
          echo "COMMIT_SHA=${COMMIT_SHA:0:7}" >> $GITHUB_OUTPUT

      - name: Build iOS Production
        id: ios-prod
        run: |
          eas build \
            --platform ios \
            --profile production \
            --non-interactive \
            --wait
        env:
          EXPO_TOKEN: ${{ secrets.EXPO_TOKEN }}

      - name: Build Android Production
        id: android-prod
        run: |
          eas build \
            --platform android \
            --profile production \
            --non-interactive \
            --wait
        env:
          EXPO_TOKEN: ${{ secrets.EXPO_TOKEN }}

      - name: Deploy OTA Update
        id: ota-deploy
        run: |
          eas update \
            --branch production \
            --message "Release v${{ steps.version.outputs.VERSION }} (${{ steps.version.outputs.COMMIT_SHA }})" \
            --non-interactive
        env:
          EXPO_TOKEN: ${{ secrets.EXPO_TOKEN }}

      - name: Notify Slack - Success
        if: success()
        uses: slackapi/slack-github-action@v1.24.0
        with:
          webhook-url: ${{ secrets.SLACK_WEBHOOK_URL }}
          payload: |
            {
              "text": "🚀 Rork Max Production Deployment Successful",
              "blocks": [
                {
                  "type": "section",
                  "text": {
                    "type": "mrkdwn",
                    "text": "*✅ Status*: Success\n*Version*: v${{ steps.version.outputs.VERSION }}\n*Commit*: <${{ github.server_url }}/${{ github.repository }}/commit/${{ github.sha }}|${{ steps.version.outputs.COMMIT_SHA }}>\n*Branch*: main"
                  }
                }
              ]
            }

      - name: Notify Slack - Failure
        if: failure()
        uses: slackapi/slack-github-action@v1.24.0
        with:
          webhook-url: ${{ secrets.SLACK_WEBHOOK_URL }}
          payload: |
            {
              "text": "❌ Rork Max Production Deployment Failed",
              "blocks": [
                {
                  "type": "section",
                  "text": {
                    "type": "mrkdwn",
                    "text": "*❌ Status*: Failed\n*Version*: v${{ steps.version.outputs.VERSION }}\n*Commit*: <${{ github.server_url }}/${{ github.repository }}/commit/${{ github.sha }}|${{ steps.version.outputs.COMMIT_SHA }}>"
                  }
                }
              ]
            }

  security-scan:
    name: Security & Dependency Check
    runs-on: ubuntu-latest
    timeout-minutes: 10
    steps:
      - name: Checkout repository
        uses: actions/checkout@v4

      - name: Run npm audit
        run: npm audit --audit-level=moderate || true

      - name: Check for vulnerable dependencies
        uses: dependabot/dependabot-cli@v1
        with:
          command: update
          directory: .
          allow-conditions: "(semver:~1.0.0)"
```

### GitHub Secrets Configuration

In your repository, go to **Settings > Secrets and variables > Actions** and add:

| Secret Name | Value |
|---|---|
| `EXPO_TOKEN` | [From expo.dev/settings/access-tokens](https://expo.dev/settings/access-tokens) |
| `SLACK_WEBHOOK_URL` | Your Slack incoming webhook (optional) |

## Chapter 6: Environment Variables and Secrets

### Create EAS Secrets

```bash
eas secret:create
```

This securely stores sensitive values like API keys, analytics tokens, and credentials on EAS servers.

### Define Secrets per Profile

```bash
# Production API key
eas secret:create --scope production --name API_KEY_PROD --value "sk_live_..."

# Staging API key
eas secret:create --scope preview --name API_KEY_STAGING --value "sk_test_..."
```

### Access in React Native Code

```javascript
import Constants from 'expo-constants';

export const config = {
  apiKey: Constants.expoConfig?.extra?.apiKey || '',
  environment: Constants.expoConfig?.extra?.environment || 'development',
  logLevel: Constants.expoConfig?.extra?.logLevel || 'warn',
};

// Usage in API calls
const headers = {
  'Authorization': `Bearer ${config.apiKey}`,
  'X-Environment': config.environment,
};
```

### Configure per Profile in eas.json

```json
{
  "build": {
    "development": {
      "env": {
        "API_BASE_URL": "http://localhost:3000",
        "ENVIRONMENT": "development",
        "LOG_LEVEL": "debug"
      }
    },
    "preview": {
      "env": {
        "API_BASE_URL": "https://staging-api.rork.dev",
        "ENVIRONMENT": "staging",
        "LOG_LEVEL": "info"
      }
    },
    "production": {
      "env": {
        "API_BASE_URL": "https://api.rork.dev",
        "ENVIRONMENT": "production",
        "LOG_LEVEL": "error"
      }
    }
  }
}
```

### GitHub Actions Environment Variables

In your workflow, pass secrets to EAS commands:

```yaml
- name: Deploy OTA Update
  env:
    API_KEY: ${{ secrets.PRODUCTION_API_KEY }}
    ANALYTICS_TOKEN: ${{ secrets.ANALYTICS_TOKEN }}
  run: eas update --branch production --non-interactive
```

<Callout type="info">
**Best Practice**: Never log secrets or pass them in command-line arguments. Use EAS CLI's built-in secret management or GitHub's encrypted environment variables exclusively.
</Callout>

## Chapter 7: Preview Builds & QA Testing

### Generate a Preview Build

```bash
eas build --platform ios --profile preview --non-interactive
```

Once complete, EAS generates a QR code. iOS testers can scan it with Expo Go to instantly test the app.

### Share Preview with Team

```bash
# Build with a descriptive message
eas build \
  --platform ios \
  --profile preview \
  --message "QA: Payment flow iteration 3" \
  --non-interactive \
  --wait
```

### Android Preview Testing

```bash
eas build --platform android --profile preview
```

The resulting APK can be distributed directly to Android testers, or uploaded to Google Play's internal testing track.

### Preview with Debug Menu

For development testing, enable the Expo dev menu:

```json
{
  "build": {
    "preview": {
      "env": {
        "EXPO_DEBUG": "true",
        "LOG_LEVEL": "debug"
      }
    }
  }
}
```

This allows testers to access performance metrics, network request logs, and reload the app without rebuilding.

## Chapter 9: Production Release Checklist

Before deploying Rork Max to production, verify:

### Pre-Build Verification

- [ ] `eas.json` production profile configured with correct settings
- [ ] Apple Team ID and Google Play service account JSON registered
- [ ] All required environment variables and secrets created in EAS
- [ ] iOS certificates and provisioning profiles valid (not expired)
- [ ] Android keystore properly configured and backed up
- [ ] `app.json` version matches intended release version

### Testing Phase

- [ ] Preview build tested on iOS and Android devices
- [ ] End-to-end tests pass (login, API calls, payments)
- [ ] Staging API integration verified
- [ ] Network error handling (offline, slow connection) tested
- [ ] Performance profiling shows acceptable metrics
- [ ] Crash analytics (Sentry, Crashlytics) configured

### Release Execution

- [ ] Git main branch updated with final code
- [ ] `npm run lint && npm run test` passes locally and in CI
- [ ] Version numbers updated in `package.json` and `app.json`
- [ ] Release notes drafted and reviewed
- [ ] Team members notified of pending deployment

### Post-Deployment Monitoring

- [ ] Crash rate monitored in Crashlytics/Sentry (< 0.1% acceptable)
- [ ] Analytics dashboard shows expected session count
- [ ] User reviews on App Store / Google Play reviewed
- [ ] Support queue monitored for new issue reports
- [ ] Rollback strategy prepared if critical bugs detected

### Rollback Procedure

If critical issues emerge:

```bash
# View recent updates
eas update --list --limit=5

# Rollback to previous version immediately
eas update --branch production --rollback-to-release-id=PREVIOUS_ID
```
