After building your app with Rork Max and implementing Apple in-app purchases, you might find that Sandbox testing passed but production payments are failing, or the restore purchases button silently does nothing, or App Store review rejected your billing flow for reasons that aren't immediately obvious.

Apple's payment system involves StoreKit's evolving APIs, a Sandbox environment with intentional quirks, and review guidelines that are stricter about billing flows than almost anything else. This guide covers the most common error patterns in Rork Max in-app purchase implementations, with Swift code and review strategy for each.

## StoreKit 2 Error Codes and Handling

Rork Max generates billing code using StoreKit 2's async/await API. Errors surface as `StoreKitError` or `Product.PurchaseError` types.

### Handling Product.PurchaseError

```swift
import StoreKit

func purchase(_ product: Product) async {
    do {
        let result = try await product.purchase()

        switch result {
        case .success(let verification):
            switch verification {
            case .verified(let transaction):
                // ✅ Purchase confirmed
                await transaction.finish()
                await updatePurchasedProducts(transaction)

            case .unverified(let transaction, let error):
                // ⚠️ JWS signature verification failed
                // In production, deny access and request support contact
                print("Verification failed: \(error)")
                showError("We couldn't verify your purchase. Please contact support.")
            }

        case .userCancelled:
            // Intentional cancellation — not an error
            print("User cancelled purchase")

        case .pending:
            // Waiting for parental approval (Ask to Buy) or payment method
            showInfo("Your purchase is awaiting approval")

        @unknown default:
            break
        }

    } catch let error as StoreKitError {
        handleStoreKitError(error)
    } catch {
        print("Unexpected error: \(error)")
    }
}

func handleStoreKitError(_ error: StoreKitError) {
    switch error {
    case .notEntitled:
        // User has a purchase but isn't entitled — needs restore
        showError("Please restore your purchases to access this content")

    case .networkError(let urlError):
        if urlError.code == .notConnectedToInternet {
            showError("Please check your internet connection")
        } else {
            showError("A network error occurred. Please try again later")
        }

    case .systemError(let nsError):
        // App Store-side issue
        print("System error: \(nsError)")
        showError("There's a problem communicating with the App Store")

    case .userCancelled:
        break  // Not an error

    case .notAvailableInStorefront:
        showError("This product isn't available in your region")

    case .purchaseNotAllowed:
        // Airplane mode, parental controls, etc.
        showError("Purchases aren't allowed under your current settings")

    @unknown default:
        showError("An unexpected error occurred")
    }
}
```

### Using Transaction.currentEntitlements

In StoreKit 2, use `Transaction.currentEntitlements` rather than manual receipt validation to check subscription status.

```swift
class SubscriptionManager: ObservableObject {
    @Published var isSubscribed = false
    @Published var subscriptionExpiry: Date?

    func checkSubscriptionStatus() async {
        for await result in Transaction.currentEntitlements {
            switch result {
            case .verified(let transaction):
                if transaction.productType == .autoRenewable {
                    isSubscribed = true
                    subscriptionExpiry = transaction.expirationDate

                    // Check for expired subscriptions
                    if let expiry = transaction.expirationDate, expiry < Date() {
                        isSubscribed = false
                    }
                }
            case .unverified:
                isSubscribed = false
            }
        }
    }

    // Call on app launch
    func listenForTransactionUpdates() {
        Task {
            for await result in Transaction.updates {
                switch result {
                case .verified(let transaction):
                    await transaction.finish()
                    await checkSubscriptionStatus()
                case .unverified:
                    break
                }
            }
        }
    }
}
```

## Sandbox Environment Quirks

### Intentional Differences Between Sandbox and Production

The Sandbox environment has deliberate differences that cause confusion when you're unaware of them.

**Accelerated renewal intervals:** A monthly subscription renews every ~5 minutes in Sandbox. A yearly subscription renews every ~1 hour. This is intentional for faster testing.

**Maximum renewal count:** Sandbox subscriptions auto-renew a maximum of 6 times before stopping. Production subscriptions continue indefinitely.

**TestFlight uses Sandbox:** Apps distributed via TestFlight use the Sandbox payment environment. You cannot test live App Store payments through TestFlight.

### Sandbox Account Setup Errors

```
This Apple ID is not available in the App Store
```

This error appears when you try to use a real Apple ID for Sandbox testing, or when the Sandbox tester account isn't correctly configured.

**Correct setup process:**

1. Go to App Store Connect → Users and Access → Sandbox Testers
2. Click "+" to create a new Sandbox tester (use a non-existent email address)
3. On device: Settings → App Store → Sandbox Account, enter the Sandbox tester credentials

**Important:** Never use your real Apple ID for Sandbox testing. It creates confusing data mixing between your production account and test transactions.

### Restore Purchases Failing in Sandbox

```swift
// ❌ Common mistake: calling purchase() for "restore"
Button("Restore Purchases") {
    Task { try? await product.purchase() }  // This triggers a new purchase
}

// ✅ Correct restore implementation with StoreKit 2
Button("Restore Purchases") {
    Task { await restorePurchases() }
}

func restorePurchases() async {
    do {
        // AppStore.sync() is the correct StoreKit 2 restore method
        try await AppStore.sync()

        // Re-check status after sync
        await checkSubscriptionStatus()

        if isSubscribed {
            showSuccess("Your purchase has been restored")
        } else {
            showInfo("No purchases found to restore")
        }
    } catch {
        showError("Restore failed: \(error.localizedDescription)")
    }
}
```

## App Store Review Rejection Patterns

### Rejection Reason 1: Missing Restore Purchases Button (Guideline 3.1.1)

Every app with in-app purchases must include a "Restore Purchases" button. If the reviewer can't find it, you'll get rejected.

**Requirements to pass review:**
- Place a clearly visible "Restore Purchases" button on your purchase screen
- Position it near the purchase button — small or hidden placement is flagged
- The button must actually work and provide user feedback

### Rejection Reason 2: Core Functionality Blocked Without Purchase (Guideline 3.1.1a)

If your app appears to require a subscription before a user can access any functionality, it will be rejected.

```swift
// ❌ Showing paywall immediately on launch (rejection risk)
struct ContentView: View {
    var body: some View {
        if !isSubscribed {
            PurchaseRequiredView()  // Blocks all access
        }
    }
}

// ✅ Offer free basic functionality; gate only premium features
struct ContentView: View {
    var body: some View {
        MainTabView()  // Free features accessible immediately
            .sheet(isPresented: $showPremiumPrompt) {
                PremiumUpgradeView()
            }
    }
}
```

### Rejection Reason 3: Unclear Pricing or Terms (Guideline 3.1.1b)

Your purchase screen must clearly display price, billing period, free trial duration, and how to cancel.

```swift
struct SubscriptionView: View {
    let product: Product

    var body: some View {
        VStack(spacing: 16) {
            Text("Premium Plan")
                .font(.title2.bold())

            // ✅ Price and period clearly stated
            Text("\(product.displayPrice) / month")
                .font(.title3)

            // ✅ Free trial info is required if offered
            if let intro = product.subscription?.introductoryOffer {
                Text("First \(intro.period.value) \(intro.period.unit == .month ? "month" : "days") free")
                    .foregroundColor(.blue)
            }

            Button("Start Premium") {
                Task { await purchase(product) }
            }
            .buttonStyle(.borderedProminent)

            Button("Restore Purchases") {
                Task { await restorePurchases() }
            }
            .foregroundColor(.secondary)

            // ✅ Cancellation instructions required
            Link("How to cancel", destination: URL(string: "https://support.apple.com/en-us/118428")!)
                .font(.caption)

            // ✅ Terms and privacy policy links
            HStack {
                Link("Terms of Service", destination: URL(string: "https://yourapp.com/terms")!)
                Text("·")
                Link("Privacy Policy", destination: URL(string: "https://yourapp.com/privacy")!)
            }
            .font(.caption)
            .foregroundColor(.secondary)
        }
    }
}
```

### Rejection Reason 4: No Ask to Buy Handling

On devices with parental controls, purchases require parental approval. The `.pending` case must be handled or you risk rejection and a confusing user experience.

The `.pending` case in the `purchase()` function shown above is required — don't omit it.

## Production Error Monitoring

### Using RevenueCat

For production billing error tracking, RevenueCat is strongly recommended. It provides a dashboard with error rates, churn metrics, and MRR.

```swift
import RevenueCat

// Configure on launch
Purchases.configure(withAPIKey: "YOUR_REVENUECAT_API_KEY")

func purchaseWithRevenueCat(package: Package) async {
    do {
        let result = try await Purchases.shared.purchase(package: package)

        if result.userCancelled { return }
        updateUI(customerInfo: result.customerInfo)

    } catch let error as RevenueCat.ErrorCode {
        switch error {
        case .purchaseCancelledError:
            break
        case .networkError:
            showError("Network error. Please try again later.")
        case .purchaseNotAllowedError:
            showError("Purchases aren't allowed under your current settings.")
        case .receiptAlreadyInUseError:
            showError("This Apple ID is associated with a different account.")
        default:
            showError("Purchase failed: \(error.localizedDescription)")
        }
    }
}
```

### Custom Error Logging

If not using RevenueCat, implement at minimum a simple error logger:

```swift
struct PurchaseErrorLogger {
    static func log(error: Error, context: [String: Any] = [:]) {
        var logData: [String: Any] = [
            "error_type": String(describing: type(of: error)),
            "error_message": error.localizedDescription,
            "timestamp": ISO8601DateFormatter().string(from: Date()),
            "app_version": Bundle.main.infoDictionary?["CFBundleShortVersionString"] as? String ?? "unknown"
        ]
        logData.merge(context) { _, new in new }
        print("[Purchase Error] \(logData)")
        // Forward to Firebase Crashlytics, Sentry, etc.
    }
}
```

## Pre-Submission Checklist

Before submitting to App Store review, verify each of the following.

**UI/UX:** Does the purchase screen clearly show price, billing period, and free trial info? Is "Restore Purchases" prominently placed? Are Terms of Service and Privacy Policy linked? Can users access basic functionality without purchasing?

**Technical:** Is the `.pending` (Ask to Buy) case handled? Is the `.unverified` case handled appropriately? Have you tested purchase, restore, and renewal in Sandbox? Is `Transaction.updates` listener set up on launch?

**Compliance:** Is pricing shown correctly for each region (tax-inclusive in Japan)? Are cancellation instructions included?

## Wrapping Up

Most in-app purchase errors in Rork Max apps fall into three categories: StoreKit 2 error codes that need proper handling, Sandbox environment behavior that differs from production in expected ways, and App Store review guidelines that are strict about billing flows.

Revenue is directly at stake with billing features, so thorough testing and review preparation are worth the extra time. We hope this guide helps your app handle payments smoothly and pass review on the first submission.
