## The Difference Between Crashes and White Screens

When developing with Rork, it's important to distinguish between two separate failure modes:

- **Crash** — The app completely closes and returns to the home screen
- **White Screen** — The app is still running, but nothing renders to the user

Both are frustrating, but they have different root causes and solutions. This guide covers both scenarios.

---

## Top 5 Root Causes of App Crashes

### 1. Out of Memory (OOM)

**Symptom**: App crashes when loading many large images or scrolling through long lists, especially on older devices.

**Root Cause**: Loading high-resolution images directly into memory without resizing.

```typescript
// ❌ Wrong: loading full-resolution images
import { View, Image, FlatList } from "react-native";

export default function PhotoGallery({ photos }: { photos: string[] }) {
  return (
    <FlatList
      data={photos}
      renderItem={({ item }) => (
        <View>
          {/* Full resolution → memory exhaustion → crash */}
          <Image
            source={{ uri: item }}
            style={{ width: "100%", height: 400 }}
          />
        </View>
      )}
      keyExtractor={(item, idx) => idx.toString()}
    />
  );
}

// ✅ Correct: resize images and optimize rendering
export default function PhotoGallery({ photos }: { photos: string[] }) {
  return (
    <FlatList
      data={photos}
      renderItem={({ item }) => (
        <View>
          {/* Explicit dimensions + caching */}
          <Image
            source={{ uri: item }}
            style={{ width: 300, height: 300 }}
            defaultSource={require("./placeholder.png")}
          />
        </View>
      )}
      keyExtractor={(item, idx) => idx.toString()}
      maxToRenderPerBatch={10}  {/* Limit items rendered per batch */}
      removeClippedSubviews={true}  {/* Unrender off-screen items */}
    />
  );
}
```

**Solution**:
- Resize images to explicit dimensions
- Use `maxToRenderPerBatch` and `removeClippedSubviews` in FlatList
- Consider image caching libraries like `react-native-image-cache-hoc`

### 2. Infinite Loops and Infinite Re-rendering

**Symptom**: App starts then immediately becomes unresponsive, CPU usage hits 100%, and crashes within seconds.

**Root Cause**: Missing or incorrect dependency array in `useEffect`, or state updates triggering endless re-renders.

```typescript
// ❌ Wrong: useEffect with no dependency array
import { useState, useEffect } from "react";
import { View, Text } from "react-native";

export default function App() {
  const [count, setCount] = useState(0);

  useEffect(() => {
    // No dependency array = runs after EVERY render
    // setCount triggers a re-render, which triggers useEffect again → infinite loop
    setCount(count + 1);
  }); // ← No dependency array!

  return <Text>{count}</Text>;
}

// ✅ Correct: specify dependency array
export default function App() {
  const [count, setCount] = useState(0);

  useEffect(() => {
    // Runs only on mount
    console.log("Component mounted");
    return () => console.log("Component unmounted");
  }, []); // ← Empty array = run once on mount

  return <Text>{count}</Text>;
}
```

**Solution**:
- Always include a dependency array in `useEffect`
- Omitting it causes the effect to run after every render
- Check for state updates that trigger endless re-renders

### 3. Native Module Compatibility Issues

**Symptom**: App crashes when using native features (camera, GPS, filesystem, etc.).

**Root Cause**: Native module version mismatches with your Expo version.

```typescript
// ❌ Installing incompatible camera library
import { Camera } from "react-native-camera"; // Wrong version

// ✅ Use Expo's compatible version
// $ npx expo install expo-camera
// $ npm install @react-native-camera-roll/camera-roll
```

**Solution**:
- Install with `npx expo install` to get Expo-compatible versions
- Check `package.json` for version numbers
- Run `expo prebuild` to rebuild native code
- Verify against Expo's official compatibility matrix

### 4. Unhandled Async Errors

**Symptom**: App crashes when network errors occur during API calls or file operations.

**Root Cause**: Missing error handling (no `.catch()` or `try-catch`).

```typescript
// ❌ Wrong: no error handling
import { useEffect, useState } from "react";
import { View, Text } from "react-native";

export default function UserProfile() {
  const [user, setUser] = useState(null);

  useEffect(() => {
    // Network error will crash the app
    fetch("https://api.example.com/user/123")
      .then((res) => res.json())
      .then((data) => setUser(data));
  }, []);

  return <Text>{user?.name}</Text>;
}

// ✅ Correct: add error handling
export default function UserProfile() {
  const [user, setUser] = useState(null);
  const [error, setError] = useState(null);

  useEffect(() => {
    fetch("https://api.example.com/user/123")
      .then((res) => res.json())
      .then((data) => setUser(data))
      .catch((err) => {
        console.error("Failed to fetch user:", err);
        setError("Unable to load user information");
      });
  }, []);

  if (error) return <Text style={{ color: "red" }}>{error}</Text>;
  if (!user) return <Text>Loading...</Text>;

  return <Text>{user?.name}</Text>;
}
```

**Solution**:
- Add `.catch()` to all promises
- Use `try-catch` in async functions
- Provide user-friendly error messages

### 5. Navigation Setup Errors

**Symptom**: App crashes during screen transitions, especially when switching tabs.

**Root Cause**: Referencing screen names that haven't been defined in the navigator.

```typescript
// ❌ Wrong: navigating to undefined screen
import { NavigationContainer } from "@react-navigation/native";
import { createNativeStackNavigator } from "@react-navigation/native-stack";

const Stack = createNativeStackNavigator();

export default function App() {
  return (
    <NavigationContainer>
      <Stack.Navigator>
        <Stack.Screen name="Home" component={HomeScreen} />
        {/* "Profile" is not defined */}
      </Stack.Navigator>
    </NavigationContainer>
  );
}

function HomeScreen({ navigation }: any) {
  return (
    <Button
      title="Go to Profile"
      onPress={() => navigation.navigate("Profile")} {/* Crash! */}
    />
  );
}

// ✅ Correct: define screens before navigating to them
export default function App() {
  return (
    <NavigationContainer>
      <Stack.Navigator>
        <Stack.Screen name="Home" component={HomeScreen} />
        <Stack.Screen name="Profile" component={ProfileScreen} />
      </Stack.Navigator>
    </NavigationContainer>
  );
}
```

**Solution**:
- Define all screens in `Stack.Screen` before calling `navigate()`
- Screen names are case-sensitive (`"Home"` ≠ `"home"`)

## White Screen Patterns and Fixes

### Pattern 1: White Screen on Startup

**Cause**: Splash screen configuration is missing or never hides.

```typescript
// ✅ Use Expo.SplashScreen and hide it after initialization
import * as SplashScreen from "expo-splash-screen";
import { useCallback, useEffect, useState } from "react";
import { View, Text } from "react-native";

SplashScreen.preventAutoHideAsync(); // Prevent auto-hide

export default function App() {
  const [fontsLoaded, setFontsLoaded] = useState(false);

  useEffect(() => {
    // Initialization (font loading, etc.)
    setTimeout(() => setFontsLoaded(true), 2000);
  }, []);

  const onLayoutRootView = useCallback(async () => {
    if (fontsLoaded) {
      await SplashScreen.hideAsync(); // Hide splash when ready
    }
  }, [fontsLoaded]);

  if (!fontsLoaded) return null;

  return (
    <View onLayout={onLayoutRootView} style={{ flex: 1 }}>
      <Text>App loaded</Text>
    </View>
  );
}
```

### Pattern 2: White Screen on Specific Screens

**Cause**: Error occurring in a child component, not caught by an error boundary.

**Fix**: See "[Complete Guide to Fixing Runtime Errors](/en/articles/rork-dev/rork-react-native-expo-runtime-errors-fix)" for error boundary implementation.

### Pattern 3: White Screen After Update

**Cause**: Stale cache from previous version.

```bash
# Clear Expo cache
npx expo start --clear

# Then restart on Expo Go
```

## Rork Preview vs Real Device

Rork's browser simulator and actual iOS/Android devices behave differently:

- **Preview** — Faster CPU, abundant memory, masks some bugs
- **Real device** — Realistic performance metrics, reveals actual issues

**Best practice**: Move to real device testing early in development to catch performance issues.

## Getting Crash Reports

Rork Companion (Mac app) automatically saves crash logs:

1. **Open Rork Companion**
2. **Reproduce the crash while testing**
3. **Check the "Crash Logs" tab**

Crash logs include:
- **Error message** — What went wrong
- **Stack trace** — Which file and line number
- **Device info** — Device type, OS version, etc.

Use this information to pinpoint the root cause.

## Quality Testing Checklist

Prevent crashes before they happen:

- **Test on multiple devices** — iPhone and Android, old and new models
- **Simulate low memory** — Use device simulator "Low Memory" mode
- **Test network failures** — Try offline mode and slow networks
- **Verify all screen transitions** — Manually test every navigation path
- **Monitor long sessions** — Watch for memory leaks over time

## FAQ

**Q1: The app works in Rork preview but crashes on a real device.**

A: The preview environment is faster and has more memory than real devices. When it crashes on real hardware, memory optimization and performance tuning are needed.

**Q2: My app crashed. How do I explain it to users?**

A: Standard messaging: "We're sorry. An error occurred in the app. Please try updating to the latest version or clearing the app cache."

**Q3: I can't get crash logs from Rork Companion.**

A: Make sure Rork Companion is running and connected to the same Wi-Fi. Restart the app and reproduce the crash again.

## Debugging Workflow (Step by Step)

1. **Read the error log** — Identify what happened
2. **Locate the error source** — Use stack trace to find file and line number
3. **Examine the code** — Check for initialization issues, infinite loops, missing error handling
4. **Write a fix** — Follow best practices from CLAUDE.md
5. **Test on multiple devices** — Verify the fix works everywhere
6. **Deploy to production** — Release when testing is complete

## Summary

Most app crashes and white screens fall into one of these five categories:

1. **Memory exhaustion** — Resize images, manage cache
2. **Infinite loops** — Check useEffect dependency arrays
3. **Native module incompatibility** — Use `npx expo install` and `prebuild`
4. **Missing error handling** — Add `.catch()` and `try-catch`
5. **Navigation setup errors** — Define all screens before navigating

Master the debugging tools (Rork Companion, crash logs, error boundaries) and your development efficiency will skyrocket.
