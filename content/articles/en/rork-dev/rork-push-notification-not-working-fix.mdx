You launched your Rork app, wired up push notifications, and... nothing. Or maybe they worked fine during development but stopped after a TestFlight or production release. Push notifications are one of the most powerful tools for re-engaging users, but they're also notoriously tricky to get right — especially when juggling iOS APNs, Android FCM, Expo credentials, and device permissions.

This guide walks you through the most common reasons push notifications fail in Rork apps and exactly how to fix each one.

## Identifying the Symptom Pattern

Before diving into solutions, narrow down what's actually happening. These are the most common failure patterns:

- **Pattern A**: Notifications arrive only when the app is in the foreground
- **Pattern B**: Works on simulator/emulator but not on a real device
- **Pattern C**: Works on iOS but not Android (or vice versa)
- **Pattern D**: Works in development but fails in the TestFlight or production build
- **Pattern E**: Send requests return success but the device never receives anything
- **Pattern F**: Permission dialog never appears, or the user previously denied it

Identifying the exact pattern will dramatically narrow the list of possible causes.

## Root Cause Analysis

### Cause 1: Push Token Not Being Retrieved

Every push notification flow starts with obtaining an **Expo Push Token** (or a native APNs/FCM token). If this step silently fails, no amount of correct server-side code will help.

Common reasons the token retrieval fails:

- Running on an iOS Simulator (APNs is not supported on simulators)
- Forgetting to call `requestPermissionsAsync()` before requesting the token
- `projectId` in `app.json` doesn't match the Expo dashboard
- Version mismatch in `expo-notifications`

### Cause 2: iOS APNs Certificate Issues

iOS push notifications require a valid **APNs certificate or key** registered with your Apple Developer account. Development and production builds use different certificate environments — mixing them is a common source of silent failures.

- Development builds → APNs Sandbox
- TestFlight and App Store builds → APNs Production

If you're using EAS Build, the credentials stored in your EAS project must match the bundle identifier of your app.

### Cause 3: Android FCM Configuration Errors

Android notifications are routed through **Firebase Cloud Messaging (FCM)**. An outdated or mismatched `google-services.json` file — particularly if the `package_name` inside doesn't match `android.package` in `app.json` — will prevent FCM token registration from succeeding.

### Cause 4: Missing app.json / app.config.js Settings

Several fields in your app config are easy to overlook but critical for push notifications:

- `extra.eas.projectId` — required for Expo push token generation
- `ios.infoPlist.UIBackgroundModes` including `"remote-notification"` — required for background notifications on iOS
- `android.googleServicesFile` — must point to a valid `google-services.json`

### Cause 5: Android Notification Channel Not Configured

Android 8.0 (API level 26) and above require a **notification channel** to display notifications. If your app targets API 26+ without a configured channel, notifications are silently dropped.

## Step-by-Step Fix Guide

### Step 1: Verify Token Retrieval

Add the following to your app to confirm whether the token is being obtained successfully:

```typescript
import * as Notifications from 'expo-notifications';
import * as Device from 'expo-device';
import Constants from 'expo-constants';

async function registerForPushNotificationsAsync() {
  // Simulators can't receive push notifications
  if (!Device.isDevice) {
    console.warn('⚠️ Push tokens require a physical device.');
    return null;
  }

  // Request permission
  const { status: existingStatus } = await Notifications.getPermissionsAsync();
  let finalStatus = existingStatus;

  if (existingStatus !== 'granted') {
    const { status } = await Notifications.requestPermissionsAsync();
    finalStatus = status;
  }

  if (finalStatus !== 'granted') {
    console.error('❌ Push notification permission denied');
    return null;
  }

  // Retrieve the Expo push token
  try {
    const projectId = Constants.expoConfig?.extra?.eas?.projectId;
    if (!projectId) {
      console.error('❌ projectId missing in app.json');
      return null;
    }
    const tokenData = await Notifications.getExpoPushTokenAsync({ projectId });
    console.log('✅ Expo Push Token:', tokenData.data);
    return tokenData.data;
  } catch (error) {
    console.error('❌ Token retrieval failed:', error);
    return null;
  }
}
```

**Expected output**: A token in the format `ExponentPushToken[xxxxxxxxxxxxxxxxxxxxxx]`. If you see an error instead, this is your starting point.

### Step 2: Review Your app.json Configuration

```json
{
  "expo": {
    "name": "YourApp",
    "slug": "your-app",
    "extra": {
      "eas": {
        "projectId": "YOUR_EAS_PROJECT_ID"
      }
    },
    "ios": {
      "bundleIdentifier": "com.yourcompany.yourapp",
      "infoPlist": {
        "UIBackgroundModes": ["remote-notification"]
      }
    },
    "android": {
      "package": "com.yourcompany.yourapp",
      "googleServicesFile": "./google-services.json"
    },
    "plugins": [
      [
        "expo-notifications",
        {
          "icon": "./assets/notification-icon.png",
          "color": "#ffffff",
          "defaultChannel": "default"
        }
      ]
    ]
  }
}
```

Key things to verify:

- `projectId` matches what you see in your Expo dashboard under **Project Settings**
- `bundleIdentifier` (iOS) and `package` (Android) match your Apple and Google configurations exactly
- `UIBackgroundModes` includes `"remote-notification"` for background delivery on iOS

### Step 3: Reconfigure iOS APNs Credentials

If you're using EAS Build, check and update your credentials with:

```bash
# View current credential status
eas credentials

# Reconfigure iOS credentials interactively
eas credentials --platform ios

# Confirm your project ID
eas project:info
```

For APNs, prefer **APNs Auth Keys (.p8)** over certificates — they don't expire annually and work across all apps in your Apple Developer account. Generate one in the Apple Developer portal under **Certificates, Identifiers & Profiles → Keys**.

### Step 4: Fix the Android google-services.json

Download the latest `google-services.json` from Firebase Console and place it in your project root. Then verify the package name matches:

```bash
# Check the package name inside google-services.json
python3 -c "
import json
with open('google-services.json') as f:
    data = json.load(f)
for client in data['client']:
    print(client['client_info']['android_client_info']['package_name'])
"

# Compare with app.json
cat app.json | python3 -c "import sys,json; d=json.load(sys.stdin); print(d['expo']['android']['package'])"
```

Both values must be identical. Even a single character mismatch will prevent FCM registration.

### Step 5: Send a Test Notification via Expo's Push Tool

Once you have a token, send a test notification using the [Expo Push Notification Tool](https://expo.dev/notifications) or via `curl`:

```bash
curl -H "Content-Type: application/json" -X POST "https://exp.host/--/api/v2/push/send" \
  -d '{
    "to": "ExponentPushToken[YOUR_TOKEN_HERE]",
    "title": "Test Notification",
    "body": "If you see this, push is working!",
    "data": { "test": true }
  }'
```

A response of `"status": "ok"` means Expo's servers accepted the request. If the notification still doesn't arrive on the device, the problem is between the push service (APNs/FCM) and the device — most likely a credentials or configuration issue.

### Step 6: Configure the Android Notification Channel

Android 8.0+ silently drops notifications if no channel is configured. Add channel setup at app launch:

```typescript
import * as Notifications from 'expo-notifications';
import { Platform } from 'react-native';

// Set the global notification handler
Notifications.setNotificationHandler({
  handleNotification: async () => ({
    shouldShowAlert: true,
    shouldPlaySound: true,
    shouldSetBadge: true,
  }),
});

// Create the Android notification channel
async function setupAndroidChannel() {
  if (Platform.OS === 'android') {
    await Notifications.setNotificationChannelAsync('default', {
      name: 'General Notifications',
      importance: Notifications.AndroidImportance.MAX,
      vibrationPattern: [0, 250, 250, 250],
      lightColor: '#FF231F7C',
      sound: 'default',
    });
    console.log('✅ Android notification channel created');
  }
}
```

## How to Verify the Fix

**iOS verification steps**:
1. Install a development or TestFlight build on a physical device
2. Launch the app — the permission dialog should appear; tap **Allow**
3. Confirm `ExponentPushToken[...]` appears in your console logs
4. Send a test notification via the Expo Push Tool
5. Background the app and confirm the notification arrives in the system notification tray

**Android verification steps**:
1. Install the app on a real device or API 33+ emulator
2. Check: Settings → Apps → [App Name] → Notifications → All turned on
3. Send a test via curl and confirm notification appears in the status bar
4. Check that the "General Notifications" channel appears in the notification settings

**Interpreting Expo Push API response codes**:

- `"status": "ok"` — Delivered to Expo's push service successfully
- `"details.error": "DeviceNotRegistered"` — Token is invalid or outdated; re-fetch and update
- `"details.error": "InvalidCredentials"` — APNs key or FCM configuration problem

## Prevention: Best Practices

### Implement Token Refresh Logic

APNs and FCM can rotate device tokens. Always re-fetch the token on app launch and sync it with your server if it has changed:

```typescript
useEffect(() => {
  const syncToken = async () => {
    const freshToken = await registerForPushNotificationsAsync();
    if (freshToken && freshToken !== storedToken) {
      await saveTokenToServer(freshToken);
      setStoredToken(freshToken);
    }
  };
  syncToken();
}, []);
```

### Check Notification Permission Status on Foreground Resume

Users can revoke notification permissions at any time from device settings. Use `AppState` to re-check permission status when the app returns to the foreground, and prompt re-authorization if needed. The [push notification implementation guide](/articles/rork-dev/rork-push-notifications) covers the full permission flow in detail.

### Set Up Error Monitoring

Wrap token retrieval and send logic in try-catch blocks and report failures to an error monitoring service. This gives you visibility into how many of your users are silently failing to receive notifications in production.

### Track APNs Certificate Expiry

If you're using APNs certificates (.p12) rather than keys (.p8), they expire after one year. Set a calendar reminder well in advance. Migrating to APNs Auth Keys (.p8) eliminates this concern entirely.

## FAQ

**Q1. Can I test push notifications on the iOS Simulator?**

No — Apple's APNs infrastructure doesn't support the iOS Simulator. For local notification testing, `Notifications.scheduleNotificationAsync()` works on the simulator. For actual push delivery, you need a physical device. Android emulators with Google Play Services do support FCM push notifications. See the [Expo local notifications guide](/articles/rork-dev/rork-local-notifications-expo-complete-guide) for simulator-friendly notification testing.

**Q2. Notifications work in development but fail in TestFlight. What's wrong?**

TestFlight uses the APNs production environment, not the sandbox. In EAS, confirm that your **distribution** credentials include a production APNs key. Run `eas credentials` to inspect. Also verify that your `eas.json` production profile has `"credentialsSource": "remote"` and is configured for distribution builds.

**Q3. The notification arrives but nothing shows on screen. Why?**

On Android, check that the notification channel's `importance` is set to `AndroidImportance.HIGH` or `MAX` — lower values suppress the heads-up display. On iOS, if the app is in the foreground, `shouldShowAlert: true` must be set in your `Notifications.setNotificationHandler` callback. Also check if the device is in Do Not Disturb or Silent mode.

**Q4. My sends return success but some users consistently don't receive notifications**

This is often a `DeviceNotRegistered` error being swallowed silently. After each batch send, inspect the response body for per-token errors and remove invalid tokens from your database immediately. For high-volume sending patterns, see the [advanced push notification segmentation guide](/articles/rork-dev/rork-push-notification-advanced-segmentation-automation).

**Q5. How do I delete stale tokens to keep my send list clean?**

After each push send, parse the Expo Push API receipts and delete any tokens that return `DeviceNotRegistered`. Schedule a periodic cleanup job to remove tokens that haven't been refreshed in 90+ days. This keeps your send lists lean and improves delivery rates.

## Summary

Push notification failures in Rork apps almost always come down to one of five root causes: token retrieval failure, APNs certificate misconfiguration, FCM package name mismatch, missing `app.json` settings, or a missing Android notification channel. Work through the steps in order:

1. Confirm the token is being retrieved on a real device
2. Validate `projectId`, `UIBackgroundModes`, and `googleServicesFile` in `app.json`
3. Reconfigure APNs credentials via `eas credentials` for iOS production issues
4. Verify `google-services.json` package name matches `app.json` for Android
5. Set up the Android notification channel with `AndroidImportance.MAX`

Once notifications are working reliably, add token refresh logic and error monitoring so your delivery rates stay high over time.
