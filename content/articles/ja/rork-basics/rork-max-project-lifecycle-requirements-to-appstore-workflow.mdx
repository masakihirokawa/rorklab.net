## はじめに — なぜプロジェクトライフサイクル全体を設計するべきなのか

Rork Max は AI を活用してネイティブアプリを素早く構築できるツールですが、「アイデアをプロンプトに入力して生成ボタンを押す」だけでは、本当にユーザーに届くアプリは作れません。App Store で継続的に評価されるアプリを作るには、要件定義からリリース後の運用改善まで、プロジェクト全体を見渡した設計が不可欠です。

この記事では、Rork Max を軸にしたアプリ開発の全工程を、実際のプロジェクトで使えるワークフローとして体系的に解説します。個人開発者がプロレベルのアプリをリリースし、継続的に改善していくための実践的なロードマップです。

対象読者は、Rork の基本操作は理解しており、初めてのアプリをストアに公開したい方、あるいはすでに公開経験があり開発プロセスを体系化したい方を想定しています。Rork Max の基本操作にまだ不安がある方は、先に[非エンジニアのための Rork Max アプリ開発入門](/articles/rork-basics/rork-max-non-engineer-beginner-guide)をご覧ください。

## フェーズ 1: 要件定義 — アプリの「Why」を明確にする

### ユーザーストーリーの作成

プロジェクトの出発点は、「誰の」「どんな課題を」「どう解決するか」を明文化することです。Rork Max で開発を始める前に、以下のフレームワークでユーザーストーリーを整理しましょう。

```
ユーザーストーリーテンプレート:
[ペルソナ] として、
[目的/課題] を達成するために、
[機能/アクション] を行いたい。
なぜなら [理由/期待する価値] だからである。

例:
「忙しい社会人」として、
「日々の支出を素早く記録」するために、
「ワンタップで金額とカテゴリを入力」したい。
なぜなら「月末のまとめ作業を30分以内に終わらせたい」からである。
```

### MVP（Minimum Viable Product）の定義

すべての機能を最初から実装しようとするのは、個人開発では最もよくある失敗パターンです。MVP として「これがなければアプリとして成立しない」コア機能を3〜5個に絞り込みます。

```typescript
// MVP機能の優先度マトリクス（例: 支出管理アプリ）
const mvpFeatures = {
  must: [
    '支出の記録（金額・カテゴリ・日付）',
    'カテゴリ別の月間集計表示',
    'データのローカル永続化',
  ],
  should: [
    '月間レポートのグラフ表示',
    'CSV エクスポート',
  ],
  could: [
    '複数通貨対応',
    'クラウド同期',
    'レシート OCR 読み取り',
  ],
  wont: [
    '家族間の共有機能（v2以降）',
    'AI による支出予測（v2以降）',
  ],
};
// must → should の順に実装し、could 以降は v2 のバックログへ
```

### 競合分析の実施

App Store で類似アプリを5〜10本ダウンロードし、以下の観点で分析します。

- **UI/UX パターン**: オンボーディングの流れ、メイン画面の構成、ナビゲーション方式
- **収益モデル**: 無料、フリーミアム、サブスクリプション、買い切り
- **レビュー傾向**: 星1〜2のレビューに共通する不満点（＝差別化のチャンス）
- **機能の過不足**: 自分のアプリで補えるギャップ

この分析結果は、後述する Rork Max へのプロンプト設計に直接活きてきます。

## フェーズ 2: アーキテクチャ設計 — Rork Max プロジェクトの骨格を作る

### 画面遷移図の作成

Rork Max にプロンプトを投げる前に、アプリ全体の画面遷移を設計図として描きます。紙のスケッチでも Figma でも構いませんが、最低限以下の要素を含めましょう。

- **画面一覧**: 各画面の名称と主要な役割
- **遷移パス**: どの画面からどの画面に移動できるか
- **データフロー**: 各画面で表示・入力するデータは何か
- **認証フロー**: ログイン状態で表示が変わる画面はどれか

```
画面遷移の例（支出管理アプリ）:

[スプラッシュ] → [オンボーディング(初回のみ)] → [ホーム(月間サマリー)]
                                                      ├→ [記録追加]
                                                      ├→ [カテゴリ一覧] → [カテゴリ別詳細]
                                                      ├→ [レポート(グラフ)]
                                                      └→ [設定]
                                                           ├→ [カテゴリ管理]
                                                           ├→ [データ管理(エクスポート/削除)]
                                                           └→ [アプリ情報]
```

### データモデルの設計

Rork Max が生成するコードの品質は、データ構造の設計に大きく左右されます。事前に TypeScript の型定義レベルでデータモデルを設計しておくと、プロンプトの精度が格段に上がります。アーキテクチャ設計をさらに深掘りしたい方は、[Rork アプリ設計パターン完全ガイド](/articles/rork-dev/rork-app-architecture-patterns-mvvm-clean)も参考にしてください。

```typescript
// データモデル設計の例
interface Expense {
  id: string;           // UUID
  amount: number;       // 金額（整数、通貨の最小単位）
  currency: 'JPY' | 'USD';
  categoryId: string;   // Category.id への参照
  note?: string;        // 任意のメモ
  date: string;         // ISO 8601 形式 'YYYY-MM-DD'
  createdAt: string;    // ISO 8601 タイムスタンプ
  updatedAt: string;
}

interface Category {
  id: string;
  name: string;         // 表示名
  icon: string;         // アイコン識別子
  color: string;        // HEX カラーコード
  budget?: number;      // 月間予算（任意）
  sortOrder: number;    // 表示順
}

interface MonthlyReport {
  yearMonth: string;    // 'YYYY-MM'
  totalAmount: number;
  byCategory: Array<{
    categoryId: string;
    amount: number;
    percentage: number; // 全体に対する割合
    overBudget: boolean;
  }>;
}
```

### 技術選定チェックリスト

Rork Max プロジェクトでは、以下の技術的な判断を事前に行っておきます。

- **データ永続化**: AsyncStorage（軽量）、SQLite（構造化データ）、Supabase（クラウド同期）
- **状態管理**: React の useState/useContext（小規模）、Zustand（中規模）、Redux Toolkit（大規模）
- **ナビゲーション**: React Navigation（Stack + Tab の組み合わせが主流）
- **認証**: Supabase Auth、Firebase Auth、または認証なし（ローカルオンリー）
- **通知**: Expo Notifications（プッシュ通知）、ローカル通知のみ
- **収益化**: サブスクリプション（StoreKit 2 / Google Billing）、広告（AdMob）、買い切り

## フェーズ 3: AI 駆動の反復開発 — Rork Max プロンプト設計のベストプラクティス

### 段階的プロンプト戦略

Rork Max でアプリ全体を一度に生成しようとすると、コードの品質が低下しやすくなります。以下の段階的アプローチを推奨します。

**ステップ 1: 基盤レイヤーの生成**

```
プロンプト例（第1段階）:
「支出管理アプリの基盤を作成してください。
- React Navigation (Stack + Bottom Tab) のナビゲーション構造
- 5つのタブ: ホーム、記録追加、カテゴリ、レポート、設定
- グローバルな状態管理（Zustand）のストア定義
- AsyncStorage を使ったデータ永続化レイヤー
- TypeScript の型定義（Expense, Category, MonthlyReport）
UI はプレースホルダーで構いません。構造の正確さを優先してください。」
```

**ステップ 2: 画面ごとの詳細実装**

```
プロンプト例（第2段階）:
「ホーム画面を実装してください。
- 今月の支出合計を大きく表示
- カテゴリ別の支出割合を円グラフで表示（react-native-chart-kit）
- 直近5件の支出をリスト表示（FlatList）
- 「記録追加」への FloatingActionButton
- Pull-to-Refresh で月間データを再計算
- ダークモード対応のカラースキーム」
```

**ステップ 3: インタラクションとアニメーション**

```
プロンプト例（第3段階）:
「記録追加画面のUXを改善してください。
- 金額入力はテンキーパッドUI（電卓風）
- カテゴリ選択はアイコングリッド（2列）+ 選択時のスケールアニメーション
- 日付はカレンダーピッカー（デフォルトは今日）
- 保存成功時に Haptic Feedback + チェックマークアニメーション
- 入力バリデーション: 金額0以下は保存不可、カテゴリ未選択時は警告表示」
```

### プロンプトの CRAFT 原則

効果的な Rork Max プロンプトには、以下の5要素を含めます。

- **C**ontext（文脈）: アプリの目的、対象ユーザー、技術スタック
- **R**equirements（要件）: 具体的な機能仕様、画面構成
- **A**rchitecture（設計方針）: 使用するライブラリ、デザインパターン、型定義
- **F**eedback（フィードバック）: 前回の生成結果で改善が必要な点
- **T**esting（テスト観点）: 期待する動作、エッジケース

```typescript
// CRAFT原則に基づくプロンプト構造化の例
const promptTemplate = `
## Context
${appDescription}
${targetUser}
${techStack}

## Requirements
${featureSpecifications}

## Architecture
${designPatterns}
${typeDefinitions}

## Feedback (前回の改善点)
${previousIterationFeedback}

## Testing
${expectedBehavior}
${edgeCases}
`;
```

### 反復サイクルの管理

Rork Max での開発は、以下の反復サイクルで進めます。1サイクルは1〜3時間を目安にしましょう。

1. **プロンプト作成**（10〜15分）: CRAFT 原則に基づいて具体的なプロンプトを設計
2. **コード生成**（5〜10分）: Rork Max でコードを生成
3. **レビュー＆テスト**（20〜30分）: 生成されたコードを実機/シミュレーターで検証
4. **改善点の記録**（5分）: 次のサイクルのフィードバックとして言語化
5. **マージ＆コミット**（5分）: 動作確認済みのコードをメインブランチに統合

1日に3〜5サイクルを回すことで、1〜2週間で MVP を完成させるのが現実的なタイムラインです。

### 実機テストチェックリスト

App Store 申請前に、以下の項目を実機で確認します。

- **デバイスバリエーション**: iPhone SE（小画面）、iPhone 16 Pro Max（大画面）、iPad
- **OS バージョン**: 最新 iOS と1つ前のメジャーバージョン
- **ネットワーク状態**: Wi-Fi、4G/5G、機内モード（オフライン動作）
- **ダークモード**: 全画面でテキストが読めるか、コントラスト比は十分か
- **アクセシビリティ**: VoiceOver で全操作が可能か、Dynamic Type でレイアウトが崩れないか
- **パフォーマンス**: 起動時間、画面遷移のスムーズさ、メモリ使用量
- **ローカライズ**: 日本語と英語で文字切れ、レイアウト崩れがないか

## フェーズ 5: App Store 申請 — 一発承認を目指す準備

### App Store Connect の事前準備

Rork Max の2クリック公開機能を活用する前に、以下の情報を揃えておきます。

```
App Store Connect 必須情報:
├── アプリ名（30文字以内、キーワードを含む）
├── サブタイトル（30文字以内）
├── カテゴリ（プライマリ + セカンダリ）
├── 説明文（4000文字以内、最初の3行が最重要）
├── キーワード（100文字以内、カンマ区切り）
├── スクリーンショット
│   ├── 6.9インチ（iPhone 16 Pro Max）× 最大10枚
│   ├── 6.7インチ（iPhone 15 Plus）× 最大10枚
│   └── 6.5インチ以下（iPhone SE等）× 最大10枚
├── App プレビュー動画（任意、30秒以内推奨）
├── プライバシーポリシー URL
├── サポート URL
└── 年齢制限の設定
```

### App Store Review ガイドライン対策

Apple のレビューで最もリジェクトされやすいポイントと、その対策を整理します。

- **4.0 Design（デザイン）**: テンプレート感のある UI はリジェクトされやすい。Rork Max の生成結果をそのまま使うのではなく、ブランドカラーやカスタムアイコンで差別化する
- **4.2 Minimum Functionality（最低限の機能）**: Web サイトをそのままラップしただけのアプリは却下される。ネイティブ機能（通知、カメラ、ローカルストレージ等）を最低1つは活用する
- **5.1.1 Data Collection and Storage（データ収集）**: App Tracking Transparency フレームワークの実装が必須（iOS 14.5以降）。サードパーティ SDK を使う場合は必ずプライバシーマニフェストを更新する
- **2.1 Performance: App Completeness（完成度）**: クラッシュ、プレースホルダーテキスト、デッドリンクは即リジェクト。TestFlight で十分な事前テストを行う

### スクリーンショット最適化

App Store での最大のコンバージョン要因はスクリーンショットです。以下の原則で作成します。

```
スクリーンショット構成（5枚の場合）:
1枚目: アプリの最大の価値提案（ヒーローショット）
  → 「月の支出を30秒で把握」+ メイン画面のスクリーンショット
2枚目: コア機能のデモ
  → 「ワンタップで簡単記録」+ 記録追加画面
3枚目: ビジュアルが映える機能
  → 「美しいグラフで支出を可視化」+ レポート画面
4枚目: 差別化ポイント
  → 「カテゴリ別予算管理」+ カテゴリ設定画面
5枚目: 信頼性の訴求
  → 「データは端末に安全に保存」+ 設定画面
```

## フェーズ 6: リリース管理 — 段階的ロールアウトと初動対応

### バージョニング戦略

```
セマンティックバージョニング: MAJOR.MINOR.PATCH
  1.0.0 — 初回リリース（MVP）
  1.0.1 — バグ修正（クラッシュ対応等）
  1.1.0 — 機能追加（CSV エクスポート等）
  2.0.0 — 大型アップデート（クラウド同期導入等）

Build Number: リリースごとにインクリメント
  1.0.0 (1) → 1.0.0 (2) → 1.0.1 (3) → 1.1.0 (4)
```

### 段階的ロールアウト

App Store では段階的リリースが可能です。初回リリースでは以下のスケジュールを推奨します。

- **Day 1〜3**: 全ユーザーに公開（初回は段階的リリースが使えないため）
- **v1.0.1 以降**: 段階的リリースを有効にし、1%→2%→5%→10%→20%→50%→100% で配信

### 初動モニタリング

リリース後72時間は、以下の指標を集中的に監視します。

```typescript
// 監視すべき指標と閾値の例
const monitoringThresholds = {
  // クラッシュ率: 1%以下が目標
  crashFreeRate: { target: 99.0, alert: 98.0 },
  // アプリ起動時間: 2秒以内
  coldStartTime: { target: 2000, alert: 3000 }, // ms
  // DAU（日次アクティブユーザー）リテンション
  day1Retention: { target: 40, alert: 25 },      // %
  // App Store レーティング
  averageRating: { target: 4.5, alert: 3.5 },
};

// Rork Max + EAS Update で緊急修正をOTAデプロイ可能
// ネイティブコード変更がなければ App Store審査なしで即時配信
```

## フェーズ 7: 継続的改善 — リリース後の成長サイクル

### ユーザーフィードバックループ

```
フィードバック収集の仕組み:
1. App Store レビュー → 週次で分析、★3以下は48時間以内に返信
2. アプリ内フィードバック → 設定画面にフィードバックフォーム設置
3. クラッシュレポート → Sentry または Firebase Crashlytics で自動収集
4. 利用データ → ログイン率、機能使用率、離脱ポイントを分析
```

### 改善の優先度フレームワーク（ICE スコア）

収集したフィードバックを、以下のフレームワークで優先順位付けします。

```typescript
// ICE スコアリング
interface Improvement {
  description: string;
  impact: number;      // 1-10: ユーザー体験への影響度
  confidence: number;  // 1-10: 改善効果の確信度
  ease: number;        // 1-10: 実装の容易さ
  iceScore: number;    // = impact × confidence × ease
}

// 例: 支出管理アプリの改善候補
const improvements: Improvement[] = [
  {
    description: 'カテゴリ別予算アラート機能',
    impact: 8,      // 多数のレビューで要望あり
    confidence: 9,  // 競合アプリで効果実証済み
    ease: 7,        // Rork Max で比較的容易に実装可
    iceScore: 504,  // 8 × 9 × 7 = 最優先
  },
  {
    description: '多通貨対応',
    impact: 6,      // 海外ユーザーからの要望
    confidence: 5,  // ニーズの規模が不明
    ease: 4,        // 為替レートAPIの統合が必要
    iceScore: 120,  // 後回し可
  },
];
```

### Rork Max を活用した高速イテレーション

リリース後の改善サイクルでこそ、Rork Max の真価が発揮されます。

```
改善サイクル（1スプリント = 1週間）:
月曜: フィードバック分析 + 改善項目の ICE スコアリング
火曜: 最優先の改善をプロンプト化 → Rork Max で生成
水曜: コードレビュー + テスト
木曜: TestFlight 配信 → 内部テスト
金曜: EAS Update（OTA）または App Store 提出
```

EAS Update を活用すれば、JavaScript バンドルの変更はストア審査を経ずに即時配信できます。ネイティブコードの変更を伴う場合のみ、通常のストア審査フローに乗せます。

## FAQ

### Q: Rork Max でアプリを作る場合、プログラミング知識はどの程度必要ですか？

A: MVP の段階では、TypeScript の型定義が読める程度の知識があれば十分です。ただし、App Store に公開して継続的に改善していくには、React Native の基本概念（コンポーネント、状態管理、ナビゲーション）とデバッグの基礎知識が必要です。この記事で紹介したワークフローに沿って進めれば、開発しながら自然に知識が身についていきます。

### Q: 要件定義からApp Store公開まで、どのくらいの期間が必要ですか？

A: アプリの規模と個人のスキルレベルによりますが、MVP レベルのアプリであれば、要件定義に1〜2日、開発に1〜2週間、テストと申請準備に3〜5日が目安です。Rork Max を活用すれば、従来の開発と比べて開発フェーズを50〜70%短縮できます。ただし、テストと申請準備の時間は短縮しないことを強く推奨します。

### Q: Rork Max で生成されたコードをそのまま使っても大丈夫ですか？

A: 基本的な動作は問題ありませんが、プロダクション品質にするには必ずレビューとテストが必要です。特に注意すべきポイントは、エラーハンドリングの抜け漏れ、メモリリークの可能性があるイベントリスナー、ハードコードされた値（本来は設定ファイルや環境変数に外出しすべきもの）です。この記事のフェーズ4で紹介したテスト戦略を実践することで、品質を担保できます。

### Q: App Store でリジェクトされた場合、どう対処すべきですか？

A: まず Apple からのリジェクト理由を正確に読み解きます。最も多いリジェクト理由は「Design（4.0）」と「Performance（2.1）」です。リジェクト理由に具体的な修正指示がある場合はそれに従い、曖昧な場合は Resolution Center から質問することもできます。修正後の再申請は通常1〜3営業日で結果が出ます。Rork Max で素早く修正コードを生成し、テストを経て再申請するサイクルを回しましょう。

## まとめ

Rork Max を使ったアプリ開発は、要件定義・アーキテクチャ設計・AI 駆動の反復開発・テスト戦略・App Store 申請・リリース後の継続改善という6つのフェーズで構成されます。どのフェーズも省略せず、体系的に進めることが、ストアで評価されるアプリを生み出す鍵です。

特に重要なのは、Rork Max への入力（プロンプト）の質を高めるために、事前の設計をしっかり行うことです。CRAFT 原則に基づくプロンプト設計と、段階的な反復開発サイクルを組み合わせることで、個人開発者でもプロレベルのアプリを効率よくリリースできます。
