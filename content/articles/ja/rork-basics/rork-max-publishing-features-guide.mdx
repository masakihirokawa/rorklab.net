Rork Max でアプリを開発した後、最大の課題が「どうやって公開するのか」だ。ノーコードツールだからこそ、公開フロー（パブリッシング）がシンプルに設計されているが、実際のApp Store・Google Play申請には細かい要件がある。

この記事では、Rork Max の**パブリッシング機能の実装状況**から、実際のアプリ公開までの全ステップを解説する。公式ドキュメントには書かれていない実装のコツも含めた、現在進行形のガイドだ。

## Rork Max パブリッシング機能の現在地

Rork Max には、アプリをバイナリ（IPA / APK）にビルドする機能が実装されている。ただし注意すべき点は：

- **App Storeへの直接提出機能は未実装**（2026年4月時点）
- **Google Play へも直接アップロード機能はなし**
- **ビルド出力は IPA / APK ファイル生成まで**

つまり、Rork Max は「開発からビルドまで」を担当し、実際の申請は**開発者が各ストアへ直接行う設計**になっている。これは一見不便に見えるが、実は大きなメリットがある。

## パブリッシング機能のメリット：自由度の高さ

App Store / Google Play の申請には、ビルド後の微調整が必須だ。Rork Max のこの「ビルドして手渡す」設計により：

1. **IPA / APK をローカルで検証できる**
   - TestFlight （iOS）で事前テストが可能
   - Google Play内部テストで Android版を配信前テスト可能

2. **ビルド設定をカスタマイズできる**
   - バージョン番号・ビルド番号を細かく調整
   - Code Signing 証明書を自分で管理

3. **複数のストアに同時申請できる**
   - 同じビルドから IPA と APK を生成可能
   - リリース日の調整も自由

つまり、ノーコードツールにありがちな「ストアに申請したら終わり」ではなく、**デベロッパーとしての自由度を保ったまま、開発フローだけを楽にする**設計になっている。

## Rork Max でのパブリッシング手順

### Step 1: アプリ設定の最終確認

ビルド前に、以下の項目をRork Max で確認・設定する必要がある：

- **アプリ名** — App Store での表示名（15文字以内推奨）
- **バンドル ID** — iOS: `com.yourcompany.appname`、Android: `com.yourcompany.appname`（一致させること）
- **バージョン番号** — 初回リリースは `1.0.0` から開始
- **ビルド番号** — iOS は整数（1, 2, 3...）、Android は自由だが統一推奨
- **アプリアイコン** — 1024×1024px PNG（透過なし・角丸なし）

```
📌 重要: バンドル ID は一度申請すると変更不可
初回設定を間違えると、新規アプリとしての再申請が必須になる
アップデートでバンドル ID を変えることも不可
```

### Step 2: ビルド＆ダウンロード

Rork Max のダッシュボードから：

1. **「Build」または「Publish」ボタンをクリック**
2. **プラットフォームを選択**
   - iOS のみ
   - Android のみ
   - iOS + Android（推奨：同日リリースできる）
3. **ビルド開始**（所要時間：5〜10分）
4. **IPA / APK をダウンロード**

ダウンロード後は、ローカルに保存。このファイルが「配信物」になる。

### Step 3: TestFlight（iOS）による事前検証

App Store へ申請する前に、TestFlight でテストすることを**強く推奨**する。

```
iOS の場合のプロセス：
1. Apple Developer Account にログイン
2. App Store Connect → My Apps で新規アプリを作成
3. 「ビルド」セクションで IPA を TestFlight にアップロード
4. 内部テスター（自分）を追加
5. TestFlight アプリから、実機でテスト
6. 問題なければ、そのビルドを App Store 本番へ提出
```

この段階で、アプリが実機でクラッシュしないか、UI が想定通りか、通信が正常か を確認する。テストなしで申請すると、**審査中にリジェクトされて時間を失う**。

### Step 4: Google Play 内部テストによる事前検証（Android）

Android も同様に、Google Play Console から事前テストが可能だ。

```
Android の場合のプロセス：
1. Google Play Console にログイン
2. 新規アプリを作成
3. 「テスト」→「内部テスト」で APK をアップロード
4. テスターアカウント（自分）を追加
5. Google Play ストアアプリから、テスト版をインストール＆検証
6. OKであれば本番トラック（ベータまたは本公開）へ申請
```

### Step 5: App Store 審査申請

審査に向けて、以下を準備する：

- **アプリ説明文**（170文字以内）
- **キーワード**（最大100文字、コンマ区切り）
- **スクリーンショット**（iOS: 5枚、解像度は機種ごとに異なる）
- **プライバシーポリシー URL**（GDPR 対応が必須）
- **サポートメール**
- **年齢指定**（4+ / 12+ / 17+）

これらを App Store Connect に入力し、「審査に提出」をクリック。**審査所要時間は通常 24〜48時間**。

### Step 6: Google Play 審査申請

Google Play は App Store より審査が緩いが、同様に以下を準備：

- **アプリ説明文**（4,000文字まで）
- **スクリーンショット**（2枚以上、5枚以下）
- **プライバシーポリシー**（データ収集がある場合は必須）
- **プロモーション用動画**（オプション）
- **年齢指定**
- **コンテンツレーティング アンケート**（自動）

Google Play は「自動公開」モードが選べるため、申請直後に公開される場合もある。

## よくある落とし穴

### ❌ 落とし穴1: ビルドなしでアップロードしようとする

Rork Max には「ワンクリック申請」機能がない。必ず IPA / APK をダウンロードしてから、App Store Connect / Google Play Console に手動でアップロードする必要がある。

**解決**: ビルド → ダウンロード → App Store Connect へ という3ステップを習慣化する。

### ❌ 落とし穴2: バンドル ID を変える

一度 App Store に申請したアプリは、バンドル ID を変えることができない。変えると「新しいアプリ」扱いになり、レビュー履歴やレーティングが全部リセットされる。

**解決**: 初回ビルド前に、バンドル ID を必ず確認する。`com.yourname.appname` という形で、社名や個人名を含めた一意のものにする。

### ❌ 落とし穴3: アプリアイコンが 1024×1024 でない

App Store では 1024×1024 PNG が必須。JPEG や透過付き PNG は受け付けない。Figma で書き出す際に解像度・形式を確認することが重要。

**解決**: アイコン用に Figma で 1024×1024 の正方形フレームを作り、エクスポートを PNG + 100% 倍率で行う。

### ❌ 落とし穴4: TestFlight でテストしない

「ノーコードだから多分動く」という思い込みで、いきなり App Store に申請するケースが多い。実機テストなしだと、本来なら2日で審査が通るはずが、リジェクトで1週間失うことになる。

**解決**: 必ず TestFlight で自分の iPhone にインストールしてテスト。ネットワーク環境（Wi-Fi と LTE）の両方で確認する。

## プライバシーポリシーの最小限テンプレート

App Store・Google Play 両方で「プライバシーポリシー」の URL を求められる。簡単なテンプレートを以下に示す：

```
【アプリ名】プライバシーポリシー

本アプリは以下の情報を収集します：

1. 位置情報（オプション機能の場合）
2. カメラロール（写真アップロード機能の場合）
3. ユーザーが入力したテキスト（サーバー保存の場合）

これらの情報は、[あなたのサーバー / Supabase / Firebase]
に保存され、サードパーティと共有されません。

ユーザーは設定から、これらの権限を取り消せます。

更新日: 2026年4月14日
```

Notion や GitHub Pages で公開するだけで十分。HTML ファイルも、シンプルなマークダウンも問題ない。

## リリース後の運用：バージョン更新

アプリをリリースしたら、定期的にアップデートを配信することが推奨される。

**バージョン番号の付け方**:
- 初回: `1.0.0`
- バグ修正: `1.0.1`, `1.0.2`, ...
- 機能追加: `1.1.0`
- 大型アップデート: `2.0.0`

Rork Max でコードを修正 → ビルド → バージョン番号を上げる → IPA / APK をダウンロード → App Store Connect へアップロード、という流れを繰り返す。

## Rork Max パブリッシング機能の今後

2026年内に、以下の機能が計画されている可能性がある（公式アナウンスは未確認）：

- **App Store Connect への直接アップロード**（OAuth 連携）
- **Google Play Console への自動申請**
- **TestFlight 自動配信**

ただ現時点では、「ビルドまで」がRork Max の責任範囲。それ以降は、デベロッパー自身が App Store・Google Play の規則に従う必要がある。

## まとめ

Rork Max のパブリッシング機能は「ビルドから申請までの自動化」ではなく、「開発からビルドまでの簡素化」を目指している。

一見すると不便に見えるが、実は：
- テスト環境と本番環境を分離できる
- ビルド設定をカスタマイズできる
- 複数ストアへの同時申請が可能

という利点がある。つまり、**ノーコード開発の利便性と、ネイティブアプリの自由度を両立させた設計**なのだ。
