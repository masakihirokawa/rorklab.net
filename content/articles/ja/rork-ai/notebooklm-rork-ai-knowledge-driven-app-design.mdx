## はじめに

NotebookLMは単なる「ドキュメント読み込み機能」ではありません。あなたのアプリ開発チームが蓄積する競合分析、設計仕様書、ユーザーリサーチをすべてNotebookLMに「学習」させることで、**AI知識ベース駆動のプロトタイピング**が可能になります。

このガイドでは、NotebookLM Enterprise APIとRorkを組み合わせ、以下を実現する方法を解説します。

- **競合アプリ分析の自動化**: PDFスクリーンショット、デザイン仕様をNotebookLMに投入し、UI/UXパターン抽出
- **設計仕様書の活用**: フロントエンドアーキテクチャ、API設計をNotebookLMに学習させ、開発着手前に質問可能
- **ユーザーリサーチの統合**: インタビュー記録、ユーザーフローをNotebookLMに集約し、要件定義の質向上
- **知識駆動型コード生成**: NotebookLMの分析結果をRorkへ自動送信し、コンテキストに基づいたコード生成

## 1. NotebookLM Enterprise APIの基礎とRork連携アーキテクチャ

### 1.1 NotebookLM APIの全体像

NotebookLM Enterprise APIは、以下の機能を提供します。

| 機能 | 説明 | Rork連携時の活用 |
|---|---|---|
| **Document Upload** | PDF/EPUB/CSV/画像（OCR対応）をNotebookLMに登録 | 設計仕様書・競合アプリのスクリーンショットをアップロード |
| **Knowledge Base** | 複数ドキュメントをまとめた知識ベース作成 | プロジェクト単位で知識ベース作成、チーム共有 |
| **Q&A API** | テキストクエリを投げ、知識ベースから回答取得 | アプリ開発中に「このUI パターンはどう実装する？」と質問 |
| **Podcast API** | ドキュメント内容をポッドキャスト形式で生成 | 通勤時間に設計仕様書をオーディオ化 |
| **Notebook Export** | JSON/CSV形式で知識抽出 | Rork側で構造化された知識を取得 |
| **MCP Integration** | Claude MCPプロトコルでAPI連携 | Rork+Claude Coworkで設計→実装の連続運用 |

## 2. NotebookLMでの競合分析ワークフロー

### 2.1 競合アプリのスクリーンショットをOCR・分析

実装例：競合アプリのUIスクリーンショット群をNotebookLMに投入し、パターンを抽出

```python
# NotebookLM API integration for competitive analysis
import anthropic
import base64
import json
from pathlib import Path

class CompetitiveAnalysisAgent:
    """
    NotebookLMでの競合分析を自動化。
    UIスクリーンショット → OCR → パターン抽出 → Rorkへフィード
    """

    def __init__(self, api_key: str, notebook_id: str):
        self.client = anthropic.Anthropic(api_key=api_key)
        self.notebook_id = notebook_id

    def upload_competitor_screenshots(self, screenshots_dir: str) -> list[dict]:
        """
        競合アプリのスクリーンショットをNotebookLMに投入
        """
        uploads = []
        for screenshot_path in Path(screenshots_dir).glob("*.png"):
            with open(screenshot_path, "rb") as f:
                image_data = base64.b64encode(f.read()).decode()

            # NotebookLM APIでスクリーンショットをアップロード
            result = self.client.messages.create(
                model="claude-opus-4-1",
                max_tokens=1500,
                messages=[
                    {
                        "role": "user",
                        "content": [
                            {
                                "type": "image",
                                "source": {
                                    "type": "base64",
                                    "media_type": "image/png",
                                    "data": image_data,
                                },
                            },
                            {
                                "type": "text",
                                "text": f"""
このUIスクリーンショットを分析してください。以下を JSON形式で出力:
{{
  "app_name": "アプリ名",
  "screen_type": "ホーム|検索|プロフィール|設定等",
  "ui_patterns": ["パターン1", "パターン2"],
  "color_scheme": "色スキーム説明",
  "typography": "フォント・サイズ",
  "interaction_cues": ["CTA1", "CTA2"],
  "notable_features": ["特徴1", "特徴2"],
  "accessibility_notes": "アクセシビリティに関する観察"
}}
"""
                            }
                        ],
                    }
                ]
            )

            uploads.append({
                "file": screenshot_path.name,
                "analysis": result.content[0].text
            })

        return uploads

    def extract_ui_patterns(self, competitor_analyses: list[dict]) -> dict:
        """
        複数の競合分析から共通UIパターンを抽出
        """
        analyses_text = "\n".join([
            f"### {item['file']}\n{item['analysis']}"
            for item in competitor_analyses
        ])

        response = self.client.messages.create(
            model="claude-opus-4-1",
            max_tokens=2000,
            messages=[
                {
                    "role": "user",
                    "content": f"""
以下の複数アプリのUI分析から、業界共通パターンと革新的パターンを抽出してください。
JSON形式で返してください。

{analyses_text}

{{
  "common_patterns": {{"pattern_name": "説明・実装例"}},
  "innovative_patterns": {{"pattern_name": "説明・採用理由"}},
  "gaps": ["市場で見つからないパターン1"],
  "implementation_recommendations": ["推奨実装1"]
}}
"""
                }
            ]
        )

        return json.loads(response.content[0].text)

# 使用例
agent = CompetitiveAnalysisAgent(
    api_key="your-anthropic-key",
    notebook_id="your-notebook-id"
)

# Step 1: 競合アプリのスクリーンショットをアップロード
analyses = agent.upload_competitor_screenshots("./competitor_screenshots")

# Step 2: パターンを抽出
patterns = agent.extract_ui_patterns(analyses)

# Step 3: 結果をRorkの設計ツールに送信
print(json.dumps(patterns, ensure_ascii=False, indent=2))
```

## 3. NotebookLMの知識ベースをRorkアプリに統合

### 3.1 MCP経由での双方向連携

NotebookLMのMCP integration を使うことで、Rorkアプリ開発中に リアルタイムで設計仕様書を参照できます。

```python
# MCP Server実装例：NotebookLM知識ベースを Rork内で参照可能に

from mcp.server import Server
from mcp.types import TextContent, Tool
import anthropic
import json

class NotebookLMMCPServer:
    """
    MCP経由でNotebookLM知識ベースを公開。
    Rork内で「設計仕様書から実装案を生成」という質問が可能に。
    """

    def __init__(self, notebook_id: str, api_key: str):
        self.server = Server("notebooklm-rork")
        self.notebook_id = notebook_id
        self.client = anthropic.Anthropic(api_key=api_key)
        self.setup_tools()

    def setup_tools(self):
        """MCPで公開するToolを定義"""

        @self.server.list_tools()
        async def list_tools():
            return [
                Tool(
                    name="query_design_spec",
                    description="設計仕様書からUIパターン・API仕様を検索",
                    inputSchema={
                        "type": "object",
                        "properties": {
                            "question": {
                                "type": "string",
                                "description": "設計仕様書に対する質問。例：『LoginフローのAPI設計は？』"
                            }
                        },
                        "required": ["question"]
                    }
                ),
                Tool(
                    name="suggest_implementation",
                    description="NotebookLMの知識に基づき、実装案を提案",
                    inputSchema={
                        "type": "object",
                        "properties": {
                            "component_name": {
                                "type": "string",
                                "description": "実装対象のコンポーネント名"
                            },
                            "technology": {
                                "type": "string",
                                "description": "使用技術（React Native/Expo等）"
                            }
                        },
                        "required": ["component_name", "technology"]
                    }
                ),
                Tool(
                    name="extract_data_models",
                    description="設計仕様書からデータモデル定義を抽出",
                    inputSchema={
                        "type": "object",
                        "properties": {
                            "entity": {
                                "type": "string",
                                "description": "抽出対象のエンティティ（User/Post/Comment等）"
                            }
                        },
                        "required": ["entity"]
                    }
                )
            ]

        @self.server.call_tool()
        async def call_tool(name: str, arguments: dict):
            if name == "query_design_spec":
                return await self.query_design_spec(arguments["question"])
            elif name == "suggest_implementation":
                return await self.suggest_implementation(
                    arguments["component_name"],
                    arguments["technology"]
                )
            elif name == "extract_data_models":
                return await self.extract_data_models(arguments["entity"])

    async def query_design_spec(self, question: str) -> str:
        """NotebookLM知識ベースに質問を投げ、回答を取得"""
        response = self.client.messages.create(
            model="claude-opus-4-1",
            max_tokens=2000,
            messages=[
                {
                    "role": "user",
                    "content": f"""
あなたはこのアプリの設計仕様書を全て理解しています。
以下の質問に、設計仕様書の内容に基づいて答えてください。

質問：{question}

返答形式：
- APIエンドポイントが関わる場合は、パス・メソッド・パラメータを明記
- UIコンポーネントが関わる場合は、構造と責務を明記
- データフローが関わる場合は、図解を含める
"""
                }
            ]
        )
        return response.content[0].text

    async def suggest_implementation(self, component: str, tech: str) -> str:
        """
        NotebookLM知識ベース + 指定技術に基づく実装案を生成
        """
        response = self.client.messages.create(
            model="claude-opus-4-1",
            max_tokens=3000,
            messages=[
                {
                    "role": "user",
                    "content": f"""
設計仕様書に記載されている '{component}' を、{tech} で実装してください。

要件：
- 設計仕様書の要件を完全に満たす
- ベストプラクティスに従う
- TypeScript型定義を含める
- 部分的ではなく、完全な実装例を提示

返答形式：
\`\`\`{tech.lower()}
// 完全なコンポーネント実装
\`\`\`

設計上の注意点：
- 設計仕様書との整合性
- パフォーマンス最適化
- アクセシビリティ対応
"""
                }
            ]
        )
        return response.content[0].text

    async def extract_data_models(self, entity: str) -> str:
        """設計仕様書からエンティティのスキーマを抽出"""
        response = self.client.messages.create(
            model="claude-opus-4-1",
            max_tokens=1500,
            messages=[
                {
                    "role": "user",
                    "content": f"""
設計仕様書から '{entity}' エンティティの完全なデータモデル定義を抽出してください。

返答形式（TypeScript）：
\`\`\`typescript
interface {entity} {{
  // フィールド定義
}}

// バリデーションロジック
export const validate{entity} = (data: unknown): {entity} => {{
  // バリデーション実装
}}
\`\`\`

必ず含めるもの：
- 全フィールドの型定義
- NULL可能性の明記
- デフォルト値
- バリデーション規則
- API通信時の変換ロジック（あれば）
"""
                }
            ]
        )
        return response.content[0].text

    async def run(self):
        """MCPサーバーを起動"""
        async with self.server:
            print(f"NotebookLM MCP Server running for notebook: {self.notebook_id}")

# 実行例
if __name__ == "__main__":
    import asyncio
    server = NotebookLMMCPServer(
        notebook_id="your-notebook-id",
        api_key="your-anthropic-key"
    )
    asyncio.run(server.run())
```

```typescript
import { MCPClient } from '@rork/mcp-client';

interface ComponentGenerationRequest {
  name: string;
  technology: 'react-native' | 'web' | 'flutter';
  context: string;
}

async function generateComponentWithDesignSpec(
  request: ComponentGenerationRequest
): Promise<string> {
  const mcpClient = new MCPClient('notebooklm-rork');

  const finalCode = await generateFinalComponent({
    implementation,
    dataModels,
    userContext: request.context,

  });

  return finalCode;
}
```

### 5.1 NotebookLM → Rork → デプロイのエンドツーエンド

```

1. 競合アプリのスクリーンショット 10枚 → NotebookLMにアップロード
2. 設計仕様書（PDF 50ページ） → NotebookLMにアップロード

【Day 3-4: 実装計画】
```

### 5.2 継続的な知識ベース更新

```typescript
// Rork内での自動ログ・フィードバック収集とNotebookLM知識ベース更新

interface DevelopmentLog {
  date: string;
  component: string;
  decision: string;
  rationale: string;
  outcome: 'success' | 'partial' | 'failed';
  lessons: string[];
}

async function updateNotebookLMKnowledgeBase(log: DevelopmentLog) {
  /**
   * アプリ開発中の決定・学習を記録。
   * NotebookLMの知識ベースに定期的に追加。
   * 次のプロジェクトで「我々のチームは過去このパターンで失敗した」という学習を反映。
   */

  const documentContent = `
# ${log.component} - ${log.date}

## 決定内容
${log.decision}

## 根拠
${log.rationale}

## 結果
${log.outcome}

## 得られた教訓
${log.lessons.map(l => `- ${l}`).join('\n')}
`;

  // NotebookLM APIで新しいドキュメントを追加
  const response = await fetch('https://notebooklm.googleapis.com/v1/notebooks/{notebookId}/documents', {
    method: 'POST',
    headers: {
      'Authorization': `Bearer ${API_KEY}`,
      'Content-Type': 'application/json'
    },
    body: JSON.stringify({
      title: `Development Decision: ${log.component}`,
      content: documentContent
    })
  });

  console.log('Added to NotebookLM knowledge base:', response.ok);
}
```

## 6. FAQ

### Q1. NotebookLMのコスト？複数プロジェクトで共有可能か？

NotebookLM Enterpriseは月額契約モデル。複数ノートブック（知識ベース）を作成でき、プロジェクト / チーム単位での分け方は自由です。

- **1ノートブック** = 1プロジェクト（推奨）
- **複数ノートブック** = 複数プロジェクト共有も可

コスト最適化のため、年1-2本のメジャープロジェクトなら1ノートブック＋定期アップデートが効率的。

### Q2. NotebookLM知識ベースをRork以外のツール（Figma、Notion）と連携させたい

MCPプロトコルはオープン標準です。

- **Figma**: Figma Plugin APIと組み合わせ、NotebookLMから抽出した「推奨UIパターン」をFigmaコンポーネントライブラリに自動反映
- **Notion**: Notion APIで知識ベースの更新結果をNotion DBに自動保存

詳細は [MCP Official Integration Guide](/articles/rork-ai/ai-code-generation-workflow) を参照。

### Q3. OCR精度が悪い（日本語テキスト認識率が低い）場合の対策

NotebookLMのOCR（Google Document AI）は英語が最適化済み。日本語は70-80%精度です。

**改善案**:
1. **事前処理**: スクリーンショット画質を上げる（300 DPI推奨）
2. **ハイブリッド**: テキスト部分を別途テキストファイルで追加
3. **代替案**: PDF生成時に埋め込みテキストレイヤーを含める

```bash
# スクリーンショット画質向上（ImageMagick）
convert input.png -resize 200% output.png

# PDFテキスト埋め込み（macOS）
pdftotext input.pdf temp.txt  # テキスト抽出
# → テキスト確認後、NotebookLMに投入
```

### Q4. セキュリティ：Rork開発情報がNotebookLMに保存される。データ漏洩リスク？

NotebookLMはGoogle Cloud上で運用。エンタープライズプランでは：

- **暗号化**: TLS 1.3 + at-rest encryption
- **アクセス制御**: Google SSO、IP制限、Team Workspaceでのロール分離
- **監査ログ**: 全アクセスを記録
- **データ保持ポリシー**: 契約期間満了後、90日以内に削除可能

**推奨設定**：
- ノートブック単位でチームを分ける（重要な機密情報は別ノートブック）
- ユーザーAPIキーは環境変数で管理（コード内に埋め込まない）
- 月次監査ログ確認

## 7. まとめと推奨実装パス

NotebookLM × Rorkを組み合わせることで、以下が実現します。

| 従来型開発 | NotebookLM駆動型開発 |
|---|---|
| 設計仕様書を読む（1-2時間） | NotebookLMに質問・分析（5-10分） |

| 競合分析は別プロセス | 統合分析（プロジェクト全体で一貫性） |
| コード生成に重複が生じる | Rorkが知識ベースから最適案を生成 |

1. **Week 1**: NotebookLM Enterpriseアカウント取得 → 既存設計仕様書を1冊アップロード → 質問テスト
2. **Week 2**: MCP Server実装 → Rorkとの連携テスト → 競合分析1回実施
3. **Week 3**: ユーザーリサーチをNotebookLMに統合 → 要件検証
4. **Week 4+**: 本番プロジェクトでNotebookLM駆動開発を実施
