## なぜ「プロンプトの書き方」がRork開発の分水嶺になるか

Rork AIのエンジンは内部的に大規模言語モデル（LLM）を使用しています。LLMは「確率的に最もそれらしい出力」を返すため、入力の質が出力の質を直接決定します。曖昧なプロンプトには曖昧な実装が返ってくる——これはRorkの制限ではなく、生成AI全体の本質的な特性です。

一方で、LLMには「適切な文脈が与えられれば、シニアエンジニア水準のコードを生成できる」という驚異的な能力もあります。この能力を引き出すのがプロンプトエンジニアリングです。

実際、同じ「チャット機能を作って」というタスクでも、プロンプトの質によって結果は大きく変わります。

**❌ 品質の低いプロンプトの例:**
```
チャット機能を作ってください
```

**✅ 品質の高いプロンプトの例:**
```
【機能要件】
リアルタイムチャット画面を実装してください。

【技術スタック】
- Supabase Realtime でメッセージを購読
- メッセージモデル: { id, userId, content, createdAt }
- FlatList で過去メッセージを表示（最新が下）

【UI仕様】
- 画面下部に固定テキスト入力 + 送信ボタン
- 自分のメッセージは右寄せ（青背景）、相手は左寄せ（グレー背景）
- キーボード表示時に入力欄が隠れないよう KeyboardAvoidingView を使用

【禁止事項】
- モックデータは使わず、Supabaseから実際に取得すること
- TypeScript の any 型は使用しないこと
```

同じ2秒で書けるようになるまで慣れれば、生成品質は劇的に向上します。以降のセクションで、この「品質の高いプロンプト」を体系的に作れるようになるための技術を解説します。

## Rork AIが"理解する"仕組みを知る ― コンテキストウィンドウとトークン管理

効果的なプロンプトを書くには、Rork AIがどのように入力を処理しているかを理解する必要があります。

### コンテキストウィンドウとは

LLMには「コンテキストウィンドウ」と呼ばれる、一度に処理できるテキスト量の上限があります。Rorkが内部で使用するモデルによって異なりますが、一般的に数万〜数十万トークン（1トークン≒1〜2文字）です。

重要なのは、**コンテキストウィンドウは会話全体の累積である**という点です。チャットの最初から最後までの全メッセージが蓄積されており、ウィンドウが溢れると古い情報から切り捨てられます。

### 長い会話セッションでの品質低下を防ぐ

Rorkで1つのプロジェクトを長期間編集し続けると、生成品質が落ちてくることがあります。これはコンテキストウィンドウの汚染が原因です。

**対策1: 「リセットプロンプト」を使う**
```
これまでの実装を整理します。
現在のプロジェクト状態:
- 実装済み: [完成した機能一覧]
- 現在のファイル構成: [主要ファイルのリスト]
- 使用中のライブラリ: [依存関係]

上記の文脈で、次のタスクに取り組んでください:
[新しい指示]
```

**対策2: 機能ごとに新しいチャットを開始する**
大きな機能追加は、新しいセッションで「現在の実装の概要」を共有してから始めると安定します。

### Rorkがコードを「読む」方法

Rorkはプロンプトだけでなく、既存のコードベースもコンテキストとして参照します。このため:

- **コード内のコメントは重要な情報源**になります。適切なコメントを書かれたコードは、後続の生成精度を高めます
- **ファイル名・変数名の一貫性**がAIの理解を助けます
- **型定義（TypeScript）**があると、AIが正確な実装を生成しやすくなります

## プロンプト構造の7原則 ― 思い通りのUIを初回で生成するために

長年の試行錯誤から導き出した、Rork向けプロンプトの7つの基本原則を紹介します。

### 原則1: 「何を」「どのように」「なぜ」を分けて書く

```
【WHAT: 何を作るか】
ユーザープロフィール編集画面

【HOW: どのように実装するか】
- フォーム: react-hook-form + zod でバリデーション
- 保存先: Supabaseのprofilesテーブル
- 画像: Expo Image Picker でアバター変更可能

【WHY: なぜその実装か（制約・背景）】
- 既存のAuthContextからuserオブジェクトを参照すること
- 保存成功後はhome画面に遷移する
```

### 原則2: 禁止事項を明示する

AIはデフォルトで「それらしい実装」を選びます。意図しない選択を避けるために禁止事項を明記します。

```
【禁止事項】
- useState によるローカル状態管理のみは禁止（Zustandを使うこと）
- console.logのデバッグコードを残さないこと
- 外部に公開されたAPIキーを直接コードに書かないこと
- 既存の [fileName].tsx を削除・上書きしないこと
```

### 原則3: 期待する出力形式を指定する

```
以下の形式で実装してください:
1. まず実装方針を3行で説明する
2. 次にコードを提示する
3. 最後に「テスト手順」を箇条書きで書く
コードはTypeScriptで、型定義を省略しないこと。
```

### 原則4: サイズと複雑度を制御する

AIは「多ければ多いほど良い」と判断しがちです。不必要なコードの追加を防ぐために範囲を限定します。

```
変更範囲を次のファイルのみに限定してください:
- src/screens/ProfileScreen.tsx
- src/hooks/useProfile.ts

他のファイルは触らないこと。既存の実装を大幅に書き換えないこと。
```

### 原則5: 既存コードの参照方法を指定する

```
現在の実装を参考にしてください。
既存のButtonコンポーネント（src/components/Button.tsx）と同じスタイル規則を踏襲し、
既存のuseAuth（src/hooks/useAuth.ts）から認証情報を取得してください。
新しいコンポーネントは同じパターンで実装すること。
```

### 原則6: エラーハンドリングを必ず要求する

```
以下のエラーケースを必ずハンドリングしてください:
- ネットワークエラー時: ユーザーへのフィードバック表示
- 認証エラー時: ログイン画面へのリダイレクト
- バリデーションエラー時: フィールドごとのエラーメッセージ表示
- loading状態: ローディングスピナーの表示
```

### 原則7: ステップバイステップで進める

複雑な機能を一度に実装しようとすると、生成品質が下がります。段階的に進めましょう。

```
この機能を3つのフェーズに分けて実装します。
今回はフェーズ1のみ実装してください:

フェーズ1: UIコンポーネントのみ（データは仮）
フェーズ2: APIとの接続
フェーズ3: エラーハンドリングとエッジケース対応
```

## 画面・コンポーネント生成の実践プロンプトパターン集

ここからは、具体的なユースケースごとのプロンプトパターンを解説します。

### パターン1: 新規画面の作成

```
【新規画面: タスク一覧画面】

コンポーネント名: TaskListScreen
ファイルパス: src/screens/TaskListScreen.tsx

【表示データ】
type Task = {
  id: string;
  title: string;
  completed: boolean;
  dueDate: Date;
  priority: 'low' | 'medium' | 'high';
};

【UI要件】
- FlatListでタスクを表示
- 完了済みタスクはチェックマーク + 取り消し線
- priorityに応じた左ボーダー色（low=グレー, medium=黄, high=赤）
- スワイプで削除（react-native-gesture-handler）
- 空の場合はイラスト + 「タスクがありません」テキスト

【ナビゲーション】
- 右上の「+」ボタンでTaskCreateScreenへ遷移
- タスクタップでTaskDetailScreenへ遷移（idをパラメータで渡す）

【スタイル方針】
- NativeWindを使用（Tailwind CSS記法）
- テーマカラー: primary=#4F46E5, background=#F9FAFB
```

### パターン2: 既存コンポーネントの改修

```
【改修依頼: ボタンコンポーネントのバリアント追加】

既存ファイル: src/components/Button.tsx

【追加するバリアント】
variant: 'primary' | 'secondary' | 'danger' | 'ghost'

現在: primaryのみ実装済み
追加: secondary（白背景・ボーダー）、danger（赤背景）、ghost（背景なし・テキストのみ）

【要件】
- 既存のprimaryの実装は変更しないこと
- loading propのスピナーを全バリアントで動作させること
- TypeScriptの型定義を更新すること
- 変更はButtonコンポーネントファイルのみに限定すること

【使用例（参考）】
<Button variant="danger" onPress={handleDelete} loading={isDeleting}>
  削除する
</Button>
```

### パターン3: フォームの実装

```
【フォーム実装: ユーザー登録フォーム】

ライブラリ: react-hook-form + zod

【バリデーションルール】
- email: 有効なメールアドレス形式
- password: 8文字以上、英数字混在
- confirmPassword: passwordと一致
- username: 3〜20文字、英数字とアンダースコアのみ

【UX要件】
- リアルタイムバリデーション（onChangeでの即座フィードバック）
- サブミット前に全フィールドが有効でなければボタンをdisabled
- 送信中はボタンにローディング表示
- エラーメッセージはフィールド直下に赤テキストで表示

【送信後の処理】
- 成功: サーバーへのAPIコール → HomeScreenへ遷移
- 失敗: エラーメッセージをトースト通知で表示（react-native-toast-messageを使用）
```

### パターン4: リスト + 無限スクロール

```
【無限スクロールリストの実装】

データソース: Supabase（postsテーブル）
ページサイズ: 20件

【実装要件】
- 初期表示: 最新20件を取得
- スクロール末尾に到達したら次の20件をフェッチ
- フェッチ中: フッターにActivityIndicator表示
- 全件取得済み: フッターにメッセージ表示
- Pull-to-Refresh: 先頭から再取得

【パフォーマンス要件】
- FlatListのinitialNumToRenderとmaxToRenderPerBatchを最適化
- getItemLayoutを実装（固定高さの場合）
- keyExtractorは必ずid属性を使用

【エラーハンドリング】
- フェッチ失敗時: フッターに「再試行」ボタン表示
- ネットワーク切断時: オフライン通知バナー表示
```

## API連携・状態管理のプロンプト設計技術

バックエンドとの連携は、プロンプトの書き方が特に重要です。

### API連携の基本パターン

```
【API連携実装: 商品一覧取得】

エンドポイント: GET /api/products
レスポンス型:
{
  data: Array<{
    id: string;
    name: string;
    price: number;
    imageUrl: string;
    category: string;
  }>;
  total: number;
  page: number;
}

【実装方針】
- TanStack Query（react-query）を使用すること
- queryKey: ['products', { page, category }]
- staleTime: 5分（商品データはそれほど頻繁に変わらない）
- エラー時: クエリが3回自動リトライされること

【環境変数】
- APIのベースURLは process.env.EXPO_PUBLIC_API_URL から取得
- APIキーは Authorization ヘッダーに Bearer トークンとして付与
  （トークンはAsyncStorageに保存済みの認証トークンを使用）

【Hooks設計】
- useProducts(page, category) カスタムフックとして実装
- { products, isLoading, error, refetch, fetchNextPage } を返す
```

### 状態管理（Zustand）のプロンプトパターン

```
【Zustandストア実装: カートストア】

ファイル: src/stores/cartStore.ts

【状態定義】
type CartItem = {
  productId: string;
  name: string;
  price: number;
  quantity: number;
};

type CartState = {
  items: CartItem[];
  // 派生状態（getters）
  totalItems: number;
  totalPrice: number;
  // アクション
  addItem: (product: Product) => void;
  removeItem: (productId: string) => void;
  updateQuantity: (productId: string, quantity: number) => void;
  clearCart: () => void;
};

【実装要件】
- zustand/middleware の persist でAsyncStorageに永続化
- totalItems, totalPrice は computed（メモ化が必要）
- addItem: 既に存在する場合はquantityを+1
- updateQuantity: quantity=0になったら自動でremoveItem
- immer ミドルウェアを使用して不変性を保証

【テスト可能性】
- テスト用にストアをリセットできるresetCart関数を追加
```

## エラー修正とリファクタリングのプロンプトテクニック

エラー修正こそ、プロンプトの質が最も問われる場面です。

### エラー修正の正しいプロンプト

エラーが起きたとき、ただエラーメッセージを貼り付けるだけでは不十分です。コンテキストを添えることで、的確な修正が得られます。

```
【エラー修正依頼】

エラーメッセージ:
```
TypeError: Cannot read properties of undefined (reading 'map')
  at ProductList.tsx:45
```

発生条件:
- アプリ起動直後のProductList画面で発生
- データが取得される前に画面がレンダリングされるタイミング

現在のコード（問題のある部分）:
```typescript
const ProductList = () => {
  const { products } = useProducts();
  return (
    <FlatList
      data={products.data}  // ← ここでエラー
      ...
    />
  );
};
```

修正方針:
- useProductsの戻り値が初期状態でundefinedにならないよう修正
- または、products.dataが存在しない場合のfallback処理を追加
- loading状態中はスケルトンローダーを表示
```

### リファクタリングのプロンプトパターン

```
【リファクタリング依頼】

対象ファイル: src/screens/HomeScreen.tsx
現在の問題:
- コンポーネントが400行超えで読みにくい
- ビジネスロジックとUIが混在している
- 同じデータフェッチのロジックが3箇所に重複

リファクタリング方針:
1. カスタムフック（useHomeData）にデータフェッチを分離
2. 以下のサブコンポーネントに分割:
   - HeroBanner.tsx
   - FeaturedProductsSection.tsx
   - RecentOrdersSection.tsx
3. 重複しているfetchWithErrorHandling関数をutils/api.tsに移動

【制約】
- 外部から見た動作・表示は変えないこと
- TypeScriptの型を強化する（anyをなくす）
- コメントを追加して意図を明確にする
```

### デバッグ支援のプロンプト

```
【デバッグ支援依頼】

問題: 決済ボタンを押すと、2回目以降に押したときだけ決済が完了する

再現手順:
1. 商品詳細画面で「購入する」ボタンを押す
2. 1回目: ローディング後にエラー（ただしコンソールにエラーなし）
3. 2回目: 正常に決済完了

関連ファイル:
- src/screens/ProductDetailScreen.tsx
- src/hooks/useCheckout.ts
- src/api/stripe.ts

デバッグのために追加してほしいログ:
- checkoutハンドラーの各ステップでのstate値
- Stripe APIコール前後のレスポンス
- useCheckoutフック内の状態変化

予測される原因（確認してほしい点）:
- 非同期処理の競合（二重送信防止の実装漏れ？）
- セッション初期化の非同期タイミング問題
```

## 本番品質を保証するプロンプトテンプレート20選

以下に、そのままコピーして使えるプロンプトテンプレートを用途別にまとめます。

### 【認証系】

**テンプレート1: ソーシャルログイン実装**
```
Apple SignIn + Google SignIn を実装してください。
- @clerk/clerk-expo を使用
- ログイン成功後: onLoginSuccess() コールバックを呼ぶ
- エラー時: platform-specific のエラーメッセージを表示
- ローディング中: 各ボタンをdisabledにしてスピナー表示
- 既存のEmailPasswordLoginとの共存を壊さないこと
```

**テンプレート2: 認証状態の永続化**
```
アプリ再起動後も認証状態を維持するために:
- セキュアストレージ（expo-secure-store）にトークンを保存
- AppLoadingコンポーネントでトークン検証
- 有効: HomeScreenへ、無効/期限切れ: LoginScreenへ
- リフレッシュトークンの自動更新ロジックも実装
```

### 【データ表示系】

**テンプレート3: グラフ・チャートの実装**
```
Victory Native を使って売上グラフを実装してください。
- 折れ線グラフ（週次データ、過去8週間）
- タップしたポイントにツールチップ表示
- X軸: 「4/1」形式の日付
- Y軸: 「¥1,000」形式の通貨表示
- アニメーション: 表示時にポイントが順番に現れる
- データがない場合: 「データがありません」の空状態表示
```

**テンプレート4: 検索機能の実装**
```
Algolia を使ったリアルタイム検索を実装してください。
- テキスト入力から300msの debounce
- 検索中: テキストフィールド右にインジケーター
- 結果: SearchResultCardコンポーネントで表示
- 入力クリアボタン（×）
- 検索結果0件時の空状態UI
- 最近の検索履歴（最大5件、AsyncStorageで永続化）
```

### 【パフォーマンス系】

**テンプレート5: 画像最適化**
```
重い画像リストのパフォーマンスを改善してください。
- expo-image を使用（react-native-fast-imageの代替）
- placeholder: ブラー効果付きのplaceholder
- contentFit: 'cover' でアスペクト比維持
- recyclingKey でメモリ効率化
- FlatListのgetItemLayoutで高さを固定してスクロールを滑らかに
- 画像URLを変換してWebPフォーマットを優先
```

**テンプレート6: アニメーション**
```
react-native-reanimated を使ってカード展開アニメーションを実装してください。
- 初期状態: カード高さ60px（タイトルのみ）
- タップ後: カード高さ240pxに滑らかに展開（duration: 300ms）
- イージング: Easing.out(Easing.cubic)
- 展開中は矢印アイコンが180度回転
- useSharedValue + useAnimatedStyle で実装
- Android/iOSの両方で動作確認すること
```

### 【セキュリティ系】

**テンプレート7: 生体認証**
```
expo-local-authentication を使って生体認証ゲートを実装してください。
- アプリ起動時 + バックグラウンドから復帰時に認証要求
- Face ID/Touch ID が利用可能かチェック
- 非対応デバイス: PINコード入力フォールバック
- 3回失敗: 一定時間ロックアウト
- 認証成功まで画面コンテンツを隠すオーバーレイ
```

**テンプレート8: APIキー保護**
```
APIキーを安全に管理する仕組みを実装してください。
- 全てのシークレットは .env ファイルと process.env.EXPO_PUBLIC_* で管理
- コードにAPIキーを直書きしないこと（git historyからも除外）
- クライアントに公開できないキー（Secret Key等）はバックエンドProxy経由でアクセス
- expo-secure-store でランタイムのトークン管理
```

## よくある失敗パターンとその対処法

プロンプトエンジニアリングを習得する過程でよく起きる失敗と、その回避方法を解説します。

### 失敗1: 複数の変更を一度に依頼する

**問題のある依頼:**
```
ユーザー認証を実装して、プロフィール画面を作って、
設定画面も追加して、プッシュ通知も設定してください。
```

**改善策:** 機能を1つずつ依頼し、各ステップで動作確認してから次に進む。

### 失敗2: スタイルの表現が抽象的すぎる

**問題のある依頼:**
```
いい感じのおしゃれなデザインにしてください
```

**改善策:**
```
Apple Human Interface Guidelinesに準拠したシンプルなデザインで:
- フォント: SF Pro（システムフォント）
- カラー: ライトモード #1C1C1E / ダークモード #FFFFFF
- 角丸: 12px
- シャドウ: なし
- ボタン高さ: 50px
- 余白: 16pxグリッド基準
```

### 失敗3: 「直して」だけでは伝わらない

**問題のある依頼:**
```
なんか動かないので直してください
```

**改善策:**
```
問題: [具体的な症状]
発生タイミング: [いつ起きるか]
期待する動作: [こうなってほしい]
エラーメッセージ: [あれば貼り付け]
試したこと: [すでに試した解決策]
```

### 失敗4: 既存実装との整合性を伝えない

新機能を追加する際、既存のアーキテクチャを無視した実装が生成されることがあります。

**改善策:**
```
プロジェクトの現在のアーキテクチャに合わせて実装してください:
- 状態管理: Zustand（Reduxは使っていない）
- スタイリング: NativeWind
- ナビゲーション: expo-router（React Navigationではない）
- データフェッチ: TanStack Query
- バックエンド: Supabase
```

## プロンプトを"開発資産"として蓄積する方法

優れたプロンプトは一度書いたら財産です。再利用可能な形で管理することで、開発効率が複利的に向上します。

### プロンプトライブラリの構築

プロジェクトのルートに `prompts/` フォルダを作り、用途別にプロンプトを保存します。

```
prompts/
├── components/
│   ├── form-with-validation.txt
│   ├── infinite-scroll-list.txt
│   └── modal-with-animation.txt
├── api/
│   ├── rest-api-integration.txt
│   └── realtime-subscription.txt
└── debugging/
    ├── typescript-error.txt
    └── rendering-performance.txt
```

### 「プロジェクトコンテキストプロンプト」の作成

プロジェクト固有の情報を1つのファイルにまとめ、新機能を依頼する際に最初に貼り付ける「コンテキストプロンプト」を作ります。

```
# プロジェクトコンテキスト

## 概要
[アプリ名]: [一言説明]

## 技術スタック
- React Native + Expo SDK 52
- ナビゲーション: expo-router v4
- 状態管理: Zustand + TanStack Query
- UI: NativeWind v4 + shadcn/ui
- バックエンド: Supabase
- 決済: RevenueCat + Stripe

## コーディング規約
- 全てTypeScript、anyは禁止
- コンポーネントは関数コンポーネントのみ
- カスタムフックで副作用を分離する
- エラーはError Boundaryでキャッチ

## ファイル命名規則
- コンポーネント: PascalCase（例: UserCard.tsx）
- フック: camelCase（例: useUser.ts）
- ユーティリティ: camelCase（例: formatDate.ts）
```

このコンテキストプロンプトを会話の最初に毎回貼り付けることで、AIが常にプロジェクトの文脈を把握した状態で回答するようになります。

## よくある質問（FAQ）

**Q1. プロンプトが長くなると、かえって生成品質が落ちることがありませんか？**

A. 確かに、関係のない情報が多いとAIが混乱することがあります。長いプロンプトを書く際は「今回のタスクに直接関係する情報だけ」に絞ることが重要です。フォーマットは構造化（見出し・箇条書き）して読みやすくし、矛盾する指示が含まれていないか確認しましょう。原則的に、1回のプロンプトでは「1つの機能・1つの問題」に集中するのが最も効果的です。

**Q2. 日本語プロンプトと英語プロンプトで品質差はありますか？**

A. Rork AIの基盤モデルは英語データで主に学習されているため、技術的なプロンプト（特にコード仕様の記述）は英語のほうが精度が上がる場合があります。ただし、最新のモデルでは日本語でも十分な品質が得られます。試してみて、品質に不満を感じる場合は英語に切り替えるのが現実的なアプローチです。コードのコメントを英語にするだけでも効果があります。

**Q3. Rork Max と Rork の通常版でプロンプト戦略に違いはありますか？**

A. Rork Maxはネイティブコード（Swift/SwiftUI）も生成できるため、より詳細なネイティブ固有の仕様をプロンプトに含めることが有効です。また、Rork Maxはより大きなコンテキストウィンドウを持つため、長めの実装仕様書を渡しても品質が維持されやすいです。通常版Rorkはクロスプラットフォームのコンポーネント開発に特化しているため、React Native / Expoのエコシステムに沿った記述が有効です。詳しくは[Rork Max プロンプトエンジニアリングのコツ](/articles/rork-dev/rork-max-prompt-tips)もご参照ください。

本記事で解説したプロンプトエンジニアリングの技術を習得すると、Rork AIとの関係が変わります。「AIに指示を出す」から「AIと協業する」感覚へ。曖昧な要望を投げて都合よく解釈してもらうのではなく、プロとして明確な仕様をAIに伝え、期待通りの成果物を引き出す——それがプロンプトエンジニアリングの本質です。
