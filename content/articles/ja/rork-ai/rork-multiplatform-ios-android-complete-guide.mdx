「Rorkで作ったアプリ、iOSしか動かないんですけど」という相談を何度か受けました。確かに、Rorkのデフォルトの挙動を理解しないまま開発を進めると、気づいたときには「iOS専用のコード」ができあがっていることがあります。

この記事では、最初からiOS・Android両対応を意識して開発する方法と、後から両対応に直す方法を解説します。

## RorkのiOS/Android対応の仕組みを理解する

Rorkが生成するコードは基本的にReact Native（Expo）です。React Nativeはコードベースを共有しながらiOS・Android両対応のアプリを作れるフレームワークです。

ただし、Rorkのデフォルト設定と、指示の仕方によっては「iOS専用の動作」が混入することがあります。

### よくあるiOS専用コードが混入するケース

**UIフォント指定**: 「San Francisco フォントを使って」「システムフォントで」という指示。San FranciscoはiOS専用フォントです。

**iOS固有の画面デザイン**: 「iPhoneのような見た目で」という指示。Androidでは不自然になります。

**SafeAreaの扱い**: iPhoneのノッチ・Dynamic Islandへの対応コードが、Androidで余白が広すぎる問題を起こすことがあります。

## 両対応を前提にした正しいプロンプトの書き方

Rorkへの指示を最初から両対応前提にするだけで、後から直す手間が大幅に減ります。

### 良い例

```
iOSとAndroidの両方で動作する[アプリ名]を作ってください。
プラットフォーム固有のデザインよりも、両方のユーザーにとって
自然に見えるデザインを優先してください。
フォントはシステムデフォルトを使用し、
SafeAreaは両プラットフォームで適切に対応してください。
```

### 良くない例

```
iPhoneアプリを作ってください（→ iOS前提になる）
iOSらしいデザインで（→ Material Designが省略される）
San Franciscoフォントで（→ Android非対応）
```

## プラットフォーム固有の差異を吸収するパターン

どうしてもプラットフォームごとに異なる実装が必要な場合は、Platform APIを使います。

```javascript
// components/PlatformAwareButton.tsx
import { Platform, StyleSheet, TouchableOpacity, TouchableNativeFeedback, View } from 'react-native';

interface ButtonProps {
  onPress: () => void;
  children: React.ReactNode;
  style?: object;
}

export function PlatformButton({ onPress, children, style }: ButtonProps) {
  // Androidではネイティブのリップルエフェクトを使用
  if (Platform.OS === 'android') {
    return (
      <TouchableNativeFeedback
        onPress={onPress}
        background={TouchableNativeFeedback.Ripple('#00000020', false)}
      >
        <View style={[styles.button, style]}>
          {children}
        </View>
      </TouchableNativeFeedback>
    );
  }
  
  // iOSではOpacityフィードバックを使用
  return (
    <TouchableOpacity onPress={onPress} style={[styles.button, style]}>
      {children}
    </TouchableOpacity>
  );
}

const styles = StyleSheet.create({
  button: {
    padding: 16,
    borderRadius: 8,
    backgroundColor: '#007AFF',
    alignItems: 'center',
  },
});
```

### SafeArea の正しい扱い方

```javascript
// 全画面共通のSafeArea設定
import { SafeAreaProvider, SafeAreaView } from 'react-native-safe-area-context';

// App.tsx
export default function App() {
  return (
    <SafeAreaProvider>
      <AppNavigator />
    </SafeAreaProvider>
  );
}

// 各画面コンポーネント
function HomeScreen() {
  return (
    // edges を指定することで特定の方向のみ適用
    <SafeAreaView edges={['top', 'bottom']} style={{ flex: 1 }}>
      {/* コンテンツ */}
    </SafeAreaView>
  );
}
```

### フォントの両対応

```javascript
// styles/typography.ts — プラットフォーム対応フォント設定
import { Platform } from 'react-native';

export const typography = {
  // システムフォントを使用（iOS: San Francisco, Android: Roboto）
  body: {
    fontFamily: Platform.select({
      ios: 'System',
      android: 'Roboto',
    }),
    fontSize: 16,
    lineHeight: 24,
  },
  heading: {
    fontFamily: Platform.select({
      ios: 'System',
      android: 'Roboto',
    }),
    fontSize: 24,
    fontWeight: '700' as const,
  },
  // カスタムフォントを使う場合は両プラットフォーム向けに埋め込む
  custom: {
    fontFamily: 'NotoSansJP', // assets/fonts/ に配置したフォント
    fontSize: 16,
  },
};
```

## Rorkでの動作確認 — 両プラットフォームで確認する方法

Rorkの「Companion App」機能を使うと、実機でのプレビューが簡単にできます。iOSとAndroid両方の実機で確認することを強く推奨します。

### よく見落とされる両対応確認ポイント

**テキスト入力のキーボード対応**: iOSとAndroidでキーボードの出現・消去の挙動が異なります。`KeyboardAvoidingView` の設定がプラットフォームごとに違います。

```javascript
import { KeyboardAvoidingView, Platform } from 'react-native';

function FormScreen() {
  return (
    <KeyboardAvoidingView
      behavior={Platform.OS === 'ios' ? 'padding' : 'height'}
      style={{ flex: 1 }}
    >
      {/* フォームコンテンツ */}
    </KeyboardAvoidingView>
  );
}
```

**スクロール慣性**: iOSは慣性スクロールが強く、Androidは弱めです。`decelerationRate` で調整できます。

**バックボタン**: Androidには物理的なバックボタンがあり、適切に処理しないとアプリが終了します。

```javascript
import { BackHandler } from 'react-native';
import { useEffect } from 'react';

// Androidバックボタンの処理
function ScreenWithBackHandler({ navigation }: any) {
  useEffect(() => {
    const backAction = () => {
      navigation.goBack();
      return true; // trueを返すとデフォルト動作（アプリ終了）を防ぐ
    };
    
    const subscription = BackHandler.addEventListener('hardwareBackPress', backAction);
    return () => subscription.remove();
  }, [navigation]);
  
  return <View>{/* コンテンツ */}</View>;
}
```

## App Store と Google Play への同時公開

### App Store（iOS）公開の流れ

Rork Maxを使えば、2クリックでApp Storeに公開できます。ただし、事前に以下が必要です。

1. Apple Developer Program への登録（年間99ドル）
2. App Store Connect でのアプリ登録
3. スクリーンショットの用意（iPhone 6.7インチ必須、iPad対応する場合は追加）
4. プライバシーポリシーのURL（データを収集する場合は必須）

Rorkのプロンプトで「App Storeのメタデータを生成して」と指示すると、説明文やキーワードの下書きを生成してくれます。

### Google Play（Android）公開の流れ

Google Play Storeへの初回公開は、App StoreよりReview時間が長い（最大7日間）場合があります。

1. Google Play Console への登録（25ドル 一生涯）
2. AAB形式（Android App Bundle）でのビルド
3. コンテンツレーティングの設定
4. プライバシーポリシーのURL

Rorkで生成したReact Nativeアプリは、`expo build:android` コマンドでAABファイルを生成できます。

### 同時公開のタイムライン

```
理想的なスケジュール:
Day 1-3:  両プラットフォームでのテスト完了
Day 4:    App Store に申請（Review 1-2日）
Day 4:    Google Play に申請（Review 3-7日）
Day 5-6:  App Store 承認
Day 7-11: Google Play 承認
Day 12:   両ストアで同時リリース
```

Apple と Google のレビュー期間が異なるため、同じ日に申請しても公開日にズレが生じることがあります。余裕を持ってスケジュールを組んでください。

## 両対応開発で最も重要な1つのアドバイス

両対応開発で最も時間を失いがちな原因は「実機確認を後回しにすること」です。Rorkのプレビューはシミュレーター/エミュレーターですが、本物のデバイスでは全く異なる問題が出ることがあります。

できれば開発初期から、iOSの実機とAndroidの実機の両方でRork Companionを使って確認する習慣をつけてください。問題が小さいうちに気づければ、修正コストが格段に下がります。

両対応のアプリは、片方のプラットフォームだけに絞ったアプリより2倍の市場にリーチできます。その価値は投資に見合うものです。ぜひ最初から両対応前提で開発してみてください。
