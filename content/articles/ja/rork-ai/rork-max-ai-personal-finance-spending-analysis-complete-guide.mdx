家計管理アプリは App Store でも Google Play でも常に上位にいますが、その多くが「記録する」だけで終わっています。問題は、記録した数字を見ても、次にどうすればいいかが分からないことではないでしょうか。

Rork Max で AI を組み合わせると、単なる記録ツールを超えた「考えてくれる家計管理アプリ」が作れます。レシートを撮ると自動で分類し、月末には「今月、外食費が先月の1.4倍になっています。来月の外食予算を少し調整してみませんか？」と提案してくれる。そんなアプリです。

このガイドでは、その実装を一通りカバーします。単純なプロンプトのやり取りではなく、ユーザーの支出履歴をコンテキストとして Gemini に渡すパターン、セキュアなローカルストレージ設計、そして審査でよく引っかかる金融アプリ特有のポイントまで含めています。

## 今日作るアプリの設計と全体像

最初に完成形と技術選択を示しておきます。こう決めた理由も含めて説明するので、自分のアプリに合わせて変えやすくなるはずです。

**アプリの機能一覧:**

- レシート撮影 → Gemini Vision で金額・店名・カテゴリを自動抽出
- テキスト入力でも自然言語登録可能（「さっきコンビニで430円」）
- 月次・週次の支出チャート
- カテゴリ別予算設定と達成率表示
- 月末に Gemini による個別アドバイス生成

**技術スタック:**

- **ローカルDB**: expo-sqlite（オフライン優先、プライバシー重視）
- **AI**: Gemini 1.5 Flash（レシートOCR・自然言語解析）、Gemini 1.5 Pro（アドバイス生成）
- **セキュリティ**: expo-secure-store（APIキー管理）
- **チャート**: Victory Native（[インタラクティブなチャート実装の詳細はこちら](/articles/rork-dev/rork-victory-native-charts-dashboard)）

なぜ Firebase や Supabase ではなくローカルDBを選んだか、という点について触れておきます。家計データは極めてプライベートな情報で、「クラウドに送りたくない」と感じるユーザーが一定数います。まずローカル完結で動くものを作り、オプションでクラウド同期を提供する設計の方が、審査通過率と星評価の両方に好影響が出ます。実際、私がリリースした家計アプリでも「データがデバイスから出ない」という説明文がレビューで高評価をもらうことが多かったです。

## データモデルとローカルDB設計

SQLite のスキーマ設計は後から変えづらいので、最初にしっかり決めます。以下が基本のデータモデルです。

```typescript
// types/finance.ts
export type ExpenseCategory =
  | 'food'          // 食費（スーパー・コンビニ）
  | 'dining'        // 外食（レストラン・カフェ）
  | 'transport'     // 交通費
  | 'shopping'      // 買い物（衣料品・家電）
  | 'utilities'     // 光熱費・通信費
  | 'healthcare'    // 医療・健康
  | 'entertainment' // 娯楽
  | 'education'     // 教育
  | 'other';        // その他

export interface Expense {
  id: string;
  amount: number;            // 金額（円）
  category: ExpenseCategory;
  description: string;       // 店名・メモ
  date: string;              // ISO 8601
  receiptImageUri?: string;  // レシート画像のローカルパス
  aiCategorized: boolean;    // AIが分類したかどうか
  createdAt: string;
}

export interface Budget {
  id: string;
  category: ExpenseCategory;
  monthlyLimit: number;  // 月次上限（円）
  month: string;         // "2026-04" 形式
}
```

次に、SQLite のテーブル初期化と CRUD 操作です。エラーハンドリングを含めた実用的なコードを示します。

```typescript
// lib/database.ts
import * as SQLite from 'expo-sqlite';

const db = SQLite.openDatabaseSync('finance.db');

export async function initDatabase(): Promise<void> {
  try {
    await db.execAsync(`
      PRAGMA journal_mode = WAL;

      CREATE TABLE IF NOT EXISTS expenses (
        id TEXT PRIMARY KEY,
        amount REAL NOT NULL,
        category TEXT NOT NULL,
        description TEXT NOT NULL,
        date TEXT NOT NULL,
        receipt_image_uri TEXT,
        ai_categorized INTEGER DEFAULT 0,
        created_at TEXT NOT NULL
      );

      CREATE TABLE IF NOT EXISTS budgets (
        id TEXT PRIMARY KEY,
        category TEXT NOT NULL,
        monthly_limit REAL NOT NULL,
        month TEXT NOT NULL,
        UNIQUE(category, month)
      );

      CREATE INDEX IF NOT EXISTS idx_expenses_date ON expenses(date);
      CREATE INDEX IF NOT EXISTS idx_expenses_category ON expenses(category);
    `);
  } catch (error) {
    console.error('Database initialization failed:', error);
    throw new Error('データベースの初期化に失敗しました');
  }
}

export async function addExpense(expense: Expense): Promise<void> {
  try {
    await db.runAsync(
      `INSERT INTO expenses VALUES (?, ?, ?, ?, ?, ?, ?, ?)`,
      [
        expense.id,
        expense.amount,
        expense.category,
        expense.description,
        expense.date,
        expense.receiptImageUri ?? null,
        expense.aiCategorized ? 1 : 0,
        expense.createdAt,
      ]
    );
  } catch (error) {
    if (error instanceof Error && error.message.includes('UNIQUE constraint')) {
      throw new Error('同じ支出データが既に存在します');
    }
    throw new Error('支出データの保存に失敗しました');
  }
}

export async function getExpensesByMonth(month: string): Promise<Expense[]> {
  // month は "2026-04" 形式
  try {
    const rows = await db.getAllAsync<any>(
      `SELECT * FROM expenses WHERE date LIKE ? ORDER BY date DESC`,
      [`${month}%`]
    );
    return rows.map(row => ({
      id: row.id,
      amount: row.amount,
      category: row.category as ExpenseCategory,
      description: row.description,
      date: row.date,
      receiptImageUri: row.receipt_image_uri ?? undefined,
      aiCategorized: row.ai_categorized === 1,
      createdAt: row.created_at,
    }));
  } catch (error) {
    console.error('Failed to fetch expenses:', error);
    return []; // エラー時は空配列を返し、UIをクラッシュさせない
  }
}

export async function getCategoryTotals(
  month: string
): Promise<Record<ExpenseCategory, number>> {
  try {
    const rows = await db.getAllAsync<{ category: string; total: number }>(
      `SELECT category, SUM(amount) as total
       FROM expenses
       WHERE date LIKE ?
       GROUP BY category`,
      [`${month}%`]
    );
    const result = {} as Record<ExpenseCategory, number>;
    rows.forEach(row => {
      result[row.category as ExpenseCategory] = row.total;
    });
    return result;
  } catch (error) {
    console.error('Failed to get category totals:', error);
    return {} as Record<ExpenseCategory, number>;
  }
}
```

`PRAGMA journal_mode = WAL` を最初に設定しているのには理由があります。デフォルトのロールバックジャーナルモードと比べて、WAL（Write-Ahead Logging）は読み取りと書き込みが同時に発生してもブロックしません。家計アプリでは「支出入力しながら集計を表示」というシナリオが多いので、これが実用上の差になります。インデックスも最初から設定しておくことを強くおすすめします。6ヶ月分のデータが溜まってから追加すると、追加時のパフォーマンスペナルティが発生します。

## 支出自動分類エンジン: Gemini Vision API を使ったレシート解析

レシートの画像から金額・店名・カテゴリを抽出する部分です。実装してみて分かったことがあります。Gemini に「このレシートから金額とカテゴリを教えて」とシンプルに聞くと、答え方がバラバラになって後処理が難しくなります。構造化された JSON で返すよう指示するのがコツです。

```typescript
// lib/gemini.ts
import { GoogleGenerativeAI } from '@google/generative-ai';
import * as SecureStore from 'expo-secure-store';
import * as FileSystem from 'expo-file-system';

const VALID_CATEGORIES: ExpenseCategory[] = [
  'food', 'dining', 'transport', 'shopping',
  'utilities', 'healthcare', 'entertainment', 'education', 'other',
];

export interface ReceiptAnalysisResult {
  amount: number;
  storeName: string;
  category: ExpenseCategory;
  confidence: number; // 0-1
  rawText?: string;
}

export async function analyzeReceiptImage(
  imageUri: string
): Promise<ReceiptAnalysisResult> {
  // APIキーはハードコードせず、SecureStore から取得
  const apiKey = await SecureStore.getItemAsync('gemini_api_key');
  if (\!apiKey) {
    throw new Error('Gemini APIキーが設定されていません。設定画面からAPIキーを登録してください。');
  }

  const genAI = new GoogleGenerativeAI(apiKey);
  const model = genAI.getGenerativeModel({ model: 'gemini-1.5-flash' });

  // 画像を Base64 に変換
  let base64Image: string;
  try {
    base64Image = await FileSystem.readAsStringAsync(imageUri, {
      encoding: FileSystem.EncodingType.Base64,
    });
  } catch {
    throw new Error('画像の読み込みに失敗しました');
  }

  const prompt = `
このレシート画像を分析して、以下のJSON形式で回答してください。
JSON以外のテキストは含めないでください。

{
  "amount": <金額（数値・円単位）>,
  "storeName": "<店名>",
  "category": "<food/dining/transport/shopping/utilities/healthcare/entertainment/education/other のいずれか>",
  "confidence": <確信度 0.0〜1.0>,
  "rawText": "<レシートから読み取れた主要テキスト>"
}

カテゴリ判断基準:
- food: スーパー・コンビニ（食品・日用品）
- dining: レストラン・カフェ・テイクアウト
- transport: 電車・バス・タクシー・ガソリン
- shopping: 衣料品・家電・非食品日用品
- utilities: 電気・ガス・水道・通信費
- healthcare: 薬局・病院・スポーツジム
- entertainment: 映画・ゲーム・書籍・趣味
- education: 学習サービス・教育書籍
- other: 上記に当てはまらないもの
`;

  let responseText: string;
  try {
    const result = await model.generateContent([
      prompt,
      {
        inlineData: {
          mimeType: 'image/jpeg',
          data: base64Image,
        },
      },
    ]);
    responseText = result.response.text();
  } catch (error) {
    throw new Error(
      'Gemini APIの呼び出しに失敗しました。ネットワーク接続を確認してください。'
    );
  }

  // JSON の抽出と検証
  try {
    // Gemini がコードブロックで返す場合にも対応
    const jsonMatch = responseText.match(/\{[\s\S]*\}/);
    if (\!jsonMatch) throw new Error('JSON not found in response');

    const parsed = JSON.parse(jsonMatch[0]);

    if (typeof parsed.amount \!== 'number' || parsed.amount <= 0) {
      throw new Error('Invalid amount');
    }
    if (\!VALID_CATEGORIES.includes(parsed.category)) {
      parsed.category = 'other';
    }
    const confidence = Math.min(1, Math.max(0, parsed.confidence ?? 0.5));

    return {
      amount: Math.round(parsed.amount),
      storeName: parsed.storeName || '不明',
      category: parsed.category,
      confidence,
      rawText: parsed.rawText,
    };
  } catch {
    // JSON パース失敗時のフォールバック — クラッシュさせずに手入力に誘導
    return {
      amount: 0,
      storeName: '不明',
      category: 'other',
      confidence: 0,
    };
  }
}
```

confidence が 0.5 以下の場合はユーザーに確認を促す UI にすると、AIの誤分類を自然に修正できます。この修正データを蓄積しておくと、将来的にカスタムマッピングの改善にも活かせます。

[レシート OCR の別アプローチについてはこちらの記事](/articles/rork-ai/rork-ai-receipt-scanner-budget-app)でも紹介しています。レシートのみに特化したシンプルな実装なので、まず動くものを作りたい方の参考になります。

## テキスト入力からの自然言語支出登録

レシートがない支出（電子決済・交通系ICカード等）向けに、テキスト入力でも登録できるようにします。「さっきコンビニで430円使った」という日本語を解析して構造化するパターンです。

```typescript
// lib/nlpExpense.ts
import { GoogleGenerativeAI } from '@google/generative-ai';
import * as SecureStore from 'expo-secure-store';

export async function parseNaturalLanguageExpense(
  input: string,
  currentDate: string
): Promise<Partial<Expense> | null> {
  const apiKey = await SecureStore.getItemAsync('gemini_api_key');
  if (\!apiKey) return null;

  const genAI = new GoogleGenerativeAI(apiKey);
  const model = genAI.getGenerativeModel({ model: 'gemini-1.5-flash' });

  const prompt = `
以下のテキストから支出情報を抽出してJSON形式で返してください。
テキスト: "${input}"
現在の日付: ${currentDate}

{
  "amount": <金額（円）、不明なら 0>,
  "category": "<カテゴリ>",
  "description": "<内容の簡潔な説明>",
  "date": "<日付 ISO 8601形式、今日の場合は ${currentDate}>"
}

抽出できない・支出情報でない場合は null を返してください。
`;

  try {
    const result = await model.generateContent(prompt);
    const text = result.response.text();
    const jsonMatch = text.match(/\{[\s\S]*\}|null/);
    if (\!jsonMatch || jsonMatch[0] === 'null') return null;

    const parsed = JSON.parse(jsonMatch[0]);
    if (\!parsed.amount || parsed.amount <= 0) return null;

    return {
      amount: parsed.amount,
      category: VALID_CATEGORIES.includes(parsed.category)
        ? parsed.category
        : 'other',
      description: parsed.description || input.slice(0, 50),
      date: parsed.date || currentDate,
    };
  } catch {
    return null;
  }
}
```

この関数を使う UI では、「解析中…」のローディング中も入力欄をフリーズさせないよう非同期で処理し、解析失敗時は通常の手動入力フォームにフォールバックするのが自然です。

## パーソナルAIアドバイザー: 支出履歴をコンテキストに注入する

ここがこのアプリの核心です。単に「節約しましょう」と言うだけのAIではなく、そのユーザーの実際の支出パターンを踏まえた具体的なアドバイスを生成します。

ポイントは Gemini に渡すコンテキストの設計です。全履歴をそのまま渡すと高コスト・低精度になります。代わりに「集計済みのサマリー」を構造化して渡す方が精度も上がり、トークンコストも抑えられます。

```typescript
// lib/financialAdvisor.ts
import { GoogleGenerativeAI } from '@google/generative-ai';
import * as SecureStore from 'expo-secure-store';

interface MonthlySpendingSummary {
  month: string;
  totalAmount: number;
  categoryBreakdown: Partial<Record<ExpenseCategory, number>>;
  budgets: Partial<Record<ExpenseCategory, number>>;
  topExpenses: Array<{ description: string; amount: number; category: string }>;
}

const CATEGORY_LABELS: Record<ExpenseCategory, string> = {
  food: '食費',
  dining: '外食',
  transport: '交通費',
  shopping: '買い物',
  utilities: '光熱費',
  healthcare: '医療・健康',
  entertainment: '娯楽',
  education: '教育',
  other: 'その他',
};

export async function generateFinancialAdvice(
  summary: MonthlySpendingSummary,
  previousMonth?: MonthlySpendingSummary
): Promise<string> {
  const apiKey = await SecureStore.getItemAsync('gemini_api_key');
  if (\!apiKey) {
    return '設定からGemini APIキーを設定すると、個別アドバイスが受け取れます。';
  }

  const genAI = new GoogleGenerativeAI(apiKey);
  const model = genAI.getGenerativeModel({
    model: 'gemini-1.5-pro',
    generationConfig: {
      maxOutputTokens: 600, // アドバイスは簡潔に
      temperature: 0.7,
    },
  });

  // 前月比の計算
  const comparisonLines = Object.entries(summary.categoryBreakdown).map(
    ([cat, amount]) => {
      const label = CATEGORY_LABELS[cat as ExpenseCategory] ?? cat;
      const prev =
        previousMonth?.categoryBreakdown[cat as ExpenseCategory] ?? 0;
      const change =
        prev > 0 ? `（前月比${(((amount\! - prev) / prev) * 100).toFixed(0)}%）` : '';
      return `- ${label}: ${amount\!.toLocaleString()}円${change}`;
    }
  );

  // 予算超過カテゴリの特定
  const overBudgetLines = Object.entries(summary.categoryBreakdown)
    .filter(([cat, amount]) => {
      const budget = summary.budgets[cat as ExpenseCategory];
      return budget && amount\! > budget;
    })
    .map(([cat, amount]) => {
      const budget = summary.budgets[cat as ExpenseCategory]\!;
      const label = CATEGORY_LABELS[cat as ExpenseCategory] ?? cat;
      return `- ${label}: ${amount\!.toLocaleString()}円（予算${budget.toLocaleString()}円を${(amount\! - budget).toLocaleString()}円超過）`;
    });

  const prompt = `
あなたは親切な家計アドバイザーです。
以下のユーザーの支出データを分析し、具体的で実行可能なアドバイスを提供してください。

【${summary.month}の支出サマリー】
合計: ${summary.totalAmount.toLocaleString()}円

カテゴリ別支出:
${comparisonLines.join('\n')}

${
    overBudgetLines.length > 0
      ? `予算超過カテゴリ:\n${overBudgetLines.join('\n')}`
      : '全カテゴリが予算内に収まっています。'
  }

主な支出トップ3:
${summary.topExpenses
    .slice(0, 3)
    .map(e => `- ${e.description}: ${e.amount.toLocaleString()}円`)
    .join('\n')}

【指示】
- 数字を具体的に引用しながら分析してください
- 実行できる具体的な行動提案を2〜3個示してください
- 批判的にならず、励ましを含めてください
- 投資推奨や金融商品の購入提案は一切しないでください
- 日本語で 200〜250 文字程度で回答してください
`;

  try {
    const result = await model.generateContent(prompt);
    return result.response.text();
  } catch (error) {
    console.error('Advice generation failed:', error);
    // API失敗時はローカルデータから簡易サマリーを返す
    return `${summary.month}の合計支出は${summary.totalAmount.toLocaleString()}円でした。詳細な分析には、インターネット接続が必要です。`;
  }
}
```

`maxOutputTokens: 600` を設定しているのは、アドバイスが長文になりすぎると読まれなくなるからです。200〜250文字の簡潔なアドバイスの方が、実際にユーザーが行動を変えるきっかけになります。

プロンプトに「投資推奨や金融商品の購入提案は一切しないでください」という制約を入れているのも重要です。これを省略すると、Gemini が「余剰資金は投資信託に回しましょう」という提案を生成することがあり、App Store 審査で問題になります。

[支出の月次チャート実装については、こちらのガイド](/articles/rork-dev/rork-budget-tracker-data-visualization)に実用的なサンプルがあります。上記のデータモデルとそのまま組み合わせることができます。

## セキュリティとプライバシー: 金融データを安全に扱う

金融アプリで最もよく審査でコメントされるのがこの部分です。特に注意が必要な点が3つあります。

### 1. APIキーは SecureStore で管理する

Rork Max で生成したコードにAPIキーをハードコードする誘惑に負けないでください。App Store のビルド解析ツールで簡単に検出されます。

```typescript
// lib/secureStorage.ts
import * as SecureStore from 'expo-secure-store';

export async function saveApiKey(key: string): Promise<void> {
  // iOS: Keychain に保存
  // Android: EncryptedSharedPreferences に保存
  await SecureStore.setItemAsync('gemini_api_key', key, {
    keychainAccessible: SecureStore.WHEN_UNLOCKED_THIS_DEVICE_ONLY,
    // デバイスのロック解除時のみアクセス可能
    // iCloud バックアップから除外（他デバイスへのキー漏洩を防止）
  });
}

export async function getApiKey(): Promise<string | null> {
  try {
    return await SecureStore.getItemAsync('gemini_api_key');
  } catch {
    // Keychain へのアクセスが失敗した場合（デバイスが暗号化されていない等）
    return null;
  }
}

// 全ユーザーデータの完全削除（プライバシーポリシーで約束した機能）
export async function deleteAllUserData(db: SQLite.SQLiteDatabase): Promise<void> {
  const errors: string[] = [];

  // SQLite データ削除
  try {
    await db.execAsync('DELETE FROM expenses; DELETE FROM budgets;');
  } catch {
    errors.push('支出データの削除に失敗');
  }

  // SecureStore の APIキー削除
  try {
    await SecureStore.deleteItemAsync('gemini_api_key');
  } catch {
    errors.push('APIキーの削除に失敗');
  }

  // レシート画像の削除
  try {
    const receiptDir = FileSystem.documentDirectory + 'receipts/';
    const dirInfo = await FileSystem.getInfoAsync(receiptDir);
    if (dirInfo.exists) {
      await FileSystem.deleteAsync(receiptDir, { idempotent: true });
    }
  } catch {
    errors.push('レシート画像の削除に失敗');
  }

  if (errors.length > 0) {
    throw new Error(`一部データの削除に失敗しました: ${errors.join(', ')}`);
  }
}
```

`WHEN_UNLOCKED_THIS_DEVICE_ONLY` の設定が重要です。このデバイスにロックされているため、iCloud バックアップ経由で他のデバイスにキーが流出しません。

### 2. レシート画像の保存場所

`FileSystem.documentDirectory` と `FileSystem.cacheDirectory` の違いは審査に影響します。ドキュメントディレクトリに保存した画像はバックアップ対象になります。レシート画像のようにセンシティブなデータはキャッシュディレクトリに保存するか、解析後すぐに削除するかを選択してください。プライバシーポリシーにもその扱いを明記する必要があります。

### 3. Gemini APIへのデータ送信の透明性

レシート画像を Gemini に送ることは、ユーザーの画像データが Google のサーバーで処理されることを意味します。この事実を：

- アプリ初回起動時のオンボーディングで説明する
- プライバシーポリシーに「AI分析のためにGemini APIを使用しており、画像はGoogleのサーバーで処理されます」と明記する
- 画像の処理後にGoogleが保持するか否かの方針へのリンクを設ける

この3点を実装しておかないと、審査で「プライバシー情報が不十分」という指摘を受けます。

## App Store 審査で金融アプリが引っかかる5つのポイント

金融系アプリを複数審査に出してきた経験から、実際に問題になったポイントをまとめます。

**① 「投資アドバイス」と取られかねない表現**

「この銘柄に投資すると良い」のような表現は、金融商品取引法に関わる表現として審査で指摘されます。AIのアドバイスが投資推奨に聞こえないよう、上記プロンプトに制約を入れることに加え、UIのラベルも「アドバイス」ではなく「支出の振り返り」のような表現にする方が安全です。

**② カメラ使用の説明文が具体的でない**

`NSCameraUsageDescription`（Info.plist）や Android のパーミッション説明文に、「レシートを撮影してAIで自動分類するため」という具体的な目的を書く必要があります。「カメラを使います」だけでは却下されます。

**③ 財務データの第三者共有申告漏れ**

ユーザーの支出データをアナリティクスツール（Mixpanel, Amplitude等）に送る場合、App Store の「データの収集」セクションで「財務情報」カテゴリの申告が必要です。アナリティクスSDKを入れる際に見落としやすい部分です。

**④ プレミアム機能の表現**

AIアドバイス機能を有料にする場合、無料版でできることと有料版でできることを明確に分ける必要があります。「試してから課金を促す」導線、つまり一定回数は無料で使えてその後プレミアムを案内する設計が、審査通過率と実際の課金率の両方で最も良い結果が出ます。

**⑤ データ削除機能の実装と記載**

上記コードで示しましたが、設定画面に「すべてのデータを削除」ボタンを置き、プライバシーポリシーにその旨を記載することが必須です。家計アプリのように金融情報を扱う場合、この機能の欠如はリジェクト理由として明示されることがあります。

## よくある実装の落とし穴

実際に開発・運用してみて詰まった点を整理します。

**落とし穴1: Gemini が毎回異なる JSON 構造を返す**

プロンプトに例示を入れても、Gemini 1.5 Flash はたまに違う構造を返します。上記コードで示したように、正規表現で JSON 部分を抽出し、存在するフィールドのみを使う防御的パースが必須です。`if (\!parsed.amount || parsed.amount <= 0) return null;` のような early return を各フィールドに入れておくと、予期しないデータ形式でもクラッシュしません。

**落とし穴2: expo-sqlite の execAsync と runAsync の混同**

`execAsync` は複数のSQL文を一度に実行できますが戻り値は void です。INSERT/UPDATE の成否を確認したい場合は `runAsync` を使い、戻り値の `changes` プロパティで変更行数を確認します。0 の場合は該当レコードが存在しなかったことを意味します。

**落とし穴3: 大量のレシート画像でストレージが膨らむ**

レシート画像を保持し続けると、3ヶ月で数百MBになることがあります。解析完了後はオリジナル画像を削除してサムネイルのみ保持するか、または「解析が終わったら画像は削除されます」という仕様にしてユーザーに説明する設計が現実的です。

**落とし穴4: 月次集計クエリが6ヶ月後に重くなる**

インデックスなしで `SUM(amount) GROUP BY category` を走らせると、データが増えるにつれ体感で遅くなります。`CREATE INDEX idx_expenses_date` を最初から設定しておくことと、さらに月次集計を別テーブルに事前集計して保持する設計が長期的には有効です。

**落とし穴5: APIキー未設定時のクラッシュ**

APIキーが設定されていない状態でレシートスキャンがタップされると、`SecureStore.getItemAsync` が null を返してそのまま API 呼び出しに進むとクラッシュします。上記コードで示したように、null チェックと設定画面への誘導メッセージを最初に入れることで、クラッシュではなくユーザーフレンドリーなエラーメッセージに変えられます。

**落とし穴6: 日本語テキストの OCR 精度**

日本のレシートは縦書き・小さいフォント・特殊な記号（税率区分の☆や＊）が含まれていることがあります。Gemini Vision は精度が高いですが、古いコンビニのレシートや手書き領収書では confidence が低くなりがちです。confidence < 0.6 の場合は金額フィールドを空欄にして「確認してください」と表示する UI にすると、誤登録が防げます。

## まとめ: アプリを動かした後の次の一手

ここまで実装できると、「記録するだけ」の家計アプリとは一線を画したものができます。次にやるべきことは一つ、**TestFlight に出して実際のユーザーに1週間使ってもらうこと**です。

自分で使っていると気づきにくい「レシートの撮り方が分からない」「AIが分類を間違えた時に修正できない」「月次アドバイスが表示されるタイミングが分かりにくい」といったUX問題が、実際の使い手からのフィードバックで浮かび上がります。それを修正してから App Store 本審査に出す方が、リジェクトリスクを大幅に下げられます。

AIアドバイス機能は、ユーザーが修正した分類データが蓄積されるほど個人に合ったものになっていきます。最初から完璧にしようとせず、まず動くものを TestFlight に出すことを優先してください。
