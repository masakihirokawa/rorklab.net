Rorkは単なる「プロンプトをコードに変換するツール」ではありません。その裏側では、**Gemini 2.0 Flash** を中心とした高度な AI エージェントが、自律的にアプリを構築しています。このガイドでは、Rork の AI エンジンの仕組みを理解したうえで、より高品質なアプリを素早く生成するための実践的なテクニックを紹介します。

## Rork が Gemini 2.0 Flash を採用している理由

Rork の AI エンジン（RorkAgent）は、2026年現在、**Google の Gemini 2.0 Flash** をデフォルトモデルとして採用しています。なぜ数ある AI モデルの中から Gemini 2.0 Flash が選ばれたのでしょうか？

主な理由は次の3点です。

**1. マルチターンのツール呼び出しに最適化されている**

Rork の内部では、アプリ生成時に最大11種類のツール（ファイル作成・編集・依存関係解決・プレビュービルドなど）を順番に呼び出す「アジェンティックループ」が走ります。Gemini 2.0 Flash は、このような複数ステップにわたるツール呼び出しを高速かつ低コストで実行できる設計になっており、レスポンス速度と精度のバランスが優れています。

**2. コード生成の精度が高い**

React Native（Expo）のコードを正確に生成するには、コンポーネントの構造、ナビゲーション、状態管理といった複数の概念を同時に把握する必要があります。Gemini 2.0 Flash はコード生成タスクにおいて競合モデルに匹敵するパフォーマンスを示しており、特に React Native Expo のエコシステムに最適化されたトレーニングが施されています。

**3. コストパフォーマンス**

Gemini 2.0 Flash は処理速度が速く、トークンあたりのコストが低いため、Rork が大量のユーザーに対してリアルタイムでアプリを生成する際のインフラコストを抑えることができます。これが Rork の価格競争力にも直結しています。

## RorkAgent の仕組み：11 ステップのアジェンティックループ

Rork でプロンプトを送信すると、裏側では以下のような流れで処理が行われます。

```
ユーザープロンプト
  ↓
RorkAgent（Gemini 2.0 Flash）
  ↓
[アジェンティックループ 最大11ターン]
  1. ファイル構造の設計
  2. package.json の生成（依存関係解決）
  3. App.tsx / エントリーポイントの作成
  4. ナビゲーション構造の定義
  5. 画面コンポーネントの生成
  6. 状態管理ロジックの実装
  7. API連携コードの追加
  8. スタイルシートの適用
  9. エラーチェックと自動修正
  10. プレビュービルドの生成
  11. 最終コードレビュー
  ↓
完成したReact Native（Expo）アプリ
```

この一連のプロセスが数十秒〜数分で自動的に完了します。ユーザーが「タスク管理アプリを作って」と入力するだけで、Gemini 2.0 Flash がアーキテクチャから実装まですべてを担うわけです。

## Gemini 2.0 Flash と Claude の違い：Rork での使い分け

Rork のエンジンは **GeminiProvider** と **ClaudeProvider** の2つをサポートしており、状況に応じて切り替えが可能です（Rork Max では Claude Opus が活用されています）。

| 項目 | Gemini 2.0 Flash | Claude |
|------|-----------------|--------|
| 処理速度 | 非常に高速 | 中〜高速 |
| コード生成精度 | 高い | 非常に高い |
| 長文コンテキスト | 最大100万トークン | 最大20万トークン |
| 得意な用途 | 素早いプロトタイプ・反復開発 | 複雑なロジック・高精度な実装 |
| Rork での位置づけ | デフォルト（Standard プラン） | Rork Max で活用 |

シンプルなアプリや素早くプロトタイプを作りたいときは Gemini 2.0 Flash、複雑なビジネスロジックや Native Swift アプリには Rork Max（Claude Opus）という使い分けが効果的です。

詳しくは「[Rork Max の AI 機能完全ガイド](/articles/rork-ai/rork-max-ai-features)」も参照してください。

## Gemini 2.0 Flash を最大限に活用するプロンプト術

Rork で Gemini 2.0 Flash が最大限のパフォーマンスを発揮するには、プロンプトの書き方が重要です。以下のテクニックを実践してみましょう。

### 1. 画面ごとに構造を明示する

悪い例:
```
チャットアプリを作って
```

良い例:
```
以下の3画面を持つチャットアプリを作って：
1. ログイン画面（メール・パスワード入力、Supabase認証）
2. チャット一覧画面（会話のリスト表示、新規チャット開始ボタン）
3. チャット詳細画面（メッセージ表示・送信フォーム、リアルタイム更新）
```

Gemini 2.0 Flash はマルチターンの文脈理解が優れているため、画面の構造を最初から明示するとアジェンティックループの精度が格段に向上します。

### 2. 技術スタックを指定する

```
Expo Router v3 を使ったナビゲーション構成で、
状態管理は Zustand を使い、
データベースは Supabase（PostgreSQL）を使って
タスク管理アプリを作って
```

Gemini 2.0 Flash は指定した技術スタックに合わせてコードを生成します。指定がない場合はデフォルトのスタックが選ばれますが、自分のプロジェクトに合わせた指定をすることで後から修正する手間が省けます。

### 3. デザインの雰囲気を言語化する

```
デザインはミニマルでダークモード対応。
カラースキームはダークグレー（#1a1a2e）をベースに
アクセントカラーは紫（#6c63ff）。
フォントはシステムフォントで統一。
```

Gemini 2.0 Flash は自然言語のデザイン指示を StyleSheet に変換するのが得意です。Figma のデザインを言語化することで、ビジュアルの再現精度が向上します。

### 4. Fix Now 機能との組み合わせ

生成されたコードにエラーが発生した場合は、**Fix Now** 機能を使いましょう。Gemini 2.0 Flash がエラーメッセージを解析し、自動的に修正コードを生成します。

```
エラー: Cannot read properties of undefined (reading 'map')

Fix Now ボタンをクリック → Gemini 2.0 Flash が原因を特定 →
null チェックとフォールバック値を自動追加 → 解決
```

Fix Now の詳細な使い方は「[Fix Now 自動デバッグ機能の使い方](/articles/rork-ai/fix-now-auto-debug)」を参照してください。

## 実践例：Gemini 2.0 Flash で TODO アプリを作る

以下は実際に Rork で生成される React Native コードの例です。Gemini 2.0 Flash が生成する典型的なコードパターンを確認しましょう。

```typescript
// app/(tabs)/index.tsx
// Gemini 2.0 Flash が生成するTODOリストのコンポーネント例

import { useState } from 'react';
import {
  View,
  Text,
  TextInput,
  TouchableOpacity,
  FlatList,
  StyleSheet,
} from 'react-native';

// TODOアイテムの型定義
interface Todo {
  id: string;
  text: string;
  completed: boolean;
  createdAt: Date;
}

export default function TodoScreen() {
  const [todos, setTodos] = useState<Todo[]>([]);
  const [inputText, setInputText] = useState('');

  // 新しいTODOを追加する関数
  const addTodo = () => {
    if (!inputText.trim()) return; // 空白入力を無視

    const newTodo: Todo = {
      id: Date.now().toString(),
      text: inputText.trim(),
      completed: false,
      createdAt: new Date(),
    };

    setTodos(prev => [newTodo, ...prev]); // 先頭に追加
    setInputText(''); // 入力フィールドをクリア
  };

  // 完了状態をトグルする関数
  const toggleTodo = (id: string) => {
    setTodos(prev =>
      prev.map(todo =>
        todo.id === id ? { ...todo, completed: !todo.completed } : todo
      )
    );
  };

  // TODO を削除する関数
  const deleteTodo = (id: string) => {
    setTodos(prev => prev.filter(todo => todo.id !== id));
  };

  return (
    <View style={styles.container}>
      <Text style={styles.title}>My TODOs</Text>

      {/* 入力フォーム */}
      <View style={styles.inputRow}>
        <TextInput
          style={styles.input}
          value={inputText}
          onChangeText={setInputText}
          placeholder="新しいタスクを入力..."
          placeholderTextColor="#888"
          onSubmitEditing={addTodo}
        />
        <TouchableOpacity style={styles.addButton} onPress={addTodo}>
          <Text style={styles.addButtonText}>追加</Text>
        </TouchableOpacity>
      </View>

      {/* TODOリスト */}
      <FlatList
        data={todos}
        keyExtractor={item => item.id}
        renderItem={({ item }) => (
          <View style={styles.todoItem}>
            <TouchableOpacity
              style={styles.checkBox}
              onPress={() => toggleTodo(item.id)}
            >
              <Text style={styles.checkMark}>{item.completed ? '✓' : '○'}</Text>
            </TouchableOpacity>
            <Text
              style={[styles.todoText, item.completed && styles.completed]}
            >
              {item.text}
            </Text>
            <TouchableOpacity onPress={() => deleteTodo(item.id)}>
              <Text style={styles.deleteButton}>削除</Text>
            </TouchableOpacity>
          </View>
        )}
        ListEmptyComponent={
          <Text style={styles.emptyText}>タスクはまだありません</Text>
        }
      />
    </View>
  );
}

const styles = StyleSheet.create({
  container: { flex: 1, backgroundColor: '#1a1a2e', padding: 20 },
  title: { fontSize: 28, fontWeight: 'bold', color: '#fff', marginBottom: 20 },
  inputRow: { flexDirection: 'row', marginBottom: 16, gap: 8 },
  input: {
    flex: 1,
    backgroundColor: '#16213e',
    color: '#fff',
    borderRadius: 8,
    padding: 12,
    fontSize: 16,
  },
  addButton: {
    backgroundColor: '#6c63ff',
    borderRadius: 8,
    padding: 12,
    justifyContent: 'center',
  },
  addButtonText: { color: '#fff', fontWeight: 'bold' },
  todoItem: {
    flexDirection: 'row',
    alignItems: 'center',
    backgroundColor: '#16213e',
    borderRadius: 8,
    padding: 12,
    marginBottom: 8,
    gap: 12,
  },
  checkBox: { width: 28 },
  checkMark: { fontSize: 18, color: '#6c63ff' },
  todoText: { flex: 1, color: '#fff', fontSize: 16 },
  completed: { textDecorationLine: 'line-through', color: '#888' },
  deleteButton: { color: '#ff6b6b', fontSize: 14 },
  emptyText: { color: '#888', textAlign: 'center', marginTop: 40, fontSize: 16 },
});
```

**期待する出力:**
- ダークテーマ（#1a1a2e）のTODOリスト画面
- テキスト入力 → 「追加」ボタンでリストに追加
- チェックボックスタップで完了状態をトグル（完了項目は打ち消し線表示）
- 「削除」ボタンでアイテムを削除
- リストが空の場合は「タスクはまだありません」と表示

このコードは Rork が実際に生成するコードに近く、TypeScript の型安全性・Expo 互換のコンポーネント・わかりやすいコメントが自動的に付与されます。

## Gemini の長文コンテキストを活かす高度な使い方

Gemini 2.0 Flash の最大の特長のひとつが、**最大100万トークンのコンテキストウィンドウ**です。これを Rork のイテレーティブな開発スタイルと組み合わせると、非常に強力です。

### プロジェクトの継続的な拡張

1回のセッションで複数の機能を追加しても、Gemini は過去の会話すべてを記憶したまま処理を続けます。

```
セッション1: 「ユーザー認証機能を追加して」
セッション2: 「さっき作った認証を使ってプロフィール画面を作って」
セッション3: 「プロフィール画面に画像アップロード機能を追加して（Supabase Storage使用）」
```

既存コードの文脈を保持したまま機能追加できるため、整合性の取れたコードが生成されます。

### コードのリファクタリング依頼

```
「このプロジェクト全体を見て、
重複しているロジックをカスタムフックに切り出して。
特に認証まわりの処理は useAuth フックに統合して」
```

Gemini 2.0 Flash の長文理解力により、プロジェクト全体を俯瞰したリファクタリング提案が可能です。

## Rork Max における Gemini と Claude の協調動作

[Rork Max](/articles/rork-ai/rork-max-ai-features) では、Gemini 2.0 Flash と Claude Opus が役割分担しながら協調動作します。

- **Gemini 2.0 Flash**: 素早いプロトタイプ生成、反復修正、エラー検出
- **Claude Opus**: 複雑なビジネスロジック、Swift ネイティブコードの生成、セキュリティ要件の厳格な実装

特にセキュリティが重要な認証処理や、Apple の API を深く利用する機能（HealthKit、Siri Intents など）では Claude が前面に出てコードを生成します。

また、プロンプトが最初から明確であれば Gemini 2.0 Flash が単独で完成度の高いアプリを生成しますが、複雑な要件を持つ場合は自動的に Claude にバトンタッチします。この切り替えはユーザーが意識することなくバックグラウンドで行われます。

## よくある質問（FAQ）

**Q. Rork で使う AI モデルを自分で選べますか？**

A. 現在（2026年3月時点）、Standard プランでは Gemini 2.0 Flash が自動的に使用されます。Claude を使いたい場合は Rork Max プランへのアップグレードが必要です。将来的には Standard プランでもモデル選択オプションが追加される可能性があります。詳しくは「[Rork の料金プランを徹底解説](/articles/rork-basics/rork-pricing-plans)」を参照してください。

**Q. Gemini 2.0 Flash の生成結果が期待通りにならない場合はどうすればいいですか？**

A. 以下の手順を試してみましょう。①プロンプトをより具体的にする（画面数・使用する技術スタック・デザインの指示を追加する）、② Fix Now 機能でエラーを自動修正する、③セクションごとに段階的に機能を追加する。一度に多くの機能を要求するよりも、段階的に積み上げていく方が精度が上がります。

**Q. Gemini 2.0 Flash で生成したコードは本番環境で使えますか？**

A. はい、使えます。Rork が生成するコードは Expo（React Native）の標準的な書き方に準拠しており、TypeScript で型安全に書かれています。App Store / Google Play への公開も問題なく行えます。ただし、生成されたコードは必ずレビューし、セキュリティ上重要な処理（APIキーの管理・認証トークンの取り扱いなど）は追加でチェックすることを推奨します。

**Q. Rork の Gemini 統合は他の AI アプリビルダーと何が違いますか？**

A. Rork の特徴は「モバイルアプリ専用」に最適化されている点です。Bolt や Lovable が Web アプリ向けに設計されているのに対し、Rork は Expo・React Native のエコシステムに特化したプロンプト処理・ツール呼び出しが行われます。Gemini 2.0 Flash を使っていても、その呼び出し方や文脈の持たせ方が他サービスとは異なります。

**Q. Gemini 2.0 Flash は日本語のプロンプトに対応していますか？**

A. はい、完全に対応しています。日本語でプロンプトを書いても、Gemini 2.0 Flash は正確に意図を解釈してコードを生成します。ただし、コードのコメントは英語で生成されることが多いため、日本語コメントが必要な場合は「コメントは日本語で書いて」と追加指示するといいでしょう。
