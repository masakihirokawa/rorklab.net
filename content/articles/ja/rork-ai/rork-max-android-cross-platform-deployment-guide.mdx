「Rork Max を契約したのですが、Android 版はいつ出せますか？」という相談を、最近とても多くいただきます。結論から言うと、Rork Max は SwiftUI を出力するため iOS 専用で、現時点では Android 版の自動生成には対応していません。これは制約というよりも、Rork Max が「ネイティブ品質の iOS アプリを最短で作る」ことに振り切っている設計上の選択です。

ただし、これは「Rork のエコシステムで Android 版が作れない」という意味ではありません。私自身、最近リリースした個人開発のアプリでは Rork（無印）で React Native 版を、Rork Max で iOS ネイティブ版を、それぞれ別々にビルドして両 OS に展開しています。この記事では、その過程で見えてきた現実的な判断軸と、つまずきやすいポイントをまとめます。

## Rork と Rork Max の出力先がそもそも違う

混乱の元になっているので、まず整理しておきます。

- **Rork（無印）**: React Native + Expo を出力します。1 つのコードベースから iOS と Android の両方をビルドできます。クロスプラットフォーム前提の設計です
- **Rork Max**: SwiftUI を出力します。Apple のエコシステムに最適化された、純正のネイティブコードが生成されます。iOS 専用です

Rork Max のリリース時、SwiftUI を選んだ理由は「Apple Intelligence や WidgetKit、ARKit などの最新 API を即座に使えるネイティブ品質」を最優先したためだと、公式ブログで説明されています。React Native のラッパーを介すると、どうしても新 API の対応が数週間〜数ヶ月遅れます。Rork Max はその遅延を許容しない選択をしたわけです。

その結果、Android は出力対象から外れました。Android で同等の AI 生成体験を得るには、別のフロー、つまり Rork（無印）の React Native 出力を使う必要があります。

## 個人開発者にとっての3つの戦略

「両 OS に出したい」という前提で考えると、現実的には次の3つの戦略があります。

### 戦略1: Rork（無印）だけで両 OS を一気にカバーする

最もシンプルです。Rork のチャットで生成した React Native + Expo のコードベースから、iOS 版と Android 版を同時にビルドします。EAS Build を使えば両プラットフォームのビルドが1コマンドで走ります。

```bash
# iOS と Android 両方を一度にクラウドビルド
npx eas build --platform all --profile production

# プラットフォーム個別に走らせる場合
npx eas build --platform ios --profile production
npx eas build --platform android --profile production
```

このコマンドは Expo Application Services（EAS）の認証が済んでいる前提で、両 OS 用のバイナリを EAS のクラウド上で生成します。完了するとダウンロードリンクがメールで届きます。

このアプローチの利点は、コードベースが1つで済むこと、機能追加が両 OS に同時反映されること、そして多くの個人開発者にとってこれで十分なことです。欠点は、iOS のネイティブ機能（最新の WidgetKit や Live Activities など）の実装が React Native のラッパー対応待ちになることです。

### 戦略2: Rork Max で iOS、Rork（無印）で Android の二刀流

iOS には絶対にネイティブ品質を求めたい場合の選択肢です。Rork Max で iOS 版（SwiftUI）を、Rork（無印）で Android 版（React Native）を、それぞれ独立に開発します。

実際にやってみて分かったのは、機能を二度実装することよりも、**仕様の同期**の方が大変だという点でした。両者で UI のレイアウトや遷移のニュアンスが微妙にずれるため、プロダクトオーナー的な視点で「これは両 OS で揃える、これは OS 文化に合わせて変える」という線引きを最初に決めておくことが重要です。

私の場合は、画面遷移の方向（プッシュ vs モーダル）や戻るボタンの位置などは OS の慣習に合わせ、コア機能のロジック（料金プラン、課金フロー、通知文言など）は完全に揃える、というルールにしています。

### 戦略3: iOS 先行 → Android 後追い

リリース速度を最優先するなら、まず Rork Max で iOS 版を作って App Store に出し、初期ユーザーの反応を見てから Rork（無印）で Android 版を後追いで作る、という順序です。

個人開発の収益化観点でも、これは合理的です。日本市場では iOS と Android のユーザー比率がほぼ半々ですが、課金率は iOS が圧倒的に高い傾向があります。先に iOS を出して PMF（Product-Market Fit）を確認できれば、Android 版の開発に投じる時間が無駄になるリスクを減らせます。

ただし、Android 後追いには落とし穴があります。後述します。

## Android 後追いで詰まりやすい3つのポイント

戦略3を選んだ場合、私が実際に詰まったポイントを共有します。

**ひとつめは、Stripe や Google Play 課金の実装差です。** iOS では StoreKit 2、Android では Google Play Billing Library と、それぞれ全く別の API になります。Rork（無印）であれば `react-native-iap` や RevenueCat の SDK で抽象化できますが、Rork Max で書いた StoreKit 直叩きのコードはそのままでは Android に持っていけません。

**ふたつめは、プッシュ通知の設計です。** iOS は APNs、Android は FCM ですが、APNs では通知の表示方法を細かく制御できる一方、FCM は通知ペイロードの構造が異なり、画像付きリッチ通知の実装方法も別物です。Rork（無印）で開発する場合は Expo Notifications が両方を抽象化してくれます。

**みっつめは、UI コンポーネントの挙動差です。** SwiftUI で当たり前の磨りガラス調のマテリアル（`.background(.ultraThinMaterial)` など）は Android には存在しません。Material Design 3 の同等表現（`androidx.compose.material3` の `Surface` + tonal elevation）に置き換える必要があります。Rork（無印）の React Native であれば、最初から両 OS 共通の見た目を意識した UI を生成してくれるので、こうした差異は自然に吸収されます。

## どの戦略を選ぶべきか — 私の判断軸

ここまで読んで「結局どれがいいの？」と思われたかもしれません。私が個人開発者向けにおすすめしているのは、ほとんどのケースで **戦略1（Rork 無印で両 OS 同時）** です。

理由は単純で、個人開発の最大のリソース制約は「時間」だからです。コードベースを1つに保てるなら、機能追加もバグ修正も1回で済みます。これは長期運用において強烈な複利効果を生みます。

戦略2（二刀流）を選ぶべきなのは、iOS でしか実現できない体験（Apple Vision Pro 連携、Live Activities、WidgetKit のフル活用など）が事業のコア価値になっているケースに限ります。たとえば写真編集アプリで Apple Pencil の最新機能を使いたい、というような状況です。

戦略3（iOS 先行 → Android 後追い）は、個人開発でかつ初リリースのアプリにおすすめです。「Android 版が動くこと」を確認するために最初から実装するより、まず iOS 1 本に集中して PMF を確認する方が、結果として両 OS への展開が早くなります。

なお、各戦略の費用感や Rork Max の料金プランについては [Rork Max 料金プラン 2026年版：個人開発者の判断ガイド](/articles/rork-business/rork-max-pricing-2026-decision-guide) で詳しく整理しています。React Native と SwiftUI の選択基準については [Rork vs Flutter / React Native：2026年の比較](/articles/rork-basics/rork-vs-flutter-react-native-comparison-2026) も参考になるかもしれません。

## Android 版で意外と評価される機能

これは余談ですが、Android 版を後から出してみて意外だったのは、Android ユーザーから「ホーム画面ウィジェット」や「アプリショートカット」へのリクエストが多かったことです。iOS の WidgetKit は派手で目立ちますが、Android のウィジェット文化はそれより歴史が長く、ユーザーの定着度も高いという印象です。

Rork（無印）で Android 版を作る際は、`expo-router` のディープリンクと組み合わせてアプリショートカットを実装すると、Android ユーザーからのレビューが顕著に良くなりました。R&D 視点で参考になれば幸いです。

クロスプラットフォーム開発の理論的背景や設計判断について、より体系的に学びたい方には [『モバイルアプリ開発の教科書 React Native + Expo編』（日経BP）](https://www.amazon.co.jp/dp/B0CHZQXNTM/?tag=pinocchio-22) が、現場のリアルな知見を補完してくれます。

## 結びに

Rork Max を契約したからといって、必ずしも iOS 専用で進める必要はありません。むしろ Rork（無印）と Rork Max を「同じエコシステム内の役割分担」と捉えて使い分けることで、個人開発の機動力は何倍にもなります。

今日できる具体的な一歩としておすすめしたいのは、まず手元のアプリ案について「これは iOS のネイティブ体験が事業の核か？」と自問してみることです。答えが「核ではない」なら、Rork（無印）1本で両 OS に出すのが最速です。「核である」なら、戦略3で iOS 先行のリリース計画を立ててみてください。
