「Rork Max は SwiftUI ネイティブアプリを生成できる」という話を聞いて期待した方は多いと思います。私自身も最初は「これがあれば iOS アプリ開発が劇的に変わる」と思ったクチでした。

ただ、実際に Rork Max で SwiftUI コードを生成して、Xcode に取り込み、App Store の審査を通すところまでやってみると、**「生成 = ゴール」ではなく「生成 = スタートライン」** だという現実が見えてきます。この記事では、過大な期待でも過小な評価でもない、**実際に Rork Max を使い込んだ個人開発者の本音** を書きます。

私はこれまで Rork Max で 3 つの SwiftUI アプリを生成し、うち 2 つは現在も App Store で配信中です。その経験ベースで、**「使う前に知っておきたかった」と思うポイントを優先順** に整理しました。

## 第1章：Rork Max の SwiftUI 生成、本当のところ何ができるか

最初に率直な評価から書きます。Rork Max の SwiftUI 生成は **「叩き台としては優秀、本番品質には人間の手が必須」** というのが私の結論です。

### 生成精度の現状（2026年4月時点）

具体的な精度を表にまとめると次のようになります。これは私が 3 つのアプリで観測した実績値です。

| カテゴリ | 生成成功率 | コメント |
|---|---|---|
| 基本UI（リスト・フォーム・ナビゲーション） | 95% | ほぼそのまま使える |
| アニメーション・トランジション | 70% | 細かい調整が必要 |
| データ永続化（SwiftData） | 80% | スキーマ設計は要レビュー |
| ネットワーク通信 | 75% | エラーハンドリングが甘い |
| プッシュ通知 | 60% | Apple Developer 設定との連携部分が弱い |
| In-App Purchase（StoreKit 2） | 50% | 自動生成は最低限。本番投入は手動実装推奨 |
| ユニバーサルリンク・ディープリンク | 40% | 公式ドキュメントを見て手動実装が確実 |

ご覧の通り、**簡単な UI ほど精度が高く、Apple のエコシステム連携が必要なほど精度が落ちる** 傾向が明確です。これは生成 AI 全般の特性で、Web 上に「動くコード例」が大量にある領域は得意、特殊な権限設定や Apple Developer Portal との連携が絡む領域は苦手、という構図です。

### 生成されるコードの典型例

実際に Rork Max が出力する SwiftUI コードがどのようなレベルか、簡単な「タスク管理アプリ」を例に見てみましょう。プロンプトは以下のようなものです。

```
プロンプト:
「シンプルなタスク管理アプリを SwiftUI で作って。
- タスク一覧を表示
- タスクを追加できる（タイトル + 期限）
- タスクを完了状態に切り替えられる
- タスクを削除できる
- データは SwiftData で永続化
- iOS 17 以降対応」
```

この指示で Rork Max が生成する典型的な構造はこんな感じです。

```swift
import SwiftUI
import SwiftData

@Model
final class Task {
    var title: String
    var dueDate: Date
    var isCompleted: Bool
    var createdAt: Date

    init(title: String, dueDate: Date) {
        self.title = title
        self.dueDate = dueDate
        self.isCompleted = false
        self.createdAt = Date()
    }
}

struct ContentView: View {
    @Environment(\.modelContext) private var modelContext
    @Query(sort: \Task.dueDate) private var tasks: [Task]
    @State private var showingAddTask = false

    var body: some View {
        NavigationStack {
            List {
                ForEach(tasks) { task in
                    TaskRow(task: task)
                }
                .onDelete(perform: deleteTasks)
            }
            .navigationTitle("タスク")
            .toolbar {
                ToolbarItem(placement: .primaryAction) {
                    Button(action: { showingAddTask = true }) {
                        Image(systemName: "plus")
                    }
                }
            }
            .sheet(isPresented: $showingAddTask) {
                AddTaskView()
            }
        }
    }

    private func deleteTasks(at offsets: IndexSet) {
        for index in offsets {
            modelContext.delete(tasks[index])
        }
    }
}

struct TaskRow: View {
    @Bindable var task: Task

    var body: some View {
        HStack {
            Button(action: { task.isCompleted.toggle() }) {
                Image(systemName: task.isCompleted ? "checkmark.circle.fill" : "circle")
            }
            .buttonStyle(.borderless)

            VStack(alignment: .leading) {
                Text(task.title)
                    .strikethrough(task.isCompleted)
                Text(task.dueDate, style: .date)
                    .font(.caption)
                    .foregroundStyle(.secondary)
            }
        }
    }
}

struct AddTaskView: View {
    @Environment(\.modelContext) private var modelContext
    @Environment(\.dismiss) private var dismiss
    @State private var title = ""
    @State private var dueDate = Date()

    var body: some View {
        NavigationStack {
            Form {
                TextField("タイトル", text: $title)
                DatePicker("期限", selection: $dueDate, displayedComponents: .date)
            }
            .navigationTitle("新しいタスク")
            .toolbar {
                ToolbarItem(placement: .cancellationAction) {
                    Button("キャンセル") { dismiss() }
                }
                ToolbarItem(placement: .confirmationAction) {
                    Button("保存") {
                        modelContext.insert(Task(title: title, dueDate: dueDate))
                        dismiss()
                    }
                    .disabled(title.isEmpty)
                }
            }
        }
    }
}
```

これだけ見ると **「もう完成では？」** と感じるかもしれません。私も最初はそう思いました。

しかし、このコードを実際に Xcode に取り込んでビルドすると、いくつかの問題が出てきます。次の章でその対処法を解説します。

## 第2章：Xcode への取り込み手順と最初に直すべき5つの典型エラー

Rork Max が生成したコードを Xcode プロジェクトとして動かすまでの手順を、現実的な障害も含めて整理します。

### 取り込み手順

**Step 1**：Rork Max の「Export」ボタンから ZIP をダウンロード
**Step 2**：ZIP を解凍し、Xcode で `.xcodeproj` を開く
**Step 3**：Signing & Capabilities タブで自分の Apple Developer Team を設定
**Step 4**：Bundle Identifier を自分のドメインベースのユニークな ID に変更
**Step 5**：シミュレータで一度ビルドを通す

ここまでで **半分以上のケースでビルドエラー** が出ます。それぞれの対処法を見ていきましょう。

### 典型エラー 1：iOS Deployment Target の不一致

Rork Max は最新の iOS を前提に出力することが多く、`@Observable` マクロや `#Preview` マクロが iOS 17 以降に依存しています。プロジェクト設定の `iOS Deployment Target` を 17.0 以降に明示的に上げる必要があります。

### 典型エラー 2：Info.plist の権限記述漏れ

ネットワーク通信、カメラ、写真ライブラリ、位置情報などを使うコードが生成されているのに、`Info.plist` の `NSCameraUsageDescription` などの記述が漏れていることがあります。**ビルドは通るが実行時にクラッシュ** するため、初回テスト時に必ず確認します。

### 典型エラー 3：Asset Catalog の参照ミス

`Image("AppIcon")` のような参照が、Asset Catalog に存在しないアイコン名を指していることがあります。Assets.xcassets を開いて、参照されている画像が実在するか確認します。

### 典型エラー 4：String Catalog（.xcstrings）が空

ローカライズ対応がされた風の構造になっているのに、`.xcstrings` ファイルが空のまま、というケースがあります。多言語対応をするなら自分で翻訳を入れる必要があります。

### 典型エラー 5：SwiftData のスキーマ衝突

開発中にモデルクラスを変更すると、既存の SwiftData データベースとスキーマが衝突してクラッシュします。開発初期は **シミュレータをアプリごと削除** してから再ビルドするのが確実です。

## 第3章：App Store 審査を通すための必須手動修正リスト

ここからが本記事で一番伝えたいパートです。Rork Max が生成したアプリを **そのまま App Store に提出すると、確実にリジェクトされます**。最低限手を入れるべき項目を、Apple の審査ガイドラインに紐づけて整理します。

### 必須修正 1：プライバシーマニフェスト（PrivacyInfo.xcprivacy）

2024年以降、Apple は SDK と app 本体の **プライバシーマニフェスト** を必須化しました。Rork Max はこれを生成しないことが多いので、手動で追加する必要があります。

```xml
<!-- PrivacyInfo.xcprivacy の最小例 -->
<?xml version="1.0" encoding="UTF-8"?>
<plist version="1.0">
<dict>
    <key>NSPrivacyAccessedAPITypes</key>
    <array>
        <dict>
            <key>NSPrivacyAccessedAPIType</key>
            <string>NSPrivacyAccessedAPICategoryUserDefaults</string>
            <key>NSPrivacyAccessedAPITypeReasons</key>
            <array>
                <string>CA92.1</string>
            </array>
        </dict>
    </array>
    <key>NSPrivacyTracking</key>
    <false/>
</dict>
</plist>
```

### 必須修正 2：App Tracking Transparency への対応

広告 SDK を入れる場合は、`NSUserTrackingUsageDescription` を Info.plist に追加し、`ATTrackingManager.requestTrackingAuthorization` を実装する必要があります。Rork Max はこの対応をデフォルトで生成しません。

### 必須修正 3：エラーハンドリングの強化

生成されたネットワーク通信コードは、エラー時のユーザー体験が雑なケースが多いです。たとえば、API 呼び出し失敗時に「Error」とだけ表示する、リトライボタンがない、といった状態だと審査で「ユーザー体験が不十分」として弾かれることがあります。

私は、以下のような最低限のエラー UI を追加するようにしています。

```swift
// 最低限のエラー表示パターン
.alert("通信エラー", isPresented: $showingError, presenting: errorMessage) { _ in
    Button("再試行") { Task { await retry() } }
    Button("キャンセル", role: .cancel) {}
} message: { error in
    Text(error)
}
```

### 必須修正 4：空状態（Empty State）の UI

データが何もないときの画面が「ただの白い画面」になっているケースが多いです。「まだタスクがありません。右上のボタンから追加してください」のような **空状態のメッセージ** を必ず追加します。これがないと審査で「不完全なアプリ」と判定されることがあります。

### 必須修正 5：ローディング状態の明示

非同期処理中の `ProgressView` の表示が抜けていることが多いです。何も表示せずに 1〜2 秒待たされるのはユーザーにとって最悪の体験です。

## 第4章：SwiftUI ネイティブと React Native 出力の使い分け

Rork Max は SwiftUI ネイティブだけでなく、React Native の出力もできます。私の使い分けの判断基準は次の通りです。

### SwiftUI ネイティブを選ぶケース

- **App Store 中心で配信したい**（iOS 限定でも構わない）
- **iOS 固有の機能をフル活用したい**（HealthKit、ARKit、Live Activities、ウィジェット）
- **長期メンテナンス前提のプロダクト**
- **アプリのパフォーマンスが最重要**（ゲーム、写真編集、AR）

### React Native を選ぶケース

- **iOS と Android を 1 つのコードベースで配信したい**
- **アプリの機能が UI 中心で、デバイス固有 API への依存が少ない**
- **Web エンジニアがメンテナンスする予定**
- **MVP（最小限のプロダクト）として早く出して反応を見たい**

私自身は、**最初の MVP は React Native で出して、ユーザーが定着してきたら SwiftUI 版を別途作る** という戦略を取ることが多いです。この場合、Rork Max の React Native 出力で素早く検証し、本格運用フェーズで SwiftUI ネイティブに移行する、というフローが合理的です。

## 第5章：本番運用 1 か月で見えてきた長期的なメリットと限界

最後に、Rork Max で生成したアプリを 1 か月以上運用してきた立場から、長期目線での評価を書きます。

### 良かった点

- **MVP までの時間が圧倒的に短縮**：手書きで 2 週間かかるアプリが、生成 + 修正で 3 日
- **「ありがちな UI パターン」の実装ミスが減った**：標準的な NavigationStack の使い方など、慣れていなかった部分が学習教材としても役立った
- **既存アプリのリファクタリングのヒントになる**：Rork Max が生成する構造を見て、「あ、こう書いた方がきれいだ」と気づくことがある

### 限界を感じた点

- **アプリが大規模になると修正不能に近づく**：生成された 50 ファイル超のプロジェクトを後から手動修正していくのは現実的ではない
- **アップデート時の差分管理が難しい**：「機能追加して」と再生成すると、既存コードの構造ごと変わることがある
- **Apple の最新機能（毎年の WWDC 新機能）への追従は数か月遅れる**：新しい SwiftUI API を即座に使いたい場合は手書きが速い

私の現時点での最適解は、**「Rork Max で初期生成 → 1 週間以内に Xcode で本格的にメンテナンスを開始 → 以降は人間が主導で開発」** というフローです。「Rork Max に依存し続ける」のではなく、「Rork Max を出発点として使う」という距離感が、長期的には最も健全だと感じています。
