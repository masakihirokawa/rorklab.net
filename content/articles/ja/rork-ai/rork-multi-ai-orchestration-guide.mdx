1つのAIモデルだけに依存するアプリ開発の時代は終わりつつあります。Claude は論理的推論と長文生成に強く、Gemini はマルチモーダル処理と Google エコシステム連携に優れ、GPT-4o は汎用的なタスクで安定した品質を発揮します。2026年現在、これらの AI を**タスクの特性に応じて使い分ける「マルチAIオーケストレーション」**が、プロダクション品質のアプリに不可欠なアーキテクチャとして注目されています。

本記事では、Rork アプリ内で複数の AI プロバイダーを統合し、インテリジェントなルーティング・フォールバック・コスト最適化を実現する設計手法を、実装コード付きで徹底解説します。

## マルチAIオーケストレーションとは

マルチAIオーケストレーションとは、複数のAIプロバイダーのAPIを単一のアプリケーション内で統合管理し、リクエストの種類や状況に応じて最適なモデルへ自動的に振り分ける設計パターンです。

この設計が求められる主な理由は3つあります。

1つ目は**耐障害性**です。単一プロバイダーに依存していると、そのAPIがダウンした時点でアプリ全体が停止します。フォールバック先があれば、ユーザーは障害に気づくことなくアプリを使い続けられます。

2つ目は**コスト最適化**です。すべてのリクエストを最も高性能（で高価）なモデルに送る必要はありません。簡単な分類タスクにはコストの低いモデルを使い、複雑な推論が必要な場面でのみ高性能モデルを投入することで、API費用を大幅に削減できます。

3つ目は**品質の最大化**です。各AIモデルには得意分野があります。コード生成にはClaude、画像理解にはGemini、汎用チャットにはGPT-4oといった使い分けにより、すべてのタスクで最高品質のレスポンスを引き出せます。

## オーケストレーション層のアーキテクチャ設計

Rork アプリでマルチAIオーケストレーションを実装する際の推奨アーキテクチャは、3層構造です。

```
┌─────────────────────────────────┐
│        Rork アプリ (UI)          │
├─────────────────────────────────┤
│    オーケストレーション層         │  ← ルーティング・フォールバック・キャッシュ
│    ┌───────────┐                │
│    │  Router   │ ← タスク分類    │
│    │  Monitor  │ ← ヘルスチェック │
│    │  Cache    │ ← レスポンス保存 │
│    └───────────┘                │
├─────────────────────────────────┤
│  Claude API │ Gemini API │ GPT-4o API  │
└─────────────────────────────────┘
```

この構造により、UI層はどのAIモデルが応答しているかを意識する必要がなく、オーケストレーション層がすべてのルーティングロジックをカプセル化します。

### プロバイダー抽象化レイヤーの実装

まず、各AIプロバイダーを統一的なインターフェースで扱える抽象化レイヤーを実装します。

```typescript
// ai-orchestrator.ts — プロバイダー抽象化とルーティング

// 各プロバイダーの統一インターフェース
interface AIProvider {
  name: string;
  sendMessage(prompt: string, options?: RequestOptions): Promise<AIResponse>;
  checkHealth(): Promise<boolean>;
  estimateCost(tokenCount: number): number;
}

interface AIResponse {
  content: string;
  provider: string;
  latencyMs: number;
  tokenUsage: { input: number; output: number };
  cost: number;
}

interface RequestOptions {
  maxTokens?: number;
  temperature?: number;
  taskType?: TaskType;
}

// タスク分類（ルーティングの判断基準）
type TaskType =
  | 'reasoning'       // 論理的推論 → Claude優先
  | 'code-generation' // コード生成 → Claude優先
  | 'multimodal'      // 画像・動画理解 → Gemini優先
  | 'translation'     // 翻訳 → GPT-4o優先
  | 'chat'            // 汎用チャット → コスト最安優先
  | 'summarization';  // 要約 → Gemini優先

// Claude プロバイダー実装
class ClaudeProvider implements AIProvider {
  name = 'claude';

  async sendMessage(prompt: string, options?: RequestOptions): Promise<AIResponse> {
    const startTime = Date.now();
    // Anthropic SDK を使用してリクエスト送信
    const response = await fetch('https://api.anthropic.com/v1/messages', {
      method: 'POST',
      headers: {
        'x-api-key': process.env.ANTHROPIC_API_KEY!,
        'content-type': 'application/json',
        'anthropic-version': '2023-06-01',
      },
      body: JSON.stringify({
        model: 'claude-sonnet-4-6',
        max_tokens: options?.maxTokens ?? 2048,
        messages: [{ role: 'user', content: prompt }],
      }),
    });
    const data = await response.json();
    const latencyMs = Date.now() - startTime;

    return {
      content: data.content[0].text,
      provider: this.name,
      latencyMs,
      tokenUsage: {
        input: data.usage.input_tokens,
        output: data.usage.output_tokens,
      },
      cost: this.estimateCost(data.usage.input_tokens + data.usage.output_tokens),
    };
  }

  async checkHealth(): Promise<boolean> {
    try {
      const res = await fetch('https://api.anthropic.com/v1/messages', {
        method: 'POST',
        headers: {
          'x-api-key': process.env.ANTHROPIC_API_KEY!,
          'content-type': 'application/json',
          'anthropic-version': '2023-06-01',
        },
        body: JSON.stringify({
          model: 'claude-sonnet-4-6',
          max_tokens: 10,
          messages: [{ role: 'user', content: 'ping' }],
        }),
      });
      return res.ok;
    } catch {
      return false;
    }
  }

  estimateCost(tokenCount: number): number {
    return tokenCount * 0.000003; // Sonnet の概算コスト（USD/token）
  }
}
```

上記の `ClaudeProvider` と同じインターフェースで `GeminiProvider`、`GPT4oProvider` も実装します。ポイントは、すべてのプロバイダーが `AIProvider` インターフェースに準拠することです。

## インテリジェントルーティングの実装

オーケストレーション層の核心は、タスクの種類に応じて最適なプロバイダーを選択するルーターです。

```typescript
// router.ts — タスクベースのインテリジェントルーティング

// 各タスクタイプに対するプロバイダー優先度マップ
const ROUTING_TABLE: Record<TaskType, string[]> = {
  'reasoning':       ['claude', 'gpt4o', 'gemini'],
  'code-generation': ['claude', 'gpt4o', 'gemini'],
  'multimodal':      ['gemini', 'gpt4o', 'claude'],
  'translation':     ['gpt4o', 'claude', 'gemini'],
  'chat':            ['gemini', 'gpt4o', 'claude'],  // コスト順
  'summarization':   ['gemini', 'claude', 'gpt4o'],
};

class AIRouter {
  private providers: Map<string, AIProvider>;
  private healthStatus: Map<string, boolean>;
  private lastHealthCheck: number = 0;
  private readonly HEALTH_CHECK_INTERVAL = 60_000; // 60秒

  constructor(providers: AIProvider[]) {
    this.providers = new Map(providers.map(p => [p.name, p]));
    this.healthStatus = new Map(providers.map(p => [p.name, true]));
  }

  // タスクに最適なプロバイダーを選択して実行
  async route(prompt: string, taskType: TaskType, options?: RequestOptions): Promise<AIResponse> {
    await this.refreshHealthIfNeeded();

    const priorityList = ROUTING_TABLE[taskType];

    for (const providerName of priorityList) {
      const provider = this.providers.get(providerName);
      const isHealthy = this.healthStatus.get(providerName);

      if (!provider || !isHealthy) continue;

      try {
        const response = await provider.sendMessage(prompt, {
          ...options,
          taskType,
        });

        // レスポンス品質の簡易チェック
        if (response.content.length < 10 && taskType !== 'chat') {
          console.warn(`[Router] ${providerName}: 短すぎるレスポンス — フォールバック`);
          continue;
        }

        console.log(`[Router] ${taskType} → ${providerName} (${response.latencyMs}ms, $${response.cost.toFixed(4)})`);
        return response;

      } catch (error) {
        console.error(`[Router] ${providerName} 失敗:`, error);
        this.healthStatus.set(providerName, false);
        // 次のプロバイダーへフォールバック
        continue;
      }
    }

    throw new Error('すべてのAIプロバイダーが利用不可です');
  }

  // 定期的なヘルスチェック
  private async refreshHealthIfNeeded(): Promise<void> {
    if (Date.now() - this.lastHealthCheck < this.HEALTH_CHECK_INTERVAL) return;

    const checks = Array.from(this.providers.entries()).map(
      async ([name, provider]) => {
        const healthy = await provider.checkHealth();
        this.healthStatus.set(name, healthy);
      }
    );

    await Promise.allSettled(checks);
    this.lastHealthCheck = Date.now();
  }
}
```

このルーターは、タスクタイプに基づいて最適なプロバイダーを選択し、障害時には自動的に次の候補にフォールバックします。

## コスト最適化エンジン

マルチAIを運用する上で避けて通れないのがAPIコストの管理です。以下のコスト最適化エンジンは、月間の予算上限を設定し、使用状況に応じてプロバイダーの選択を動的に調整します。

```typescript
// cost-optimizer.ts — 予算管理とコスト追跡

interface UsageRecord {
  provider: string;
  taskType: TaskType;
  cost: number;
  timestamp: number;
}

class CostOptimizer {
  private usageLog: UsageRecord[] = [];
  private monthlyBudget: number; // USD
  private readonly COST_PER_TOKEN: Record<string, { input: number; output: number }> = {
    claude:  { input: 0.000003, output: 0.000015 },
    gemini:  { input: 0.0000005, output: 0.0000015 },
    gpt4o:   { input: 0.0000025, output: 0.00001 },
  };

  constructor(monthlyBudget: number = 50) {
    this.monthlyBudget = monthlyBudget;
  }

  recordUsage(record: UsageRecord): void {
    this.usageLog.push(record);
  }

  getCurrentMonthSpend(): number {
    const startOfMonth = new Date();
    startOfMonth.setDate(1);
    startOfMonth.setHours(0, 0, 0, 0);

    return this.usageLog
      .filter(r => r.timestamp >= startOfMonth.getTime())
      .reduce((sum, r) => sum + r.cost, 0);
  }

  getBudgetRemaining(): number {
    return this.monthlyBudget - this.getCurrentMonthSpend();
  }

  // 予算残量に応じてプロバイダー優先度を調整
  adjustRouting(originalPriority: string[]): string[] {
    const remaining = this.getBudgetRemaining();
    const spendRatio = this.getCurrentMonthSpend() / this.monthlyBudget;

    // 予算の80%以上を消費 → コスト最安のプロバイダーを優先
    if (spendRatio > 0.8) {
      return [...originalPriority].sort((a, b) => {
        const costA = this.COST_PER_TOKEN[a]?.output ?? 0;
        const costB = this.COST_PER_TOKEN[b]?.output ?? 0;
        return costA - costB; // 安い順
      });
    }

    return originalPriority;
  }

  // 日次コストレポートを生成
  generateDailyReport(): string {
    const today = new Date();
    today.setHours(0, 0, 0, 0);

    const todayUsage = this.usageLog.filter(r => r.timestamp >= today.getTime());
    const byProvider: Record<string, number> = {};

    todayUsage.forEach(r => {
      byProvider[r.provider] = (byProvider[r.provider] || 0) + r.cost;
    });

    // 期待する出力例:
    // === 日次コストレポート ===
    // Claude: $1.23
    // Gemini: $0.45
    // GPT-4o: $0.78
    // 合計: $2.46 / 月間予算: $50.00 (残: $32.10)
    let report = '=== 日次コストレポート ===\n';
    Object.entries(byProvider).forEach(([provider, cost]) => {
      report += `${provider}: $${cost.toFixed(2)}\n`;
    });
    report += `合計: $${todayUsage.reduce((s, r) => s + r.cost, 0).toFixed(2)}`;
    report += ` / 月間予算: $${this.monthlyBudget.toFixed(2)}`;
    report += ` (残: $${this.getBudgetRemaining().toFixed(2)})`;

    return report;
  }
}
```

## レスポンス品質の自動評価

複数のAIプロバイダーを運用していると、「どのモデルがどのタスクで最も高品質なレスポンスを返しているか」を定量的に把握する必要があります。

```typescript
// quality-evaluator.ts — レスポンス品質の自動評価

interface QualityMetrics {
  relevance: number;   // 0-1: プロンプトとの関連性
  completeness: number; // 0-1: 回答の完全性
  latency: number;     // レイテンシ（ms）
  costEfficiency: number; // 品質/コスト比
}

class QualityEvaluator {
  private metricsHistory: Map<string, QualityMetrics[]> = new Map();

  // レスポンスの品質を簡易評価
  evaluate(prompt: string, response: AIResponse): QualityMetrics {
    const relevance = this.calculateRelevance(prompt, response.content);
    const completeness = this.calculateCompleteness(response.content);
    const costEfficiency = (relevance + completeness) / 2 / Math.max(response.cost, 0.0001);

    const metrics: QualityMetrics = {
      relevance,
      completeness,
      latency: response.latencyMs,
      costEfficiency,
    };

    // 履歴に記録
    const history = this.metricsHistory.get(response.provider) || [];
    history.push(metrics);
    if (history.length > 100) history.shift(); // 直近100件を保持
    this.metricsHistory.set(response.provider, history);

    return metrics;
  }

  // プロバイダーごとの平均品質スコアを取得
  getProviderScores(): Record<string, number> {
    const scores: Record<string, number> = {};
    this.metricsHistory.forEach((history, provider) => {
      const avg = history.reduce((sum, m) =>
        sum + (m.relevance + m.completeness) / 2, 0
      ) / history.length;
      scores[provider] = avg;
    });
    return scores;
  }

  private calculateRelevance(prompt: string, response: string): number {
    // プロンプト内のキーワードがレスポンスに含まれる割合
    const keywords = prompt.toLowerCase().split(/\s+/).filter(w => w.length > 3);
    if (keywords.length === 0) return 0.5;
    const found = keywords.filter(kw => response.toLowerCase().includes(kw));
    return found.length / keywords.length;
  }

  private calculateCompleteness(response: string): number {
    // 文字数ベースの完全性スコア（短すぎる/長すぎるレスポンスを減点）
    const len = response.length;
    if (len < 50) return 0.2;
    if (len < 200) return 0.5;
    if (len < 2000) return 0.9;
    return 1.0;
  }
}
```

## Rork アプリへの統合

これまでに実装したコンポーネントを、Rork アプリのReact Nativeコードに統合します。以下は、AIオーケストレーターを使用するカスタムフックの例です。

```typescript
// hooks/useAIOrchestrator.ts — Rorkアプリ用カスタムフック

import { useState, useCallback, useRef } from 'react';

interface UseAIOrchestratorResult {
  sendQuery: (prompt: string, taskType: TaskType) => Promise<void>;
  response: string | null;
  isLoading: boolean;
  activeProvider: string | null;
  error: string | null;
  costInfo: { totalToday: number; budgetRemaining: number } | null;
}

export function useAIOrchestrator(): UseAIOrchestratorResult {
  const [response, setResponse] = useState<string | null>(null);
  const [isLoading, setIsLoading] = useState(false);
  const [activeProvider, setActiveProvider] = useState<string | null>(null);
  const [error, setError] = useState<string | null>(null);
  const [costInfo, setCostInfo] = useState<{ totalToday: number; budgetRemaining: number } | null>(null);

  const routerRef = useRef<AIRouter | null>(null);
  const costOptimizerRef = useRef(new CostOptimizer(50)); // 月間$50予算

  const getRouter = useCallback(() => {
    if (!routerRef.current) {
      routerRef.current = new AIRouter([
        new ClaudeProvider(),
        new GeminiProvider(),
        new GPT4oProvider(),
      ]);
    }
    return routerRef.current;
  }, []);

  const sendQuery = useCallback(async (prompt: string, taskType: TaskType) => {
    setIsLoading(true);
    setError(null);
    setActiveProvider(null);

    try {
      const router = getRouter();
      const result = await router.route(prompt, taskType);

      setResponse(result.content);
      setActiveProvider(result.provider);

      // コスト記録
      costOptimizerRef.current.recordUsage({
        provider: result.provider,
        taskType,
        cost: result.cost,
        timestamp: Date.now(),
      });

      setCostInfo({
        totalToday: costOptimizerRef.current.getCurrentMonthSpend(),
        budgetRemaining: costOptimizerRef.current.getBudgetRemaining(),
      });

    } catch (err) {
      setError(err instanceof Error ? err.message : 'AI リクエストに失敗しました');
    } finally {
      setIsLoading(false);
    }
  }, [getRouter]);

  return { sendQuery, response, isLoading, activeProvider, error, costInfo };
}
```

この `useAIOrchestrator` フックを使えば、Rork アプリのどの画面からでも、タスクタイプを指定するだけでインテリジェントなAIルーティングが利用できます。

## よくある質問（FAQ）

### Q: マルチAIオーケストレーションはどの程度のアプリ規模から導入すべきですか？

月間のAPIリクエスト数が1,000回を超えるあたりから、コスト最適化の効果が目に見えて現れます。ただし、耐障害性の観点では、リクエスト数に関係なく導入する価値があります。本番環境で単一プロバイダーに障害が発生した場合のビジネスインパクトを考えると、早い段階でフォールバック機構を用意しておくことをおすすめします。

### Q: APIキーの管理はどうすれば安全ですか？

Rork アプリではクライアントサイドにAPIキーを埋め込まないでください。バックエンドサーバー（Supabase Edge Functions や Cloudflare Workers など）にプロキシAPIを配置し、そこにキーを保管するのが最善です。環境変数やシークレットマネージャーを利用し、キーのローテーションも定期的に行いましょう。

### Q: ルーティングテーブルのカスタマイズはどのように行いますか？

`ROUTING_TABLE` は用途に合わせて自由に変更可能です。運用を開始したら、`QualityEvaluator` が蓄積するプロバイダーごとの品質スコアデータを分析し、定期的にルーティングテーブルを更新するのが理想的です。将来的には、品質データを元にルーティングテーブルを自動更新する仕組みも実装可能です。

### Q: レスポンスのキャッシュはどのように実装しますか？

同じプロンプトに対するレスポンスは、AsyncStorage や MMKV にキャッシュすることでAPIコストを削減できます。ただし、キャッシュの有効期限（TTL）を適切に設定し、古い情報を返さないよう注意してください。ニュースや株価のようなリアルタイム性が求められるタスクではキャッシュを無効化するのが望ましいです。
