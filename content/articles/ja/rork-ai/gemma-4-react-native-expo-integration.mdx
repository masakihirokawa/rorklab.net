「スマートフォンアプリにAI機能を入れたいが、API呼び出しのたびに通信費がかかるのが気になる」「オフラインでも動く音声認識を実装したい」——こういった悩みに対して、Gemma 4 の E2B/E4B モデルは現実的な答えを出してくれます。

Apache 2.0 ライセンスなので App Store や Google Play への公開に制約はありません。モデルウェイトを端末に持てば、推論は完全にローカルで動く。この記事では React Native（Expo）環境での実装方法を3つのパターンで解説します。

## React NativeでオンデバイスAIを動かす3つのアプローチ

Gemma 4 を React Native アプリで使う方法は、プロジェクトの要件によって選択肢が変わる。

**アプローチ1: llama.rn（推奨・iOS/Android両対応）**

[llama.rn](https://github.com/mybigday/llama.rn) は llama.cpp の React Native バインディングで、GGUF 形式のモデルをネイティブ推論エンジンで実行できます。Gemma 4 の GGUF 版は Hugging Face で公開されています。

```bash
# インストール
npm install llama.rn
cd ios && pod install
```

```typescript
import { LlamaContext, initLlama } from "llama.rn";

let context: LlamaContext | null = null;

async function loadGemma4(): Promise<void> {
  // モデルファイルはアプリバンドルまたはドキュメントディレクトリに配置
  const modelPath = `${RNFS.DocumentDirectoryPath}/gemma4-4b.gguf`;

  context = await initLlama({
    model: modelPath,
    use_mlock: true,
    n_ctx: 4096,      // コンテキスト窓
    n_threads: 4,     // CPUスレッド数
    n_gpu_layers: 32, // GPU利用（Metal/Vulkan対応端末）
  });
}

async function generateText(prompt: string): Promise<string> {
  if (\!context) throw new Error("モデルが読み込まれていません");

  const result = await context.completion({
    prompt: `<start_of_turn>user
${prompt}<end_of_turn>
<start_of_turn>model
`,
    n_predict: 512,
    temperature: 0.7,
    top_p: 0.9,
    stop: ["<end_of_turn>"],
  });

  return result.text;
}
```

**アプローチ2: ONNX Runtime Mobile（軽量・クロスプラットフォーム）**

ONNX形式に変換したGemma 4を使う方法。モデルサイズを最小限に抑えたい場合や、既存のMLパイプラインとの互換性が必要な場合に有効です。

```typescript
import { InferenceSession, Tensor } from "onnxruntime-react-native";

async function runOnnxInference(inputText: string): Promise<string> {
  const session = await InferenceSession.create(
    `${RNFS.DocumentDirectoryPath}/gemma4-e2b.onnx`
  );

  // トークナイゼーション（別途tokenizer実装が必要）
  const inputIds = tokenize(inputText);
  const inputTensor = new Tensor("int64", inputIds, [1, inputIds.length]);

  const outputs = await session.run({ input_ids: inputTensor });
  return decode(outputs.logits);
}
```

**アプローチ3: Ollama + ローカルサーバー（開発・プロトタイプ向け）**

デバイス上ではなく同一ネットワーク上のサーバー（開発機やRaspberry Pi）で Ollama を動かし、アプリからAPIで叩くパターン。開発速度を優先する場合に有効です。

```typescript
const OLLAMA_BASE_URL = "http://192.168.1.100:11434"; // ローカルサーバーのIP

async function queryGemma4(prompt: string): Promise<string> {
  const response = await fetch(`${OLLAMA_BASE_URL}/api/generate`, {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({
      model: "gemma4:4b",
      prompt,
      stream: false,
    }),
  });
  const data = await response.json();
  return data.response;
}
```

## E2B/E4Bモデルのファイルサイズと端末要件

アプリにモデルを同梱する場合、ファイルサイズと端末スペックの考慮が必要です。

GGUF 量子化の選択基準:

- **Q4_K_M**（推奨）: E2B約1.4GB、E4B約2.5GB。精度と速度のバランスが良い。最低メモリ: 3GB
- **Q5_K_M**: E2B約1.8GB、E4B約3.1GB。品質重視。最低メモリ: 4GB
- **Q8_0**: E2B約2.8GB。最高品質だがサイズが大きいです。最低メモリ: 5GB

```typescript
import RNFS from "react-native-fs";

async function downloadModel(quantType: string = "Q4_K_M"): Promise<void> {
  const modelUrl = `https://huggingface.co/your-org/gemma4-gguf/resolve/main/gemma4-e4b-${quantType}.gguf`;
  const destPath = `${RNFS.DocumentDirectoryPath}/gemma4-e4b.gguf`;

  // 既にダウンロード済みか確認
  const exists = await RNFS.exists(destPath);
  if (exists) return;

  // バックグラウンドダウンロード（進捗表示付き）
  const download = RNFS.downloadFile({
    fromUrl: modelUrl,
    toFile: destPath,
    progressInterval: 500,
    begin: () => console.log("ダウンロード開始"),
    progress: (res) => {
      const progress = res.bytesWritten / res.contentLength;
      console.log(`進捗: ${(progress * 100).toFixed(1)}%`);
    },
  });

  await download.promise;
  console.log("モデルのダウンロード完了");
}
```

## App Store / Google Play 審査の注意点

モデルファイルをアプリに同梱する場合、App Store と Google Play それぞれに注意点があります。

**App Store**: 初回ダウンロード時のアプリサイズが 4GB を超える場合、App Thinning（App Slicing）の設定が必要になります。モデルファイルを On-Demand Resource（ODR）として分離することで、インストールサイズを抑えられます。

**Google Play**: APK/AAB のサイズ制限（100MB）があるため、モデルファイルはアプリ内ダウンロードで対処する。Play Asset Delivery（PAD）を使うと、初回起動時にモデルを自動取得できます。

```typescript
// Expo での動的ダウンロード実装例
import { Asset } from "expo-asset";
import * as FileSystem from "expo-file-system";

export async function ensureModelLoaded(): Promise<string> {
  const modelDir = `${FileSystem.documentDirectory}models/`;
  await FileSystem.makeDirectoryAsync(modelDir, { intermediates: true });

  const modelPath = `${modelDir}gemma4-e4b-q4.gguf`;
  const info = await FileSystem.getInfoAsync(modelPath);

  if (\!info.exists) {
    console.log("モデルをダウンロード中...");
    const { uri } = await FileSystem.downloadAsync(
      "https://your-cdn.example.com/gemma4-e4b-q4.gguf",
      modelPath
    );
    return uri;
  }
  return modelPath;
}
```

## 実装してみて分かること

llama.rn でGemma 4 E4B（Q4_K_M量子化）を動かした場合の実測感覚として、iPhone 15 Pro（A17 Pro）では 10〜20 トークン/秒程度の生成速度が目安になります。リアルタイムチャットには少し遅く感じるが、音声入力 → テキスト生成 → 要約 → 通知 というバックグラウンドタスクには十分な速度です。

Gemma 4 の音声入力は E2B/E4B に搭載されているが、llama.rn 経由での音声処理は現時点では別途音声前処理が必要になる場合が多いです。テキスト生成タスクから始めて、音声処理は Whisper.rn 等と組み合わせるパターンが実装しやすい。

Rork でのビジュアル設計を終えた後、AI機能を追加するタイミングで llama.rn を取り込む流れが、現時点での最も現実的なアプローチです。
