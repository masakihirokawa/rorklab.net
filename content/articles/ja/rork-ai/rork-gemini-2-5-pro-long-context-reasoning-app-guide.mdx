## はじめに：なぜ Gemini 2.5 Pro なのか

2025年に登場したGemini 2.5 Proは、AIアプリ開発の風景を大きく変えました。最大100万トークン（約75万語）のコンテキストウィンドウと、思考プロセスを可視化できる「Thinking Mode」は、従来のAIモデルでは不可能だった複雑なタスクを現実のアプリに組み込むことを可能にしました。

RorkでGemini 2.5 Proを活用すれば、以下のようなアプリが構築できます。

- 契約書全文（100ページ超）をリアルタイムでレビューするリーガルテック
- 数千行のコードベースを一括でリファクタリング提案するDev Tool
- 財務報告書や研究論文を深く分析するビジネスインテリジェンスアプリ
- 複数のドキュメントを横断比較して矛盾点を抽出するコンプライアンスチェッカー

この記事では、Gemini 2.5 ProのAPIをRork Max環境で実装する具体的な手順から、コスト最適化のための設計パターンまで、実装に必要な知識をすべて網羅します。対象読者は、RorkとTypeScriptの基本を習得済みで、本格的なAI機能の実装に挑戦したい中〜上級者です。

## Gemini 2.5 Pro の主要機能と活用シナリオ

### 100万トークンコンテキストの実力

GPT-4oが128,000トークン、Claude 3.7が200,000トークンであるのに対し、Gemini 2.5 Proは最大**1,000,000トークン**（約750,000ワード）のコンテキストを一度に処理できます。これは一般的な小説10冊分、あるいは15,000行のソースコードに相当します。

このコンテキスト量があれば、以下のような処理が単一のAPIコールで完結します。

- **法的文書**: 契約書全文＋関連判例＋社内ガイドラインを同時入力して整合性チェック
- **コードレビュー**: 大規模リポジトリのファイル群を一括で解析して技術的負債を特定
- **財務分析**: 複数年度の決算書をすべて入力して異常値とトレンドを抽出
- **学術研究**: 数十本の論文を同時に読み込んでメタ分析レポートを生成

### Thinking Mode：透明な推論プロセス

Thinking Mode（`thinkingConfig`で有効化）は、モデルが答えを出す前の「考えるプロセス」を出力するオプション機能です。数学的証明、法的推論、ビジネス戦略立案など、単純な情報検索ではなく**多段階の論理的思考**が必要なタスクで特に効果を発揮します。

ユーザーにとっては「なぜそう判断したのか」が見えるため、AIへの信頼性向上につながります。エンタープライズ向けアプリでは、この透明性が導入の決め手になるケースも多いです。

### Flash vs. Pro：適切なモデルの選択

Gemini 2.5 Proは高性能ですが、すべてのクエリに使う必要はありません。コスト効率を考えると、シンプルなタスクにはGemini 2.5 Flashを使い、複雑なタスクにのみProを使うルーティング設計が理想的です。

| 用途 | 推奨モデル |
|---|---|
| 短い質問・FAQ応答 | Gemini 2.5 Flash |
| 単一文書のサマリー（50ページ未満） | Gemini 2.5 Flash |
| 複数文書の横断分析 | Gemini 2.5 Pro |
| 法的・医療的判断を要するタスク | Gemini 2.5 Pro + Thinking Mode |
| 10,000行超のコードレビュー | Gemini 2.5 Pro |

## 環境構築とAPIキーの設定

### Google AI Studio でのプロジェクトセットアップ

まず[Google AI Studio](https://aistudio.google.com/)にアクセスし、APIキーを取得します。

1. Google AI Studio にサインイン
2. 右上の「Get API Key」をクリック
3. 新しいプロジェクトを作成するか、既存プロジェクトを選択
4. 「Create API Key」でキーを生成

> **注意**: APIキーは絶対にソースコードに直接埋め込まないでください。Rork Maxでは環境変数機能を使って安全に管理します。

### Rork Max の環境変数設定

Rorkのプロジェクト設定画面で以下の環境変数を追加します。

```
GOOGLE_AI_API_KEY=YOUR_GEMINI_API_KEY
```

実際のAPIキーの代わりに `YOUR_GEMINI_API_KEY` というプレースホルダーを使ってコードを管理し、Rorkのシークレット管理機能で実際の値を設定してください。

### パッケージのインストール

Rork Maxのターミナルで以下を実行します。

```bash
npm install @google/generative-ai
```

### モデルルーターの実装

タスクの複雑さに応じてFlashとProを自動的に切り替えるルーターを実装します。

```typescript
// src/lib/gemini-router.ts
import { geminiFlash, geminiPro } from "./gemini";

interface AnalysisTask {
  content: string;          // 分析対象のコンテンツ
  taskType: "simple" | "complex" | "legal" | "financial" | "code";
  requiresThinking?: boolean;
}

interface AnalysisResult {
  response: string;
  modelUsed: "flash" | "pro";
  tokenCount?: number;
}

/**
 * タスクの複雑さに応じてFlash / Proを自動選択するルーター
 * Expected output: { response: "分析結果テキスト", modelUsed: "pro", tokenCount: 15000 }
 */
export async function analyzeWithRouter(task: AnalysisTask): Promise<AnalysisResult> {
  const shouldUsePro = 
    task.taskType === "legal" ||
    task.taskType === "financial" ||
    task.taskType === "code" ||
    task.content.length > 50000 ||  // 50,000文字超はPro
    task.requiresThinking === true;

  const model = shouldUsePro ? geminiPro : geminiFlash;
  const modelUsed = shouldUsePro ? "pro" : "flash";

  const result = await model.generateContent(task.content);
  const response = result.response.text();

  return {
    response,
    modelUsed,
    tokenCount: result.response.usageMetadata?.totalTokenCount,
  };
}
```

## 長文ドキュメント分析アプリの実装

### ドキュメントアップロードと処理パイプライン

Gemini 2.5 Proの真価は、巨大なドキュメントをまるごと処理できる点にあります。以下は、PDFや長文テキストを受け取り、深い分析レポートを生成する実装例です。

```typescript
// src/services/document-analyzer.ts
import { geminiPro } from "../lib/gemini";

interface DocumentAnalysisRequest {
  documentText: string;   // ドキュメントの全テキスト
  analysisType: "summary" | "legal_review" | "financial_audit" | "risk_assessment";
  userQuestion?: string;  // ユーザーの特定の質問（オプション）
}

interface DocumentAnalysisResult {
  summary: string;
  keyFindings: string[];
  risks: string[];
  recommendations: string[];
  rawResponse: string;
}

const ANALYSIS_PROMPTS = {
  summary: `以下のドキュメントを詳しく分析し、以下の形式でJSON出力してください：
    - summary: 要点を3〜5文で
    - keyFindings: 重要な発見事項（配列）
    - risks: リスク・問題点（配列）
    - recommendations: 推奨アクション（配列）`,
  
  legal_review: `法律の専門家として、以下の契約書・法的文書を精査し、以下の形式でJSON出力してください：
    - summary: 文書の概要と目的
    - keyFindings: 重要な条項・権利・義務（配列）
    - risks: 法的リスクや不利な条件（配列）
    - recommendations: 修正・交渉すべき点（配列）`,
  
  financial_audit: `財務アナリストとして、以下の財務データを分析し、JSON形式で出力してください：
    - summary: 財務状況の総括
    - keyFindings: 重要な財務指標・トレンド（配列）
    - risks: 財務リスクや懸念事項（配列）
    - recommendations: 改善策・投資判断（配列）`,
  
  risk_assessment: `リスクマネジメントの専門家として、以下の文書からリスクを洗い出し、JSON形式で出力してください：
    - summary: リスク評価の概要
    - keyFindings: 特定されたリスク要因（配列）
    - risks: 高・中・低リスクに分類したリスク一覧（配列）
    - recommendations: リスク軽減策（配列）`,
};

/**
 * 長文ドキュメントをGemini 2.5 Proで分析する
 * Expected output: { summary: "...", keyFindings: [...], risks: [...], recommendations: [...] }
 */
export async function analyzeDocument(
  request: DocumentAnalysisRequest
): Promise<DocumentAnalysisResult> {
  const systemPrompt = ANALYSIS_PROMPTS[request.analysisType];
  
  const userPrompt = request.userQuestion
    ? `${systemPrompt}\n\n特に以下の点に注目してください：${request.userQuestion}\n\n---\n\n${request.documentText}`
    : `${systemPrompt}\n\n---\n\n${request.documentText}`;

  const result = await geminiPro.generateContent({
    contents: [{ role: "user", parts: [{ text: userPrompt }] }],
    generationConfig: {
      responseMimeType: "application/json",
    },
  });

  const rawResponse = result.response.text();
  
  // JSON パース（エラーハンドリング付き）
  let parsed: DocumentAnalysisResult;
  try {
    parsed = JSON.parse(rawResponse);
  } catch {
    // JSON パース失敗時のフォールバック
    parsed = {
      summary: rawResponse,
      keyFindings: [],
      risks: [],
      recommendations: [],
      rawResponse,
    };
  }

  return { ...parsed, rawResponse };
}
```

### ストリーミング実装で UX を向上させる

長文ドキュメントの分析は数秒〜数十秒かかります。ストリーミングAPIを使えば、生成中のテキストをリアルタイムでユーザーに表示できます。

```typescript
// src/services/streaming-analyzer.ts
import { geminiPro } from "../lib/gemini";

interface StreamingCallbacks {
  onChunk: (text: string) => void;      // テキストチャンクを受信するたびに呼ばれる
  onComplete: (fullText: string) => void; // 完了時に呼ばれる
  onError: (error: Error) => void;       // エラー時に呼ばれる
}

/**
 * ストリーミングでドキュメントを分析し、リアルタイムで結果を返す
 */
export async function analyzeDocumentStream(
  prompt: string,
  callbacks: StreamingCallbacks
): Promise<void> {
  try {
    const result = await geminiPro.generateContentStream(prompt);
    
    let fullText = "";
    
    for await (const chunk of result.stream) {
      const chunkText = chunk.text();
      fullText += chunkText;
      callbacks.onChunk(chunkText);
    }
    
    callbacks.onComplete(fullText);
  } catch (error) {
    callbacks.onError(error as Error);
  }
}
```

## Thinking Mode の実装：推論プロセスを可視化する

### Thinking Mode を有効にした API 呼び出し

Thinking ModeはGemini 2.5 Pro特有の機能で、モデルが答えを出す前の内部思考プロセスを出力します。法的判断や複雑な分析で「なぜそう判断したか」を示す場合に特に有効です。

```typescript
// src/services/thinking-analyzer.ts
import { GoogleGenerativeAI } from "@google/generative-ai";

const genAI = new GoogleGenerativeAI(process.env.GOOGLE_AI_API_KEY!);

interface ThinkingAnalysisResult {
  thinkingProcess: string;  // モデルの推論プロセス
  finalAnswer: string;      // 最終的な回答
}

/**
 * Thinking Mode で複雑な質問を分析する
 * Expected output: { thinkingProcess: "まず前提を確認...", finalAnswer: "結論として..." }
 */
export async function analyzeWithThinking(
  question: string,
  context?: string
): Promise<ThinkingAnalysisResult> {
  // Thinking Mode 対応モデルの初期化
  const model = genAI.getGenerativeModel({
    model: "gemini-2.5-pro",
  });

  const fullPrompt = context
    ? `背景情報:\n${context}\n\n質問:\n${question}`
    : question;

  const result = await model.generateContent({
    contents: [{ role: "user", parts: [{ text: fullPrompt }] }],
    generationConfig: {
      // Thinking Mode の設定（APIバージョンによって異なる場合があります）
      temperature: 1,  // Thinking Mode推奨の設定
    },
  });

  const response = result.response;
  const text = response.text();
  
  // 思考プロセスと最終回答を分割（実装はAPIバージョンに合わせて調整）
  const thinkingMatch = text.match(/<think>([\s\S]*?)<\/think>/);
  const thinkingProcess = thinkingMatch ? thinkingMatch[1].trim() : "";
  const finalAnswer = thinkingProcess
    ? text.replace(/<think>[\s\S]*?<\/think>/, "").trim()
    : text;

  return { thinkingProcess, finalAnswer };
}
```

## コスト最適化と本番運用のベストプラクティス

### トークン使用量の監視とコスト計算

Gemini 2.5 Proは高性能ですが、大量のトークンを処理するとコストが積み上がります。本番環境では使用量を常に監視することが重要です。

```typescript
// src/lib/cost-tracker.ts

// 2026年4月時点の参考価格（USD）
// 実際の価格はGoogle AI Studioで確認してください
const COST_PER_TOKEN = {
  "gemini-2.5-pro": {
    input: 0.000003,   // $3 per million tokens (参考値)
    output: 0.000015,  // $15 per million tokens (参考値)
  },
  "gemini-2.5-flash": {
    input: 0.0000003,  // $0.30 per million tokens (参考値)
    output: 0.0000015, // $1.50 per million tokens (参考値)
  },
};

interface UsageMetrics {
  model: "gemini-2.5-pro" | "gemini-2.5-flash";
  inputTokens: number;
  outputTokens: number;
}

export function calculateCost(usage: UsageMetrics): {
  inputCost: number;
  outputCost: number;
  totalCost: number;
} {
  const rates = COST_PER_TOKEN[usage.model];
  const inputCost = usage.inputTokens * rates.input;
  const outputCost = usage.outputTokens * rates.output;
  
  return {
    inputCost,
    outputCost,
    totalCost: inputCost + outputCost,
  };
}

export function estimateTokenCount(text: string): number {
  // 大まかな推定：英語は約4文字/トークン、日本語は約1.5文字/トークン
  const japaneseCharCount = (text.match(/[\u3000-\u9FFF]/g) || []).length;
  const otherCharCount = text.length - japaneseCharCount;
  return Math.ceil(japaneseCharCount / 1.5 + otherCharCount / 4);
}
```

### レート制限とリトライ戦略

```typescript
// src/lib/retry-handler.ts

interface RetryConfig {
  maxRetries: number;
  initialDelayMs: number;
  backoffMultiplier: number;
}

const DEFAULT_RETRY_CONFIG: RetryConfig = {
  maxRetries: 3,
  initialDelayMs: 1000,
  backoffMultiplier: 2,
};

/**
 * エクスポネンシャルバックオフでAPIコールをリトライする
 * Expected output: APIの正常なレスポンス、または最大リトライ後にエラーをスロー
 */
export async function withRetry<T>(
  apiCall: () => Promise<T>,
  config: RetryConfig = DEFAULT_RETRY_CONFIG
): Promise<T> {
  let lastError: Error;
  let delay = config.initialDelayMs;
  
  for (let attempt = 0; attempt <= config.maxRetries; attempt++) {
    try {
      return await apiCall();
    } catch (error) {
      lastError = error as Error;
      
      // レート制限エラーの場合はリトライ
      if (error instanceof Error && error.message.includes("429")) {
        console.log(`レート制限: ${delay}ms 後にリトライ（${attempt + 1}/${config.maxRetries}）`);
        await new Promise(resolve => setTimeout(resolve, delay));
        delay *= config.backoffMultiplier;
      } else {
        // その他のエラーはリトライしない
        throw error;
      }
    }
  }
  
  throw lastError!;
}
```

### キャッシュ戦略でコストを削減する

同じドキュメントへの繰り返しアクセスに対して、レスポンスをキャッシュすることでAPIコストを大幅に削減できます。

```typescript
// src/lib/response-cache.ts
import AsyncStorage from "@react-native-async-storage/async-storage";
import { createHash } from "crypto";

const CACHE_TTL_MS = 24 * 60 * 60 * 1000; // 24時間

function generateCacheKey(prompt: string): string {
  // プロンプトのハッシュをキャッシュキーとして使用
  return `gemini_cache_${createHash("sha256").update(prompt).digest("hex").substring(0, 16)}`;
}

export async function getCachedResponse(prompt: string): Promise<string | null> {
  const key = generateCacheKey(prompt);
  
  try {
    const cached = await AsyncStorage.getItem(key);
    if (!cached) return null;
    
    const { response, timestamp } = JSON.parse(cached);
    
    // TTL チェック
    if (Date.now() - timestamp > CACHE_TTL_MS) {
      await AsyncStorage.removeItem(key);
      return null;
    }
    
    return response;
  } catch {
    return null;
  }
}

export async function setCachedResponse(prompt: string, response: string): Promise<void> {
  const key = generateCacheKey(prompt);
  
  await AsyncStorage.setItem(
    key,
    JSON.stringify({ response, timestamp: Date.now() })
  );
}
```

## よくあるエラーと対処法

長文コンテキストや高度な機能を使う際には、いくつかの特有のエラーに遭遇することがあります。

**コンテキスト長の超過エラー**: 100万トークン制限を超えた場合は `CONTEXT_LENGTH_EXCEEDED` エラーが返ります。ドキュメントを分割して複数回に分けてAPIを呼び出すか、要約→詳細分析の二段階アプローチを採用してください。

**レート制限エラー（429）**: APIの呼び出し頻度が制限を超えた場合に発生します。上記の `withRetry` 実装のようにエクスポネンシャルバックオフを実装し、ユーザーには「分析中」の状態を適切に表示してください。

**JSON パースエラー**: `responseMimeType: "application/json"` を指定しても、複雑な構造では稀にパースエラーが発生します。`try/catch` での対処に加え、フォールバックとして文字列レスポンスをそのまま表示する実装を必ず入れてください。

**タイムアウト**: 大量のドキュメント処理では応答に30秒以上かかる場合があります。クライアント側のタイムアウトを60〜120秒に設定し、ストリーミングAPIを活用して途中経過を表示することでUXを改善してください。

## FAQ

**Q. Gemini 2.5 Pro と Flash はどう使い分けるべきですか？**

A. シンプルな質問応答・短いドキュメントのサマリー・チャットボットの日常会話にはFlashを使い、法的・医療・財務の専門的判断、10,000文字を超えるドキュメントの分析、Thinking Modeが必要な複雑な推論タスクにはProを使うのが最適です。この記事で紹介した `analyzeWithRouter` のようなルーター実装を使えば、コストを大幅に削減できます。

**Q. 100万トークンのコンテキストを実際に使い切ることはありますか？**

A. 通常のアプリ開発では、一般的な契約書（20〜30ページ）は約15,000〜25,000トークン、コードベース（10,000行程度）は約40,000〜60,000トークンです。100万トークン全体を使うのは学術研究や大規模なリーガルテックなど特殊なケースに限られますが、50,000〜200,000トークン規模のタスクで大きな差が出ます。

**Q. 日本語のドキュメントでも精度は問題ありませんか？**

A. はい、Gemini 2.5 Proは日本語の理解・生成精度が非常に高く、英語と遜色のない品質で動作します。ただし、日本語は英語に比べてトークン消費量が多い（1文字が0.5〜1トークン）ため、コストの見積もり時は注意が必要です。

**Q. Thinking Mode は常に有効にするべきですか？**

A. いいえ。Thinking Mode は回答生成に時間がかかり、トークン消費量も増加します。法的判断・医療アドバイス・複雑な数学的問題など、「なぜその答えになったか」の透明性が重要なタスクに限定して使うことを推奨します。一般的なQ&Aや情報検索タスクではFlashをそのまま使った方が速く安く済みます。
