Rork AIを使ってアプリを生成する際、プロンプトが期待通りの結果を生み出さないことがあります。「こんなアプリを作りたい」という要望があっても、プロンプトの書き方次第で生成されるアプリは大きく変わります。本記事では、効果的なプロンプト作成のコツと、よくある悩みへの対処法をQ&A形式で解説します。

## プロンプト作成の基本原則

Rork AIで高品質なアプリを生成するには、以下の3つの要素を意識します。

1. **具体性**: 曖昧な表現ではなく、具体的な機能・デザイン要件を明記
2. **段階性**: 複雑な要求は段階的に分割
3. **検証性**: 生成後に実際に動作するか確認し、改善を反復

## よくある質問（FAQ）

### Q1: プロンプトを書いても、まったく違う機能のアプリが生成されました。何が悪いのでしょうか？

**A1: プロンプトが曖昧すぎます。「AI」「検索」「SNS」のような単語だけでは、AIが何を優先すべきか判断できません。**

具体的なプロンプトの書き方：

```
❌ 悪い例：
「タスク管理アプリを作って」

✅ 良い例：
「以下の機能を持つタスク管理アプリを作成してください：
- タスクをカテゴリ分け（仕事、プライベート、買い物）
- 期限日時を設定できる
- 完了したタスクはチェックマークで表示
- カテゴリごとに進捗ゲージを表示
- ダークモード対応
- ホーム画面は『今日のタスク』を優先表示」
```

重要なポイント：

- **機能リスト化**: 「～ができる」という要素を箇条書きで列挙
- **優先順位明示**: 「メインは～、補助として～」と優先順位を示す
- **デザイン指定**: 「カード型レイアウト」「シンプルで落ち着いた配色」など
- **ユースケース記述**: 「毎朝、昨日の完了数をチェックする」といった使い方を書く

### Q2: 生成されたアプリのデザインが安っぽく見えます。どうしたら改善できますか？

**A2: デザイン要件をプロンプトに明示する必要があります。AIは「デザイン指定がない=シンプルなものを」と解釈します。**

デザイン指定プロンプトの例：

```
「以下のデザイン要件でアプリを生成してください：
- カラースキーム：深い紺色（#1a2847）とコーラルピンク（#ff6b6b）
- フォント：見出しは San Francisco Bold、本文は Regular
- 間隔：大きめの余白（padding 16px 以上）
- アイコン：Heroicons を使用
- シャドウ効果：iOS ライクな淡いシャドウ
- ボタン：コーナー丸め 12px、背景は主要色、テキスト白」
```

参考になるデザイン指定例：

- 「Material Design 3 のガイドラインに従う」
- 「Apple iOS 18 のヒューマンインターフェースガイドラインに従う」
- 「TailwindCSS の色体系（slate, blue, indigo など）を使用」

### Q3: アプリは生成されたけれど、内部の計算ロジックが間違っている気がします。

**A3: ロジック要件を詳細に記述する必要があります。特にビジネスロジック（計算・判定）は数式やアルゴリズムで明示しましょう。**

ロジック指定のプロンプト例：

```
「以下の計算ロジックで家計簿アプリを生成してください：

【支出カテゴリごとの集計】
- カテゴリ（食費、交通、エンタメ）ごとに月間支出を合計

【月間予算対比】
- 実績 / 予算 × 100 でパーセンテージ表示
- 100% を超えた場合は警告表示（赤）

【貯蓄額計算】
- 毎月の貯蓄額 = 月収 - 月支出
- 年間貯蓄率 = 年間貯蓄額 / 年間収入 × 100」
```

### Q4: ボタンをタップしてもアクションが起動しません。どう指定すればいいですか？

**A4: インタラクション要件を明確に指定しましょう。「タップしたらどうなるか」を具体的に書きます。**

インタラクション指定の例：

```
「タスク登録画面で『追加』ボタンをタップした時の動作：
1. 画面バリデーション：タスク名が空でないか確認
2. 空の場合：トースト通知『タスク名を入力してください』を表示（1秒後に消える）
3. 入力ある場合：
   - タスク情報をローカルストレージに保存
   - メイン画面に戻る
   - ホーム画面のタスク一覧を自動更新
4. 保存中は『追加』ボタンをdisabled状態にし、ローディングスピナー表示」
```

### Q5: APIと連携させたいけど、APIキーの処理はどう指定したらいいですか？

**A5: 詳細なAPI統合要件を記述し、セキュリティに配慮しましょう。**

API連携プロンプトの例：

```
「以下の設定でOpenWeather APIと連携してください：

【API呼び出し仕様】
- エンドポイント：https://api.openweathermap.org/data/2.5/weather
- メソッド：GET
- タイムアウト：10秒

【APIキー管理】
- APIキーは環境変数から取得
- .env ファイルに記述（コード内に直書きしない）
- APIキーを画面やログに出力しない

【レスポンス処理】
- 成功：main, temp, humidity を抽出して表示
- エラー：『都市が見つかりません』と表示
- ネットワークエラー：『接続に失敗しました』と表示」
```

### Q6: デバッグモードやテスト用の機能を追加したいのですが？

**A6: 開発用機能と本番用機能は別のプロンプトで指定するか、条件分岐で明示します。**

開発環境指定の例：

```
「以下の開発用機能を追加してください（本番環境では非表示）：

【デバッグパネル】
- 画面右下に隠しボタン（5回タップで表示）
- パネル内容：
  - 現在のアプリバージョン
  - ローカルストレージのデータ一覧（JSON形式で表示）
  - 最後のAPI呼び出し情報（URL, レスポンスステータス）

【ロギング】
- すべてのAPI呼び出しを console.log に出力
- ユーザーアクション（ボタンタップ、画面遷移）をログに記録
- エラー発生時はスタックトレースを出力

【環境判別】
- 環境変数で development / production を判別」
```

### Q7: 生成されたアプリは動くけど、動作がすごく遅いです。なぜですか？

**A7: パフォーマンス要件が未指定の場合、AIは動作確認できる程度の最小限の最適化しかしません。**

パフォーマンス指定の例：

```
「以下のパフォーマンス要件を満たすようにしてください：

【画面読み込み】
- ホーム画面の初期表示：1秒以内

【リスト表示】
- 100件のアイテムリストをスムーズにスクロール
- 仮想スクロール（virtualized list）を使用

【画像処理】
- ダウンロード時に圧縮（最大幅 800px）
- ローカルキャッシュして重複ダウンロード防止

【API呼び出し】
- ページネーション（1ページ20件）で分割取得
- キャッシング戦略：5分以内は同じデータを再取得しない」
```

### Q8: アプリのアイコンやスプラッシュスクリーンもAIに指定できますか？

**A8: はい。ロゴ/アイコン生成要件を記述できます。ただし詳細なビジュアルはデザインツール（Figmaなど）で後調整することをお勧めします。**

ビジュアル要素の指定例：

```
「以下のビジュアル要素を生成してください：

【アプリアイコン】
- 形状：正方形（角丸め）
- デザイン：シンプルな電卓のシンボル（+-×÷）
- 背景色：グラデーション（紺 → 水色）

【スプラッシュスクリーン】
- 背景：グラデーション（紺 → 深紫）
- 中央：アプリロゴ
- 下部：アプリ名（ロード中...）
- アニメーション：ロゴが2秒かけてフェードイン」
```

### Q9: 同じプロンプトで何度生成しても、毎回違うアプリになってしまいます。

**A9: AIの生成には確率的なばらつきがあります。要件を『さらに詳細に』『より制約的に』指定することで再現性が向上します。**

再現性向上のコツ：

```
❌ 曖昧（毎回変わる可能性が高い）：
「シンプルな買い物リストアプリ」

✅ 詳細（再現性向上）：
「買い物リストアプリ（以下に厳密に従う）
【機能要件】
1. ホーム画面：リスト（項目名、数量、チェックボックス）
2. 新規追加：『+』ボタン → 入力フォーム → 追加
3. 削除：左スワイプ → 『削除』確認 → 削除
4. データ永続化：AsyncStorage で保存
【UI要件】
- ホーム画面：リスト、ヘッダー『買い物リスト』、『+』ボタン
- 色：白背景、テキスト黒
- フォント：サンセリフ」
```

### Q10: プロンプトを何度も修正しながら、AIに『ここを直して』とお願いしていますが、効率的な方法はありますか？

**A10: 段階的な反復改善（イテラティブ・プロンプティング）が効果的です。以下の流れで進めましょう。**

効率的なイテラティブ改善フロー：

```
1. 【初期プロンプト】
   基本機能を明確に記述（1回目の生成）

2. 【評価フェーズ】
   「アプリは起動する？」「期待した機能は動く？」を確認

3. 【修正要求】
   問題を特定したら、修正内容を明確に：
   ❌「デザインをもっと良くして」
   ✅「ホームボタンを大きく（50x50px）、背景を濃い青（#1a3a52）に変更」

4. 【段階的拡張】
   基本機能が完成したら、一つずつ機能追加：
   初回：基本CRUD機能
   2回目：バリデーション
   3回目：デザイン調整
   4回目：パフォーマンス最適化

5. 【ドキュメント化】
   最終的な完成形プロンプトをテキストで保存
   次回の参考資料に
```

## よくある失敗パターンと対策

アプリ生成で失敗する主なパターンと対策：

- **失敗**: アプリが起動しない → **対策**: プロンプトに「Expo SDK 50+」など最小環境を明記
- **失敗**: 期待と違う機能 → **対策**: 箇条書きで機能リスト化、優先順位明示
- **失敗**: UIが崩れている → **対策**: 「iPhone 14, 15, iPad Pro で動作確認」と指定
- **失敗**: パフォーマンスが悪い → **対策**: キャッシング、ページネーション具体指定
- **失敗**: セキュリティ不備 → **対策**: APIキー、PII データの取扱い方法を明記

## まとめ

Rork AIで期待通りのアプリを生成するコツは、**具体性、段階性、検証性** の3点に尽きます。

1. **最初は基本機能のみ記述** → シンプルに動くものを作る
2. **デザイン要件を次に追加** → 見栄えを整える
3. **インタラクション・ロジック詳細化** → 細部の挙動を調整
4. **パフォーマンス・セキュリティ強化** → 本番対応にする
