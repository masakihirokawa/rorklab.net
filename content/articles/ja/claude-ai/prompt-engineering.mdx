## プロンプトエンジニアリングとは

プロンプトエンジニアリングは、Claude からより正確で高品質な回答を引き出すためにプロンプトを設計・改良する技術です。質問の言い回しをわずかに変えるだけで、まったく異なる結果が得られることがあります。Claude の能力と制限を理解することで、一貫して高品質な出力を生成するプロンプトを作成できます。

<Callout type="info">
優れたプロンプトエンジニアリングは、難解で複雑なプロンプトを作ることではありません。明確で具体的で構造化されたプロンプトを作り、Claude が正確に何が必要かを理解できるようにすることです。
</Callout>

## システムプロンプト

システムプロンプトは会話全体のコンテキストと動作を設定します。実際のクエリの前に処理され、Claude の回答をガイドする見えない指示として機能します。

### システムプロンプトの使用方法

```
システムプロンプト:
あなたはソフトウェアドキュメンテーション専門の技術ライターです。
常に明確な見出しと簡潔な説明で回答を構成してください。
複雑な概念を説明するために例を使用してください。
専門用語は最初に定義してから使用してください。
```

システムプロンプトは以下の場合に特に効果的です:
- 専門的なトーンやパーソナリティの確立
- アシスタントの役割や専門知識の定義
- 出力形式の要件の指定
- 制約やガイドラインの設定

Claude.ai ではプロジェクト設定でシステムプロンプトを設定できます。API では、リクエストの `system` パラメータを使用します。

## 少数ショット例 (コンテキスト内学習)

少数ショット例は、実際の問題を解く前に目的のパターンを示すことで、Claude に応答方法を教えます。この技法は長い説明よりも効果的です。

### 例: テキスト分類

```
感情分類の例を以下に示します:

入力: 「この製品は素晴らしい! 大好きです。」
分類: POSITIVE

入力: 「1週間で壊れました。とても失望しました。」
分類: NEGATIVE

入力: 「まあまあですね。特に特別なものではありませんが、機能します。」
分類: NEUTRAL

次のテキストを分類してください: 「カスタマーサービスは役に立たず、失礼でした。」
分類:
```

少数ショットプロンプトが効果的なのは、Claude が具体的な例からパターンを学ぶからです。特に以下の場合に役立ちます:
- タスクが特定のフォーマット要件を持つ場合
- 多くのアイテム全体で一貫したカテゴリ分けが必要な場合
- カスタムカテゴリやラベルを定義する必要がある場合
- 標準的な指示が目的の形式を生成していない場合

## 思考の連鎖 (チェーン・オブ・ソート)

思考の連鎖プロンプトは、Claude に最終答を提供する前にステップごとに推論を説明するよう求めます。このテクニックは、複雑な問題解決と推論タスクの精度を向上させます。

### 思考の連鎖なし

```
Q: 列車が時速 40 マイルで 120 マイルを移動します。旅行にかかる時間は?
A: 3時間
```

### 思考の連鎖あり

```
Q: 列車が時速 40 マイルで 120 マイルを移動します。旅行にかかる時間は?

ステップごとに考えてください:
1. わかっていることを特定する
2. 関連する公式を書く
3. 計算を実行する
4. 答えを述べる

A:
```

Claude は自然に各ステップを説明し、より正確で検証可能な結果をもたらします。これは以下の場合に非常に価値があります:
- 複雑な計算と推論
- 意思決定プロセス
- デバッグとトラブルシューティング
- 学問的または技術的な問題

<Callout type="tip">
思考の連鎖は「声に出して考える」とも言えます。Claude に推論を言語化するよう求めることで、エラーを見つけやすくなり、回答の信頼性が向上します。
</Callout>

## 構造化のための XML タグ

XML タグは複雑なプロンプトを整理し、リクエストのさまざまな部分を明確に分離するのに役立ちます。プロンプトを理解しやすく、解析しやすくするビジュアルデリミタとして機能します。

### XML タグを使った構造化リクエスト

```
<task>
ワイヤレスヘッドフォンの製品説明を書く
</task>

<requirements>
- 最大150文字
- ターゲットオーディエンス: プロフェッショナル
- バッテリー駆動時間と音質を強調する
- 主要な技術仕様を1つ含める
</requirements>

<tone>
プロフェッショナルだが親しみやすい
</tone>

<format>
見出しで始めて、その後2つのパラグラフを続ける
</format>
```

XML タグが効果的な理由:
- リクエストのさまざまなタイプの情報を明確に分離する
- Claude がリクエストの構造を簡単に識別できる
- 複雑なマルチパートリクエストに特に効果的
- 複数のプロンプトで使用する場合の一貫性を向上させる

## 出力フォーマット仕様

目的の出力形式を明示することで、結果の品質が大幅に向上します。Claude が正しい形式を推測することを期待する代わりに、求めるものを正確に指定してください。

### JSON 出力の指定

```
テック スタートアップのための 5 つのクリエイティブなビジネス名のアイデアを生成します。
このような構造の JSON として返します:

{
  "names": [
    {
      "name": "string",
      "reasoning": "string"
    }
  ]
}
```

### Markdown テーブルの指定

```
Python、JavaScript、Go を 5 つの次元にわたって比較してください。
言語を列、次元を行として markdown テーブルとしてフォーマットします。
```

### リストと箇条書きの指定

```
リモートワーカーのための 10 大生産性のヒントを列挙します。
簡潔な 1 行の説明付きの番号付きリストとしてフォーマットします。
各アイテムには実践的なアクションステップが含まれている必要があります。
```

## テクニックの組み合わせ: 完全な例

```
あなたはエキスパートな Python コードレビュアーです。あなたの役割は、バグを特定し、
改善を提案し、改善を明確に説明することです。

コードをレビューする方法は次のとおりです:

1. バグまたは論理エラーがないか確認する
2. パフォーマンスの改善を提案する
3. ベストプラクティスを推奨する
4. 各提案を簡単に説明する

レビューを次のようにフォーマットしてください:
- **見つかったバグ:** [問題の一覧]
- **パフォーマンス:** [最適化の提案]
- **ベストプラクティス:** [推奨事項]

このコードをレビューしてください:
[コード スニペットをここに]

レビューを提供する前に、潜在的な問題をステップごとに考えてください。
```

この例では以下が組み合わされています:
- システムプロンプト (役割の定義)
- 少数ショット例 (フォーマットのデモンストレーション)
- 出力仕様 (構造化形式)
- 思考の連鎖 (段階的な思考)

## テストと反復

プロンプトエンジニアリングはワンショットのプロセスではありません。最良のプロンプトはテストを通じて改善されます:

1. **シンプルに始める** — 明確で基本的なリクエストから始めます
2. **バリエーションをテストする** — さまざまな言い回しと構造を試してみます
3. **品質を評価する** — 出力はニーズを満たしていますか?
4. **反復する** — 結果に基づいて改善します
5. **ドキュメント化する** — 再利用するために最適なプロンプトを保存します

一般的なタスクに対して機能するプロンプトのコレクションを保持してください。時間がたつにつれて、Claude で何が機能するかについての直感が発展します。

## 避けるべき一般的な落とし穴

- **曖昧すぎる**: 「AI について何か書いて」は「AI 入門ガイド (300 語) を書いて」ほど効果的ではありません
- **複雑すぎる**: 長いプロンプトが常に良いわけではありません。明確さが冗長さに勝ります
- **コンテキストを想定する**: Claude が特定の用語やバックグラウンドを知っていると想定しないでください
- **フォーマットが一貫しない**: 構造化された出力を求める場合、形式について正確に指定します
- **トーンを無視する**: トーンは重要です。「フォーマル」、「フレンドリー」、「テクニカル」など指定してください
