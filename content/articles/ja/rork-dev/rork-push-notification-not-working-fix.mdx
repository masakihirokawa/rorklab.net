Rork で作ったアプリを公開したあと、「プッシュ通知が届かない」「通知が表示されない」「テスト送信しても何も起きない」——こうした問題に直面した経験はないでしょうか。プッシュ通知はモバイルアプリのユーザーエンゲージメントを支える重要な機能である一方、iOS と Android の違い、証明書・権限まわりの設定ミスなど、躓くポイントが非常に多い機能でもあります。

本記事では、Rork アプリで発生しやすいプッシュ通知トラブルのパターンを整理し、原因の特定から解決までのステップを丁寧にご説明します。

## プッシュ通知が届かない症状のパターン

まず、どんな症状が起きているかを正確に把握することが大切です。以下のパターンに分類して考えてみましょう。

- **パターンA**: アプリが完全にフォアグラウンドのときだけ通知が届く
- **パターンB**: シミュレーター/エミュレーターでは届くが実機で届かない
- **パターンC**: iOS では届くが Android では届かない（またはその逆）
- **パターンD**: リリース版（本番環境）では届かないが開発環境では届く
- **パターンE**: 送信リクエスト自体は成功しているが端末に届かない
- **パターンF**: 通知許可ダイアログが表示されないまたはユーザーが拒否した

症状のパターンが特定できると、原因の絞り込みが大幅に楽になります。ログを確認しながら一つずつ確認していきましょう。

## 主な原因の分析

### 原因1: デバイストークンが取得できていない

プッシュ通知を送るためには、まずデバイスから **Expo Push Token** または **APNs/FCM ネイティブトークン** を取得する必要があります。このトークン取得に失敗していると、どれだけ送信処理を正しく書いても通知は届きません。

よくある原因:

- `expo-notifications` パッケージのバージョン不整合
- 権限リクエスト（`requestPermissionsAsync`）の呼び出しを忘れている
- iOS シミュレーターでトークン取得を試みている（シミュレーターは APNs 非対応）
- `projectId` が Expo のダッシュボード設定と一致していない

### 原因2: iOS APNs 証明書・プロビジョニングプロファイルの不備

iOS でプッシュ通知を送るには、**APNs（Apple Push Notification service）** の証明書設定が正しく行われている必要があります。開発用と本番用で証明書が分かれており、混同すると通知が届きません。

- 開発環境: APNs Sandbox 証明書
- 本番環境（App Store / TestFlight）: APNs Production 証明書

EAS Build を使っている場合は `eas.json` の credentials 設定が影響します。

### 原因3: Android FCM の設定ミス

Android では **FCM（Firebase Cloud Messaging）** を経由して通知が配信されます。`google-services.json` ファイルが最新でなかったり、Firebase プロジェクトの設定が誤っていたりすると、通知が届きません。

### 原因4: app.json / app.config.js の設定漏れ

`app.json` または `app.config.js` において、プッシュ通知に必要な設定が漏れているケースも多く見られます。特に `experienceId` や `projectId` の設定は見落としやすいポイントです。

### 原因5: バックグラウンド通知の設定不足

アプリがバックグラウンド状態のときに通知を受け取るには、iOS では `UIBackgroundModes` の設定が必要です。この設定が抜けていると、フォアグラウンドでのみ通知が届くという挙動になります。

## 解決手順（Step by Step）

### Step 1: デバイストークンの取得を確認する

まず、トークンが正しく取得できているか確認します。以下のコードをアプリに組み込み、コンソールログを確認してください。

```typescript
import * as Notifications from 'expo-notifications';
import * as Device from 'expo-device';
import Constants from 'expo-constants';

async function registerForPushNotificationsAsync() {
  if (!Device.isDevice) {
    console.warn('⚠️ 実機が必要です。シミュレーターではプッシュ通知トークンを取得できません。');
    return null;
  }

  // 通知権限のリクエスト
  const { status: existingStatus } = await Notifications.getPermissionsAsync();
  let finalStatus = existingStatus;

  if (existingStatus !== 'granted') {
    const { status } = await Notifications.requestPermissionsAsync();
    finalStatus = status;
  }

  if (finalStatus !== 'granted') {
    console.error('❌ 通知権限が拒否されています');
    return null;
  }

  // Expo Push Token の取得
  try {
    const projectId = Constants.expoConfig?.extra?.eas?.projectId;
    if (!projectId) {
      console.error('❌ projectId が設定されていません（app.json を確認）');
      return null;
    }
    const tokenData = await Notifications.getExpoPushTokenAsync({ projectId });
    console.log('✅ Expo Push Token:', tokenData.data);
    return tokenData.data;
  } catch (error) {
    console.error('❌ トークン取得エラー:', error);
    return null;
  }
}
```

**確認ポイント**: コンソールに `ExponentPushToken[xxxx...]` の形式でトークンが表示されれば成功です。

### Step 2: app.json の設定を確認する

`app.json` にプッシュ通知関連の設定が含まれているか確認します。

```json
{
  "expo": {
    "name": "YourApp",
    "slug": "your-app",
    "extra": {
      "eas": {
        "projectId": "YOUR_EAS_PROJECT_ID"
      }
    },
    "ios": {
      "bundleIdentifier": "com.yourcompany.yourapp",
      "infoPlist": {
        "UIBackgroundModes": ["remote-notification"]
      }
    },
    "android": {
      "package": "com.yourcompany.yourapp",
      "googleServicesFile": "./google-services.json"
    },
    "plugins": [
      [
        "expo-notifications",
        {
          "icon": "./assets/notification-icon.png",
          "color": "#ffffff",
          "defaultChannel": "default"
        }
      ]
    ]
  }
}
```

### Step 3: iOS APNs 証明書を再設定する

EAS Build を利用している場合は、以下のコマンドで証明書の状態を確認・再設定できます。

```bash
# 証明書の状態確認
eas credentials

# 本番用 APNs キーの再設定（対話式メニューが起動します）
eas credentials --platform ios

# プロジェクトIDの確認
eas project:info
```

証明書の有効期限が切れていたり、Bundle ID と証明書が一致していない場合は再発行が必要です。Apple Developer ポータルで APNs Key（.p8 形式）を新たに作成し、EAS に登録し直してください。

### Step 4: Android の google-services.json を確認する

Firebase Console から最新の `google-services.json` をダウンロードし、プロジェクトルートに配置します。

```bash
# google-services.json の配置確認
ls -la ./google-services.json

# package name が app.json の android.package と一致しているか確認
cat google-services.json | python3 -c "import sys, json; data=json.load(sys.stdin); print([c['client_info']['android_client_info']['package_name'] for c in data['client']])"
```

package name が `app.json` の `android.package` と完全一致していない場合、FCM のトークン取得・通知配信が失敗します。

### Step 5: Expo Push Tool でテスト送信する

トークンが取得できたら、[Expo の Push Notification Tool](https://expo.dev/notifications) を使って手動テスト送信を試みます。

```bash
# curl でのテスト送信（ターミナルから直接確認できます）
curl -H "Content-Type: application/json" -X POST "https://exp.host/--/api/v2/push/send" \
  -d '{
    "to": "ExponentPushToken[YOUR_TOKEN_HERE]",
    "title": "テスト通知",
    "body": "この通知が届けば設定は正常です",
    "data": {"test": true}
  }'
```

レスポンスに `"status": "ok"` が返ってくれば、Expo サーバーまでは通知が届いています。それでも端末に届かない場合は、APNs/FCM の設定を再確認してください。

### Step 6: Android の通知チャンネルを設定する

Android 8.0（API 26）以降、**通知チャンネル** の設定が必須です。チャンネルが設定されていないと、通知が届いても表示されない場合があります。

```typescript
import * as Notifications from 'expo-notifications';
import { Platform } from 'react-native';

// アプリ起動時に通知ハンドラとチャンネルを設定する
Notifications.setNotificationHandler({
  handleNotification: async () => ({
    shouldShowAlert: true,
    shouldPlaySound: true,
    shouldSetBadge: true,
  }),
});

async function setupAndroidChannel() {
  if (Platform.OS === 'android') {
    await Notifications.setNotificationChannelAsync('default', {
      name: 'デフォルト通知',
      importance: Notifications.AndroidImportance.MAX,
      vibrationPattern: [0, 250, 250, 250],
      lightColor: '#FF231F7C',
      sound: 'default',
    });
    console.log('✅ Android 通知チャンネルを設定しました');
  }
}
```

## 解決できたかの確認方法

以下の手順で、プッシュ通知が正常に動作していることを確認します。

**iOS の確認手順**:
1. 実機（TestFlight ビルドまたは開発ビルド）でアプリを起動する
2. 通知権限ダイアログが表示され「許可」をタップする
3. コンソールログで `ExponentPushToken[...]` が表示されることを確認する
4. Expo Push Tool でテスト送信し、端末に通知が届くことを確認する
5. アプリをバックグラウンドにした状態でも通知が届くことを確認する

**Android の確認手順**:
1. 実機または Android エミュレーター（API 33 以上推奨）でアプリを起動する
2. 設定 → アプリ → [アプリ名] → 通知 から、通知が許可されているか確認する
3. Expo Push Tool でテスト送信し、通知バーに通知が届くことを確認する
4. 通知チャンネルが「デフォルト通知」として表示されていることを確認する

**Expo Push API のレスポンスコードの見方**:

- `"status": "ok"` — 送信成功
- `"status": "error"`, `"details.error": "DeviceNotRegistered"` — トークンが無効（再取得が必要）
- `"status": "error"`, `"details.error": "InvalidCredentials"` — APNs/FCM 証明書の問題

## 予防策（再発防止のベストプラクティス）

一度解決できたとしても、同じ問題が再発しないよう、以下の対策を取り入れておくことをおすすめします。

**1. トークン再登録ロジックを実装する**

APNs/FCM はデバイストークンを定期的に更新することがあります。アプリ起動時に毎回トークンを取得し、サーバーのトークンと比較して変更があれば更新するロジックを組み込んでおきましょう。

```typescript
// アプリ起動時のトークン同期
useEffect(() => {
  const syncToken = async () => {
    const newToken = await registerForPushNotificationsAsync();
    if (newToken && newToken !== storedToken) {
      await updateTokenOnServer(newToken); // サーバーにトークンを保存
      setStoredToken(newToken);
    }
  };
  syncToken();
}, []);
```

**2. 通知権限の状態を定期的に確認する**

ユーザーが設定からアプリの通知を無効にした場合にも対応できるよう、アプリのフォアグラウンド復帰時に権限状態を確認するロジックを追加しましょう。[プッシュ通知の実装ガイド](/articles/rork-dev/rork-push-notifications)では通知許可フローの詳細を解説しています。

**3. エラーログを Sentry で監視する**

通知関連のエラーを自動的に検出・通知するために、エラー監視ツールを導入しておくと安心です。トークン取得失敗や送信エラーを catch した際に Sentry に送信する仕組みを整えましょう。

**4. 証明書の有効期限をカレンダーに登録する**

APNs 証明書（p12 形式）は1年間で失効します。APNs Key（p8 形式）は失効しませんが、Apple Developer アカウントの更新に注意が必要です。証明書の有効期限を事前にカレンダーに登録し、失効による通知停止を防ぎましょう。

## FAQ

**Q1. シミュレーターでプッシュ通知のテストをしたいのですが、できますか？**

iOS シミュレーターは APNs に非対応のため、実際のプッシュ通知は受信できません。ただし、`expo-notifications` の `scheduleNotificationAsync` を使ったローカル通知は動作します。プッシュ通知のテストは必ず実機で行ってください。Android エミュレーター（Google Play Services 搭載）はプッシュ通知のテストが可能です。詳しくは [Expo ローカル通知の完全ガイド](/articles/rork-dev/rork-local-notifications-expo-complete-guide)もご参照ください。

**Q2. 開発ビルドでは届くのに TestFlight ビルドで届かない場合はどうすればよいですか？**

TestFlight は本番環境の APNs を使用します。EAS の credentials 設定で本番用の APNs Key（Distribution 証明書）が正しく設定されているか確認してください。`eas credentials` コマンドで確認・更新が可能です。また、`eas.json` の `production` プロファイルに `"credentialsSource": "remote"` が設定されているかも確認しましょう。

**Q3. 通知は届いているのに内容が表示されない場合は？**

Android の場合、通知チャンネルの `importance` が `Notifications.AndroidImportance.LOW` や `NONE` になっている可能性があります。`MAX` または `HIGH` に設定してください。また、端末の「サイレントモード」や「おやすみモード」がオンになっていると通知音・ポップアップが抑制されます。デバイスの設定もご確認ください。

**Q4. 大量のユーザーに一括送信したときだけ通知が届かないことがあります**

Expo Push API には1リクエストあたり100件のトークン制限があります。それ以上のユーザーへ送信する場合は、バッチ分割と再送ロジックが必要です。詳しい実装方法は [プッシュ通知の高度なセグメンテーション自動化ガイド](/articles/rork-dev/rork-push-notification-advanced-segmentation-automation)で解説しています。

**Q5. `DeviceNotRegistered` エラーが頻繁に出る場合は？**

アプリをアンインストールしたユーザーのトークンや、長期間使用されていないトークンを送信先リストから定期的に削除する仕組みが必要です。送信後に `DeviceNotRegistered` エラーが返ってきたトークンは即座にデータベースから削除するようにしましょう。

## まとめ

Rork アプリでプッシュ通知が届かないトラブルは、原因を一つひとつ切り分けていくことで必ず解決できます。本記事で解説した手順を要約すると:

1. デバイストークンが正しく取得できているか確認する
2. `app.json` の設定（`projectId`・`UIBackgroundModes`・`googleServicesFile`）を見直す
3. iOS は APNs 証明書、Android は `google-services.json` の package name を確認する
4. Expo Push Tool で手動テスト送信して原因の範囲を絞り込む
5. Android は通知チャンネルの設定と importance レベルを確認する

一度正常に動作する環境を整えたら、トークン再登録ロジックや Sentry によるエラー監視を導入して継続的に安定運用できる仕組みを作っていきましょう。
