## はじめに — なぜユーザー行動分析が必要なのか

アプリをリリースした後、「ダウンロード数は増えているのに、なぜか継続率が低い」「どの画面でユーザーが離脱しているのかわからない」といった課題に直面することは少なくありません。こうした問題を解決するために不可欠なのが、ユーザー行動分析です。

Firebase Analyticsは、Googleが提供する無料の分析ツールで、アプリ内のユーザー行動をリアルタイムに追跡できます。さらにFirebase Remote Configと組み合わせることで、A/Bテストを実施し、UIの改善やコンバージョン率の最適化をデータに基づいて進められます。

この記事では、Rork Maxで構築したアプリにFirebase Analyticsを統合し、カスタムイベントの設計からA/Bテストの実装まで、ステップバイステップで解説します。

## 前提知識と環境準備

この記事を進めるにあたって、以下の環境が必要です。

- **Rork Max**のアカウント（アプリが作成済みであること）
- **Firebase**のプロジェクト（[Firebase Console](https://console.firebase.google.com/)で作成可能）
- **Node.js** 18以上
- React Native / Expoの基本知識

Rork Maxはネイティブモジュールへのアクセスが可能なため、Firebase SDKをフルに活用できます。まだFirebaseプロジェクトを作成していない場合は、Firebase Consoleから新しいプロジェクトを作成し、iOSおよびAndroidアプリを登録しておきましょう。

## Firebase Analytics のセットアップ

### パッケージのインストール

Rork Maxのプロジェクトで、以下のパッケージをインストールします。

```bash
# Firebase Analytics と Remote Config をインストール
npm install @react-native-firebase/app @react-native-firebase/analytics @react-native-firebase/remote-config
```

### 初期化コードの記述

アプリのエントリーポイントで Firebase を初期化します。

```typescript
// app/_layout.tsx
import analytics from '@react-native-firebase/analytics';
import remoteConfig from '@react-native-firebase/remote-config';
import { useEffect } from 'react';

export default function RootLayout() {
  useEffect(() => {
    const initFirebase = async () => {
      // Remote Config のデフォルト値を設定
      await remoteConfig().setDefaults({
        onboarding_variant: 'control',
        cta_button_color: '#007AFF',
        show_premium_banner: false,
      });

      // Remote Config を取得（キャッシュは5分）
      await remoteConfig().setConfigSettings({
        minimumFetchIntervalMillis: 300000,
      });
      await remoteConfig().fetchAndActivate();

      // Analytics のセッション開始をログ
      await analytics().logEvent('app_session_start', {
        timestamp: new Date().toISOString(),
      });
    };

    initFirebase();
  }, []);

  // ... レイアウトの返却
}
```

上記のコードで、アプリ起動時にRemote Configのデフォルト値が設定され、最新の設定値がサーバーから取得されます。同時に、セッション開始イベントがAnalyticsに送信されます。

## カスタムイベントの設計と実装

Firebase Analyticsの真価は、アプリ固有のカスタムイベントを設計することで発揮されます。以下は、実際のアプリでよく使われるイベント設計の例です。

### イベント設計のベストプラクティス

```typescript
// utils/analytics.ts — Analytics ヘルパー関数
import analytics from '@react-native-firebase/analytics';

// 画面遷移を追跡
export const trackScreenView = async (screenName: string) => {
  await analytics().logScreenView({
    screen_name: screenName,
    screen_class: screenName,
  });
};

// ボタンタップを追跡
export const trackButtonTap = async (
  buttonName: string,
  screenName: string,
  additionalParams?: Record<string, string>
) => {
  await analytics().logEvent('button_tap', {
    button_name: buttonName,
    screen_name: screenName,
    ...additionalParams,
  });
};

// コンバージョンイベント（課金・登録など）
export const trackConversion = async (
  conversionType: string,
  value?: number,
  currency?: string
) => {
  await analytics().logEvent('conversion', {
    conversion_type: conversionType,
    value: value ?? 0,
    currency: currency ?? 'JPY',
  });
};

// ファネル追跡（オンボーディングの各ステップなど）
export const trackFunnelStep = async (
  funnelName: string,
  stepNumber: number,
  stepName: string
) => {
  await analytics().logEvent('funnel_step', {
    funnel_name: funnelName,
    step_number: stepNumber,
    step_name: stepName,
  });
};
```

### 実装例 — オンボーディングファネルの追跡

```typescript
// screens/Onboarding.tsx
import { trackFunnelStep, trackConversion } from '@/utils/analytics';

export default function OnboardingScreen() {
  const handleStepComplete = async (step: number) => {
    // 各ステップの完了を記録
    await trackFunnelStep('onboarding', step, `step_${step}_complete`);
  };

  const handleOnboardingFinish = async () => {
    // オンボーディング完了をコンバージョンとして記録
    await trackConversion('onboarding_complete');
  };

  // ... UIコンポーネント
}
```

期待する結果として、Firebase Consoleの「イベント」タブで `funnel_step` イベントが記録され、各ステップの完了率をファネルレポートとして可視化できるようになります。

## A/Bテストの設計と実装

### Remote Config でバリアントを管理する

Firebase Remote Configを使うと、サーバー側からアプリの表示内容や動作を変更できます。これをA/Bテストに活用しましょう。

```typescript
// hooks/useABTest.ts — A/Bテスト用カスタムフック
import { useState, useEffect } from 'react';
import remoteConfig from '@react-native-firebase/remote-config';
import analytics from '@react-native-firebase/analytics';

interface ABTestConfig {
  parameterKey: string;   // Remote Config のキー
  experimentName: string; // 実験名（Analyticsログ用）
}

export function useABTest({ parameterKey, experimentName }: ABTestConfig) {
  const [variant, setVariant] = useState<string>('control');

  useEffect(() => {
    const fetchVariant = async () => {
      // Remote Config から現在のバリアントを取得
      const value = remoteConfig().getValue(parameterKey);
      const currentVariant = value.asString() || 'control';
      setVariant(currentVariant);

      // どのバリアントが表示されたか Analytics に記録
      await analytics().logEvent('ab_test_exposure', {
        experiment_name: experimentName,
        variant: currentVariant,
      });
    };

    fetchVariant();
  }, [parameterKey, experimentName]);

  return variant;
}
```

### 実装例 — CTAボタンのA/Bテスト

```typescript
// components/CTAButton.tsx
import { TouchableOpacity, Text, StyleSheet } from 'react-native';
import { useABTest } from '@/hooks/useABTest';
import { trackButtonTap } from '@/utils/analytics';

export default function CTAButton() {
  // A/Bテスト: ボタンの文言を切り替え
  const ctaVariant = useABTest({
    parameterKey: 'cta_button_text',
    experimentName: 'cta_text_experiment_v1',
  });

  const buttonText =
    ctaVariant === 'variant_a'
      ? '今すぐ無料で始める'
      : ctaVariant === 'variant_b'
        ? 'アプリを作ってみる'
        : '始めましょう'; // control

  const handlePress = async () => {
    await trackButtonTap('cta_main', 'home', {
      variant: ctaVariant,
    });
    // ... ナビゲーション処理
  };

  return (
    <TouchableOpacity style={styles.button} onPress={handlePress}>
      <Text style={styles.text}>{buttonText}</Text>
    </TouchableOpacity>
  );
}

const styles = StyleSheet.create({
  button: {
    backgroundColor: '#007AFF',
    paddingVertical: 16,
    paddingHorizontal: 32,
    borderRadius: 12,
    alignItems: 'center',
  },
  text: {
    color: '#FFFFFF',
    fontSize: 18,
    fontWeight: '700',
  },
});
```

Firebase ConsoleのA/B Testingセクションで実験を作成し、`cta_button_text`パラメータに `control`、`variant_a`、`variant_b` の3つのバリアントを設定すれば、ユーザーごとに異なるCTAテキストが表示され、コンバージョン率の違いを測定できます。

## データの可視化と分析のポイント

Firebase Analyticsのデータは、Firebase Consoleだけでなく、BigQueryにエクスポートして高度な分析を行うこともできます。

### Firebase Console で確認すべき指標

まず確認すべき基本指標は以下のとおりです。

- **DAU / MAU比率**: 日次アクティブユーザーを月次で割った値。0.2以上が健全な水準です
- **セッション継続時間**: ユーザーが1回のセッションでアプリを使う平均時間
- **イベント完了率**: カスタムイベントのファネルで、各ステップの完了率を追跡
- **リテンション率**: 1日後、7日後、30日後にアプリに戻ってくるユーザーの割合

### BigQuery エクスポートの活用

より詳細な分析が必要な場合は、Firebase ConsoleからBigQueryへのエクスポートを有効にします。SQLでユーザー行動を直接クエリでき、たとえば「オンボーディングを完了したユーザーの30日リテンション率」を算出するといった分析が可能です。

```sql
-- オンボーディング完了ユーザーの7日リテンション率
SELECT
  COUNT(DISTINCT CASE WHEN retention_day = 7 THEN user_pseudo_id END)
    / COUNT(DISTINCT user_pseudo_id) AS retention_7d
FROM (
  SELECT
    user_pseudo_id,
    DATE_DIFF(event_date, first_open_date, DAY) AS retention_day
  FROM `your_project.analytics_events`
  WHERE event_name = 'session_start'
    AND user_pseudo_id IN (
      SELECT user_pseudo_id
      FROM `your_project.analytics_events`
      WHERE event_name = 'conversion'
        AND event_params.value.string_value = 'onboarding_complete'
    )
)
```

## よくあるエラーと対処法

Firebase Analyticsを導入する際に起こりやすいトラブルと対処法をまとめました。

### イベントが Firebase Console に表示されない

イベント送信後、Firebase Consoleに反映されるまで最大24時間かかることがあります。リアルタイムで確認したい場合は、Firebase Consoleの **DebugView** を使用してください。

```bash
# iOSシミュレータで DebugView を有効にする
# Xcodeのスキーム設定で以下の引数を追加
-FIRAnalyticsDebugEnabled
```

### Remote Config の値が更新されない

Remote Configにはキャッシュ機構があり、デフォルトでは12時間キャッシュされます。開発中は `minimumFetchIntervalMillis` を短く設定しましょう（先述のコード例では5分に設定済み）。本番環境では、頻繁なフェッチはパフォーマンスに影響するため、3,600,000ミリ秒（1時間）程度が推奨です。

### A/Bテストのサンプルサイズが不足する

統計的に有意な結果を得るには、一般的に各バリアントあたり最低1,000〜5,000のイベント数が必要です。DAUが少ないアプリでは、テスト期間を2〜4週間に設定するか、テストするバリアント数を減らして集約することをおすすめします。

## FAQ

### Q1. Firebase Analytics は無料で使えますか？

はい、Firebase Analyticsは完全無料です。イベント数やユーザー数に制限はありません。ただし、BigQueryへのエクスポートを利用する場合は、BigQueryの利用料金が別途かかります（無料枠あり）。

### Q2. A/Bテストの実施期間はどのくらいが適切ですか？

一般的には2〜4週間が推奨されます。短すぎると統計的な有意性が得られず、長すぎると市場環境の変化が結果に影響します。Firebase ConsoleのA/B Testingでは、十分なデータが集まった時点で自動的に「勝者バリアント」を提案してくれます。

### Q3. Rork（無印版）でも Firebase Analytics は使えますか？

Rorkの無印版はReact Native / Expoベースですが、Expo GoではネイティブモジュールであるFirebase SDKが動作しません。`expo-firebase-analytics`パッケージを使用するか、EAS Buildでカスタムビルドを作成する必要があります。Rork Maxであれば、ネイティブモジュールに直接アクセスできるため、Firebase SDKをフルに活用できます。

### Q4. どのイベントを最初に追跡すべきですか？

最初は以下の3つから始めるのがおすすめです。（1）画面遷移（`screen_view`）でユーザーの動線を把握、（2）主要アクションのタップ（`button_tap`）でエンゲージメントを測定、（3）コンバージョンイベント（課金や登録の完了）で収益に直結する指標を追跡。これらのデータが揃えば、改善すべきポイントが見えてきます。

## まとめ

Firebase AnalyticsとRemote Configを活用することで、Rork Maxアプリのユーザー行動を可視化し、データに基づいた改善を継続的に進められるようになります。特にA/Bテストは、「感覚」ではなく「数値」でUI改善の効果を検証できる強力な手法です。

まずはカスタムイベントの設計から始めて、ユーザーがアプリ内でどのような行動をとっているかを把握しましょう。そのデータを元にA/Bテストを設計し、コンバージョン率やリテンション率の向上を目指していくことが、アプリの成長につながります。
