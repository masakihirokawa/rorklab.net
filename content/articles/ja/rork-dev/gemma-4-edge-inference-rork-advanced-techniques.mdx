## なぜエッジ推論なのか — クラウド AI の限界を超える

Rork でアプリを開発して App Store や Google Play にリリースしたあと、多くの開発者が直面するのが **API コスト問題**です。ユーザーが増えるたびにクラウド AI の呼び出しコストが膨らみ、アプリの収益をサーバー代が上回る — いわゆる「AIコスト地獄」に陥るケースは珍しくありません。

Gemma 4 のエッジモデル（E2B / E4B）は、この問題を根本から解決します。ユーザーのデバイス上でモデルが動作するため、**API 呼び出しコストはゼロ**です。さらに以下のメリットがあります。

- **完全オフライン動作**: ネットワーク接続なしで AI 機能が使える。地下鉄、飛行機、山間部でもアプリが動く
- **プライバシー保護**: ユーザーのデータがデバイスを離れないため、GDPR・APPI などのプライバシー規制への対応が容易
- **低レイテンシ**: ネットワーク往復が不要なため、応答速度が 10〜50ms と体感的にゼロ遅延
- **スケーラビリティ**: ユーザーが100万人に増えてもサーバー負荷は増えない

この記事では、Gemma 4 のエッジ推論を Rork アプリに本格実装するための上級テクニックを網羅的に解説します。量子化によるモデルの軽量化、MediaPipe との統合、オフラインファースト設計パターン、カスタムファインチューニング、そしてバッテリー消費の最適化まで、プロダクション品質の実装を目指します。

## Gemma 4 エッジモデルのアーキテクチャ理解

エッジ推論を正しく実装するには、Gemma 4 のエッジモデルがどのように設計されているかを理解する必要があります。

### E2B と E4B の内部構造

Gemma 4 E2B（約1.2Bパラメータ）と E4B（約4Bパラメータ）は、Gemini 3 の知識蒸留（Knowledge Distillation）技術をベースに構築されています。大規模な教師モデルの知識を小さなモデルに凝縮することで、パラメータ数に対して非常に高い精度を実現しています。

- **入力モダリティ**: テキスト + 画像 + 音声（E2B / E4B ともに3モダリティ対応）
- **コンテキスト長**: 最大 8,192 トークン（エッジモデル。クラウドモデルの 256K とは異なる）
- **推論エンジン**: Google AI Edge SDK / MediaPipe LLM Inference API
- **対応プラットフォーム**: iOS（Core ML / Metal）、Android（GPU delegate / NNAPI）

### メモリ要件の見積もり

```
E2B FP16:  ~2.4 GB RAM（iPhone 15 以降で動作）
E2B INT8:  ~1.2 GB RAM（iPhone 13 以降で動作）
E2B INT4:  ~0.6 GB RAM（ほぼ全ての現行スマートフォンで動作）

E4B FP16:  ~8.0 GB RAM（iPad Pro / ハイエンド Android のみ）
E4B INT8:  ~4.0 GB RAM（iPhone 15 Pro 以降）
E4B INT4:  ~2.0 GB RAM（iPhone 14 以降で動作）
```

Rork アプリのターゲットデバイスに応じて、**モデルサイズ × 量子化レベル** の組み合わせを選択することが重要です。

## INT4 / INT8 量子化でモデルを最適化する

量子化は、モデルの重みを低精度の数値表現に変換し、ファイルサイズと推論速度を大幅に改善する技術です。Gemma 4 は公式に INT4 と INT8 の量子化モデルを提供しています。

### 量子化モデルのダウンロードと変換

```python
# quantize_gemma4.py — Gemma 4 の量子化モデルを準備するスクリプト
# 実行環境: Python 3.10+, pip install ai-edge-torch mediapipe

import ai_edge_torch
from ai_edge_torch.generative.quantize import quant_recipes

# Gemma 4 E2B を INT4 量子化で変換
# Hugging Face から事前学習済みモデルをダウンロード
model_path = "google/gemma-4-e2b"

# INT4 量子化レシピの定義
quant_config = quant_recipes.full_int4_weight_only()

# TFLite 形式に変換（モバイルデプロイ用）
edge_model = ai_edge_torch.generative.convert(
    model_path=model_path,
    quant_config=quant_config,
    target="tflite",
)

# 変換後のモデルを保存
edge_model.export("gemma4_e2b_int4.tflite")
print(f"Model size: {edge_model.size_mb:.1f} MB")
# 期待する出力: Model size: 620.0 MB

# INT8 量子化（精度重視の場合）
quant_config_int8 = quant_recipes.full_int8_weight_only()
edge_model_int8 = ai_edge_torch.generative.convert(
    model_path=model_path,
    quant_config=quant_config_int8,
    target="tflite",
)
edge_model_int8.export("gemma4_e2b_int8.tflite")
print(f"INT8 Model size: {edge_model_int8.size_mb:.1f} MB")
# 期待する出力: INT8 Model size: 1200.0 MB
```

### 量子化による精度への影響

量子化はモデルサイズを削減する代わりに、わずかな精度低下を伴います。

- **INT8**: FP16 に対して精度低下は 1〜2%。ほとんどのユースケースで FP16 と区別がつかない
- **INT4**: FP16 に対して精度低下は 3〜5%。単純な分類・要約・チャットでは十分だが、複雑な推論タスクでは差が出る場合がある

Rork アプリの用途に応じて、**INT4（軽量・高速優先）** か **INT8（精度優先）** を選択してください。

## MediaPipe LLM Inference API で Rork アプリに統合する

Google の MediaPipe は、Gemma 4 をモバイルデバイスで動かすための公式推論エンジンです。React Native / Expo ベースの Rork アプリからネイティブモジュールを介して呼び出します。

### ネイティブモジュールの実装（iOS）

```swift
// ios/GemmaModule.swift — iOS ネイティブモジュール
import MediaPipeTasksGenAI
import React

@objc(GemmaModule)
class GemmaModule: NSObject {
  private var llmInference: LlmInference?
  
  @objc
  func initialize(_ modelPath: String,
                  resolver resolve: @escaping RCTPromiseResolveBlock,
                  rejecter reject: @escaping RCTPromiseRejectBlock) {
    do {
      // MediaPipe LLM Inference の設定
      let options = LlmInference.Options()
      options.modelPath = modelPath
      options.maxTokens = 2048
      options.temperature = 0.7
      
      // GPU アクセラレーションを有効化（Metal）
      options.preferredBackend = .gpu
      
      self.llmInference = try LlmInference(options: options)
      resolve(["status": "initialized", "backend": "gpu"])
    } catch {
      reject("INIT_ERROR", "Failed to initialize Gemma 4: \(error)", error)
    }
  }
  
  @objc
  func generateResponse(_ prompt: String,
                        resolver resolve: @escaping RCTPromiseResolveBlock,
                        rejecter reject: @escaping RCTPromiseRejectBlock) {
    guard let inference = self.llmInference else {
      reject("NOT_INIT", "Model not initialized", nil)
      return
    }
    
    DispatchQueue.global(qos: .userInitiated).async {
      do {
        let startTime = CFAbsoluteTimeGetCurrent()
        let response = try inference.generateResponse(inputText: prompt)
        let elapsed = CFAbsoluteTimeGetCurrent() - startTime
        
        resolve([
          "text": response,
          "latencyMs": Int(elapsed * 1000),
        ])
      } catch {
        reject("GEN_ERROR", "Generation failed: \(error)", error)
      }
    }
  }
}
```

```kotlin
import com.google.mediapipe.tasks.genai.llminference.LlmInference

                    putInt("latencyMs", elapsed.toInt())
                }
                promise.resolve(result)

                promise.reject("GEN_ERROR", "Generation failed", e)
```

### React Native ブリッジ（TypeScript）

```typescript
// services/gemmaEdge.ts — Gemma 4 エッジ推論ブリッジ
import { NativeModules, Platform } from 'react-native';
import * as FileSystem from 'expo-file-system';

const { GemmaModule } = NativeModules;

// モデルファイルの保存先
const MODEL_DIR = `${FileSystem.documentDirectory}models/`;
const MODEL_FILENAME = 'gemma4_e2b_int4.tflite';

export interface EdgeResponse {
  text: string;
  latencyMs: number;
}

/**
 * Gemma 4 エッジモデルの初期化
 * - 初回起動時にモデルをダウンロード（約600MB）
 * - 2回目以降はローカルキャッシュを使用
 */
export async function initializeGemmaEdge(
  onProgress?: (progress: number) => void
): Promise<void> {
  const modelPath = `${MODEL_DIR}${MODEL_FILENAME}`;
  
  // モデルファイルの存在チェック
  const fileInfo = await FileSystem.getInfoAsync(modelPath);
  
  if (!fileInfo.exists) {
    // 初回: モデルをダウンロード
    await FileSystem.makeDirectoryAsync(MODEL_DIR, { intermediates: true });
    
    const downloadUrl = 'https://your-cdn.example.com/models/gemma4_e2b_int4.tflite';
    const downloadResumable = FileSystem.createDownloadResumable(
      downloadUrl,
      modelPath,
      {},
      (downloadProgress) => {
        const progress = downloadProgress.totalBytesWritten /
                         downloadProgress.totalBytesExpectedToWrite;
        onProgress?.(progress);
      }
    );
    
    await downloadResumable.downloadAsync();
  }

  // ネイティブモジュールでモデルを初期化
  await GemmaModule.initialize(modelPath);
}

/**
 * テキスト生成（エッジ推論）
 */
export async function generateEdgeResponse(prompt: string): Promise<EdgeResponse> {
  return await GemmaModule.generateResponse(prompt);
}
```

## オフラインファースト設計パターン

エッジ推論の最大のメリットは**完全なオフライン動作**です。しかし、単にモデルをローカルで動かすだけでは不十分です。アプリ全体をオフラインファーストで設計する必要があります。

### ハイブリッド AI アーキテクチャ

ネットワーク状態に応じて、エッジ推論とクラウド API を動的に切り替える設計が最も実用的です。

```typescript
// services/hybridAI.ts — ハイブリッド AI マネージャー
import NetInfo from '@react-native-community/netinfo';
import { generateEdgeResponse, EdgeResponse } from './gemmaEdge';
import { callGemma4 } from './gemma4';

type AIMode = 'edge' | 'cloud' | 'auto';

interface HybridConfig {
  mode: AIMode;
  edgeMaxTokens: number;    // エッジモデルの出力上限
  cloudFallback: boolean;   // エッジ失敗時にクラウドへフォールバック
  qualityThreshold: number; // 品質スコア閾値（0-1）
}

const DEFAULT_CONFIG: HybridConfig = {
  mode: 'auto',
  edgeMaxTokens: 512,
  cloudFallback: true,
  qualityThreshold: 0.7,
};

export async function hybridGenerate(
  prompt: string,
  config: Partial<HybridConfig> = {}
): Promise<{ text: string; source: 'edge' | 'cloud'; latencyMs: number }> {
  const cfg = { ...DEFAULT_CONFIG, ...config };
  
  // ネットワーク状態を確認
  const netState = await NetInfo.fetch();
  const isOnline = netState.isConnected && netState.isInternetReachable;
  
  // モード判定
  const useEdge = cfg.mode === 'edge' ||
                  cfg.mode === 'auto' && !isOnline ||
                  cfg.mode === 'auto' && prompt.length < 500;
  
  if (useEdge) {
    try {
      const startTime = Date.now();
      const edgeResult = await generateEdgeResponse(prompt);
      return {
        text: edgeResult.text,
        source: 'edge',
        latencyMs: Date.now() - startTime,
      };
    } catch (error) {
      // エッジ推論失敗時、クラウドにフォールバック
      if (cfg.cloudFallback && isOnline) {
        const cloudResult = await callGemma4({ text: prompt });
        return { text: cloudResult.text, source: 'cloud', latencyMs: 0 };
      }
      throw error;
    }
  }
  
  // クラウド推論
  const startTime = Date.now();
  const cloudResult = await callGemma4({ text: prompt });
  return {
    text: cloudResult.text,
    source: 'cloud',
    latencyMs: Date.now() - startTime,
  };
}
```

### オフラインキューイング

ネットワーク復帰後にクラウド処理を自動実行するキューイングシステムも重要です。

```typescript
// services/offlineQueue.ts — オフライン時のリクエストキュー
import AsyncStorage from '@react-native-async-storage/async-storage';
import NetInfo from '@react-native-community/netinfo';

interface QueuedRequest {
  id: string;
  prompt: string;
  timestamp: number;
  priority: 'high' | 'normal' | 'low';
}

const QUEUE_KEY = 'gemma4_offline_queue';

export async function enqueueRequest(
  prompt: string,
  priority: 'high' | 'normal' | 'low' = 'normal'
): Promise<string> {
  const id = `req_${Date.now()}_${Math.random().toString(36).slice(2)}`;
  const queue = await getQueue();
  queue.push({ id, prompt, timestamp: Date.now(), priority });
  
  // 優先度でソート（high → normal → low）
  queue.sort((a, b) => {
    const order = { high: 0, normal: 1, low: 2 };
    return order[a.priority] - order[b.priority];
  });
  
  await AsyncStorage.setItem(QUEUE_KEY, JSON.stringify(queue));
  return id;
}

// ネットワーク復帰時に自動実行
NetInfo.addEventListener(async (state) => {
  if (state.isConnected && state.isInternetReachable) {
    await processQueue();
  }
});

async function processQueue(): Promise<void> {
  const queue = await getQueue();
  if (queue.length === 0) return;
  
  for (const request of queue) {
    try {
      const result = await callGemma4({ text: request.prompt });
      // 結果をローカルストレージに保存
      await AsyncStorage.setItem(`result_${request.id}`, JSON.stringify(result));
    } catch {
      break; // ネットワークエラーの場合は中断
    }
  }
  
  // 処理済みリクエストをキューから削除
  const remaining = await getQueue();
  const processed = remaining.filter(r =>
    !queue.some(q => q.id === r.id)
  );
  await AsyncStorage.setItem(QUEUE_KEY, JSON.stringify(processed));
}

async function getQueue(): Promise<QueuedRequest[]> {
  const data = await AsyncStorage.getItem(QUEUE_KEY);
  return data ? JSON.parse(data) : [];
}
```

## カスタムファインチューニングで独自モデルを構築する

Gemma 4 は Apache 2.0 ライセンスのため、商用目的のファインチューニングが自由に行えます。アプリ固有のドメイン知識を組み込むことで、汎用モデルでは得られない精度を実現できます。

### LoRA によるパラメータ効率的なファインチューニング

```python
# fine_tune_gemma4.py — LoRA ファインチューニングスクリプト
# 実行環境: Python 3.10+, GPU 推奨（A100 / L4）
# pip install transformers peft datasets torch accelerate

from transformers import AutoModelForCausalLM, AutoTokenizer, TrainingArguments
from peft import LoraConfig, get_peft_model, TaskType
from datasets import load_dataset
import torch

# Gemma 4 E4B をベースモデルとしてロード
model_name = "google/gemma-4-e4b"
tokenizer = AutoTokenizer.from_pretrained(model_name)
model = AutoModelForCausalLM.from_pretrained(
    model_name,
    torch_dtype=torch.bfloat16,
    device_map="auto",
)

# LoRA 設定（メモリ効率の良いファインチューニング）
lora_config = LoraConfig(
    task_type=TaskType.CAUSAL_LM,
    r=16,                  # LoRA のランク（8-64、高いほど表現力が上がるがメモリ増）
    lora_alpha=32,         # スケーリング係数
    lora_dropout=0.05,     # 過学習防止
    target_modules=[       # アテンション層のみ更新
        "q_proj", "k_proj", "v_proj", "o_proj",
    ],
)

# LoRA アダプターを適用
model = get_peft_model(model, lora_config)
print(f"Trainable parameters: {model.print_trainable_parameters()}")
# 期待する出力: trainable params: 6,553,600 || all params: 4,000,000,000 || trainable%: 0.1638

# カスタムデータセットの準備（例: 料理レシピアプリ向け）
dataset = load_dataset("json", data_files="recipe_training_data.jsonl")

# トレーニング設定
training_args = TrainingArguments(
    output_dir="./gemma4-recipe-lora",
    num_train_epochs=3,
    per_device_train_batch_size=4,
    gradient_accumulation_steps=4,
    learning_rate=2e-4,
    bf16=True,
    logging_steps=10,
    save_strategy="epoch",
)

# ファインチューニング実行
from transformers import Trainer
trainer = Trainer(
    model=model,
    args=training_args,
    train_dataset=dataset["train"],
    tokenizer=tokenizer,
)
trainer.train()

# LoRA アダプターを保存（ベースモデルとは別に保存 — 約25MB）
model.save_pretrained("./gemma4-recipe-lora-adapter")
```

### ファインチューニング済みモデルのエッジ変換

LoRA アダプターを適用した状態で、再度 INT4 量子化 → TFLite 変換を行うことで、カスタムモデルをデバイス上で動かせます。

```python
# merge_and_convert.py — LoRA マージ + エッジ変換
from peft import PeftModel
from transformers import AutoModelForCausalLM
import ai_edge_torch
from ai_edge_torch.generative.quantize import quant_recipes

# ベースモデル + LoRA アダプターをマージ
base_model = AutoModelForCausalLM.from_pretrained("google/gemma-4-e4b")
lora_model = PeftModel.from_pretrained(base_model, "./gemma4-recipe-lora-adapter")
merged_model = lora_model.merge_and_unload()

# マージ済みモデルを保存
merged_model.save_pretrained("./gemma4-recipe-merged")

# INT4 量子化 + TFLite 変換
quant_config = quant_recipes.full_int4_weight_only()
edge_model = ai_edge_torch.generative.convert(
    model_path="./gemma4-recipe-merged",
    quant_config=quant_config,
    target="tflite",
)
edge_model.export("gemma4_recipe_e4b_int4.tflite")
print(f"Custom model size: {edge_model.size_mb:.1f} MB")
# 期待する出力: Custom model size: 2050.0 MB
```

## バッテリー消費の最適化

エッジ推論はデバイスの GPU / NPU を集中的に使用するため、バッテリー消費への配慮が不可欠です。

### 推論スケジューリング

```typescript
// services/batteryAware.ts — バッテリー配慮型推論マネージャー
import * as Battery from 'expo-battery';
import { AppState, AppStateStatus } from 'react-native';

interface InferencePolicy {
  maxConcurrent: number;
  maxTokensPerRequest: number;
  cooldownMs: number;       // 連続推論間の最小間隔
  backgroundAllowed: boolean;
}

// バッテリー残量に応じたポリシー設定
function getPolicy(batteryLevel: number, isCharging: boolean): InferencePolicy {
  if (isCharging) {
    return { maxConcurrent: 3, maxTokensPerRequest: 2048, cooldownMs: 0, backgroundAllowed: true };
  }
  if (batteryLevel > 0.5) {
    return { maxConcurrent: 2, maxTokensPerRequest: 1024, cooldownMs: 500, backgroundAllowed: true };
  }
  if (batteryLevel > 0.2) {
    return { maxConcurrent: 1, maxTokensPerRequest: 512, cooldownMs: 2000, backgroundAllowed: false };
  }
  // バッテリー残量 20% 以下: 推論を最小限に制限
  return { maxConcurrent: 1, maxTokensPerRequest: 256, cooldownMs: 5000, backgroundAllowed: false };
}

export class BatteryAwareInference {
  private lastInferenceTime = 0;
  private activeCount = 0;

  async canInfer(): Promise<{ allowed: boolean; reason?: string }> {
    const batteryLevel = await Battery.getBatteryLevelAsync();
    const batteryState = await Battery.getBatteryStateAsync();
    const isCharging = batteryState === Battery.BatteryState.CHARGING;
    const policy = getPolicy(batteryLevel, isCharging);

    // バックグラウンド状態のチェック
    if (AppState.currentState !== 'active' && !policy.backgroundAllowed) {
      return { allowed: false, reason: 'バッテリー節約のためバックグラウンド推論を停止中' };
    }

    // 同時実行数のチェック
    if (this.activeCount >= policy.maxConcurrent) {
      return { allowed: false, reason: '処理中のリクエストがあります' };
    }

    // クールダウンのチェック
    const elapsed = Date.now() - this.lastInferenceTime;
    if (elapsed < policy.cooldownMs) {
      return { allowed: false, reason: `${Math.ceil((policy.cooldownMs - elapsed) / 1000)}秒後に再試行してください` };
    }

    return { allowed: true };
  }

  async inferWithPolicy(prompt: string): Promise<EdgeResponse> {
    const check = await this.canInfer();
    if (!check.allowed) {
      throw new Error(check.reason);
    }

    this.activeCount++;
    this.lastInferenceTime = Date.now();

    try {
      return await generateEdgeResponse(prompt);
    } finally {
      this.activeCount--;
    }
  }
}
```

### サーマルスロットリング対策

長時間の推論でデバイスが過熱すると、OS がパフォーマンスを自動的に制限（サーマルスロットリング）します。これを事前に検出し、推論の頻度を調整する機能も実装しておくと良いでしょう。

```typescript
// utils/thermalMonitor.ts — サーマル状態の監視
// iOS: ProcessInfo.thermalState を監視
// Android: BatteryManager.EXTRA_TEMPERATURE を監視

export type ThermalState = 'nominal' | 'fair' | 'serious' | 'critical';

export function adjustInferenceForThermal(state: ThermalState): {
  shouldDefer: boolean;
  reducedTokens: number;
} {
  switch (state) {
    case 'nominal':
      return { shouldDefer: false, reducedTokens: 2048 };
    case 'fair':
      return { shouldDefer: false, reducedTokens: 1024 };
    case 'serious':
      return { shouldDefer: true, reducedTokens: 512 };
    case 'critical':
      return { shouldDefer: true, reducedTokens: 0 }; // 推論を完全停止
  }
}
```

## パフォーマンスベンチマークと最適化指標

エッジ推論のパフォーマンスを定量的に把握し、継続的に改善するためのベンチマーク手法を紹介します。

```typescript
// utils/benchmark.ts — 推論パフォーマンス計測
export interface BenchmarkResult {
  modelName: string;
  quantization: 'fp16' | 'int8' | 'int4';
  device: string;
  tokensPerSecond: number;
  firstTokenLatencyMs: number;
  peakMemoryMB: number;
  batteryDrainPerHour: number; // パーセント/時
}

export async function runBenchmark(
  modelName: string,
  testPrompts: string[]
): Promise<BenchmarkResult> {
  const results: number[] = [];
  
  for (const prompt of testPrompts) {
    const start = performance.now();
    const response = await generateEdgeResponse(prompt);
    const elapsed = performance.now() - start;
    
    // トークン数を概算（英語は4文字/token、日本語は2文字/token）
    const estimatedTokens = response.text.length / 3;
    results.push(estimatedTokens / (elapsed / 1000));
  }
  
  const avgTps = results.reduce((a, b) => a + b, 0) / results.length;
  
  return {
    modelName,
    quantization: 'int4',
    device: Platform.OS === 'ios' ? 'iPhone' : 'Android',
    tokensPerSecond: Math.round(avgTps * 10) / 10,
    firstTokenLatencyMs: 0, // 実測値を入れる
    peakMemoryMB: 0,        // 実測値を入れる
    batteryDrainPerHour: 0, // 実測値を入れる
  };
}
```

## FAQ

**Q: エッジ推論に対応している最低限のデバイススペックは？**

A: Gemma 4 E2B INT4 の場合、約 600MB の RAM を使用するため、RAM 4GB 以上のデバイスが推奨です。具体的には、iOS では iPhone 12 以降、Android では 2022年以降のミッドレンジ以上のスマートフォンで動作します。E4B は iPhone 14 以降または RAM 8GB 以上の Android が推奨です。

**Q: アプリのバンドルサイズが大きくなりすぎないか心配です。どう対処すべきですか？**

A: モデルファイルをアプリバンドルに含めるのではなく、初回起動時に CDN からダウンロードする方式を推奨します。ダウンロード進捗を表示するプログレスバーを用意し、ユーザー体験を損なわないようにしましょう。この記事の `initializeGemmaEdge` 関数がこのパターンを実装しています。

**Q: ファインチューニングにはどのくらいのデータが必要ですか？**

A: LoRA ファインチューニングは比較的少量のデータで効果を発揮します。500〜1,000 サンプルのドメイン固有データで顕著な改善が見られることが多いです。ただし、データの品質が量よりも重要です。高品質なプロンプト-応答ペアを用意することに注力してください。

**Q: エッジ推論中にアプリがクラッシュする場合の対処法は？**

A: 主な原因はメモリ不足（OOM）です。対処法は3つあります。(1) より小さいモデル（E2B）またはより低精度の量子化（INT4）を使う、(2) 他のアプリがメモリを解放するよう OS に要求する（`didReceiveMemoryWarning` ハンドリング）、(3) 推論前にデバイスの利用可能メモリをチェックし、不足時はクラウドにフォールバックします。

**Q: App Store / Google Play の審査でエッジ AI モデルに関する制限はありますか？**

A: 現時点では、オンデバイス AI モデルの使用を禁止する審査ガイドラインはありません。ただし、モデルダウンロードサイズが大きい場合は Wi-Fi 接続時のみダウンロードすることを推奨します（Apple は 200MB 以上のダウンロードでセルラー通信を制限します）。また、プライバシーポリシーに AI 機能の説明を含めることをお勧めします。
