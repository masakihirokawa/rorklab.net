## WebRTC の基本アーキテクチャを理解する

### ブラウザ技術からモバイルへ

WebRTC（Web Real-Time Communication）は、もともとブラウザ間でプラグインなしにリアルタイム通信を実現するために開発されたオープンソースプロジェクトです。Google が 2011 年に公開して以来、ビデオ通話・音声通話・データ転送の標準技術として広く採用されています。

React Native でも `react-native-webrtc` ライブラリを通じてネイティブの WebRTC 機能にアクセスでき、Rork Max のコード生成機能と組み合わせることで効率的にビデオ通話アプリを構築できます。

### P2P 接続の 3 つの柱

WebRTC の P2P（Peer-to-Peer）接続は、以下の 3 つのプロトコルで成り立っています。

**SDP（Session Description Protocol）:** 通話の「契約書」のようなもので、使用するコーデック、解像度、帯域幅などのメディア情報を相手に伝えます。発信側が「Offer」を作成し、受信側が「Answer」を返すことでネゴシエーションが完了します。

**ICE（Interactive Connectivity Establishment）:** NAT やファイアウォールを越えて最適な通信経路を見つけるためのフレームワークです。複数の候補（ICE Candidate）を収集し、接続テストを行って最も効率的なパスを選択します。

**STUN/TURN サーバー:** STUN（Session Traversal Utilities for NAT）はパブリック IP アドレスを発見するために使用し、TURN（Traversal Using Relays around NAT）は直接接続ができない場合にリレーサーバーとして中継します。

```typescript
// WebRTC接続の基本フロー
// 1. RTCPeerConnectionを作成
const configuration: RTCConfiguration = {
  iceServers: [
    // STUN: パブリックIPの発見用（無料）
    { urls: 'stun:stun.l.google.com:19302' },
    { urls: 'stun:stun1.l.google.com:19302' },
    // TURN: NAT越えリレー用（本番では自前またはサービスを利用）
    {
      urls: 'turn:your-turn-server.example.com:3478',
      username: 'YOUR_TURN_USERNAME',
      credential: 'YOUR_TURN_CREDENTIAL',
    },
  ],
};

const peerConnection = new RTCPeerConnection(configuration);

// 2. ローカルメディアストリームを取得してトラックを追加
const localStream = await mediaDevices.getUserMedia({
  audio: true,
  video: {
    facingMode: 'user', // フロントカメラ
    width: { ideal: 1280 },
    height: { ideal: 720 },
  },
});

localStream.getTracks().forEach((track) => {
  peerConnection.addTrack(track, localStream);
});

// 3. ICE候補が見つかるたびにシグナリング経由で相手に送信
peerConnection.onicecandidate = (event) => {
  if (event.candidate) {
    sendToSignalingServer({
      type: 'ice-candidate',
      candidate: event.candidate,
    });
  }
};

// 4. リモートストリームを受信
peerConnection.ontrack = (event) => {
  // event.streams[0] をリモートビデオ表示に設定
  setRemoteStream(event.streams[0]);
};
```

上記のコードが WebRTC 接続の骨格です。ここで重要なのは、`RTCPeerConnection` 自体は通信経路のネゴシエーションを担当しますが、**SDP や ICE 候補を相手に届ける仕組み（シグナリング）は WebRTC の仕様には含まれていない**という点です。次のセクションで、このシグナリングの設計パターンを詳しく見ていきましょう。

## Supabase Realtime でシグナリングサーバーを構築する

### なぜ Supabase Realtime が最適なのか

シグナリングサーバーは SDP や ICE 候補をリアルタイムで中継する役割を担います。WebSocket、Firebase Realtime Database、Socket.io など選択肢は多数ありますが、Rork エコシステムとの親和性を考えると **Supabase Realtime** が最も効率的です。

その理由は 3 つあります。まず、Supabase は認証・データベース・ストレージが統合されているため、ユーザー管理から通話履歴の保存まで一貫した開発体験が得られます。次に、Realtime の Broadcast 機能は、特定のチャンネルに参加しているクライアント間でメッセージを即座に送受信でき、シグナリングに必要な低遅延通信を実現します。最後に、Row Level Security（RLS）との組み合わせで、通話のプライバシーをデータベースレベルで保護できます。

### データベーススキーマの設計

```sql
-- 通話セッションを管理するテーブル
CREATE TABLE call_sessions (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  room_id TEXT NOT NULL UNIQUE,
  caller_id UUID REFERENCES auth.users(id) NOT NULL,
  callee_id UUID REFERENCES auth.users(id),
  status TEXT NOT NULL DEFAULT 'waiting'
    CHECK (status IN ('waiting', 'ringing', 'active', 'ended', 'missed')),
  started_at TIMESTAMPTZ,
  ended_at TIMESTAMPTZ,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  -- メタデータ（通話時間、品質スコア等）
  metadata JSONB DEFAULT '{}'::jsonb
);

-- RLS: 参加者のみアクセス可能
ALTER TABLE call_sessions ENABLE ROW LEVEL SECURITY;

CREATE POLICY "参加者のみ閲覧可能" ON call_sessions
  FOR SELECT USING (
    auth.uid() = caller_id OR auth.uid() = callee_id
  );

CREATE POLICY "認証済みユーザーが通話を開始可能" ON call_sessions
  FOR INSERT WITH CHECK (auth.uid() = caller_id);

CREATE POLICY "参加者のみ更新可能" ON call_sessions
  FOR UPDATE USING (
    auth.uid() = caller_id OR auth.uid() = callee_id
  );
```

### シグナリングフローの実装

```typescript
import { createClient } from '@supabase/supabase-js';

// シグナリングマネージャー
class SignalingManager {
  private supabase;
  private channel;
  private roomId: string;

  constructor(supabaseUrl: string, supabaseKey: string, roomId: string) {
    this.supabase = createClient(supabaseUrl, supabaseKey);
    this.roomId = roomId;
  }

  // チャンネルに参加してシグナリングメッセージを待受
  async join(userId: string, handlers: SignalingHandlers) {
    this.channel = this.supabase.channel(`call:${this.roomId}`, {
      config: {
        broadcast: { self: false }, // 自分自身にはブロードキャストしない
      },
    });

    this.channel
      .on('broadcast', { event: 'offer' }, ({ payload }) => {
        handlers.onOffer(payload);
      })
      .on('broadcast', { event: 'answer' }, ({ payload }) => {
        handlers.onAnswer(payload);
      })
      .on('broadcast', { event: 'ice-candidate' }, ({ payload }) => {
        handlers.onIceCandidate(payload);
      })
      .on('broadcast', { event: 'hangup' }, ({ payload }) => {
        handlers.onHangup(payload);
      })
      .subscribe();
  }

  // SDP Offerを送信
  async sendOffer(offer: RTCSessionDescriptionInit) {
    await this.channel.send({
      type: 'broadcast',
      event: 'offer',
      payload: { sdp: offer },
    });
  }

  // SDP Answerを送信
  async sendAnswer(answer: RTCSessionDescriptionInit) {
    await this.channel.send({
      type: 'broadcast',
      event: 'answer',
      payload: { sdp: answer },
    });
  }

  // ICE候補を送信
  async sendIceCandidate(candidate: RTCIceCandidate) {
    await this.channel.send({
      type: 'broadcast',
      event: 'ice-candidate',
      payload: { candidate },
    });
  }

  // 通話終了
  async hangup(userId: string) {
    await this.channel.send({
      type: 'broadcast',
      event: 'hangup',
      payload: { userId },
    });
    this.channel.unsubscribe();
  }
}

// 使用するハンドラーの型定義
interface SignalingHandlers {
  onOffer: (payload: { sdp: RTCSessionDescriptionInit }) => void;
  onAnswer: (payload: { sdp: RTCSessionDescriptionInit }) => void;
  onIceCandidate: (payload: { candidate: RTCIceCandidate }) => void;
  onHangup: (payload: { userId: string }) => void;
}
```

このシグナリングマネージャーは、Supabase の Broadcast 機能を使って通話ルームごとにチャンネルを作成し、SDP や ICE 候補をリアルタイムで中継します。`self: false` の設定により、自分が送信したメッセージを自分で受信してしまうループを防いでいます。

## Rork Max でビデオ通話 UI を構築する

### コンポーネント構成

ビデオ通話画面は複数のレイヤーで構成されます。Rork Max の AI コード生成を活用しつつ、通話品質に直結する部分は手動で最適化するアプローチが効果的です。

```typescript
import React, { useState, useEffect, useRef, useCallback } from 'react';
import {
  View,
  StyleSheet,
  TouchableOpacity,
  Text,
  Dimensions,
  Animated,
} from 'react-native';
import {
  RTCView,
  mediaDevices,
  RTCPeerConnection,
  RTCSessionDescription,
  RTCIceCandidate,
} from 'react-native-webrtc';

const { width: SCREEN_WIDTH, height: SCREEN_HEIGHT } = Dimensions.get('window');

interface VideoCallScreenProps {
  roomId: string;
  userId: string;
  isCaller: boolean;
  onCallEnd: () => void;
}

export const VideoCallScreen: React.FC<VideoCallScreenProps> = ({
  roomId,
  userId,
  isCaller,
  onCallEnd,
}) => {
  const [localStream, setLocalStream] = useState<MediaStream | null>(null);
  const [remoteStream, setRemoteStream] = useState<MediaStream | null>(null);
  const [isMuted, setIsMuted] = useState(false);
  const [isCameraOff, setIsCameraOff] = useState(false);
  const [isFrontCamera, setIsFrontCamera] = useState(true);
  const [callDuration, setCallDuration] = useState(0);
  const [connectionState, setConnectionState] = useState<string>('new');

  const peerConnection = useRef<RTCPeerConnection | null>(null);
  const signalingManager = useRef<SignalingManager | null>(null);
  const durationTimer = useRef<NodeJS.Timeout | null>(null);

  // 通話時間のフォーマット
  const formatDuration = useCallback((seconds: number): string => {
    const mins = Math.floor(seconds / 60);
    const secs = seconds % 60;
    return `${mins.toString().padStart(2, '0')}:${secs
      .toString()
      .padStart(2, '0')}`;
  }, []);

  useEffect(() => {
    initializeCall();
    return () => {
      cleanup();
    };
  }, []);

  const initializeCall = async () => {
    try {
      // ローカルメディアの取得
      const stream = await mediaDevices.getUserMedia({
        audio: {
          echoCancellation: true,
          noiseSuppression: true,
          autoGainControl: true,
        },
        video: {
          facingMode: 'user',
          width: { ideal: 1280 },
          height: { ideal: 720 },
          frameRate: { ideal: 30, max: 30 },
        },
      });
      setLocalStream(stream);

      // PeerConnectionの初期化
      const pc = new RTCPeerConnection({
        iceServers: [
          { urls: 'stun:stun.l.google.com:19302' },
          { urls: 'stun:stun1.l.google.com:19302' },
        ],
      });

      // トラックを追加
      stream.getTracks().forEach((track) => {
        pc.addTrack(track, stream);
      });

      // リモートストリームの受信
      pc.ontrack = (event) => {
        if (event.streams && event.streams[0]) {
          setRemoteStream(event.streams[0]);
          // 通話タイマーを開始
          startDurationTimer();
        }
      };

      // 接続状態の監視
      pc.onconnectionstatechange = () => {
        setConnectionState(pc.connectionState);
        if (
          pc.connectionState === 'disconnected' ||
          pc.connectionState === 'failed'
        ) {
          handleCallEnd();
        }
      };

      peerConnection.current = pc;

      // シグナリングの初期化
      const sm = new SignalingManager(
        'YOUR_SUPABASE_URL',
        'YOUR_SUPABASE_ANON_KEY',
        roomId
      );

      // ICE候補の送信
      pc.onicecandidate = (event) => {
        if (event.candidate) {
          sm.sendIceCandidate(event.candidate);
        }
      };

      await sm.join(userId, {
        onOffer: async ({ sdp }) => {
          await pc.setRemoteDescription(new RTCSessionDescription(sdp));
          const answer = await pc.createAnswer();
          await pc.setLocalDescription(answer);
          sm.sendAnswer(answer);
        },
        onAnswer: async ({ sdp }) => {
          await pc.setRemoteDescription(new RTCSessionDescription(sdp));
        },
        onIceCandidate: async ({ candidate }) => {
          await pc.addIceCandidate(new RTCIceCandidate(candidate));
        },
        onHangup: () => {
          handleCallEnd();
        },
      });

      signalingManager.current = sm;

      // 発信者の場合はOfferを作成
      if (isCaller) {
        const offer = await pc.createOffer({
          offerToReceiveAudio: true,
          offerToReceiveVideo: true,
        });
        await pc.setLocalDescription(offer);
        sm.sendOffer(offer);
      }
    } catch (error) {
      console.error('通話の初期化に失敗:', error);
    }
  };

  const startDurationTimer = () => {
    durationTimer.current = setInterval(() => {
      setCallDuration((prev) => prev + 1);
    }, 1000);
  };

  const toggleMute = () => {
    if (localStream) {
      localStream.getAudioTracks().forEach((track) => {
        track.enabled = !track.enabled;
      });
      setIsMuted(!isMuted);
    }
  };

  const toggleCamera = () => {
    if (localStream) {
      localStream.getVideoTracks().forEach((track) => {
        track.enabled = !track.enabled;
      });
      setIsCameraOff(!isCameraOff);
    }
  };

  const switchCamera = () => {
    if (localStream) {
      localStream.getVideoTracks().forEach((track) => {
        // @ts-ignore — React Native WebRTC固有のメソッド
        track._switchCamera();
      });
      setIsFrontCamera(!isFrontCamera);
    }
  };

  const handleCallEnd = () => {
    signalingManager.current?.hangup(userId);
    cleanup();
    onCallEnd();
  };

  const cleanup = () => {
    if (durationTimer.current) {
      clearInterval(durationTimer.current);
    }
    localStream?.getTracks().forEach((track) => track.stop());
    peerConnection.current?.close();
  };

  return (
    <View style={styles.container}>
      {/* リモートビデオ（全画面表示） */}
      {remoteStream ? (
        <RTCView
          streamURL={remoteStream.toURL()}
          style={styles.remoteVideo}
          objectFit="cover"
          zOrder={0}
        />
      ) : (
        <View style={styles.waitingContainer}>
          <Text style={styles.waitingText}>接続中...</Text>
          <Text style={styles.connectionState}>{connectionState}</Text>
        </View>
      )}

      {/* ローカルビデオ（ピクチャーインピクチャー） */}
      {localStream && (
        <RTCView
          streamURL={localStream.toURL()}
          style={styles.localVideo}
          objectFit="cover"
          mirror={isFrontCamera}
          zOrder={1}
        />
      )}

      {/* 通話時間表示 */}
      {remoteStream && (
        <View style={styles.durationContainer}>
          <Text style={styles.durationText}>
            {formatDuration(callDuration)}
          </Text>
        </View>
      )}

      {/* コントロールバー */}
      <View style={styles.controlBar}>
        <TouchableOpacity
          style={[styles.controlButton, isMuted && styles.activeButton]}
          onPress={toggleMute}
        >
          <Text style={styles.controlIcon}>{isMuted ? '🔇' : '🔊'}</Text>
          <Text style={styles.controlLabel}>
            {isMuted ? 'ミュート中' : 'ミュート'}
          </Text>
        </TouchableOpacity>

        <TouchableOpacity
          style={[styles.controlButton, styles.hangupButton]}
          onPress={handleCallEnd}
        >
          <Text style={styles.hangupIcon}>📞</Text>
          <Text style={styles.controlLabel}>終了</Text>
        </TouchableOpacity>

        <TouchableOpacity
          style={[styles.controlButton, isCameraOff && styles.activeButton]}
          onPress={toggleCamera}
        >
          <Text style={styles.controlIcon}>
            {isCameraOff ? '📷' : '📹'}
          </Text>
          <Text style={styles.controlLabel}>
            {isCameraOff ? 'カメラOFF' : 'カメラ'}
          </Text>
        </TouchableOpacity>

        <TouchableOpacity style={styles.controlButton} onPress={switchCamera}>
          <Text style={styles.controlIcon}>🔄</Text>
          <Text style={styles.controlLabel}>切替</Text>
        </TouchableOpacity>
      </View>
    </View>
  );
};

const styles = StyleSheet.create({
  container: {
    flex: 1,
    backgroundColor: '#000',
  },
  remoteVideo: {
    flex: 1,
    width: SCREEN_WIDTH,
    height: SCREEN_HEIGHT,
  },
  localVideo: {
    position: 'absolute',
    top: 60,
    right: 16,
    width: 120,
    height: 160,
    borderRadius: 12,
    overflow: 'hidden',
    borderWidth: 2,

  },

    flex: 1,
    justifyContent: 'center',

  },
```

### 着信画面の実装

通話体験で見落とされがちなのが着信画面です。ユーザーがアプリをアクティブに使っていないときでも、プッシュ通知と連動して着信を知らせる仕組みが必要です。

```typescript
import React, { useEffect, useRef } from 'react';
import {
  View,
  Text,
  TouchableOpacity,
  StyleSheet,
  Animated,
  Vibration,
} from 'react-native';

interface IncomingCallScreenProps {
  callerName: string;
  callerAvatar?: string;
  onAccept: () => void;
  onReject: () => void;
}

export const IncomingCallScreen: React.FC<IncomingCallScreenProps> = ({
  callerName,
  onAccept,
  onReject,
}) => {
  const pulseAnim = useRef(new Animated.Value(1)).current;

  useEffect(() => {
    // バイブレーションパターン（着信風）
    const vibrationPattern = [0, 500, 200, 500];
    Vibration.vibrate(vibrationPattern, true);

    // パルスアニメーション
    const pulse = Animated.loop(
      Animated.sequence([
        Animated.timing(pulseAnim, {
          toValue: 1.3,
          duration: 800,
          useNativeDriver: true,
        }),
        Animated.timing(pulseAnim, {
          toValue: 1,
          duration: 800,
          useNativeDriver: true,
        }),
      ])
    );
    pulse.start();

    return () => {
      Vibration.cancel();
      pulse.stop();
    };
  }, []);

  return (
    <View style={incomingStyles.container}>
      <Text style={incomingStyles.label}>着信中</Text>
      <Text style={incomingStyles.callerName}>{callerName}</Text>

      <Animated.View
        style={[
          incomingStyles.avatarRing,
          { transform: [{ scale: pulseAnim }] },
        ]}
```

## 通話品質の最適化テクニック

### アダプティブビットレート制御

モバイル環境ではネットワーク状況が刻々と変化します。Wi-Fi からモバイル回線への切り替え、電波の弱いエリアへの移動など、帯域幅は常に変動しています。WebRTC にはアダプティブビットレート機能が組み込まれていますが、追加の最適化を行うことで体験を大幅に改善できます。

```typescript
// ネットワーク品質に応じた映像パラメータの動的調整
class QualityManager {
  private peerConnection: RTCPeerConnection;
  private currentQuality: 'high' | 'medium' | 'low' = 'high';

  // 映像品質プリセット
  private readonly qualityPresets = {
    high: {
      maxBitrate: 2500000, // 2.5 Mbps
      maxFramerate: 30,
      scaleResolutionDownBy: 1,
    },
    medium: {
      maxBitrate: 1000000, // 1 Mbps
      maxFramerate: 24,
      scaleResolutionDownBy: 1.5,
    },
    low: {
      maxBitrate: 400000, // 400 Kbps
      maxFramerate: 15,
      scaleResolutionDownBy: 2,
    },
  };

  constructor(pc: RTCPeerConnection) {
    this.peerConnection = pc;
  }

  // ネットワーク統計を監視して品質を自動調整
  async startMonitoring(intervalMs: number = 3000) {
    setInterval(async () => {
      const stats = await this.peerConnection.getStats();
      let packetsLost = 0;
      let packetsReceived = 0;
      let roundTripTime = 0;

      stats.forEach((report) => {
        if (report.type === 'inbound-rtp' && report.kind === 'video') {
          packetsLost = report.packetsLost || 0;
          packetsReceived = report.packetsReceived || 0;
        }
        if (report.type === 'candidate-pair' && report.state === 'succeeded') {
          roundTripTime = report.currentRoundTripTime || 0;
        }
      });

      // パケットロス率とRTTに基づいて品質を決定
      const lossRate =
        packetsReceived > 0
          ? packetsLost / (packetsLost + packetsReceived)
          : 0;

      let targetQuality: 'high' | 'medium' | 'low';
      if (lossRate < 0.02 && roundTripTime < 0.15) {
        targetQuality = 'high';
      } else if (lossRate < 0.05 && roundTripTime < 0.3) {
        targetQuality = 'medium';
      } else {
        targetQuality = 'low';
      }

      if (targetQuality !== this.currentQuality) {
        await this.applyQuality(targetQuality);
        this.currentQuality = targetQuality;
      }
    }, intervalMs);
  }

  private async applyQuality(quality: 'high' | 'medium' | 'low') {
    const preset = this.qualityPresets[quality];
    const sender = this.peerConnection
      .getSenders()
      .find((s) => s.track?.kind === 'video');

    if (sender) {
      const params = sender.getParameters();
      if (!params.encodings) {
        params.encodings = [{}];
      }
      params.encodings[0].maxBitrate = preset.maxBitrate;
      params.encodings[0].maxFramerate = preset.maxFramerate;
      params.encodings[0].scaleResolutionDownBy =
        preset.scaleResolutionDownBy;
      await sender.setParameters(params);
      console.log(`映像品質を ${quality} に変更`);
    }
  }
}
```

### 接続断からの自動復旧

モバイルアプリでは、トンネル進入時の一時的な回線断やWi-Fiの切り替えが頻繁に起こります。ユーザーに「切断されました」と表示して終了するのではなく、自動的に再接続を試みる仕組みが重要です。

```typescript
// ICE再起動による接続復旧
class ConnectionRecovery {
  private peerConnection: RTCPeerConnection;
  private signalingManager: SignalingManager;
  private maxRetries = 3;
  private retryCount = 0;
  private retryTimeoutId: NodeJS.Timeout | null = null;

  constructor(pc: RTCPeerConnection, sm: SignalingManager) {
    this.peerConnection = pc;
    this.signalingManager = sm;
    this.setupMonitoring();
  }

  private setupMonitoring() {
    this.peerConnection.oniceconnectionstatechange = () => {
      const state = this.peerConnection.iceConnectionState;

      switch (state) {
        case 'disconnected':
          // 短時間の切断は自然復旧を待つ（5秒）
          this.retryTimeoutId = setTimeout(() => {
            if (
              this.peerConnection.iceConnectionState === 'disconnected'
            ) {
              this.attemptRecovery();
            }
          }, 5000);
          break;
        case 'failed':
          // 完全な失敗時はICE再起動を試行
          this.attemptRecovery();
          break;
        case 'connected':
          // 接続復旧時にリトライカウンターをリセット
          this.retryCount = 0;
          if (this.retryTimeoutId) {
            clearTimeout(this.retryTimeoutId);
          }
          break;
      }
    };
  }

  private async attemptRecovery() {
    if (this.retryCount >= this.maxRetries) {
      console.log('再接続の上限に達しました。通話を終了します。');
      return;
    }

    this.retryCount++;
    console.log(
      `再接続を試行中... (${this.retryCount}/${this.maxRetries})`
    );

    try {
      // ICE再起動: 新しいICE候補を収集して接続を再確立
      const offer = await this.peerConnection.createOffer({
        iceRestart: true,
      });
      await this.peerConnection.setLocalDescription(offer);
      await this.signalingManager.sendOffer(offer);
    } catch (error) {
      console.error('再接続に失敗:', error);
      // 指数バックオフで再試行
      setTimeout(
        () => this.attemptRecovery(),
        Math.pow(2, this.retryCount) * 1000
      );
    }
  }
}
```

## 画面共有機能を実装する

画面共有はビデオ通話アプリの付加価値を大きく高める機能です。技術サポート、プレゼンテーション、ペアプログラミングなど、多くのユースケースで活用されます。

### iOS での画面キャプチャ

iOS では `RPScreenRecorder` を使って画面キャプチャを取得し、WebRTC のメディアトラックとして送信します。React Native では `react-native-webrtc` がこの機能をサポートしています。

```typescript
import { mediaDevices } from 'react-native-webrtc';

class ScreenShareManager {
  private peerConnection: RTCPeerConnection;
  private screenStream: MediaStream | null = null;
  private originalVideoTrack: MediaStreamTrack | null = null;
  private originalVideoSender: RTCRtpSender | null = null;

  constructor(pc: RTCPeerConnection) {
    this.peerConnection = pc;
  }

  async startScreenShare(): Promise<boolean> {
    try {
      // 画面キャプチャストリームを取得
      // iOS: RPScreenRecorder、Android: MediaProjection APIを内部で使用
      this.screenStream = await mediaDevices.getDisplayMedia({
        video: true,
        audio: false, // システム音声も共有する場合は true
      });

      const screenTrack = this.screenStream.getVideoTracks()[0];

      // 既存のビデオトラックを画面共有トラックに置き換え
      const videoSender = this.peerConnection
        .getSenders()
        .find((s) => s.track?.kind === 'video');

      if (videoSender) {
        this.originalVideoTrack = videoSender.track;
        this.originalVideoSender = videoSender;
        await videoSender.replaceTrack(screenTrack);
      }

      // 画面共有が終了した場合（ユーザーが停止ボタンを押した場合）
      screenTrack.onended = () => {
        this.stopScreenShare();
      };

      return true;
    } catch (error) {
      console.error('画面共有の開始に失敗:', error);
      return false;
    }
  }

  async stopScreenShare(): Promise<void> {
    // 元のカメラ映像に戻す
    if (this.originalVideoSender && this.originalVideoTrack) {
      await this.originalVideoSender.replaceTrack(this.originalVideoTrack);
    }

    // 画面共有ストリームを停止
    this.screenStream?.getTracks().forEach((track) => track.stop());
    this.screenStream = null;
    this.originalVideoTrack = null;
    this.originalVideoSender = null;
  }
}
```

画面共有時に重要なのは、`replaceTrack()` メソッドを使うことです。新しいトラックを追加するのではなく、既存のビデオトラックを置き換えることで、SDP の再ネゴシエーションなしにシームレスに切り替えられます。

## グループ通話への拡張 — SFU アーキテクチャ

### P2P の限界と SFU の必要性

1 対 1 通話では P2P 接続で十分ですが、3 人以上のグループ通話では各参加者が他の全員と個別に P2P 接続を維持する必要があり、端末の負荷とネットワーク帯域幅が急激に増大します。4 人の通話で各端末が 3 本の送信ストリームを維持するのは、モバイルデバイスには現実的ではありません。

この問題を解決するのが **SFU（Selective Forwarding Unit）** です。SFU は参加者から 1 本のストリームを受け取り、他の参加者に選択的に転送するサーバーです。各端末は 1 本の送信と N-1 本の受信だけで済み、クライアントの負荷を大幅に削減できます。

### SFU 接続のクライアント実装

```typescript
// SFUベースのグループ通話マネージャー
class GroupCallManager {
  private localStream: MediaStream | null = null;
  private publisherConnection: RTCPeerConnection | null = null;
  // 参加者ごとの受信用PeerConnection
  private subscriberConnections: Map<string, RTCPeerConnection> = new Map();
  private participants: Map<string, MediaStream> = new Map();
  private sfuEndpoint: string;
  private roomId: string;

  constructor(sfuEndpoint: string, roomId: string) {
    this.sfuEndpoint = sfuEndpoint;
    this.roomId = roomId;
  }

  // 自分のメディアをSFUに送信（Publisher）
  async publishLocalStream(stream: MediaStream): Promise<void> {
    this.localStream = stream;

    const pc = new RTCPeerConnection({
      iceServers: [{ urls: 'stun:stun.l.google.com:19302' }],
    });

    stream.getTracks().forEach((track) => {
      pc.addTrack(track, stream);
    });

    // SFUとのネゴシエーション
    const offer = await pc.createOffer();
    await pc.setLocalDescription(offer);

    const response = await fetch(`${this.sfuEndpoint}/publish`, {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({
        roomId: this.roomId,
        sdp: offer.sdp,
      }),
    });

    const { sdp: answerSdp } = await response.json();
    await pc.setRemoteDescription(
      new RTCSessionDescription({ type: 'answer', sdp: answerSdp })
    );

    this.publisherConnection = pc;
  }

  // 特定の参加者のストリームをSFUから受信（Subscriber）
  async subscribeToParticipant(participantId: string): Promise<MediaStream> {
    const pc = new RTCPeerConnection({
      iceServers: [{ urls: 'stun:stun.l.google.com:19302' }],
    });

    // トランシーバーを受信専用で追加
    pc.addTransceiver('video', { direction: 'recvonly' });
    pc.addTransceiver('audio', { direction: 'recvonly' });

    const remoteStream = new MediaStream();
    pc.ontrack = (event) => {
      remoteStream.addTrack(event.track);
    };

    const offer = await pc.createOffer();
    await pc.setLocalDescription(offer);

    const response = await fetch(`${this.sfuEndpoint}/subscribe`, {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({
        roomId: this.roomId,
        participantId,
        sdp: offer.sdp,
      }),
    });

    const { sdp: answerSdp } = await response.json();
    await pc.setRemoteDescription(
      new RTCSessionDescription({ type: 'answer', sdp: answerSdp })
    );

    this.subscriberConnections.set(participantId, pc);
    this.participants.set(participantId, remoteStream);

    return remoteStream;
  }

  // 参加者の退出処理
  removeParticipant(participantId: string): void {
    const pc = this.subscriberConnections.get(participantId);
    if (pc) {
      pc.close();
      this.subscriberConnections.delete(participantId);
    }
    this.participants.delete(participantId);
  }

  // 全接続のクリーンアップ
  async disconnect(): Promise<void> {
    this.publisherConnection?.close();
    this.subscriberConnections.forEach((pc) => pc.close());
    this.subscriberConnections.clear();
    this.participants.clear();
    this.localStream?.getTracks().forEach((track) => track.stop());
  }
}
```

グループ通話のサーバー側には、Cloudflare の **Calls API** や **LiveKit**、**Janus** といった OSS の SFU サーバーが利用できます。Rork Max で構築するアプリのスケールに応じて、マネージドサービスかセルフホストかを選択しましょう。

## 通話録画機能の実装

通話を録画して後から確認したいというニーズは、教育・コンサルティング・カスタマーサポートの分野で特に強くあります。

```typescript
// クライアントサイドでの通話録画（MediaRecorder API）
class CallRecorder {
  private mediaRecorder: MediaRecorder | null = null;
  private recordedChunks: Blob[] = [];
  private isRecording = false;

  // ローカルとリモートのストリームをミックスして録画
  async startRecording(
    localStream: MediaStream,
    remoteStream: MediaStream
  ): Promise<void> {
    // AudioContextでストリームをミックス
    const audioContext = new AudioContext();
    const destination = audioContext.createMediaStreamDestination();

    // ローカル音声
    if (localStream.getAudioTracks().length > 0) {
      const localSource = audioContext.createMediaStreamSource(
        new MediaStream(localStream.getAudioTracks())
      );
      localSource.connect(destination);
    }

    // リモート音声
    if (remoteStream.getAudioTracks().length > 0) {
      const remoteSource = audioContext.createMediaStreamSource(
        new MediaStream(remoteStream.getAudioTracks())
      );
      remoteSource.connect(destination);
    }

    // 映像はリモートストリームを録画（相手の映像）
    const mixedStream = new MediaStream([
      ...remoteStream.getVideoTracks(),
      ...destination.stream.getAudioTracks(),
    ]);

    this.mediaRecorder = new MediaRecorder(mixedStream, {
      mimeType: 'video/webm;codecs=vp8,opus',
      videoBitsPerSecond: 2500000,
    });

    this.mediaRecorder.ondataavailable = (event) => {
      if (event.data.size > 0) {
        this.recordedChunks.push(event.data);
      }
    };

    this.mediaRecorder.start(1000); // 1秒ごとにチャンクを取得
    this.isRecording = true;
  }

  async stopRecording(): Promise<Blob | null> {
    return new Promise((resolve) => {
      if (!this.mediaRecorder || !this.isRecording) {
        resolve(null);
        return;
      }

      this.mediaRecorder.onstop = () => {
        const blob = new Blob(this.recordedChunks, {
          type: 'video/webm',
        });
        this.recordedChunks = [];
        this.isRecording = false;
        resolve(blob);
      };

      this.mediaRecorder.stop();
    });
  }
}
```

録画データは Supabase Storage にアップロードして保存するのが効率的です。大容量の動画ファイルはマルチパートアップロードで分割送信し、ストレージのバケットポリシーで参加者のみがアクセスできるよう制限しましょう。

## 本番運用で押さえるべきポイント

### TURN サーバーの選定

本番環境では、信頼性の高い TURN サーバーが不可欠です。企業のファイアウォール内からの接続や、モバイル回線の NAT タイプが Symmetric NAT の場合、STUN だけでは接続できないケースが 10〜15% 程度発生します。

主な選択肢としては、**Cloudflare TURN**（低遅延・グローバル分散）、**Twilio Network Traversal Service**（従量課金・実績豊富）、**coturn**（OSS・セルフホスト）があります。コスト面ではセルフホストの coturn が有利ですが、運用負荷を考えるとマネージドサービスが現実的です。

### パーミッション管理のベストプラクティス

カメラとマイクのパーミッションは、ユーザー体験に直結する重要なポイントです。いきなりパーミッションダイアログを表示するのではなく、なぜ必要なのかを事前に説明する「プリパーミッション」画面を挟むことで許可率を大幅に向上できます。

```typescript
import { Platform, PermissionsAndroid } from 'react-native';
import { check, request, PERMISSIONS, RESULTS } from 'react-native-permissions';

async function requestMediaPermissions(): Promise<boolean> {
  if (Platform.OS === 'ios') {
    const cameraStatus = await request(PERMISSIONS.IOS.CAMERA);
    const micStatus = await request(PERMISSIONS.IOS.MICROPHONE);
    return (
      cameraStatus === RESULTS.GRANTED && micStatus === RESULTS.GRANTED
    );
  }

  if (Platform.OS === 'android') {
    const grants = await PermissionsAndroid.requestMultiple([
      PermissionsAndroid.PERMISSIONS.CAMERA,
      PermissionsAndroid.PERMISSIONS.RECORD_AUDIO,
    ]);
    return (
      grants[PermissionsAndroid.PERMISSIONS.CAMERA] ===
        PermissionsAndroid.RESULTS.GRANTED &&
      grants[PermissionsAndroid.PERMISSIONS.RECORD_AUDIO] ===
        PermissionsAndroid.RESULTS.GRANTED
    );
  }

  return false;
}
```

エラーハンドリングの設計パターンについてさらに深く知りたい方は、[Rork Max エラーハンドリング・デバッグ・本番監視の実践パターン](/articles/rork-dev/rork-max-error-handling-debugging-production-monitoring)が参考になります。

## よくある質問（FAQ）

### Q1: WebRTC の通話でどのくらいのデータ通信量が発生しますか？

映像品質によりますが、720p・30fps の 1 対 1 ビデオ通話で、1 時間あたり約 1.5〜2.5 GB のデータ通信量が発生します。アダプティブビットレート制御を実装することで、ネットワーク状況に応じて自動的に画質が調整され、モバイル回線でも安定した通話が可能です。音声のみの通話であれば、1 時間あたり約 30〜50 MB に抑えられます。

### Q2: STUN/TURN サーバーなしでも P2P 接続は可能ですか？

同じローカルネットワーク内（同じ Wi-Fi など）であれば、STUN/TURN サーバーなしでも接続可能なケースがあります。しかし、インターネット経由の通信では NAT（Network Address Translation）が介在するため、STUN サーバーによるパブリック IP の発見が必要です。さらに、Symmetric NAT 環境では TURN サーバーによるリレーが必須となります。本番アプリでは必ず TURN サーバーを用意してください。

### Q3: WebRTC と Twilio / Agora などの有料サービスの違いは何ですか？

WebRTC は低レベルの通信プロトコルで、シグナリングやサーバーインフラを自前で構築する必要があります。Twilio Video や Agora はこれらをフルマネージドで提供するサービスです。自前実装はカスタマイズ性とコスト面で有利ですが、開発・運用コストがかかります。月間の通話時間が少ない場合や、早期に市場投入したい場合はマネージドサービスが現実的な選択肢です。

### Q4: グループ通話の最大参加人数はどのくらいですか？

P2P 方式では 3〜4 人が実用的な上限です。SFU を導入すれば、クライアント端末のスペックに依存しますが 20〜50 人程度までスケールできます。ただし、モバイルアプリの場合はバッテリー消費やメモリ制約を考慮して、映像表示は 4〜9 人のグリッドレイアウトにし、それ以上はアクティブスピーカーのみを大きく表示する設計が推奨されます。

## まとめ

この記事では、Rork Max で WebRTC ベースのビデオ通話アプリを構築する方法を、P2P 接続の基礎からグループ通話、画面共有、録画機能まで体系的に解説しましました。

WebRTC の技術的な複雑さは決して小さくありませんが、Supabase Realtime をシグナリングに活用し、Rork Max の AI コード生成で UI を効率的に構築することで、個人開発者でもプロダクション品質のビデオ通話機能を実現できます。

まずは 1 対 1 の通話機能から実装を始め、ユーザーのフィードバックを得ながらグループ通話や画面共有へと段階的に拡張していくアプローチがおすすめです。

リアルタイム通信の基盤を深く理解したい方には「[ハイパフォーマンス ブラウザネットワーキング（Ilya Grigorik 著）](https://www.amazon.co.jp/dp/4873116767/?tag=pinocchio-22)」がおすすめです。WebRTC の内部アーキテクチャを含め、ネットワークプロトコルの本質を学べる一冊です。
