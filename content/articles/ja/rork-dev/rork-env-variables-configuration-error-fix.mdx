## Rorkアプリで環境変数がundefinedになるエラーとは

Rorkで生成したアプリを開発していると、API キーやバックエンドの URL を `.env` ファイルに設定したはずなのに、アプリ側で `undefined` として読み込まれるトラブルに遭遇することがあります。これは Expo / React Native 特有の環境変数の仕組みを正しく理解していないと解決が難しく、多くの開発者がつまずくポイントです。

このガイドでは、Rork アプリで環境変数が機能しない代表的なパターンを網羅し、原因の特定から解決までを Step by Step で解説します。

## 環境変数が読み込めない主な原因

Rorkアプリ（Expo / React Native ベース）で環境変数に問題が起きる原因は、大きく分けて以下の5つです。

**1. `EXPO_PUBLIC_` プレフィックスの欠落**

Expo SDK 49 以降では、クライアント側（アプリ内）で参照する環境変数には必ず `EXPO_PUBLIC_` プレフィックスが必要です。このプレフィックスがない変数はビルド時にバンドルに含まれず、アプリから参照できません。

**2. `.env` ファイルの配置ミス**

`.env` ファイルがプロジェクトルートではなく、サブディレクトリに配置されていると読み込みに失敗します。

**3. Metro バンドラーのキャッシュ**

`.env` を変更した後に Metro のキャッシュをクリアしないと、古い値が使われ続けます。

**4. ビルド時変数とランタイム変数の混同**

EAS Build で使う環境変数（`eas.json` の `env`）と、ローカル開発で使う `.env` ファイルは別のメカニズムです。混同するとビルドは通るがアプリ内で変数が読めない、という状況になります。

**5. EAS Secrets の設定ミス**

EAS Secrets に登録した変数が正しくビルドプロセスに渡されていないケースです。

## Step by Stepの解決手順

### ステップ1: `.env` ファイルの確認と修正

まず、プロジェクトルートに `.env` ファイルが正しく配置されているか確認します。

```bash
# プロジェクトルートにいることを確認
ls -la .env*

# .env ファイルの内容を確認
cat .env
```

正しい `.env` ファイルの形式は以下の通りです。

```bash
# ✅ 正しい形式（クライアント側で使う変数）
EXPO_PUBLIC_API_URL=https://api.example.com
EXPO_PUBLIC_SUPABASE_URL=https://your-project.supabase.co
EXPO_PUBLIC_SUPABASE_ANON_KEY=your_anon_key_here

# ✅ 正しい形式（サーバー側・ビルド時のみ使う変数）
SENTRY_AUTH_TOKEN=your_sentry_token
EAS_PROJECT_ID=your_project_id
```

よくある間違いパターンを確認してください。

```bash
# ❌ 間違い: EXPO_PUBLIC_ プレフィックスがない
API_URL=https://api.example.com
SUPABASE_URL=https://your-project.supabase.co

# ❌ 間違い: 値をクォーテーションで囲んでいる
EXPO_PUBLIC_API_URL="https://api.example.com"

# ❌ 間違い: スペースが含まれている
EXPO_PUBLIC_API_URL = https://api.example.com
```

### ステップ2: アプリ内での参照方法を確認

Expo では `process.env` を通じて環境変数にアクセスします。

```typescript
// ✅ 正しい参照方法
const apiUrl = process.env.EXPO_PUBLIC_API_URL;
const supabaseUrl = process.env.EXPO_PUBLIC_SUPABASE_URL;

// デバッグ用: 値が正しく読み込まれているか確認
console.log('API URL:', process.env.EXPO_PUBLIC_API_URL);
console.log('Supabase URL:', process.env.EXPO_PUBLIC_SUPABASE_URL);
```

`react-native-dotenv` や `expo-constants` 経由で読み込む方法もありますが、Expo SDK 49 以降では `process.env` が推奨されています。

```typescript
// ⚠️ 旧方式（非推奨だが動作する）
import Constants from 'expo-constants';
const apiUrl = Constants.expoConfig?.extra?.apiUrl;

// ✅ 新方式（推奨）
const apiUrl = process.env.EXPO_PUBLIC_API_URL;
```

### ステップ3: Metro キャッシュのクリア

`.env` ファイルを変更した後は、必ず Metro バンドラーのキャッシュをクリアしてから再起動します。

```bash
# Metro キャッシュをクリアして再起動
npx expo start --clear

# より徹底的にクリアする場合
rm -rf node_modules/.cache
npx expo start --clear
```

キャッシュをクリアせずに `npx expo start` を実行すると、変更前の `.env` の値が使われ続けます。これは環境変数がビルド時に静的に埋め込まれるためです。

### ステップ4: EAS Build での環境変数設定

EAS Build（クラウドビルド）で環境変数を使う場合は、`eas.json` で明示的に設定する必要があります。

```json
{
  "build": {
    "development": {
      "env": {
        "EXPO_PUBLIC_API_URL": "https://dev-api.example.com",
        "EXPO_PUBLIC_SUPABASE_URL": "https://your-project.supabase.co"
      }
    },
    "preview": {
      "env": {
        "EXPO_PUBLIC_API_URL": "https://staging-api.example.com",
        "EXPO_PUBLIC_SUPABASE_URL": "https://your-project.supabase.co"
      }
    },
    "production": {
      "env": {
        "EXPO_PUBLIC_API_URL": "https://api.example.com",
        "EXPO_PUBLIC_SUPABASE_URL": "https://your-project.supabase.co"
      }
    }
  }
}
```

### ステップ5: EAS Secrets でシークレットを管理する

API キーなどの機密情報は `eas.json` に直接書かず、EAS Secrets を使って安全に管理します。

```bash
# EAS Secrets にシークレットを登録
eas secret:create --name EXPO_PUBLIC_SUPABASE_ANON_KEY \
  --value "your_anon_key_here" \
  --scope project

# 登録済みのシークレット一覧を確認
eas secret:list

# シークレットを更新する場合
eas secret:delete --name EXPO_PUBLIC_SUPABASE_ANON_KEY
eas secret:create --name EXPO_PUBLIC_SUPABASE_ANON_KEY \
  --value "new_anon_key_here" \
  --scope project
```

EAS Secrets に登録した値は、EAS Build 実行時に自動的に環境変数として注入されます。ローカル開発では `.env` ファイル、クラウドビルドでは EAS Secrets という使い分けが基本です。

## 正しく設定できたかの確認方法

修正が完了したら、以下の手順で環境変数が正しく読み込まれているか検証します。

```typescript
// app/debug-env.tsx（デバッグ用一時ファイル）
import { View, Text, ScrollView } from 'react-native';

export default function DebugEnvScreen() {
  const envVars = {
    API_URL: process.env.EXPO_PUBLIC_API_URL,
    SUPABASE_URL: process.env.EXPO_PUBLIC_SUPABASE_URL,
    SUPABASE_ANON_KEY: process.env.EXPO_PUBLIC_SUPABASE_ANON_KEY
      ? '✅ Set (hidden)'
      : '❌ Not set',
  };

  return (
    <ScrollView style={{ padding: 20 }}>
      <Text style={{ fontSize: 20, fontWeight: 'bold', marginBottom: 16 }}>
        Environment Variables Debug
      </Text>
      {Object.entries(envVars).map(([key, value]) => (
        <View key={key} style={{ marginBottom: 12 }}>
          <Text style={{ fontWeight: 'bold' }}>{key}</Text>
          <Text style={{ color: value ? '#22c55e' : '#ef4444' }}>
            {value || 'undefined ❌'}
          </Text>
        </View>
      ))}
    </ScrollView>
  );
}
```

確認後は必ずこのデバッグ画面を削除してください。機密情報の漏洩を防ぐために、本番ビルドには含めないことが重要です。

## 再発防止のベストプラクティス

環境変数のトラブルを繰り返さないために、以下のルールをプロジェクトに導入しましょう。

**`.env.example` をリポジトリに含める**

チーム開発や将来の自分のために、必要な環境変数のテンプレートをリポジトリに含めておきます。

```bash
# .env.example（値は空またはダミー）
EXPO_PUBLIC_API_URL=
EXPO_PUBLIC_SUPABASE_URL=
EXPO_PUBLIC_SUPABASE_ANON_KEY=
```

**`.gitignore` に `.env` を追加する**

機密情報を含む `.env` ファイルは必ず `.gitignore` に追加して、リポジトリにコミットされないようにします。

```bash
# .gitignore
.env
.env.local
.env.production
```

**型安全な環境変数アクセス**

TypeScript プロジェクトでは、環境変数を型安全にアクセスするユーティリティを作成すると、設定ミスを早期に発見できます。

```typescript
// utils/env.ts
function getEnvVar(key: string): string {
  const value = process.env[key];
  if (\!value) {
    console.warn(`⚠️ Environment variable ${key} is not set`);
  }
  return value ?? '';
}

export const ENV = {
  API_URL: getEnvVar('EXPO_PUBLIC_API_URL'),
  SUPABASE_URL: getEnvVar('EXPO_PUBLIC_SUPABASE_URL'),
  SUPABASE_ANON_KEY: getEnvVar('EXPO_PUBLIC_SUPABASE_ANON_KEY'),
} as const;
```

アプリ全体でこの `ENV` オブジェクトを使えば、変数名のタイプミスを防ぎ、未設定の変数を早期に検出できます。

## よくある質問（FAQ）

### Q: `EXPO_PUBLIC_` プレフィックスなしでクライアント側から環境変数を読み込む方法はありますか？

`EXPO_PUBLIC_` プレフィックスなしの環境変数をクライアント側から直接読むことはできません。これはセキュリティ上の設計です。サーバー側のシークレット（データベースのパスワードなど）がアプリのバンドルに含まれることを防いでいます。どうしてもプレフィックスを付けたくない場合は、`app.config.js` の `extra` フィールド経由で渡す方法がありますが、`process.env` 方式の方がシンプルで推奨されます。

### Q: ローカルでは動くのに EAS Build でだけ環境変数が undefined になります。

ローカルでは `.env` ファイルから読み込まれますが、EAS Build ではクラウド上でビルドが実行されるため `.env` ファイルは存在しません。`eas.json` の `env` フィールドに値を設定するか、`eas secret:create` で EAS Secrets に登録する必要があります。特に `EXPO_PUBLIC_` で始まる変数は `eas.json` の `env` に書くのが確実です。

### Q: `.env.local` と `.env.production` の優先順位はどうなりますか？

Expo では `.env.local` が最も優先度が高く、次に `.env` が読み込まれます。ただし `.env.production` のような環境別ファイルの自動切り替えは、Expo の標準機能ではサポートされていません。環境ごとに異なる値を使いたい場合は、`eas.json` の `env` フィールドでビルドプロファイルごとに設定するのが正しいアプローチです。

### Q: Supabase の `service_role_key` を環境変数に入れても大丈夫ですか？

`service_role_key` は管理者権限を持つ鍵であり、絶対にクライアント側のコードに含めてはいけません。`EXPO_PUBLIC_` プレフィックスを付けて `.env` に入れると、アプリのバンドルに埋め込まれてしまい、リバースエンジニアリングで抽出される危険があります。`service_role_key` はサーバーサイド（Supabase Edge Functions や自前の API サーバー）でのみ使用してください。

## まとめ

Rorkアプリで環境変数が読み込めない問題は、`EXPO_PUBLIC_` プレフィックスの欠落、Metro キャッシュ、EAS Build とローカル開発の設定の違いが主な原因です。この記事で紹介した手順に沿って確認すれば、ほとんどのケースで問題を解決できます。

環境変数の管理は地味な作業ですが、セキュリティと開発効率の両面で非常に重要です。`.env.example` によるテンプレート化と TypeScript による型安全なアクセスを導入して、チーム全体で安定した開発環境を維持しましょう。
