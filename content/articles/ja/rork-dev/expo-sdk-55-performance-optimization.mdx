## Expo SDK 55とは？パフォーマンス革新の背景

2026年2月にリリースされたExpo SDK 55は、単なるマイナーアップデートではなく、モバイルアプリのパフォーマンスに革新をもたらす重要なリリースです。最大の特徴は**Hermes V1がデフォルトエンジン**になったこと。これにより、すべてのExpoアプリが自動的にパフォーマンス向上の恩恵を受けます。

従来のJavaScriptエンジンと比べてHermes V1は、実行速度が大幅に向上し、メモリ使用量を削減。特に低スペックなAndroidデバイスでの実行が劇的に改善されます。さらにExpo Observe、EASビルドキャッシング、react-native-screensのアップグレードなど、開発体験とアプリ品質を同時に向上させる機能が多数搭載されています。

この記事では、Expo SDK 55の最新機能を実装し、あなたのRorkアプリをさらに高速化・高最適化する方法を詳しく解説します。

## Hermes V1エンジンの性能向上と実装方法

### Hermes V1がもたらす実質的なメリット

Hermes V1は以下の大幅な改善をもたらします。

| 改善項目 | 効果 |
|---------|------|
| 実行速度 | 従来比で平均15～25%高速化 |
| メモリ使用量 | 初期化時メモリを約30%削減 |
| 起動時間 | アプリ起動時間を500ms～1秒短縮 |
| ES6サポート | async/await、const/let、クラス構文が完全サポート |

Hermes V1の動作原理は、JavaScriptをバイトコードにプリコンパイルすることで、実行時の解析処理を最小限にする点です。これは特にCPUリソースが限定されたAndroidデバイスで威力を発揮します。

### Expo SDK 55での標準設定と確認方法

Expo SDK 55を使用している場合、Hermes V1は自動的に有効になっています。設定を確認するには、以下のコマンドで現在のExpoバージョンを確認してください。

```bash
expo --version
# 出力例: 52.0.0 (SDK 55対応)
```

`app.json`にHermes設定を明示的に記述することもできます。

```json
{
  "expo": {
    "plugins": [
      [
        "expo-build-properties",
        {
          "android": {
            "usesCleartextTraffic": false,
            "enableHermes": true
          },
          "ios": {
            "deploymentTarget": "13.0"
          }
        }
      ]
    ]
  }
}
```

Hermes有効化後は、Android側でバイトコード生成が自動実行され、APKサイズが約2～3MB削減されることが多いです。

## EASビルドキャッシングで開発速度3倍高速化

### キャッシング機能がもたらす時間短縮

EASビルドキャッシングは、同一の依存関係・設定でビルドする際に、前回のビルド結果を再利用する機能です。その効果は劇的です。

- **初回ビルド**: 10～15分
- **キャッシュ利用時**: 3～5分（約70%削減）

特にCI/CDパイプラインで頻繁にビルドする環境では、月単位で数十時間の削減につながります。

### EASキャッシング設定と使用例

EASキャッシングはEAS Build v2以上で標準提供されており、追加の設定は不要です。ただし、キャッシュの効率を最大化するには以下の設定を推奨します。

```bash
# eas.json の設定例
{
  "build": {
    "production": {
      "node": "20.10.0",
      "npm": "10.2.4",
      "cache": {
        "cacheDisabled": false,
        "customPaths": [
          ".next",
          "build",
          "dist"
        ]
      }
    }
  }
}
```

customPathsに自身のビルド成果物ディレクトリを指定することで、さらなる高速化が実現できます。

```bash
eas build --platform ios --cache
eas build --platform android --cache
```

キャッシュは最大14日間保持されるため、2週間以内のビルド再実行時に有効です。

## モーダル表示の改善と react-native-screens 4.23.0

### なぜモーダル最適化が重要なのか

Rorkアプリでは、ボトムシート、ダイアログ、ナビゲーションスタックなど、モーダル表示が頻繁に使用されます。react-native-screens 4.23.0では、ネイティブスクリーンライフサイクルが完全に改善され、以下の効果が得られます。

- モーダル開閉時のジャンク削減
- メモリリークの防止
- スムーズなアニメーション実行

### 実装例：ボトムシートの正しい構築

react-native-screenを活用した最新のボトムシート実装例を示します。

```typescript
import { NavigationContainer } from '@react-navigation/native';
import { createNativeStackNavigator } from '@react-navigation/native-stack';
import { GestureHandlerRootView } from 'react-native-gesture-handler';
import BottomSheet from '@gorhom/bottom-sheet';

const Stack = createNativeStackNavigator();

export default function ModalScreen() {
  const [isOpen, setIsOpen] = React.useState(false);
  const snapPoints = React.useMemo(() => [200, 400], []);

  return (
    <GestureHandlerRootView style={{ flex: 1 }}>
      <NavigationContainer>
        <Stack.Navigator
          screenOptions={{
            presentation: 'modal',
            animationEnabled: true,
            cardStyle: { backgroundColor: 'transparent' },
          }}
        >
          <Stack.Group>
            <Stack.Screen name="Home" component={HomeScreen} />
          </Stack.Group>
          <Stack.Group
            screenOptions={{
              presentation: 'transparentModal',
              cardOverlayEnabled: true,
              cardStyle: { backgroundColor: 'rgba(0,0,0,0.5)' },
            }}
          >
            <Stack.Screen name="Modal" component={ModalContent} />
          </Stack.Group>
        </Stack.Navigator>
      </NavigationContainer>
      
      {isOpen && (
        <BottomSheet snapPoints={snapPoints} onClose={() => setIsOpen(false)}>
          <View style={{ flex: 1, padding: 16 }}>
            <Text>ボトムシートコンテンツ</Text>
          </View>
        </BottomSheet>
      )}
    </GestureHandlerRootView>
  );
}
```

react-native-screens 4.23.0では、モーダルスタックのメモリ管理が自動化され、複数のモーダルを積み重ねた場合でもメモリリークが発生しにくくなります。

## Expo Observe：本番環境のパフォーマンス監視

### Expo Observeとは

Expo Observeはプライベートプレビュー段階の新機能で、本番アプリのパフォーマンスをリアルタイムで監視するダッシュボード機能です。以下のメトリクスを自動収集します。

- フレームレート（FPS）
- メモリ使用量の推移
- CPUバッテリー消費率
- ネットワークリクエスト時間
- カスタムメトリクス（ユーザー定義）

### Expo Observeの有効化と基本設定

Expo Observeを使用するには、EAS Accountでプロジェクトを登録し、以下のコマンドで初期化します。

```bash
eas update:configure
expo prebuild --clean
```

その後、アプリコード内で以下のようにObserve SDKを初期化します。

```typescript
import { useExpoObserve } from 'expo-observe';

export default function App() {
  React.useEffect(() => {
    // パフォーマンス監視を開始
    const observer = useExpoObserve();
    
    // カスタムメトリクスを記録
    observer.recordMetric('user_action_duration', 1250);
    observer.recordMetric('api_response_time', 350);
    
    return () => observer.cleanup();
  }, []);

  return <MainApp />;
}
```

データはEAS Dashboardにアップロードされ、任意の時間幅でパフォーマンス推移を確認できます。

## android-blur の効率化：RenderNode APIの採用

### 背景ぼかし処理の課題と改善

Rorkアプリで背景ぼかし効果（glassmorphism）を使用する場合、従来はNativeModuleで複雑なフレームバッファ操作が必要でした。Expo SDK 55ではAndroid 12以上でRenderNode APIが活用され、GPU最適化が実現しています。

効果は以下の通りです。

| 環境 | 処理時間 | メモリ使用量 |
|-----|---------|-----------|
| 従来手法 | 16ms/フレーム | 4.5MB |
| RenderNode API | 4ms/フレーム | 1.2MB |

## FAQ：よくある質問と回答

**Q1: Hermes V1は本当に必要ですか？デバッグに支障はありませんか？**

A: Hermes V1はプロダクション環境での性能向上に特化しており、デバッグモードではJavaScriptエンジンの自動選択が行われます。デバッグ中はHermesを無効化することも可能です。パフォーマンス測定の観点からはHermes有効化を強く推奨します。

**Q2: EASキャッシュはいつ無効化すべきですか？**

A: キャッシュは以下の場合に手動クリアを推奨します：(1) ネイティブ依存関係の更新後、(2) node_modules削除後、(3) 不可解なビルドエラー発生時。手動クリアは `eas build --clear-cache` で実行できます。

**Q3: Expo Observeはどの程度のオーバーヘッドがありますか？**

A: Expo Observeのオーバーヘッドは平均1～2%程度のCPU使用量追加に抑制されています。本番環境での採用は安全です。

**Q4: 古いバージョンのExpoからSDK 55へのアップグレードは安全ですか？**

A: SDK 55への移行は一般的に安全ですが、react-native-screens 4.23.0の仕様変更に伴い、カスタムモーダル実装が影響を受ける可能性があります。アップグレード前に十分なテストを推奨します。

**Q5: Hermes V1とJavaScript新構文（async/await）の組み合わせは安定していますか？**

A: 完全に安定しており、既存のPromiseベースコードも問題なく動作します。
