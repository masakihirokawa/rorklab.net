## Rork アプリで画像・メディアが表示されない症状とは

Rork で生成したアプリを実機やシミュレーターで動作確認していると、「画像が一部だけ表示されない」「読み込みインジケーターが回り続ける」「大量の画像を表示するとアプリがクラッシュする」といった問題に遭遇することがあります。

これらの画像・メディア読み込みエラーは、原因が多岐にわたるため特定しづらいのが厄介なところです。ネットワークの問題なのか、画像フォーマットの互換性なのか、キャッシュの破損なのか、あるいはメモリ不足なのか——この記事では、発生パターンごとに原因を分析し、具体的な解決手順をステップバイステップで解説していきます。

Rork が生成する React Native（Expo）コードは `Image` コンポーネントや `expo-image` ライブラリを使用しますが、それぞれに固有の注意点があります。本記事を参考に、画像表示トラブルを効率的に解消していきましょう。

## 画像が表示されない主な原因パターン

画像の読み込み失敗にはいくつかの典型的なパターンがあります。まずは自分が直面している問題がどのパターンに該当するかを確認してください。

**パターン1: URL・パスの指定ミス**

最も基本的な原因です。リモート画像のURLにタイプミスがある、ローカル画像のパスが間違っている、あるいは `require()` の記述が正しくないケースです。特に Rork が自動生成したコードでは、画像パスがプレースホルダーのまま残っていることがあります。

**パターン2: HTTPS/HTTPの混在（iOS App Transport Security）**

iOS では App Transport Security（ATS）により、HTTP（非暗号化）の通信がデフォルトでブロックされます。画像サーバーが HTTPS に対応していない場合、iOS 実機で画像が表示されません。Android では問題なく表示されるため、iOS 固有の問題として見落としがちです。

**パターン3: 画像フォーマットの非互換**

WebP 画像は Android では問題なく表示されますが、古い iOS バージョン（14以前）ではサポートが不完全です。また、HEIC/HEIF フォーマットの画像を Android で表示しようとすると失敗するケースがあります。

**パターン4: CORS（クロスオリジン）エラー**

Web プレビューで画像が表示されない場合、CORS ポリシーが原因であることが多いです。画像サーバー側で適切な `Access-Control-Allow-Origin` ヘッダーが設定されていないと、ブラウザがリクエストをブロックします。

**パターン5: メモリ不足によるクラッシュ**

高解像度の画像を大量にリスト表示すると、メモリ使用量が急増してアプリがクラッシュします。これは特に `FlatList` や `ScrollView` 内で多数の画像を一度にレンダリングする場合に発生しやすい問題です。

## 解決手順 Step 1: URL とパスの検証

まず、画像の URL やパスが正しいかを確認します。以下のデバッグ用コンポーネントを使って、画像の読み込み状態を可視化しましょう。

```tsx
// components/DebugImage.tsx
// 画像の読み込み状態をデバッグするためのラッパーコンポーネント
import React, { useState } from 'react';
import { Image, View, Text, StyleSheet } from 'react-native';

interface DebugImageProps {
  uri: string;
  width: number;
  height: number;
}

export function DebugImage({ uri, width, height }: DebugImageProps) {
  const [status, setStatus] = useState<'loading' | 'success' | 'error'>('loading');
  const [errorMsg, setErrorMsg] = useState('');

  return (
    <View style={styles.container}>
      <Image
        source={{ uri }}
        style={{ width, height }}
        onLoad={() => setStatus('success')}
        onError={(e) => {
          setStatus('error');
          setErrorMsg(e.nativeEvent.error || 'Unknown error');
        }}
      />
      {status === 'loading' && <Text style={styles.info}>読み込み中: {uri}</Text>}
      {status === 'error' && (
        <Text style={styles.error}>エラー: {errorMsg}{'\n'}URL: {uri}</Text>
      )}
    </View>
  );
}

const styles = StyleSheet.create({
  container: { marginVertical: 8 },
  info: { fontSize: 12, color: '#666' },
  error: { fontSize: 12, color: '#ff4444', backgroundColor: '#fff0f0', padding: 8 },
});

// 期待する出力:
// 正常時 → 画像が表示され、テキストは非表示
// エラー時 → 「エラー: [具体的なエラーメッセージ]」と URL が赤字で表示される
```

このコンポーネントを使うことで、どの画像の読み込みが失敗しているか、そのエラー内容が何かを即座に把握できます。

ローカル画像の場合は、`require()` のパスが正しいか確認します。

```tsx
// ✅ 正しいローカル画像の読み込み
<Image source={require('../assets/images/logo.png')} />

// ❌ よくある間違い — パスが存在しない
<Image source={require('./images/logo.png')} />

// ❌ よくある間違い — 変数を使った require は動的解決できない
const imageName = 'logo';
<Image source={require(`../assets/images/${imageName}.png`)} /> // 動作しない
```

## 解決手順 Step 2: ネットワークとセキュリティ設定の確認

iOS で HTTP 画像が読み込めない場合は、`app.json`（または `app.config.js`）で ATS の例外設定を追加します。

```json
{
  "expo": {
    "ios": {
      "infoPlist": {
        "NSAppTransportSecurity": {
          "NSAllowsArbitraryLoads": false,
          "NSExceptionDomains": {
            "your-image-server.com": {
              "NSExceptionAllowsInsecureHTTPLoads": true,
              "NSIncludesSubdomains": true
            }
          }
        }
      }
    }
  }
}
```

ただし、`NSAllowsArbitraryLoads: true` を設定するのはセキュリティ上推奨されません。できる限り画像サーバーを HTTPS に移行し、個別のドメインのみ例外として許可するのがベストプラクティスです。

CORS エラーの場合は、画像サーバー側のレスポンスヘッダーを確認します。Supabase Storage や Cloudflare R2 を使っている場合は、バケットの CORS 設定を見直しましょう。

```bash
# 画像URLのレスポンスヘッダーを確認するコマンド
curl -I https://your-bucket.supabase.co/storage/v1/object/public/images/photo.jpg

# 期待する出力（CORS設定済みの場合）:
# Access-Control-Allow-Origin: *
# Content-Type: image/jpeg
# Content-Length: 245890
```

## 解決手順 Step 3: expo-image への移行で安定性を向上

React Native 標準の `Image` コンポーネントにはキャッシュ制御の不備やメモリ管理の弱さがあります。Rork が生成するプロジェクトで画像表示に問題が頻発する場合は、`expo-image` への移行を検討してください。

```bash
# expo-image のインストール
npx expo install expo-image
```

```tsx
// expo-image を使った画像表示（キャッシュ自動管理付き）
import { Image } from 'expo-image';

// プレースホルダー付きの安定した画像表示
function OptimizedImage({ uri }: { uri: string }) {
  return (
    <Image
      source={{ uri }}
      style={{ width: 300, height: 200 }}
      contentFit="cover"
      placeholder={require('../assets/placeholder.png')}
      transition={200}
      cachePolicy="memory-disk"
      onError={(e) => console.warn('画像読み込みエラー:', e.error)}
    />
  );
}

// 期待する動作:
// 1. placeholder 画像が即座に表示される
// 2. リモート画像の読み込みが完了すると 200ms のフェードで切り替わる
// 3. 2回目以降はキャッシュから瞬時に表示される
```

`expo-image` は内部で `SDWebImage`（iOS）と `Glide`（Android）を使用しており、標準の `Image` コンポーネントより大幅に安定しています。主なメリットは以下のとおりです。

- **自動キャッシュ管理**: メモリキャッシュとディスクキャッシュを自動で最適化
- **プレースホルダー対応**: 読み込み中にぼかし画像やスケルトンを表示可能
- **WebP/AVIF サポート**: モダンな画像フォーマットをクロスプラットフォームで対応
- **メモリ効率**: 画面外の画像を自動でメモリから解放

## 解決手順 Step 4: 大量画像のメモリ管理

`FlatList` や `SectionList` で画像を大量表示する場合、適切なメモリ管理を行わないとアプリがクラッシュします。以下の設定を確認してください。

```tsx
// メモリ効率の良い画像リスト表示
import { FlatList } from 'react-native';
import { Image } from 'expo-image';

function ImageGallery({ images }: { images: string[] }) {
  return (
    <FlatList
      data={images}
      keyExtractor={(item, index) => `img-${index}`}
      // ✅ 画面外の要素を早めに解放してメモリを節約
      removeClippedSubviews={true}
      // ✅ レンダリングする画面外アイテム数を制限
      windowSize={5}
      // ✅ 初期表示数を制限
      initialNumToRender={6}
      // ✅ バッチレンダリング数を制限
      maxToRenderPerBatch={4}
      renderItem={({ item }) => (
        <Image
          source={{ uri: item }}
          style={{ width: '100%', height: 200 }}
          contentFit="cover"
          // ✅ 低品質プレースホルダーで体感速度向上
          placeholder={{ blurhash: 'LKN]Rv%2Tw=w]~RBVZRi};RPxuwH' }}
          recyclingKey={item}
        />
      )}
    />
  );
}
```

特に重要なのは `removeClippedSubviews` と `windowSize` の設定です。デフォルトでは `FlatList` は画面外の要素もメモリに保持し続けますが、これらの設定で不要なメモリ消費を抑制できます。

画像の解像度も見落としがちな問題です。サーバーから 4000×3000 ピクセルの元画像をそのまま配信していると、1枚あたり約 48MB のメモリを消費します。表示サイズに合わせたリサイズ済み画像を配信するか、CDN のリサイズ機能を活用しましょう。

## 確認方法 — 問題が解決できたかの検証手順

修正を適用した後は、以下の手順で問題が解決されたかを検証します。

1. **Expo Go でのプレビュー確認**: `npx expo start` で開発サーバーを起動し、iOS・Android 両方のデバイスで画像表示を確認する
2. **ネットワーク状態のテスト**: 機内モード→復帰後に画像が再読み込みされるか確認。キャッシュが機能していれば、2回目の表示は瞬時に行われる
3. **メモリ使用量の監視**: React Native Debugger や Xcode Instruments の Memory グラフで、画像リストをスクロールした際のメモリ推移を確認する
4. **エラーログの確認**: `console.warn` や `onError` コールバックで新たなエラーが発生していないことを確認する

```bash
# React Native のログをリアルタイムで確認
npx react-native log-ios
# または
npx react-native log-android

# 画像関連のエラーだけをフィルタリング
npx react-native log-ios 2>&1 | grep -i "image\|photo\|media\|load"
```

## 予防策 — 画像・メディアエラーの再発を防ぐベストプラクティス

画像関連のトラブルを未然に防ぐための設計指針をまとめます。

**1. 画像配信の最適化を設計段階で組み込む**

開発初期の段階で、画像のリサイズ・圧縮・フォーマット変換のパイプラインを構築しておきましょう。Supabase Storage や Cloudflare R2 を使う場合は、アップロード時に自動リサイズする Edge Function を組み合わせると効果的です。詳しい実装方法は[Rork + Cloudflare R2 でファイルストレージを構築する完全ガイド](/articles/rork-dev/rork-cloudflare-r2-file-storage-guide)で解説しています。

**2. エラーバウンダリで画像エラーを隔離する**

1つの画像の読み込み失敗がアプリ全体をクラッシュさせないよう、画像コンポーネントに fallback UI を設定します。`expo-image` の `placeholder` プロパティや、React の Error Boundary を活用してください。

**3. 画像フォーマットの統一ルールを決める**

プロジェクト内で使用する画像フォーマットを統一し、ドキュメント化しておきます。基本的には WebP を推奨しますが、iOS 14 以前のサポートが必要な場合は JPEG/PNG を併用する戦略を取ります。画像の最適化とキャッシュ戦略については[Rork アプリの画像最適化とキャッシュ戦略](/articles/rork-dev/rork-image-optimization-cache-strategy)が参考になります。

**4. 定期的なキャッシュクリアの仕組みを実装する**

古いキャッシュが破損して画像が表示されなくなるケースがあります。アプリの設定画面に「キャッシュクリア」ボタンを用意しておくと、ユーザー自身で問題を解決できます。

```tsx
// expo-image のキャッシュをクリアする関数
import { Image } from 'expo-image';

async function clearImageCache() {
  await Image.clearDiskCache();
  await Image.clearMemoryCache();
  // キャッシュクリア完了をユーザーに通知
  Alert.alert('完了', '画像キャッシュをクリアしました');
}
```

## よくある質問（FAQ）

### Q1: Rork が生成したコードで画像が一部だけ表示されません。何を確認すべきですか？

まず `DebugImage` コンポーネント（本記事の Step 1 参照）を使って、表示されない画像のエラー内容を特定してください。多くの場合、URL のタイプミスか、画像サーバーの CORS 設定が原因です。Rork が自動生成するプレースホルダー URL（`https://via.placeholder.com/...` など）がそのまま残っている可能性もあるため、すべての `source` プロパティに正しい URL が設定されているかを確認しましょう。

### Q2: Android では表示されるのに iOS だけ画像が出ません。原因は何ですか？

iOS の App Transport Security（ATS）が HTTP リクエストをブロックしている可能性が高いです。本記事の Step 2 で解説している `app.json` の ATS 設定を確認してください。また、画像が HEIC フォーマットの場合、Android 側で表示できないケースもあります。JPEG または WebP での配信に統一するのが安全な対処法です。

### Q3: 画像リストをスクロールするとアプリがクラッシュします。どう修正しますか？

メモリ不足が原因の可能性が高いです。Step 4 で解説した `FlatList` の最適化設定（`removeClippedSubviews`、`windowSize`、`maxToRenderPerBatch`）を適用してください。また、サーバーから配信する画像の解像度を表示サイズに合わせてリサイズすることも重要です。4000×3000 ピクセルの画像をサムネイル（300×200）で表示している場合、画像1枚あたり約 48MB のメモリを無駄に消費しています。

### Q4: expo-image と React Native の Image コンポーネント、どちらを使うべきですか？

新規プロジェクトでは `expo-image` を強く推奨します。キャッシュ管理、プレースホルダー表示、WebP サポート、メモリ効率のすべてにおいて標準 `Image` を上回ります。既存プロジェクトで標準 `Image` を使っている場合も、画像表示に問題が頻発するなら移行を検討してください。API の互換性が高いため、移行コストは比較的低いです。

## まとめ

Rork アプリの画像・メディア読み込みエラーは、原因が多岐にわたるため体系的なアプローチで切り分けることが大切です。URL の検証 → ネットワーク設定の確認 → ライブラリの最適化 → メモリ管理の順で対処していけば、ほとんどの問題は解決できます。

特に `expo-image` への移行は、画像関連のトラブルを大幅に減らせる効果的な対策です。画像ギャラリーや EC アプリなど画像を多用するアプリでは、開発初期の段階で導入しておくことをおすすめします。
