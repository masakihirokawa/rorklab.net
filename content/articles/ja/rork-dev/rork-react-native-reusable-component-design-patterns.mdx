## はじめに — なぜコンポーネント設計が重要なのか

Rork でアプリ開発を進めていくと、画面数が増えるにつれて「同じようなUIパーツを何度も書いている」と感じる場面が出てきます。ボタン、カード、入力フォーム、リストアイテムなど、似た構造のコンポーネントがプロジェクト内に散在すると、修正が必要になったとき全ての箇所を手作業で直す必要があり、バグの温床になりがちです。

再利用可能なコンポーネント設計は、こうした問題を根本から解決します。一度しっかり設計したコンポーネントは、Props を変えるだけで異なる画面で使い回せるため、開発速度が飛躍的に向上します。本記事では、Rork での開発に即した形で、React Native のコンポーネント設計パターンを実践的に解説していきます。

この記事の対象読者は、Rork で1〜2個のアプリを作った経験があり、次のステップとしてコード品質を高めたい方です。

## Atomic Design の考え方を Rork に適用する

コンポーネント設計のフレームワークとして広く知られている **Atomic Design** は、UIを5つの階層に分類する手法です。Rork で生成されるReact Nativeコードにも、この考え方を自然に取り入れることができます。

Rork のプロジェクトでは、以下の3階層を意識するのが現実的です。

**Atoms（原子）** は、それ以上分解できない最小のUI要素です。ボタン、テキスト入力、アイコン、ラベルなどが該当します。

**Molecules（分子）** は、Atomsを組み合わせた小さな機能単位です。検索バー（テキスト入力＋アイコン＋ボタン）、カードコンポーネント（画像＋テキスト＋ボタン）などが該当します。

**Organisms（有機体）** は、Moleculesを組み合わせた画面の大きなセクションです。ヘッダー、フッター、商品リスト、フォーム全体などが該当します。

Rork にプロンプトで指示する際も、「検索バーコンポーネントを作って」ではなく「TextInput と Icon の Atom を組み合わせた SearchBar の Molecule を作って」と階層を意識した指示を出すと、再利用しやすい構造のコードが生成されやすくなります。

## Props 設計の基本 — 柔軟性と型安全性の両立

再利用可能なコンポーネントの鍵は **Props の設計** にあります。Props が適切に設計されていれば、同じコンポーネントを異なるコンテキストで柔軟に使い回せます。

以下は、Rork で生成した汎用ボタンコンポーネントの Props 設計例です。

```typescript
// components/atoms/Button.tsx
import React from 'react';
import { TouchableOpacity, Text, StyleSheet, ViewStyle, TextStyle } from 'react-native';

// Props の型定義 — 必須と任意を明確に分ける
interface ButtonProps {
  label: string;               // 必須: ボタンのラベル
  onPress: () => void;         // 必須: タップ時の処理
  variant?: 'primary' | 'secondary' | 'outline';  // 任意: スタイルバリエーション
  size?: 'small' | 'medium' | 'large';            // 任意: サイズ
  disabled?: boolean;          // 任意: 無効状態
  style?: ViewStyle;           // 任意: カスタムスタイル
}

export const Button: React.FC<ButtonProps> = ({
  label,
  onPress,
  variant = 'primary',    // デフォルト値を設定
  size = 'medium',
  disabled = false,
  style,
}) => {
  return (
    <TouchableOpacity
      onPress={onPress}
      disabled={disabled}
      style={[styles.base, styles[variant], styles[size], disabled && styles.disabled, style]}
    >
      <Text style={[styles.label, styles[`${variant}Label`]]}>
        {label}
      </Text>
    </TouchableOpacity>
  );
};

// 期待される使い方:
// <Button label="送信" onPress={handleSubmit} />
// <Button label="キャンセル" onPress={handleCancel} variant="outline" size="small" />
```

この設計のポイントは3つあります。まず、`label` と `onPress` は必須にしつつ、スタイルに関するプロパティは任意にしてデフォルト値を持たせること。次に、`variant` や `size` にリテラル型を使って、指定できる値を制限すること。そして、`style` プロパティで外部からのカスタマイズ余地を残すことです。

## コンポーネント合成パターン — children と Render Props

より複雑なコンポーネントでは、**コンポーネント合成（Composition）** パターンが威力を発揮します。

```typescript
// components/molecules/Card.tsx
import React, { ReactNode } from 'react';
import { View, StyleSheet, ViewStyle } from 'react-native';

interface CardProps {
  children: ReactNode;       // 内部コンテンツを自由に差し替え可能
  padding?: number;
  shadow?: boolean;
  style?: ViewStyle;
}

export const Card: React.FC<CardProps> = ({
  children,
  padding = 16,
  shadow = true,
  style,
}) => {
  return (
    <View style={[styles.container, shadow && styles.shadow, { padding }, style]}>
      {children}
    </View>
  );
};

// Card.Header, Card.Body, Card.Footer をサブコンポーネントとして定義
Card.Header = ({ children }: { children: ReactNode }) => (
  <View style={styles.header}>{children}</View>
);

Card.Body = ({ children }: { children: ReactNode }) => (
  <View style={styles.body}>{children}</View>
);

Card.Footer = ({ children }: { children: ReactNode }) => (
  <View style={styles.footer}>{children}</View>
);

// 期待される使い方:
// <Card>
//   <Card.Header><Text>タイトル</Text></Card.Header>
//   <Card.Body><Text>本文</Text></Card.Body>
//   <Card.Footer><Button label="詳細" onPress={handlePress} /></Card.Footer>
// </Card>
```

`children` を使うことで、Card の内部構造を利用側が自由に決められます。商品カード、プロフィールカード、通知カードなど、見た目は異なっても同じ Card コンポーネントを土台にできるのが利点です。

## カスタムフックでロジックを分離する

UIコンポーネントからビジネスロジックを分離することも、再利用性を高める重要な手法です。React のカスタムフックを活用すると、状態管理やAPI通信をコンポーネントの外に切り出せます。

```typescript
// hooks/useToggle.ts
import { useState, useCallback } from 'react';

// シンプルなトグル状態を管理するカスタムフック
export const useToggle = (initialValue: boolean = false) => {
  const [value, setValue] = useState(initialValue);

  const toggle = useCallback(() => setValue(prev => !prev), []);
  const setTrue = useCallback(() => setValue(true), []);
  const setFalse = useCallback(() => setValue(false), []);

  return { value, toggle, setTrue, setFalse };
};

// 使用例:
// const { value: isVisible, toggle: toggleModal } = useToggle(false);
// <Button label={isVisible ? '閉じる' : '開く'} onPress={toggleModal} />
```

もう少し実践的な例として、フォームの入力バリデーションをカスタムフックにまとめるケースを見てみましょう。

```typescript
// hooks/useFormField.ts
import { useState, useCallback } from 'react';

interface UseFormFieldOptions {
  initialValue?: string;
  validate?: (value: string) => string | null;  // null = バリデーション成功
}

export const useFormField = ({ initialValue = '', validate }: UseFormFieldOptions = {}) => {
  const [value, setValue] = useState(initialValue);
  const [error, setError] = useState<string | null>(null);
  const [touched, setTouched] = useState(false);

  const onChange = useCallback((text: string) => {
    setValue(text);
    if (touched && validate) {
      setError(validate(text));
    }
  }, [touched, validate]);

  const onBlur = useCallback(() => {
    setTouched(true);
    if (validate) {
      setError(validate(value));
    }
  }, [value, validate]);

  return { value, error, touched, onChange, onBlur };
};

// 期待される出力:
// { value: 'test@example.com', error: null, touched: true, onChange: [Function], onBlur: [Function] }
```

このフックを使えば、メールアドレス入力、パスワード入力、ユーザー名入力など、あらゆるテキスト入力フィールドのバリデーションロジックを統一的に扱えます。

## Rork のプロンプトでコンポーネント設計を指示するコツ

Rork でアプリを生成する際、プロンプトの書き方次第でコンポーネントの再利用性が大きく変わります。以下のポイントを意識すると、質の高い設計になりやすいです。

**具体的な Props を指示する**: 「ボタンを作って」ではなく「variant（primary/secondary/outline）、size（small/medium/large）、disabled の Props を持つ汎用ボタンコンポーネントを作って」と指定します。

**ファイル構造を指定する**: 「components/atoms/、components/molecules/、components/organisms/ のフォルダ構成にして」と明示します。

**共通テーマを定義させる**: 「colors、typography、spacing の定数を theme.ts にまとめ、全コンポーネントで参照して」と指示します。これにより、デザインの一貫性が保たれます。

**既存コンポーネントの活用を促す**: 新しい画面を追加する際は「既存の Button、Card、Input コンポーネントを使って構築して」と指示すると、コードの重複を防げます。

より体系的なアーキテクチャパターンについて詳しく知りたい方は、[Rork アプリ設計パターン完全ガイド](/articles/rork-dev/rork-app-architecture-patterns-mvvm-clean) で MVVM や Clean Architecture の実装方法を解説しています。

## FAQ

**Q: Rork が生成するコンポーネントをそのまま使っても良いですか？**

A: Rork が生成するコードは動作する状態で出力されるため、プロトタイプとしてはそのまま使えます。ただし、プロジェクトが成長してきたら、本記事で紹介した設計パターンに沿ってリファクタリングすることをおすすめします。特に Props の型定義とカスタムフックの分離は、保守性の向上に直結します。

**Q: Atomic Design は小規模なアプリでも必要ですか？**

A: 画面が2〜3個の小規模アプリでは、厳密な Atomic Design を導入する必要はありません。ただし、「Button や Input は共通化する」「ロジックはフックに分離する」という基本原則だけでも取り入れると、後から機能を追加する際にスムーズです。

**Q: TypeScript の型定義が難しく感じます。どこから始めればいいですか？**

A: まずは Props の `interface` 定義から始めるのがおすすめです。`string`、`number`、`boolean`、`() => void` の4つの型を使えれば、ほとんどのコンポーネントの Props は定義できます。Rork にプロンプトで「TypeScript の型定義を必ず含めて」と指示すれば、型付きのコードを生成してくれます。

**Q: コンポーネントのテストはどう書けばいいですか？**

A: React Native Testing Library を使うのが標準的です。再利用可能なコンポーネントは入出力が明確なため、テストが書きやすいという利点があります。「Props にこの値を渡したら、この表示になる」というテストを Atom コンポーネントから順に追加していくと効率的です。

## まとめ

再利用可能なコンポーネント設計は、Rork でのアプリ開発を次のレベルに引き上げる重要なスキルです。Atomic Design の階層を意識した構造化、Props による柔軟性の確保、カスタムフックによるロジック分離の3つの柱を押さえることで、保守しやすく拡張性の高いアプリが作れるようになります。

最初から完璧な設計を目指す必要はありません。まずは共通ボタンとカードコンポーネントを作ることから始め、徐々にパターンを広げていくのが実践的です。Rork のプロンプトにコンポーネント設計の指示を含めることで、AIが生成するコードの品質も向上します。
