## アプリセキュリティは「鍵をかける」だけでは足りない

モバイルアプリ開発で後回しにされがちなのがセキュリティだ。動くものを作り、デザインを磨き、App Store に申請する――その過程で「セキュリティは後で考えよう」と思った経験は、おそらく多くの開発者にあるだろう。

しかし現実は厳しい。2025年の調査では、Google Play に公開されているアプリの83%が OWASP Mobile Top 10 の脆弱性を少なくとも1つ含んでいた。App Store のレビューも年々厳格化しており、不適切なデータ保存や平文通信が検出されるとリジェクトされるケースが増えている。

ゼロトラストセキュリティの考え方は「何も信頼しない」ことを前提とする。ネットワークを信頼しない、デバイスを信頼しない、ユーザー入力を信頼しない。この前提に立つと、アプリのセキュリティ設計は根本から変わる。

この記事では、Rork Max で SwiftUI ネイティブアプリを開発する際に、OWASP MASVS（Mobile Application Security Verification Standard）の L2 レベルに準拠するセキュリティアーキテクチャを段階的に構築する。理論だけでなく、コピー&ペーストで使えるコードと、実装時に私がぶつかった落とし穴を交えて解説する。

## OWASP MASVS を理解する — なぜ「基準」が必要なのか

OWASP MASVS は、モバイルアプリのセキュリティ要件を体系化した国際標準だ。L1（標準セキュリティ）と L2（多層防御）の2つのレベルがある。

L1 は一般的な商用アプリに求められる最低限の要件で、認証・ネットワーク通信・データストレージの基本的な保護をカバーする。L2 は金融・ヘルスケア・エンタープライズアプリ向けで、リバースエンジニアリング耐性やランタイム保護まで含む。

私の経験では、個人開発のアプリでも L2 を目指す価値がある。理由は単純で、**攻撃者はアプリの規模で標的を選ばない**からだ。自動化ツールで脆弱性をスキャンし、見つかれば規模に関係なく攻撃する。小規模アプリだからこそ、開発者が一人しかいないぶん、インシデント対応の負荷が致命的になる。

MASVS の主要カテゴリは以下の8つだ。

- **MASVS-STORAGE**: データの安全な保存
- **MASVS-CRYPTO**: 暗号化の適切な使用
- **MASVS-AUTH**: 認証と認可
- **MASVS-NETWORK**: ネットワーク通信の保護
- **MASVS-PLATFORM**: プラットフォーム連携の安全性
- **MASVS-CODE**: コード品質とビルド設定
- **MASVS-RESILIENCE**: リバースエンジニアリング耐性
- **MASVS-PRIVACY**: プライバシー保護

この記事では、Rork Max の文脈で特に重要度が高い STORAGE・NETWORK・RESILIENCE・CODE の4領域を深掘りする。

## 脅威モデリングから始める — 「何を守るか」を明確にする

セキュリティ実装の前に、まず脅威モデリングを行う。闇雲に対策を積み上げても、重要な部分が抜け落ちて些末な部分に工数を使ってしまう。

### STRIDE モデルによる脅威分類

Rork Max で開発する典型的なアプリを想定して、STRIDE モデルで脅威を整理する。

```swift
// 脅威モデルの定義 — アプリ起動時に初期化
enum ThreatCategory: String, CaseIterable {
    case spoofing = "なりすまし"       // 他ユーザーへのなりすまし
    case tampering = "改ざん"          // データや通信の改ざん
    case repudiation = "否認"          // 操作の否認
    case informationDisclosure = "情報漏洩" // 機密データの流出
    case denialOfService = "サービス妨害"   // 機能の妨害
    case elevationOfPrivilege = "権限昇格"  // 不正な権限取得
}

struct ThreatModel {
    let category: ThreatCategory
    let asset: String        // 守るべき資産
    let attackVector: String // 攻撃経路
    let impact: Impact       // 影響度
    let likelihood: Likelihood // 発生可能性
    
    enum Impact: Int { case low = 1, medium = 2, high = 3, critical = 4 }
    enum Likelihood: Int { case unlikely = 1, possible = 2, likely = 3, certain = 4 }
    
    /// リスクスコア = 影響度 × 発生可能性
    var riskScore: Int { impact.rawValue * likelihood.rawValue }
}

// 具体的な脅威を定義
let threats: [ThreatModel] = [
    ThreatModel(
        category: .informationDisclosure,
        asset: "ユーザーの認証トークン",
        attackVector: "端末のファイルシステムから平文のトークンを読み取る",
        impact: .critical,
        likelihood: .likely
    ),
    ThreatModel(
        category: .tampering,
        asset: "API リクエスト",
        attackVector: "中間者攻撃でリクエストを改ざんする",
        impact: .high,
        likelihood: .possible
    ),
    ThreatModel(
        category: .elevationOfPrivilege,
        asset: "課金ステータス",
        attackVector: "アプリを改ざんして課金チェックをバイパスする",
        impact: .high,
        likelihood: .likely
    ),
]

// リスクスコアの高い順にソートして対策の優先順位を決める
let prioritized = threats.sorted { $0.riskScore > $1.riskScore }
for threat in prioritized {
    print("[\(threat.riskScore)] \(threat.category.rawValue): \(threat.asset)")
    // 出力例:
    // [12] 情報漏洩: ユーザーの認証トークン
    // [9] 権限昇格: 課金ステータス
    // [6] 改ざん: API リクエスト
}
```

このスコアリングで優先順位が明確になる。「認証トークンの保護」が最優先で、次に「課金バイパスの防止」、そして「通信の改ざん防止」と続く。

### なぜリスクベースで対策するのか

全ての脅威に同じ工数をかけるのは非効率だ。リスクスコアが4以下の脅威は、アプリのフェーズによっては後回しにしてもよい。逆にスコア9以上は、リリース前に必ず対処すべきだ。

ゼロトラストの考え方では「全てを疑う」が原則だが、実装の順序はリスクの大きさで決める。これは矛盾ではない。限られたリソースで最大のセキュリティ効果を得るための現実的な戦略だ。

## 暗号化ストレージの実装 — MASVS-STORAGE 準拠

### Keychain Services を正しく使う

iOS の Keychain は、認証トークン・パスワード・暗号化キーを安全に保存するための唯一の正解だ。`UserDefaults` や `FileManager` での平文保存は L1 ですら要件を満たさない。

Rork Max で SwiftUI アプリを構築する際、Keychain 操作をラップするユーティリティを作る。

```swift
import Foundation
import Security

/// Keychain 操作のエラーを明示的に定義
enum KeychainError: Error, LocalizedError {
    case duplicateItem
    case itemNotFound
    case unexpectedStatus(OSStatus)
    case dataConversionFailed
    
    var errorDescription: String? {
        switch self {
        case .duplicateItem: return "Keychain にアイテムが既に存在します"
        case .itemNotFound: return "Keychain にアイテムが見つかりません"
        case .unexpectedStatus(let status):
            return "Keychain エラー: \(SecCopyErrorMessageString(status, nil) ?? "不明" as CFString)"
        case .dataConversionFailed: return "データの変換に失敗しました"
        }
    }
}

final class SecureKeychain {
    
    private let serviceName: String
    private let accessGroup: String?
    
    /// - Parameters:
    ///   - serviceName: アプリの Bundle Identifier を推奨
    ///   - accessGroup: App Group でデータ共有する場合に指定
    init(serviceName: String = Bundle.main.bundleIdentifier ?? "com.app.default",
         accessGroup: String? = nil) {
        self.serviceName = serviceName
        self.accessGroup = accessGroup
    }
    
    /// 値を保存する（既存の場合は上書き）
    func save(key: String, data: Data,
              accessibility: CFString = kSecAttrAccessibleWhenUnlockedThisDeviceOnly) throws {
        
        var query: [String: Any] = [
            kSecClass as String: kSecClassGenericPassword,
            kSecAttrService as String: serviceName,
            kSecAttrAccount as String: key,
            kSecValueData as String: data,
            // ★ kSecAttrAccessibleWhenUnlockedThisDeviceOnly を指定することで、
            //    デバイスがロック中はアクセス不可＋バックアップに含まれない
            kSecAttrAccessible as String: accessibility,
        ]
        
        if let group = accessGroup {
            query[kSecAttrAccessGroup as String] = group
        }
        
        let status = SecItemAdd(query as CFDictionary, nil)
        
        if status == errSecDuplicateItem {
            // 既存アイテムを更新
            let searchQuery: [String: Any] = [
                kSecClass as String: kSecClassGenericPassword,
                kSecAttrService as String: serviceName,
                kSecAttrAccount as String: key,
            ]
            let updateAttributes: [String: Any] = [
                kSecValueData as String: data,
                kSecAttrAccessible as String: accessibility,
            ]
            let updateStatus = SecItemUpdate(
                searchQuery as CFDictionary,
                updateAttributes as CFDictionary
            )
            guard updateStatus == errSecSuccess else {
                throw KeychainError.unexpectedStatus(updateStatus)
            }
        } else if status \!= errSecSuccess {
            throw KeychainError.unexpectedStatus(status)
        }
    }
    
    /// 値を取得する
    func load(key: String) throws -> Data {
        let query: [String: Any] = [
            kSecClass as String: kSecClassGenericPassword,
            kSecAttrService as String: serviceName,
            kSecAttrAccount as String: key,
            kSecReturnData as String: true,
            kSecMatchLimit as String: kSecMatchLimitOne,
        ]
        
        var result: AnyObject?
        let status = SecItemCopyMatching(query as CFDictionary, &result)
        
        guard status == errSecSuccess else {
            if status == errSecItemNotFound {
                throw KeychainError.itemNotFound
            }
            throw KeychainError.unexpectedStatus(status)
        }
        
        guard let data = result as? Data else {
            throw KeychainError.dataConversionFailed
        }
        return data
    }
    
    /// 値を削除する
    func delete(key: String) throws {
        let query: [String: Any] = [
            kSecClass as String: kSecClassGenericPassword,
            kSecAttrService as String: serviceName,
            kSecAttrAccount as String: key,
        ]
        
        let status = SecItemDelete(query as CFDictionary)
        guard status == errSecSuccess || status == errSecItemNotFound else {
            throw KeychainError.unexpectedStatus(status)
        }
    }
}

// 使用例 — 認証トークンの安全な保存
let keychain = SecureKeychain()

// 保存
let token = "eyJhbGciOiJIUzI1NiIs..."
try keychain.save(key: "auth_token", data: Data(token.utf8))

// 取得
let storedData = try keychain.load(key: "auth_token")
let storedToken = String(data: storedData, encoding: .utf8)
print(storedToken ?? "取得失敗")
// 期待出力: eyJhbGciOiJIUzI1NiIs...
```

### よくある間違い: `kSecAttrAccessibleAlways` の使用

`kSecAttrAccessible` に `kSecAttrAccessibleAlways` を指定すると、デバイスがロック中でも Keychain データにアクセスできてしまう。これは MASVS-STORAGE の要件違反だ。正しくは `kSecAttrAccessibleWhenUnlockedThisDeviceOnly` を使う。「ThisDeviceOnly」が付くと iTunes バックアップにも含まれなくなる。

もう一つよくある間違いは、トークンを `UserDefaults` に保存してしまうことだ。`UserDefaults` は plist ファイルとして平文保存されるため、Jailbreak された端末では簡単に読み取れる。Rork Max のプロジェクトで `UserDefaults` を検索して、機密情報が紛れ込んでいないか確認してほしい。

## ネットワーク防御の実装 — MASVS-NETWORK 準拠

### 証明書ピニングの仕組みと実装

TLS/SSL だけでは中間者攻撃を完全には防げない。攻撃者がルート証明書をデバイスにインストールすれば、正規の TLS 通信を傍受できてしまう。証明書ピニングは、アプリが信頼するサーバー証明書（またはその公開鍵）をアプリ内にハードコードし、接続先がその証明書を持っているかを検証する仕組みだ。

```swift
import Foundation
import CryptoKit

/// 証明書ピニングを実装する URLSession デリゲート
final class CertificatePinningDelegate: NSObject, URLSessionDelegate {
    
    /// ピニングする公開鍵の SHA-256 ハッシュ（Base64 エンコード）
    /// ★ 本番環境では、プライマリとバックアップの2つ以上を設定すること
    private let pinnedKeyHashes: Set<String> = [
        "BBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBB=",  // プライマリ証明書の公開鍵ハッシュ
        "CCCCCCCCCCCCCCCCCCCCCCCCCCCCCCCCCCCCCCCCCCC=",  // バックアップ証明書の公開鍵ハッシュ
    ]
    
    func urlSession(
        _ session: URLSession,
        didReceive challenge: URLAuthenticationChallenge,
        completionHandler: @escaping (URLSession.AuthChallengeDisposition, URLCredential?) -> Void
    ) {
        guard challenge.protectionSpace.authenticationMethod == NSURLAuthenticationMethodServerTrust,
              let serverTrust = challenge.protectionSpace.serverTrust else {
            completionHandler(.cancelAuthenticationChallenge, nil)
            return
        }
        
        // サーバー証明書チェーンを検証
        guard SecTrustEvaluateWithError(serverTrust, nil) else {
            completionHandler(.cancelAuthenticationChallenge, nil)
            return
        }
        
        // 証明書チェーン内の公開鍵を検証
        let certCount = SecTrustGetCertificateCount(serverTrust)
        var foundMatch = false
        
        for index in 0..<certCount {
            guard let certificate = SecTrustCopyCertificateChain(serverTrust)?
                    .unsafelyUnwrapped as? [SecCertificate],
                  index < certificate.count else { continue }
            
            let cert = certificate[index]
            
            // 証明書から公開鍵を抽出
            guard let publicKey = SecCertificateCopyKey(cert),
                  let publicKeyData = SecKeyCopyExternalRepresentation(publicKey, nil) as? Data else {
                continue
            }
            
            // SHA-256 ハッシュを計算して比較
            let hash = SHA256.hash(data: publicKeyData)
            let hashBase64 = Data(hash).base64EncodedString()
            
            if pinnedKeyHashes.contains(hashBase64) {
                foundMatch = true
                break
            }
        }
        
        if foundMatch {
            let credential = URLCredential(trust: serverTrust)
            completionHandler(.useCredential, credential)
        } else {
            // ★ ピニング失敗 — 接続を拒否
            // 本番では、このイベントをセキュリティログに記録する
            print("⚠️ 証明書ピニング失敗: \(challenge.protectionSpace.host)")
            completionHandler(.cancelAuthenticationChallenge, nil)
        }
    }
}

// 使用例 — ピニング付きの API クライアント
let pinningDelegate = CertificatePinningDelegate()
let session = URLSession(
    configuration: .default,
    delegate: pinningDelegate,
    delegateQueue: nil
)

// このセッション経由の全リクエストで証明書ピニングが有効になる
let url = URL(string: "https://api.example.com/v1/user")\!
let (data, response) = try await session.data(from: url)
// ピニング検証に失敗した場合、リクエストは自動的にキャンセルされる
```

### 証明書ピニングの落とし穴

証明書ピニングで最も危険な間違いは、**バックアップ鍵を設定しないこと**だ。サーバー証明書は定期的にローテーションされる。プライマリ鍵しかピニングしていないと、証明書更新の瞬間にアプリが全く通信できなくなる。

もう一つの落とし穴は、開発環境と本番環境で同じピニング設定を使うことだ。開発時にプロキシツール（Charles Proxy や mitmproxy）を使ってデバッグしたい場面がある。`#if DEBUG` ディレクティブでデバッグビルド時のみピニングを無効化する設定を入れておかないと、開発効率が著しく下がる。

```swift
#if DEBUG
// デバッグビルドではピニングをスキップ（開発時のみ）
completionHandler(.performDefaultHandling, nil)
#else
// リリースビルドではピニングを強制
if foundMatch {
    completionHandler(.useCredential, URLCredential(trust: serverTrust))
} else {
    completionHandler(.cancelAuthenticationChallenge, nil)
}
#endif
```

ただし、**この `#if DEBUG` ブロックがリリースビルドに混入しないよう、CI/CD パイプラインで Build Configuration を必ず確認する**こと。Rork Max の EAS Build を使っている場合は、`eas.json` の本番プロファイルで `-DDEBUG` フラグが含まれていないことを検証する。

## ランタイム保護の実装 — MASVS-RESILIENCE 準拠

### Jailbreak / Root 検出

Jailbreak されたデバイスでは、アプリのサンドボックスが無効化される。他のアプリがファイルシステムにアクセスしたり、実行中のプロセスにアタッチしたりできてしまう。完璧な検出は不可能だが、複数のシグナルを組み合わせることで検出精度を上げる。

```swift
import Foundation
import UIKit

/// 複数のシグナルを組み合わせた Jailbreak 検出
struct JailbreakDetector {
    
    /// 総合判定 — 1つでも検出されたら true
    static func isJailbroken() -> Bool {
        return checkSuspiciousFiles()
            || checkSuspiciousURLSchemes()
            || checkWriteAccess()
            || checkDynamicLibraries()
            || checkFork()
    }
    
    /// 1. Jailbreak ツールが作成する典型的なファイルの存在確認
    private static func checkSuspiciousFiles() -> Bool {
        let suspiciousPaths = [
            "/Applications/Cydia.app",
            "/Applications/Sileo.app",
            "/usr/sbin/sshd",
            "/usr/bin/ssh",
            "/etc/apt",
            "/var/lib/cydia",
            "/private/var/stash",
            "/usr/libexec/sftp-server",
            "/usr/sbin/frida-server",
            "/usr/bin/cycript",
        ]
        
        for path in suspiciousPaths {
            if FileManager.default.fileExists(atPath: path) {
                return true
            }
        }
        return false
    }
    
    /// 2. Cydia / Sileo の URL スキームが開けるか確認
    private static func checkSuspiciousURLSchemes() -> Bool {
        let schemes = ["cydia://", "sileo://", "zbra://"]
        for scheme in schemes {
            if let url = URL(string: scheme),
               UIApplication.shared.canOpenURL(url) {
                return true
            }
        }
        return false
    }
    
    /// 3. サンドボックス外への書き込みが可能か確認
    private static func checkWriteAccess() -> Bool {
        let testPath = "/private/jailbreak_test_\(UUID().uuidString)"
        do {
            try "test".write(toFile: testPath, atomically: true, encoding: .utf8)
            try FileManager.default.removeItem(atPath: testPath)
            return true  // 書き込みに成功 = サンドボックスが破られている
        } catch {
            return false  // 書き込み失敗 = 正常
        }
    }
    
    /// 4. 不審な動的ライブラリの読み込みを検出
    private static func checkDynamicLibraries() -> Bool {
        let suspiciousLibs = [
            "FridaGadget",
            "frida-agent",
            "cynject",
            "libcycript",
            "SubstrateLoader",
        ]
        
        let count = _dyld_image_count()
        for i in 0..<count {
            if let name = _dyld_get_image_name(i) {
                let imageName = String(cString: name)
                for lib in suspiciousLibs {
                    if imageName.contains(lib) {
                        return true
                    }
                }
            }
        }
        return false
    }
    
    /// 5. fork() が成功するか確認（Jailbreak なしでは失敗する）
    private static func checkFork() -> Bool {
        let pid = fork()
        if pid >= 0 {
            // fork 成功 = Jailbreak の可能性が高い
            if pid > 0 {
                kill(pid, SIGTERM)  // 子プロセスを終了
            }
            return true
        }
        return false
    }
}

// 使用例 — アプリ起動時のセキュリティチェック
func applicationDidFinishLaunching() {
    if JailbreakDetector.isJailbroken() {
        // ★ 即座にアプリを終了するのではなく、機能を制限する
        // 理由: 即終了はユーザー体験が悪く、App Store レビューで問題になることがある
        SecurityManager.shared.enableRestrictedMode()
        // 制限モード: API キーを Keychain から削除、課金機能を無効化、
        // ユーザーに「セキュリティ上の理由で一部機能が制限されています」と通知
        print("⚠️ Jailbreak 検出 — 制限モードで起動")
        // 期待出力: ⚠️ Jailbreak 検出 — 制限モードで起動
    }
}
```

### デバッガアタッチ検出

攻撃者がデバッガ（lldb 等）をアプリプロセスにアタッチして、実行中のメモリを読み取ったり、関数の戻り値を書き換えたりする攻撃がある。

```swift
import Foundation

/// デバッガのアタッチを検出する
struct DebuggerDetector {
    
    /// sysctl でデバッガフラグを確認
    static func isBeingDebugged() -> Bool {
        var info = kinfo_proc()
        var size = MemoryLayout<kinfo_proc>.stride
        var mib: [Int32] = [CTL_KERN, KERN_PROC, KERN_PROC_PID, getpid()]
        
        let result = sysctl(&mib, UInt32(mib.count), &info, &size, nil, 0)
        guard result == 0 else { return false }
        
        // P_TRACED フラグが立っていればデバッガがアタッチされている
        return (info.kp_proc.p_flag & P_TRACED) \!= 0
    }
    
    /// ptrace でデバッガのアタッチを拒否する
    /// ★ この関数はアプリ起動の極めて早い段階で呼ぶこと
    static func denyDebuggerAttach() {
        #if \!DEBUG
        // PT_DENY_ATTACH を直接呼び出す
        let PT_DENY_ATTACH: CInt = 31
        ptrace(PT_DENY_ATTACH, 0, nil, 0)
        #endif
    }
}

// アプリ起動時に呼ぶ
// ★ @main の init() か、AppDelegate の最初で実行するのが最も効果的
DebuggerDetector.denyDebuggerAttach()
if DebuggerDetector.isBeingDebugged() {
    print("⚠️ デバッガ検出")
    // 期待出力（デバッガがアタッチされている場合）: ⚠️ デバッガ検出
}
```

### ランタイム保護でやりがちな失敗

**Jailbreak 検出で即座に `exit(0)` を呼ぶのは悪手だ。** App Store のレビューガイドラインでは、アプリが突然クラッシュすることは望ましくないとされている。また、攻撃者は `exit` 関数をフックして無効化できるため、防御としても弱い。

代わりに「制限モード」を実装する。機密データを Keychain から消去し、課金機能を無効化し、ユーザーには丁寧に通知する。これなら App Store レビューでも問題にならず、実質的なセキュリティ効果も高い。

もう一つの失敗は、**検出ロジックを1つの関数にまとめてしまうこと**。攻撃者はその関数をフックして戻り値を `false` に書き換えるだけで全ての検出を無効化できる。上記のコードのように複数の独立した検出関数に分割し、それぞれ異なるタイミングで呼び出すことで耐性を上げる。

## コード難読化とバイナリ保護 — MASVS-CODE 準拠

### Swift コードの難読化戦略

Swift コンパイラは最適化レベル `-O` でビルドすると、ある程度の難読化効果がある。しかし、文字列リテラル（API キー、エンドポイント URL）はバイナリ内に平文で残る。

```swift
/// 文字列の実行時復号化 — バイナリ解析で API エンドポイントを発見されにくくする
struct ObfuscatedString {
    
    private let encrypted: [UInt8]
    private let key: UInt8
    
    /// XOR 暗号化で文字列を保護する
    /// ★ 完璧な暗号化ではないが、strings コマンドでの即座の発見を防ぐ
    init(_ plainText: String, key: UInt8 = 0xAB) {
        self.key = key
        self.encrypted = plainText.utf8.map { $0 ^ key }
    }
    
    /// 使用時に復号化する
    var value: String {
        let decrypted = encrypted.map { $0 ^ key }
        return String(bytes: decrypted, encoding: .utf8) ?? ""
    }
}

// 使用例 — API エンドポイントの保護
// ❌ 悪い例: バイナリ内に平文で残る
// let apiEndpoint = "https://api.myapp.com/v2/secure"

// ✅ 良い例: 実行時まで復号化されない
let apiEndpoint = ObfuscatedString("https://api.myapp.com/v2/secure")
let url = URL(string: apiEndpoint.value)\!
print(url.absoluteString)
// 期待出力: https://api.myapp.com/v2/secure
```

### App Transport Security の厳格化

`Info.plist` の ATS（App Transport Security）設定は、意図せず緩和されていることが多い。Rork Max でプロジェクトを生成した場合、デフォルトで `NSAllowsArbitraryLoads` が `true` になっている場合がある。本番ビルドでは必ず以下の設定に修正する。

```xml
<\!-- Info.plist — ATS の厳格設定 -->
<key>NSAppTransportSecurity</key>
<dict>
    <\!-- 全体で任意のHTTP接続を禁止 -->
    <key>NSAllowsArbitraryLoads</key>
    <false/>
    
    <\!-- 特定ドメインのみ例外を設定（必要な場合のみ） -->
    <key>NSExceptionDomains</key>
    <dict>
        <key>legacy-api.example.com</key>
        <dict>
            <key>NSExceptionAllowsInsecureHTTPLoads</key>
            <true/>
            <key>NSExceptionMinimumTLSVersion</key>
            <string>TLSv1.2</string>
            <\!-- ★ 例外ドメインには必ずコメントで理由を残す -->
            <\!-- レガシー API がHTTPSに対応するまでの暫定措置 -->
        </dict>
    </dict>
</dict>
```

## セキュリティテストの自動化 — CI/CD に組み込む

### セキュリティチェックを GitHub Actions で自動実行する

セキュリティは「実装して終わり」ではない。継続的にテストし、退行を防ぐ仕組みが必要だ。

```yaml
# .github/workflows/security-check.yml
name: Security Audit

on:
  pull_request:
    branches: [main]
  push:
    branches: [main]

jobs:
  security-audit:
    runs-on: macos-14
    steps:
      - uses: actions/checkout@v4
      
      # 1. 依存関係の脆弱性チェック
      - name: Audit npm dependencies
        run: npm audit --audit-level=high
        continue-on-error: true  # 警告は許容、high 以上で失敗
      
      # 2. 平文シークレットの検出
      - name: Scan for secrets
        uses: trufflesecurity/trufflehog@main
        with:
          extra_args: --only-verified
      
      # 3. ATS 設定の検証
      - name: Verify ATS configuration
        run: |
          # NSAllowsArbitraryLoads が true でないことを確認
          if grep -r "NSAllowsArbitraryLoads" ios/ | grep -q "true"; then
            echo "❌ ATS が無効化されています"
            exit 1
          fi
          echo "✅ ATS 設定は正常"
      
      # 4. UserDefaults での機密データ保存を検出
      - name: Check for insecure storage
        run: |
          # UserDefaults に token/password/secret を保存するコードを検出
          INSECURE=$(grep -rn "UserDefaults.*\(token\|password\|secret\|apiKey\)" \
            --include="*.swift" . || true)
          if [ -n "$INSECURE" ]; then
            echo "❌ UserDefaults に機密データが保存される可能性があります:"
            echo "$INSECURE"
            exit 1
          fi
          echo "✅ 不安全なストレージ使用は検出されませんでした"
      
      # 5. デバッグコードの残存チェック
      - name: Check for debug remnants
        run: |
          # リリースビルドに print/NSLog が残っていないか確認
          DEBUG_LOGS=$(grep -rn "print(\|NSLog(" --include="*.swift" . \
            | grep -v "// DEBUG" \
            | grep -v "#if DEBUG" \
            | grep -v "test/" || true)
          if [ -n "$DEBUG_LOGS" ]; then
            echo "⚠️ デバッグログが検出されました（確認してください）:"
            echo "$DEBUG_LOGS" | head -20
          fi
```

### 自動化で見つけられないもの

CI/CD のセキュリティチェックはベースラインを守るために有効だが、ロジックの脆弱性は検出できない。例えば、「課金ステータスの判定をクライアント側だけで行っている」「API レスポンスの署名を検証していない」といった設計上の問題は、コードスキャンでは発見しにくい。

四半期に一度は、OWASP MASVS のチェックリストを手動で確認する時間を確保することを強く推奨する。チェックリストは [OWASP の公式サイト](https://mas.owasp.org/MASVS/) で公開されている。

## よくある間違いと落とし穴

### 1. 「リリースビルドだからデバッグ情報は入らない」という誤解

Swift の Release ビルドでも、デフォルトでは DWARF デバッグ情報がバイナリに含まれる。`Strip Debug Symbols During Copy` を `Yes` に設定し、`Debug Information Format` を `dwarf` （`dwarf-with-dsym` ではなく）に変更すること。ただし、クラッシュレポートの解析には dSYM ファイルが必要なので、dSYM は別途アーカイブして保管する。

### 2. 証明書ピニングのハッシュを間違える

公開鍵のハッシュ計算で、DER エンコーディングされた公開鍵全体ではなく、Subject Public Key Info（SPKI）のハッシュを使うべき場面がある。ライブラリによって期待するフォーマットが異なるため、必ずサーバー証明書からテスト用のハッシュを生成して検証すること。

```bash
# サーバー証明書の公開鍵ハッシュを取得するコマンド
openssl s_client -connect api.example.com:443 2>/dev/null \
  | openssl x509 -pubkey -noout \
  | openssl pkey -pubin -outform DER \
  | openssl dgst -sha256 -binary \
  | base64
```

### 3. Keychain のアクセスグループ設定漏れ

App Extension（Widget、Notification Service Extension 等）とメインアプリで Keychain データを共有する場合、`kSecAttrAccessGroup` の設定が必要だ。設定を忘れると、Widget がトークンを読めずに認証エラーになる。Entitlements ファイルの `Keychain Access Groups` にも同じグループを追加すること。

### 4. ネットワーク応答の検証を怠る

API レスポンスの JSON をデコードするとき、Swift の `Codable` は型の不一致をエラーにしてくれるが、**値の妥当性**は検証しない。例えば、`"price": -100` のようなレスポンスを受け入れてしまうと、課金ロジックが壊れる。重要なフィールドには必ずバリデーションを追加する。

### 5. セキュリティ対策を全て起動時に集中させる

Jailbreak 検出、デバッガ検出、整合性チェックを全て `applicationDidFinishLaunching` で実行するのは、攻撃者にとって「ここだけフックすれば全対策を無効化できる」ことを意味する。検出ロジックはアプリのライフサイクル全体に分散させる。課金チェック時、API コール時、バックグラウンド復帰時など、複数のタイミングで異なる検出を実行する。

## 実践的な応用: 課金バイパス防止のアーキテクチャ

個人開発アプリで最も被害が大きいのが課金バイパスだ。攻撃者がアプリを改ざんして `isPremium` フラグを常に `true` に書き換える手法が広く知られている。

```swift
/// 課金バイパスを防止するサーバーサイド検証付き課金チェッカー
actor PremiumVerifier {
    
    private let keychain = SecureKeychain()
    private let session: URLSession
    
    // ★ ローカルのフラグだけに頼らない — サーバーで必ず検証する
    private var cachedStatus: PremiumStatus?
    private var lastVerified: Date?
    private let cacheExpiry: TimeInterval = 300  // 5分でキャッシュ失効
    
    struct PremiumStatus: Codable {
        let isPremium: Bool
        let expiresAt: Date?
        let signature: String  // サーバーが署名した検証トークン
    }
    
    init() {
        let pinning = CertificatePinningDelegate()
        self.session = URLSession(
            configuration: .default,
            delegate: pinning,
            delegateQueue: nil
        )
    }
    
    /// プレミアムステータスを検証する
    /// - Returns: ユーザーがプレミアム会員かどうか
    func verifyPremium() async throws -> Bool {
        // 1. キャッシュが有効ならキャッシュを返す
        if let cached = cachedStatus,
           let lastCheck = lastVerified,
           Date().timeIntervalSince(lastCheck) < cacheExpiry {
            return cached.isPremium
        }
        
        // 2. Keychain からトークンを取得
        guard let tokenData = try? keychain.load(key: "auth_token"),
              let token = String(data: tokenData, encoding: .utf8) else {
            return false  // トークンがない = 未認証 = 非プレミアム
        }
        
        // 3. サーバーに問い合わせ（証明書ピニング付き）
        var request = URLRequest(url: URL(string: "https://api.example.com/v1/verify-premium")\!)
        request.setValue("Bearer \(token)", forHTTPHeaderField: "Authorization")
        request.setValue(generateDeviceFingerprint(), forHTTPHeaderField: "X-Device-Fingerprint")
        
        let (data, response) = try await session.data(for: request)
        
        guard let httpResponse = response as? HTTPURLResponse,
              httpResponse.statusCode == 200 else {
            // ★ サーバーに到達できない場合は deny by default
            // 理由: 機内モードでの課金バイパスを防止する
            return cachedStatus?.isPremium ?? false
        }
        
        let status = try JSONDecoder().decode(PremiumStatus.self, from: data)
        
        // 4. サーバー署名を検証（改ざん検出）
        guard verifySignature(status) else {
            // レスポンスが改ざんされている
            return false
        }
        
        // 5. キャッシュを更新
        cachedStatus = status
        lastVerified = Date()
        
        return status.isPremium
    }
    
    /// デバイスフィンガープリントを生成（リプレイ攻撃防止）
    private func generateDeviceFingerprint() -> String {
        let vendorID = UIDevice.current.identifierForVendor?.uuidString ?? "unknown"
        let timestamp = Int(Date().timeIntervalSince1970)
        let nonce = UUID().uuidString
        return "\(vendorID):\(timestamp):\(nonce)"
    }
    
    /// サーバー署名の検証（簡易版）
    private func verifySignature(_ status: PremiumStatus) -> Bool {
        // 本番では HMAC-SHA256 や RSA 署名を使う
        // ここでは概念を示すための簡易実装
        return \!status.signature.isEmpty
    }
}

// 使用例
let verifier = PremiumVerifier()
let isPremium = try await verifier.verifyPremium()
if isPremium {
    // プレミアムコンテンツを表示
    print("✅ プレミアム検証完了")
    // 期待出力: ✅ プレミアム検証完了
} else {
    // アップグレード画面を表示
    print("🔒 プレミアムコンテンツはアップグレードが必要です")
}
```

このアーキテクチャのポイントは3つある。第一に、ローカルフラグだけでなくサーバーで必ず検証する。第二に、オフライン時は deny by default（疑わしければ拒否）。第三に、レスポンスの署名検証で中間者攻撃を防ぐ。

この設計思想は、[Rork Max アプリのセキュリティ実装ベストプラクティス](/articles/rork-dev/rork-max-security-best-practices)や [Keychain を使った認証情報の安全な保存ガイド](/articles/rork-dev/rork-max-security-keychain-guide) でも触れている基礎概念を発展させたものだ。
