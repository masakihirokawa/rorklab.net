## はじめに：なぜ Neon + Drizzle ORM なのか

Rork MaxでAIがアプリのフロントエンドを瞬時に生成できる時代でも、バックエンドの設計と実装は依然として開発者の腕の見せどころです。特にデータベース周りの設計は、アプリの信頼性・スケーラビリティ・開発速度に直結します。

2026年現在、モバイルアプリバックエンドの新標準として急速に普及しているのが **Neon Serverless Postgres** と **Drizzle ORM** の組み合わせです。

**Neon** はサーバーレス向けに最適化されたPostgreSQLサービスで、コールドスタートがなく、Cloudflare WorkersのようなEdge環境からHTTP経由で直接接続できます。従来のPostgreSQLが苦手としていた「コネクション数の上限」「サーバーレス環境との相性」を根本から解決しています。

**Drizzle ORM** はTypeScriptファーストのORMで、SQLの表現力を保ちながら完全な型安全性を提供します。ORMによくある「マジックすぎて何が起きているか分からない」という問題がなく、生成されるSQLが予測可能で、パフォーマンスチューニングもしやすいのが特徴です。

Supabase（Postgres as a Service）と比較した場合、Neon + Drizzle ORMの構成は以下の優位性があります。

- ベンダーロックインが少なく、標準Postgresクエリが使える
- Cloudflare Workers / Vercel Edge Functions との相性が抜群
- DX（開発者体験）が非常に高い型安全なクエリビルダー
- オートスケールと一時停止機能によるコスト最適化

本ガイドでは、Rork Maxで生成したアプリのバックエンドとして、この最新スタックをゼロから実装する全手順を解説します。実用的なコード例を豊富に盛り込み、本番運用まで視野に入れた設計指針をお届けします。

## Neon データベースの設定

### Neonプロジェクトの作成

[Neonコンソール](https://console.neon.tech/) にログインし、以下の手順でデータベースを作成します。

1. 「New Project」をクリック
2. プロジェクト名（例: `rork-app-backend`）を入力
3. リージョンを選択（日本ユーザーは `Asia Pacific (Singapore)` または `Asia Pacific (Tokyo)` を推奨）
4. Postgres バージョンは最新（16系）を選択

作成後、接続文字列をコピーします。形式は以下の通りです。

```
postgres://[user]:[password]@[endpoint-id].us-east-2.aws.neon.tech/[dbname]?sslmode=require
```

Cloudflare Workersの `wrangler.toml` には接続文字列を直接書かず、Secretとして管理します。

```bash
# プロンプトに従ってNeonの接続文字列を貼り付け
```

## Drizzle ORM スキーマ定義

### TypeScriptファーストのスキーマ設計

Drizzle ORMの最大の特徴は、スキーマをTypeScriptで定義することで、クエリの結果型が自動的に推論される点です。

```typescript
// src/db/schema.ts
import { pgTable, text, timestamp, boolean, integer, uuid, index, uniqueIndex } from "drizzle-orm/pg-core";
import { createInsertSchema, createSelectSchema } from "drizzle-zod";
import { z } from "zod";

// ユーザーテーブル
export const users = pgTable("users", {
  id: uuid("id").primaryKey().defaultRandom(),
  email: text("email").notNull(),
  displayName: text("display_name"),
  avatarUrl: text("avatar_url"),
  isPremium: boolean("is_premium").default(false).notNull(),
  stripeCustomerId: text("stripe_customer_id"),
  createdAt: timestamp("created_at", { withTimezone: true }).defaultNow().notNull(),
  updatedAt: timestamp("updated_at", { withTimezone: true }).defaultNow().notNull(),
}, (table) => ({
  emailIdx: uniqueIndex("users_email_idx").on(table.email),
}));

// 投稿テーブル（サンプル：SNS/ブログ系アプリを想定）
export const posts = pgTable("posts", {
  id: uuid("id").primaryKey().defaultRandom(),
  userId: uuid("user_id").notNull().references(() => users.id, { onDelete: "cascade" }),
  title: text("title").notNull(),
  content: text("content").notNull(),
  isPublished: boolean("is_published").default(false).notNull(),
  likeCount: integer("like_count").default(0).notNull(),
  createdAt: timestamp("created_at", { withTimezone: true }).defaultNow().notNull(),
  updatedAt: timestamp("updated_at", { withTimezone: true }).defaultNow().notNull(),
}, (table) => ({
  userIdIdx: index("posts_user_id_idx").on(table.userId),
  createdAtIdx: index("posts_created_at_idx").on(table.createdAt),
}));

// いいねテーブル
export const likes = pgTable("likes", {
  id: uuid("id").primaryKey().defaultRandom(),
  userId: uuid("user_id").notNull().references(() => users.id, { onDelete: "cascade" }),
  postId: uuid("post_id").notNull().references(() => posts.id, { onDelete: "cascade" }),
  createdAt: timestamp("created_at", { withTimezone: true }).defaultNow().notNull(),
}, (table) => ({
  uniqueLike: uniqueIndex("likes_unique_idx").on(table.userId, table.postId),
}));

// Zodスキーマ（バリデーション用）
export const insertUserSchema = createInsertSchema(users, {
  email: z.string().email(),
  displayName: z.string().min(1).max(50).optional(),
});

export const insertPostSchema = createInsertSchema(posts, {
  title: z.string().min(1).max(200),
  content: z.string().min(1).max(10000),
});

// 型エクスポート
export type User = typeof users.$inferSelect;
export type NewUser = typeof users.$inferInsert;
export type Post = typeof posts.$inferSelect;
export type NewPost = typeof posts.$inferInsert;
```

このスキーマ定義から、`User` 型や `Post` 型が自動的に推論されます。IDEでのオートコンプリートも完全に機能し、型エラーを事前に検出できます。

### DB接続設定（Cloudflare Workers + Neon HTTP対応）

```typescript
// src/db/index.ts
import { drizzle } from "drizzle-orm/neon-http";
import { neon } from "@neondatabase/serverless";
import * as schema from "./schema";

export type Env = {
  DATABASE_URL: string;
};

// Cloudflare Workersのリクエストコンテキスト内でDB接続を初期化
export function createDb(env: Env) {
  // neon() は HTTP ドライバーを返す（WebSocket不要・エッジ対応）
  const sql = neon(env.DATABASE_URL);
  return drizzle(sql, { schema });
}

export type Database = ReturnType<typeof createDb>;
```

**重要なポイント**: Cloudflare Workersでは永続的なTCP接続が使えないため、`@neondatabase/serverless` の `neon()` 関数（HTTPモード）を使います。これにより、コネクションプールなしでエッジ環境から安全にPostgreSQLにアクセスできます。

### マイグレーションの実行フロー

```bash
# スキーマからマイグレーションファイルを生成
npx drizzle-kit generate

# 生成されたSQLを確認（実行前に必ずチェック）
cat src/db/migrations/0000_initial.sql

# Neonデータベースにマイグレーションを適用
npx drizzle-kit migrate

# 現在のDBスキーマを確認（Drizzle Studioを起動）
npx drizzle-kit studio
```

生成されるマイグレーションSQLの例：

```sql
-- src/db/migrations/0000_initial.sql
CREATE TABLE IF NOT EXISTS "users" (
  "id" uuid PRIMARY KEY DEFAULT gen_random_uuid() NOT NULL,
  "email" text NOT NULL,
  "display_name" text,
  "avatar_url" text,
  "is_premium" boolean DEFAULT false NOT NULL,
  "stripe_customer_id" text,
  "created_at" timestamptz DEFAULT now() NOT NULL,
  "updated_at" timestamptz DEFAULT now() NOT NULL

CREATE UNIQUE INDEX "users_email_idx" ON "users" ("email");
```

スキーマ変更時は `drizzle-kit generate` を再度実行し、新しいマイグレーションファイルを生成します。**既存のマイグレーションファイルは絶対に手動編集しない**ことが鉄則です。

### 型安全なクエリ実装例（ユーザールート）

```typescript
// src/routes/users.ts
import { Hono } from "hono";
import { zValidator } from "@hono/zod-validator";
import { eq, desc } from "drizzle-orm";
import { createDb, type Env } from "../db";
import { users, posts, insertUserSchema } from "../db/schema";

const route = new Hono<{ Bindings: Env }>();

// ユーザー一覧取得（ページネーション付き）
route.get("/", async (c) => {
  const db = createDb(c.env);
  const page = parseInt(c.req.query("page") ?? "1");
  const limit = parseInt(c.req.query("limit") ?? "20");
  const offset = (page - 1) * limit;

  // Drizzleクエリビルダー：型安全かつSQLに近い記述
  const result = await db
    .select({
      id: users.id,
      email: users.email,
      displayName: users.displayName,
      isPremium: users.isPremium,
      createdAt: users.createdAt,
    })
    .from(users)
    .orderBy(desc(users.createdAt))
    .limit(limit)
    .offset(offset);

  return c.json({ users: result, page, limit });
});

// ユーザー詳細取得（投稿数を含む）
route.get("/:id", async (c) => {
  const db = createDb(c.env);
  const userId = c.req.param("id");

  // JOINを使った集計クエリ
  const [user] = await db
    .select({
      id: users.id,
      email: users.email,
      displayName: users.displayName,
      avatarUrl: users.avatarUrl,
      isPremium: users.isPremium,
      createdAt: users.createdAt,
    })
    .from(users)
    .where(eq(users.id, userId))
    .limit(1);

  if (!user) {
    return c.json({ error: "User not found" }, 404);
  }

  // 投稿数を別クエリで取得（シンプルに保つ）
  const postCount = await db.$count(posts, eq(posts.userId, userId));

  return c.json({ ...user, postCount });
});

// ユーザー作成（バリデーション付き）
route.post(
  "/",
  zValidator("json", insertUserSchema),
  async (c) => {
    const db = createDb(c.env);
    const body = c.req.valid("json");

    const [newUser] = await db
      .insert(users)
      .values(body)
      .returning({
        id: users.id,
        email: users.email,
        displayName: users.displayName,
        createdAt: users.createdAt,
      });

    return c.json(newUser, 201);
  }
);

export default route;
```

### 投稿の取得・作成・更新（トランザクション対応）

```typescript
// src/routes/posts.ts（抜粋）
import { eq, desc, and } from "drizzle-orm";
import { posts, likes, users } from "../db/schema";

// いいね機能（トランザクションで原子的に更新）
route.post("/:id/like", async (c) => {
  const db = createDb(c.env);
  const postId = c.req.param("id");
  const userId = c.req.header("X-User-Id"); // 認証ミドルウェアから取得

  if (!userId) {
    return c.json({ error: "Unauthorized" }, 401);
  }

  // Drizzle ORMのトランザクション
  const result = await db.transaction(async (tx) => {
    // すでにいいね済みかチェック
    const [existing] = await tx
      .select()
      .from(likes)
      .where(and(eq(likes.userId, userId), eq(likes.postId, postId)))
      .limit(1);

    if (existing) {
      // いいね解除
      await tx.delete(likes).where(eq(likes.id, existing.id));
      const [updated] = await tx
        .update(posts)
        .set({ likeCount: posts.likeCount - 1 })  // ← 型安全な更新
        .where(eq(posts.id, postId))
        .returning({ likeCount: posts.likeCount });
      return { liked: false, likeCount: updated.likeCount };
    } else {
      // いいね追加
      await tx.insert(likes).values({ userId, postId });
      const [updated] = await tx
        .update(posts)
        .set({ likeCount: posts.likeCount + 1 })
        .where(eq(posts.id, postId))
        .returning({ likeCount: posts.likeCount });
      return { liked: true, likeCount: updated.likeCount };
    }
  });

  return c.json(result);
});
```

## 認証との統合

### Clerk を使ったJWT検証

Rork Maxで生成したアプリにClerkを使っている場合、Honoミドルウェアでトークンを検証します。

```typescript
// src/middleware/auth.ts
import { createMiddleware } from "hono/factory";
import { verifyToken } from "@clerk/backend";

export const clerkAuth = createMiddleware<{ Bindings: Env; Variables: { userId: string } }>(
  async (c, next) => {
    const authHeader = c.req.header("Authorization");
    if (!authHeader?.startsWith("Bearer ")) {
      return c.json({ error: "Unauthorized" }, 401);
    }

    const token = authHeader.slice(7);
    try {
      const payload = await verifyToken(token, {
        secretKey: c.env.CLERK_SECRET_KEY,
      });
      c.set("userId", payload.sub);
      await next();
    } catch {
      return c.json({ error: "Invalid token" }, 401);
    }
  }
);
```

ルートに適用：

```typescript
// 認証が必要なルートに適用
route.use("/*", clerkAuth);

// 認証済みユーザーのIDをコンテキストから取得
route.get("/me", async (c) => {
  const userId = c.get("userId");
  const db = createDb(c.env);
  // ... ユーザー情報を取得
});
```

## パフォーマンス最適化とキャッシュ戦略

### Neonのコネクションプーリング活用

Neonのコンソールから「Connection Pooling」を有効化し、`pgbouncer` モードの接続URLを取得します。高トラフィック時は通常の接続URLをPooler URLに切り替えるだけで対応できます。

```toml
# wrangler.toml
[vars]
# PoolerのURLを使用（-pooler サフィックスのエンドポイント）
# DATABASE_URL はシークレットで設定済み
```

### Cloudflare Cache API によるレスポンスキャッシュ

```typescript
// よく参照される公開データのキャッシュ
route.get("/popular", async (c) => {
  const cacheKey = new Request("https://cache/popular-posts", c.req.raw);
  const cache = caches.default;

  // キャッシュヒット確認
  let response = await cache.match(cacheKey);
  if (response) {
    return response;
  }

  // DBからデータ取得
  const db = createDb(c.env);
  const popularPosts = await db
    .select()
    .from(posts)
    .where(eq(posts.isPublished, true))
    .orderBy(desc(posts.likeCount))
    .limit(10);

  response = c.json(popularPosts);

  // 60秒キャッシュ（Cache-Control ヘッダーで制御）
  const cachedResponse = new Response(response.body, response);
  cachedResponse.headers.set("Cache-Control", "public, max-age=60");
  c.executionCtx.waitUntil(cache.put(cacheKey, cachedResponse.clone()));

  return response;
});
```

### よくあるエラーと対処法

DBとの接続でよく発生するエラーと対処法をまとめます。

**エラー: `NeonDbError: endpoint is disabled`**
- 原因：Neonの無料プランでは非アクティブ時にブランチが停止する
- 対処：Neonコンソールでブランチのオートサスペンドを設定変更、または有料プランへアップグレード

**エラー: `drizzle-orm: column X does not exist`**
- 原因：マイグレーションが適用されていないか、スキーマ定義が実際のDBと不一致
- 対処：`npx drizzle-kit migrate` を再実行し、マイグレーション状態を確認する

**エラー: `Too many connections`**
- 原因：Cloudflare Workersがリクエストごとに新しい接続を作成している
- 対処：Neonのコネクションプーリング（pgbouncer）を有効化し、Pooler URLを使用する

**エラー: `CORS policy blocked`**
- 原因：Rork MaxのアプリのオリジンがバックエンドのCORS設定に含まれていない
- 対処：`wrangler.toml` の `ALLOWED_ORIGINS` 変数にアプリのURLを追加する

### GitHub Actions による CI/CD 自動化

```yaml
# .github/workflows/deploy.yml
name: Deploy to Cloudflare Workers

on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: "20"
      - run: npm ci
      - name: Run DB migrations
        env:
          DATABASE_URL: ${{ secrets.DATABASE_URL }}
        run: npx drizzle-kit migrate
      - name: Deploy to Workers
        env:
          CLOUDFLARE_API_TOKEN: ${{ secrets.CF_API_TOKEN }}
        run: npx wrangler deploy
```

## よくある質問（FAQ）

**Q1. Supabase と Neon はどちらを選ぶべきですか？**

Supabaseはオールインワンのバックエンドサービスで、認証・ストレージ・リアルタイム機能が最初から含まれています。一方、Neon + Drizzle ORMはデータベース層のみに特化した構成です。Rork Maxアプリで「リアルタイム同期が重要」「すぐ使えるRLSが欲しい」場合はSupabase、「バックエンドを細かくコントロールしたい」「Cloudflare Workersで完結させたい」場合はNeon + Drizzle ORMが向いています。両者を組み合わせる（Neonをストレージとして使い、Supabase Realtimeだけ利用する）設計も可能です。

**Q2. Drizzle ORM のクエリはパフォーマンスが良いですか？**

はい、Drizzle ORMはSQLに1対1で対応するクエリビルダーなので、生成されるSQLが予測可能で無駄がありません。N+1問題を防ぐためには `leftJoin` や `inArray` を適切に使います。`db.$client.query()` で生のSQL実行も可能なので、パフォーマンスクリティカルなクエリは生SQL + EXPLAIN ANALYZEで最適化できます。

**Q3. Rork Maxで既にFirebase/Supabaseを使っているアプリに後からNeon + Drizzleを追加できますか？**

可能です。Neon + Drizzle ORMは既存のFirebase・Supabaseと共存できます。例えば「認証はFirebase Auth、複雑なビジネスロジックを含むデータはNeon Postgres」という使い分けも現実的です。Rork MaxのAIに「既存のFirebase認証を維持しつつ、新しいエンドポイントだけHono + Neonで実装する」と指示することで、段階的な移行も可能です。

**Q4. 本番環境でのコストはどれくらいかかりますか？**

Neonの無料プランは月0.5 GB storage + 1 compute unitまで無料で、個人開発の初期フェーズには十分です。Cloudflare Workersも100,000リクエスト/日まで無料です。トラフィックが増えてきたらNeonの$19/月プラン（10 GB storage + 10 compute units）へ移行するのが一般的です。

**Q5. データベースの変更（カラム追加・削除）はどう管理しますか？**

スキーマを変更したら `npx drizzle-kit generate` でマイグレーションファイルを生成し、`npx drizzle-kit migrate` で適用します。カラム削除は既存データが失われるため、まず `isDeprecated` フラグを立てて段階的に廃止するのが安全です。マイグレーションファイルは必ずGitで管理し、レビュープロセスを経てから本番適用することを強く推奨します。

## まとめ

Neon Serverless Postgres + Drizzle ORM の組み合わせは、Rork Maxで生成したReact Nativeアプリのバックエンドとして、2026年現在で最も優れた選択肢の一つです。

本ガイドで解説した内容をまとめます。

- **Neon** はCloudflare Workers対応のサーバーレスPostgresで、エッジ環境からHTTP経由で接続できる
- **Drizzle ORM** はTypeScriptファーストで型安全なクエリビルダー、SQLに近い記述で予測可能な動作
- **スキーマ定義** をTypeScriptで書くことで、APIのリクエスト/レスポンス型も自動推論される
- **Drizzle Kit** でマイグレーションを管理し、スキーマ変更を安全に本番に反映できる
- **Hono + Cloudflare Workers** でAPIサーバーを実装し、Rork MaxのReact Nativeアプリから呼び出す
- **トランザクション・キャッシュ・認証** の実装パターンを習得することで、本番品質のバックエンドが構築できる

次のステップとして、[Rork Max Hono + Cloudflare Workers REST APIガイド](/articles/rork-dev/rork-hono-cloudflare-workers-rest-api-guide) でHonoの応用パターンを、[Rork Max tRPC + Cloudflare Workers 型安全バックエンドガイド](/articles/rork-dev/rork-trpc-cloudflare-workers-type-safe-backend-guide) でさらに高度な型安全APIを学ぶことをお勧めします。
