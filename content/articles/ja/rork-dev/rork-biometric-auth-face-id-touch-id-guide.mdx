## はじめに — なぜモバイルアプリに生体認証が必要なのか

スマートフォンの普及とともに、モバイルアプリにおけるセキュリティの重要性はますます高まっています。パスワードだけではフィッシングや漏洩のリスクを完全に防ぐことはできず、ユーザー体験の面でも毎回の入力は煩わしいものです。

そこで注目されるのが **生体認証（Biometric Authentication）** です。iOSの **Face ID** や **Touch ID**、Androidの **指紋認証** や **顔認証** を活用すれば、ユーザーは一瞬でアプリにログインでき、セキュリティと利便性を同時に実現できます。

Rork を使えば、React Native / Expo ベースの `expo-local-authentication` ライブラリを活用して、数十行のコードでこの仕組みをアプリに組み込めます。この記事では、生体認証の導入手順からセキュリティ設計のベストプラクティスまでを体系的に解説します。

## expo-local-authentication の概要と仕組み

Rork アプリで生体認証を実装する際に使うのが `expo-local-authentication` パッケージです。このパッケージは iOS と Android の両プラットフォームで統一的なAPIを提供し、以下の機能を備えています。

- **デバイスの生体認証対応チェック**: Face ID / Touch ID / 指紋センサーの有無を判定
- **認証レベルの確認**: ハードウェアレベルの生体認証とデバイスレベルの認証（PINやパターン）を区別
- **認証ダイアログの表示**: OSネイティブの認証UIを呼び出し、結果を取得

内部的には、iOSでは `LAContext`（LocalAuthentication.framework）、Androidでは `BiometricPrompt` API を利用しています。React Native / Expo のブリッジを経由するため、プラットフォーム固有のコードを書く必要がありません。

```typescript
// expo-local-authentication のインポート例
import * as LocalAuthentication from 'expo-local-authentication';

// 利用可能な認証タイプを確認
const types = await LocalAuthentication.supportedAuthenticationTypesAsync();
// [1] = FINGERPRINT, [2] = FACIAL_RECOGNITION, [3] = IRIS
console.log('利用可能な認証:', types);
```

## ステップ1 — プロジェクトのセットアップ

まずは Rork プロジェクトに `expo-local-authentication` をインストールします。

```bash
# Rork プロジェクトのターミナルで実行
npx expo install expo-local-authentication

# iOS の場合、Info.plist への権限追記が必要
# app.json または app.config.js に以下を追加
```

`app.json` に Face ID の使用理由を記述します（App Store審査で必須）。

```json
{
  "expo": {
    "ios": {
      "infoPlist": {
        "NSFaceIDUsageDescription": "セキュアなログインのためにFace IDを使用します"
      }
    }
  }
}
```

Android側は特別な設定は不要です。`expo-local-authentication` が自動的に `USE_BIOMETRIC` と `USE_FINGERPRINT` パーミッションをマニフェストに追加します。

## ステップ2 — 生体認証の対応チェック

ユーザーのデバイスが生体認証に対応しているかを事前に確認することは非常に重要です。対応していない場合のフォールバックも設計しておきましょう。

```typescript
import * as LocalAuthentication from 'expo-local-authentication';
import { Alert } from 'react-native';

// 生体認証の利用可能性をチェックする関数
async function checkBiometricSupport(): Promise<{
  isAvailable: boolean;
  authTypes: LocalAuthentication.AuthenticationType[];
  isEnrolled: boolean;
}> {
  // ハードウェアが生体認証に対応しているか
  const isAvailable = await LocalAuthentication.hasHardwareAsync();
  
  // 対応している認証タイプ一覧
  const authTypes = await LocalAuthentication.supportedAuthenticationTypesAsync();
  
  // 生体情報が登録されているか（Face ID / 指紋が設定済みか）
  const isEnrolled = await LocalAuthentication.isEnrolledAsync();

  return { isAvailable, authTypes, isEnrolled };
}

// 使用例
const support = await checkBiometricSupport();

if (!support.isAvailable) {
  Alert.alert('お知らせ', 'このデバイスは生体認証に対応していません');
} else if (!support.isEnrolled) {
  Alert.alert(
    '設定が必要です',
    'デバイスの設定からFace IDまたはTouch IDを登録してください'
  );
}
// 期待する出力例:
// { isAvailable: true, authTypes: [2], isEnrolled: true }
// authTypes: [2] は FACIAL_RECOGNITION（Face ID）を示す
```

## ステップ3 — 認証処理の実装

実際の認証処理を実装します。ユーザーにネイティブの認証ダイアログを表示し、成功・失敗を判定するコアロジックです。

```typescript
import * as LocalAuthentication from 'expo-local-authentication';

interface AuthResult {
  success: boolean;
  error?: string;
}

async function authenticateUser(): Promise<AuthResult> {
  try {
    const result = await LocalAuthentication.authenticateAsync({
      // ダイアログに表示するメッセージ
      promptMessage: 'アプリのロックを解除',
      // 生体認証失敗時にパスコード入力を許可するか
      fallbackLabel: 'パスコードで認証',
      // 認証失敗時にキャンセルボタンを表示するか
      cancelLabel: 'キャンセル',
      // デバイスのパスコードをフォールバックとして使用
      disableDeviceFallback: false,
    });

    if (result.success) {
      console.log('認証成功');
      return { success: true };
    } else {
      console.log('認証失敗:', result.error);
      return {
        success: false,
        error: result.error ?? '認証に失敗しました',
      };
    }
  } catch (error) {
    console.error('認証エラー:', error);
    return {
      success: false,
      error: '認証処理中にエラーが発生しました',
    };
  }
}
// 期待する出力（成功時）: { success: true }
// 期待する出力（失敗時）: { success: false, error: "user_cancel" }
```

`disableDeviceFallback` を `false` にしておくと、生体認証が何度か失敗した場合にデバイスのパスコード（PINやパターン）で認証できるようになります。セキュリティレベルを維持しつつ、ユーザーがロックアウトされることを防ぐバランスのよい設定です。

## ステップ4 — React コンポーネントへの統合

認証ロジックをReactコンポーネントとして組み立て、アプリのロック画面を実装します。

```tsx
import React, { useState, useEffect, useCallback } from 'react';
import { View, Text, TouchableOpacity, StyleSheet, ActivityIndicator } from 'react-native';
import * as LocalAuthentication from 'expo-local-authentication';

export default function BiometricLockScreen({ onAuthenticated }: {
  onAuthenticated: () => void;
}) {
  const [isLoading, setIsLoading] = useState(true);
  const [biometricType, setBiometricType] = useState<string>('');
  const [errorMessage, setErrorMessage] = useState('');

  useEffect(() => {
    (async () => {
      const types = await LocalAuthentication.supportedAuthenticationTypesAsync();
      if (types.includes(LocalAuthentication.AuthenticationType.FACIAL_RECOGNITION)) {
        setBiometricType('Face ID');
      } else if (types.includes(LocalAuthentication.AuthenticationType.FINGERPRINT)) {
        setBiometricType('Touch ID');
      } else {
        setBiometricType('パスコード');
      }
      setIsLoading(false);
    })();
  }, []);

  const handleAuthenticate = useCallback(async () => {
    setErrorMessage('');
    const result = await LocalAuthentication.authenticateAsync({
      promptMessage: `${biometricType}で認証してください`,
      fallbackLabel: 'パスコードを使用',
      disableDeviceFallback: false,
    });

    if (result.success) {
      onAuthenticated();
    } else if (result.error !== 'user_cancel') {
      setErrorMessage('認証に失敗しました。もう一度お試しください。');
    }
  }, [biometricType, onAuthenticated]);

  // 初回マウント時に自動で認証ダイアログを表示
  useEffect(() => {
    if (!isLoading && biometricType) {
      handleAuthenticate();
    }
  }, [isLoading, biometricType, handleAuthenticate]);

  if (isLoading) {
    return (
      <View style={styles.container}>
        <ActivityIndicator size="large" color="#007AFF" />
      </View>
    );
  }

  return (
    <View style={styles.container}>
      <Text style={styles.lockIcon}>🔒</Text>
      <Text style={styles.title}>アプリがロックされています</Text>
      <Text style={styles.subtitle}>
        {biometricType}で認証してください
      </Text>
      {errorMessage ? (
        <Text style={styles.error}>{errorMessage}</Text>
      ) : null}
      <TouchableOpacity style={styles.button} onPress={handleAuthenticate}>
        <Text style={styles.buttonText}>{biometricType}で解除</Text>
      </TouchableOpacity>
    </View>
  );
}

const styles = StyleSheet.create({
  container: {
    flex: 1, justifyContent: 'center',
    alignItems: 'center', backgroundColor: '#F5F5F7',
  },
  lockIcon: { fontSize: 64, marginBottom: 24 },
  title: { fontSize: 22, fontWeight: '700', marginBottom: 8, color: '#1D1D1F' },
  subtitle: { fontSize: 16, color: '#6E6E73', marginBottom: 24 },
  error: { fontSize: 14, color: '#FF3B30', marginBottom: 16 },
  button: {
    backgroundColor: '#007AFF', paddingHorizontal: 32,
    paddingVertical: 14, borderRadius: 12,
  },
  buttonText: { color: '#FFFFFF', fontSize: 16, fontWeight: '600' },
});
```

## セキュリティ設計のベストプラクティス

生体認証を導入する際には、単にAPIを呼ぶだけでなく、セキュリティ全体の設計を考慮することが大切です。

### 認証トークンの安全な保存

生体認証に成功した後、アクセストークンやセッション情報を安全に保管する必要があります。React Native では `expo-secure-store` を使うことで、iOSの **Keychain** やAndroidの **Encrypted SharedPreferences** にデータを暗号化して保存できます。

```typescript
import * as SecureStore from 'expo-secure-store';

// 認証成功後にトークンを安全に保存
async function saveAuthToken(token: string): Promise<void> {
  await SecureStore.setItemAsync('auth_token', token, {
    keychainAccessible: SecureStore.WHEN_UNLOCKED_THIS_DEVICE_ONLY,
  });
}

// アプリ起動時にトークンを取得
async function getAuthToken(): Promise<string | null> {
  return await SecureStore.getItemAsync('auth_token');
}
// 期待する出力: "eyJhbGciOiJIUzI1NiIs..." のようなJWTトークン文字列
```

### セキュリティレベルの段階設計

アプリ内の機能ごとにセキュリティレベルを変えることで、利便性とセキュリティのバランスを最適化できます。

- **レベル1（低）**: アプリの起動時 — デバイスパスコードでも認証OK
- **レベル2（中）**: 個人情報の閲覧 — 生体認証を要求
- **レベル3（高）**: 決済・送金 — 生体認証 + サーバー側の追加検証

### タイムアウトと再認証のポリシー

バックグラウンドからの復帰時に再認証を求めるかどうかは、アプリの性質によって判断します。金融系アプリなら即座に再認証すべきですし、メモアプリなら5分間の猶予を設けるのが自然です。

```typescript
// バックグラウンド復帰時の再認証ロジック例
import { AppState, AppStateStatus } from 'react-native';

const AUTH_TIMEOUT_MS = 5 * 60 * 1000; // 5分
let lastAuthTime = Date.now();

AppState.addEventListener('change', (state: AppStateStatus) => {
  if (state === 'active') {
    const elapsed = Date.now() - lastAuthTime;
    if (elapsed > AUTH_TIMEOUT_MS) {
      // 再認証を要求
      requireReAuthentication();
    }
  }
});

function requireReAuthentication() {
  // ロック画面を表示
  console.log('セッションタイムアウト: 再認証が必要です');
}
```

## フォールバック戦略 — 生体認証が使えない場合

すべてのユーザーが生体認証を利用できるわけではありません。以下のケースに対応するフォールバック設計が不可欠です。

- 古いデバイスで生体認証ハードウェアが非搭載
- ユーザーがFace ID / Touch IDを設定していない
- 生体情報の変更後（指紋の追加など）にリセットが必要な場合
- 怪我や手袋などで一時的に生体認証が使えない場合

フォールバックの優先順位は次のようにするのが一般的です。

1. **生体認証**（Face ID / Touch ID / 指紋）
2. **デバイスパスコード**（PIN / パターン）
3. **アプリ内PIN**（独自の4〜6桁コード）
4. **パスワード認証**（最終手段）

このような認証フローの設計は、[Rork × Firebase / Supabase 認証ガイド](/articles/rork-dev/rork-auth-firebase-supabase) と組み合わせることで、バックエンド連携を含めた包括的な認証システムを構築できます。

## テストとデバッグのポイント

生体認証はシミュレーター/エミュレーターでもテスト可能です。

- **iOS シミュレーター**: `Features > Face ID > Enrolled` で有効化し、`Features > Face ID > Matching Face / Non-matching Face` で認証結果を制御
- **Android エミュレーター**: `Extended Controls > Fingerprint` でシミュレーション可能

実機テストでは、以下のケースを必ず確認してください。

- Face ID / Touch ID の初回認証成功
- 認証失敗後のフォールバック（パスコード入力）
- ユーザーキャンセル時の挙動
- バックグラウンド復帰後の再認証
- 生体情報が未登録の状態でアプリを起動

Rork のプレビュー機能を使えば、実機での動作確認がスムーズに行えます。Rork Max をお使いの場合は、[Rork Max セキュリティベストプラクティス](/articles/rork-dev/rork-max-security-best-practices) も参考に、本番環境に適したセキュリティ設定を確認しておくとよいでしょう。

## よくある質問（FAQ）

### Q1. Face ID と Touch ID を同時にサポートする必要がありますか？

はい。`expo-local-authentication` は内部的にプラットフォームのAPIを自動選択するため、コード上で Face ID と Touch ID を区別する必要はありません。`authenticateAsync()` を呼ぶだけで、デバイスに搭載されている認証方式が自動的に使われます。ただし、UIのラベルやアイコンをユーザーフレンドリーにするために、`supportedAuthenticationTypesAsync()` で認証タイプを確認し、表示を切り替えることを推奨します。

### Q2. 生体認証だけでアプリのセキュリティは十分ですか？

生体認証はローカルデバイスでの本人確認には優れていますが、サーバー側の認証（JWT、セッション管理など）と組み合わせて使うのがベストプラクティスです。生体認証はあくまで「デバイスを持っている本人」を確認する手段であり、ネットワーク経由のなりすましには別の対策が必要です。

### Q3. 生体認証の認証結果をサーバーに送信する際の注意点は？

生体認証の成功/失敗はクライアント側で完結します。サーバーに「生体認証が成功した」というフラグだけを送るのはセキュリティ上危険です。正しい設計は、生体認証の成功後に `expo-secure-store` から取り出した認証トークン（JWTなど）をサーバーに送信し、サーバー側でトークンの有効性を検証する方式です。

### Q4. App Store / Google Play の審査で注意すべき点はありますか？

iOSでは `NSFaceIDUsageDescription`（Face ID の使用理由）を `Info.plist` に必ず記載してください。この記述がないとApp Storeの審査でリジェクトされます。また、生体認証を必須にする場合は、対応していないデバイスのユーザーがアプリを利用できなくなるため、フォールバック手段を必ず用意してください。

## まとめ

Rork と `expo-local-authentication` を組み合わせることで、Face ID や Touch ID による生体認証をモバイルアプリに手軽に導入できます。ポイントをまとめると以下の通りです。

- `hasHardwareAsync()` と `isEnrolledAsync()` で事前チェックを行い、フォールバックを設計する
- `authenticateAsync()` のオプションでユーザー体験を細かくカスタマイズできる
- 認証トークンは `expo-secure-store` で安全に保管し、サーバー側の認証と組み合わせる
- バックグラウンド復帰時の再認証ポリシーを、アプリの用途に合わせて適切に設計する

セキュリティはアプリの信頼性に直結する重要な要素です。ぜひこの記事を参考に、ユーザーにとって安全で快適な認証体験を実装してみてください。
