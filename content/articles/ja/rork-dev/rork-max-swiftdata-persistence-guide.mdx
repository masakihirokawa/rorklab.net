## SwiftData とは何か

SwiftData は Apple が WWDC23 で発表した次世代のデータ永続化フレームワークです。従来の Core Data が持っていた複雑な設定やボイラープレートコードを大幅に削減し、Swift のマクロシステムを活用した宣言的なモデル定義を実現しています。

Rork Max で生成されるネイティブ Swift アプリでは、iOS 17 以降をターゲットにすることで SwiftData を採用でき、より少ないコードで堅牢なデータ管理を実装できます。Core Data と比べてラーニングコストが低く、SwiftUI との統合もシームレスなため、個人開発者にとって特に魅力的な選択肢です。

---

## Core Data との違い

SwiftData と Core Data はどちらも Apple 純正の永続化ソリューションですが、開発体験には大きな差があります。

Core Data では `.xcdatamodeld` ファイルでエンティティを定義し、`NSManagedObjectContext` や `NSFetchRequest` など専用の API を使ってデータを操作する必要がありましました。設定ファイルの管理やコンテキストの受け渡しなど、理解すべき概念が多い点がハードルでしました。

一方 SwiftData では、通常の Swift クラスに `@Model` マクロを付けるだけでモデルを定義できます。フェッチには `@Query` プロパティラッパーを使い、SwiftUI のビューから直接データにアクセスできます。設定ファイルは不要で、コードだけでスキーマを完結させられるのが最大の利点です。

既存の Core Data プロジェクトがある場合でも、SwiftData は内部的に Core Data のストレージレイヤーを使用しているため、段階的な移行が可能です。

## @Model マクロによるモデル定義

SwiftData の核となるのが `@Model` マクロです。通常の Swift クラスをデータモデルとして扱えるようにします。

```swift
import SwiftData

@Model
class Book {
    var title: String
    var author: String
    var genre: String
    var finishedDate: Date?
    var memo: String
    var rating: Int
    var createdAt: Date

    init(
        title: String,
        author: String,
        genre: String = "未分類",
        finishedDate: Date? = nil,
        memo: String = "",
        rating: Int = 0
    ) {
        self.title = title
        self.author = author
        self.genre = genre
        self.finishedDate = finishedDate
        self.memo = memo
        self.rating = rating
        self.createdAt = Date()
    }
}
```

Core Data のように `.xcdatamodeld` ファイルを編集する必要はありません。プロパティの型がそのままスキーマになるため、コードとデータ構造の不整合が起きにくいのが特長です。

### 属性のカスタマイズ

特定のプロパティにインデックスを追加したり、ユニーク制約を設定することもできます。

```swift
@Model
class Book {
    @Attribute(.unique) var isbn: String
    @Attribute(.spotlight) var title: String
    var author: String
    // ...
}
```

`@Attribute(.unique)` を付けると、同じ値を持つレコードの重複挿入を防げます。`@Attribute(.spotlight)` は Spotlight 検索にインデックスを登録します。

## リレーションシップの定義

モデル間のリレーションシップも、Swift の通常のプロパティとして定義できます。

```swift
@Model
class Author {
    var name: String
    var bio: String

    @Relationship(deleteRule: .cascade)
    var books: [Book] = []

    init(name: String, bio: String = "") {
        self.name = name
        self.bio = bio
    }
}

@Model
class Book {
    var title: String
    var author: Author?
    // ...
}
```

`@Relationship` マクロで削除ルールを指定できます。`.cascade` を設定すると、著者を削除したときに関連する本のデータも一緒に削除されます。`.nullify`（デフォルト）では関連プロパティが `nil` に設定されるだけで、子レコードは残ります。

## ModelContainer と ModelContext

SwiftData でデータを操作するには、`ModelContainer`（ストレージの設定）と `ModelContext`（操作のコンテキスト）が必要です。

### アプリのエントリーポイントで設定

```swift
import SwiftUI
import SwiftData

@main
struct ReadingApp: App {
    var body: some Scene {
        WindowGroup {
            ContentView()
        }
        .modelContainer(for: [Book.self, Author.self])
    }
}
```

`.modelContainer(for:)` モディファイアを使うと、指定したモデル型に対応するストレージが自動的に構成されます。この一行で、Core Data でいうところの `NSPersistentContainer` の設定が完了します。

### ModelContext の取得と操作

ビュー内では `@Environment` プロパティラッパーで `ModelContext` を取得し、CRUD 操作を実行します。

```swift
struct AddBookView: View {
    @Environment(\.modelContext) private var context

    func addBook() {
        let newBook = Book(
            title: "Swift実践入門",
            author: "田中太郎"
        )
        context.insert(newBook)
    }

    func deleteBook(_ book: Book) {
        context.delete(book)
    }
}
```

`context.insert()` でデータを追加し、`context.delete()` で削除します。変更は自動的に保存されるため、明示的な `save()` 呼び出しは通常不要です。

## @Query によるデータフェッチ

SwiftData の `@Query` プロパティラッパーを使うと、データの取得とビューの更新が自動的に連動します。

```swift
struct BookListView: View {
    @Query(sort: \Book.finishedDate, order: .reverse)
    private var books: [Book]

    var body: some View {
        List(books) { book in
            BookRowView(book: book)
        }
    }
}
```

`@Query` に渡せるパラメータは豊富です。

```swift
// フィルタ付きクエリ
@Query(
    filter: #Predicate<Book> { $0.rating >= 4 },
    sort: \Book.finishedDate,
    order: .reverse
)
private var highRatedBooks: [Book]

// 件数制限
@Query(
    sort: \Book.createdAt,
    order: .reverse,
    limit: 10
)
private var recentBooks: [Book]
```

`#Predicate` マクロを使ったフィルタリングは型安全で、コンパイル時にエラーを検出できます。Core Data の `NSPredicate` と比べて、Swift ネイティブの式で直感的に条件を記述できる点が大きなメリットです。

---

## 動的フィルタリングの実装

ユーザーの入力に応じてクエリ条件を変えたい場合は、ビューの初期化時に `@Query` を動的に構成します。

```swift
struct FilteredBookListView: View {
    @Query private var books: [Book]

    init(genre: String) {
        let predicate = #Predicate<Book> {
            $0.genre == genre
        }
        _books = Query(
            filter: predicate,
            sort: \Book.finishedDate,
            order: .reverse
        )
    }

    var body: some View {
        List(books) { book in
            BookRowView(book: book)
        }
    }
}
```

親ビューから `FilteredBookListView(genre: "小説")` のように呼び出すことで、ジャンルに応じた結果が表示されます。

## iCloud 同期の有効化

SwiftData は CloudKit との連携もシンプルです。`ModelConfiguration` で iCloud コンテナを指定するだけで、デバイス間のデータ同期が実現します。

```swift
@main
struct ReadingApp: App {
    var body: some Scene {
        WindowGroup {
            ContentView()
        }
        .modelContainer(for: [Book.self, Author.self], cloudKitDatabase: .automatic)
    }
}
```

Rork Max のプロンプトに「iCloud 同期を有効にしてほしい」と追記するだけで、この設定を含むコードが生成されます。

iCloud 同期を使う場合の注意点として、`@Attribute(.unique)` 制約は CloudKit と互換性がない場合があります。ユニーク制約が必要な場合は、アプリ側でバリデーションロジックを実装するのがベターです。

## データマイグレーション

アプリのアップデートに伴ってモデルの構造を変更する場合、SwiftData はスキーママイグレーションの仕組みを提供しています。

```swift
enum BookSchemaV1: VersionedSchema {
    static var versionIdentifier: Schema.Version = Schema.Version(1, 0, 0)
    static var models: [any PersistentModel.Type] {
        [BookV1.self]
    }

    @Model
    class BookV1 {
        var title: String
        var author: String
        init(title: String, author: String) {
            self.title = title
            self.author = author
        }
    }
}

enum BookSchemaV2: VersionedSchema {
    static var versionIdentifier: Schema.Version = Schema.Version(2, 0, 0)
    static var models: [any PersistentModel.Type] {
        [BookV2.self]
    }

    @Model
    class BookV2 {
        var title: String
        var author: String
        var genre: String
        init(title: String, author: String, genre: String = "未分類") {
            self.title = title
            self.author = author
            self.genre = genre
        }
    }
}
```

`SchemaMigrationPlan` を定義して、バージョン間の移行ステップを明示できます。軽量マイグレーション（プロパティの追加やデフォルト値の設定）であれば、SwiftData が自動的に処理してくれます。

## パフォーマンス最適化のポイント

SwiftData を使ったアプリのパフォーマンスを最適化するためのポイントをまとめます。

**遅延読み込みの活用**: SwiftData はリレーションシップのデータを遅延読み込みします。大量のデータを持つリレーションがあっても、実際にアクセスされるまでメモリを消費しません。

**フェッチ件数の制限**: `@Query` に `limit` パラメータを設定し、一度に取得するデータ量を制御します。無限スクロールのような UI では、ページネーションを意識した設計が重要です。

**バックグラウンドでの操作**: 大量のデータをインポートする場合は、メインスレッドをブロックしないよう `ModelActor` を活用します。

```swift
@ModelActor
actor BookImporter {
    func importBooks(from data: [BookData]) {
        for item in data {
            let book = Book(
                title: item.title,
                author: item.author
            )
            modelContext.insert(book)
        }
        try? modelContext.save()
    }
}
```

`@ModelActor` マクロを使うことで、バックグラウンドスレッドで安全にデータ操作を実行できます。UI のレスポンスを損なうことなく、数千件規模のデータインポートにも対応可能です。

## 実装時のよくある注意点

SwiftData を Rork Max で活用する際に知っておきたいポイントをいくつか紹介します。

**iOS 17 以上が必須**: SwiftData は iOS 17 以降でのみ利用可能です。iOS 16 以前もサポートする必要がある場合は、Core Data を選択してください。Rork Max のプロンプトでターゲット OS バージョンを明示すると、適切なフレームワークが選択されます。

**プレビューでのテスト**: SwiftUI Previews で SwiftData を使う場合、インメモリコンテナを設定すると便利です。

```swift
#Preview {
    BookListView()
        .modelContainer(for: Book.self, inMemory: true)
}
```

**Optional の扱い**: SwiftData のモデルプロパティで `Optional` を使う場合、フェッチ結果のフィルタリングで `nil` チェックが必要になります。可能な限りデフォルト値を設定し、Optional を避ける設計がシンプルです。

## まとめ

SwiftData は Apple のデータ永続化を根本から簡素化するフレームワークであり、Rork Max との相性も抜群です。`@Model` マクロによる宣言的なモデル定義、`@Query` による自動更新、そして iCloud 同期の容易さは、個人開発者がアプリのデータ層を素早く構築するうえで大きな助けになります。

Core Data の経験がなくても、Swift の基本文法を理解していれば SwiftData はすぐに使いこなせます。Rork Max のプロンプトに「SwiftData を使ってデータを保存してほしい」と一言添えるだけで、適切な実装が生成されるので、ぜひ次のアプリ開発で試してみてください。
