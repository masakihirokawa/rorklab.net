「テストビルドできたよ」と Slack に書き込んで、.ipa や .apk を手動でアップロードして、インストール方法を説明して——個人開発でこのサイクルを繰り返しているうちに、本来コードを書く時間がどこかへ消えていく経験、きっと一度はあるのではないでしょうか。

しかもクラッシュが出た日には「どの画面で落ちましたか？」「何をタップしましたか？」と聞いて回ることになります。テスターが覚えていればラッキーで、大抵は「なんか落ちました」の一言で終わります。

Rork で生成したアプリをプロダクション品質に持っていくには、この QA の非効率さを解消することが最初の壁です。Firebase App Distribution と Crashlytics、そして GitHub Actions の三つを組み合わせると、push するだけでテスターに届き、クラッシュが起きた瞬間に詳細なログが手元に来る仕組みが作れます。私が実際に個人開発アプリで試した経験を踏まえながら、設定から本番運用まで丁寧に解説していきます。

## なぜこの三つの組み合わせなのか

Firebase App Distribution はベータ配布の自動化ツール、Crashlytics はクラッシュ解析ツール、GitHub Actions は CI/CD の基盤——それぞれ単体で使うことはできますが、三つを連携させたときに「1人でも回せる品質保証サイクル」が完成します。

[Sentry を使ったクラッシュ監視](/articles/rork-dev/rork-sentry-crash-monitoring-setup-guide)も有力な選択肢ですが、Firebase を選ぶ理由は別にあります。App Distribution、Crashlytics、Performance Monitoring が同じ Firebase コンソールで完結するため、ツールをまたいで情報を追う必要がありません。また、`@react-native-firebase` ライブラリが Rork で生成したコードと素直に共存します。

もう一点：Crashlytics は永続的に無料です。この点は Sentry の無料枠（イベント数に上限あり）と大きく異なります。個人開発者として費用対効果を意識するなら、クラッシュ分析ツールに毎月費用がかからないのは意外と大きいです。

三つの役割をまとめると、

- **GitHub Actions**：push をトリガーに自動でビルドを走らせる司令塔
- **Firebase App Distribution**：ビルド成果物をテスターへ自動配布するデリバリー担当
- **Crashlytics**：本番・ベータ問わずクラッシュをリアルタイムで収集する見張り番

この分業体制が、1人での開発でも品質を保ち続けられる仕組みの核心です。

## Firebase プロジェクトの作成と Rork への統合

まず Firebase コンソール（console.firebase.google.com）でプロジェクトを作成し、iOS と Android 両方のアプリを登録します。

iOS の場合、バンドル ID は Rork の `app.json` に書いてある `bundleIdentifier` と完全に一致させます。Android は `package` の値を使います。

```json
// app.json（Rork が生成する形式）
{
  "expo": {
    "name": "MyApp",
    "slug": "my-app",
    "ios": {
      "bundleIdentifier": "com.yourname.myapp"
    },
    "android": {
      "package": "com.yourname.myapp"
    }
  }
}
```

Firebase コンソールから `GoogleService-Info.plist`（iOS）と `google-services.json`（Android）をダウンロードします。この2ファイルをどこに置くかで後々詰まる人が多いのですが、Expo managed workflow の場合はプロジェクトルートに置き、`app.json` で参照します。

```json
// app.json に追加する Firebase 設定
{
  "expo": {
    "plugins": [
      "@react-native-firebase/app",
      "@react-native-firebase/crashlytics"
    ],
    "ios": {
      "googleServicesFile": "./GoogleService-Info.plist"
    },
    "android": {
      "googleServicesFile": "./google-services.json"
    }
  }
}
```

次に必要なパッケージをインストールします。Rork Max を使っている場合、ネイティブモジュールは EAS Build で処理されます。

```bash
# Rork プロジェクトのルートで実行
npx expo install @react-native-firebase/app @react-native-firebase/crashlytics

# Performance Monitoring も使う場合（推奨）
npx expo install @react-native-firebase/perf
```

ここで注意点が一つあります。`@react-native-firebase` はネイティブコードを含むため、Expo Go では動作しません。EAS Build でカスタムビルドが必要です。「Rork Companion でプレビューしたら動かない」という問い合わせの多くはここが原因なので、EAS Build の設定を先に済ませておくことをお勧めします。EAS Build の基本設定については[Rork + EAS Build の基本ガイド](/articles/rork-dev/rork-github-actions-eas-build-cicd)も参考になります。

## Crashlytics の初期設定：クラッシュを自動的に捕捉する

パッケージを入れただけでは Crashlytics は機能しません。アプリのエントリポイント（通常は `App.tsx` または `_layout.tsx`）に初期化コードを追加します。

```typescript
// app/_layout.tsx（Expo Router を使っている場合）
import { useEffect } from 'react';
import crashlytics from '@react-native-firebase/crashlytics';
import { Stack } from 'expo-router';

export default function RootLayout() {
  useEffect(() => {
    // Crashlytics の収集を明示的に有効化
    crashlytics().setCrashlyticsCollectionEnabled(true);

    // ユーザー ID を設定（認証済みの場合のみ）
    // これにより「このユーザーだけクラッシュしている」が追えるようになる
    const setupCrashlytics = async () => {
      try {
        // 独自の認証ストアから取得する例
        const userId = await getStoredUserId();
        if (userId) {
          await crashlytics().setUserId(userId);
        }
        // アプリのバージョンもカスタムキーに記録しておく
        await crashlytics().setAttribute(
          'app_version',
          process.env.EXPO_PUBLIC_APP_VERSION ?? 'unknown'
        );
      } catch (e) {
        // Crashlytics の初期化失敗でアプリを落とさない
        console.warn('Crashlytics setup failed:', e);
      }
    };

    setupCrashlytics();
  }, []);

  return <Stack />;
}
```

これだけで、次回クラッシュ時に Firebase コンソールにレポートが届きます。ただし「届くこと」と「原因がわかること」は別物です。デフォルトのクラッシュレポートはスタックトレースだけで、「なぜそこに来たか」という文脈がありません。

### Crashlytics を開発環境でテストする方法

`__DEV__` モードではクラッシュレポートが送信されません。テスト時は以下のコードで強制的にクラッシュを起こしてレポートを確認できます。

```typescript
// 開発時のテスト専用ボタン（本番では除去する）
import crashlytics from '@react-native-firebase/crashlytics';

const testCrash = async () => {
  // 非致命的エラーのテスト
  await crashlytics().recordError(new Error('Test non-fatal error'));

  // 致命的クラッシュのテスト（アプリが落ちる）
  crashlytics().crash();
};
```

Firebase コンソールにレポートが届くまで数分かかることがあります。テスト後に Crashlytics タブを確認して、イベントが記録されていれば設定は正常です。

## カスタムキーとエラーバウンダリーで文脈を記録する

Crashlytics の真価はカスタムキーにあります。クラッシュが起きた時点での「アプリの状態」を一緒に送ることで、再現が格段に楽になります。

よく使うパターンは「画面遷移のたびに現在の画面名を記録する」ことです。

```typescript
// hooks/useCrashlyticsTracking.ts
import { useEffect } from 'react';
import { usePathname } from 'expo-router';
import crashlytics from '@react-native-firebase/crashlytics';

export function useCrashlyticsTracking() {
  const pathname = usePathname();

  useEffect(() => {
    const track = async () => {
      try {
        // 現在の画面をカスタムキーに記録
        await crashlytics().setAttribute('current_screen', pathname);
        // Crashlytics のブレッドクラムログにも残す
        // （最後の N 件がクラッシュレポートに含まれる）
        crashlytics().log(`Screen: ${pathname} at ${new Date().toISOString()}`);
      } catch {
        // トラッキング失敗でアプリに影響させない
      }
    };

    track();
  }, [pathname]);
}

// RootLayout などで呼び出す
// useCrashlyticsTracking();
```

エラーバウンダリーと組み合わせると、JavaScript レイヤーで捕捉したエラーも Crashlytics に送れます。

```typescript
// components/CrashBoundary.tsx
import React, { Component, ErrorInfo, ReactNode } from 'react';
import crashlytics from '@react-native-firebase/crashlytics';

interface Props {
  children: ReactNode;
  fallback?: ReactNode;
  screenName?: string;
}

interface State {
  hasError: boolean;
}

export class CrashBoundary extends Component<Props, State> {
  constructor(props: Props) {
    super(props);
    this.state = { hasError: false };
  }

  static getDerivedStateFromError(): State {
    return { hasError: true };
  }

  componentDidCatch(error: Error, errorInfo: ErrorInfo) {
    // Crashlytics に非致命的エラーとして記録
    crashlytics().recordError(error);

    // コンポーネントスタックと画面名をログに追加
    crashlytics().log(
      `[${this.props.screenName ?? 'Unknown'}] ` +
      `ComponentStack: ${errorInfo.componentStack?.slice(0, 500) ?? 'n/a'}`
    );

    // エラー発生時のコンテキストをカスタムキーに記録
    crashlytics().setAttributes({
      error_boundary: 'triggered',
      screen: this.props.screenName ?? 'unknown',
      // message は 1024 文字以内に収める
      error_summary: error.message.slice(0, 256),
    });
  }

  render() {
    if (this.state.hasError) {
      return this.props.fallback ?? null;
    }
    return this.props.children;
  }
}
```

`recordError()` で送ったエラーは「致命的でないエラー（Non-fatal）」として分類されます。アプリが落ちなくても「表示がおかしい」「API エラーが出た」といった問題を追跡でき、クラッシュ率を下げながら体験品質も改善できます。

一点だけ注意：`error.message` に個人情報が含まれることがあります。エラーメッセージをそのまま送る前に、メールアドレスや電話番号といったセンシティブな文字列が混入していないか確認してください。

## GitHub Actions で配布を完全自動化する

ここからが本番です。`main` ブランチへの push をトリガーに、自動で EAS Build → Firebase App Distribution への配布まで行うワークフローを作ります。

```yaml
# .github/workflows/distribute.yml
name: Build and Distribute (Android)

on:
  push:
    branches: [main]
  workflow_dispatch: # 手動実行も可能にしておく

jobs:
  build-and-distribute:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'

      - name: Install dependencies
        run: npm ci

      - name: Setup Expo / EAS
        uses: expo/expo-github-action@v8
        with:
          eas-version: latest
          token: ${{ secrets.EXPO_TOKEN }}

      # google-services.json をシークレットから復元
      - name: Restore google-services.json
        run: |
          echo "${{ secrets.GOOGLE_SERVICES_JSON_BASE64 }}" \
            | base64 --decode > ./google-services.json

      # EAS Build で APK をビルド（preview プロファイル）
      - name: Build Android APK
        id: build
        run: |
          eas build \
            --platform android \
            --profile preview \
            --non-interactive \
            --output ./dist/app.apk
        env:
          EAS_NO_VCS: 1

      # Firebase App Distribution に配布
      - name: Upload to Firebase App Distribution
        uses: wzieba/Firebase-Distribution-Github-Action@v1
        with:
          appId: ${{ secrets.FIREBASE_APP_ID_ANDROID }}
          token: ${{ secrets.FIREBASE_TOKEN }}
          groups: qa-testers
          file: ./dist/app.apk
          releaseNotes: |
            Branch: ${{ github.ref_name }}
            Commit: ${{ github.sha }}
            ${{ github.event.head_commit.message }}

      # Slack 通知（任意）
      - name: Notify success
        if: success()
        uses: slackapi/slack-github-action@v1.26.0
        with:
          channel-id: ${{ secrets.SLACK_CHANNEL_ID }}
          slack-message: |
            ✅ テストビルド配布完了
            *${{ github.event.head_commit.message }}*
        env:
          SLACK_BOT_TOKEN: ${{ secrets.SLACK_BOT_TOKEN }}

      - name: Notify failure
        if: failure()
        uses: slackapi/slack-github-action@v1.26.0
        with:
          channel-id: ${{ secrets.SLACK_CHANNEL_ID }}
          slack-message: |
            ❌ ビルド失敗
            *${{ github.event.head_commit.message }}*
            詳細: ${{ github.server_url }}/${{ github.repository }}/actions/runs/${{ github.run_id }}
        env:
          SLACK_BOT_TOKEN: ${{ secrets.SLACK_BOT_TOKEN }}
```

GitHub Secrets に設定が必要な値の一覧です。

- `EXPO_TOKEN`：expo.dev のアクセストークン（Account Settings → Access Tokens）
- `FIREBASE_TOKEN`：`firebase login:ci` で取得したトークン
- `FIREBASE_APP_ID_ANDROID`：Firebase コンソール → プロジェクト設定 → アプリ ID
- `GOOGLE_SERVICES_JSON_BASE64`：`base64 google-services.json` の出力値

`eas.json` にも `preview` プロファイルを追加しておきます。

```json
// eas.json
{
  "build": {
    "preview": {
      "distribution": "internal",
      "android": {
        "buildType": "apk",
        "gradleCommand": ":app:assembleRelease"
      }
    },
    "production": {
      "android": {
        "buildType": "app-bundle"
      },
      "ios": {
        "autoIncrement": true
      }
    }
  }
}
```

この設定で、`main` に push するたびに自動でビルドが走り、登録したテスターへ配布通知が届きます。EAS Build のキューに入ってから配布完了まで、私の環境では大体6〜8分程度です。

### iOS の自動配布を追加する場合

iOS ビルドは Android の2〜3倍時間がかかります。すべての push で iOS もビルドすると費用がかさむため、`release` ブランチへの merge か手動実行（`workflow_dispatch`）のみ iOS を走らせる使い分けをお勧めします。

```yaml
# iOS 配布ジョブ（ビルドが成功した後に実行）
build-ios:
  runs-on: macos-latest
  if: github.ref == 'refs/heads/release'
  steps:
    - uses: actions/checkout@v4
    - name: Build iOS IPA
      run: |
        eas build \
          --platform ios \
          --profile preview \
          --non-interactive \
          --output ./dist/app.ipa
    - name: Upload iOS to Firebase
      uses: wzieba/Firebase-Distribution-Github-Action@v1
      with:
        appId: ${{ secrets.FIREBASE_APP_ID_IOS }}
        token: ${{ secrets.FIREBASE_TOKEN }}
        groups: ios-testers
        file: ./dist/app.ipa
```

## Firebase Performance Monitoring でボトルネックを可視化する

クラッシュが出ない場合でも「なんか遅い」という体験はユーザー離脱につながります。Performance Monitoring を組み込むことで、API レスポンスタイムや画面の表示速度を継続的に計測できます。

```typescript
// utils/trackedFetch.ts
import perf from '@react-native-firebase/perf';
import crashlytics from '@react-native-firebase/crashlytics';

/**
 * パフォーマンスを計測しながら非同期処理を実行するラッパー。
 * トレース名は Firebase コンソールに表示されるので、
 * 機能が判別できる名前をつけること。
 *
 * 使い方：
 *   const products = await trackedAsync('fetch_product_list', () =>
 *     supabase.from('products').select('*').limit(20)
 *   );
 */
export async function trackedAsync<T>(
  traceName: string,
  asyncFn: () => Promise<T>
): Promise<T> {
  const trace = await perf().startTrace(traceName);

  try {
    const result = await asyncFn();
    await trace.putAttribute('status', 'success');
    return result;
  } catch (error) {
    await trace.putAttribute('status', 'error');

    // Crashlytics にも転送して一元管理
    const err = error instanceof Error ? error : new Error(String(error));
    crashlytics().recordError(err);
    crashlytics().log(`Error in trace "${traceName}": ${err.message}`);

    throw error;
  } finally {
    // 成功・失敗どちらでも trace を必ず停止する
    await trace.stop();
  }
}
```

注意点として、トレース名は 100 文字以内で、先頭にアンダースコアを使わないことが条件です。また、1アプリあたり 300 種類のカスタムトレースが上限なので、API エンドポイントをすべて個別にトレースするのではなく「機能単位」でまとめることをお勧めします。

Performance Monitoring を数週間運用すると「特定の画面だけ応答が遅い」「アップデート後にレスポンスが遅くなった」といった変化をグラフで追えるようになります。Crashlytics でクラッシュが減った後の「次の品質改善ターゲット」として使える、地味に便利なツールです。

## よくある落とし穴：詰まりやすいポイントと解決策

この構成を実際に設定した際に私が詰まったポイントと、開発者コミュニティでよく見る問題をまとめます。

### 1. Hermes とスタックトレースの可読性問題

React Native 0.70 以降は Hermes がデフォルトになっていますが、Hermes の出力するスタックトレースは minified 形式です。Firebase コンソールに届いたスタックトレースが読めない場合、ソースマップのアップロードが必要です。EAS Build を使っている場合は Crashlytics の Gradle プラグインが自動処理してくれますが、`app.json` に `jsEngine` を明示していないと機能しないことがあります。

```json
// app.json
{
  "expo": {
    "jsEngine": "hermes"
  }
}
```

### 2. iOS の GoogleService-Info.plist が EAS Build に反映されない

`app.json` で `googleServicesFile` を指定しても、EAS Build のキャッシュが古い場合に無視されることがあります。`--clear-cache` フラグをつけてビルドすると解決します。

```bash
eas build --platform ios --profile preview --clear-cache
```

### 3. Firebase Token の突然の失効

`firebase login:ci` で取得したトークンは通常永続的ですが、Google アカウントのセキュリティポリシー変更で無効になることがあります。GitHub Actions が突然失敗し始めた場合、まずこれを疑ってください。Firebase CLI を再ログインして新しいトークンを取得し、GitHub Secrets を更新します。

### 4. App Distribution のテスターが招待メールを見落とす

Firebase App Distribution の配布通知はメール経由です。テスターによっては迷惑メールフォルダに入ることがあります。初回配布時に「Firebase からのメールを確認してアプリを登録してください」と一言伝えておくことで、「インストールできない」という問い合わせをかなり減らせます。

### 5. Android リリースビルドとデバッグビルドの Crashlytics 設定差異

デバッグビルドの Crashlytics はデフォルト無効です（`setCrashlyticsCollectionEnabled(false)`）。EAS Build の `preview` プロファイルで生成したリリースビルドでは有効になっていますが、ローカルでデバッグビルドを走らせてテストしても Crashlytics にデータが来ないため「設定が壊れている？」と勘違いしやすいです。

```typescript
// 開発環境では無効化するパターン（明示的に書くことで意図が明確になる）
crashlytics().setCrashlyticsCollectionEnabled(\!__DEV__);
```

## 個人開発者のためのコスト設計：Firebase 無料枠の現実

Firebase の料金体系について、実際の使用感から補足します。

Spark（無料）プランでQAに使うサービスの上限は次の通りです。

- **Crashlytics**：完全無料（上限なし）
- **App Distribution**：月 2GB のストレージ（APK 1本が 50MB 前後なら約 40 回分）
- **Performance Monitoring**：完全無料
- **Hosting**（ダッシュボードなどを置く場合）：月 10GB のデータ転送

個人開発で月間アクティブユーザーが数千人規模であれば、この QA パイプラインを Spark プランで完全に運用できます。課金が必要になるのは主に Firestore や Cloud Functions を使う場合で、テスト配布とクラッシュ監視の目的に限れば無料枠を超えることはほぼありません。

EAS Build の費用については別途かかります（個人プランは月 30 ビルドまで無料）。ビルド回数が増えてきたら、`main` push のたびにビルドするのをやめて、Pull Request マージ時のみにトリガーを絞ると消費を抑えられます。

## クラッシュフリーレートの目標値と改善サイクル

Crashlytics の「クラッシュフリーセッション率」は、全セッションのうちクラッシュが起きなかった割合です。App Store / Google Play の品質評価に影響するため、この数字を定期的に確認する習慣が大切です。

実際の目安としては、99.5% 以上をキープできていれば良好です。これは App Store のフィーチャー獲得を狙える水準でもあります。99.0% を下回ると、ユーザーからのネガティブレビューが増え始めます。

改善サイクルの現実的な回し方は、毎週月曜日に Crashlytics コンソールを確認し、先週の上位3件のクラッシュに絞って対処することです。全件を一気に直そうとすると疲弊します。「今週は1件だけ完全に潰す」くらいのペースが、長く続けるコツです。

Crashlytics の「Velocity Alerts」機能を有効にしておくと、急激にクラッシュが増えた際にメールで通知が来ます。アップデート直後のリグレッションを早期に発見できるので、必ず設定しておくことをお勧めします。設定は Firebase コンソール → Crashlytics → 右上のベルアイコンから行えます。

## まとめ：今日から始めるQA自動化の最初の一歩

Firebase App Distribution + Crashlytics + GitHub Actions の構成は、一度組んでしまえばほぼメンテナンスフリーで回り続けます。

今日すぐ始めるなら、まず Firebase プロジェクトを作って **Crashlytics だけ組み込む** ところから始めてください。App Distribution の自動化は後からでも追加できますが、Crashlytics は今すぐ入れておくことで「昨日のアップデートから急にクラッシュが増えた」という事実をデータで確認できるようになります。

品質を守れるかどうかは、ツールより習慣の問題だと思っています。週に一度 Crashlytics を確認するだけで、Rork で生成したアプリは確実に磨かれていきます。まずは Crashlytics の初期設定だけでも、今日中に終わらせてみてください。
