## RorkアプリがGoogle Playで審査リジェクトされる主な症状

Rorkで生成したAndroidアプリをGoogle Play Consoleに申請したとき、次のようなエラーメッセージを受け取ることがあります。

- 「お客様のアプリはポリシーに違反しています」というメール通知
- Google Play Consoleのダッシュボードに赤いアラートが表示される
- アプリのステータスが「リジェクト」や「申請却下」に変わる
- 「メタデータポリシー違反」「広告ポリシー違反」「権限ポリシー違反」といった具体的な違反内容が記載されている

これらは初めて経験すると非常に焦りますが、Rorkで生成されたReact Native（Expo）アプリに特有のパターンが多く、対処方法を知っておけば落ち着いて修正できます。本記事では、よくある違反パターンとその解決手順を詳しく解説します。

---

## 原因の分析：Google Playポリシー違反の主なパターン

### パターン1：メタデータの不備・誤解を招く内容

Google Playでもっとも多いリジェクト理由のひとつが、アプリのメタデータ（アプリ名、説明文、スクリーンショット）に関するものです。

よくある違反例は以下のとおりです。

- **アプリ名や説明に「No.1」「ベスト」「最高」などの根拠のない優位性表現を使っている**
- **スクリーンショットがアプリの実際の動作を反映していない（デザインモックが混入している）**
- **説明文に他アプリのブランド名やキーワードを不適切に含めている**
- **アプリのアイコン・スクリーンショットに文字が過剰に含まれている**

### パターン2：権限の過剰要求

RorkアプリはReact Native（Expo）をベースにしているため、使用していない機能の権限が`AndroidManifest.xml`に含まれることがあります。

```xml
<!-- 問題のある例：位置情報を使わないのに権限が含まれている -->
<uses-permission android:name="android.permission.ACCESS_FINE_LOCATION" />
<uses-permission android:name="android.permission.READ_CONTACTS" />
```

Googleはアプリが実際に使用しない権限を宣言することを禁止しています。特に「危険な権限（Dangerous Permissions）」が含まれると高い確率でリジェクトされます。

### パターン3：広告の実装方法に関する違反

AdMobなどの広告を実装している場合、以下のような問題でリジェクトされることがあります。

- **インタースティシャル広告が頻繁すぎる（ボタンタップのたびに全画面広告が出るなど）**
- **広告の閉じるボタンが小さすぎて押しづらい、または見えにくい**
- **誤クリックを誘発するような広告配置（操作ボタンの直下など）**
- **広告をコンテンツと区別しにくいデザイン**

### パターン4：ターゲット年齢層の不一致

Google Play Familyポリシーに違反するケースです。アプリが子ども向けコンテンツを含むのに「全年齢」以外の対象年齢を設定していたり、逆に子ども向けに設定しているのに成人向けの広告SDK（AdMob）を使用している場合に違反となります。

### パターン5：データセーフティセクションの不備

2022年以降、Google Playではデータセーフティセクションの正確な申告が必須です。Rorkアプリでよくある問題が以下のものです。

- **SupabaseやFirebaseの認証で収集されるユーザーデータを申告していない**
- **プッシュ通知に使用するデバイストークンの取り扱いを申告していない**
- **サードパーティSDKが収集するデータを把握していない**

## 解決手順：Step by Stepの修正方法

### Step 1：Google Play Consoleでリジェクト理由を正確に確認する

```
Google Play Console → [アプリ名] → ポリシーのステータス → 問題を表示
```

「ポリシーの問題」のセクションを開き、**具体的な違反条項のリンク**をクリックしてGoogleのポリシードキュメントを読みます。曖昧な表現でも、リンク先のポリシーページに詳細が記載されています。

### Step 2：メタデータ違反の修正

アプリ名・説明文を修正する場合、Google Play Console上で直接編集できます。

修正のポイントは次のとおりです。

- **根拠のない優位性表現を削除する**：「世界一」「No.1」→「シンプルで使いやすい」に変える
- **説明文はアプリの実際の機能を具体的に書く**：「すごいアプリ」ではなく「タスクを3ステップで完了できるToDoアプリ」のように
- **スクリーンショットは必ず実機で撮影したものを使う**：Rorkのプレビュー画面ではなく、実際のAndroidデバイスやエミュレーターでの画面を使用する

### Step 3：権限の過剰要求を修正する

Rorkで生成されたアプリの`AndroidManifest.xml`を確認するには、Expo/EASのビルド設定で調整します。

```json
// app.json または app.config.js
{
  "expo": {
    "android": {
      "permissions": [
        "INTERNET",
        "NOTIFICATIONS"
        // 使用しない権限はここに列挙しない
      ]
    }
  }
}
```

不要な権限を削除した後、EAS Build（またはRorkのエクスポート機能）でAPK/AABを再ビルドします。

```bash
# EAS Buildで再ビルド（Rork MaxのExpoプロジェクトの場合）
eas build --platform android --profile production
```

### Step 4：広告実装の修正

AdMobインタースティシャル広告の表示頻度を制限します。

```javascript
// 広告表示に最低60秒のインターバルを設ける
let lastAdShownTime = 0;
const AD_INTERVAL_MS = 60000; // 60秒

function showInterstitialAd() {
  const now = Date.now();
  if (now - lastAdShownTime < AD_INTERVAL_MS) {
    console.log('広告インターバル中のためスキップ');
    return;
  }
  // 広告表示処理
  interstitial.show();
  lastAdShownTime = now;
}
```

広告配置のチェックポイントは以下のとおりです。

- **閉じるボタンは48dp以上**（Googleの最小タップターゲット推奨値）
- **操作ボタンから48dp以上の距離を確保**
- **「広告」「AD」のラベルを表示する**（AdMobは自動で表示するが、確認が必要）

### Step 5：データセーフティセクションの修正

Google Play Console → データセーフティ → 編集

修正時の確認事項は次のとおりです。

- Supabase/Firebaseが収集するデータ（メールアドレス、ユーザーID）を申告する
- プッシュ通知に使うデバイストークンを「デバイスまたはその他の識別子」として申告する
- データを第三者（AdMobなど）と共有している場合は「データの共有」を「はい」にする

## 確認方法：修正できたかの検証手順

修正後、Google Playに再申請する前に以下のチェックを行いましょう。

**Google Play Policyガイドラインとの照合**

[Google Playデベロッパープログラムポリシー](https://play.google.com/about/developer-content-policy/)を開き、リジェクト理由となったポリシーページと自分のアプリを照合します。

**Pre-launch Reportの確認**

Google Play ConsoleのPre-launch Reportは、実機でアプリを自動テストした結果を提供します。

```
Google Play Console → テスト → Pre-launch Report
```

クラッシュや権限エラーが検出された場合は修正してから再申請します。

**内部テストトラックで先行確認**

本番申請の前に「内部テスト」トラックに提出すると、ポリシー違反の事前チェックが行われ、問題を早期発見できます。

## 予防策：再発防止のベストプラクティス

### 公式ポリシーを事前に読む習慣をつける

Google Playのポリシーは定期的に改訂されます。特に以下のページをブックマークして定期確認することをお勧めします。

- [Google Playデベロッパープログラムポリシー](https://play.google.com/about/developer-content-policy/)
- [Google Playポリシー ヘルプセンター](https://support.google.com/googleplay/android-developer/)

### リリース前のチェックリストを作る

Rorkアプリのリリース前に毎回確認するチェックリストを用意しましょう。最低限チェックすべき項目は以下のとおりです。

- [ ] メタデータに誇大表現がない
- [ ] 使用していない権限がない
- [ ] データセーフティセクションが最新の状態
- [ ] 広告の閉じるボタンが十分な大きさ
- [ ] ターゲット年齢と実際のコンテンツが一致している

### 内部テストを積極活用する

本番申請直前ではなく、開発初期から内部テストトラックを使うことで、リジェクトリスクを大幅に減らせます。Rorkで出力したAABファイルを内部テストに登録するのに数分しかかかりません。

## FAQ

**Q. ポリシー違反の警告メールが届いたが、アプリはまだ公開されています。すぐに対応が必要ですか？**

はい、できるだけ早く対応することをお勧めします。Googleは警告後に一定期間の猶予を設けることが多いですが、期限内に修正しないとアプリが削除される場合があります。メールに記載された期限日を必ず確認してください。

**Q. 同じ違反で2回リジェクトされましました。どうすれば審査を通過できますか？**

まず、Google Play ConsoleのPre-launch Reportで問題が解消されていることを確認してください。それでも通過しない場合は、Google Playのサポートに「ポリシー違反の異議申し立て（Appeal）」を行い、どのように修正したかを具体的に説明することが有効です。

**Q. RorkアプリはExpoベースですが、AndroidManifest.xmlを直接編集できますか？**

Rork Maxを使っている場合、Expoのプラグインシステムを通じて`app.json`の`android.permissions`フィールドを調整できます。直接`AndroidManifest.xml`を編集したい場合はEASのカスタムビルド設定が必要になりますが、多くのケースは`app.json`の変更で対応可能です。

**Q. データセーフティセクションを「後で入力する」にしていたら警告を受けましました。どうすればよいですか？**

データセーフティセクションは申請時に必ず完了している必要があります。Google Play ConsoleでアプリのデータセーフティセクションをWEBから編集し、「承認済み」状態にしてから再提出してください。

## まとめ

RorkアプリのGoogle Playポリシー違反で多いのは、メタデータの誤解を招く表現・不要な権限の宣言・データセーフティセクションの未申告の3つです。それぞれに明確な修正方法があり、Googleのポリシードキュメントを参照しながら丁寧に対応すれば必ず解消できます。

特に大切なのは、**公開前に内部テストトラックを活用してポリシー問題を早期発見すること**と、**定期的にGoogleのポリシー改訂をチェックする習慣**です。

Androidアプリのリリース運用をさらに体系的に学びたい方には、[Androidアプリ開発の教科書 第3版（田渕義人 著）](https://www.amazon.co.jp/dp/4798171808/?tag=pinocchio-22)がおすすめです。Google Playの申請フローからポリシー対応まで、実践的な内容が網羅されています。

アプリのGoogle Play公開手順そのものについては、[RorkアプリをGoogle Playに公開する完全ガイド](/articles/rork-dev/google-play-publishing)もあわせてご覧ください。
