## はじめに：EASとは何か？

Expo Application Services（EAS）は、Rork Max（React Native / Expo ベース）で構築したモバイルアプリケーションをクラウドで自動ビルド・デプロイするためのマネージドサービスです。従来、iOS アプリのビルドには macOS と Xcode、Android アプリのビルドには Android Studio が必須でしたが、EAS を使うことでこれらの環境構築は不要。Linux サーバーや Windows マシンからも直接 iOS/Android アプリケーションをビルドできます。

### Rork Max 開発者にとって EAS が重要な理由

1. **ローカル環境の複雑性を排除**: Xcode や Android Studio をインストール・管理する手間が削減
2. **高速なリリースサイクル**: CI/CD パイプラインにより、コミット → テスト → ビルド → デプロイが自動化
3. **OTA（Over-The-Air）アップデート**: ストア審査なしで JavaScript バンドルを即座に配信可能
4. **チーム開発の加速**: プレビュービルドを QR コードで共有し、iOS/Android デバイスで即座にテスト
5. **本番環境の堅牢性**: 失敗時の自動ロールバック、複数プロファイルによる環境分離

このガイドでは、Rork Max アプリを EAS でフル CI/CD パイプライン化し、本番環境にまで持ち込む実装方法を、コード例を交えて解説します。

## 第1章：EAS セットアップと初期設定

### ステップ 1：EAS CLI のインストール

```bash
npm install -g eas-cli
```

バージョン確認：

```bash
eas --version
```

### ステップ 2：Expo アカウントへのログイン

```bash
eas login
```

プロンプトに従ってブラウザで Expo アカウントにログインします。既存アカウントがない場合は、[expo.dev](https://expo.dev) から無料会員登録してください。

<Callout type="info">
**ヒント**: Rork Max プロジェクトが複数ある場合、各プロジェクトで同じアカウントを使用するか、あるいは環境変数 `EXPO_TOKEN` に API トークンを設定して、CI/CD 環境での自動認証を可能にします。
</Callout>

### ステップ 3：EAS ビルド設定の初期化

```bash
eas build:configure
```

このコマンドを実行すると、プロジェクトルートに `eas.json` ファイルが作成されます。以下は典型的な設定例です。

### esa.json の詳細設定例

```json
{
  "cli": {
    "version": ">= 5.0.0"
  },
  "build": {
    "development": {
      "distribution": "internal",
      "ios": {
        "buildType": "development"
      },
      "android": {
        "buildType": "development"
      }
    },
    "preview": {
      "distribution": "internal",
      "ios": {
        "buildType": "ad-hoc",
        "provisioning": {
          "teamId": "YOUR_APPLE_TEAM_ID"
        }
      },
      "android": {
        "buildType": "apk"
      }
    },
    "production": {
      "distribution": "store",
      "ios": {
        "buildType": "app-store",
        "provisioning": {
          "teamId": "YOUR_APPLE_TEAM_ID"
        }
      },
      "android": {
        "buildType": "aab"
      }
    }
  },
  "submit": {
    "production": {
      "ios": {
        "appleId": "your-email@example.com",
        "ascAppId": "1234567890",
        "appleTeamId": "YOUR_APPLE_TEAM_ID"
      },
      "android": {
        "serviceAccount": "path/to/service-account.json",
        "track": "production"
      }
    }
  }
}
```

**各プロファイルの説明**：

- **development**: ローカル開発用。内部配布のみ
- **preview**: QA チーム向けテストビルド。ストア申請なしで TestFlight（iOS）または Google Play Internal Testing（Android）経由で配布可能
- **production**: App Store / Google Play へ本番リリースするビルド。AAB（Android App Bundle）と App Store 用 iOS ビルドを生成

## 第2章：EAS Build — iOS ビルドの実装

### iOS ビルドの準備

iOS アプリを App Store にリリースするには、Apple Developer Account が必須です。アカウント取得後、以下の手順で進めます。

#### 1. ビルド認証情報の設定

```bash
eas credentials
```

このコマンドでインタラクティブに以下を設定できます：
- Apple ID とパスワード（またはアプリパスワード）
- 開発者証明書（Certificate）
- プロビジョニングプロファイル

<Callout type="info">
**セキュリティ上の注意**: Apple パスワードは絶対に GitHub や他の公開場所に保存しないこと。EAS が暗号化して安全に管理します。
</Callout>

#### 2. 初めての iOS ビルド実行

```bash
eas build --platform ios --profile production
```

初回実行時、EAS はビルド ID をコンソールに表示し、ビルドキューに登録します。進捗を確認するには：

```bash
eas build --status
```

ビルド完了後、TestFlight 用の .ipa ファイルが生成されます。

#### 3. TestFlight への配布

```bash
eas submit --platform ios --profile production
```

このコマンドで自動的に App Store Connect にアップロードされ、TestFlight で内部テスターが利用可能になります。

### iOS ビルドのベストプラクティス

```json
{
  "build": {
    "production": {
      "ios": {
        "buildType": "app-store",
        "provisioning": {
          "teamId": "YOUR_APPLE_TEAM_ID",
          "signingCertificate": "YOUR_CERT_ID"
        },
        "entitlements": {
          "com.apple.developer.pass-type-identifiers": [
            "pass.com.example.rorkmax"
          ]
        }
      }
    }
  }
}
```

## 第3章：EAS Build — Android ビルドの実装

### Android ビルド認証情報の設定

```bash
eas credentials
```

Android の場合、キーストア（Keystore）情報が必要です。初回実行時に自動生成することも、既存のキーストアをアップロードすることもできます。

#### キーストアの手動作成（既に持っている場合）

```bash
keytool -genkey -v -keystore my-release-key.keystore \
  -keyalg RSA -keysize 2048 -validity 10000 \
  -alias my-key-alias
```

生成されたキーストアを EAS に登録します。

### Android APK / AAB ビルド

```bash
# APK ビルド（プレビュー用）
eas build --platform android --profile preview

# AAB ビルド（Google Play 本番用）
eas build --platform android --profile production
```

### Android ビルド設定例

```json
{
  "build": {
    "production": {
      "android": {
        "buildType": "aab",
        "gradleCommand": ":app:bundleRelease",
        "withoutCredentials": false,
        "env": {
          "TARGET_SDK_VERSION": "34",
          "MIN_SDK_VERSION": "24"
        }
      }
    },
    "preview": {
      "android": {
        "buildType": "apk",
        "gradleCommand": ":app:assembleRelease"
      }
    }
  }
}
```

### Google Play への配布

```bash
eas submit --platform android --profile production
```

Google Play Console の認証情報（サービスアカウント JSON ファイル）が必要です。以下の手順で準備します。

1. [Google Cloud Console](https://console.cloud.google.com) で新規プロジェクトを作成
2. Google Play Android Developer API を有効化
3. サービスアカウント キーを JSON 形式でダウンロード
4. esa.json の `submit.production.android` に `serviceAccount` パスを記載

```json
{
  "submit": {
    "production": {
      "android": {
        "serviceAccount": "./path/to/service-account.json",
        "track": "production"
      }
    }
  }
}
```

## 第4章：EAS Update — OTA（Over-The-Air）アップデート

### OTA アップデートの可能範囲と制限

EAS Update は、ネイティブコード（Objective-C、Kotlin、Java）の変更を除く、JavaScript バンドルと静的アセット（画像、フォント）の更新を可能にします。

**更新可能**:
- React Native / Rork Max のビジネスロジック
- UI コンポーネント
- API 呼び出しロジック
- CSS スタイル（StyleSheet）

**更新不可**:
- ネイティブモジュール追加
- パーミッション変更
- Expo SDK のメジャーバージョンアップ

### OTA アップデートの実行

```bash
eas update --branch production --message "Critical bug fix: crash on user login"
```

このコマンドは以下の処理を実行します：
1. 最新の JavaScript バンドルをビルド
2. EAS サーバーにアップロード
3. `production` ブランチの全デバイスに即座にロール会員ード

### リリースノート付きアップデート

```bash
eas update --branch production \
  --message "v1.2.1: Fixed dashboard load time" \
  --metadata "{\"environment\": \"production\", \"rollback_available\": true}"
```

### 複数ブランチ戦略

```bash
# ステージング環境へのロールアウト
eas update --branch staging --message "Beta: new payment flow"

# 段階的ロールアウト（A/B テスト）
eas update --branch production \
  --message "Variant A: new onboarding" \
  --percent 25
```

`--percent` オプションで、ユーザーの何パーセントにアップデートを提供するかを制御できます。段階的ロールアウトにより、問題が見つかった場合の影響を最小化できます。

### ロールバック戦略

```bash
# 最新リリースを表示
eas update --list

# 特定のリリースにロールバック
eas update --branch production --rollback-to-release-id=abc123def456
```

<Callout type="info">
**本番運用のコツ**: OTA アップデートをリリース前に、必ずステージングブランチで検証してください。不具合が見つかった場合は、`--rollback-to-release-id` で即座に前のバージョンに戻せます。
</Callout>

## 第5章：GitHub Actions による CI/CD パイプライン

### ワークフロー YAML の実装

プロジェクトルートに `.github/workflows/eas-ci.yml` を作成します。

```yaml
name: Rork Max EAS CI/CD

on:
  push:
    branches:
      - main
      - develop
  pull_request:
    branches:
      - main

jobs:
  test:
    name: Run Tests
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v3

      - name: Setup Node.js
        uses: actions/setup-node@v3
        with:
          node-version: "18.x"
          cache: "npm"

      - name: Install dependencies
        run: npm ci

      - name: Run linter
        run: npm run lint

      - name: Run tests
        run: npm run test -- --coverage

  build-preview:
    name: Build Preview (iOS & Android)
    runs-on: ubuntu-latest
    needs: test
    if: github.event_name == 'push' && github.ref == 'refs/heads/develop'
    steps:
      - name: Checkout code
        uses: actions/checkout@v3

      - name: Setup Node.js
        uses: actions/setup-node@v3
        with:
          node-version: "18.x"
          cache: "npm"

      - name: Setup EAS CLI
        run: npm install -g eas-cli

      - name: Install dependencies
        run: npm ci

      - name: Build iOS Preview
        run: eas build --platform ios --profile preview --non-interactive
        env:
          EXPO_TOKEN: ${{ secrets.EXPO_TOKEN }}

      - name: Build Android Preview
        run: eas build --platform android --profile preview --non-interactive
        env:
          EXPO_TOKEN: ${{ secrets.EXPO_TOKEN }}

      - name: Post build status
        if: always()
        uses: actions/github-script@v6
        with:
          script: |
            github.rest.issues.createComment({
              issue_number: context.issue.number,
              owner: context.repo.owner,
              repo: context.repo.repo,
              body: `✅ Preview builds completed! Check EAS dashboard for download links.`
            })

  build-production:
    name: Build Production & Deploy OTA
    runs-on: ubuntu-latest
    needs: test
    if: github.event_name == 'push' && github.ref == 'refs/heads/main'
    steps:
      - name: Checkout code
        uses: actions/checkout@v3

      - name: Setup Node.js
        uses: actions/setup-node@v3
        with:
          node-version: "18.x"
          cache: "npm"

      - name: Setup EAS CLI
        run: npm install -g eas-cli

      - name: Install dependencies
        run: npm ci

      - name: Build iOS Production
        run: eas build --platform ios --profile production --non-interactive
        env:
          EXPO_TOKEN: ${{ secrets.EXPO_TOKEN }}

      - name: Build Android Production
        run: eas build --platform android --profile production --non-interactive
        env:
          EXPO_TOKEN: ${{ secrets.EXPO_TOKEN }}

      - name: Deploy OTA Update
        run: |
          eas update \
            --branch production \
            --message "Release: $(git describe --tags --abbrev=0 2>/dev/null || echo 'v0.0.1')" \
            --non-interactive
        env:
          EXPO_TOKEN: ${{ secrets.EXPO_TOKEN }}

      - name: Notify Slack
        if: always()
        uses: slackapi/slack-github-action@v1.24.0
        with:
          webhook-url: ${{ secrets.SLACK_WEBHOOK }}
          payload: |
            {
              "text": "🚀 Rork Max Production Deployment",
              "blocks": [
                {
                  "type": "section",
                  "text": {
                    "type": "mrkdwn",
                    "text": "*Status*: ${{ job.status }}\n*Branch*: main\n*Commit*: <${{ github.server_url }}/${{ github.repository }}/commit/${{ github.sha }}|${{ github.sha }}>"
                  }
                }
              ]
            }
```

### GitHub Secrets の設定

リポジトリの Settings > Secrets and variables > Actions で以下を登録します：

- `EXPO_TOKEN`: Expo アカウントの API トークン（[expo.dev/settings/access-tokens](https://expo.dev/settings/access-tokens) で生成）
- `SLACK_WEBHOOK`: Slack 通知用（オプション）

<Callout type="info">
**CI/CD 本番運用**: `--non-interactive` フラグを使用すると、ユーザー入力を待たずにコマンドが実行されます。GitHub Actions 環境での必須オプションです。
</Callout>

## 第6章：環境変数とシークレット管理

### EAS Secret の作成

```bash
eas secret:create
```

プロンプトに従って以下の情報を入力します：
- シークレット名（例：`API_KEY_PRODUCTION`）
- シークレット値

作成されたシークレットは EAS クラウドで暗号化されます。

### app.json での環境変数定義

```json
{
  "expo": {
    "name": "RorkMax App",
    "slug": "rorkmax-app",
    "version": "1.0.0",
    "plugins": [
      [
        "expo-build-properties",
        {
          "ios": {
            "deploymentTarget": "14.0"
          },
          "android": {
            "targetSdkVersion": 34
          }
        }
      ]
    ]
  }
}
```

### React Native コード内での環境変数使用

```javascript
import Constants from 'expo-constants';

export const config = {
  apiBaseUrl: Constants.manifest?.extra?.API_BASE_URL || 'https://api.example.com',
  environment: Constants.manifest?.extra?.ENVIRONMENT || 'development',
  analyticsToken: Constants.manifest?.extra?.ANALYTICS_TOKEN || '',
};

// API 呼び出し
const response = await fetch(`${config.apiBaseUrl}/users`, {
  headers: {
    'Authorization': `Bearer ${config.analyticsToken}`,
  },
});
```

### プロファイル別環境変数の設定

esa.json で各プロファイルに異なる環境変数を指定：

```json
{
  "build": {
    "development": {
      "env": {
        "API_BASE_URL": "http://localhost:3000",
        "ENVIRONMENT": "development",
        "DEBUG": "true"
      }
    },
    "preview": {
      "env": {
        "API_BASE_URL": "https://staging-api.example.com",
        "ENVIRONMENT": "staging"
      }
    },
    "production": {
      "env": {
        "API_BASE_URL": "https://api.example.com",
        "ENVIRONMENT": "production"
      }
    }
  }
}
```

GitHub Actions 環境で環境変数を利用する場合：

```yaml
- name: Deploy to Production
  env:
    API_BASE_URL: ${{ secrets.PRODUCTION_API_URL }}
    ANALYTICS_TOKEN: ${{ secrets.ANALYTICS_TOKEN }}
  run: eas update --branch production --non-interactive
```

## 第7章：プレビュービルドの QA テスト活用

### プレビュービルドの作成と配布

```bash
eas build --platform ios --profile preview --non-interactive
```

プレビュービルド完成後、EAS が QR コードを生成します。iOS デバイスから Expo Go アプリで QR コードをスキャンすると即座にテスト可能。

### 複数チームメンバーへの配布

```bash
eas build --platform ios --profile preview --non-interactive \
  --message "QA Review: Payment flow v2" \
  --auto-submit
```

`--auto-submit` を指定すると、ビルド完了時に自動的に TestFlight（iOS）に登録され、登録済みテスターに通知されます。

### プレビュー環境での環境変数

```json
{
  "build": {
    "preview": {
      "env": {
        "API_BASE_URL": "https://staging-api.example.com",
        "ENABLE_DEV_MENU": "true",
        "LOG_LEVEL": "debug"
      }
    }
  }
}
```

<Callout type="info">
**QA チーム向けのヒント**: プレビュービルドには本番環境とは異なるサーバー（ステージング）を指すよう設定し、デバッグメニューを有効化します。これにより、デバイスで直接ネットワークリクエストやロジックを検証できます。
</Callout>

## 第8章：EAS 料金プラン・制限事項

### EAS Free プラン

- **無料ビルド時間**: 30 分/月
- **ストレージ**: 1 GB
- **同時ビルド**: 1 個
- **更新配信**: 無制限

### EAS Pro プラン（$99/月）

- **ビルド時間**: 500 分/月
- **ストレージ**: 100 GB
- **同時ビルド**: 2 個
- **優先サポート**: あり

本番環境で週次以上のリリースを行う場合は、Pro プランの導入が現実的です。

### ビルド時間の削減のコツ

```bash
# キャッシュを活用
eas build --platform ios --profile production

# 前回のビルドログを確認
eas build --status --detailed
```

<Callout type="info">
**コスト最適化**: 開発段階ではプレビュービルドを多用し、本番リリース時のみ本格ビルドを行うことで、月々のビルド時間を圧縮できます。
</Callout>

## 第9章：本番運用チェックリスト

Rork Max アプリを EAS で本番環境にリリースする際の確認事項：

### ビルド前の確認

- [ ] `eas.json` で production プロファイルが正しく設定されているか
- [ ] Apple Developer Team ID および Google Play サービスアカウント JSON が登録済みか
- [ ] 全環境変数（API_KEY、ANALYTICS_TOKEN など）が EAS Secret に登録済みか
- [ ] iOS の Provisioning Profile と Certificate が有効期限内か

### テスト段階

- [ ] プレビュービルドを iOS/Android で実行し、動作確認
- [ ] ステージング環境の API に接続してエンドツーエンドテスト
- [ ] ネットワーク遅延・オフライン状態での挙動をテスト
- [ ] 複数バージョンの iOS/Android デバイスで検証

### リリース実行

- [ ] Git リポジトリに最新コードが反映されているか確認
- [ ] `npm run lint` / `npm run test` で全テストをパス
- [ ] バージョン番号を `package.json` / `app.json` で更新
- [ ] リリースノートを準備

### リリース後の監視

- [ ] Crashlytics でクラッシュレートを監視
- [ ] Analytics でセッション数・アクティブユーザー数を追跡
- [ ] ユーザーレビュー・サポート問い合わせを監視
- [ ] 問題が見つかった場合は速やかに OTA ロールバック

## まとめ

Rook Max アプリの EAS CI/CD パイプライン化により、以下が実現できます：

1. **開発環境の簡素化**: ローカル Xcode/Android Studio 不要
2. **リリースサイクルの加速**: コミット → テスト → ビルド → デプロイが自動化
3. **OTA 更新による迅速なバグ修正**: ストア審査を待たずに即座に配信
4. **チーム開発の効率化**: プレビュービルドで QA テストを並列化
5. **本番環境の信頼性**: 失敗時の自動ロールバック、複数環境の分離管理

EAS の導入によって、Rork Max エンジニアチームは本来の開発に集中でき、インフラの煩雑さから解放されます。段階的なロールアウトと監視体制を整備することで、信頼性の高い本番運用を実現してください。
