Rork でアプリを作り、リリース後しばらく使っていると「だんだん動作が重くなる」「バックグラウンドから復帰するとクラッシュする」という報告を受けることがあります。こうした症状の多くは、メモリリークが原因です。

AI が生成するコードは全体的な動作には問題がないことが多いのですが、イベントリスナーの登録解除や非同期処理のキャンセルまで完璧に面倒を見てくれるわけではありません。「動くコード」と「メモリを適切に解放するコード」は別物です。この記事では、Rork が出力しやすいメモリリークのパターンを整理し、実際に診断・修正するところまで手を動かして確認してみましょう。

## なぜ Rork のコードでメモリリークが起きやすいのか

Rork は React Native + Expo の構成でコードを生成します。React Native 自体は適切に書けばメモリは管理できますが、AI が生成するコードには以下のような傾向があります。

- `useEffect` でイベントリスナーを登録するがクリーンアップ関数を返さない
- `setInterval` や `setTimeout` を使うがコンポーネントアンマウント時にクリアしない
- 非同期処理の結果をアンマウント後もステートに書き込もうとする
- Supabase や Firebase のリアルタイム購読を解除しない

これらはコードとして「動く」のですが、長時間使い続けると確実にメモリが積み上がっていきます。

## Flipper でメモリを可視化する

最初のステップは「実際にリークしているのか」を確認することです。感覚だけで修正してもキリがありません。

**Flipper の準備（Mac が必要）**

```bash
# Flipper は Expo 開発者向けの公式デバッグツール
# https://fbflipper.com/ からダウンロードしてインストール

# EAS ビルドではなく、開発用ビルドで接続します
npx expo start --dev-client
```

Flipper を起動して端末またはシミュレーターと接続したら、左メニューの「React DevTools」→「Profiler」タブを開きます。アプリを数分操作し、画面遷移を繰り返してから Profiler の記録を停止します。

**確認すべき指標**

Flipper の「Memory」タブで以下を確認してください。

- **Heap Size**（ヒープサイズ）: 操作を続けるほど右肩上がりなら要注意
- **Retained Size**（保持サイズ）: ある特定のコンポーネントが突出して大きければそこが犯人

数値の見方に迷ったら「同じ画面を 10 回開いて閉じた後のヒープサイズ」を最初の値と比べてみてください。リークがなければほぼ同じ値に戻るはずです。戻らなければリークしています。

## よくある 4 パターンと修正コード

### パターン 1：useEffect のクリーンアップ忘れ

Rork が生成するコードで最も多いのがこれです。

```typescript
// ❌ リークするコード（クリーンアップなし）
useEffect(() => {
  const subscription = someEventEmitter.addListener('update', handleUpdate);
  // コンポーネントがアンマウントされても subscription が残り続ける
}, []);

// ✅ 修正版（クリーンアップ関数を返す）
useEffect(() => {
  const subscription = someEventEmitter.addListener('update', handleUpdate);
  return () => {
    subscription.remove(); // アンマウント時に必ず解除
  };
}, []);
```

Rork に修正を依頼する場合は「useEffect 内のイベントリスナーにクリーンアップ関数を追加してください」とプロンプトで指示すると、ほとんどのケースで正しく修正してくれます。

### パターン 2：タイマーのクリア忘れ

定期更新や遅延処理を実装するときによく現れるパターンです。

```typescript
// ❌ リークするコード
useEffect(() => {
  const timer = setInterval(() => {
    fetchLatestData(); // 画面を離れた後も延々と実行される
  }, 5000);
  // タイマーをクリアしていない
}, []);

// ✅ 修正版
useEffect(() => {
  const timer = setInterval(() => {
    fetchLatestData();
  }, 5000);
  return () => clearInterval(timer); // これだけで解決する
}, []);
```

`setTimeout` も同様です。`clearTimeout(timer)` をクリーンアップ関数に入れてください。

### パターン 3：アンマウント後のステート更新

API コールや非同期処理が完了する前にユーザーが画面を離れた場合、コールバックがアンマウント済みのコンポーネントにステートを書き込もうとして React の警告が出ることがあります。（React 18 以降は警告が出なくなりましたが、リソースが無駄に消費されます）

```typescript
// ✅ isMounted フラグで制御する方法
useEffect(() => {
  let isMounted = true;

  const fetchData = async () => {
    const result = await api.getItems();
    if (isMounted) { // マウント中のときだけ更新
      setItems(result);
    }
  };

  fetchData();
  return () => {
    isMounted = false; // アンマウント時にフラグを切る
  };
}, []);
```

AbortController を使う方法もありますが、isMounted フラグのほうがコードがシンプルで読みやすく、Rork のコードに追記しやすいです。

### パターン 4：Supabase リアルタイム購読の解除忘れ

Supabase を使ったリアルタイムチャットや通知機能でよく起きます。

```typescript
// ❌ リークするコード
useEffect(() => {
  const channel = supabase
    .channel('messages')
    .on('postgres_changes', { event: '*', schema: 'public', table: 'messages' }, handleChange)
    .subscribe();
  // channel.unsubscribe() がない
}, []);

// ✅ 修正版
useEffect(() => {
  const channel = supabase
    .channel('messages')
    .on('postgres_changes', { event: '*', schema: 'public', table: 'messages' }, handleChange)
    .subscribe();
  return () => {
    supabase.removeChannel(channel); // これで完全に購読を解除できる
  };
}, []);
```

Firebase の `onSnapshot` も同様で、返り値の `unsubscribe` 関数をクリーンアップで呼ぶ必要があります。

## Xcode Instruments でさらに深掘りする（iOS）

Flipper での確認が完了したら、iOS の場合は Xcode Instruments の「Leaks」テンプレートでも検証してみましょう。Flipper よりも詳細なスタックトレースが取れます。

```bash
# Rork Max の場合、Xcode プロジェクトが生成されているので直接 Instruments を起動できます
# Product → Profile → Leaks テンプレートを選択
# アプリを操作して赤い「\!」マークが出たらリーク箇所の詳細が確認できます
```

Instruments は学習コストが少し高いですが、「どのオブジェクトが解放されていないか」がひと目でわかります。Flipper でリークの存在は確認できたが原因が特定できないときに活用してみてください。

## 修正後の確認方法

修正を加えたら、必ず以下の手順で改善を確認してください。

1. Flipper を接続した状態でアプリを起動
2. 問題の画面を開く → 操作する → 戻る、を 10 回繰り返す
3. Memory タブの Heap Size が最初の値に近い水準に戻っているか確認

戻っていれば修正成功です。まだ増え続けているようであれば、別の箇所に同様のパターンが残っている可能性があります。Rork に「すべての useEffect にクリーンアップ関数が設定されているか確認して修正してください」と依頼すると、ひととおりチェックしてくれます。

## Rork への修正依頼プロンプトの書き方

自分でコードを修正するのが難しい場合は、Rork に直接依頼しましょう。効果的なプロンプトの例を紹介します。

```
以下のファイルを確認して、メモリリークの原因となっている箇所をすべて修正してください：
- useEffect 内のイベントリスナーにクリーンアップ関数が設定されているか
- setInterval / setTimeout のクリアが漏れていないか
- Supabase のリアルタイム購読を解除しているか
- 非同期処理の結果をアンマウント後に書き込もうとしていないか

修正後、変更箇所とその理由を教えてください。
```

このプロンプトを `src/screens/YourScreen.tsx` のような特定のファイルとセットで渡すと、的確に修正してくれます。

## 予防のための習慣

メモリリークは後から修正するより、最初から防ぐほうが楽です。Rork でコードを生成したあとに毎回確認するべき点をまとめました。

- `useEffect` を見つけたら、必ず「クリーンアップ関数が必要か」を考える
- タイマー系の処理（`setInterval`, `setTimeout`）には自動的にクリアを追加する習慣をつける
- 外部サービス（Supabase, Firebase, WebSocket）の購読はすべてクリーンアップで解除する
- 重要な画面を変更したあとは Flipper で 10 回往復テストをする

アプリが育ってくると「なぜか重くなってきた」という問題が必ず出てきます。その時に「診断の手順がわかっている」だけで、対処のスピードがまったく変わってきます。ぜひ今のうちに Flipper の使い方に慣れておいてください。

メモリリークの修正に取り組む過程で、あわせてアプリ全体のパフォーマンスも見直しておくと効果的です。[Rork アプリの動作が重くなったときのトラブルシューティング](/articles/rork-dev/rork-app-slow-performance-troubleshooting-fix)や、[React Native のビルドエラーを解決するガイド](/articles/rork-dev/rork-react-native-build-errors-fix)も参考にしてみてください。

まずは自分のアプリで `useEffect` を全文検索し、クリーンアップ関数が返されていない箇所がないか確認してみましょう。おそらく 1〜2 箇所は見つかるはずです。
