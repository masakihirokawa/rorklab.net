## はじめに

予約管理アプリは、飲食店・美容室・フィットネスジム・医療クリニックなど、あらゆるサービス業で需要の高いアプリジャンルです。従来はバックエンドの予約ロジック、決済連携、通知システムなど、複数の専門知識が求められましたが、**Rork Max** を使えば、AIの力で個人でも本格的な予約アプリを構築できます。

この記事では、以下の機能を持つフル機能の予約管理アプリをゼロから作成します。

- **カレンダーベースの予約UI**（月・週・日ビュー切り替え）
- **リアルタイム空き枠管理**（Supabase Realtime）
- **Stripe決済**（予約時の事前決済・キャンセルポリシー対応）
- **プッシュ通知**（予約確認・リマインダー・キャンセル通知）
- **管理者ダッシュボード**（予約一覧・売上レポート）

Rork Max の基本操作に慣れている方を対象としています。まだ Rork を使ったことがない方は、まず [Rorkの始め方ガイド](/articles/rork-basics/getting-started) をご確認ください。

## 概念と設計思想

### データベース設計

予約アプリの核となるのはデータベース設計です。正しい設計を最初に行うことで、後の開発がスムーズになります。

予約システムの主要テーブルは以下の4つです。

1. **services**（サービスマスタ）— 提供するサービスの種類・料金・所要時間
2. **time_slots**（時間枠）— 予約可能な日時の枠
3. **bookings**（予約）— ユーザーの予約情報
4. **payments**（決済）— Stripe 決済の記録

### 予約フローの設計

ユーザーが予約を完了するまでの流れは次のとおりです。

1. サービス選択 → 2. 日付選択 → 3. 時間枠選択 → 4. 確認画面 → 5. Stripe 決済 → 6. 予約確定 → 7. 確認通知

この流れをRork Maxで段階的に実装していきます。

## ステップバイステップ実装

### Step 1: Supabase テーブル作成

Supabase のダッシュボードで SQL Editor を開き、以下のスキーマを実行します。

```sql
-- サービステーブル
CREATE TABLE services (
  id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  name TEXT NOT NULL,
  description TEXT,
  duration_minutes INTEGER NOT NULL DEFAULT 60,
  price INTEGER NOT NULL,
  category TEXT,
  image_url TEXT,
  is_active BOOLEAN DEFAULT true,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- 時間枠テーブル
CREATE TABLE time_slots (
  id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  service_id UUID REFERENCES services(id),
  date DATE NOT NULL,
  start_time TIME NOT NULL,
  end_time TIME NOT NULL,
  max_bookings INTEGER DEFAULT 1,
  current_bookings INTEGER DEFAULT 0,
  is_available BOOLEAN DEFAULT true,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE(service_id, date, start_time)
);

-- 予約テーブル
CREATE TABLE bookings (
  id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  user_id UUID REFERENCES auth.users(id),
  service_id UUID REFERENCES services(id),
  time_slot_id UUID REFERENCES time_slots(id),
  status TEXT DEFAULT 'pending'
    CHECK (status IN ('pending','confirmed','cancelled','completed')),
  notes TEXT,
  stripe_payment_intent_id TEXT,
  total_price INTEGER NOT NULL,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW()
);

-- 決済テーブル
CREATE TABLE payments (
  id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  booking_id UUID REFERENCES bookings(id),
  stripe_payment_intent_id TEXT NOT NULL,
  amount INTEGER NOT NULL,
  currency TEXT DEFAULT 'jpy',
  status TEXT DEFAULT 'pending'
    CHECK (status IN ('pending','succeeded','failed','refunded')),
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- RLS（Row Level Security）ポリシー
ALTER TABLE bookings ENABLE ROW LEVEL SECURITY;

CREATE POLICY "Users can view own bookings"
  ON bookings FOR SELECT
  USING (auth.uid() = user_id);

CREATE POLICY "Users can create bookings"
  ON bookings FOR INSERT
  WITH CHECK (auth.uid() = user_id);

ALTER TABLE time_slots ENABLE ROW LEVEL SECURITY;

CREATE POLICY "Anyone can view available slots"
  ON time_slots FOR SELECT
  USING (is_available = true);
```

Rork Max のチャットで「Supabase の接続設定を追加して」と指示すれば、AI が環境変数と接続コードを自動生成します。

### Step 2: カレンダーUIの実装

カレンダーUIは予約アプリの要です。Rork Max に以下のプロンプトを送信します。

```text
予約画面にカレンダーコンポーネントを追加してください。
- 月表示で、日付をタップすると空き時間枠の一覧を表示する
- 予約可能な日は青色、満席の日はグレーで表示
- 選択中の日付はハイライト表示
- 下部に時間枠のリストを表示し、タップで予約フローに進む
- react-native-calendars ライブラリを使用
```

Rork Max が生成するカレンダーコンポーネントは、Supabase からリアルタイムでデータを取得し、空き状況を視覚的に表示します。生成されたコードの主要ポイントは次のとおりです。

```typescript
// カレンダーの日付マーキング生成
const generateMarkedDates = (slots: TimeSlot[]) => {
  const marked: Record<string, MarkedDateProps> = {};

  slots.forEach(slot => {
    const dateStr = slot.date;
    if (slot.current_bookings >= slot.max_bookings) {
      marked[dateStr] = {
        marked: true,
        dotColor: '#9CA3AF',
        customStyles: {
          container: { backgroundColor: '#F3F4F6' }
        }
      };
    } else {
      marked[dateStr] = {
        marked: true,
        dotColor: '#3B82F6',
        customStyles: {
          container: { backgroundColor: '#EFF6FF' }
        }
      };
    }
  });
  return marked;
};
```

// Supabase Realtime で空き枠を監視
useEffect(() => {

### Step 4: Stripe 決済の統合

予約時に事前決済を行うことで、無断キャンセルを防止します。Supabase Edge Function で決済サーバーを構築します。

```typescript
// supabase/functions/create-payment-intent/index.ts
import Stripe from 'https://esm.sh/stripe@14.14.0';

const stripe = new Stripe(Deno.env.get('STRIPE_SECRET_KEY')!);

Deno.serve(async (req) => {
  const { booking_id, amount } = await req.json();

  const paymentIntent = await stripe.paymentIntents.create({
    amount: amount,
    currency: 'jpy',
    metadata: { booking_id },
    automatic_payment_methods: { enabled: true },
  });

  const supabase = createClient(
    Deno.env.get('SUPABASE_URL')!,
    Deno.env.get('SUPABASE_SERVICE_ROLE_KEY')!
  );

  await supabase.from('payments').insert({
    booking_id,
    stripe_payment_intent_id: paymentIntent.id,
    amount,
    status: 'pending',
  });

  return new Response(JSON.stringify({
    clientSecret: paymentIntent.client_secret,
  }));
});
```

Rork Max のチャットで「Stripe Payment Sheet を予約確認画面に統合して」と指示すれば、クライアントサイドのコードも自動生成されます。Apple Pay と Google Pay も `automatic_payment_methods` を有効にすることで自動的に利用可能になります。

### Step 5: プッシュ通知の設定

予約確認やリマインダーの通知を設定します。

```sql
-- 24時間前リマインダーを毎時チェック（Supabase pg_cron）
SELECT cron.schedule(
  'booking-reminders',
  '0 * * * *',
  $$
  SELECT net.http_post(
    url := 'https://your-project.supabase.co/functions/v1/send-reminder',
    body := json_build_object(
      'bookings',
      (SELECT json_agg(row_to_json(b))
       FROM bookings b
       JOIN time_slots ts ON b.time_slot_id = ts.id
       WHERE b.status = 'confirmed'
       AND ts.date = CURRENT_DATE + INTERVAL '1 day'
       AND NOT b.reminder_sent)
    )::jsonb
  );
  $$
);
```

Rork Max に「Expo Notifications でプッシュ通知を送信する Edge Function を作って」と指示すれば、通知送信側のロジックも自動生成されます。

### キャンセルポリシーの自動適用

ビジネスに合わせたキャンセルポリシーを自動適用する仕組みも重要です。

```typescript
const calculateRefundAmount = (
  booking: Booking,
  slot: TimeSlot
): number => {
  const now = new Date();
  const bookingDate = new Date(
    `${slot.date}T${slot.start_time}`
  );
  const hoursUntil =
    (bookingDate.getTime() - now.getTime()) / (1000 * 60 * 60);

  if (hoursUntil >= 48) return booking.total_price;
  if (hoursUntil >= 24) return booking.total_price * 0.5;
  return 0;
};
```

48時間前までは全額返金、24時間前までは50%返金、それ以降は返金不可というルールを自動適用します。Stripe の Refund API と組み合わせることで、キャンセルから返金までを完全自動化できます。

## トラブルシューティング

### Supabase Realtime が動作しない

RLS ポリシーが正しく設定されているか確認してください。Realtime は RLS に従ってデータをフィルタリングします。`time_slots` テーブルに `SELECT` ポリシーが設定されていることが必要です。また、Supabase ダッシュボードの Replication 設定で、対象テーブルが有効になっているか確認しましょう。

### Stripe Payment Sheet が表示されない

Expo で Stripe を使う場合、`@stripe/stripe-react-native` のインストールと EAS ビルドが必要です。Expo Go ではネイティブモジュールが動作しないため、必ず `eas build --profile development` で開発ビルドを作成してテストしてください。

### ダブルブッキングが発生する

`time_slots` テーブルの `current_bookings` 更新時に、PostgreSQL のトランザクション分離レベルを活用します。`UPDATE ... WHERE current_bookings < max_bookings` を使い、更新行数が0の場合は「満席」として処理することで、同時アクセスによるダブルブッキングを防止できます。

## FAQ

**Q: 予約の同時アクセス（ダブルブッキング）を防ぐにはどうすればよいですか？**

Supabase のトランザクションとユニーク制約を活用します。`time_slots` テーブルの `current_bookings` を `UPDATE ... WHERE current_bookings < max_bookings` で更新し、更新行数が0の場合は「満席」として処理します。楽観的ロックで安全に制御できます。

**Q: オフラインでも予約内容を確認できますか？**

Rork Max でオフライン対応を有効にすれば、直近の予約データをローカルにキャッシュし、ネットワーク復帰時に自動同期する仕組みを構築できます。詳しくは [オフラインファーストアーキテクチャガイド](/articles/rork-dev/rork-max-offline-first-architecture) をご覧ください。

**Q: 管理者と一般ユーザーの権限をどう分けますか？**

Supabase の RLS ポリシーとカスタムクレーム（`app_metadata`）を使い分けます。管理者は全予約の閲覧・編集が可能に、一般ユーザーは自分の予約のみアクセス可能に設定します。

**Q: 複数店舗に対応させるにはどうすればよいですか？**

`services` テーブルに `shop_id` カラムを追加し、店舗ごとにサービスと時間枠を管理します。テナントIDベースの RLS ポリシーを設定すれば、マルチテナント対応の SaaS 型予約システムに発展させることが可能です。

**Q: テスト用のダミーデータはどう用意しますか？**

Supabase の SQL Editor でシードデータを一括挿入するのが効率的です。Rork Max のチャットで「テスト用のダミーサービスと時間枠データを30件生成するSQLを書いて」と指示すれば、AIがSEED文を自動生成します。

**Q: 予約完了後のメール通知も送れますか？**

Supabase Edge Functions と Resend（メール送信API）を組み合わせれば、予約確認メールやリマインダーメールを自動送信できます。Rork Maxで「Resend連携でメール通知を追加して」と指示するだけで実装の骨組みが生成されます。
