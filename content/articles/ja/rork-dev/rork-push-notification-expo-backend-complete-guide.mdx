## プッシュ通知で収益が変わった話

個人でアプリを開発してきた経験から言うと、プッシュ通知は「後回し」にされがちな機能です。でも、実装した途端にリテンション率が上がり、課金転換にも直結するという現実があります。

私が最初にプッシュ通知を実装したとき、DAU（日次アクティブユーザー）が2週間で1.4倍になりました。Rork で生成したアプリにこの機能を加える方法を、実装の詳細からバックエンドの設計まで全部お伝えします。

## Rork のコードベースと Expo Notifications の関係

Rork で生成されたアプリは React Native（Expo）ベースです。つまり、Expo が提供するプッシュ通知エコシステムをそのまま活用できます。

Expo のプッシュ通知は2層で構成されています：

**1. Expo Push API**（推奨）
Expo がサーバーとして APNs（Apple）と FCM（Google）を抽象化してくれます。開発者は Expo の API に通知を投げるだけで、iOS/Android 両方に送信できます。

**2. 直接連携**（上級者向け）
自前で APNs / FCM と連携する方法。制御は細かいですが実装コストが高い。最初は Expo Push API で十分です。

## 実装ステップ1：デバイストークンの取得

まず Rork に「通知許可を求めるコード」を追加してもらいます。Rork への指示例：

```
アプリ起動時（または onboarding 完了時）に、Expo のプッシュ通知許可を求めるコードを追加してください。

要件:
1. expo-notifications パッケージを使う
2. 許可が取れたら Expo Push Token を取得する
3. トークンを AsyncStorage に保存する
4. 同時にバックエンドの /api/register-token エンドポイントに POST する
5. 許可を拒否された場合はサイレントに失敗（エラー表示しない）
```

生成されるコードのイメージ：

```typescript
// hooks/usePushNotifications.ts
import { useEffect } from 'react';
import * as Notifications from 'expo-notifications';
import AsyncStorage from '@react-native-async-storage/async-storage';

export const usePushNotifications = () => {
  useEffect(() => {
    registerForPushNotifications();
  }, []);
  
  const registerForPushNotifications = async () => {
    try {
      const { status } = await Notifications.requestPermissionsAsync();
      if (status \!== 'granted') return;
      
      const token = (await Notifications.getExpoPushTokenAsync()).data;
      await AsyncStorage.setItem('pushToken', token);
      
      // バックエンドに登録
      await fetch('https://your-api.com/api/register-token', {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({ 
          token, 
          userId: await AsyncStorage.getItem('userId'),
          platform: Platform.OS
        })
      });
    } catch (error) {
      // サイレント失敗
    }
  };
};
```

## 実装ステップ2：バックエンドのトークン管理

トークンを受け取るサーバーサイドの実装です。私は Node.js + PostgreSQL の構成で管理しています。

```javascript
// routes/notifications.js (Node.js / Express)
const { Pool } = require('pg');
const pool = new Pool({ connectionString: process.env.DATABASE_URL });

// テーブル作成（初回のみ）
// CREATE TABLE push_tokens (
//   id SERIAL PRIMARY KEY,
//   user_id VARCHAR(255),
//   token TEXT UNIQUE NOT NULL,
//   platform VARCHAR(10),
//   is_active BOOLEAN DEFAULT true,
//   created_at TIMESTAMP DEFAULT NOW(),
//   last_used_at TIMESTAMP DEFAULT NOW()
// );

app.post('/api/register-token', async (req, res) => {
  const { token, userId, platform } = req.body;
  
  try {
    await pool.query(`
      INSERT INTO push_tokens (user_id, token, platform)
      VALUES ($1, $2, $3)
      ON CONFLICT (token) 
      DO UPDATE SET 
        user_id = $1,
        is_active = true,
        last_used_at = NOW()
    `, [userId, token, platform]);
    
    res.json({ success: true });
  } catch (error) {
    res.status(500).json({ error: error.message });
  }
});
```

## 実装ステップ3：通知の送信

Expo Push API を使った送信ロジックです。

```javascript
// services/pushNotification.js
const { Expo } = require('expo-server-sdk');
const expo = new Expo();

async function sendPushNotifications(tokens, message) {
  // バッチに分割（Expo の制限: 100件/リクエスト）
  const messages = tokens
    .filter(token => Expo.isExpoPushToken(token))
    .map(token => ({
      to: token,
      title: message.title,
      body: message.body,
      data: message.data || {},
      sound: 'default',
      badge: 1
    }));
  
  const chunks = expo.chunkPushNotifications(messages);
  const tickets = [];
  
  for (const chunk of chunks) {
    try {
      const ticketChunk = await expo.sendPushNotificationsAsync(chunk);
      tickets.push(...ticketChunk);
    } catch (error) {
      console.error('Send error:', error);
    }
  }
  
  // エラートークンを無効化（重要！）
  await handleTicketErrors(tickets, tokens);
  
  return tickets;
}

async function handleTicketErrors(tickets, tokens) {
  const errorTokens = tickets
    .filter(t => t.status === 'error' && t.details?.error === 'DeviceNotRegistered')
    .map((t, i) => tokens[i]);
  
  if (errorTokens.length > 0) {
    // 無効なトークンをDBで非アクティブ化
    await pool.query(
      'UPDATE push_tokens SET is_active = false WHERE token = ANY($1)',
      [errorTokens]
    );
  }
}
```

`DeviceNotRegistered` エラーの処理は非常に重要です。これを放置すると、無効なトークンへの送信が積み重なり、Expo からの送信制限を受けることがあります。

## セグメント配信の設計

全ユーザーに同じ通知を送るのは最善策ではありません。私が使っているセグメント設計です。

```javascript
// ユーザーセグメント別の通知戦略

// 1. アクティブユーザー（直近7日以内にアプリを開いた）
// 目的: エンゲージメント強化・新機能紹介
// 頻度: 週2〜3回まで
// 例: 「新機能が追加されました」「今週のおすすめ」

// 2. 休眠ユーザー（7〜30日間アプリを開いていない）
// 目的: 再アクティブ化
// 頻度: 2〜3日に1回（最大2週間試みる）
// 例: 「お帰りなさい！〇〇の新しい○○をチェックしてみてください」

// 3. 無料ユーザー（課金未実施）
// 目的: プレミアム転換促進
// 頻度: 週1回以下（しつこくしない）
// 例: 「プレミアム機能を7日間無料で試す」

// 4. 課金ユーザー
// 目的: ロイヤルティ強化・アップセル
// 頻度: 最小限（UX 重視）
// 例: 「いつもご支援ありがとうございます。新機能先行アクセスのご案内」

async function getSegmentTokens(segment) {
  const queries = {
    active: `
      SELECT t.token FROM push_tokens t
      JOIN users u ON t.user_id = u.id
      WHERE u.last_active_at > NOW() - INTERVAL '7 days'
      AND t.is_active = true
    `,
    dormant: `
      SELECT t.token FROM push_tokens t
      JOIN users u ON t.user_id = u.id
      WHERE u.last_active_at BETWEEN NOW() - INTERVAL '30 days' AND NOW() - INTERVAL '7 days'
      AND t.is_active = true
    `,
    free: `
      SELECT t.token FROM push_tokens t
      JOIN users u ON t.user_id = u.id
      WHERE u.subscription_status IS NULL
      AND t.is_active = true
    `
  };
  
  const result = await pool.query(queries[segment]);
  return result.rows.map(r => r.token);
}
```

## 開封率を上げる文面設計

どんなに精巧なシステムを作っても、通知文面が悪ければ開封されません。私の経験から効果的だったパターンです。

**効果的だったパターン:**
- 数字を入れる: 「3つの新機能」「あと2日」「50%オフ」
- 緊急感を出す（乱用禁止）: 「今日限り」「残り10枠」
- ユーザー名を使う: 「田中さん、お久しぶりです」
- 疑問形で興味を引く: 「〇〇したいとき、どうしていますか？」

**避けるべきパターン:**
- 曖昧すぎる: 「お知らせがあります」
- 押しつけすぎ: 「今すぐアップグレード！」
- 毎回同じ文面（慣れると開封しなくなる）

## 分析基盤の構築

送ったら終わりではなく、改善のためのデータ収集も大切です。

```typescript
// アプリ側: 通知タップ時のイベント記録
Notifications.addNotificationResponseReceivedListener(async (response) => {
  const { notificationId } = response.notification.request.content.data;
  
  // バックエンドに開封を記録
  await fetch('/api/notification-opened', {
    method: 'POST',
    body: JSON.stringify({ notificationId, openedAt: new Date().toISOString() })
  });
  
  // 目的のページへ遷移
  navigation.navigate(response.notification.request.content.data.screen);
});
```

計測すべき主な指標：
- **許可取得率**: 通知許可を求めたユーザーのうち、許可したユーザーの割合
- **開封率**: 送信数に対して開封（タップ）された割合（良い値: 5〜15%）
- **コンバージョン率**: 通知から目的のアクション（課金・シェアなど）を完了した割合

## 次のステップ

基本的なプッシュ通知システムが整ったら、次は **ローカル通知**（サーバー不要でアプリ内から送信）との組み合わせを検討してください。リマインダー・習慣化系アプリでは、ローカル通知の方が効果的な場面も多いです。

また、通知疲れ（Notification Fatigue）には常に注意してください。送りすぎてユーザーがオフにしてしまうのが、最も避けるべき状態です。「送らない勇気」も重要な設計判断です。
