RorkでアプリにFirebaseまたはSupabaseを組み込んでいると、「ある日突然ログインできなくなった」「本番ビルドだけ認証が通らない」といったトラブルに直面することがあります。認証エラーはユーザー体験に直結するため、原因の特定と迅速な対処が求められます。

本記事では、Rorkアプリで頻発するFirebase・Supabase認証エラーのパターンを整理し、それぞれの解決手順をStep by Stepで解説します。

## 症状別：よくある認証エラーの種類

Rorkで認証エラーが発生するとき、コンソールやネットワークログには主に以下のようなメッセージが現れます。

**Firebase認証の代表的なエラー:**
- `auth/network-request-failed` — ネットワーク接続の問題
- `auth/user-token-expired` — トークンの有効期限切れ
- `auth/invalid-api-key` — APIキーが無効またはプロジェクトの設定ミス
- `auth/unauthorized-domain` — 許可されていないドメインからのリクエスト
- `auth/too-many-requests` — 短時間に多数の認証リクエストが発生

**Supabase認証の代表的なエラー:**
- `AuthApiError: Invalid login credentials` — メール/パスワードの不一致
- `JWT expired` — JWTトークンの有効期限切れ
- `AuthSessionMissingError` — セッション情報が見つからない
- `new row violates row-level security policy` — RLSポリシーによるブロック
- HTTP `401 Unauthorized` または `403 Forbidden`

症状に合わせて後述の解決手順を参照してください。

## 原因の分析：なぜ認証エラーが起きるのか

認証エラーには複数のパターンがあります。主な原因を整理します。

**パターン1: APIキーや接続情報の設定ミス**

Rorkでは `app.json` や `.env` ファイルにAPIキーを記述しますが、開発環境と本番環境で値が混在していたり、コピーミスでキーが正しく反映されていないケースがよくあります。

**パターン2: トークンの有効期限切れ（自動リフレッシュ未設定）**

FirebaseのIDトークンは約1時間で失効します。Supabaseのアクセストークンも同様にTTL（有効期限）があります。トークンの自動リフレッシュ処理を実装していないと、長時間アプリを開いたままにしたユーザーが認証エラーに遭遇します。

**パターン3: CORSポリシーの違反（Web Preview時）**

RorkのWeb Preview機能でアプリを確認している場合、ブラウザのCORSポリシーにより認証リクエストがブロックされることがあります。特にSupabaseでは、許可するOriginの設定が必要です。

**パターン4: RLSポリシーの過剰制限**

SupabaseのRow Level Security（RLS）を有効にしている場合、ポリシーの設定が厳しすぎてログイン後のデータアクセスが403エラーになるケースがあります。

**パターン5: ネットワーク環境の問題（iOS/Androidビルド）**

ネイティブビルドでのみ認証が失敗する場合、`info.plist`（iOS）や `AndroidManifest.xml`（Android）のネットワーク設定が原因のことがあります。特にiOSではATSポリシーが厳しく、HTTPS以外の通信がブロックされます。

## 解決手順：Step by Step

### Step 1: APIキーと接続情報を確認する

まず最も基本的な設定ミスを排除します。

**Firebaseの場合:**

```javascript
// firebaseConfig.js の内容を確認
const firebaseConfig = {
  apiKey: "YOUR_API_KEY",          // Firebaseコンソールで確認
  authDomain: "yourapp.firebaseapp.com",
  projectId: "your-project-id",
  storageBucket: "your-project.appspot.com",
  messagingSenderId: "123456789",
  appId: "1:123456789:web:abcdef"
};

// 設定が正しいか確認するテストコード
import { initializeApp, getApps } from 'firebase/app';

const app = getApps().length === 0
  ? initializeApp(firebaseConfig)
  : getApps()[0];

console.log("Firebase initialized:", app.name); // "[DEFAULT]" と表示されれば正常
```

**Supabaseの場合:**

```javascript
// supabaseClient.js
import { createClient } from '@supabase/supabase-js';

const supabaseUrl = process.env.EXPO_PUBLIC_SUPABASE_URL;
const supabaseAnonKey = process.env.EXPO_PUBLIC_SUPABASE_ANON_KEY;

if (!supabaseUrl || !supabaseAnonKey) {
  throw new Error('Supabase URL または Anon Key が設定されていません');
}

export const supabase = createClient(supabaseUrl, supabaseAnonKey);
```

環境変数の設定は `.env` ファイルで行いますが、Expoでは `EXPO_PUBLIC_` プレフィックスが必須です。設定後は必ず `npx expo start --clear` でキャッシュをクリアしてください。

### Step 2: トークンの自動リフレッシュを実装する

**Firebaseの自動トークンリフレッシュ:**

```javascript
import { getAuth, onAuthStateChanged } from 'firebase/auth';

const auth = getAuth();

// アプリ起動時にセッションを復元
export function useAuthState() {
  const [user, setUser] = useState(null);
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    const unsubscribe = onAuthStateChanged(auth, (currentUser) => {
      setUser(currentUser);
      setLoading(false);
    });

    // クリーンアップ
    return () => unsubscribe();
  }, []);

  return { user, loading };
}
```

**SupabaseのセッションリフレッシュとAsyncStorage連携:**

```javascript
import { createClient } from '@supabase/supabase-js';
import AsyncStorage from '@react-native-async-storage/async-storage';
import { AppState } from 'react-native';

export const supabase = createClient(supabaseUrl, supabaseAnonKey, {
  auth: {
    storage: AsyncStorage,           // セッションをAsyncStorageに永続化
    autoRefreshToken: true,          // トークン自動リフレッシュ
    persistSession: true,            // セッション永続化
    detectSessionInUrl: false,       // React Nativeでは必ずfalse
  },
});

// アプリがフォアグラウンドに戻ったときにセッションをリフレッシュ
AppState.addEventListener('change', (state) => {
  if (state === 'active') {
    supabase.auth.startAutoRefresh();
  } else {
    supabase.auth.stopAutoRefresh();
  }
});
```

### Step 3: Firebaseの許可ドメインを設定する

Web Preview環境でFirebase Authが動作しない場合は、Firebaseコンソールで許可ドメインを追加します。

1. [Firebase Console](https://console.firebase.google.com/) を開く
2. プロジェクトを選択 → **Authentication** → **Settings**（設定）タブ
3. **承認済みドメイン**に以下を追加:
   - `localhost`
   - Rorkのプレビュー用URL（`*.rork.app` 形式）
   - 本番ドメイン（例: `yourapp.com`）

### Step 4: SupabaseのRLSポリシーを確認・修正する

RLSが原因のエラーは、まず一時的にRLSを無効にして問題が再現するか確認します。

```sql
-- テスト用（本番では必ず元に戻すこと）
ALTER TABLE your_table DISABLE ROW LEVEL SECURITY;
```

問題が解消した場合、ポリシーの修正が必要です。ログイン済みユーザーが自身のデータを読み書きできるポリシーの例:

```sql
-- 認証済みユーザーが自身のデータにアクセスできるポリシー
CREATE POLICY "Users can access their own data"
ON your_table
FOR ALL
USING (auth.uid() = user_id)
WITH CHECK (auth.uid() = user_id);
```

### Step 5: iOSのATSポリシーを確認する

iOS実機ビルドでのみ認証エラーが出る場合、`app.json` のATS設定を確認します。

```json
{
  "expo": {
    "ios": {
      "infoPlist": {
        "NSAppTransportSecurity": {
          "NSAllowsArbitraryLoads": false,
          "NSExceptionDomains": {
            "your-supabase-url.supabase.co": {
              "NSExceptionAllowsInsecureHTTPLoads": false,
              "NSIncludesSubdomains": true
            }
          }
        }
      }
    }
  }
}
```

基本的にFirebaseもSupabaseもHTTPSを使用するため `NSAllowsArbitraryLoads` は `false` のままで問題ありませんが、カスタムAPIを使用している場合はドメインを明示的に許可する必要があります。

## 確認方法：解決できたかの検証

修正後は以下の手順で動作を確認します。

**1. ネットワークリクエストのログ確認**

```javascript
// Supabaseのリクエストをデバッグログで確認
const supabase = createClient(supabaseUrl, supabaseAnonKey, {
  global: {
    fetch: (...args) => {
      console.log('[Supabase] Request:', args[0]);
      return fetch(...args).then(res => {
        console.log('[Supabase] Response:', res.status);
        return res;
      });
    }
  }
});
```

**2. セッション状態の確認**

```javascript
// 現在のセッション情報を取得（Supabase）
const { data: { session } } = await supabase.auth.getSession();
console.log('Session:', session ? 'Active' : 'None');
console.log('Expires at:', session?.expires_at
  ? new Date(session.expires_at * 1000).toISOString()
  : 'N/A');
```

**3. 認証フローのE2Eテスト**

開発ビルド、TestFlightビルド（iOS）、内部テストビルド（Android）の3環境で順番にログイン・ログアウト・セッション復元を確認することで、環境依存の問題を切り分けられます。

## 予防策：再発防止のベストプラクティス

認証エラーを未然に防ぐために、以下の実装を標準化することをおすすめします。

**エラーハンドリングの統一:**

```javascript
async function signIn(email, password) {
  try {
    const { data, error } = await supabase.auth.signInWithPassword({
      email,
      password,
    });

    if (error) {
      // エラーコードに応じてユーザーフレンドリーなメッセージを表示
      if (error.message.includes('Invalid login credentials')) {
        throw new Error('メールアドレスまたはパスワードが正しくありません');
      } else if (error.message.includes('Email not confirmed')) {
        throw new Error('メールアドレスの確認が完了していません');
      } else {
        throw new Error(`ログインに失敗しました: ${error.message}`);
      }
    }

    return data.user;
  } catch (err) {
    // Sentryなどのエラートラッキングに送信
    console.error('[Auth] Sign in error:', err.message);
    throw err;
  }
}
```

**定期的なセッション有効性チェック:**

アプリの重要な画面遷移時にセッションの有効性を確認し、失効している場合はログイン画面にリダイレクトするガード処理を実装することで、ユーザーが不意に操作できなくなる事態を防げます。

Rorkアプリのパフォーマンスやエラー監視については、[RorkアプリにSentryでクラッシュ監視を導入する方法](/articles/rork-dev/rork-sentry-crash-monitoring-setup-guide)も合わせてご覧ください。

## FAQ

**Q1. 開発環境では動くのに本番ビルドだけ認証が失敗します**

本番ビルドではデバッグモードが無効になるため、一部のエラーが表示されなくなります。`console.log` の代わりにエラートラッキング（SentryやFirebase Crashlyticsなど）を本番環境でも有効にしておくことで、原因特定が格段に早くなります。また、本番用と開発用のAPIキー・プロジェクトが分かれている場合は、本番用の設定が `app.config.js` や環境変数に正しく反映されているか確認してください。

**Q2. Supabaseで「new row violates row-level security policy」が出ます**

RLSポリシーが原因です。Supabaseダッシュボードの**Authentication → Policies**で、対象テーブルのポリシーを確認してください。`INSERT` 操作が許可されていない場合は、適切なポリシーを追加する必要があります。なお、`auth.uid()` が `null` になっている（つまり未ログイン状態）でデータ操作を試みている可能性もあるため、操作前に認証状態を確認するコードも追加することをおすすめします。

**Q3. Firebase AuthでGoogle/Apple Sign-Inが iOS実機だけ失敗します**

iOSでのOAuth認証には、Xcodeでの設定が必要です。具体的には、Firebase ConsoleからダウンロードしたGoogleService-Info.plistがプロジェクトに正しく含まれているか、Bundle IDがFirebaseコンソールの設定と一致しているかを確認してください。RorkからEASビルドを使用している場合は、`eas.json` の `buildProfile` にGoogleService-Info.plistのパスが正しく設定されているかも確認が必要です。

**Q4. JWTトークンが頻繁に失効してユーザーが強制ログアウトされます**

SupabaseのデフォルトのアクセストークンTTLは1時間ですが、ダッシュボードの**Authentication → Settings**から最大604800秒（7日間）まで延長できます。また、`autoRefreshToken: true` と `AppState` を組み合わせたセッション管理（Step 2参照）を実装することで、バックグラウンドからの復帰時にもトークンが自動更新されます。

**Q5. Rorkのプレビューとネイティブビルドで挙動が違います**

Rorkのプレビュー（Web）とネイティブビルドではJavaScriptエンジンや利用可能なAPIが異なります。特に `AsyncStorage` はWebでは `localStorage` にフォールバックしますが、動作保証の範囲外です。認証まわりの動作確認は、できる限り実機またはシミュレーター上で行うことを強くおすすめします。

## まとめ

RorkアプリのFirebase・Supabase認証エラーの主な原因と解決手順を解説しました。

- **APIキー・接続情報のミス** → コンソールで最新の値を再確認
- **トークン失効** → `autoRefreshToken` と `AppState` の組み合わせで自動リフレッシュを実装
- **CORS/ドメイン問題** → 許可ドメインの設定を見直す
- **RLSポリシーのブロック** → ポリシーを段階的に確認・修正
- **iOS ATS設定** → `app.json` の `NSAppTransportSecurity` を確認

認証は一度動いてもアップデートや設定変更で突然壊れることがあります。エラー監視ツールを早めに導入し、問題を即座に検知できる体制を整えることが、長期的なアプリ運用の安定につながります。

Supabaseの認証とリアルタイム機能の詳しい実装については、[SupabaseとRorkでリアルタイム認証アプリを作る](/articles/rork-dev/rork-supabase-auth-realtime-guide)もあわせてご覧ください。

モバイルアプリのバックエンド設計をさらに深く学びたい方には、[Firebase + React Native in Action（実践ガイド）](https://www.amazon.co.jp/dp/4873119677/?tag=pinocchio-22)（O'Reilly Japan）がおすすめです。認証フローの設計から本番運用まで体系的に学べます。
