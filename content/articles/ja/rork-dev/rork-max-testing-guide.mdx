## はじめに：Rork Maxアプリのテストがなぜ重要か

Rork MaxはAIプロンプトやビジュアルビルダーからReact Nativeコードを生成し、モバイルアプリ開発を加速します。しかし、AI生成コードには独特のテスト課題があります：自動生成されたコンポーネントは微細なバグを含む可能性があり、依存関係は急速にバージョン更新され、統合ポイントは慎重な検証が必要です。**生成されたアプリであっても、本番品質にはテストが不可欠です**。

モバイルアプリのテストピラミッドは3つのレイヤーで構成されます：

1. **ユニットテスト（70%）** — 個別のコンポーネント、フック、ユーティリティを隔離してテスト
2. **統合テスト（20%）** — 複数のコンポーネントの連携、API呼び出しをテスト
3. **E2Eテスト（10%）** — 実デバイス/シミュレータ上で完全なユーザーフローをテスト

このガイドでは、**Jest**（ユニット/統合テスト）と**Detox**（E2Eテスト）を使用し、GitHub Actionsで継続的な品質保証を実装します。

<Callout type="info">
Rork Maxプロジェクトは通常、ExpoとJestが事前設定されています。このガイドは生成されたRork Maxアプリ構造（Expo管理ワークフロー）から始まることを想定しています。
</Callout>

## Part 1：ユニットテスト用Jest設定

### テスト依存関係のインストール

Rork Maxプロジェクトは既にJestを持っていますが、必要なライブラリがすべてあることを確認します：

```bash
npm install --save-dev @testing-library/react-native @testing-library/jest-native jest-expo babel-jest
```

またはYarnで：

```bash
yarn add --dev @testing-library/react-native @testing-library/jest-native jest-expo babel-jest
```

### jest.config.js の設定

プロジェクトルートに `jest.config.js` を作成または更新します：

```javascript
module.exports = {
  preset: 'jest-expo',
  testEnvironment: 'node',
  setupFilesAfterEnv: ['<rootDir>/jest.setup.js'],
  testPathIgnorePatterns: ['/node_modules/', '/android/', '/ios/'],
  collectCoverageFrom: [
    'app/**/*.{ts,tsx}',
    '!app/**/*.d.ts',
    '!app/**/index.ts',
  ],
  coverageThreshold: {
    global: {
      branches: 60,
      functions: 60,
      lines: 60,
      statements: 60,
    },
  },
};
```

### jest.setup.js の作成

```javascript
import '@testing-library/jest-native/extend-expect';
```

### 最初のテストの実行

テストファイル `app/__tests__/App.test.tsx` を作成します：

```typescript
import React from 'react';
import { render, screen } from '@testing-library/react-native';
import App from '../App';

describe('Appコンポーネント', () => {
  it('ウェルカムメッセージをレンダリングする', () => {
    render(<App />);
    expect(screen.getByText(/Welcome/i)).toBeTruthy();
  });
});
```

テストを実行します：

```bash
npm test
```

<Callout type="warning">
「Cannot find module 'react-native'」エラーが表示される場合は、jest.config.jsのプリセットが「jest-expo」であることを確認してください。「jest-expo」プリセットは自動的にReact Nativeをモックします。
</Callout>

## Part 2：コンポーネントのユニットテスト

### ボタンコンポーネントのテスト

ほとんどのRork Maxアプリはボタンコンポーネントを含みます。カスタムボタンをテストしましょう：

```typescript
// app/components/Button.tsx
import React from 'react';
import { TouchableOpacity, Text } from 'react-native';

interface ButtonProps {
  title: string;
  onPress: () => void;
  disabled?: boolean;
}

export const Button: React.FC<ButtonProps> = ({ title, onPress, disabled }) => (
  <TouchableOpacity onPress={onPress} disabled={disabled}>
    <Text>{title}</Text>
  </TouchableOpacity>
);
```

テストを作成します：

```typescript
// app/__tests__/Button.test.tsx
import React from 'react';
import { render, screen, fireEvent } from '@testing-library/react-native';
import { Button } from '../components/Button';

describe('ボタンコンポーネント', () => {
  it('正しいタイトルでレンダリングされる', () => {
    render(<Button title="押してください" onPress={jest.fn()} />);
    expect(screen.getByText('押してください')).toBeTruthy();
  });

  it('押下時にonPressが呼ばれる', () => {
    const mockPress = jest.fn();
    render(<Button title="押してください" onPress={mockPress} />);

    const button = screen.getByRole('button', { name: /押してください/i });
    fireEvent.press(button);

    expect(mockPress).toHaveBeenCalledTimes(1);
  });

  it('無効化時はonPressが呼ばれない', () => {
    const mockPress = jest.fn();
    render(
      <Button title="押してください" onPress={mockPress} disabled={true} />
    );

    const button = screen.getByRole('button');
    fireEvent.press(button);

    expect(mockPress).not.toHaveBeenCalled();
  });
});
```

### フォームのテスト

フォームはアプリで重要です。フォーム送信をテストします：

```typescript
// app/__tests__/LoginForm.test.tsx
import React from 'react';
import { render, screen, fireEvent, waitFor } from '@testing-library/react-native';
import { LoginForm } from '../components/LoginForm';

describe('ログインフォームコンポーネント', () => {
  it('メールとパスワードでフォームが送信される', async () => {
    const mockSubmit = jest.fn();
    render(<LoginForm onSubmit={mockSubmit} />);

    const emailInput = screen.getByPlaceholderText('メール');
    const passwordInput = screen.getByPlaceholderText('パスワード');
    const submitButton = screen.getByText('サインイン');

    fireEvent.changeText(emailInput, 'user@example.com');
    fireEvent.changeText(passwordInput, 'password123');
    fireEvent.press(submitButton);

    await waitFor(() => {
      expect(mockSubmit).toHaveBeenCalledWith({
        email: 'user@example.com',
        password: 'password123',
      });
    });
  });

  it('メールが無効な場合はエラーを表示', async () => {
    render(<LoginForm onSubmit={jest.fn()} />);

    const emailInput = screen.getByPlaceholderText('メール');
    fireEvent.changeText(emailInput, 'invalid-email');

    await waitFor(() => {
      expect(screen.getByText(/無効なメール/i)).toBeTruthy();
    });
  });
});
```

## Part 3：renderHookでフックをテストする

カスタムフックはReact Nativeアプリで一般的です。`renderHook`でテストします：

```typescript
// app/hooks/useAuth.ts
import { useState, useCallback } from 'react';

export const useAuth = () => {
  const [user, setUser] = useState(null);
  const [loading, setLoading] = useState(false);

  const login = useCallback(async (email: string, password: string) => {
    setLoading(true);
    try {
      // API呼び出し
      const response = await fetch('/api/login', {
        method: 'POST',
        body: JSON.stringify({ email, password }),
      });
      const data = await response.json();
      setUser(data.user);
    } finally {
      setLoading(false);
    }
  }, []);

  const logout = useCallback(() => {
    setUser(null);
  }, []);

  return { user, loading, login, logout };
};
```

フックをテストします：

```typescript
// app/__tests__/useAuth.test.ts
import { renderHook, act, waitFor } from '@testing-library/react-native';
import { useAuth } from '../hooks/useAuth';

describe('useAuthフック', () => {
  beforeEach(() => {
    jest.clearAllMocks();
  });

  it('nullユーザーとfalseロード状態で初期化される', () => {
    const { result } = renderHook(() => useAuth());

    expect(result.current.user).toBeNull();
    expect(result.current.loading).toBe(false);
  });

  it('ログイン中はloadingがtrueになる', async () => {
    const { result } = renderHook(() => useAuth());

    act(() => {
      result.current.login('user@example.com', 'password123');
    });

    expect(result.current.loading).toBe(true);
  });

  it('ログイン成功後、ユーザーが更新される', async () => {
    const mockUser = { id: '123', email: 'user@example.com' };

    global.fetch = jest.fn(() =>
      Promise.resolve({
        json: () => Promise.resolve({ user: mockUser }),
      })
    ) as jest.Mock;

    const { result } = renderHook(() => useAuth());

    await act(async () => {
      await result.current.login('user@example.com', 'password123');
    });

    await waitFor(() => {
      expect(result.current.user).toEqual(mockUser);
      expect(result.current.loading).toBe(false);
    });
  });

  it('ログアウト時はユーザーがクリアされる', () => {
    const { result } = renderHook(() => useAuth());

    act(() => {
      result.current.logout();
    });

    expect(result.current.user).toBeNull();
  });
});
```

## Part 4：APIとサービスのモッキング

外部サービス（Supabaseなど）のモッキングは必須です。`jest.mock()`を使用します：

```typescript
// app/services/supabaseClient.ts
import { createClient } from '@supabase/supabase-js';

export const supabase = createClient(
  process.env.EXPO_PUBLIC_SUPABASE_URL!,
  process.env.EXPO_PUBLIC_SUPABASE_KEY!
);
```

テストでモックします：

```typescript
// app/__tests__/supabaseService.test.ts
import { supabase } from '../services/supabaseClient';

jest.mock('../services/supabaseClient', () => ({
  supabase: {
    from: jest.fn(),
  },
}));

describe('Supabaseサービス', () => {
  beforeEach(() => {
    jest.clearAllMocks();
  });

  it('投稿を正常に取得する', async () => {
    const mockPosts = [
      { id: 1, title: '投稿1' },
      { id: 2, title: '投稿2' },
    ];

    (supabase.from as jest.Mock).mockReturnValue({
      select: jest.fn().mockResolvedValue({
        data: mockPosts,
        error: null,
      }),
    });

    const { data, error } = await supabase
      .from('posts')
      .select('*');

    expect(data).toEqual(mockPosts);
    expect(error).toBeNull();
  });

  it('エラーを正常に処理する', async () => {
    const mockError = { message: 'ネットワークエラー' };

    (supabase.from as jest.Mock).mockReturnValue({
      select: jest.fn().mockResolvedValue({
        data: null,
        error: mockError,
      }),
    });

    const { data, error } = await supabase
      .from('posts')
      .select('*');

    expect(data).toBeNull();
    expect(error).toEqual(mockError);
  });
});
```

<Callout type="info">
テスト中のネットワーク呼び出しを避けるため、常に外部サービスをモックしてください。`jest.clearAllMocks()`を`beforeEach`で使用し、テスト間でモック状態をリセットしましょう。
</Callout>

## Part 5：スナップショットテスト

スナップショットテストはコンポーネント出力をキャプチャし、回帰を検出します：

```typescript
// app/__tests__/ProfileCard.snapshot.test.tsx
import React from 'react';
import { render } from '@testing-library/react-native';
import { ProfileCard } from '../components/ProfileCard';

describe('プロフィールカードスナップショット', () => {
  it('正しくレンダリングされる', () => {
    const tree = render(
      <ProfileCard
        name="太郎"
        bio="開発者"
        avatar="https://example.com/avatar.jpg"
      />
    ).toJSON();

    expect(tree).toMatchSnapshot();
  });
});
```

`npm test -- -u`を実行すると、Jestはスナップショットファイル（`ProfileCard.snapshot.test.tsx.snap`）を保存します。その後の実行では、Jestは出力をスナップショットと比較します。

**スナップショットを使うべき場合：**
- ✅ 構造が安定したUIコンポーネント
- ✅ デザイン変更の回帰検出

**スナップショットを避けるべき場合：**
- ❌ 動的コンテンツ（タイムスタンプ、ID）
- ❌ 頻繁に変わるUI

## Part 6：Detox E2Eセットアップ

Detoxは実デバイスまたはシミュレータ上でアプリをエンドツーエンドテストします。インストールします：

```bash
npm install --save-dev detox-cli detox detox-build-configuration
```

### .detoxrc.js の設定

プロジェクトルートに`.detoxrc.js`を作成します：

```javascript
module.exports = {
  testRunner: 'jest',
  apps: {
    ios: {
      type: 'ios.app',
      binaryPath: 'artifacts/build/Build/Products/Release-iphonesimulator/YourApp.app',
      build: 'xcodebuild -workspace ios/YourApp.xcworkspace -scheme YourApp -configuration Release -sdk iphonesimulator -derivedDataPath artifacts/build',
    },
  },
  configurations: {
    'ios.sim.debug': {
      device: {
        type: 'iPhone 14',
      },
      app: 'ios',
    },
  },
  testRunner: 'jest',
};
```

### アプリのビルド

```bash
detox build-framework-cache
detox build-app --configuration ios.sim.debug
```

### 最初のE2Eテストの実行

`e2e/firstTest.e2e.js`を作成します：

```javascript
describe('最初のE2Eテスト', () => {
  beforeAll(async () => {
    await device.launchApp();
  });

  beforeEach(async () => {
    await device.reloadReactNative();
  });

  it('ウェルカムスクリーンを表示すべき', async () => {
    await expect(element(by.text('Welcome'))).toBeVisible();
  });
});
```

テストを実行します：

```bash
detox test e2e/firstTest.e2e.js --configuration ios.sim.debug
```

<Callout type="warning">
DetoxはmacOS上でXcodeとiOSシミュレータが必要です。Androidテストの場合は`android.emu.debug`設定を使用します。Detoxセットアップは複雑になる可能性があります。プラットフォーム固有の問題については、公式Detoxドキュメントを参照してください。
</Callout>

## Part 7：Detox E2Eテストの例

### ログインフロー テスト

```javascript
// e2e/loginFlow.e2e.js
describe('ログインフロー', () => {
  beforeAll(async () => {
    await device.launchApp();
  });

  beforeEach(async () => {
    await device.reloadReactNative();
  });

  it('有効な認証情報でログインに成功すべき', async () => {
    // ログイン画面に移動
    await element(by.id('loginButton')).multiTap();

    // メールを入力
    await element(by.id('emailInput')).typeText('user@example.com');

    // パスワードを入力
    await element(by.id('passwordInput')).typeText('password123');

    // サインインボタンをタップ
    await element(by.id('signInButton')).multiTap();

    // ダッシュボードをチェックして、ログイン成功を確認
    await waitFor(element(by.text('ダッシュボード')))
      .toBeVisible()
      .withTimeout(5000);
  });

  it('無効なメールでエラーを表示すべき', async () => {
    await element(by.id('loginButton')).multiTap();
    await element(by.id('emailInput')).typeText('invalid-email');
    await element(by.id('signInButton')).multiTap();

    await expect(element(by.text('無効なメール'))).toBeVisible();
  });
});
```

### ナビゲーション フロー テスト


### フォーム送信テスト

```javascript

describe('フォーム送信', () => {
  beforeAll(async () => {
    await device.launchApp();
```

## Part 8：カバレッジレポート

### カバレッジレポートの生成

カバレッジ付きでJestを実行します：

```bash
npm test -- --coverage
```

これにより、`coverage/`ディレクトリにカバレッジレポートが生成されます。カバレッジメトリクスには以下が含まれます：

- **Statements** — 実行されたステートメントの割合
- **Branches** — テストされた条件分岐の割合
- **Functions** — 呼ばれた関数の割合
- **Lines** — 実行された行の割合

### カバレッジしきい値の設定

`jest.config.js`で最小しきい値を設定します：

```javascript
coverageThreshold: {
  global: {
    branches: 70,
    functions: 75,
    lines: 80,
    statements: 80,
  },
  './app/services/': {
    branches: 90,
    functions: 90,
    lines: 90,
    statements: 90,
  },
};
```

カバレッジがしきい値以下の場合、Jestはコード1で終了し、ビルドが失敗します。

### ブラウザでカバレッジレポートを表示

カバレッジの実行後、ブラウザで`coverage/lcov-report/index.html`を開き、詳細でインタラクティブなカバレッジレポートを確認します。

## Part 9：GitHub Actions CI統合

### .github/workflows/test.yml を作成

```yaml
name: Jest テスト

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main, develop]

jobs:
  test:
    runs-on: ubuntu-latest

    strategy:
      matrix:
        node-version: [18.x, 20.x]

    steps:
      - uses: actions/checkout@v4

      - name: Node.js ${{ matrix.node-version }} をセットアップ
        uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node-version }}
          cache: 'npm'

      - name: 依存関係をインストール
        run: npm ci

      - name: テストを実行
        run: npm test -- --coverage

      - name: Codecovにカバレッジをアップロード
        uses: codecov/codecov-action@v3
        with:
          files: ./coverage/lcov.info
          flags: unittests
          name: codecov-umbrella
          fail_ci_if_error: false
```

### すべてのPRで実行

ワークフローは自動的にプルリクエストで実行されます。テストが失敗するとPRは赤いXを表示します。テストが成功すると、緑のチェックマークが表示されます。

<Callout type="info">
READMEにバッジを追加して、テストステータスを表示しましょう：

```markdown
![テスト](https://github.com/YOUR_ORG/YOUR_REPO/workflows/Jest%20Tests/badge.svg)
```
</Callout>

## Part 10：CI内でのE2Eテスト

CIでDetoxを実行するにはAndroid/iOSビルドツールが必要です。iOSの最小例：

```yaml
name: Detox E2E テスト

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: macos-latest

    steps:
      - uses: actions/checkout@v4

      - name: Node.js をセットアップ
        uses: actions/setup-node@v4
        with:
          node-version: '20.x'
          cache: 'npm'

      - name: 依存関係をインストール
        run: npm ci

      - name: Detoxをビルド
        run: |
          detox build-framework-cache
          detox build-app --configuration ios.sim.debug

      - name: E2Eテストを実行
        run: detox test e2e --configuration ios.sim.debug --cleanup
```

<Callout type="warning">
macOS実行環境はLinuxより高額です。CI費用を削減するため、E2Eテストはメインブランチのみで実行し、すべてのPRでは実行しないことを検討してください。
</Callout>

## まとめ：Rork Maxアプリの成功するテスト戦略

Rork Maxアプリが本番対応であることを確認するために：

1. **ユニットテスト（Jest）** — コンポーネント、フック、ユーティリティをカバーします。80%以上の行カバレッジを目指しましょう。
2. **コンポーネントテスト** — ユーザーインタラクション（ボタンクリック、フォーム送信）をテストします。
3. **モッキング** — 外部APIとサービスをモックして、ロジックを隔離します。
4. **E2Eテスト（Detox）** — 重要なユーザーフロー（ログイン、支払い、コア機能）をテストします。
5. **CI/CD** — GitHub Actionsですべてのプルリクエストでテストを自動化します。
6. **カバレッジレポート** — カバレッジを追跡し、しきい値を実装します。
