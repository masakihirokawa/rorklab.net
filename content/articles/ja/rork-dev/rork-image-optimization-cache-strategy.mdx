## はじめに — なぜ画像最適化がアプリの成否を分けるのか

モバイルアプリにおいて、画像はユーザー体験を大きく左右する要素です。美しい画像はアプリの魅力を高めますが、最適化されていない画像はアプリの動作を著しく遅くし、ユーザーの離脱を招きます。

実際のデータによると、画像の読み込みに3秒以上かかると約40%のユーザーがアプリを離脱するとされています。Rork で開発するアプリでも、画像の扱い方次第でユーザー満足度やストアの評価が大きく変わります。

この記事では、Rork アプリにおける画像最適化の基本から、キャッシュ戦略、メモリ管理、レスポンシブ対応まで、パフォーマンスを劇的に改善する実践テクニックを解説します。React Native / Expo 環境で使えるコード例も豊富に掲載していますので、すぐに自分のプロジェクトに取り入れていただけます。

## Expo Image を使った高速な画像表示

Rork のアプリは React Native + Expo をベースにしているため、画像表示には `expo-image` ライブラリの活用が効果的です。従来の React Native の `Image` コンポーネントに比べて、`expo-image` はディスクキャッシュやメモリキャッシュを内蔵しており、パフォーマンス面で大きなアドバンテージがあります。

```tsx
// expo-image を使った基本的な画像表示
import { Image } from 'expo-image';

// ブラーハッシュによるプレースホルダー付き画像コンポーネント
const OptimizedImage = ({ uri, blurhash, width, height }) => {
  return (
    <Image
      source={{ uri }}
      placeholder={{ blurhash }}
      contentFit="cover"
      transition={300}
      style={{ width, height, borderRadius: 8 }}
      cachePolicy="memory-disk"
      recyclingKey={uri}
    />
  );
};

// 使用例
// <OptimizedImage
//   uri="https://example.com/photo.webp"
//   blurhash="LKO2:N%2Tw=w]~RBVZRi};RPxuwH"
//   width={300}
//   height={200}
// />
// → ブラーハッシュが先に表示され、画像読み込み完了後にフェードイン
```

`cachePolicy` を `"memory-disk"` に設定することで、一度読み込んだ画像はメモリとディスクの両方にキャッシュされます。同じ画像を再表示する際にはネットワークリクエストが発生しないため、表示速度が格段に向上します。

## キャッシュ戦略の設計 — 3層キャッシュアーキテクチャ

効率的な画像キャッシュを実現するには、用途に応じた3層のキャッシュ戦略が有効です。

**メモリキャッシュ（L1）** は最も高速なキャッシュ層です。現在表示中の画面やすぐに表示される可能性の高い画像をメモリ上に保持します。ただしメモリは有限なので、LRU（Least Recently Used）方式で古い画像から自動的に解放されます。

**ディスクキャッシュ（L2）** はアプリのローカルストレージに画像を保存する層です。アプリを再起動しても画像が残るため、2回目以降の表示が高速になります。`expo-image` はこの層を自動で管理してくれます。

**CDNキャッシュ（L3）** はサーバーサイドのキャッシュです。画像をCDN（Content Delivery Network）経由で配信することで、ユーザーに近いエッジサーバーから画像が配信され、初回読み込みも高速化できます。

```tsx
// 3層キャッシュを活用した画像マネージャー
import { Image } from 'expo-image';
import * as FileSystem from 'expo-file-system';

const IMAGE_CACHE_DIR = FileSystem.cacheDirectory + 'images/';

// キャッシュディレクトリの初期化
const ensureCacheDir = async () => {
  const dirInfo = await FileSystem.getInfoAsync(IMAGE_CACHE_DIR);
  if (!dirInfo.exists) {
    await FileSystem.makeDirectoryAsync(IMAGE_CACHE_DIR, {
      intermediates: true,
    });
  }
};

// 画像のプリフェッチ（次の画面で使う画像を先読み）
const prefetchImages = async (urls: string[]) => {
  await Promise.all(
    urls.map((url) => Image.prefetch(url))
  );
};

// キャッシュサイズの取得と管理
const getCacheSize = async () => {
  const dirInfo = await FileSystem.getInfoAsync(IMAGE_CACHE_DIR, {
    size: true,
  });
  const sizeMB = (dirInfo.size || 0) / (1024 * 1024);
  return sizeMB; // MB単位で返す
};

// キャッシュのクリア（設定画面などから呼び出し）
const clearImageCache = async () => {
  await Image.clearDiskCache();
  await Image.clearMemoryCache();
  console.log('画像キャッシュをクリアしました');
};

// 期待する出力:
// prefetchImages 完了後、該当画像の表示が即座に行われる
// getCacheSize() → 例: 45.2 (MB)
```

## プレースホルダーとプログレッシブ表示

画像の読み込み中にユーザーに何も表示しないのは、体験として最も良くないパターンです。プレースホルダーを活用することで、読み込み中でもアプリがレスポンシブに感じられます。

**ブラーハッシュ（BlurHash）** は、画像をわずか20〜30文字の文字列にエンコードする技術です。サーバー側で事前に計算しておき、APIのレスポンスに含めることで、画像の読み込み前にぼやけたプレビューを即座に表示できます。

```tsx
// ブラーハッシュ付きの画像リストコンポーネント
import { Image } from 'expo-image';
import { FlatList, View, StyleSheet } from 'react-native';

const ImageGrid = ({ images }) => {
  const renderItem = ({ item }) => (
    <View style={styles.imageContainer}>
      <Image
        source={{ uri: item.url }}
        placeholder={{ blurhash: item.blurhash }}
        contentFit="cover"
        transition={250}
        style={styles.image}
        cachePolicy="memory-disk"
        // 画面外の画像は低優先度で読み込む
        priority={item.isVisible ? 'high' : 'low'}
      />
    </View>
  );

  return (
    <FlatList
      data={images}
      renderItem={renderItem}
      keyExtractor={(item) => item.id}
      numColumns={2}
      // 画面外の描画領域を制限してメモリ節約
      windowSize={5}
      maxToRenderPerBatch={6}
      removeClippedSubviews={true}
    />
  );
};

const styles = StyleSheet.create({
  imageContainer: {
    flex: 1,
    margin: 4,
    borderRadius: 8,
    overflow: 'hidden',
  },
  image: {
    width: '100%',
    aspectRatio: 1,
  },
});

// 期待する動作:
// 1. グリッド表示時にブラーハッシュが即座に表示される
// 2. 画像読み込み完了後に250msのフェードインで切り替わる
// 3. 画面外の画像は遅延読み込みされメモリを節約
```

## レスポンシブ画像 — デバイスに最適なサイズを配信する

同じ画像でも、iPhone SE の小さな画面と iPad Pro の大きな画面では必要な解像度が異なります。フル解像度の画像を全デバイスに配信するのは帯域の無駄遣いです。

画像のリサイズをサーバーサイドで行い、デバイスの画面サイズに応じた最適なサイズをリクエストする方法が効果的です。Cloudflare Images や Imgix などの画像変換サービスを利用すると、URLパラメータでリサイズが可能です。

```tsx
// デバイスに最適な画像URLを生成するユーティリティ
import { Dimensions, PixelRatio } from 'react-native';

const getOptimizedImageUrl = (
  baseUrl: string,
  displayWidth: number
) => {
  // デバイスのピクセル密度を考慮した実際の必要ピクセル数
  const pixelWidth = Math.round(
    displayWidth * PixelRatio.get()
  );

  // 最大幅を制限（過度な解像度は不要）
  const targetWidth = Math.min(pixelWidth, 1200);

  // Cloudflare Images の例
  // /cdn-cgi/image/width=600,format=webp/original-image.jpg
  return `${baseUrl}?w=${targetWidth}&format=webp&q=80`;
};

// 使用例
const screenWidth = Dimensions.get('window').width;
const imageUrl = getOptimizedImageUrl(
  'https://example.com/images/hero.jpg',
  screenWidth
);
// iPhone 15 Pro (PixelRatio: 3, width: 393) の場合:
// → "https://example.com/images/hero.jpg?w=1179&format=webp&q=80"
// iPhone SE (PixelRatio: 2, width: 375) の場合:
// → "https://example.com/images/hero.jpg?w=750&format=webp&q=80"
```

WebP フォーマットを使用することで、JPEG に比べて25〜35%のファイルサイズ削減が期待できます。画質パラメータ `q=80` は、多くのケースで視覚的に劣化がわからない範囲でファイルサイズを抑えるバランスの良い設定です。

## メモリ管理 — アプリクラッシュを防ぐ

画像を大量に扱うアプリでは、メモリ管理が特に重要です。メモリリークや過剰なメモリ使用は、アプリのクラッシュに直結します。

```tsx
// メモリを考慮した画像リスト実装
import { useCallback, useEffect, useRef } from 'react';
import { Image } from 'expo-image';
import { FlatList, AppState } from 'react-native';

const MemoryEfficientImageList = ({ images }) => {
  const flatListRef = useRef(null);

  // アプリがバックグラウンドに移行した際にメモリを解放
  useEffect(() => {
    const subscription = AppState.addEventListener(
      'change',
      (nextAppState) => {
        if (nextAppState === 'background') {
          // メモリキャッシュをクリアして解放
          Image.clearMemoryCache();
        }
      }
    );
    return () => subscription.remove();
  }, []);

  const renderItem = useCallback(
    ({ item }) => (
      <Image
        source={{ uri: item.url }}
        style={{ width: '100%', height: 200 }}
        contentFit="cover"
        cachePolicy="disk"
        recyclingKey={item.id}
      />
    ),
    []
  );

  // getItemLayout で FlatList のスクロール性能を向上
  const getItemLayout = useCallback(
    (data, index) => ({
      length: 200,
      offset: 200 * index,
      index,
    }),
    []
  );

  return (
    <FlatList
      ref={flatListRef}
      data={images}
      renderItem={renderItem}
      keyExtractor={(item) => item.id}
      getItemLayout={getItemLayout}
      windowSize={3}
      maxToRenderPerBatch={4}
      removeClippedSubviews={true}
      initialNumToRender={5}
    />
  );
};

// 期待する動作:
// - スクロール時にメモリ使用量が安定（急増しない）
// - バックグラウンド移行時にメモリキャッシュが自動解放
// - removeClippedSubviews で画面外の画像がアンマウントされる
```

`recyclingKey` を設定することで、`expo-image` が内部でビューのリサイクルを効率的に行い、スクロール時のメモリ使用量を抑えられます。`cachePolicy` を `"disk"` にすることで、メモリキャッシュを使わずディスクキャッシュのみを使用する設定も可能です。大量の画像を扱うリスト画面では、この設定がメモリ節約に有効です。

## パフォーマンス計測と改善サイクル

最適化の効果を正しく評価するには、具体的な数値で計測することが大切です。Rork アプリのパフォーマンスを計測するための基本的なアプローチを紹介します。

```tsx
// 画像読み込み時間を計測するカスタムフック
import { useState, useCallback } from 'react';

const useImageLoadMetrics = () => {
  const [metrics, setMetrics] = useState({
    loadCount: 0,
    totalLoadTime: 0,
    averageLoadTime: 0,
    cacheHitRate: 0,
    cacheHits: 0,
  });

  const onLoadStart = useCallback(() => {
    return Date.now();
  }, []);

  const onLoadEnd = useCallback(
    (startTime: number, fromCache: boolean) => {
      const loadTime = Date.now() - startTime;
      setMetrics((prev) => {
        const newCount = prev.loadCount + 1;
        const newTotal = prev.totalLoadTime + loadTime;
        const newCacheHits = prev.cacheHits + (fromCache ? 1 : 0);
        return {
          loadCount: newCount,
          totalLoadTime: newTotal,
          averageLoadTime: Math.round(newTotal / newCount),
          cacheHits: newCacheHits,
          cacheHitRate: Math.round(
            (newCacheHits / newCount) * 100
          ),
        };
      });
    },
    []
  );

  return { metrics, onLoadStart, onLoadEnd };
};

// 使用例
// const { metrics } = useImageLoadMetrics();
// console.log(metrics);
// → { loadCount: 50, totalLoadTime: 8500,
//     averageLoadTime: 170, cacheHits: 35,
//     cacheHitRate: 70 }
```

平均読み込み時間が200ms以下、キャッシュヒット率が60%以上を目標にすると、ユーザーが快適に感じるアプリになります。さらに詳しいパフォーマンス最適化の手法については、[Rork アプリのパフォーマンス最適化 — レンダリング・メモリ・ネットワークの実践テクニック](/articles/rork-dev/advanced-rork-app-performance-optimization) で体系的に解説しています。

## よくある質問（FAQ）

### Q: expo-image と React Native 標準の Image コンポーネントの違いは？

`expo-image` は、ディスク・メモリの2層キャッシュを内蔵し、BlurHash / ThumbHash によるプレースホルダー、WebP / AVIF 対応、ビューのリサイクルなど、パフォーマンスに関する機能が充実しています。標準の `Image` コンポーネントは基本的なキャッシュのみで、大量の画像を扱うアプリでは `expo-image` の方が明らかに高速です。新規プロジェクトでは `expo-image` を使うことをおすすめします。

### Q: キャッシュサイズの上限はどのくらいに設定すべき？

一般的には、ディスクキャッシュの上限を100〜200MB程度に設定するのが適切です。ユーザーのデバイスストレージを圧迫しない範囲で、よく表示される画像がキャッシュに収まるサイズが理想です。写真アプリのように大量の画像を扱う場合は300MB程度まで拡張しても問題ありません。`expo-image` のデフォルト設定は多くのケースで十分です。

### Q: WebP と JPEG のどちらを使うべき？

2026年現在、モバイルアプリでは WebP の使用を強くおすすめします。WebP は JPEG に比べて25〜35%小さいファイルサイズで同等の画質を実現でき、透過（アルファチャンネル）もサポートしています。iOS / Android ともに WebP を完全サポートしているため、互換性の問題もありません。サーバー側で画像変換サービスを使い、リクエスト時に `format=webp` を指定するのが最も手軽な導入方法です。

### Q: Rork のプロンプトで画像最適化を指示するコツは？

Rork にアプリを生成させる際、「画像表示には expo-image を使い、BlurHash プレースホルダーとディスクキャッシュを有効にしてください」と明示的に指示すると、最適化された実装が生成されやすくなります。また「FlatList の windowSize を 5 に設定し、removeClippedSubviews を有効にしてください」のように具体的なパラメータを伝えると、より高品質なコードが得られます。
