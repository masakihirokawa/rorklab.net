Rorkでアプリを開発していると、データフェッチの管理が複雑になってくることがあります。ローディング状態・エラー処理・キャッシュ・バックグラウンド更新……これらを自前で実装するのは意外と手間がかかります。そこで活躍するのが **TanStack Query（旧 React Query）v5** です。

本記事では、RorkアプリにTanStack Queryを導入し、データ取得から更新・キャッシュ最適化までを一通り実践できるよう丁寧に解説します。

## TanStack Query とは？Rorkアプリで使う理由

TanStack Query（v5）は、React / React Native アプリのサーバー状態管理に特化したライブラリです。「サーバー状態」とは、APIやデータベースから取得するデータのことで、ローカルのUIステートとは性質が異なります。

Rork は React Native（Expo）をベースにしており、TanStack Query は Expo 環境でも問題なく動作します。通常の `useState` + `useEffect` によるデータフェッチに比べて、以下の点で優れています。

- **自動キャッシュ**: 一度取得したデータはメモリ内にキャッシュされ、再レンダリング時に不要なAPIリクエストを防ぐ
- **バックグラウンド再フェッチ**: アプリがフォアグラウンドに戻ったとき、古いデータを自動で更新
- **ローディング・エラー状態の一元管理**: `isLoading` / `isError` / `data` がシンプルに取り出せる
- **楽観的更新（Optimistic Update）**: UXを損なわずにデータ変更を即座に反映
- **無限スクロール・ページネーション**: `useInfiniteQuery` で対応可能

「useState + useEffect で十分では？」と思うかもしれません。しかし、アプリの画面数が増えるにつれて、同じデータを複数箇所でフェッチしたり、キャッシュの整合性が取れなくなるケースが出てきます。TanStack Queryはそこを一手に引き受けてくれます。

## セットアップ：Rorkプロジェクトへの導入

まず、必要なパッケージをインストールします。

```bash
# TanStack Query v5 をインストール
npx expo install @tanstack/react-query

# オプション：DevTools（開発時のデバッグに便利）
npx expo install @tanstack/react-query-devtools
```

インストールが完了したら、アプリのルートコンポーネント（`app/_layout.tsx` など）に `QueryClientProvider` を設定します。

```tsx
// app/_layout.tsx
import { QueryClient, QueryClientProvider } from "@tanstack/react-query";
import { Stack } from "expo-router";

// QueryClient をモジュールレベルで作成（コンポーネント外）
const queryClient = new QueryClient({
  defaultOptions: {
    queries: {
      // データが「古い」と見なすまでの時間（ミリ秒）
      staleTime: 1000 * 60 * 5, // 5分
      // キャッシュをメモリに保持する時間
      gcTime: 1000 * 60 * 10, // 10分
      // エラー時の自動リトライ回数
      retry: 2,
    },
  },
});

export default function RootLayout() {
  return (
    <QueryClientProvider client={queryClient}>
      <Stack />
    </QueryClientProvider>
  );
}
```

> **ポイント**: `staleTime` のデフォルトは `0`（常に古い扱い）です。APIコールを減らしたい場合は用途に応じて調整しましょう。

## useQuery の基本：データ取得をシンプルに

最も基本的な使い方は `useQuery` フックです。タスク一覧を取得する例を見てみましょう。

```tsx
// hooks/useTasks.ts
import { useQuery } from "@tanstack/react-query";

// データ取得関数（非同期関数であれば何でもOK）
async function fetchTasks() {
  const response = await fetch("https://api.example.com/tasks");
  if (!response.ok) {
    // エラーを throw することで TanStack Query がキャッチ
    throw new Error("タスクの取得に失敗しました");
  }
  return response.json();
}

export function useTasks() {
  return useQuery({
    queryKey: ["tasks"], // キャッシュのキー（ユニークな識別子）
    queryFn: fetchTasks,
  });
}
```

```tsx
// components/TaskList.tsx
import { View, Text, ActivityIndicator } from "react-native";
import { useTasks } from "@/hooks/useTasks";

export function TaskList() {
  const { data, isLoading, isError, error } = useTasks();

  if (isLoading) {
    return <ActivityIndicator />;
  }

  if (isError) {
    return <Text>エラー: {error.message}</Text>;
  }

  return (
    <View>
      {data.map((task: { id: string; title: string }) => (
        <Text key={task.id}>{task.title}</Text>
      ))}
    </View>
  );
}
```

`queryKey` は配列で指定します。同じ `queryKey` を持つクエリはキャッシュを共有するため、異なるコンポーネントで同じデータを取得しても、APIリクエストは1回だけ発生します。

## queryKey の設計：動的パラメータの扱い方

ユーザーIDや検索クエリなど、動的なパラメータに基づいてデータを取得する場合は、`queryKey` にパラメータを含めます。

```tsx
// hooks/useTaskDetail.ts
import { useQuery } from "@tanstack/react-query";

async function fetchTaskById(taskId: string) {
  const response = await fetch(`https://api.example.com/tasks/${taskId}`);
  if (!response.ok) throw new Error("タスクが見つかりません");
  return response.json();
}

export function useTaskDetail(taskId: string) {
  return useQuery({
    queryKey: ["tasks", taskId], // taskId が変わると別のキャッシュエントリになる
    queryFn: () => fetchTaskById(taskId),
    enabled: !!taskId, // taskId が空の場合はフェッチしない
  });
}
```

`enabled` オプションを使うと、条件が満たされるまでクエリの実行を遅延できます。ログイン後にのみデータを取得するケースなどで便利です。

## useMutation：データの作成・更新・削除

サーバーへのデータ書き込みには `useMutation` を使います。新しいタスクを追加する例です。

```tsx
// hooks/useCreateTask.ts
import { useMutation, useQueryClient } from "@tanstack/react-query";

async function createTask(newTask: { title: string; completed: boolean }) {
  const response = await fetch("https://api.example.com/tasks", {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify(newTask),
  });
  if (!response.ok) throw new Error("タスクの作成に失敗しました");
  return response.json();
}

export function useCreateTask() {
  const queryClient = useQueryClient();

  return useMutation({
    mutationFn: createTask,
    onSuccess: () => {
      // 成功したら "tasks" のキャッシュを無効化して再フェッチを促す
      queryClient.invalidateQueries({ queryKey: ["tasks"] });
    },
    onError: (error) => {
      console.error("ミューテーションエラー:", error.message);
    },
  });
}
```

```tsx
// components/AddTaskForm.tsx
import { useState } from "react";
import { View, TextInput, Button, Text } from "react-native";
import { useCreateTask } from "@/hooks/useCreateTask";

export function AddTaskForm() {
  const [title, setTitle] = useState("");
  const { mutate, isPending, isError } = useCreateTask();

  const handleAdd = () => {
    if (!title.trim()) return;
    mutate(
      { title, completed: false },
      {
        onSuccess: () => {
          setTitle(""); // 送信後にフォームをリセット
        },
      }
    );
  };

  return (
    <View>
      <TextInput
        value={title}
        onChangeText={setTitle}
        placeholder="タスクを入力"
      />
      <Button title={isPending ? "追加中..." : "追加"} onPress={handleAdd} />
      {isError && <Text>エラーが発生しました</Text>}
    </View>
  );
}
```

`invalidateQueries` を呼ぶことで、`queryKey: ["tasks"]` に一致するキャッシュが無効化され、次にその画面が表示されたときに自動で最新データが取得されます。

## Supabase との連携

Rorkアプリでは Supabase をバックエンドとして使うケースが多いです。Supabase のクライアントメソッドは Promise を返すため、TanStack Query とシームレスに連携できます。

```tsx
// lib/supabase.ts（既存の設定ファイル）
import { createClient } from "@supabase/supabase-js";

export const supabase = createClient(
  process.env.EXPO_PUBLIC_SUPABASE_URL!,
  process.env.EXPO_PUBLIC_SUPABASE_ANON_KEY!
);
```

```tsx
// hooks/useSupabaseTasks.ts
import { useQuery, useMutation, useQueryClient } from "@tanstack/react-query";
import { supabase } from "@/lib/supabase";

// Supabase からタスクを取得
async function fetchSupabaseTasks() {
  const { data, error } = await supabase
    .from("tasks")
    .select("*")
    .order("created_at", { ascending: false });

  if (error) throw new Error(error.message);
  return data;
}

export function useSupabaseTasks() {
  return useQuery({
    queryKey: ["supabase", "tasks"],
    queryFn: fetchSupabaseTasks,
    staleTime: 1000 * 30, // 30秒間はキャッシュを新鮮と見なす
  });
}

// タスクの完了状態を更新
export function useToggleTask() {
  const queryClient = useQueryClient();

  return useMutation({
    mutationFn: async ({
      id,
      completed,
    }: {
      id: string;
      completed: boolean;
    }) => {
      const { error } = await supabase
        .from("tasks")
        .update({ completed })
        .eq("id", id);

      if (error) throw new Error(error.message);
    },
    onSuccess: () => {
      queryClient.invalidateQueries({ queryKey: ["supabase", "tasks"] });
    },
  });
}
```

Supabase と組み合わせた実践的な認証フローについては、[Rork × Supabase 認証・リアルタイム完全ガイド](/articles/rork-dev/rork-supabase-auth-realtime-guide) も参考にしてください。

## キャッシュ戦略：staleTime と gcTime を理解する

TanStack Queryのキャッシュ動作を理解することは、パフォーマンス最適化に直結します。

**staleTime（新鮮期間）**: データを「新鮮（fresh）」と見なす時間。この期間内は、再マウント時やウィンドウフォーカス時にAPIリクエストを発行しません。

**gcTime（ガーベジコレクション時間、旧 cacheTime）**: 使われていないキャッシュをメモリから削除するまでの時間。

```tsx
// 用途別のキャッシュ設定例
const queries = {
  // ユーザープロフィール（滅多に変わらないデータ）
  userProfile: {
    staleTime: 1000 * 60 * 60, // 1時間
    gcTime: 1000 * 60 * 60 * 24, // 24時間
  },
  // 商品在庫（頻繁に変わるデータ）
  inventory: {
    staleTime: 0, // 常に再フェッチ
    gcTime: 1000 * 60 * 5, // 5分
  },
  // ニュースフィード（中程度の更新頻度）
  newsFeed: {
    staleTime: 1000 * 60 * 5, // 5分
    gcTime: 1000 * 60 * 30, // 30分
  },
};
```

データの性質に合わせて `staleTime` を調整することで、不必要なAPIコールを削減しつつ、常に最新のデータをユーザーに届けられます。

## よくあるエラーと対処法

### クエリが実行されない

`enabled` が `false` になっていないか確認してください。また、`queryClient` が `QueryClientProvider` のスコープ内で使われているかも確認しましょう。

```tsx
// NG: enabled に undefined が渡っている
useQuery({ queryKey: ["user", userId], queryFn: ..., enabled: userId });

// OK: 明示的に boolean に変換
useQuery({ queryKey: ["user", userId], queryFn: ..., enabled: !!userId });
```

### キャッシュが更新されない

`invalidateQueries` の `queryKey` がフェッチ時のキーと完全に一致しているか確認してください。部分一致でも動作しますが、意図しない範囲を無効化する可能性があります。

```tsx
// "tasks" で始まる全てのキャッシュを無効化（["tasks", "1"] なども含む）
queryClient.invalidateQueries({ queryKey: ["tasks"] });

// 完全一致のみ無効化
queryClient.invalidateQueries({
  queryKey: ["tasks", specificId],
  exact: true,
});
```

### TypeScript の型エラー

v5 では型推論が強化されています。`queryFn` の戻り値から型が自動推論されますが、明示的に型を指定することもできます。

```tsx
// 明示的な型指定
useQuery<Task[], Error>({
  queryKey: ["tasks"],
  queryFn: fetchTasks,
});
```

## よくある質問（FAQ）

**Q. TanStack Query と Redux / Zustand は競合しますか？**

競合しません。TanStack Query は「サーバー状態（APIから来るデータ）」を管理するのに対し、Redux / Zustand は「クライアント状態（UIの状態やユーザー入力）」を管理します。両者を組み合わせることが推奨されます。たとえば、カートの中身は Zustand、商品一覧は TanStack Query という使い分けが典型的です。

**Q. React Native の AppState 変化時にも自動再フェッチできますか？**

はい、`@tanstack/react-query` はブラウザの `window.focus` イベントを監視しますが、React Native では動作が異なります。`focusManager.setEventListener` を使って AppState の変化に対応させる設定が必要です。

```tsx
// lib/queryFocusManager.ts
import { focusManager } from "@tanstack/react-query";
import { AppState, AppStateStatus } from "react-native";

export function setupFocusManager() {
  return AppState.addEventListener("change", (state: AppStateStatus) => {
    focusManager.setFocused(state === "active");
  });
}
```

これを `app/_layout.tsx` の `useEffect` 内で呼び出すことで、ホーム画面から戻ったときに自動再フェッチが動作します。

**Q. オフライン時のデータ取得はどうなりますか？**

v5 では `networkMode` オプションでオフライン時の動作を制御できます。`networkMode: "offlineFirst"` に設定すると、オフライン時もキャッシュからデータを返します。ローカルストレージとの連携については、[Rork × AsyncStorage / MMKV / SQLite ローカルデータ管理ガイド](/articles/rork-dev/rork-local-storage-asyncstorage-mmkv-sqlite-guide) もご参照ください。

## まとめ

本記事では、Rork × TanStack Query v5 を使ったデータフェッチ・キャッシュ管理の全体像を解説しました。

- `useQuery` でシンプルなデータ取得とキャッシュ管理
- `useMutation` + `invalidateQueries` でデータの更新と再フェッチの連動
- `staleTime` / `gcTime` でキャッシュ戦略をチューニング
- Supabase との組み合わせでバックエンドとのやり取りをスマートに
