# Rork Maxアプリのクラッシュ解析・監視完全ガイド

本番環境では、開発中に見つけられないエラーが必ず発生します。低速ネットワークでのタイムアウト、想定外のエッジケース—こうしたエラーはサイレントに消えていき、ユーザーはアプリを削除するか不満を抱えたままになります。

本ガイドでは、**Sentry**と**Firebase Crashlytics**をRork Maxアプリに統合し、本当に起きているエラーを捕捉し、ユーザー影響を追跡し、問題を早期に察知するアラート設定まで、実装例を交えて解説します。

## クラッシュ監視が必須な理由

すべてのクラッシュは機会喪失です：アプリ削除、低評価、エンゲージメント低下。監視がなければ、あなたは何も見えていません。

**監視なしで見えないもの：**
- **クラッシュ頻度**：ユーザー全体の何％が影響を受けているのか？
- **ユーザー影響**：どのユーザーか？どのデバイス・OSバージョンか？
- **エラー傾向**：最新リリース後、クラッシュが増加していないか？
- **根本原因**：クラッシュ時刻でのスタックトレース、ネットワーク状態、デバイスメモリ
- **ブレッドクラム**：クラッシュに至るまでのユーザーアクション
- **パフォーマンス低下**：クラッシュしないもののユーザーを困らせる遅さ

クラッシュ監視ツールは、こうした情報を自動的に捕捉・集約し、サイレントな失敗を実行可能なデータに変えます。

## Sentry vs Firebase Crashlytics

| 機能 | Sentry | Firebase Crashlytics |
|------|--------|----------------------|
| **クラッシュ報告** | ✅ 優秀 | ✅ 優秀 |
| **パフォーマンス監視** | ✅ ビルトイン | ⚠️ 別製品（Performance） |
| **エラーブレッドクラム** | ✅ 詳細 | ⚠️ 基本的 |
| **ユーザーコンテキスト** | ✅ リッチメタデータ対応 | ✅ 良好 |
| **ソースマップ** | ✅ 自動アップロード | ✅ 自動アップロード |
| **無料枠** | ✅ 月5Kイベント | ✅ 無制限 |
| **料金** | 月額$29〜（従量課金） | 無料 + 分析機能は有料 |
| **セットアップ難度** | 低 | 低 |
| **React Native対応** | ✅ 一級対応 | ✅ 一級対応 |

**Sentryを選ぶべき場合：** 高度なエラー集約、パフォーマンス監視、詳細ブレッドクラムが必要—特に可視性を重視するチーム向け。

**Firebase Crashlyticsを選ぶべき場合：** 既にFirebaseを使用中、Googleエコシステムを好む、無料ソリューションで十分なチーム向け。

## Sentryの統合手順

### ステップ1：インストール

Expoを使用してSentryを追加：

```bash
npx expo install @sentry/react-native
```

<Callout type="info">
SentryのReact Native SDKはExpoと完全互換です。EAS Buildを使用している場合、ビルド完了後、Sentryが自動的にソースマップをアップロードします。
</Callout>

### ステップ2：アプリケーション起動時に初期化

アプリのエントリーポイントで、他のコード実行前にSentryを初期化：

```javascript
import * as Sentry from "@sentry/react-native";
import { NavigationContainer } from "@react-navigation/native";
import { createNativeStackNavigator } from "@react-navigation/native-stack";

// 最初にSentryを初期化
Sentry.init({
  dsn: "https://your-key@sentry.io/your-project-id",
  environment: process.env.NODE_ENV || "development",
  // app.jsonからリリースバージョンを取得
  release: "1.0.0",
  // 本番環境では低めに、開発環境では100%に
  tracesSampleRate: process.env.NODE_ENV === "production" ? 0.1 : 1.0,
  // HTTPクライアント相互作用を記録
  integrations: [
    new Sentry.ReactNativeTracing(),
  ],
});

const Stack = createNativeStackNavigator();

function App() {
  return (
    <NavigationContainer>
      <Stack.Navigator>
        {/* スクリーン定義 */}
      </Stack.Navigator>
    </NavigationContainer>
  );
}

// Sentryでラップ
export default Sentry.wrap(App);
```

### ステップ3：例外を手動で捕捉

Sentryが未捕捉例外を自動的に捕捉する一方、多くの場合、明示的にエラーを捕捉したいもの：

```javascript
import * as Sentry from "@sentry/react-native";

async function loadUserProfile(userId) {
  try {
    const response = await fetch(`/api/users/${userId}`);
    const data = await response.json();
    return data;
  } catch (error) {
    // コンテキスト付きでエラーを捕捉
    Sentry.captureException(error, {
      tags: {
        user_action: "load_profile",
      },
      extra: {
        userId,
        timestamp: new Date().toISOString(),
      },
    });
    throw error;
  }
}
```

### ステップ4：メッセージを送信

例外ではなく、重要なイベントを記録：

```javascript
import * as Sentry from "@sentry/react-native";

function processPayment(orderId, amount) {
  if (amount < 0) {
    Sentry.captureMessage("マイナス金額の支払いが試みられました", {
      level: "warning",
      extra: { orderId, amount },
    });
    return false;
  }
  // 支払い処理...
}
```

### ステップ5：Reactエラーバウンダリ

クラッシュの可能性があるコンポーネントをラップ：

```javascript
import * as Sentry from "@sentry/react-native";
import { Component } from "react";
import { View, Text } from "react-native";

class ErrorBoundary extends Component {
  state = { hasError: false };

  static getDerivedStateFromError(error) {
    return { hasError: true };
  }

  componentDidCatch(error, errorInfo) {
    // Sentryにエラーを送信
    Sentry.captureException(error, {
      contexts: {
        react: {
          componentStack: errorInfo.componentStack,
        },
      },
    });
  }

  render() {
    if (this.state.hasError) {
      return (
        <View style={{ flex: 1, justifyContent: "center", alignItems: "center" }}>
          <Text>問題が発生しました。対応中です。</Text>
        </View>
      );
    }

    return this.props.children;
  }
}

export default Sentry.withErrorBoundary(ErrorBoundary, {
  fallback: <Text>コンポーネント読込エラー</Text>,
});
```

## Sentry：ユーザーコンテキスト・プライバシー

どのユーザーがエラーを経験したかを追跡することは優先順位付けに必須：

```javascript
import * as Sentry from "@sentry/react-native";

function setUserContext(user) {
  Sentry.setUser({
    id: user.id,
    email: user.email,
    username: user.username,
    // カスタムフィールド—個人識別情報（PII）送信に注意
  });
}

function clearUserContext() {
  Sentry.setUser(null);
}
```

<Callout type="warning">
**プライバシー・GDPR対応**：パスワード、クレジットカード、社会保障番号などの機密情報をSentryに送らないこと。可能な限り匿名化されたユーザーIDを使用してください。プライバシーポリシーを確認し、ユーザーが監視に同意しているか確保してください。
</Callout>

デバッグ用の追加コンテキストを設定：

```javascript
Sentry.setContext("device", {
  model: "iPhone 13",
  os_version: "15.2",
});

Sentry.setTag("subscription_level", "premium");
```

## Sentry：パフォーマンス監視

遅いスクリーンと操作を追跡：

```javascript
import * as Sentry from "@sentry/react-native";

async function fetchListingDetails(listingId) {
  // トランザクションを開始
  const transaction = Sentry.startTransaction({
    name: "fetch_listing_details",
    op: "http",
  });

  const span = transaction.startChild({
    op: "fetch",
    description: "GET /listings/:id",
  });

  try {
    const response = await fetch(`/api/listings/${listingId}`);
    const data = await response.json();

    span.finish();
    transaction.finish();
    return data;
  } catch (error) {
    span.status = "error";
    span.finish();
    transaction.finish();
    Sentry.captureException(error);
    throw error;
  }
}
```

## Firebase Crashlyticsの統合

### ステップ1：依存関係をインストール

```bash
npx expo install @react-native-firebase/app @react-native-firebase/crashlytics
```

`app.json`でFirebaseを設定（またはAndroid用に`google-services.json`）：

```json
{
  "plugins": [
    [
      "@react-native-firebase/app",
      {
        "analytics": false,
        "database": false,
        "messaging": false,
        "storage": false,
        "crashlytics": true
      }
    ]
  ]
}
```

### ステップ2：Crashlyticsを初期化

```javascript
import crashlytics from "@react-native-firebase/crashlytics";

async function initializeCrashlytics() {
  // クラッシュをユーザーIDにリンク
  await crashlytics().setUserId("user_12345");

  // 収集を有効化（デバッグビルドではデフォルト無効）
  if (!__DEV__) {
    await crashlytics().setCrashlyticsCollectionEnabled(true);
  }
}

// アプリ起動時に呼び出し
initializeCrashlytics();
```

### ステップ3：エラーを記録

```javascript
import crashlytics from "@react-native-firebase/crashlytics";

try {
  riskyOperation();
} catch (error) {
  // 非致命的エラーを記録
  crashlytics().recordError(error);
}
```

例外なしでエラーを記録：

```javascript
crashlytics().log("支払いゲートウェイタイムアウト");
```

### ステップ4：カスタム属性・ブレッドクラム

クラッシュレポートにコンテキストを追加：

```javascript
import crashlytics from "@react-native-firebase/crashlytics";

function startCheckout(orderId, total) {
  crashlytics().setAttribute("current_order_id", orderId);
  crashlytics().setAttribute("checkout_total", String(total));

  // ブレッドクラム
  crashlytics().log(`ユーザーが注文 ${orderId} のチェックアウトを開始`);
}
```

## 致命的・非致命的エラー：戦略

エラーの分類方法を決定：

| エラータイプ | Sentry | Crashlytics | 例 |
|-----------|--------|-------------|-----|
| **致命的クラッシュ** | `captureException()` | 自動捕捉 | アプリ凍結、強制終了 |
| **非致命的例外** | `captureException()` | `recordError()` | キャッチして処理されたネットワークタイムアウト |
| **ユーザーアクション警告** | `captureMessage()` | `log()` | フォーム送信失敗 |
| **パフォーマンスアラート** | トランザクション（遅い場合） | n/a | スクリーン読込に5秒以上 |

```javascript
// 非致命的ネットワークエラー—アプリは継続
async function fetchData() {
  try {
    return await fetch("/api/data");
  } catch (error) {
    Sentry.captureException(error, { level: "warning" });
    return null; // アプリが適切に処理
  }
}

// 致命的エラー—アプリが続行できない
function validateAuth(token) {
  if (!token) {
    const error = new Error("認証トークンなし");
    Sentry.captureException(error, { level: "fatal" });
    throw error; // ここでアプリ停止
  }
}
```

## アラート・通知設定

### Sentryのアラート設定

1. **sentry.io にログイン**し、プロジェクトへ移動
2. **Alerts → Create Alert Rule**
3. **条件を選択：**
   - エラー率 > 過去1時間で5%
   - 新しい問題が発生
   - エラー数 > 100

4. **アクションを設定：**
   - Slack送信：`/slack https://hooks.slack.com/...`
   - メール通知
   - 重大問題はPagerDuty連携

```
条件: 「新しい問題が作成される」
アクション: Slackチャネル #app-errors、メール team@company.com
```

### Firebase Crashlyticsのアラート設定

1. **Firebase Console** → Crashlytics
2. **Setup → Alerts**
3. **新規アラート：**
   - クラッシュフリーユーザー率が99%未満に低下
   - 特定バージョンでのクラッシュスパイク

<Callout type="info">
両ツールともノイズの多いエラーをミュート可能です。「解決済み」にマークすれば、以後同様のエラーを分離追跡できます。
</Callout>

## 選択ガイド：チーム向けSentry vs Crashlytics

**Sentryを使用すべき場合：**
- **高度なエラー集約**が必要（例：「古いAndroidデバイスでのクラッシュパターン」を検出）
- **パフォーマンス監視**（遅いスクリーン、HTTP要求タイミング）が必須
- **詳細ブレッドクラム**が必要（クラッシュ前の最後の10アクション）
- Firebaseを使用していない、スタンドアロン解決策を好む
- **カスタム課題ワークフロー**と統合が必要

**Firebase Crashlyticsを使用すべき場合：**
- 既に**Google・Firebaseエコシステム**を採用中
- **無料ソリューション**（従量課金なし）を求める
- クラッシュが主に**ネイティブコード**（NDKエラー）からの場合
- シンプルでストレートフォワードなクラッシュ報告で十分
- 既にFirebase Analyticsを使用、統一レポーティングを希望

**両方を使用すべき場合：**
- ミッションクリティカルアプリで複雑さが許容される
- **冗長性**を確保したい（1サービスがダウンしても別で継続）
- Sentryがアプリクラッシュを、Crashlyticsがネイティブフレームワーク問題をキャッチ

## まとめ

クラッシュ監視は、見えないエラーを実行可能なデータに変えます。Sentryは可視性とパフォーマンスに優れ、Firebase Crashlyticsは無料でGoogle統合を提供します。

Rork Maxアプリの場合：

1. **まずはひとつから**：高度な機能はSentry、シンプルさはCrashlytics
2. **段階的に計測**：自動クラッシュ捕捉から始め、ユーザーコンテキスト、パフォーマンス監視へ
3. **早期にアラート設定**：規模拡大前にスパイクを察知
4. **定期的にレビュー**：週1回のダッシュボード確認がトレンド早期発見に
5. **プライバシー第一**：ユーザー同意を得て、機密情報を避ける

ユーザーはクラッシュ監視に気付きませんが、3週間前に報告したバグを修正したときには感謝するはずです。
