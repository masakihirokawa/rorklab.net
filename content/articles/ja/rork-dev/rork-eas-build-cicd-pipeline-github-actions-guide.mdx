Rork でアプリを作り始めると、最初のうちは「できた！App Store に出してみよう」という熱量で手動ビルドを繰り返せます。でも3本目、5本目のアプリを並行管理するようになったとき、その手動作業が一気に重荷になってきます。

私自身、ある週末に「あれ、今週ビルドに費やした時間の合計が6時間超えてる…」と気づいた瞬間がありました。コード書く時間より、Xcode を開いてアーカイブしてアップロードして待って……を繰り返す時間の方が長い。これは明らかにおかしい、と。

そこから EAS Build + GitHub Actions でCI/CDパイプラインを整えたのですが、最初のセットアップさえ乗り越えれば、あとは `git push` だけでTestFlightに届き、マージボタンを押せばGoogle Play の内部テストに配信される世界が待っています。

この記事では、Rork で生成したアプリを前提に、**開発・ステージング・本番の3環境を持つ完全自動化CI/CDパイプライン**を0から構築する手順を、実際に詰まったポイントも含めて丁寧に解説します。

## なぜ「手動ビルド → 自動化」が今すぐ必要なのか

自動化の恩恵を話す前に、手動ビルド運用のコストを整理してみます。

Rork はコード生成が得意なので、「アプリを作ること」自体のコストはどんどん下がっています。問題は、**作った後のデリバリーコスト**が昔のまま残っていることです。

手動運用でかかるコスト:

- **毎回のリリース作業**: Xcode でアーカイブ → App Store Connect へアップロード → 処理完了まで待機（15〜30分）
- **証明書・プロビジョニングの管理**: 更新タイミングを見落とすと突然ビルドが通らなくなる
- **環境変数の混入リスク**: 本番用のAPIキーをテスト環境に入れてしまう、その逆も然り
- **複数アプリの並行管理**: アプリが増えるほど、この作業が乗算で増える

CI/CD を整えると、これらが全て解消されます。コードを書いてプッシュするだけで済みます。証明書は自動更新。環境変数は secrets に閉じ込めて、開発者が直接触ることはありません。

## 全体アーキテクチャを把握する

まず何を作るのかを明確にしておきます。構築するパイプラインの全体像は以下の通りです:

- **main ブランチへのマージ** → GitHub Actions トリガー → EAS Build（production） → TestFlight（iOS）/ Google Play 内部テスト（Android）→ Slack通知
- **develop ブランチへのマージ** → GitHub Actions トリガー → EAS Build（staging） → TestFlightステージンググループ / Play Console 内部テスト
- **feature/* ブランチ → develop へのPR** → GitHub Actions トリガー → EAS Build（development） → PRコメントにビルド結果を投稿

3つのブランチ戦略で、3つの環境を完全に分離します。

## Step 1: 前提条件と初期セットアップ

### 必要なアカウント・ツール

- Expo アカウント（EAS Build 用）
- GitHub リポジトリ（Rork のエクスポートコードを格納）
- Apple Developer Program（iOS 用、年 $99）
- Google Play Console（Android 用、初回 $25）
- Slack ワークスペース（通知先、任意）

### EAS CLI のインストールと認証

```bash
# EAS CLI をグローバルインストール
npm install -g eas-cli

# Expo アカウントでログイン
eas login

# ログイン確認
eas whoami
# → your-email@example.com と表示されれば成功
```

### Rork プロジェクトのEAS設定

Rork からエクスポートしたコードのルートディレクトリで初期設定を行います:

```bash
# EAS の初期設定（初回のみ・プロジェクトIDが割り当てられる）
eas build:configure
```

これにより `eas.json` が生成されます。デフォルトのままでは3環境に対応できないので、次のステップで詳細設定を行います。

## Step 2: eas.json で3環境を設計する（最重要）

ここが最もつまずきやすく、かつ最も重要な箇所です。**`eas.json` の設計を間違えると、後から直すのが非常に大変**になります。以下のテンプレートをベースに各プロジェクトに合わせて調整してください。

```json
{
  "cli": {
    "version": ">= 10.0.0"
  },
  "build": {
    "development": {
      "developmentClient": true,
      "distribution": "internal",
      "ios": {
        "simulator": false,
        "resourceClass": "m-medium"
      },
      "android": {
        "buildType": "apk",
        "resourceClass": "medium"
      },
      "env": {
        "APP_ENV": "development",
        "API_URL": "https://dev-api.your-app.com"
      }
    },
    "staging": {
      "distribution": "internal",
      "ios": {
        "resourceClass": "m-medium"
      },
      "android": {
        "buildType": "apk",
        "resourceClass": "medium"
      },
      "env": {
        "APP_ENV": "staging",
        "API_URL": "https://staging-api.your-app.com"
      }
    },
    "production": {
      "autoIncrement": true,
      "ios": {
        "resourceClass": "m-large"
      },
      "android": {
        "buildType": "app-bundle",
        "resourceClass": "large"
      },
      "env": {
        "APP_ENV": "production",
        "API_URL": "https://api.your-app.com"
      }
    }
  },
  "submit": {
    "production": {
      "ios": {
        "appleId": "your-apple-id@example.com",
        "ascAppId": "1234567890",
        "appleTeamId": "XXXXXXXXXX"
      },
      "android": {
        "serviceAccountKeyPath": "./google-services/play-store-key.json",
        "track": "internal"
      }
    }
  }
}
```

### 重要: APIキー類は env に書かない

上記の `eas.json` で `API_URL` を環境ごとに書いていますが、**APIキーやシークレット値は絶対に `eas.json` に書かないでください。**

`eas.json` は Git に含まれます。シークレット値は EAS の Secrets 管理機能か GitHub Secrets で管理します:

```bash
# EAS Secrets にシークレットを登録（プロジェクトスコープ）
eas secret:create \
  --scope project \
  --name STRIPE_SECRET_KEY \
  --value "sk_prod_xxxxx"

# 登録済み secrets の確認
eas secret:list
```

## Step 3: app.config.js で環境を動的に切り替える

Rork が生成する `app.json` は静的ですが、環境ごとにアプリ名・バンドルIDを変えたい場合は `app.config.js` に変換します。`app.json` を削除（または名前変更）してから、同じディレクトリに `app.config.js` を作成します:

```javascript
// app.config.js
const IS_DEV = process.env.APP_ENV === "development";
const IS_STAGING = process.env.APP_ENV === "staging";

export default {
  name: IS_DEV
    ? "MyApp (Dev)"
    : IS_STAGING
    ? "MyApp (Staging)"
    : "MyApp",
  slug: "my-app",
  version: "1.0.0",
  orientation: "portrait",
  // 環境ごとに別アイコンを使う（混乱防止）
  icon: IS_DEV
    ? "./assets/icon-dev.png"
    : "./assets/icon.png",
  ios: {
    // バンドルIDを環境ごとに分ける（デバイスに3つ共存できる）
    bundleIdentifier: IS_DEV
      ? "com.yourcompany.myapp.dev"
      : IS_STAGING
      ? "com.yourcompany.myapp.staging"
      : "com.yourcompany.myapp",
    buildNumber: "1",
    supportsTablet: false,
  },
  android: {
    package: IS_DEV
      ? "com.yourcompany.myapp.dev"
      : IS_STAGING
      ? "com.yourcompany.myapp.staging"
      : "com.yourcompany.myapp",
    versionCode: 1,
  },
  // アプリ内で環境変数を参照するための設定
  extra: {
    apiUrl: process.env.API_URL,
    appEnv: process.env.APP_ENV,
    eas: {
      projectId: "your-eas-project-id",
    },
  },
  plugins: ["expo-router"],
};
```

これにより、開発・ステージング・本番で**アイコンもアプリ名もバンドルIDも別のアプリ**として管理できます。デバイスに3つを並べても混乱しません。

## Step 4: GitHub Actions ワークフローを設定する

ここからが自動化の核心部分です。`.github/workflows/` ディレクトリを作成し、ワークフローファイルを配置します。

### 本番ビルド + 自動提出ワークフロー

`main` ブランチへのプッシュで本番ビルドを起動し、TestFlight と Google Play 内部テストへ自動提出します:

```yaml
# .github/workflows/production-build.yml
name: Production Build & Submit

on:
  push:
    branches:
      - main

jobs:
  build-and-submit-ios:
    name: Build iOS (Production)
    runs-on: ubuntu-latest
    steps:
      - name: Checkout repository
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: "20"
          cache: "npm"

      - name: Install dependencies
        run: npm ci

      - name: Setup EAS
        uses: expo/expo-github-action@v8
        with:
          eas-version: latest
          token: ${{ secrets.EXPO_TOKEN }}

      - name: Build iOS for production
        run: |
          eas build \
            --platform ios \
            --profile production \
            --non-interactive
        env:
          STRIPE_SECRET_KEY: ${{ secrets.STRIPE_SECRET_KEY }}
          SUPABASE_SERVICE_KEY: ${{ secrets.SUPABASE_SERVICE_KEY }}

      - name: Submit to App Store (TestFlight)
        run: |
          eas submit \
            --platform ios \
            --profile production \
            --latest \
            --non-interactive
        env:
          EXPO_APPLE_APP_SPECIFIC_PASSWORD: ${{ secrets.APPLE_APP_SPECIFIC_PASSWORD }}

  build-and-submit-android:
    name: Build Android (Production)
    runs-on: ubuntu-latest
    steps:
      - name: Checkout repository
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: "20"
          cache: "npm"

      - name: Install dependencies
        run: npm ci

      - name: Setup Google Play key
        run: |
          mkdir -p ./google-services
          echo '${{ secrets.GOOGLE_PLAY_SERVICE_ACCOUNT_JSON }}' \
            > ./google-services/play-store-key.json

      - name: Setup EAS
        uses: expo/expo-github-action@v8
        with:
          eas-version: latest
          token: ${{ secrets.EXPO_TOKEN }}

      - name: Build Android for production
        run: |
          eas build \
            --platform android \
            --profile production \
            --non-interactive
        env:
          STRIPE_SECRET_KEY: ${{ secrets.STRIPE_SECRET_KEY }}
          SUPABASE_SERVICE_KEY: ${{ secrets.SUPABASE_SERVICE_KEY }}

      - name: Submit to Google Play
        run: |
          eas submit \
            --platform android \
            --profile production \
            --latest \
            --non-interactive

  notify-slack:
    name: Notify Slack
    runs-on: ubuntu-latest
    needs: [build-and-submit-ios, build-and-submit-android]
    if: always()
    steps:
      - name: Send Slack notification
        uses: slackapi/slack-github-action@v1.26.0
        with:
          payload: |
            {
              "text": "🚀 Production build完了\niOS: ${{ needs.build-and-submit-ios.result }}\nAndroid: ${{ needs.build-and-submit-android.result }}\nCommit: ${{ github.sha }}\nAuthor: ${{ github.actor }}"
            }
        env:
          SLACK_WEBHOOK_URL: ${{ secrets.SLACK_WEBHOOK_URL }}
```

### ステージングビルドワークフロー

`develop` ブランチへのマージで自動的にステージング版をビルドします:

```yaml
# .github/workflows/staging-build.yml
name: Staging Build

on:
  push:
    branches:
      - develop

jobs:
  build-staging:
    name: Build Staging (iOS + Android)
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: "20"
          cache: "npm"

      - run: npm ci

      - uses: expo/expo-github-action@v8
        with:
          eas-version: latest
          token: ${{ secrets.EXPO_TOKEN }}

      - name: Build staging for both platforms
        run: |
          eas build \
            --platform all \
            --profile staging \
            --non-interactive
        env:
          STRIPE_SECRET_KEY: ${{ secrets.STRIPE_SECRET_KEY_STAGING }}
          SUPABASE_SERVICE_KEY: ${{ secrets.SUPABASE_SERVICE_KEY_STAGING }}
```

## Step 5: GitHub Secrets の設定

GitHub リポジトリの **Settings → Secrets and variables → Actions** に以下を登録します。

Expo 関連:
- `EXPO_TOKEN`: Expo ダッシュボードの Account Settings → Access tokens で生成（**有効期限なし**を選ぶこと）

Apple 関連:
- `APPLE_APP_SPECIFIC_PASSWORD`: [appleid.apple.com](https://appleid.apple.com) → セキュリティ → アプリ用パスワードで生成

Google Play 関連:
- `GOOGLE_PLAY_SERVICE_ACCOUNT_JSON`: Google Cloud Console で生成したサービスアカウントのJSONキー内容をそのまま貼り付け

アプリのシークレット:
- `STRIPE_SECRET_KEY`: 本番用 Stripe シークレットキー
- `STRIPE_SECRET_KEY_STAGING`: ステージング用 Stripe シークレットキー
- `SUPABASE_SERVICE_KEY`: 本番用 Supabase サービスロールキー
- `SUPABASE_SERVICE_KEY_STAGING`: ステージング用 Supabase サービスロールキー

Slack 関連:
- `SLACK_WEBHOOK_URL`: Slack の Incoming Webhook URL

### 注意: Secrets の命名規則

GitHub Secrets は大文字・アンダースコアのみ使用可能です。ハイフン（`-`）は使えません。`STRIPE-SECRET-KEY` と書くとワークフロー実行時に値が空になってビルドが失敗します。エラーメッセージが「Unauthorized」などの認証エラーになるため、原因特定に時間がかかります。

## Step 6: iOS 証明書の自動管理

EAS Build の最大の利点のひとつが、**証明書管理を完全に代行してくれること**です。Fastlane Match のような複雑なセットアップは不要で、`eas.json` に一行追加するだけで済みます:

```json
{
  "build": {
    "production": {
      "ios": {
        "credentialsSource": "remote"
      }
    }
  }
}
```

`credentialsSource: "remote"` を設定すると、EAS が Apple Developer Portal との連携を自動で処理します。証明書の期限管理も EAS が行うため、突然「証明書が切れてビルドできない」という事故がなくなります。

初回だけは Apple の2ファクタ認証が必要なため、手動でセットアップします:

```bash
# 証明書・プロビジョニングプロファイルの設定（初回のみ）
eas credentials

# Interactive モードで以下を選択:
# > iOS
# > production
# > Set up credentials automatically
```

一度設定してしまえば、以降は GitHub Actions から完全自動で実行されます。

## Step 7: Google Play API 権限の設定（最も複雑な箇所）

Android の自動提出は、Google Play Console の API 設定が最も複雑です。実際に詰まった手順を順を追って説明します。

### サービスアカウントの作成手順

1. [Google Cloud Console](https://console.cloud.google.com/) にアクセスし、Google Play Console と同じ Google アカウントでログイン
2. 新しいプロジェクトを作成（または既存のものを使用）
3. 「APIとサービス」→「ライブラリ」→「Google Play Android Developer API」を有効化
4. 「認証情報」→「サービスアカウントを作成」
5. サービスアカウント名を入力（例: `play-store-ci`）
6. ロール: 「オーナー」（後で制限しても可）
7. 「キーを追加」→「新しいキーを作成」→ **JSON** 形式でダウンロード

### Google Play Console でのサービスアカウント招待

1. Google Play Console → 「ユーザーと権限」→「ユーザーを招待」
2. 作成したサービスアカウントのメールアドレスを入力（例: `play-store-ci@your-project.iam.gserviceaccount.com`）
3. 権限レベル: **管理者**（ビルドの提出には高い権限が必要）
4. 対象アプリを選択して保存

### 詰まりポイント: 権限の反映に最大24時間かかる

サービスアカウントを招待してもすぐには機能しないことがあります。EAS submit が失敗した場合、エラーの種別で原因を切り分けます:

```bash
# ローカルでテスト提出を試みて原因を確認
eas submit --platform android --profile production --latest

# 「Permission denied」「403」→ Google Play Console の権限の問題（24時間待つ）
# 「invalid credentials」「401」→ JSON キーの問題（Google Cloud で再生成）
# 「package not found」→ Play Console にアプリが未作成（手動で初回登録が必要）
```

最初のAPK/AABは手動でアップロードする必要があります。自動提出は2回目以降からです。

## Step 8: OTA アップデートで小さな修正を即時配信する

UI の文言修正やバグフィックス程度の変更には、毎回ストアに提出しなくてもOTAアップデート（Over The Air）で即時配信できます。ストアの審査なしで数分以内に全ユーザーへ届きます。

```yaml
# .github/workflows/ota-update.yml
name: OTA Update

on:
  push:
    branches:
      - main
    paths:
      - "app/**"
      - "components/**"
      - "hooks/**"
      - "lib/**"
      - "constants/**"

jobs:
  update:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: "20"
          cache: "npm"

      - run: npm ci

      - uses: expo/expo-github-action@v8
        with:
          eas-version: latest
          token: ${{ secrets.EXPO_TOKEN }}

      - name: Publish OTA update to production channel
        run: |
          eas update \
            --channel production \
            --message "Update: ${{ github.event.head_commit.message }}" \
            --auto
```

`paths:` でアプリコードが変更されたときだけOTAアップデートを発行する設定にしています。`eas.json` や `package.json` の変更のみではトリガーしません。

**OTAが使えないケース**: ネイティブモジュールの追加・変更（新しい expo-modules の追加など）はOTAでは配信できず、フルビルドが必要です。`eas.json`、`app.config.js`、`ios/`、`android/` ディレクトリへの変更は自動的にフルビルドに切り替わります。

## Step 9: 環境ごとのデバッグ設定と Sentry 連携

本番では非表示にしたいデバッグ情報を、開発・ステージング環境では有効にする設定です。Sentry のサンプリングレートも環境で変えることで、開発中はすべてのエラーをキャプチャし、本番では負荷を抑えます。

```typescript
// utils/config.ts
import Constants from "expo-constants";

const APP_ENV =
  (Constants.expoConfig?.extra?.appEnv as string) ?? "production";

export const Config = {
  appEnv: APP_ENV,
  apiUrl: Constants.expoConfig?.extra?.apiUrl as string,
  isProduction: APP_ENV === "production",
  isDevelopment: APP_ENV === "development",
  isStaging: APP_ENV === "staging",
} as const;

export const debugLog = (...args: unknown[]) => {
  if (\!Config.isProduction) {
    console.log(`[DEBUG][${Config.appEnv}]`, ...args);
  }
};
```

```typescript
// app/_layout.tsx（Expo Router）
import * as Sentry from "@sentry/react-native";
import { Config } from "@/utils/config";

Sentry.init({
  dsn: process.env.EXPO_PUBLIC_SENTRY_DSN,
  // 本番は20%、ステージングは50%、開発は100%
  tracesSampleRate: Config.isProduction
    ? 0.2
    : Config.isStaging
    ? 0.5
    : 1.0,
  environment: Config.appEnv,
  enabled: \!Config.isDevelopment, // 開発中は無効（ノイズ削減）
});
```

## よくある詰まりポイントと対処法

実際にセットアップした際に詰まった箇所を、解決策とあわせてまとめます。

**① EAS Token の期限切れ**

GitHub Secrets に登録した `EXPO_TOKEN` は有効期限があります（デフォルト30日）。期限切れになると Actions が無言で失敗します。Expo ダッシュボードで **No expiry** オプションで再生成し、Secrets を更新してください。定期的に確認する習慣をつけるか、カレンダーにリマインダーを設定しておくと安全です。

**② `--no-wait` と `eas submit` の競合**

ビルドをキューに入れた後すぐ `eas submit` を実行すると、ビルド完了前に実行されて失敗します。`--no-wait` を外してビルド完了を待つか、別のジョブに分けて `needs:` で依存関係を設定してください。

**③ Android の `play-store-key.json` のパス問題**

`eas.json` に書いたパスと、GitHub Actions の `echo` で書き出したパスがズレると権限エラーになります。ワークフロー内でデバッグするときは `ls -la ./google-services/` を追加して実際のパスを確認してください。

**④ Slack 通知が飛ばない**

`needs:` に指定したジョブが `failure` で終了すると、後続ジョブがデフォルトでスキップされます。通知ジョブには `if: always()` を必ず付けてください。

**⑤ Podfile.lock の不整合でiOSビルドが失敗する**

```
error: CocoaPods lockfile was modified manually
```

`npm install` や Expo のバージョンアップ後に発生しやすいです。ローカルで `cd ios && pod install` を実行し、生成された `Podfile.lock` をコミットしてください。

**⑥ Gradle バージョン不一致でAndroidビルドが失敗する**

```
Unsupported class file major version 65
```

`android/build.gradle` に Java バージョンを明示することで解消します:

```groovy
compileOptions {
  sourceCompatibility JavaVersion.VERSION_17
  targetCompatibility JavaVersion.VERSION_17
}
kotlinOptions {
  jvmTarget = "17"
}
```

## まとめ: まず staging から自動化してみる

EAS Build + GitHub Actions によるCI/CDパイプラインは、セットアップに最初の半日〜1日かかりますが、それ以降のリリース作業は `git push` だけになります。

アプリが1本のうちはそこまで恩恵を感じにくいかもしれませんが、2本、3本と増えていくと「なぜ早くやっておかなかったのか」と思う類の投資です。

まずは `staging` ブランチだけ自動化して動作確認してから、`main` ブランチへの本番自動提出を有効にするのがリスクの少ない順序です。本番への自動提出は、一度手動で提出フローを理解してから有効にする方が安心できます。

まず Rork でアプリを作り、次のリリースを `git push` だけで完結させてみてください。
