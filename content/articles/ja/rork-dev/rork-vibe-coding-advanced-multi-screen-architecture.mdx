## はじめに

バイブコーディングの真価は、大規模アプリケーションのアーキテクチャ設計にこそあります。複数の画面、複雑な状態管理、API 連携を含むエンタープライズレベルのアプリを、自然言語で指定可能です。

このガイドでは、実務で使用される高度なパターンを解説します。対象は、Rork でのアプリ開発経験があり、中規模以上のプロジェクトを計画している方です。

## 大規模アプリのアーキテクチャ設計原則

### 1. 画面の論理的分割

バイブコーディングで複数画面を管理する際は、以下の原則に従います：

**画面分類の 3 層モデル:**

- **プレゼンテーション層**：UI のみ
- **ビジネスロジック層**：API 呼び出し、データ変換
- **データ層**：Firestore / Supabase との通信

バイブコーディングで複数画面を定義するときは、この分離を保つプロンプトを設計します：

eコマースアプリの例：

【画面1: 商品一覧】
- Firestore から商品データを取得（カテゴリフィルター付き）
- 各商品はカード表示（画像・名前・価格）

【画面2: 商品詳細】
- 商品の詳細情報（説明・レビュー・在庫）
- 「カートに追加」ボタン

【画面3: ショッピングカート】
- カート内の商品一覧（数量・小計表示）
- 数量の増減機能

【画面4: チェックアウト】
- 配送先住所入力フォーム
- 支払い方法選択（Stripe 統合）

このプロンプトから Rork が生成するアーキテクチャは以下のようになります：

```
src/
├── screens/
│   ├── ProductList.tsx
│   ├── ProductDetail.tsx
│   ├── ShoppingCart.tsx
│   └── Checkout.tsx
├── context/
│   ├── CartContext.tsx
│   └── ProductContext.tsx
├── services/
│   ├── firestoreService.ts
│   ├── stripeService.ts
│   └── authService.ts
└── navigation/
    └── RootNavigator.tsx
```

### 2. グローバル状態管理パターン

複雑な状態を扱う場合は、Context + useReducer パターンを推奨します。複数の画面で共有するデータを管理：

- ログイン状態（ユーザー情報）
- ショッピングカート（商品リスト）
- フィルター状態（選択したカテゴリ）

画面1で選択したカテゴリは、ページ遷移後も保持されるようにプロンプトで指定します。

## 複雑なデータフローの設計

### パターン 1: API キャッシング

商品データは Firestore から取得しますが、同じユーザーセッション中はメモリキャッシュに保持してください。キャッシュの有効期限は 5 分です。

### パターン 2: 楽観的更新（Optimistic Update）

ユーザーが「カートに追加」ボタンをタップしたとき：
1. 即座に UI に商品を追加表示（楽観的更新）
2. 同時に Firestore に非同期で保存
3. 失敗時はロールバック

## 実装のベストプラクティス

### 1. 型安全性の確保

TypeScript を活用して、実行時エラーを防ぎます。すべての API レスポンスに TypeScript インターフェースを定義してください。

### 2. エラーハンドリングの一元化

API エラーは全て統一したエラーハンドラーで処理してください。ユーザーにはわかりやすいメッセージを表示し、内部的には詳細なログを Firebase Analytics に送信してください。

### 3. パフォーマンス最適化

商品一覧画面で 1000 件以上の商品がある場合、仮想スクロール（VirtualList）を使用してください。

## よくある質問（FAQ）

### Q1: バイブコーディングで生成されたコードは、後から手動で修正できるか？

A: はい、完全にできます。Rork が生成したコードは標準的な React Native コードであり、任意の IDE で編集できます。

### Q2: 複数の開発者がバイブコーディングで同じプロジェクトを編集する場合、マージコンフリクトが起きないか？

A: 通常の Git ワークフローと同じようにコンフリクト管理ができます。複数人が同時に異なる画面をバイブコーディングする場合は、プロンプトを分割し、各画面を独立して定義することを推奨します。

### Q3: バイブコーディングの自動生成と、既存の手書きコンポーネントライブラリとの相性は？

A: 良好です。Rork では、自動生成画面と手書きコンポーネント（Design System）を混在できます。
