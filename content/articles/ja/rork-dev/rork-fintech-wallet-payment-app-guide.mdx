フィンテック（FinTech）アプリの開発は複雑に見えますが、Rork Maxを使えば、ウォレット・送金・経費精算といった機能を持つアプリを効率的に構築できます。この記事では、実際の開発フローとコンプライアンス上の注意点を合わせて解説します。

## フィンテックアプリの種類と選択

「フィンテックアプリ」といっても、扱う機能によって法的要件が大きく異なります。Rork Maxで構築できる主なユースケースを整理しましょう。

**コンプライアンス要件が比較的低いもの:**
- 経費精算・申請ワークフロー（社内ツール）
- 家計簿・支出記録アプリ
- ポイント・スタンプ管理（自社ポイント）
- 見積もり・請求書作成ツール

**法的規制の確認が必要なもの:**
- 実際の送金機能（資金移動業登録が必要）
- 仮想通貨の取扱い
- ローン・融資機能

この記事では、コンプライアンス要件を把握しながら、経費精算アプリとウォレット的なポイント管理アプリの構築を説明します。

## 経費精算アプリの設計

社内向けの経費精算アプリはフィンテックの入門として最適です。

**必要な機能:**
- 領収書の撮影・アップロード
- 経費カテゴリの選択（交通費・飲食費・備品など）
- 申請ワークフロー（申請→承認→精算）
- 承認者への通知
- 月次の精算レポート出力

**Supabaseスキーマ:**

```
expenses テーブル:
id              uuid
employee_id     uuid
amount          integer   (金額、円)
category        text      (transportation/food/supplies/other)
receipt_url     text      (Supabase Storageの画像URL)
description     text      (用途説明)
submitted_at    timestamp
status          text      (draft/submitted/approved/rejected/paid)
approver_id     uuid
approved_at     timestamp
notes           text      (承認者コメント)
```

## Rork Maxでの経費精算アプリ構築

Rork Maxに次のプロンプトを入力します。

```
社内向け経費精算アプリを作成してください。

機能:
1. 経費の新規申請画面（カメラで領収書撮影、金額・カテゴリ・用途入力）
2. 申請一覧画面（自分の申請履歴、ステータス別フィルター）
3. 承認者用画面（未承認の申請一覧、承認/差戻しボタン）
4. 申請が承認・差戻しされたらプッシュ通知

データ: Supabase
認証: Supabase Auth（メール認証）
ロール: employee（申請者）と approver（承認者）の2種類
```

承認者ロールの識別には、Supabaseのusersテーブルにroleカラムを追加し、RLSで承認者のみが他者の申請を閲覧できるようにします。

## ポイントウォレットアプリの構築

実際のお金を動かさない「ポイント」や「社内通貨」の管理アプリであれば、資金移動業の登録なしに構築できます。

**活用シーン:**
- 社内表彰・ポイント付与システム
- ショップのスタンプカードアプリ
- イベントの参加ポイント管理
- コミュニティのコイン経済圏

**スキーマ設計:**

```
wallets テーブル:
id              uuid
user_id         uuid
balance         integer   (ポイント残高)
updated_at      timestamp

transactions テーブル:
id              uuid
from_user_id    uuid      (送り手、nullならシステム付与)
to_user_id      uuid      (受け手)
amount          integer   (ポイント数)
type            text      (earn/transfer/spend)
description     text      (理由・説明)
created_at      timestamp
```

残高の整合性を保つために、トランザクション処理が重要です。SupabaseのPostgreSQL関数（RPC）を使って、送金と残高更新を原子的に実行する設計が理想的です。

## Rork Maxでのポイント送金実装

```
ポイントウォレットアプリを作成してください。

機能:
1. ウォレット画面（残高表示、取引履歴）
2. ポイント送金画面（送り先ユーザー検索、金額入力、確認）
3. QRコードで素早く送金（送り先のQRをスキャン）
4. 管理者によるポイント付与機能

残高不足時は送金できないようにしてください。
取引は全てtransactionsテーブルに記録してください。
```

## Stripe連携で実際の決済も追加する

ポイントのチャージに実際のカード決済を組み込む場合はStripeを使います。ただし、ポイントを現金に換金する機能は資金移動業に該当する可能性があるため、「ポイントは購入のみで、返金・換金は不可」という設計が一般的です。

```
ポイントチャージ機能を追加してください。
500円 → 500ポイント、1000円 → 1100ポイント（10%ボーナス）
Stripe Checkoutで決済後、walletsテーブルの残高を更新してください。
チャージ完了後はtransactionsテーブルに記録してください。
```

## 法的・セキュリティ上の注意点

フィンテックアプリ開発では、技術だけでなく法的要件への理解が欠かせません。

実際の法定通貨を「預かる」「送金する」機能は資金移動業法の適用対象となります。事業として提供する場合は、必ず法律の専門家や金融当局への確認をお勧めします。

また、セキュリティ面ではSupabaseのRLSを厳格に設定し、ユーザーが他者のウォレット情報にアクセスできないよう設計することが基本です。監査ログとして全トランザクションを記録することも重要です。

## まとめ

Rork Maxを使えば、経費精算やポイント管理といったフィンテック系の機能を持つアプリを効率的に構築できます。実際のお金を動かさない範囲であれば、法的なハードルも比較的低く、社内ツールやコミュニティアプリとして活用できます。

本格的な送金・決済サービスを展開する際は、必ず法的要件の確認と適切な資格・登録を行ったうえで進めましょう。
