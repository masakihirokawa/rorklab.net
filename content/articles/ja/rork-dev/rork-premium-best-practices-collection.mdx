### StoreKit 2 での IAP 検証 — サーバーレス検証パターン

**ベストプラクティス：AppSync や Serverless JWT で StoreKit 2 の Transaction.listen をサポートし、サーバー側では Product.SubscriptionInfo で自動更新の期限を一元管理する。**

StoreKit 2 では Transaction が JWS トークンとして署名され、サーバーが Apple から直接検証できます。ローカルでの Transaction.listen は UI 部分に、サーバー検証は Revenue に関わる部分に分けるのが定石です。

```typescript
// React Native + TypeScript での Transaction 検証パターン
import { useEffect, useState } from 'react';
import StoreKit from 'react-native-storekit2';

export function IAPManager() {
  const [transactions, setTransactions] = useState<string[]>([]);

  useEffect(() => {
    // Transaction listener は UI sync 用
    const unlistener = StoreKit.onTransaction(async (tx) => {
      // JWS トークンをサーバーに送信
      const response = await fetch('/api/verify-transaction', {
        method: 'POST',
        body: JSON.stringify({ transactionJWS: tx.jws }),
      });
      const verified = await response.json();
      if (verified.isValid) {
        setTransactions([...transactions, tx.id]);
      }
    });
    return () => unlistener();
  }, []);

  return <View>{/* UI */}</View>;
}
```

### RevenueCat での Offering / Package 設計

**ベストプラクティス：Offering/Package の階層設計（Trial → 月額 → 年額）は最初に決定し、A/B テストは Remote Config と併用する。**

RevenueCat では Apple/Google/Stripe の複数プラットフォームの価格を一元管理できます。重要なのは、アプリ側では RevenueCat API を叩くだけで、複雑なロジックはサーバー側に持つことです。

```typescript
// RevenueCat での Offering / Entitlement チェック
import Purchases from 'react-native-purchases';

async function checkSubscriptionStatus() {
  const offerings = await Purchases.getOfferings();
  const currentOffering = offerings?.current;

  if (!currentOffering) {
    console.log('No offerings available');
    return;
  }

  // premium パッケージを取得（A/B テストなら Remote Config で変更）
  const packages = currentOffering.availablePackages;
  const monthlyPkg = packages.find((p) => p.identifier === 'premium_monthly');

  // Entitlement チェック
  const customerInfo = await Purchases.getCustomerInfo();
  const isPremium = customerInfo.entitlements.active['premium'] !== undefined;

  return { monthlyPkg, isPremium };
}
```

### Stripe サブスクリプション管理 — Webhook 同期

**ベストプラクティス：Webhook で Payment Intent → Subscription → Invoice の状態遷移をすべて記録し、冪等性を確保する。**

Stripe では invoice.payment_succeeded / invoice.payment_failed が重要な Webhook です。これを元に DB を更新し、クライアント側は疎結合に保ちます。

```typescript
// Webhook ハンドラー（Node.js Express）
app.post('/webhooks/stripe', express.raw({ type: 'application/json' }), async (req, res) => {
  const sig = req.headers['stripe-signature'];
  let event;

  try {
    event = stripe.webhooks.constructEvent(req.body, sig, STRIPE_WEBHOOK_SECRET);
  } catch (err) {
    res.status(400).send(`Webhook Error: ${err}`);
    return;
  }

  // 冪等キーで重複実行を防ぐ
  const idempotencyKey = event.id;
  const existing = await db.webhookEvent.findUnique({ where: { idempotencyKey } });
  if (existing) {
    res.json({ received: true });
    return;
  }

  switch (event.type) {
    case 'invoice.payment_succeeded': {
      const invoice = event.data.object;
      await db.subscription.update({
        where: { stripeSubId: invoice.subscription },
        data: { status: 'active', nextBillingDate: new Date(invoice.next_payment_attempt * 1000) },
      });
      break;
    }
  }

  res.json({ received: true });
});
```

### 収益モデル設計 — フリーミアム導線

**ベストプラクティス：無料体験（7日 Trial） → 月額 $4.99 → 年額 $39.99 の3段階モデルとし、「制限」で段階的に課金へ導く。**

成功するフリーミアムアプリの特徴は、無料でコア機能を十分に体験させつつ、高度な機能は課金で提供することです。初回タップで課金せず、2-3回目で「この機能ロック解除にはプレミアムが必要」と表示する設計が標準的です。

```typescript
// フリーミアム ロジック
interface FeatureGate {
  name: string;
  requiredLevel: 'free' | 'premium';
  trialDaysLeft: number;
}

export function canUseFeature(feature: FeatureGate, userStatus: UserStatus): boolean {
  if (feature.requiredLevel === 'free') return true;

  // Premium 判定
  if (userStatus.subscription?.isActive) return true;

  // Trial 中？
  if (userStatus.subscription?.trialEndsAt && userStatus.subscription.trialEndsAt > new Date()) {
    return true;
  }

  return false;
}
```

### React memo による レンダリング最適化

**ベストプラクティス：大規模リスト（100 項目以上）の子要素に React.memo を適用し、props オブジェクトは useMemo でメモ化する。**

React Native では不必要な再レンダリングがパフォーマンス低下に直結します。FlatList の renderItem 内では必ず memo を使用します。

```typescript
// 最適化されたリストアイテム
const ListItem = React.memo(({ item, onPress }: ListItemProps) => (
  <TouchableOpacity onPress={() => onPress(item.id)}>
    <Text>{item.name}</Text>
  </TouchableOpacity>
), (prevProps, nextProps) => {
  // カスタム比較：item.id と onPress のみ確認
  return prevProps.item.id === nextProps.item.id &&
         prevProps.onPress === nextProps.onPress;
});

// 親コンポーネントで useMemo で props を安定化
export function ArticleList({ articles }: ArticleListProps) {
  const handlePress = useCallback((id: string) => {
    navigation.push('Detail', { articleId: id });
  }, [navigation]);

  const memoizedItems = useMemo(
    () => articles.map((a) => ({ ...a })),
    [articles],
  );

  return (
    <FlatList
      data={memoizedItems}
      renderItem={({ item }) => (
        <ListItem item={item} onPress={handlePress} />
      )}
      keyExtractor={(item) => item.id}
    />
  );
}
```

### オフラインファースト — SQLite + CRDT 同期

**ベストプラクティス：SQLite でローカル DB を保持し、CRDT アルゴリズムで競合を自動解決、ネットワーク復帰時にバッチ同期する。**

オフラインファースト設計では、ローカル DB が真実の源泉となり、サーバーは次の状態を確認するリポジトリの役割を持ちます。

```typescript
// SQLite + CRDT 同期パターン
import SQLite from 'react-native-sqlite-storage';
import { Automerge } from '@automerge/automerge';

class OfflineSync {
  private db: SQLiteDatabase;
  private automerge = Automerge.new();

  async syncWithServer() {
    // ローカル変更をサーバーに送信
    const localChanges = await this.db.executeSql(
      'SELECT * FROM changes WHERE synced = 0',
    );

    const response = await fetch('/api/sync', {
      method: 'POST',
      body: JSON.stringify({
        changes: localChanges[0].rows._array,
        clientId: this.clientId,
      }),
    });

    const serverChanges = await response.json();

    // CRDT で競合を解決
    const merged = Automerge.merge(this.automerge, serverChanges.doc);

    // DB を更新
    await this.db.executeSql('UPDATE changes SET synced = 1');
  }
}
```

### Gemini API ストリーミング チャットボット

**ベストプラクティス：Server-Sent Events (SSE) でリアルタイム表示し、50ms ごとにバッファをフラッシュしてチャプターリング効果を避ける。**

```typescript
// Gemini API ストリーミング実装
async function streamChatResponse(message: string) {
  const response = await fetch('https://generativelanguage.googleapis.com/v1/models/gemini-pro:streamGenerateContent', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({
      contents: [{ parts: [{ text: message }] }],
    }),
  });

  const reader = response.body.getReader();
  const decoder = new TextDecoder();
  let buffer = '';

  while (true) {
    const { done, value } = await reader.read();
    if (done) break;

    buffer += decoder.decode(value);
    const lines = buffer.split('\n');

    // 完全な行のみ処理
    for (let i = 0; i < lines.length - 1; i++) {
      const line = lines[i];
      if (line.startsWith('data: ')) {
        const data = JSON.parse(line.slice(6));
        if (data.candidates?.[0]?.content?.parts?.[0]?.text) {
          updateChatUI(data.candidates[0].content.parts[0].text);
        }
      }
    }
    buffer = lines[lines.length - 1];
  }
}
```

### Cloudflare Workers AI での レート制限 + キャッシュ

**ベストプラクティス：AI Gateway でレート制限（100req/min）を設定し、D1 でセッションを永続化してコスト削減。**

エッジ処理を活用することで、レイテンシと コストを最適化できます。

```typescript
// Cloudflare Worker での AI Gateway 利用
export default {
  async fetch(request: Request, env: Env) {
    const cache = caches.default;
    const cacheKey = new Request(request.url, { method: 'GET' });
    let response = await cache.match(cacheKey);

    if (!response) {
      // AI Gateway へリクエスト（レート制限内）
      response = await fetch('https://api.cloudflare.com/client/v4/accounts/xxx/ai/run/text_generation', {
        method: 'POST',
        headers: { Authorization: `Bearer ${env.CF_API_TOKEN}` },
        body: JSON.stringify({ prompt: 'こんにちは' }),
      });

      // 5分キャッシュ
      response = new Response(response.body, response);
      response.headers.set('Cache-Control', 'max-age=300');
      await cache.put(cacheKey, response.clone());
    }

    return response;
  },
};
```

### React Native + Expo AI 開発 — Typed Interface

**ベストプラクティス：AI レスポンスは Zod スキーマで型検証し、不完全なレスポンスは自動リトライ。**

AI の出力は予測不可能なため、型安全性が重要です。

```typescript
import { z } from 'zod';

// AI レスポンスの型定義
const AIResponseSchema = z.object({
  suggestion: z.string(),
  confidence: z.number().min(0).max(1),
  relatedTopics: z.array(z.string()),
});

type AIResponse = z.infer<typeof AIResponseSchema>;

async function generateAIInsight(userInput: string): Promise<AIResponse> {
  let retries = 0;
  while (retries < 3) {
    try {
      const raw = await fetch('/api/ai/generate', {
        method: 'POST',
        body: JSON.stringify({ prompt: userInput }),
      }).then((r) => r.json());

      return AIResponseSchema.parse(raw);
    } catch (error) {
      retries++;
      if (retries === 3) throw error;
      await new Promise((r) => setTimeout(r, 1000 * retries));
    }
  }
  throw new Error('Failed to generate AI response');
}
```
