## はじめに — なぜローカルデータ永続化が重要なのか

モバイルアプリでは、ユーザー設定やキャッシュデータ、オフライン対応のためのローカルデータ管理が不可欠です。Rork でアプリを開発する際にも、データをどこにどう保存するかはアプリの品質とパフォーマンスに直結します。

たとえば、ユーザーのログイン状態を保持したり、フォームの入力途中データを復元したり、API レスポンスをキャッシュして表示速度を上げたりと、ローカルストレージの活用場面は多岐にわたります。

この記事では、React Native / Expo ベースの Rork アプリで利用できる代表的な3つのローカルストレージライブラリ — **AsyncStorage**、**MMKV**、**SQLite** — を比較し、ユースケース別の選び方と具体的な実装方法をコード例付きで解説します。

**この記事の対象読者:**

- Rork でアプリを開発し始めたが、データの保存方法に迷っている方
- AsyncStorage は使っているが、パフォーマンスに不満を感じている方
- オフライン対応やデータ検索機能をアプリに追加したい方

## AsyncStorage — もっともシンプルなキーバリューストア

### 特徴と仕組み

AsyncStorage は React Native の公式コミュニティが提供するキーバリュー型ストレージです。Rork のプロジェクトでもデフォルトで利用しやすく、学習コストが非常に低いのが特徴です。

- **データ形式**: 文字列のキーバリューペア（JSON.stringify でオブジェクトも保存可能）
- **非同期API**: すべての操作が Promise ベース
- **容量制限**: Android では SQLite バックエンド（6MB デフォルト）、iOS では plist ファイル
- **スレッドセーフ**: 内部でシリアライズされるため安全

### 実装例

```typescript
import AsyncStorage from '@react-native-async-storage/async-storage';

// ユーザー設定を保存する
const saveUserPreferences = async (preferences: {
  theme: 'light' | 'dark';
  language: string;
  notifications: boolean;
}) => {
  try {
    const jsonValue = JSON.stringify(preferences);
    await AsyncStorage.setItem('@user_preferences', jsonValue);
    console.log('設定を保存しました');
  } catch (error) {
    console.error('保存に失敗しました:', error);
  }
};

// ユーザー設定を読み込む
const loadUserPreferences = async () => {
  try {
    const jsonValue = await AsyncStorage.getItem('@user_preferences');
    if (jsonValue !== null) {
      return JSON.parse(jsonValue);
    }
    // デフォルト値を返す
    return { theme: 'light', language: 'ja', notifications: true };
  } catch (error) {
    console.error('読み込みに失敗しました:', error);
    return null;
  }
};

// 期待される出力:
// saveUserPreferences({ theme: 'dark', language: 'ja', notifications: true })
// => "設定を保存しました"
// loadUserPreferences()
// => { theme: 'dark', language: 'ja', notifications: true }
```

### AsyncStorage が適しているケース

- ユーザー設定（テーマ、言語、通知のON/OFF）
- 認証トークンの一時保存（※機密データには Keychain 推奨）
- 少量のキャッシュデータ（数十件程度）
- プロトタイプやMVP段階のアプリ

### 注意点

AsyncStorage は手軽ですが、パフォーマンス面では限界があります。大量のデータ（数百件以上）を頻繁に読み書きする場合、体感できるレベルで遅延が発生することがあります。また、非同期のみの API は、アプリの起動時にデータを即座に利用したいケースでは扱いにくい面もあります。

## MMKV — 高速な同期ストレージ

### 特徴と仕組み

react-native-mmkv は、WeChat（微信）チームが開発した MMKV ライブラリの React Native バインディングです。メモリマップドファイル（mmap）を使い、AsyncStorage と比べて**約30倍**高速な読み書きを実現します。

- **データ形式**: キーバリュー型（文字列・数値・真偽値をネイティブ型のまま保存）
- **同期 + 非同期API**: `storage.getString()` のように同期的にアクセスでき、アプリ起動直後から利用可能
- **暗号化対応**: AES-128 による暗号化をオプションで有効化
- **複数インスタンス**: 用途別に複数のストレージインスタンスを作成可能

### 実装例

```typescript
import { MMKV } from 'react-native-mmkv';

// ストレージインスタンスを作成
const storage = new MMKV({
  id: 'app-storage',
  // 暗号化を有効にする場合（オプション）
  // encryptionKey: 'your-encryption-key',
});

// 同期的にデータを保存・取得する
const saveTheme = (theme: 'light' | 'dark') => {
  storage.set('user.theme', theme);
  console.log(`テーマを ${theme} に設定しました`);
};

const loadTheme = (): 'light' | 'dark' => {
  const theme = storage.getString('user.theme');
  return (theme as 'light' | 'dark') ?? 'light';
};

// オブジェクトを保存する場合
const saveCartItems = (items: Array<{ id: string; quantity: number }>) => {
  storage.set('cart.items', JSON.stringify(items));
};

const loadCartItems = (): Array<{ id: string; quantity: number }> => {
  const json = storage.getString('cart.items');
  return json ? JSON.parse(json) : [];
};

// 期待される出力:
// saveTheme('dark')
// => "テーマを dark に設定しました"
// loadTheme()
// => "dark"
// saveCartItems([{ id: 'item-1', quantity: 2 }])
// loadCartItems()
// => [{ id: 'item-1', quantity: 2 }]
```

### MMKV と React の状態管理を統合する

MMKV は `useMMKVStorage` フックを使って React の状態と直接連動させることもできます。

```typescript
import { useMMKVString, useMMKVBoolean } from 'react-native-mmkv';

function SettingsScreen() {
  // MMKV の値を React の状態として扱う
  const [username, setUsername] = useMMKVString('user.name');
  const [darkMode, setDarkMode] = useMMKVBoolean('user.darkMode');

  return (
    <View>
      <Text>ユーザー名: {username ?? '未設定'}</Text>
      <Switch
        value={darkMode ?? false}
        onValueChange={(value) => setDarkMode(value)}
      />
    </View>
  );
  // => 値を変更するとMMKVに自動保存され、コンポーネントが再レンダリングされる
}
```

### MMKV が適しているケース

- パフォーマンスが重要なアプリ（起動直後にデータが必要）
- ユーザー設定・フラグ・トークンなど頻繁にアクセスするデータ
- 暗号化が必要な機密データの保存
- [Rork アプリの状態管理パターン](/articles/rork-dev/rork-advanced-state-management-patterns) で解説されているグローバルステート管理と組み合わせると、特に効果的です

## SQLite — リレーショナルデータの本格管理

### 特徴と仕組み

expo-sqlite は、端末上でリレーショナルデータベースを扱える SQLite のラッパーです。テーブル定義、SQL クエリ、インデックス作成など、サーバーサイドのデータベースに近い機能をローカルで実現できます。

- **データ形式**: テーブル（行と列）によるリレーショナル構造
- **クエリ**: SQL 文でデータの検索・集計・結合が可能
- **インデックス**: 大量データでも高速な検索を実現
- **トランザクション**: データ整合性を保証

### 実装例

```typescript
import * as SQLite from 'expo-sqlite';

// データベースを開く
const db = await SQLite.openDatabaseAsync('myapp.db');

// テーブルを作成する
await db.execAsync(`
  CREATE TABLE IF NOT EXISTS tasks (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    title TEXT NOT NULL,
    completed INTEGER DEFAULT 0,
    category TEXT,
    created_at TEXT DEFAULT (datetime('now'))
  );
  CREATE INDEX IF NOT EXISTS idx_tasks_category ON tasks(category);
`);

// タスクを追加する
const addTask = async (title: string, category: string) => {
  const result = await db.runAsync(
    'INSERT INTO tasks (title, category) VALUES (?, ?)',
    [title, category]
  );
  console.log(`タスクを追加しました (ID: ${result.lastInsertRowId})`);
  return result.lastInsertRowId;
};

// カテゴリ別にタスクを取得する
const getTasksByCategory = async (category: string) => {
  const tasks = await db.getAllAsync(
    'SELECT * FROM tasks WHERE category = ? ORDER BY created_at DESC',
    [category]
  );
  return tasks;
};

// 完了率を集計する
const getCompletionStats = async () => {
  const stats = await db.getFirstAsync(`
    SELECT
      COUNT(*) as total,
      SUM(CASE WHEN completed = 1 THEN 1 ELSE 0 END) as done
    FROM tasks
  `);
  console.log(`完了率: ${stats.done}/${stats.total}`);
  return stats;
};

// 期待される出力:
// await addTask('買い物リスト作成', '日常')
// => "タスクを追加しました (ID: 1)"
// await getTasksByCategory('日常')
// => [{ id: 1, title: '買い物リスト作成', completed: 0, category: '日常', created_at: '2026-03-29 ...' }]
// await getCompletionStats()
// => "完了率: 0/1"
```

### SQLite が適しているケース

- 数百〜数万件のレコードを管理する必要がある
- 条件検索・ソート・集計が頻繁に発生する（タスク管理、家計簿、在庫管理など）
- リレーション（テーブル結合）が必要な構造化データ
- オフラインファーストのアプリで、サーバーとの同期前にローカルで完結させたい場合

オフライン対応を本格的に実装する場合は、[Rork Max のオフライン・ファースト・アーキテクチャ](/articles/rork-dev/rork-max-offline-first-architecture) も参考にしてみてください。ローカルDBとサーバー間の同期エンジンや競合解決の設計パターンが詳しく解説されています。

## 3つのストレージを比較する

用途やデータ量に応じて最適なストレージは異なります。以下のポイントで比較してみましょう。

**読み書き速度**: MMKV が圧倒的に高速（AsyncStorage の約30倍）。SQLite はクエリの複雑さに依存しますが、インデックスを適切に設定すれば数万件でも高速に動作します。AsyncStorage は少量データなら十分ですが、件数が増えると顕著に遅くなります。

**データ構造**: AsyncStorage と MMKV はキーバリュー型で、単純な値やJSONオブジェクトの保存に向いています。SQLite はテーブル構造で、複雑なデータ関係や検索条件がある場合に真価を発揮します。

**学習コスト**: AsyncStorage がもっとも簡単で、MMKV も API が似ているためすぐに移行できます。SQLite は SQL の知識が必要ですが、expo-sqlite の API はシンプルに設計されています。

**暗号化**: MMKV は組み込みの AES-128 暗号化をサポート。SQLite は SQLCipher で暗号化可能。AsyncStorage は暗号化機能を持たないため、機密データには不向きです。

**おすすめの選択基準**: まず AsyncStorage で始め、パフォーマンスが気になったら MMKV に移行する。構造化データや検索が必要になったら SQLite を導入する — というステップアップがもっとも現実的です。

## 実践パターン — 複数のストレージを組み合わせる

実際のアプリでは、1つのストレージだけでなく複数を組み合わせるのが効果的です。以下は、Rork で開発するタスク管理アプリを例にした使い分けパターンです。

```typescript
import { MMKV } from 'react-native-mmkv';
import * as SQLite from 'expo-sqlite';

// MMKV: アプリ設定・UIステート（高速アクセスが必要）
const settingsStorage = new MMKV({ id: 'settings' });

// SQLite: タスクデータ（検索・集計が必要）
let db: SQLite.SQLiteDatabase;

const initializeStorage = async () => {
  // MMKV は同期的に即座に利用可能
  const theme = settingsStorage.getString('theme') ?? 'light';
  const lastOpenedTab = settingsStorage.getString('lastTab') ?? 'today';

  // SQLite はテーブル初期化が必要
  db = await SQLite.openDatabaseAsync('tasks.db');
  await db.execAsync(`
    CREATE TABLE IF NOT EXISTS tasks (
      id INTEGER PRIMARY KEY AUTOINCREMENT,
      title TEXT NOT NULL,
      due_date TEXT,
      priority INTEGER DEFAULT 0,
      completed INTEGER DEFAULT 0
    );
  `);

  console.log(`初期化完了: テーマ=${theme}, タブ=${lastOpenedTab}`);
  return { theme, lastOpenedTab };
};

// UIステートの保存（同期・高速）
const saveLastTab = (tab: string) => {
  settingsStorage.set('lastTab', tab);
};

// タスクの検索（SQL クエリで柔軟に）
const searchTasks = async (keyword: string) => {
  return await db.getAllAsync(
    'SELECT * FROM tasks WHERE title LIKE ? ORDER BY priority DESC',
    [`%${keyword}%`]
  );
};

// 期待される出力:
// await initializeStorage()
// => "初期化完了: テーマ=light, タブ=today"
// saveLastTab('week')  ← 同期的に即座に保存
// await searchTasks('買い物')
// => [{ id: 1, title: '買い物リスト', due_date: '2026-03-30', ... }]
```

このように、**高速アクセスが必要な設定値は MMKV**、**検索や集計が必要な業務データは SQLite** という役割分担が効率的です。

## Rork でのセットアップのコツ

Rork でアプリを作成する際、AI プロンプトにデータ永続化の要件を明確に伝えることで、適切なストレージ構成を生成してもらえます。

効果的なプロンプトの例:

```
ユーザー設定（テーマ・言語・通知）は MMKV で同期的に管理してください。
タスクデータは expo-sqlite でテーブル管理し、カテゴリ別の検索とソートに対応してください。
アプリ起動時にMMKVから設定を即座に読み込み、SQLiteのマイグレーションは非同期で実行してください。
```

このように、何を保存するか・どのライブラリを使うか・アクセスパターンはどうかを具体的に指定すると、Rork の AI が最適なコードを生成してくれます。

認証トークンの安全な保存については [Rork でユーザー認証を実装する](/articles/rork-dev/rork-auth-firebase-supabase) も併せて確認してください。

## よくある質問（FAQ）

### Q1: AsyncStorage から MMKV への移行は大変ですか？

API の構造が似ているため、移行は比較的簡単です。`getItem` / `setItem` を `getString` / `set` に置き換え、非同期の `await` を外すのが主な変更点です。既存データの移行には、初回起動時に AsyncStorage から読み取って MMKV に書き込むマイグレーション関数を実装するのがおすすめです。

### Q2: SQLite のデータベースファイルはアプリ削除時にどうなりますか？

アプリを削除すると、SQLite のデータベースファイルも一緒に削除されます。重要なデータはサーバーにバックアップを取る設計にしておくと安心です。Supabase や Firebase と組み合わせたオフライン同期パターンが有効です。

### Q3: Expo の Managed Workflow でも MMKV は使えますか？

Expo SDK 50 以降では、Config Plugins を利用して react-native-mmkv を Managed Workflow でも使えます。Rork のプロジェクトでは Expo ベースの構成が多いため、`expo-dev-client` と組み合わせて利用するのが一般的です。`app.json` に Config Plugin を追加するだけで導入できます。

### Q4: セキュリティが重要なデータはどのストレージを使うべきですか？

認証トークンや個人情報など機密性の高いデータには、MMKV の暗号化オプションまたは expo-secure-store（Keychain / Keystore ラッパー）の利用を推奨します。[Rork Max セキュリティベストプラクティス](/articles/rork-dev/rork-max-security-best-practices) で詳しい実装パターンを解説しています。

## まとめ

Rork アプリのローカルデータ永続化は、データの特性とアクセスパターンに合わせて最適なライブラリを選ぶことが大切です。シンプルな設定保存なら AsyncStorage、パフォーマンス重視なら MMKV、構造化データの管理なら SQLite というのが基本的な指針です。実際のアプリでは複数のストレージを組み合わせて、それぞれの強みを活かす設計がもっとも実用的です。

まずは Rork のプロンプトでストレージ要件を明確に伝えることから始めてみてください。AI がプロジェクトの規模やユースケースに合った最適なデータ管理コードを生成してくれるはずです。
