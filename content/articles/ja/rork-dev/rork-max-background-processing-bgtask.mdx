**Rork Max で Swift ネイティブアプリを開発する際、最も複雑で重要な機能の1つがバックグラウンド処理です。** BGTaskScheduler、Background App Refresh、Background Download/Upload、LocalNotifications を組み合わせることで、ユーザーがアプリを開いていない時でも動作し続ける信頼性の高いアプリが実現できます。

しかし、バックグラウンド処理は iOS のセキュリティモデルと制約が厳しく、理解を間違えるとアプリがクラッシュしたり、審査で リジェクト されたりします。このガイドでは、**Rork Max で生成されたコードをベースに、バックグラウンド処理の全パターンを実装のコツとともに解説** します。

## バックグラウンド処理の4つの仕組み

iOS では、アプリがフォアグラウンドから離れた後のコード実行方法が4種類あります。それぞれの用途と制約を理解することが、正しい実装の第一歩です。

### 1. BGTaskScheduler — スケジュール実行タスク

**BGTaskScheduler** は、iOS 13 以降で推奨される「バックグラウンドで定期的にタスクを実行」する仕組みです。データベースのクリーンアップ、同期処理、レポート生成など、「リアルタイムである必要はないが、定期的に実行すべき処理」に最適です。

```swift
import BackgroundTasks

// Step 1: タスク登録（AppDelegate または SceneDelegate）
func registerBackgroundTasks() {
    let success = BGTaskScheduler.shared.register(
        forTaskWithIdentifier: "com.example.app.cleanup",
        using: nil
    ) { (task) in
        self.handleCleanupTask(task: task as! BGProcessingTask)
    }

    if !success {
        print("Failed to register background task")
    }
}

// Step 2: タスク処理
func handleCleanupTask(task: BGProcessingTask) {
    // タスクがシステムに強制終了されるのを防ぐため、
    // 処理が完了したら必ずsetTaskCompleted()を呼び出す
    task.expirationHandler = {
        print("Task expired, cleaning up...")
    }

    // データベース最適化処理
    DispatchQueue.global().async {
        self.optimizeDatabase()
        task.setTaskCompleted(success: true)
    }
}

// Step 3: タスクスケジュール
func scheduleCleanupTask() {
    let request = BGProcessingTaskRequest(
        identifier: "com.example.app.cleanup"
    )
    request.requiresNetworkConnectivity = false
    request.requiresExternalPower = false

    do {
        try BGTaskScheduler.shared.submit(request)
        print("Cleanup task scheduled")
    } catch {
        print("Failed to schedule cleanup: \(error)")
    }
}
```

**重要な制約**:
- タスク実行時刻は **OS が決定** します（「だいたい数時間ごと」の精度）
- タスク実行は 30 秒以内に完了する必要があります
- デバイスが充電されている状態で実行される傾向があります
- `Info.plist` に `BGTaskSchedulerPermittedIdentifiers` キーを設定し、アプリが許可するタスク ID を明記する必要があります

### 2. Background App Refresh — 低頻度の定期更新

**Background App Refresh** は、BGTaskScheduler よりさらに低頻度（1日数回程度）で、アプリのメインロジックを実行したい場合に使います。ニュースアプリの「バックグラウンド記事更新」が典型例です。

```swift
import BackgroundTasks
import UserNotifications

func scheduleBackgroundRefresh() {
    let request = BGAppRefreshTaskRequest(identifier: "com.example.app.refresh")
    request.earliestBeginDate = Date(timeIntervalSinceNow: 15 * 60) // 15分後

    do {
        try BGTaskScheduler.shared.submit(request)
    } catch {
        print("Failed to schedule background refresh: \(error)")
    }
}

// タスク処理
func handleBackgroundRefresh(task: BGAppRefreshTask) {
    task.expirationHandler = {
        task.setTaskCompleted(success: false)
    }

    // 非同期処理（例：API呼び出し、データ取得）
    fetchLatestData { success in
        task.setTaskCompleted(success: success)
    }
}
```

### 3. Background Download/Upload — ファイル転送

**URLSessionDownloadTask** と `URLSessionConfiguration.background` を組み合わせることで、大容量ファイルのダウンロード・アップロードをバックグラウンドで実行できます。アプリが強制終了されても、OSがダウンロードを継続します。

```swift
import Foundation

func startBackgroundDownload(url: URL, destination: String) {
    var config = URLSessionConfiguration.background(
        withIdentifier: "com.example.app.download"
    )
    config.isDiscretionary = true // バッテリー節約時は実行を延期
    config.sessionSendsLaunchEvents = true // アプリ起動時にコールバック実行

    let session = URLSession(
        configuration: config,
        delegate: DownloadDelegate(),
        delegateQueue: .main
    )

    let downloadTask = session.downloadTask(with: url)
    downloadTask.resume()
}

// URLSessionDownloadDelegate の実装
class DownloadDelegate: NSObject, URLSessionDownloadDelegate {
    func urlSession(
        _ session: URLSession,
        downloadTask: URLSessionDownloadTask,
        didFinishDownloadingTo location: URL
    ) {
        // ダウンロード完了時の処理
        let fileManager = FileManager.default
        let documentsPath = fileManager.urls(
            for: .documentDirectory,
            in: .userDomainMask
        )[0]

        let destinationURL = documentsPath.appendingPathComponent("downloaded.zip")

        do {
            try fileManager.moveItem(at: location, to: destinationURL)
            print("Download completed: \(destinationURL)")
        } catch {
            print("Error moving downloaded file: \(error)")
        }
    }

    func urlSession(
        _ session: URLSession,
        downloadTask: URLSessionDownloadTask,
        didFailWithError error: Error
    ) {
        print("Download failed: \(error)")
    }
}
```

### 4. Local Notifications — バックグラウンドからのユーザー通知

バックグラウンド処理の結果をユーザーに通知する場合、**UNUserNotificationCenter** を使い、LocalNotification を送信します。

```swift
import UserNotifications

func scheduleLocalNotification(
    title: String,
    body: String,
    delay: TimeInterval = 5
) {
    let content = UNMutableNotificationContent()
    content.title = title
    content.body = body
    content.sound = .default
    content.badge = NSNumber(value: UIApplication.shared.applicationIconBadgeNumber + 1)

    // カスタムデータを追加（タップ時にアプリで処理可能）
    content.userInfo = ["articleID": "12345"]

    let trigger = UNTimeIntervalNotificationTrigger(
        timeInterval: delay,
        repeats: false
    )

    let request = UNNotificationRequest(
        identifier: UUID().uuidString,
        content: content,
        trigger: trigger
    )

    UNUserNotificationCenter.current().add(request) { error in
        if let error = error {
            print("Failed to schedule notification: \(error)")
        }
    }
}

// ユーザーが通知をタップした時の処理
func userNotificationCenter(
    _ center: UNUserNotificationCenter,
    didReceive response: UNNotificationResponse,
    withCompletionHandler completionHandler: @escaping () -> Void
) {
    let userInfo = response.notification.request.content.userInfo

    if let articleID = userInfo["articleID"] as? String {
        // 記事ページを表示する等の処理
        navigateToArticle(id: articleID)
    }

    completionHandler()
}
```

## Rork Max で推奨するバックグラウンド処理パターン

### パターン 1: 1日1回のデータ同期

キャッシュデータや設定の同期を 1 日 1 回程度実行したい場合。

```swift
// Rork Max プロンプト例：
// "BGTaskScheduler を使い、1日1回メインデータベースと同期するコードを生成してください"

class SyncService {
    static func setupDailySync() {
        let identifier = "com.rorkapp.daily-sync"

        BGTaskScheduler.shared.register(
            forTaskWithIdentifier: identifier,
            using: .main
        ) { task in
            Self.performSync(task: task as! BGProcessingTask)
        }
    }

    static func scheduleNextSync() {
        let request = BGProcessingTaskRequest(
            identifier: "com.rorkapp.daily-sync"
        )
        request.requiresNetworkConnectivity = true

        do {
            try BGTaskScheduler.shared.submit(request)
        } catch {
            print("Sync scheduling failed: \(error)")
        }
    }

    static func performSync(task: BGProcessingTask) {
        task.expirationHandler = {
            task.setTaskCompleted(success: false)
        }

        // API呼び出し
        URLSession.shared.dataTask(with: URL(string: "https://api.example.com/sync")!) {
            data, response, error in

            if error == nil, let data = data {
                // データをローカルDBに保存
                self.saveToDatabase(data)
                task.setTaskCompleted(success: true)
            } else {
                task.setTaskCompleted(success: false)
            }

            // 次回のタスクをスケジュール
            self.scheduleNextSync()
        }.resume()
    }

    static func saveToDatabase(_ data: Data) {
        // Realm や CoreData で保存
    }
}
```

### パターン 2: 大容量ファイルのバックグラウンドダウンロード

地図データやアプリリソースの事前ダウンロード。

```swift
class BackgroundDownloadManager: NSObject, URLSessionDownloadDelegate {
    static let shared = BackgroundDownloadManager()

    var session: URLSession!

    override init() {
        super.init()

        let config = URLSessionConfiguration.background(
            withIdentifier: "com.rorkapp.bgdownload"
        )
        config.isDiscretionary = true
        self.session = URLSession(
            configuration: config,
            delegate: self,
            delegateQueue: nil
        )
    }

    func downloadLargeFile(url: URL) {
        let task = session.downloadTask(with: url)
        task.resume()
    }

    func urlSession(
        _ session: URLSession,
        downloadTask: URLSessionDownloadTask,
        didFinishDownloadingTo location: URL
    ) {
        let fileManager = FileManager.default
        let docsPath = fileManager.urls(for: .documentDirectory, in: .userDomainMask)[0]
        let fileName = downloadTask.originalRequest?.url?.lastPathComponent ?? "download"
        let destination = docsPath.appendingPathComponent(fileName)

        do {
            try fileManager.moveItem(at: location, to: destination)

            // ダウンロード完了の通知
            notifyDownloadComplete(fileName: fileName)
        } catch {
            print("Error: \(error)")
        }
    }

    func notifyDownloadComplete(fileName: String) {
        let content = UNMutableNotificationContent()
        content.title = "ダウンロード完了"
        content.body = "\(fileName) がダウンロードされました"
        content.sound = .default

        let trigger = UNTimeIntervalNotificationTrigger(timeInterval: 1, repeats: false)
        let request = UNNotificationRequest(
            identifier: UUID().uuidString,
            content: content,
            trigger: trigger
        )

        UNUserNotificationCenter.current().add(request)
    }
}
```

### パターン 3: バックグラウンドでローカルデータ処理

写真加工、ファイル圧縮、データキャッシュの削除など、ネットワーク不要な処理。

```swift
class DataProcessingService {
    static func scheduleOfflineProcessing() {
        let identifier = "com.rorkapp.offline-process"

        BGTaskScheduler.shared.register(
            forTaskWithIdentifier: identifier,
            using: nil
        ) { task in
            self.processLocalData(task: task as! BGProcessingTask)
        }
    }

    static func requestProcessing() {
        let request = BGProcessingTaskRequest(
            identifier: "com.rorkapp.offline-process"
        )
        request.requiresNetworkConnectivity = false
        request.requiresExternalPower = false

        do {
            try BGTaskScheduler.shared.submit(request)
        } catch {
            print("Processing request failed: \(error)")
        }
    }

    static func processLocalData(task: BGProcessingTask) {
        task.expirationHandler = {
            task.setTaskCompleted(success: false)
        }

        DispatchQueue.global(qos: .background).async {
            // 例: 30日以上前のキャッシュを削除
            let docsPath = FileManager.default.urls(
                for: .documentDirectory,
                in: .userDomainMask
            )[0]

            let cache = docsPath.appendingPathComponent("Cache")

            do {
                let files = try FileManager.default.contentsOfDirectory(
                    at: cache,
                    includingPropertiesForKeys: [.contentModificationDateKey]
                )

                let thirtyDaysAgo = Date(timeIntervalSinceNow: -30 * 24 * 3600)

                for file in files {
                    let attrs = try FileManager.default.attributesOfItem(atPath: file.path)
                    if let modDate = attrs[.modificationDate] as? Date,
                       modDate < thirtyDaysAgo {
                        try FileManager.default.removeItem(at: file)
                    }
                }

                task.setTaskCompleted(success: true)
            } catch {
                task.setTaskCompleted(success: false)
            }
        }
    }
}
```

## Info.plist 設定（必須）

バックグラウンド処理を有効化するには、プロジェクトの **Info.plist** に以下を追加します。

```xml
<key>BGTaskSchedulerPermittedIdentifiers</key>
<array>
    <string>com.rorkapp.daily-sync</string>
    <string>com.rorkapp.offline-process</string>
</array>
```

また、Xcode の **Signing & Capabilities** タブで **Background Modes** を有効にし、以下をチェック:
- ✅ Background App Refresh
- ✅ Background Processing
- ✅ Background Fetch（オプション）

## よくある質問（FAQ）

**Q1: BGTaskScheduler のタスク実行時刻は予測できますか？**

いいえ。OS が「デバイスの状態（充電中か、WiFi接続状態か）」「ユーザーの使用パターン」「バッテリー節約モード」などを判断して決定します。タスク実行が「いつ」行われるかは約束されません。

もし **正確な時刻実行** が必要な場合は、フォアグラウンド（ユーザーがアプリを開いている時）に処理を実行する設計に変更するか、バックエンド（サーバー側）でスケジュール実行を行い、プッシュ通知でアプリに通知する方法を検討してください。

**Q2: バックグラウンド処理でネットワーク接続が切れたら？**

URLSession は自動的に再接続を試みます。`requiresNetworkConnectivity = true` で設定したタスクは、ネットワークが利用可能になるまで延期されます。

処理途中でネットワークが切れた場合は `expirationHandler` が呼ばれます。ここで「部分的に保存したデータ」をロールバックするなど、安全な終了処理を実装してください。

**Q3: バックグラウンドタスク内で複数の API 呼び出しをしてもいい？**

30秒以内なら可能ですが、**2–3個の軽量API呼び出し程度** にとどめることをおすすめします。大量のAPI呼び出しが必要な場合は、バックエンド（サーバー側）で処理を集約し、結果をまとめてアプリに返す設計にしましょう。

**Q4: バックグラウンドタスク中にローカル通知を送信できますか？**

はい。タスク処理中（30秒以内）なら `UNUserNotificationCenter.current().add(request)` で通知を送信できます。

**Q5: テストでバックグラウンドタスクの動作を確認するには？**

Xcode の **Scheme > Run > Pre-actions** でシミュレータからバックグラウンドタスクをトリガーできます。実機テストの場合は `e -l objc -- (void)[[BGTaskScheduler sharedScheduler] _simulateLaunchForTaskWithIdentifier:@"com.example.task"]` をコンソールで実行することで、タスク実行をシミュレートできます。

## ベストプラクティス

1. **タスク登録は起動時に一度だけ** — `application(_:didFinishLaunchingWithOptions:)` で呼び出し
2. **expirationHandler は必ず実装** — タスク強制終了時の安全な後片付け
3. **30秒ルールを守る** — タスク処理は常に 30 秒以内に完了
4. **ネットワーク接続をチェック** — `requiresNetworkConnectivity` で明示的に設定
5. **Local Notifications と組み合わせる** — バックグラウンド処理の結果をユーザーに通知
6. **Instruments で監視** — System Trace で実際のタスク実行時刻を確認
