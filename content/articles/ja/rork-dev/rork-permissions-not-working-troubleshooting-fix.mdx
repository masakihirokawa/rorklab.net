Expo Goでは正常に動いていたカメラ機能が、TestFlightビルドを渡したとたんに「ダイアログが出ない」「アクセスが拒否される」と報告されてくる——これはRorkでアプリを開発するときに驚くほど多くの開発者がぶつかる壁だ。

権限（パーミッション）系のバグは再現環境が限られ、かつエラーメッセージが不親切なため、原因特定に時間がかかりやすい。この記事では、カメラ・位置情報・プッシュ通知を中心に、Rorkのコード生成特性と合わせてパーミッション問題を根本から整理する。

## なぜRorkのAI生成コードで権限エラーが起きやすいのか

RorkのAIはコンポーネントを個別に生成するため、権限リクエストのロジックが分散しがちだ。たとえばカメラ画面を生成したとき、`useCameraPermissions` のフックは追加されるが、`app.json` や `Info.plist` の `NSCameraUsageDescription` が漏れることがある。

iOS/Android の権限系エラーには次の3つのパターンがある。

**パターン①「ダイアログ自体が出ない」**
ネイティブ側の宣言（`Info.plist` / `AndroidManifest.xml`）が欠けているため、OSが権限を認識しておらず、リクエストを無視している状態。エラーも出ないので最もはまりやすい。

**パターン②「初回は出たが、以降出なくなった」**
ユーザーが一度「許可しない」を選択した場合、iOSは同じアプリから再度ダイアログを出す手段を持たない。コードが正しくても、設定アプリへ誘導する処理がなければアプリとして機能しなくなる。

**パターン③「Expo Goでは動くが実ビルドで失敗する」**
Expo Goには多数の権限が事前に宣言されており、それに乗っかって動いていた可能性がある。独立したビルドでは自分のアプリの宣言だけが有効になるため、Expo Go環境でしか検証していないと本番で初めて気づく。

## iOS・Androidで必要な設定の違い

権限の宣言方法はプラットフォームで異なる。Rorkが生成した `app.json` を確認し、以下の項目が揃っているか確認しよう。

**iOS（`app.json` の `expo.ios.infoPlist`）**

```json
{
  "expo": {
    "ios": {
      "infoPlist": {
        "NSCameraUsageDescription": "写真・動画の撮影に使用します",
        "NSMicrophoneUsageDescription": "動画撮影時の音声に使用します",
        "NSLocationWhenInUseUsageDescription": "現在地の近くのスポットを表示するために使用します",
        "NSPhotoLibraryUsageDescription": "写真を保存・読み込みするために使用します",
        "NSPhotoLibraryAddUsageDescription": "写真を保存するために使用します"
      }
    }
  }
}
```

**Android（`app.json` の `expo.android.permissions`）**

```json
{
  "expo": {
    "android": {
      "permissions": [
        "CAMERA",
        "RECORD_AUDIO",
        "ACCESS_FINE_LOCATION",
        "ACCESS_COARSE_LOCATION",
        "READ_MEDIA_IMAGES",
        "READ_MEDIA_VIDEO"
      ]
    }
  }
}
```

`NSLocationAlwaysAndWhenInUseUsageDescription`（バックグラウンド位置情報）は設定するだけでApp Storeの審査で説明を求められる。本当に必要でなければ `NSLocationWhenInUseUsageDescription` だけにすること。バックグラウンド位置情報は審査基準が厳しく、不要な宣言は却下につながる。

## 正しい権限リクエストの実装パターン

Rorkが生成するコードはシンプルなケースに最適化されているため、「一度拒否された後の処理」が抜けていることが多い。以下は実用的な完全パターンだ。

```typescript
import { Alert, Linking } from 'react-native';
import { useCameraPermissions, PermissionStatus } from 'expo-camera';

export function useCameraWithFallback() {
  const [permission, requestPermission] = useCameraPermissions();

  const requestWithFallback = async () => {
    // まだ確認されていない場合はダイアログを表示
    if (permission?.status === PermissionStatus.UNDETERMINED) {
      const result = await requestPermission();
      return result.granted;
    }

    // 一度拒否されている場合は設定アプリへ誘導
    if (permission?.status === PermissionStatus.DENIED) {
      Alert.alert(
        'カメラへのアクセスが必要です',
        'このアプリでカメラを使うには、設定アプリから許可してください。',
        [
          { text: 'キャンセル', style: 'cancel' },
          {
            text: '設定を開く',
            onPress: () => Linking.openSettings(),
          },
        ]
      );
      return false;
    }

    // 許可済みの場合はそのまま使用可
    return permission?.granted ?? false;
  };

  return { permission, requestPermission: requestWithFallback };
}
```

このパターンのポイントは `UNDETERMINED`（未確認）と `DENIED`（拒否済み）を明確に分岐している点だ。Rorkが生成するコードは `requestPermission()` を直接呼ぶだけのシンプルな実装が多いが、それだと拒否後に何も起きなくなる。

## 位置情報権限特有のはまりポイント

位置情報は権限の粒度が細かく、混乱しやすい。

iOSでは「アプリ使用中のみ」から「常に許可」へのアップグレードを段階的に行う仕様になっている。最初から「常に許可」のダイアログを出そうとするコードは、iOSが内部的に「アプリ使用中のみ」に書き換えてしまうことがある。

```typescript
import * as Location from 'expo-location';

async function requestLocationPermission() {
  // まず「アプリ使用中のみ」を要求
  const { status: foregroundStatus } =
    await Location.requestForegroundPermissionsAsync();

  if (foregroundStatus \!== 'granted') {
    console.log('前景位置情報が拒否されました');
    return false;
  }

  // バックグラウンドが本当に必要な場合のみ追加要求
  // 不要なら以下のブロックごと削除する
  const { status: backgroundStatus } =
    await Location.requestBackgroundPermissionsAsync();

  if (backgroundStatus \!== 'granted') {
    // 前景だけで動作する機能に限定する
    return 'foreground-only';
  }

  return 'full';
}
```

## Expo Goと実ビルドの権限テスト方法

権限のテストはExpo Goだけでは不十分だ。特に以下の2点は実ビルドでないと確認できない。

**確認ポイント①：「許可しない」を選んだ後の挙動**
Expo Goだと権限を一度リセットしてテストしにくい。TestFlightやEAS Buildで実際のスタンドアロンアプリを使い、「許可しない」→設定アプリへの誘導が正しく表示されるか必ず確認する。

**確認ポイント②：権限説明文の表示**
Appleの審査では、ダイアログに表示される説明文が「この権限が必要な理由を具体的に説明していること」を求める。Rorkが自動生成した宣言文をそのまま使っている場合、英語のままになっていることがある。`app.json` の説明文を確認してプッシュする前に必ず上書きしよう。日本語ターゲットのアプリなら説明文も日本語で書くことが推奨される。

## ストア審査で権限が原因で却下されるケース

権限に関してApp StoreとGoogle Playで審査に引っかかるパターンをまとめておく。

**App Store（Apple）の主な却下理由**

- 使っていない権限が `Info.plist` に宣言されている（Guideline 5.1.1）
- 説明文が不十分または不自然（Guideline 2.5.4）
- バックグラウンド位置情報を使っているのに正当な理由がない（Guideline 2.5.4）

**Google Play（Android）の主な却下理由**

- ターゲットAPIレベルが34以上で古い権限名を使っている（Android 13以降は `READ_MEDIA_IMAGES` が必要で、`READ_EXTERNAL_STORAGE` は非推奨）
- 「危険な権限」（カメラ・マイク・連絡先など）をランタイムで要求せずにインストール時の宣言だけで済ませようとしている

特にAndroidは権限APIの変更が頻繁なため、Rorkが生成するコードのターゲットAPIレベルと権限名が一致しているか確認する習慣をつけておこう。

## まとめ：権限エラーを事前につぶすチェックリスト

権限問題は機能実装のタイミングで潰しておくのが最もコストが低い。機能を追加したら直後に `app.json` の宣言を確認し、コード側に `UNDETERMINED` / `DENIED` の分岐を書き、TestFlight実機で「拒否後フロー」を一度通してみる——この手順を習慣にするだけで、リリース後に「カメラが使えない」と報告が来るケースの大半は防げる。

Rorkのパーミッション実装についてさらに詳しく学びたい場合は、[Expo公式ドキュメントのPermissionsガイド](https://docs.expo.dev/guides/permissions/)が体系的にまとまっておりおすすめだ。
