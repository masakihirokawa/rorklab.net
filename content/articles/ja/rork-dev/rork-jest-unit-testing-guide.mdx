## はじめに — なぜ Rork アプリにテストが必要なのか

Rork や Rork Max で生成したアプリは、AI が素早くコードを書いてくれる反面、**ロジックの正確性を保証するのは開発者自身の責任**です。アプリが成長し、機能追加やリファクタリングを行うたびに「どこかが壊れていないか」を手動で確認し続けるのは現実的ではありません。

そこで活躍するのが **単体テスト（ユニットテスト）** です。単体テストを導入すると次のメリットが得られます。

- バグの早期発見（コードを書いた直後に検出できる）
- リファクタリング時の安心感（テストが通れば動作は変わっていない）
- 仕様の文書化（テストコードが「この関数はどう動くか」を示す）
- CI/CD パイプラインへの組み込み（自動化による品質担保）

この記事では、Rork 生成アプリへの **Jest** と **React Native Testing Library（RNTL）** の導入から、実践的なテストの書き方まで、ステップバイステップで解説します。対象読者は React Native の基礎を理解していて、テストを初めて書く方や書き慣れていない方です。

なお、E2E テスト（アプリ全体を実機で動かすテスト）については、[Rork Max E2E テスト完全ガイド（Detox・Maestro）](/articles/rork-dev/rork-max-e2e-testing-detox-maestro-guide) で詳しく解説しています。

## Jest と React Native Testing Library とは

### Jest

**Jest** は Meta（旧 Facebook）が開発した JavaScript のテストフレームワークです。React Native プロジェクトでは標準的に使われており、Expo で生成されたプロジェクトにはデフォルトで設定済みです。

- テストランナー（テストを実行する仕組み）
- アサーション（`expect(value).toBe(...)` のような検証構文）
- モック（外部依存を仮のオブジェクトに差し替える）
- スナップショットテスト（UI の意図しない変更を検出）

の4役をひとつのパッケージで担います。

### React Native Testing Library（RNTL）

**React Native Testing Library** は、コンポーネントを「ユーザー視点」でテストするためのライブラリです。内部実装ではなく「画面に何が表示されているか」「ボタンを押したら何が起きるか」という観点でテストを書けるため、壊れにくいテストが書けます。

```bash
# Rork Expo プロジェクトでの確認（すでにインストール済みの場合あり）
npx expo install @testing-library/react-native @testing-library/jest-native
```

## Rork アプリへのテスト環境構築

### Step 1 — package.json の確認

Rork で生成した Expo プロジェクトを開き、`package.json` に以下が含まれているか確認します。

```json
{
  "devDependencies": {
    "jest": "^29.0.0",
    "jest-expo": "^51.0.0",
    "@testing-library/react-native": "^12.0.0",
    "@testing-library/jest-native": "^5.0.0"
  },
  "jest": {
    "preset": "jest-expo"
  }
}
```

含まれていない場合は次のコマンドでインストールします。

```bash
# Expo プロジェクト推奨インストール方法
npx expo install jest-expo jest @testing-library/react-native @testing-library/jest-native
```

### Step 2 — jest.config.js の設定

プロジェクトルートに `jest.config.js` を作成（または確認）します。

```js
// jest.config.js
module.exports = {
  preset: 'jest-expo',
  setupFilesAfterFramework: ['@testing-library/jest-native/extend-expect'],
  transformIgnorePatterns: [
    'node_modules/(?!((jest-)?react-native|@react-native(-community)?)|expo(nent)?|@expo(nent)?/.*|@expo-google-fonts/.*|react-navigation|@react-navigation/.*|@unimodules/.*|unimodules|sentry-expo|native-base|react-native-svg)',
  ],
  moduleFileExtensions: ['ts', 'tsx', 'js', 'jsx'],
  testPathPattern: '(/__tests__/.*|(\\.|/)(test|spec))\\.[jt]sx?$',
};
```

> **ポイント**: `transformIgnorePatterns` は Expo ネイティブモジュールを Jest が正しく変換できるようにするための設定です。これが不足していると `SyntaxError` が頻発します。

### Step 3 — テストファイルの配置

テストファイルは以下のどちらかのパターンで配置します。

- `__tests__/` フォルダを作成してまとめる（規模が大きい場合）
- テスト対象ファイルと同じ場所に `*.test.tsx` を置く（シンプルで見つけやすい）

Rork 生成プロジェクトでは後者が扱いやすいでしょう。

```
src/
  components/
    Button.tsx
    Button.test.tsx   ← テストファイルをここに置く
  utils/
    formatPrice.ts
    formatPrice.test.ts
```

## 基本的な単体テストの書き方

まずは純粋な JavaScript 関数のテストから始めましょう。UI が絡まないため、最も書きやすいテストです。

### ユーティリティ関数のテスト例

```ts
// src/utils/formatPrice.ts
export function formatPrice(price: number, currency: string = 'JPY'): string {
  if (price < 0) {
    throw new Error('価格は0以上の値を指定してください');
  }
  return new Intl.NumberFormat('ja-JP', {
    style: 'currency',
    currency,
  }).format(price);
}
```

```ts
// src/utils/formatPrice.test.ts
import { formatPrice } from './formatPrice';

describe('formatPrice', () => {
  test('正常な価格をJPY形式でフォーマットする', () => {
    expect(formatPrice(1000)).toBe('¥1,000');
    // 期待する出力: ¥1,000
  });

  test('USD通貨でフォーマットする', () => {
    expect(formatPrice(9.99, 'USD')).toBe('$9.99');
    // 期待する出力: $9.99
  });

  test('0円をフォーマットする', () => {
    expect(formatPrice(0)).toBe('¥0');
  });

  test('負の値でエラーをスローする', () => {
    expect(() => formatPrice(-100)).toThrow('価格は0以上の値を指定してください');
  });
});
```

テストを実行します。

```bash
npx jest src/utils/formatPrice.test.ts

# 期待する出力:
# PASS  src/utils/formatPrice.test.ts
#   formatPrice
#     ✓ 正常な価格をJPY形式でフォーマットする (3ms)
#     ✓ USD通貨でフォーマットする (1ms)
#     ✓ 0円をフォーマットする
#     ✓ 負の値でエラーをスローする (1ms)
#
# Tests: 4 passed, 4 total
```

## コンポーネントテストの実践

React Native Testing Library を使ったコンポーネントテストは、「レンダリングされた画面に何が表示されているか」を検証します。

### シンプルなボタンコンポーネントのテスト

```tsx
// src/components/PrimaryButton.tsx
import React from 'react';
import { TouchableOpacity, Text, StyleSheet } from 'react-native';

type Props = {
  label: string;
  onPress: () => void;
  disabled?: boolean;
};

export function PrimaryButton({ label, onPress, disabled = false }: Props) {
  return (
    <TouchableOpacity
      style={[styles.button, disabled && styles.disabled]}
      onPress={onPress}
      disabled={disabled}
      accessibilityRole="button"
      accessibilityLabel={label}
    >
      <Text style={styles.label}>{label}</Text>
    </TouchableOpacity>
  );
}

const styles = StyleSheet.create({
  button: { backgroundColor: '#007AFF', padding: 16, borderRadius: 8 },
  disabled: { opacity: 0.5 },
  label: { color: '#fff', textAlign: 'center', fontWeight: '600' },
});
```

```tsx
// src/components/PrimaryButton.test.tsx
import React from 'react';
import { render, fireEvent } from '@testing-library/react-native';
import { PrimaryButton } from './PrimaryButton';

describe('PrimaryButton', () => {
  test('ラベルが正しく表示される', () => {
    const { getByText } = render(
      <PrimaryButton label="送信" onPress={() => {}} />
    );
    // 「送信」テキストが画面に存在することを確認
    expect(getByText('送信')).toBeTruthy();
  });

  test('タップ時に onPress が呼ばれる', () => {
    const mockOnPress = jest.fn(); // モック関数
    const { getByRole } = render(
      <PrimaryButton label="送信" onPress={mockOnPress} />
    );

    fireEvent.press(getByRole('button'));

    // onPress が1回呼ばれたことを確認
    expect(mockOnPress).toHaveBeenCalledTimes(1);
  });

  test('disabled の場合は onPress が呼ばれない', () => {
    const mockOnPress = jest.fn();
    const { getByRole } = render(
      <PrimaryButton label="送信" onPress={mockOnPress} disabled={true} />
    );

    fireEvent.press(getByRole('button'));

    // disabled なので onPress は呼ばれないことを確認
    expect(mockOnPress).not.toHaveBeenCalled();
  });
});
```

### 非同期処理を含むコンポーネントのテスト

API 呼び出しなど非同期処理があるコンポーネントは、`waitFor` を使います。

```tsx
// src/components/UserProfile.test.tsx
import React from 'react';
import { render, waitFor } from '@testing-library/react-native';
import { UserProfile } from './UserProfile';

// fetch をモック化
global.fetch = jest.fn(() =>
  Promise.resolve({
    json: () => Promise.resolve({ name: '山田太郎', email: 'yamada@example.com' }),
  })
) as jest.Mock;

describe('UserProfile', () => {
  test('ユーザー情報が取得後に表示される', async () => {
    const { getByText } = render(<UserProfile userId="123" />);

    // API レスポンス待ち（最大1000ms）
    await waitFor(() => {
      expect(getByText('山田太郎')).toBeTruthy();
    }, { timeout: 1000 });
  });
});
```

## カスタムフックのテスト

Rork アプリでよく使うカスタムフックもテストできます。`renderHook` を使います。

```ts
// src/hooks/useCounter.ts
import { useState, useCallback } from 'react';

export function useCounter(initialValue: number = 0) {
  const [count, setCount] = useState(initialValue);
  const increment = useCallback(() => setCount(c => c + 1), []);
  const decrement = useCallback(() => setCount(c => c - 1), []);
  const reset = useCallback(() => setCount(initialValue), [initialValue]);
  return { count, increment, decrement, reset };
}
```

```ts
// src/hooks/useCounter.test.ts
import { renderHook, act } from '@testing-library/react-native';
import { useCounter } from './useCounter';

describe('useCounter', () => {
  test('初期値が0になる', () => {
    const { result } = renderHook(() => useCounter());
    expect(result.current.count).toBe(0);
  });

  test('increment でカウントが増える', () => {
    const { result } = renderHook(() => useCounter());
    act(() => {
      result.current.increment();
    });
    expect(result.current.count).toBe(1);
  });

  test('decrement でカウントが減る', () => {
    const { result } = renderHook(() => useCounter(5));
    act(() => {
      result.current.decrement();
    });
    expect(result.current.count).toBe(4);
  });

  test('reset で初期値に戻る', () => {
    const { result } = renderHook(() => useCounter(10));
    act(() => {
      result.current.increment();
      result.current.increment();
      result.current.reset();
    });
    expect(result.current.count).toBe(10);
  });
});
```

## よくあるエラーと対処法

### エラー1: `SyntaxError: Cannot use import statement`

**原因**: Jest が ES Modules を正しく変換できていない。

**対処法**: `jest.config.js` の `transformIgnorePatterns` を確認し、問題のあるパッケージ名を除外リストに追加します。

```js
// よくある追加パターン
transformIgnorePatterns: [
  'node_modules/(?!((jest-)?react-native|@react-native(-community)?)|expo(nent)?|@expo(nent)?/.*|react-navigation|@react-navigation/.*|nativewind|react-native-reanimated)',
],
```

### エラー2: `Unable to find an element with the text`

**原因**: テキストの大文字小文字の違いや、ローディング中に検索している可能性がある。

**対処法**: 

```tsx
// 大文字小文字を無視する場合
getByText(/送信/i)

// 非同期でレンダリングされる場合
await waitFor(() => getByText('送信'));
```

### エラー3: `act(...)` 警告

**原因**: 状態更新が `act()` の外で発生している。

**対処法**: `fireEvent` や非同期処理を `act()` または `waitFor` でラップします。

```tsx
// 修正前（警告が出る）
fireEvent.press(button);
expect(mockFn).toHaveBeenCalled();

// 修正後
await act(async () => {
  fireEvent.press(button);
});
expect(mockFn).toHaveBeenCalled();
```

## テストカバレッジの確認

どの程度のコードがテストされているかは、カバレッジレポートで確認できます。

```bash
# カバレッジレポートを生成
npx jest --coverage

# 期待する出力（例）:
# ----------|---------|----------|---------|---------|
# File      | % Stmts | % Branch | % Funcs | % Lines |
# ----------|---------|----------|---------|---------|
# All files |   78.5  |   72.3   |   80.1  |   79.2  |
```

`package.json` にスクリプトを追加しておくと便利です。

```json
{
  "scripts": {
    "test": "jest",
    "test:watch": "jest --watch",
    "test:coverage": "jest --coverage"
  }
}
```

カバレッジの目標は**重要なビジネスロジックとユーティリティ関数は 80%以上**、UI コンポーネントは主要な操作をカバーする、という方針がおすすめです。

## FAQ

**Q. Rork で生成したコードはそのままテストできますか？**

A. はい、テストできます。ただし Rork が生成したコンポーネントに Native モジュール（カメラ・位置情報など）が含まれている場合は、その部分をモック化する必要があります。たとえば `expo-camera` は `jest.mock('expo-camera')` で差し替えられます。

**Q. スナップショットテストは有効ですか？**

A. シンプルな UI の変更検出には有効ですが、Rork のように AI がコードを頻繁に更新する環境では、スナップショットが頻繁に壊れて更新コストが高くなる傾向があります。スナップショットよりも「特定のテキストが表示されているか」「ボタンを押したら関数が呼ばれるか」といった動作テストを優先するのがおすすめです。

**Q. TypeScript プロジェクトで型エラーが出ます**

A. `tsconfig.json` の `compilerOptions` に `"types": ["jest", "@testing-library/jest-native"]` を追加してください。また `@types/jest` と `ts-jest` が devDependencies に含まれているか確認してください。

**Q. CI/CD でテストを自動化するには？**

A. GitHub Actions に以下のステップを追加することで、プッシュのたびにテストが自動実行されます。

```yaml
- name: Run tests
  run: npx jest --ci --coverage
```

E2E テストを含む本格的な CI/CD 自動化は、[Rork Max E2E テスト完全ガイド（Detox・Maestro）](/articles/rork-dev/rork-max-e2e-testing-detox-maestro-guide) で解説しています。

## まとめ

この記事では、Rork で生成した React Native アプリに **Jest と React Native Testing Library** を導入し、単体テストとコンポーネントテストを書く方法を解説しました。

ポイントをおさらいすると以下のとおりです。

- `jest-expo` プリセットで Expo プロジェクトのテスト環境を素早く構築できる
- ユーティリティ関数は `describe` + `test` で読みやすくシンプルに書ける
- コンポーネントは「ユーザー視点」で `getByText` / `getByRole` を使ってテストする
- 非同期処理は `waitFor` で確実に待ってからアサーションする
- カスタムフックは `renderHook` + `act` で状態変化を検証できる

テストを書く習慣は最初こそ時間がかかりますが、アプリが育つほど**「どこかが壊れていないか」という不安から解放されます**。Rork の AI 生成コードと組み合わせることで、スピードと品質を両立した開発体験を実現してみてください。

さらに実践的なテスト戦略（E2E テスト・CI 連携・本番環境モニタリング）は [Rork Max テスト完全ガイド](/articles/rork-dev/rork-max-testing-guide) で体系的に解説しています。テストを本格的に活用したい方はぜひご覧ください。
