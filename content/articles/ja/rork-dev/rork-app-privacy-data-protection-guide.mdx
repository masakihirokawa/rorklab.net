## はじめに — なぜプライバシー対策がアプリ開発の最優先事項なのか

アプリを公開する際に避けて通れないのが、ユーザーのプライバシー保護です。Apple は 2021 年に App Tracking Transparency（ATT）フレームワークを導入して以来、プライバシーに対する要求を年々強化しています。2026 年現在では、プライバシーラベルの正確な申告、ユーザーデータの最小限収集、そして各国の個人情報保護法への準拠が、App Store 審査を通過するうえで不可欠な要件となっています。

Rork でアプリを開発する場合も例外ではありません。むしろ AI が生成するコードの中に、意図せずユーザーデータを収集するロジックが含まれていないか確認する視点が大切です。この記事では、Rork アプリで押さえるべきプライバシー対策を体系的に整理し、実装からストア申請までの具体的な手順を解説します。

## App Tracking Transparency（ATT）の基本と実装

ATT は、ユーザーの IDFA（広告識別子）にアクセスする前に、明示的な許可を得ることを義務づけるフレームワークです。広告 SDK や分析ツールを導入している場合、ATT 対応は必須になります。

### ATT が必要なケース

- AdMob や Facebook Ads などの広告 SDK を組み込んでいる
- Firebase Analytics でユーザー属性を広告目的で利用している
- サードパーティの分析ツールがデバイス識別子を収集している

### ATT が不要なケース

- 広告を一切表示しない
- 分析データをデバイス識別子と紐づけていない
- ファーストパーティデータのみを利用している

### Rork アプリでの ATT 実装例

Rork で React Native ベースのアプリを作成している場合、`react-native-tracking-transparency` ライブラリを使うのが最もシンプルな方法です。

```typescript
// ATT許可リクエストの実装例
import { requestTrackingPermission, getTrackingStatus } from 'react-native-tracking-transparency';

const requestTracking = async () => {
  const status = await getTrackingStatus();

  if (status === 'not-determined') {
    // ユーザーにトラッキング許可ダイアログを表示
    const result = await requestTrackingPermission();
    console.log('Tracking permission:', result);
    // 期待する出力: 'authorized' | 'denied' | 'restricted' | 'not-determined'
  }

  return status;
};

// アプリ起動時に呼び出す
// 注意: 広告SDKの初期化はATT許可取得後に行うこと
```

ポイントは、ATT ダイアログを表示するタイミングです。アプリ起動直後に表示すると、ユーザーはまだアプリの価値を理解していないため拒否率が高くなります。数回使ってもらった後に、なぜトラッキングが必要なのかを説明してからダイアログを表示すると、許可率が向上することが知られています。

## App Store プライバシーラベルの正しい設定方法

App Store Connect でアプリを提出する際、プライバシーラベル（プライバシー栄養ラベルとも呼ばれます）の申告が求められます。これはアプリが収集するデータの種類と用途をユーザーに開示するものです。

### 申告が必要なデータカテゴリ

- **連絡先情報**: メールアドレス、電話番号、住所
- **ヘルスケア＆フィットネス**: 健康データ、歩数、心拍数
- **財務情報**: 支払い情報、クレジットカード情報
- **位置情報**: 正確な位置情報、おおまかな位置情報
- **ユーザーコンテンツ**: 写真、動画、テキスト入力
- **閲覧履歴**: アプリ内での閲覧パターン
- **識別子**: ユーザーID、デバイスID
- **使用状況データ**: アプリ起動回数、機能使用率
- **診断**: クラッシュログ、パフォーマンスデータ

### Rork アプリでよくあるパターン

Rork で生成したアプリが Supabase を使っている場合、認証で収集するメールアドレスは「連絡先情報」として申告が必要です。また、Firebase Analytics を導入していれば「使用状況データ」と「識別子」の申告が必要になります。

プライバシーラベルの申告内容が実際のアプリの動作と一致していないと、審査でリジェクトされます。Rork が生成したコードの中に、どのようなデータ収集ロジックが含まれているかを把握しておくことが重要です。

### 確認のためのチェックリスト

- **`info.plist` の使用目的キー**: カメラ、位置情報、マイクなどにアクセスする場合は、利用目的の説明文（`NSCameraUsageDescription` など）が必須
- **サードパーティ SDK の確認**: 導入している SDK がどのデータを収集するか、各 SDK のプライバシーマニフェストを確認する
- **サーバーへの送信データ**: API リクエストの中に不要な個人情報が含まれていないかレビューする

## ユーザーデータの安全な取り扱い — 最小限収集の原則

プライバシー対策の基本方針は「必要なデータだけを、必要な期間だけ収集する」ことです。これを「データ最小化の原則」と呼びます。

### 安全なデータ保存の実装パターン

```typescript
// ユーザーデータを安全に保存する例
import * as SecureStore from 'expo-secure-store';

// ✅ 良い例: 機密データはSecureStoreに保存
const saveAuthToken = async (token: string) => {
  await SecureStore.setItemAsync('auth_token', token);
  // SecureStoreはiOSのKeychainを利用するため暗号化される
};

// ❌ 悪い例: AsyncStorageに機密データを保存しない
// AsyncStorageは暗号化されていないプレーンテキストで保存される
// import AsyncStorage from '@react-native-async-storage/async-storage';
// await AsyncStorage.setItem('auth_token', token); // 危険！

// ✅ 良い例: 不要になったデータは速やかに削除
const clearUserData = async () => {
  await SecureStore.deleteItemAsync('auth_token');
  await SecureStore.deleteItemAsync('user_profile');
  console.log('User data cleared successfully');
  // 期待する出力: User data cleared successfully
};
```

アプリのセキュリティをさらに強化したい場合は、[Rork Max セキュリティ実装ガイド — Keychain とデータ保護](/articles/rork-dev/rork-max-security-keychain-guide)も参考にしてください。

### API 通信の保護

サーバーとの通信では、HTTPS は当然として、不要なデータをリクエストに含めない設計が大切です。

```typescript
// ✅ 良い例: 必要なフィールドのみ送信
const updateProfile = async (displayName: string) => {
  const response = await fetch('https://api.example.com/profile', {
    method: 'PATCH',
    headers: {
      'Content-Type': 'application/json',
      'Authorization': `Bearer ${await getToken()}`,
    },
    body: JSON.stringify({
      display_name: displayName,
      // 位置情報や端末情報は不要なら送らない
    }),
  });
  return response.json();
};
```

## GDPR・個人情報保護法への対応

アプリをグローバルに公開する場合、EU の GDPR（一般データ保護規則）や日本の個人情報保護法など、各地域のデータ保護法への準拠が必要です。

### GDPR で求められる主な対応

- **同意の取得**: データ収集前にユーザーの明示的な同意を得る
- **データアクセス権**: ユーザーが自分のデータを閲覧・ダウンロードできる仕組み
- **削除権（忘れられる権利）**: ユーザーが自分のデータの完全削除を要求できる仕組み
- **データポータビリティ**: ユーザーが自分のデータを別のサービスに移行できる形式で提供する

### Rork アプリでの実装ポイント

Supabase をバックエンドに使っている場合、ユーザーデータの削除は比較的シンプルに実装できます。

```typescript
// アカウント削除（GDPR対応）の実装例
const deleteUserAccount = async (userId: string) => {
  // 1. ユーザーのコンテンツを削除
  const { error: contentError } = await supabase
    .from('user_content')
    .delete()
    .eq('user_id', userId);

  // 2. ユーザープロフィールを削除
  const { error: profileError } = await supabase
    .from('profiles')
    .delete()
    .eq('id', userId);

  // 3. Supabase Authからユーザーを削除（Edge Functionから実行）
  // クライアントからは直接削除できないため
  // Edge Functionでservice_roleキーを使用する
  const { error: authError } = await supabase.functions.invoke(
    'delete-user',
    { body: { userId } }
  );

  if (!contentError && !profileError && !authError) {
    console.log('Account deleted successfully');
    // 期待する出力: Account deleted successfully
  }
};
```

Apple は 2022 年以降、アカウント作成機能を持つアプリにはアカウント削除機能の実装を義務づけています。Rork で認証機能を含むアプリを作る場合は、この機能を忘れずに組み込みましょう。

より詳しいセキュリティ実装パターンについては、[Rork Max エラーハンドリング・デバッグ・本番監視の実践パターン](/articles/rork-dev/rork-max-error-handling-debugging-production-monitoring)でも堅牢なアプリ構築のためのテクニックを解説しています。

## 子どものプライバシー保護 — COPPA 対応

アプリが13歳未満の子どもを対象にしている場合、米国の COPPA（児童オンラインプライバシー保護法）への対応が必要です。

### COPPA で求められる主な対応

- 保護者の同意なしに子どもの個人情報を収集しない
- 子ども向けの広告配信に制限を設ける
- 収集するデータの種類を必要最小限に抑える

App Store Connect で「子ども向け」カテゴリを選択すると、サードパーティの分析ツールやトラッキング SDK の使用が大幅に制限されます。Rork でプロンプトを書く際に子ども向けアプリであることを明記し、不要なデータ収集ロジックが生成されないようにすることが重要です。

## よくある質問（FAQ）

### Q1: Rork で生成したアプリに広告を入れていなくても ATT 対応は必要ですか？

広告 SDK を組み込んでおらず、IDFA を利用するサードパーティ SDK も使っていなければ、ATT 対応は不要です。ただし、Firebase Analytics の「広告ID」オプションを有効にしている場合は対応が必要になるため、設定を確認してください。

### Q2: プライバシーラベルの申告を間違えた場合はどうなりますか？

Apple の審査でリジェクトされる可能性があります。リジェクトされた場合、App Store Connect でプライバシーラベルを修正し、再提出すれば問題ありません。ただし、虚偽の申告を繰り返すとアカウントへの影響が出る可能性があるため、正確な申告を心がけましょう。

### Q3: GDPR 対応はグローバル公開しない場合も必要ですか？

日本国内のみの公開であれば GDPR への対応は法的には不要ですが、日本の個人情報保護法への対応は必要です。また、App Store はグローバルに公開されるため、将来的な海外展開を見据えてGDPR に準拠した設計にしておくことを推奨します。アカウント削除機能は Apple のガイドラインで必須化されているため、いずれにしても実装が必要です。

### Q4: Rork のプロンプトでプライバシーを意識するにはどうすればよいですか？

プロンプトに「ユーザーデータは最小限のみ収集し、機密情報は SecureStore に保存すること」「広告 SDK を使用しない場合は ATT を実装しないこと」などの指示を追加すると効果的です。生成されたコードをレビューし、不要なデータ収集がないかチェックする習慣をつけましょう。

プライバシー対策は、ユーザーからの信頼を得るための土台であり、App Store 審査を通過するための必須条件です。この記事で解説した ATT の適切な実装、プライバシーラベルの正確な申告、データ最小化の原則、GDPR 対応を押さえておけば、安心してアプリを公開できます。
