# Rorkでバックエンド連携：SupabaseとFirebaseの統合ガイド

現代的なアプリケーション開発には、認証、データストレージ、リアルタイム同期を処理できる堅牢なバックエンドインフラが不可欠です。Rorkは複数のバックエンドサービスをネイティブに統合できます。このガイドでは、最も人気のある2つのオプション、SupabaseとFirebaseについて説明します。

## Rorkのバックエンド選択肢の柔軟性

Rorkは複数のバックエンドプラットフォームをサポートしています：
- **Supabase**: PostgreSQL ベースで Row Level Security 対応
- **Firebase**: Google の NoSQL アプローチと Firestore
- **Airtable**: ラピッドプロトタイピング向けスプレッドシート型データベース
- **カスタム API**: REST または GraphQL エンドポイントの統合

この柔軟性により、プロジェクトの要件、チームの専門知識、スケーリング戦略に最適なバックエンドを選択できます。

## Supabase の統合

SupabaseはPostgreSQLをベースとしたオープンソースのFirebase代替です。構造化されたデータが必要で、複雑なクエリときめ細かいアクセス制御が必要なアプリケーションに最適です。

### Supabaseを選ぶ理由

**PostgreSQL基盤**: 完全な機能を備えたリレーショナルデータベースにACID準拠を備えています。テーブル、リレーションシップ、制約を含むスキーマを事前に定義します。

**Row Level Security（RLS）**: データベースレベルのセキュリティポリシーを実装できます。追加のアプリケーションロジックなしで、ユーザーは自分のデータのみの表示・変更が可能です。

**リアルタイムサブスクリプション**: Supabaseはビルトイン型のリアルタイム機能を提供します。データベースの変更をサブスクライブすると、更新が接続されたすべてのクライアントに即座に伝播します。

```
// 例：Rorkでのリアルタイムサブスクリプション
const subscription = supabase
  .from('tasks')
  .on('*', payload => {
    console.log('変更受信:', payload)
  })
  .subscribe()
```

2. **Rork でプロンプトインターフェイスを使用**: 「Supabase データベースに接続」または「Supabase バックエンドを追加」
3. **認証情報を入力**:
   - Project URL
   - Public anon key
   - Service role key（サーバー側操作用）

4. **スキーマを定義** Supabase SQL エディタまたはマイグレーションで
5. **RLS ポリシーを有効化** 各テーブルのデータアクセスルール実装に

### Supabaseの認証

Supabaseは複数の認証方法をサポートしています：

```
// メール/パスワードサインアップ
const { user, session, error } = await supabase.auth.signUp({
  email: 'user@example.com',
  password: 'password123'
})

// ソーシャルログイン（OAuth）
const { user, session, error } = await supabase.auth.signInWithOAuth({
  provider: 'github'
})
```

Supabaseはメール/パスワード、マジックリンク、OAuth（GitHub、Google、Discord）、エンタープライズアプリケーション向けSAMLをサポートしています。

## Firebase の統合

FirebaseはGoogleのスケーラブルなアプリケーション構築プラットフォームです。Firestoreを使用した異なるアプローチで、スキーマの事前定義が不要なNoSQLドキュメントデータベースです。

### Firebaseを選ぶ理由

**NoSQLの柔軟性**: Firestoreはドキュメントのコレクションとしてデータを保存します。テーブルやマイグレーションを事前作成する必要はありません。データ要件の変化に応じて動的にフィールドを追加できます。

**ジャストインタイムコレクション**: コレクションは初めてデータを書き込むときに作成されます。このラピッドプロトタイピングアプローチはスタートアップと高速で動くプロジェクトに最適です。

**Firebase認証**: メール/パスワード、電話認証、OAuth、匿名ログインをサポートしています。

**Cloud Functions**: データベースイベント、HTTPリクエスト、スケジュール済みタスクによってトリガーされるサーバーレス関数を記述できます。

```
// 例：RorkでのFirestoreクエリ
const tasksRef = collection(db, 'users', userId, 'tasks')
const q = query(tasksRef, where('completed', '==', false))
const snapshot = await getDocs(q)
```

### Rork で Firebase をセットアップする

1. **Firebase プロジェクトを作成** firebase.google.com で
2. **Firebase コンソールで Firestore を有効化**
3. **Rork でプロンプトインターフェイスを使用**: 「Firebase に接続」または「Firebase バックエンドを追加」
4. **Firebase 設定を入力**:
   - Project ID

5. **Firestore セキュリティルールを設定** データアクセスを制限
6. **Firebase コンソールで認証方法を有効化**

### Firebaseの認証

Firebaseは複数の認証オプションを提供しています：

```
// メール/パスワードログイン
const userCredential = await signInWithEmailAndPassword(
  auth,
  email,
  password
)

// Googleサインイン
const provider = new GoogleAuthProvider()
const result = await signInWithPopup(auth, provider)

// トライアルユーザー向け匿名ログイン
const result = await signInAnonymously(auth)
```

## Supabase vs Firebase の比較

| 機能 | Supabase | Firebase |
|------|----------|----------|
| データベースモデル | リレーショナル（PostgreSQL） | NoSQL（Firestore） |
| スキーマ定義 | 事前定義 | 動的/柔軟 |
| 複雑なクエリ | ネイティブ SQL サポート | クエリ機能限定 |
| リアルタイム | ビルトインサブスクリプション | リアルタイムリスナー |
| 認証 | 複数の方法 | 複数の方法 |
| コストモデル | 使用量ベース | 操作ごと |
| セルフホスティング | 可能 | Google Cloud のみ |

**Supabaseを選ぶべき場合**:
- データに複雑なリレーションシップがある
- 強力なクエリ機能が必要
- 構造化スキーマを好む
- セルフホスティングしたい

**Firebaseを選ぶべき場合**:
- ラピッドプロトタイピングが必要
- データ構造が頻繁に変わる
- Google Cloud と統合したい
- 完全に管理されたサービスを好む

## Rork のプロンプトで バックエンドを接続する

Rork の AI は、バックエンド設定の自然言語指示を理解します：

- 「ユーザーノート保存用の Supabase データベースを追加」
- 「リアルタイムメッセージング用 Firebase を接続」
- 「メール及び Google ログイン認証をセットアップ」
- 「製品インベントリ用 Firestore コレクションを作成」

AI は必要な設定コード、環境変数、統合ボイラープレートを自動生成します。

## ベストプラクティス

**1. 環境変数**: API キーをハードコードしないでください。開発時は `.env.local` を使用して機密認証情報を保存します。

```
SUPABASE_URL=your_url
SUPABASE_ANON_KEY=your_key
FIREBASE_PROJECT_ID=your_project
```

**2. 型安全性**: Supabase と Firebase はどちらも TypeScript をサポートしています。型を使用して開発時にエラーをキャッチします。

**3. データ検証**: クライアントとサーバーの両方でデータを検証します。フロントエンド検証のみに依存しないでください。

**4. セキュリティルール**: セキュリティルール または RLS ポリシー書き込みに時間をかけます。本番前に徹底的にテストしてください。

**5. モニタリング**: ロギングとモニタリングをセットアップし、認証イベント、データベースエラー、パフォーマンスの問題を追跡します。

**6. バックアップ戦略**: データを定期的にバックアップします。Supabase と Firebase の両方がバックアップソリューションを提供しています。

## まとめ

RorkはSupabaseとFirebaseをネイティブにサポートすることで、バックエンド統合を簡素化しています。Supabaseはリレーショナルデータ構造と複雑なクエリが必要な場合に優れており、FirebaseはラピッドプロトタイピングとNoSQLの柔軟性で輝きます。現在のニーズに合ったプラットフォームから始めてください。アプリケーションの進化に応じて、Rorkはバックエンド間の移行を簡単にします。
