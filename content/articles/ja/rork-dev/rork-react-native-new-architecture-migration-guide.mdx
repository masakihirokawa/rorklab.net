## はじめに — なぜ今 New Architecture が重要なのか

2024年10月にリリースされた React Native 0.76 で、長年開発が続けられてきた **New Architecture** がついにデフォルト有効化されましました。Expo SDK 52（2024年11月リリース）もこれに追随し、新規プロジェクトではすべて New Architecture が標準で適用されています。

Rork で生成されたアプリも Expo ベースで構成されているため、2025年以降に作成されたプロジェクトはほぼ確実に New Architecture 環境で動作しています。しかし、**「なぜ移行が必要なのか」「旧アーキテクチャと何が変わったのか」「既存アプリに何か影響があるか」**を正確に理解している開発者はまだ少ないのが現状です。

このガイドでは、New Architecture の設計思想から始まり、Rork アプリへの実践的な対応方法、互換性問題の診断と解決、さらにパフォーマンスを最大限に引き出すための最適化テクニックまでを体系的に解説します。

### 対象読者

このガイドは以下の方に向けています。

- Rork を使ってアプリを開発しており、ビルドエラーや動作異常の原因が New Architecture かどうか判断したい方
- 既存アプリを New Architecture に対応させて長期的な保守性を高めたい方
- React Native の内部アーキテクチャを深く理解し、パフォーマンスチューニングに活かしたい方

## New Architecture を構成する4つの技術

New Architecture は単一の変更ではなく、4つの相互補完的な技術の集合体です。

### 1. JSI（JavaScript Interface）

旧アーキテクチャでは、JavaScript と Native コードの間の通信に **非同期ブリッジ** が使われていましました。すべてのメッセージはシリアライズ（JSON化）されてキューに積まれ、非同期で処理されていたため、ネイティブ機能を呼び出すたびに必然的に遅延が生じていましました。

JSI はこのブリッジを廃止し、JavaScript エンジン（Hermes）が **C++ オブジェクトへの直接参照** を保持できる仕組みを提供します。これにより：

- JavaScript から Native への呼び出しが **同期的** に行える
- シリアライズのオーバーヘッドがゼロ
- JavaScript オブジェクトが Native メモリを直接参照できる（ゼロコピー）

```cpp
// JSI ネイティブバインディングの例（C++）
// JavaScript から直接呼び出せる関数を登録する
runtime.global().setProperty(
  runtime,
  "myNativeFunction",
  jsi::Function::createFromHostFunction(
    runtime,
    jsi::PropNameID::forAscii(runtime, "myNativeFunction"),
    1,
    [](jsi::Runtime& rt, const jsi::Value& thisVal,
       const jsi::Value* args, size_t count) -> jsi::Value {
      // ブリッジなしで直接実行される
      double result = args[0].asNumber() * 2;
      return jsi::Value(result);
    }
  )
);
// 期待する動作: JS側から myNativeFunction(21) を呼ぶと即座に 42 が返る
```

### 2. TurboModules

旧来の Native Modules は、アプリ起動時にすべてのモジュールが一括で初期化されていましました。カメラ、位置情報、Bluetooth など、実際には使っていないモジュールもすべてメモリに読み込まれるため、起動時間が長くなる原因になっていましました。

TurboModules は **遅延初期化**（Lazy Loading）を採用しています。

- モジュールは初めてアクセスされた時点で初期化される
- JSI 経由で型安全な非同期・同期の両インターフェースを提供
- TypeScript の型定義（Flow / TypeScript Spec）から Native コードが自動生成される（Codegen 連携）

```typescript
// TurboModule の TypeScript Spec 定義例
// このファイルから C++ / Swift / Java コードが自動生成される
import type { TurboModule } from 'react-native';
import { TurboModuleRegistry } from 'react-native';

export interface Spec extends TurboModule {
  // 同期メソッド - JSI により真の同期呼び出しが可能
  getDeviceId(): string;

  // 非同期メソッド - Promise を返す
  requestPermission(permissionType: string): Promise<boolean>;

  // コールバック形式
  addListener(eventType: string): void;
  removeListeners(count: number): void;
}

export default TurboModuleRegistry.getEnforcing<Spec>('MyTurboModule');
```

### 3. Fabric（新レンダラー）

Fabric は UI レンダリングパイプラインの全面刷新です。旧アーキテクチャでは、React の仮想DOM処理、Shadow Tree の計算、Native View の更新がそれぞれ別スレッドで非同期に連携していましました。

Fabric では：

- **C++ で統一された Shadow Tree**：JavaScript スレッド・UI スレッド・バックグラウンドスレッドから同一のデータ構造に安全にアクセスできる
- **同期的なレイアウト計算**：`onLayout` ハンドラなどでレイアウト情報を即座に取得できる
- **並行レンダリング対応**：React 18 の Concurrent Features（`useTransition`・`useDeferredValue`）が真の意味で機能する

```typescript
// Fabric の同期レイアウト測定例
// 旧アーキテクチャでは非同期でしか取得できなかった値が
// New Architecture では ref から同期的に取得できる
import { useRef, useCallback } from 'react';
import { View, Text } from 'react-native';

export function MeasurableComponent() {
  const viewRef = useRef<View>(null);

  const handlePress = useCallback(() => {
    // measureInWindow が即座に結果を返す（コールバック不要）
    viewRef.current?.measureInWindow((x, y, width, height) => {
      console.log(`位置: x=${x}, y=${y}, サイズ: ${width}x${height}`);
      // 期待する出力例: 位置: x=16, y=84, サイズ: 343x48
    });
  }, []);

  return (
    <View ref={viewRef}>
      <Text onPress={handlePress}>位置を測定する</Text>
    </View>
  );
}
```

### 4. Codegen（コード自動生成）

Codegen は TypeScript / Flow で書かれた型定義ファイルから、Native コードのインターフェース（C++ ヘッダー、Java/Kotlin、Swift/ObjC のスタブ）を自動生成するツールです。

- 手書きのブリッジコードが不要になり、型の不一致によるランタイムクラッシュがゼロに
- ビルド時に型安全性が検証される
- Native モジュールの実装が大幅に簡素化される

## Rork 生成アプリでの New Architecture 確認と設定

### 現在の状態を確認する

Rork で生成されたアプリが New Architecture で動作しているかを確認するには、`app.json`（または `app.config.js`）を見ます。

```json
// app.json - Expo SDK 52 以降の新規プロジェクト
{
  "expo": {
    "name": "MyApp",
    "slug": "my-app",
    "sdkVersion": "52.0.0",
    "newArchEnabled": true
  }
}
```

Expo SDK 52 以降では `newArchEnabled` のデフォルト値が `true` になっています。明示的に記載がなくても、SDK 52+ であれば New Architecture が有効です。

```bash
# 現在の Expo SDK バージョンを確認するコマンド
cat package.json | grep '"expo"'
# 出力例: "expo": "^52.0.0"

# react-native のバージョン確認
cat package.json | grep '"react-native"'
# 出力例: "react-native": "0.76.5"
```

### 一時的に旧アーキテクチャを使う方法

互換性の問題が解決するまで一時的に旧アーキテクチャを使いたい場合は `app.json` で無効化できます。ただし、この設定は将来的には廃止される予定のため、長期的な解決策にはなりません。

```json
{
  "expo": {
    "newArchEnabled": false
  }
}
```

## 互換性チェック — 使用ライブラリを診断する

New Architecture への移行で最も多い問題は、**サードパーティライブラリの未対応** です。

### 公式互換性チェックツールの使い方

```bash
# react-native-community/directory の互換性チェック
npx react-native-community-cli check-android
npx react-native-community-cli check-ios

# または、依存関係を一括スキャン
npx @react-native-community/upgrade-helper
```

### よくある互換性問題とその対処法

**問題1: 旧来の NativeModules / NativeEventEmitter を使うライブラリ**

旧形式の Native Modules は New Architecture でも Interop Layer を経由して動作しますが、パフォーマンス上の恩恵は受けられません。

```typescript
// ❌ 旧来の NativeModules アクセス（非推奨）
import { NativeModules } from 'react-native';
const { MyLegacyModule } = NativeModules;

// ✅ New Architecture 対応のアクセス方法
import { TurboModuleRegistry } from 'react-native';
const MyTurboModule = TurboModuleRegistry.get('MyTurboModule');
```

**問題2: `react-native-camera`（旧来の camera ライブラリ）**

New Architecture 未対応の `react-native-camera` は `expo-camera` または `react-native-vision-camera` v4 に移行します。

```bash
# 旧ライブラリの削除
npx expo uninstall react-native-camera

# New Architecture 対応の代替ライブラリをインストール
npx expo install expo-camera
```

**問題3: Reanimated v3 未満のバージョン**

`react-native-reanimated` は v3 から New Architecture に完全対応しています。

```bash
# バージョン確認
cat package.json | grep reanimated
# 出力例: "react-native-reanimated": "^3.16.0"

# v3 未満の場合はアップグレード
npx expo install react-native-reanimated@latest
```

### New Architecture 対応状況の一覧（主要ライブラリ）

主要ライブラリの対応状況を整理します。

- **対応済み**: react-navigation v7、react-native-reanimated v3+、expo-camera、react-native-gesture-handler v2+、react-native-mmkv v3+、@shopify/flash-list
- **Interop Layer 経由で動作**: react-native-svg（v15未満）、react-native-maps（v1.10未満）
- **要注意（独自確認が必要）**: 長期間メンテナンスされていないコミュニティライブラリ

## ビルドエラーのトラブルシューティング

New Architecture 移行時によく発生するビルドエラーとその解決策を示します。

### iOS ビルドエラー: `Undefined symbol: _OBJC_CLASS_$_RCTBridge`

旧来のブリッジクラスへの参照が残っているライブラリが原因です。

```bash
# Podfile.lock をクリアして再インストール
cd ios
pod deintegrate
pod install --repo-update
cd ..

# キャッシュクリア
npx expo prebuild --clean
```

### Android ビルドエラー: `Cannot find symbol: class ReactPackage`

```kotlin
// ❌ 旧来の ReactPackage 実装（New Architecture では非推奨）
class MyPackage : ReactPackage {
    override fun createNativeModules(reactContext: ReactApplicationContext)
        = listOf(MyModule(reactContext))
}

// ✅ TurboReactPackage への移行
class MyPackage : TurboReactPackage() {
    override fun getModule(name: String, reactContext: ReactApplicationContext)
        = if (name == MyTurboModule.NAME) MyTurboModule(reactContext) else null
}
```

### Metro バンドラーエラー: `Cannot use 'import' outside a module`

`package.json` の `"type": "module"` が設定されている場合に発生します。

```json
// metro.config.js を更新
const { getDefaultConfig } = require('expo/metro-config');
const config = getDefaultConfig(__dirname);

// sourceExts に mjs を追加
config.resolver.sourceExts.push('mjs');

module.exports = config;
```

## パフォーマンス測定と最適化

New Architecture の恩恵を最大限に受けるためのテクニックを紹介します。

### React DevTools Profiler での計測

```typescript
// パフォーマンス計測用のユーティリティ
import { performance } from 'react-native';

export function measureRender(label: string, fn: () => void): void {
  const start = performance.now();
  fn();
  const end = performance.now();
  console.log(`[${label}] レンダリング時間: ${(end - start).toFixed(2)}ms`);
  // 期待する出力例: [MyComponent] レンダリング時間: 2.34ms
}
```

### Concurrent Features の活用

React 18 の Concurrent Features は New Architecture でのみ真の効果を発揮します。

```typescript
import { useState, useTransition, useDeferredValue } from 'react';
import { FlatList, TextInput, Text, View } from 'react-native';

interface Item {
  id: string;
  name: string;
}

export function SearchableList({ data }: { data: Item[] }) {
  const [query, setQuery] = useState('');
  const [isPending, startTransition] = useTransition();

  // 重い絞り込み処理を低優先度で実行
  const deferredQuery = useDeferredValue(query);

  const filteredData = data.filter(item =>
    item.name.toLowerCase().includes(deferredQuery.toLowerCase())
  );

  return (
    <View>
      <TextInput
        value={query}
        onChangeText={text => {
          // 入力の更新は高優先度（即座に反映）
          setQuery(text);
        }}
        placeholder="検索..."
      />
      {/* isPending: true の間は前の結果を薄く表示 */}
      <FlatList
        data={filteredData}
        style={{ opacity: isPending ? 0.5 : 1 }}
        keyExtractor={item => item.id}
        renderItem={({ item }) => <Text>{item.name}</Text>}
      />
    </View>
  );
}
```

### JSI を活用したアニメーションの最適化

```typescript
import Animated, {
  useSharedValue,
  useAnimatedStyle,
  withSpring,
  runOnJS,
} from 'react-native-reanimated';

// Reanimated v3 + Fabric による UI スレッド直接アニメーション
// JS スレッドのブロックなしにスムーズな 60fps アニメーションを実現
export function OptimizedCard() {
  const scale = useSharedValue(1);
  const opacity = useSharedValue(1);

  const animatedStyle = useAnimatedStyle(() => ({
    transform: [{ scale: scale.value }],
    opacity: opacity.value,
  }));

  const handlePress = () => {
    // worklet: UI スレッドで直接実行（JS スレッドの関与ゼロ）
    scale.value = withSpring(0.95, { damping: 15 }, () => {
      scale.value = withSpring(1);
    });
  };

  return (
    <Animated.View style={[{ padding: 20, backgroundColor: '#fff' }, animatedStyle]}>
      <Animated.Text onPress={handlePress}>タップしてください</Animated.Text>
    </Animated.View>
  );
}
```

## よくあるエラーと対処法

New Architecture 移行後に発生しやすいランタイムエラーのパターンと解決策を紹介します。

**エラー1: `TypeError: null is not an object (evaluating 'module.default')`**

CommonJS 形式のモジュールが ESM 環境で正しく読み込まれない場合に発生します。

```javascript
// metro.config.js に以下の設定を追加
config.resolver.unstable_enablePackageExports = true;
config.resolver.unstable_conditionNames = ['require', 'default'];
```

**エラー2: `Invariant Violation: No suitable renderer found for...`**

Fabric 未対応のカスタムコンポーネントが原因です。`UIManager` への直接アクセスを避け、`refs` を通じたアクセスに変更します。

```typescript
// ❌ UIManager への直接アクセス（非推奨）
import { UIManager, findNodeHandle } from 'react-native';
UIManager.measure(findNodeHandle(ref.current), callback);

// ✅ ref の measureInWindow を使用
ref.current?.measureInWindow((x, y, width, height) => {
  // Fabric 対応の計測方法
});
```

**エラー3: カスタム Native モジュールがクラッシュする**

Interop Layer はすべての旧形式モジュールをサポートしているわけではありません。特に、コールバックを複数回呼び出すモジュールは問題が起きやすいです。

```typescript
// ❌ コールバックを複数回呼び出す（New Architecture では禁止）
RCT_EXPORT_METHOD(streamData:(RCTResponseSenderBlock)callback) {
  callback(@[@"first"]);
  callback(@[@"second"]); // クラッシュの原因！
}

// ✅ EventEmitter を使用して複数の値を送信する
- (void)startStream {
  [self sendEventWithName:@"dataReceived" body:@{@"value": @"first"}];
  [self sendEventWithName:@"dataReceived" body:@{@"value": @"second"}];
}
```

## 移行チェックリスト

移行作業を安全に進めるためのチェックリストです。

**移行前の準備**

- Expo SDK を 52 以上（または React Native を 0.76 以上）にアップグレード
- 使用中のすべての Native モジュールの New Architecture 対応状況を確認
- `main` ブランチとは別の `feature/new-architecture` ブランチを作成

**移行作業**

- `app.json` に `"newArchEnabled": true` を追加（または確認）
- `pod install` を実行（iOS の CocoaPods を更新）
- Android の `gradle.properties` で `newArchEnabled=true` を確認
- `npx expo prebuild --clean` でネイティブコードを再生成

**検証**

- iOS シミュレーターでフル機能テストを実施
- Android エミュレーターでフル機能テストを実施
- アニメーション・ジェスチャー・カメラ・位置情報の動作確認
- クラッシュが発生した場合、Hermes のスタックトレースで原因を特定

## FAQ

**Q. Rork で生成したアプリは自動的に New Architecture に対応していますか？**

A. Rork は Expo ベースでプロジェクトを生成するため、Expo SDK 52 以降を使用している場合は New Architecture がデフォルトで有効になっています。ただし、Rork がインストールするサードパーティライブラリがすべて対応しているとは限りません。`app.json` で `"newArchEnabled": true` を確認し、問題が発生した場合は本ガイドのトラブルシューティングを参照してください。

**Q. 旧アーキテクチャはいつまで使えますか？**

A. React Native チームの公式発表では、旧アーキテクチャのサポートは段階的に縮小される予定です。2026年時点では Interop Layer により多くの旧ライブラリが動作しますが、新機能（Concurrent Features、JSI 直接バインディングなど）は New Architecture でのみ利用可能です。新規プロジェクトはもちろん、既存プロジェクトも早期の移行を強く推奨します。

**Q. New Architecture にすると必ずパフォーマンスが向上しますか？**

A. 必ずしも即座に体感できるわけではありません。特に軽量なアプリでは差異が少ない場合もあります。ただし、リスト描画の大量データ処理、複雑なアニメーション、JavaScript とネイティブ間の高頻度通信が多いアプリでは顕著な改善が見られます。React DevTools Profiler や Xcode Instruments でプロファイリングを行い、改善効果を定量的に測定することをお勧めします。

**Q. カスタム Native モジュールを持つ既存アプリの移行はどう進めますか？**

A. まず Interop Layer で動作するかを確認します。動作しない場合は、TurboModule 形式への移行が必要です。TypeScript で Spec ファイルを作成し、Codegen で Native スタブを生成してから実装を移行する手順になります。公式ドキュメントの「Migrating a Legacy Native Module to TurboModule」を参考にしてください。移行作業が複雑な場合は、まず `newArchEnabled: false` で安定させながら段階的に対応することを検討してください。

## まとめ

React Native New Architecture は、5年以上にわたる大規模な再設計の成果であり、**JSI によるブリッジレス通信**、**TurboModules の遅延ロード**、**Fabric の同期レンダリング**、**Codegen による型安全なバインディング**という4つの柱で構成されています。

Rork アプリ開発者にとって重要なポイントをまとめると：

- Expo SDK 52+ / React Native 0.76+ ではデフォルト有効
- 主要ライブラリの大部分は既に対応済み
- 未対応ライブラリの多くは Interop Layer で動作するが、長期的には移行が必要
- Concurrent Features（`useTransition`・`useDeferredValue`）と Reanimated v3 の組み合わせで圧倒的なパフォーマンスを実現できる

New Architecture を深く理解することで、ユーザーに快適な体験を届けるアプリを継続的に維持・改善していくことができます。移行に際して不安な点があれば、まず開発環境でのテストを丁寧に行い、一歩ずつ確認しながら進めることをお勧めします。

New Architecture への移行後は、[Rork アプリのパフォーマンス最適化 — レンダリング・メモリ・ネットワークの実践テクニック](/articles/rork-dev/advanced-rork-app-performance-optimization) で紹介しているプロファイリング手法と組み合わせることで、より具体的な改善効果を測定できます。また、[Rork Max × EAS CI/CD 完全ガイド](/articles/rork-dev/rork-max-eas-cicd) のビルドパイプラインに New Architecture のビルド検証を組み込むことで、デプロイのたびに互換性を自動チェックできる体制が整います。さらに、[Rork Maxアプリのテスト完全ガイド](/articles/rork-dev/rork-max-testing-guide) を参考にして E2E テストを New Architecture 環境でも実行することで、移行後の品質を継続的に担保できます。

React Native のアーキテクチャをより体系的に学びたい方には、[React Native アプリ設計の実践（Evan Bacon、Brent Vatne 著）](https://www.amazon.co.jp/dp/4839973453/?tag=pinocchio-22) がおすすめです。コアメンテナーによる知見が豊富に収録されており、New Architecture の理解を深める上でも非常に役立ちます。
