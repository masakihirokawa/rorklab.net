リリースした翌朝、サポートメールが届いていました。「決済ボタンが押せない」という報告が複数件。コードを確認したら、前日のコミットで分岐条件を一行間違えていました。

修正自体は5分もかかりません。しかし App Store の審査には最短でも24時間、長いときは数日かかります。その間ずっと、ユーザーが壊れたアプリを使い続けることになります。

EAS Update（OTA アップデート）を使っていれば、この状況は完全に変わります。JavaScript/TypeScript のコード変更であれば、審査なしでユーザーのデバイスに即座に届けることができます。

## EAS Update とは何か

EAS Update は、Expo が提供するオーバー・ザ・エア（OTA）アップデート配信サービスです。React Native アプリの JavaScript バンドルとアセット（画像・フォントなど）を、App Store や Google Play の審査を経ずにユーザーへ届けることができます。

Rork で作ったアプリも React Native / Expo ベースなので、EAS Update がそのまま機能します。

### 何が更新できて、何が更新できないか

EAS Update で更新できるのは JavaScript/TypeScript のコードと静的アセットのみです。ネイティブコード（Swift・Kotlin・C++）の変更や、新しい権限の追加、ネイティブモジュールの追加には通常のアプリ審査が必要です。

- ✅ ロジックのバグ修正
- ✅ UI テキスト・色・レイアウトの調整
- ✅ API エンドポイントの変更
- ✅ 画像・フォントの差し替え
- ✅ ライブラリの JavaScript 部分のアップデート
- ❌ カメラ・マイクなど新しい権限の追加
- ❌ ネイティブモジュールの変更
- ❌ アプリアイコン・スプラッシュスクリーンの変更（ネイティブ側）

日常的なバグ修正やテキスト修正の大半は EAS Update でカバーできます。

## EAS Update の仕組み

EAS Update は「チャンネル」と「ブランチ」という概念で更新を管理します。

```
ビルド（アプリ本体）
  └─ チャンネル: production  ← App Store / Google Play のユーザーが見る
  └─ チャンネル: staging     ← テスターが見る（TestFlight / 内部テスト）

チャンネルはブランチを参照する
ブランチ: main              ← 最新の更新が入る場所
```

ユーザーがアプリを起動すると、アプリは EAS のサーバーに接続し「自分のチャンネルに新しい更新があるか」を確認します。更新があれば JavaScript バンドルをバックグラウンドでダウンロードし、次回起動時（または即時）に適用します。

ユーザー側では何も操作が不要です。アプリを再起動するだけで最新版になります。

## セットアップ手順

### 1. Expo EAS CLI のインストール

```bash
npm install -g eas-cli

# ログイン（Expo アカウントが必要）
eas login
```

### 2. app.json の設定

Rork で作ったアプリの `app.json` に以下を追加します。

```json
{
  "expo": {
    "name": "Your App Name",
    "slug": "your-app-slug",
    "runtimeVersion": {
      "policy": "sdkVersion"
    },
    "updates": {
      "url": "https://u.expo.dev/YOUR_PROJECT_ID",
      "enabled": true,
      "fallbackToCacheTimeout": 0,
      "checkAutomatically": "ON_LOAD"
    }
  }
}
```

`YOUR_PROJECT_ID` は `eas init` を実行すると自動で設定されます。

```bash
# プロジェクトを EAS に登録
eas init
```

### 3. eas.json にチャンネルを設定

`eas.json` の各プロファイルにチャンネルを追加します。

```json
{
  "build": {
    "development": {
      "developmentClient": true,
      "distribution": "internal",
      "channel": "development"
    },
    "staging": {
      "distribution": "internal",
      "channel": "staging"
    },
    "production": {
      "distribution": "store",
      "channel": "production"
    }
  }
}
```

チャンネルを分けることで、テスターには staging チャンネルの更新を、一般ユーザーには production チャンネルの更新だけを届けることができます。

### 4. 更新のデプロイ

コードを修正したあと、以下のコマンド一発でデプロイできます。

```bash
# staging チャンネルに更新を配信
eas update --channel staging --message "決済ボタンの修正"

# 問題なければ production にも配信
eas update --channel production --message "決済ボタンの修正 (本番)"
```

出力例:
```
✔ Published\!
  Channel: production
  Branch: main
  Update ID: abc123...
  Message: 決済ボタンの修正 (本番)
```

このコマンドが完了してから数分以内に、ユーザーのデバイスに更新が届き始めます。

## チャンネル管理とロールバック

### 更新の確認

```bash
# production チャンネルの更新履歴を確認
eas update:list --channel production
```

### ロールバック

問題のある更新を配信してしまった場合でも、以前のバージョンに即座に戻せます。

```bash
# 特定の更新 ID にロールバック
eas update:republish --channel production --update-id PREVIOUS_UPDATE_ID
```

これが EAS Update の最大の強みのひとつです。アプリのバイナリを再ビルドする必要なく、5分以内に問題のある更新を取り消せます。

### ステージングで確認してから本番に反映

```bash
# まず staging に配信してテスト
eas update --channel staging --message "新機能: プロフィール画像アップロード"

# テスト完了後、同じブランチを production に向ける
eas channel:edit production --branch main
```

または、staging で確認した特定の更新を production に再公開する方法もあります。

```bash
eas update:republish \
  --source-channel staging \
  --channel production \
  --update-id STAGING_UPDATE_ID
```

## よくある落とし穴と対処法

EAS Update を導入してから実際に遭遇した問題をまとめます。

### ① runtimeVersion の不一致でアップデートが届かない

最も多いトラブルです。ネイティブコードを変更してアプリをリビルドした場合、`runtimeVersion` が変わります。古いビルドのユーザーには新しい更新が届きません。

```json
// ❌ 手動で固定していると更新漏れが起きやすい
"runtimeVersion": "1.0.0"

// ✅ SDK バージョンに連動させる（推奨）
"runtimeVersion": {
  "policy": "sdkVersion"
}
```

`sdkVersion` ポリシーを使うと、Expo SDK のバージョンが変わったときだけ `runtimeVersion` が更新されます。意図せず runtimeVersion が上がることを防げます。

### ② `checkAutomatically: "ON_LOAD"` と UX のバランス

デフォルト設定ではアプリ起動時に更新を確認し、バックグラウンドでダウンロードします。更新は**次回起動時**に適用されます。

緊急のバグ修正を即座に反映したい場合は、`expo-updates` の `reloadAsync()` を使って強制的に再読み込みすることもできます。

```typescript
import * as Updates from 'expo-updates';

async function checkAndApplyUpdate() {
  try {
    const update = await Updates.checkForUpdateAsync();
    if (update.isAvailable) {
      await Updates.fetchUpdateAsync();
      // ユーザーに再起動を促すダイアログを表示してから実行
      await Updates.reloadAsync();
    }
  } catch (error) {
    // 更新チェック失敗時は既存のバージョンで続行（サイレント処理）
    console.warn('Update check failed:', error);
  }
}
```

ただし `reloadAsync()` はアプリが突然再起動するためユーザー体験を損ないます。深刻なバグでない限り、次回起動時適用（デフォルト動作）のほうが自然です。

### ③ 開発環境で更新が反映されない

Expo Go や Development Build では EAS Update が動作しません（意図的な仕様です）。

実際の更新配信をテストするには `staging` ビルドを TestFlight / 内部テストトラックで配布し、そちらで確認してください。

## 本番運用のベストプラクティス

実際に数ヶ月運用してわかったことをまとめます。

**コミットメッセージと同じ粒度でアップデートメッセージを書く**。EAS Update のダッシュボードで過去の更新を見返すとき、「バグ修正」だけでは何を修正したか追えません。「決済フロー: ローカル変数の初期化タイミング修正 (#123)」のように具体的に書くと、ロールバック判断が速くなります。

**必ず staging で一定時間テストしてから production に配信する**。staging で自分のデバイスに入れて15分使ってから本番配信するルールを設けるだけで、重大なリグレッションをほぼ防げます。

**GitHub Actions と組み合わせて自動化する**。`main` ブランチへのマージをトリガーに自動で staging チャンネルへデプロイする設定は、[GitHub Actions と EAS Build CI/CD 自動化ガイド](/articles/rork-dev/rork-github-actions-eas-build-cicd) を参考にしてください。

**更新ログをチームで共有する**。一人開発でも、Slack や Discord に `eas update` の結果を自動投稿しておくと、「いつどんな修正をしたか」の記録が残ります。

EAS Update を使い始めてから、リリース後のストレスが大幅に減りました。完璧なコードは存在しません。問題が起きたとき、5分で直せる環境を整えておくことが個人開発を長く続けるコツだと感じています。

Rork Companion での実機テストの方法については [Rork Companion iOS 実機テストガイド](/articles/rork-basics/rork-companion-ios-real-device-test-guide) も合わせてご覧ください。
