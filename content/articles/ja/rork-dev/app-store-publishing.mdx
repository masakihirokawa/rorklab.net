# Rorkで作ったアプリをApp StoreとGoogle Playに公開する方法

Rorkアプリケーションをアプリストアに公開することは、ユーザーに到達するための重要なステップです。このガイドはアプリストアアカウント設定、ビルド準備、提出、発表後の管理を含む、Apple App StoreとGoogle Play Storeの完全な公開プロセスを実説します。

## Rork オリジナル vs Rork Max

Rorkは異なる機能と価格設定の2つの公開パスを提供します。

### Rork オリジナル

Rork オリジナルは、提出プロセスに対する完全な制御を備えてApp StoreとGoogle Play Storeの両方に公開できます。デベロッパーアカウント、提出、アプリストアページを直接管理します。

**機能**:
- iOS（App Store）と Android（Google Play）の両方に公開
- アプリストアリスティングの完全なカスタマイズ
- 更新スケジューリングの直接的な制御
- プラットフォーム制限なし

**要件**:
- Apple Developer Program メンバーシップ（99ドル/年）
- Google Play Developer アカウント（25ドル一回限り）
- コード生成用 Rork アカウント

### Rork Max

Rork Maxは2クリックでシンプルな公開プロセスを簡素化します。しかし、現在はApple App Storeの公開のみをサポートしています。

**機能**:
- 2クリック App Store 公開
- Rork が提出ロジスティクスを処理
- シンプルなバージョン管理
- Apple 最適化リリースプロセス

**制限**:
- App Store のみ（iOS）
- Rork Original より高い 1 アプリあたりコスト
- ストアリスティングのカスタマイズが少ない

**推奨**: iOS と Android の両方の存在が必要な場合は Rork Original を使用します。簡単な iOS 公開が必要で Android が不要な場合は Rork Max を使用します。

## デベロッパーアカウントのセットアップ

### Apple Developer Program

1. **developer.apple.com にアクセス** して「Account」をクリック
2. **Apple ID でサインイン** するか、新しいアカウントを作成
3. **Apple Developer Program に登録** （99ドル/年）
4. **デベロッパープロフィールを完成** させる（企業情報）
5. **アプリ開発の法的同意書に同意**
6. **承認を待つ** （通常即座、時々24～48時間）

承認後、以下ができます：
- アプリ識別子を作成
- 証明書とプロビジョニングプロファイルをセットアップ
- 開発および配布証明書を管理

### Google Play Console

1. **play.google.com/console にアクセス** して Google アカウントでサインイン
2. **新しい開発者アカウントを作成** （25ドル一回限り）
3. **デベロッパープロフィールをセットアップ** （名前と連絡先情報）
4. **Google Play ポリシーに同意** してデベロッパー契約に同意
5. **Google Wallet アカウントを作成** （支払い用）

### 証明書とプロビジョニングファイル

iOS アプリの場合：
- **開発証明書**: テストと Expo Go 開発中に使用
- **配布証明書**: App Store 提出に必須
- **プロビジョニングプロファイル**: アプリ ID、証明書、デバイスをリンク

Android アプリの場合：
- **アップロードキー**: Google Play 用アプリに署名するために使用
- **アプリ署名キー**: セキュリティ用に Google Play が管理

Rork と EAS（Expo Application Services）は、これらのファイルを自動的に生成・管理できます。

## ビルド準備

アプリを提出する前に本番対応であることを確認します：

### コードレビュー

1. **iOS と Android デバイスで徹底的にテスト**
2. **プラットフォーム固有の問題をチェック** （パーミッション、戻るボタン、セーフエリア）
3. **すべてのユーザーフローをテスト** （サインアップから主要フィーチャーまで）
4. **パフォーマンステスト** 古いデバイスでスムーズな 60 FPS を確保
5. **メモリテスト** 長期使用でリークをチェック

### アプリ設定

`app.json` で正しいメタデータを更新します：

```json
{
  "expo": {
    "name": "My Awesome App",
    "slug": "my-awesome-app",
    "version": "1.0.0",
    "description": "A description of your app",
    "icon": "./assets/icon.png",
    "splash": {
      "image": "./assets/splash.png",
      "resizeMode": "contain",
      "backgroundColor": "#ffffff"
    },
    "ios": {
      "bundleIdentifier": "com.yourcompany.myapp",
      "supportsTabletAndPhone": true
    },
    "android": {
      "package": "com.yourcompany.myapp",
      "versionCode": 1
    }
  }
}
```

### バージョン番号

- **iOS**: セマンティックバージョニング使用（1.0.0、1.0.1、1.1.0）
- **Android**: versionCode（内部、自動インクリメント）と versionName（ユーザーに表示）の両方を使用

各更新はバージョン番号のインクリメントが必須です。

## EAS でビルドする

```bash
eas build --platform ios
eas build --platform android
```

EAS は React Native コードをネイティブバイナリにコンパイルします：
- すべてのネイティブコードコンパイルを処理
- 証明書でのコード署名を管理
- `.ipa`（iOS）および `.aab`（Android）ファイルを生成
- EAS ダッシュボードにビルドを保存（簡単アクセス）

## App Store に提出する（iOS）

### アプリレコードを作成

1. **App Store Connect にログイン** （appstoreconnect.apple.com）
2. **「My Apps」をクリック** して **「+ New App」**
3. **iOS をプラットフォームとして選択**
4. **アプリ情報を入力**:
   - App Name
   - Bundle ID（app.json の `bundleIdentifier` と一致する必要があります）
   - SKU（あなたの記録用の一意の識別子）
   - Primary Category
   - Secondary Category（オプション）

### アプリ情報を入力

**価格と利用可能性**:
- 価格を設定（無料または層を選択）
- アプリが利用可能な地域を選択
- 利用可能日を選択

**ローカライズ**:
- 各言語向けにアプリプレビュー、スクリーンショット、説明を追加
- 高品質な 5.5 インチ iPhone スクリーンショットを使用
- 説得力があり、正直な説明を書く

**アプリプライバシー**:
- データ収集実践を宣言
- 分析、広告、トラッキング実装をリスト
- プライバシー実践について正直に

### 提出準備

1. **ビルド番号**: まず TestFlight でビルドを作成
   - Xcode または EAS でビルドをアップロード
   - 処理を待つ（数分かかる場合があります）

2. **一般アプリ情報**:
   - Category
   - Content rating questionnaire
   - Age rating（正確に入力）

3. **アプリレビュー情報**:
   - デモアカウント認証情報（アプリがログイン機能を持つ場合）
   - レビュアー向けメモ
   - 連絡先情報
   - ウェブサイト URL とプライバシーポリシー

### レビュー向けて提出

1. **すべての必須フィールドを完成** （赤いアスタリスク付き）
2. **アプリ情報をレビュー** （正確性）
3. **App Store Review Guidelines に同意**
4. **「Submit for Review」をクリック**
5. **レビューを待つ** （通常 24～48 時間）

## Google Play に提出する

### Google Play Console でアプリを作成

1. **Google Play Console に移動** してサインイン
2. **「Create App」をクリック**
3. **アプリ名を入力** し、デフォルト言語を選択
4. **コンテンツに関する宣言に同意**

### ストアリスティングを入力

1. **スクリーンショットを追加**:
   - 電話向けに 4～8 個のスクリーンショットをアップロード
   - ランドスケープおよびタブレットスクリーンショットはオプション
   - マーケティング画像ではなく、実際のアプリスクリーンショットを使用

2. **アプリ説明を書く**:
   - Short description（80 文字以内）
   - Full description（4,000 文字以内）
   - 機能と利点を明確にする

3. **アプリアイコンを追加**:
   - 512x512 PNG 画像
   - テキストなし、きれいなアイコンデザインのみ

4. **カテゴリとコンテンツレーティングを選択**:
   - Primary category
   - Content rating questionnaire

### アプリ署名を設定

Google Play は配布用署名を処理します：
- 署名された AAB（Android App Bundle）をアップロード
- Google Play は各デバイス構成の最終 APK に署名
- 手動署名との比較でセキュリティが向上

### リリースを管理

1. **「Release」セクションに移動**
2. **本番トラックで新しいリリースを作成**
3. **署名された AAB を追加** （EAS で生成）
4. **このバージョンのリリースノートを書く**
5. **ロールアウト割合を選択**:
   - 5～10% から開始（テスト）
   - クラッシュレポートを監視
   - 1 週間かけて 100% に徐々に増加

### レビュー向けて提出

Google Play は Apple より厳しくないレビュー：
- 通常数時間以内に承認
- 自動チェックが即座に実行
- ポリシー違反の場合は手動レビュー

## よくある却下理由

### App Store の却下

1. **クラッシュやパフォーマンスの問題**: 実デバイスで徹底的にテスト
2. **誤解を招くまたは不完全なメタデータ**: スクリーンショットと説明がアプリ機能と一致していることを確認
3. **アプリプライバシーポリシーが不足**: シンプルなアプリでもプライバシーポリシーが必須
4. **機能が動作しない**: 壊れた機能を含めない
5. **iOS ガイドラインに違反**: iOS Human Interface Guidelines に従う
6. **API の未認可使用**: Apple が承認した API のみを使用
7. **重複機能**: 同じアプリのバリエーションを提出しない

### Google Play の却下

1. **欺くような動作**: アプリは主張する内容を実行する必要があります
2. **知的財産権侵害**: 著作権と商標を尊重
3. **マルウェアまたはセキュリティの問題**: セキュリティ問題は即座削除
4. **成人向けコンテンツ**: 適切に評価され制限される必要があります
5. **スパム行為**: 過度の広告または誤解を招くコンテンツ
6. **壊れたダウンロードリンク**: すべての外部リンクが機能する必要があります

## アプリメタデータの準備

### スクリーンショット

- マーケティングモックアップではなく実アプリスクリーンショットを使用
- アプリの主要フィーチャーと利点を表示
- App Store: 言語ごとに 5～8 個のスクリーンショット
- Google Play: 言語ごとに 2～8 個のスクリーンショット
- 高品質（ぼやけたまたはピクセレーション画像なし）

### アプリタイトルとサブタイトル

- **App Name**: 30 文字（iOS）、50 文字（Android）
- **Subtitle**（iOS）: 追加 30 文字
- アプリが実行内容を明確にする
- ユーザーが検索するキーワードを含める

### キーワードとカテゴリ

- 最も関連性の高いカテゴリを選択
- iOS: 100 文字のコンマ区切りキーワード
- Android: キーワードは重要性が低い（説明が検索対象）

### プライバシーポリシーとウェブサイト

両ストアが必須：
- アクセス可能なプライバシーポリシー（URL）
- 実際のデータ慣行を反映するプライバシーポリシー
- ウェブサイトまたはサポート連絡先情報
- 多くはテンプレートプライバシーポリシーを開始点として使用

## 発表後：更新と保守

### Rork での定期更新

アプリがライブになった後、更新は同じプロセスに従います：

1. **Rork でコード変更を加える**
2. **app.json でバージョン番号をインクリメント**
3. **EAS でビルド**: `eas build --platform ios && eas build --platform android`
4. **リリースノート付きでストアに提出**
5. **クラッシュレポートを監視** ユーザーレビュー
6. **ユーザーの割合にリリース** 最初に、その後スケールアップ

### パフォーマンスを監視

**追跡するメトリクス**:
- クラッシュレート
- ユーザー評価
- ダウンロード数
- レビューセンチメント
- セッション期間
- フィーチャー使用

**利用可能なツール**:
- App Store Analytics（App Store Connect）
- Google Play Console Analytics
- Firebase Analytics（統合されている場合）
- クラッシュレポート（Firebase Crashlytics または Sentry）

### レビューに対応

1. **ユーザーレビューを定期的に監視**
2. **レビューで報告されたバグに対応**
3. **ポジティブレビュアーに感謝** 懸念に対処
4. **問題を素早く修正** 評価を維持
5. **一般的な苦情に対応する更新をプッシュ**

### バージョン戦略

- **パッチリリース**（1.0.1、1.0.2）: バグ修正
- **マイナーリリース**（1.1.0、1.2.0）: 新しいフィーチャー
- **メジャーリリース**（2.0.0）: 大きな変更

十分な頻度で更新して関連性を保つが、過度に更新しない（1 日に複数回ではなく）。

## まとめ

App Store と Google Play への公開は Rork でシンプルです。ほとんどの開発者にとって、Rork Original は iOS と Android ユーザーの両方に到達するための、制御とシンプルさのバランスを最適に提供します。良質なアプリを作成し、優れたメタデータで、提出前に徹底的にテストし、レビュープロセスに対応する準備をしてください。ライブになった後、ユーザーフィードバックを監視し、ユーザー満足度とアプリストアランキングを維持するために定期的な更新をプッシュしてください。
