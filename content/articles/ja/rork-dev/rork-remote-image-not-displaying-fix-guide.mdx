Rork でアプリを生成して TestFlight に上げた瞬間、ローカルのアセット画像はちゃんと出るのに、Supabase Storage や CDN から取ってくる「リモート画像」だけが真っ白なまま――。私自身、この症状にここ数ヶ月で3回ぶつかりました。原因はそのたびに違いましたが、調べる順番だけは固定したほうが早く片付くと感じています。この記事では、その「調べる順番」を 4 つのチェックポイントにまとめてお伝えします。

最初に申し上げておきますと、リモート画像が出ないトラブルの 9 割は、画像コンポーネントの問題ではありません。「URL の届き方」と「OS 側の遮断ポリシー」のどちらかです。Rork が裏で生成している React Native + Expo のコードは正しく書かれていることが多いので、いきなり `<Image>` を疑わず、外側から内側へ順に切り分けるのが結局いちばん速い、というのが私の実感です。

## 症状を 30 秒で切り分ける

まず、ブラウザのアドレスバーにその画像 URL を貼って、PC のブラウザで普通に開けるかを確認してください。これだけで原因の半分が決まります。

- 開ける → URL とサーバーは生きています。次はクライアント側（ATS・URL エンコード・拡張子）を疑います
- 開けない / 認証要求が出る → サーバー側の公開設定の問題です。ストレージ側のバケットポリシーや署名 URL の有効期限を見直してください

ここで「シミュレータでは出るのに実機で出ない」という症状が出たら、ほぼ確実に次の章の **iOS App Transport Security（ATS）** が犯人です。逆に「Android では出るのに iOS だけ出ない」も同じ原因のことが多いです。

## 原因 1: iOS の ATS が HTTP 画像を遮断している

iOS は `https://` ではない URL（つまり `http://`）からのデータ取得を、デフォルトで全部ブロックします。これは Apple のセキュリティポリシーで、Rork 側の問題ではありません。

確認方法は簡単で、画像 URL の先頭が `http://` で始まっていたらまずこれを疑います。Rork のプレビュー（Web）では出るのに、iOS の Companion アプリや TestFlight ビルドで出ない、というパターンはほぼ ATS です。

正しい対処は **画像配信側を HTTPS にする** ことです。Cloudinary、Supabase Storage、Firebase Storage、imgix などのほぼ全ての主要 CDN は HTTPS で配信できます。HTTP 配信のままになっているのは、自前のサーバーで Let's Encrypt を入れていないケースがほとんどです。

どうしても HTTP のまま使いたい場合は、`app.json` に例外を追加できますが、本番アプリでは絶対におすすめしません。App Store 審査でも理由を聞かれます。

```json
{
  "expo": {
    "ios": {
      "infoPlist": {
        "NSAppTransportSecurity": {
          "NSAllowsArbitraryLoads": true
        }
      }
    }
  }
}
```

このコードは「全 HTTP 接続を許可する」という強い設定です。本来は特定ドメインだけを許可する `NSExceptionDomains` を使うべきで、暫定的なデバッグ以外では入れないほうが安全です。

## 原因 2: URL に空白や日本語が混ざっている

意外と多いのが、画像のファイル名や URL パスに空白・括弧・日本語が混ざっていて、エンコード漏れで読み込みに失敗するケースです。Rork が画像 URL をプロンプトから組み立てる場合、ユーザーがアップロードしたファイル名がそのまま入ることがあります。

問題のあるコード例:

```tsx
// ❌ ファイル名に空白や日本語が含まれる場合に壊れる
<Image source={{ uri: `${BASE_URL}/uploads/${fileName}` }} />
```

正しい書き方は `encodeURIComponent` を通すことです:

```tsx
// ✅ ファイル名側だけをエンコード（パスの "/" を壊さない）
<Image
  source={{ uri: `${BASE_URL}/uploads/${encodeURIComponent(fileName)}` }}
  style={{ width: 200, height: 200 }}
/>
```

`encodeURI` ではなく `encodeURIComponent` を使うのが要点です。前者はスラッシュをエンコードしないので、ファイル名部分の安全性が担保されません。私はこのバグでアップロード機能を1日溶かしたことがあります。アップロード時にサーバー側でランダムな ASCII 文字列にリネームしてから保存する設計にしておくと、根本的に発生しなくなります。

## 原因 3: 画像にサイズが指定されていない

これは「URL は正しいのに画像が見えない」もう一つの定番パターンです。React Native の `<Image>` コンポーネントは、リモート画像の場合は `width` と `height` を明示しないと **表示領域が 0×0 になります**。ローカル画像（`require(...)`）は自動でサイズを推定しますが、リモートURLは推定しません。

```tsx
// ❌ サイズ未指定 — 画像はロードされているのに見えない
<Image source={{ uri: 'https://example.com/photo.jpg' }} />

// ✅ 明示的にサイズを指定
<Image
  source={{ uri: 'https://example.com/photo.jpg' }}
  style={{ width: '100%', height: 240, borderRadius: 12 }}
  resizeMode="cover"
/>
```

「画像 URL を変えても何も変わらない」「Inspector で見ても要素はあるのに何も表示されない」場合は、このサイズ未指定を疑ってください。可変サイズで使うなら `aspectRatio` を `style` に渡すと、幅に合わせて高さを自動計算してくれます。

## 原因 4: キャッシュと CORS — Web プレビュー特有の落とし穴

Rork の Web プレビューだけで画像が出ないとき、原因はほぼ CORS（Cross-Origin Resource Sharing）です。ネイティブビルド（iOS/Android）では関係ありませんが、Web 版ではブラウザが厳格にチェックします。

S3 や Supabase Storage を使っている場合、ストレージ側で CORS の許可ヘッダー（`Access-Control-Allow-Origin`）が `*` または該当ドメインで返るように設定する必要があります。Supabase なら Storage の設定画面、AWS なら S3 のバケット → 「アクセス許可」 → 「CORS 設定」で編集できます。

また、画像の更新が反映されない（古い画像が出続ける）症状は、CDN とブラウザのキャッシュが原因です。URL に `?v=タイムスタンプ` のような **キャッシュバスター** を付けて回避します:

```tsx
// 画像更新時に URL を変えてキャッシュを破棄
const cacheBuster = `?v=${user.avatarUpdatedAt}`;
<Image
  source={{ uri: `${user.avatarUrl}${cacheBuster}` }}
  style={{ width: 64, height: 64, borderRadius: 32 }}
/>
```

ユーザーがアバターを更新したのに古い画像が表示され続ける、というクレームは個人開発でも頻出します。`updatedAt` のような変更検知できるタイムスタンプを必ず URL に付与する設計にしておくと、サポート対応がぐっと減ります。

## ロード状態とエラー状態を可視化する

ここまでの 4 つの原因を切り分けるには、画像の読み込み状態を画面に出してしまうのが一番早いです。`onError` と `onLoad` を使ってログを出し、本当に「読み込めていない」のか「読み込めているが見えない」のかを区別します。

```tsx
import { Image } from 'expo-image'; // expo-image を推奨

export function RemoteAvatar({ uri }: { uri: string }) {
  return (
    <Image
      source={{ uri }}
      style={{ width: 80, height: 80, borderRadius: 40 }}
      contentFit="cover"
      transition={200}
      onLoad={() => console.log('[Image] loaded:', uri)}
      onError={(e) => console.log('[Image] error:', uri, e?.error)}
      placeholder={require('../assets/avatar-fallback.png')}
    />
  );
}
```

期待する出力は、成功時に `[Image] loaded: https://...`、失敗時に `[Image] error: ... (エラー詳細)` です。`expo-image` を使うと、エラー時のフォールバック画像、ブラー付きトランジション、メモリキャッシュが標準で備わっており、ネットワーク不安定環境でも UX が大きく改善します。Rork が生成するコードに `Image` がそのまま使われていたら、`expo-image` への置き換えを検討する価値はあります。

ネットワーク全般の切り分けについては、[Rork のネットワーク・APIリクエストをデバッグする実践ガイド](/articles/rork-basics/rork-network-api-request-debug-guide)も合わせて確認すると、画像以外の通信トラブルにも応用が効きます。Supabase Storage を使っている場合は、[Rork × Supabase Storage で画像アップロード機能を作る](/articles/rork-dev/rork-image-upload-gallery-supabase-storage)で公開バケットの設定例を載せていますので参考にしてください。

React Native の画像周りやネイティブモジュールの挙動をもう少し体系的に押さえたい方には、[React Nativeの実践（Bonfire Books）](https://www.amazon.co.jp/dp/B09T2X9ZFC/?tag=pinocchio-22)が良い入り口になります。Image・Animation・パフォーマンスの章が、特にトラブルシューティング時の地図として役に立ちます。

リモート画像のトラブルを今日のうちに切り分けるなら、まずブラウザでその URL を直接開いてみてください。それだけで「サーバー側の問題」か「クライアント側の問題」かが 5 秒で分かります。クライアント側だと判明したら、この記事の順序で（ATS → URL エンコード → サイズ指定 → CORS）潰していけば、3 つ以内のどれかで必ず原因に当たります。
