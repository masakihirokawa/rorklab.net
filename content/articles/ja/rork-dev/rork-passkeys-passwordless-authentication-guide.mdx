## パスワードを捨てる決断が、アプリの離脱率を変える

ログイン画面でユーザーが離脱する最大の理由は「パスワードを思い出せない」ことです。2026 年現在、Apple・Google・Microsoft が推進する Passkeys（パスキー）は、この問題を根本から解決する。Face ID や指紋認証だけでログインが完了し、フィッシング攻撃にも強い。

Rork で作ったアプリに Passkeys を導入しようとすると、最初にぶつかる壁があります。Expo Go では動かないという事実です。Passkeys はネイティブモジュールに依存するため、Development Build への切り替えが前提になります。この記事では、その切り替えも含めた実装手順を具体的に示す。

## Development Build への移行が必須な理由

Passkeys は OS のネイティブ認証 API（iOS の `ASAuthorizationController`、Android の Credential Manager）を呼び出す。Expo Go はサンドボックス化された環境でネイティブモジュールを追加できないため、Passkeys ライブラリが動作しません。

Rork でプロジェクトを生成した後、以下のコマンドで Development Build に切り替える:

```bash
# Expo Development Build のセットアップ
npx expo install expo-dev-client
npx expo prebuild

# iOS シミュレータでビルド
npx expo run:ios

# Android エミュレータでビルド
npx expo run:android
```

この時点で `ios/` と `android/` ディレクトリが生成されます。以降はネイティブモジュールを自由に追加できる状態になります。

## react-native-passkeys のセットアップ

Passkeys の実装には `react-native-passkeys` を使います。このライブラリは iOS・Android・Web で同一の API を提供しており、Rork アプリのクロスプラットフォーム開発と相性がいい。

```bash
# ライブラリのインストール
npx expo install react-native-passkeys
```

### ドメイン検証ファイルの設置

Passkeys は「どのドメインがこのアプリと紐づいているか」を OS に証明する必要があります。これを省略すると、登録も認証も無言で失敗します。

**iOS 向け — Apple App Site Association（AASA）ファイル:**

```json
// https://your-domain.com/.well-known/apple-app-site-association
{
  "webcredentials": {
    "apps": [
      "TEAM_ID.com.yourcompany.yourapp"
    ]
  }
}
```

**Android 向け — Digital Asset Links:**

```json
// https://your-domain.com/.well-known/assetlinks.json
[
  {
    "relation": ["delegate_permission/common.handle_all_urls"],
    "target": {
      "namespace": "android_app",
      "package_name": "com.yourcompany.yourapp",
      "sha256_cert_fingerprints": ["YOUR_SHA256_FINGERPRINT"]
    }
  }
]
```

ここで多くの開発者がつまずくのが、**ローカル開発中はこの検証が通らない**という点です。iOS シミュレータでは Associated Domains の `webcredentials:` エンタイトルメントを `app.json` に追加し、`applinks:` と同じドメインを指定する必要があります。

## Passkey の登録フローを実装する

以下は、ユーザーが初回登録する際のコード全体です。サーバー側で生成したチャレンジを受け取り、デバイスの生体認証で鍵ペアを作成する:

```typescript
// hooks/usePasskeyAuth.ts
import { useState, useCallback } from 'react';
import { Passkeys } from 'react-native-passkeys';
import { Alert, Platform } from 'react-native';

const RP_ID = 'your-domain.com';
const API_BASE = 'https://your-domain.com/api/auth';

type PasskeyState = {
  isLoading: boolean;
  error: string | null;
};

export function usePasskeyAuth() {
  const [state, setState] = useState<PasskeyState>({
    isLoading: false,
    error: null,
  });

  // Passkey の登録（初回サインアップ時）
  const register = useCallback(async (userId: string, userName: string) => {
    setState({ isLoading: true, error: null });

    try {
      // 1. サーバーからチャレンジを取得
      const optionsRes = await fetch(`${API_BASE}/passkey/register-options`, {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({ userId, userName }),
      });

      if (\!optionsRes.ok) {
        throw new Error('チャレンジの取得に失敗しました');
      }

      const options = await optionsRes.json();

      // 2. デバイスの生体認証で鍵ペアを生成
      const credential = await Passkeys.create({
        publicKey: {
          challenge: options.challenge,
          rp: { id: RP_ID, name: 'Your App' },
          user: {
            id: options.userId,
            name: userName,
            displayName: userName,
          },
          pubKeyCredParams: [
            { alg: -7, type: 'public-key' },   // ES256
            { alg: -257, type: 'public-key' },  // RS256
          ],
          authenticatorSelection: {
            userVerification: 'required',
            residentKey: 'required',
          },
          timeout: 60000,
        },
      });

      // 3. 公開鍵をサーバーに送信して保存
      const verifyRes = await fetch(`${API_BASE}/passkey/register-verify`, {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({
          userId,
          credential: {
            id: credential.id,
            rawId: credential.rawId,
            response: credential.response,
            type: credential.type,
          },
        }),
      });

      if (\!verifyRes.ok) {
        throw new Error('登録の検証に失敗しました');
      }

      setState({ isLoading: false, error: null });
      return true;
    } catch (err) {
      const message =
        err instanceof Error ? err.message : '登録中にエラーが発生しました';
      setState({ isLoading: false, error: message });
      Alert.alert('登録エラー', message);
      return false;
    }
  }, []);

  // Passkey によるログイン
  const authenticate = useCallback(async () => {
    setState({ isLoading: true, error: null });

    try {
      const optionsRes = await fetch(`${API_BASE}/passkey/auth-options`, {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
      });

      if (\!optionsRes.ok) {
        throw new Error('認証チャレンジの取得に失敗しました');
      }

      const options = await optionsRes.json();

      const assertion = await Passkeys.get({
        publicKey: {
          challenge: options.challenge,
          rpId: RP_ID,
          userVerification: 'required',
          timeout: 60000,
        },
      });

      const verifyRes = await fetch(`${API_BASE}/passkey/auth-verify`, {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({
          credential: {
            id: assertion.id,
            rawId: assertion.rawId,
            response: assertion.response,
            type: assertion.type,
          },
        }),
      });

      if (\!verifyRes.ok) {
        throw new Error('認証に失敗しました');
      }

      const { token } = await verifyRes.json();
      setState({ isLoading: false, error: null });
      return token;
    } catch (err) {
      const message =
        err instanceof Error ? err.message : '認証中にエラーが発生しました';
      setState({ isLoading: false, error: message });
      return null;
    }
  }, []);

  return { ...state, register, authenticate };
}
```

このフックを使えば、ログイン画面は驚くほどシンプルになる:

```tsx
// screens/LoginScreen.tsx
import { View, Text, TouchableOpacity, ActivityIndicator } from 'react-native';
import { usePasskeyAuth } from '../hooks/usePasskeyAuth';

export default function LoginScreen() {
  const { isLoading, authenticate } = usePasskeyAuth();

  const handleLogin = async () => {
    const token = await authenticate();
    if (token) {
      // ログイン成功 → メイン画面に遷移
    }
  };

  return (
    <View style={{ flex: 1, justifyContent: 'center', padding: 24 }}>
      <Text style={{ fontSize: 28, fontWeight: 'bold', marginBottom: 32 }}>
        おかえりなさい
      </Text>

      <TouchableOpacity
        onPress={handleLogin}
        disabled={isLoading}
        style={{
          backgroundColor: '#007AFF',
          padding: 16,
          borderRadius: 12,
          alignItems: 'center',
        }}
      >
        {isLoading ? (
          <ActivityIndicator color="#fff" />
        ) : (
          <Text style={{ color: '#fff', fontSize: 18, fontWeight: '600' }}>
            Face ID / 指紋でログイン
          </Text>
        )}
      </TouchableOpacity>
    </View>
  );
}
```

## 実装で陥りやすい 3 つの落とし穴

**1. AASA ファイルの Content-Type 設定ミス**

AASA ファイルは `application/json` で配信する必要があります。Cloudflare Pages や Vercel でホスティングしている場合、`.well-known` ディレクトリのレスポンスヘッダーを明示的に設定しないと、iOS が検証に失敗する。Cloudflare の場合は `_headers` ファイルで対応できます。

**2. Android の `assetlinks.json` と署名鍵の不一致**

開発ビルドとリリースビルドで署名鍵が異なるため、`sha256_cert_fingerprints` を両方登録しておかないと本番で認証が通らない。EAS Build を使う場合、`eas credentials` で確認できるフィンガープリントと一致しているか必ず照合します。

**3. `userVerification: 'required'` の罠**

`userVerification` を `required` にすると、生体認証に対応していないデバイスではエラーになります。古い Android 端末をサポートする場合は `preferred` に変更し、フォールバック認証（PIN 入力など）を用意する判断が必要です。

## 従来の認証との段階的な併用

Passkeys はまだ全ユーザーが対応デバイスを持っているわけではありません。現実的なアプローチとして、既存の認証方法と並行して提供し、対応デバイスのユーザーに Passkeys を推奨する設計が有効です。

[Rork の生体認証ガイド](/articles/rork-dev/rork-biometric-auth-face-id-touch-id-guide)で解説している Face ID / Touch ID 認証は、Passkeys が使えない場合のフォールバックとして機能する。また、[認証全般の設計](/articles/rork-dev/rork-auth-firebase-supabase)で扱っている Firebase Auth や Supabase Auth と Passkeys を組み合わせることで、ユーザーに選択肢を与えつつ、最もセキュアな方法への移行を促せる。

## パスキーが変えるアプリ体験

Passkeys の導入で得られる最大の変化は、ログインのコンバージョン率です。パスワード入力が不要になることで、復帰ユーザーのログイン成功率は大幅に改善します。

実装のハードルは Development Build への移行とドメイン検証の設定だが、一度乗り越えれば以降のメンテナンスコストは従来のパスワード管理より低い。次のステップとして、[アプリ全体のセキュリティ設計](/articles/rork-dev/rork-app-security-complete-implementation-guide)を見直し、Passkeys を中心とした認証アーキテクチャを検討してほしい。
