## はじめに — なぜE2Eテストがモバイルアプリに不可欠か

Rork Maxアプリをリリースする際、「ローカルテストでは動作するのにApp Storeで次々とリジェクトされる」という経験をしていませんか？

その原因は、開発環境では気づきにくい**リアルデバイス上での複雑な操作フロー**にあります。ユーザーが実際に使う順序でタップし、スワイプし、テキストを入力し、ネットワーク待機を経験する。こうした**エンドツーエンド（E2E）のシナリオを自動テストする**ことで、リリース前に品質問題を捕捉できます。

Rork Maxプロジェクトでは、以下の課題を抱えることが多いです。

- **手動テストの再現性欠落**: 新バージョンごとに手作業でテストし、テスト忘れやタイミング依存バグを見落とす
- **リグレッション検出遅延**: 古い機能への影響をリリース後に発見し、緊急パッチが必要になる
- **スクリーンショット更新負担**: UI変更のたびに全デバイス×全言語のスクリーンショットを手動撮影
- **App Store品質基準の未充足**: アクセシビリティやデバイス互換性の確認が不十分

本記事では、**Detox** と **Maestro** という2つのE2Eテストフレームワークを使い、CI/CDパイプラインに統合して、品質を確保する実践的な方法を解説します。

## Detox vs Maestro — 特徴比較、選定基準、プロジェクト規模別おすすめ

Rork Maxで採用可能なE2Eテストフレームワークは、大きく2つのアプローチに分かれます。

### Detox — ホワイトボックステスト（開発環境統合型）

**特徴**:
- アプリのネイティブコードに直接アクセス。UI要素を内部状態から識別
- React Native とも統合し、コンポーネントレベルでの同期テスト可能
- テスト用に特別なビルド設定が必要（`detox` target構成）
- 実行が高速（同期ベース、待機が少ない）

**メリット**:
- 要素の特定が確実。HTMLセレクタのような `testID` プロップで素早く操作
- ネットワークモックが容易で、API依存を排除したテスト実施可能
- React Nativeアプリのロジック検証と密接に連携

**デメリット**:
- **テスト用ビルド構成が複雑**: iOS/Android両方のDetoxビルド設定を保守必要
- **開発環境依存**: アプリのコードに `testID` を埋め込む必要（本番コードが肥大化する可能性）
- **Detoxデーモンの管理**: エミュレータ起動とのタイミング同期が手間

### Maestro — ブラックボックステスト（ユーザー体験重視）

**特徴**:
- 標準ビルド（App Store向けと同じ）をそのままテスト。特別な設定不要
- YAMLで**視認性の高い**テストフロー定義。技術者でなくても読み書き可能
- ビジュアル認識（OCR）とUI自動化の組み合わせ。画像マッチングでUI検証
- クラウドでテスト実行可能（Maestro Cloud）

**メリット**:
- **セットアップが簡潔**: 標準ビルドなので既存CI/CDと親和性が高い
- **エンドユーザー体験に近い**: 実際のApp Storeビルドをテストすることで、リリース後の問題を事前捕捉
- **ビジュアルテストが得意**: スクリーンショットの自動比較でUI回帰を検出
- **チーム全体が参加しやすい**: YAMLの直感性が高い

**デメリット**:
- **実行速度**: OCR処理のため、Detoxより遅い傾向
- **環境依存性**: OS言語やフォント、デバイスの物理的スペック影響を受けやすい
- **ネットワーク依存**: モック化が限定的。実API呼び出しが必要な場面が多い

### 選定基準

| 選定軸 | Detox推奨 | Maestro推奨 |
| --- | --- | --- |
| **チーム規模** | 8名以上のQA専任がいる | 3名以下のスタートアップ |
| **App Store品質重視** | 中程度 | **高（ユーザー視点重視）** |
| **CI/CD統合難度** | 中（ビルド設定増) | 低（標準ビルドのみ） |
| **テスト開発速度** | 遅い（コード記述） | **高速（YAML定義）** |
| **ビジュアルテスト** | 限定的 | **得意（OCR＆画像比較）** |
| **ネットワークモック要件** | **高い必要性** | 低 |

**Rork Maxでのおすすめ**:
- **最初はMaestro**で快速にテストカバレッジを確保。YAML駆動のため非開発者もテスト追加可能
- **重要なビジネスロジック**（決済フロー、認証など）のみ**Detoxで詳細テスト**
- **2つを組み合わせる**アプローチが現実的

## Detoxセットアップと基本テスト

Rork Maxプロジェクトで、重要な支払いフロー検証にDetoxを採用する場合のセットアップを見ていきます。

### インストールと設定

まずはDetox CLIとiOS/Androidドライバをインストール。

```bash
# Detox CLIグローバルインストール
npm install -g detox-cli

# Rork Maxプロジェクトにdetoxドライバ追加
cd /path/to/rork-max
npm install --save-dev detox detox-cli detox-test-utils
# iOSビルド用
npm install --save-dev detox-ios

# Androidの場合
npm install --save-dev detox-android
```

次に、**detox.config.js** をプロジェクト直下に作成。

```javascript
// detox.config.js
module.exports = {
  testRunner: 'jest',
  apps: {
    ios: {
      type: 'ios.app',
      binaryPath: 'ios/build/Build/Products/Release-iphonesimulator/RorkMax.app',
      build: 'xcodebuild -workspace ios/RorkMax.xcworkspace -scheme RorkMax -configuration Release -sdk iphonesimulator -derivedDataPath ios/build'
    },
    android: {
      type: 'android.apk',
      binaryPath: 'android/app/build/outputs/apk/release/app-release.apk',
      build: 'cd android && ./gradlew assembleRelease assembleAndroidTest -DtestBuildType=release'
    }
  },
  configurations: {
    ios: {
      device: {
        type: 'iPhone14'
      },
      app: 'ios'
    },
    android: {
      device: {
        type: 'Pixel4'
      },
      app: 'android'
    }
  },
  testRunner: 'jest'
};
```

### testID埋め込み

Rork Maxのコンポーネントに `testID` を埋め込み、Detoxが要素を正確に識別できるようにします。

```typescript
// src/screens/PaymentScreen.tsx
import React, { useState } from 'react';
import { View, Text, TextInput, TouchableOpacity } from 'react-native';

export const PaymentScreen = () => {
  const [amount, setAmount] = useState('');

  const handlePayment = () => {
    // 決済処理
  };

  return (
    <View testID="payment-screen">
      <Text testID="payment-title">支払い額を入力</Text>
      <TextInput
        testID="amount-input"
        placeholder="金額"
        value={amount}
        onChangeText={setAmount}
        keyboardType="decimal-pad"
      />
      <TouchableOpacity
        testID="confirm-payment-button"
        onPress={handlePayment}
      >
        <Text>確認</Text>
      </TouchableOpacity>
    </View>
  );
};
```

### 初めてのテストケース

Detoxテストは Jest構文で記述。支払いフロー検証の例。

```typescript
// e2e/paymentFlow.e2e.ts
describe('Payment Flow', () => {
  beforeAll(async () => {
    await device.launchApp();
  });

  beforeEach(async () => {
    await device.reloadReactNative();
  });

  it('should complete payment with valid amount', async () => {
    // 支払い画面に遷移
    await element(by.id('payment-screen')).multiTap();

    // 金額入力
    await element(by.id('amount-input')).typeText('1000');

    // 確認ボタンタップ
    await element(by.id('confirm-payment-button')).multiTap();

    // 完了画面の表示確認
    await expect(element(by.text('支払い完了'))).toBeVisible();
  });

  it('should show error for zero amount', async () => {
    await element(by.id('amount-input')).typeText('0');
    await element(by.id('confirm-payment-button')).multiTap();

    await expect(element(by.text('金額は1円以上で入力してください'))).toBeVisible();
  });
});
```

## Maestroセットアップと基本テスト

Maestroは標準ビルドでテスト可能。セットアップが簡潔で、Rork Maxのオンボーディングフロー検証に最適。

### インストール

```bash
# Maestro CLIインストール
curl -Ls "https://get.maestro.mobile/install.sh" | bash

# システムパス確認
maestro --version
```

### YAMLフロー定義の基本

Maestroのテストは `.yaml` ファイルで定義。以下は登録フロー検証の例。

```yaml
# e2e/flows/onboarding.yaml
appId: com.rork.max
---
- launchApp
- tapOn:
    text: "始める"
- inputText: "user@example.com"
- tapOn:
    text: "次へ"
- inputText: "password123"
- tapOn:
    text: "アカウント作成"
- assertVisible:
    text: "ようこそ、ユーザーさん"
```

### 高度なYAML定義 — エラーハンドリングと条件分岐

```yaml
# e2e/flows/loginWithRetry.yaml
appId: com.rork.max
---
- launchApp

# ログイン試行
- tapOn:
    text: "ログイン"

- inputText: "user@example.com"

# パスワード入力（最大3回リトライ）
- repeat:
    times: 3
    commands:
      - inputText: "wrongpassword"
      - tapOn:
          text: "ログイン"
      - waitForAnimationToEnd
      - if:
          condition:
            visible:
              text: "パスワードが正しくありません"
          then:
            - tapOn:
                text: "パスワードをリセット"
            - exitRepeat

# ホーム画面表示確認
- assertVisible:
    text: "ダッシュボード"
```

### コマンド体系の早見表

| コマンド | 説明 | 例 |
| --- | --- | --- |
| `launchApp` | アプリ起動 | `- launchApp` |
| `tapOn` | UI要素タップ | `- tapOn: { text: "ボタン" }` |
| `inputText` | テキスト入力 | `- inputText: "example@mail.com"` |
| `scroll` | スクロール | `- scroll: { direction: down }` |
| `assertVisible` | 表示確認 | `- assertVisible: { text: "完了" }` |
| `waitFor` | 待機 | `- waitFor: { timeout: 5000 }` |
| `swipe` | スワイプ | `- swipe: { direction: left }` |
| `repeat` | ループ | `- repeat: { times: 3, commands: [...] }` |

## テストシナリオ設計パターン

Rork Maxアプリで現実的に発生する操作フローをパターン化。各パターンはDetox・Maestroいずれでも実装可能。

### パターン1 — 認証フロー（マルチステップフォーム）

ユーザーがメールアドレス → パスワード → 2要素認証 → プロフィール完了までの一連の流れを検証。

```yaml
# e2e/flows/completeRegistration.yaml
appId: com.rork.max
---
- launchApp
- tapOn:
    text: "新規登録"

# ステップ1: メール入力
- tapOn:
    id: "email-input"
- inputText: "newuser@example.com"

# ステップ2: パスワード設定
- tapOn:
    id: "next-button"
- tapOn:
    id: "password-input"
- inputText: "SecurePass123!"

# ステップ3: 認証コード入力（メール受信をシミュレート）
- tapOn:
    id: "next-button"
- inputText: "123456"  # メール送信されたコード

# ステップ4: プロフィール入力
- tapOn:
    id: "next-button"
- tapOn:
    id: "username-input"
- inputText: "newuser"
- tapOn:
    id: "complete-button"

# 完了確認
- assertVisible:
    text: "登録完了しました"
```

### パターン2 — CRUD操作フロー（データ操作）

ユーザーが項目を作成 → 編集 → 削除する流れ。リスト操作の回帰検出に有効。

```typescript
// e2e/crudOperations.e2e.ts (Detox例)
describe('CRUD Operations', () => {
  it('should create, edit, and delete a task', async () => {
    // CREATE: 新規項目追加
    await element(by.id('add-task-button')).multiTap();
    await element(by.id('task-name-input')).typeText('新規プロジェクト');
    await element(by.id('priority-dropdown')).multiTap();
    await element(by.text('高')).multiTap();
    await element(by.id('save-task-button')).multiTap();

    // リスト表示確認
    await expect(element(by.text('新規プロジェクト'))).toBeVisible();

    // UPDATE: 項目編集
    await element(by.text('新規プロジェクト')).multiTap();
    await element(by.id('task-name-input')).clearText();
    await element(by.id('task-name-input')).typeText('更新されたプロジェクト');
    await element(by.id('save-task-button')).multiTap();

    // 編集確認
    await expect(element(by.text('更新されたプロジェクト'))).toBeVisible();

    // DELETE: 項目削除
    await element(by.text('更新されたプロジェクト')).longPress();
    await element(by.text('削除')).multiTap();
    await element(by.id('confirm-delete-button')).multiTap();

    // 削除確認
    await expect(element(by.text('更新されたプロジェクト'))).not.toBeVisible();
  });
});
```

### パターン3 — ナビゲーションフロー（画面遷移）

### パターン4 — エッジケース（境界条件）

極端な入力値・ネットワーク不安定・メモリ不足などの異常系を検証。

```typescript
// e2e/edgeCases.e2e.ts
describe('Edge Cases', () => {
  it('should handle very long text input', async () => {
    const longText = 'a'.repeat(5000);
    await element(by.id('text-input')).typeText(longText);
    await element(by.id('submit-button')).multiTap();
    // エラー表示または自動カット確認
    await expect(element(by.text('入力が長すぎます'))).toBeVisible();
  });

  it('should handle offline condition', async () => {
    // ネットワークを意図的に無効化
    await device.disableNetwork();
    await element(by.id('refresh-button')).multiTap();
    await expect(element(by.text('ネットワーク接続がありません'))).toBeVisible();

    // ネットワーク復帰
    await device.enableNetwork();
    await element(by.id('retry-button')).multiTap();
    await expect(element(by.text('読み込み中'))).not.toBeVisible();
  });

  it('should handle rapid consecutive taps', async () => {
    // ダブルクリック防止のテスト
    await element(by.id('purchase-button')).multiTap();
    await element(by.id('purchase-button')).multiTap();
    // 1件の決済のみ確認（2重決済防止）
    await expect(element(by.text('決済済み'))).toBeVisible();
  });
});
```

## スクリーンショットテストとビジュアルリグレッション

Maestroの強みの1つが**ビジュアル自動化**。UI変更時に自動的にスクリーンショットを比較し、意図しない見た目変更を検出します。

### スクリーンショット自動キャプチャ

Maestroでテスト実行時、各ステップのスクリーンショットを自動保存。

```yaml
# e2e/flows/visualRegression.yaml
appId: com.rork.max
---
- launchApp
- takeScreenshot: "01-home-screen"

- tapOn:
    text: "プロフィール"
- takeScreenshot: "02-profile-screen"

- tapOn:
    id: "edit-button"
- takeScreenshot: "03-profile-edit-screen"

- inputText: "新しいユーザー名"
- takeScreenshot: "04-username-input"

- tapOn:
    text: "保存"
- assertVisible:
    text: "保存完了"
- takeScreenshot: "05-save-complete"
```

### ピクセル比較と閾値設定

初回実行でベースラインスクリーンショットを作成。以降の実行では自動比較。

```bash
# 初回実行（ベースライン記録）
maestro test e2e/flows/visualRegression.yaml --record-baselines

# 2回目以降（比較実行）
maestro test e2e/flows/visualRegression.yaml --tolerance 0.01
```

`--tolerance 0.01` により、1%以下の差分は許容。フォント微調整やアンチエイリアス依存の誤検出を回避。

### GitHub Actionsでの自動キャプチャ保存

CI/CDパイプラインで失敗時のスクリーンショットを自動アップロード。

```yaml
# .github/workflows/maestro-visual-tests.yml
name: Maestro Visual Tests

on:
  pull_request:
  push:
    branches: [main]

jobs:
  test:
    runs-on: macos-latest
    steps:
      - uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: 18

      - name: Build app
        run: |
          cd ios
          xcodebuild -workspace RorkMax.xcworkspace -scheme RorkMax -configuration Release

      - name: Run Maestro tests
        run: maestro test e2e/flows/ --output-dir maestro-output

      - name: Upload test artifacts
        if: always()
        uses: actions/upload-artifact@v3
        with:
          name: maestro-screenshots
          path: maestro-output/
```

## CI/CD統合 — GitHub Actions × EAS Build

Rork Maxのテスト自動化が効力を発揮するのは、CI/CDパイプラインに統合してからです。コミットのたびにテストが走り、問題があれば即座に検出される体制を整備します。

### GitHub Actions ワークフロー（Detox + Maestro統合）

```yaml
# .github/workflows/e2e-tests.yml
name: E2E Tests (Detox + Maestro)

on:
  pull_request:
  push:
    branches: [main, develop]

env:
  CACHE_VERSION: v1

jobs:
  build-and-test:
    runs-on: macos-13
    strategy:
      matrix:
        platform: [ios, android]

    steps:
      - uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: 18
          cache: 'npm'

      - name: Install dependencies
        run: npm ci

      - name: Install Detox CLI
        run: npm install -g detox-cli

      - name: Setup iOS environment
        if: matrix.platform == 'ios'
        run: |
          brew install applesimutils
          detox build-framework-cache

      - name: Build app (iOS - Detox)
        if: matrix.platform == 'ios'
        run: detox build-ios --configuration Release

      - name: Run Detox tests (iOS)
        if: matrix.platform == 'ios'
        run: detox test e2e --configuration ios --cleanup --record-logs all

      - name: Build app (Android)
        if: matrix.platform == 'android'
        run: |
          cd android
          ./gradlew assembleRelease assembleAndroidTest

      - name: Run Maestro tests
        run: |
          maestro test e2e/flows/ \
            --output-dir maestro-output \
            --format junit

      - name: Upload test reports
        if: always()
        uses: actions/upload-artifact@v3
        with:
          name: test-reports-${{ matrix.platform }}
          path: |
            artifacts/
            maestro-output/

      - name: Comment test results on PR
        if: always() && github.event_name == 'pull_request'
        uses: actions/github-script@v7
        with:
          script: |
            const fs = require('fs');
            const reportPath = 'maestro-output/report.xml';

            if (fs.existsSync(reportPath)) {
              const report = fs.readFileSync(reportPath, 'utf8');
              // JUnit XMLをパースしてコメント投稿
              github.rest.issues.createComment({
                issue_number: context.issue.number,
                owner: context.repo.owner,
                repo: context.repo.repo,
                body: `✅ E2E Tests Passed\n\n詳細は [Artifacts](https://github.com/${context.repo.owner}/${context.repo.repo}/actions/runs/${context.runId}) を参照してください。`
              });
            }
```

### EAS Build との統合

## アクセシビリティテスト — VoiceOver/TalkBack互換性

App Storeリジェクト理由の上位に「アクセシビリティ不対応」があります。Rork Maxが多くのユーザーに使われるなら、スクリーンリーダー（VoiceOver on iOS、TalkBack on Android）に対応する必須です。

### アクセシビリティ要素の確認テスト

Detoxで `accessibilityLabel` と `accessible` を検証。

```typescript
// e2e/accessibility.e2e.ts
describe('Accessibility Tests', () => {
  it('all interactive elements should have accessibility labels', async () => {
    // ボタン要素
    await expect(element(by.id('submit-button'))).toHaveToggleValue(false);

    // アクセシビリティラベルが存在するか
    const submitButton = element(
      by.id('submit-button').and(by.type('RCTTouchableOpacity'))

    // スクリーンリーダーに読み上げられる内容確認
    // RN/Expoではdetoxで直接ラベルチェックは限定的。
    // 実運用ではXcodeのAccessibility Inspectorで手動確認推奨
  });

  it('text contrast should meet WCAG AA standard', async () => {
    // Detoxでは色コントラスト測定が直接非対応
    // 別途 Lighthouse や axe DevTools（ブラウザ版）で確認
  });
});
```

### Maestroでのアクセシビリティ検証

Maestroはビジュアル検証が強いため、レイアウトやテキストサイズ検証が容易。

```yaml
# e2e/flows/accessibility.yaml
appId: com.rork.max
---
# スクリーンリーダー対応確認
- launchApp

# ボタンテキストが十分大きいか（視認性）
- assertVisible:
    text: "ログイン"  # テキストが見える = 最小フォントサイズ確保

# タップ領域が十分か（最小44x44pt推奨）
- tapOn:
    id: "compact-button"  # 成功 = タップ可能サイズ

# 色覚多様性対応（カラーコントラスト）
# Maestroでは視覚的に検証
- assertVisible:
    text: "重要な警告"  # テキスト視認可能 = コントラスト合格
```

### アクセシビリティのベストプラクティス

Rork Maxコンポーネント実装時に心がけるべき点。

```typescript
// src/components/AccessibleButton.tsx
import React from 'react';
import { TouchableOpacity, Text } from 'react-native';

interface AccessibleButtonProps {
  label: string;
  hint?: string;
  onPress: () => void;
}

export const AccessibleButton: React.FC<AccessibleButtonProps> = ({
  label,
  hint,
  onPress,
}) => {
  return (
    <TouchableOpacity
      onPress={onPress}
      accessibilityRole="button"
      accessibilityLabel={label}
      accessibilityHint={hint}
      // 最小サイズ: 44x44pt (Apple HIG)
      style={{ minHeight: 44, minWidth: 44, padding: 12 }}
    >
      <Text style={{ fontSize: 16 }}>{label}</Text>
    </TouchableOpacity>
  );
};
```

アクセシビリティ対応チェックリスト:

- **タップ領域**: 最小 44×44 pt（Apple HIG）
- **フォントサイズ**: 最小 12pt（視認性）
- **コントラスト比**: 最小 4.5:1（テキスト）、3:1（グラフィック）
- **アクセシビリティラベル**: ボタン・アイコンに必須
- **キーボード操作**: Bluetooth キーボード対応
- **スクリーンリーダー対応**: VoiceOver/TalkBack で操作可能

## FAQ

### Q1: DetoxとMaestro、どちらを先に導入すべき？

**A**: Maestroから始めることをお勧めします。理由は3つ：

1. **セットアップが簡潔**: ビルド設定の変更がない
2. **チーム全体参加可能**: YAMLで非エンジニアもテスト追加可能
3. **ROI が高い**: 少ない工数で広範なUIテストカバレッジを確保

その後、支払い・認証など**重要なビジネスロジック**のみDetoxで詳細テストするハイブリッド戦略が現実的です。

### Q2: スクリーンショットテストの自動化で、環境依存による誤検出が多い。どう対処？

**A**: 以下の戦略で偽陽性を削減：

1. **`--tolerance` パラメータ**: 1〜2%の差分を許容
2. **ベースライン更新**: UI意図的変更時は `--record-baselines` で再記録
3. **マスキング**: 環境依存箇所（タイムスタンプ、ステータスバー）を除外
4. **複数デバイステスト**: 異環境での一貫性を確保

```bash
maestro test e2e/flows/ \
  --tolerance 0.02 \
  --exclude-regions "0,0,414,20"  # ステータスバー除外
```

### Q3: ローカルテストでは成功、CI/CDで失敗する。なぜ？

**A**: 以下の4つの原因が多いです：

1. **タイムアウト値**: CI環境はローカルより遅い。`waitFor` タイムアウトを増やす
2. **ネットワーク**: API通信が不安定。Detoxのモック化を活用
3. **フォント/言語**: CI実行環境の言語設定がローカルと異なる
4. **メモリ不足**: 長時間テストでメモリリークが顕在化

```yaml
# タイムアウト調整例
- waitFor:
    timeout: 10000  # 10秒に延長
```

### Q4: テスト開発コストが高い。加速化方法は？

**A**: 以下の優先度でテスト追加を進める：

1. **重要フロー（P0）**: 支払い、認証、データ保存（即座にテスト化）
2. **一般フロー（P1）**: ナビゲーション、フォーム入力（次週）
3. **エッジケース（P2）**: 異常系、境界条件（余裕があれば）

また、テスト用データファクトリを整備。セットアップコードを削減。

```typescript
// e2e/factories/userFactory.ts
export const createTestUser = async () => {
  return {
    email: `test-${Date.now()}@example.com`,
    password: 'TestPassword123!',
  };
};
```

### Q5: App Storeリジェクト理由が「アクセシビリティ不足」。Rork Maxで何をテストすべき？

**A**: 以下3つの項目は必須：

1. **スクリーンリーダー対応**: 全ボタン・入力欄に `accessibilityLabel` を設定
2. **タップサイズ**: 全タップ要素が 44×44pt 以上
3. **色コントラスト**: テキスト 4.5:1 以上、グラフィック 3:1 以上

これらは Detox/Maestro テストでは限定的。iOS Xcode の Accessibility Inspector や Android の Accessibility Scanner を併用し、リリース前の手動検証が重要です。

## まとめ

Rork Maxアプリの品質保証は、単なる手動テストでは賄えません。DetoxとMaestroを組み合わせ、CI/CDパイプラインに統合することで：

✅ **リグレッション検出を自動化**: コミットのたびにテストが走り、古い機能への影響を即座に発見

✅ **App Storeリジェクト率を削減**: ビジュアルテストとアクセシビリティ検証で品質を確保

✅ **テスト保守コストを削減**: YAMLベースのMaestroで非エンジニアも参加可能

✅ **エンドユーザー体験に近づく**: 実ビルドをテストすることで、ローカル環境とのズレを排除

リスク の高い機能（決済、認証、データ同期）からDetoxで詳細テストを、オンボーディングやナビゲーションはMaestroで快速に、という**ハイブリッド戦略**が実用的です。

3ヶ月で基本テストスイートを構築し、その後は新機能追加のたびにテストを加増することで、継続的な品質向上が実現できます。Rork Maxの信頼性向上に、ぜひ本記事の知見を活用してください。
