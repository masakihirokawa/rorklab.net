## はじめに

2026年2月に React Native 0.84 がリリースされ、JavaScript エンジンが JavaScriptCore (JSC) から **Hermes V1 へと完全に移行** しましました。さらに Expo SDK 55 では Legacy Architecture が廃止され、**New Architecture のみが選択肢** となりましました。これは React Native 開発の歴史においても重要な転換点です。

本ガイドでは、React Native 0.84 と Hermes V1 の詳細、Expo SDK 55 への対応方法、そして Rork Max アプリ開発への実践的な影響を解説します。

## React Native 0.84 の主要変更点

### Hermes V1がデフォルトエンジンに

React Native 0.84 の最大の変更は、**Hermes V1 がデフォルト JavaScript エンジン** となったことです。従来の JavaScriptCore (JSC) からの完全な移行を意味します。

Hermes は Facebook（現 Meta）によって開発された JavaScript エンジンで、モバイルアプリケーションに特化した設計になっています。0.84 以降、すべての新規プロジェクトおよび既存プロジェクトのアップグレードでは、Hermes V1 が標準となります。

### New Architecture の必須化

React Native 0.84 と共に、**New Architecture が事実上の標準** となりましました。Legacy Architecture (Old Architecture) はサポート対象外となり、開発チームは New Architecture への移行を余儀なくされています。

New Architecture は以下の特徴を持ちます：
- **TurboModules**: ネイティブモジュール呼び出しの高速化
- **Fabric**: レンダリングエンジンの最適化による高速UI更新
- **JSI (JavaScript Interface)**: JavaScript と ネイティブコードの直接通信

## Hermes V1 が実現するパフォーマンス革命

### Hermes V1 と JSC の性能比較

| 指標 | Hermes V1 | JSC | 改善率 |
|------|----------|-----|--------|
| アプリ起動時間 | 850ms | 1200ms | 29% 高速化 |
| メモリ使用量 | 45MB | 72MB | 38% 削減 |
| バンドルサイズ | 2.1MB | 2.8MB | 25% 削減 |
| フレームレート安定性 | 59.8 fps | 57.2 fps | 4.5% 向上 |

Hermes V1 はこれらの指標において著しい改善を実現しています。特に：

- **アプリ起動時間**: バイトコード最適化により約 30% 高速化
- **メモリ効率**: Garbage Collection の改善で 40% 近い削減
- **バンドルサイズ**: Dead code elimination と tree shaking の強化
- **フレームレート**: 安定した 60fps 維持がより容易に

### バイトコードプリコンパイル

Hermes V1 の革新的な機能は、ビルド時に JavaScript を **バイトコード** に事前コンパイルすることです。これにより：

1. アプリ起動時の JavaScript パースが不要に
2. メモリ効率が向上
3. 実行速度が向上

従来の JSC では、アプリ起動時に毎回 JavaScript をパースしていたため、大規模アプリではボトルネックとなっていましました。

### ガベージコレクションの改善

Hermes V1 は改善されたガベージコレクション アルゴリズムを搭載：
- **Generational GC**: 世代別ガベージコレクション
- **Incremental GC**: 段階的なガベージコレクション
- **背景 GC**: UIスレッドへの干渉を最小化

これらにより、フレームドロップの減少とメモリ使用量の最適化が実現します。

## Expo SDK 55 での变更

### Legacy Architecture 廃止

Expo SDK 55 は **Legacy Architecture を完全に廃止** しましました。これにより：

```
❌ Legacy Architecture: もうサポートされません
✅ New Architecture: 唯一の選択肢
```

Legacy Architecture を使用していたアプリは必ず移行が必要です。新規プロジェクトでは移行の手間は不要ですが、既存プロジェクトの更新は必須です。

### newArchEnabled フラグの削除

従来の `app.json` では以下のように New Architecture を有効にしていました：

```json
{
  "expo": {
    "plugins": [
      [
        "react-native-worklets-core"
      ]
    ],
    "newArchEnabled": true
  }
}
```

**Expo SDK 55 では `newArchEnabled` フラグが削除** され、デフォルトで New Architecture が有効化されます。`app.json` からこのフラグを削除する必要があります。

移行後の `app.json` は以下のようにシンプルになります：

```json
{
  "expo": {
    "plugins": [
      [
        "react-native-worklets-core"
      ]
    ]
  }
}
```

### EAS Build の高速化

Expo SDK 55 では EAS Build に改善が加えられ、**キャッシング機能により約 30% のビルド時間削減** が実現しています。

ビルド時間の削減：
- キャッシュヒット率: 約 70%
- 平均ビルド時間: 8分 → 5.6分
- 大規模プロジェクト: 最大 40% の削減が報告されている

## Node.js v22.11+ の要件

React Native 0.84 と Expo SDK 55 では、**Node.js v22.11 以上** が必須となりましました。

### 現在のバージョン確認

```bash
node --version
```

v22.11 未満の場合は、以下の手順でアップグレードしてください：

### Node.js のアップグレード方法

**macOS での nvm を使用した更新:**

```bash
nvm install 22.11
nvm use 22.11
nvm alias default 22.11
```

**Windows での Chocolatey を使用:**

```powershell
choco upgrade nodejs --version=22.11.0
```

**Linux (Ubuntu/Debian):**

```bash
curl -fsSL https://deb.nodesource.com/setup_22.x | sudo -E bash -
sudo apt-get install -y nodejs=22.11.0-1nodesource1
```

Node.js v22.11 では以下の改善が含まれています：
- V8 エンジンの最適化
- NPM 11.0 の改善
- より厳密な ESM サポート

## 既存プロジェクトからの移行手順

### ステップ 1: 依存関係の更新

```bash
npx expo@latest upgrade
```

または、手動で `package.json` を更新：

```bash
npm install react-native@0.84 expo@55 @react-native/core@0.84
npm install --save-dev @react-native-community/cli@14
```

### ステップ 2: newArchEnabled フラグの削除

`app.json` から `newArchEnabled: true` を削除します：

```diff
{
  "expo": {
    "name": "MyApp",
-   "newArchEnabled": true,
    "plugins": [
      ["react-native-worklets-core"]
    ]
  }
}
```

### ステップ 3: ネイティブコードの更新（iOS）

`ios/Podfile` を確認し、以下の設定を確認：

```ruby
post_install do |installer|
  react_native_post_install(
    installer,
    config[:reactNativePath],
    :mac_catalyst_enabled => false,
    :architecture => :new
  )
end
```

### ステップ 4: ネイティブコードの更新（Android）

`android/build.gradle` で以下を確認：

```gradle
project.ext.react = [
  enableHermes: true,
  newArchEnabled: false, // デフォルトで無視されます
]
```

### ステップ 5: 依存する外部ライブラリの確認

新しい Architecture に対応していないライブラリがないか確認します：

```bash
npx expo doctor
```

### ステップ 6: テストとデバッグ

```bash
# iOS
npx react-native run-ios

# Android
npx react-native run-android

# Expo で開発
npx expo start
```

### ステップ 7: ビルドと検証

```bash
# iOS での Hermes バイトコードビルド
npx react-native run-ios --mode=Release

# Android での確認
npx react-native run-android --mode=release
```

## トラブルシューティング

### 問題 1: Hermes が有効化されない

**原因**: `newArchEnabled` フラグが遺残しているか、ビルドキャッシュが残っている可能性

**解決策**:
```bash
# キャッシュをクリア
rm -rf ios/Pods node_modules
npm install
cd ios && pod install && cd ..

# 再度ビルド
npx react-native run-ios --mode=Release
```

### 問題 2: Legacy Architecture のライブラリが互換性エラー

**原因**: 古いネイティブライブラリが New Architecture に対応していない

**解決策**:
```bash
# 互換性チェック
npx expo doctor --fix

# または手動でライブラリを更新
npm update <library-name>
```

### 問題 3: EAS Build が失敗する

**原因**: ローカルビルドでは成功し、EAS では失敗する場合、環境の差異が考えられる

**解決策**:
```bash
# ローカルでプロダクションビルドをシミュレート
eas build --local --platform ios

# キャッシュをリセット
eas build --clear-cache
```
