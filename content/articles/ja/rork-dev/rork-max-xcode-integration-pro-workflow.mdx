Rork Max で iOS アプリを素早く立ち上げられるようになりましたが、App Store 審査を意識したプロダクションレベルの品質に仕上げるには、Xcode 側での手作業が依然として必要です。本記事では、Rork Max が出力した SwiftUI プロジェクトを Xcode で開き、プロ品質に仕上げるまでのワークフローを、工程ごとに整理してお届けします。

個人アプリ開発者として Rork Max を半年ほど本格運用してきた経験から、「ここで時間を溶かした」というハマりどころも合わせて共有します。

## Rork Max 生成プロジェクトの構造を理解する

まず Xcode で Rork Max のプロジェクトを開いたときに、ディレクトリ構成を正しく把握することが出発点です。

典型的な構成は次のような形になります。

```
MyApp/
├── MyApp.xcodeproj/
├── Sources/
│   ├── App/              ← アプリケーションのエントリポイント
│   ├── Views/            ← SwiftUI View 定義
│   ├── Models/           ← データモデル・Codable 定義
│   ├── Services/         ← API クライアント・永続化層
│   └── Utils/            ← 拡張関数・ヘルパー
├── Resources/
│   ├── Assets.xcassets/
│   ├── Localizations/
│   └── Fonts/
└── .rork/                ← Rork 生成メタデータ（重要）
```

この中で最も注意が必要なのは `.rork/` ディレクトリです。これは Rork Max がプロジェクトの再生成や更新に使う内部メタデータで、触ると Rork 側からの再生成がおかしくなります。絶対に手動で編集しないでください。

Sources 配下のファイルは自由に編集できますが、次に Rork Max を使ってプロジェクトを更新するとき、構造を大きく変えたファイルは「衝突あり」として扱われます。後述するマージ戦略を必ず押さえてから改変してください。

## Instruments で診断すべき 3 つのプロファイル

Rork Max が生成するコードは十分に動きますが、初期状態では「とりあえず動く」レベルに留まっていることが多いです。プロダクション品質に仕上げるには、Instruments を使った診断が必須です。特に見るべきプロファイルは次の 3 つです。

### 1. Time Profiler — CPU 時間の見直し

アプリ起動時やタブ切り替え時の体感速度が遅いと感じたら、最初に Time Profiler で計測します。Rork 生成コードで典型的に遅いのは、次のパターンです。

```swift
// 遅い: 毎回すべての要素を再計算
var body: some View {
    List(items) { item in
        Row(item: item, score: computeScore(for: item))
    }
}
```

`computeScore` が重い計算を含んでいると、スクロール時に毎フレーム再計算が走って体感速度が落ちます。対策は `@State` でキャッシュするか、コンピュテッドプロパティを事前計算しておくことです。

```swift
// 速い: 事前計算してキャッシュ
@State private var cachedScores: [UUID: Double] = [:]

var body: some View {
    List(items) { item in
        Row(item: item, score: cachedScores[item.id] ?? 0)
    }
    .task {
        await computeAllScores()
    }
}
```

### 2. Allocations — メモリ使用量の把握

Rork Max が生成するコードは、画像の扱いで不要なメモリを消費するケースがあります。Allocations プロファイルで「Transient」ではなく「Persistent」なメモリの増加を追うと、解放漏れが見えてきます。

特に `AsyncImage` や `UIImage(data:)` を多用するリストは要注意です。画像のダウンサンプリングを入れるだけで、メモリ使用量が 1/5 になることもあります。

```swift
// 画像のダウンサンプリング関数
func downsample(image data: Data, to pointSize: CGSize, scale: CGFloat) -> UIImage? {
    let imageSourceOptions = [kCGImageSourceShouldCache: false] as CFDictionary
    guard let imageSource = CGImageSourceCreateWithData(data as CFData, imageSourceOptions) else {
        return nil
    }
    let maxDimensionInPixels = max(pointSize.width, pointSize.height) * scale
    let downsampleOptions = [
        kCGImageSourceCreateThumbnailFromImageAlways: true,
        kCGImageSourceShouldCacheImmediately: true,
        kCGImageSourceCreateThumbnailWithTransform: true,
        kCGImageSourceThumbnailMaxPixelSize: maxDimensionInPixels
    ] as CFDictionary
    guard let downsampledImage = CGImageSourceCreateThumbnailAtIndex(imageSource, 0, downsampleOptions) else {
        return nil
    }
    return UIImage(cgImage: downsampledImage)
}
```

### 3. SwiftUI — ビュー更新の可視化

SwiftUI は宣言的なゆえに、どの View がどのタイミングで再描画されるかが直感と合わないことがあります。SwiftUI Instrument を使うと、予期しない再描画が可視化されます。

Rork 生成コードで典型的に起こる問題は、`ObservableObject` のプロパティを監視している View が、関係ないプロパティの変更でも再描画されることです。これは `@ObservedObject` を `@Published` 個別の値に分解して渡す、あるいは `EquatableView` を使うことで改善できます。

## Rork Max アップデートを取り込みつつ最適化を維持する

実運用で最も悩ましいのが、Rork 側のアップデートを取り込むときに、Xcode 側で加えた最適化を失わないようにすることです。

私が採っているのは、次の 2 分岐アプローチです。

```
main ブランチ           ← Xcode 最適化を反映した手動ブランチ
rork-generated ブランチ ← Rork Max が生成したそのままのコード
```

`rork-generated` ブランチは、Rork Max でプロジェクトを再生成するたびに force push します。そして `main` ブランチに `rork-generated` を merge することで、Rork 側の更新を自分の最適化と統合します。

この運用のポイントは、**手動で加える最適化を、できるだけ独立したファイルに分離しておくこと**です。`Sources/App/AppDelegate.swift` を直接改変するのではなく、`Sources/Extensions/AppDelegate+Optimizations.swift` のような拡張ファイルを作っておくと、Rork 側のファイルとのコンフリクトを大幅に減らせます。

### 衝突が起きたときの対処

Git の merge で衝突が発生したら、基本方針は「Rork 側のロジックは Rork に任せ、自分の最適化は拡張点に閉じ込める」です。どうしても本体に手を入れなければならないときは、コメントで `// RORK:ORIGINAL_KEPT` や `// DOLICE:OPTIMIZATION` のようにマーカーを入れておくと、後で追跡しやすくなります。

## App Store 審査前のチェックリスト

最後に、Rork Max で作ったアプリを App Store に出す前に、私が必ず確認する項目を列挙します。

- **Bundle Identifier の最終確認**: Rork 生成時のデフォルト値（`com.rork.appname`）のまま出すとリジェクトされるので、自分の組織ドメインに変更
- **Info.plist のプライバシー説明**: カメラ・位置情報・フォトライブラリを使う場合、`NSCameraUsageDescription` 等の説明文を実運用レベルで書き直す
- **Push Notifications の entitlement**: Rork 側で有効化しても、Apple Developer Portal での Certificate 紐付けが別途必要
- **Dark Mode 対応の目視確認**: Rork 生成時に Dark Mode の色指定が抜けていることがあり、Xcode のプレビューで両モード確認
- **VoiceOver 対応**: アクセシビリティラベルが生成時に不足しがちなので、Accessibility Inspector で再確認
- **起動時間の実機計測**: TestFlight で実機にインストールし、ストップウォッチで起動時間を測る（シミュレータと実機で倍以上違うことがある）
