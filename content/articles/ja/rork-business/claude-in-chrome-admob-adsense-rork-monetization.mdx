## はじめに

Rork でモバイルアプリを構築し、ユーザー数が増えてきたら、次のステップは収益化です。広告収益（AdMob）とウェブサイト訪問者からの収益（AdSense）は、多くの開発者にとって主要な収入源となります。

しかし、複数のダッシュボードを毎日チェックし、データを分析し、最適化するのは時間がかかります。Claude in Chrome を使えば、これらのタスクを自動化でき、あなたは本来の仕事—ユーザーに価値を提供するアプリ開発に専念できます。

本記事では、Rork で開発したアプリの広告収益を最大化するための、Claude in Chrome を活用した自動化戦略を詳しく解説します。

## Rork アプリの収益化パイプライン

### 広告フォーマット別の収益構造

Rork で構築したモバイルアプリには、複数の広告フォーマットを統合できます。各フォーマットの特性を理解することが、収益最大化の第一歩です。

| 広告フォーマット | 説明 | eCPM | ユーザー体験 | 推奨配置 |
|----------|------|------|---------|---------|
| **バナー広告** | 画面上部・下部の小さな広告 | $1-5 | 低い（常時表示） | ホーム、リスト |
| **インタースティシャル** | フルスクリーン広告 | $3-10 | 中（操作の途中で表示） | レベル終了、タブ切り替え |
| **リワード広告** | 視聴でアプリ内通貨獲得 | $5-15 | 高い（ユーザー主動） | ボーナス、ガチャ |
| **アプリ内広告SDK** | 特定のアドネットワークの広告 | 変動 | 中〜高 | プレイ間のインターバル |

Rork アプリが複数フォーマットを採用している場合、各フォーマットの収益性をトラッキングし、最適な配置を見極めることが重要です。

### 収益化パイプラインの全体像

```
┌─────────────────────────────────────────┐
│ Rork アプリ（ユーザーが利用）            │
└──────────────┬──────────────────────────┘
               │
      ┌────────┴─────────┐
      ↓                  ↓
   AdMob            AdSense
 （モバイル）      （Web：ランディングページ）
      │                  │
      └────────┬─────────┘
               ↓
   Claude in Chrome
   （自動監視・最適化）
```

## Claude in Chrome で AdMob Console を自動操作

### 広告ユニットの一括作成

Rork アプリをリリースする際、複数のプラットフォーム（iOS、Android）やゲームステージごとに、複数の広告ユニットを作成する必要があります。

**例：Rork ゲームアプリの広告ユニット一括作成**

```
Claude in Chrome への指示：
「AdMob Console https://admob.google.com にアクセスし、
Rork アプリ 'rork-puzzle-game' に以下の広告ユニットを作成してください：

【Android ユニット】
1. 'android_home_banner'
   - フォーマット: バナー広告（320×50）
   - 配置: ホーム画面

2. 'android_stage_interstitial'
   - フォーマット: インタースティシャル
   - 配置: ステージ失敗時

3. 'android_hint_rewarded'
   - フォーマット: リワード広告
   - 配置: ヒント獲得

【iOS ユニット】
4. 'ios_home_banner'
   - フォーマット: バナー広告（320×50）
   - 配置: ホーム画面

5. 'ios_stage_interstitial'
   - フォーマット: インタースティシャル
   - 配置: ステージ失敗時

6. 'ios_hint_rewarded'
   - フォーマット: リワード広告
   - 配置: ヒント獲得

各ユニット作成後、以下の形式で出力してください：
{
  \"android_home_banner\": \"ca-app-pub-xxxxxxxxxxxxxxxx/xxxxxxxx\",
  \"android_stage_interstitial\": \"...\",
  ...
}」
```

Claude in Chrome がこれらのユニットを順序立てて作成し、構造化されたデータを出力します。それを Rork アプリコードに統合するだけで、広告配信がすぐに開始されます。

### eCPM・収益レポートの自動取得

広告ユニットを設定したら、各ユニットの eCPM（1000インプレッションあたりの推定収益）と実際の収益を毎日トラッキングします。

**例：日次収益レポート自動取得**

```
「AdMob Console の Reports セクションで、
過去7日間の以下を広告ユニット別に取得してください：

1. 推定収益（USD）
2. インプレッション数
3. クリック数
4. CTR（%）
5. eCPM（USD）
6. フィルレート（%）

データを CSV 形式で出力し、
以下の分析も追加してください：

【eCPM 分析】
- 最も eCPM の高いユニット
- 最も eCPM の低いユニット
- 平均 eCPM

【フィルレート分析】
- フィルレートが高い（>80%）ユニット
- フィルレートが低い（<50%）ユニット
- フィルレート改善の推奨アクション」
```

こうしたレポートを毎日自動取得することで、以下が見えてきます：

- どの広告フォーマット・配置が最も収益性が高いか
- フィルレート（広告配信率）が低いユニットはないか
- eCPM が下降傾向にないか

### メディエーション（複数アドネットワーク最適化）

AdMob だけに頼るのではなく、複数のアド・ネットワークから最良の入札価格を自動選択するメディエーション機能を活用します。

**例：メディエーション・グループの設定と監視**

```
「AdMob Mediation セクションで、
'rork-puzzle-game' 用のメディエーション・グループを作成してください：

【グループ設定】
グループ名: 'all_formats_mediation'
対象フォーマット: バナー、インタースティシャル、リワード

【アドネットワーク チェーン】
優先順位順：
1. Google AdMob ネットワーク（優先度：最高）
   - アルゴリズム: 最高入札価格を自動選択

2. Meta Audience Network
   - CPM フロア: $2（$2未満の入札は無視）

3. Pangle（ByteDance）
   - CPM フロア: $1.5

4. Unity Ads
   - CPM フロア: $1.0

【テスト設定】
テスト装置を登録し、実際の配信をテスト

設定後、各ネットワークのパフォーマンスを表示してください：
- ネットワーク名
- 填充率（eCPM で選ばれた率）
- 平均 eCPM」
```

メディエーション・グループを適切に設定することで、AdMob のみの場合と比較して、eCPM を 20-40% 向上させることが可能です。

### A/B テスト結果の確認

異なるプレースメント位置や広告フォーマット、アドネットワークの組み合わせで収益が変わります。これを検証するため、A/B テストを実施します。

**例：バナー広告の配置最適化テスト**

```
「AdMob で以下の A/B テストを実施・監視してください：

テスト名: 'banner_placement_optimization'
期間: 過去14日間

【テスト内容】
コントロール（現在の設定）:
- バナー広告の配置: 画面下部
- バナーサイズ: 320×50

テストバリアント:
- バナー広告の配置: 画面上部
- バナーサイズ: 320×50

【測定結果】
各バリアント別：
1. インプレッション数
2. クリック数
3. CTR（%）
4. 推定収益
5. eCPM

どちらのバリアントが勝利したか判定し、
推奨される次のアクションを提案してください」
```

A/B テストを定期的に実施することで、同じユーザーベースでも、工夫次第で収益を 10-30% 増加させることができます。

## Firebase Analytics × AdMob の統合分析

Firebase Analytics と AdMob を連携させることで、ユーザー行動と広告収益の関係を詳細に分析できます。

### ARPU（ユーザーあたり収益）のモニタリング

```
「Firebase Console の Analytics セクションで、
以下の指標を確認してください：

【基本ユーザー指標】
- DAU（日次アクティブユーザー）
- MAU（月次アクティブユーザー）
- 新規ユーザー数

【収益指標（AdMob統合時）】
- ARPU: （合計推定収益） / DAU
- ARPPU: （合計推定収益） / 課金ユーザー数
- LTV: ユーザーのライフタイム価値

【例】
- DAU: 10,000
- 推定収益: $300
- ARPU: $0.03（= $300 / 10,000）

【分析】
- 過去7日間の ARPU トレンド
- ARPU が上昇しているか下降しているか
- 国別の ARPU 比較（ある場合）」
```

ARPU が高いほど、ユーザーあたりの収益性が高いアプリです。継続的に ARPU を向上させることが、事業成長の鍵となります。

### コホート分析（ユーザーセグメント別分析）

インストール日が異なるユーザーグループ（コホート）ごとに、継続率と収益性を分析します。

```
「Firebase Analytics で、以下のコホート分析を実施してください：

【コホート定義】
インストール日別にグループ化

【測定指標】
- Day 1 リテンション（%）
- Day 7 リテンション（%）
- Day 30 リテンション（%）
- 各コホートの平均 ARPU

【質問】
1. 最も継続率の高いコホートはどれか
2. 最も継続率の低いコホートはどれか
3. 新しいコホートは古いコホートと比較して
   継続率が向上しているか下降しているか
4. ARPU が高いコホートに共通する特性

【改善施策の検討】
継続率や ARPU が低いコホートに対して、
どのような施策が有効か提案してください」
```

コホート分析から、「新しいユーザーは継続率が低い」「特定の期間のユーザーは ARPU が高い」といった傾向が見えます。これらの情報は、ゲームのアップデート計画や広告配置の最適化に活用できます。

## よくある質問（FAQ）

### Q1. バナー広告とインタースティシャル広告、どちらを優先すべきですか？

以下の基準で判断してください：

| 指標 | バナー | インタースティシャル |
|-----|--------|-------------|
| eCPM | $1-5 | $3-10 |
| ユーザー体験 | 低い（常時表示） | 中（割り込み） |
| クリック率 | 低い（0.5-1%） | 高い（2-5%） |

**推奨：** 両者を組み合わせる
- バナー: 常時表示で継続的な収益
- インタースティシャル: ユーザーの自然な操作完了時に表示（リテンション損傷を最小化）

バナーとインタースティシャルの比率は 60:40 または 70:30 から開始し、A/B テストで最適化してください。

### Q2. eCPM が低い場合、どのように改善できますか？

以下の施策を試してください（優先順位順）：

1. **メディエーション設定の見直し**
   - AdMob 以外のアド・ネットワーク（Meta、Pangle など）を追加
   - CPM フロア値を適切に設定
   - 結果として eCPM が 20-40% 向上することが多い

2. **トラフィック品質の向上**
   - 不正ユーザー・ボットフィルター設定
   - 低質なトラフィック源を排除

3. **地理的ターゲティング**
   - 高 eCPM 国（米国、カナダ、北欧など）からのトラフィック増加
   - 低 eCPM 国のトラフィック削減

4. **広告フォーマットの最適化**
   - リワード広告を増やす（eCPM が高いため）
   - バナー広告の配置を変更（テスト必須）

### Q3. Rork で複数のアプリを公開しており、全て Claude in Chrome で監視したいです。どうしますか？

以下の方法が推奨されます：

**方法 1: アプリ別に複数タスク作成**
```
- task-1: Rork アプリ A の監視（毎朝8:00）
- task-2: Rork アプリ B の監視（毎朝8:15）
- task-3: Rork アプリ C の監視（毎朝8:30）
```

**方法 2: 統合タスク作成（全アプリを1つのレポートに）**
```
- task-1: 全アプリの収益を集約したダッシュボード生成（毎朝8:00）
```

方法 2 の場合、1つのタスク実行時間が長くなるため、複雑さを避ける（詳細分析は削減）ことが重要です。

### Q4. AdMob のポリシー違反で収益が停止されました。Claude in Chrome で防げますか？

残念ながら、Claude in Chrome は AdMob ポリシー違反を防ぐことはできません。ただし、以下の定期監視により、問題を早期に発見できます：

```
「AdMob Console の Policy Center セクションを毎週チェックし、
警告や制限がないか確認してください」
```

**予防策：**
- ユーザーベースを理解する（子ども向けアプリなど）
- クリック詐欺やボット対策を強化
- 広告ネットワークのポリシーを定期的に確認

### Q5. リワード広告を見すぎたユーザーがいて、広告うんざりになっていないか心配です。どう監視しますか？

Firebase Analytics のカスタムイベントで、ユーザーのリワード広告視聴数を追跡し、過剰視聴者の比率を分析できます。

```
Claude in Chrome への指示：
「Firebase Analytics で、ユーザーセグメント別に以下を分析してください：

- リワード広告平均視聴数（1日あたり）
- ユーザーの日数ごとの継続率
- リワード広告視聴数が多いユーザー（>5回/日）の
  Day 1/7/30 リテンション

リワード広告視聴数が多いユーザーの継続率が
低い場合は、広告疲れが起きている可能性があります」
```

リワード広告の視聴数に上限を設ける、または視聴後にクールダウン時間を設けることで、ユーザー満足度を保ちながら継続的な収益を確保できます。
