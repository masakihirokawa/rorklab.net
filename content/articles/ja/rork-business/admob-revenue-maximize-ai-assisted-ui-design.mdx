## はじめに

AdMob で稼ぐアプリ開発者にとって、最大の課題は「どうやって広告収益を上げるのか」。

多くの開発者は、アプリをリリースしたら広告を貼るだけで終わり。だが、ほんの少しの UI 設計の工夫で、AdMob 収益を **2 倍～ 3 倍に伸ばす** ことは珍しくない。

本記事では、Rork を使ってデータドリブンに高収益 UI を設計する方法を、ステップバイステップで解説する。Firebase Analytics、A/B テスト、そして実際の改善事例まで、すべて含む。

## なぜ UI 設計がAdMob 収益を左右するのか——データが示す事実

### AdMob 収入の仕組み

AdMob で稼ぐメカニズムは簡単：

**収入 = インプレッション数（表示回数）× CTR（クリック率）× CPM（クリックあたりの単価）**

ほとんどの開発者は「CPM を上げたい」と思うが、CPM はGoogle が決めるので、開発者には制御不可。

**制御できる 2 つの要因：**

1. **インプレッション数**：1 ユーザーセッション内で、広告がいくつ表示されるか。
2. **CTR（クリック率）**：表示された広告のうち、いくつクリックされるか。

### 典型的な改善前後の数字

改善前：
- 日次アクティブユーザー（DAU）：5,000。
- セッション長：3 分。
- インプレッション数/セッション：1.2。
- CTR：3%。
- CPM：50円。
- **日次収益**：900円。

改善後（UI 最適化）：
- DAU：5,000（変わらず）。
- セッション長：3.5 分（UI が邪魔にならず、ユーザーが長く遊ぶ）。
- インプレッション数/セッション：2.0（広告配置を増やす）。
- CTR：5%（広告が目立つ位置 + 高クリック誘発デザイン）。
- CPM：50円（変わらず）。
- **日次収益**：2,500円。

**差分**：900円 → 2,500円 = **2.8 倍**。

この改善のほぼすべてが、UI デザインで実現できる。

## 高収益 UI の条件——広告配置の原則と科学的根拠

### 広告の「黄金配置」

広告が効果的な位置は、心理学と UX 研究に基づいている。

**最高のパフォーマンス：ゲーム終了画面**

ユーザーがゲームオーバー画面を見たときが、最もリラックスしており、かつ「何か起こるのを待っている」状態。ここにインタースティシャル広告（全画面広告）を表示すると：

- CTR：8-12%（通常の 3-5% から上昇）。
- ユーザーのいら立ち：低い（自然な流れ）。

**次点：メニュー画面やポーズ中**

ゲームを一度中断したユーザーは、再開ボタンを押すまでの「待ち時間」に広告を見る心理的準備ができている。

例：「再開」ボタンの上に 300x250 バナー広告。

**避けるべき：コンテンツの真ん中**

ユーザーが集中している最中に広告を割り込ませると、UX が台無し。評価が下がり、離脱率が上昇。

### 広告フォーマットの選択基準

#### バナー広告（320x50, 728x90）

**メリット**：
- 邪魔が少ない。
- CTR は低い（1-2%）が、常に表示されるのでインプレッション数が多い。

**デメリット**：
- スマートフォンの小さい画面では領域を圧迫。

**向き**：
- 画面上部または下部。
- ゲーム画面のスコア表示エリアと一緒など、不可侵領域外。

#### インタースティシャル広告（全画面）

**メリット**：
- 大きく目立つ。
- CTR が高い（8-15%）。
- 単価も高い（CPM 100-200円）。

**デメリット**：
- タイミングを間違えると、ユーザーがアプリを削除する。
- 頻度が高いと、評価が 2星に落ちる。

**向き**：
- ゲーム終了時。
- レベル完了時。
- メニュー画面に戻る際。

#### リワード広告（報酬付き）

**メリット**：
- ユーザーに明確な利益（ゲーム内通貨、ボーナス）がある。
- CTR が超高い（20-40%）。

**デメリット**：
- ユーザーはわかって見ているので、CPM が低い（30-50円）。
- ゲーム内経済バランスを壊す（無料ユーザーがリワード広告だけでクリアできてしまう）。

**向き**：
- 復活回数が限定されるゲーム（「死亡後、広告を見て復活」）。
- ゲーム内通貨の購入代替え。

### 推奨レイアウト：バナー + インタースティシャル + リワードの 3 層構成

```
[Game Screen]
================
|      Game     |
|     ←タップ   |
================
|  [バナー広告]  |  ← 下部の固定領域、常時表示
================

[Menu Screen]
================
| [バナー広告]   |  ← 上部
================
|  再開ボタン    |
| レベル選択     |
| 設定           |
================

[Game Over Screen]
================
|             |
| [インタースティ|  ← タップで閉じるまで表示
|  シャル広告]   |
|             |
================
```

この構成で：
- バナー広告：常時インプレッション数を稼ぐ。
- インタースティシャル：ゲーム終了のタイミングで高 CTR を獲得。
- リワード広告：ユーザーが選択的に見るので、エンゲージメント低下なし。

## Rork でのAIアシスト型 UI 設計——プロンプトから高収益レイアウトを生成

RorkのAIを使えば、「高収益 UI」を自動設計できる。コツは、プロンプトに「広告配置」を明示的に指定すること。

### Step 1: 高収益仕様をプロンプトに盛り込む

例：ゲームアプリの場合

```
Create an iOS game app called "Block Rush" with the following:

**Game Mechanics**:
- Stack colorful blocks from falling.
- Game over when blocks reach the top.
- Score increases by 10 per block placed.
- 3 lives per game.

**Ad Monetization (HIGH REVENUE)**:
- Bottom banner ad (320x50) always visible during gameplay.
  - Place below game score display.
  - Don't overlap with game area.
- Fullscreen interstitial ad (Admob) on game over.
  - Show immediately when user loses.
  - Provide "Retry" and "Menu" buttons after ad closes.
- Rewarded ad option on game over.
  - Button text: "Watch Ad + 1 Extra Life"
  - Only show if user has < 3 lives remaining.

**UI Design**:
- Minimalist, dark theme (black background, white text).
- Score at top-left, lives indicator at top-right.
- Large, tappable buttons (44pt minimum).
- No text overlapping ad areas.
- Dark mode support.

**Analytics ready**:
- Track: game_start, game_over, ad_impression, ad_click
- Include Firebase event logging hooks in code comments.

Priority: Maximize ad revenue while keeping UX smooth.
```

このプロンプトでRorkが生成するコードは、広告の配置が最適化されたレイアウトになる。

### Step 2: レイアウト確認＆反復

生成されたコードをダウンロード。シミュレーターで動作確認。

チェックリスト：
- [ ] バナー広告が下部に固定されている。
- [ ] バナーがゲーム画面と重なっていない。
- [ ] ゲーム終了時にインタースティシャル広告が表示される。
- [ ] 広告を閉じたら、メニューに戻る。
- [ ] リワード広告のボタンが明確。
- [ ] 画面向き（縦 / 横）を切り替えても、広告位置が正しい。

問題があれば、Rork に指示：

```
The banner ad at the bottom overlaps the game controls. 
Move the game area up by 60 pixels to accommodate the ad.
```

### Step 3: コードに Firebase イベントログを追加

デフォルトでは、Rorkが生成するコードに分析タグが不足していることがある。Firebase Analytics を追加：

```swift
// In Swift (iOS)
import FirebaseAnalytics

// Game start
Analytics.logEvent("game_start", parameters: nil)

// Game over
Analytics.logEvent("game_over", parameters: [
  "score": score,
  "level": currentLevel
])

// Ad impression
Analytics.logEvent("ad_impression", parameters: [
  "ad_type": "interstitial",
  "position": "game_over"
])

// Ad click
Analytics.logEvent("ad_click", parameters: [
  "ad_type": "interstitial"
])
```

## 広告フォーマット選択——バナー・インタースティシャル・リワード の使い分け

前述した 3 フォーマットの詳細なKPI 比較：

| フォーマット | CTR | CPM | 推奨配置 | チャーン影響 |
|---|---|---|---|---|
| バナー | 1-2% | 30-70円 | 上部・下部固定 | 低（許容範囲） |
| インタースティシャル | 8-12% | 100-200円 | ゲーム終了 | 中（頻度に注意） |
| リワード | 20-40% | 30-50円 | ユーザー選択時 | 低（選択制） |

### 数値最適化：各フォーマットの組み合わせ

**パターン A：バナーのみ**
- インプレッション数/セッション：0.8（バナーは常時 1 回）。
- CTR：1.5%。
- 日次収益：1,200円。

**パターン B：バナー + インタースティシャル（毎ゲーム終了時）**
- インプレッション数/セッション：2.5（バナー 1 回 + インタースティシャル平均 1.5 回）。
- CTR：5%（ゲーム終了の高 CTR）。
- 日次収益：3,750円。

**パターン C：バナー + インタースティシャル（3 ゲームごと）+ リワード（ユーザー選択）**
- インプレッション数/セッション：2.2（バナー 1 回 + インタースティシャル 0.5 回 + リワード 0.7 回）。
- CTR：6%（リワード高 CTR が全体を押上げ）。
- 日次収益：3,300円。

**結論**：パターン B（毎ゲーム終了時）が最も効率的。ただし、ユーザー離脱とのバランスを確認必須。

## A/B テスト実践——Rork と Firebase A/B Testing を連携する方法

単純に UI を変更して「これが良いはず」と思い込むのは危険。必ず A/B テストで検証。

### Step 1: Firebase でA/Bテスト設定

Firebase Console（console.firebase.google.com）：

1. **プロジェクト選択 → Grow セクション → A/B Testing**。
2. **新しいテスト作成**。
3. **テスト名**：`test_ad_placement_v2`。
4. **ターゲット**：ユーザーの 50% を対象。
5. **期間**：14 日。

### Step 2: Remote Config で UI バリアント定義

Remote Config（Firebase の設定管理機能）で、広告配置の異なる 2 パターンを定義：

```json
// Variant A (Control) — 現在の配置
{
  "banner_ad_position": "bottom",
  "interstitial_frequency": 1.0,
  "show_reward_button": true
}

// Variant B (Test) — 新しい配置
{
  "banner_ad_position": "top",
  "interstitial_frequency": 0.5,
  "show_reward_button": true
}
```

アプリ起動時、Firebase から設定を取得：

```swift
let remoteConfig = RemoteConfig.remoteConfig()
remoteConfig.fetch { status, error in
  remoteConfig.activate()
  
  let bannerPosition = remoteConfig.configValue(forKey: "banner_ad_position").stringValue
  if bannerPosition == "top" {
    // Variant B: トップに広告配置
    placeBannerAtTop()
  } else {
    // Variant A: ボトムに広告配置
    placeBannerAtBottom()
  }
}
```

### Step 3: 改善指標を定義

テスト期間中に測定する指標：

| 指標 | 定義 | 目標値 |
|---|---|---|
| **ARPU (Average Revenue Per User)** | 1 ユーザーあたりの日平均収益 | Variant B が Variant A より +15% |
| **セッション長** | ユーザーがアプリを使う平均時間 | Variant B が Variant A と同等以上 |
| **チャーン率** | アプリをアンインストールするユーザー% | Variant B が Variant A と同等以下 |
| **CTR** | 広告クリック率 | Variant B が Variant A より +20% |

### Step 4: 結果判定＆導入

14 日後、Firebase で結果を確認：

- **ARPU 差分**：1,500円（A） vs. 1,800円（B） → Variant B が+20%。
- **セッション長**：A = 4 分、B = 3.9 分 → 同等（むしろ B がわずかに短いが、ARPU 向上で実質勝ち）。
- **チャーン**：A = 2.5%、B = 2.4% → 同等。
- **CTR**：A = 4.2%、B = 5.1% → B が勝ち。

**結論**：Variant B を全体展開。ARPU が+20% = 月収が +20%。

- 結果：B が+20% ARPU。

- Month 2：+15% = 2,070円。

## Firebase Analytics × AdMobデータを読み解く——改善すべき指標の優先順位

Firebase と AdMob のデータを組み合わせることで、どこを改善すべきかが見える。

### 読むべき主要指標

**1. ARPU（Average Revenue Per User）**

定義：総広告収益 ÷ ユーザー数。

最重要指標。これを上げれば、あらゆる経営判断が良くなる。

サンプル：
```
AdMob ダッシュボード（過去 30 日）
- 推定収益：150,000円
- 広告リクエスト：500,000
- 有効なインプレッション：450,000
- クリック：22,500
- CTR：5%
- CPM：333円
```

DAU が 5,000 なら、ARPU = 150,000円 ÷ 5,000 = 30円。

月収 = 30円 × 150 日 = 4,500,000円（想定）。

**2. CTR（Click-Through Rate）**

広告がクリックされる率。高いほど良い。

- CTR < 2%：広告配置が悪い。再配置を検討。
- CTR 2-5%：平均的。改善の余地あり。
- CTR > 5%：優秀。現状を維持。

**3. CPM（Cost Per Mille）**

1,000 インプレッションあたりの広告収益。

- CPM が低下 → 広告フォーマットを変更するか、ターゲット国を確認。
- CPM が上昇 → 広告フォーマットが合致している。

**4. セッション長**

ユーザーが 1 回のセッションでアプリを使う時間。

- セッション長が短縮 → 広告が邪魔している可能性。CTR が高くてもセッション長が短いなら、UI を修正。
- セッション長が延伸 → UI が改善されている信号。ARPU も上昇しやすい。

### 優先度付けチェックリスト

```
1. ARPU が月 5% 以上低下？
   → 原因：新広告配置 or CPM 低下。
   → 対策：A/B テスト or 広告フォーマット見直し。

2. CTR が下降（3% → 1.5%）？
   → 原因：広告配置の悪化 or ユーザー疲れ。
   → 対策：配置変更 or インタースティシャル頻度低下。

3. セッション長が短縮（4 分 → 2.5 分）？
   → 原因：広告の割り込みが多すぎる。
   → 対策：インタースティシャル頻度低下 or バナー移動。

4. CPM が低下（500円 → 200円）？
   → 原因：広告フォーマットの変更 or ターゲット地域の変更。
   → 対策：過去 30 日の国別データを確認。海外トラフィックが減った可能性。

5. アンインストール率が上昇？
   → 原因：広告が多すぎる or 新機能のバグ。
   → 対策：アプリレビューを読む。「広告が多い」コメントが多いなら、配置削減。
```

## 月次改善サイクルの回し方——個人開発者が無理なく継続できる仕組み

A/B テストと改善は、毎月繰り返す。だが、個人開発者は時間が限られている。効率的に回すコツ。

### 推奨スケジュール

**Week 1: 仮説立案＋テスト設計**
- Firebase / AdMob のデータを 1 時間分析。
- 「今月はどこを改善するか」を決定。
- 仮説を書く（「バナー位置を上に移動すると、CTR が 5% から 7% に上昇する」）。

**Week 2: Rork で UI 生成＋テスト設定**
- Rork でプロンプト指示：新しい広告配置。
- Firebase Remote Config で バリアント定義。
- TestFlight または内部テストで動作確認。

**Week 3: テスト実施**
- 新 UI をテスト対象（50%）にリリース。
- 監視（バグなし）。

**Week 4: 結果分析＋導入**
- テスト結果を Firebase / AdMob で確認。
- 勝ったバリアントを全体展開。

### ツール最小化

- **分析**: Firebase Console + AdMob ダッシュボード。外部ツール不要。
- **テスト設計**: Firebase A/B Testing。ビルト・イン。
- **UI 生成**: Rork のプロンプト。手動コード修正なし。

月 1 回のサイクルなら、1 週間あたり 5-10 時間の投資で十分。

---

Rork と Firebase を組み合わせることで、個人開発者でもデータドリブンなAdMob 最適化が可能。小さな改善の積み重ねが、3 ヶ月で 50% 以上の ARPU 向上につながる。始めるなら、今がその時。
