個人開発者がB2Bアプリで収益を上げることの難しさは、技術的な問題ではない。「法人相手に営業なんて自分には無理」という思い込みが最大の壁だ。

しかし現実を見ると、法人向けアプリの平均契約単価は個人向けの20〜100倍になる。月980円のサブスクリプションを100人集めるより、月10万円の法人契約を1社取る方が、同じ収益でもストレスは圧倒的に少ない。Rorkで作ったアプリが解決できる業務課題は、あなたが思っているより身近なところにある。

## なぜ個人開発者こそB2Bアプリに向いているのか

### B2CとB2Bの根本的な違いを理解する

B2C（一般消費者向け）の世界では、競合する相手はメルカリ・Lineなどのメガアプリだ。マーケティング予算なしに戦い続けるのは消耗する。一方B2Bの世界は、ニッチな業種の課題を解決するアプリに対して、大手企業が本気で取り組まないケースが多い。

たとえば、工務店向けの現場写真管理アプリを考えてみよう。工務店は日本に約48,000社あり、現場写真の管理に日々悩んでいる。しかし規模が小さいためSalesforceやkintoneのような大手SaaSでは過剰仕様になる。こういう「ちょうどいい粒度の課題」こそ、個人開発者が勝てるB2Bの舞台だ。

**B2CとB2Bの比較:**

- **顧客単価**: B2C 月480〜980円 / B2B 月5万〜50万円
- **解約率**: B2C 月5〜15% / B2B 月0.5〜2%（法人は一度導入すると変えにくい）
- **口コミ**: B2C 偶発的 / B2B 業界内紹介が高確率で発生
- **開発要件**: B2C 常に新機能が必要 / B2B 安定性・サポートが最重要

### 個人開発者の強みがB2Bで活きる

大手SaaSベンダーと個人開発者を比べると、個人は以下の点で明確に有利だ。

**意思決定の速さ**：顧客から「この機能を追加してほしい」と言われたとき、大企業は半年後のスプリントに入れるかを検討する。個人開発者は翌週に対応できる。この速度差が法人顧客から絶大な信頼を生む。

**料金の柔軟性**：標準プランに収まらない顧客にも、カスタム料金を出せる。「50ユーザーで月8万円」「年間一括払いで10%割引」といった交渉が即座にできる。

**担当者との直接関係**：窓口がCEO（あなた）と開発者（あなた）が同一人物なので、担当者は安心感を持つ。問題が起きてもすぐ本人が動く。

### どんなRorkアプリがB2B向けか

B2B化しやすいカテゴリを具体的に挙げる：

1. **現場系**: 建設・製造・農業の現場チェックリスト、写真記録、品質管理
2. **サービス業**: 美容室・整体の予約・顧客カルテ管理、報告書作成
3. **士業サポート**: 税理士・社労士事務所向けドキュメント共有・承認フロー
4. **小売・飲食**: 在庫管理、発注自動化、スタッフシフト管理
5. **教育機関**: 塾・スポーツスクールの出欠管理、進捗レポート自動生成

あなたの既存アプリが上記のどれかに当てはまるなら、B2B展開の可能性がある。

## B2B向けRorkアプリの設計原則

### マルチテナント設計：1つのアプリで複数法人を管理する

B2Bアプリの根幹はマルチテナント設計だ。「テナント」とは法人顧客1社のこと。A社のデータとB社のデータを完全に分離しつつ、1つのバックエンドで管理する仕組みが必要になる。

SupabaseのRow Level Security（RLS）を使うと、Rork Maxで安全なマルチテナント実装ができる。

```typescript
-- Supabase: マルチテナント対応のテーブル設計
-- テナント（法人）テーブル
CREATE TABLE tenants (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name TEXT NOT NULL,
  plan TEXT NOT NULL DEFAULT 'starter', -- starter, professional, enterprise
  created_at TIMESTAMPTZ DEFAULT now()
);

-- ユーザーとテナントの紐付け
CREATE TABLE tenant_users (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id UUID REFERENCES tenants(id) ON DELETE CASCADE,
  user_id UUID REFERENCES auth.users(id) ON DELETE CASCADE,
  role TEXT NOT NULL DEFAULT 'member', -- owner, admin, member
  created_at TIMESTAMPTZ DEFAULT now(),
  UNIQUE(tenant_id, user_id)
);

-- RLSポリシー: 自分のテナントのデータのみアクセス可能
ALTER TABLE tenant_users ENABLE ROW LEVEL SECURITY;

CREATE POLICY "Users can see their own tenant memberships"
  ON tenant_users FOR SELECT
  USING (user_id = auth.uid());

-- 業務データテーブル（例: 現場チェック記録）
CREATE TABLE site_checks (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id UUID REFERENCES tenants(id) ON DELETE CASCADE,
  title TEXT NOT NULL,
  status TEXT DEFAULT 'pending',
  created_by UUID REFERENCES auth.users(id),
  created_at TIMESTAMPTZ DEFAULT now()
);

ALTER TABLE site_checks ENABLE ROW LEVEL SECURITY;

-- RLS: 同じテナントのメンバーのみ閲覧・編集可能
CREATE POLICY "Tenant members can access site checks"
  ON site_checks FOR ALL
  USING (
    tenant_id IN (
      SELECT tenant_id FROM tenant_users
      WHERE user_id = auth.uid()
    )
  );
```

このSQL定義をSupabaseに設定すれば、Rorkアプリ側で特別な条件分岐を書かなくても、自動的にテナント単位でデータが分離される。

### 管理者ダッシュボードの実装

法人顧客が最初に求める機能は「管理者画面」だ。「誰が何を使っているか見たい」「ユーザーを追加・削除したい」というニーズは100%存在する。

```typescript
// Rork Max: テナント管理APIクライアント
// src/lib/tenant-admin.ts

import { createClient } from '@supabase/supabase-js';

const supabase = createClient(
  process.env.EXPO_PUBLIC_SUPABASE_URL\!,
  process.env.EXPO_PUBLIC_SUPABASE_ANON_KEY\!
);

// テナントのメンバー一覧取得
export async function getTenantMembers(tenantId: string) {
  const { data, error } = await supabase
    .from('tenant_users')
    .select(`
      id,
      role,
      created_at,
      users:user_id (
        id,
        email,
        raw_user_meta_data->>full_name as name,
        last_sign_in_at
      )
    `)
    .eq('tenant_id', tenantId)
    .order('created_at', { ascending: true });

  if (error) {
    console.error('メンバー取得エラー:', error.message);
    throw new Error(`メンバー情報の取得に失敗しました: ${error.message}`);
  }
  return data ?? [];
}

// メンバー招待（メールアドレスで招待リンクを送る）
export async function inviteMember(
  tenantId: string,
  email: string,
  role: 'admin' | 'member' = 'member'
) {
  // 1. Supabase Auth でユーザーを招待（メール送信）
  const { data: inviteData, error: inviteError } =
    await supabase.auth.admin.inviteUserByEmail(email, {
      data: { tenant_id: tenantId, role },
      redirectTo: `${process.env.EXPO_PUBLIC_APP_URL}/auth/callback`
    });

  if (inviteError) {
    throw new Error(`招待メールの送信に失敗: ${inviteError.message}`);
  }

  // 2. tenant_users テーブルにレコードを事前作成（招待中の表示用）
  const { error: insertError } = await supabase
    .from('tenant_users')
    .insert({
      tenant_id: tenantId,
      user_id: inviteData.user.id,
      role
    });

  if (insertError) {
    throw new Error(`テナント登録エラー: ${insertError.message}`);
  }

  return { success: true, email };
}

// メンバーのロール変更
export async function updateMemberRole(
  tenantId: string,
  userId: string,
  newRole: 'admin' | 'member'
) {
  const { error } = await supabase
    .from('tenant_users')
    .update({ role: newRole })
    .eq('tenant_id', tenantId)
    .eq('user_id', userId);

  if (error) {
    throw new Error(`ロール更新エラー: ${error.message}`);
  }
}
```

### 監査ログの実装（法人顧客が必ず求める機能）

法人顧客、特に士業や医療・金融系は「誰がいつ何をしたか」の記録を求めてくる。これを「監査ログ」または「アクティビティログ」と呼ぶ。実装のポイントは、業務処理と同時にログを記録するパターンを徹底することだ。

```typescript
// 監査ログ記録ユーティリティ
// src/lib/audit-log.ts

type AuditAction =
  | 'member_invited'
  | 'member_removed'
  | 'record_created'
  | 'record_updated'
  | 'record_deleted'
  | 'report_exported'
  | 'setting_changed';

interface AuditLogEntry {
  tenant_id: string;
  user_id: string;
  action: AuditAction;
  target_type: string;  // 'user' | 'site_check' | 'report' など
  target_id?: string;
  metadata?: Record<string, unknown>;
}

export async function recordAuditLog(entry: AuditLogEntry) {
  const { error } = await supabase
    .from('audit_logs')
    .insert({
      ...entry,
      ip_address: null, // モバイルアプリでは取得不可の場合多い
      user_agent: `Rork/${process.env.EXPO_PUBLIC_APP_VERSION}`,
      created_at: new Date().toISOString()
    });

  if (error) {
    // ログ記録の失敗はサイレントに処理（業務処理をブロックしない）
    console.warn('監査ログ記録失敗:', error.message);
  }
}

// 使用例: レコード作成時に自動でログを記録
export async function createSiteCheckWithLog(
  tenantId: string,
  userId: string,
  data: { title: string; description: string }
) {
  // 業務処理
  const { data: record, error } = await supabase
    .from('site_checks')
    .insert({ tenant_id: tenantId, created_by: userId, ...data })
    .select()
    .single();

  if (error) throw new Error(`作成エラー: ${error.message}`);

  // 監査ログを非同期で記録（業務処理の遅延を防ぐ）
  recordAuditLog({
    tenant_id: tenantId,
    user_id: userId,
    action: 'record_created',
    target_type: 'site_check',
    target_id: record.id,
    metadata: { title: data.title }
  }).catch(console.warn);

  return record;
}
```

## 法人向け料金設計：適切な価格設定が全てを決める

### 3つの課金モデルと使い分け

**ユーザー数課金（Per Seat）**
月額 × ユーザー数で課金する最もシンプルなモデル。予測可能な収益になりやすく、法人担当者も予算計画が立てやすい。

- 適している: ユーザー全員が同様に使うツール（チェックリスト、勤怠、経費）
- 例: 1ユーザー月5,000円、最低5ユーザーから（月2.5万円〜）

**機能別課金（Tiered Feature）**
「スタータープラン」「プロフェッショナルプラン」「エンタープライズプラン」と機能で価格を分けるモデル。上位プランへのアップセルが狙いやすい。

- 適している: 機能の豊富さが差別化になるツール
- 例: スターター月3万円（10ユーザー）/ プロ月8万円（50ユーザー＋API連携）/ エンタープライズ要見積もり

**使用量課金（Usage-Based）**
実際の利用量（処理件数、APIコール数、ストレージ量）で課金するモデル。導入ハードルが低く、成長とともに収益が増える。

- 適している: AI機能を含むアプリ、データ処理量が変動するツール
- 例: 基本料月1万円 + 処理件数1件あたり10円

### StripeでのB2B請求書払い対応

法人顧客の多くが「請求書払い（後払い）」を求めてくる。Stripeはこれに対応しており、Rorkアプリからプログラムで請求書を発行できる。

```typescript
// B2B請求書の自動発行
// src/app/api/billing/invoice/route.ts（Next.js互換のCloudflare Workers）

import Stripe from 'stripe';

const stripe = new Stripe(process.env.STRIPE_SECRET_KEY\!, {
  apiVersion: '2024-06-20'
});

export async function POST(request: Request) {
  const { tenantId, planId, userCount } = await request.json();

  try {
    // テナント情報をDBから取得
    const tenant = await getTenantById(tenantId);

    // Stripeカスタマーが未作成なら作成
    let customerId = tenant.stripe_customer_id;
    if (\!customerId) {
      const customer = await stripe.customers.create({
        name: tenant.company_name,
        email: tenant.billing_email,
        metadata: {
          tenant_id: tenantId,
          app: 'your-app-name'
        }
      });
      customerId = customer.id;

      // テナントにStripeカスタマーIDを保存
      await updateTenantStripeId(tenantId, customerId);
    }

    // 請求書アイテムの追加
    const pricePerUser = 5000; // 1ユーザー月5,000円
    await stripe.invoiceItems.create({
      customer: customerId,
      amount: pricePerUser * userCount * 100, // Stripeは円（最小単位）
      currency: 'jpy',
      description: `プロフェッショナルプラン（${userCount}ユーザー）`
    });

    // 請求書の作成と送付
    const invoice = await stripe.invoices.create({
      customer: customerId,
      collection_method: 'send_invoice',
      days_until_due: 30, // 30日後支払い
      metadata: { tenant_id: tenantId, plan_id: planId }
    });

    // 請求書を確定して送付
    const finalizedInvoice = await stripe.invoices.finalizeInvoice(invoice.id);
    await stripe.invoices.sendInvoice(finalizedInvoice.id);

    return Response.json({
      success: true,
      invoice_url: finalizedInvoice.hosted_invoice_url,
      invoice_pdf: finalizedInvoice.invoice_pdf
    });

  } catch (error) {
    console.error('請求書作成エラー:', error);
    return Response.json(
      { error: '請求書の作成に失敗しました' },
      { status: 500 }
    );
  }
}
```

この実装で、管理画面から「請求書発行」ボタンを押すだけで法人担当者にPDFつき請求書メールが届く。支払い確認もStripeのWebhookで自動化できる。

### 年間契約で収益を安定させる

月額課金に加えて年間一括払いオプションを用意すると、以下のメリットがある：

- **開発者側**：年間収益が確定し、キャッシュフローが安定
- **法人側**：通常10〜20%の割引が得られ、予算処理が楽

年間契約への誘導は、最初の提案時に「年間プランだと月額換算でN%お得です」と明示するだけで、30〜40%の法人が選んでくれる経験則がある。

## 最初の法人契約を獲得する3つのルート

### ルート1：既存の個人ユーザーをアップセル

最も成功率が高い方法は、すでに個人プランを使っているユーザーに「法人プランへの移行」を提案することだ。特に個人ユーザーが「チームで使いたい」とApp Storeのレビューや問い合わせフォームに書いてきたら、即座にこのアプローチが有効だ。

**アップセルメールのテンプレート（参考）:**
```
件名: [アプリ名] を会社でご利用いただけます

○○様

いつも[アプリ名]をご利用いただきありがとうございます。

「チームで使いたい」というご要望をいくつかいただいており、
法人向けプランを用意いたしました。

法人プランでは以下が可能です：
- チームメンバー全員のアカウント一括管理
- 管理者ダッシュボードでの利用状況確認
- 請求書払い対応
- 専用サポート窓口

まずは2週間の無料トライアルをご用意しています。
ご興味があればお気軽にご返信ください。

[あなたの名前]
```

返信率は業種にもよるが、個人ユーザーへのメールは10〜20%が来ることが多い。

### ルート2：業界コミュニティへの参加と情報提供

ターゲットの業種が集まるコミュニティ（Facebookグループ、業界団体の懇親会、Slackコミュニティなど）に参加し、課題を持つ人が現れたときに「こういうアプリがあるんですが…」と自然に会話を始める方法だ。

重要なのは、最初から「売ろう」としないことだ。「こんな課題ありませんか？ちょうど解決するものを作っていて、使ってみてフィードバックをもらえれば」という姿勢が信頼を生む。

### ルート3：コールドアウトリーチ（小規模から始める）

ターゲット企業を10〜20社リストアップし、LinkedInや会社のお問い合わせフォームから連絡する方法。返信率は低い（1〜5%）が、量をこなすことで最初の契約につながることがある。

**効果的なコールドメールの3原則：**
1. 「何ができるか」ではなく「相手の課題は何か」から始める
2. メールは5〜6行以内に収める（長いメールは読まれない）
3. 具体的な次のステップを1つだけ提示する（「30分だけ話せますか？」）

## 契約書・NDA・プライバシーポリシーの現実的な対応

### 最低限必要な書類

法人顧客から「利用規約を見せてほしい」「NDAを締結したい」「個人情報保護法への対応は？」と言われた場合の対処法だ。

**個人開発者が最低限用意すべき書類：**

1. **サービス利用規約**：何ができて何ができないか、免責事項、解約条件
2. **プライバシーポリシー**：取得するデータ、利用目的、第三者提供、削除対応
3. **DPA（データ処理契約）**：GDPRや個人情報保護法に対応するための契約書

NDAは相手から提示されることが多い。相手のNDAに署名するのが最もシンプルだ。自分でNDAを作成する場合は、「秘密情報の定義」「有効期間（通常2〜3年）」「例外事項」の3点を押さえれば問題ない。

**テンプレートリソース：**
- 「電子契約 NDA テンプレート」で検索すると、日本の弁護士が公開している無料テンプレートが複数ある
- 弥生やfreeeの書類テンプレートも参考になる
- 本格的なB2B展開を想定するなら、クラウドサインやDocuSignでの電子署名対応が望ましい

### 意外と怖くない契約交渉

「法人から契約変更を求められたらどうしよう」と思うかもしれないが、実際のところ、小規模な法人（社員50名以下）は標準の利用規約で問題ないケースがほとんどだ。

変更を求められるとしたら「SLAの明記」「データの保管場所（日本国内限定）」「解約後のデータ返還」あたりだ。これらは事前に対応方針を決めておけばその場で答えられる。

## 法人向けサポートをひとりでこなすシステム設計

### サポート体制の基本構成

法人顧客は個人ユーザーと違い、「問題が起きたらすぐに直してほしい」というプレッシャーをかけてくる。これをひとりでさばくには、問い合わせが来る前に自己解決できる仕組みを作ることが重要だ。

**推奨するサポート体制：**

1. **FAQ/ヘルプドキュメント**：NotionをPublicで公開し、よくある質問を網羅
2. **チャット窓口**：Intercom（月100ドル〜）またはSlack shared channel（無料で作れる）
3. **メール窓口**：support@yourdomain.com を設定し、Notionのインボックスと連携
4. **障害通知**：Statuspage.io（無料プランあり）で稼働状況を公開

### 定期レポートの自動化

法人顧客は「ちゃんと使われているか」を経営者に報告する義務がある。毎月の利用レポートを自動で送ると、担当者から大変喜ばれる。

```typescript
// 月次利用レポートの自動生成と送信
// src/lib/monthly-report.ts

import { Resend } from 'resend';

const resend = new Resend(process.env.RESEND_API_KEY\!);

export async function sendMonthlyReport(tenantId: string) {
  const tenant = await getTenantById(tenantId);
  const now = new Date();
  const lastMonth = new Date(now.getFullYear(), now.getMonth() - 1, 1);
  const lastMonthEnd = new Date(now.getFullYear(), now.getMonth(), 0);

  // 利用統計を集計
  const { data: stats } = await supabase
    .from('audit_logs')
    .select('action, count', { count: 'exact' })
    .eq('tenant_id', tenantId)
    .gte('created_at', lastMonth.toISOString())
    .lte('created_at', lastMonthEnd.toISOString());

  const totalActions = stats?.length ?? 0;
  const activeUsers = new Set(stats?.map(s => s.user_id)).size;

  // HTMLレポートを生成
  const reportHtml = `
    <h1>${tenant.company_name} 様 月次利用レポート</h1>
    <p>集計期間: ${lastMonth.toLocaleDateString('ja-JP')} 〜 ${lastMonthEnd.toLocaleDateString('ja-JP')}</p>
    <table>
      <tr><td>アクティブユーザー数</td><td>${activeUsers}名</td></tr>
      <tr><td>総アクション数</td><td>${totalActions}件</td></tr>
      <tr><td>ご契約プラン</td><td>${tenant.plan}</td></tr>
    </table>
    <p>今月もご利用いただきありがとうございました。</p>
  `;

  // メール送信
  const { error } = await resend.emails.send({
    from: 'report@yourdomain.com',
    to: tenant.billing_email,
    subject: `[月次レポート] ${tenant.company_name} 様 ${lastMonth.getMonth() + 1}月のご利用状況`,
    html: reportHtml
  });

  if (error) {
    console.error('レポート送信エラー:', error);
    throw new Error('月次レポートの送信に失敗しました');
  }

  console.log(`✅ 月次レポート送信完了: ${tenant.company_name}`);
}

// [[triggers.crons]]
// cron = "0 9 1 * *"  # 毎月1日9:00 UTC
```

このコードをCloudflare Workers + Cronトリガーで動かせば、毎月1日に全法人顧客へ自動でレポートメールが届く。

## よくある落とし穴：B2Bで失敗する7つのパターン

個人開発者がB2Bに挑戦するとき、繰り返し同じ失敗が起きる。事前に知っておくだけで回避できる。

### 1. 価格を低く設定しすぎる

「まず安くして実績を作ろう」という考えは、B2Bでは裏目に出る。安すぎる価格は「大丈夫なのか？」という不信感につながる。また、一度設定した価格を後から上げるのは契約更新時の解約リスクになる。

最初から適切な価格（月3〜5万円以上）を設定し、「高いですが、これだけの価値があります」と説明できる準備をする。

### 2. 無制限のカスタマイズ要求に対応する

「うちのフローに合わせて少し変えてほしい」という要望が必ず来る。1社のためにカスタマイズを続けると、コードが複雑になり、他社への展開が難しくなる。

対応策：カスタマイズは別途費用（時間単価5万〜10万円）として明示し、汎用的な機能として実装できない場合はお断りする勇気を持つ。

### 3. フリートライアルを長くしすぎる

「30日間無料」は個人向けでは有効だが、法人向けでは「30日で終わる前に稟議が通らない」という問題が起きる。法人の意思決定には平均2〜3ヶ月かかることを理解し、「2週間の集中トライアル」か「トライアルなし・即本契約」にする方が結果的に早く収益化できる。

### 4. サポートSLAを約束しすぎる

「24時間以内に対応します」という約束を安易にすると、休日・深夜の対応が発生する。現実的なSLAは「営業時間内（9-18時）の問い合わせは翌営業日以内に初回回答」程度から始める。

### 5. 一社に依存しすぎる

最初の法人顧客が収益の80%以上を占める状態は危険だ。その顧客が解約したとき、事業が一気に傾く。理想は月10万円の顧客を10社以上持つ分散状態だ。

### 6. 口頭での約束を書面にしない

「この機能は来月までに追加します」という口頭の約束が後でトラブルの元になる。機能追加の約束は必ずメールで残し、「現時点では未対応ですが、ロードマップに入れて検討します」という表現で曖昧にしておくのが安全だ。

### 7. 競合の存在を無視する

「うちにしかないアプリ」と思っていても、顧客は必ず他のSaaSと比較する。主要な競合（大手SaaSや他の個人開発者のツール）を把握し、「なぜうちを選ぶべきか」を3点に絞って即答できるようにしておく。

## 収益スケールのロードマップ：月10万円から100万円へ

### フェーズ1：0→1（最初の法人顧客を獲得する・目安3〜6ヶ月）

- 既存の個人ユーザーに法人プランの案内メールを送る
- App Storeのスクリーンショットに「チーム向け」の文言を追加
- 業界コミュニティ（業種ごとのFacebookグループなど）に参加し、関係構築
- **目標**: 月5〜10万円の法人顧客を1〜2社獲得

### フェーズ2：1→10（リファレンス顧客を積み上げる・目安6〜12ヶ月）

- 導入事例（ケーススタディ）をまとめてWebサイトに掲載（顧客の許可を得て）
- 最初の顧客から紹介をもらう（「同業者に紹介してもらえますか？」と直接聞く）
- 1社あたりの単価を上げる（ユーザー数増加時の自動アップセル）
- **目標**: 合計月収50〜100万円

### フェーズ3：10→100（パートナー・代理店戦略・目安1〜2年）

- 業界の有力者（コンサルタント、士業の先生など）にリセラーとして動いてもらう
- 代理店向けに20〜30%のマージンを提供
- OEM版（ホワイトラベル）として他社ブランドで提供する
- **目標**: 合計月収200〜500万円（ひとりでの上限ライン）

## まとめ：最初の一歩を踏み出す

B2Bアプリビジネスは一夜にして成功しない。しかし「最初の1社」を獲得できれば、確実に次の2社、3社につながっていく。技術力はRorkで補える。あとはあなたが「営業」という体験を積むだけだ。

まず今日できることは一つ：既存の個人ユーザーのメールアドレスリストを確認し、「チームで使いたい」と書いたことがある人を探してみる。そこにB2Bへの第一歩がある。

Rorkで作ったアプリは、あなたが思っているより多くの法人課題を解決できる力を持っている。
