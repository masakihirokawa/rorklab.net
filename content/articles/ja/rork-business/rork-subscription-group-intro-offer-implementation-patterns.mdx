Rorkで作ったアプリで月額¥980のサブスクリプションを公開してみたら、初月の解約率が60%を超えていて頭を抱えた、という話を最近いくつか聞きました。私自身も似たような数字を出したことがあります。原因を一つずつ潰していくと、ほぼ全てのケースで「Subscription Group の設計とイントロオファーの出し方」に問題が集中していました。

この記事は、Rorkで生成したアプリにStoreKit 2 ベースのサブスクを実装している方向けに、Subscription Group をどう階層化し、3種類のイントロオファー（Free Trial / Pay As You Go / Pay Up Front）をどう使い分けるかを、実装コード込みでまとめたものです。テクニカルな実装パターンを示しつつ、なぜそう設計するのかという判断基準まで踏み込みます。

公式ドキュメントには断片的な情報がありますが、「個人開発者がRorkで月額アプリを作って公開する」という具体的な文脈に落とし込んだ情報は意外と少ないので、3年ほど自分のアプリで試行錯誤してきた結果も交えて書いていきます。

## Subscription Group が「アプリの収益設計の土台」になる理由

Subscription Group はAppleが定めた「複数のサブスクリプションプランを互いに排他的なグループにまとめる仕組み」です。同じグループ内のプランは「ユーザーがいずれか1つだけアクティブにできる」という制約があり、プラン間の切り替え（アップグレード／ダウングレード）が自動で処理されます。

ここで重要なのは、Subscription Group の設計が単なる技術的な分類ではなく、「ユーザーの購入動線そのもの」を決定するということです。Rorkで生成されるStoreKit 2のひな形コードはシンプルなので、グループ設計を後から変えるのも比較的楽ですが、一度ストアに公開してアクティブな課金ユーザーがいる状態で大幅な再設計をすると、既存ユーザーの料金体系をどう移行するかで悩むことになります。最初の設計が後々まで効いてきます。

私が過去に陥った典型的な失敗は「全プランを1つのグループに突っ込む」というものです。月額・年額の他にも「ライフタイム買い切り」を同じグループに混ぜたり、機能が大きく違う上位プラン（プロ版）まで同じグループに入れたりすると、ユーザーが選択肢を比較しづらくなり、結局「とりあえず月額」を選んで初月で解約していくパターンが頻出します。

### 階層化の3つのパターン

私が今おすすめしているのは、以下の3つのパターンのいずれかです。

- **シンプル型**: グループ1つ・プラン2つ（月額・年額）。機能差は無く、価格差のみ。新規アプリや初リリース版に向いています
- **アップセル型**: グループ1つ・プラン4つ（月額・年額・月額プロ・年額プロ）。プロ版にだけ含まれる機能を明確に分けます
- **マルチグループ型**: グループ2つ。例: グループA（基本サブスク）+ グループB（追加機能のアドオン）。本格的に成長フェーズに入ったアプリ向け

最初のリリースでは「シンプル型」をおすすめします。プロ版を出したくなったら、後から同じグループに追加するか、別グループとして公開するかを売上データを見ながら判断できます。

## イントロオファーは「3種類」あることを正しく知る

イントロオファーには3つの形式があります。Rorkで生成されたコードはどれもサポートしますが、いつどれを使うべきかは公式ドキュメントには書かれていません。それぞれの特性と、私が現場で使っている判断基準を書いておきます。

- **Free Trial（無料トライアル）**: 一定期間無料で全機能を使える。期間終了時に自動で本料金に切り替わる。**最も新規獲得力が高いが、解約率も高い**
- **Pay As You Go（割引導入）**: 一定期間だけ通常価格より安い金額を毎週／毎月課金。例: 通常¥980/月 → 最初の3ヶ月は¥480/月。**継続率が高く、後半で本料金に上がる時の解約も少なめ**
- **Pay Up Front（一括前払い割引）**: 一定期間分を割引価格で一括前払い。例: 6ヶ月分¥3,980（通常¥5,880）。**解約しづらいので継続率が一番高いが、初回の心理的ハードルが高い**

それぞれを「アプリの種類」ごとにマッピングすると次のようになります。

- **習慣化系（フィットネス・瞑想・学習）**: Pay As You Go が相性◎。「最初の3ヶ月¥480」のような価格で習慣化させる
- **コンテンツ消費系（ニュース・動画・電子書籍）**: Free Trial が定番。1〜2週間試して残るかどうかが分かりやすい
- **ツール系（生産性・写真編集・タスク管理）**: Pay Up Front が向くケースが多い。半年〜1年単位で必要性を判断する人が多いため
- **ニッチ・専門ツール**: Free Trialよりも、最初から月額プランで売る方が「本気のユーザー」だけが残ってLTVが高くなることもある

## 実装1: Subscription Group をRorkアプリで読み込む

ここからは実装に入ります。RorkがStoreKit 2のひな形を出してくれている前提で、それを実用品質に拡張するパターンを書きます。

まず、アプリ起動時に App Store Connect で定義した全Productを読み込み、Subscription Group ごとに分類します。Rorkが出してくれるコードは「productIDの配列を渡してProductを取得する」までは書かれていますが、グループごとに分類するロジックまではないので、これを追加します。

```swift
// SubscriptionGroupManager.swift
// 目的: App Store Connect から取得した Product を Subscription Group ごとに分類し、
//       UI が「グループ単位でプラン選択画面を出す」ことを可能にする
import StoreKit

@MainActor
final class SubscriptionGroupManager: ObservableObject {
    /// グループID（App Store Connect で発行される String）→ そのグループに属する Product
    @Published private(set) var groupedProducts: [String: [Product]] = [:]

    /// 取得対象の全 Product ID（Rork が environment 変数から読むなら .env で管理）
    private let allProductIDs: [String]

    init(productIDs: [String]) {
        self.allProductIDs = productIDs
    }

    /// アプリ起動時または Paywall 表示直前に呼ぶ
    func loadProducts() async {
        do {
            let products = try await Product.products(for: allProductIDs)
            // subscription が nil の Product（消耗型 IAP など）は除外する
            let subscriptionProducts = products.filter { $0.subscription != nil }

            // Subscription Group ID 単位で分類する
            let grouped = Dictionary(grouping: subscriptionProducts) { product in
                product.subscription?.subscriptionGroupID ?? "unknown"
            }

            // 各グループ内で価格の安い順に並べる（年額が一番安く、月額が次、最上位プランは別グループで）
            self.groupedProducts = grouped.mapValues { products in
                products.sorted { $0.price < $1.price }
            }
        } catch {
            // ネットワークエラーや App Store 障害時のフォールバック
            // → エラーログに送信（Sentry 等）し、ユーザーには「再試行ボタン付きエラー画面」を出す
            print("[SubscriptionGroupManager] Failed to load products: \(error)")
            self.groupedProducts = [:]
        }
    }
}

// 期待される動作:
// loadProducts() 完了後、groupedProducts は次のような構造になる
// [
//   "21482710": [Product(月額¥980), Product(年額¥9,800)],   ← 基本グループ
//   "21482711": [Product(プロ月額¥1,980), Product(プロ年額¥19,800)] ← プロ版グループ
// ]
```

`subscriptionGroupID` は App Store Connect でサブスクを作成した時点で自動採番される文字列です。コード上ではこの ID を `groupID` 定数として `Constants.swift` などに定義しておくと、UI 側で「このグループのプランだけを表示」というフィルタリングが楽になります。

ここで「なぜ Dictionary に分類してから処理するのか」というと、後で出てくるイントロオファー判定が「ユーザーが過去にそのグループ内のいずれかのプランを利用していたか」をチェックする仕組みになっているからです。グループ単位で考えるのが StoreKit 2 の自然な使い方です。

## 実装2: イントロオファーの「資格」を判定する

イントロオファーは「初回購入者のみ」あるいは「特定の条件を満たすユーザーのみ」に表示する必要があります。ここをサーバー側で雑に処理すると、解約→再加入で何度も無料トライアルを繰り返されたり、Apple のレビューで「資格判定が不正確」とリジェクトされたりします。

StoreKit 2 では `Product.SubscriptionInfo.isEligibleForIntroOffer(for:)` で資格判定ができます。これを使った実装が以下です。

```swift
// IntroOfferEligibilityChecker.swift
// 目的: 表示しようとしているサブスク Product のイントロオファーを、
//       現在のユーザーが利用可能かどうかを返す
import StoreKit

struct IntroOfferEligibilityChecker {
    /// 指定 Product のイントロオファーがこのユーザーに適用可能かどうか
    /// - Returns: true なら表示してよい / false なら通常価格を表示する
    static func isEligible(for product: Product) async -> Bool {
        guard let subscriptionInfo = product.subscription else {
            // サブスク商品ではない（消耗型 IAP 等）
            return false
        }
        guard let groupID = product.subscription?.subscriptionGroupID else {
            return false
        }
        // 同じ Subscription Group の過去履歴をチェック
        // → 過去にこのグループのいずれかを購入していたら、イントロオファー対象外
        let isEligible = await Product.SubscriptionInfo.isEligibleForIntroOffer(
            for: groupID
        )
        return isEligible
    }
}

// 使用例: Paywall 表示時
struct PaywallView: View {
    @StateObject private var groupManager = SubscriptionGroupManager(
        productIDs: ["com.example.app.monthly", "com.example.app.yearly"]
    )
    @State private var introEligibility: [String: Bool] = [:]

    var body: some View {
        VStack {
            ForEach(groupManager.groupedProducts.values.flatMap { $0 }, id: \.id) { product in
                PlanRow(
                    product: product,
                    showIntroOffer: introEligibility[product.id] ?? false
                )
            }
        }
        .task {
            await groupManager.loadProducts()
            // 各 Product のイントロオファー資格を一括判定
            for product in groupManager.groupedProducts.values.flatMap({ $0 }) {
                introEligibility[product.id] = await IntroOfferEligibilityChecker.isEligible(for: product)
            }
        }
    }
}

// 期待される動作:
// 初回ユーザー: introEligibility = ["...monthly": true, "...yearly": true] → イントロオファー表示
// 過去購入歴あり: introEligibility = ["...monthly": false, ...] → 通常価格のみ表示
```

ここで気をつけたいのは、`isEligibleForIntroOffer` は「同じ Subscription Group」単位で判定されることです。つまり、ユーザーが過去に「月額プラン」を一度でも契約していれば、同じグループの「年額プラン」のイントロオファーも適用外になります。これは Apple の仕様であり、サーバー側で判定をオーバーライドすることはできません（してはいけません）。

「お客さん同士で家族間共有して何度も無料トライアル取られないか？」という質問もよく受けますが、StoreKit 2 はApple ID単位（より正確にはApple StoreのSubscription Family単位）で判定しているので、よほどのことがない限り抜け道はありません。

## 実装3: Paywall に「最適なオファー」を動的に表示する

ここまでで「資格」は判定できました。次は、Paywall に「複数あるイントロオファーのうち最も訴求力のあるもの」を動的に選んで表示するロジックを書きます。

Apple のドキュメントでは、1つの Subscription Product に複数の `IntroductoryOffer` を設定することはできず、`introductoryOffer` プロパティは単一の Optional です。つまり「1つのプランにつき、提示できるイントロオファーは1つだけ」です。複数オファーを試したい場合は **Promotional Offers** を使うのですが、これはサーバー側で署名が必要で、Rork で生成したコードに後から組み込むには手間が大きいので、本記事ではIntroductory Offer に絞ります。

```swift
// PlanRow.swift
// 目的: 1つのプランカードを表示し、イントロオファー資格があれば
//       その内容を「人間が読みやすい文言」で表示する
import StoreKit

struct PlanRow: View {
    let product: Product
    let showIntroOffer: Bool

    var body: some View {
        VStack(alignment: .leading, spacing: 8) {
            Text(product.displayName)
                .font(.headline)

            if showIntroOffer, let intro = product.subscription?.introductoryOffer {
                introOfferLabel(for: intro)
                    .foregroundColor(.accentColor)
                Text("その後 \(product.displayPrice)/\(periodSuffix(for: product.subscription))")
                    .font(.caption)
                    .foregroundColor(.secondary)
            } else {
                Text("\(product.displayPrice)/\(periodSuffix(for: product.subscription))")
                    .font(.subheadline)
            }
        }
        .padding()
        .background(Color(uiColor: .secondarySystemBackground))
        .cornerRadius(12)
    }

    /// イントロオファーを「無料」「割引」「前払い」と人間語で出す
    @ViewBuilder
    private func introOfferLabel(for intro: Product.SubscriptionOffer) -> some View {
        switch intro.paymentMode {
        case .freeTrial:
            Text("最初の \(periodDescription(intro.period)) 無料")
                .font(.subheadline)
                .bold()
        case .payAsYouGo:
            Text("最初の \(periodDescription(intro.period)) は \(intro.displayPrice)/\(periodSuffix(intro.period))")
                .font(.subheadline)
                .bold()
        case .payUpFront:
            Text("\(periodDescription(intro.period))分まとめて \(intro.displayPrice)")
                .font(.subheadline)
                .bold()
        @unknown default:
            EmptyView()
        }
    }

    private func periodDescription(_ period: Product.SubscriptionPeriod) -> String {
        let unit: String
        switch period.unit {
        case .day: unit = "日間"
        case .week: unit = "週間"
        case .month: unit = "ヶ月"
        case .year: unit = "年"
        @unknown default: unit = ""
        }
        return "\(period.value)\(unit)"
    }

    private func periodSuffix(_ subscription: Product.SubscriptionInfo?) -> String {
        guard let period = subscription?.subscriptionPeriod else { return "" }
        return periodDescription(period)
    }

    private func periodSuffix(_ period: Product.SubscriptionPeriod) -> String {
        return periodDescription(period)
    }
}

// 期待される動作:
// Free Trial の場合:
//   "最初の 7日間 無料"
//   "その後 ¥980/ヶ月"
// Pay As You Go の場合:
//   "最初の 3ヶ月 は ¥480/ヶ月"
//   "その後 ¥980/ヶ月"
// Pay Up Front の場合:
//   "6ヶ月分まとめて ¥3,980"
//   "その後 ¥980/ヶ月"
```

`paymentMode` を switch で分岐するのは Apple の API の構造上ほぼ必須です。これを忘れると「最初の3ヶ月無料」と表示しているのに実は「割引価格で課金される」というバグが起こり、レビューで★1の山になります。私は一度これをやらかして、リリース当日にホットフィックスを出した記憶があります。

## よくある間違いと落とし穴

ここからは、私や周囲のRorkユーザーが実際に踏んだ落とし穴を3つ紹介します。これらは公式ドキュメントには明示されていないか、書かれていても「実装段階で気付くのは難しい」性質のものです。

### 落とし穴1: Sandbox環境ではイントロオファーが「常に資格あり」になる

開発中は Sandbox 環境でテストすることになりますが、Sandbox では多くの場合、`isEligibleForIntroOffer` が常に `true` を返します。「これでイントロオファー判定は完璧」と思って本番にリリースすると、本番環境では `false` を返すユーザーが大量にいて Paywall の表示が崩れる、ということが起こります。

対策は次の2つです。

- 必ず TestFlight 環境（Sandbox とは挙動が違います）で実機テストしてください。同じApple IDで一度購入→解約→再表示のフローを通すと安心です
- 本番リリース前に「家族・友人」に協力してもらい、実際のApple IDでイントロオファーが正しく出るか／出ないかをチェックしてもらう

### 落とし穴2: ローカライズで通貨記号が崩れる

`product.displayPrice` は Apple が自動でロケールごとの通貨記号を付けてくれますが、自前でフォーマットしたい場合（「最初の3ヶ月¥480」のような文言を組み立てる時）に、`product.priceFormatStyle` を使わずに `String(format:)` で組むと、米国ロケールで通貨記号が「JPY」と出てしまう、ヨーロッパで小数点が「,」になっていない、といった問題が出ます。

```swift
// 悪い例
Text("\(intro.price)円")

// 良い例
Text(intro.price.formatted(product.priceFormatStyle))
// または
Text(intro.displayPrice)
```

`displayPrice` はApple が完全にローカライズした文字列を返してくれるので、これを使うのが最も安全です。

### 落とし穴3: グループを後から分割するとユーザーの履歴が「リセットされない」

リリース後に「やっぱりプロ版は別グループにしよう」と思ってグループを分割した時、過去にそのグループにいたユーザーは新しいグループに対しては「初回購入扱い」になります。これは「ユーザー獲得には有利」に見えますが、実は逆効果になるケースが多いです。

例えば、月額を契約していたユーザーが新しいグループのプロ月額に切り替えようとすると、**自動アップグレード処理が働かなくなる**のです。同じグループ内なら StoreKit が日割り計算で差額を清算してくれますが、別グループだと「既存サブスクは継続したまま、新しいサブスクが追加で発生する」状態になります。サポート問い合わせが急増します。

対策としては、グループ分割は本当に最後の手段にしてください。可能なら同じグループに「上位プラン」として追加し、機能差で差別化する設計をおすすめします。

## 実用シナリオ：個人開発のヘルスケアアプリでの設計例

ここまで抽象論が続いたので、具体的なシナリオで設計を組み立てます。例として「日々の運動と食事を記録するヘルスケアアプリ」を Rork で作って公開する場合を考えます。

このアプリの特性は次の通りです。

- 習慣化系（毎日アプリを開いてもらいたい）
- 競合は無料アプリも多く、課金にはハードルがある
- 個人開発で、初期はサーバー機能を最小化したい
- 課金してくれるユーザーには「グラフ可視化」「データのエクスポート」「AIアドバイス機能」を提供する

この前提で、Subscription Group とイントロオファーは次のように設計します。

- **Subscription Group**: 1グループのみ（シンプル型）
- **プラン**: 月額¥580 / 年額¥3,800（年額の方が割安なことを明示）
- **イントロオファー**: 月額に「Pay As You Go」を設定し、最初の2ヶ月¥280。年額には「Free Trial 1週間」を設定

なぜこの設計にするか。

- 習慣化系なので、まず「2ヶ月割引」で続けさせて、3ヶ月目以降に本料金に上がるパターンが効きます。1週間無料で課金開始する Free Trial だと、初週で抜ける人が多い
- 年額プランは購入意欲の高いユーザー向けなので、Free Trial で「全機能を1週間試して納得したら年額¥3,800」という流れが自然
- 月額と年額で異なるオファー形式を使うのは Apple のガイドライン上問題ありません（同じ Subscription Group 内でもプランごとに別のイントロオファーを設定できる）

実装では、Paywall を開いた時に「あなたのおすすめプラン」として年額（Free Trial付き）を上に大きく表示し、月額（割引付き）を下に小さく表示する、というレイアウトにすると、年額への誘導が効きやすくなります。

## まとめ — 今日試すなら、まず1つだけ

ここまで読んでくださった方が今日すぐにできることを1つだけ挙げるなら、**「現在のSubscription Groupに何種類のプランが入っているか」を確認し、もし4つ以上あるなら整理を検討する**ことです。

具体的には、過去30日間のApp Store Connect レポートを見て、「ほぼ売れていないプラン」が含まれていれば、それは選択肢を増やしているだけで購入率を下げている可能性があります。Subscription Group 内のプラン数は2〜3が最もコンバージョンが安定する、というのが私の経験則です。

そこから先は、「自分のアプリの種類は習慣化系／コンテンツ系／ツール系のどれか」を考えて、最適なイントロオファー形式を1つ選んで小さく実装してみてください。設計から実装まで全部いっぺんに変えるよりも、一つずつ仮説検証していく方が、本当に効いた変更が分かりやすくなります。
