# Rork アプリがストアに出せない？審査リジェクト対策 FAQ

Rorkでアプリを生成して、App StoreやGoogle Playに提出したのに、「リジェクト（却下）」されてしまった—これは多くの初心者が経験する悔しい瞬間です。Appleの審査基準は厳しく、小さな問題でもアプリが拒否されることがあります。このFAQでは、Rorkユーザーが最もよく遭遇する7つのリジェクト理由と、その対策方法を分かりやすく説明します。

## Q1: 「Guideline 4.2 — Minimum Functionality」で拒否されました

**A:** 「Minimum Functionality」のリジェクトは、Appleが「このアプリは実用的な価値が不十分である」と判断したことを意味しています。

### なぜこのリジェクトが発生するのか

Apple のガイドライン 4.2 では、アプリには「明確で有用な機能」が必要です。以下のようなアプリはリジェクトされやすいです：

- **プレースホルダーのみのアプリ** — ボタンやテキストが機能していない
- **不完全な機能** — ログイン機能がありながらデータベースに接続していない
- **実用的ではない機能** — ユーザーが実際に使用できない単なるデモ

### 解決方法

**1. 核となる機能が完全に動作するか確認する**

アプリを自分で使用して、以下をテストしてください：
- すべてのボタンが動作しているか
- フォーム入力がデータベースに保存されているか
- ナビゲーションが正常に機能しているか

```javascript
// 悪い例：ボタンが機能していない
<Button title="Save" onPress={() => console.log('Save pressed')} />

// 良い例：ボタンが実際にデータを保存
<Button title="Save" onPress={() => saveDataToDatabase(userData)} />
```

**2. アプリの説明を明確にする**

App Store のアプリ説明に、このアプリが何をするのか、ユーザーが何ができるのかを明確に記述します。

```
× 悪い説明: 「楽しいアプリです」

○ 良い説明: 「このアプリはユーザーがタスクを作成・管理し、
完了したタスクにチェックマークをつけることができます。
リアルタムでデータクラウドに保存されます。」
```

**3. スクリーンショットで実機能を表示する**

App Store に提出するスクリーンショットに、実際に機能しているアプリの画面を含めます。

:::warning
コンセプトアート、文字だけのスクリーンショット、またはプレースホルダー画像を使用しないでください。実際のアプリの機能を表示する必要があります。
:::

## Q2: プライバシーポリシーが無い、または不十分です

**A:** プライバシーポリシーはApp Store 審査の必須項目です。Appleはこれを非常に厳格に評価します。

### プライバシーポリシーが必要な理由

Appleは、アプリがユーザーデータをどのように収集・使用・保護するのかを理解する必要があります。プライバシーポリシーがないアプリはリジェクトされます。

### 解決方法

**1. プライバシーポリシーを作成する**

プライバシーポリシーは、以下の内容を含む必要があります：

```markdown
# プライバシーポリシー

## 収集されるデータ
- ユーザー名、メールアドレス（ユーザーが入力した場合）
- デバイスID（分析のため）
- 位置情報（許可した場合のみ）

## データの使用方法
- ユーザー認証
- サービス改善のための分析
- エラー報告

## データの保護
- すべてのデータはSSL/TLSで暗号化
- サーバーは定期的にバックアップ
- 第三者への共有はしない
```

**2. プライバシーポリシーを公開する**

App Store の「プライバシーポリシーURL」フィールドに、プライバシーポリシーへのリンクを追加します。

**3. ポリシーを最新に保つ**

アプリに新しい機能を追加したときは、プライバシーポリシーも更新します。

:::tip
無料のプライバシーポリシージェネレーター（例：iubenda、Termly）を使用して、簡単にプライバシーポリシーを生成できます。ただし、あなたのアプリに合わせてカスタマイズすることが重要です。
:::

**4. App Store の「App Privacy」セクションを正しく設定する**

Appleは 2020年以降、「App Privacy」（アプリのプライバシーラベル）の設定を義務付けています。以下を正確に回答します：

- アプリが収集するデータタイプ
- そのデータが追跡に使用されるかどうか
- ユーザーが削除できるかどうか

## Q3: スクリーンショットが実際のアプリと違う

**A:** App Store に提出するスクリーンショットがアプリの実際の機能を正確に表現していない場合、リジェクトされます。

### よくあるミス

- **古いスクリーンショット** — アプリが更新されたが、スクリーンショットは古いまま
- **デモ画像** — 実際のアプリではなく、モックアップやコンセプト画像を使用
- **修正されていないバグ** — スクリーンショットに表示されているバグが実際のアプリにも存在

### 解決方法

**1. 最新のスクリーンショットを撮影する**

```bash
# テスト用デバイスでアプリをビルド
eas build --platform ios

# 提供されたテストデバイスでアプリを実行
# アプリの重要な画面をスクリーンショット
```

**2. スクリーンショットが実際のアプリを反映しているか確認**

以下の項目を確認します：
- ボタンが表示されているか、位置は正確か
- テキストが正しく表示されているか
- 画像やアイコンが表示されているか
- レイアウトが正確か

**3. アプリの全画面をカバーするスクリーンショットを準備**

iOS アプリの場合、以下の画面をスクリーンショットに含めます：
- ログイン/スタート画面
- メイン画面
- 主要な機能の画面
- 設定画面

:::warning
Rorkで生成したアプリで、表示されない画像やバグがある場合は、アプリを修正してからスクリーンショットを撮影してください。スクリーンショットとアプリの矛盾はリジェクト理由になります。
:::

## Q4: クラッシュログが検出されました

**A:** Appleがアプリをテストして、クラッシュ（エラーで起動しない）を検出した場合、リジェクトされます。

### クラッシュの一般的な原因

- **メモリリーク** — アプリがメモリを解放しない
- **初期化エラー** — データベース接続やAPIが初期化されていない
- **型エラー** — 変数の型が正しくない
- **ネットワークエラー** — インターネット接続がないときの処理ができていない

### 解決方法

**1. テスト用デバイスでクラッシュをテストする**

```bash
# iOS でテストビルドをインストール
eas build --platform ios --profile preview

# Android でテストビルドをインストール
eas build --platform android --profile preview

# デバイスでアプリを実行し、すべての機能をテスト
```

**2. エラーログを確認する**

```javascript
// エラー処理を追加
try {
  const result = await fetchDataFromAPI();
  setData(result);
} catch (error) {
  console.error('API Error:', error);
  // ユーザーにエラーを表示
  alert('データの読み込みに失敗しました。再度試してください。');
}
```

**3. ネットワーク接続がない状況をシミュレートする**

```javascript
// ネットワークエラーのハンドリング
const fetchWithErrorHandling = async (url) => {
  try {
    const response = await fetch(url);
    if (!response.ok) {
      throw new Error('Network error');
    }
    return await response.json();
  } catch (error) {
    console.error('Fetch error:', error);
    return null; // またはキャッシュされたデータを返す
  }
};
```

**4. Xcode でクラッシュログを確認**

Appleの TestFlight を使用してベータテストを実施します。クラッシュが発生した場合、クラッシュログが表示されます。

:::tip
本番提出前に、複数のデバイスでテストして、クラッシュやバグがないか確認してください。特に古いiOSバージョンやAndroidバージョンでテストすることが重要です。
:::

## Q5: In-App Purchase (アプリ内課金) の設定ミス

**A:** In-App Purchase (IAP) 機能を使用する場合、App Store での設定が複雑で、ミスしやすい項目です。

### よくあるIAP設定ミス

- **App Store Connect での IAP 設定なし** — コードでは IAP を参照しているが、App Store Connect で設定されていない
- **テスト用の IAP 設定がない** — テスト用のサンドボックス IAP がない
- **価格の不一致** — Rork で設定した価格と App Store で設定した価格が異なる

### 解決方法

**1. App Store Connect で In-App Purchase を設定**

```
App Store Connect > マイアプリ > [アプリ名] > In-App Purchases

新しい In-App Purchase を作成：
- Product ID: "com.example.app.premium"
- Tier: $0.99 (例)
- ローカライズされた説明を追加
```

**2. Rork で IAP コードが正しく記述されているか確認**

```javascript
// React Native In-App Purchase の例
import RNIap from 'react-native-iap';

const requestPurchase = async () => {
  try {
    await RNIap.requestPurchase({
      skus: ['com.example.app.premium'], // App Store Connect で設定した ID
    });
  } catch (error) {
    console.error('Purchase error:', error);
  }
};
```

**3. テスト用の Apple ID を用意**

App Store Connect で、テスト用の Apple ID を「テスター」として追加します。このアカウントで TestFlight 経由で IAP をテストします。

:::warning
実際のお金を課金されないようにするため、必ずテスト環境で IAP をテストしてください。本番提出前に、テスト用 Apple ID で購入をシミュレートします。
:::

## Q6: 年齢制限レーティングの設定忘れ

**A:** すべてのアプリには年齢制限レーティングが必要です。設定がないと審査でリジェクトされます。

### 年齢制限レーティングとは

Appleは、アプリが含むコンテンツに基づいて、推奨年齢を決定します。例えば：
- 暴力的なコンテンツ → 12才以上
- 恋愛要素 → 12才以上
- 露出度の高いコンテンツ → 17才以上

### 解決方法

**1. App Store Connect でレーティングアンケートに回答**

```
App Store Connect > マイアプリ > [アプリ名] > 情報 > コンテンツ

以下の質問に回答：
- 暴力的なコンテンツが含まれていますか？
- 恋愛要素が含まれていますか？
- アルコール、タバコ、ドラッグが含まれていますか？
- 言語は不適切ですか？
```

**2. 正確に回答する**

多くの初心者は、すべての質問に「いいえ」と回答して、年齢制限を回避しようとしますが、Appleはアプリの内容を確認するため、不正直な回答はすぐにばれます。

:::tip
ほとんどの基本的なアプリ（ゲーム、生産性ツール、ユーティリティ）は「4才以上」のレーティングが適切です。
:::

## Q7: APIキーがハードコードされている

**A:** APIキー、パスワード、シークレットをソースコードに直接書き込むことは、セキュリティリスクです。Appleはこのようなアプリをリジェクトします。

### なぜこれが危険なのか

ハードコードされたAPIキーは、アプリをリバースエンジニアリングすれば簡単に抽出できます。その結果：
- 不正アクセス
- サーバー攻撃
- 認証情報の盗難

### 解決方法

**1. 環境変数を使用する**

```javascript
// 悪い例：APIキーがハードコード
const API_KEY = "sk-1234567890abcdef";

// 良い例：環境変数を使用
const API_KEY = process.env.EXPO_PUBLIC_API_KEY;
```

**2. .env ファイルを使用**

```
// .env ファイル
EXPO_PUBLIC_API_KEY=sk-1234567890abcdef
EXPO_PUBLIC_API_URL=https://api.example.com
```

```javascript
// app.json または app.config.js で読み込む
const dotenv = require('dotenv');
dotenv.config();

module.exports = {
  expo: {
    extra: {
      apiKey: process.env.EXPO_PUBLIC_API_KEY,
      apiUrl: process.env.EXPO_PUBLIC_API_URL,
    },
  },
};
```

**3. バックエンドサーバーを使用**

最も安全な方法は、APIキーをバックエンドサーバーに保管し、アプリはサーバーを経由してリクエストを送信することです。

```
[Rork App] → [Your Server] → [Third-party API]
          ↑ (APIキーを含まない)
          ↑ (サーバーのみがAPIキーを持つ)
```

:::warning
本番提出前に、すべてのシークレットがソースコードに含まれていないか確認してください。`git grep` や環境変数チェッカーを使用してスキャンします。
:::

## まとめ

Rork で生成したアプリが App Store で審査リジェクトされた場合、以下を確認してください：

1. **アプリが完全に動作しているか** — すべての機能が実装されていることを確認
2. **プライバシーポリシーが含まれているか** — プライバシーポリシーの URL が正確に設定されている
3. **スクリーンショットが最新であるか** — 実際のアプリの画面を反映している
4. **クラッシュがないか** — 複数のデバイスでテストしている
5. **セキュリティが確保されているか** — APIキーなどのシークレットが公開されていない

これらの項目をチェックすれば、ほとんどのリジェクト理由を事前に防ぐことができます。リジェクトされた場合は、Apple から送られてくるメッセージをよく読んで、具体的な理由を理解し、修正した上で再提出してください。
