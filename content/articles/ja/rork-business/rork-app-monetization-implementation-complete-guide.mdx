Rork で作ったアプリが公開されて、最初の1万円が出たところから、本格的な収益化の設計が始まります。実は、収益化の成否はアプリそのものの品質以上に、**「どの収益化モデルを選び、どう組み合わせるか」**で決まります。同じ品質のアプリでも、モデルの選び方と実装の丁寧さで、売上が5倍以上変わることも珍しくありません。

この記事では、私が個人でアプリ開発を10年以上続けてきた経験と、Rork を本格的に使って新しいアプリを仕上げる中で得た学びをもとに、収益化モデルの選択から実装の具体まで、運用で崩れないための設計を書きます。広告・サブスク・アプリ内課金（IAP）・Stripe 連携をどう使い分けるか、どう組み合わせるかを、実装レベルで追っていきます。

## アプリの性格ごとに「主軸」を決める

収益化で最も大事なのは、アプリの性格に合った「主軸」を決めることです。全部をバランスよく入れようとするアプリは、どれも中途半端になり、結果的に売上が伸びません。

アプリの性格を大きく4つに分類すると、それぞれに向いた収益化モデルが見えてきます。

**タイプA: 毎日短時間使うユーティリティ**
家計簿・タスク管理・健康記録など。1回あたりの滞在時間は短いが、開く頻度が高い。**AdMob バナー + リワード動画** が主軸になりやすいタイプ。IAP やサブスクを入れる場合は、「広告を非表示にするプレミアム」など控えめな形が向きます。

**タイプB: 集中して使うコンテンツアプリ**
瞑想・語学学習・フィットネス・読書など。1回の滞在時間が長く、体験が連続的。**サブスクリプション** が主軸になりやすいタイプ。広告はユーザー体験を壊しやすく、使っても動画広告を極少量に留めます。

**タイプC: 創作・編集・生産ツール**
画像編集・動画編集・メモ・イラスト制作など。高度な機能を求めるユーザーが明確に存在する。**IAP（買い切り機能解放） + サブスク** の二段構えが向きます。広告は基本的に入れないほうが、高単価ユーザーの体験を守れます。

**タイプD: ゲーム**
ジャンルにより大きく異なるが、基本的には **IAP（消耗型） + リワード動画 + バナー** の複合。サブスクは一部のジャンル（遊び放題パス等）でのみ効く。

Rork で作るアプリは、タイプ A・B・C のいずれかに収まることが多いはずです。自分のアプリがどのタイプかを最初に決めると、不要な収益化要素を削ぎ落とせます。

## 広告収益化 — AdMob の実装と配置の勘所

AdMob を入れる際の設計ポイントを、実装レベルで整理します。

**バナー広告**は、画面下部の固定配置が王道です。ただし、アプリの主要操作が画面下部にある場合（例: タブバー、主要ボタン）は、衝突を避けるために上部配置か、特定画面のみ表示とします。

```typescript
// React Native での AdMob バナー実装の要点
import { BannerAd, BannerAdSize, TestIds } from "react-native-google-mobile-ads";

const bannerUnitId = __DEV__
  ? TestIds.BANNER
  : "ca-app-pub-XXXXXXXXXXXXXXXX/YYYYYYYYYY";

export function AdBanner() {
  return (
    <BannerAd
      unitId={bannerUnitId}
      size={BannerAdSize.ANCHORED_ADAPTIVE_BANNER}
      requestOptions={{ requestNonPersonalizedAdsOnly: false }}
    />
  );
}
```

**インタースティシャル広告（全画面）** は、ユーザー体験を最も壊しやすい広告です。私のルールは「1セッションに最大1回、主要な達成イベントの直後」です。起動直後や画面遷移のたびに出すのは絶対に避けます。

**リワード動画**は、ユーザーが自発的に選んで見る広告なので、体験を壊しません。配置として有効なのは「プレミアム機能を1回だけ解放」「追加コンテンツをロック解除」といった形です。

AdMob の収益を最大化するうえで最も効くのは、実は**広告の種類選びより、アプリ自体のリテンション**です。DAU が安定していれば、広告単価が多少低くても積み上がります。収益化に凝る前に、まずユーザーに毎日開いてもらえるアプリにするほうが長期的には効きます。

## サブスクリプション — RevenueCat で実装をシンプルに

ネイティブアプリでサブスクを実装するなら、**RevenueCat** の利用が現代的な選択です。iOS/Android 両方のストア課金をラップし、Webhook・復元・レシート検証を一元化してくれます。自前実装は個人開発者の消耗を招きます。

RevenueCat の基本的な組み込み:

```typescript
import Purchases from "react-native-purchases";

// 初期化
await Purchases.configure({ apiKey: process.env.REVENUECAT_API_KEY });

// ユーザー ID の紐付け
await Purchases.logIn(userId);

// オファリング取得
const offerings = await Purchases.getOfferings();
const monthly = offerings.current?.availablePackages[0];

// 購入
const { customerInfo } = await Purchases.purchasePackage(monthly!);

// 権限確認
const isSubscribed = customerInfo.entitlements.active["pro"] !== undefined;
```

サブスクで一番の肝は、**トライアル期間の設計**です。3日〜7日の無料トライアルが業界標準ですが、アプリのタイプによって最適値は変わります。瞑想アプリのように「習慣化までに時間がかかる」カテゴリは7日〜14日、使用頻度が高いツール系なら3日で十分です。

トライアル終了直前の「トライアル期限リマインダー通知」は必ず実装します。ユーザーが「トライアルだったと忘れて課金される」経験は、ストアレビューで酷評される最大の原因の1つです。正直に「明日から月額◯◯◯円になります」と伝えるほうが、長期的な信頼を維持できます。

## 解約率を下げるためのサブスクUX

サブスクの収益を最大化するのは、解約率の低下です。個人開発者が入れるべき施策は次の3つです。

**1. トライアル初日の体験設計**
ユーザーがトライアル開始直後に「これは続ける価値がある」と感じる瞬間を作る必要があります。「トライアル開始時に、初回オンボーディングで1つだけ確実な価値体験を届ける」設計が有効です。

**2. 解約前のダウングレード提案**
解約ボタンを押そうとしたユーザーに、「月額の半額で機能制限版に切り替えできます」という選択肢を出します。完全解約よりも部分的に残ってもらうほうが、LTV は大きく向上します。

**3. 離脱ユーザーへの復帰キャンペーン**
解約後30日経過したユーザーに、「30日間の無料再開」をメールでオファーします。復帰率は5〜10%程度ですが、コストはゼロに近いので必ず仕掛ける価値があります。

```typescript
// 解約前にダウングレード案内を挟む例
async function handleCancelAttempt() {
  const currentPlan = await getCurrentPlan();
  if (currentPlan === "pro") {
    const accepted = await showDowngradeDialog({
      message: "Pro をやめる前に、月額半額の Lite プランはいかがですか？",
      options: ["Lite に変更", "やはり解約する"],
    });
    if (accepted) {
      await changePlan("lite");
      return;
    }
  }
  await openCancellationFlow();
}
```

## IAP（アプリ内課金） — 設計と価格の組み立て

買い切り型の IAP は、特定の機能解放や追加コンテンツに向いています。サブスクに比べて心理的ハードルが低く、ユーザーにとっても「一度払えば一生使える」という安心感があります。

IAP の価格設計で私が使っている型:

- **最小エントリー**: 120〜250円。試しに課金してみる価格帯
- **標準プレミアム**: 600〜980円。主要機能の解放
- **完全版**: 1,800〜2,500円。全機能解放の買い切り

この3段階の設計にすると、「まず小さく払ってみる → 気に入ったら大きく払う」というアンカリング効果が働きます。最初から完全版のみを提示するより、小刻みに段階を用意したほうが総売上は伸びる傾向があります。

ただし、IAP とサブスクの共存は、ユーザーの混乱を招きやすいので慎重に設計します。「月額 480円（全機能使い放題）vs 買い切り 2,400円（全機能永続使用）」のように、**サブスクと買い切りが重複しない切り分け**にするのがコツです。サブスクには継続アップデートを、買い切りには現時点での機能を、と役割を分けるとユーザーは選びやすくなります。

## Stripe 連携 — Web連動やコンテンツ販売で使う

ネイティブアプリのストア課金だけでは対応できないケースで、Stripe 連携が役に立ちます。典型的なのは、以下の3パターンです。

**1. Web版と連動するアプリ**
Webとアプリで同じサブスクを提供する場合、Web側は Stripe、アプリ側は RevenueCat という二本立てにします。RevenueCat は外部購入（Web経由）の情報も一元管理できるので、どちらで契約したかに関わらず、同じ権限で扱えます。

**2. デジタルコンテンツの外販**
アプリが生成したコンテンツ（動画・PDF・画像等）をユーザーが販売できる機能を提供する場合、Stripe Connect を使います。マーケットプレイス型の課金フローを実装できます。

**3. 法人向けサブスクの直販**
App Store / Google Play のサブスクは個人向けの使い勝手に最適化されていて、法人請求には不向きです。法人案件は Stripe で直接請求書を出したほうが、手数料も低く、経理処理もスムーズです。

```typescript
// Stripe Checkout Session を Cloudflare Workers で生成
export async function createCheckoutSession(userEmail: string, env: Env) {
  const stripe = new Stripe(env.STRIPE_SECRET_KEY);
  return stripe.checkout.sessions.create({
    mode: "subscription",
    customer_email: userEmail,
    line_items: [{ price: env.STRIPE_PRICE_ID_PRO, quantity: 1 }],
    success_url: `${env.APP_URL}/billing/thanks`,
    cancel_url: `${env.APP_URL}/billing/cancel`,
    metadata: { source: "rork-app-web" },
  });
}
```

Stripe の Webhook で受け取ったイベントを RevenueCat に反映させる場合は、RevenueCat の Virtual Currencies や Custom Entitlements を活用すると、アプリ側のロジックをシンプルに保てます。

## 収益化UIのタイミング設計

どんなに良い収益化モデルを用意しても、**提示するタイミングが悪いと売れません**。私が意識しているのは次の3点です。

**タイミング1: 価値を実感した直後に見せる**
アプリの主要操作を初めて成功させた直後、「もっと便利に使うには Pro プランがおすすめです」という案内を出します。価値の体験と課金の提示が時間的に近いほど、成約率は上がります。

**タイミング2: 機能制限に当たった瞬間**
無料枠の上限に達した瞬間に、「Pro プランなら無制限に使えます」と自然に見せます。このタイミングは、ユーザー自身が「もっと使いたい」と思った瞬間なので、訴求が一番刺さります。

**タイミング3: アプリを深く使い始めた段階**
連続3日以上利用しているユーザーは、アプリに価値を見出している証拠です。この段階で、トライアル案内を出すと好意的に受け取られやすくなります。逆に初回起動時のトライアル強制提示は、警戒感を生むので避けたほうが無難です。

## 収益 KPI の見方と改善の優先順位

収益化を本格化させると、数字を見る回数が増えます。個人開発者が追うべき KPI は、次の5つに絞って管理すれば十分です。

| KPI | 目標値の目安 | 改善の優先度 |
|---|---|---|
| DAU / MAU（粘着率） | 20% 以上 | 最優先 |
| Day 1 リテンション | 30% 以上 | 高 |
| Day 7 リテンション | 15% 以上 | 高 |
| ARPU（1ユーザー平均収益） | カテゴリによる | 中 |
| サブスク解約率（月次） | 10% 以下 | 中 |

改善の優先順位は、**まずリテンションから**です。リテンションが低いアプリに収益化施策を積み重ねても、穴の空いたバケツに水を注ぐようなものです。リテンションが一定以上に達してから、ARPU や解約率の改善に進みます。

## 税務・経理の現実的な注意点

個人開発者がアプリで本格的に稼ぎ始めると、税務の問題が避けられません。これはかなり軽視されがちですが、早めに整理しておくと後が楽です。

- **アプリ収益は事業所得（または雑所得）として申告**: 年間20万円以上なら確定申告が必要
- **AdMob、Apple、Google、Stripe それぞれからの支払いを別々に記録**: 通貨も入金通貨で記録する
- **源泉徴収分（AdMob 等）は必ず把握**: 二重課税とならないよう、税制上の扱いを理解しておく
- **開業届と青色申告を早めに**: 年の途中で申請しても、その年から65万円控除が受けられる

確定申告が近づく前に、月次で入金と経費を記録しておくと、年度末にバタバタしなくて済みます。freee や マネーフォワード のような会計 SaaS に、AdMob・Apple・Google の入金を月1回手入力するだけで十分です。完璧を目指すより、継続できる最小ルーティンを作るほうが大事です。

## 3ヶ月・6ヶ月・1年の目標線

個人開発者が Rork でアプリを収益化する場合、目標感は次のように持つと現実的です。

- **3ヶ月後**: 初アプリから月5,000〜10,000円の収益。リテンションの基礎を固める期間
- **6ヶ月後**: 2本目のアプリを公開。月20,000〜50,000円。経験と改善の速度が上がる
- **1年後**: 3〜4本のアプリで月100,000〜300,000円。安定収益と新規開発を両立する段階

この数字はあくまで目安で、アプリの性格や時代の波によって大きく変動します。重要なのは、**数字を毎月振り返る習慣を作ること**です。半年前の自分と比べて前進しているか、どの施策が効いたかを定期的に確認することで、次の一手が見えてきます。

Rork という道具を得た個人開発者にとって、収益化は「作れるかどうか」よりも「やりきれるかどうか」の戦いになります。今日の1アプリから半年後の3アプリへ、毎月コツコツ積み上げていく感覚で、この道を楽しんでいきましょう。
