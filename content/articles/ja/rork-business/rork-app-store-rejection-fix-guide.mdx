# Rork アプリ App Store リジェクト — よくある理由と再審査通過の対処法

Rorkでアプリを完成させてApp Storeに提出したのに、「リジェクト（拒否）」通知が来た — そんな時の絶望感は、app開発者なら誰もが経験したことがあるでしょう。悪いニュースと良いニュースがあります。悪いニュースは、リジェクトは誰にでも起こること。良いニュースは、ほぼ全てのリジェクトは**明確な理由**があり、その理由が理解できれば**必ず対応できる**ということです。

このガイドでは、Rorkユーザーが最も多く遭遇する6つのリジェクト理由と、それぞれの対処法、そして実際に使える再審査申し立てテンプレートを紹介します。

## ガイドライン 4.3 — スパム・重複アプリ

### 問題：Rorkの「テンプレートの似通い」が重複判定される理由

Rorkはテンプレートベースのアプリ生成なので、他のRorkユーザーが作ったアプリと「構造」や「レイアウト」が類似していることがあります。Appleはこれを検出すると、以下の理由で拒否します：

- UI/UXが他のアプリと酷似している
- 機能が単なるテンプレートの焼き直しに見える
- ブランディングやカスタマイズが不足している

### 対処法

**1. UIデザインの独自カスタマイズを施す**

Rorkの初期テンプレートそのままではなく、以下の変更を加えます：

- カラースキーム、フォント、アイコンをオリジナル化（ブランドガイドラインに沿わせる）
- ロゴ、スプラッシュスクリーン、起動画面をカスタム制作
- レイアウトの大幅な変更（ボトムナビから、サイドメニューに変更するなど）

```javascript
// Rork生成後のカスタマイズ例
// デフォルトカラーを上書き
const themeColors = {
  primary: '#1F4788',      // 独自ブランドカラー
  secondary: '#FFA500',    // アクセントカラー
  background: '#F5F5F5'    // 背景
};

// テンプレートのデフォルト色を完全に置き換える
export default StyleSheet.create({
  container: {
    backgroundColor: themeColors.background,
  },
  button: {
    backgroundColor: themeColors.primary,
  }
});
```

**2. 独自機能を上乗せする**

テンプレート機能だけでなく、以下のような独自機能を追加します：

- AIパワード機能（例：Rork Max × Claude APIで生成されたコンテンツ推奨）
- ソーシャル共有機能（ユーザーが生成したコンテンツをSNS共有）
- オフライン機能（インターネット接続なしでも機能）
- カスタムアナリティクス

これらは「このアプリならではの価値」をAppleに明示します。

**3. アプリ説明文で差別化を強調する**

App Storeの説明文で「何がユニークなのか」を明確に記述：

```markdown
× 悪い例：
「Rorkで作ったタスク管理アプリです」

○ 良い例：
「このアプリはAIを使ってあなたの優先度を自動判定します。
毎日のタスクを入力するだけで、重要度に基づいて自動ソート。
さらにチームメンバーとリアルタイム共有できます。」
```

### 再審査申し立てテンプレート

Appleから重複判定で拒否された場合、以下のテンプレートを使って異議申し立てます：

```
件名：App Store Connect — [アプリ名] リジェクト審査異議申し立て

本文：
---

Apple Review Team様

[アプリ名]（Bundle ID: [bundle-id]）のリジェクト（拒否）決定に対し、
異議申し立てさせていただきます。

指摘：「このアプリは既存のアプリと重複しており、十分なユニークな価値がない」

対応内容：

1. UIデザインの完全オリジナル化
   - カラースキーム：[ブランド名]の独自色「#1F4788」を採用
   - ロゴ・アイコン：[デザイナー名]によるオリジナル制作
   - レイアウト：ボトムナビゲーションから、サイドバー形式に全面刷新

2. 独自機能の実装
   - AI搭載タスク優先度判定：ユーザーが入力したタスクから、
     自動で重要度を分析しソート（既存テンプレートにはない機能）
   - リアルタイムチーム共有：複数ユーザー間でのライブデータ同期

3. App Storeスクリーンショットの刷新
   - 実機での動作画面のみを使用
   - プレースホルダーなし、完全な機能デモンストレーション

詳細は添付スクリーンショット及び、以下のApp Store説明を参照ください。
本アプリは十分なユニークな価値を持つと確信しており、
再度の審査をお願いいたします。

敬具

---
[署名]
```

## ガイドライン 2.1 — アプリの不完全性

### 問題：プレースホルダー機能、未実装の画面

Rorkで生成したアプリに、以下の問題があるとリジェクトされます：

- ログイン画面は実装されているがバックエンド接続がない
- 「設定」ボタンは存在するが、タップしても何も起こらない
- リンクが複数の画面に向かっておらず、機能が半実装状態

### 対処法

**1. 全ナビゲーションが正常に動作するか確認**

自分自身でアプリを隅から隅まで使用し、以下をテストします：

```javascript
// テスト項目チェックリスト
const testChecklist = {
  "全画面遷移": false,      // ← 全てのボタン・タップでナビゲーション
  "データ保存": false,       // ← フォーム入力がデータベースに確認できる
  "入力エラー": false,       // ← 無効な入力時にエラー表示される
  "ログアウト": false,       // ← ログイン機能が完全に動作する
  "画像読み込み": false,     // ← 画像が全て正常に表示される
  "オフライン対応": false    // ← インターネット接続なしでの動作を確認
};

// 全てtrueになるまでテスト
```

**2. プレースホルダーコンテンツを完全に削除**

Rorkのテンプレートには、以下のようなプレースホルダーが残っていないか確認：

```javascript
// 悪い例：テンプレートテキストがそのまま
<Text>"Lorem ipsum dolor sit amet..."</Text>

// 良い例：実際のアプリコンテンツ
<Text>"タスク管理を始めましょう"</Text>
```

**3. テスターがアプリを使うために必要な全情報を提供**

Appleのテスターがアプリを評価するには、ログイン認証情報やテストデータが必要です。以下を App Store Connect の「App Preview Information」に記載：

```
テスターアカウント情報：
---
メールアドレス: tester@example.com
パスワード: TestPassword123!
テストデータ：
- タスク「サンプルタスク1」が既にアプリ内に存在します
- 画像ギャラリーに3枚のサンプル画像が表示されます
注意：ログイン後、全ての機能は手動操作で動作確認できます
```

## ガイドライン 5.1.1 — データ収集とプライバシー

### 問題：プライバシーポリシー不足、ユーザーデータの扱いが不明確

Appleが最も厳格に審査する項目です。Rorkアプリがユーザーデータを利用する場合、明確なプライバシーポリシーが必須です。

### 対処法

**1. プライバシーポリシーを作成し、アクセス可能な URL に配置**

プライバシーポリシーは単にApp Storeに記述するのではなく、**WebサイトのURL（https://例.com/privacy）** として提供する必要があります。

```markdown
# [アプリ名] プライバシーポリシー

更新日：2026年3月26日

## 1. 収集されるデータ

このアプリが収集するデータ：
- ユーザー名、メールアドレス（ユーザーが入力）
- デバイスID（クラッシュレポート、分析用）
- 位置情報（ユーザーが位置情報許可を与えた場合のみ）
- アプリ内で作成されたコンテンツ

## 2. データの目的

収集されたデータは以下の目的のみに使用されます：
- ユーザーアカウント管理
- クラッシュレポート及びバグ修正
- アプリの改善・分析（Google Analytics）
- カスタマーサポート

## 3. 第三者への共有

ユーザーデータを以下の第三者に共有します：
- Google Analytics（ユーザー行動分析）
- Firebase（クラッシュレポート）
- AWS（データベースホスティング）

詳細は各サービスのプライバシーポリシーをご参照ください。

## 4. データの保護

全てのデータはSSL/TLSで暗号化され、
Amazonのセキュアサーバー上で管理されます。

## 5. ユーザーの権利

ユーザーはいつでも以下をリクエストできます：
- 自身のデータの確認（アプリ設定から「データ書き出し」）
- アカウント削除（サポート: support@example.com に連絡）
```

**2. App Tracking Transparency (ATT) 対応**

iOSアプリが位置情報、カメラ、マイク、または連絡先などの個人情報へのアクセスを求める場合、Info.plistに使用理由を記述し、ユーザーの明示的な許可を得る必要があります。

```xml
<!-- Info.plist -->
<key>NSLocationWhenInUseUsageDescription</key>
<string>このアプリはマップ表示機能であなたの現在地が必要です。
位置情報はアプリ内でのみ使用され、第三者と共有されません。</string>

<key>NSCameraUsageDescription</key>
<string>プロフィール写真を撮影する時のみカメラが必要です。</string>

<key>NSMicrophoneUsageDescription</key>
<string>音声メッセージを送信するのに必須です。</string>
```

**3. App Store Connect でプライバシー情報を完全に登録**

```
App Store Connect → アプリ情報 → プライバシー
---
- プライバシーポリシーURL: https://yourapp.com/privacy
- ユーザーのメールアドレスを収集する: YES
- デバイスの位置情報を使用: YES（位置情報許可を取得）
- 健康・フィットネス情報を使用: NO

データ共有パートナー：
 ☑ Google Analytics
 ☑ Firebase Crashlytics
 ☑ Amazon Web Services (データベースホスティング)
```

## ガイドライン 4.0 — 設計とUIの品質

### 問題：最小限の機能しかない、デザインが不完成

Appleはアプリの「品質」と「完成度」を評価します。Rorkで生成したテンプレートをそのまま提出すると、以下の理由で拒否されることがあります：

- 画面数が少なすぎる（3画面未満）
- デザインが洗練されていない
- ユーザー体験（UX）が悪い（遅い、直感的でない）

### 対処法

**1. アプリに最低限の画面数と機能を実装**

```
推奨最小構成：
✓ 起動画面（スプラッシュ）
✓ ホーム画面（メイン機能）
✓ 詳細画面（アイテムをタップしたときの詳細表示）
✓ ユーザー認証画面（ログイン / サインアップ）
✓ ユーザープロフィール画面
✓ 設定画面（言語、通知、アカウント削除）

最低でも6画面以上が目安です。3画面以下は
「不完全なアプリ」と判断されやすいです。
```

**2. パフォーマンス最適化**

遅いアプリはAppleから拒否されます。以下を確認：

```javascript
// 悪い例：大量のデータを一度にロード
async function loadAllData() {
  const users = await fetch('/api/users');      // 10,000ユーザー全部
  const posts = await fetch('/api/posts');      // 100,000投稿全部
  // アプリが数秒フリーズする...
}

// 良い例：ページネーション・遅延ロード
async function loadData(page = 1) {
  const data = await fetch(`/api/posts?page=${page}&limit=20`);
  // 最初は20件だけ読み込む
  // ユーザーがスクロールしたら次の20件を読み込む
}
```

**3. デザイン品質を高める**

テンプレートのデフォルトカラーやフォントでなく、以下をカスタマイズ：

- **フォント**: システムデフォルトでなく、Googleフォントなど独自フォント採用
- **カラー**: ブランドガイドラインに基づく統一感のあるカラーパレット
- **アイコン**: Material Icons や SF Symbols で統一

## ガイドライン 2.3.1 — 隠れた機能・制限事項

### 問題：重要な機能がメニューに見当たらない

Appleのテスターは限られた時間でアプリを評価します。見つけにくい機能は「実装されていない機能」と判断されてリジェクトされます。

### 対処法

**1. 全ての機能をナビゲーションメニューから直接アクセス可能に**

```
悪い例：
- ホーム画面に「設定」ボタンしかない
- エクスポート機能が3階層のメニュー奥深くにある

良い例：
- ボトムナビゲーションに「ホーム」「検索」「作成」「お気に入り」「設定」が見える
- よく使う機能は最大3タップで到達可能
```

**2. App Preview で全機能のデモンストレーション**

App Store に提出する際、「App Preview」（15秒〜30秒の動画）で、アプリの全主要機能を順番に示します。

```markdown
# App Preview スクリプト例（15秒）

0-3秒: スプラッシュ画面 → ホーム画面にロード
3-6秒: ホーム画面で「新規作成」ボタンをタップ
6-9秒: データ作成フローの実行
9-12秒: 作成されたアイテムをタップして詳細表示
12-15秒: 設定画面で個人カスタマイズを表示
```

## 再審査がリジェクトされた場合：最終チェックリスト

Appleに一度リジェクトされた後の再審査でも拒否される場合、以下を確認：

```markdown
# 再リジェクト対策チェックリスト

□ テスターアカウントが正常に動作するか、自分で再度ログイン確認
□ プレースホルダーテキストが完全に削除されたか
□ 全ナビゲーションが機能するか（全てのリンク・ボタンをテスト）
□ プライバシーポリシー URL が実在し、アクセス可能か
□ クラッシュレポートが含まれていないか（Xcode Organizer確認）
□ スクリーンショットが実機で撮影されたものか（モックアップでないか）
□ App Store 説明文に過大な誇大広告がないか
□ 外部SDKやAPIキーが正常に動作しているか
□ iOS最新バージョンでテストされたか
```

## まとめ

Rorkで作ったアプリがリジェクトされるのは、決して失敗ではなく、単なる**改善のステップ**です。Appleの厳しい基準は、App Storeのユーザー体験を守るためのもので、これをクリアすることで、あなたのアプリの品質も格段に上がります。

再審査申し立てを送るときのコツ：

1. **Appleの指摘を正確に理解する** —「何が問題なのか」を読み込む
2. **対応内容を具体的に説明する** — テンプレートのようなぼんやりした返答は避ける
3. **スクリーンショットで証拠を示す** — 「修正しました」だけでなく、修正後の画像を添付

Amazon で Rork関連のガイドを探すなら、[開発者向けテクニック書籍](https://www.amazon.co.jp/dp/B0D6FH9KNG/?tag=pinocchio-22)も参考になります。
