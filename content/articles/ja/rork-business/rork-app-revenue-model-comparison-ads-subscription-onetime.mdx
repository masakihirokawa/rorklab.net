アプリを作った後の最大の分岐点は、どうやって収益化するかです。2014年から個人でアプリ開発をしてきた経験から言えるのは、「正解は1つではない」ということ。でも、明確に「不正解」なパターンは確実に存在します。

このガイドでは、Rork で開発したアプリを広告、サブスクリプション、買い切りの3つのモデルで比較し、実際の市場データに基づいたシミュレーションを示します。あなたのアプリのジャンルと目指すビジネス規模に応じて、どのモデルが最適かを判断するための実践的な知識を提供します。

## 3つの収益モデル — 特性と選択基準

まず全体像を整理しましょう。

**広告（Ad-Based）** は、ユーザーが無料でアプリを使える代わりに、広告を表示して収益を得るモデルです。スケールが生まれやすく、MAU 5,000程度の規模でも月数万円の収入が期待できます。ただし、ユーザー体験を損なわないための工夫が必須です。

**サブスクリプション（Subscription）** は、定期課金でユーザーから直接収益を得るモデル。利益率が高く、継続的な収入が見込め、LTV（ライフタイムバリュー）も大きくなります。ただし、「継続的な価値提供」が不可欠で、チャーン（解約率）との戦いになります。

**買い切り（One-Time Purchase）** は、ユーザーが1回だけ支払うモデル。実装がシンプルで、アップデート後の価格変更も容易です。ただし、スケールに向かい難く、新規ユーザーの獲得がすべてになります。

それぞれのモデルに向いたアプリのジャンルは異なります。ニュースアプリやミニゲーム集は広告向き。生産性ツールやフォーカス系アプリはサブスク向き。特殊なツール系は買い切り向きです。

## モデル1: 広告収益（AdMob）の実態と最適化

広告収益で重要なのは **eCPM**（1,000インプレッション当たりの収益）と **フィルレート**（実際に広告が配信される比率）です。

日本のeCPMは、ジャンルにもよりますが、おおよそ $0.50〜$2.00 程度。ゲーム系は比較的高く、ツール系は低い傾向があります。一方、北米や西欧は $2.00〜$8.00、東南アジアは $0.20〜$0.80 という相場感です。

AdMob の収益を最大化するには、広告を「邪魔」だと感じさせない配置が鍵になります。

### リワード広告の最適配置

リワード広告は、ユーザーが「広告を見る」ことを主体的に選ぶため、クリックスルーレートが高く、eCPMも相対的に良好です。

ミニゲーム系アプリなら、各ゲーム終了後のスコア画面に「コイン +100 をもらう代わりに広告を見る」というボタンを配置するのが効果的です。ユーザーは得をする選択肢として認識し、離脱率も低くなります。

実装の例としては、以下のような流れになります。

```typescript
// React Native (Expo) での AdMob リワード広告実装例
import * as AdMob from 'expo-ads-admob';
import { RewardedAd, RewardedAdEventType } from 'google-mobile-ads-react-native';

const RewardAdButton = ({ onReward }) => {
  const rewardedInterstitial = RewardedAd.createForAdRequest(
    'ca-app-pub-3940256099942544/5224354917', // テスト用 Ad Unit ID
    {
      requestNonPersonalizedAdsOnly: false,
    }
  );

  const handleShowReward = async () => {
    try {
      await rewardedInterstitial.load();
      
      // ユーザーが広告視聴を完了した時
      rewardedInterstitial.addAdEventListener(
        RewardedAdEventType.EARNED_REWARD,
        () => {
          console.log('ユーザーがリワードを獲得しました');
          onReward(100); // コイン +100
        }
      );

      // 広告を表示
      rewardedInterstitial.show();
    } catch (error) {
      console.error('リワード広告の読み込み失敗:', error);
    }
  };

  return (
    <TouchableOpacity
      style={styles.button}
      onPress={handleShowReward}
    >
      <Text>広告を見てコイン +100</Text>
    </TouchableOpacity>
  );
};
```

このパターンは、生産性ツールなら「プレミアム機能を1回無料で使えるチケットを獲得」という形に応用できます。

### インタースティシャル広告の頻度設計

インタースティシャル広告（全画面広告）は、eCPMが高いため収益の主体になりやすいのですが、配置のバランスが難しいです。

一般的に、ゲームなら「ステージクリア後」、ツール系なら「処理完了後」など、ユーザーが自然に次のアクションに進む地点に配置します。ただし、連続で表示してしまうとユーザー離脱につながります。

実装戦略としては「2アクション以上の間隔を開ける」が目安です。つまり、ユーザーが少なくとも2つの別の機能を使うまでは、同じユーザーに広告を表示しないということです。

```typescript
// インタースティシャル広告の間隔管理
class AdFrequencyManager {
  private lastAdTime: number = 0;
  private actionCount: number = 0;
  private minIntervalMs: number = 30000; // 30秒
  private minActionsBetweenAds: number = 2;

  canShowInterstitial(): boolean {
    const timeSinceLastAd = Date.now() - this.lastAdTime;
    return (
      timeSinceLastAd > this.minIntervalMs &&
      this.actionCount >= this.minActionsBetweenAds
    );
  }

  recordAction(): void {
    this.actionCount++;
  }

  recordAdShown(): void {
    this.lastAdTime = Date.now();
    this.actionCount = 0;
  }
}

// 使用例
const adManager = new AdFrequencyManager();

const handleGameComplete = async () => {
  if (adManager.canShowInterstitial()) {
    await showInterstitialAd();
    adManager.recordAdShown();
  }
  adManager.recordAction();
  // ゲーム結果画面へ遷移
};
```

### eCPM を2倍にするウォーターフォール設計

複数の広告ネットワークを積み重ねる「ウォーターフォール」戦略を使うと、eCPM を大きく改善できます。

AdMob の次点として Google Ad Manager（Premium）、さらに Ironsource や AppLovin などの代替ネットワークを設定することで、各リクエストで最高入札額のネットワークから広告を引き出せます。これにより、理想的には eCPM が 50〜100% 向上します。

実装には Mediation SDK を使うのが一般的です。Expo では、複数の広告プロバイダーを管理するカスタムソリューションが必要になる場合もあります。

```typescript
// Mediation ネットワークの優先度設定（疑似コード）
const mediationConfig = {
  networks: [
    { id: 'admob', priority: 1, timeout: 3000 },
    { id: 'ironsource', priority: 2, timeout: 3000 },
    { id: 'applovin', priority: 3, timeout: 3000 },
  ],
};

const selectBestAdNetwork = async (): Promise<AdNetwork> => {
  const results = await Promise.all(
    mediationConfig.networks.map(async (network) => ({
      network,
      bid: await fetchBid(network.id),
    }))
  );
  
  return results.reduce((best, current) =>
    current.bid > best.bid ? current : best
  ).network;
};
```

## モデル2: サブスクリプション — 継続性の価値

サブスクリプションモデルの最大の利点は、ユーザーが月ごと、または年ごとに一定額を支払ってくれるため、 LTV が大きくなることです。

しかし「継続する理由」がなければ、ユーザーはすぐ解約します。成功するサブスクには、以下の3つの条件があります。

1. **明確な価値提供** — サブスク限定機能が「あると便利」ではなく「ないと困る」レベル
2. **継続的なアップデート** — 少なくとも月1回は何か新しい要素が追加される
3. **無料トライアル** — 最初の3日〜7日間を無料にして、実際の価値を体験させる

### 価格設計と市場相場

日本の相場感では、以下が目安です。

- **¥480/月** — 軽い生産性ツール、シンプルなユーティリティ
- **¥980/月** — 本格的な画像編集、ノート・TODOアプリ
- **¥1,980/月以上** — 専門的なツール、継続的なコンテンツ配信

一方、北米の App Store では以下が一般的です。

- **$2.99/月** — ライト向け
- **$4.99/月** — 標準向け
- **$9.99/月以上** — プレミアム向け

重要なのは「年払い」の選択肢を必ず用意することです。年払いは月払いの 10倍程度の粗利が生まれるため、ユーザーの 20% が年払いに転換するだけで、全体収益が 20% 向上します。

### StoreKit 2 を使った実装

React Native では、`react-native-purchases`（RevenueCat）がサブスク実装の標準になっています。ただし Expo 環境では制限があるため、EAS Build を使うか、Expo Modules API で直接 StoreKit 2 を叩く方が安定します。

```typescript
// Expo Modules での StoreKit 2 実装例（疑似コード）
import * as StoreKit from 'expo-store-kit';

const SubscriptionManager = {
  async startFreeTrial() {
    try {
      const product = await StoreKit.fetchProduct('premium_monthly');
      const result = await StoreKit.requestPurchase(product, {
        offerToken: product.introductoryPrice.offerToken, // 無料トライアル
      });
      return result;
    } catch (error) {
      console.error('購入エラー:', error);
    }
  },

  async checkSubscriptionStatus() {
    const transactions = await StoreKit.getTransactions();
    const activeSubscription = transactions.find(
      (tx) => tx.isActive && tx.productId === 'premium_monthly'
    );
    return !!activeSubscription;
  },

  async restorePurchases() {
    await StoreKit.syncPurchases();
  },
};
```

### 無料ユーザーから有料への転換フロー

サブスク成功の鍵は、無料トライアルをどうやって有料への転換に結びつけるかです。

効果的なパターンは、以下の流れです。

1. ユーザーがアプリをインストール → 無料で基本機能を使える
2. 3日目に「プレミアム機能」へのアクセス試行 → トライアル開始の案内
3. トライアル中の 5〜6日目に push 通知で「明日から課金されます」と案内
4. トライアル終了日の朝に「プレミアムの価値」をリマインド
5. 解約の場合は、「3日後に再度試す」オプションを提示

重要なのは、ユーザーが「騙された」と感じさせないことです。Apple と Google は「欺瞞的な実装」に厳しく対処します。

```typescript
// トライアル→有料転換フロー
const SubscriptionFlow = {
  async initiateFreeTrial(userId: string) {
    const trialEndDate = new Date();
    trialEndDate.setDate(trialEndDate.getDate() + 7); // 7日間

    // ユーザーの設定に記録
    await saveUserData(userId, {
      trialStartedAt: new Date(),
      trialEndsAt: trialEndDate,
      isTrialActive: true,
    });

    // 5日目に通知予約
    scheduleNotification({
      date: new Date(trialEndDate.getTime() - 2 * 24 * 60 * 60 * 1000),
      title: 'プレミアムはいかがでしたか？',
      body: 'あと2日でプレミアム機能をすべて利用できます',
    });

    return trialEndDate;
  },

  async handleTrialExpired(userId: string) {
    const userData = await getUserData(userId);
    
    // サブスクが確定する
    await StoreKit.requestPurchase(premiumProduct);
  },

  async handleTrialCancelled(userId: string) {
    // ユーザーが解約を選んだ場合、3日後に再度案内
    scheduleNotification({
      date: new Date(Date.now() + 3 * 24 * 60 * 60 * 1000),
      title: 'プレミアムを再度試してみませんか？',
      body: '初回なら7日間無料です',
    });
  },
};
```

## モデル3: 買い切り — シンプルさの価値

買い切りモデルは実装がシンプルです。App Store Pricing Tier を選んで、アプリの内部で機能を解放するだけです。

ただし、スケーラビリティという点では課題があります。買い切りの売上は、新規ユーザー数に完全に依存するため、一度ピークを迎えると減速します。サブスクや広告のように「既存ユーザーからの継続的な収入」が期待できません。

### 買い切り向きのジャンル

以下のようなアプリは買い切りに向いています。

- **ユーティリティ系** — 単機能の便利ツール（バーコードスキャナー、単位変換、PDF圧縮など）
- **教育・参考書系** — 講座や参考資料を「買う」という心理的バリア
- **ゲーム系** — 特に「完結型」のゲーム。MMOやソーシャル系は向きません
- **デザイン素材系** — Procreate、Affinity Photo など

### 価格帯別の売上傾向

私の経験と市場データを組み合わせると、以下の傾向があります。

| 価格 | ダウンロード数（相対） | 月間予想売上（MAU 5,000） |
|---|---|---|
| $1.99 | 100% | $300〜600 |
| $4.99 | 50% | $400〜800 |
| $9.99 | 20% | $300〜600 |
| $19.99 | 8% | $250〜400 |

つまり、単純な掛け算ではなく、価格が上がるごとに購買数が大きく落ちるため、一定の「最適価格」が存在します。多くのユーティリティは **$2.99〜$4.99** の帯域で最大売上を達成します。

## 3モデルの収益シミュレーション — MAU 5,000 の場合

では、同じアプリを3つの異なるモデルで展開した場合、月次収益がどう変わるかを見てみましょう。

前提条件：
- MAU（月間アクティブユーザー）: 5,000
- 広告表示による DAU（日間アクティブユーザー）: 3,000
- 1 DAU あたりの広告インプレッション: 8
- eCPM: $1.50（日本ターゲット、ゲーム系）
- サブスク価格: ¥480/月（約 $3.50）
- サブスク転換率: 5%（一般的な相場）
- サブスク月間チャーン率: 8%（標準的）
- 買い切り価格: $3.99
- 買い切り転換率: 3%

### 広告モデルの月次収益

```
DAU: 3,000
1 DAU あたりインプレッション: 8
合計インプレッション: 3,000 × 8 = 24,000

eCPM: $1.50
月次収益: (24,000 / 1,000) × $1.50 = $36

日本円換算（1 USD = 150 円）: ¥5,400
```

**月次純利益（30% プラットフォーム手数料後）**: 約 ¥3,780

広告モデルは「スケールしやすい」が「1ユーザーあたりの価値は低い」という特性が出ています。

### サブスクモデルの月次収益

```
MAU: 5,000
初月転換率: 5% = 250ユーザー
初月チャーン率: 8% = -20ユーザー
正味新規: 230ユーザー

定常状態（2ヶ月目以降）では、流入と流出が均衡：
流入: 5,000 × 5% = 250ユーザー
流出: 基盤 × 8%

定常状態の総サブスク数：
250 / 0.08 = 3,125ユーザー

月次収益: 3,125 × ¥480 = ¥1,500,000

Apple / Google 手数料（30%）後: ¥1,050,000
```

**月次純利益**: 約 ¥1,050,000

サブスクモデルは、初期段階では広告より低いですが、スケールすると圧倒的に有利になります。

### 買い切りモデルの月次収益

```
MAU: 5,000
購買転換率: 3% = 150ユーザー/月
購買価格: $3.99

月次収益: 150 × $3.99 = $598.50

日本円換算: 約 ¥90,000
Apple / Google 手数料（30%）後: ¥63,000
```

**月次純利益**: 約 ¥63,000

買い切りモデルは短期的には悪くありませんが、スケール性が低いため、MAU が 10,000 になっても新規転換分しか増えません。

### 3モデルの比較表

| 指標 | 広告 | サブスク | 買い切り |
|---|---|---|---|
| 初月純利益 | ¥3,780 | ¥350,000 | ¥63,000 |
| 定常月純利益 | ¥3,780 | ¥1,050,000 | ¥63,000 |
| 実装難易度 | 中 | 高 | 低 |
| ユーザー体験への負荷 | 高 | 低 | なし |
| スケーラビリティ | 低 | 高 | 低 |

## ハイブリッドモデル — 広告＋サブスクの設計

実務的には、純粋に1つのモデルに絞るのではなく、**ハイブリッド**を採用することが多いです。

典型的なパターンは「無料＋広告」と「サブスク（広告なし＋追加機能）」の組み合わせです。このモデルなら、マネタイズ意志がないユーザーは無料で使い続け、貴重なユーザーはサブスクで囲い込めます。

```typescript
// ハイブリッド課金モデルの実装例
export const FeatureAccess = {
  // 機能ごとのアクセス制御
  async canUseFeature(userId: string, featureId: string) {
    const user = await getUserData(userId);
    
    // プレミアム会員ならすべての機能が使える
    if (user.isPremium) return true;
    
    // 無料ユーザーはベーシック機能のみ
    const basicFeatures = ['core-editing', 'export-jpg'];
    return basicFeatures.includes(featureId);
  },

  // プレミアム限定機能へのアクセス試行
  async handlePremiumFeatureAccess(userId: string, featureId: string) {
    const user = await getUserData(userId);
    
    if (!user.isPremium) {
      // アップセル UI を表示
      showPremiumUpgradeSheet({
        feature: featureId,
        onProceed: async () => {
          // トライアル開始フロー
          await SubscriptionManager.startFreeTrial();
        },
      });
      
      return false;
    }
    return true;
  },

  // 広告非表示へのアップグレード
  async shouldShowAds(userId: string): Promise<boolean> {
    const user = await getUserData(userId);
    
    // プレミアム会員と広告削除購入済みなら非表示
    if (user.isPremium || user.adsRemoved) return false;
    
    // 無料ユーザーには表示
    return true;
  },
};
```

このモデルでは、サブスク転換を促進するため、以下のポイントが重要です。

1. **段階的な機能制限** — 最初の数回は全機能が使えるが、その後は限定される
2. **心理的トリガー** — 「あと1回この機能を使うにはプレミアムが必要」という設計
3. **価値の可視化** — プレミアム機能が「どれだけ生産性を上げるか」を数値で示す

## Rork での実装における留意点

### State Management との連携

課金状態（サブスク、広告削除など）を Redux や Zustand で管理すると、各コンポーネントからアクセスしやすくなります。

```typescript
// store.ts (Zustand の例)
import { create } from 'zustand';

interface MonetizationState {
  isPremium: boolean;
  adsRemoved: boolean;
  purchasedArticles: string[];
  
  // サブスク状態を更新
  setPremiumStatus: (status: boolean) => void;
  addPurchasedArticle: (articleId: string) => void;
}

export const useMonetization = create<MonetizationState>((set) => ({
  isPremium: false,
  adsRemoved: false,
  purchasedArticles: [],
  
  setPremiumStatus: (status) => set({ isPremium: status }),
  addPurchasedArticle: (id) =>
    set((state) => ({
      purchasedArticles: [...state.purchasedArticles, id],
    })),
}));
```

## ジャンル別の最適解マトリックス

最後に、アプリのジャンルごとに「どのモデルが最適か」をまとめました。

| ジャンル | 最適モデル | 理由 |
|---|---|---|
| ゲーム（カジュアル） | 広告＋サブスク（広告削除） | 短時間プレイで広告が邪魔になりやすい |
| ゲーム（コア）| 買い切り | 継続的なコンテンツ投資が必要 |
| 生産性ツール | サブスク | 毎日使うため LTV が高い |
| 写真編集 | サブスク | 継続的なフィルター追加で価値向上 |
| ユーティリティ | 買い切り | 単機能で更新頻度が低い |
| ニュース・コンテンツ | サブスク | 継続的なコンテンツ供給が前提 |
| SNS・コミュニティ | 広告 | スケール重視、マネタイズはあと |

## まとめ — 段階的な戦略が成功の鍵

10年以上、個人でアプリ開発をしてきた経験から言うと、最初から「完璧な収益モデル」を設計しようとするのは時間の無駄です。

大切なのは、以下の段階的なアプローチです。

1. **初期段階（MAU <1,000）** — 広告を実装し、ユーザーが定着するか検証する
2. **成長段階（MAU 1,000〜5,000）** — サブスクを追加し、有料転換のポテンシャルを測定
3. **スケール段階（MAU >5,000）** — サブスクを主軸に、広告を補助的に使う

Rork で開発したアプリなら、この流れは比較的スムーズに進めることができます。まず無料版で市場適合性を確認し、その後ビジネスモデルを追加していく。その過程で、あなたのアプリにとって「本当に正解」なモデルが見えてくるはずです。
