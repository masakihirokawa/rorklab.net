Rork でアプリを作り終えたあと、多くの人がぶつかる壁が「App Store 申請」です。開発は順調に進んだのに、審査でリジェクトされて途方に暮れた——という経験をした方は少なくありません。

このガイドでは、Rork で作ったアプリを App Store にリリースするまでの流れを、申請準備から審査通過まで順を追って解説します。リジェクトされやすいポイントと対処法も含めているので、初めてのリリースでも落ち着いて進められるはずです。

## 申請前に必ず確認すること

審査に出す前に、いくつかの準備が必要です。この段階を丁寧にやっておくと、リジェクトのリスクを大幅に下げられます。

### Apple Developer アカウントの準備

App Store にアプリを公開するには、Apple Developer Program への加入（年間 $99）が必要です。まだ加入していない場合は [Apple Developer](https://developer.apple.com) から登録しましょう。

加入後は以下を確認します:

- **Bundle ID の登録**: `com.あなたの名前.アプリ名` の形式で登録する
- **証明書とプロビジョニングプロファイルの設定**: Xcode の Signing & Capabilities で自動管理を有効にしておくのが最も安全
- **App Store Connect でのアプリ登録**: 新しいアプリ (App) を作成し、基本情報を入力する

### アプリ情報の準備

App Store Connect で必要な情報を事前に用意しておきましょう。

**必須項目:**

- アプリ名（最大 30 文字）
- サブタイトル（最大 30 文字）
- 説明文（最大 4,000 文字）
- キーワード（最大 100 文字、カンマ区切り）
- カテゴリ（メイン・サブ）
- 年齢制限レーティング
- プライバシーポリシーの URL

**スクリーンショット（必須）:**

iPhone 向けには最低限以下のサイズが必要です:

- 6.9 インチ（iPhone 16 Pro Max 相当）: 1320 × 2868px
- 6.5 インチ（iPhone 14 Plus 相当）: 1242 × 2688px

Canva や Figma でモックアップを作る方法が一般的ですが、Rork 自体のプレビュー画面をスクリーンショットして使う方法もシンプルです。

## Rork アプリを Xcode でビルドする

Rork は生成したコードを Xcode プロジェクトとして書き出せます。以下の手順でビルドし、App Store Connect へアップロードします。

### 手順

```
1. Rork でアプリが完成したら「Export」からコードをダウンロードする
2. ダウンロードした .zip を展開し、.xcodeproj または .xcworkspace を Xcode で開く
3. Signing & Capabilities でチームを選択する（Apple Developer アカウントに紐付いたもの）
4. Bundle Identifier が App Store Connect に登録したものと一致しているか確認する
5. スキームを「Any iOS Device」に設定し、Product → Archive を実行する
6. Organizer が開いたら「Distribute App」→「App Store Connect」→「Upload」を選択する
7. アップロード完了後、App Store Connect の「TestFlight」または「App Store」タブに反映される
```

ビルドが完了したら、App Store Connect の「ビルド」セクションにアプリが表示されるまで数分〜30分ほど待ちます。

## 審査申請の手順

ビルドがアップロードできたら、App Store Connect で審査に提出します。

```
1. App Store Connect → 対象アプリ → 「+バージョン」で新バージョンを作成する
2. アプリ情報（名前・説明・スクリーンショット等）を入力する
3. 「ビルド」セクションでアップロードしたビルドを選択する
4. 「審査のために提出」ボタンを押す
```

審査期間は通常 24〜48 時間ですが、繁忙期（年末年始・WWDC 前後など）は 1 週間以上かかることもあります。

## よくあるリジェクト理由と対処法

初回リリースで審査に落ちることは珍しくありません。落ち着いて理由を確認し、修正して再提出しましょう。

### リジェクト理由1: Guideline 2.1 — 機能が不十分

「アプリとして完成していない、または機能が少なすぎる」という理由です。

**対処法**: アプリの主要機能が一通り動作すること、ダミーデータだけでなく実際のコンテンツがあることを確認します。シンプルなアプリでも「このアプリは何のためにあるか」が明確に伝わる UI になっていれば通ることがほとんどです。

### リジェクト理由2: Guideline 5.1.1 — プライバシーポリシーの問題

ユーザーデータを収集する機能（ログイン・位置情報・連絡先など）がある場合、プライバシーポリシーは必須です。

**対処法**: [PrivacyPolicyGenerator.info](https://www.privacypolicygenerator.info) などの無料サービスで最低限のプライバシーポリシーを作成し、URL を App Store Connect に入力します。日本語のポリシーで通過できるケースも多いです。

### リジェクト理由3: Guideline 4.0 — デザインの問題

UI の品質が低い、または Apple の Human Interface Guidelines に沿っていないと判断された場合です。

**対処法**: Rork が生成するコードは基本的に HIG に準拠していますが、独自 UI を多用している場合は注意が必要です。審査員がすべての画面を触って確認するため、クラッシュするシナリオがないか事前にテストしましょう。

### リジェクト理由4: ログイン機能がある場合のデモアカウント

ログイン機能があるアプリでデモアカウント情報を提供していないと、審査員がアプリを試せずリジェクトされます。

**対処法**: App Store Connect の「App Review Information」セクションにデモ用のメールアドレスとパスワードを入力しておきます。

## リリース後にやること

審査が通過してアプリが公開されたら、以下を確認しましょう。

**1. クラッシュログの確認**: Xcode の Organizer または App Store Connect の「クラッシュ」タブで、公開後のクラッシュレポートを定期的にチェックします。

**2. レーティングとレビューの管理**: 最初のレビューが付くまでが一番不安ですが、ネガティブなレビューにも丁寧に返信することで、ユーザーの信頼を得られます。

**3. アップデートの計画**: App Store は定期的に更新されるアプリを評価します。小さな改善でもこまめにアップデートを出す習慣をつけましょう。

## まとめ

Rork で作ったアプリを App Store にリリースするプロセスは、最初は複雑に見えますが、一度経験すると次からはスムーズに進められます。

一番大切なのは「リジェクトを恐れすぎないこと」です。リジェクトは失敗ではなく、Apple からのフィードバックです。理由を読んで修正して再提出する、というサイクルを回すことが、App Store リリースの現実です。

最初の1本を出し切ることで、2本目・3本目は格段に楽になります。まずはリリースまでの道のりを完走することを目標に、一歩ずつ進んでいきましょう。
