壁紙アプリを最初にリリースしたのは、スマートフォンが普及し始めた頃でした。当時は「シンプルに作って、AdMob を貼るだけで稼げる」という時代でもあり、実際そうでした。ところが今は状況が違います。App Store には無数の壁紙アプリが存在し、品質の低いアプリは審査前に弾かれるか、ユーザーに無視されるかのどちらかです。

Rork の登場で実装コストは大幅に下がりました。ただ、実装が簡単になった分だけ競合も増えます。この状況で長期的に収益を上げるには、「ビジネスモデルの設計」と「コンテンツ運用の仕組み化」が重要です。

この記事では、私が壁紙系アプリを複数運営してきた経験をベースに、Rork での実装コードと合わせてご紹介します。初めて壁紙アプリを作る方にも、すでに運営していてマネタイズを改善したい方にも、実際に使える内容をお届けできれば嬉しいです。

## 壁紙アプリのビジネスモデル設計

壁紙アプリには大きく3つのビジネスモデルがあります。どれが正解かは「コンテンツの更新頻度」と「ターゲットユーザー層」によって変わります。

### freemium + プレミアムアンロック

無料で50〜100枚を提供し、それ以上はサブスクリプションまたは買い切りで解放するモデルです。最も一般的で、個人開発者に向いています。

重要なのは「無料コンテンツをケチらない」ことです。私の経験上、無料部分の品質が低いと「有料版もどうせこの程度」と判断されてしまいます。最も自信のあるコンテンツを無料で出すくらいの気持ちで設計するほうが、コンバージョン率が上がります。

### 完全無料 + リワード広告

すべての壁紙を無料で提供し、ダウンロード時に AdMob のリワード広告を表示するモデルです。「広告を1本見るとダウンロードできる」という UX は、課金への心理的ハードルがなく、DAU の多いアプリに向いています。

### 広告なし買い切り

$0.99〜$2.99 の買い切りで全壁紙を解放するシンプルなモデルです。コンテンツ追加が少なく、写真集のようなコンセプトのアプリに向いています。ただし継続収益にはならないため、長期運用には不向きです。

**私のおすすめは freemium + AdMob リワード広告の組み合わせ**です。無料ユーザーには「広告を見てダウンロード」の選択肢を提供し、プレミアム会員には広告なしの快適な体験を提供する。この設計が LTV（顧客生涯価値）を最大化しやすいと感じています。

## Rork でのプロジェクト設計とアーキテクチャ

### 技術スタックの選定

壁紙アプリに必要な技術を整理すると、以下のようになります。

- コンテンツストレージ: **Supabase Storage**（画像の CDN 配信）
- データベース: **Supabase**（カテゴリ・タグ・お気に入り管理）
- 課金: **RevenueCat**（サブスクリプションまたは買い切り）
- 広告: **Google AdMob**（リワード広告）
- 状態管理: **Zustand**（ダウンロード状態・お気に入り）

Rork に渡すプロンプトは、一度に全機能を要求するのではなく、画面・機能ごとに分けて依頼するのが効率的です。

```bash
# 段階的なプロンプト設計の例

# Step 1: 基本構造（データはモック）
"""
壁紙ギャラリーアプリを作ってください。
- ホーム: カテゴリタブ（自然・都市・抽象・ミニマル）
- 一覧: 2カラムグリッド、サムネイル表示
- 詳細: フルスクリーン、ダウンロードボタン、お気に入りボタン
- データはモックで OK。技術: React Native + Expo
"""

# Step 2: Supabase 連携（Step1完了後）
"""
先ほどの壁紙アプリの wallpapers テーブルを
Supabase から取得するよう変更してください。
useQuery で SWR キャッシュも使ってください。
"""

# Step 3: 課金機能（Step2完了後）
"""
RevenueCat を使って、isPremium フラグが true の壁紙は
プレミアム会員のみダウンロードできるよう制御してください。
"""
```

このように「動く状態を保ちながら段階的に機能追加する」アプローチが、Rork での開発品質を安定させるコツです。一度に複雑な要求をすると、AI が中途半端な実装を生成しやすくなります。

### Supabase テーブル設計

壁紙データの設計は、後から変更しにくい部分なので最初に慎重に考えましょう。

```sql
-- Supabase での壁紙テーブル設計

create table wallpapers (
  id uuid default gen_random_uuid() primary key,
  title text not null,
  category text not null check (
    category in ('nature', 'city', 'abstract', 'minimal')
  ),
  tags text[] default '{}',
  -- thumbnail と full を分けることで通信量を最適化
  thumbnail_url text not null,   -- 低解像度プレビュー (400×700px)
  full_url text not null,        -- 高解像度本体 (1080×1920px以上)
  is_premium boolean default false,
  is_active boolean default true,
  published_at timestamptz default now(),  -- 予約投稿に使用
  download_count integer default 0,
  created_at timestamptz default now()
);

-- RLS: 認証不要で公開壁紙を読み取り可能
alter table wallpapers enable row level security;

create policy "Public wallpapers are readable by anyone"
  on wallpapers for select
  using (is_active = true and published_at <= now());

-- ダウンロード数インクリメント用のストアドファンクション
create or replace function increment_download_count(wallpaper_id uuid)
returns void as $$
  update wallpapers
  set download_count = download_count + 1
  where id = wallpaper_id;
$$ language sql security definer;
```

`published_at` カラムを設けることで「予約投稿」的な運用が可能になります。週に一度まとめてコンテンツをアップロードし、毎日少しずつ公開するという仕組みです。これが後述の「コンテンツ更新の自動化」に繋がります。

## コンテンツ管理: Supabase Storage の活用

アプリバンドルに画像を直接含める実装を見かけることがありますが、壁紙アプリでは避けた方が無難です。理由は単純で、コンテンツを追加するたびに App Store 審査が必要になり、運用コストが積み上がっていくからです。

Supabase Storage に画像を置けば、アプリ審査なしで新しい壁紙を追加できます。これは長期運用においてとても重要な設計判断です。

### ストレージバケットの設定

```sql
-- Supabase Storage のバケット作成（SQL Editor で実行）

-- サムネイル用バケット（CDN キャッシュ有効化推奨）
insert into storage.buckets (id, name, public, file_size_limit, allowed_mime_types)
values (
  'wallpaper-thumbnails',
  'wallpaper-thumbnails',
  true,  -- パブリックアクセス
  2097152,  -- 2MB 上限
  array['image/jpeg', 'image/webp']
);

-- 高解像度画像用バケット
insert into storage.buckets (id, name, public, file_size_limit, allowed_mime_types)
values (
  'wallpaper-full',
  'wallpaper-full',
  true,
  20971520,  -- 20MB 上限
  array['image/jpeg', 'image/webp', 'image/png']
);

-- RLS ポリシー（認証不要で読み取り可能）
create policy "Public read access for thumbnails"
  on storage.objects for select
  using (bucket_id = 'wallpaper-thumbnails');

create policy "Public read access for full images"
  on storage.objects for select
  using (bucket_id = 'wallpaper-full');
```

Supabase の Storage URL は `{project_url}/storage/v1/object/public/{bucket}/{path}` の形式になります。この URL を `wallpapers` テーブルの `thumbnail_url` と `full_url` に保存します。

画像形式は **WebP** を優先することをおすすめします。JPEG と比べて20〜30%ファイルサイズが小さく、通信量とストレージコストを節約できます。

### ダウンロード機能の実装

ダウンロード処理は `expo-media-library` を使います。パーミッション処理とアルバム整理を省略すると審査でリジェクトされやすいので、この実装パターンをそのまま使ってください。

```tsx
// hooks/useWallpaperDownload.ts

import * as MediaLibrary from 'expo-media-library';
import * as FileSystem from 'expo-file-system';
import { useState, useCallback } from 'react';

interface DownloadResult {
  success: boolean;
  error?: string;
}

export function useWallpaperDownload() {
  const [isDownloading, setIsDownloading] = useState(false);
  const [progress, setProgress] = useState(0);  // 0〜100

  const downloadWallpaper = useCallback(async (
    wallpaperId: string,
    fullUrl: string,
  ): Promise<DownloadResult> => {
    setIsDownloading(true);
    setProgress(0);

    const tempFileUri = `${FileSystem.cacheDirectory}wallpaper_${wallpaperId}.jpg`;

    try {
      // 1. メディアライブラリの権限を確認・要求
      const { status } = await MediaLibrary.requestPermissionsAsync();
      if (status \!== 'granted') {
        return {
          success: false,
          error: '写真へのアクセス許可が必要です。設定 › プライバシー › 写真 から許可してください。',
        };
      }

      // 2. 高解像度画像をキャッシュディレクトリに一時ダウンロード
      const downloadResumable = FileSystem.createDownloadResumable(
        fullUrl,
        tempFileUri,
        {},
        (downloadProgress) => {
          const pct =
            downloadProgress.totalBytesWritten /
            downloadProgress.totalBytesExpectedToWrite;
          setProgress(Math.round(pct * 100));
        },
      );

      const result = await downloadResumable.downloadAsync();
      if (\!result?.uri) {
        throw new Error('ダウンロードファイルの取得に失敗しました');
      }

      // 3. カメラロールに保存
      const asset = await MediaLibrary.saveToLibraryAsync(result.uri);

      // 4. 「Wallpapers」アルバムに整理（なければ作成）
      //    ⚠️ この手順を省略すると写真がランダムに散らばり UX が悪化する
      let album = await MediaLibrary.getAlbumAsync('Wallpapers');
      if (\!album) {
        await MediaLibrary.createAlbumAsync('Wallpapers', asset, false);
      } else {
        await MediaLibrary.addAssetsToAlbumAsync([asset], album, false);
      }

      // 5. ダウンロード数インクリメント（失敗しても UX に影響なし）
      supabase
        .rpc('increment_download_count', { wallpaper_id: wallpaperId })
        .then()
        .catch((e) => console.warn('increment_download_count failed:', e));

      return { success: true };

    } catch (error) {
      console.error('Wallpaper download error:', error);
      return {
        success: false,
        error: error instanceof Error
          ? error.message
          : 'ダウンロードに失敗しました。しばらく時間をおいて再試行してください。',
      };

    } finally {
      setIsDownloading(false);
      setProgress(0);
      // 一時ファイルをクリーンアップ
      FileSystem.deleteAsync(tempFileUri, { idempotent: true }).catch(() => {});
    }
  }, []);

  return { downloadWallpaper, isDownloading, progress };
}
```

よくある間違いとして、`MediaLibrary.saveToLibraryAsync()` だけで終わらせてしまうケースがあります。写真がカメラロールに保存されても「Wallpapers アルバム」に整理されないと、ユーザーが後から壁紙を探しにくくなります。アルバム整理のステップは必ず含めてください。

## アプリ内課金の実装（RevenueCat）

RevenueCat を使った基本的な実装は [Rork RevenueCat アプリ内課金ガイド](/articles/rork-dev/rork-revenuecat-inapp-purchase) に詳しくまとめています。ここでは壁紙アプリ特有のパターンに絞ってご紹介します。

### プレミアム状態を管理するカスタムフック

```tsx
// hooks/usePremiumStatus.ts

import Purchases, { CustomerInfo } from 'react-native-purchases';
import { useEffect, useState } from 'react';

interface PremiumStatus {
  isPremium: boolean;
  isLoading: boolean;
  expirationDate?: Date | null;
}

export function usePremiumStatus(): PremiumStatus {
  const [isPremium, setIsPremium] = useState(false);
  const [isLoading, setIsLoading] = useState(true);
  const [expirationDate, setExpirationDate] = useState<Date | null>(null);

  const updateFromCustomerInfo = (info: CustomerInfo) => {
    // RevenueCat の Entitlement ID を "premium" に設定している前提
    const entitlement = info.entitlements.active['premium'];
    if (entitlement) {
      setIsPremium(true);
      setExpirationDate(
        entitlement.expirationDate ? new Date(entitlement.expirationDate) : null,
      );
    } else {
      setIsPremium(false);
      setExpirationDate(null);
    }
  };

  useEffect(() => {
    // 初回ロード
    setIsLoading(true);
    Purchases.getCustomerInfo()
      .then(updateFromCustomerInfo)
      .catch((e) => console.warn('RevenueCat getCustomerInfo failed:', e))
      .finally(() => setIsLoading(false));

    // 購入完了・リストア後のリアルタイム更新
    const listener = Purchases.addCustomerInfoUpdateListener(updateFromCustomerInfo);
    return () => { listener.remove(); };
  }, []);

  return { isPremium, isLoading, expirationDate };
}
```

このフックを壁紙詳細画面で使うと、プレミアムゲートの実装が次のようにシンプルになります。

```tsx

  <DownloadButton
    onPress={() => downloadWallpaper(wallpaper.id, wallpaper.fullUrl)}
    isLoading={isDownloading}

  />
);
```

### RevenueCat の製品設計

壁紙アプリの課金製品は、シンプルに以下の2種類から始めることをおすすめします。

- **月額プラン**: $0.99〜$1.99/月。継続収益の柱になります。
- **買い切りプラン**: $3.99〜$7.99。「定期的に課金したくない」層向けです。

App Store のサブスクリプションは「年間プランの方が月換算で安い」設計にすると、年額への誘導が自然にできます。

## AdMob リワード広告との組み合わせ

AdMob の詳細な実装は [Rork AdMob 広告収益化完全ガイド](/articles/rork-business/rork-admob-ad-monetization-complete-guide) を参照してください。ここでは壁紙アプリ特有の設計パターンをご紹介します。

### リワード広告でダウンロードを制御するフック

```tsx
// hooks/useDownloadWithRewardAd.ts

import {
  RewardedAd,
  RewardedAdEventType,
  TestIds,
} from 'react-native-google-mobile-ads';
import { useCallback, useEffect, useRef, useState } from 'react';
import { useWallpaperDownload } from './useWallpaperDownload';
import { usePremiumStatus } from './usePremiumStatus';

const AD_UNIT_ID = __DEV__
  ? TestIds.REWARDED
  : (process.env.EXPO_PUBLIC_ADMOB_REWARDED_ID ?? TestIds.REWARDED);

export function useDownloadWithRewardAd() {
  const { isPremium } = usePremiumStatus();
  const { downloadWallpaper, isDownloading, progress } = useWallpaperDownload();

  // ダウンロード対象を保持する ref（広告コールバック内で使用）
  const pendingDownload = useRef<{ id: string; url: string } | null>(null);

  const [ad] = useState(() =>
    RewardedAd.createForAdRequest(AD_UNIT_ID, {
      requestNonPersonalizedAdsOnly: true,  // GDPR 対応
    }),
  );

  const [adLoaded, setAdLoaded] = useState(false);

  useEffect(() => {
    const unsubscribeLoaded = ad.addAdEventListener(
      RewardedAdEventType.LOADED,
      () => setAdLoaded(true),
    );

    const unsubscribeEarned = ad.addAdEventListener(
      RewardedAdEventType.EARNED_REWARD,
      () => {
        // リワード獲得 = 広告視聴完了 → ダウンロード実行
        if (pendingDownload.current) {
          const { id, url } = pendingDownload.current;
          pendingDownload.current = null;
          downloadWallpaper(id, url);
        }
      },
    );

    // 初回広告をロード
    ad.load();

    return () => {
      unsubscribeLoaded();
      unsubscribeEarned();
    };
  }, [ad, downloadWallpaper]);

  const handleDownload = useCallback(async (
    wallpaperId: string,
    fullUrl: string,
  ) => {
    if (isPremium) {
      // プレミアム会員は広告なしで即ダウンロード
      await downloadWallpaper(wallpaperId, fullUrl);
      return;
    }

    // 無料ユーザー: 広告をロード済みなら表示
    if (adLoaded) {
      pendingDownload.current = { id: wallpaperId, url: fullUrl };
      setAdLoaded(false);
      ad.show().catch(() => {
        // 広告の表示に失敗した場合はダウンロードにフォールバック
        pendingDownload.current = null;
        downloadWallpaper(wallpaperId, fullUrl);
        ad.load();
      });
    } else {
      // 広告未ロードの場合もダウンロードを完了させる（UX を損なわない）
      await downloadWallpaper(wallpaperId, fullUrl);
      ad.load();  // 次回のために非同期でロード
    }
  }, [ad, adLoaded, isPremium, downloadWallpaper]);

  return { handleDownload, isDownloading, progress };
}
```

**よくある落とし穴**: `RewardedAd.show()` は `async/await` では待機できません。`EARNED_REWARD` イベントを listener で監視する必要があります。また、リワード広告は視聴途中でキャンセルした場合は `EARNED_REWARD` が発火しません。「広告をキャンセルしたらダウンロードできない」という設計にするか、「どちらにしてもダウンロードできる（広告はオプション）」という設計にするかは、アプリのコンセプトに合わせて決めてください。私は後者を採用しています。理由は、広告視聴を強制すると低評価レビューにつながりやすいからです。

## App Store 審査で壁紙アプリが引っかかるポイント

壁紙アプリは、一般的なアプリと比べて審査で特に確認される項目があります。過去の経験から、注意すべき点をまとめました。

### 著作権とライセンスの証明

審査チームから「この画像の使用権はありますか？」という質問が来ることがあります。Unsplash・Pexels などのフリー素材を使う場合でも、ライセンス証明を提出できる準備をしておいてください。AI 生成画像（Midjourney・Stable Diffusion など）を使う場合は、そのツールの商用利用ポリシーを事前に確認することが必須です。

### プライバシーマニフェストの設定

`expo-media-library`（写真へのアクセス）を使う壁紙アプリでは、`app.json` への記述が欠かせません。以下の両方を設定してください。

```json
{
  "expo": {
    "ios": {
      "infoPlist": {
        "NSPhotoLibraryAddUsageDescription": "壁紙を写真アルバムに保存するために使用します。",
        "NSPhotoLibraryUsageDescription": "保存済みの壁紙を「Wallpapers」アルバムに整理するために写真へのアクセスが必要です。"
      }
    }
  }
}
```

説明文は具体的である必要があります。「アプリの機能に使用します」のような曖昧な説明は審査で指摘されます。

### コンテンツポリシーの確認

壁紙アプリは「主にコンテンツを提供するアプリ」として審査されます。以下のコンテンツは明確にNGです。

- 成人向けコンテンツ（4+/12+/17+ レーティングの適切な設定も必要）
- 実在する著名人の写真（タレント・スポーツ選手など）
- ロゴ・商標が含まれる画像（ブランドの壁紙は著作権侵害）
- フォトリアルな武器・暴力表現

ゲームアセット風のキャラクターや架空のシーンは問題ありませんが、「どのレーティング対象者が使うか」を意識してコンテンツを選ぶようにしましょう。

## コンテンツ更新の自動化と長期収益化

壁紙アプリで収益を継続させる鍵は「コンテンツの鮮度」です。毎週新しい壁紙を追加することで、ユーザーが定期的にアプリを開く動機を作れます。

### 予約投稿とプッシュ通知の連携

先ほど設計した `published_at` カラムを活用した予約投稿の仕組みと、OneSignal を使った通知の連携方法をご説明します。

Supabase の Database Webhooks（または Edge Functions のスケジューラ）を使って、「今日公開された壁紙があれば OneSignal に通知を送る」ワークフローを構築できます。

```typescript
// supabase/functions/notify-new-wallpapers/index.ts
// Supabase Edge Function（週1回のスケジュール実行）

import { createClient } from '@supabase/supabase-js';

const supabase = createClient(
  Deno.env.get('SUPABASE_URL')\!,
  Deno.env.get('SUPABASE_SERVICE_ROLE_KEY')\!,
);

Deno.serve(async () => {
  // 過去24時間以内に公開された壁紙を取得
  const yesterday = new Date();
  yesterday.setDate(yesterday.getDate() - 1);

  const { data: newWallpapers, error } = await supabase
    .from('wallpapers')
    .select('id, title, thumbnail_url')
    .gte('published_at', yesterday.toISOString())
    .eq('is_active', true)
    .order('published_at', { ascending: false })
    .limit(5);

  if (error || \!newWallpapers?.length) {
    return new Response('No new wallpapers', { status: 200 });
  }

  // OneSignal へ通知リクエスト
  const onesignalResponse = await fetch(
    'https://onesignal.com/api/v1/notifications',
    {
      method: 'POST',
      headers: {
        'Content-Type': 'application/json',
        'Authorization': `Basic ${Deno.env.get('ONESIGNAL_REST_API_KEY')}`,
      },
      body: JSON.stringify({
        app_id: Deno.env.get('ONESIGNAL_APP_ID'),
        included_segments: ['All'],
        contents: { en: 'New wallpapers added\!', ja: '新しい壁紙が追加されました' },
        headings: { en: 'Wallpaper Update', ja: 'ウォレットアップデート' },
        big_picture: newWallpapers[0].thumbnail_url,  // 通知に画像を添付
      }),
    },
  );

  const result = await onesignalResponse.json();
  return new Response(JSON.stringify(result), { status: 200 });
});
```

「新しい壁紙が追加されました」という通知は、開封率が高い通知カテゴリの一つです。週に一度のタイミングで送るだけで、週間 DAU が安定しやすくなります。

### コンテンツ更新サイクルの設計

長期運営を見据えると、コンテンツ追加のワークフローを最初から設計しておくことが大切です。私が採用しているサイクルは次の通りです。

- 月〜水: 次週分の壁紙を選定・加工（WebP 変換・サイズ最適化）
- 木曜日: Supabase Storage にアップロード、`wallpapers` テーブルに `published_at` を来週月〜日で設定
- 月〜日: 自動で1日3〜5枚ずつ公開、月曜日に OneSignal 通知

この仕組みを一度作れば、週2〜3時間の作業でコンテンツを維持できます。

## まとめ — 最初に取り組むべき1つのこと

壁紙アプリの実装で最初にやってほしいことは1つだけです。**画像を Supabase Storage に外出しすること**です。

アプリバンドルに画像を直接含めてしまうと、壁紙を追加するたびに App Store 審査が必要になります。最初の数十枚は問題なくても、100枚・200枚とコンテンツが増えるにつれて運用コストが重くなっていきます。

Supabase Storage + `published_at` の予約投稿の仕組みを最初に作っておけば、コンテンツの追加はアップロードだけで完結します。これがあるかないかで、半年後の運用負荷がまったく違います。

アプリの実装から始めたくなる気持ちはよく分かりますが、まずコンテンツ基盤を整えることが長期的な安定運用への近道だと感じています。
