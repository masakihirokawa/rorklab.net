今週も Rork Lab をご覧いただき、ありがとうございます。

4月最終週から5月最初の金曜日にかけての一週間は、**Rork で生成したアプリを「本番品質のプロダクト」に押し上げるための周辺機能の記事が一気に厚みを増した**一週間になりました。Rork Max 自体の使い方ではなく、その先の iOS / Android のネイティブ世界とどう繋ぐか、ここを正面から書いていく時期に入ってきた気がしています。

並行して、Rork のことを半年から1年使い込んだ上での正直なレビュー記事を2本公開しました。新しめのツールはどうしても「使ってみた」記事が先行しがちですが、長く触ってきた立場からあらためて評価軸を整理する記事は、これからの Rork ユーザーにとっても役立つはずだ、と思って書いたものです。

## 今週の中心テーマ：ネイティブ機能の本番運用

今週もっとも記事数が多かったのは、Rork のアプリにネイティブの周辺機能を組み込んでいくための実装ガイドでした。

[Rork アプリで Bluetooth Low Energy（BLE）通信を実装する](/articles/rork-dev/rork-bluetooth-low-energy-ble-implementation-guide) では、`react-native-ble-plx` を使った IoT デバイス連携を、スキャン・接続・キャラクタリスティック読み書きまで一通り扱いました。BLE は「とりあえず接続できた」までは早いものの、再接続戦略やバックグラウンド挙動で詰まりやすい領域なので、本番運用で頻繁に踏むポイントを優先して書いています。

[Rork × Vision Camera v4](/articles/rork-dev/rork-vision-camera-v4-frame-processor-production-guide) は、QR・OCR・ML 推論まで通すフレームプロセッサの実装ノートです。Vision Camera は v4 で API がかなり整理された一方、フレームプロセッサ周りはまだ情報が少なく、本番投入したときに性能が出ない問題に当たりやすい部分でした。私自身、撮影系アプリで実際に使った経験ベースで書いたので、机上の解説だけにならないように意識しています。

[Rork Max で Share Extension を実装する完全ガイド](/articles/rork-dev/rork-max-share-extension-implementation-guide) と、[Rork Max でスクリーンタイム制御アプリを作る完全ガイド](/articles/rork-dev/rork-max-screentime-family-controls-app-complete-guide) は、Rork Max が SwiftUI ネイティブを生成できるようになって初めて現実的になった領域を扱った記事です。Share Extension は App Group と URL Scheme、Screen Time は Family Controls / DeviceActivity / ManagedSettings と、それぞれ Apple 純正フレームワークの組み合わせを正面から書きました。**Rork Max 以前は個人開発で手を出しづらかった領域に、確実に手が届くようになってきている**のを実感します。

[Rork Max で画面録画と配信を実装する](/articles/rork-dev/rork-max-replaykit-screen-recording-broadcast-guide) では、ReplayKit と Android の MediaProjection をペアで扱い、ゲーム配信やチュートリアルアプリで需要が高い「画面録画＋配信」機能の本番運用パターンをまとめました。[Rork × CallKit + PushKit で VoIP 通話アプリを実装する](/articles/rork-dev/rork-callkit-pushkit-voip-implementation-guide) はその延長線上の記事で、ネイティブ着信 UI と専用 Push の組み合わせは、Rork が生成する JS レイヤーだけでは絶対に到達できない部分です。

そして個人的に「これは記事として残しておきたかった」のが [Rork × MetricKit でアプリの本番品質を計測する完全実装ガイド](/articles/rork-dev/rork-metrickit-production-diagnostics-complete-guide) です。MetricKit はクラッシュ・ハング・電池消費を Apple が直接計測して返してくれる仕組みですが、Rork のような AI ビルダーで生成したアプリほど「ユーザーの端末で何が起きているか」を可視化する基盤が大事になります。アプリを出した後の運用品質を一段引き上げるための一本としてご活用ください。

## AI 連携の実装が、確実に深いレイヤーへ進んでいます

AI 連携の記事もこの一週間で大きく進みました。

[Rork × ElevenLabs で『話せるAIキャラクター』アプリを作る実践ガイド](/articles/rork-ai/rork-elevenlabs-voice-ai-companion-app-guide) は、テキスト入力の AI チャットからもう一段先、リアルタイム音声でキャラクターと会話するアプリを設計するための記事です。コンパニオン系アプリは個人開発でもっとも投資対効果が高いジャンルの一つだと思っていて、収益化の見込みも含めて書きました。

[Rork で Apple FoundationModels を使う](/articles/rork-ai/rork-foundation-models-on-device-llm-offline-ai-guide) は、iOS 26 で利用可能になった Apple のオンデバイス LLM をプロダクトに組み込む実装ガイドです。クラウド LLM の API コストが利益を圧迫しがちな個人開発において、**端末側で完結する LLM の選択肢が現実になった**のは大きな転換点でした。プライバシー要件が強いアプリでも導入しやすくなります。

そして今週もっとも長い記事になったのが [Rork で『AI が自動でアルバム化する写真整理アプリ』を作る完全実装ガイド](/articles/rork-ai/rork-ai-photo-organizer-clip-vision-vector-search-guide) です。Vision API + CLIP + pgvector を組み合わせたベクトル検索パイプラインを、本番品質で設計するためのプレミアム記事として書きました。AI で作るアプリといっても、実装が「外部 API を叩くだけ」で止まっているケースが多い中で、**意味検索やクラスタリングまで踏み込むと一気に体験の質が変わる**ということを伝えたかった記事です。

少し毛色が違うところでは、[Rork × AI モデレーション完全実装](/articles/rork-dev/rork-ugc-ai-moderation-reporting-blocking-production-guide) も AI 連携の重要な記事でした。UGC 系アプリで通報・自動審査・段階的制裁を本番投入するための実装をまとめています。AI を「コンテンツ生成」だけでなく「運営の安全弁」としても使えるようになったのは大きな前進です。

## サブスクリプション基盤を「自分で持つ」選択肢が現実的になってきました

今週は、サブスクリプション関連の意欲作も2本公開しました。

[RevenueCat の固定費が気になり始めた人のための、App Store Server Notifications V2 自前運用ガイド](/articles/rork-dev/rork-app-store-server-notifications-v2-self-hosted-subscription) は、サブスク収益が伸びてきた個人開発者ほど刺さる内容になっていると思います。**RevenueCat は最初の半年は神様のように便利ですが、収益が伸びるほど月額固定費が重くなる**のが正直なところで、自前運用に切り替える判断材料を実装パターンとあわせて書きました。

そしてその次のレイヤーとして書いたのが [サブスクの『今・誰が・どの権利を持つか』を取り違えない](/articles/rork-dev/rork-subscription-entitlement-state-machine-design-guide) です。エンタイトルメント（権利）の状態機械を設計するという、地味ですがバグると致命的な領域を扱ったプレミアム記事です。サブスク・買い切り・トライアル・返金が絡むと急激に複雑度が上がるので、状態遷移を設計図として残しておく価値はかなり高いです。

[Rork で iOS アプリ申請を通すための App Tracking Transparency 実装](/articles/rork-dev/rork-app-tracking-transparency-att-prompt-guide) も、収益化と表裏一体のテーマです。ATT は審査リジェクトの大きな原因の一つですが、許可率がそのまま広告収益に直結するので、両立させる実装の勘所を整理しました。

## クロスプラットフォーム展開の現実

Android 周りの記事も今週はまとまった本数を出しました。

[Rork Max で Android 版アプリを出すには？クロスプラットフォーム展開の現実解](/articles/rork-ai/rork-max-android-cross-platform-deployment-guide) は、Rork Max が SwiftUI ネイティブを生成できるようになった一方、Android はどうなっているのか、という多くの方が気になっている部分に正面から答えた記事です。**「両 OS で完全に同じ体験を作る」ことを諦め、目的に応じて重みづけを変える**現実解を書きました。

[Rork 製アプリで iOS 18 のダーク／ティントアイコンと Android のテーマアイコンに対応する](/articles/rork-dev/rork-themed-app-icon-ios-android-guide) も今週公開したクロスプラットフォーム記事で、最近のホーム画面トレンドに対応するための一本です。ティントアイコンは「対応しているアプリ」と「していないアプリ」の見栄えがホーム画面ではっきり差として出るので、地味ですがブランディング上の効果は意外と大きいです。

## Rork に対する正直なレビューを2本書きました

そして今週の見どころが、Rork そのものを評価しなおすレビュー記事を2本続けて公開した点です。

[Rork AI アプリビルダーを実際に使い込んだ正直レビュー — 2026年4月時点の強み・弱み・向いている人](/articles/rork-basics/rork-ai-app-builder-honest-review-2026) は、新規にこれから Rork を試そうとしている方向けに、**強みと弱みを公平に書く**ことを意識した記事です。良い面だけ強調する記事はもう読者に見抜かれているので、合わない用途・避けたほうが良いケースも具体的に書きました。

もう一本の [Rork は『AIモバイルアプリビルダー』として何を変えたのか — 半年運用してわかった評価軸](/articles/rork-ai/rork-ai-mobile-app-builder-review-2026) は、より長く使い込んだ立場から、AI モバイルアプリビルダーというカテゴリ自体への評価軸を提案する記事です。Lovable・Bolt・Claude Design・FlutterFlow と並べたときに、Rork のどこが本質的に違うのかを言語化することを試みました。

このタイプの記事は普段あまり書かないのですが、**読者の判断材料を提供することと、過度に持ち上げないこと**を両立できると、長期的には信頼につながると思っています。
